# 博客系列写作助手

这是一个通用的博客系列写作工具，帮助你从零开始创建结构化的技术博客系列。

## 何时使用

当你需要：

- 创建一个系列技术博客（如"React 进阶"、"NestJS 实战"等）
- 制定详细的博客写作计划
- 按照统一格式规范撰写博客文章
- 自动处理文章间的链接关系

## 写作核心理念

### 你在为谁写作

读者是混合受众：有 1-2 门语言经验的开发者，想快速上手新技术。

- 入门部分用简单语言引入，不假设读者了解当前技术
- 深入部分要满足有经验开发者的期望，不能停留在表面

### 写作三原则

1. **先说为什么，再说怎么做**
   - 每个知识点都从"为什么需要它"开始
   - 解释它解决了什么问题、没有它会怎样
   - 对比替代方案，说明选择它的理由

2. **用一个完整案例贯穿全文**
   - 每篇文章选一个真实场景作为主线案例
   - 案例要有真实感（业务场景、数据结构、边界条件）
   - 随着文章深入，案例逐步演进和复杂化

3. **像和朋友聊天一样写技术**
   - 用"你可能会遇到..."、"这里有个坑..."的对话语气
   - 可以用类比解释抽象概念
   - 避免文档式的干巴巴叙述

## 反模式清单（写作时必须避免）

以下是写作中必须主动避免的问题：

### 内容层面

- ❌ **文档搬运** — 只是把官方文档翻译一遍，没有自己的理解和补充
- ❌ **代码堆砌** — 代码占比超过 50%，解释性文字太少
- ❌ **玩具示例** — 九九乘法表、Hello World、学生成绩表这类教科书式例子
- ❌ **空洞总结** — 用复选框重复章节标题作为总结
- ❌ **假 FAQ** — 常见问题部分是编造的，不是基于真实困惑
- ❌ **只说怎么做** — 介绍了 API 用法，但不说为什么这样设计、什么时候该用

### 风格层面

- ❌ **机械模板** — 每篇文章都一模一样的"前言→概念→实战→最佳实践→常见问题→总结"
- ❌ **AI 腔调** — 大量使用 emoji 装饰、过度使用分隔线、千篇一律的语气
- ❌ **假亲切** — 用 emoji 和感叹号假装友好，但内容没有温度
- ❌ **面面俱到** — 试图覆盖所有知识点，导致每个都讲不深

### 代码层面

- ❌ **完整代码贴图** — 贴 50+ 行完整代码，只说一句"这是基本用法"
- ❌ **无上下文代码** — 代码前后没有引导语和解读
- ❌ **万能数据** — 用 `{ id: 1, name: "Alice" }` 这类假数据

## 文章结构指南

### 灵活模板（非固定模板）

不要机械地套用固定结构。根据主题特点灵活组织，但每篇文章必须包含以下要素：

#### 1. 开头：用一个真实问题抓住读者（必须）

从一个具体的、读者可能遇到的问题或困惑开始：

❌ 错误示范：

> "本文将介绍 Go 的控制流语句，包括 if、switch、for 等。"

✅ 正确示范：

> "你刚接手一个 Go 项目，看到前辈写了这样的代码：
> `for range ch { ... }`，心里嘀咕这到底是啥？
> 别急，Go 的循环和你在 Java 里写的还真不一样。"

#### 2. 中间：渐进式展开（核心）

按照"为什么 → 是什么 → 怎么用 → 进阶 → 踩坑"的节奏展开，但不需要每个知识点都走完全部步骤。

**简单知识点**：为什么需要 → 怎么用 → 注意事项（3 步够了）
**核心知识点**：为什么需要 → 概念解释 → 代码演示 → 进阶用法 → 踩坑（5 步）
**复杂知识点**：为什么需要 → 概念解释 → 设计思路 → 代码实现 → 进阶优化 → 对比替代方案 → 生产建议（7 步）

#### 3. 代码示例：服务于叙述，而非反过来

- 代码是用来论证观点的，不是文章的主体
- 代码比例控制在 40% 以内
- 每段代码前要有引导语（为什么要看这段代码）
- 每段代码后要有解读（这段代码的关键点是什么）
- 不要堆砌完整代码，只展示关键部分，省略无关的样板代码

❌ 错误：贴一整段 50 行代码，下面写一句"这就是基本用法"
✅ 正确：先说"注意这里我们用到了 X 来解决 Y 问题"，再展示关键 10 行

#### 4. 结尾：trade-off 总结（必须）

不要用复选框式总结重复章节标题。要回答：

- 这个技术/方案的 **适用场景** 和 **不适用场景**
- 和其他方案的 **对比**（什么时候该用 A，什么时候该用 B）
- 一个 **实践建议**（来自真实经验，不是泛泛而谈）

❌ 错误：

> "本文介绍了... ✓ if语句 ✓ switch语句 ✓ for循环"

✅ 正确：

> "什么时候用 switch 而不是 if-else？当你有 3 个以上
> 离散值需要匹配时，switch 更清晰。但如果你的条件
> 涉及范围判断（score > 60），老老实实用 if。"

### Frontmatter 格式

每篇文章的 frontmatter 保持以下格式：

```yaml
---
title: "序号 文章标题"
date: "YYYY-MM-DD"
summary: "一句话总结（100字以内）"
tags: ["标签1", "标签2"]
category: "{主题}"
cover: ""
draft: false
series: "{系列名称}"
seriesOrder: N
---
```

## 案例设计指南

### 如何选择案例

每篇文章需要一个**贯穿全文的主线案例**。选择标准：

1. **来自真实业务场景**，不是教科书式的题目
2. **足够简单**让读者快速理解，但**足够真实**包含边界情况
3. **能自然演进** — 随着文章深入，案例可以逐步叠加复杂度

### 案例主题参考

不同技术领域适合不同类型的案例：

| 技术领域 | 好的案例                     | 差的案例                |
| -------- | ---------------------------- | ----------------------- |
| Web 框架 | 博客评论系统、待办事项 API   | Hello World、打印字符串 |
| 数据库   | 电商订单查询、用户权限管理   | 学生成绩表、员工信息表  |
| 缓存     | 热门商品库存、秒杀场景       | 设置 key-value、计数器  |
| 并发     | 聊天室消息分发、限流器       | 数字求和、交替打印      |
| 类型系统 | 表单验证器、API 响应类型     | 简单条件类型、联合类型  |
| ORM      | 博客文章与标签关联、订单流水 | 用户 CRUD、商品 CRUD    |
| 消息队列 | 订单状态变更通知、邮件发送   | 生产者消费者模型        |

### 案例演进模式

主线案例要随着文章逐步演进。示例：

```
第1篇（基础）：实现一个简单的用户注册接口
第2篇（参数校验）：给注册接口加上输入验证
第3篇（数据库）：把用户数据存到数据库
第4篇（错误处理）：处理注册失败的各种情况
第5篇（进阶）：加上邮箱验证和密码重置
```

每篇开头的案例从上一篇结束的状态继续，读者能感受到自己在"构建一个真实项目"。

### 案例中的数据要真实

❌ 错误：`{ id: 1, name: "Alice", age: 25 }`
✅ 正确：`{ userId: "usr_8x9k2m", username: "zhang_wei", plan: "pro", createdAt: "2025-03-15T08:30:00Z" }`

数据要像真实生产环境中的数据：有业务含义、有合理的命名、有边界值。

## 系列文章连贯性要求

### 知识依赖链

系列文章不是多篇独立文章的集合，而是一条渐进的学习路径。

**每篇文章必须做到：**

1. **开头回顾前篇** — 用 2-3 句话自然地回顾上一篇的核心内容，不要机械重复
   - ✅ "上一篇我们用 `useEffect` 拉取了数据，但你可能发现每次组件重渲染都会重新请求。今天来解决这个性能问题。"
   - ❌ "上一篇我们介绍了 useEffect，本文将介绍 useMemo。"

2. **引用前篇的案例** — 如果前篇有主线案例，本篇要在其基础上继续演进
   - 让读者感受到自己在持续构建一个项目
   - 不需要重复前篇的完整代码，只展示新增/修改的部分

3. **预告下一篇** — 结尾自然过渡到下一篇要解决的问题
   - ✅ "现在我们的接口能正常工作了，但密码还是明文存储的。下一篇我们来看看如何安全地处理用户密码。"
   - ❌ "下一篇我们将介绍数据库操作。"

### 跨篇知识引用格式

当需要引用前面文章讲过的知识点时，使用这种格式：

> 之前在 [第 3 篇：错误处理](/posts/{category}/{slug}) 中我们讲过，
> Go 的 error 是值类型。这里我们用同样的思路来处理...

不要假设读者一定读过前面的文章，给一个简短的上下文提示。

### 系列整体节奏

计划书阶段就要规划好系列的整体节奏：

```
基础篇（2-3篇）：快速上手，建立直觉
  → 每篇一个独立的小案例，降低入门门槛

核心篇（3-5篇）：深入原理，主线案例开始
  → 引入一个贯穿后续所有文章的完整案例

进阶篇（3-5篇）：真实场景，案例持续演进
  → 案例越来越复杂，接近生产环境

实战篇（1-2篇）：完整项目，整合所有知识
  → 用一个完整项目收尾，展示如何综合运用
```

前几篇可以用独立小案例降低门槛，从核心篇开始再用贯穿案例。

## 质量自检清单

每篇文章写完后，逐项检查：

### 必须全部通过

- [ ] 文章是否从一个具体问题或场景开始，而不是"本文将介绍..."
- [ ] 每个核心知识点是否解释了"为什么需要它"
- [ ] 是否有一个贯穿全文的真实案例（不是玩具示例）
- [ ] 代码占比是否在 40% 以内
- [ ] 每段代码前是否有引导语，代码后是否有解读
- [ ] 结尾是否有 trade-off 分析（适用/不适用场景、对比替代方案）
- [ ] 是否回顾了前篇内容并在其基础上继续（系列文章）
- [ ] 案例中的数据是否像真实生产环境的数据

### 建议通过（至少满足 5/7）

- [ ] 是否用了类比来解释抽象概念
- [ ] 是否包含了"踩坑"经验或常见错误
- [ ] 是否对比了不同方案的优劣
- [ ] 文章是否有自然的起承转合，而不是机械分段
- [ ] 是否标注了技术版本要求（如"Go 1.21+"）
- [ ] 进阶部分是否足够深入，能满足有经验的开发者
- [ ] 整体读起来是否像人在分享经验，而不是文档翻译

## 使用前准备

在激活此 skill 前，请准备以下信息：

1. **博客主题**（如：React、Vue、TypeScript、Node.js 等）
2. **技术栈**（可选，列出要覆盖的核心技术）
3. **实战项目**（可选，一个贯穿系列的项目示例）

## 工作流程

### 第一阶段：制定计划书

**目标**：创建详细的博客系列计划文档

**步骤**：

1. **创建计划文件**
   - 文件路径：`content/posts/{主题}/plan.md`
   - 参考格式：`content/posts/nestjs/plan.md`

2. **计划书内容结构**：

```markdown
# {主题} 渐进式学习计划

## 背景

- 说明为什么选择这个主题
- 技术栈选择标准
- 核心技术列表

## 博客目录结构

列出所有文章的文件名和标题

## 学习阶段划分

根据难度和主题，将文章分为多个阶段（基础→核心→进阶→实战→部署）

## 每篇文章大纲

为每篇文章提供：

- 学习目标
- 核心问题（这篇文章要解决什么问题）
- 主线案例（贯穿全文的真实场景）
- 内容要点
- 与前篇的关联（如何承接上一篇）

## 系列节奏规划

- 基础篇：用独立小案例降低门槛
- 核心篇：引入贯穿案例
- 进阶篇：案例持续演进
- 实战篇：完整项目收尾

## 预期效果

读者学完后将具备的能力
```

3. **确认计划**
   - 展示计划书给用户确认
   - 根据反馈调整

### 第二阶段：撰写博客文章

**目标**：按计划逐篇撰写博客

撰写每篇文章时，遵循以下优先级：

1. 阅读计划书中本篇的大纲
2. 如果不是系列第一篇，阅读上一篇的完整内容，了解案例进展
3. 按照【文章结构指南】和【案例设计指南】撰写
4. 撰写完成后，按照【质量自检清单】逐项检查
5. 保存文件

### 第三阶段：命名规范

**文件命名规则**：

- 格式：`序号-kebab-case标题.mdx`
- 示例：`01-getting-started.mdx`、`02-advanced-concepts.mdx`
- 位置：`content/posts/{category}/`

**标题命名规则**：

- 格式：`序号 标题`
- 示例：`01 环境搭建与第一个项目`

## 代码示例标准

1. **关键优先** — 只展示关键代码，省略样板代码，不要贴 50+ 行完整代码
2. **有引导有解读** — 代码前说明目的，代码后点出关键点
3. **渐进式** — 从简单到复杂，逐步深入
4. **注释精准** — 只在真正需要解释的地方加注释，不要每行都注释
5. **错误处理** — 展示正确的错误处理方式
6. **版本标注** — 注明代码适用的技术版本（如 "// Go 1.21+"）

## 格式要求

- 使用 Markdown 格式
- 代码块指定语言（`typescript`、`go`、`bash` 等）
- 适当使用表格、列表、引用等格式
- 添加适当的空行提升可读性，但不要过度使用分隔线

## 自动化任务

当用户请求写博客时，按以下步骤执行：

1. **询问主题信息**
   - 主题名称
   - 技术栈
   - 实战项目（可选）

2. **生成计划书**
   - 根据主题和学习路径，自动生成计划书
   - 参考类似主题的计划书结构
   - 划分学习阶段，规划系列节奏

3. **用户确认**
   - 展示计划书
   - 等待用户反馈和确认

4. **创建目录结构**

   ```
   content/posts/{主题}/
   ├── plan.md
   ├── 01-xxx.mdx
   ├── 02-xxx.mdx
   └── ...
   ```

5. **自动连续撰写**
   - **重要：一旦开始撰写，自动完成所有文章，不需要用户确认**
   - 按计划书顺序逐篇撰写
   - 每篇撰写前先阅读上一篇内容（确保连贯性）
   - 每篇完成后立即开始下一篇
   - 保存到对应目录
   - 每篇完成后简单提示进度

6. **完成提示**
   - 显示所有已完成的文章列表
   - 提示如何预览和发布

## 示例

用户输入：

```
我想写一个 React Hooks 系列博客，面向初学者
```

Agent 执行：

1. 生成 `content/posts/react/plan.md`
2. 确认计划后，创建 `01-hooks-introduction.mdx`
3. 撰写内容，遵循写作核心理念、案例设计指南、质量自检清单
4. 继续撰写后续文章...

## 技能输出

### 输出物

1. **计划书** (`plan.md`)
2. **系列博客文章** (`.mdx` 文件)
3. **目录结构** (按规范组织)

### 质量保证

- 所有代码示例关键且有意义
- 文章有真实案例和 trade-off 分析
- 系列文章前后连贯、渐进深入
- 通过质量自检清单

## 注意事项

- 写作过程中随时可以暂停，用户可以要求修改
- 每篇文章写完后应保存，避免丢失
- 保持系列文章的一致性（术语、风格、难度、主线案例）
- 代码示例要考虑兼容性（注明版本要求）

## 禁止事项

- **禁止在文章中手动添加"上一篇/下一篇"导航链接**：系统已有自动的"下一篇"导航组件（在 `page.tsx` 中实现），会根据文件序号自动生成。在文章末尾手动添加导航链接是冗余的且会导致 404 错误。
- **禁止使用 `.mdx` 文件相对路径作为链接**：如 `./02-xxx.mdx`、`../02-xxx.mdx` 等。这些路径在 MDX 编译后不会被转换为正确的 URL，会导致 404。如果需要在文章中引用其他文章，使用博客 URL 格式：`/posts/{category}/{slug}`。
- **禁止在文章末尾添加"继续阅读"、"下一步"等指向 `.mdx` 文件的链接**：同样会导致 404。
- **禁止使用复选框式总结**：不要用 `✅ xxx` 的列表重复章节标题作为总结。
- **禁止使用玩具示例**：不要用九九乘法表、Hello World、学生成绩表等教科书式例子。
- **禁止堆砌完整代码**：代码占比不超过 40%，只展示关键部分。
