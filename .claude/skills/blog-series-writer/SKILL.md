# 博客系列写作助手

这是一个通用的博客系列写作工具，帮助从零开始创建结构化的博客系列，覆盖技术、健康、心理、投资、生活等各类主题。

## 博客位置

- 路径：`~/Desktop/code/wy5211.github.io/content/posts/{category}/`
- 文件格式：`.mdx`
- 项目构建：`cd ~/Desktop/code/wy5211.github.io && npm run build`

## 何时使用

当你需要：

- 创建一个系列博客（技术类或非技术类）
- 制定详细的博客写作计划
- 按照统一格式规范撰写博客文章
- 自动处理文章间的链接关系

## 写作核心理念

### 你在为谁写作

**主要服务自己。** 博客是你整理思路、加深理解的工具。写的过程本身就是学习——如果你写完一篇文章觉得自己对某个话题理解更深了，这篇就成功了。

次要是给有同样困惑的人看的。你的读者可能是一个遇到类似问题、想找点思路的人，不一定是专业领域的从业者。

### 写作三原则

1. **先说为什么，再说怎么做**
   - 每个知识点都从"为什么需要关注它"开始
   - 解释它解决了什么问题、没有它会怎样
   - 对比替代方案，说明选择它的理由

2. **用一个真实场景贯穿全文**
   - 每篇文章选一个真实场景作为主线
   - 场景要有真实感，能让读者（或未来的自己）产生代入感
   - 随着文章深入，场景逐步展开和复杂化

3. **像和自己对话一样写**
   - 用"你可能会遇到..."、"这里有个坑..."的语气
   - 可以用类比解释抽象概念
   - 不装权威，不堆砌术语，说人话

## 文章类型与写作指南

根据主题类型，选择对应的写作策略：

### 类型 A：技术类

适用：编程语言、框架、数据库、DevOps、AI/LLM 等

- 读者是有一定开发经验的人，想快速上手新技术
- 入门部分用简单语言引入，深入部分满足有经验者的期望
- 代码示例服务于叙述，代码占比控制在 40% 以内
- 每段代码前有引导语（为什么要看这段），代码后有解读（关键点是什么）
- 注明技术版本要求（如 "Go 1.21+"）

### 类型 B：健康/身体类

适用：身体信号解读、运动、营养、睡眠、精力管理等

- 用第一人称身体感受开头，让读者产生共鸣（"你是不是也有过这种感觉..."）
- 科学原理用通俗语言解释，避免堆砌医学术语
- 给出可执行的日常建议，而不是空泛的"注意健康"
- 区分"正常现象"和"需要就医的信号"，不制造健康焦虑
- 案例用日常场景：加班后的疲惫、体检报告的困惑、睡眠质量下降等

### 类型 C：心理学/认知类

适用：情绪管理、认知偏差、习惯养成、人际沟通等

- 用一个具体的日常困境开头（不敢拒绝同事、拖延症、情绪循环等）
- 心理学概念用生活场景解释，不照搬教科书定义
- 区分"正常的心理反应"和"需要关注的信号"
- 每篇文章给一个具体可操作的方法，而不是泛泛的"要积极"
- 案例用真实的人际场景和工作生活场景

### 类型 D：投资/理财类

适用：投资理念、指数基金、资产配置、财务自由等

- 用真实数据说话，不贩卖焦虑也不画饼
- 先讲"为什么"（通胀、复利、风险），再讲"怎么做"
- 对比不同方案的优劣，说明适用条件
- 案例用真实的市场数据和可操作的策略
- 避免推荐具体产品，讲思路和方法论

### 类型 E：生活/运营类

适用：极简主义、社交媒体运营、职业发展等

- 从一个具体的痛点或观察切入
- 分享真实经验和方法，不写鸡汤
- 给出可操作的步骤，不要泛泛而谈

## 反模式清单（写作时必须避免）

### 通用内容层面

- ❌ **文档搬运** — 只是翻译或复述已有内容，没有自己的理解和补充
- ❌ **空洞总结** — 用复选框重复章节标题作为总结
- ❌ **假 FAQ** — 常见问题部分是编造的，不是基于真实困惑
- ❌ **只说怎么做** — 介绍了方法，但不说为什么这样想、什么时候该用
- ❌ **贩卖焦虑** — 用极端案例制造恐慌，却不给解决思路
- ❌ **鸡汤灌水** — 用大量励志语录填充篇幅，没有实质内容

### 通用风格层面

- ❌ **机械模板** — 每篇文章一模一样的"前言→概念→实战→总结"
- ❌ **AI 腔调** — 大量 emoji 装饰、过度使用分隔线、千篇一律的语气
- ❌ **假亲切** — 用 emoji 和感叹号假装友好，但内容没有温度
- ❌ **面面俱到** — 试图覆盖所有知识点，导致每个都讲不深

### 技术类代码层面（仅类型 A）

- ❌ **完整代码贴图** — 贴 50+ 行完整代码，只说一句"这是基本用法"
- ❌ **无上下文代码** — 代码前后没有引导语和解读
- ❌ **万能数据** — 用 `{ id: 1, name: "Alice" }` 这类假数据
- ❌ **玩具示例** — 九九乘法表、Hello World、学生成绩表

## 文章结构指南

### 灵活模板（非固定模板）

不要机械套用固定结构。根据主题特点灵活组织，但每篇文章必须包含以下要素：

#### 1. 开头：用一个真实问题抓住读者（必须）

从一个具体的、读者可能遇到的问题或困惑开始。

**技术类示例：**

> "你刚接手一个 Go 项目，看到前辈写了 `for range ch { ... }`，
> 心里嘀咕这到底是啥？别急，Go 的循环和你在 Java 里写的还真不一样。"

**健康类示例：**

> "体检报告一切正常，但就是觉得不对劲？
> 你可能正处在亚健康和慢性炎症的灰色地带。"

**心理类示例：**

> "那天晚上十点半，我刚关上电脑准备回家，
> 隔壁工位的同事走过来说：'帮我看看这个方案呗。'
> 我其实连看一眼的力气都没有了。但我听到自己说：'行，你发我吧。'"

#### 2. 中间：渐进式展开（核心）

按照"为什么 → 是什么 → 怎么用 → 进阶 → 踩坑"的节奏展开，但不需要每个知识点都走完全部步骤。

**简单知识点**：为什么需要 → 怎么用 → 注意事项（3 步够了）
**核心知识点**：为什么需要 → 概念解释 → 代码演示/案例展开 → 进阶用法 → 踩坑（5 步）
**复杂知识点**：为什么需要 → 概念解释 → 设计思路 → 代码实现/深入分析 → 进阶优化 → 对比替代方案 → 实践建议（7 步）

#### 3. 结尾：trade-off 总结（必须）

不要用复选框式总结重复章节标题。要回答：

- 这个方法/技术的 **适用场景** 和 **不适用场景**
- 和其他方案的 **对比**（什么时候该用 A，什么时候该用 B）
- 一个 **实践建议**（来自真实经验，不是泛泛而谈）

❌ 错误：> "本文介绍了... ✓ if语句 ✓ switch语句 ✓ for循环"

✅ 正确：> "什么时候用 switch 而不是 if-else？当你有 3 个以上

> 离散值需要匹配时，switch 更清晰。但如果你的条件
> 涉及范围判断（score > 60），老老实实用 if。"

### Frontmatter 格式

每篇文章的 frontmatter 保持以下格式：

```yaml
---
title: "序号 文章标题"
date: "YYYY-MM-DD"
summary: "一句话总结（100字以内）"
tags: ["标签1", "标签2"]
category: "{主题}"
cover: ""
draft: false
series: "{系列名称}"
seriesOrder: N
---
```

## 案例设计指南

### 通用原则

每篇文章需要一个**贯穿全文的主线案例/场景**。选择标准：

1. **来自真实场景**，不是教科书式的题目
2. **足够简单**让读者快速代入，但**足够真实**包含值得思考的点
3. **能自然演进** — 随着文章深入，场景可以逐步展开

### 按类型的案例参考

| 类型             | 好的案例                     | 差的案例                |
| ---------------- | ---------------------------- | ----------------------- |
| 技术类（Web）    | 博客评论系统、待办事项 API   | Hello World、打印字符串 |
| 技术类（数据库） | 电商订单查询、用户权限管理   | 学生成绩表、员工信息表  |
| 技术类（缓存）   | 热门商品库存、秒杀场景       | 设置 key-value、计数器  |
| 技术类（ORM）    | 博客文章与标签关联、订单流水 | 用户 CRUD、商品 CRUD    |
| 健康/身体        | 加班后的疲惫感、体检报告困惑 | "要保持健康"            |
| 心理学           | 不敢拒绝同事、拖延到深夜     | "要积极面对生活"        |
| 投资             | 10年前的50万现在值多少       | "投资很重要"            |
| 习惯养成         | 每天早起失败的20次尝试       | "养成好习惯"            |

### 案例中的数据要真实

**技术类**：`{ userId: "usr_8x9k2m", username: "zhang_wei", plan: "pro" }`
**投资类**：用真实的市场数据（如沪深300指数历史收益率）
**健康类**：用日常可感知的指标（睡眠时长、体重变化、精力评分）

### 系列案例演进模式

主线案例要随着文章逐步演进：

```
技术类：
第1篇（基础）：实现一个简单的用户注册接口
第2篇（参数校验）：给注册接口加上输入验证
第3篇（数据库）：把用户数据存到数据库

健康类：
第1篇：识别身体发出的信号
第2篇：理解信号背后的原因
第3篇：从饮食开始调整
第4篇：运动和睡眠的配合
```

## 系列文章连贯性要求

### 知识依赖链

系列文章不是多篇独立文章的集合，而是一条渐进的路径。

**每篇文章必须做到：**

1. **开头回顾前篇** — 用 2-3 句话自然地回顾上一篇的核心内容
   - ✅ "上一篇我们了解了皮质醇是怎么在体内循环的，今天来看看怎么通过饮食和运动来调节它。"
   - ❌ "上一篇我们介绍了皮质醇，本文将介绍调节方法。"

2. **引用前篇的案例** — 如果前篇有主线案例，本篇要在其基础上继续
   - 让读者感受到自己在持续深入一个话题
   - 不需要重复前篇的完整内容，只展示新增/演进的部分

3. **预告下一篇** — 结尾自然过渡到下一篇要解决的问题
   - ✅ "现在我们知道怎么识别身体信号了，但光知道还不够。下一篇来看看具体怎么做才能改善。"
   - ❌ "下一篇我们将介绍改善方法。"

### 跨篇知识引用格式

当需要引用前面文章讲过的知识点时，使用博客 URL 格式：

> 之前在 [第 3 篇：错误处理](/posts/{category}/{slug}) 中我们讲过，
> Go 的 error 是值类型。这里我们用同样的思路来处理...

不要假设读者一定读过前面的文章，给一个简短的上下文提示。

### 系列整体节奏

计划书阶段就要规划好系列的整体节奏：

```
入门篇（2-3篇）：快速建立直觉，用日常场景引入
  → 降低门槛，让读者觉得"这个话题和我有关"

核心篇（3-5篇）：深入原理，主线场景展开
  → 引入一个贯穿后续所有文章的完整视角

进阶篇（3-5篇）：真实场景，案例持续演进
  → 越来越接近真实生活中的复杂情况

实践篇（1-2篇）：整合所有知识，给出行动指南
  → 用一个完整的实践方案收尾
```

## 质量自检清单

每篇文章写完后，逐项检查：

### 必须全部通过

- [ ] 文章是否从一个具体问题或场景开始，而不是"本文将介绍..."
- [ ] 每个核心知识点是否解释了"为什么需要关注它"
- [ ] 是否有一个贯穿全文的真实场景（不是泛泛而谈）
- [ ] 结尾是否有 trade-off 分析（适用/不适用场景、对比替代方案）
- [ ] 是否回顾了前篇内容并在其基础上继续（系列文章）
- [ ] 文章读起来是否自然流畅，像在和朋友聊天而不是在写文档

### 技术类额外检查（仅类型 A）

- [ ] 代码占比是否在 40% 以内
- [ ] 每段代码前是否有引导语，代码后是否有解读
- [ ] 案例中的数据是否像真实生产环境的数据

### 建议通过（至少满足 5/7）

- [ ] 是否用了类比来解释抽象概念
- [ ] 是否包含了"踩坑"经验或常见错误
- [ ] 是否对比了不同方案的优劣
- [ ] 文章是否有自然的起承转合，而不是机械分段
- [ ] 进阶部分是否足够深入
- [ ] 是否给了一个具体的行动项（读者/未来的自己看完能做什么）
- [ ] 整体读起来是否像人在分享经验，而不是搬运文档

## 工作流程

### 第一阶段：制定计划书

**目标**：创建详细的博客系列计划文档

**步骤**：

1. **创建计划文件**
   - 文件路径：`~/Desktop/code/wy5211.github.io/content/posts/{category}/plan.md`

2. **计划书内容结构**：

```markdown
# {主题} 系列计划

## 背景

- 为什么写这个主题（自己想搞清楚什么）
- 核心话题列表
- 文章类型（A/B/C/D/E）

## 博客目录结构

列出所有文章的文件名和标题

## 阶段划分

根据深度和逻辑，将文章分为多个阶段（入门→核心→进阶→实践）

## 每篇文章大纲

为每篇文章提供：

- 学习目标（读完这篇能搞清楚什么）
- 核心问题（这篇文章要回答什么）
- 主线场景（贯穿全文的真实场景）
- 内容要点
- 与前篇的关联（如何承接上一篇）

## 系列节奏规划

- 入门篇：用日常场景引入，建立关联感
- 核心篇：深入原理，主线场景展开
- 进阶篇：场景持续演进，接近真实复杂度
- 实践篇：整合所有知识，给出行动指南
```

3. **确认计划**
   - 展示计划书给用户确认
   - 根据反馈调整

### 第二阶段：撰写博客文章

**目标**：按计划逐篇撰写博客

撰写每篇文章时，遵循以下优先级：

1. 阅读计划书中本篇的大纲
2. 如果不是系列第一篇，阅读上一篇的完整内容，了解场景进展
3. 根据文章类型选择对应的写作指南（类型 A/B/C/D/E）
4. 按照【文章结构指南】和【案例设计指南】撰写
5. 撰写完成后，按照【质量自检清单】逐项检查
6. 保存文件

### 第三阶段：命名规范

**文件命名规则**：

- 格式：`序号-kebab-case标题.mdx`
- 示例：`01-getting-started.mdx`、`02-advanced-concepts.mdx`
- 位置：`~/Desktop/code/wy5211.github.io/content/posts/{category}/`

**标题命名规则**：

- 格式：`序号 标题`
- 示例：`01 环境搭建与第一个项目`

### 第四阶段：构建验证

- 运行 `cd ~/Desktop/code/wy5211.github.io && npm run build`
- 如果构建失败，根据错误信息修复文章，重新构建
- 重复直到构建通过

### 第五阶段：提交推送

- commit message 格式：`feat: add {系列名称} series with N articles`
- 推送到远程仓库（`git push`）
- 显示推送结果和所有已完成的文章列表

## 自动化任务

当用户请求写博客时，按以下步骤执行：

1. **询问主题信息**
   - 主题名称
   - 文章类型（技术/健康/心理/投资/生活）
   - 核心话题（可选）

2. **生成计划书**
   - 根据主题和类型，自动生成计划书
   - 划分阶段，规划节奏
   - 根据文章类型选择对应的写作策略

3. **用户确认**
   - 展示计划书
   - 等待用户反馈和确认

4. **创建目录结构**

   ```
   ~/Desktop/code/wy5211.github.io/content/posts/{category}/
   ├── plan.md
   ├── 01-xxx.mdx
   ├── 02-xxx.mdx
   └── ...
   ```

5. **自动连续撰写**
   - **重要：一旦开始撰写，自动完成所有文章，不需要用户确认**
   - 按计划书顺序逐篇撰写
   - 每篇撰写前先阅读上一篇内容（确保连贯性）
   - 每篇完成后立即开始下一篇
   - 保存到对应目录
   - 每篇完成后简单提示进度

6. **构建验证**
   - 所有文章撰写完成后，运行构建检查
   - 修复问题直到构建通过

7. **提交并推送**

## 注意事项

- 每篇文章写完后应保存，避免丢失
- 保持系列文章的一致性（术语、风格、难度、主线场景）
- 技术类代码示例要注明版本兼容性

## 禁止事项

- **禁止在文章中手动添加"上一篇/下一篇"导航链接**：系统已有自动的"下一篇"导航组件，手动添加会导致 404。
- **禁止使用 `.mdx` 文件相对路径作为链接**：如 `./02-xxx.mdx`。使用博客 URL 格式：`/posts/{category}/{slug}`。
- **禁止在文章末尾添加指向 `.mdx` 文件的链接**：同样会导致 404。
- **禁止使用复选框式总结**：不要用 `✅ xxx` 的列表重复章节标题。
- **禁止使用玩具示例**（技术类）：不要用九九乘法表、Hello World、学生成绩表等。
- **禁止堆砌完整代码**（技术类）：代码占比不超过 40%。
- **禁止贩卖焦虑**（非技术类）：不要用极端案例制造恐慌。
- **禁止鸡汤灌水**（非技术类）：不要用大量励志语录填充篇幅。
