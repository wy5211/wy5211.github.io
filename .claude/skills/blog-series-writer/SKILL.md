# 博客系列写作助手

帮你从零创建结构化的博客系列，覆盖技术、健康、心理、投资、生活等主题。

## 基本信息

- 文章路径：`~/Desktop/code/wy5211.github.io/content/posts/{category}/`
- 文件格式：`.mdx`，命名 `序号-kebab-case标题.mdx`（如 `01-getting-started.mdx`）
- 标题格式：`序号 标题`（如 `01 环境搭建与第一个项目`）
- 构建命令：`cd ~/Desktop/code/wy5211.github.io && pnpm run build`

### Frontmatter 模板

```yaml
---
title: "序号 文章标题"
date: "YYYY-MM-DD"
summary: "一句话总结（100字以内）"
tags: ["标签1", "标签2"]
category: "{主题}"
cover: ""
draft: false
series: "{系列名称}"
seriesOrder: N
---
```

## 核心理念

**主要服务自己。** 博客是整理思路、加深理解的工具。写完一篇如果你对话题理解更深了，这篇就成功了。次要是给有同样困惑的人看——读者可能只是遇到类似问题想找点思路，不一定是从业者。

### 写作四原则

1. **先给具体，再做解释** —— 每个章节用一个场景、一段对话、一组数据、一个代码块开场，先摆"东西"再解释它意味着什么。
   - ❌ "缓存是一种用空间换时间的优化手段。"
   - ✅ "秒杀开始 3 秒，数据库 CPU 飙到 100%，这时候缓存能救命——它的本质是用空间换时间。"
2. **先说为什么，再说怎么做** —— 从"为什么需要关注它"开始，解释它解决了什么问题、没有它会怎样，对比替代方案。
3. **用一个真实场景贯穿全文** —— 每篇选一个有代入感的真实场景作主线，随文章深入逐步展开。
4. **像和自己对话一样写** —— 用"你可能会遇到…""这里有个坑…"的语气，用类比解释抽象概念，不装权威，说人话。

## 语调捕捉（写新系列前必做）

目标是让新文章听起来像「你」，而不是通用 AI。先分清两层东西：

- **你真实的语气**：偏短句还是长句、爱不爱反问、会不会自嘲、有没有个人观点——值得保留。
- **旧 skill 留下的套路**：固定开场结构、千篇一律的过渡、复选框总结——这是 AI 残留，不是你，必须剥离。

不加区分地「读旧文模仿语气」会把套路一起继承，越写越像，最终收敛成平均化的 AI 模子。所以参考来源有优先级：

1. **最可信：你手写的东西**（聊天记录、笔记、评论、社交媒体、和我对话的原话）——真实指纹，优先从这里提取语气。
2. **系列内连贯：写非首篇时，必读同系列上一篇**——保证同系列语气统一。
3. **跨系列的旧文：当「避雷镜」，不当「模板」**——读它们是为了找出反复出现的套路和口头禅，列入本系列要主动避开的清单；绝不照抄结构和措辞。

**提取并复用这些特征**：句子长短和节奏、平实/口语/犀利的语气、修辞习惯（括号补充、反问、短句、打比方）、对幽默和个人观点的接受度、排版习惯。

**主动对抗雷同**：每个新系列换一种开场和结构节奏；让「内容」制造独特性（真实场景、你自己的判断、具体数据每篇天然不同）；读完旧文若发现它们长得一个样，那是要打破的信号，不是要继承的模板。

如果没有任何手写素材可参考，默认用「实干者」语气：具体、务实、不浮夸，先摆事实再下判断。

## 文章类型指南

根据主题选择对应策略：

**类型 A · 技术类**（编程、框架、数据库、DevOps、AI/LLM）

- 读者是有开发经验、想快速上手的人；入门部分简单引入，深入部分满足老手期望
- 代码占比 ≤ 40%，每段代码前有引导语（为什么看这段）、后有解读（关键点）
- 注明版本要求（如 "Go 1.21+"）

**类型 B · 健康/身体类**（身体信号、运动、营养、睡眠、精力）

- 用第一人称身体感受开头引发共鸣；科学原理用通俗语言解释
- 给可执行的日常建议；区分"正常现象"和"需就医信号"，不制造健康焦虑

**类型 C · 心理学/认知类**（情绪、认知偏差、习惯、沟通）

- 用一个具体日常困境开头；概念用生活场景解释，不照搬教科书
- 每篇给一个可操作的方法，不泛泛说"要积极"

**类型 D · 投资/理财类**（投资理念、指数基金、资产配置）

- 用真实数据说话，不贩卖焦虑也不画饼；先讲为什么（通胀、复利、风险）再讲怎么做
- 讲思路和方法论，避免推荐具体产品

**类型 E · 生活/运营类**（极简、自媒体运营、职业发展）

- 从具体痛点或观察切入；分享真实经验，不写鸡汤

## 文章结构

不要机械套用固定模板，但每篇必须包含三个要素：

**1. 开头——用真实问题抓住读者（必须）**
从一个具体的、读者可能遇到的困惑开始。例：

> "体检报告一切正常，但就是觉得不对劲？你可能正处在亚健康和慢性炎症的灰色地带。"

**2. 中间——渐进式展开（核心）**
按"为什么 → 是什么 → 怎么用 → 进阶 → 踩坑"的节奏，但按知识点复杂度灵活取舍：

- 简单知识点：为什么 → 怎么用 → 注意事项（3 步）
- 核心知识点：+ 概念解释 + 代码/案例展开（5 步）
- 复杂知识点：+ 设计思路 + 进阶优化 + 对比替代方案（7 步）

**3. 结尾——trade-off 总结（必须）**
不要用复选框重复章节标题。要回答：适用 / 不适用场景、和其他方案的对比、一个来自真实经验的实践建议。

> ✅ "什么时候用 switch 而不是 if-else？有 3 个以上离散值要匹配时 switch 更清晰；但涉及范围判断（score > 60）就老实用 if。"
> ❌ "本文介绍了 ✓ if 语句 ✓ switch 语句 ✓ for 循环"

## 案例设计

每篇需要一个**贯穿全文的主线场景**：来自真实场景、足够简单让人代入、能随文章自然演进。

| 类型           | 好的案例                   | 差的案例                |
| -------------- | -------------------------- | ----------------------- |
| 技术（Web）    | 博客评论系统、待办 API     | Hello World、打印字符串 |
| 技术（数据库） | 电商订单查询、用户权限     | 学生成绩表、员工信息表  |
| 技术（缓存）   | 热门商品库存、秒杀场景     | 设置 key-value、计数器  |
| 技术（ORM）    | 文章与标签关联、订单流水   | 用户 CRUD、商品 CRUD    |
| 健康/身体      | 加班后的疲惫、体检报告困惑 | "要保持健康"            |
| 心理学         | 不敢拒绝同事、拖延到深夜   | "要积极面对生活"        |
| 投资           | 10 年前的 50 万现在值多少  | "投资很重要"            |
| 习惯养成       | 每天早起失败的 20 次尝试   | "养成好习惯"            |

数据要真实：技术类用 `{ userId: "usr_8x9k2m", plan: "pro" }` 这类拟真数据；投资类用真实市场数据；健康类用可感知指标（睡眠时长、精力评分）。

主线场景随系列演进，例如技术类：第 1 篇实现简单注册接口 → 第 2 篇加输入校验 → 第 3 篇存入数据库。

## 系列连贯性

系列不是多篇独立文章的集合，而是一条渐进的路径。每篇必须做到：

1. **开头自然回顾前篇**（2-3 句，融入语境而非生硬罗列）
   - ✅ "上一篇我们了解了皮质醇怎么在体内循环，今天来看怎么通过饮食和运动调节它。"
   - ❌ "上一篇我们介绍了皮质醇，本文将介绍调节方法。"
2. **延续前篇的主线场景**，只展示新增/演进的部分。
3. **结尾自然预告下一篇**要解决的问题，而非干巴巴一句"下一篇介绍 X"。

**跨篇引用知识点**用博客 URL 格式，并给一句简短上下文（别假设读者读过前文）：

> 之前在 [第 3 篇：错误处理](/posts/{category}/{slug}) 讲过 Go 的 error 是值类型，这里用同样的思路…

**整体节奏**在计划书阶段就规划好：入门篇（2-3 篇，用日常场景建立关联）→ 核心篇（3-5 篇，深入原理、主线展开）→ 进阶篇（3-5 篇，场景演进到真实复杂度）→ 实践篇（1-2 篇，整合知识给行动指南）。

## 禁用词与反模式

写作和自检时，看到下面这些直接删掉重写。

**禁用词与套路开头**（AI 味最重）：

- 空话开场：「在当今快速发展的时代」「随着…的不断发展」「众所周知」「不可否认」
- 凑数过渡：「此外」「值得一提的是」「综上所述」「总而言之」「不难看出」
- 炒作词汇：「颠覆」「革命性」「游戏规则改变者」「黑科技」「天花板」「yyds」
- 正确的废话：「具体情况具体分析」「凡事都有两面性」「适合自己的才是最好的」
- 空洞收尾：「希望这篇文章对你有帮助」「让我们一起加油」「未来可期」

替换原则：能用一个具体例子说清的，就别用一句正确的废话。

**内容反模式**：文档搬运（只翻译复述，无自己理解）｜空洞总结（复选框重复章节标题）｜假 FAQ（编造而非真实困惑）｜只说怎么做不说为什么和何时用｜贩卖焦虑（极端案例制造恐慌却不给思路）｜鸡汤灌水（励志语录填充篇幅）。

**风格反模式**：机械模板（每篇"前言→概念→实战→总结"）｜AI 腔调（堆 emoji、滥用分隔线、千篇一律）｜假亲切（emoji 和感叹号装友好但内容没温度）｜面面俱到（覆盖所有点导致每个都不深）。

**技术类代码反模式**（仅类型 A）：贴 50+ 行完整代码只说一句"这是基本用法"｜代码前后无引导和解读｜用 `{ id: 1, name: "Alice" }` 这类假数据｜九九乘法表、Hello World 等玩具示例。

**MDX 链接禁忌**（违反会导致 404）：

- 禁止手动添加"上一篇/下一篇"导航链接——系统已有自动导航组件。
- 禁止用 `.mdx` 相对路径作链接（如 `./02-xxx.mdx`），一律用 URL 格式 `/posts/{category}/{slug}`。
- 禁止在文末添加指向 `.mdx` 文件的链接。

**MDX 构建陷阱**（违反会导致 build 失败）：

- **frontmatter 的 summary 字段里不要嵌套引号**。YAML 解析器无法正确处理 `"他说"好"就走了"` 这种值。中文文章的 summary 经常自然地想用引号包裹术语（如 `"价值"因子`），但这会导致 `YAMLException: can not read a block mapping entry`。解决：去掉内部引号，或把整个值改用单引号包裹。写完 summary 后检查一遍是否包含转义引号 `\"`。
- **MDX 正文中 `<` 后跟非字母会被解析为 JSX 标签**。例如 `<10亿`、`< 5%` 会导致 `Unexpected character before name` 编译错误。解决：用中文替代（"小于10亿"、"低于5%"），或在 `<` 前后加空格使其不被识别为标签。检查方法：搜索正则 `<[^a-zA-Z/!]`。

## 质量自检

每篇写完逐项检查，以下必须全过：

- 从具体问题/场景开始，而非"本文将介绍…"
- 每个核心知识点解释了"为什么需要关注它"，且先给具体再解释
- 有一个贯穿全文的真实主线场景
- 清除了所有【禁用词与反模式】
- 语气和你的旧文/上一篇一致，不是通用 AI 腔
- 结尾有 trade-off 分析（适用/不适用、对比替代方案）
- 回顾了前篇并在其基础上继续（系列文章）
- 读起来像和朋友聊天，不像写文档

技术类额外检查：代码占比 ≤ 40%｜每段代码前有引导语、后有解读｜案例数据像真实生产环境。

建议尽量满足：用类比解释抽象概念｜包含踩坑经验｜对比不同方案优劣｜起承转合自然｜进阶部分够深入｜给一个具体行动项。

## 工作流程

写博客时按此执行。**一旦开始撰写，自动连续完成所有文章，无需逐篇确认。**

**1. 确认主题信息** —— 主题名称、文章类型（A/B/C/D/E）、核心话题（可选）。

**1.5 检查内容重叠（必须）** —— 新系列确定方向后，先读取已有相关系列的 plan.md 和文章内容，确认不会重复。如果发现已有系列覆盖了计划中的核心话题，调整方向或聚焦在互补内容上，而不是重写一遍。在 plan.md 的「与已有系列的关系」表格中明确列出重叠内容和处理方式。

**2. 生成计划书** —— 写入 `content/posts/{category}/plan.md`，结构如下：

```markdown
# {主题} 系列计划

## 背景

- 为什么写这个主题（自己想搞清楚什么）
- 核心话题列表
- 文章类型（A/B/C/D/E）

## 博客目录结构

列出所有文章的文件名和标题

## 阶段划分

入门 → 核心 → 进阶 → 实践

## 每篇文章大纲

每篇提供：学习目标、核心问题、主线场景、内容要点、与前篇的关联

## 系列节奏规划

按上文【系列连贯性】的节奏规划各阶段
```

**3. 通知用户确认（必须等待）** —— 计划书写完后，**主动通知用户查看并等待明确确认**，不要自作主张开始写正文。即使用户之前说了"开始写"，也要先把计划书发给他们过目。计划书可能需要调整——用户看到大纲后往往会补充角度、调整顺序、修正数据。收到确认后才进入步骤 4。

**4. 连续撰写** ——

- 首篇开写前，先按【语调捕捉】从你的手写素材锁定语气，旧文只当避雷镜
- 按计划书顺序逐篇写；非首篇先读上一篇（保证连贯性和语气一致）
- 根据文章类型套用对应【文章类型指南】，按【文章结构】【案例设计】撰写
- 每篇写完按【质量自检】逐项检查，保存文件，简单提示进度
- **并行写文章（用 write_file 成对写，默认策略）**：同一轮里对两篇非相邻文章同时调用 write_file，效率接近 delegate_task 但不受模型限流影响。相邻文章保持串行（后一篇需要引用前一篇的具体措辞）。这是首选策略——实践中 delegate_task 因模型限流（429）频繁失败，自己成对写反而更快更可靠。
- **delegate_task 仅作为大量文章时的补充**：仅在文章总数超过 10 篇且自己写太慢时尝试，且降低并行度（每次最多 2 个子任务）。delegate_task 可能因模型限流（429）失败（尤其是 glm 系列模型），失败后立即切回自己写。delegate_task context 必须包含：语调规则、禁用词清单、MDX 格式陷阱、完整大纲、所需数据、frontmatter 模板。delegate_task 写完的文件要验证是否实际写入（429 可能导致文件未生成或为空）。
- **数据预取**：如果系列需要真实市场数据，先在主会话用 execute_code 批量拉取所有数据（Sina Finance API 等），再作为 context 传入 delegate_task 或直接用于自己写文章。避免重复请求数据。多 ETF 对比指标（收益/波动率/回撤/夏普）的计算流程见 [references/investment-metrics-computation.md](references/investment-metrics-computation.md)。
- **跨系列并行写作**：同时写两个系列时，把两个系列的同序号文章（如 A-01 + B-01）放在同一轮 write_file 里并行——它们互不相邻所以不依赖彼此的具体措辞，效率翻倍。实测 14 篇（2 系列 × 7 篇）可以在 7 轮内完成，全程不需要 delegate_task。
- **进度通知（必须，用户多次强调，屡次违反）**：连续撰写阶段每 2-3 篇（或约 30%）**必须**发一条独立消息通知进度（如"4/8 篇写完，继续"）。**全部写完后、构建通过后、推送完成后，每个节点都要发独立通知。** 遇到阻塞（API 限流、构建报错、数据拉不到）不能默默 retry，必须告知用户卡在哪了。**通知是和写文章同等优先级的步骤，不是"想起来才做"的可选项。** 写完最后一批文章后，不要继续执行构建——先发通知告诉用户"正文全部写完，开始构建"，然后才继续。违反此规则等于任务没完成。

**4.5 MDX 格式预检（批量写完后必须做）** —— 用 [scripts/check-mdx.py](scripts/check-mdx.py) 对新系列所有 .mdx 文件做批量检查，覆盖：summary 嵌套引号、JSX 标签陷阱（`<[^a-zA-Z/!]`）、禁用词扫描。在跑 `npm run build` 之前先扫一遍，避免 build 失败后逐文件排查。

**5. 构建验证** —— 确认 .env.local 存在且含 BLOG_ACCESS_PASSWORD（缺失会导致 build 报环境变量错误），然后运行构建命令，按错误信息修复直到通过。

**6. 提交推送** —— commit message 用 `feat: add {系列名称} series with N articles`，`git push`，然后**立即发一条独立的完成通知**（不要把通知埋在对话回复里等用户发现）。通知包含：系列名称、篇数、构建结果。用户必须在第一时间知道任务完成了——不能等他们来问"任务状态"，问了才回不叫主动通知。

## 投资类文章的数据获取

Type D（投资/理财）文章要求"用真实市场数据"。写投资类博客时，**不要用搜索引擎拿行情数据**（Google/百度/Bing 在浏览器自动化中几乎必然触发验证码）。

数据源：

- **中国 ETF/基金：** Sina K线/报价 API、Eastmoney 基金详情/搜索 API，详见 [references/china-etf-fund-data-apis.md](references/china-etf-fund-data-apis.md)。
- **国际指数（日经225/标普500/上证等）：** Yahoo Finance chart API（`query1.finance.yahoo.com/v8/finance/chart/{symbol}`），详见 [references/international-index-data-apis.md](references/international-index-data-apis.md)。Sina API 只能拉 A 股 ETF，国际指数用 Yahoo。

## 注意事项

- 写作过程中可随时暂停，用户可要求修改。
- 每篇写完即保存，避免丢失。
- 保持系列内一致性：术语、风格、难度、主线场景。
- 写投资/理财类（类型 D）文章需要拉取市场数据时，参考 `references/a-share-financial-data.md`——里面记录了东方财富 datacenter API 的用法、字段映射和多因子评分模型。

## 系列归档

当用户要求归档某个系列时：

1. `mv content/posts/{series}/ content/posts/_archived/{series}/`（`_` 前缀目录被 Next.js content loader 自动忽略）
2. **必须修复死链**：用 `search_files` 在 `content/posts/` 下搜索归档系列的 category 名称，找到所有引用它的文章，将指向已归档文章的链接改为不依赖死链的表述（保留核心观点，去掉链接）。
3. 归档 plan.md 一起移走。
4. 构建验证确认页面数减少、无 404。

## 构建前置检查

构建前确认 `.env.local` 存在且包含 `BLOG_ACCESS_PASSWORD`。缺失会报 `缺少环境变量 BLOG_ACCESS_PASSWORD` 错误。该文件已被 .gitignore 忽略，不会提交到 git。如果用户提供了密码值，用 `printf 'BLOG_ACCESS_PASSWORD=值\n' > .env.local` 写入（注意：密码值会被工具输出过滤，用 `wc -c` 验证字节数确认写入正确）。
