## 为什么需要 Redis

你在一个 Node.js 项目中第一次接触到 Redis。代码里到处是 `redis.get()` 和 `redis.set()`，但你不知道为什么有些查询走了 Redis，有些没走。同事说 Redis 是缓存，但你也看到用它存 Session、做消息队列、实现排行榜。Redis 到底是什么？

让我用最简单的话告诉你：Redis 是一个把数据存在内存里的数据库。你的硬盘读写速度大概是 100MB/s，而内存读写速度可以达到几 GB/s。这就是 Redis 比传统数据库快的根本原因。

但这只是表象。真正让 Redis 成为利器的是它丰富的数据结构。想象一下，你在做一个实时互动直播平台，需要实时统计在线人数、弹幕消息、礼物排行、点赞计数、用户关注关系。如果用 MySQL，你需要设计复杂的表结构，写很多 SQL，还要考虑并发问题。而用 Redis，这些功能可能只需要一行命令。

这就是我们学习 Redis 的原因：它既快，又好用。

## Redis 是什么

Redis（Remote Dictionary Server）是一个开源的内存数据结构存储系统。它可以用作数据库、缓存和消息中间件。这几个角色并不冲突，取决于你如何使用它。

作为数据库，Redis 可以持久化数据到磁盘，重启后数据不会丢失。作为缓存，它可以把热点数据放在内存，极大提升查询速度。作为消息中间件，它的发布订阅功能可以实现实时消息推送。

但所有这些能力都建立在一个核心特性上：数据存储在内存中。

## 为什么 Redis 这么快

你可能会问：MySQL 也可以把数据加载到内存，为什么 Redis 还是要快很多？

首先是数据结构。Redis 的数据结构是专门为内存操作设计的，没有 MySQL 那样的 B+ 树遍历开销。其次是单线程模型。Redis 的网络请求和命令处理是单线程的，避免了多线程的锁竞争和上下文切换。虽然现代版本支持多线程 I/O，但核心命令处理还是单线程。

最后是减少网络 I/O。Redis 支持批量操作（pipeline），一次网络往返可以执行多个命令。

这些设计让 Redis 的 QPS（每秒查询次数）可以达到 10 万以上，而 MySQL 一般只有几千到 1 万。

## Redis 的应用场景

让我用一个真实的案例来说明。假设你在做一个类似 Bilibili 的直播平台，有这些需求：

实时统计在线人数：每当用户进入或离开直播间，在线人数就要更新。如果用 MySQL，每次都要更新数据库，并发大的时候会成为瓶颈。用 Redis 的计数器，更新是原子操作，QPS 轻松达到 10 万。

礼物排行榜：用户送礼时要实时更新排行榜，显示前 10 名。用 Redis 的 Sorted Set，更新分数和查询 Top 10 都是 O(log N) 时间复杂度。

弹幕消息：主播发一条弹幕，所有在线观众要实时看到。用 Redis 的 Pub/Sub，发布一次，所有订阅者都能收到。

用户关注关系：需要实现"共同关注"、"推荐好友"功能。用 Redis 的 Set，交集、并集操作一行命令搞定。

Session 存储：用户登录信息要存在服务器，每次请求都要验证。用 Redis 存储，设置过期时间，自动清理。

这就是 Redis 的威力：不同的场景用不同的数据结构，每个数据结构都是为此类场景优化的。

## 环境搭建

好，说了这么多，让我们先把 Redis 跑起来。

### Docker 方式（推荐）

Docker 是最简单的方式：

```bash
# 拉取 Redis 7 镜像
docker pull redis:7-alpine

# 启动 Redis 容器
docker run -d \
  --name redis \
  -p 6379:6379 \
  redis:7-alpine

# 测试连接
docker exec -it redis redis-cli ping
# 应该返回 PONG
```

这个命令做了什么：`-d` 表示后台运行，`--name redis` 给容器命名，`-p 6379:6379` 把容器的 6379 端口映射到主机的 6379 端口。

### Homebrew 方式（macOS）

如果你是 Mac 用户，用 Homebrew 更方便：

```bash
# 安装 Redis
brew install redis

# 启动 Redis
brew services start redis

# 测试连接
redis-cli ping
```

### Linux 方式

Ubuntu 或 Debian 系统：

```bash
sudo apt update
sudo apt install redis-server
sudo systemctl start redis
```

## 第一个 Redis 命令

连接上 Redis 后，让我们尝试第一个命令：

```bash
# 连接到 Redis
redis-cli

# 设置一个键值对
SET livestream:room:abc123 "wang_dev的直播间"

# 获取值
GET livestream:room:abc123
# 返回 "wang_dev的直播间"

# 设置过期时间（单位是秒）
EXPIRE livestream:room:abc123 3600

# 查看剩余过期时间
TTL livestream:room:abc123
# 返回剩余秒数，比如 3598

# 删除键
DEL livestream:room:abc123
```

这些命令很简单，但已经包含了 Redis 最核心的概念：键值对。键是你给数据的名字，值是数据本身。

## 在 Node.js 中使用 Redis

在实际项目中，你不会用 redis-cli 手动执行命令，而是用客户端库。Node.js 生态最流行的是 ioredis。

先安装：

```bash
npm install ioredis
```

然后创建一个客户端：

```typescript
// redis-client.ts
import Redis from "ioredis";

const redis = new Redis({
  host: "localhost",
  port: 6379,
});

async function demo() {
  // 设置直播间信息
  await redis.set("livestream:room:abc123", "wang_dev的直播间");

  // 获取直播间信息
  const room = await redis.get("livestream:room:abc123");
  console.log(room); // "wang_dev的直播间"

  // 设置过期时间
  await redis.expire("livestream:room:abc123", 3600);

  // 查看剩余时间
  const ttl = await redis.ttl("livestream:room:abc123");
  console.log(`剩余时间: ${ttl} 秒`);

  await redis.quit();
}

demo().catch(console.error);
```

这个例子展示了基本的 CRUD 操作。但 Redis 的真正威力在于它的数据结构操作。

## 直播平台案例：计数器

让我们用直播平台的真实场景来练习。假设要统计直播间点赞数：

```typescript
async function likeRoom(roomId: string, userId: string) {
  const key = `livestream:room:${roomId}:likes`;

  // 原子递增，不需要先读后写
  const count = await redis.incr(key);

  // 第一次设置时添加过期时间
  if (count === 1) {
    await redis.expire(key, 86400); // 24小时过期
  }

  console.log(`用户 ${userId} 点赞了直播间 ${roomId}，当前 ${count} 个赞`);
  return count;
}

// 使用
await likeRoom("abc123", "user_888");
await likeRoom("abc123", "user_999");
```

注意这里的 `incr` 命令。它是一个原子操作，不需要先读取当前值，再加 1，再写回。这在并发场景下非常重要，可以避免竞态条件。

## 直播平台案例：用户 Session

另一个常见场景是存储用户 Session：

```typescript
async function createSession(userId: string, username: string) {
  const sessionId = `usr_${generateRandomId()}`;
  const key = `session:${sessionId}`;

  // 存储 Session 数据
  await redis.hset(key, {
    userId,
    username,
    role: "user",
    createdAt: Date.now().toString(),
  });

  // 24小时后过期
  await redis.expire(key, 86400);

  return sessionId;
}

async function getSession(sessionId: string) {
  const key = `session:${sessionId}`;
  const session = await redis.hgetall(key);

  if (!session || Object.keys(session).length === 0) {
    return null;
  }

  // 每次访问时刷新过期时间
  await redis.expire(key, 86400);

  return session;
}
```

这里我们用了 Hash 数据结构，而不是 String(JSON)。为什么呢？因为 Hash 可以让你只获取需要的字段，而且当字段少的时候，Redis 会用更节省内存的方式存储。

## Trade-off：什么时候用 Redis

现在你已经知道 Redis 很快，但这不代表所有数据都要放 Redis。你需要权衡几个因素：

首先是数据重要性。Redis 即使开启持久化，也有丢失数据的风险（取决于持久化策略）。如果数据是核心业务数据，比如订单、支付记录，还是要存 MySQL，Redis 只做缓存。

其次是内存成本。内存比硬盘贵多了。你不能把所有数据都放内存，只能放热点数据。

最后是数据结构复杂度。Redis 不是关系型数据库，不支持 JOIN，不支持复杂查询。如果需要多表关联、复杂聚合，MySQL 更合适。

那什么时候用 Redis 呢？记住这几个场景：读写频繁的热点数据、不需要持久化的临时数据、需要实时统计的计数器、需要排序的排行榜、需要实时推送的消息。

这就是 Redis 的定位：它是 MySQL 的好搭档，不是替代品。

今天我们学了 Redis 的基本概念和操作，但只是冰山一角。Redis 有 5 种核心数据结构，我们今天只用了 String 和 Hash。下一篇我会带你深入了解这些数据结构，以及如何用它们解决实际问题。

记住，学习 Redis 不要死记命令，要理解每个数据结构的设计意图。就像工具箱里的工具，每个工具都有它最合适的场景。
