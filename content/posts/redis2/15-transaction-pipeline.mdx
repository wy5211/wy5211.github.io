上篇文章我们用 Pub/Sub 实现了直播间的实时消息推送。现在有个新需求：用户送礼物时，要做几个操作：

1. 更新礼物排行榜（Sorted Set）
2. 增加主播收益（String INCR）
3. 记录礼物日志（List）

这三个操作要么全部成功，要么全部失败，不能出现只更新了排行榜但没增加收益的情况。这就需要用到 Redis 的事务。

但 Redis 的事务和关系数据库的事务不太一样，而且还有 Pipeline 和 Lua 脚本两种优化方案。今天我们就来聊聊这三者的区别和使用场景。

## Redis 事务

Redis 的事务通过 MULTI/EXEC 实现：

```typescript
import Redis from "ioredis";

const redis = new Redis();

async function sendGift(
  roomId: string,
  userId: string,
  giftName: string,
  amount: number
) {
  // 开启事务
  const multi = redis.multi();

  // 1. 更新礼物排行榜
  multi.zincrby(`livestream:room:${roomId}:gifts`, amount, userId);

  // 2. 增加主播收益
  multi.incrbyfloat(`livestream:room:${roomId}:revenue`, amount);

  // 3. 记录礼物日志
  multi.lpush(
    `livestream:room:${roomId}:gifts:log`,
    JSON.stringify({
      userId,
      giftName,
      amount,
      timestamp: Date.now(),
    })
  );

  // 执行事务
  const results = await multi.exec();
  console.log("事务执行结果：", results);
}
```

MULTI 开启事务后，后续命令不会立即执行，而是放入队列。EXEC 时才会按顺序执行所有命令。

### 事务的 ACID

Redis 事务的 ACID 特性和关系数据库不太一样：

**原子性（Atomicity）**：

- Redis 事务保证要么全部执行，要么全部不执行
- 但如果某个命令执行失败，其他命令仍会执行（这是和数据库事务的区别）

```typescript
const multi = redis.multi();
multi.set("key1", "value1");
multi.incr("key2"); // key2 不是数字，会失败
multi.set("key3", "value3");

const results = await multi.exec();
// 结果：key1 和 key3 会设置成功，key2 的操作失败
```

**一致性（Consistency）**：

- 事务执行前后数据一致
- 但如果某个命令失败，可能处于不一致状态

**隔离性（Isolation）**：

- 事务执行期间不会被其他命令打断
- 但 MULTI 之后 EXEC 之前，其他客户端仍能执行命令

**持久性（Durability）**：

- 取决于持久化配置（RDB/AOF）

### WATCH 命令

WATCH 可以实现乐观锁，监听某个 key，如果在事务执行期间这个 key 被修改，事务会失败：

```typescript
async function transfer(from: string, to: string, amount: number) {
  // 监视账户余额
  await redis.watch(`account:${from}`);

  const balance = parseFloat((await redis.get(`account:${from}`)) || "0");

  if (balance < amount) {
    await redis.unwatch();
    throw new Error("余额不足");
  }

  // 开启事务
  const multi = redis.multi();
  multi.decrby(`account:${from}`, amount);
  multi.incrby(`account:${to}`, amount);

  const results = await multi.exec();

  if (!results) {
    // 事务被取消（监听的 key 被修改），重试
    console.log("事务被取消，重试...");
    return await transfer(from, to, amount);
  }

  console.log("转账成功");
  return results;
}
```

WATCH 的机制是：EXEC 时检查监听的 key 是否被修改，如果被修改了，事务会失败返回 null。应用层需要捕获这个失败并重试。

## Pipeline

Pipeline 是批量执行命令的优化方案，可以减少网络往返次数（RTT）：

```typescript
// 不使用 Pipeline：1000 次网络往返
async function batchSetWithoutPipeline() {
  const startTime = Date.now();

  for (let i = 0; i < 1000; i++) {
    await redis.set(`key:${i}`, `value:${i}`);
  }

  console.log(`耗时：${Date.now() - startTime}ms`);
  // 约 1000ms（取决于网络延迟）
}

// 使用 Pipeline：1 次网络往返
async function batchSetWithPipeline() {
  const startTime = Date.now();

  const pipeline = redis.pipeline();
  for (let i = 0; i < 1000; i++) {
    pipeline.set(`key:${i}`, `value:${i}`);
  }
  await pipeline.exec();

  console.log(`耗时：${Date.now() - startTime}ms`);
  // 约 20-50ms
}
```

Pipeline 的原理是把多个命令打包一起发送，Redis 一起处理并返回结果，大大减少网络往返。

### Pipeline vs 事务

Pipeline 和事务看起来很像，但本质不同：

```typescript
// Pipeline：减少网络往返，但不保证原子性
const pipeline = redis.pipeline();
pipeline.set("key1", "value1");
pipeline.set("key2", "value2");
await pipeline.exec();

// 事务：保证原子性，但也会减少网络往返
const multi = redis.multi();
multi.set("key1", "value1");
multi.set("key2", "value2");
await multi.exec();
```

区别：

- Pipeline：命令会立即执行，只是批量发送网络请求
- 事务：命令会排队，EXEC 时才一起执行

性能对比：

- Pipeline：更快，因为没有事务的开销
- 事务：稍慢，但有原子性保证

选择建议：

- 需要原子性：用事务
- 只需要批量操作：用 Pipeline

## Lua 脚本

Lua 脚本是 Redis 的终极武器：保证原子性，支持复杂逻辑，还能复用：

```typescript
// 原子性的"检查并设置"操作
async function checkAndSet(
  key: string,
  expectedValue: string,
  newValue: string
) {
  const script = `
    local current = redis.call("GET", KEYS[1])
    if current == ARGV[1] then
      redis.call("SET", KEYS[1], ARGV[2])
      return 1
    else
      return 0
    end
  `;

  const result = await redis.eval(script, 1, key, expectedValue, newValue);
  return result === 1;
}
```

Lua 脚本的执行是原子的，Redis 保证在脚本执行期间不会执行其他命令。

### 复杂业务逻辑

Lua 脚本可以执行复杂的业务逻辑：

```typescript
// 限流：每个用户每分钟最多 100 次请求
async function rateLimit(userId: string, limit: number, window: number) {
  const key = `ratelimit:${userId}`;
  const script = `
    local key = KEYS[1]
    local limit = tonumber(ARGV[1])
    local window = tonumber(ARGV[2])

    local current = redis.call("GET", key)
    if current == false then
      redis.call("SET", key, 1, "EX", window)
      return 1
    end

    current = tonumber(current)
    if current < limit then
      redis.call("INCR", key)
      return 1
    else
      return 0
    end
  `;

  const allowed = await redis.eval(script, 1, key, limit, window);
  return allowed === 1;
}

// 使用
const canRequest = await rateLimit("usr_a1b2c3", 100, 60);
if (!canRequest) {
  throw new Error("请求过于频繁");
}
```

### 脚本缓存

Lua 脚本执行后会缓存，后续可以用 SHA1 调用，避免重复传输脚本：

```typescript
const script = `
  local value = redis.call("GET", KEYS[1])
  redis.call("DEL", KEYS[1])
  return value
`;

// 第一次执行，加载脚本
const sha1 = await redis.script("load", script);
console.log("脚本 SHA1：", sha1);

// 后续用 SHA1 执行
const value = await redis.evalsha(sha1, 1, "mykey");
```

## 实际应用：礼物系统

让我们用这些知识实现一个完整的礼物系统：

```typescript
class GiftService {
  constructor(private redis: Redis) {}

  // 使用事务确保原子性
  async sendGiftWithTransaction(
    roomId: string,
    userId: string,
    giftName: string,
    amount: number
  ) {
    const multi = this.redis.multi();

    // 1. 更新礼物排行榜
    multi.zincrby(`livestream:room:${roomId}:gifts`, amount, userId);

    // 2. 增加主播收益
    multi.incrbyfloat(`livestream:room:${roomId}:revenue`, amount);

    // 3. 记录礼物日志
    multi.lpush(
      `livestream:room:${roomId}:gifts:log`,
      JSON.stringify({
        userId,
        giftName,
        amount,
        timestamp: Date.now(),
      })
    );

    // 4. 限制日志列表长度（只保留最近 100 条）
    multi.ltrim(`livestream:room:${roomId}:gifts:log`, 0, 99);

    const results = await multi.exec();

    if (results) {
      console.log("礼物发送成功");
      // 发布消息通知
      await this.publishGiftNotification(roomId, userId, giftName, amount);
    }

    return results;
  }

  // 使用 Lua 脚本实现更复杂的逻辑
  async sendGiftWithLua(
    roomId: string,
    userId: string,
    giftName: string,
    amount: number
  ) {
    const script = `
      local room = KEYS[1]
      local user = ARGV[1]
      local gift = ARGV[2]
      local amount = tonumber(ARGV[3])

      -- 1. 更新礼物排行榜
      redis.call("ZINCRBY", room .. ":gifts", amount, user)

      -- 2. 增加主播收益
      redis.call("INCRBYFLOAT", room .. ":revenue", amount)

      -- 3. 记录礼物日志
      local log = room .. ":gifts:log"
      redis.call("LPUSH", log, cjson.encode({
        userId = user,
        giftName = gift,
        amount = amount,
        timestamp = ARGV[4]
      }))
      redis.call("LTRIM", log, 0, 99)

      -- 4. 检查是否触发特效（比如连续送礼）
      local lastGift = redis.call("GET", room .. ":last_gift:" .. user)
      if lastGift then
        local elapsed = tonumber(ARGV[4]) - tonumber(lastGift)
        if elapsed < 5000 then -- 5 秒内连续送礼
          return {1, "combo"} -- 返回连击标记
        end
      end
      redis.call("SET", room .. ":last_gift:" .. user, ARGV[4], "EX", 10)

      return {0, "success"}
    `;

    const result = await this.redis.eval(
      script,
      1,
      `livestream:room:${roomId}`,
      userId,
      giftName,
      amount,
      Date.now().toString()
    );

    const [isCombo, status] = result as [number, string];

    if (isCombo === 1) {
      console.log("触发连击特效");
      // 触发连击动画
    }

    return status;
  }

  // 使用 Pipeline 批量查询多个房间数据
  async getRoomStatsBatch(roomIds: string[]) {
    const pipeline = this.redis.pipeline();

    for (const roomId of roomIds) {
      pipeline.zcard(`livestream:room:${roomId}:gifts`);
      pipeline.get(`livestream:room:${roomId}:revenue`);
      pipeline.llen(`livestream:room:${roomId}:gifts:log`);
    }

    const results = await pipeline.exec();

    const stats = [];
    for (let i = 0; i < roomIds.length; i++) {
      const baseIndex = i * 3;
      stats.push({
        roomId: roomIds[i],
        giftCount: results[baseIndex][1],
        revenue: results[baseIndex + 1][1],
        logCount: results[baseIndex + 2][1],
      });
    }

    return stats;
  }

  private async publishGiftNotification(
    roomId: string,
    userId: string,
    giftName: string,
    amount: number
  ) {
    await this.redis.publish(
      `livestream:room:${roomId}:gifts`,
      JSON.stringify({
        userId,
        giftName,
        amount,
        timestamp: Date.now(),
      })
    );
  }
}
```

## 性能对比

让我们对比一下三种方案的性能：

```typescript
// 测试：1000 次 SET 操作

// 1. 普通执行：约 1000ms
async function normal() {
  for (let i = 0; i < 1000; i++) {
    await redis.set(`key:${i}`, `value:${i}`);
  }
}

// 2. Pipeline：约 20-50ms
async function withPipeline() {
  const pipeline = redis.pipeline();
  for (let i = 0; i < 1000; i++) {
    pipeline.set(`key:${i}`, `value:${i}`);
  }
  await pipeline.exec();
}

// 3. 事务：约 50-100ms
async function withTransaction() {
  const multi = redis.multi();
  for (let i = 0; i < 1000; i++) {
    multi.set(`key:${i}`, `value:${i}`);
  }
  await multi.exec();
}

// 4. Lua 脚本：约 30-80ms
async function withLua() {
  const script = `
    for i = 0, 999 do
      redis.call("SET", "key:" .. i, "value:" .. i)
    end
  `;
  await redis.eval(script, 0);
}
```

性能排序：Pipeline > Lua 脚本 > 事务 > 普通执行

## Trade-off 分析

三种方案各有优劣：

**事务（MULTI/EXEC）**：

- 优势：保证原子性，易于理解
- 劣势：性能不如 Pipeline，回滚机制不完善
- 适用：需要原子性但逻辑简单的场景

**Pipeline**：

- 优势：性能最好，减少网络往返
- 劣势：不保证原子性，不支持命令间依赖
- 适用：批量操作，不需要原子性的场景

**Lua 脚本**：

- 优势：保证原子性，支持复杂逻辑，可复用
- 劣势：调试困难，可能阻塞 Redis
- 适用：复杂业务逻辑，需要原子性的场景

选择建议：

- 简单原子操作：用事务
- 批量操作：用 Pipeline
- 复杂逻辑：用 Lua 脚本

对于直播平台的礼物系统，推荐使用 Lua 脚本：既能保证原子性，又能实现复杂的业务逻辑（比如连击检测）。而且脚本可以缓存，性能也不错。

但要注意：Lua 脚本不要写太复杂，避免阻塞 Redis。如果脚本执行时间超过 5 秒，Redis 会拒绝执行。可以把复杂脚本拆分成多个小脚本，或者用 Pipeline 分批执行。
