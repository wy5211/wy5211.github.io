# Redis 渐进式学习计划

## 背景

你在一个 Node.js 项目中第一次接触到 Redis。代码里到处是 `redis.get()` 和 `redis.set()`，但你不知道为什么有些查询走了 Redis，有些没走。同事说 Redis 是缓存，但你也看到用它存 Session、做消息队列、实现排行榜。Redis 到底是什么？

这个系列从"为什么需要 Redis"出发，带你逐步掌握 Redis 的核心能力和生产实践。

### 技术栈

| 技术       | 版本   | 为什么选它                      |
| ---------- | ------ | ------------------------------- |
| Redis      | 7.x    | 最新稳定版，有更多数据结构      |

## 基础篇：快速上手（01-05）

### 01. Redis 快速入门：为什么需要它

**核心问题**：数据库查询很慢，听说加个 Redis 就能快。但 Redis 是什么？为什么它比数据库快？用它缓存数据，数据不一致怎么办？

**主线案例**：搭建 Redis 环境，用最简单的命令操作，体验"从内存读写"的感觉。

**内容要点**：

- Redis 是什么：内存数据库、缓存、消息中间件
- 为什么比数据库快：内存 vs 磁盘、单线程模型
- 典型应用场景：缓存、Session、排行榜、计数器、消息队列
- 环境搭建：Docker、Homebrew、编译安装
- 第一个命令：SET/GET/EXISTS/DEL
- CLI 连接：redis-cli 的基本使用
- ioredis 客户端：Node.js 中如何连接 Redis

**与前篇关联**：系列第一篇。

---

### 02. CLI 命令与键操作

**核心问题**：你连上了 Redis，但不知道里面存了什么。怎么查看所有的键？怎么按模式搜索？键怎么设置过期时间？

**主线案例**：管理一个直播平台的 Redis 数据，查看在线用户、清理过期 Session、统计键数量。

**内容要点**：

- 键操作：KEYS（危险）、SCAN（安全）、EXISTS、TYPE、RENAME
- 过期时间：TTL、EXPIRE、EXPIREAT、PERSIST
- 键命名规范：冒号分隔、前缀统一
- 服务器命令：DBSIZE、INFO、FLUSHDB/FLUSHALL
- 批量操作：为什么不能用 `KEYS + foreach`
- 键空间通知：监听键的事件

**与前篇关联**：上一篇学会了基本命令，这一篇掌握键管理的全套技能。

---

### 03. 五大数据类型概览

**核心问题**：Redis 不只是存字符串。Hash、List、Set、Sorted Set 各是什么场景？为什么需要这么多类型？

**主线案例**：为直播平台的不同场景选择合适的数据类型。

**内容要点**：

- String：字符串、数字、二进制（最通用）
- Hash：对象存储、购物车（适合字段多的对象）
- List：队列、栈、最新列表（保留顺序）
- Set：去重、交集、并集（无序集合）
- Sorted Set：排行榜、范围查询（有序集合）
- 类型选择决策树：什么场景用什么类型

**与前篇关联**：上一篇操作了键，这一篇深入理解值的数据结构。

---

### 04. 键设计与命名规范

**核心问题**：你的 Redis 里键名乱七八糟：`user1`、`User:2`、`usr_003`。团队协作时根本看不懂。怎么设计一套清晰的键命名体系？

**主线案例**：为直播平台设计统一的键命名规范。

**内容要点**：

- 命名规范：冒号分隔、小写、前缀统一
- 键的层级设计：业务域:实体:标识
- 过期策略：不同类型数据的 TTL 设计
- 内存优化：短键名 vs 可读性
- 键分离：不同业务用不同 DB 或前缀
- 大 Key 问题：什么是大 Key、怎么避免

**与前篇关联**：上一篇了解了数据类型，这一篇学会如何组织键结构。

---

### 05. 持久化机制：RDB 与 AOF

**核心问题**：Redis 是内存数据库，服务器重启数据就没了。怎么把数据持久化到磁盘？RDB 和 AOF 有什么区别？

**主线案例**：为直播平台配置持久化，确保重启后数据不丢失。

**内容要点**：

- 为什么需要持久化：内存的易失性
- RDB：fork、写时复制、全量快照
- AOF：追加日志、重写机制
- 混合持久化：RDB + AOF 结合
- 配置选择：根据场景选择持久化方案
- 恢复数据：从 RDB/AOF 文件恢复

**与前篇关联**：前几篇用默认配置开发，这一篇关心数据安全。

## 数据结构篇：深入实战（06-10）

### 06. String 进阶：计数器与限流

**核心问题**：String 不只是存字符串。怎么用它做计数器？怎么实现 API 限流？

**主线案例**：直播间点赞数、礼物积分统计、API 访问限流。

**内容要点**：

- INCR/DECR：原子递增递减
- INCRBY：批量递增
- 限流实现：固定窗口、滑动窗口
- 分布式 ID 生成：INCR 的妙用
- 数值范围：INCR 的溢出处理

**与前篇关联**：上一篇讲了持久化，这一篇深入 String 的实战应用。

### 07. Hash 实战：对象存储与购物车

**核心问题**：用户对象、商品信息存成 String(JSON) 还是 Hash？Hash 和 JSON 各有什么优劣？

**主线案例**：存储用户信息、实现购物车功能。

**内容要点**：

- Hash vs String(JSON)：内存占用、灵活性
- HSET/HGET：字段操作
- HMGET/MSET：批量操作
- 购物车实现：商品 ID、数量、价格
- HINCRBY：购物车数量增减
- Hash 优缺点：适合字段固定的对象

**与前篇关联**：上一篇用 String 做计数器，这一篇用 Hash 存对象。

### 08. List 队列：消息队列实现

**核心问题**：需要实现一个异步任务队列。Redis 的 List 怎么做队列？LPUSH + RPOP 还是 RPUSH + LPOP？

**主线案例**：直播平台的弹幕消息队列、礼物通知队列。

**内容要点**：

- LPUSH/RPOP、RPUSH/LPOP：队列方向
- 阻塞操作：BLPOP/BRPOP
- 栈实现：同侧进同侧出
- 消息队列：基本实现、局限性
- List vs 专门 MQ：RabbitMQ、Kafka

**与前篇关联**：上一篇用 Hash 存对象，这一篇用 List 做队列。

---

### 09. Set 集合运算：社交功能

**核心问题**：怎么实现"共同关注"、"推荐好友"？Set 的交集、并集怎么用？

**主线案例**：用户关注关系、共同好友、标签推荐。

**内容要点**：

- SADD/SMEMBERS：添加和查询成员
- SINTER：交集（共同关注）
- SUNION：并集（所有关注）
- SDIFF：差集（关注了但没关注）
- 社交关系：关注图、推荐算法
- Set 去重：自动去重的特性

**与前篇关联**：上一篇用 List 做队列，这一篇用 Set 做关系运算。

**与前篇关联**：上一篇用 Set 做关系，这一篇用 Sorted Set 做排序。

## 缓存策略篇（11-13）

### 11. Cache-Aside 缓存模式

**核心问题**：Redis 和数据库怎么配合？先查 Redis 还是先查数据库？更新时先更新哪个？

**主线案例**：为直播平台的用户信息接口加缓存。

**内容要点**：

- Cache-Aside 模式：读流程、写流程
- Cache-Aside vs Read-Through vs Write-Through
- 缓存更新策略：先更新数据库还是先删除缓存
- 延迟双删：确保一致性
- 缓存粒度：缓存整个对象还是字段

**与前篇关联**：上一篇完成了 Sorted Set，这一篇开始讲系统级的缓存模式。

---

### 12. 缓存穿透、击穿、雪崩

**核心问题**：缓存上线后，反而出现更多问题：数据库压力更大、缓存命中率低。这是缓存穿透、击穿、雪崩。

**主线案例**：诊断和解决直播平台的缓存问题。

**内容要点**：

- 缓存穿透：查询不存在的数据，布隆过滤器
- 缓存击穿：热点 Key 过期，互斥锁、热点 Key 不过期
- 缓存雪崩：大量 Key 同时过期，过期时间加随机值
- 监控指标：缓存命中率、响应时间

**与前篇关联**：上一篇讲了缓存模式，这一篇处理缓存的问题。

---

**核心问题**：多台服务器同时跑定时任务，怎么保证只执行一次？分布式锁怎么做？

**主线案例**：直播间礼物结算、定时任务互斥。

**内容要点**：

- 为什么需要分布式锁：多进程并发
- SET NX EX：加锁基本实现
- 锁的续期：看门狗机制
- Redlock 算法：多节点锁
- Lua 脚本：保证原子性
- 锁的超时：防止死锁

**与前篇关联**：上一篇解决了缓存问题，这一篇解决并发问题。
