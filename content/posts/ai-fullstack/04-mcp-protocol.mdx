上一篇你用 Function Calling 给代码审查助手加了几个工具——搜索 NPM 文档、查 GitHub、检查安全漏洞。能跑，但有个问题：这些工具是你在这个应用里手写的。如果你要做一个新的 AI 应用，想复用这些工具怎么办？复制粘贴？那 10 个应用、每个 5 个工具，就是 50 份重复代码。

更麻烦的是，你的同事在另一个项目里也想用你的"博客内容查询"工具。他怎么用？你又不能把代码直接塞给他，因为他的技术栈可能和你不一样。

这就是 MCP（Model Context Protocol）要解决的问题：**定义一个标准协议，让 AI 工具可以跨应用、跨技术栈地复用。** 就像 USB 统一了外设接口一样，MCP 想统一 AI 工具接口。

## MCP 的核心思想

在 [第 3 篇](/posts/ai-fullstack/03-function-calling) 中，Function Calling 的工作方式是：你在代码里定义工具 → 注册到 API 调用中 → 模型调用时你的代码执行。工具和 AI 应用是紧耦合的。

MCP 的思路是把这个过程拆成三层：

```
Host（AI 应用）←→ Client（MCP 客户端）←→ Server（MCP 服务端）
```

- **Host**：你的 Next.js 应用、Claude Desktop、Cursor 等 AI 客户端
- **Client**：Host 内嵌的 MCP 客户端，负责和 Server 通信
- **Server**：独立运行的服务，提供 Tools 和 Resources

关键变化是：**Server 是独立的**。它不依赖你的 AI 应用，任何支持 MCP 的 Host 都可以连接。你写了一个"查询博客内容"的 MCP Server，Claude Desktop 能用，Cursor 能用，你的 Next.js 应用也能用。

## MCP 提供两种能力

MCP Server 可以向 Client 暴露两种能力：

**Tools（工具）**：可以执行的操作。类似 REST API 的 POST 请求——调用它会有副作用（比如创建文件、发送邮件）。

**Resources（资源）**：可以读取的数据。类似 REST API 的 GET 请求——读取数据，不产生副作用。

举个例子，一个"博客查询" MCP Server 可以提供：

- Resource：`blog://articles` — 返回所有文章列表
- Resource：`blog://article/{id}` — 返回指定文章内容
- Tool：`search_articles` — 按关键词搜索文章
- Tool：`get_articles_by_tag` — 按标签筛选文章

Resources 是只读的、结构化的数据源，Tools 是可执行的、有副作用的操作。

## 开发一个博客内容 MCP Server

用 `@modelcontextprotocol/sdk` 开发一个 MCP Server。这个 Server 提供你博客的内容查询能力——让任何 MCP 客户端都能搜索你的 340+ 篇文章。

先初始化项目：

```bash
mkdir blog-mcp-server && cd blog-mcp-server
pnpm init
pnpm add @modelcontextprotocol/sdk zod
```

MCP Server 的基本结构：

```typescript
// index.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "blog-content-server",
  version: "1.0.0",
});

// 注册 Tools 和 Resources（下面展开）
// ...

// 启动 Server，使用 stdio 传输
const transport = new StdioServerTransport();
await server.connect(transport);
```

这里用了 **stdio 传输**——MCP Server 通过标准输入输出和 Client 通信。这是最简单的传输方式，适合本地运行的工具。还有 HTTP+SSE 传输方式，适合远程部署的 Server。

### 注册 Tool：搜索文章

```typescript
server.tool(
  "search_articles",
  "搜索博客文章。当用户询问博客中是否有关于某个主题的文章时使用。",
  {
    query: z.string().describe("搜索关键词"),
    tag: z
      .string()
      .optional()
      .describe("按标签筛选，如 'nestjs'、'typescript'"),
    limit: z.number().optional().describe("返回数量，默认 5"),
  },
  async ({ query, tag, limit = 5 }) => {
    // 实际项目中这里查数据库，这里用模拟数据
    const articles = await db.query.articles.findMany({
      where: and(
        tag ? eq(schema.articles.category, tag) : undefined,
        or(
          ilike(schema.articles.title, `%${query}%`),
          ilike(schema.articles.content, `%${query}%`)
        )
      ),
      limit,
    });

    return {
      content: [
        {
          type: "text" as const,
          text: JSON.stringify(
            articles.map((a) => ({
              title: a.title,
              category: a.category,
              date: a.date,
              summary: a.summary,
            }))
          ),
        },
      ],
    };
  }
);
```

Tool 的注册和上一篇的 Function Calling 很像——都有 name、description、parameters。区别在于 MCP Tool 的返回值格式是固定的 `{ content: [{ type: "text", text: "..." }] }`。

### 注册 Resource：文章列表

```typescript
server.resource(
  "articles",
  "blog://articles",
  "博客文章列表，包含所有分类的文章",
  async (uri) => {
    const categories = await db.query.articles.findMany({
      columns: { category: true },
      distinct: true,
    });

    return {
      contents: categories.map((c) => ({
        uri: `blog://articles?category=${c.category}`,
        mimeType: "application/json",
        text: JSON.stringify({ category: c.category }),
      })),
    };
  }
);

server.resource(
  "article-detail",
  "blog://article/{slug}",
  "博客文章详情",
  async (uri) => {
    const slug = uri.pathname.split("/").pop();
    const article = await db.query.articles.findFirst({
      where: eq(schema.articles.slug, slug),
    });

    if (!article) {
      return { contents: [] };
    }

    return {
      contents: [
        {
          uri: uri.toString(),
          mimeType: "text/markdown",
          text: article.content,
        },
      ],
    };
  }
);
```

Resource 用 URI 标识，支持模板参数（`{slug}`）。Client 可以通过 URI 获取对应的数据。

## 在 Claude Desktop 中使用你的 MCP Server

MCP 最实用的场景之一是接入 Claude Desktop。编辑 Claude Desktop 的配置文件：

```json
// macOS: ~/Library/Application Support/Claude/claude_desktop_config.json
{
  "mcpServers": {
    "blog-content": {
      "command": "node",
      "args": ["/path/to/blog-mcp-server/dist/index.js"]
    }
  }
}
```

重启 Claude Desktop 后，你就可以直接问它"我的博客里有没有关于 NestJS 依赖注入的文章"，它会调用你的 MCP Server 搜索博客内容。

这就是 MCP 的价值——**你写一次 Server，所有 MCP 客户端都能用**。不需要每个 AI 应用都重新实现一遍博客查询逻辑。

## 在 Next.js 中集成 MCP Client

如果你想在自建的 AI 应用中使用 MCP Server，需要集成 MCP Client。Vercel AI SDK 提供了 `experimental_createMCPClient`：

```typescript
import { experimental_createMCPClient } from "ai";

export async function POST(req: Request) {
  const { messages } = await req.json();

  const mcpClient = await experimental_createMCPClient({
    transport: {
      type: "stdio",
      command: "node",
      args: ["/path/to/blog-mcp-server/dist/index.js"],
    },
  });

  const tools = await mcpClient.tools();

  const result = streamText({
    model: anthropic("claude-sonnet-4-20250514"),
    system: SYSTEM_PROMPT,
    messages,
    tools,
    maxSteps: 5,
  });

  return result.toDataStreamResponse();
}
```

`mcpClient.tools()` 会自动发现 MCP Server 提供的所有 Tools，转换成 Vercel AI SDK 的 tool 格式。这样你上一篇写的 Function Calling 代码几乎不用改——只是 tools 的来源从"手写"变成了"从 MCP Server 获取"。

## stdio vs HTTP 传输

前面用的是 stdio 传输——Client 启动 Server 进程，通过标准输入输出通信。这适合本地场景，但如果你想把 MCP Server 部署到远程服务器、让多个 Client 共享，就需要 HTTP+SSE 传输：

```typescript
// Server 端
import { SSEServerTransport } from "@modelcontextprotocol/sdk/server/sse.js";

const app = express();

app.get("/sse", async (req, res) => {
  const transport = new SSEServerTransport("/messages", res);
  await server.connect(transport);
});

app.post("/messages", async (req, res) => {
  // 处理 Client 发来的消息
});
```

选择标准很简单：**本地开发、个人工具用 stdio**（简单、不需要部署），**多人共享、远程服务用 HTTP**（需要部署，但可以多 Client 共享）。

## MCP 的实际局限

MCP 是 2024 年底才发布的协议，目前还在快速迭代中。几个需要注意的点：

**SDK 稳定性**：`@modelcontextprotocol/sdk` 的 API 可能会变化，Vercel AI SDK 中的 MCP 集成还标记为 `experimental`。生产环境使用需要关注版本更新。

**生态还在早期**：目前社区贡献的 MCP Server 还不多，很多场景需要自己写。但这个方向是对的——就像 npm 生态早期一样，先有人写包，生态才会繁荣。

**不是所有工具都适合 MCP**：简单的工具（比如"获取当前时间"、"格式化 JSON"）直接用 Function Calling 写在代码里更简单，不需要包装成独立的 MCP Server。MCP 适合的是那些**有复用价值、有独立状态、可能被多个应用使用的工具**。

## trade-off：什么时候用 MCP，什么时候用 Function Calling

| 场景                       | 推荐方案         | 理由                     |
| -------------------------- | ---------------- | ------------------------ |
| 应用专属的 1-2 个工具      | Function Calling | 简单直接，不需要额外进程 |
| 工具需要被多个 AI 应用复用 | MCP Server       | 写一次，到处用           |
| 工具有独立的状态或数据源   | MCP Server       | 封装数据访问逻辑         |
| 工具逻辑很简单，几行代码   | Function Calling | MCP 的包装成本不值得     |
| 团队共享的工具集           | MCP Server       | 标准化接口，降低协作成本 |

实际项目中，两者可以混合使用——核心的、复用价值高的工具用 MCP Server 包装，应用特有的简单逻辑用 Function Calling 直接写在代码里。
