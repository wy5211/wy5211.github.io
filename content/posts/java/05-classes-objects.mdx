从这篇开始，我们要做一个博客 CMS 系统。

不是那种"概念演示用的小例子"——一个 Student 类、一个 Dog 类、一个 Rectangle 类，学完就扔。从现在起到第 20 篇，所有 Java 知识都会围绕一个叫 TechBlog CMS 的项目展开。每篇文章引入新概念时，都会在这个项目里找到真实的使用场景。

先把丑话说在前头：如果你之前用 JS 的 class 写过代码，你可能会觉得 Java 的 class 语法"差不多"。但它们背后的运行机制完全不同。JS 的 class 是原型链的语法糖，运行时本质上还是个函数；Java 的 class 编译后是一个独立的 `.class` 字节码文件，JVM 拿到它就能精确知道这个类有哪些字段、哪些方法、谁可以访问。这不是"差不多"，是两个世界的东西。

## CMS 数据模型

TechBlog CMS 的核心实体有四个：Post（文章）、Tag（标签）、User（用户）、Comment（评论）。今天我们先定义 Post 和 Tag，User 和 Comment 会在后续文章中逐步引入。

先看 Post 需要哪些数据：

```
Post:    id, title, slug, content(markdown), status(DRAFT/PUBLISHED),
         viewCount, createdAt, publishedAt

Tag:     id, name, slug, color
```

一个文章有标题、slug（URL 友好的标识符）、正文内容（markdown 格式）、发布状态、浏览量、创建时间和发布时间。标签相对简单，有名称、slug 和颜色。

这些东西用 JS 对象怎么表示？一行搞定：

```js
const post = {
  title: "我的第一篇博客",
  slug: "my-first-post",
  content: "# Hello World",
  status: "DRAFT",
};
```

随便加属性、随便改结构、运行时动态增删字段。这在 JS 里完全没问题。但 Java 不允许——你必须先告诉编译器"这个类长什么样"，然后才能创建对象。

## 从 JS class 到 Java class：本质差异

在深入写代码之前，先搞清楚 JS 的 class 和 Java 的 class 到底有什么不同。

JS 的 class 是 ES6 引入的语法糖。你写 `class Post { ... }`，经过 Babel 编译后（或者 V8 内部处理），它变成一个构造函数 + 原型链。你甚至可以在运行时给一个实例动态添加属性，class 定义里没有的字段也不影响。

Java 的 class 是一等公民。编译器会把每个 public class 编译成一个独立的 `.class` 文件（字节码）。JVM 加载这个类的时候，会在内存中构建一个完整的类型描述——字段名、字段类型、方法签名、访问权限——全部精确记录。运行时不能给对象添加类定义之外的属性，编译器会直接拒绝。

```java
// Java: Post.java 编译后生成 Post.class
public class Post {
    private String title;  // 编译器精确知道：title 是 String 类型，private 访问
    private String status; // 运行时不能随便加个新字段

    public Post(String title, String status) {
        this.title = title;
        this.status = status;
    }
}

// 运行时尝试 post.xxx = "abc"？编译直接报错
```

| 特性         | JavaScript class       | Java class                             |
| ------------ | ---------------------- | -------------------------------------- |
| 编译产物     | 函数（原型链）         | `.class` 字节码文件                    |
| 类型信息     | 运行时松散             | 编译时严格检查                         |
| 字段动态添加 | 可以                   | 不行                                   |
| 方法重载     | 不支持（同名方法覆盖） | 支持（参数签名不同）                   |
| 构造器       | constructor 方法       | 可以有多个（参数不同）                 |
| 访问控制     | public / #（私有字段） | public / protected / default / private |

这张表后面还会反复提到。先有个整体印象，下面在写代码的过程中逐一感受。

## 定义 Post 类

现在正式开始。CMS 系统的第一行代码是 Post 类的定义。

Java 要求每个 public 类放在一个同名文件中——`Post` 类对应 `Post.java`，`Tag` 类对应 `Tag.java`。这在 JS 里没有对应的概念，你一个文件里可以 export 几十个 class，没有任何限制。

```java
public class Post {
    // 字段：用 private 封装，外部不能直接访问
    private Long id;
    private String title;
    private String slug;
    private String content;
    private String status;       // "DRAFT" 或 "PUBLISHED"，枚举后面再讲
    private long viewCount;
    private String createdAt;
    private String publishedAt;

    // 无参构造器：创建一个"空白"的 Post 对象
    public Post() {
        this.status = "DRAFT";
        this.viewCount = 0;
        this.createdAt = java.time.LocalDateTime.now().toString();
    }

    // 全参构造器：创建时就提供所有数据
    public Post(Long id, String title, String slug, String content,
                String status, long viewCount, String createdAt, String publishedAt) {
        this.id = id;
        this.title = title;
        this.slug = slug;
        this.content = content;
        this.status = status;
        this.viewCount = viewCount;
        this.createdAt = createdAt;
        this.publishedAt = publishedAt;
    }
}
```

几个关键点要解释：

**字段类型**。`Long id` 而不是 `long id`——这里用了包装类型 Long（对象），而不是基本类型 long。因为数据库里的 id 可能为 null（新创建还没保存到数据库时），基本类型不能为 null。`viewCount` 用 `long` 就够了，浏览量不会是 null，默认值是 0。这个基本类型和包装类型的区别，在第 02 篇讲过，这里第一次在实际场景中遇到。

**构造器重载**。一个类可以有多个构造器，只要参数列表不同。无参构造器用于"先创建空对象，后面再设置属性"的场景（比如从数据库读取后逐个字段赋值）；全参构造器用于"一次性传入所有数据"的场景。JS 只能有一个 constructor，Java 可以有任意多个。

**this 关键字**。`this.title = title` 中的第一个 `title` 是字段（成员变量），第二个 `title` 是构造器的参数。不加 `this.` 的话，编译器会认为你在把参数赋给自己。这在 JS 里也是一样的写法，但 Java 编译器在这方面更严格——如果你写了 `title = title` 而没加 `this`，编译不会报错，但字段永远不会被赋值。

### getter 和 setter

字段是 private 的，外部怎么读写？答案是 getter 和 setter 方法。

```java
public class Post {
    // ... 字段和构造器 ...

    // Getter 方法：提供"只读"访问
    public String getTitle() {
        return title;
    }

    public String getStatus() {
        return status;
    }

    public long getViewCount() {
        return viewCount;
    }

    // Setter 方法：提供"受控"写入
    public void setTitle(String title) {
        if (title == null || title.isBlank()) {
            throw new IllegalArgumentException("标题不能为空");
        }
        this.title = title;
    }

    public void setStatus(String status) {
        if (!"DRAFT".equals(status) && !"PUBLISHED".equals(status)) {
            throw new IllegalArgumentException("状态只能是 DRAFT 或 PUBLISHED");
        }
        this.status = status;
    }

    public void incrementViewCount() {
        this.viewCount++;
    }
}
```

这里有个值得注意的细节：`setStatus` 里做参数校验，非法状态直接抛异常。这就是 getter/setter 比直接暴露字段强大的地方——你可以控制"怎么改"。如果 title 是 public 的，任何人都能把它设成 null 或空字符串，你完全没办法拦截。

在 JS 里，你可以用 getter/setter 语法做类似的事，但大部分时候大家直接暴露字段（或者用 TypeScript 但也不加校验逻辑）。Java 社区的惯例是：字段一律 private，通过 getter/setter 访问。这个惯例不是教条，而是因为 Java 生态里的工具（序列化框架、ORM、Bean 验证器）都依赖这个约定。

还有一个方法 `incrementViewCount()` 没有对应的 setter。浏览量不应该被外部直接设置（`setViewCount(99999)` 这种操作毫无意义），只应该通过自增来变化。这也是封装的意义——不是所有字段都需要 setter，暴露什么行为，取决于业务逻辑。

### 业务方法：发布文章

Post 不只是一个数据容器，它还应该有行为。最典型的就是"发布"操作——把状态从 DRAFT 改成 PUBLISHED，同时记录发布时间。

```java
public class Post {
    // ... 字段、构造器、getter/setter ...

    /**
     * 发布文章：将状态改为 PUBLISHED，并记录发布时间
     */
    public void publish() {
        if ("PUBLISHED".equals(this.status)) {
            throw new IllegalStateException("文章已经发布过了");
        }
        this.status = "PUBLISHED";
        this.publishedAt = java.time.LocalDateTime.now().toString();
    }

    /**
     * 判断文章是否已发布
     */
    public boolean isPublished() {
        return "PUBLISHED".equals(this.status);
    }

    /**
     * 生成文章摘要（取前 100 个字符）
     */
    public String getExcerpt() {
        if (this.content == null || this.content.length() <= 100) {
            return this.content;
        }
        return this.content.substring(0, 100) + "...";
    }
}
```

`publish()` 方法做了两件事：校验状态（不能重复发布）和更新数据（改状态、记时间）。这种"数据 + 行为绑定在一起"的模式，就是面向对象的精髓。在 JS 里你可能会写一个独立的 `publishPost(post)` 函数，数据和行为是分开的。Java 的风格是行为跟着数据走——post 自己知道怎么发布自己。

字符串比较用 `"PUBLISHED".equals(this.status)` 而不是 `this.status == "PUBLISHED"`。还记得上一篇讲的吗？`==` 比较引用，`equals` 比较内容。而且把常量 `"PUBLISHED"` 放在前面调用 equals，即使 `this.status` 是 null 也不会抛 NullPointerException——这是 Java 程序员的一个防御性编程习惯。

## 定义 Tag 类

Tag 比 Post 简单得多，但套路是一样的：私有字段、构造器、getter/setter。

```java
public class Tag {
    private Long id;
    private String name;
    private String slug;
    private String color;

    // 无参构造器
    public Tag() {
    }

    // 常用构造器：创建时指定 name 和 slug
    public Tag(String name, String slug) {
        this.name = name;
        this.slug = slug;
    }

    // 全参构造器
    public Tag(Long id, String name, String slug, String color) {
        this.id = id;
        this.name = name;
        this.slug = slug;
        this.color = color;
    }

    // Getter 和 Setter
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }

    public String getName() { return name; }
    public void setName(String name) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("标签名不能为空");
        }
        this.name = name;
    }

    public String getSlug() { return slug; }
    public void setSlug(String slug) {
        // slug 应该是小写字母和连字符
        if (slug != null && !slug.matches("^[a-z0-9-]+$")) {
            throw new IllegalArgumentException("slug 只能包含小写字母、数字和连字符");
        }
        this.slug = slug;
    }

    public String getColor() { return color; }
    public void setColor(String color) { this.color = color; }

    @Override
    public String toString() {
        return "Tag{id=" + id + ", name='" + name + "', color='" + color + "'}";
    }
}
```

注意 `setSlug` 里的正则校验——`^[a-z0-9-]+$` 确保 slug 只包含小写字母、数字和连字符。`"java-stream"` 合法，`"Java Stream"` 不合法。这种输入校验在 JS 里经常被忽略，但在 Java 里已经成为社区习惯，因为类型系统强迫你提前思考"这个字段应该接受什么值"。

`toString()` 方法加了 `@Override` 注解。这个方法继承自 `Object` 类（Java 所有类的根），我们在这里重写它，让 `System.out.println(tag)` 打印出可读的内容而不是 `Tag@7a81197d` 这种内存地址。`@Override` 告诉编译器"我确实要重写父类的方法"，如果方法签名写错了，编译器会报错，而不是默默创建一个新方法。

## 把 Post 和 Tag 用起来

类定义好了，现在创建对象来测试。

```java
public class Main {
    public static void main(String[] args) {
        // 创建一篇新文章
        Post post = new Post();
        post.setTitle("Java 类与对象入门");
        post.setSlug("java-classes-and-objects");
        post.setContent("# Java 的 class\n\nJava 的 class 是真家伙...");
        // status 默认是 "DRAFT"，createdAt 在构造器里已设置

        System.out.println("标题: " + post.getTitle());
        System.out.println("状态: " + post.getStatus());
        System.out.println("摘要: " + post.getExcerpt());

        // 发布文章
        post.publish();
        System.out.println("发布后状态: " + post.getStatus());
        System.out.println("发布时间: " + post.getPublishedAt());

        // 浏览量自增
        post.incrementViewCount();
        post.incrementViewCount();
        System.out.println("浏览量: " + post.getViewCount());

        System.out.println("---");

        // 创建标签
        Tag java = new Tag("Java", "java");
        java.setColor("#E76F00");
        Tag oop = new Tag("面向对象", "oop");
        oop.setColor("#2E8B57");

        System.out.println(java);
        System.out.println(oop);

        // 尝试发布已发布的文章
        post.publish(); // 抛出 IllegalStateException
    }
}
```

注意 `new Post()` 创建对象的过程。在 JS 里，`new Post()` 调用 constructor 函数，返回一个新对象。在 Java 里，`new Post()` 做了三件事：在堆内存分配空间、执行构造器初始化字段、返回引用。关键字都是 `new`，但底层机制完全不同。

运行后，`post.publish()` 第二次调用会抛异常。在 JS 里你可能用 `if (post.status !== 'PUBLISHED')` 来避免，但 Java 的风格是让方法自己做校验，调用方不用操心——"post 自己知道怎么发布自己"。

## 构造器的 this() 调用

当一个类有多个构造器时，它们之间经常有重复的初始化逻辑。Java 允许一个构造器通过 `this()` 调用另一个构造器：

```java
public class Post {
    private Long id;
    private String title;
    private String slug;
    private String content;
    private String status;
    private long viewCount;
    private String createdAt;
    private String publishedAt;

    public Post() {
        this(null, null, null, null, "DRAFT", 0,
             java.time.LocalDateTime.now().toString(), null);
    }

    public Post(String title, String slug, String content) {
        this(null, title, slug, content, "DRAFT", 0,
             java.time.LocalDateTime.now().toString(), null);
    }

    public Post(Long id, String title, String slug, String content,
                String status, long viewCount, String createdAt, String publishedAt) {
        this.id = id;
        this.title = title;
        this.slug = slug;
        this.content = content;
        this.status = status;
        this.viewCount = viewCount;
        this.createdAt = createdAt;
        this.publishedAt = publishedAt;
    }
}
```

`this(...)` 必须是构造器的第一条语句。它让"轻量"的构造器把参数补全后交给"全量"的构造器处理，避免到处重复赋值代码。JS 没有这个机制——你只能把公共逻辑抽成一个普通方法，然后在 constructor 里调用。

这种设计在 CMS 项目里很实用。从数据库查出完整数据时用全参构造器，用户提交表单创建新文章时用三个参数的构造器，某些框架（比如 Jackson 反序列化 JSON）需要无参构造器。三个入口，最终都走同一条路。

## Trade-off：严格 vs 自由

Java 的 class 设计比 JS 严格得多：必须声明字段类型、一个文件一个 public 类、不能运行时加属性、构造器必须和类名一致。这种严格换来的是编译期的错误检查——大部分类型错误、访问权限问题、方法签名不匹配，在编译阶段就能发现，不用等到线上才爆炸。

代价是写起来啰嗦。一个 Post 类要写几十行代码，同样的数据结构在 JS 里一个对象字面量就搞定了。如果你在写一个快速原型，这种严格确实是一种负担。但当你维护一个多人协作的项目，几十个类互相调用时，编译器的严格检查能帮你挡住很多低级错误。

getter/setter 的繁琐是 Java 被吐槽最多的点之一。好消息是，下一篇讲的 Lombok 可以用注解自动生成这些方法。但即使不借助工具，理解 getter/setter 背后的封装思想也很重要——它不是形式主义，而是"控制数据访问边界"的手段。

本篇我们正式进入了面向对象的领域，完成了 CMS 项目的基础类设计：Post 类封装了文章的数据和行为（创建、发布、生成摘要），Tag 类封装了标签数据和格式校验。同时第一次感受到 Java class 和 JS class 的本质差异——Java 的 class 编译后是独立的字节码，JVM 对类型的理解比 JS 引擎精确得多。
