# Lambda 与函数式接口

在 CMS 中，我们经常需要按条件过滤文章：查找已发布的、某个标签下的、某个作者的。在 JS 里，一个 `filter` 加箭头函数就搞定了。Java 也有类似的写法，但背后的机制和限制完全不同。

## 从匿名内部类到 Lambda

先看 Java 8 之前的做法。假设我们要给 CMS 的文章列表排序：

```java
// Java 8 之前：匿名内部类
posts.sort(new Comparator<Post>() {
    @Override
    public int compare(Post a, Post b) {
        return b.getCreatedAt().compareTo(a.getCreatedAt());
    }
});
```

5 行代码，真正有意义的只有一行比较逻辑。Java 8 引入 Lambda 表达式后，可以简化为：

```java
// Lambda 写法
posts.sort((a, b) -> b.getCreatedAt().compareTo(a.getCreatedAt()));
```

Lambda 的语法是 `(参数列表) -> { 方法体 }`。当方法体只有一条语句时，大括号和 `return` 都可以省略。看起来和 JS 的箭头函数 `(a, b) => b.createdAt - a.createdAt` 很像，但本质上有重要区别。

## 函数式接口：Lambda 的类型基础

JS 中，函数是一等公民，可以赋值给变量、作为参数传递、作为返回值。Java 不是这样——Java 的 Lambda 必须依附于一个**函数式接口**（FunctionalInterface）才能存在。

函数式接口是只包含一个抽象方法的接口。`@FunctionalInterface` 注解不是必须的，但加上后编译器会帮你检查：

```java
@FunctionalInterface
interface PostFilter {
    boolean test(Post post);
}
```

当你写一个 Lambda 时，Java 编译器会根据上下文推断它属于哪个函数式接口。上面的排序例子中，`sort` 方法需要一个 `Comparator<Post>` 参数，编译器看到 Lambda 的签名 `(Post, Post) -> int` 匹配 `Comparator` 的唯一抽象方法 `compare`，就完成了绑定。

这意味着同一个 Lambda 表达式，在不同的上下文中可以是不同的类型：

```java
// 同一个 Lambda，在不同上下文中类型不同
Comparator<String> cmp = (a, b) -> a.length() - b.length();  // Comparator
Predicate<String> isEmpty = s -> s.isEmpty();                  // Predicate
```

在 JS 中不存在这种区分——函数就是函数，`typeof` 永远返回 `"function"`。

## 四个核心函数式接口

`java.util.function` 包下有四个最常用的函数式接口，覆盖了大部分场景：

**Predicate\<T\>**——判断，返回 `boolean`：

```java
Predicate<Post> isPublished = post -> post.getStatus() == PostStatus.PUBLISHED;
boolean result = isPublished.test(somePost);
```

**Function\<T, R\>**——转换，接受 T 返回 R：

```java
Function<Post, String> toTitle = post -> post.getTitle();
String title = toTitle.apply(somePost);
```

**Consumer\<T\>**——消费，接受 T 无返回值：

```java
Consumer<Post> printTitle = post -> System.out.println(post.getTitle());
printTitle.accept(somePost);
```

**Supplier\<T\>**——供应，无参数返回 T：

```java
Supplier<Post> newDraft = () -> new Post("未命名草稿", "");
Post draft = newDraft.get();
```

它们都可以通过 `.and()`、`.or()`、`.negate()`、`.andThen()`、`.compose()` 等方法进行组合：

```java
// 组合条件：已发布 且 标题不为空
Predicate<Post> filter = isPublished.and(post -> !post.getTitle().isBlank());
```

## 方法引用（::）

当 Lambda 的方法体只是调用一个已有方法时，可以用方法引用进一步简化：

```java
// Lambda 写法
posts.stream().map(post -> post.getTitle()).forEach(title -> System.out.println(title));

// 方法引用写法
posts.stream().map(Post::getTitle).forEach(System.out::println);
```

方法引用有四种形式：

| 形式                 | 语法                      | 示例                  |
| -------------------- | ------------------------- | --------------------- |
| 静态方法             | `ClassName::staticMethod` | `Integer::parseInt`   |
| 实例方法（特定对象） | `instance::method`        | `System.out::println` |
| 实例方法（任意对象） | `ClassName::method`       | `String::toUpperCase` |
| 构造器               | `ClassName::new`          | `ArrayList::new`      |

方法引用不只是语法糖。`Post::getTitle` 实际上是编译器帮你生成了一个实现了 `Function<Post, String>` 的 Lambda。选择哪种写法取决于可读性——如果方法引用让人一眼看懂意图，就用它；如果 Lambda 更清晰，就用 Lambda。

## Lambda 的限制：effectively final

这是 JS 开发者最常踩的坑。在 JS 中，箭头函数可以自由修改外部变量：

```javascript
let count = 0;
const increment = () => {
  count++;
}; // 完全合法
increment();
console.log(count); // 1
```

但 Java 的 Lambda 只能访问**effectively final** 的外部变量——即声明后不再被修改的变量：

```java
int count = 0;
// 编译错误：Variable used in lambda expression should be final or effectively final
Consumer<Integer> increment = n -> count += n;
```

如果要"修改"外部变量，需要用容器包装：

```java
int[] holder = {0};  // 用数组做容器
Consumer<Integer> increment = n -> holder[0] += n;
increment.accept(5);
System.out.println(holder[0]);  // 5
```

或者用原子类（上篇刚学的）：

```java
AtomicInteger count = new AtomicInteger(0);
Consumer<Integer> increment = n -> count.addAndGet(n);
increment.accept(5);
System.out.println(count.get());  // 5
```

这个限制的根本原因是：Lambda 在 Java 中会被编译为匿名内部类（或方法句柄），它捕获外部变量时是**值捕获**，不是引用捕获。如果允许修改外部变量，语义会变得混乱——你修改的是 Lambda 内的副本，还是外部的原值？

JS 的箭头函数不存在这个问题，因为它通过闭包直接引用外层作用域的变量。

## CMS 实战：标签匹配的多种写法

假设 CMS 需要根据用户选择的标签过滤文章。我们来看几种实现方式：

```java
import java.util.*;
import java.util.function.*;
import java.util.stream.*;

public class PostFilterService {
    private List<Post> posts;

    public PostFilterService(List<Post> posts) {
        this.posts = posts;
    }

    // 方式一：传统方法，传入标签名
    public List<Post> byTag(String tagName) {
        List<Post> result = new ArrayList<>();
        for (Post post : posts) {
            if (post.getTags().stream().anyMatch(tag -> tag.getName().equals(tagName))) {
                result.add(post);
            }
        }
        return result;
    }

    // 方式二：接受 Predicate，调用者自定义过滤逻辑
    public List<Post> filter(Predicate<Post> predicate) {
        return posts.stream()
            .filter(predicate)
            .collect(Collectors.toList());
    }

    // 方式三：组合多个条件
    public List<Post> filterComposite(List<Predicate<Post>> conditions) {
        return posts.stream()
            .filter(conditions.stream().reduce(Predicate::and).orElse(post -> true))
            .collect(Collectors.toList());
    }
}
```

调用时可以灵活组合：

```java
PostFilterService service = new PostFilterService(allPosts);

// 查找标签为 "Java" 的文章
List<Post> javaPosts = service.byTag("Java");

// 用 Predicate 自定义：已发布的、浏览量超过 1000 的
List<Post> popular = service.filter(
    post -> post.getStatus() == PostStatus.PUBLISHED && post.getViewCount() > 1000
);

// 组合多个条件
List<Predicate<Post>> conditions = Arrays.asList(
    post -> post.getStatus() == PostStatus.PUBLISHED,
    post -> post.getViewCount() > 100,
    post -> post.getTags().stream().anyMatch(t -> t.getName().equals("Java"))
);
List<Post> results = service.filterComposite(conditions);
```

方式二是 Java 中**高阶函数**的典型用法：把判断逻辑作为参数传入。在 JS 中这很自然，在 Java 中通过函数式接口实现了同样的效果。

## JS vs Java 对比

| 维度     | JavaScript 箭头函数                      | Java Lambda                            |
| -------- | ---------------------------------------- | -------------------------------------- |
| 类型     | 函数是一等公民，`typeof` 是 `"function"` | 必须依附于函数式接口，有明确的类型     |
| 闭包     | 引用捕获，可修改外部变量                 | 值捕获，外部变量必须 effectively final |
| `this`   | 继承外层 `this`（不绑定自己的 `this`）   | 继承外层 `this`（和匿名内部类不同）    |
| 方法引用 | 无对应概念                               | `ClassName::method`                    |
| 函数组合 | 手动组合或用 lodash/fp                   | 内置 `.andThen()`、`.compose()` 等     |
| 返回值   | 可以返回任何类型                         | 返回类型由函数式接口决定               |

## Trade-off

Java 的 Lambda 比 JS 箭头函数多了一层"函数式接口"的约束，这看起来是限制，但实际上提供了更好的类型安全。编译器能在编译期检查 Lambda 的参数类型和返回类型是否匹配，而 JS 只能在运行时才发现类型错误。

`effectively final` 的限制虽然让习惯了 JS 闭包的开发者不适应，但它从根本上避免了并发场景下对外部变量的意外修改。在上篇讲并发安全时我们看到，多线程下对外部变量的无保护修改是 bug 的重要来源。

方法引用是 Java 的独有优势——JS 没有等价物。`Post::getTitle` 比写 Lambda 更简洁，也比传递函数名更明确。在 Stream 操作链中大量使用方法引用，可以让代码的可读性接近自然语言。
