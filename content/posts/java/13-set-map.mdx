CMS 后台有一个标签输入框，用户可以给文章添加标签。但用户可能输入重复标签——"Java"、"java"、"Java "（带空格）。你需要去重，还要统计每个标签下有多少篇文章。另一个场景：首页加载文章时，如果每次都查数据库，响应会越来越慢。你需要一个按 slug 缓存文章的机制。

这两个需求分别对应 Set 和 Map：Set 解决"存不存在"的问题，Map 解决"按 key 找 value"的问题。

## Set：保证唯一性

### HashSet 的工作原理

HashSet 的底层其实是一个 HashMap——每次 `add(e)` 时，HashSet 把元素作为 key 放进一个内部的 HashMap，value 是一个固定的空对象 `PRESENT`。所以 HashSet 的性能特性和 HashMap 完全一致。

判断两个元素是否"相同"的流程：

1. 先计算 `hashCode()`，如果哈希值不同，直接判定为不同元素
2. 如果哈希值相同，再调用 `equals()` 做精确比较
3. 两个都相同，才认为是重复元素

这意味着如果你自定义了类并放入 HashSet，必须同时重写 `hashCode()` 和 `equals()`，而且要保证：**两个 equals 相等的对象，hashCode 必须相同**。违反这个规则会导致 HashSet 中出现重复元素。

```java
public class Tag {
    private final String name;

    public Tag(String name) {
        this.name = name.trim().toLowerCase(); // 标准化处理
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Tag)) return false;
        return name.equals(((Tag) o).name);
    }

    @Override
    public int hashCode() {
        return name.hashCode();
    }

    public String getName() {
        return name;
    }
}
```

注意 Tag 的构造函数里做了 `trim().toLowerCase()`——这就是解决开头"Java"、"java"、"Java "三种写法的问题。在数据进入 Set 之前就标准化，比在 equals 里做更可靠。

### CMS 标签去重

```java
public class TagService {
    // 标签去重
    public Set<Tag> deduplicateTags(List<String> rawTags) {
        Set<Tag> uniqueTags = new LinkedHashSet<>();
        for (String raw : rawTags) {
            uniqueTags.add(new Tag(raw));
        }
        return uniqueTags;
    }

    // 统计标签使用频率
    public Map<Tag, Integer> countTagUsage(List<List<Tag>> postTags) {
        Map<Tag, Integer> count = new HashMap<>();
        for (List<Tag> tags : postTags) {
            for (Tag tag : tags) {
                count.merge(tag, 1, Integer::sum);
            }
        }
        return count;
    }
}
```

`count.merge(key, 1, Integer::sum)` 是 Java 8 的写法，等价于"如果 key 不存在就设为 1，存在就加 1"。比手写 `getOrDefault` + `put` 更简洁。

### HashSet vs TreeSet vs LinkedHashSet

| 特性           | HashSet   | LinkedHashSet      | TreeSet                |
| -------------- | --------- | ------------------ | ---------------------- |
| 底层           | HashMap   | HashMap + 双向链表 | 红黑树（NavigableMap） |
| 插入/查找/删除 | O(1) 均摊 | O(1) 均摊          | O(log n)               |
| 迭代顺序       | 不保证    | 插入顺序           | 排序顺序               |
| null 元素      | 允许      | 允许               | 不允许（需要比较）     |
| 线程安全       | 否        | 否                 | 否                     |

CMS 的标签展示场景：如果需要按插入顺序显示用户添加的标签（UX 上更直觉），用 `LinkedHashSet`；如果需要按字母序显示（比如标签云），用 `TreeSet`；如果不关心顺序，用 `HashSet`（性能最好）。

```java
// 按字母序排列的标签云
Set<Tag> tagCloud = new TreeSet<>(Comparator.comparing(Tag::getName));

// 保持用户添加顺序
Set<Tag> userTags = new LinkedHashSet<>();
```

TreeSet 要求元素实现 `Comparable`，或者在构造时传入 `Comparator`。这就是为什么上面 Tag 类没有实现 `Comparable`——因为排序规则是使用场景决定的，不应该写死在类里。

## Map：键值映射

### HashMap 的内部结构

HashMap 是 Java 中使用频率最高的集合类。它的内部结构在 JDK 8 之后是"数组 + 链表 + 红黑树"：

```
HashMap 内部结构（简化）：
┌─────────────────────────────────────────┐
│  Node[] table（桶数组）                   │
│  ┌──────┐  ┌──────┐  ┌──────┐          │
│  │  0   │  │  1   │  │  2   │  ...     │
│  └──┬───┘  └──┬───┘  └──┬───┘          │
│     │         │         │               │
│     ▼         ▼         ▼               │
│  [K,V]→[K,V]  null    [K,V]            │
│  链表或红黑树             │              │
│                          ▼              │
│                       [K,V]→[K,V]      │
└─────────────────────────────────────────┘
```

当某个桶中的链表长度超过 8 且数组总长度超过 64 时，链表会转为红黑树，将最坏情况的查找复杂度从 O(n) 降到 O(log n)。

`put(key, value)` 的过程：

1. 计算 key 的 `hashCode()`，然后通过 `(n - 1) & hash` 定位到桶索引（`n` 是数组长度，必须是 2 的幂）
2. 如果桶为空，直接放入
3. 如果桶不为空，遍历链表（或红黑树），用 `equals()` 比较key：
   - 找到相同的 key，替换 value
   - 没找到，在链表尾部追加新节点
4. 如果元素数量超过 `容量 * 负载因子（默认 0.75）`，触发扩容（数组长度翻倍，重新分配所有元素）

### CMS 文章缓存

```java
public class PostCache {
    private final Map<String, Post> cache = new HashMap<>();

    public Optional<Post> getBySlug(String slug) {
        // HashMap 的 get 是 O(1) 均摊
        return Optional.ofNullable(cache.get(slug));
    }

    public void put(Post post) {
        cache.put(post.getSlug(), post);
    }

    public void putAll(List<Post> posts) {
        for (Post post : posts) {
            cache.put(post.getSlug(), post);
        }
    }

    public void remove(String slug) {
        cache.remove(slug);
    }

    // 批量查找：给定一组 slug，返回找到的文章
    public Map<String, Post> getBySlugs(Collection<String> slugs) {
        Map<String, Post> result = new HashMap<>();
        for (String slug : slugs) {
            Post post = cache.get(slug);
            if (post != null) {
                result.put(slug, post);
            }
        }
        return result;
    }
}
```

这里用 slug（URL 友好的标识符）作为 key 而不是文章 ID，是因为 slug 是用户可见的，在 URL 中直接使用，省去了一次查找。HashMap 的 `get()` 均摊 O(1)，即使缓存了上万篇文章，查找速度也不会明显下降。

### Map 的遍历

Map 有三种遍历方式，效率不同：

```java
Map<String, Post> cache = new HashMap<>();

// 方式 1：entrySet —— 推荐，同时拿到 key 和 value
for (Map.Entry<String, Post> entry : cache.entrySet()) {
    String slug = entry.getKey();
    Post post = entry.getValue();
}

// 方式 2：forEach（Java 8+）—— 更简洁
cache.forEach((slug, post) -> {
    System.out.println(slug + " -> " + post.getTitle());
});

// 方式 3：keySet + get —— 不推荐，每次 get 都要再哈希一次
for (String slug : cache.keySet()) {
    Post post = cache.get(slug); // 额外的哈希计算
}
```

方式 3 的问题在于：`keySet()` 遍历每个 key 时，`cache.get(slug)` 又要重新计算哈希、定位桶、遍历链表。虽然均摊还是 O(1)，但常数因子是方式 1 的两倍。数据量大时差距明显。

### HashMap vs TreeMap vs LinkedHashMap

| 特性           | HashMap            | LinkedHashMap      | TreeMap            |
| -------------- | ------------------ | ------------------ | ------------------ |
| 底层           | 数组 + 链表/红黑树 | HashMap + 双向链表 | 红黑树             |
| 查找/插入/删除 | O(1) 均摊          | O(1) 均摊          | O(log n)           |
| 迭代顺序       | 不保证             | 插入顺序或访问顺序 | 按 key 排序        |
| null key       | 允许一个           | 允许一个           | 不允许             |
| 适用场景       | 通用缓存、查找表   | 需要保持顺序的映射 | 范围查询、有序遍历 |

CMS 中的选择：文章缓存用 HashMap（不需要顺序，需要最快的查找）；文章按时间排列的归档页用 LinkedHashMap（保持插入顺序，即按时间）；按标签名排序的标签索引用 TreeMap（需要有序遍历和范围查询）。

LinkedHashMap 还有一个特殊能力——通过 `accessOrder` 参数实现 LRU 缓存：

```java
// accessOrder=true 时，每次 get 操作会把访问的元素移到链表尾部
Map<String, Post> lruCache = new LinkedHashMap<>(100, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, Post> eldest) {
        return size() > 100; // 超过 100 篇就淘汰最早的
    }
};
```

每次 `get(slug)` 都会更新访问顺序，最久没被访问的文章会被自动淘汰。这是一个轻量级的 LRU 实现，不需要引入额外的缓存库。

### HashMap 的 hashCode 冲突

两个不同的 key 算出了相同的哈希值，就会落到同一个桶里形成链表。String 的 `hashCode()` 实现很好，冲突概率很低，但如果你自定义类作为 key，需要注意 hashCode 的质量。

一个常见的错误：只用了对象的一个字段计算 hashCode：

```java
// 错误：只用 name 计算 hashCode
@Override
public int hashCode() {
    return name.hashCode();
}

// 正确：使用所有影响 equals 的字段
@Override
public int hashCode() {
    return Objects.hash(name, version);
}
```

如果两个不同版本的文章有相同的 name，错误写法会导致它们映射到同一个桶，退化成链表遍历。`Objects.hash()` 是最安全的选择。

## JS vs Java 对比

| 概念         | JavaScript                   | Java                               |
| ------------ | ---------------------------- | ---------------------------------- |
| 去重         | `new Set(array)`             | `new HashSet<>(list)`              |
| 键值映射     | `Map`（ES6）或 `Object`      | `HashMap` / `TreeMap`              |
| Set 迭代顺序 | 插入顺序                     | HashSet 不保证，LinkedHashSet 保证 |
| Map 遍历     | `map.forEach()` / `for...of` | `entrySet()` / `forEach()`         |
| 判等         | `===`（值比较）              | `hashCode()` + `equals()`          |
| null key     | Map 允许，Object 不允许      | HashMap 允许一个，TreeMap 不允许   |

JS 的 Set 和 Map 底层实现取决于引擎（V8 用有序哈希表），API 更简洁，但缺少 Java 那些针对不同场景的变体（LinkedHashSet、TreeMap 等）。在 Java 中选择哪个实现是一个有意义的决策，在 JS 中你只需要用 `Set` 和 `Map`。

## 回顾与预告

上篇我们深入了 List：ArrayList vs LinkedList 的选择、Comparator 排序和 subList 分页。本篇讲了 Set 和 Map——Set 通过 `hashCode()` + `equals()` 保证唯一性，Map 用同样的机制实现 O(1) 的键值查找。

核心取舍：HashSet/HashMap 的性能依赖于正确的 `hashCode()` 和 `equals()` 实现；LinkedHashSet/LinkedHashMap 多用一点内存换来顺序保证；TreeSet/TreeMap 用 O(log n) 换来排序能力。选哪个取决于你是否需要顺序。
