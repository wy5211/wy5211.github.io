Post 类有个字段叫 `status`，从第 5 篇开始就一直是 String 类型，值要么是 `"DRAFT"` 要么是 `"PUBLISHED"`。这没问题，但存在一个隐患：没有任何东西阻止你写出 `post.setStatus("PUBLSIHED")`——拼错一个字母，编译器不会报错，运行时逻辑就悄悄错了。

上一篇文章结尾提到过这个话题：status 这种只有有限几个合法值的字段，用字符串表示太脆弱了。今天来做两件事——用枚举（enum）彻底解决这个问题，以及顺带认识内部类（inner class），它是实现 Builder 模式的基础。

## JS 里怎么表示有限值集合

在 JavaScript 中，表示有限值集合有几种常见做法，各有优劣。

最朴素的方式是用纯字符串：

```js
// 最简单，也最危险
post.status = "DRAFT";
post.status = "PUBLSIHED"; // 拼错了，运行时才出问题
post.status = "DELETE"; // 不存在的状态，没人拦得住
```

稍微好一点，用常量对象：

```js
const PostStatus = {
  DRAFT: "draft",
  PUBLISHED: "published",
};

post.status = PostStatus.DRAFT;
// 但 TypeScript 之外，没有类型约束
post.status = "any string"; // 依然合法
```

TypeScript 用联合类型（union type）能加上编译期检查：

```ts
type PostStatus = "DRAFT" | "PUBLISHED";

function publish(post: { status: PostStatus }) {
  post.status = "PUBLISHED"; // OK
  post.status = "DELETED"; // TS 编译报错
}
```

TypeScript 的联合类型在编译期确实能帮你拦住错误值，但它本质上是字符串的子集，运行时不存在 `PostStatus.DRAFT` 这样的实体对象。你无法给 `"DRAFT"` 附加额外信息（比如中文显示名、是否允许编辑），也无法定义方法（比如"当前状态是否可以转到另一个状态"）。

Java 的 enum 能同时解决这两个问题。

## Java enum：不只是常量集合

先看最简单的枚举定义：

```java
public enum PostStatus {
    DRAFT,
    PUBLISHED,
    ARCHIVED
}
```

把它用在 Post 类里，替代原来的 String：

```java
public class Post {
    // 之前：private String status;
    // 现在：
    private PostStatus status;

    public Post() {
        this.status = PostStatus.DRAFT; // 类型安全，编译器强制检查
    }

    public void setStatus(PostStatus status) {
        this.status = status;
    }

    public PostStatus getStatus() {
        return status;
    }

    public void publish() {
        if (this.status == PostStatus.PUBLISHED) {
            throw new IllegalStateException("文章已经发布过了");
        }
        this.status = PostStatus.PUBLISHED;
        this.publishedAt = java.time.LocalDateTime.now().toString();
    }
}
```

和之前的字符串版本比，变化不大，但安全性完全不同。`setStatus` 的参数类型是 `PostStatus`，传入任何其他类型编译器直接拒绝——不存在拼错、不存在"DELETE"这种非法值。调用方写 `post.setStatus(PostStatus.DRAFT)` 时，IDE 会自动补全枚举值，不用去记到底有哪些合法状态。

注意枚举比较用 `==` 就够了，不需要 `equals()`。每个枚举值在 JVM 中只有一个实例（类似单例），所以 `==` 比较的就是同一个对象引用。这和字符串恰好相反——字符串必须用 `equals()`，枚举值用 `==` 更简洁。

## 枚举不只是常量：字段和方法

如果 enum 只能做到"限定合法值"，那它和 TypeScript 的联合类型差别不大。Java enum 的真正威力在于：**枚举值本身可以携带数据和行为**。

CMS 的文章有三种状态：草稿、已发布、已归档。每种状态有中文显示名，而且状态之间的转换有规则——草稿可以发布，已发布可以归档，但不能从草稿直接跳到归档。

```java
public enum PostStatus {
    DRAFT("草稿"),
    PUBLISHED("已发布"),
    ARCHIVED("已归档");

    private final String displayName;

    // 枚举的构造器：每个枚举值在定义时就确定了自己的 displayName
    PostStatus(String displayName) {
        this.displayName = displayName;
    }

    public String getDisplayName() {
        return displayName;
    }

    /**
     * 判断当前状态是否允许转换到目标状态
     * DRAFT -> PUBLISHED (允许)
     * PUBLISHED -> ARCHIVED (允许)
     * DRAFT -> ARCHIVED (不允许)
     * 其他情况也不允许
     */
    public boolean canTransitionTo(PostStatus target) {
        return switch (this) {
            case DRAFT -> target == PUBLISHED;
            case PUBLISHED -> target == ARCHIVED;
            default -> false;
        };
    }
}
```

这段代码里有几个 JS/TS 开发者可能觉得新鲜的地方。

**枚举的构造器是私有的**。你不能写 `new PostStatus("草稿")`——枚举值在类加载时就已经创建好了，`DRAFT`、`PUBLISHED`、`ARCHIVED` 就是全部实例。每个实例通过 `DRAFT("草稿")` 这种语法调用构造器，括号里的参数传给 `PostStatus(String displayName)` 构造器。

**switch 表达式**。`switch (this)` 后面用了 Java 14 引入的 switch 表达式（箭头语法）。和传统的 switch 语句不同，switch 表达式有返回值，而且不需要 break。更重要的是，当 switch 的目标是枚举类型时，**IDE 和编译器会检查是否穷举了所有枚举值**。如果你只处理了 `DRAFT` 和 `PUBLISHED` 而漏了 `ARCHIVED`，IDE 会给出警告。这在 JS 的 switch 里是不可能做到的——JS 的 switch 对字符串值没有任何穷举检查。

把状态转换逻辑放在枚举里，Post 类的代码就变得非常干净：

```java
public class Post {
    private PostStatus status = PostStatus.DRAFT;

    /**
     * 安全地转换状态：先检查是否允许，不允许则抛异常
     */
    public void transitionTo(PostStatus target) {
        if (!this.status.canTransitionTo(target)) {
            throw new IllegalStateException(
                "不能从 " + this.status.getDisplayName()
                + " 转换到 " + target.getDisplayName()
            );
        }
        this.status = target;

        if (target == PostStatus.PUBLISHED) {
            this.publishedAt = java.time.LocalDateTime.now().toString();
        }
    }
}
```

`transitionTo` 方法不知道也不关心具体的转换规则——它只问枚举"能不能转"。如果以后新增了一个 `REVIEWING` 状态，只需要在 `PostStatus` 里加一行枚举值、调整 `canTransitionTo` 的逻辑，Post 类一行都不用改。这就是**把行为放在数据旁边**的思想：状态转换是"状态"自己的事，不是 Post 的事。

对比一下 TS 的做法——同样的逻辑你得写在外部函数里：

```ts
type PostStatus = "DRAFT" | "PUBLISHED" | "ARCHIVED";

// 转换规则散落在某个工具函数里，或者直接写在 Post 类的条件判断里
function canTransition(from: PostStatus, to: PostStatus): boolean {
  if (from === "DRAFT") return to === "PUBLISHED";
  if (from === "PUBLISHED") return to === "ARCHIVED";
  return false;
}
```

TS 的字符串联合类型只是值的约束，行为只能游离在外面。Java enum 把值和行为绑在了一起。

## 枚举实现接口

enum 还可以实现接口。这在 CMS 中也有实际用途——假设我们希望所有"可显示的枚举"都提供一个统一的 `getDisplayName()` 方法：

```java
public interface Displayable {
    String getDisplayName();
}

public enum PostStatus implements Displayable {
    DRAFT("草稿"),
    PUBLISHED("已发布"),
    ARCHIVED("已归档");

    private final String displayName;

    PostStatus(String displayName) {
        this.displayName = displayName;
    }

    @Override
    public String getDisplayName() {
        return displayName;
    }
}
```

实现接口后，`PostStatus` 就可以被当作 `Displayable` 类型使用。这在后续讲集合框架时会更有用——比如你有一个 `List<Displayable>`，里面可以放 `PostStatus`、`UserRole`（用户角色枚举）等各种实现了 `Displayable` 的枚举，统一调用 `getDisplayName()`。

每个枚举值还可以覆写方法，实现"每个值有不同的行为"：

```java
public enum PostStatus {
    DRAFT("草稿") {
        @Override
        public String getActionText() {
            return "提交审核";
        }
    },
    PUBLISHED("已发布") {
        @Override
        public String getActionText() {
            return "归档文章";
        }
    },
    ARCHIVED("已归档") {
        @Override
        public String getActionText() {
            return "重新发布";
        }
    };

    private final String displayName;

    PostStatus(String displayName) {
        this.displayName = displayName;
    }

    public String getDisplayName() {
        return displayName;
    }

    // 抽象方法：每个枚举值必须提供自己的实现
    public abstract String getActionText();
}
```

这种"每个枚举值有不同的行为"的模式，其实就是策略模式的一种简写。在 JS 里你需要写一个映射对象来实现类似效果，没有语言层面的支持。

## 内部类：嵌套在类中的类

Java 允许在一个类的内部定义另一个类，称为内部类（inner class）。内部类有四种形态，但日常开发中最常用的是两种：**静态内部类**和**匿名内部类**。

### 静态内部类

静态内部类用 `static` 修饰，它和外部类的关系是"命名空间上的归属"，不依赖外部类的实例。最典型的用途是实现 Builder 模式。

Post 的字段越来越多（id、title、slug、content、status、authorId、viewCount、createdAt、publishedAt），构造器参数列表也越来越长。Builder 模式可以解决这个问题——用链式调用逐步设置每个字段，最后调用 `build()` 生成对象：

```java
public class Post {
    private Long id;
    private String title;
    private String slug;
    private String content;
    private PostStatus status;
    private Long authorId;
    private long viewCount;
    private String createdAt;
    private String publishedAt;

    private Post() {
        // 私有构造器，外部不能直接 new，只能通过 Builder 创建
    }

    // 静态内部类 Builder
    public static class Builder {
        private String title;
        private String slug;
        private String content;
        private PostStatus status = PostStatus.DRAFT;
        private Long authorId;

        public Builder title(String title) {
            this.title = title;
            return this;
        }

        public Builder slug(String slug) {
            this.slug = slug;
            return this;
        }

        public Builder content(String content) {
            this.content = content;
            return this;
        }

        public Builder status(PostStatus status) {
            this.status = status;
            return this;
        }

        public Builder authorId(Long authorId) {
            this.authorId = authorId;
            return this;
        }

        public Post build() {
            Post post = new Post();
            post.title = this.title;
            post.slug = this.slug;
            post.content = this.content;
            post.status = this.status;
            post.authorId = this.authorId;
            post.createdAt = java.time.LocalDateTime.now().toString();
            return post;
        }
    }
}
```

使用的时候，代码读起来几乎像自然语言：

```java
Post post = new Post.Builder()
    .title("Java 枚举入门")
    .slug("java-enum-guide")
    .content("# 枚举\n\nJava 的 enum 比 JS 的常量对象强在哪？")
    .status(PostStatus.DRAFT)
    .authorId(1L)
    .build();
```

`Post.Builder` 是一个静态内部类——它不需要访问 Post 的实例字段（构造器是私有的，Builder 在 `build()` 方法里才创建 Post 实例）。把它放在 Post 内部的好处是：命名空间清晰（`Post.Builder` 一看就知道是给 Post 服务的），而且 Builder 可以访问 Post 的私有构造器和私有字段，不用额外暴露 setter。

在 JS/TS 里没有内部类的概念，类似的 Builder 通常写成一个独立的类或者工厂函数：

```ts
class PostBuilder {
  private title = "";
  private slug = "";
  private content = "";

  setTitle(title: string) {
    this.title = title;
    return this;
  }
  setSlug(slug: string) {
    this.slug = slug;
    return this;
  }
  setContent(content: string) {
    this.content = content;
    return this;
  }

  build() {
    return { title: this.title, slug: this.slug, content: this.content };
  }
}
```

功能上没有本质区别，但 `PostBuilder` 是一个独立的顶级类，散落在文件里没有归属感。Java 的静态内部类让代码的组织更内聚。

### 匿名内部类

匿名内部类没有名字，通常用于"只需要用一次"的场景。它最常见的身份是 Lambda 表达式的前身——在 Java 8 之前，如果你需要传一个接口的实现，就得写匿名内部类。

在 CMS 中，假设你要对文章列表做排序，用 `Comparator` 接口：

```java
import java.util.Comparator;
import java.util.ArrayList;
import java.util.List;

List<Post> posts = new ArrayList<>();
// ... 添加文章 ...

// 匿名内部类：就地创建一个 Comparator 实现，只用一次
posts.sort(new Comparator<Post>() {
    @Override
    public int compare(Post a, Post b) {
        return b.getViewCount() - (int) a.getViewCount(); // 按浏览量降序
    }
});
```

`new Comparator<Post>() { ... }` 就是一个匿名内部类——它实现了 `Comparator` 接口但没有名字。在 Java 8 之后，这段代码可以用 Lambda 简写为：

```java
posts.sort((a, b) -> (int) (b.getViewCount() - a.getViewCount()));
```

Lambda 更简洁，但理解匿名内部类很重要——它是 Lambda 的底层机制，而且在某些场景下（比如需要实现多个方法、或者需要引用 `this` 指向匿名类自身而非外部类）匿名内部类仍然是唯一选择。

JS 没有匿名内部类的概念，类似的需求用回调函数或箭头函数解决。事实上 JS 的异步回调（`fs.readFile(path, (err, data) => { ... })`）在概念上和 Java 的匿名内部类很相似——都是"传一段代码作为参数，在某个时机执行"。

### 非静态内部类和局部内部类

另外两种内部类（非静态成员内部类、局部内部类）在实际开发中用得较少，简单提一下：

**非静态内部类**持有外部类的实例引用，可以直接访问外部类的字段和方法。它的创建必须依赖外部类实例：`Outer.Inner inner = outer.new Inner()`。这种隐式持有的外部类引用容易造成内存泄漏（内部类实例比外部类活得更久时），所以除非确实需要访问外部类实例状态，否则优先用静态内部类。

**局部内部类**定义在方法内部，作用域仅限该方法。现在基本被 Lambda 表达式取代，了解即可。

## JS vs Java 对比

| 特性            | JavaScript        | TypeScript            | Java                        |
| --------------- | ----------------- | --------------------- | --------------------------- |
| 有限值集合      | 字符串 / 常量对象 | 联合类型 `"A" \| "B"` | enum（真正的类型）          |
| 编译期值检查    | 无                | 有（TS 编译器）       | 有（javac 编译器）          |
| 运行时类型安全  | 无                | 无（编译后都是 JS）   | 有（JVM 类型检查）          |
| 枚举携带数据    | 需要额外对象      | 需要额外对象          | 枚举自带字段                |
| 枚举定义行为    | 需要外部函数      | 需要外部函数          | 枚举自带方法                |
| switch 穷举检查 | 无                | 部分（TS never 类型） | 有（IDE + 编译器）          |
| 内部类          | 无                | 无                    | 静态 / 非静态 / 匿名 / 局部 |
| Builder 模式    | 独立类或工厂函数  | 独立类或工厂函数      | 静态内部类                  |

这张表的核心信息是：JS/TS 对"有限值集合"的支持停留在"值的约束"层面，Java 的 enum 把值、数据、行为打包成了一个完整的类型。这不是语法糖层面的差异，而是类型系统能力的差异。

## Trade-off

enum 的代价是什么？

首先是序列化。枚举在 Java 中序列化（比如存到数据库、转成 JSON）时，默认存储的是枚举值的名称字符串（`"DRAFT"`、`"PUBLISHED"`）。如果有一天你把 `DRAFT` 改名成 `NEW_DRAFT`，数据库里的旧数据就反序列化不回来了。这个问题在 JS 的常量对象中同样存在，但 JS 开发者通常不关心，因为反正存的就是字符串值。Java 开发者需要特别注意：一旦 enum 值被外部系统依赖，就不要改名，只能废弃（加 `@Deprecated`）并新增。

其次是枚举的单例特性。每个枚举值在 JVM 中只有一个实例，不能 clone、不能反射创建新实例、不能反序列化出新实例。这通常是优点（节省内存、可以用 `==` 比较），但如果你的业务需要"同名的不同状态实例"（虽然这几乎不存在），enum 做不到。

内部类的 trade-off 更直接：它让代码组织更内聚，但也增加了类的嵌套层级，阅读时需要跳来跳去。静态内部类几乎没有副作用，可以放心用。非静态内部类要小心内存泄漏，Android 开发中这是经典坑点。

## 回顾

上一篇我们学了接口与抽象类，给 CMS 引入了 `Taggable` 和 `Publishable` 两个接口，Post 同时实现了它们。我们理解了 Java 的 interface 是运行时类型（JVM 通过接口类型做方法分派），而 TypeScript 的 interface 只是编译期的类型声明。default 方法让接口可以提供默认实现，避免了"加一个方法所有实现类都要改"的问题。

本篇做了两件事。第一，把 Post 的 status 从 String 升级为 `PostStatus` 枚举——编译器现在能拦住所有非法状态值，枚举自身携带了 `displayName` 数据和 `canTransitionTo()` 方法，状态转换规则不再散落在 Post 类的条件判断里。第二，认识了内部类，重点是用静态内部类实现了 Post.Builder 构建者模式。

下一篇是异常处理——CMS 的文章发布流程中有很多可能出错的地方：标题为空、状态转换不合法、slug 格式不对。Java 的 checked exception 机制强制你显式处理这些错误，这在 JS/TS 中是没有对应概念的，也是 Java 和 JS 最大的设计分歧之一。
