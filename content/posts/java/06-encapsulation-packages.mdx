上一篇我们定义了 Post 和 Tag 类，所有字段都是 private，通过 getter/setter 控制访问。你可能会问：Java 的访问控制就这一种吗？上一篇对比表里明明写了有四种。

现实情况是，上一篇把 Post 和 Tag 都写在同一个"默认包"里——没有 package 声明，所有类挤在一个平面。这在写 demo 时没问题，但 CMS 项目继续往下走，你需要把模型类、服务类、工具类分开管理。一旦涉及多个包，Java 的访问控制就开始发挥真正的威力。

这篇文章要解决的核心问题是：**为什么 Java 要搞四种访问修饰符？** 这个问题在 JS 里几乎不存在——JS 在 ES2022 才引入 `#` 私有字段，在此之前所有属性都是公开的，你全靠"自觉"不直接访问下划线开头的属性。Java 从第一天起就把访问控制做到了语言层面，不是可选的规范，是编译器强制执行的规则。

## CMS 的包结构

在组织代码之前，先看看 CMS 项目最终会长什么样。Java 的 package 既是命名空间，也是目录结构——包名 `com.techblog.model` 对应文件系统中的 `com/techblog/model/` 目录。

```
src/
├── com/techblog/
│   ├── model/          ← 数据模型
│   │   ├── Post.java
│   │   └── Tag.java
│   ├── service/        ← 业务逻辑
│   │   └── PostService.java
│   └── util/           ← 工具类
│       └── SlugGenerator.java
└── Main.java
```

这跟 JS 项目的目录组织很像，但有一个根本区别：JS 的 `import { Post } from './model/Post'` 只是文件路径，跟代码逻辑没有强绑定；Java 的 `package com.techblog.model` 是语言级别的命名空间，编译器会根据包名决定哪些类能访问哪些成员。这个区别后面会反复体现。

现在给 Post 和 Tag 加上包声明。注意包名全小写，这是 Java 的惯例。

```java
// 文件：com/techblog/model/Post.java
package com.techblog.model;

public class Post {
    private Long id;
    private String title;
    private String slug;
    private String content;
    private String status;
    private long viewCount;
    private String createdAt;
    private String publishedAt;

    // ... 构造器、getter/setter、publish() 等方法不变 ...
}

// 文件：com/techblog/model/Tag.java
package com.techblog.model;

public class Tag {
    private Long id;
    private String name;
    private String slug;
    private String color;

    // ... 构造器、getter/setter 不变 ...
}
```

每个 `.java` 文件的第一行（除了注释和空行）必须是 `package` 声明。Java 编译器用它来确定类的全限定名——`com.techblog.model.Post`，而不是简单的 `Post`。这个全限定名在 JVM 中是唯一的，就像 JS 中模块路径保证唯一一样，只不过 Java 在编译期就完成了这个检查。

## 四种访问修饰符

Java 的访问修饰符控制"谁能看到这个成员"。四种级别从严格到宽松依次是：

| 修饰符          | 同一个类 | 同一个包 | 不同包的子类 | 所有类 |
| --------------- | -------- | -------- | ------------ | ------ |
| `private`       | 可以     | 不行     | 不行         | 不行   |
| default（不写） | 可以     | 可以     | 不行         | 不行   |
| `protected`     | 可以     | 可以     | 可以         | 不行   |
| `public`        | 可以     | 可以     | 可以         | 可以   |

这张表看起来抽象，放到 CMS 场景里就清晰了。我们在 Post 类中用四种修饰符各定义一个成员，然后从不同位置访问它们。

```java
package com.techblog.model;

public class Post {
    // private：只有 Post 类自己能访问
    private String title;

    // default（不写修饰符）：同一个包内的类能访问
    String internalNote;  // 内部备注，仅供 model 包内使用

    // protected：子类和同一个包内的类能访问
    protected String rawContent;  // 原始内容，子类可能需要处理

    // public：所有类都能访问
    public static final String STATUS_DRAFT = "DRAFT";

    // 构造器和 getter/setter 保持不变
    public Post() {
        this.status = STATUS_DRAFT;
        this.createdAt = java.time.LocalDateTime.now().toString();
    }

    // private 方法：外部完全看不到
    private String sanitizeTitle(String title) {
        return title.trim().replaceAll("<[^>]*>", "");
    }

    // default 方法：model 包内可见
    void logInternalState() {
        System.out.println("Post内部状态: title=" + title);
    }

    // protected 方法：子类可以重写
    protected void beforePublish() {
        this.slug = generateSlug(this.title);
    }
}
```

为什么 Post 需要这么多访问级别？每个级别都有真实的业务场景：

- `private` 的 `title` 和 `sanitizeTitle()` 是核心数据和行为，外部只能通过 `setTitle()` 间接修改，不能绕过校验。
- `default` 的 `internalNote` 是 model 包内的调试信息，其他包不需要知道。
- `protected` 的 `rawContent` 和 `beforePublish()` 是为未来的子类预留的——下一篇我们会创建 `Content` 基类，子类可能需要直接操作原始内容。
- `public static final` 的 `STATUS_DRAFT` 是常量，所有地方都能用。

### 跨包访问：谁行谁不行

写一个 `PostService` 类放在 `com.techblog.service` 包里，看看哪些成员能访问，哪些不能。

```java
package com.techblog.service;

import com.techblog.model.Post;

public class PostService {

    public void createDraft(String title) {
        Post post = new Post();

        post.setTitle(title);             // OK：public 方法
        post.logInternalState();          // 编译错误：default 方法，service 包看不到
        post.rawContent = "hello";        // 编译错误：protected 字段，service 包看不到
        // post.title = "hack";           // 编译错误：private 字段，只有 Post 自己能访问
        String status = Post.STATUS_DRAFT; // OK：public static final 常量
    }
}
```

编译器会直接拒绝最后三行。这不是"建议你最好别这么写"，是编译都过不了。在 JS 里，你要靠团队规范（"下划线开头的属性不要碰"）或者 TypeScript 的 `private`（运行时无效，编译后照常访问）来维持类似的效果。Java 的访问控制在编译期就是铁律。

protected 在不同包的子类中才能访问，普通类即使在其他包也不行。这个细节经常被忽略——很多人以为 protected 就是"包外也能访问"，但实际上只有通过继承才能跨包访问 protected 成员。

### JS 的 # 私有字段

JS 在 ES2022 引入了 `#` 私有字段，和 Java 的 `private` 在概念上接近，但实现方式完全不同。

```js
// JS 的 # 私有字段
class Post {
  #title; // 真正的私有字段，外部访问会报错
  #status = "DRAFT";

  constructor(title) {
    this.#title = title; // 类内部可以访问
  }

  get title() {
    return this.#title;
  }
}

const post = new Post("Hello");
post.#title; // SyntaxError: Private field '#title' must be declared
post["_title"]; // undefined，但不会报错——# 字段甚至不在这个对象上
```

Java 的 `private` 是编译期检查——字节码里私有字段还在，反射可以访问。JS 的 `#` 是语言级别的私有——字段根本不在实例上，用 `Object.keys()` 都看不到。Java 的方式更"软"，但配合模块系统已经够用；JS 的方式更"硬"，但来得太晚了，大部分 JS 代码还是靠 `_` 前缀的惯例。

| 特性          | Java private     | JS # (ES2022) |
| ------------- | ---------------- | ------------- |
| 何时引入      | Java 1.0 (1996)  | ES2022        |
| 保护机制      | 编译期检查       | 语法级私有    |
| 反射/动态访问 | 可以（反射 API） | 不可能        |
| 子类访问      | 不行             | 不行          |
| 社区使用率    | 几乎所有字段     | 仍在普及中    |

## package 与 ES Module

Java 的 package 和 JS 的 ES Module 都解决"代码组织"的问题，但设计哲学完全不同。

### Java 的 package：命名空间 + 目录结构

Java 的 package 首先是命名空间——两个类都叫 `Post`，只要在不同包里就不会冲突（`com.techblog.model.Post` vs `com.example.blog.Post`）。其次是目录结构——编译器和 JVM 都要求包名和目录路径对应。

```java
// 导入单个类
import com.techblog.model.Post;
import com.techblog.model.Tag;

// 或者导入整个包（不推荐，导入太多类会让代码意图不清）
import com.techblog.model.*;
```

`import` 只是告诉编译器"后面用到 `Post` 时，全限定名是 `com.techblog.model.Post`"。编译完成后，字节码里存的是全限定名，import 语句本身不会出现在 `.class` 文件中。也就是说，Java 的 import 是纯粹的编译期便利，不影响运行时行为。

### JS 的 ES Module：文件路径 + 依赖图

JS 的模块系统基于文件路径，`import` 语句在运行时（或打包时）构建依赖图。

```js
// JS：import 是文件路径
import { Post } from "./model/Post.js";
import { PostService } from "../service/PostService.js";

// 具名导出
export class Post {
  /* ... */
}

// 默认导出
export default class PostService {
  /* ... */
}
```

JS 模块的边界是文件——一个文件就是一个模块，你 export 什么，外面就能 import 什么。没有 package 这种额外的组织层级。项目大了之后，JS 社区用目录约定（`models/`、`services/`、`utils/`）来模拟 package 的效果，但这只是约定，不是语言规则。

### 核心差异

| 特性           | Java package              | JS ES Module                    |
| -------------- | ------------------------- | ------------------------------- |
| 组织单位       | 包（可含多个类）          | 文件（一个模块）                |
| 命名空间       | 全限定名唯一              | 文件路径唯一                    |
| 访问控制       | 四种修饰符 + 包边界       | export 的是公开的，其他是私有的 |
| 循环依赖       | 可以（但设计上要避免）    | 可以                            |
| 运行时         | import 仅编译期生效       | import 动态构建依赖图           |
| 一个文件多个类 | 可以（但只能一个 public） | 可以 export 多个                |

最关键的区别在第三行：Java 的访问控制是**成员级别**的（一个类里，public 方法和 private 方法可以混在一起），JS 的访问控制是**模块级别**的（export 了就是公开的，没 export 就拿不到）。在 JS 里，你不能做到"一个模块导出了一个类，但这个类里的某些方法是私有的"——你只能靠文档或 `_` 前缀来约定。

## static：属于类而非实例

到目前为止，Post 的所有字段和方法都绑定在具体实例上——每个 Post 对象有自己的 title、自己的 status。但有些东西不应该属于某个实例，而应该属于整个类。这就是 `static` 的作用。

### static 字段：文章计数器

假设 CMS 需要统计总共创建了多少篇 Post 对象。这个数字不属于任何单篇文章，而是属于 Post 类本身。

```java
package com.techblog.model;

public class Post {
    private Long id;
    private String title;
    private String slug;
    private String content;
    private String status;
    private long viewCount;
    private String createdAt;
    private String publishedAt;

    // static 字段：属于 Post 类，不属于某个 Post 实例
    private static long instanceCount = 0;

    // public static final：常量，所有类都能用
    public static final String STATUS_DRAFT = "DRAFT";
    public static final String STATUS_PUBLISHED = "PUBLISHED";
    public static final int MAX_TITLE_LENGTH = 200;

    public Post() {
        Post.instanceCount++;  // 注意：用类名访问 static 字段
        this.status = STATUS_DRAFT;
        this.createdAt = java.time.LocalDateTime.now().toString();
    }

    public Post(String title, String slug, String content) {
        this();
        setTitle(title);
        setSlug(slug);
        this.content = content;
    }

    // static 方法：不需要 Post 实例就能调用
    public static long getInstanceCount() {
        return instanceCount;
    }

    // static 方法不能访问 this，也不能访问实例字段
    // public static String getTitle() {
    //     return this.title;  // 编译错误：static 方法里没有 this
    // }

    // ... 其他 getter/setter 不变 ...
}
```

`instanceCount` 是 `static` 的，所以无论创建多少个 Post 对象，内存中只有一份 `instanceCount`。而 `title`、`status` 这些非 static 字段，每个 Post 对象各自有一份。调用方式也不同：

```java
Post post1 = new Post("第一篇", "first", "...");
Post post2 = new Post("第二篇", "second", "...");

// 通过类名访问 static 成员
System.out.println(Post.getInstanceCount());       // 2
System.out.println(Post.STATUS_DRAFT);             // "DRAFT"
System.out.println(Post.MAX_TITLE_LENGTH);         // 200

// 通过实例也能访问（但不推荐，编译器会警告）
System.out.println(post1.getInstanceCount());      // 2，但容易让人误以为跟 post1 有关
```

通过类名调用 static 方法是 Java 的惯例。虽然用实例引用也能调（`post1.getInstanceCount()`），但这会误导读者——让人以为结果跟 `post1` 有关。IDE 通常会对这种写法发出警告。

### static 方法：工具方法

工具方法不需要操作任何实例状态，只根据输入参数返回结果。上一篇的 `incrementViewCount()` 是实例方法（它操作 `this.viewCount`），但"生成 slug"这种操作跟具体实例无关，适合做成 static 方法。

```java
package com.techblog.util;

public class SlugGenerator {
    // 私有构造器：工具类不应该被实例化
    private SlugGenerator() {
        throw new UnsupportedOperationException("工具类不能实例化");
    }

    /**
     * 将标题转为 slug：小写、空格转连字符、移除特殊字符
     */
    public static String generate(String title) {
        if (title == null || title.isBlank()) {
            return "";
        }
        return title.toLowerCase()
                    .replaceAll("[^a-z0-9\\s-]", "")
                    .replaceAll("\\s+", "-")
                    .replaceAll("-+", "-")
                    .replaceAll("^-|-$", "");
    }
}
```

`SlugGenerator` 没有 static 字段，所有方法都是 static 的，构造器是 private 的。这种"纯工具类"在 Java 项目里非常常见——`java.util.Arrays`、`java.util.Collections`、`java.lang.Math` 都是这种模式。

JS 没有静态方法的概念。你通常会写一个导出纯函数的模块：

```js
// slugGenerator.js
export function generate(title) {
  if (!title) return "";
  return title
    .toLowerCase()
    .replace(/[^a-z0-9\s-]/g, "")
    .replace(/\s+/g, "-")
    .replace(/-+/g, "-")
    .replace(/^-|-$/g, "");
}

// 使用
import { generate } from "./slugGenerator.js";
const slug = generate("My First Post");
```

功能一样，但 JS 的方式是"模块即命名空间"——函数属于模块，而不是属于类。Java 的方式是"类即命名空间"——方法属于类。两种方式各有利弊，后面讲 trade-off 时会分析。

### static 代码块：类加载时执行

static 代码块在类第一次被 JVM 加载时执行一次，通常用于初始化 static 字段。

```java
package com.techblog.model;

import java.util.HashMap;
import java.util.Map;

public class Post {
    public static final String STATUS_DRAFT = "DRAFT";
    public static final String STATUS_PUBLISHED = "PUBLISHED";

    // 用 static 代码块初始化复杂的 static 字段
    private static final Map<String, String> STATUS_DISPLAY_NAMES;

    static {
        STATUS_DISPLAY_NAMES = new HashMap<>();
        STATUS_DISPLAY_NAMES.put(STATUS_DRAFT, "草稿");
        STATUS_DISPLAY_NAMES.put(STATUS_PUBLISHED, "已发布");

        System.out.println("Post 类已加载，初始化状态映射表");
    }

    public static String getStatusDisplayName(String status) {
        return STATUS_DISPLAY_NAMES.getOrDefault(status, "未知");
    }

    // ... 其他代码不变 ...
}
```

static 代码块只执行一次——在类第一次被使用时。如果你在同一个程序里创建了 100 个 Post 对象，static 代码块只会在创建第一个时执行。这和构造器不同，构造器每次 `new` 都会执行。

在 JS 里，模块顶层的代码在第一次 import 时执行，效果类似 static 代码块：

```js
// post.js
const STATUS_DISPLAY_NAMES = {
  DRAFT: "草稿",
  PUBLISHED: "已发布",
}; // 模块第一次被 import 时执行一次

export function getStatusDisplayName(status) {
  return STATUS_DISPLAY_NAMES[status] || "未知";
}
```

Java 的 static 代码块可以写多条，按声明顺序执行。但通常一条就够用了，写多了反而难维护。

## 用 CMS 完整串联

把上面的知识串起来，看看 `PostService` 怎么使用不同包中的类。

```java
package com.techblog.service;

import com.techblog.model.Post;
import com.techblog.model.Tag;
import com.techblog.util.SlugGenerator;

public class PostService {

    public Post createPost(String title, String content) {
        Post post = new Post();
        post.setTitle(title);
        post.setContent(content);
        // 用工具类的 static 方法生成 slug
        post.setSlug(SlugGenerator.generate(title));
        return post;
    }

    public void publishPost(Post post) {
        post.publish();
        System.out.println("文章已发布，当前总文章数: " + Post.getInstanceCount());
    }

    public static void main(String[] args) {
        PostService service = new PostService();

        Post p1 = service.createPost("Java 入门", "# Java 入门指南...");
        Post p2 = service.createPost("封装详解", "# 封装是 OOP 的基础...");

        service.publishPost(p1);
        System.out.println("标题: " + p1.getTitle());
        System.out.println("Slug: " + p1.getSlug());
        System.out.println("状态: " + p1.getStatus());

        // 跨包访问：只能用 public 成员
        // p1.internalNote = "测试";  // 编译错误：default，service 包看不到
        // p1.rawContent = "...";     // 编译错误：protected，service 包看不到
        System.out.println("状态常量: " + Post.STATUS_DRAFT);  // OK：public static final
    }
}
```

`PostService` 在 `com.techblog.service` 包，`Post` 和 `Tag` 在 `com.techblog.model` 包，`SlugGenerator` 在 `com.techblog.util` 包。三个包之间通过 `public` 方法交互，private/default/protected 成员被包边界挡住了。这就是 Java 访问控制的实际效果——不是靠文档约定，是编译器帮你守门。

## Trade-off：严格边界 vs 自由组合

Java 的封装体系比 JS 严格得多。四种访问修饰符 + 包边界 + static 分离，初学者觉得繁琐，但这种严格在项目规模增长后价值巨大。

**好处**。当你打开一个类，看到 `private` 字段就知道"这个数据只有这个类自己能改"，看到 `public` 方法就知道"这是给外部调用的接口"。团队协作时，你不需要阅读整个类的实现就能安全地使用它的 public API——编译器保证了你不会不小心碰到不该碰的东西。这比 JS 的"靠命名约定区分公开/私有"靠谱得多。

**代价**。写起来啰嗦。一个包含 8 个字段的 Post 类，光 getter/setter 就要写几十行。而且访问控制粒度太细时（default vs protected），你需要提前设计包结构，一旦包划分不合理，后面改起来很麻烦——牵一发动全身。JS 的模块系统更灵活，随时可以拆分文件、调整 import 路径，重构成本更低。

static 的使用也是一个需要权衡的点。static 字段在整个 JVM 进程中共享，多线程环境下容易出问题（第 18 篇会讲）。工具类全是 static 方法虽然方便，但不好 mock 测试（第 10 篇讲异常处理时会提到）。JS 的模块级变量虽然也有类似问题，但 JS 社区的函数式风格（纯函数 + 不可变数据）天然减少了对可变 static 状态的依赖。

## 回顾与预告

上一篇我们完成了 CMS 的基础类设计：Post 类有 8 个 private 字段、构造器重载、带校验的 getter/setter、以及 `publish()`/`isPublished()`/`getExcerpt()` 三个业务方法；Tag 类有 4 个 private 字段和格式校验。同时第一次感受到 Java class 编译后是独立字节码，与 JS 的原型链语法糖有本质区别。

本篇我们把这些类放进了包结构中，理解了 Java 四种访问修饰符的实际效果：private 保护核心数据、default 在包内共享内部实现、protected 为子类预留扩展点、public 定义外部接口。同时用 static 实现了文章计数器和工具方法，看到了"属于类"和"属于实例"的区别。

下一篇进入面向对象的第二个核心特性——继承。我们会创建一个 `Content` 基类，让 Post 继承它；理解 `extends` 和 JS 的 prototype 链有什么不同；搞清楚方法重写（Override）和重载（Overload）的区别。继承是把 Post 和未来的 Comment、Page 等类型联系起来的关键机制。
