CMS 系统里，Post 有 id、title、slug、createdAt、viewCount 这几个字段。未来加 Comment 的时候，Comment 也有 id、content、createdAt。再加个 Page（独立页面），同样需要 id、title、slug、createdAt。三份代码里，这几个字段和它们的 getter/setter 会重复写三遍。

这不是懒不懒的问题。当"创建时间"的格式要从 `String` 改成 `LocalDateTime` 的时候，你要改三个类。忘改一个就是一个 bug。

Java 给出的方案是继承：把共有字段抽到一个基类里，让 Post、Comment、Page 都从这个基类"继承"。JS 里也能做类似的事——通过原型链。但 Java 的 `extends` 和 JS 的 prototype 链，虽然都是"复用代码"的手段，机制和约束完全不同。搞清楚这些差异，是本篇的核心。

## 继承的本质：编译期契约 vs 运行时链

先用一段 JS 代码回顾原型链是怎么工作的：

```js
// JS：原型链继承
class Content {
  constructor(id, title) {
    this.id = id;
    this.title = title;
  }
  getExcerpt() {
    return this.title + "...";
  }
}

class Post extends Content {
  constructor(id, title, content) {
    super(id, title);
    this.content = content;
  }
}

const post = new Post(1, "Hello", "World");
// post.__proto__ === Post.prototype
// Post.prototype.__proto__ === Content.prototype
// 访问 post.id → 先找 post 自身 → 没有 → 找 Post.prototype → 没有 → 找 Content.prototype → 找到了
```

JS 的继承是运行时的链式查找。访问 `post.id` 时，引擎沿着 `__proto__` 一级一级往上找。你甚至可以在运行时修改原型链：

```js
Post.prototype.newField = "surprise";
console.log(post.newField); // "surprise" —— 运行时动态加的
```

Java 的 `extends` 是编译期的类型关系。编译器在编译阶段就精确知道 Post 继承了 Content 的哪些字段和方法。运行时不存在"链式查找"——JVM 直接知道 `post` 对象的内存布局里，哪些是 Content 的字段、哪些是 Post 自己的字段。你也无法在运行时给 Content 类"加个字段"然后让所有子类突然拥有它。

| 特性       | JavaScript                  | Java                                                 |
| ---------- | --------------------------- | ---------------------------------------------------- |
| 继承方式   | prototype 链                | extends 关键字                                       |
| 方法重写   | 直接覆盖 prototype 上的方法 | @Override + 方法签名必须匹配                         |
| 多态       | 原型链查找                  | 编译期类型 + 运行时动态绑定                          |
| super      | super.method() 调用父类方法 | super() 调用父类构造器 / super.method() 调用父类方法 |
| 多继承     | 混入（mixin）可以模拟       | 不支持，用接口代替                                   |
| 运行时修改 | 可以修改 prototype          | 不行，类型在编译期确定                               |

另一个关键区别：Java 只允许单继承（一个类只能 `extends` 一个父类），JS 的 prototype 也只有一条链（一个对象只有一个 `__proto__`），所以单继承这点上两者一样。但 JS 可以通过 mixin 或手动复制属性来模拟多继承，Java 则明确不支持——多继承的场景用接口（下一篇讲）来替代。

## 抽取 Content 基类

回到 CMS 的实际问题。Post 当前的 8 个字段里，有 5 个是所有内容类型共有的：

```
共有字段：id, title, slug, createdAt, viewCount
Post 特有：content, status, publishedAt, authorId
Comment 特有（未来）：content, postId, authorId, parentId
```

把这些共有字段抽到 Content 基类里：

```java
// Content.java —— 所有内容类型的基类
public class Content {
    private Long id;
    private String title;
    private String slug;
    private String createdAt;
    private long viewCount;

    public Content(Long id, String title, String slug) {
        this.id = id;
        this.title = title;
        this.slug = slug;
        this.createdAt = java.time.LocalDateTime.now().toString();
        this.viewCount = 0;
    }

    public Long getId() { return id; }
    public String getTitle() { return title; }
    public String getSlug() { return slug; }
    public String getCreatedAt() { return createdAt; }
    public long getViewCount() { return viewCount; }

    public void incrementViewCount() {
        this.viewCount++;
    }

    /**
     * 生成摘要：子类可以重写此方法提供自定义逻辑
     */
    public String getExcerpt() {
        return this.title;
    }
}
```

注意 `getExcerpt()` 方法。基类提供了一个默认实现——直接返回标题。子类如果有自己的摘要逻辑（比如 Post 要从正文中截取），可以重写这个方法。这就是"模板方法"的雏形：基类定义行为框架，子类填充具体细节。

Post 类现在继承 Content，只保留自己特有的字段：

```java
// Post.java
public class Post extends Content {
    private String content;
    private String status;
    private String publishedAt;
    private Long authorId;

    public Post(Long id, String title, String slug,
                String content, String status, Long authorId) {
        super(id, title, slug);  // 必须是第一条语句
        this.content = content;
        this.status = status;
        this.authorId = authorId;
    }

    // Post 特有的 getter/setter
    public String getContent() { return content; }
    public void setContent(String content) { this.content = content; }

    public String getStatus() { return status; }
    public void setStatus(String status) {
        if (!"DRAFT".equals(status) && !"PUBLISHED".equals(status)) {
            throw new IllegalArgumentException("状态只能是 DRAFT 或 PUBLISHED");
        }
        this.status = status;
    }

    public String getPublishedAt() { return publishedAt; }
    public Long getAuthorId() { return authorId; }
    public void setAuthorId(Long authorId) { this.authorId = authorId; }

    public void publish() {
        if ("PUBLISHED".equals(this.status)) {
            throw new IllegalStateException("文章已经发布过了");
        }
        this.status = "PUBLISHED";
        this.publishedAt = java.time.LocalDateTime.now().toString();
    }

    public boolean isPublished() {
        return "PUBLISHED".equals(this.status);
    }

    // 重写基类的 getExcerpt()，提供 Post 专属的摘要逻辑
    @Override
    public String getExcerpt() {
        if (this.content == null || this.content.length() <= 100) {
            return this.content;
        }
        return this.content.substring(0, 100) + "...";
    }
}
```

几个要点需要解释。

**super() 必须是第一条语句**。子类构造器中调用 `super()` 传递共有字段给 Content 的构造器，这条语句必须是子类构造器的第一行。这不是建议，是编译器强制要求的——因为父类必须先完成初始化，子类才能使用父类的字段。如果父类没有无参构造器（Content 确实没有），编译器不会自动帮你加 `super()`，你必须显式调用，否则编译失败。

**@Override 注解**。`getExcerpt()` 上的 `@Override` 告诉编译器："我要重写父类的这个方法，请帮我检查签名对不对。"如果把方法名拼成 `getExcErpt()`，没有 `@Override` 的话编译器不会报错——它以为你在定义一个全新的方法。有了 `@Override`，编译器立刻告诉你"父类没有这个方法"，在编译期就拦住 bug。

**字段可见性**。Content 的字段是 `private` 的。Post 继承 Content 后，能继承的是 Content 的 public/protected 方法和 public/protected 字段，而不是 private 字段。Post 通过 `getTitle()`、`getId()` 这些继承来的 getter 访问父类字段，不能直接写 `this.title`（除非 Content 里把 title 改成 protected）。这是上一篇讲封装时提到的访问控制——private 字段只在定义它的类内部可见，子类也不行。

## 方法重写 vs 方法重载

继承带来了两个容易混淆的概念：重写（override）和重载（overload）。它们的中文名很像，但完全是两回事。

**重写**是子类重新定义父类的方法，方法签名完全相同。上面 Post 的 `getExcerpt()` 就是重写了 Content 的 `getExcerpt()`。

**重载**是同一个类中，方法名相同但参数列表不同。上一篇讲过的构造器重载就是重载——多个构造器，名字都叫 Post，参数不同。

```java
public class Post extends Content {
    // 重写（override）：方法签名和父类 getExcerpt() 完全一致
    @Override
    public String getExcerpt() {
        if (this.content == null || this.content.length() <= 100) {
            return this.content;
        }
        return this.content.substring(0, 100) + "...";
    }

    // 重载（overload）：方法名相同，参数列表不同
    public String getExcerpt(int maxLength) {
        if (this.content == null || this.content.length() <= maxLength) {
            return this.content;
        }
        return this.content.substring(0, maxLength) + "...";
    }
}
```

调用 `post.getExcerpt()` 走重写版本，调用 `post.getExcerpt(50)` 走重载版本。编译器根据参数类型和数量决定调用哪个重载方法（编译期绑定），根据实际对象的类型决定执行哪个重写方法（运行时绑定）。

JS 里没有重载的概念——同名方法直接覆盖。想模拟重载只能在方法体里判断参数类型和数量，这也是 TypeScript 需要用函数签名重载来弥补的。

## 多态：父类引用指向子类对象

多态这个词听起来很学术，但核心就一句话：用父类的类型来引用子类的对象，调用方法时执行的是子类的版本。

```java
// 用 Content 类型引用 Post 对象
Content c = new Post(1L, "Java 入门", "java-intro",
                     "# Java 入门\n\n这是正文...", "DRAFT", 100L);

// 编译时，c 的类型是 Content，所以只能调用 Content 声明的方法
String title = c.getTitle();     // OK，Content 有这个方法
long views = c.getViewCount();   // OK
// c.getContent();               // 编译错误，Content 没有 getContent()
// c.publish();                  // 编译错误，Content 没有 publish()

// 运行时，c 实际指向 Post 对象，所以 getExcerpt() 执行 Post 的版本
String excerpt = c.getExcerpt();
// 输出的是从正文截取的摘要，不是基类的"直接返回标题"
```

这就是动态绑定：编译器看声明的类型（Content）决定哪些方法可以调用，JVM 在运行时看实际对象的类型（Post）决定执行哪个版本的方法实现。

多态的实际价值在于统一处理。CMS 里要统计所有内容的总浏览量，不管它是 Post 还是 Comment：

```java
public class ContentStats {
    public static long totalViewCount(Content[] contents) {
        long total = 0;
        for (Content c : contents) {
            total += c.getViewCount();
        }
        return total;
    }

    public static void main(String[] args) {
        Content[] contents = {
            new Post(1L, "Java 入门", "java-intro",
                     "# Java\n\n正文...", "PUBLISHED", 100L),
            new Post(2L, "Spring Boot", "spring-boot",
                     "# Spring\n\n正文...", "PUBLISHED", 100L),
            // 未来：new Comment(3L, "写得好！", 2L, 200L)
        };

        // 给第二篇文章加浏览量
        // 需要先转型才能调用 Post 特有的方法
        if (contents[1] instanceof Post) {
            Post p = (Post) contents[1];
            p.publish();       // Post 特有方法
            p.incrementViewCount();  // 继承自 Content 的方法
        }

        // 统一统计总浏览量——不需要关心每个元素具体是什么类型
        System.out.println("总浏览量: " + totalViewCount(contents));
    }
}
```

`totalViewCount` 方法接收 `Content[]` 参数。不管数组里放的是 Post、Comment 还是未来的 Page，都能统一处理，因为它们都继承自 Content，都有 `getViewCount()` 方法。不需要为每种类型写一个统计方法，也不需要在循环里用 `if-else` 判断类型。

当确实需要调用子类特有的方法时（比如 `publish()`），用 `instanceof` 检查类型后强制转型。`instanceof` 在 Java 16+ 可以配合模式匹配简化写法：

```java
// Java 16+ 模式匹配
if (contents[1] instanceof Post p) {
    p.publish();  // 直接使用 p，不需要手动转型
}
```

## instanceof 与类型安全

向上转型（子类赋给父类引用）是安全的，不需要任何特殊语法：

```java
Content c = new Post(...);  // 自动向上转型
```

向下转型（父类引用转回子类）可能失败，因为父类引用实际指向的不一定是你期望的子类：

```java
Content c = new Post(...);
Comment comment = (Comment) c;  // 运行时抛出 ClassCastException
```

这就是 `instanceof` 存在的意义——转型之前先检查，避免运行时崩溃。在 CMS 项目里，当你从数据库查出一条记录，类型是 Content，但你需要根据具体类型执行不同操作时，`instanceof` 就是你的安全网。

JS 里没有这种严格的类型检查。你可以随便把一个对象当任何类型用，访问不存在的方法也只是在运行时返回 `undefined`（或者抛 TypeError）。Java 的转型机制强制你在编译期就考虑"这个对象到底是什么类型"，把错误提前暴露。

## Trade-off：继承的收益与代价

继承解决了代码重复的问题，也带来了多态的能力——用统一的接口处理不同类型。在 CMS 里，`Content[]` 数组能存放所有内容类型，`totalViewCount()` 不需要关心具体类型，这就是多态的威力。

但继承的代价也不小。首先是耦合：Post 的行为依赖于 Content 的实现细节。如果 Content 的构造器签名变了，所有子类都要改。其次是脆弱的基类问题——修改基类可能意外影响所有子类，尤其是当子类重写方法时依赖了基类的某些隐含假设。最后是 Java 的单继承限制——Post 只能 extends 一个类。如果未来 Post 既需要继承 Content，又需要复用另一个类的逻辑，extends 做不到。

正因为这些代价，Java 社区有一条经验法则：优先用组合（composition），谨慎用继承。当你犹豫"该继承还是该组合"的时候，问自己一个问题：B 是不是 A 的一种（is-a 关系）？Post 是 Content 的一种——是的，这个继承关系合理。但如果 Post 要"复用" Tag 的 slug 校验逻辑，不应该让 Post 继承 Tag，而是把校验逻辑抽成一个工具方法。

关于"多继承做不到怎么办"——Java 的答案是接口。一个类只能 extends 一个父类，但可以 implements 多个接口。Tag 类将来可以 implements Taggable 接口，Post 也可以 implements Taggable。下一篇就讲这个。

## 回顾与预告

上一篇我们学了封装与包，搞清楚了 Java 的四种访问修饰符、package 作为命名空间的作用、以及 static 成员属于类而非实例。本篇用上了这些知识：Content 的字段是 private 的，子类通过继承来的 getter 访问；`incrementViewCount()` 是 public 方法，所有子类共享这个行为。

本篇我们从 CMS 中抽取了 Content 基类，让 Post 继承它，理解了 Java 的 `extends` 是编译期的类型关系、`super()` 调用父类构造器的规则、`@Override` 的编译期检查、以及多态通过"父类引用指向子类对象"实现统一处理的机制。同时看到了 Java 的单继承限制。

下一篇进入接口与抽象类——TS 的 interface 和 Java 的 interface 看着像，实际差异很大。我们会给 CMS 加上 Taggable 接口，让 Post 和 Tag 都能被统一管理，同时理解为什么 Java 需要接口来弥补单继承的不足。
