前六篇我们把 Drizzle 的核心能力过了一遍——Schema 建模、CRUD、联表、事务迁移，还接上了 tRPC 做端到端类型安全。但这些都还停留在"库"的层面，你总得有个框架来承载它们。这一篇我们专注 Next.js App Router，因为它是 Drizzle 最常见的搭档，也是踩坑最多的地方。

我会用一个博客系统贯穿全文：首页加载文章列表（Server Component）、文章详情页带评论（关联查询）、评论提交（Server Action）、管理后台的 CRUD。过程中你会看到连接管理的各种坑——热重载泄漏连接、Vercel 冷启动耗尽连接池——以及怎么逐一解决。

## 博客系统的 Schema 回顾

后面所有例子都基于这组表，如果你跟过前面的文章应该很熟悉了。这里快速列一下，确保我们在同一页上。

```typescript
// src/lib/db/schema.ts
import {
  pgTable,
  serial,
  varchar,
  text,
  integer,
  timestamp,
  boolean,
} from "drizzle-orm/pg-core";

export const users = pgTable("users", {
  id: serial("id").primaryKey(),
  email: varchar("email", { length: 255 }).notNull().unique(),
  name: text("name"),
  createdAt: timestamp("created_at").defaultNow(),
});

export const posts = pgTable("posts", {
  id: serial("id").primaryKey(),
  title: varchar("title", { length: 255 }).notNull(),
  slug: varchar("slug", { length: 255 }).notNull().unique(),
  content: text("content").notNull(),
  excerpt: text("excerpt"),
  published: boolean("published").default(false),
  authorId: integer("author_id")
    .references(() => users.id, { onDelete: "cascade" })
    .notNull(),
  createdAt: timestamp("created_at").defaultNow(),
  updatedAt: timestamp("updated_at").defaultNow(),
});

export const comments = pgTable("comments", {
  id: serial("id").primaryKey(),
  content: text("content").notNull(),
  postId: integer("post_id")
    .references(() => posts.id, { onDelete: "cascade" })
    .notNull(),
  authorId: integer("author_id")
    .references(() => users.id, { onDelete: "cascade" })
    .notNull(),
  createdAt: timestamp("created_at").defaultNow(),
});

export const tags = pgTable("tags", {
  id: serial("id").primaryKey(),
  name: varchar("name", { length: 100 }).notNull().unique(),
});

export const postToTags = pgTable("post_to_tags", {
  postId: integer("post_id")
    .references(() => posts.id, { onDelete: "cascade" })
    .notNull(),
  tagId: integer("tag_id")
    .references(() => tags.id, { onDelete: "cascade" })
    .notNull(),
});
```

## 连接管理：第一个要搞对的事

很多人上来就写查询，结果开发环境跑着跑着就报 "too many connections"。原因很简单——Next.js 的热重载（Fast Refresh）会反复重新执行模块级别的代码。如果你在模块顶层 `new postgres()`，每次热重载都会创建一个新的连接池，旧的不会被释放。

### 开发环境：单例保护

```typescript
// src/lib/db/index.ts
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema";

function getConnectionString() {
  const url = process.env.DATABASE_URL;
  if (!url) throw new Error("DATABASE_URL is not set");
  return url;
}

// 开发环境用 globalThis 保证单例
const globalForDb = globalThis as unknown as {
  client: postgres.Sql<{}> | undefined;
};

const client =
  globalForDb.client ??
  postgres(getConnectionString(), {
    max: 1, // 开发环境只开一个连接，方便观察问题
    idle_timeout: 20,
    connect_timeout: 10,
  });

if (process.env.NODE_ENV !== "production") {
  globalForDb.client = client;
}

export const db = drizzle(client, { schema });
```

关键点是 `globalForDb` 这层缓存。Next.js 在开发模式下会对页面做 Fast Refresh，但 `globalThis` 不会被重置，所以连接池只创建一次。生产环境不需要这层——每次冷启动都是全新进程，模块级变量天然是单例。

### 生产环境：连接池参数调优

```typescript
// 生产环境直接用模块级变量，不需要 globalThis
const client = postgres(getConnectionString(), {
  max: 10, // 最大连接数
  idle_timeout: 20, // 空闲 20 秒释放
  connect_timeout: 10, // 连接超时 10 秒
  max_lifetime: 60 * 30, // 连接最多活 30 分钟
});
```

如果你部署在 Vercel 上，有个更隐蔽的坑：Serverless 函数是按需创建实例的，每个实例都会建立自己的连接池。如果你的 `max` 设成 10，同时有 5 个函数实例在跑，数据库那边就是 50 个连接。对于 Postgres 默认 100 个连接上限的免费套餐来说，很容易打满。

解决方案有两个：

- 用连接池代理（比如 Supabase 的 PgBouncer、Neon 的 pooler），把 `max` 调小到 2-3
- 或者换成 HTTP 驱动（如 `@neondatabase/serverless`），走 HTTP 协议而不是 TCP，根本不存在连接池的概念

```typescript
// Vercel + Neon 的推荐方式：HTTP 驱动，无连接池问题
import { neon } from "@neondatabase/serverless";
import { drizzle } from "drizzle-orm/neon-http";

const sql = neon(process.env.DATABASE_URL!);
export const db = drizzle(sql, { schema });
```

### 环境变量：启动时校验

别等到查询失败才发现 `DATABASE_URL` 没配。用 Zod 在启动时校验：

```typescript
// src/lib/env.ts
import { z } from "zod";

const envSchema = z.object({
  DATABASE_URL: z.string().min(1),
  NODE_ENV: z
    .enum(["development", "production", "test"])
    .default("development"),
});

export const env = envSchema.parse(process.env);
```

然后在 `db/index.ts` 里用 `env.DATABASE_URL` 替代 `process.env.DATABASE_URL`。如果缺了，应用直接启动失败，比运行时报错好排查得多。

## Server Components：在组件里直接查数据库

App Router 最大的变化是组件可以在服务端直接跑 async 函数。这意味着你不用写 API 路由，不用序列化/反序列化，直接在组件里 `await db.select()` 就行。

### 首页：文章列表

```typescript
// src/app/page.tsx
import { db } from "@/lib/db";
import { posts, users } from "@/lib/db/schema";
import { eq, desc } from "drizzle-orm";
import Link from "next/link";

export const revalidate = 60; // ISR：每 60 秒重新生成

export default async function HomePage() {
  const recentPosts = await db
    .select({
      id: posts.id,
      title: posts.title,
      slug: posts.slug,
      excerpt: posts.excerpt,
      createdAt: posts.createdAt,
      authorName: users.name,
    })
    .from(posts)
    .innerJoin(users, eq(posts.authorId, users.id))
    .where(eq(posts.published, true))
    .orderBy(desc(posts.createdAt))
    .limit(10);

  return (
    <main>
      <h1>最新文章</h1>
      <ul>
        {recentPosts.map((post) => (
          <li key={post.id}>
            <Link href={`/posts/${post.slug}`}>
              <h2>{post.title}</h2>
              <p>{post.excerpt}</p>
              <span>{post.authorName} / {post.createdAt?.toLocaleDateString()}</span>
            </Link>
          </li>
        ))}
      </ul>
    </main>
  );
}
```

这里 `revalidate = 60` 让页面做 ISR（Incremental Static Regeneration）。用户第一次访问时渲染并缓存，之后的 60 秒内都返回缓存，60 秒后的下一次访问触发后台重新渲染。

注意我们只 select 了需要的字段，而不是 `select()` 全部拿。这在字段多、有 text 大字段时差别明显。

### 文章详情页：关联查询 + 评论

```typescript
// src/app/posts/[slug]/page.tsx
import { db } from "@/lib/db";
import { posts, users, comments, tags, postToTags } from "@/lib/db/schema";
import { eq } from "drizzle-orm";
import { notFound } from "next/navigation";
import { CommentForm } from "./comment-form";

export default async function PostPage({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;

  // 查文章 + 作者
  const [post] = await db
    .select({
      id: posts.id,
      title: posts.title,
      content: posts.content,
      createdAt: posts.createdAt,
      authorId: users.id,
      authorName: users.name,
      authorEmail: users.email,
    })
    .from(posts)
    .innerJoin(users, eq(posts.authorId, users.id))
    .where(eq(posts.slug, slug))
    .limit(1);

  if (!post) notFound();

  // 查标签
  const postTags = await db
    .select({ name: tags.name })
    .from(postToTags)
    .innerJoin(tags, eq(postToTags.tagId, tags.id))
    .where(eq(postToTags.postId, post.id));

  // 查评论（含评论者信息）
  const postComments = await db
    .select({
      id: comments.id,
      content: comments.content,
      createdAt: comments.createdAt,
      authorName: users.name,
    })
    .from(comments)
    .innerJoin(users, eq(comments.authorId, users.id))
    .where(eq(comments.postId, post.id))
    .orderBy(comments.createdAt);

  return (
    <article>
      <h1>{post.title}</h1>
      <p>By {post.authorName} / {post.createdAt?.toLocaleDateString()}</p>
      <div>{postTags.map((t) => t.name).join(", ")}</div>
      <div>{post.content}</div>

      <section>
        <h2>评论 ({postComments.length})</h2>
        {postComments.map((c) => (
          <div key={c.id}>
            <strong>{c.authorName}</strong>
            <p>{c.content}</p>
          </div>
        ))}
      </section>

      <CommentForm postId={post.id} />
    </article>
  );
}
```

这里发了三条查询。你可能会问为什么不用一个大的 join 把文章、标签、评论全查出来？因为那样会产生笛卡尔积——一篇有 10 条评论和 5 个标签的文章，join 结果是 50 行。分开查反而更清晰，类型推断也更干净。

如果你觉得三次 await 有性能焦虑，可以用 `Promise.all` 并行化：

```typescript
const [postTags, postComments] = await Promise.all([
  db
    .select({ name: tags.name })
    .from(postToTags)
    .innerJoin(tags, eq(postToTags.tagId, tags.id))
    .where(eq(postToTags.postId, post.id)),
  db
    .select({
      id: comments.id,
      content: comments.content,
      createdAt: comments.createdAt,
      authorName: users.name,
    })
    .from(comments)
    .innerJoin(users, eq(comments.authorId, users.id))
    .where(eq(comments.postId, post.id))
    .orderBy(comments.createdAt),
]);
```

## Server Actions：在服务端写数据

Server Actions 是 App Router 里处理表单提交的推荐方式。它在服务端执行，不需要手动写 API 路由。

### 提交评论

```typescript
// src/app/posts/[slug]/actions.ts
"use server";

import { db } from "@/lib/db";
import { comments } from "@/lib/db/schema";
import { revalidatePath } from "next/cache";
import { z } from "zod";

const commentSchema = z.object({
  content: z.string().min(1, "评论不能为空").max(2000, "评论不超过 2000 字"),
  postId: z.number().int().positive(),
  authorId: z.number().int().positive(),
});

export async function submitComment(formData: FormData) {
  const raw = {
    content: formData.get("content") as string,
    postId: Number(formData.get("postId")),
    authorId: Number(formData.get("authorId")),
  };

  const parsed = commentSchema.safeParse(raw);
  if (!parsed.success) {
    return { error: parsed.error.flatten().fieldErrors };
  }

  await db.insert(comments).values(parsed.data);

  // 刷新文章详情页的缓存，让新评论立即可见
  revalidatePath(`/posts/${formData.get("slug")}`);

  return { success: true };
}
```

`revalidatePath` 是这里的关键。它告诉 Next.js："这个路径的缓存作废了，下次访问时重新渲染"。没有它，ISR 的缓存页面上看不到新评论。

### 前端表单

```typescript
// src/app/posts/[slug]/comment-form.tsx
"use client";

import { useActionState } from "react";
import { submitComment } from "./actions";

export function CommentForm({ postId }: { postId: number }) {
  const [state, formAction, isPending] = useActionState(submitComment, null);

  return (
    <form action={formAction}>
      <input type="hidden" name="postId" value={postId} />
      <input type="hidden" name="authorId" value={1} />
      <textarea name="content" required placeholder="写下你的评论..." />
      {state?.error && <p>{JSON.stringify(state.error)}</p>}
      {state?.success && <p>评论已发布</p>}
      <button type="submit" disabled={isPending}>
        {isPending ? "提交中..." : "发表评论"}
      </button>
    </form>
  );
}
```

`useActionState` 是 React 19 的 API，用来管理 Server Action 的返回值和 pending 状态。比之前 `useFormState` 命名更清晰。

### 管理后台：完整的 CRUD

管理后台的增删改查比较常规，但有几个细节值得注意。

```typescript
// src/app/admin/actions.ts
"use server";

import { db } from "@/lib/db";
import { posts, postToTags, tags } from "@/lib/db/schema";
import { eq } from "drizzle-orm";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";

export async function createPost(formData: FormData) {
  const title = formData.get("title") as string;
  const content = formData.get("content") as string;
  const tagNames = (formData.get("tags") as string)
    .split(",")
    .map((t) => t.trim())
    .filter(Boolean);

  if (!title || !content) {
    return { error: "标题和内容不能为空" };
  }

  const slug = title
    .toLowerCase()
    .replace(/\s+/g, "-")
    .replace(/[^\w-]/g, "");

  // 用事务保证一致性
  const result = await db.transaction(async (tx) => {
    const [post] = await tx
      .insert(posts)
      .values({ title, content, slug, authorId: 1, published: true })
      .returning();

    // 处理标签：已存在的复用，不存在的创建
    if (tagNames.length > 0) {
      for (const tagName of tagNames) {
        const [tag] = await tx
          .insert(tags)
          .values({ name: tagName })
          .onConflictDoNothing()
          .returning();

        const existingTag =
          tag ??
          (await tx
            .select()
            .from(tags)
            .where(eq(tags.name, tagName))
            .limit(1)
            .then((r) => r[0]));

        if (existingTag) {
          await tx
            .insert(postToTags)
            .values({ postId: post.id, tagId: existingTag.id });
        }
      }
    }

    return post;
  });

  revalidatePath("/");
  revalidatePath("/admin");
  redirect(`/posts/${result.slug}`);
}

export async function deletePost(formData: FormData) {
  const id = Number(formData.get("id"));
  await db.delete(posts).where(eq(posts.id, id));
  revalidatePath("/");
  revalidatePath("/admin");
  redirect("/admin");
}
```

标签处理那个循环看起来有点啰嗦，但它解决了一个真实问题：标签名要唯一，用户输入的标签可能已经存在。`onConflictDoNothing` + 回查是 Drizzle 处理 "insert if not exists" 的标准写法。如果你用 Postgres 15+，也可以用 `ON CONFLICT ... DO UPDATE` 配合 CTE 来优化成单条语句，但大多数场景下这个循环性能够用。

## ISR 和动态渲染：什么时候用哪个

App Router 里页面的渲染策略由你导出的变量和函数决定：

| 导出                                     | 行为                       | 适用场景       |
| ---------------------------------------- | -------------------------- | -------------- |
| 什么都不导出                             | 默认缓存（类似 SSG）       | 静态页面       |
| `export const revalidate = 60`           | ISR，每 N 秒重新验证       | 首页、列表页   |
| `export const dynamic = "force-dynamic"` | 每次请求都渲染（类似 SSR） | 实时数据页面   |
| `cookies()` / `headers()` 调用           | 自动 opt-in 动态渲染       | 需要认证的页面 |

博客首页适合 ISR——文章不会秒秒变，60 秒的延迟可以接受。管理后台适合动态渲染——作者需要看到最新状态。文章详情页看情况：如果评论量大且要求实时，用动态渲染；否则 ISR 就够了。

```typescript
// 管理后台：强制动态渲染
export const dynamic = "force-dynamic";

export default async function AdminPage() {
  const allPosts = await db.select().from(posts).orderBy(desc(posts.createdAt));
  // ...
}
```

## 和传统 API Routes 的对比

你可能会问：有了 Server Components 和 Server Actions，还需要 API Routes 吗？

答案是"看场景"。

Server Components 适合页面级的数据加载——组件渲染时自动获取数据，不需要额外的 API 端点。Server Actions 适合表单提交和简单的数据变更。

但如果你需要给移动端 App 提供接口、或者要做 Webhook 接收、或者需要第三方 OAuth 回调，API Routes 还是不可替代的。Drizzle 在 API Routes 里用起来和普通 Node.js 服务没区别：

```typescript
// src/app/api/posts/route.ts
import { NextResponse } from "next/server";
import { db } from "@/lib/db";
import { posts, users } from "@/lib/db/schema";
import { eq, desc } from "drizzle-orm";

export async function GET() {
  const result = await db
    .select({
      id: posts.id,
      title: posts.title,
      slug: posts.slug,
      authorName: users.name,
      createdAt: posts.createdAt,
    })
    .from(posts)
    .innerJoin(users, eq(posts.authorId, users.id))
    .orderBy(desc(posts.createdAt))
    .limit(20);

  return NextResponse.json(result);
}
```

简单说：页面展示优先用 Server Components，数据变更优先用 Server Actions，对外接口用 API Routes。不要混着用——同一个数据流不要一半走 Server Component 一半走 `fetch('/api/...')`，那样既多了一次网络往返，又丢失了类型推断。

## Trade-off 分析

**Server Components + Drizzle 的优势：**

- 零 API 样板代码，组件里直接 `await`，开发效率极高
- 类型从数据库到组件全程推断，改一个字段编译器替你报错
- ISR 做页面级缓存非常方便，一个 `revalidate = 60` 搞定
- 不需要额外的数据获取层（React Query、SWR 等可以省掉）

**代价和限制：**

- 和 Next.js 耦合深——Server Components 和 Server Actions 都是 Next.js 的 API，换框架要重写
- 数据获取逻辑和 UI 组件混在一起，时间久了容易变成"上帝组件"，需要主动拆分（把查询逻辑抽到单独的 data 层函数里）
- Server Actions 的错误处理比较原始——没有标准化的错误码机制，也没有中间件层，复杂场景不如 tRPC 灵活
- `revalidatePath` 是全页面级别的失效，做不到"只刷新列表里某一条"这种精细控制

**我的建议：**

如果你的项目就是 Next.js 全栈应用（不是给移动端提供 API），Server Components + Server Actions + Drizzle 是最高效的组合。但如果你需要服务多种客户端（Web + App + 小程序），还是在前一篇的 tRPC 架构上走更稳妥。

下一篇我们聊一个更激进的部署方案——把整个博客系统搬到 Edge 上。你的用户遍布全球，Vercel 的 Serverless 函数跑在固定区域，亚洲用户访问美区的 API 要多几百毫秒。Cloudflare Workers 有 300+ 个全球节点，而 Drizzle 是少数在 Edge 上跑得好的 ORM。下一篇见。
