前几篇我们把博客系统的查询功能都搞定了——CRUD、JOIN、关系查询，读数据已经没什么问题。但有一个坑一直没提：**发布一篇文章，要同时往 `posts` 表插入文章、往 `post_to_tags` 表插入标签关联，如果标签不存在还要先创建标签。** 这三步操作必须要么全部成功、要么全部回滚，否则就会出现"有文章但标签丢了"的脏数据。

还有一个问题：Schema 改了之后，怎么把变更同步到数据库？开发环境和生产环境的表结构怎么保持一致？手动写 SQL 迁移脚本？那太原始了。

今天这两件事一起搞定。

## 事务：多步操作的原子化保障

### 从一个真实的 Bug 说起

假设我们写了一个发布文章的函数，没加事务：

```typescript
async function publishPost(data: {
  title: string;
  content: string;
  authorId: number;
  tagNames: string[];
}) {
  // 第一步：创建文章
  const [post] = await db
    .insert(posts)
    .values({
      title: data.title,
      content: data.content,
      authorId: data.authorId,
      slug: generateSlug(data.title),
    })
    .returning();

  // 第二步：为每个标签创建关联
  for (const tagName of data.tagNames) {
    const [tag] = await db
      .insert(tags)
      .values({ name: tagName, slug: generateSlug(tagName) })
      .onConflictDoNothing()
      .returning();

    const existingTag =
      tag ??
      (await db.select().from(tags).where(eq(tags.name, tagName)).limit(1))[0];

    await db.insert(postsToTags).values({
      postId: post.id,
      tagId: existingTag.id,
    });
  }

  return post;
}
```

如果第二步中间某个 `insert` 抄错了字段名，抛了异常怎么办？文章已经写进去了，但标签关联没完成。数据库里就多了一篇"裸奔"的文章。更糟糕的是，你可能完全不知道这件事——因为文章确实创建了，接口没报错，只是标签悄悄丢了。

这就是事务要解决的问题。

### db.transaction() 的基本用法

Drizzle 的事务 API 很简洁——`db.transaction()` 接收一个回调，回调里的所有操作要么全部提交，要么全部回滚：

```typescript
await db.transaction(async (tx) => {
  // tx 和 db 拥有完全一样的查询 API
  // 区别在于：tx 里的操作都在同一个事务中

  const [post] = await tx
    .insert(posts)
    .values({ title: "Drizzle 事务指南", content: "...", authorId: 1 })
    .returning();

  // 如果下面这行报错，上面的 insert 也会被回滚
  await tx.insert(postsToTags).values({ postId: post.id, tagId: 1 });
});
```

在回调里**抛出任何错误**（包括你自己 throw 的），整个事务都会回滚。回调正常结束，事务自动提交。你不需要手动调用 commit 或 rollback。

### 用事务重写发布文章

```typescript
async function publishPost(data: {
  title: string;
  content: string;
  authorId: number;
  tagNames: string[];
}) {
  return await db.transaction(async (tx) => {
    const [post] = await tx
      .insert(posts)
      .values({
        title: data.title,
        content: data.content,
        authorId: data.authorId,
        slug: generateSlug(data.title),
      })
      .returning();

    for (const tagName of data.tagNames) {
      // 标签不存在就创建，存在就复用
      const [tag] = await tx
        .insert(tags)
        .values({ name: tagName, slug: generateSlug(tagName) })
        .onConflictDoNothing()
        .returning();

      const actualTag =
        tag ??
        (
          await tx.select().from(tags).where(eq(tags.name, tagName)).limit(1)
        )[0];

      await tx.insert(postsToTags).values({
        postId: post.id,
        tagId: actualTag.id,
      });
    }

    return post;
  });
}
```

现在，如果循环中任何一步出错——比如标签名违反了唯一约束，或者关联表插入失败——文章的插入也会被回滚。数据库始终保持干净的状态。

### 主动回滚：用 throw 做业务校验

有时候不是程序出错，而是业务规则不允许继续。比如一个用户最多发布 100 篇文章：

```typescript
await db.transaction(async (tx) => {
  const [{ count: postCount }] = await tx
    .select({ count: sql<number>`count(*)` })
    .from(posts)
    .where(eq(posts.authorId, userId));

  if (Number(postCount) >= 100) {
    throw new Error("每个用户最多发布 100 篇文章");
  }

  // ... 继续插入
});
```

throw 出去的错误会被 Drizzle 捕获，触发回滚，然后重新抛出。你在外层 catch 它就行。

### 嵌套事务和保存点

PostgreSQL 支持 SAVEPOINT，Drizzle 也对应提供了嵌套事务的能力。这在某些场景下很有用——比如批量操作中，你希望某一条失败不影响其他条，但又不想丢失整批操作：

```typescript
await db.transaction(async (tx) => {
  // 外层事务：创建文章
  const [post] = await tx
    .insert(posts)
    .values({ title: "批量导入", content: "...", authorId: 1 })
    .returning();

  // 批量导入评论，单条失败不影响文章创建
  for (const comment of importedComments) {
    try {
      // 嵌套事务（自动创建 SAVEPOINT）
      await tx.transaction(async (innerTx) => {
        await innerTx.insert(comments).values({
          postId: post.id,
          content: comment.content,
          authorId: comment.authorId,
        });
      });
    } catch {
      console.warn(`跳过无效评论: ${comment.content.slice(0, 20)}...`);
      // 这条评论失败了，但外层事务不会回滚
    }
  }
});
```

内层 `tx.transaction()` 会创建一个 SAVEPOINT。内层失败只回滚到 SAVEPOINT，不影响外层。这个模式在数据导入、批量处理时特别好用。

### 隔离级别和行级锁

如果你有并发写入的场景（比如文章浏览计数、库存扣减），需要关注隔离级别。Drizzle 支持通过选项设置：

```typescript
await db.transaction(
  async (tx) => {
    // 读取并锁定文章行
    const [post] = await tx
      .select()
      .from(posts)
      .where(eq(posts.id, postId))
      .for("update");

    await tx
      .update(posts)
      .set({ viewCount: sql`${posts.viewCount} + 1` })
      .where(eq(posts.id, postId));
  },
  { isolationLevel: "serializable" }
);
```

`for("update")` 会给选中的行加排他锁，其他事务不能同时修改这行。PostgreSQL 默认的隔离级别是 READ COMMITTED，大多数场景够用。只有在涉及"读后写"的并发竞争时，才需要考虑更高的隔离级别或行级锁。

## 数据库迁移：从开发到生产

事务解决了运行时的一致性问题。但还有一类问题发生在开发阶段——你修改了 Schema（加了个字段、改了个类型），怎么把变更同步到数据库？

### 三条命令的工作流

Drizzle Kit 是官方的迁移工具，核心就三条命令：

```bash
pnpm add -D drizzle-kit
```

先配置 `drizzle.config.ts`：

```typescript
import type { Config } from "drizzle-kit";

export default {
  schema: "./src/db/schema/*.ts",
  out: "./drizzle",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
} satisfies Config;
```

然后三条命令覆盖所有场景：

```bash
# 1. 根据Schema变更生成迁移文件
pnpm drizzle-kit generate

# 2. 在开发环境直接推送Schema到数据库（不生成迁移文件）
pnpm drizzle-kit push

# 3. 在生产环境执行已有的迁移文件
pnpm drizzle-kit migrate
```

`generate` 对比你的 Schema 和上一次的状态，生成一个 SQL 文件。`push` 则跳过迁移文件，直接把 Schema 同步到数据库。它们的分工是：**开发用 push，生产用 migrate**。

### 开发环境：push 的快节奏

开发阶段 Schema 变动频繁，今天加个字段明天又删了。如果每次都生成迁移文件，`drizzle/` 目录里会堆满没用的文件。`push` 就是干这个的——直接把当前的 Schema 推送到数据库，不管中间经历了什么：

```bash
# 给 posts 表加了个 excerpt 字段
pnpm drizzle-kit push
```

Drizzle Kit 会计算差异，直接执行 ALTER TABLE。没有中间文件，没有历史记录。适合开发阶段快速迭代。

### 生产环境：migrate 的严谨流程

上线之前，你需要把 Schema 变更变成可审查、可回滚的迁移文件：

```bash
pnpm drizzle-kit generate
```

这会在 `drizzle/` 目录下生成类似这样的文件：

```
drizzle/
  0000_initial.sql
  0001_add_excerpt_to_posts.sql
  0002_create_comments.sql
  meta/
    _journal.json
```

打开 `0001_add_excerpt_to_posts.sql` 看看：

```sql
ALTER TABLE "posts" ADD COLUMN "excerpt" text;
```

迁移文件就是纯 SQL。你可以审查它、修改它、在 CI 里跑它。生产环境执行迁移：

```typescript
import { migrate } from "drizzle-orm/postgres-js/migrator";
import { db } from "./db";

await migrate(db, { migrationsFolder: "./drizzle" });
```

或者在应用启动时自动执行（适合小项目）：

```typescript
if (process.env.NODE_ENV === "production") {
  await migrate(db, { migrationsFolder: "./drizzle" });
}
```

### 常见迁移场景

**添加字段**——最简单的场景。Schema 里加一列，`generate` 就会生成 `ALTER TABLE ADD COLUMN`：

```typescript
// schema 中新增字段
export const posts = pgTable("posts", {
  // ... 已有字段
  excerpt: text("excerpt"), // 新增
});
```

```bash
pnpm drizzle-kit generate
# 生成: ALTER TABLE "posts" ADD COLUMN "excerpt" text;
```

**修改类型**——比如把 `varchar(100)` 改成 `text`。Drizzle Kit 会生成相应的 ALTER 语句，但要注意 PostgreSQL 对某些类型转换有要求，可能需要先用 USING 子句做数据转换。

**重命名列**——这个比较特殊。Drizzle Kit 目前可能无法自动检测"重命名"和"删除旧列+添加新列"的区别。如果你在意数据，迁移文件生成后一定要检查，必要的时候手动把 `DROP + ADD` 改成 `RENAME`。

**添加索引和外键**——Schema 里定义好索引后，`generate` 会创建对应的 `CREATE INDEX` 语句。生产环境加索引要注意大表的锁问题，可以考虑 `CONCURRENTLY`（需要手动编辑迁移文件）。

### 一个容易忽略的实践

开发阶段用 push 很方便，但有个坑：如果你一直 push，从没 generate 过，上线时想一次性 generate 所有的迁移文件，结果可能不太理想——因为 Drizzle Kit 是对比当前 Schema 和上次 generate 的快照来生成差异的，没有历史快照时，它只能生成一个包含所有表的"初始化"迁移。

所以建议的做法是：**项目一开始就 generate 一次初始迁移**，之后的变更用 push 快速迭代，每次准备发版前再 generate 一次增量迁移。

## 种子数据

开发阶段需要测试数据。手动插入太痛苦，写个 seed 脚本一劳永逸：

```typescript
// src/db/seed.ts
import { db } from "./db";
import * as schema from "./schema";

async function seed() {
  // 创建测试用户
  const [alice] = await db
    .insert(schema.users)
    .values({
      email: "alice@blog.dev",
      name: "Alice Chen",
    })
    .returning();

  const [bob] = await db
    .insert(schema.users)
    .values({
      email: "bob@blog.dev",
      name: "Bob Wang",
    })
    .returning();

  // 创建标签
  const [tsTag] = await db
    .insert(schema.tags)
    .values({ name: "TypeScript", slug: "typescript" })
    .returning();

  const [pgTag] = await db
    .insert(schema.tags)
    .values({ name: "PostgreSQL", slug: "postgresql" })
    .returning();

  // 用事务创建带标签的文章
  const [post] = await db
    .insert(schema.posts)
    .values({
      title: "Drizzle ORM 实战指南",
      slug: "drizzle-orm-guide",
      content: "从 Schema 定义到生产部署...",
      authorId: alice.id,
    })
    .returning();

  await db.insert(schema.postsToTags).values([
    { postId: post.id, tagId: tsTag.id },
    { postId: post.id, tagId: pgTag.id },
  ]);

  // 评论
  await db.insert(schema.comments).values({
    postId: post.id,
    authorId: bob.id,
    content: "写得真好，学到了事务的正确用法",
  });

  console.log("Seed data created");
}

seed()
  .then(() => process.exit(0))
  .catch((e) => {
    console.error(e);
    process.exit(1);
  });
```

在 `package.json` 里加个脚本方便调用：

```json
{
  "scripts": {
    "db:seed": "tsx src/db/seed.ts",
    "db:push": "drizzle-kit push",
    "db:generate": "drizzle-kit generate",
    "db:migrate": "drizzle-kit migrate"
  }
}
```

日常开发流程就是：改 Schema -> `pnpm db:push` -> `pnpm db:seed`。

## Trade-off 分析

**什么时候必须用事务？** 涉及多表写入、且数据之间有依赖关系时——比如发布文章+关联标签、创建用户+初始化配置、订单+扣库存。单表的单次写入不需要事务。

**push 还是 migrate？** 开发阶段用 push，别犹豫。但如果你在做多人协作的项目，建议在 CI 里跑 generate + validate，确保迁移文件不会冲突。生产环境永远用 migrate，而且迁移文件要跟着代码一起进 Git。

**种子数据要不要放进迁移文件？** 不要。迁移文件只管表结构变更，种子数据单独管理。原因很简单：迁移是幂等的（跑一次和跑多次结果一样），但种子数据往往不是（重复插入会报错）。把它们混在一起，迁移会变得很难维护。

下一篇我们要解决一个更上层的问题：数据从数据库到前端，怎么保证全程类型安全？裸写 REST API 的话，请求参数和返回值都得手写类型，跟数据库 Schema 还容易脱节。Drizzle + tRPC 的组合是目前 TypeScript 全栈里类型安全最彻底的方案——下一篇来看看怎么搭建。
