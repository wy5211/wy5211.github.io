上一篇我们把博客系统的 Schema 设计好了——users、posts、comments、tags、post_to_tags 五张表，字段、索引、外键都就位了。现在面临一个很现实的问题：表建好了，数据呢？

你可能会说，不就是 INSERT 几条数据嘛。但实际做博客系统的时候，光"插入数据"这件事就有不少门道：批量导入文章的时候 slug 冲突怎么办？文章列表怎么分页？浏览量怎么原子递增？删除文章是真删还是假删？

今天就用"博客文章管理"这个场景，把 Drizzle ORM 的增删改查完整过一遍。

## 插入数据：从单条到批量

### 基本插入和 returning

最简单的场景：注册一个新用户。

```typescript
import { db } from "./db";
import { users } from "./schema";
import { eq } from "drizzle-orm";

const [newUser] = await db
  .insert(users)
  .values({
    email: "wang@example.com",
    username: "wang_dev",
    passwordHash: "hashed_xxx",
    name: "老王",
  })
  .returning();

// newUser 的类型被自动推断为：
// { id: number; email: string; username: string; ... }
// 不需要手写任何类型
```

`returning()` 是 PostgreSQL 的特性，INSERT 的同时把新行的数据返回给你。不加 `returning()` 的话，你拿不到新插入的 id，这在后续关联插入的时候会很麻烦。

### 批量导入文章

场景来了：你从 WordPress 迁移博客，需要一次性导入 50 篇文章。Drizzle 支持直接传数组：

```typescript
import { posts } from "./schema";

const importedPosts = await db
  .insert(posts)
  .values([
    {
      title: "Drizzle ORM 入门",
      slug: "drizzle-orm-intro",
      content: "正文内容...",
      authorId: 1,
      status: "PUBLISHED",
      publishedAt: new Date("2026-01-15"),
    },
    {
      title: "TypeScript 5.x 新特性",
      slug: "typescript-5-features",
      content: "正文内容...",
      authorId: 1,
      status: "PUBLISHED",
      publishedAt: new Date("2026-02-20"),
    },
    // ... 更多文章
  ])
  .returning();
```

Drizzle 会生成一条 `INSERT INTO posts ... VALUES (...), (...)` 语句，而不是循环执行 50 条 INSERT。性能差别很大。

### 冲突处理：onConflictDoUpdate

但迁移的时候大概率会遇到问题：某篇文章的 slug 已经存在了。slug 上有 unique 约束，直接插入会报错。

Drizzle 提供了 `onConflictDoUpdate` 来处理这种情况——冲突时更新，不冲突时插入：

```typescript
await db
  .insert(posts)
  .values({
    title: "Drizzle ORM 入门（修订版）",
    slug: "drizzle-orm-intro",
    content: "更新后的正文...",
    authorId: 1,
  })
  .onConflictDoUpdate({
    target: posts.slug,
    set: {
      title: "Drizzle ORM 入门（修订版）",
      content: "更新后的正文...",
      updatedAt: new Date(),
    },
  });
```

生成的 SQL 等价于：

```sql
INSERT INTO posts (title, slug, content, author_id)
VALUES ('Drizzle ORM 入门（修订版）', 'drizzle-orm-intro', ...)
ON CONFLICT (slug) DO UPDATE SET
  title = EXCLUDED.title,
  content = EXCLUDED.content,
  updated_at = NOW();
```

如果你不想更新，只是想"有就跳过"，用 `onConflictDoNothing`：

```typescript
await db
  .insert(tags)
  .values([{ name: "TypeScript", slug: "typescript" }])
  .onConflictDoNothing({
    target: tags.slug,
  });
```

这在批量导入标签的时候特别好用——标签已存在就跳过，不存在就创建，不需要先 SELECT 再决定 INSERT 还是 UPDATE。

## 查询数据：从全表到精准筛选

### select 的基本姿势

查所有文章的标题和发布状态：

```typescript
const allPosts = await db
  .select({
    id: posts.id,
    title: posts.title,
    status: posts.status,
    createdAt: posts.createdAt,
  })
  .from(posts);
```

注意这里我明确指定了要查的字段，而不是 `db.select().from(posts)` 查所有列。文章表的 `content` 字段可能存了好几 KB 的内容，列表页根本用不到，全查出来浪费带宽和内存。养成习惯：列表查询只 select 你需要的字段。

### where 条件：各种姿势的筛选

Drizzle 的条件操作符都是从 `drizzle-orm` 导入的函数，拼接方式很接近 SQL 的心理模型。

**精确匹配和范围查询：**

```typescript
import { eq, gt, lt, between } from "drizzle-orm";

// 查某个用户的所有文章
const userPosts = await db.select().from(posts).where(eq(posts.authorId, 1));

// 查 2026 年 1 月之后发布的文章
const recentPosts = await db
  .select()
  .from(posts)
  .where(gt(posts.publishedAt, new Date("2026-01-01")));

// 查浏览量在 100-500 之间的文章
const midRange = await db
  .select()
  .from(posts)
  .where(between(posts.viewCount, 100, 500));
```

**模糊搜索——博客的搜索功能：**

```typescript
import { ilike, or, and } from "drizzle-orm";

const searchResults = await db
  .select({
    id: posts.id,
    title: posts.title,
    excerpt: posts.excerpt,
  })
  .from(posts)
  .where(
    and(
      eq(posts.status, "PUBLISHED"),
      or(ilike(posts.title, "%drizzle%"), ilike(posts.content, "%drizzle%"))
    )
  );
```

`ilike` 是 PostgreSQL 特有的不区分大小写的 LIKE。如果你想区分大小写，用 `like`。不过大多数搜索场景用 `ilike` 更符合用户预期。

**inArray——按多个标签筛选：**

```typescript
import { inArray } from "drizzle-orm";

// 查状态为 DRAFT 或 ARCHIVED 的文章
const draftOrArchived = await db
  .select()
  .from(posts)
  .where(inArray(posts.status, ["DRAFT", "ARCHIVED"]));
```

### 排序和分页

博客列表页的标配：按发布时间倒序排列，每页 10 条。

```typescript
import { desc, asc, count } from "drizzle-orm";

const page = 1;
const pageSize = 10;

const [items, [{ total }]] = await Promise.all([
  db
    .select({
      id: posts.id,
      title: posts.title,
      excerpt: posts.excerpt,
      publishedAt: posts.publishedAt,
      viewCount: posts.viewCount,
    })
    .from(posts)
    .where(eq(posts.status, "PUBLISHED"))
    .orderBy(desc(posts.publishedAt))
    .limit(pageSize)
    .offset((page - 1) * pageSize),

  db
    .select({ total: count() })
    .from(posts)
    .where(eq(posts.status, "PUBLISHED")),
]);
```

这里用 `Promise.all` 并行查数据和总数，比串行快。分页计算 `offset = (page - 1) * pageSize` 是标准做法，没什么花哨的。

`orderBy` 支持多字段排序。比如先按是否精选排（精选在前），再按发布时间倒序：

```typescript
// 注意：isFeatured 默认 false，asc 排序时 false 在前
// 要精选在前，得用 desc
db.select()
  .from(posts)
  .orderBy(desc(posts.isFeatured), desc(posts.publishedAt));
```

这种细节，TypeScript 不会帮你检查，需要自己对 SQL 排序规则有理解。

### 动态查询构建

实际业务中，筛选条件往往是动态的——用户可能只选了分类，可能同时选了分类和关键词，也可能什么都没选。

```typescript
import { SQL, and, or, ilike, eq, desc } from "drizzle-orm";

async function getPostList(filters: {
  status?: "DRAFT" | "PUBLISHED" | "ARCHIVED";
  authorId?: number;
  keyword?: string;
  page?: number;
}) {
  const conditions: SQL[] = [];

  if (filters.status) {
    conditions.push(eq(posts.status, filters.status));
  }
  if (filters.authorId) {
    conditions.push(eq(posts.authorId, filters.authorId));
  }
  if (filters.keyword) {
    conditions.push(
      or(
        ilike(posts.title, `%${filters.keyword}%`),
        ilike(posts.excerpt, `%${filters.keyword}%`)
      )!
    );
  }

  const page = filters.page ?? 1;
  const pageSize = 10;

  return db
    .select()
    .from(posts)
    .where(conditions.length > 0 ? and(...conditions) : undefined)
    .orderBy(desc(posts.createdAt))
    .limit(pageSize)
    .offset((page - 1) * pageSize);
}
```

把条件收集到数组里，最后用 `and(...conditions)` 组装。没有条件时传 `undefined`，Drizzle 会忽略 where 子句。这种模式在实际项目中用得非常多。

## 更新数据：不只是改个字段

### 基本更新

修改文章标题是常规操作：

```typescript
const [updated] = await db
  .update(posts)
  .set({
    title: "Drizzle ORM 入门（第二版）",
    updatedAt: new Date(),
  })
  .where(eq(posts.id, 1))
  .returning();
```

`set()` 里只写要改的字段就行，其他字段保持不变。注意 `updatedAt` 需要手动维护——Drizzle 不会帮你自动更新这个字段，除非你在 Schema 里用 `.$onUpdate(() => new Date())` 钩子（Drizzle 0.38+ 支持这个特性）。

### 浏览量原子自增

文章被阅读一次，viewCount 加 1。你不能先 SELECT 再 UPDATE——并发时会丢计数。得用 SQL 表达式在数据库层面完成：

```typescript
import { sql } from "drizzle-orm";

await db
  .update(posts)
  .set({
    viewCount: sql`${posts.viewCount} + 1`,
  })
  .where(eq(posts.id, postId));
```

`sql` 模板标签是 Drizzle 的"逃生舱"——当你需要写原生 SQL 片段，但又不想完全脱离类型系统时，用它嵌入 SQL 表达式。生成的 SQL：

```sql
UPDATE posts SET view_count = view_count + 1 WHERE id = $1
```

这是一个原子操作，不管多少并发请求同时进来，每次浏览都会被正确计数。

### 批量更新

"把所有草稿文章标记为归档"——一次更新符合条件的所有行：

```typescript
await db
  .update(posts)
  .set({ status: "ARCHIVED", updatedAt: new Date() })
  .where(
    and(eq(posts.status, "DRAFT"), lt(posts.createdAt, new Date("2025-01-01")))
  );
```

一条 SQL 搞定，不需要先查出来再逐条更新。

**一个危险的注意点**：如果 `update` 不加 `where`，会更新整张表。Drizzle 不会阻止你这么干。这是个设计哲学问题——Drizzle 信任开发者，不像某些 ORM 会默认加 LIMIT 1。

## 删除数据：真删还是假删

### 物理删除

直接从数据库删除记录：

```typescript
// 删除某篇文章
await db.delete(posts).where(eq(posts.id, 42));

// 删除某用户的所有草稿
await db
  .delete(posts)
  .where(and(eq(posts.authorId, 3), eq(posts.status, "DRAFT")));
```

同样，不加 `where` 会清空整张表。

### 软删除：更安全的做法

对于博客系统，直接 DELETE 一篇文章太危险了——万一误删呢？常见的做法是软删除：不真删，而是打个标记。

我们的 posts 表没有 `deletedAt` 字段，需要先加上：

```typescript
// schema 里新增字段
deletedAt: timestamp("deleted_at"),
```

然后软删除就是一次 UPDATE：

```typescript
// 软删除
await db.update(posts).set({ deletedAt: new Date() }).where(eq(posts.id, 42));

// 查询时排除已删除的文章
const activePosts = await db
  .select()
  .from(posts)
  .where(sql`${posts.deletedAt} IS NULL`);
```

这里又用到了 `sql` 模板标签。Drizzle 提供了 `isNull()` 函数，写法是 `sql`${posts.deletedAt} IS NULL``或者`isNull(posts.deletedAt)`，效果一样，选你觉得顺眼的就行。

## 当 Drizzle API 不够用：sql 模板标签

前面已经用到了几次 `sql` 标签，这里正式介绍一下。Drizzle 的 API 覆盖了 90% 的日常查询，但有些场景它确实没有对应的函数，比如：

```typescript
import { sql } from "drizzle-orm";

// PostgreSQL 全文搜索——Drizzle 没有内置 API
const searchResults = await db
  .select()
  .from(posts)
  .where(
    sql`to_tsvector('simple', ${posts.title} || ' ' || ${posts.content})
        @@ to_tsquery('simple', ${searchTerm})`
  );

// 类型安全的计算字段
const postWithReadingTime = await db
  .select({
    id: posts.id,
    title: posts.title,
    readingTime: sql<number>`ceil(length(${posts.content}) / 1500)`.as(
      "reading_time"
    ),
  })
  .from(posts);
```

`sql` 标签会自动处理参数绑定（防止 SQL 注入），同时让 Drizzle 知道返回值的类型。`sql<number>` 告诉 TypeScript 这个表达式的结果是 number。

这个逃生舱的存在很重要——它意味着你不会遇到"Drizzle 做不到，只能换回原始 SQL 驱动"的窘境。任何 PostgreSQL 能做的查询，你都能通过 `sql` 标签在 Drizzle 里写。

## Prepared Statements：复用查询计划

如果你的接口被高频调用（比如文章列表），每次都让数据库解析 SQL、生成执行计划是浪费。Prepared Statements 让数据库缓存查询计划，只替换参数值。

```typescript
// 定义预编译查询
const getPostsByStatus = db
  .select({
    id: posts.id,
    title: posts.title,
    publishedAt: posts.publishedAt,
  })
  .from(posts)
  .where(eq(posts.status, sql.placeholder("status")))
  .orderBy(desc(posts.publishedAt))
  .limit(10)
  .prepare("get_posts_by_status");

// 使用时传入参数
const published = await getPostsByStatus.execute({ status: "PUBLISHED" });
const drafts = await getPostsByStatus.execute({ status: "DRAFT" });
```

`prepare("name")` 创建一个预编译语句，`sql.placeholder("status")` 标记参数位置。后续调用 `execute()` 只传参数，数据库不需要重新解析 SQL。

在 Serverless 环境（比如 Vercel Edge Functions）里，每个请求都是新的连接，Prepared Statements 的优势不那么明显。但在长连接场景（传统服务器、PM2 集群）下，效果显著。

## 查询构建器风格 vs 关系查询风格

到目前为止我们用的都是查询构建器风格——`db.select().from(posts).where(...)`。Drizzle 还有一种关系查询风格，用 `with` API 直接获取关联数据：

```typescript
// 查询构建器风格（本篇用的）
const result = await db
  .select({
    postId: posts.id,
    authorName: users.name,
  })
  .from(posts)
  .innerJoin(users, eq(posts.authorId, users.id));

// 关系查询风格（下一篇重点讲）
const result = await db.query.posts.findMany({
  with: {
    author: true,
    tags: true,
  },
});
```

关系查询风格的代码更简洁，但需要在 Schema 里定义 `relations`（上一篇已经定义了）。它的底层实现不是魔法——Drizzle 会自动帮你处理 JOIN 和数据组装，甚至解决了 N+1 查询问题。具体怎么做到的，下一篇细说。

## Trade-off 分析

Drizzle 的 CRUD API 有几个值得权衡的设计决策：

**查询构建器 vs 关系查询。** 查询构建器给你完全的控制力——你写的每一段 SQL 都在你脑子里。关系查询（`db.query`）更简洁，但当查询出问题时，调试难度更高，因为你看到的不是你写的。建议：简单关联用关系查询，复杂报表用查询构建器加 `sql` 标签。

**`sql` 模板标签的代价。** 它是逃生舱，也是技术债。过度使用 `sql` 标签意味着你在绕过 Drizzle 的类型系统，TypeScript 对嵌入的 SQL 片段几乎无能为力。一个项目中如果到处都是 `sql` 片段，你可能需要重新评估是否应该用 Drizzle。正确用法是：90% 用 Drizzle API，10% 用 `sql` 标签处理特殊需求。

**Prepared Statements 的适用场景。** 它不是银弹。在 Serverless 环境下连接本身就不复用，预编译的优势为零。在连接池模式下才有实际收益。不要盲目对所有查询都 prepare——只对高频、参数化、结构固定的查询使用。

**不带 `where` 的 update/delete。** Drizzle 允许你不加 where 就执行 update 或 delete，这会操作整张表。有些 ORM 默认阻止这种操作。Drizzle 的立场是"开发者知道自己在做什么"。如果你担心误操作，可以在应用层封装一个 `safeDelete` 函数，强制要求传 where 条件。
