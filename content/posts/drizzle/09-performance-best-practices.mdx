前八篇我们从 Schema 设计到 Edge 部署，把博客系统从头搭了一遍。但"能跑"和"跑得快"是两回事——当文章数到 10 万、并发用户到 1000，你可能发现首页加载从 200ms 变成了 2 秒。

今天不讲理论，我们直接从博客系统上线后遇到的真实性能问题出发，逐个诊断和解决。

## 首页文章列表为什么变慢了

博客上线两个月，文章从 50 篇涨到了 5000 篇，首页加载从 200ms 慢到了 800ms。打开 Drizzle 的查询日志看了一眼：

```typescript
// 开启 Drizzle 的 logger
import { drizzle } from "drizzle-orm/node-postgres";

export const db = drizzle(pool, {
  schema,
  logger: true, // 控制台打印每条 SQL
});
```

日志显示首页加载触发了这条查询：

```sql
SELECT id, title, slug, content, excerpt, cover_image, author_id,
       published, view_count, created_at, updated_at, published_at
FROM posts
ORDER BY created_at DESC
LIMIT 20
```

两个问题一目了然：第一，`content` 字段被查出来了——文章正文动辄几千字，首页列表根本用不到；第二，`ORDER BY created_at DESC` 没有索引，5000 条数据做了全表排序。

### 只查需要的字段

Drizzle 的 `select()` 不传参数等同于 `SELECT *`。首页列表只需要标题、摘要、封面这几个字段：

```typescript
const posts = await db
  .select({
    id: postsTable.id,
    title: postsTable.title,
    slug: postsTable.slug,
    excerpt: postsTable.excerpt,
    coverImage: postsTable.coverImage,
    createdAt: postsTable.createdAt,
  })
  .from(postsTable)
  .orderBy(desc(postsTable.createdAt))
  .limit(20);
```

光这一步，查询返回的数据量从每条约 5KB 降到了约 200 字节，网络传输时间直接砍掉 90%。

### 给排序字段加索引

没有索引的 `ORDER BY created_at DESC` 迫使 PostgreSQL 对全表做排序。解决方案是在 Schema 里加上索引：

```typescript
import {
  pgTable,
  serial,
  text,
  integer,
  boolean,
  timestamp,
  index,
} from "drizzle-orm/pg-core";

export const posts = pgTable(
  "posts",
  {
    id: serial("id").primaryKey(),
    title: text("title").notNull(),
    slug: text("slug").notNull().unique(),
    content: text("content").notNull(),
    excerpt: text("excerpt"),
    coverImage: text("cover_image"),
    published: boolean("published").default(false),
    authorId: integer("author_id")
      .references(() => users.id)
      .notNull(),
    viewCount: integer("view_count").default(0),
    createdAt: timestamp("created_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
    updatedAt: timestamp("updated_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
    publishedAt: timestamp("published_at", { withTimezone: true }),
  },
  (table) => ({
    // 给 ORDER BY created_at DESC 加索引
    createdAtIndex: index("posts_created_at_idx").on(table.createdAt.desc()),
    // WHERE published = true 经常用到，和 created_at 组成复合索引
    publishedCreatedIdx: index("posts_published_created_idx").on(
      table.published,
      table.createdAt.desc()
    ),
  })
);
```

修改 Schema 后运行迁移：

```bash
npx drizzle-kit generate
npx drizzle-kit migrate
```

复合索引 `posts_published_created_idx` 的好处是，当你查询 `WHERE published = true ORDER BY created_at DESC` 时，PostgreSQL 可以直接走索引扫描，不需要回表排序。

改完这两处，首页查询从 800ms 降到了 15ms。

## 热门文章的评论出现了 N+1 问题

博客有一篇阅读量 10 万的爆款文章，评论有 300 条。文章详情页的加载时间突然飙升到 1.5 秒。

问题出在评论的嵌套展示逻辑上。原始代码是这样的：

```typescript
// 查询文章的所有评论
const comments = await db
  .select()
  .from(commentsTable)
  .where(eq(commentsTable.postId, postId));

// 为每条评论查作者信息——这就是 N+1
for (const comment of comments) {
  const [author] = await db
    .select({ id: users.id, name: users.name, avatar: users.avatar })
    .from(users)
    .where(eq(users.id, comment.authorId))
    .limit(1);

  comment.author = author;
}
```

300 条评论意味着 301 次数据库查询（1 次查评论 + 300 次查作者）。这就是经典的 N+1 问题。

### 用 JOIN 一次搞定

```typescript
const commentsWithAuthors = await db
  .select({
    id: commentsTable.id,
    content: commentsTable.content,
    createdAt: commentsTable.createdAt,
    parentId: commentsTable.parentId,
    status: commentsTable.status,
    author: {
      id: users.id,
      name: users.name,
      avatar: users.avatar,
    },
  })
  .from(commentsTable)
  .innerJoin(users, eq(commentsTable.authorId, users.id))
  .where(eq(commentsTable.postId, postId))
  .orderBy(desc(commentsTable.createdAt));
```

一次查询拿到所有评论和作者信息。查询次数从 301 降到 1，加载时间从 1.5 秒降到 50ms。

识别 N+1 的方法很简单：如果你在循环里做数据库查询，大概率就是 N+1。解决办法就是 JOIN 或者批量 IN 查询。

## 批量导入文章时连接数爆了

运营团队需要把旧博客系统的 2000 篇文章迁移过来。同事写了个迁移脚本：

```typescript
// 危险操作：循环里逐条插入，每条都是独立的数据库连接
for (const post of legacyPosts) {
  await db.insert(postsTable).values({
    title: post.title,
    slug: post.slug,
    content: post.content,
    authorId: post.authorId,
    createdAt: new Date(post.created_at),
    updatedAt: new Date(),
  });
}
```

跑到 200 篇的时候，数据库报错 `too many connections`。

### 用批量插入替代循环单条

Drizzle 的 `insert().values()` 支持传入数组，一次 SQL 插入多行：

```typescript
// 每批 100 条，避免单条 SQL 过长
const BATCH_SIZE = 100;

for (let i = 0; i < legacyPosts.length; i += BATCH_SIZE) {
  const batch = legacyPosts.slice(i, i + BATCH_SIZE);

  await db.insert(postsTable).values(
    batch.map((post) => ({
      title: post.title,
      slug: post.slug,
      content: post.content,
      authorId: post.authorId,
      createdAt: new Date(post.created_at),
      updatedAt: new Date(),
    }))
  );
}
```

2000 篇文章从 2000 次查询变成了 20 次，而且不再有连接数问题。

如果批量操作涉及更新和删除，用事务包裹能进一步提升性能：

```typescript
await db.transaction(async (tx) => {
  // 事务内的所有操作共享同一个连接
  for (let i = 0; i < legacyPosts.length; i += BATCH_SIZE) {
    const batch = legacyPosts.slice(i, i + BATCH_SIZE);
    await tx.insert(postsTable).values(batch.map(transformPost));
  }
});
```

事务的好处是所有操作在同一个连接上执行，而且要么全部成功、要么全部回滚。

## 连接池参数调优

解决批量导入的问题后，我们顺手调了一下连接池参数。博客系统用的是 `postgres.js` 驱动，连接池配置直接影响并发能力。

```typescript
import postgres from "postgres";
import { drizzle } from "drizzle-orm/postgres-js";

const client = postgres(process.env.DATABASE_URL!, {
  max: 10, // 最大连接数，根据数据库规格调整
  idle_timeout: 20, // 空闲连接 20 秒后回收
  connect_timeout: 10, // 建立连接超时 10 秒
  max_lifetime: 60 * 30, // 连接最长存活 30 分钟，防止内存泄漏
});

export const db = drizzle(client, { schema });
```

几个参数的解释：

- `max` 设多大取决于你的数据库能承受多少并发连接。PostgreSQL 默认最大 100 个连接，如果应用有多个实例（比如 5 个 Pod），每个实例 `max` 设 10 就占了 50 个连接，留些余量给运维工具和临时查询。
- `idle_timeout` 太小会导致频繁建连接，太大则浪费数据库资源。20 秒是个比较安全的值。
- `max_lifetime` 是个容易被忽略的参数。长时间复用同一个连接可能遇到数据库端的内存泄漏或状态异常，30 分钟重建一次是合理的折中。

不同环境的配置思路不同：

```typescript
const isDev = process.env.NODE_ENV === "development";

const client = postgres(process.env.DATABASE_URL!, {
  max: isDev ? 1 : 10, // 开发环境单连接，避免热重载连接泄漏
  idle_timeout: isDev ? 5 : 20,
  prepare: !isDev, // 开发环境关闭 prepared statements，方便调试
});
```

开发环境用单连接有个好处：每次 HMR 热重载不会创建新连接，避免开发时连接数疯涨。

## Prepared Statements：预编译查询

博客首页是访问量最大的页面，每次都编译同一条 SQL 有些浪费。Drizzle 支持 Prepared Statements，把 SQL 编译一次、多次执行：

```typescript
import { sql } from "drizzle-orm";

// 定义预编译查询
const getLatestPosts = db
  .select({
    id: postsTable.id,
    title: postsTable.title,
    slug: postsTable.slug,
    excerpt: postsTable.excerpt,
    coverImage: postsTable.coverImage,
    createdAt: postsTable.createdAt,
  })
  .from(postsTable)
  .where(eq(postsTable.published, sql.placeholder("published")))
  .orderBy(desc(postsTable.createdAt))
  .limit(sql.placeholder("limit"))
  .prepare("getLatestPosts");

// 每次请求直接执行，跳过 SQL 编译
const posts = await getLatestPosts.execute({ published: true, limit: 20 });
```

`prepare("getLatestPosts")` 中的字符串是语句标识符，`postgres.js` 驱动会按这个标识符缓存编译结果。注意标识符要全局唯一，不同的 prepare 调用不能用相同的名字。

Prepared Statements 最适合高频查询场景——首页列表、热门文章、用户信息查询这些每次请求都会触发的操作。对于只执行一次的管理后台查询，加了反而多余。

## 慢查询监控和 Drizzle Logger

优化做完了，但线上随时可能出现新的慢查询。我们需要一个持续运行的监控机制。

Drizzle 的自定义 logger 可以捕获每条查询的执行时间：

```typescript
export const db = drizzle(client, {
  schema,
  logger: {
    logQuery: (query, params) => {
      const start = Date.now();
      // 返回一个回调，查询执行完后调用
      return () => {
        const duration = Date.now() - start;
        if (duration > 100) {
          console.warn(`[Slow Query] ${duration}ms | ${query}`);
        }
      };
    },
  },
});
```

超过 100ms 的查询会被记录下来。你可以把 `console.warn` 替换成发送到监控系统（比如 Datadog、Grafana）的逻辑。

这里有个容易踩的坑：Drizzle 的 `logger: true` 只是打印 SQL 语句，不包含执行时间。要拿到执行时间，必须用 `logQuery` 自定义 logger，像上面那样手动计时。

一个实用的健康检查接口也值得加上：

```typescript
import { sql } from "drizzle-orm";

export async function healthCheck() {
  const start = Date.now();
  await db.execute(sql`SELECT 1`);
  const latency = Date.now() - start;

  return {
    database: { connected: true, latency: `${latency}ms` },
    status: latency < 50 ? "healthy" : "degraded",
  };
}
```

部署后把这个接口接到负载均衡器的健康检查上，数据库一慢就能第一时间知道。

## 缓存策略：别让每个请求都打到数据库

热门文章每小时访问量 1000 次，每次都查数据库太浪费。引入缓存层可以大幅减少数据库压力。

### 应用层内存缓存

对于数据变化不频繁的内容，进程内 LRU 缓存最简单：

```typescript
import { LRUCache } from "lru-cache";

const postCache = new LRUCache<string, any>({
  max: 200, // 最多缓存 200 篇文章
  ttl: 1000 * 60 * 5, // 5 分钟过期
});

async function getPostBySlug(slug: string) {
  const cached = postCache.get(`post:${slug}`);
  if (cached) return cached;

  const [post] = await db
    .select()
    .from(postsTable)
    .where(eq(postsTable.slug, slug))
    .limit(1);

  if (post) {
    postCache.set(`post:${slug}`, post);
  }
  return post;
}
```

注意内存缓存只适合单实例部署。如果你的应用跑多个 Pod，每个 Pod 的缓存是独立的，可能出现不一致。这种情况用 Redis 做共享缓存更合适。

### HTTP 缓存

对于公开内容（文章列表、文章详情），CDN 层的 HTTP 缓存比应用缓存更高效：

```typescript
// Next.js App Router 示例
export async function GET(request: Request) {
  const posts = await getLatestPosts.execute({ published: true, limit: 20 });

  return Response.json(posts, {
    headers: {
      "Cache-Control": "public, s-maxage=300, stale-while-revalidate=60",
    },
  });
}
```

`stale-while-revalidate=60` 的意思是：缓存过期后的 60 秒内，先返回旧数据，后台异步刷新。用户永远不会感受到缓存未命中。

缓存有个原则：写操作必须主动失效缓存，不能只靠 TTL 过期。文章更新或发布时，清除对应的缓存 key：

```typescript
async function publishPost(postId: number) {
  const [post] = await db
    .update(postsTable)
    .set({ published: true, publishedAt: new Date() })
    .where(eq(postsTable.id, postId))
    .returning();

  // 清缓存
  postCache.delete(`post:${post.slug}`);
  postCache.delete("posts:latest");

  return post;
}
```

## 查看索引使用情况

加了一堆索引之后，怎么知道哪些在用、哪些是摆设？PostgreSQL 自带了索引使用统计：

```sql
-- 查看每个索引的扫描次数
SELECT indexname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE tablename = 'posts'
ORDER BY idx_scan ASC;
```

`idx_scan = 0` 的索引从来没被用过，不仅浪费存储空间，还会拖慢写入速度（每次 INSERT/UPDATE 都要维护索引）。找到没用过的索引，该删就删。

也可以用 `EXPLAIN ANALYZE` 看具体查询的执行计划：

```sql
EXPLAIN ANALYZE
SELECT id, title, slug FROM posts
WHERE published = true
ORDER BY created_at DESC
LIMIT 20;
```

如果输出里有 `Seq Scan`（全表扫描）而不是 `Index Scan`，说明查询没走到索引，需要检查 WHERE 条件和索引定义是否匹配。

## Trade-off 分析

性能优化没有银弹，每个方案都有代价：

**选字段查询 vs SELECT \***：选字段查询减少传输量，但如果业务经常需要切换展示字段，维护 select 列表会增加代码复杂度。折中方案是定义好几种常用的 select preset（列表用、详情用、后台管理用），按场景选用。

**索引数量 vs 写入性能**：每个索引加速一种查询模式，但拖慢 INSERT 和 UPDATE。一篇博客系统的文章表，3-5 个索引是合理的（主键、slug 唯一索引、发布时间排序索引、作者索引、标签关联索引）。超过 10 个就该想想是不是在为不存在的查询模式建索引。

**缓存 vs 数据一致性**：缓存越久性能越好，但数据越可能过期。文章内容可以缓存久一些（5-10 分钟），评论需要更短的 TTL（30 秒-1 分钟）。关键是用 `stale-while-revalidate` 让用户无感知地过渡。

**Prepared Statements vs 灵活性**：预编译查询快，但参数必须是固定的结构。如果你的查询条件动态拼接（比如多条件筛选），prepared statements 反而不如动态查询灵活。对高频固定查询用 prepared statements，对低频动态查询用普通查询就好。

**批量操作 vs 内存占用**：批量插入 1000 条比循环插入 1000 次快 10 倍以上，但如果单条数据很大（比如包含长文章内容），一批 1000 条可能占用大量内存。根据单条数据大小动态调整 batch size：小数据每批 500-1000 条，大数据每批 50-100 条。
