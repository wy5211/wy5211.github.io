上一篇我们把博客系统跑在了 Next.js + Vercel 上。Server Components 和 Server Actions 用起来很爽，但有一个问题你可能已经感受到了——如果你的用户遍布全球，Vercel 的 Serverless 函数部署在固定区域（通常是美区），亚洲用户访问一次 API 就要多吃几百毫秒的网络延迟。

Cloudflare Workers 可以把你的代码部署到离用户最近的节点——全球 300+ 个 POP。而 Drizzle 是少数在 Edge 上跑得舒服的 ORM。这一篇我们把博客系统的 API 层从 Node.js + PostgreSQL 迁移到 Cloudflare Workers + D1（SQLite at Edge），用 KV 缓存热门文章，实现全球低延迟。

## 为什么 Drizzle 在 Edge 上有天然优势

先说结论：绝大多数 ORM 根本跑不了 Edge。

Prisma 依赖一个用 Rust 写的 Query Engine 二进制文件，体积接近 2MB，Edge 运行时既加载不了也跑不动。TypeORM 依赖 Node.js 的 `net` 模块做 TCP 连接，而 Workers 的运行时没有这些 API。

Drizzle 完全不同：

- **零原生依赖**——纯 TypeScript 实现，没有二进制文件，没有 native addon
- **体积极小**——D1 driver 压缩后大约 30KB，Prisma 的 engine 光下载就要好几秒
- **基于 Web 标准**——用 Fetch API、Web Crypto 等 Edge 原生支持的 API
- **Driver 可插拔**——`drizzle-orm/d1`、`drizzle-orm/neon-http` 等按需引入，不需要引入所有数据库的适配器

这意味着你在 Edge 上用 Drizzle 和在 Node.js 上用几乎没有体验差异。Schema 定义完全一样，查询 API 完全一样，只是底层 driver 从 `postgres-js` 换成了 `d1`。

### Schema 定义（SQLite 版本）

D1 底层是 SQLite，所以 Schema 要用 `drizzle-orm/sqlite-core` 而不是 `drizzle-orm/pg-core`。这是迁移过程中最大的变化——数据类型和约束有一些差异。

```typescript
// src/db/schema.ts
import { sqliteTable, text, integer } from "drizzle-orm/sqlite-core";

export const users = sqliteTable("users", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  email: text("email").notNull().unique(),
  name: text("name"),
  createdAt: integer("created_at", { mode: "timestamp" })
    .notNull()
    .$defaultFn(() => new Date()),
});

export const posts = sqliteTable("posts", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  title: text("title").notNull(),
  slug: text("slug").notNull().unique(),
  content: text("content").notNull(),
  excerpt: text("excerpt"),
  published: integer("published", { mode: "boolean" }).default(false),
  authorId: integer("author_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  createdAt: integer("created_at", { mode: "timestamp" })
    .notNull()
    .$defaultFn(() => new Date()),
  updatedAt: integer("updated_at", { mode: "timestamp" })
    .notNull()
    .$defaultFn(() => new Date()),
});

export const comments = sqliteTable("comments", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  content: text("content").notNull(),
  postId: integer("post_id")
    .notNull()
    .references(() => posts.id, { onDelete: "cascade" }),
  authorId: integer("author_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  createdAt: integer("created_at", { mode: "timestamp" })
    .notNull()
    .$defaultFn(() => new Date()),
});

export const tags = sqliteTable("tags", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  name: text("name").notNull().unique(),
});

export const postToTags = sqliteTable("post_to_tags", {
  postId: integer("post_id")
    .notNull()
    .references(() => posts.id, { onDelete: "cascade" }),
  tagId: integer("tag_id")
    .notNull()
    .references(() => tags.id, { onDelete: "cascade" }),
});
```

和 PostgreSQL 版本的主要差异：

- `serial` 变成了 `integer().primaryKey({ autoIncrement: true })`
- `varchar` / `text` 统一用 `text()`
- `boolean` 变成了 `integer({ mode: "boolean" })`——SQLite 没有原生布尔类型，存的是 0/1
- `timestamp` 变成了 `integer({ mode: "timestamp" })`——SQLite 存的是 Unix 时间戳（秒）
- 没有 `defaultNow()`，改用 `$defaultFn(() => new Date())`

## 在 Workers 里写 API

Cloudflare Workers 的入口是一个 `fetch` handler，类似一个极简的 HTTP 框架。路由要自己用 URL pathname 匹配。

### 入口和路由

```typescript
// src/index.ts
import { drizzle } from "drizzle-orm/d1";
import { eq, desc } from "drizzle-orm";
import * as schema from "./db/schema";

export interface Env {
  DB: D1Database;
  CACHE: KVNamespace;
}

export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const url = new URL(request.url);
    const db = drizzle(env.DB, { schema });

    // 文章列表
    if (url.pathname === "/api/posts" && request.method === "GET") {
      return getPosts(db, env.CACHE);
    }

    // 文章详情
    const postMatch = url.pathname.match(/^\/api\/posts\/([^/]+)$/);
    if (postMatch && request.method === "GET") {
      return getPostBySlug(db, env.CACHE, postMatch[1]);
    }

    // 提交评论
    if (url.pathname === "/api/comments" && request.method === "POST") {
      return createComment(request, db, env.CACHE);
    }

    // 标签列表
    if (url.pathname === "/api/tags" && request.method === "GET") {
      return getTags(db, env.CACHE);
    }

    return new Response("Not Found", { status: 404 });
  },
};
```

注意 `const db = drizzle(env.DB, { schema })` 放在 handler 内部而不是模块顶层。Workers 的机制是：同一个 isolate 可能复用处理多个请求，但 `env` 对象是每次请求传入的。如果放在模块顶层，冷启动时 `env` 还不存在，会直接报错。

### 文章列表（带 KV 缓存）

```typescript
async function getPosts(db: ReturnType<typeof drizzle>, cache: KVNamespace) {
  // 先查 KV 缓存
  const cached = await cache.get("posts:list", "json");
  if (cached) {
    return Response.json(cached, { headers: { "X-Cache": "HIT" } });
  }

  // 缓存未命中，查 D1
  const result = await db
    .select({
      id: schema.posts.id,
      title: schema.posts.title,
      slug: schema.posts.slug,
      excerpt: schema.posts.excerpt,
      authorName: schema.users.name,
      createdAt: schema.posts.createdAt,
    })
    .from(schema.posts)
    .innerJoin(schema.users, eq(schema.posts.authorId, schema.users.id))
    .where(eq(schema.posts.published, true))
    .orderBy(desc(schema.posts.createdAt))
    .limit(20);

  // 写入 KV，60 秒过期
  await cache.put("posts:list", JSON.stringify(result), { expirationTtl: 60 });

  return Response.json(result, { headers: { "X-Cache": "MISS" } });
}
```

KV 的读取延迟在 1ms 以内，D1 的查询延迟通常在 5-15ms。热数据走 KV，冷数据走 D1，这是 Edge 架构的核心策略。

### 文章详情（按 slug 查询）

```typescript
async function getPostBySlug(
  db: ReturnType<typeof drizzle>,
  cache: KVNamespace,
  slug: string
) {
  const cacheKey = `post:${slug}`;
  const cached = await cache.get(cacheKey, "json");
  if (cached) {
    return Response.json(cached, { headers: { "X-Cache": "HIT" } });
  }

  const [post] = await db
    .select({
      id: schema.posts.id,
      title: schema.posts.title,
      content: schema.posts.content,
      createdAt: schema.posts.createdAt,
      authorName: schema.users.name,
    })
    .from(schema.posts)
    .innerJoin(schema.users, eq(schema.posts.authorId, schema.users.id))
    .where(eq(schema.posts.slug, slug))
    .limit(1);

  if (!post) {
    return new Response("Post not found", { status: 404 });
  }

  // 单篇文章缓存久一点，5 分钟
  await cache.put(cacheKey, JSON.stringify(post), { expirationTtl: 300 });

  return Response.json(post, { headers: { "X-Cache": "MISS" } });
}
```

### 提交评论（写入后清缓存）

```typescript
async function createComment(
  request: Request,
  db: ReturnType<typeof drizzle>,
  cache: KVNamespace
) {
  const body = (await request.json()) as {
    content: string;
    postId: number;
    authorId: number;
    postSlug: string;
  };

  if (!body.content || !body.postId || !body.authorId) {
    return new Response("Missing required fields", { status: 400 });
  }

  await db.insert(schema.comments).values({
    content: body.content,
    postId: body.postId,
    authorId: body.authorId,
  });

  // 写入后清除相关缓存
  await cache.delete(`post:${body.postSlug}`);
  await cache.delete("posts:list");

  return new Response("Created", { status: 201 });
}
```

这里有个值得思考的问题：KV 是最终一致性的，`cache.delete` 之后在全球范围内生效可能需要 60 秒。也就是说用户刚提交完评论，刷新页面可能暂时看不到——这和传统数据库的"写后立即读"不一样。如果你的场景要求强一致性，写入后应该直接返回新数据而不是依赖缓存刷新。

## D1 的 SQLite 语法差异和注意事项

从 PostgreSQL 迁移到 D1 不只是换个 driver 那么简单。SQLite 和 PostgreSQL 在 SQL 层面有一些差异会影响你的查询写法。

### 日期处理

SQLite 没有 `timestamp` 类型。Drizzle 的 `integer({ mode: "timestamp" })` 自动帮你做 `Date <-> Unix timestamp` 的转换。但在 raw SQL 里你需要自己处理：

```typescript
import { sql } from "drizzle-orm";

// 在 raw SQL 中手动转换
const result = await db.run(sql`
  SELECT * FROM posts
  WHERE created_at > unixepoch('now', '-7 days')
  ORDER BY created_at DESC
`);
```

### JSON 操作

SQLite 支持 JSON 函数，但语法和 PostgreSQL 不同。如果你之前用了 `jsonb` 类型和 PostgreSQL 的 `->>` 操作符，需要改写：

```typescript
// PostgreSQL
sql`${posts.metadata}->>'title'`;

// SQLite
sql`json_extract(${posts.metadata}, '$.title')`;
```

### 全文搜索

如果你用 PostgreSQL 的 `tsvector` 做全文搜索，迁移到 D1 后要换成 SQLite FTS5：

```sql
-- 创建全文搜索虚拟表
CREATE VIRTUAL TABLE posts_fts USING fts5(title, content, content=posts, content_rowid=id);

-- 触发器：自动同步
CREATE TRIGGER posts_ai AFTER INSERT ON posts BEGIN
  INSERT INTO posts_fts(rowid, title, content) VALUES (new.id, new.title, new.content);
END;
```

这个迁移是最容易踩坑的地方。如果你的博客重度依赖全文搜索，认真评估一下 FTS5 是否满足你的需求——它在中文分词上不如 PostgreSQL 的 zhparser 插件。

## Edge vs 传统架构：什么时候该换

这里直接给一个对比，帮你判断自己的项目适不适合迁移。

| 维度       | Node.js + PostgreSQL                  | Cloudflare Workers + D1                    |
| ---------- | ------------------------------------- | ------------------------------------------ |
| 全球延迟   | 100-500ms（取决于用户到服务器的距离） | 10-50ms（就近节点）                        |
| 冷启动     | 几百毫秒到几秒（Vercel）              | < 5ms（Workers）                           |
| 数据库功能 | PostgreSQL 全功能                     | SQLite 子集（够用，但有边界）              |
| 事务支持   | 完整 ACID                             | D1 支持，但跨 region 有延迟                |
| 连接管理   | 需要连接池、要防泄漏                  | 不需要——D1 是内嵌的，没有连接概念          |
| 部署复杂度 | 需要管数据库实例                      | D1 是托管服务，零运维                      |
| 生态成熟度 | 非常成熟                              | 快速成长中，但比不上 PG 生态               |
| 成本       | Vercel Pro + 数据库实例，中等偏高     | Workers 免费 10 万次/天，D1 免费 5GB，很低 |

**适合迁移到 Edge 的场景：**

- 读多写少的内容型应用（博客、文档站、CMS）
- 全球用户分布，延迟敏感
- 流量波动大，需要自动弹性扩缩
- 小团队，不想管基础设施

**不适合迁移的场景：**

- 重度事务和复杂查询（多表 join + 聚合 + 窗口函数）
- 需要完整的 PostgreSQL 功能（jsonb 高级操作、LISTEN/NOTIFY、扩展插件）
- 已有庞大的 PostgreSQL 数据，迁移成本太高
- 需要强一致性的写入后立即读取

实际上很多项目不需要二选一。你可以把读密集的 API 放到 Workers（文章列表、文章详情），把写操作和复杂查询留在 Node.js（管理后台、数据分析）。用同一种 Schema 定义，不同的 driver——这就是 Drizzle 的 driver 可插拔带来的好处。

## Trade-off 分析

**Edge + D1 的优势：**

- 全球低延迟，用户体验提升是实打实的
- 零运维——不用管连接池、不用管数据库实例扩容、不用管证书更新
- 成本极低——对于中小流量的博客来说几乎免费
- 冷启动快——Workers 的 V8 isolate 比容器的冷启动快两个数量级

**Edge + D1 的代价：**

- SQLite 的功能子集——没有完整的 ALTER 支持，JSON 操作不如 PostgreSQL 方便
- 最终一致性的 KV 缓存——写入后全球生效有延迟，不适合强一致性要求的场景
- 调试体验差一些——没有直连数据库的 GUI 工具（D1 有 Dashboard 但功能有限），日志查看靠 `wrangler tail`
- Workers 的运行时限制——CPU 时间 10ms（免费）/ 30ms（付费），128MB 内存，单个请求不能跑太重的计算
- 数据库大小限制——D1 单库 10GB，对于博客来说绰绰有余，但其他类型的应用要留意

**我的建议：**

如果你的博客/内容站已经有 Node.js + PostgreSQL 的版本在跑，先别急着全面迁移。可以先把文章列表和详情这两个最热的读接口搬到 Workers + D1，用 D1 做只读副本（定期从 PostgreSQL 同步数据），管理后台继续用 Node.js。验证效果之后再把写入也迁过来。下一篇我们讲性能优化——不管你用哪种架构，查询优化和索引策略都是绕不开的话题。下一篇见。
