你用 Prisma 开发了一个博客系统，本地跑得好好的。部署到 Cloudflare Workers 的时候，构建直接炸了——Prisma 的查询引擎依赖 Node.js 运行时，根本没法在 Edge 环境里跑。你翻遍了文档，折腾了一整天，最后只能换回传统的 Node.js 服务器部署。

或者你没那么倒霉，但你的接口 P99 延迟总是卡在 100-200ms，排查了一圈发现瓶颈在 Prisma 的查询引擎层。一个简单的 `findUnique`，SQL 执行只要 5ms，Prisma 硬是把总耗时拖到了 50ms 以上。

这些问题不是偶然的。Prisma 的架构决定了它跑不快，也跑不到 Edge 上去。而 Drizzle ORM 正是为了解决这些问题而生的。

## 同一个查询任务，三个 ORM 写出来差别有多大

先别急着看特性对比表。我们用一个真实的场景——查一篇博客文章及其作者信息——直接看代码。

**Prisma 的写法：**

```typescript
const post = await prisma.post.findUnique({
  where: { id: 1 },
  include: { author: true },
});
// 耗时约 50ms（含查询引擎开销）
```

看起来很简洁。但这里有个你平时不会注意的问题：Prisma 会启动一个独立的 Rust 查询引擎进程，你的查询先经过 JS 客户端，再通过 IPC 传给查询引擎，查询引擎生成 SQL，执行后再把结果传回来。一个简单的查询，光在 IPC 上就花了不少时间。

**TypeORM 的写法：**

```typescript
const post = await repository.findOne({
  where: { id: 1 },
  relations: ["author"],
});
// 耗时约 30ms
```

TypeORM 没有查询引擎的问题，但类型推断是你永远的痛。`post` 的类型可能是 `Post | null`，但 `post.author` 在 TypeScript 眼里可能是 `undefined`——即使你在数据库层面定义了外键不为空。你得手动加类型断言，或者在运行时做判空检查。

**Drizzle 的写法：**

```typescript
const post = await db
  .select({
    id: posts.id,
    title: posts.title,
    content: posts.content,
    authorName: users.name,
  })
  .from(posts)
  .innerJoin(users, eq(posts.authorId, users.id))
  .where(eq(posts.id, 1))
  .limit(1);
// 耗时约 5ms
// 返回类型：{ id: number; title: string; content: string; authorName: string }[]
```

Drizzle 没有 IPC，没有查询引擎，函数调用直接编译成 SQL 字符串。而且返回类型是精确推断的——你 select 了哪些字段，TypeScript 就知道结果里有哪些字段。不是 `Post` 这种大杂烩类型，而是你实际需要的那几个字段的精确类型。

### 体积差异也是一个硬伤

这三个 ORM 打包后的体积差异大到让你没法忽视：

| ORM     | Gzip 后体积 | 能跑在 Cloudflare Workers 上吗 |
| ------- | ----------- | ------------------------------ |
| Prisma  | ~600KB      | 不行，光查询引擎就超了         |
| TypeORM | ~200KB      | 勉强，但冷启动慢               |
| Drizzle | ~15KB       | 完美，几乎不占空间             |

Cloudflare Workers 的打包限制是压缩后 1MB。Prisma 光自己就占了大半，再塞点业务代码和依赖，直接爆掉。这不是配置能解决的问题，是架构层面的硬限制。

## Drizzle 的核心设计理念

Drizzle 的口号是"如果你懂 SQL，你就懂 Drizzle"。这不是营销话术，它的 API 确实是 SQL 的 1:1 映射。

举个例子，你想写这个 SQL：

```sql
SELECT id, title FROM posts
WHERE status = 'PUBLISHED'
ORDER BY created_at DESC
LIMIT 10;
```

在 Drizzle 里就是：

```typescript
const result = await db
  .select({ id: posts.id, title: posts.title })
  .from(posts)
  .where(eq(posts.status, "PUBLISHED"))
  .orderBy(desc(posts.createdAt))
  .limit(10);
```

几乎是一一对应的。你不需要学一套新的查询 DSL，不需要理解 `include` 和 `select` 的区别（Prisma 开发者肯定被这两个搞晕过），更不需要在遇到复杂查询时退回到 raw SQL。

### 为什么 Drizzle 不需要代码生成

Prisma 的工作流是这样的：你先写 `.prisma` 文件，然后运行 `npx prisma generate`，它会生成一堆 TypeScript 类型文件和客户端代码。每次改了 schema，你都得重新跑 generate，否则类型就对不上。

这个流程有几个实际痛点：

- CI/CD 管线里必须加一步 `prisma generate`，忘了就构建失败
- 生成的代码在 `node_modules` 里，调试时跳进去看一堆自动生成的代码
- 如果你在 monorepo 里共享 schema，generate 的路径配置经常出幺蛾子

Drizzle 不需要 generate 这一步。你的 schema 就是用 TypeScript 写的，类型在定义的时候就确定了。改了 schema，TypeScript 编译器立刻告诉你哪些查询的类型对不上了。

```typescript
// Drizzle schema 定义——这就是纯 TypeScript 代码
import { pgTable, serial, varchar, text, timestamp } from "drizzle-orm/pg-core";

export const users = pgTable("users", {
  id: serial("id").primaryKey(),
  email: varchar("email", { length: 255 }).notNull().unique(),
  name: text("name"),
  createdAt: timestamp("created_at").defaultNow(),
});

// 这个表定义本身就是类型——不需要 generate
// users.$inferSelect 自动推断为：
// { id: number; email: string; name: string | null; createdAt: Date }
```

## 动手跑通第一个查询

说了这么多，不如上手试试。我们从零搭建一个连接 PostgreSQL 的项目，建一张 users 表，插入并查询一条数据。

### 安装依赖

```bash
mkdir drizzle-blog && cd drizzle-blog
npm init -y
npm install drizzle-orm postgres
npm install -D drizzle-kit typescript tsx @types/node
npx tsc --init
```

这里 `drizzle-orm` 是核心库，`postgres` 是 PostgreSQL 的驱动，`drizzle-kit` 是 CLI 工具，用来生成和运行迁移。你不需要额外装 Prisma 那样的查询引擎。

### 创建数据库连接

```typescript
// src/db.ts
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema";

const client = postgres(process.env.DATABASE_URL!);
export const db = drizzle(client, { schema });
```

这里有个细节值得注意：`{ schema }` 参数把所有表定义传给了 drizzle 实例。这样做的好处是后面你可以用 relational query API（类似 Prisma 的 `include` 语法）来做关联查询，后面的文章会详细讲。

### 定义 Schema

```typescript
// src/schema.ts
import { pgTable, serial, varchar, text, timestamp } from "drizzle-orm/pg-core";

export const users = pgTable("users", {
  id: serial("id").primaryKey(),
  email: varchar("email", { length: 255 }).notNull().unique(),
  username: varchar("username", { length: 50 }).notNull().unique(),
  displayName: text("display_name"),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow(),
});
```

注意列名用的是 `snake_case`（`display_name`），而 TypeScript 属性用的是 `camelCase`（`displayName`）。Drizzle 不会强制你用哪种命名，但保持数据库列名 `snake_case`、代码属性 `camelCase` 是 PostgreSQL 项目的常见惯例。

### 配置迁移工具

```typescript
// drizzle.config.ts
import { defineConfig } from "drizzle-kit";

export default defineConfig({
  schema: "./src/schema.ts",
  out: "./drizzle",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
});
```

然后生成并运行迁移：

```bash
npx drizzle-kit generate   # 生成 SQL 迁移文件
npx drizzle-kit migrate    # 执行迁移，在数据库里创建 users 表
```

### 跑通 CRUD

```typescript
// src/index.ts
import { db } from "./db";
import { users } from "./schema";
import { eq } from "drizzle-orm";

async function main() {
  // 插入一条用户
  const [newUser] = await db
    .insert(users)
    .values({
      email: "zhang_wei@example.com",
      username: "zhang_wei",
      displayName: "张伟",
    })
    .returning();

  console.log("插入结果:", newUser);
  // TypeScript 知道 newUser 的类型是：
  // { id: number; email: string; username: string;
  //   displayName: string | null; createdAt: Date | null }

  // 按 email 查询
  const [found] = await db
    .select()
    .from(users)
    .where(eq(users.email, "zhang_wei@example.com"));

  console.log("查询结果:", found);
}

main();
```

运行 `npx tsx src/index.ts`，如果你看到两条一致的输出，说明整条链路跑通了。

这里 `returning()` 是 PostgreSQL 特有的语法，插入后直接返回新插入的行。MySQL 用户可能不太熟悉，但它确实很方便，省掉了再查一次的麻烦。

## Drizzle 什么时候该用，什么时候别碰

Drizzle 不是银弹。它有自己的适用场景，也有明显的短板。

**适合用 Drizzle 的场景：**

- 你的项目需要部署到 Edge 环境（Cloudflare Workers、Deno Deploy、Vercel Edge Functions）。Prisma 和 TypeORM 在这些环境里要么跑不了，要么体验很差。
- 你对查询性能有要求，尤其是高并发的读场景。Drizzle 的零开销架构在这里优势明显。
- 你的团队 SQL 基础扎实，能直接理解 `innerJoin`、`groupBy` 这些概念。Drizzle 不帮你"翻译"SQL，它让你用 TypeScript 的方式写 SQL。
- 你在做 Serverless 应用，关心冷启动时间和包体积。

**不太适合的场景：**

- 你的团队刚从 Ruby on Rails 或者 Django 过来，对 SQL 不太熟，更习惯"模型驱动"的抽象。Prisma 的抽象层更友好。
- 项目需要一个可视化的数据库管理工具。Prisma Studio 开箱即用，Drizzle 虽然有 Studio 但功能没那么成熟。
- 你已经在用 Prisma 了，而且项目运行稳定。迁移成本需要认真评估——Drizzle 的 schema 定义、查询语法、迁移管理都和 Prisma 完全不同。
- 你需要 MongoDB 支持。Drizzle 只支持关系型数据库（PostgreSQL、MySQL、SQLite），而 TypeORM 支持 MongoDB。

**和直接写 SQL 比呢？** 如果你的项目查询简单、团队只有一两个人，直接用 `postgres` 驱动写 SQL 也没问题。Drizzle 的价值在于当项目复杂度上来之后，纯字符串 SQL 缺乏类型安全，改个字段名很容易漏掉某处查询。Drizzle 让你在保持 SQL 灵活性的同时获得编译时检查。
