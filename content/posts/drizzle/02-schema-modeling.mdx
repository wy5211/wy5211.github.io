上一篇我们搞清楚了为什么选 Drizzle，跑通了第一个查询——插入一条用户记录，再按 email 查出来。但那张 users 表只有五个字段，真实项目的表要复杂得多。字段类型怎么选、默认值怎么设、索引加在哪里、多对多关联怎么建——这些决策直接影响查询性能和数据一致性，搞错了后期改起来代价很大。

今天我们来为博客系统设计完整的数据库 Schema。这个案例会贯穿后续所有文章：用户注册、发布文章、评论互动、标签分类，后面讲 CRUD、关联查询、事务迁移的时候都会用这套表结构。

## 从需求推导表结构

先想清楚博客系统有哪些核心实体：

- **用户（users）**：注册、登录、发文章、写评论
- **文章（posts）**：标题、内容、发布状态、浏览量
- **评论（comments）**：支持嵌套回复（评论下面可以再回复评论）
- **标签（tags）**：一篇文章可以有多个标签，一个标签可以归属多篇文章——典型的多对多关系
- **文章-标签关联表（posts_to_tags）**：多对多的中间表

五张表，关系不算复杂，但已经包含了你会遇到的大部分建模问题：枚举类型、自引用外键（评论的嵌套）、多对多关联、复合索引。

## 先把枚举定义好

PostgreSQL 的枚举类型很适合表示固定的状态值。Drizzle 用 `pgEnum` 来定义，和 SQL 的 `CREATE TYPE ... AS ENUM (...)` 一一对应：

```typescript
// src/schema/enums.ts
import { pgEnum } from "drizzle-orm/pg-core";

export const postStatusEnum = pgEnum("post_status", [
  "DRAFT",
  "PUBLISHED",
  "ARCHIVED",
]);

export const commentStatusEnum = pgEnum("comment_status", [
  "PENDING",
  "APPROVED",
  "SPAM",
]);
```

这里有个设计选择的问题：为什么用 `pgEnum` 而不是 TypeScript 的 `union` 类型加 `varchar` 字段？因为 `pgEnum` 在数据库层面就约束了合法值，你不可能往 `post_status` 列里插入一个 `"DELETED"` 除非先修改枚举定义。而 `varchar` 加 union 类型只在 TypeScript 编译时有效，运行时写错值数据库不会拦你。

但 `pgEnum` 也有个麻烦的地方：PostgreSQL 的枚举类型修改起来比较笨重，加一个值需要 `ALTER TYPE ... ADD VALUE`，不能在事务里执行。如果你的状态值经常变动，用 `varchar` 加应用层校验可能更灵活。博客系统的文章状态基本是固定的，所以这里用 `pgEnum` 没问题。

## 用户表：选对字段类型比你想象的重要

```typescript
// src/schema/users.ts
import {
  pgTable,
  serial,
  varchar,
  text,
  boolean,
  timestamp,
  index,
} from "drizzle-orm/pg-core";

export const users = pgTable(
  "users",
  {
    id: serial("id").primaryKey(),
    email: varchar("email", { length: 255 }).notNull().unique(),
    username: varchar("username", { length: 50 }).notNull().unique(),
    passwordHash: varchar("password_hash", { length: 255 }).notNull(),
    displayName: varchar("display_name", { length: 100 }),
    bio: text("bio"),
    avatarUrl: varchar("avatar_url", { length: 500 }),
    isActive: boolean("is_active").default(true).notNull(),
    createdAt: timestamp("created_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
    updatedAt: timestamp("updated_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
  },
  (table) => ({
    emailIdx: index("users_email_idx").on(table.email),
    usernameIdx: index("users_username_idx").on(table.username),
  })
);
```

几个值得解释的选择：

**为什么 `email` 用 `varchar(255)` 而不是 `text`？** 技术上 `text` 和 `varchar` 在 PostgreSQL 里性能几乎一样（PostgreSQL 不会真的按长度预分配空间）。但 `varchar(255)` 有两个好处：一是在应用层做校验时有个明确的长度边界，二是某些 ORM 和迁移工具在比较 schema 差异时对 `text` 处理不够好。255 是邮箱的标准上限，不是随便拍脑袋的数字。

**为什么 `bio` 用 `text` 不设长度？** 个人简介可长可短，设一个硬上限没什么意义。PostgreSQL 的 `text` 类型最大可以存 1GB，完全够用。

**为什么单独存 `passwordHash` 而不是叫 `password`？** 命名即文档。叫 `password` 容易让人误以为存的是明文（虽然你肯定不会这么做）。叫 `passwordHash` 一眼就知道这是哈希值，新同事接手代码时不会产生误解。

**为什么 `createdAt` 和 `updatedAt` 都加了 `notNull()`？** `defaultNow()` 确保了插入时自动填入时间，但它不阻止你显式传 `null`。加上 `notNull()` 才能在数据库层面彻底杜绝空值。`updatedAt` 也设了 `defaultNow()`，这样插入记录时它也有初始值，后续更新时你手动 `.set({ updatedAt: new Date() })` 就行。

**索引策略：** `email` 和 `username` 都加了索引。这两个字段是登录和用户主页查询的高频条件，不建索引的话，用户量一上去，登录接口就会变慢。注意 `unique()` 约束本身会创建唯一索引，所以额外加的 `emailIdx` 其实是冗余的——这里保留它是为了演示索引语法，生产环境你可以省掉。

## 文章表：索引要跟着查询走

```typescript
// src/schema/posts.ts
import {
  pgTable,
  serial,
  varchar,
  text,
  integer,
  boolean,
  timestamp,
  index,
  unique,
} from "drizzle-orm/pg-core";
import { postStatusEnum } from "./enums";
import { users } from "./users";

export const posts = pgTable(
  "posts",
  {
    id: serial("id").primaryKey(),
    title: varchar("title", { length: 255 }).notNull(),
    slug: varchar("slug", { length: 255 }).notNull(),
    excerpt: varchar("excerpt", { length: 500 }),
    content: text("content").notNull(),
    coverImageUrl: varchar("cover_image_url", { length: 500 }),
    authorId: integer("author_id")
      .notNull()
      .references(() => users.id, { onDelete: "cascade" }),
    status: postStatusEnum("status").default("DRAFT").notNull(),
    viewCount: integer("view_count").default(0).notNull(),
    isFeatured: boolean("is_featured").default(false).notNull(),
    publishedAt: timestamp("published_at", { withTimezone: true }),
    createdAt: timestamp("created_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
    updatedAt: timestamp("updated_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
  },
  (table) => ({
    slugIdx: unique("posts_slug_unique").on(table.slug),
    authorIdx: index("posts_author_idx").on(table.authorId),
    statusIdx: index("posts_status_idx").on(table.status),
    publishedAtIdx: index("posts_published_at_idx").on(table.publishedAt),
    authorStatusIdx: index("posts_author_status_idx").on(
      table.authorId,
      table.status
    ),
  })
);
```

**slug 为什么用 `unique` 约束而不是普通索引？** 文章的 URL 通常是 `/posts/how-to-use-drizzle`，slug 就是 URL 的一部分。如果两条记录有相同的 slug，URL 路由就会冲突。`unique` 约束在数据库层面保证不会出现重复，比应用层校验可靠得多。

**为什么 `authorId` 的外键设了 `onDelete: "cascade"`？** 这意味着删除用户时，该用户的所有文章也会被自动删除。对于博客系统来说这是合理的——用户注销了，文章也没理由留着。但如果你做的是论坛或者内容平台，你可能需要 `onDelete: "set null"` 来保留文章（把作者设为"已注销"），或者 `onDelete: "restrict"` 来阻止删除还有文章的用户。

**复合索引 `authorStatusIdx` 是干什么的？** 博客后台最常见的查询之一是"获取某作者的所有已发布文章"：`WHERE author_id = ? AND status = 'PUBLISHED'`。单独的 `authorIdx` 或 `statusIdx` 都不是最优解，复合索引 `(author_id, status)` 才能让这个查询走索引覆盖。

这里有个坑：复合索引的列顺序很重要。`(author_id, status)` 和 `(status, author_id)` 是不同的索引。原则是"选择性高的列放前面"。如果某个作者有 1000 篇文章，但只有 10% 是 PUBLISHED 状态，那 `(author_id, status)` 更好——先通过 author_id 缩小范围，再在结果里过滤 status。

## 评论表：自引用外键实现嵌套回复

```typescript
// src/schema/comments.ts
import {
  pgTable,
  serial,
  text,
  integer,
  timestamp,
  index,
} from "drizzle-orm/pg-core";
import { commentStatusEnum } from "./enums";
import { users } from "./users";
import { posts } from "./posts";

export const comments = pgTable(
  "comments",
  {
    id: serial("id").primaryKey(),
    postId: integer("post_id")
      .notNull()
      .references(() => posts.id, { onDelete: "cascade" }),
    authorId: integer("author_id").references(() => users.id, {
      onDelete: "set null",
    }),
    parentId: integer("parent_id").references((): any => comments.id, {
      onDelete: "cascade",
    }),
    content: text("content").notNull(),
    status: commentStatusEnum("status").default("PENDING").notNull(),
    createdAt: timestamp("created_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
    updatedAt: timestamp("updated_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
  },
  (table) => ({
    postIdx: index("comments_post_idx").on(table.postId),
    authorIdx: index("comments_author_idx").on(table.authorId),
    parentIdx: index("comments_parent_idx").on(table.parentId),
  })
);
```

这张表有两个值得细说的设计决策。

**自引用外键 `parentId`：** `parentId` 引用的是同一张表的 `id`，这是实现嵌套评论的标准做法。顶级评论的 `parentId` 为 `null`，回复的 `parentId` 指向被回复的评论。查某条评论的所有回复就是 `WHERE parent_id = ?`，简单直接。

这里有个取舍：自引用外键只支持单层父子关系。如果你需要更复杂的树形结构（比如 Reddit 那样的无限嵌套），可以考虑 PostgreSQL 的 `ltree` 扩展或者单独的闭包表。但对于博客评论来说，一两层嵌套足够了，自引用外键是最简单的方案。

**`authorId` 的 `onDelete: "set null"`：** 和文章表不同，评论的作者被删除时，我们选择保留评论但把 `authorId` 设为 `null`。前端显示时如果 `authorId` 为 null，展示"已注销用户"就行。这样做保留了文章的讨论上下文——你肯定见过论坛里"此用户已注销"的评论，就是这个逻辑。

## 标签和关联表：多对多关系

```typescript
// src/schema/tags.ts
import { pgTable, serial, varchar, timestamp } from "drizzle-orm/pg-core";

export const tags = pgTable("tags", {
  id: serial("id").primaryKey(),
  name: varchar("name", { length: 50 }).notNull(),
  slug: varchar("slug", { length: 50 }).notNull(),
  createdAt: timestamp("created_at", { withTimezone: true })
    .defaultNow()
    .notNull(),
});
```

标签表很简单。`name` 存显示名（"TypeScript"），`slug` 存 URL 友好的标识（"typescript"）。

接下来是关键的多对多关联表：

```typescript
// src/schema/posts-to-tags.ts
import {
  pgTable,
  integer,
  timestamp,
  primaryKey,
  index,
} from "drizzle-orm/pg-core";
import { posts } from "./posts";
import { tags } from "./tags";

export const postsToTags = pgTable(
  "posts_to_tags",
  {
    postId: integer("post_id")
      .notNull()
      .references(() => posts.id, { onDelete: "cascade" }),
    tagId: integer("tag_id")
      .notNull()
      .references(() => tags.id, { onDelete: "cascade" }),
    createdAt: timestamp("created_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
  },
  (table) => ({
    pk: primaryKey({ columns: [table.postId, table.tagId] }),
    postIdIdx: index("posts_to_tags_post_idx").on(table.postId),
    tagIdIdx: index("posts_to_tags_tag_idx").on(table.tagId),
  })
);
```

这里有几个设计要点：

**复合主键而不是自增 ID：** 关联表不需要单独的自增 ID。`(post_id, tag_id)` 的组合就是天然的主键——同一篇文章不能重复打同一个标签。用 `primaryKey()` 定义复合主键，在数据库层面就保证了这个约束。

**两边都设了 `onDelete: "cascade"`：** 删除文章时，自动清除关联记录；删除标签时，也是。标签被删了，文章还留着，只是少了那个标签。

**为什么加 `createdAt`？** 多对多关联表加时间戳不是必须的，但知道"这篇文章是什么时候被打上这个标签的"有时候有用——比如做"最近添加的标签"推荐。不加也可以，看你有没有这个需求。

## Drizzle 的 Relations API：让关联查询更自然

到目前为止我们定义的都是表的物理结构（字段、类型、索引、外键）。Drizzle 还有一套 `relations` API，用来描述表之间的逻辑关系。它不影响数据库结构，但能让查询写起来更方便。

```typescript
// src/schema/relations.ts
import { relations } from "drizzle-orm";
import { users } from "./users";
import { posts } from "./posts";
import { comments } from "./comments";
import { tags } from "./tags";
import { postsToTags } from "./posts-to-tags";

export const usersRelations = relations(users, ({ many }) => ({
  posts: many(posts),
  comments: many(comments),
}));

export const postsRelations = relations(posts, ({ one, many }) => ({
  author: one(users, {
    fields: [posts.authorId],
    references: [users.id],
    relationName: "postAuthor",
  }),
  comments: many(comments),
  postsToTags: many(postsToTags),
}));

export const commentsRelations = relations(comments, ({ one }) => ({
  post: one(posts, {
    fields: [comments.postId],
    references: [posts.id],
    relationName: "postComments",
  }),
  author: one(users, {
    fields: [comments.authorId],
    references: [users.id],
    relationName: "commentAuthor",
  }),
  parent: one(comments, {
    fields: [comments.parentId],
    references: [comments.id],
    relationName: "commentReplies",
  }),
}));

export const tagsRelations = relations(tags, ({ many }) => ({
  postsToTags: many(postsToTags),
}));

export const postsToTagsRelations = relations(postsToTags, ({ one }) => ({
  post: one(posts, {
    fields: [postsToTags.postId],
    references: [posts.id],
  }),
  tag: one(tags, {
    fields: [postsToTags.tagId],
    references: [tags.id],
  }),
}));
```

这里最容易搞混的是 `relationName` 参数。什么时候需要它？当两张表之间有多个关系的时候。比如 `users` 和 `posts` 之间是"作者"关系，`users` 和 `comments` 之间也是"作者"关系——虽然语义不同，但外键都指向 `users.id`。Drizzle 需要你用 `relationName` 来区分"这是文章的作者"还是"评论的作者"。不加的话，Drizzle 会报关系歧义的错误。

定义了 relations 之后，你就可以用 relational query API 了：

```typescript
// 查文章时顺便带出作者和评论
const result = await db.query.posts.findMany({
  with: {
    author: true,
    comments: true,
  },
  where: eq(posts.status, "PUBLISHED"),
  limit: 10,
});
```

这种写法类似 Prisma 的 `include`，比手动 `leftJoin` 省事很多。后面讲关联查询的文章会深入讲这个 API。

## 汇总：把所有 Schema 组装到一起

实际项目中，你可以把每张表放在单独的文件里（上面已经这么做了），然后通过一个 `index.ts` 统一导出：

```typescript
// src/schema/index.ts
export * from "./enums";
export * from "./users";
export * from "./posts";
export * from "./comments";
export * from "./tags";
export * from "./posts-to-tags";
export * from "./relations";
```

这样在 `db.ts` 里就可以 `import * as schema from "./schema"` 一行搞定。

## Schema 设计的几个实践经验

**列名和属性名要一致地命名。** Drizzle 允许你把数据库列名和 TypeScript 属性名设成不同的（`displayName: text("display_name")`），这很好。但要在整个项目里保持同一个惯例——列名 `snake_case`，属性名 `camelCase`。不要有的表混着用，有的表又全用 `camelCase`，后面维护的人会疯掉。

**索引不要贪多。** 每个索引都会占用额外的存储空间，并且会在插入和更新时拖慢写入速度。只给真正频繁出现在 `WHERE`、`JOIN`、`ORDER BY` 里的列加索引。上面博客系统的索引策略就是按"跟着查询走"原则设计的。

**`notNull()` 尽可能加上。** JavaScript 里的 `null` 和 `undefined` 已经够让人头疼了，不要让数据库也加入这场混乱。能确定一个字段一定有值的，就加上 `notNull()`。需要允许为空的字段（比如评论的 `parentId`、用户的 `bio`）才不加。

**时间戳用 `withTimezone: true`。** 除非你的应用只在一个时区运行，否则始终存带时区的时间。PostgreSQL 的 `timestamptz` 会自动把时间转成 UTC 存储，查询时按客户端时区返回。不加 `withTimezone` 的 `timestamp` 存的是"本地时间"，跨时区部署时容易出 bug。
