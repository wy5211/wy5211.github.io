## 先想清楚架构再动手

很多人拿到需求就开始写代码，写到一半发现 Schema 要改、API 结构不合理、前后端类型对不上。这次我们换个思路——先花半小时想清楚整体架构，省下的可能是几天的返工时间。

博客系统的核心需求：用户可以注册登录、写文章、给文章打标签、互相评论。管理后台可以审核评论、管理用户。前端要 SSR 做好 SEO，API 要类型安全。

架构上我选 monorepo，原因很简单：Schema 定义、API 类型、工具函数这些前后端都要用的东西，放在一个仓库里共享最方便。用 pnpm workspace 管理，结构如下：

关键设计决策：`packages/database` 是独立的包，前端和 API 都引用它。Schema 只在一个地方定义，前后端共享同一套类型，不会出现"前端猜后端返回什么"的问题。这个思路来自第 2 篇的 Schema 设计和第 6 篇的 tRPC 集成。

## Schema：五张表撑起整个博客

在第 2 篇和第 4 篇里我们详细讲过 Schema 设计和关系建模。这里把完整的 Schema 整合出来：

```typescript
// packages/database/src/schema/users.ts
import { pgTable, serial, varchar, text, timestamp } from "drizzle-orm/pg-core";

export const users = pgTable("users", {
  id: serial("id").primaryKey(),
  email: varchar("email", { length: 255 }).notNull().unique(),
  username: varchar("username", { length: 50 }).notNull().unique(),
  name: text("name"),
  avatar: text("avatar"),
  bio: text("bio"),
  passwordHash: text("password_hash").notNull(),
  role: varchar("role", { length: 20 }).default("reader").notNull(),
  createdAt: timestamp("created_at", { withTimezone: true })
    .defaultNow()
    .notNull(),
});

export type User = typeof users.$inferSelect;
export type NewUser = typeof users.$inferInsert;
```

```typescript
// packages/database/src/schema/posts.ts
import {
  pgTable,
  serial,
  text,
  integer,
  boolean,
  timestamp,
  index,
} from "drizzle-orm/pg-core";
import { users } from "./users";

export const posts = pgTable(
  "posts",
  {
    id: serial("id").primaryKey(),
    title: text("title").notNull(),
    slug: text("slug").notNull().unique(),
    content: text("content").notNull(),
    excerpt: text("excerpt"),
    coverImage: text("cover_image"),
    published: boolean("published").default(false).notNull(),
    authorId: integer("author_id")
      .references(() => users.id, { onDelete: "cascade" })
      .notNull(),
    viewCount: integer("view_count").default(0).notNull(),
    createdAt: timestamp("created_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
    updatedAt: timestamp("updated_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
    publishedAt: timestamp("published_at", { withTimezone: true }),
  },
  (table) => ({
    publishedCreatedIdx: index("posts_published_created_idx").on(
      table.published,
      table.createdAt.desc()
    ),
    authorIdx: index("posts_author_idx").on(table.authorId),
  })
);
```

```typescript
// packages/database/src/schema/comments.ts
import {
  pgTable,
  serial,
  text,
  integer,
  varchar,
  timestamp,
  index,
} from "drizzle-orm/pg-core";
import { users } from "./users";
import { posts } from "./posts";

export const comments = pgTable(
  "comments",
  {
    id: serial("id").primaryKey(),
    content: text("content").notNull(),
    authorId: integer("author_id")
      .references(() => users.id, { onDelete: "cascade" })
      .notNull(),
    postId: integer("post_id")
      .references(() => posts.id, { onDelete: "cascade" })
      .notNull(),
    parentId: integer("parent_id"),
    status: varchar("status", { length: 20 }).default("pending").notNull(),
    createdAt: timestamp("created_at", { withTimezone: true })
      .defaultNow()
      .notNull(),
  },
  (table) => ({
    postIdx: index("comments_post_idx").on(table.postId),
  })
);
```

```typescript
// packages/database/src/schema/tags.ts
import { pgTable, serial, text, integer } from "drizzle-orm/pg-core";
import { posts } from "./posts";

export const tags = pgTable("tags", {
  id: serial("id").primaryKey(),
  name: text("name").notNull(),
  slug: text("slug").notNull().unique(),
});

export const postsToTags = pgTable("posts_to_tags", {
  postId: integer("post_id")
    .references(() => posts.id, { onDelete: "cascade" })
    .notNull(),
  tagId: integer("tag_id")
    .references(() => tags.id, { onDelete: "cascade" })
    .notNull(),
});
```

五张表的关系：users 一对多 posts，posts 一对多 comments，comments 自引用（parentId）实现嵌套回复，posts 和 tags 通过 postsToTags 多对多关联。索引策略来自第 9 篇的性能优化经验——给高频查询的排序字段和过滤字段加索引。

## 数据库客户端：按环境切换驱动

第 8 篇我们讨论过 Edge 环境的特殊性。数据库客户端需要根据运行环境选择不同的驱动：

```typescript
// packages/database/src/client.ts
import * as schema from "./schema";

function createDb() {
  // Edge 环境（Cloudflare Workers、Vercel Edge）
  if (process.env.DATABASE_DRIVER === "neon-http") {
    const { drizzle } = require("drizzle-orm/neon-http");
    const { neon } = require("@neondatabase/serverless");
    return drizzle(neon(process.env.DATABASE_URL!), { schema });
  }

  // Node.js 环境（Next.js SSR、传统服务器）
  const { drizzle } = require("drizzle-orm/node-postgres");
  const { Pool } = require("pg");
  const pool = new Pool({
    connectionString: process.env.DATABASE_URL,
    max: 10,
    idleTimeoutMillis: 20000,
  });
  return drizzle(pool, { schema });
}

export const db = createDb();
export { schema };
```

这里用条件导入区分环境。Edge 环境用 neon-http 无状态连接（第 8 篇详细讲过），Node.js 环境用 pg 连接池（第 9 篇调过参数）。

## tRPC Router：把 Schema 变成类型安全的 API

第 6 篇我们搭建了 tRPC + Drizzle 的类型安全管道。这里把它应用到博客系统的核心 API 上。

文章 Router 是最复杂的部分，涵盖列表查询（分页+过滤）、详情查询（关联作者和标签）、创建（事务写入多张表）、更新和删除：

```typescript
// apps/api/src/routers/posts.ts
import { z } from "zod";
import { eq, desc, and, sql } from "drizzle-orm";
import { posts, users, tags, postsToTags, comments } from "@blog/database";
import { t } from "../trpc";

export const postsRouter = t.router({
  // 首页文章列表——只查需要的字段，走索引
  list: t.procedure
    .input(
      z.object({
        page: z.number().min(1).default(1),
        limit: z.number().min(1).max(50).default(10),
        tag: z.string().optional(),
      })
    )
    .query(async ({ input, ctx }) => {
      const { page, limit, tag } = input;

      // 基础查询：只查列表需要的字段
      const baseQuery = ctx.db
        .select({
          id: posts.id,
          title: posts.title,
          slug: posts.slug,
          excerpt: posts.excerpt,
          coverImage: posts.coverImage,
          createdAt: posts.createdAt,
          author: { id: users.id, name: users.name, avatar: users.avatar },
        })
        .from(posts)
        .innerJoin(users, eq(posts.authorId, users.id))
        .where(eq(posts.published, true));

      // 标签过滤
      if (tag) {
        const [tagRecord] = await ctx.db
          .select({ id: tags.id })
          .from(tags)
          .where(eq(tags.slug, tag))
          .limit(1);

        if (!tagRecord)
          return { items: [], total: 0, page, limit, totalPages: 0 };
      }

      const items = await baseQuery
        .orderBy(desc(posts.createdAt))
        .limit(limit)
        .offset((page - 1) * limit);

      // 用 "多查一条" 代替 COUNT 查询（第 9 篇的技巧）
      const countCheck = await baseQuery
        .limit(limit + 1)
        .offset((page - 1) * limit);
      const hasMore = countCheck.length > limit;

      return {
        items: items.slice(0, limit),
        hasMore,
        page,
        limit,
      };
    }),

  // 文章详情——关联作者、标签、评论
  bySlug: t.procedure
    .input(z.object({ slug: z.string() }))
    .query(async ({ input, ctx }) => {
      const [post] = await ctx.db
        .select({
          id: posts.id,
          title: posts.title,
          content: posts.content,
          coverImage: posts.coverImage,
          createdAt: posts.createdAt,
          publishedAt: posts.publishedAt,
          author: {
            id: users.id,
            name: users.name,
            avatar: users.avatar,
            bio: users.bio,
          },
        })
        .from(posts)
        .innerJoin(users, eq(posts.authorId, users.id))
        .where(and(eq(posts.slug, input.slug), eq(posts.published, true)))
        .limit(1);

      if (!post) return null;

      // 标签
      const postTags = await ctx.db
        .select({ id: tags.id, name: tags.name, slug: tags.slug })
        .from(tags)
        .innerJoin(postsToTags, eq(tags.id, postsToTags.tagId))
        .where(eq(postsToTags.postId, post.id));

      // 评论（带作者，避免 N+1）
      const postComments = await ctx.db
        .select({
          id: comments.id,
          content: comments.content,
          createdAt: comments.createdAt,
          parentId: comments.parentId,
          author: { id: users.id, name: users.name, avatar: users.avatar },
        })
        .from(comments)
        .innerJoin(users, eq(comments.authorId, users.id))
        .where(
          and(eq(comments.postId, post.id), eq(comments.status, "approved"))
        )
        .orderBy(desc(comments.createdAt));

      // 异步增加浏览量，不阻塞响应
      ctx.db
        .update(posts)
        .set({ viewCount: sql`${posts.viewCount} + 1` })
        .where(eq(posts.id, post.id))
        .then(() => {})
        .catch(() => {});

      return { ...post, tags: postTags, comments: postComments };
    }),

  // 创建文章——事务保证原子性
  create: t.procedure
    .input(
      z.object({
        title: z.string().min(1).max(200),
        content: z.string().min(1),
        excerpt: z.string().max(500).optional(),
        tagIds: z.array(z.number()).optional(),
      })
    )
    .mutation(async ({ input, ctx }) => {
      if (!ctx.userId) throw new Error("未登录");

      const slug = input.title
        .toLowerCase()
        .replace(/[^\w\s-]/g, "")
        .replace(/\s+/g, "-")
        .slice(0, 100);

      return ctx.db.transaction(async (tx) => {
        // 第一步：插入文章
        const [post] = await tx
          .insert(posts)
          .values({
            title: input.title,
            slug,
            content: input.content,
            excerpt: input.excerpt,
            authorId: ctx.userId,
          })
          .returning();

        // 第二步：关联标签
        if (input.tagIds?.length) {
          await tx
            .insert(postsToTags)
            .values(input.tagIds.map((tagId) => ({ postId: post.id, tagId })));
        }

        return post;
      });
    }),
});
```

注意几个细节：列表查询只取需要的字段（不用 `SELECT *`），评论查询用 JOIN 避免 N+1（第 9 篇的经验），创建文章用事务包裹（第 5 篇的原子操作），浏览量更新是异步的不阻塞响应。

## 前端：Next.js App Router + Server Components

第 7 篇我们讲过 Next.js 和 Drizzle 的集成。博客前端充分利用 Server Components，首页和文章详情在服务端渲染，SEO 友好且首屏快。

```typescript
// apps/web/app/page.tsx
import { db } from "@blog/database/client";
import { posts, users } from "@blog/database";
import { eq, desc } from "drizzle-orm";
import Link from "next/link";

export const revalidate = 300; // ISR：5 分钟重新验证

export default async function HomePage() {
  const latestPosts = await db
    .select({
      id: posts.id,
      title: posts.title,
      slug: posts.slug,
      excerpt: posts.excerpt,
      coverImage: posts.coverImage,
      createdAt: posts.createdAt,
      author: { name: users.name, avatar: users.avatar },
    })
    .from(posts)
    .innerJoin(users, eq(posts.authorId, users.id))
    .where(eq(posts.published, true))
    .orderBy(desc(posts.createdAt))
    .limit(12);

  return (
    <main>
      <section>
        <h1>Blog</h1>
        <div>
          {latestPosts.map((post) => (
            <article key={post.id}>
              {post.coverImage && <img src={post.coverImage} alt={post.title} />}
              <h2>
                <Link href={`/blog/${post.slug}`}>{post.title}</Link>
              </h2>
              <p>{post.excerpt}</p>
              <span>{post.author.name}</span>
              <time>{post.createdAt.toLocaleDateString("zh-CN")}</time>
            </article>
          ))}
        </div>
      </section>
    </main>
  );
}
```

`export const revalidate = 300` 让 Next.js 做 ISR（增量静态再生），首页每 5 分钟重新生成一次。结合第 9 篇的缓存策略，大部分请求不会打到数据库。

文章详情页类似，多一步查标签和评论：

```typescript
// apps/web/app/blog/[slug]/page.tsx
import { db } from "@blog/database/client";
import { posts, users, tags, postsToTags } from "@blog/database";
import { eq } from "drizzle-orm";
import { notFound } from "next/navigation";

export default async function PostPage({ params }: { params: { slug: string } }) {
  const [post] = await db
    .select({
      id: posts.id,
      title: posts.title,
      content: posts.content,
      createdAt: posts.createdAt,
      author: { id: users.id, name: users.name, avatar: users.avatar, bio: users.bio },
    })
    .from(posts)
    .innerJoin(users, eq(posts.authorId, users.id))
    .where(eq(posts.slug, params.slug))
    .limit(1);

  if (!post) notFound();

  const postTags = await db
    .select({ name: tags.name, slug: tags.slug })
    .from(tags)
    .innerJoin(postsToTags, eq(tags.id, postsToTags.tagId))
    .where(eq(postsToTags.postId, post.id));

  return (
    <article>
      <h1>{post.title}</h1>
      <div>
        <img src={post.author.avatar} alt={post.author.name} />
        <span>{post.author.name}</span>
        <time>{post.createdAt.toLocaleDateString("zh-CN")}</time>
      </div>
      <div>
        {postTags.map((t) => (
          <span key={t.slug}>{t.name}</span>
        ))}
      </div>
      <div dangerouslySetInnerHTML={{ __html: post.content }} />
    </article>
  );
}
```

评论部分用客户端组件，通过 tRPC 发起交互请求（登录、发表评论）。

## 部署：从开发到生产

开发阶段用 Docker Compose 跑 PostgreSQL，`pnpm dev` 启动前后端。生产部署有两种主流方案。

### 方案一：Vercel + Neon（推荐新手）

前端部署到 Vercel，数据库用 Neon（Serverless PostgreSQL）。好处是零运维、自动扩缩容。Drizzle 通过 `neon-http` 驱动连接 Neon，每次请求是无状态的 HTTP 调用，天然适配 Serverless。

```bash
# 构建和部署
pnpm build
vercel deploy --prod
```

数据库迁移：

```bash
cd packages/database
npx drizzle-kit generate    # 生成迁移文件
npx drizzle-kit migrate     # 应用到 Neon
```

### 方案二：Cloudflare Workers + D1（适合 Edge 场景）

如果追求全球低延迟，API 部署到 Cloudflare Workers，数据库用 D1（SQLite at Edge）。第 8 篇详细讲过这个方案。

```bash
# 部署 Worker
cd apps/api
wrangler d1 migrations apply blog-db
wrangler deploy
```

前端继续部署到 Vercel，API 调用指向 Workers 的域名。这种混合架构在全球范围都能做到 50ms 以内的响应。

### 方案对比

Vercel + Neon 更简单，适合个人博客和小团队。Workers + D1 性能上限更高，但需要处理 SQLite 的兼容性问题（比如 D1 不支持完整的外键约束）。选哪个取决于你的流量预期和运维能力。

## 系列回顾

十篇文章，我们从零开始搭建了一个完整的博客系统。回顾一下每篇解决的核心问题：

第一篇介绍了 Drizzle ORM 是什么、为什么选它——轻量、类型安全、Edge 友好。第二篇用纯 TypeScript 定义了 users、posts、comments、tags 五张表的 Schema。第三篇掌握了 CRUD 操作和查询构建器。第四篇解决了表之间的关联查询——JOIN、一对一、一对多、多对多。第五篇用事务保证多步操作的原子性，用 Drizzle Kit 管理数据库迁移。第六篇把 Drizzle 和 tRPC 结合起来，实现了从数据库到前端的端到端类型安全。第七篇深入 Next.js 和 Nuxt 的集成，用 Server Components 做 SSR。第八篇探索了 Edge Computing 环境——Cloudflare Workers、D1、Neon。第九篇从真实性能问题出发，解决了 N+1 查询、索引优化、连接池调优、缓存策略。今天这篇把所有知识整合成了一个可部署的完整项目。

一条线串下来就是：定义数据模型 -> 操作数据 -> 关联查询 -> 保证一致性 -> 类型安全 API -> 前端渲染 -> Edge 部署 -> 性能优化 -> 项目整合。

## 进阶方向

这个系列覆盖了 Drizzle ORM 的核心用法，但还有一些值得深入的方向：

**Drizzle Studio** 是 Drizzle 官方的数据库管理 GUI，类似 pgAdmin 但更轻量。直接在浏览器里查看和编辑数据，开发调试时很方便。用 `npx drizzle-kit studio` 就能启动。

**Drizzle Kit 高级用法**：除了 `generate` 和 `migrate`，Drizzle Kit 还支持 `push`（直接推送 Schema 变更到数据库，跳过迁移文件）、`check`（检查迁移文件是否有冲突）、`up`（重放迁移）。生产环境建议走 `generate + migrate` 的正规流程，`push` 只在开发原型阶段用。

**多数据库支持**：我们这个系列主要用 PostgreSQL，但 Drizzle 的 API 是数据库无关的。同样的 Schema 定义，换个驱动就能跑在 MySQL、SQLite、Turso（libSQL）、PlanetScale（MySQL 兼容）上。迁移到 Supabase 也很自然——Supabase 底层就是 PostgreSQL，Drizzle 可以直接连。

**从 Prisma 迁移**：如果你有现成的 Prisma 项目，Drizzle 官方提供了 `prisma-generator-drizzle` 工具，可以自动把 Prisma Schema 转成 Drizzle Schema。但迁移后需要重新审视查询方式——Prisma 的 `include` 和 Drizzle 的 `with`（Relational Query API）语义不同，不是一一对应的。

**实时订阅**：Drizzle 本身不支持实时订阅（类似 Prisma Pulse 或 Supabase Realtime），但可以结合 PostgreSQL 的 LISTEN/NOTIFY 机制自己实现。或者用 Neon 的 WebSocket 实时功能。

## Trade-off 分析

**Monorepo vs 多仓库**：Monorepo 共享类型方便，但构建配置更复杂。如果你的团队只有 2-3 个人、项目不复杂，Monorepo 的收益远大于成本。如果团队大了、CI 时间变长，可以考虑用 Turborepo 或 Nx 优化构建。

**Serverless vs 传统服务器**：Serverless（Vercel + Neon）运维成本低，按请求付费，但冷启动和数据库连接是永远的痛点。传统服务器（VPS + PostgreSQL）性能更稳定，但需要自己管运维。博客系统流量波动大（白天高峰、夜间低谷），Serverless 的弹性扩缩容优势明显。

**SQLite (D1) vs PostgreSQL (Neon)**：SQLite 简单、快、零运维，但不支持完整的关系型特性（比如复杂 JOIN 优化、窗口函数）。PostgreSQL 功能完整但运维成本更高。选哪个取决于你的数据模型复杂度。博客系统的 Schema 不算复杂，D1 完全够用；但如果要做复杂的数据分析（比如用户行为统计），PostgreSQL 更合适。

**类型安全 vs 开发速度**：Drizzle + tRPC 的类型安全管道前期搭建成本高（定义 Schema、写 Router、配置类型），但后期维护成本低——改一个字段类型，编译器会在所有引用处报错。如果你的项目是一次性的或者需要快速验证想法，可以先用简单方案，等模型稳定了再补类型安全。

这个系列到这里就结束了。希望你能把这套思路应用到自己的项目里——不只是博客系统，任何需要数据库访问的 TypeScript 项目，Drizzle 都是一个值得认真考虑的选择。
