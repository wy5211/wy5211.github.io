上一篇我们搞定了事务和迁移，博客系统的数据库层已经完整了。但还有一个很现实的问题：**数据怎么从数据库流到前端？**

最直觉的做法是写 REST API 路由。但很快你就会发现几个痛点：请求参数用什么校验？返回值的类型谁来维护？数据库 Schema 加了个字段，后端返回了新数据，前端 TypeScript 还是旧类型——编译不报错，运行时拿不到新字段，排查半天才发现是类型没同步。

Drizzle + tRPC 是目前 TypeScript 全栈开发中解决这类问题最干净的方案。Drizzle 在数据库层保证了类型安全，tRPC 在 API 层保证类型从后端穿透到前端。两者结合，改了数据库 Schema，类型错误会一路传递到组件 props。

## 为什么是 tRPC 而不是 REST

先说清楚 tRPC 不是什么——它不是 REST 的替代品，不是 GraphQL 的竞争者。它是一个 **TypeScript RPC 框架**，核心就一句话：**后端定义的函数签名，前端能直接推断出参数和返回值类型**。

写 REST API 时的典型流程：

1. 后端定义路由 `GET /api/posts/:id`
2. 手写 Zod schema 校验 query 参数
3. 手写返回值类型 `interface PostResponse { ... }`
4. 前端用 axios/fetch 调用，手写类型断言 `as PostResponse`
5. 数据库 Schema 变了，上面四步全部要手动同步

tRPC 的流程：

1. 后端写一个函数，参数和返回值的类型是 Drizzle 自动推断的
2. 前端直接调用这个函数，TypeScript 自动知道参数和返回值
3. 完了

没有手写类型、没有类型断言、没有同步负担。

## 项目结构

Drizzle + tRPC 的项目推荐用 monorepo，把共享的 Schema 和类型抽到独立包里。不过也不一定要用 turborepo 这种重型方案——简单的目录分层就够了。核心思路是**数据库 Schema 和 tRPC Router 在同一个 TypeScript 项目里**，类型通过 import 直接共享：

```
src/
  db/
    schema/          # Drizzle Schema（第 2 篇定义的那些表）
      users.ts
      posts.ts
      tags.ts
      index.ts
    index.ts         # db 实例
  trpc/
    context.ts       # tRPC 上下文（注入 db 和用户信息）
    router.ts        # 合并所有子路由
    trpc.ts          # tRPC 初始化
  routers/
    posts.ts         # 文章相关 API
    comments.ts      # 评论相关 API
    auth.ts          # 认证相关 API
```

关键点：`db/schema/` 和 `routers/` 在同一个项目里，Router 可以直接 import Drizzle 的表定义和推断类型。不需要 monorepo 也能获得完整的类型安全。

## tRPC 后端搭建

### 初始化 tRPC 实例

先装依赖：

```bash
pnpm add @trpc/server zod
```

然后创建 tRPC 实例。这里会把 Drizzle 的 db 实例注入到上下文里，这样每个 Router 都能通过 `ctx.db` 访问数据库：

```typescript
// src/trpc/trpc.ts
import { initTRPC, TRPCError } from "@trpc/server";
import { db } from "../db";

// 上下文定义：每个请求都会携带这些信息
export interface Context {
  db: typeof db;
  userId?: number; // 登录后才有
}

const t = initTRPC.context<Context>().create();

export const router = t.router;
export const publicProcedure = t.procedure;

// 受认证保护的 procedure，未登录会直接报错
export const protectedProcedure = t.procedure.use(async (opts) => {
  if (!opts.ctx.userId) {
    throw new TRPCError({ code: "UNAUTHORIZED" });
  }
  return opts.next({ ctx: { ...opts.ctx, userId: opts.ctx.userId } });
});
```

`publicProcedure` 是公开接口，`protectedProcedure` 要求用户必须登录。后面的 Router 直接用这两个来定义 API。

### 创建上下文

```typescript
// src/trpc/context.ts
import type { Context } from "./trpc";

export async function createContext(opts: { req: Request }): Promise<Context> {
  // 从请求头中提取 session token，验证后拿到 userId
  const session = await verifySession(req.headers.get("cookie"));

  return {
    db, // Drizzle 实例，直接注入
    userId: session?.userId,
  };
}
```

在实际项目中，session 验证可能用 next-auth、lucia 或自己写的 JWT 中间件。核心就是从请求里拿到 userId，塞进 context。

### 从 Drizzle Schema 推导 Zod 验证

这是 Drizzle + tRPC 最优雅的部分。Drizzle 的 `drizzle-zod` 包可以直接从表定义推导出 Zod schema，不需要手写：

```bash
pnpm add drizzle-zod
```

```typescript
// src/routers/posts.ts
import { createInsertSchema, createSelectSchema } from "drizzle-zod";
import { posts, postsToTags, tags } from "../db/schema";
import { router, publicProcedure, protectedProcedure } from "../trpc/trpc";
import { eq, and, desc, sql } from "drizzle-orm";
import { z } from "zod";

// 从 Drizzle 表定义直接生成 Zod schema
const insertPostSchema = createInsertSchema(posts);

// 也可以覆盖某些字段的验证规则
const publishPostSchema = createInsertSchema(posts, {
  title: z.string().min(1).max(200),
  content: z.string().min(10),
});
```

`createInsertSchema(posts)` 会根据 `posts` 表的定义，自动生成一个 Zod schema——`title` 是必填的 string，`content` 是必填的 string，`id` 是自动生成的所以变成了 optional。你在 Schema 里定义的 `.notNull()`、`.default()` 约束，全部自动映射到 Zod 验证规则。

### 实现文章 Router

有了上面的基础设施，实现 CRUD 就很直接了：

```typescript
// src/routers/posts.ts（续）
export const postsRouter = router({
  // 获取文章列表（公开）
  list: publicProcedure
    .input(
      z.object({
        page: z.number().min(1).default(1),
        limit: z.number().min(1).max(50).default(10),
      })
    )
    .query(async ({ input, ctx }) => {
      const { page, limit } = input;
      const offset = (page - 1) * limit;

      const [items, [{ count }]] = await Promise.all([
        ctx.db
          .select({
            id: posts.id,
            title: posts.title,
            excerpt: posts.excerpt,
            createdAt: posts.createdAt,
            authorName: users.name,
          })
          .from(posts)
          .innerJoin(users, eq(posts.authorId, users.id))
          .orderBy(desc(posts.createdAt))
          .limit(limit)
          .offset(offset),
        ctx.db.select({ count: sql<number>`count(*)` }).from(posts),
      ]);

      return { items, total: Number(count), page, limit };
    }),

  // 获取文章详情（公开）
  byId: publicProcedure
    .input(z.object({ id: z.number() }))
    .query(async ({ input, ctx }) => {
      const [post] = await ctx.db
        .select()
        .from(posts)
        .where(eq(posts.id, input.id))
        .limit(1);

      if (!post) {
        throw new TRPCError({ code: "NOT_FOUND", message: "文章不存在" });
      }

      // 获取关联的标签
      const postTags = await ctx.db
        .select({ id: tags.id, name: tags.name })
        .from(postsToTags)
        .innerJoin(tags, eq(postsToTags.tagId, tags.id))
        .where(eq(postsToTags.postId, post.id));

      return { ...post, tags: postTags };
    }),

  // 发布文章（需登录）——复用上一篇的事务逻辑
  publish: protectedProcedure
    .input(
      publishPostSchema.pick({ title: true, content: true }).extend({
        tagNames: z.array(z.string()).min(1),
      })
    )
    .mutation(async ({ input, ctx }) => {
      return await ctx.db.transaction(async (tx) => {
        const [post] = await tx
          .insert(posts)
          .values({
            title: input.title,
            content: input.content,
            authorId: ctx.userId!,
            slug: generateSlug(input.title),
          })
          .returning();

        for (const tagName of input.tagNames) {
          const [tag] = await tx
            .insert(tags)
            .values({ name: tagName, slug: generateSlug(tagName) })
            .onConflictDoNothing()
            .returning();

          const actualTag =
            tag ??
            (
              await tx
                .select()
                .from(tags)
                .where(eq(tags.name, tagName))
                .limit(1)
            )[0];

          await tx.insert(postsToTags).values({
            postId: post.id,
            tagId: actualTag.id,
          });
        }

        return post;
      });
    }),

  // 删除文章（需登录，且是文章作者）
  delete: protectedProcedure
    .input(z.object({ id: z.number() }))
    .mutation(async ({ input, ctx }) => {
      const [post] = await ctx.db
        .select({ authorId: posts.authorId })
        .from(posts)
        .where(eq(posts.id, input.id))
        .limit(1);

      if (!post) {
        throw new TRPCError({ code: "NOT_FOUND" });
      }
      if (post.authorId !== ctx.userId) {
        throw new TRPCError({
          code: "FORBIDDEN",
          message: "只能删除自己的文章",
        });
      }

      await ctx.db.delete(posts).where(eq(posts.id, input.id));
      return { success: true };
    }),
});
```

注意几个设计点：

- `list` 和 `byId` 是 `.query()`——幂等查询，适合用 GET 缓存
- `publish` 和 `delete` 是 `.mutation()`——有副作用的操作
- `publish` 里复用了上一篇写的事务逻辑，直接从 tRPC mutation 调用
- `publishPostSchema.pick()` 从 Drizzle 推导的 schema 中选取字段，再 `.extend()` 加上自定义的 `tagNames`——不需要从头写验证

### 评论 Router

评论的场景更简单，展示一下嵌套路由的基本模式：

```typescript
// src/routers/comments.ts
export const commentsRouter = router({
  // 获取某篇文章的评论（公开）
  byPost: publicProcedure
    .input(z.object({ postId: z.number() }))
    .query(async ({ input, ctx }) => {
      return await ctx.db
        .select({
          id: comments.id,
          content: comments.content,
          createdAt: comments.createdAt,
          authorName: users.name,
        })
        .from(comments)
        .innerJoin(users, eq(comments.authorId, users.id))
        .where(eq(comments.postId, input.postId))
        .orderBy(desc(comments.createdAt));
    }),

  // 发表评论（需登录）
  create: protectedProcedure
    .input(
      z.object({
        postId: z.number(),
        content: z.string().min(1).max(1000),
      })
    )
    .mutation(async ({ input, ctx }) => {
      const [comment] = await ctx.db
        .insert(comments)
        .values({
          postId: input.postId,
          authorId: ctx.userId!,
          content: input.content,
        })
        .returning();

      return comment;
    }),
});
```

### 合并路由

```typescript
// src/trpc/router.ts
import { router } from "./trpc";
import { postsRouter } from "../routers/posts";
import { commentsRouter } from "../routers/comments";

export const appRouter = router({
  posts: postsRouter,
  comments: commentsRouter,
});

// 这个类型是整个方案的核心——前端通过 import 它获得所有 API 的类型
export type AppRouter = typeof appRouter;
```

`AppRouter` 这个类型导出是关键。前端拿到它之后，所有 tRPC 调用的参数和返回值都能自动推断。改了后端的任何返回值，前端的 TypeScript 立刻报错。

### 挂载到 HTTP 服务

如果你用 Next.js，这一步非常简单：

```typescript
// src/app/api/trpc/[trpc]/route.ts (App Router)
import { fetchRequestHandler } from "@trpc/server/adapters/fetch";
import { appRouter } from "@/trpc/router";
import { createContext } from "@/trpc/context";

export function GET(req: Request) {
  return fetchRequestHandler({
    endpoint: "/api/trpc",
    req,
    router: appRouter,
    createContext,
  });
}

export { GET as POST };
```

如果是独立 Node 服务（Hono、Express），也有对应的 adapter。

## 前端：类型自动穿透

### 创建 tRPC 客户端

```bash
pnpm add @trpc/client @trpc/react-query @tanstack/react-query
```

```typescript
// src/lib/trpc.ts
import { createTRPCReact } from "@trpc/react-query";
import type { AppRouter } from "@/trpc/router";

export const trpc = createTRPCReact<AppRouter>();
```

`AppRouter` 从后端 import 过来（monorepo 或者同一个 TypeScript 项目），`createTRPCReact` 会根据它生成一个完全类型安全的客户端。就这么一行，所有的类型推断就开始工作了。

### 在组件里调用

```typescript
// 文章列表组件
function PostList() {
  const { data, isLoading } = trpc.posts.list.useQuery({ page: 1, limit: 10 });

  if (isLoading) return <div>加载中...</div>;

  return (
    <ul>
      {data?.items.map((post) => (
        <li key={post.id}>
          <h3>{post.title}</h3>
          <span>by {post.authorName}</span>
        </li>
      ))}
    </ul>
  );
}
```

`data.items` 的类型是后端 `posts.list` 返回值自动推断的。`post.title` 是 `string`，`post.authorName` 是 `string | null`——这些类型全部来自 Drizzle Schema 定义，经过 tRPC Router，直接到达 React 组件。中间没有任何手写类型。

如果你在 Schema 里给 `users.name` 加了 `.notNull()`，Drizzle 推断的类型会变成 `string`，tRPC 自动把这个变更传递到前端，`post.authorName` 自动变成 `string`——你不需要改任何前端代码。

### Mutation 调用

```typescript
function PublishPost() {
  const utils = trpc.useUtils();

  const publish = trpc.posts.publish.useMutation({
    onSuccess: () => {
      // 发布成功后刷新列表
      utils.posts.list.invalidate();
    },
  });

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    publish.mutate({
      title: "Drizzle + tRPC 实战",
      content: "从数据库到前端，全程类型安全...",
      tagNames: ["TypeScript", "tRPC"],
    });
  };

  return <form onSubmit={handleSubmit}>...</form>;
}
```

`publish.mutate()` 的参数类型来自 `publishPostSchema`——Drizzle 的表定义推导出来的 Zod schema。传错了字段名或类型，编辑器直接标红。

## 错误处理

tRPC 的错误处理有一个很好的设计：后端 throw 的 `TRPCError` 会带着结构化的错误码传到前端，前端可以根据 code 做不同的处理：

```typescript
// 后端
if (!post) {
  throw new TRPCError({ code: "NOT_FOUND", message: "文章不存在" });
}
if (post.authorId !== ctx.userId) {
  throw new TRPCError({ code: "FORBIDDEN", message: "没有权限" });
}

// 前端
const { data, error } = trpc.posts.byId.useQuery({ id: postId });

if (error?.data?.code === "NOT_FOUND") {
  return <div>文章不存在</div>;
}
if (error?.data?.code === "FORBIDDEN") {
  return <div>没有权限查看</div>;
}
```

Zod 验证失败也是结构化的。如果你在 tRPC 初始化时配置了 `errorFormatter`，前端能拿到完整的字段级错误信息：

```typescript
import { ZodError } from "zod";

const t = initTRPC.context<Context>().create({
  errorFormatter({ shape, error }) {
    return {
      ...shape,
      data: {
        ...shape.data,
        zodError:
          error.cause instanceof ZodError ? error.cause.flatten() : null,
      },
    };
  },
});
```

前端拿到 `error.data.zodError`，就能精确知道哪个字段校验失败了，直接在表单上标红对应的输入框。

## 和纯 REST API 的对比

做完这些之后，回头看看如果用 REST API 写同样的功能：

| 方面            | REST + 手写类型                | Drizzle + tRPC                    |
| --------------- | ------------------------------ | --------------------------------- |
| 请求参数校验    | 手写 Zod schema 或手动校验     | `createInsertSchema` 从表定义推导 |
| 返回值类型      | 手写 interface，容易和实际脱节 | Drizzle 自动推断，tRPC 自动传递   |
| Schema 变更同步 | 手动更新后端类型 + 前端类型    | 改一处，全链路自动更新            |
| 错误处理        | 自定义错误码和消息格式         | 结构化的 TRPCError + ZodError     |
| 前端调用        | fetch/axios + 类型断言         | 类型安全的 hooks，零断言          |
| Bundle 大小     | 取决于请求库                   | tRPC client 极小，无运行时开销    |

代价是有的：tRPC 要求前后端都是 TypeScript。如果你的后端是 Go 或者前端要支持非 TS 项目，REST 仍然是更通用的选择。另外 tRPC 的学习曲线虽然不陡，但团队每个人都要理解它的概念。

## Trade-off 分析

**Drizzle + tRPC 适合什么项目？** 前后端都用 TypeScript 的全栈项目，尤其是中小型团队——一个人维护前后端时，类型自动同步的价值最大。改了数据库 Schema，TypeScript 的类型错误会引导你修复所有受影响的地方。

**不适合什么场景？** 后端不是 TypeScript（那就用不了 tRPC 的类型穿透）、需要公开 API 给第三方（REST 或 GraphQL 更合适）、团队对 tRPC 完全陌生且没有学习时间。

**drizzle-zod 的 schema 推导够用吗？** 简单的 CRUD 场景完全够用。但如果你有复杂的业务校验（比如"标题不能和已有文章重复"），还是需要手写 Zod schema。好消息是可以从 `createInsertSchema` 出发，用 `.extend()` 和 `.refine()` 叠加自定义逻辑，不用从头写。
