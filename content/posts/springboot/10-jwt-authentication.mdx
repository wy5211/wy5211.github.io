上一篇搭好了 Spring Security 的骨架——`SecurityFilterChain` 定义了路径权限，`UserDetailsService` 告诉框架用户数据从哪来，`PasswordEncoder` 强制密码哈希。但有一个关键环节缺失：**用户登录成功后，怎么拿到凭证？后续请求怎么证明身份？**

在 Express 里你的做法很直觉：装 `jsonwebtoken`，登录接口里 `jwt.sign()` 生成 token 返回给前端，后续请求在 `Authorization: Bearer <token>` 头里带上，写一个 `express-jwt` 中间件验证 token 并挂载 `req.user`。整个流程拆成三个独立的部分——生成、验证、挂载——中间用 JWT 这个字符串连接。

Spring Security + JWT 做的事情完全一样，但组织方式不同。Spring Security 没有一个"登录成功后自动生成 JWT"的开关，它提供的是过滤器链的插槽——你需要在合适的环节插入自己的 JWT 逻辑。理解这个差异，是从 Express 转到 Spring Security JWT 的核心认知。

## JWT 的本质：一个自包含的凭证

先统一概念。JWT 由三段 Base64URL 编码的字符串用 `.` 连接而成：

```
Header.Payload.Signature
```

- **Header**：算法和类型，`{"alg":"HS256","typ":"JWT"}`
- **Payload**：声明（claims），比如 `sub`（主题/用户名）、`iat`（签发时间）、`exp`（过期时间）
- **Signature**：用密钥对 Header + Payload 的签名，防止篡改

JWT 是自包含的——它不依赖服务端存储，验证 token 只需要密钥，不需要查数据库（除非你要做 token 吊销）。这也是为什么它适合前后端分离和无状态场景。

Express 和 Spring Boot 用到的 JWT 库：

| 生态    | 核心库                 | 作用           |
| ------- | ---------------------- | -------------- |
| Node.js | `jsonwebtoken`         | 生成 + 验证    |
| Node.js | `express-jwt`          | 中间件自动验证 |
| Java    | `jjwt`                 | 生成 + 验证    |
| Java    | Spring Security Filter | 接入过滤器链   |

`jjwt` 是 Java 生态最主流的 JWT 库（由 Okta 维护），和 `jsonwebtoken` 在 Node.js 生态的地位完全对等。两者的 API 设计也很相似——都是 Builder 模式生成 token，解析器模式验证 token。

## 依赖引入

`jjwt` 从 0.12.x 开始拆成了三个模块，你需要引入 API、实现和 Jackson 集成：

```xml
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>
    <version>0.12.6</version>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>
    <version>0.12.6</version>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-jackson</artifactId>
    <version>0.12.6</version>
    <scope>runtime</scope>
</dependency>
```

`jjwt-api` 是编译期依赖（你调用的接口在这里），`jjwt-impl` 和 `jjwt-jackson` 是运行时依赖。这个拆分和 `jsonwebtoken` 的单包发布形成对比——`jsonwebtoken` 一个包包含所有功能，`jjwt` 把接口和实现分开，让你在测试时可以 mock API 层。

## JwtUtil：Token 的生成与验证

JWT 的生成和验证逻辑集中在一个工具类里。它不依赖 Spring Security 的任何类，是一个纯粹的 JWT 操作工具：

```java
@Component
public class JwtUtil {

    private final SecretKey key;
    private final long accessTokenExpiration;
    private final long refreshTokenExpiration;

    public JwtUtil(
            @Value("${jwt.secret}") String secret,
            @Value("${jwt.access-token-expiration:3600000}") long accessTokenExpiration,
            @Value("${jwt.refresh-token-expiration:604800000}") long refreshTokenExpiration) {
        this.key = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
        this.accessTokenExpiration = accessTokenExpiration;
        this.refreshTokenExpiration = refreshTokenExpiration;
    }

    public String generateAccessToken(String username, Set<String> roles) {
        return Jwts.builder()
                .subject(username)
                .claim("roles", roles)
                .issuedAt(Date.from(Instant.now()))
                .expiration(Date.from(Instant.now().plusMillis(accessTokenExpiration)))
                .signWith(key)
                .compact();
    }

    public String generateRefreshToken(String username) {
        return Jwts.builder()
                .subject(username)
                .issuedAt(Date.from(Instant.now()))
                .expiration(Date.from(Instant.now().plusMillis(refreshTokenExpiration)))
                .signWith(key)
                .compact();
    }

    public Claims parseToken(String token) {
        return Jwts.parser()
                .verifyWith(key)
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }

    public String getUsername(String token) {
        return parseToken(token).getSubject();
    }

    @SuppressWarnings("unchecked")
    public Set<String> getRoles(String token) {
        return parseToken(token).get("roles", Set.class);
    }

    public boolean validateToken(String token) {
        try {
            parseToken(token);
            return true;
        } catch (ExpiredJwtException e) {
            // Token 过期，区分处理
            return false;
        } catch (JwtException e) {
            return false;
        }
    }
}
```

等价的 Express + jsonwebtoken：

```js
const jwt = require("jsonwebtoken");

function generateAccessToken(user) {
  return jwt.sign(
    { sub: user.username, roles: user.roles },
    process.env.JWT_SECRET,
    { expiresIn: "1h" }
  );
}

function generateRefreshToken(user) {
  return jwt.sign({ sub: user.username }, process.env.JWT_SECRET, {
    expiresIn: "7d",
  });
}

function validateToken(token) {
  try {
    return jwt.verify(token, process.env.JWT_SECRET);
  } catch {
    return null;
  }
}
```

两者的结构几乎一一对应。有几个关键差异值得注意：

- **密钥处理**。`jjwt` 用 `Keys.hmacShaKeyFor()` 将密钥字符串转为 `SecretKey` 对象，底层会检查密钥长度是否满足算法要求（HS256 至少 256 位）。`jsonwebtoken` 直接传字符串，不会校验长度——你传一个 3 个字符的密钥也能生成 token，只是安全性形同虚设
- **过期时间表达**。`jjwt` 用绝对时间戳（`Date.from(Instant.now().plusMillis(...))`），`jsonwebtoken` 用相对时间字符串（`"1h"`, `"7d"`）。前者更精确，后者更易读
- **异常处理**。`jjwt` 区分 `ExpiredJwtException`（过期）和其他 `JwtException`（伪造、格式错误），这让你可以对过期 token 做特殊处理（比如返回"token 已过期，请刷新"而不是"token 无效"）。`jsonwebtoken` 的 `jwt.verify()` 在过期时也抛 `TokenExpiredError`，但名字上不容易和 `JsonWebTokenError` 区分

配置项放在 `application.yml` 里，而不是硬编码在代码中：

```yaml
jwt:
  secret: "your-base64-encoded-secret-key-at-least-256-bits-long-change-this"
  access-token-expiration: 3600000 # 1 小时
  refresh-token-expiration: 604800000 # 7 天
```

## 双 Token 策略：Access Token + Refresh Token

上面的代码已经暗示了一个设计决策：生成两种 token。

**Access Token**（短命，1 小时）：携带用户身份和角色，用于日常 API 请求。过期时间短，即使泄露影响窗口也有限。

**Refresh Token**（长命，7 天）：只携带用户名，用于在 access token 过期后换取新的 access token。它只走一个专用的 `/api/auth/refresh` 接口，不参与日常 API 调用。

为什么要分两个？如果只用一个长命 token，用户退出登录后（前端清除了 token），服务端无法主动使 token 失效——JWT 是自包含的，没有服务端状态。而双 token 策略下，access token 短命意味着被盗用的风险窗口小，refresh token 长命但你可以在 `/refresh` 接口做额外校验（比如检查用户是否被封禁）。

Express 社区的做法也是这样——成熟的认证方案（如 `passport-jwt` 配合 `passport-http-bearer`）都会区分 access 和 refresh。这不是 Spring Security 特有的设计，而是 JWT 认证的最佳实践。

## JwtAuthenticationFilter：把 JWT 接入过滤器链

`JwtUtil` 只负责 token 的生成和验证，它不知道 Spring Security 的存在。真正把 JWT 和 Spring Security 连起来的是 `JwtAuthenticationFilter`：

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtUtil jwtUtil;
    private final CmsUserDetailsService userDetailsService;

    public JwtAuthenticationFilter(JwtUtil jwtUtil, CmsUserDetailsService userDetailsService) {
        this.jwtUtil = jwtUtil;
        this.userDetailsService = userDetailsService;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                   HttpServletResponse response,
                                   FilterChain filterChain) throws ServletException, IOException {
        String header = request.getHeader("Authorization");

        if (header != null && header.startsWith("Bearer ")) {
            String token = header.substring(7);

            if (jwtUtil.validateToken(token)) {
                String username = jwtUtil.getUsername(token);

                // 从数据库加载最新的用户信息和权限
                UserDetails userDetails = userDetailsService.loadUserByUsername(username);

                UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());

                SecurityContextHolder.getContext().setAuthentication(authentication);
            }
        }

        filterChain.doFilter(request, response);
    }
}
```

这个 Filter 做了三件事：

1. **提取 token**。从 `Authorization: Bearer <token>` 头中取出 token 字符串
2. **验证 token**。调用 `jwtUtil.validateToken()` 检查签名和过期时间
3. **设置认证上下文**。用 token 中的用户名从数据库加载用户详情，创建 `Authentication` 对象放入 `SecurityContextHolder`

第三步有一个容易忽略的设计点：**为什么不直接从 token 里拿权限，而要查数据库？** 因为 token 签发后，用户的角色可能已经变了（比如管理员降级为普通用户）。如果只信 token 里的角色信息，就会出现权限变更后旧 token 仍然拥有高权限的安全漏洞。每次请求都从数据库加载最新的权限，是安全性和性能之间的折中——后续用 Redis 缓存可以大幅降低数据库查询频率。

等价的 Express + express-jwt 中间件：

```js
const { expressjwt: ejwt } = require("express-jwt");
const jwksRsa = require("jwks-rsa");

// 自动验证 JWT 并挂载 req.auth
app.use(
  ejwt({
    secret: jwksRsa.expressJwtSecret({ cache: true }),
    algorithms: ["RS256"],
  })
);

// 在后续 middleware 里用 req.auth.sub 拿用户名
// 但 express-jwt 不查数据库，角色信息完全来自 token
```

`express-jwt` 默认不做数据库查询，角色完全依赖 token 里的 payload。如果你要实时校验权限，需要在后面再挂一个 middleware 手动查库。Spring Security 的方式把这些逻辑集中在一个 Filter 里，结构更清晰。

## 把 Filter 注册到 SecurityFilterChain

上一篇的 `SecurityConfig` 只配置了路径权限，没有认证机制。现在把 `JwtAuthenticationFilter` 插进去：

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtAuthenticationFilter;

    public SecurityConfig(JwtAuthenticationFilter jwtAuthenticationFilter) {
        this.jwtAuthenticationFilter = jwtAuthenticationFilter;
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .requestMatchers("/api/posts", "/api/posts/{slug}").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthenticationFilter,
                UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

`addFilterBefore` 的位置很重要。`UsernamePasswordAuthenticationFilter` 是 Spring Security 内置的表单登录过滤器（基于 Session 的认证），把它放在这个位置之前意味着：JWT 认证先于表单认证执行。对于纯 REST API + JWT 的场景，表单认证过滤器不会生效（因为 Session 是 STATELESS 的），但保留它的位置确保过滤器链的顺序符合预期。

Express 中等价的概念就是 middleware 的注册顺序：

```js
// 顺序决定一切
app.use(cors());
app.use(bodyParser.json());
app.use(jwtMiddleware); // ← JWT 验证在这里
app.use("/api/auth", authRouter); // ← 放在后面，所以不受 JWT 保护
app.use("/api/admin", adminRouter);
```

但 Express 的 middleware 和 Spring Security 的 Filter 有一个根本区别：**Express 中 `jwtMiddleware` 执行后，后面的 middleware 都能访问 `req.user`。但 Spring Security 的 Filter 设置的 `SecurityContext` 不是"向后传递"的——它是存在 ThreadLocal 里的，任何后续的 Filter、Interceptor、Controller 都能访问。** 这意味着即使 Controller 之后再执行一些逻辑，它依然能通过 `SecurityContextHolder` 拿到用户信息。

## 登录注册接口

认证的最后一环是提供登录和注册接口。它们放在 `/api/auth/**` 路径下，不需要 JWT 认证（`permitAll()`）。

DTO 定义：

```java
public record LoginDTO(
    @NotBlank String username,
    @NotBlank String password
) {}

public record RegisterDTO(
    @NotBlank @Size(min = 3, max = 30) String username,
    @NotBlank @Email String email,
    @NotBlank @Size(min = 6, max = 50) String password
) {}
```

Controller：

```java
@RestController
@RequestMapping("/api/auth")
public class AuthController {

    private final AuthenticationManager authenticationManager;
    private final JwtUtil jwtUtil;
    private final UserService userService;

    public AuthController(AuthenticationManager authenticationManager,
                         JwtUtil jwtUtil, UserService userService) {
        this.authenticationManager = authenticationManager;
        this.jwtUtil = jwtUtil;
        this.userService = userService;
    }

    @PostMapping("/login")
    public ResponseEntity<Map<String, Object>> login(@Valid @RequestBody LoginDTO dto) {
        authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(dto.username(), dto.password()));

        Set<String> roles = userService.getRolesByUsername(dto.username());
        String accessToken = jwtUtil.generateAccessToken(dto.username(), roles);
        String refreshToken = jwtUtil.generateRefreshToken(dto.username());

        Map<String, Object> data = Map.of(
            "accessToken", accessToken,
            "refreshToken", refreshToken,
            "tokenType", "Bearer"
        );

        return ResponseEntity.ok(data);
    }

    @PostMapping("/register")
    public ResponseEntity<Map<String, String>> register(@Valid @RequestBody RegisterDTO dto) {
        userService.register(dto.username(), dto.email(), dto.password());

        return ResponseEntity.status(HttpStatus.CREATED)
            .body(Map.of("message", "注册成功"));
    }

    @PostMapping("/refresh")
    public ResponseEntity<Map<String, Object>> refresh(
            @RequestHeader("Authorization") String authHeader) {
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();
        }

        String refreshToken = authHeader.substring(7);

        if (!jwtUtil.validateToken(refreshToken)) {
            return ResponseEntity.status(HttpStatus.UNAUTHORIZED)
                .body(Map.of("error", "Refresh Token 无效或已过期"));
        }

        String username = jwtUtil.getUsername(refreshToken);
        Set<String> roles = userService.getRolesByUsername(username);
        String newAccessToken = jwtUtil.generateAccessToken(username, roles);

        return ResponseEntity.ok(Map.of(
            "accessToken", newAccessToken,
            "tokenType", "Bearer"
        ));
    }
}
```

登录流程有一个值得注意的设计：**使用 `AuthenticationManager.authenticate()` 而不是手动查库比对密码。** `AuthenticationManager` 内部会调用 `UserDetailsService` 查用户，再用 `PasswordEncoder` 比对密码。这样密码验证逻辑复用了 Spring Security 的标准流程，而不是在 Controller 里另写一套。

`/refresh` 接口只验证 refresh token，签发新的 access token。注意它不签发新的 refresh token——refresh token 是一次性的"续命凭证"，用完即止。如果 refresh token 也过期了，用户只能重新登录。这是安全性和用户体验之间的平衡点。

等价的 Express 登录接口：

```js
router.post("/login", async (req, res) => {
  const { username, password } = req.body;

  const user = await User.findOne({ where: { username } });
  if (!user || !bcrypt.compareSync(password, user.passwordHash)) {
    return res.status(401).json({ error: "用户名或密码错误" });
  }

  const accessToken = generateAccessToken(user);
  const refreshToken = generateRefreshToken(user);

  res.json({ accessToken, refreshToken, tokenType: "Bearer" });
});
```

Express 版本直接在路由处理函数里查库、比对密码、生成 token，一切都在一个函数里完成。Spring Boot 版本把这些职责分散到了 `AuthenticationManager`（密码验证）、`JwtUtil`（token 生成）、`UserService`（角色查询）三个组件中。前者写起来快，后者改起来安全——密码验证策略变了只需要换 `PasswordEncoder`，不影响 Controller。

## 错误处理：JWT 认证失败时返回什么

默认情况下，JWT 验证失败（token 无效、过期、缺失）时，Spring Security 不会返回友好的 JSON 错误。它会抛出 `AuthenticationException`，由 `ExceptionTranslationFilter` 处理——默认返回 401 和一个空的响应体，或者重定向到 `/login`（在非 STATELESS 模式下）。

对于 REST API，你需要自定义认证失败的处理：

```java
@Component
public class JwtAuthenticationEntryPoint implements AuthenticationEntryPoint {

    @Override
    public void commence(HttpServletRequest request,
                         HttpServletResponse response,
                         AuthenticationException authException) throws IOException {
        response.setContentType("application/json;charset=UTF-8");
        response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);

        String error = "认证失败";
        if (authException instanceof InsufficientAuthenticationError) {
            error = "缺少认证凭证";
        }

        response.getWriter().write(
            "{\"code\":401,\"error\":\"" + error + "\"}");
    }
}
```

然后在 `SecurityConfig` 中注册：

```java
http.exceptionHandling(ex -> ex
    .authenticationEntryPoint(jwtAuthenticationEntryPoint));
```

Express 中不存在这个问题——`express-jwt` 验证失败时直接抛 `UnauthorizedError`，你的全局错误处理 middleware 统一捕获并返回 JSON：

```js
app.use((err, req, res, next) => {
  if (err.name === "UnauthorizedError") {
    return res.status(401).json({ code: 401, error: "Token 无效" });
  }
  // ...
});
```

Spring Security 的方式更正式但也更繁琐——你需要显式实现 `AuthenticationEntryPoint` 接口。这是 Spring Security "凡事都有接口" 设计哲学的体现：每个可扩展的点都有对应的接口，你只需要实现它并注册到配置中。

## 完整请求链路

一个 JWT 认证请求的完整生命周期：

```
客户端发起请求: GET /api/admin/users
  Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
        │
        ▼
  JwtAuthenticationFilter
    │  提取 Bearer token
    │  jwtUtil.validateToken(token) → true
    │  jwtUtil.getUsername(token) → "admin"
    │  userDetailsService.loadUserByUsername("admin")
    │  创建 Authentication → 放入 SecurityContextHolder
        │
        ▼
  FilterSecurityInterceptor
    │  检查当前请求路径 /api/admin/users 需要 ROLE_ADMIN
    │  检查 Authentication.getAuthorities() 包含 ROLE_ADMIN
    │  通过 ✓
        │
        ▼
  DispatcherServlet → AdminController
    │  SecurityContextHolder.getContext().getAuthentication()
    │  拿到当前用户信息，执行业务逻辑
        │
        ▼
  返回 200 + JSON 数据
```

对比 Express 的链路：

```
客户端发起请求: GET /api/admin/users
  Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
        │
        ▼
  cors middleware → body parser
        │
        ▼
  express-jwt middleware
    │  验证 token → 成功
    │  挂载 req.auth = { sub: "admin", roles: ["admin"] }
        │
        ▼
  authorize middleware
    │  检查 req.auth.roles 包含 "admin"
    │  通过 ✓
        │
        ▼
  adminRouter handler
    │  用 req.auth.sub 拿用户名
        │
        ▼
  返回 200 + JSON 数据
```

链路结构一致，差异在于执行层级和组件组织方式。

## Express vs Spring Boot JWT 对比

| 维度         | Express + jsonwebtoken                     | Spring Boot + jjwt + Spring Security                        |
| ------------ | ------------------------------------------ | ----------------------------------------------------------- |
| Token 生成   | `jwt.sign(payload, secret)`                | `Jwts.builder().subject().signWith().compact()`             |
| Token 验证   | `jwt.verify(token, secret)`                | `Jwts.parser().verifyWith(key).build().parseSignedClaims()` |
| 自动拦截     | `express-jwt` middleware                   | `JwtAuthenticationFilter` (OncePerRequestFilter)            |
| 密码验证     | `bcrypt.compare()` 手动调用                | `AuthenticationManager` 自动调用 `PasswordEncoder`          |
| 用户信息传递 | `req.auth` (请求对象属性)                  | `SecurityContextHolder` (ThreadLocal)                       |
| 认证失败处理 | 全局 error middleware                      | `AuthenticationEntryPoint` 接口                             |
| 密钥安全校验 | 无（传什么都能生成 token）                 | `Keys.hmacShaKeyFor()` 自动校验长度                         |
| 异常区分     | `TokenExpiredError` vs `JsonWebTokenError` | `ExpiredJwtException` vs `JwtException`                     |
| 配置集中度   | middleware 分散在各路由文件                | `SecurityFilterChain` 一个类集中配置                        |

## Trade-off 分析

**Spring Security JWT 的收益：**

- 认证逻辑与框架深度集成。`JwtAuthenticationFilter` 设置的 `SecurityContext` 自动被 `@PreAuthorize`、`FilterSecurityInterceptor` 等所有安全组件识别，不需要额外的胶水代码
- 密码验证复用 `AuthenticationManager`。登录接口不需要手写"查库 + 比对密码"的逻辑，`UserDetailsService` + `PasswordEncoder` 的组合在认证和注册两处复用
- 安全策略集中管理。所有路径的认证要求在 `SecurityFilterChain` 里声明，`AuthenticationEntryPoint` 统一处理认证失败响应。修改安全策略只改一个文件
- 类型安全。`record LoginDTO`、`record RegisterDTO` 配合 `@Valid` 在编译期就保证请求体结构正确，运行时由 Bean Validation 校验内容

**代价：**

- 概念密度高。`JwtAuthenticationFilter` + `SecurityContextHolder` + `AuthenticationManager` + `UserDetailsService` + `AuthenticationEntryPoint` + `SecurityFilterChain`——一个完整的 JWT 认证需要理解至少 6 个核心概念。Express 的 `express-jwt` middleware 三个概念就能搞定：`jwt.sign()`、`jwt.verify()`、`req.auth`
- 配置繁琐。`AuthenticationManager` 需要 `UserDetailsService` 和 `PasswordEncoder` 两个 Bean 才能工作，`SecurityConfig` 要显式禁用 CSRF、设置 STATELESS、注册 Filter、配置 EntryPoint。Express 里两行 middleware 搞定的事，Spring Security 要写一整个配置类
- 调试困难。JWT 验证失败时，你不知道是 Filter 没执行、token 格式错误、签名不匹配还是 `UserDetailsService` 抛了异常。Spring Security 的过滤器链很长，出问题时需要逐个排查。Express 的 middleware 链更短，加个 `console.log` 就能定位
- 灵活度受限。如果你想对不同接口用不同的 token 过期时间（比如管理员接口 15 分钟过期、普通接口 1 小时过期），在 Spring Security 中需要自定义多个 Filter 或在 `JwtUtil` 里加逻辑，不如 Express 中针对不同路由组挂不同 middleware 灵活

如果你问"Express JWT 中间件和 Spring Security JWT 怎么选"，答案取决于项目的规模和团队能力。对于小团队快速迭代的个人项目，Express + jsonwebtoken 的上手速度无可匹敌。但对于需要细粒度权限控制、多模块协作的企业级应用，Spring Security JWT 的集中管理和框架集成能力，长期来看维护成本更低。

## 回顾与预告

到这篇为止，CMS 的认证体系完整了：

- `JwtUtil` 封装了 token 的生成和验证，等价于 Express 中的 `jsonwebtoken`
- `JwtAuthenticationFilter` 把 JWT 接入 Spring Security 的过滤器链，等价于 `express-jwt` middleware
- 双 Token 策略（access + refresh）平衡了安全性和用户体验
- `AuthenticationManager` 复用了 `UserDetailsService` 和 `PasswordEncoder`，登录逻辑不需要手写密码比对
- `SecurityConfig` 集中配置了所有路径的认证要求，`AuthenticationEntryPoint` 统一处理认证失败

但有一个性能隐患一直存在：`JwtAuthenticationFilter` 每次请求都调 `userDetailsService.loadUserByUsername()` 查数据库加载用户权限。对于高频访问的公开接口（文章列表、详情），这完全没必要——用户的权限信息不会在几分钟内变化。

下一篇引入 Redis 缓存，解决两个问题：用户权限的缓存（减少数据库查询）和文章列表的缓存（减少重复 SQL 执行）。
