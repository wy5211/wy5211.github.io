CMS 的文章 CRUD、参数校验、全局异常处理都已经到位了，但有一个致命的问题：**所有接口都是裸奔的。** 任何人都能调用 `DELETE /api/posts/1` 删除文章，任何人都能创建新用户。到目前为止我们只关心了"功能能不能用"，没有关心"谁有权限用"。

在 Express 里你会怎么做？装 passport.js，写几个 middleware，在需要保护的路由上加 `passport.authenticate('jwt', { session: false })`。整个认证流程是中间件链——请求进来，一个接一个的函数检查 token、查用户、设置 `req.user`，任何一个环节失败就返回 401。

Spring Security 做的事情本质上一样，但它用的不是 middleware 链，而是 **Servlet Filter 链**。这两个概念看起来相似，但在执行层级、生命周期和配置方式上有根本区别。理解这个区别，是从 Express 转到 Spring Security 的第一步。

## 过滤器链 vs 中间件链：根本区别

Express 的中间件链运行在应用层：

```
HTTP 请求
  │
  ▼
Node.js HTTP Server（libuv）
  │
  ▼
Express app（应用层）
  │
  ▼
middleware 1 (cors)
  │
  ▼
middleware 2 (bodyParser)
  │
  ▼
passport.authenticate() ← 认证在这里
  │
  ▼
route handler ← 业务逻辑在这里
```

所有中间件都在 Express 应用内部执行，由 `app.use()` 或 `router.use()` 注册，执行顺序就是你写的顺序。

Spring Security 的过滤器链运行在 Servlet 容器层，在 DispatcherServlet 之前：

```
HTTP 请求
  │
  ▼
Tomcat（Servlet 容器）
  │
  ▼
Filter 1: CorsFilter
  │
  ▼
Filter 2: SecurityContextPersistenceFilter
  │
  ▼
Filter 3: UsernamePasswordAuthenticationFilter ← 认证在这里
  │
  ▼
Filter 4: ExceptionTranslationFilter
  │
  ▼
Filter 5: FilterSecurityInterceptor ← 授权检查在这里
  │
  ▼
DispatcherServlet ← 这才进入 Spring MVC
  │
  ▼
Controller ← 业务逻辑在这里
```

关键区别：**Spring Security 的过滤器在 Controller 之前执行，甚至在 Spring MVC 之前执行。** 它不依赖 Spring MVC 的任何组件——即使你不写 Controller，Security 的过滤器链依然工作。这意味着认证和授权是完全独立于业务逻辑的，你不可能"忘记"加认证——它在框架层面就拦截了。

对 Express 开发者来说，这就像把 passport.js 的中间件放到了 `http.createServer()` 的回调之前，而不是 `app.use()` 里。

## SecurityFilterChain：过滤器链的配置入口

Spring Security 6.x（Spring Boot 3.x 默认版本）的核心配置入口是 `SecurityFilterChain` Bean。你通过 `HttpSecurity` 的链式 API 来定义哪些路径需要认证、用什么认证方式、CSRF 策略等：

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**", "/api/posts/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            );

        return http.build();
    }
}
```

逐行拆解：

- **`csrf.disable()`** —— 禁用 CSRF 保护。REST API 用 Token 认证（而非 Cookie-Session），CSRF 防护没有意义。Web 表单提交场景才需要 CSRF。
- **`SessionCreationPolicy.STATELESS`** —— 告诉 Spring Security 不要创建 HTTP Session。每个请求独立认证，不依赖服务端状态。这是前后端分离 + JWT 的标准配置。
- **`requestMatchers("/api/auth/**").permitAll()`\*\* —— 注册和登录接口不需要认证。
- **`requestMatchers("/api/admin/**").hasRole("ADMIN")`\*\* —— 管理接口需要 ADMIN 角色。
- **`anyRequest().authenticated()`** —— 其他所有请求都需要认证。

等价的 Express + passport.js 写法：

```js
// Express 中你需要手动给每个路由组加中间件
app.use("/api/auth", authRouter); // 不需要认证
app.use("/api/posts", postRouter); // 不需要认证
app.use("/api/admin", passport.authenticate("jwt"), adminRouter); // 需要认证+ADMIN
app.use(passport.authenticate("jwt"), router); // 其他都需要认证
```

两者的表达力基本一致。区别在于 Spring Security 的配置集中在一个 `SecurityFilterChain` Bean 里，而 Express 的认证中间件分散在各个路由定义处。前者在安全策略变更时只需要改一个文件，后者需要在多个文件中搜索和修改。

## SecurityContextHolder 和 Authentication

认证成功后，Spring Security 把用户信息存在哪里？答案是 `SecurityContextHolder`。

```java
// 在 Controller 或 Service 中获取当前认证用户
Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
String username = authentication.getName();
Collection<? extends GrantedAuthority> authorities = authentication.getAuthorities();
```

`SecurityContextHolder` 底层用 `ThreadLocal` 存储认证信息。每个请求线程有自己的 SecurityContext，互不干扰。这和 Express 中 `req.user` 的作用类似——在同一个请求的生命周期内，任何地方都能通过 `SecurityContextHolder` 拿到当前用户。

但有一个重要的区别：Express 的 `req.user` 是请求对象上的属性，跟着请求走；Spring Security 的 `SecurityContextHolder` 是线程级别的。在 Web 环境下（一个请求一个线程）两者行为一致。但如果你在异步场景中（比如 `@Async` 方法或 `CompletableFuture`），子线程默认拿不到父线程的 SecurityContext，需要手动传递。这是从 Express 转过来容易忽略的坑。

`Authentication` 接口的核心方法：

```java
public interface Authentication extends Principal {
    // 认证主体，通常是用户名或 User 对象
    Object getPrincipal();

    // 凭证，密码认证后通常是 null（安全考虑），JWT 认证后是 token 字符串
    Object getCredentials();

    // 权限列表：ROLE_USER, ROLE_ADMIN 等
    Collection<? extends GrantedAuthority> getAuthorities();

    // 是否已认证
    boolean isAuthenticated();
}
```

`getPrincipal()` 返回的对象类型取决于你的 `UserDetailsService` 实现。默认情况下它是 Spring Security 的 `User` 对象（包含用户名、密码、权限）。下一篇 JWT 认证中，我们会把它换成 CMS 自己的 `User` 实体，这样在 Controller 里可以直接 `getUser().getEmail()` 而不是只拿到一个用户名。

## UserDetailsService：告诉 Spring Security "用户从哪来"

Spring Security 需要一个地方查用户信息——验证密码、获取角色。这个"地方"就是 `UserDetailsService`。它只有一个方法：

```java
public interface UserDetailsService {
    UserDetails loadUserByUsername(String username) throws UsernameNotFoundException;
}
```

对于 CMS 项目，实现非常直接——从数据库查用户，然后包装成 Spring Security 要求的 `UserDetails` 对象：

```java
@Service
public class CmsUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    public CmsUserDetailsService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        User user = userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("用户不存在: " + username));

        return org.springframework.security.core.userdetails.User
            .withUsername(user.getUsername())
            .password(user.getPasswordHash())
            .roles(user.getRole().name())
            .accountExpired(false)
            .accountLocked(false)
            .credentialsExpired(false)
            .enabled(true)
            .build();
    }
}
```

`UserDetails` 是 Spring Security 的接口，定义了"一个认证主体需要哪些信息"。你不需要实现这个接口——Spring Security 提供了一个 Builder，用 `User.withUsername()` 构造。这里把 CMS 的 `UserRole` 枚举直接转成 `roles()`，Spring Security 会自动加上 `ROLE_` 前缀。也就是说，如果 `user.getRole()` 返回 `ADMIN`，最终生成的权限是 `ROLE_ADMIN`，这和 `SecurityConfig` 里的 `.hasRole("ADMIN")` 对应。

等价的 Express + passport.js：

```js
// passport LocalStrategy 本质上也是查用户
passport.use(
  new LocalStrategy(async (username, password, done) => {
    const user = await User.findOne({ where: { username } });
    if (!user) return done(null, false, { message: "用户不存在" });
    if (!bcrypt.compareSync(password, user.passwordHash))
      return done(null, false, { message: "密码错误" });
    return done(null, user);
  })
);

// 序列化——决定 session 里存什么
passport.serializeUser((user, done) => done(null, user.id));
passport.deserializeUser(async (id, done) => {
  const user = await User.findByPk(id);
  done(null, user);
});
```

passport.js 的 `LocalStrategy` 和 `UserDetailsService.loadUserByUsername()` 做的事情完全一样：根据用户名查数据库，返回用户对象。但有一个关键区别——**passport.js 的 Strategy 只负责认证（验证密码），不负责加载权限。** 权限检查通常在后续的 middleware 里手写。而 Spring Security 的 `UserDetailsService` 同时提供了用户信息和权限列表，认证和授权用的是同一个数据源。

## PasswordEncoder：不要自己存明文密码

Spring Security 不会让你在配置里不提供 `PasswordEncoder`。如果你定义了 `UserDetailsService` 但没有 `PasswordEncoder` Bean，启动时直接报错。这个设计强制你思考密码怎么存。

```java
@Configuration
public class SecurityConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

BCrypt 是当前推荐的密码哈希算法。它有两个特点：自带盐（salt），每次生成的哈希值不同；计算成本可调（strength 参数），提高暴力破解的门槛。

注册用户时加密密码：

```java
@Service
public class UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    public UserService(UserRepository userRepository, PasswordEncoder passwordEncoder) {
        this.userRepository = userRepository;
        this.passwordEncoder = passwordEncoder;
    }

    public User register(String username, String email, String rawPassword) {
        if (userRepository.existsByUsername(username)) {
            throw new IllegalArgumentException("用户名已存在");
        }
        if (userRepository.existsByEmail(email)) {
            throw new IllegalArgumentException("邮箱已被注册");
        }

        User user = new User();
        user.setUsername(username);
        user.setEmail(email);
        user.setPasswordHash(passwordEncoder.encode(rawPassword));
        user.setRole(UserRole.USER);
        user.setCreatedAt(LocalDateTime.now());

        return userRepository.save(user);
    }
}
```

在 Express 中，同样的逻辑需要手动引入 `bcrypt`：

```js
const bcrypt = require("bcrypt");
const hash = await bcrypt.hash(rawPassword, 10); // salt rounds = 10
```

`passwordEncoder.encode()` 和 `bcrypt.hash(raw, 10)` 做的事情完全一样。Spring Security 的优势在于 `PasswordEncoder` 是接口，你可以随时切换实现（比如从 BCrypt 换成 Argon2）而不改业务代码。

## 方法级权限：@PreAuthorize

`SecurityFilterChain` 配置的是 URL 级别的权限控制。如果你想更精细——比如"只有文章作者才能删除自己的文章"——需要方法级权限控制。

```java
@Configuration
@EnableMethodSecurity
public class SecurityConfig {
    // ...
}
```

`@EnableMethodSecurity` 开启后，可以在 Service 方法上用 `@PreAuthorize`：

```java
@Service
public class PostService {

    @PreAuthorize("hasRole('ADMIN')")
    public void deleteAnyPost(Long id) {
        postRepository.deleteById(id);
    }

    @PreAuthorize("hasRole('USER') and #authorId == authentication.principal.id")
    public void updatePost(Long id, Long authorId, PostUpdateRequest request) {
        // 只有作者本人能修改
    }
}
```

`#authorId` 引用的是方法参数，`authentication.principal.id` 引用的是当前登录用户的 ID。SpEL（Spring Expression Language）让权限表达非常灵活。

Express 中等价的写法通常是在 middleware 里手写：

```js
function authorizeOwner(req, res, next) {
  if (req.user.id !== Number(req.params.authorId)) {
    return res.status(403).json({ error: "无权操作" });
  }
  next();
}
```

两者功能等价，但 `@PreAuthorize` 的优势在于：权限规则和方法绑定在一起，不需要额外的 middleware 文件。当你修改业务方法签名时，权限规则就在旁边，不容易遗忘。

## 完整对比

| 维度         | Express + passport.js             | Spring Security                                        |
| ------------ | --------------------------------- | ------------------------------------------------------ |
| 认证执行层   | Express middleware（应用层）      | Servlet Filter（容器层）                               |
| 认证入口     | `passport.authenticate(strategy)` | `SecurityFilterChain` 集中配置                         |
| 用户信息存储 | `req.user`（请求对象属性）        | `SecurityContextHolder`（ThreadLocal）                 |
| 查用户       | `Strategy.verify()` 回调          | `UserDetailsService.loadUserByUsername()`              |
| 密码哈希     | `bcrypt` 手动调用                 | `PasswordEncoder` 接口，框架强制提供                   |
| URL 级权限   | middleware 挂载位置控制           | `authorizeHttpRequests` 链式配置                       |
| 方法级权限   | 自定义 middleware                 | `@PreAuthorize` + SpEL 表达式                          |
| 配置方式     | 分散在多个文件中                  | 集中在 SecurityConfig 一个类中                         |
| 灵活度       | 高，每个路由可以完全自定义        | 中，在框架约定内配置                                   |
| 学习曲线     | 低，看几个例子就能上手            | 高，过滤器链 + Authentication + SecurityContext 概念多 |

## Trade-off 分析

**Spring Security 的收益：**

- 安全策略集中管理。`SecurityFilterChain` 一个配置类定义了所有路径的访问规则，比 Express 中分散在多个路由文件的 middleware 更容易审计和维护
- 默认安全。没有 `PasswordEncoder` Bean 就启动不了，这种"强制安全"的设计避免了不小心存明文密码
- 认证和授权的深度集成。`UserDetailsService` 同时提供用户信息和权限，`@PreAuthorize` 可以直接用 SpEL 表达复杂规则
- 社区和企业验证。Spring Security 是 Java 生态事实上的安全标准，OWASP Top 10 里大部分攻击向量它都有内置防护

**代价：**

- 概念多，学习曲线陡峭。过滤器链、SecurityContext、Authentication、UserDetails、GrantedAuthority、PasswordEncoder——光是这些概念就需要消化。passport.js 只需要理解 Strategy + serialize/deserialize
- 框架"魔数"多。一个 `@EnableWebSecurity` 注解背后注册了十几个 Filter，你不知道具体发生了什么。出问题时排查链路长
- 灵活度受限。你想在认证流程中插入一个自定义步骤（比如检查用户是否被临时封禁），需要自定义 Filter 或 AuthenticationProvider，比 Express middleware 的 `next()` 链复杂得多
- Session 状态的隐式行为。如果你忘了配置 `SessionCreationPolicy.STATELESS`，Spring Security 默认会创建 Session，在前后端分离场景下会导致诡异的问题

如果你问"passport.js 和 Spring Security 哪个更灵活"，答案是 passport.js 在灵活度上胜出——它的 Strategy 模式让你可以对接任何认证方式，每个路由的认证逻辑可以完全自定义。但 Spring Security 在"安全策略的集中管理和默认安全"上胜出——它不是让你"自由组合"，而是给你一套经过验证的安全框架，你在这个框架内配置。

## 回顾与预告

到这篇为止，CMS 的安全基础已经搭好：

- 理解了 Spring Security 的过滤器链机制——认证在 Servlet 容器层执行，比 Express middleware 更早介入请求
- `SecurityFilterChain` 集中配置了路径级别的访问规则
- `UserDetailsService` 告诉框架用户数据从哪来，`PasswordEncoder` 强制密码哈希
- `SecurityContextHolder` 提供请求级别的用户信息访问，等价于 Express 的 `req.user`

但目前的配置只定义了"谁有权限访问什么"，还没有实现真正的认证流程。用户怎么登录？登录成功后怎么获取凭证？后续请求怎么用这个凭证证明身份？这些都需要 JWT。

下一篇实现完整的 JWT 认证——用 jjwt 库生成和验证 Token，写自定义的 `JwtAuthenticationFilter` 接入 Security 过滤器链，完成 CMS 的注册、登录、Token 刷新全流程。
