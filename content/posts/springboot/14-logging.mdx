CMS 上线三天，运维群里有人发了一条消息："文章详情页偶尔返回 500，日志里找不到原因"。你打开日志文件，满屏都是 Hibernate 的 SQL 语句和 Spring 的 Bean 初始化信息——找不到那个 500 错误到底发生在哪。因为你从头到尾用的是 `System.out.println`，日志没有级别、没有上下文、没有请求 ID，本质上是把一堆字符串扔到控制台。

在 Node.js 里你可能已经用了 winston 或 pino——它们提供日志级别、格式化输出、文件轮转。Java 的日志体系比 Node.js 复杂得多：SLF4J、Logback、Log4j2、JUL，光是名字就够让人头疼。好消息是 Spring Boot 已经帮你做了选择，默认用 SLF4J + Logback。你需要做的只是理解它的工作方式，然后正确配置。

## SLF4J + Logback：门面 + 实现

Java 日志体系的核心设计是门面模式（Facade Pattern）：

```
你的代码
  |
  v
SLF4J（日志门面 / 接口）
  |
  v
Logback（日志实现）
  |-- ConsoleAppender        -> 控制台
  |-- RollingFileAppender    -> 文件，自动轮转
  +-- LogstashEncoder        -> JSON 格式，对接 ELK
```

SLF4J 是接口，Logback 是实现。你的代码只依赖 SLF4J 的 API，不直接依赖 Logback。想换成 Log4j2，只需要换依赖包，不改任何业务代码。

这和 Node.js 的日志生态形成鲜明对比。Node.js 没有类似的门面抽象——你直接使用 winston 或 pino 的 API。从 winston 换到 pino，需要改所有日志调用点。但实际上很少有人这么做，选了一个日志库基本就锁定了。

```java
// Spring Boot 中使用 SLF4J 记录日志
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class PostService {
    private static final Logger log = LoggerFactory.getLogger(PostService.class);

    public Post getPostById(Long id) {
        log.info("查询文章, id={}", id);
        return postRepository.findById(id)
            .orElseThrow(() -> {
                log.warn("文章不存在, id={}", id);
                return new PostNotFoundException(id);
            });
    }
}
```

也可以用 Lombok 的 `@Slf4j` 注解自动生成 `log` 字段，效果完全等价，只是省了手动声明那行代码。

## 日志级别与精细化控制

SLF4J 定义了五个标准级别，从低到高：`TRACE < DEBUG < INFO < WARN < ERROR`。

- **TRACE** —— 最细粒度的调试信息，生产环境不应输出
- **DEBUG** —— 开发调试信息，SQL 语句、参数绑定等，生产环境默认关闭
- **INFO** —— 关键业务节点，用户登录、文章发布等，生产环境默认开启
- **WARN** —— 潜在问题，不致命但需要关注
- **ERROR** —— 需要立即处理的错误

在 `application.yml` 中配置：

```yaml
logging:
  level:
    root: INFO
    com.example.cms: DEBUG
    com.example.cms.web: INFO
    org.springframework.security: WARN
    org.hibernate.SQL: DEBUG
    org.hibernate.type.descriptor.sql.BasicBinder: TRACE
```

`root: INFO` 是全局默认级别，`com.example.cms: DEBUG` 只对你自己的代码开启 DEBUG。Spring 框架和 Hibernate 的日志级别单独控制——调试 SQL 问题时只看 Hibernate 的 DEBUG 输出，不会被 Spring Security 的 INFO 日志淹没。

这是 Logback 相对于 Node.js 日志库的一个杀手级功能。Node.js 的 pino 只有一个全局 `level` 字段，你想只开启某个模块的 DEBUG 日志，需要借助 `debug` 库的命名空间机制（`DEBUG=app:service:*`），这不是日志库的核心功能，而是另一个库的能力。

## logback-spring.xml：结构化日志配置

Spring Boot 推荐用 `logback-spring.xml`（而不是 `logback.xml`），因为前者支持 Spring 的 profile 特性——不同环境使用不同的日志配置。

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <include resource="org/springframework/boot/logging/logback/defaults.xml"/>

    <springProperty scope="context" name="APP_NAME"
        source="spring.application.name" defaultValue="cms"/>
    <property name="LOG_PATH" value="${LOG_PATH:-./logs}"/>
    <property name="LOG_FILE" value="${LOG_PATH}/${APP_NAME}"/>

    <!-- 开发环境：只输出到控制台 -->
    <springProfile name="dev">
        <appender name="CONSOLE"
            class="ch.qos.logback.core.ConsoleAppender">
            <encoder>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %highlight(%-5level)
                    [%thread] %cyan(%logger{36}) - %msg%n</pattern>
            </encoder>
        </appender>
        <root level="INFO">
            <appender-ref ref="CONSOLE"/>
        </root>
    </springProfile>

    <!-- 生产环境：文件输出，ERROR 单独一份 -->
    <springProfile name="prod">
        <appender name="FILE"
            class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>${LOG_FILE}.log</file>
            <rollingPolicy class="ch.qos.logback.core.rolling
                .SizeAndTimeBasedRollingPolicy">
                <fileNamePattern>${LOG_FILE}.%d{yyyy-MM-dd}.%i.log.gz
                </fileNamePattern>
                <maxFileSize>100MB</maxFileSize>
                <maxHistory>30</maxHistory>
                <totalSizeCap>2GB</totalSizeCap>
            </rollingPolicy>
            <encoder>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level
                    [%thread] %logger{36} - %msg%n</pattern>
            </encoder>
        </appender>

        <appender name="ERROR_FILE"
            class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>${LOG_FILE}-error.log</file>
            <filter class="ch.qos.logback.classic.filter.ThresholdFilter">
                <level>ERROR</level>
            </filter>
            <rollingPolicy class="ch.qos.logback.core.rolling
                .SizeAndTimeBasedRollingPolicy">
                <fileNamePattern>${LOG_FILE}-error.%d{yyyy-MM-dd}.%i.log.gz
                </fileNamePattern>
                <maxFileSize>50MB</maxFileSize>
                <maxHistory>60</maxHistory>
                <totalSizeCap>1GB</totalSizeCap>
            </rollingPolicy>
            <encoder>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level
                    [%thread] %logger{36} - %msg%n</pattern>
            </encoder>
        </appender>

        <root level="INFO">
            <appender-ref ref="FILE"/>
            <appender-ref ref="ERROR_FILE"/>
        </root>
    </springProfile>
</configuration>
```

几个关键配置：

- **`SizeAndTimeBasedRollingPolicy`** —— 同时按时间和大小轮转。每天一个文件，单文件超过 100MB 自动分割，保留 30 天，总大小不超过 2GB
- **`ERROR_FILE` + `ThresholdFilter`** —— ERROR 日志单独一份文件，保留 60 天。排查线上问题直接看这个文件，不用从海量 INFO 中过滤
- **`springProfile`** —— dev 只输出到控制台，prod 输出到文件。不需要写 if/else，XML 标签天然就是条件化配置

Node.js 中实现同样的文件轮转需要 `winston-daily-rotate-file`，功能等价，但 Logback 的 XML 配置支持更丰富的策略——按日志级别分文件、按包名分文件、条件化输出。winston 的配置是 JavaScript 对象，灵活但表达力有限。

## JSON 结构化日志与 ELK 对接

生产环境的日志通常用 ELK（Elasticsearch + Logstash + Kibana）或 Loki 收集和分析，这些系统需要 JSON 格式的日志。Logback 通过 `LogstashEncoder` 支持：

```xml
<!-- pom.xml 添加依赖 -->
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

```xml
<appender name="JSON_FILE"
    class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>${LOG_FILE}.json.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling
        .SizeAndTimeBasedRollingPolicy">
        <fileNamePattern>${LOG_FILE}-json.%d{yyyy-MM-dd}.%i.log.gz
        </fileNamePattern>
        <maxFileSize>100MB</maxFileSize>
        <maxHistory>30</maxHistory>
        <totalSizeCap>2GB</totalSizeCap>
    </rollingPolicy>
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <customFields>{"app":"${APP_NAME}"}</customFields>
    </encoder>
</appender>
```

输出效果：

```json
{
  "@timestamp": "2026-03-30T10:15:30.123Z",
  "level": "INFO",
  "thread": "http-nio-8080-exec-1",
  "logger": "com.example.cms.service.PostService",
  "message": "查询文章, id=42",
  "app": "cms"
}
```

Node.js 的 pino 天生就是 JSON 格式输出——这是它最大的卖点。pino 的结构化日志开箱即用，Logback 需要额外配置 LogstashEncoder。但从另一个角度看，Logback 的文本格式默认就是可读的，开发时更友好；pino 的 JSON 在控制台里很难读，需要装 `pino-pretty` 才行。

## MDC：请求追踪的利器

MDC（Mapped Diagnostic Context）是 SLF4J 的一个关键特性——它允许你在日志中自动附加上下文信息（请求 ID、用户 ID、客户端 IP），而不需要手动在每个日志调用中传递这些参数。

先写一个 Filter，在请求入口设置 MDC：

```java
@Component
public class MdcFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain filterChain)
            throws ServletException, IOException {
        try {
            String requestId = request.getHeader("X-Request-Id");
            if (requestId == null || requestId.isEmpty()) {
                requestId = UUID.randomUUID().toString().replace("-", "");
            }

            MDC.put("requestId", requestId);
            MDC.put("clientIp", request.getRemoteAddr());
            MDC.put("method", request.getMethod());
            MDC.put("uri", request.getRequestURI());

            response.setHeader("X-Request-Id", requestId);
            filterChain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}
```

然后在 `logback-spring.xml` 的 pattern 中用 `%X{requestId}` 引用 MDC 的值：

```xml
<pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level [reqId=%X{requestId}]
    [%thread] %logger{36} - %msg%n</pattern>
```

输出效果：

```
2026-03-30 10:15:30.123 INFO  [reqId=a1b2c3d4e5f6] [http-nio-8080-exec-1] c.e.c.s.PostService - 查询文章, id=42
2026-03-30 10:15:30.156 WARN  [reqId=a1b2c3d4e5f6] [http-nio-8080-exec-1] c.e.c.s.PostService - 文章不存在, id=42
2026-03-30 10:15:30.200 ERROR [reqId=a1b2c3d4e5f6] [http-nio-8080-exec-1] c.e.c.e.GlobalExceptionHandler - 未捕获异常
```

所有属于同一个请求的日志都有相同的 `reqId`。在 ELK 中搜索这个 ID 就能还原完整的请求链路。

### 异步场景的 MDC 传递

MDC 基于 `ThreadLocal` 实现——每个线程有自己的 MDC 上下文。在异步场景中（`@Async` 方法、`CompletableFuture`），子线程默认拿不到父线程的 MDC。上一篇讲 `@Async` 时的上下文传递问题，在日志层面同样存在。

解决办法是用 `TaskDecorator` 自动传递 MDC：

```java
@Bean("cmsTaskExecutor")
public Executor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(4);
    executor.setMaxPoolSize(8);
    executor.setQueueCapacity(100);
    executor.setThreadNamePrefix("cms-async-");
    executor.setTaskDecorator(runnable -> {
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        return () -> {
            try {
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    });
    executor.initialize();
    return executor;
}
```

Node.js 中用 `AsyncLocalStorage` 实现类似效果。但 `AsyncLocalStorage` 在 Node.js 的异步环境中天然工作——事件循环单线程，上下文自动沿 Promise 链传递。MDC 在 Java 的多线程环境中需要手动处理跨线程传递，这是多线程模型的固有代价。

## CMS 实战：请求日志与慢查询检测

有了 MDC 基础设施，给 CMS 加两个实用功能：请求日志和慢查询检测。

请求日志用 AOP 切面拦截所有 Controller 方法，记录入参、耗时、结果：

```java
@Aspect
@Component
@Slf4j
public class RequestLogAspect {

    @Around("execution(* com.example.cms.web.*.*(..))")
    public Object logRequest(ProceedingJoinPoint joinPoint) throws Throwable {
        String method = joinPoint.getSignature().toShortString();
        log.info("请求开始: {}", method);
        long start = System.currentTimeMillis();
        try {
            Object result = joinPoint.proceed();
            long elapsed = System.currentTimeMillis() - start;
            log.info("请求完成: {}, 耗时 {}ms", method, elapsed);
            return result;
        } catch (Throwable ex) {
            long elapsed = System.currentTimeMillis() - start;
            log.error("请求异常: {}, 耗时 {}ms, 异常: {}",
                method, elapsed, ex.getMessage());
            throw ex;
        }
    }
}
```

慢查询检测拦截所有 Repository 方法，超过 500ms 就记录 WARN：

```java
@Aspect
@Component
@Slf4j
public class SlowQueryAspect {

    private static final long SLOW_QUERY_THRESHOLD_MS = 500;

    @Around("execution(* com.example.cms.repository.*.*(..))")
    public Object logSlowQuery(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return joinPoint.proceed();
        } finally {
            long elapsed = System.currentTimeMillis() - start;
            if (elapsed > SLOW_QUERY_THRESHOLD_MS) {
                String method = joinPoint.getSignature().toShortString();
                log.warn("慢查询检测: {} 耗时 {}ms", method, elapsed);
            }
        }
    }
}
```

利用第 8 篇学的 AOP 知识，两个切面分别拦截 Controller 和 Repository 层，不需要修改任何业务代码。结合 MDC 的 requestId，日志输出自动带上请求 ID，可以精确定位是哪个用户请求触发了慢查询。

## 完整对比

| 维度           | Node.js (winston/pino)                | Java (SLF4J + Logback)                  |
| -------------- | ------------------------------------- | --------------------------------------- |
| 架构           | 直接使用，无门面                      | 门面模式（SLF4J）+ 实现（Logback）      |
| 日志级别       | debug, info, warn, error（4 级）      | trace, debug, info, warn, error（5 级） |
| 级别精细化控制 | 全局一个级别，或用 `debug` 库命名空间 | 按包名/类名独立控制级别                 |
| 结构化日志     | pino 天生 JSON，winston 需配置        | 需 LogstashEncoder                      |
| 请求上下文     | AsyncLocalStorage（需手动封装）       | MDC（框架内置，pattern 直接引用）       |
| 文件轮转       | winston-daily-rotate-file 插件        | 内置 RollingFileAppender                |
| 异步日志       | 天然异步（事件循环）                  | 需 AsyncAppender                        |
| 配置方式       | JavaScript 对象                       | XML（logback-spring.xml）               |
| 性能           | pino 极快（流式 JSON）                | Logback 异步模式性能优秀                |

## Trade-off 分析

**SLF4J + Logback 的收益：**

- 门面模式让日志实现可替换。你写的 `log.info()` 调用永远不变，底层 Logback、Log4j2、JUL 随便换。企业级项目中不同模块可能由不同团队维护，统一日志门面避免了解析冲突
- 按包名精细化控制日志级别是杀手级功能。调试 JPA 时开 `org.hibernate.SQL: DEBUG`，调试 Security 时开 `org.springframework.security: DEBUG`，不影响其他模块。Node.js 的日志库没有这个能力
- MDC + Logback 的 pattern 机制让请求追踪变得简单。`%X{requestId}` 一行配置，所有日志自动带上请求 ID，不需要改任何业务代码
- Logback 的 RollingFileAppender 功能完善。按天 + 按大小轮转、压缩归档、总大小限制，这些都是内置功能，不需要额外插件
- Spring Boot 的 `logback-spring.xml` 支持 profile，dev 和 prod 用不同的日志配置，天然就是条件化配置

**代价：**

- 配置复杂。`logback-spring.xml` 的 XML 语法对习惯了 JavaScript 的开发者不友好。Appender、Encoder、RollingPolicy、Filter 的组合方式需要时间去理解
- MDC 在异步场景中需要手动处理。`@Async` 方法的子线程拿不到父线程的 MDC，需要配置 `TaskDecorator`。Node.js 的 `AsyncLocalStorage` 在这个方面反而更省心
- 默认输出不是 JSON。pino 开箱就是 JSON 格式，可以直接对接 ELK。Logback 需要额外引入 `logstash-logback-encoder` 并配置 JSON Appender
- 启动时日志噪音。Spring Boot 启动时会输出大量 INFO 日志（Banner、Bean 初始化、条件评估报告），在没有 profile 匹配的情况下很难过滤

如果你问"winston/pino 和 Logback 哪个更好"，这不是一个公平的比较。pino 在"轻量、快速、JSON 优先"上胜出，几乎是 Node.js 项目的最佳日志选择。Logback 在"企业级日志管理"上胜出——按包名控制级别、MDC 请求追踪、profile 匹配、文件轮转策略，这些都是企业项目需要的能力。选哪个取决于你要解决什么问题。

## 回顾与预告

到这篇为止，CMS 的可观测性基础设施已经搭建完成：

- SLF4J 是日志门面，Logback 是实现，代码中只依赖 SLF4J 的 API
- 五个日志级别（TRACE/DEBUG/INFO/WARN/ERROR）和按包名精细化控制
- `logback-spring.xml` 支持 profile 匹配，dev 用控制台，prod 用文件 + JSON
- MDC 实现请求级别的上下文传递，`%X{requestId}` 一行配置自动附加到所有日志
- AOP 切面实现请求日志和慢查询检测，结合 MDC 精确定位问题

经过 14 篇文章，CMS 已经具备了比较完整的能力：REST API、JPA 数据访问、参数校验、全局异常处理、AOP、Spring Security、JWT 认证、Redis 缓存、异步任务、WebSocket 实时通信、日志系统。但代码质量怎么样？你确定那些 Service 方法写得没问题？改了一行代码会不会导致另一个接口挂掉？
