CMS 上线后收到了一个 bug 报告："文章详情页偶尔返回 500"。你打开日志文件，看到满屏的 INFO 输出——Hibernate 的 SQL 语句、Spring 的 Bean 初始化信息、Redis 的连接日志——但找不到那个 500 错误的具体信息。因为你从头到尾用的是 `System.out.println` 和 `console.log`，日志没有级别、没有上下文、没有请求 ID。

在 Node.js 里你可能已经用了 winston 或 pino——它们提供了日志级别、格式化输出、文件轮转等功能。但 Java 的日志体系比 Node.js 复杂得多：SLF4J、Logback、Log4j2、JUL——光是这些名字就够让人头疼。好消息是 Spring Boot 已经帮你做了选择，默认用 SLF4J + Logback。你需要做的只是理解它的工作方式，然后正确配置。

## SLF4J + Logback：门面 + 实现

Java 日志体系的核心设计是一个"门面模式"（Facade Pattern）：

```
你的代码
  │
  ▼
SLF4J（日志门面/接口）
  │
  ▼
Logback（日志实现）
  │
  ├─ ConsoleAppender（输出到控制台）
  ├─ RollingFileAppender（输出到文件，自动轮转）
  └─ LogstashEncoder（输出 JSON 格式，对接 ELK）
```

SLF4J 是接口，Logback 是实现。你的代码只依赖 SLF4J 的 API，不直接依赖 Logback。这意味着你想从 Logback 换成 Log4j2，只需要换依赖包，不改任何业务代码。

这和 Node.js 的日志生态形成对比。Node.js 没有类似的门面抽象——你直接使用 winston 或 pino 的 API。从 winston 换到 pino，需要改所有日志调用点。但实际上，很少有人会这样做——选了一个日志库基本就锁定了。

```java
// Spring Boot 中使用 SLF4J 记录日志
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class PostService {
    private static final Logger log = LoggerFactory.getLogger(PostService.class);

    public Post getPostById(Long id) {
        log.info("查询文章, id={}", id);
        return postRepository.findById(id)
            .orElseThrow(() -> {
                log.warn("文章不存在, id={}", id);
                return new PostNotFoundException(id);
            });
    }
}
```

也可以用 Lombok 的 `@Slf4j` 注解自动生成 `log` 字段：

```java
@Slf4j
@Service
public class PostService {
    // @Slf4j 自动生成: private static final Logger log = LoggerFactory.getLogger(PostService.class);
}
```

两种写法完全等价，`@Slf4j` 只是省了手动声明那行代码。

## 日志级别

SLF4J 定义了五个标准级别，从低到高：

```
TRACE < DEBUG < INFO < WARN < ERROR
```

- **TRACE** —— 最细粒度的调试信息。生产环境不应该输出
- **DEBUG** —— 开发调试信息。SQL 语句、参数绑定、缓存命中率等。生产环境默认关闭
- **INFO** —— 关键业务节点。用户登录、文章发布、订单创建等。生产环境默认开启
- **WARN** —— 潜在问题。不致命但需要关注。比如缓存穿透、接口响应时间超过阈值
- **ERROR** —— 错误。需要立即处理。比如数据库连接失败、外部 API 调用异常

在 `application.yml` 中配置：

```yaml
logging:
  level:
    root: INFO
    com.example.cms: DEBUG
    com.example.cms.web: INFO
    org.springframework.security: WARN
    org.hibernate.SQL: DEBUG
    org.hibernate.type.descriptor.sql.BasicBinder: TRACE
```

`root: INFO` 是全局默认级别，`com.example.cms: DEBUG` 只对你自己的代码开启 DEBUG。Spring 框架和 Hibernate 的日志级别单独控制——这样你在调试 SQL 问题时只看 Hibernate 的 DEBUG 输出，不会被 Spring Security 的 INFO 日志淹没。

Node.js 中等价的配置（以 pino 为例）：

```js
const pino = require("pino");

const logger = pino({
  level: process.env.NODE_ENV === "production" ? "info" : "debug",
  transport:
    process.env.NODE_ENV !== "production"
      ? { target: "pino-pretty" }
      : undefined,
});
```

pino 的配置更简单——一个 level 字段控制全局级别。但缺少 Java 那种按包名精细控制的能力。你想在 Node.js 里只开启某个模块的 DEBUG 日志，通常需要用 `debug` 库的命名空间机制（`DEBUG=app:service:*`），而这不是日志库的核心功能。

## logback-spring.xml：结构化日志配置

Spring Boot 推荐用 `logback-spring.xml`（而不是 `logback.xml`）来配置 Logback，因为前者支持 Spring 的 profile 特性——不同环境使用不同的日志配置。

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- 引入 Spring Boot 默认的 Logback 配置作为基础 -->
    <include resource="org/springframework/boot/logging/logback/defaults.xml"/>

    <!-- 定义变量 -->
    <springProperty scope="context" name="APP_NAME" source="spring.application.name" defaultValue="cms"/>
    <property name="LOG_PATH" value="${LOG_PATH:-./logs}"/>
    <property name="LOG_FILE" value="${LOG_PATH}/${APP_NAME}"/>

    <!-- 控制台输出：带颜色的格式，开发环境用 -->
    <springProfile name="dev">
        <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
            <encoder>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %highlight(%-5level) [%thread] %cyan(%logger{36}) - %msg%n</pattern>
            </encoder>
        </appender>

        <root level="INFO">
            <appender-ref ref="CONSOLE"/>
        </root>
    </springProfile>

    <!-- 生产环境：文件输出 + JSON 格式 -->
    <springProfile name="prod">
        <!-- 按天轮转的日志文件 -->
        <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>${LOG_FILE}.log</file>
            <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
                <fileNamePattern>${LOG_FILE}.%d{yyyy-MM-dd}.%i.log.gz</fileNamePattern>
                <maxFileSize>100MB</maxFileSize>
                <maxHistory>30</maxHistory>
                <totalSizeCap>2GB</totalSizeCap>
            </rollingPolicy>
            <encoder>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level [%thread] %logger{36} - %msg%n</pattern>
            </encoder>
        </appender>

        <!-- 错误日志单独一份文件 -->
        <appender name="ERROR_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
            <file>${LOG_FILE}-error.log</file>
            <filter class="ch.qos.logback.classic.filter.ThresholdFilter">
                <level>ERROR</level>
            </filter>
            <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
                <fileNamePattern>${LOG_FILE}-error.%d{yyyy-MM-dd}.%i.log.gz</fileNamePattern>
                <maxFileSize>50MB</maxFileSize>
                <maxHistory>60</maxHistory>
                <totalSizeCap>1GB</totalSizeCap>
            </rollingPolicy>
            <encoder>
                <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level [%thread] %logger{36} - %msg%n</pattern>
            </encoder>
        </appender>

        <root level="INFO">
            <appender-ref ref="FILE"/>
            <appender-ref ref="ERROR_FILE"/>
        </root>
    </springProfile>
</configuration>
```

关键配置解析：

- **`SizeAndTimeBasedRollingPolicy`** —— 同时按时间和大小轮转。每天一个文件，单个文件超过 100MB 自动分割，保留 30 天历史，总大小不超过 2GB
- **`ERROR_FILE` + `ThresholdFilter`** —— ERROR 级别的日志单独输出一份文件，保留 60 天。排查线上问题时直接看这个文件，不用从海量 INFO 中过滤
- **`springProfile`** —— dev 环境只输出到控制台（方便开发），prod 环境输出到文件（方便运维）

Node.js 中实现同样的文件轮转，通常需要 winston-daily-rotate-file：

```js
const winston = require("winston");
const DailyRotateFile = require("winston-daily-rotate-file");

const transport = new DailyRotateFile({
  filename: "logs/cms-%DATE%.log",
  datePattern: "YYYY-MM-DD",
  maxSize: "100m",
  maxFiles: "30d",
});
```

功能等价，但 Logback 的 XML 配置支持更丰富的策略——比如按日志级别分文件、按包名分文件、条件化输出。winston 的配置是 JavaScript 对象，灵活但表达力有限。

## JSON 结构化日志：对接 ELK

现代生产环境的日志通常用 ELK（Elasticsearch + Logstash + Kibana）或 Loki 收集和分析。这些系统需要 JSON 格式的日志。Logback 通过 `LogstashEncoder` 支持：

```xml
<!-- 添加依赖：logstash-logback-encoder -->
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

```xml
<appender name="JSON_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>${LOG_FILE}.json.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
        <fileNamePattern>${LOG_FILE}-json.%d{yyyy-MM-dd}.%i.log.gz</fileNamePattern>
        <maxFileSize>100MB</maxFileSize>
        <maxHistory>30</maxHistory>
        <totalSizeCap>2GB</totalSizeCap>
    </rollingPolicy>
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <customFields>{"app":"${APP_NAME}"}</customFields>
    </encoder>
</appender>
```

输出效果：

```json
{
  "@timestamp": "2026-03-30T10:15:30.123Z",
  "level": "INFO",
  "thread": "http-nio-8080-exec-1",
  "logger": "com.example.cms.service.PostService",
  "message": "查询文章, id=42",
  "app": "cms"
}
```

Node.js 的 pino 天生就是 JSON 格式输出，这是它最大的卖点：

```js
const pino = require("pino");
const logger = pino();

logger.info({ postId: 42 }, "查询文章");
// 输出: {"level":30,"time":1722368130123,"postId":42,"msg":"查询文章"}
```

pino 的结构化日志开箱即用，Logback 需要额外配置 LogstashEncoder。但从另一个角度看，Logback 的文本格式默认就是可读的，开发时更友好；pino 的 JSON 格式在控制台里很难读，需要装 `pino-pretty` 才行。

## MDC：请求追踪的利器

MDC（Mapped Diagnostic Context）是 SLF4J 的一个关键特性——它允许你在日志中自动附加上下文信息（请求 ID、用户 ID、客户端 IP 等），而不需要手动在每个日志调用中传递这些参数。

这是 Logback 相对于 Node.js 日志库的一个显著优势。Node.js 的 winston/pino 没有类似的线程级上下文机制（因为 Node.js 是单线程的，不需要）。但 Node.js 可以用 `AsyncLocalStorage` 实现类似效果。

先写一个 Filter 在请求入口设置 MDC：

```java
@Component
public class MdcFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain filterChain) throws ServletException, IOException {
        try {
            // 生成请求 ID（如果客户端传了就复用，否则生成新的）
            String requestId = request.getHeader("X-Request-Id");
            if (requestId == null || requestId.isEmpty()) {
                requestId = UUID.randomUUID().toString().replace("-", "");
            }

            MDC.put("requestId", requestId);
            MDC.put("clientIp", request.getRemoteAddr());
            MDC.put("method", request.getMethod());
            MDC.put("uri", request.getRequestURI());

            // 在响应头中返回请求 ID
            response.setHeader("X-Request-Id", requestId);

            filterChain.doFilter(request, response);
        } finally {
            // 请求结束后清除 MDC，避免线程复用时数据污染
            MDC.clear();
        }
    }
}
```

然后在 `logback-spring.xml` 的 pattern 中引用 MDC 的值：

```xml
<pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level [reqId=%X{requestId}] [%thread] %logger{36} - %msg%n</pattern>
```

`%X{requestId}` 就是引用 MDC 中 `requestId` 的值。输出效果：

```
2026-03-30 10:15:30.123 INFO  [reqId=a1b2c3d4e5f6] [http-nio-8080-exec-1] c.e.c.s.PostService - 查询文章, id=42
2026-03-30 10:15:30.156 WARN  [reqId=a1b2c3d4e5f6] [http-nio-8080-exec-1] c.e.c.s.PostService - 文章不存在, id=42
2026-03-30 10:15:30.200 ERROR [reqId=a1b2c3d4e5f6] [http-nio-8080-exec-1] c.e.c.e.GlobalExceptionHandler - 未捕获异常
```

所有属于同一个请求的日志都有相同的 `reqId`。在 ELK 中搜索这个 ID 就能还原完整的请求链路。

MDC 基于 `ThreadLocal` 实现——每个线程有自己的 MDC 上下文。这意味着在异步场景中（`@Async` 方法、`CompletableFuture`），子线程默认拿不到父线程的 MDC。上一篇提到的 `@Async` 上下文传递问题，在日志层面也会出现。解决办法是在异步任务启动时手动传递：

```java
// 异步方法中手动传递 MDC
@Async("cmsTaskExecutor")
public void sendNotification(Long postId) {
    // 需要在调用方传入 requestId，子线程手动设置
    String requestId = MDC.get("requestId");
    // ... 在子线程中 MDC.get("requestId") 是 null
}
```

对于这个问题，Spring 提供了 `TaskDecorator`，可以自动将 MDC 从主线程传递到子线程：

```java
@Bean("cmsTaskExecutor")
public Executor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(4);
    executor.setMaxPoolSize(8);
    executor.setQueueCapacity(100);
    executor.setThreadNamePrefix("cms-async-");
    // 自动传递 MDC 到子线程
    executor.setTaskDecorator(runnable -> {
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        return () -> {
            try {
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    });
    executor.initialize();
    return executor;
}
```

Node.js 中用 `AsyncLocalStorage` 实现类似的请求上下文传递：

```js
const { AsyncLocalStorage } = require("async_hooks");

const asyncContext = new AsyncLocalStorage();

app.use((req, res, next) => {
  asyncContext.run({ requestId: uuid(), clientIp: req.ip }, () => next());
});

// 在任何地方获取请求 ID
function logWithContext(message, data) {
  const { requestId } = asyncContext.getStore() || {};
  logger.info({ requestId, ...data }, message);
}
```

`AsyncLocalStorage` 和 MDC 的原理相同——都是线程/协程级别的存储。但 `AsyncLocalStorage` 在 Node.js 的异步环境中天然工作（事件循环单线程，上下文自动沿 Promise 链传递），MDC 在 Java 的多线程环境中需要手动处理跨线程传递。

## CMS 实战：慢查询日志

给 CMS 加一个慢查询检测——任何数据库查询超过 500ms 就记录 WARN 日志：

```java
@Aspect
@Component
@Slf4j
public class SlowQueryAspect {

    private static final long SLOW_QUERY_THRESHOLD_MS = 500;

    @Around("execution(* com.example.cms.repository.*.*(..))")
    public Object logSlowQuery(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return joinPoint.proceed();
        } finally {
            long elapsed = System.currentTimeMillis() - start;
            if (elapsed > SLOW_QUERY_THRESHOLD_MS) {
                String method = joinPoint.getSignature().toShortString();
                log.warn("慢查询检测: {} 耗时 {}ms", method, elapsed);
            }
        }
    }
}
```

利用第 8 篇学的 AOP 知识，用切面拦截所有 Repository 方法，记录执行时间。这比在每个 Repository 方法里手写 `log.debug` 要干净得多。输出效果：

```
2026-03-30 10:15:30.123 WARN  [reqId=a1b2c3d4e5f6] [http-nio-8080-exec-1] c.e.c.a.SlowQueryAspect - 慢查询检测: PostRepository.findTop10ByStatusAndViewCountGreaterThanOrderByViewCountDesc(..) 耗时 623ms
```

结合 MDC 的 requestId，你可以精确定位是哪个用户请求触发了慢查询，然后进一步分析。

## 完整对比

| 维度           | Node.js (winston/pino)                  | Java (SLF4J + Logback)                  |
| -------------- | --------------------------------------- | --------------------------------------- |
| 架构           | 直接使用（无门面）                      | 门面模式（SLF4J）+ 实现（Logback）      |
| 日志级别       | debug, info, warn, error（4 级）        | trace, debug, info, warn, error（5 级） |
| 级别精细化控制 | 全局一个级别，或用 `debug` 库的命名空间 | 按包名/类名独立控制级别                 |
| 结构化日志     | pino 天生 JSON，winston 需配置          | 需 LogstashEncoder                      |
| 请求上下文     | AsyncLocalStorage（需手动封装）         | MDC（框架内置，pattern 直接引用）       |
| 文件轮转       | winston-daily-rotate-file 插件          | Logback 内置 RollingFileAppender        |
| 异步日志       | 天然异步（事件循环）                    | 需 AsyncAppender                        |
| 配置方式       | JavaScript 对象                         | XML（logback-spring.xml）               |
| 性能           | pino 极快（流式 JSON）                  | Logback 异步模式性能优秀                |

## Trade-off 分析

**SLF4J + Logback 的收益：**

- 门面模式的设计让日志实现可替换。你写的 `log.info()` 调用永远不变，底层的 Logback、Log4j2、JUL 随便换。这在企业级项目中很重要——不同模块可能由不同团队维护，统一日志门面避免了解析冲突
- 按包名精细化控制日志级别是杀手级功能。调试 JPA 时开 `org.hibernate.SQL: DEBUG`，调试 Security 时开 `org.springframework.security: DEBUG`，不影响其他模块的日志输出。Node.js 的日志库没有这个能力
- MDC + Logback 的 pattern 机制让请求追踪变得简单。`%X{requestId}` 一行配置，所有日志自动带上请求 ID。不需要改任何业务代码
- Logback 的 RollingFileAppender 功能完善。按天 + 按大小轮转、压缩归档、总大小限制——这些都是内置功能，不需要额外插件
- Spring Boot 的 `logback-spring.xml` 支持 profile，dev 和 prod 用不同的日志配置，非常方便

**代价：**

- 配置复杂。`logback-spring.xml` 的 XML 语法对习惯了 JavaScript 的开发者来说不友好。各种 Appender、Encoder、RollingPolicy、Filter 的组合方式需要时间去理解
- MDC 在异步场景中需要手动处理。`@Async` 方法的子线程拿不到父线程的 MDC，需要配置 `TaskDecorator`。Node.js 的 `AsyncLocalStorage` 在这个方面反而更省心
- 默认输出不是 JSON。pino 开箱就是 JSON 格式，可以直接对接 ELK。Logback 需要额外引入 `logstash-logback-encoder` 并配置 JSON Appender
- 启动时的日志噪音。Spring Boot 启动时会输出大量 INFO 日志（Banner、Bean 初始化、条件评估报告），在没有 profile 匹配的情况下很难过滤

如果你问"winston/pino 和 Logback 哪个更好"，这不是一个公平的比较。pino 在"轻量、快速、JSON 优先"上胜出，它几乎是 Node.js 项目的最佳日志选择。Logback 在"企业级日志管理"上胜出——按包名控制级别、MDC 请求追踪、profile 匹配、文件轮转策略，这些都是企业项目需要的。

## 回顾与预告

到这篇为止，CMS 的可观测性基础设施已经搭建好：

- SLF4J 是日志门面，Logback 是实现。代码中只依赖 SLF4J 的 API
- 五个日志级别（TRACE/DEBUG/INFO/WARN/ERROR）和按包名精细控制
- `logback-spring.xml` 支持 profile 匹配，dev 用控制台，prod 用文件 + JSON
- MDC 实现请求级别的上下文传递，配合 `%X{requestId}` 在所有日志中自动附加
- 慢查询检测用 AOP 切面实现，结合 MDC 的 requestId 精确定位问题

经过 14 篇文章，CMS 已经具备了比较完整的能力：REST API、JPA 数据访问、参数校验、全局异常处理、AOP、Spring Security、JWT 认证、Redis 缓存、异步任务、WebSocket 实时通信、日志系统。但代码质量怎么样？你确定那些 Service 方法写得没问题？改了一行代码会不会导致另一个接口挂掉？

下一篇讲测试策略——用 `@WebMvcTest` 测试 Controller、用 `@DataJpaTest` 测试 Repository、用 MockMvc 做集成测试，对比 Spring Boot Test 和 Jest 的差异。
