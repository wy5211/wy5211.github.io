CMS 的文章详情页上，用户 A 发了一条评论，用户 B 和用户 C 正在看同一篇文章——但他们要刷新页面才能看到新评论。这个体验在 2026 年已经不可接受了。实时推送评论是基本功能。

在 Node.js 里你大概率会用 Socket.io。npm install、几行代码、前后端都搞定。Socket.io 的自动重连、房间（Room）、命名空间（Namespace）让它几乎成了 Node.js 实时通信的事实标准。但 Socket.io 不是标准 WebSocket——它在原生 WebSocket 上封装了一层自己的协议，客户端和服务端必须都用 Socket.io 才能通信。

Spring Boot 的 WebSocket 方案走的是另一条路：原生 WebSocket + STOMP 协议。STOMP（Simple Text Oriented Messaging Protocol）是一个轻量级的消息协议，在 WebSocket 之上提供发布/订阅模型。理解 STOMP 的工作方式，是从 Socket.io 转到 Spring WebSocket 的关键。

## 原生 WebSocket vs STOMP

原生 WebSocket 只提供了一件事：一个持久的全双工 TCP 连接。你可以通过它收发消息，但"怎么定义消息格式"、"怎么路由消息到不同的处理器"、"怎么实现发布/订阅"——这些 WebSocket 协议本身不定义，全靠你自己设计。

```java
// 原生 WebSocket 处理器——你需要自己处理消息路由和分发
@Component
public class CommentWebSocketHandler extends TextWebSocketHandler {

    private final Map<String, WebSocketSession> sessions = new ConcurrentHashMap<>();

    @Override
    public void afterConnectionEstablished(WebSocketSession session) {
        sessions.put(session.getId(), session);
    }

    @Override
    protected void handleTextMessage(WebSocketSession session, TextMessage message) {
        // 你需要自己解析消息、判断目标、广播给谁
        // 这就是没有协议层时你要自己做的事情
        for (WebSocketSession s : sessions.values()) {
            if (s.isOpen()) {
                s.sendMessage(message);
            }
        }
    }
}
```

这段代码和 Socket.io 的 `io.emit()` 看起来差不多，但有一个本质区别：Socket.io 内置了 Room 机制，你可以 `io.to("post:42").emit()` 只向关注某篇文章的用户广播。原生 WebSocket 没有这个能力——你需要自己维护"谁订阅了哪个频道"的映射关系。

STOMP 解决的就是这个问题。它在 WebSocket 之上定义了消息的格式和路由规则，让你用类似 JMS（Java Message Service）的方式使用 WebSocket：

```
客户端 -> 服务端（发送消息）:
SUBSCRIBE
id:sub-1
destination:/topic/posts/42/comments

客户端 -> 服务端（发送消息）:
SEND
destination:/app/posts/42/comments
content-type:application/json

{"content":"写得好！","authorId":5}

服务端 -> 客户端（推送消息）:
MESSAGE
destination:/topic/posts/42/comments
content-type:application/json

{"id":100,"content":"写得好！","authorName":"张三","createdAt":"2026-03-30T10:00:00"}
```

`SUBSCRIBE` 订阅频道，`SEND` 发送消息到目的地，服务端处理后将结果推送到订阅者。这就是 STOMP 的全部核心——它不传输数据，只定义消息的"信封格式"和"路由规则"。

## WebSocket + STOMP 配置

添加依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-websocket</artifactId>
</dependency>
```

配置类：

```java
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        // 消息代理的目的地前缀
        // 客户端订阅 "/topic/posts/42/comments" 时，消息代理负责路由
        registry.enableSimpleBroker("/topic", "/queue");
        // 应用目的地前缀
        // 客户端发送到 "/app/posts/42/comments" 的消息，由 @MessageMapping 处理
        registry.setApplicationDestinationPrefixes("/app");
        // 用户目的地前缀（点对点消息）
        registry.setUserDestinationPrefix("/user");
    }

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        // WebSocket 握手端点
        registry.addEndpoint("/ws")
            .setAllowedOrigins("*");
        // 启用 SockJS fallback（不支持 WebSocket 的浏览器降级方案）
        registry.addEndpoint("/ws")
            .setAllowedOrigins("*")
            .withSockJS();
    }
}
```

这段配置定义了三条路由规则：

- **`/topic/**`** —— 消息代理前缀。客户端 `SUBSCRIBE /topic/posts/42/comments`订阅某篇文章的评论频道。Spring 的`SimpleBroker` 负责将消息路由到所有订阅者
- **`/app/**`** —— 应用前缀。客户端 `SEND /app/posts/42/comments`发送消息，Spring 根据这个前缀找到对应的`@MessageMapping` 方法处理
- **`/user/**`\*\* —— 用户前缀。用于点对点消息，只推送给指定用户

这和 Socket.io 的概念对应关系：

| Socket.io                      | Spring WebSocket + STOMP                                             |
| ------------------------------ | -------------------------------------------------------------------- |
| `io.on('connection')`          | `registerStompEndpoints` + `afterConnectionEstablished`              |
| `socket.join('post:42')`       | `SUBSCRIBE /topic/posts/42/comments`                                 |
| `io.to('post:42').emit()`      | `messagingTemplate.convertAndSend("/topic/posts/42/comments", data)` |
| `socket.emit('event', data)`   | `SEND /app/posts/42/comments`                                        |
| `socket.on('event', callback)` | `SUBSCRIBE /topic/...` + 自动接收 MESSAGE                            |

## CMS 评论实时推送

先定义消息 DTO：

```java
public class CommentMessage {
    private Long id;
    private String content;
    private String authorName;
    private String authorAvatar;
    private LocalDateTime createdAt;
    // getters and setters
}
```

消息控制器（等价于 Socket.io 的事件处理器）：

```java
@Controller
@Slf4j
public class CommentWebSocketController {

    private final CommentService commentService;
    private final SimpMessagingTemplate messagingTemplate;

    public CommentWebSocketController(CommentService commentService,
                                       SimpMessagingTemplate messagingTemplate) {
        this.commentService = commentService;
        this.messagingTemplate = messagingTemplate;
    }

    @MessageMapping("/posts/{postId}/comments")
    public void handleComment(CommentMessage message,
                               @DestinationVariable Long postId) {
        // 1. 持久化评论到数据库
        Comment saved = commentService.addComment(
            postId, message.getAuthorName(), message.getContent());

        // 2. 构建推送消息
        CommentMessage notification = new CommentMessage();
        notification.setId(saved.getId());
        notification.setContent(saved.getContent());
        notification.setAuthorName(saved.getAuthor().getUsername());
        notification.setAuthorAvatar(saved.getAuthor().getAvatar());
        notification.setCreatedAt(saved.getCreatedAt());

        // 3. 推送给所有订阅了这篇文章评论频道的用户
        messagingTemplate.convertAndSend(
            "/topic/posts/" + postId + "/comments", notification);
    }
}
```

`@MessageMapping` 和 `@RequestMapping` 的关系值得注意：前者处理 WebSocket 消息，后者处理 HTTP 请求。它们工作在不同的协议层，但 Spring 让它们的编程模型保持一致。

`SimpMessagingTemplate` 是向客户端推送消息的工具。`convertAndSend` 将对象序列化为 JSON 后推送到指定目的地。所有订阅了 `/topic/posts/42/comments` 的客户端都会收到这条消息。

REST 接口和 WebSocket 共存。用户提交评论的 HTTP 接口仍然需要：

```java
@RestController
@RequestMapping("/api/posts/{postId}/comments")
public class CommentController {

    private final CommentService commentService;
    private final SimpMessagingTemplate messagingTemplate;

    public CommentController(CommentService commentService,
                              SimpMessagingTemplate messagingTemplate) {
        this.commentService = commentService;
        this.messagingTemplate = messagingTemplate;
    }

    @PostMapping
    public Result<Comment> createComment(@PathVariable Long postId,
                                          @RequestBody @Valid CommentCreateRequest request,
                                          Authentication authentication) {
        Comment comment = commentService.addComment(postId,
            authentication.getName(), request.getContent());

        // 通过 WebSocket 推送给在线用户
        CommentMessage notification = new CommentMessage();
        notification.setId(comment.getId());
        notification.setContent(comment.getContent());
        notification.setAuthorName(comment.getAuthor().getUsername());
        notification.setCreatedAt(comment.getCreatedAt());
        messagingTemplate.convertAndSend(
            "/topic/posts/" + postId + "/comments", notification);

        return Result.success(comment);
    }
}
```

注意：评论可以通过两种方式提交——WebSocket 的 `@MessageMapping` 和 HTTP 的 `@PostMapping`。两者都会触发 WebSocket 推送。`SimpMessagingTemplate` 是线程安全的，你可以在任何地方（Controller、Service、定时任务）向 WebSocket 客户端推送消息。

## 认证与授权

WebSocket 连接建立后，你需要知道"这条消息是谁发的"。Spring 支持在握手阶段携带认证信息：

```java
@Configuration
public class WebSocketSecurityConfig {

    @Bean
    public ChannelInterceptor authInterceptor(JwtUtil jwtUtil) {
        return new ChannelInterceptor() {
            @Override
            public Message<?> preSend(Message<?> message, MessageChannel channel) {
                StompHeaderAccessor accessor = MessageHeaderAccessor
                    .getAccessor(message, StompHeaderAccessor.class);

                if (StompCommand.CONNECT.equals(accessor.getCommand())) {
                    String token = accessor.getFirstNativeHeader("Authorization");
                    if (token != null && token.startsWith("Bearer ")) {
                        // 验证 JWT，设置用户身份
                        String username = jwtUtil.getUsername(token.substring(7));
                        accessor.setUser(() -> username);
                    }
                }
                return message;
            }
        };
    }
}
```

注册拦截器：

```java
@Override
public void configureClientInboundChannel(ChannelRegistration registration) {
    registration.interceptors(authInterceptor(jwtUtil));
}
```

之后在 `@MessageMapping` 方法中，你可以通过 `Principal` 参数获取当前用户：

```java
@MessageMapping("/posts/{postId}/comments")
public void handleComment(CommentMessage message,
                           @DestinationVariable Long postId,
                           Principal principal) {
    // principal.getName() 就是 JWT 中的用户名
    commentService.addComment(postId, principal.getName(), message.getContent());
}
```

## 前端连接

用 `@stomp/stompjs` 连接：

```js
import { Client } from "@stomp/stompjs";

const client = new Client({
  brokerURL: "ws://localhost:8080/ws",
  reconnectDelay: 5000,
});

client.onConnect = () => {
  // 订阅文章 42 的评论频道
  client.subscribe("/topic/posts/42/comments", (message) => {
    const comment = JSON.parse(message.body);
    // 将新评论追加到页面
    appendComment(comment);
  });
};

client.activate();
```

和 Socket.io 的对比：

```js
// Socket.io 的等价写法
const socket = io("http://localhost:3000");

socket.on("connect", () => {
  socket.join("post:42"); // 加入房间
});

socket.on("newComment", (comment) => {
  appendComment(comment);
});
```

Socket.io 的 API 更简洁（`join` / `emit` / `on`），STOMP 的 API 更标准化（`SUBSCRIBE` / `SEND` / `MESSAGE`）。前者上手快，后者更适合大型系统——因为 STOMP 的消息路由和 Spring 的 `@MessageMapping` 天然集成。

## 完整对比

| 维度      | Node.js + Socket.io                         | Spring Boot + STOMP                       |
| --------- | ------------------------------------------- | ----------------------------------------- |
| 协议      | 自定义协议（非标准 WebSocket）              | STOMP 标准协议                            |
| 消息路由  | `io.to(room).emit(event)`                   | `@MessageMapping` + destination           |
| 发布/订阅 | Room 机制（服务端维护）                     | STOMP SUBSCRIBE/MESSAGE（协议层）         |
| 自动重连  | 内置                                        | `@stomp/stompjs` 内置                     |
| 降级方案  | 长轮询 fallback                             | SockJS fallback                           |
| 认证集成  | middleware 中间件                           | ChannelInterceptor                        |
| 消息格式  | 自定义（通常 JSON）                         | STOMP 帧格式 + JSON payload               |
| 横向扩展  | Redis Adapter（`@socket.io/redis-adapter`） | 外部消息代理（RabbitMQ / ActiveMQ）       |
| 学习成本  | 低，API 直观                                | 中，需要理解 STOMP 协议和 Spring 消息模型 |

## Trade-off 分析

**Spring WebSocket + STOMP 的收益：**

- 标准化。STOMP 是一个正式的协议，不是某个库的私有实现。这意味着你可以用任何支持 STOMP 的客户端（Java、JavaScript、Python）接入，不绑定特定语言的 SDK
- 和 Spring 生态深度集成。`@MessageMapping` 的编程模型和 `@RequestMapping` 一致，`SimpMessagingTemplate` 可以在任何 Spring Bean 中注入使用，`SecurityContext` 可以通过拦截器传递到 WebSocket 会话
- 横向扩展路径清晰。当单机不够时，`enableSimpleBroker` 可以换成 `enableStompBrokerRelay`，对接外部的 RabbitMQ 或 ActiveMQ。多个应用实例通过消息代理共享 WebSocket 消息

**代价：**

- 概念更多。原生 WebSocket -> STOMP -> Message Broker -> `@MessageMapping` -> `SimpMessagingTemplate`——理解整个链路需要消化多个抽象层。Socket.io 只有一个概念：`io`
- 前端代码量更多。Socket.io 的客户端 API 是面向 JavaScript 开发者设计的，简洁直观；`@stomp/stompjs` 需要你理解 STOMP 的 SUBSCRIBE/SEND 语义
- `SimpleBroker` 有局限。Spring 内置的 `SimpleBroker` 不支持集群——它把订阅关系存在内存里。多实例部署必须切换到外部消息代理。Socket.io 通过 `@socket.io/redis-adapter` 实现集群更简单
- 没有内置"房间"概念。STOMP 用 destination（目的地）做路由，等价于 Socket.io 的 Room，但语义不同。Socket.io 的 `socket.join("post:42")` 更直觉，STOMP 的 `SUBSCRIBE /topic/posts/42/comments` 更正式

如果你问"Socket.io 和 Spring WebSocket 哪个好"，答案取决于你的约束条件。小型项目、快速原型、团队熟悉 Node.js——Socket.io 是更好的选择。企业级项目、需要和 Spring Security 集成、未来可能需要消息队列做横向扩展——Spring WebSocket + STOMP 更合适。

## 回顾与预告

到这篇为止，CMS 的实时通信能力已经实现：

- 理解了原生 WebSocket 和 STOMP 的关系——前者是传输层，后者是消息协议层
- `WebSocketMessageBrokerConfigurer` 配置了消息代理和目的地前缀
- `@MessageMapping` 处理客户端发来的消息，`SimpMessagingTemplate` 向客户端推送消息
- JWT 认证通过 `ChannelInterceptor` 集成到 WebSocket 握手阶段
- REST 接口和 WebSocket 可以共存，共享同一个推送通道

但到目前为止，CMS 的"可观测性"几乎为零。出了问题只能看控制台输出，无法追踪一个请求从进入到返回的完整路径，无法区分不同环境的日志级别，无法把日志发送到集中式的日志系统。
