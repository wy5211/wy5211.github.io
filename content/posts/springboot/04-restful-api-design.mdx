CMS 项目的文章 CRUD 已经跑起来了，但如果你仔细审视当前的 API 设计，会发现几个问题：GET /api/posts 返回的是一个裸数组，前端拿不到分页元信息（总共多少条、有没有下一页）；创建文章成功返回 200 而不是 201；删除成功返回的响应体结构和其他接口不一致；按状态筛选、按时间排序这些查询参数还没有支持。

更关键的问题是：没有一个统一的响应格式。`PostController` 有时返回 `List<Post>`，有时返回 `ResponseEntity<Post>`，有时返回 `ResponseEntity<Void>`。前端对接的时候，每个接口的响应结构都不一样，解析逻辑要写三套。

这篇来解决这些问题。核心目标只有一个：**让 API 的设计从"能用"变成"规范"。**

## 路径参数 vs 查询参数：什么时候用哪个

RESTful API 设计的第一个决策点是：某个参数应该放在路径里还是查询字符串里？

规则很直接：

- **路径参数**标识资源。`/api/posts/42` 里的 `42` 唯一确定了一篇文章，去掉它 URL 就没有意义了。路径参数对应 Spring Boot 的 `@PathVariable`，Express 的 `req.params`。
- **查询参数**过滤、排序、分页。`/api/posts?status=PUBLISHED&page=1&size=10` 里，去掉任何一个参数 URL 仍然有效——只是返回的结果集不同。查询参数对应 Spring Boot 的 `@RequestParam`，Express 的 `req.query`。

对于 CMS 的文章查询，URL 应该设计成这样：

```
GET /api/posts?page=1&size=10&status=PUBLISHED&sort=createdAt,desc
```

`page`、`size`、`status`、`sort` 都是查询参数。它们不标识具体资源，而是对 `/api/posts` 这个资源集合做筛选和分页。如果客户端想看第 42 篇文章，才用路径参数：`GET /api/posts/42`。

一个容易犯的错误是把查询条件塞进路径：`GET /api/posts/status/PUBLISHED/page/1`。这在功能上可以工作，但语义上把筛选条件和资源定位混在了一起，也增加了路由匹配的复杂度。查询字符串才是 HTTP 协议为这类参数设计的归属。

## 统一响应格式：ApiResponse\<T\>

前端最怕的就是每个接口返回结构不一样。一个统一的响应包装类能解决这个问题。

```java
public class ApiResponse<T> {

    private int code;
    private String message;
    private T data;
    private LocalDateTime timestamp;

    private ApiResponse(int code, String message, T data) {
        this.code = code;
        this.message = message;
        this.data = data;
        this.timestamp = LocalDateTime.now();
    }

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(200, "success", data);
    }

    public static <T> ApiResponse<T> success(String message, T data) {
        return new ApiResponse<>(200, message, data);
    }

    public static ApiResponse<Void> created() {
        return new ApiResponse<>(201, "created", null);
    }

    public static ApiResponse<Void> noContent() {
        return new ApiResponse<>(204, "no content", null);
    }

    public static ApiResponse<Void> error(int code, String message) {
        return new ApiResponse<>(code, message, null);
    }

    // getter 省略
}
```

这个类用泛型 `T` 包装实际数据，所有接口返回 `ApiResponse<T>` 而不是裸对象。前端拿到的每个响应都有 `code`、`message`、`data`、`timestamp` 四个字段，解析逻辑只需要写一套。

和 Express 的对比：Express 里没有强制约束，全靠团队约定。你可以写一个 `res.success(data)` 辅助函数，也可以在每个 handler 里 `res.json({ code: 200, data })`。Spring Boot 的泛型类提供了编译期的类型检查——如果你声明了 `ApiResponse<Post>`，`data` 字段就只能是 `Post`，放一个 `List<Post>` 进去编译器直接报错。这种约束在项目变大之后价值极高。

## 分页响应：PageResult\<T\>

分页是列表接口的核心功能。前端不仅需要数据列表，还需要知道分页元信息：当前第几页、每页多少条、总共多少条、有没有下一页。

```java
public class PageResult<T> {

    private List<T> items;
    private int page;
    private int size;
    private long total;
    private int totalPages;

    public PageResult(List<T> items, int page, int size, long total) {
        this.items = items;
        this.page = page;
        this.size = size;
        this.total = total;
        this.totalPages = (int) Math.ceil((double) total / size);
    }

    public boolean hasNext() {
        return page < totalPages;
    }

    public boolean hasPrevious() {
        return page > 1;
    }

    // getter 省略
}
```

`PageResult` 封装了分页的全部信息。`hasNext()` 和 `hasPrevious()` 是给前端用的便捷方法——前端不需要自己算 `page < totalPages`，直接判断就行。

在 Express 里，分页通常是在 handler 里手动拼装：

```js
const page = parseInt(req.query.page) || 1;
const size = parseInt(req.query.size) || 10;
const offset = (page - 1) * size;
const [rows, count] = await Promise.all([
  Post.findAndCountAll({ offset, limit: size }),
  Post.count(),
]);
res.json({
  items: rows,
  page,
  size,
  total: count,
  totalPages: Math.ceil(count / size),
});
```

每个列表接口都要重复这段逻辑。Spring Boot 虽然没有内置的 `PageResult`（那是 Spring Data JPA 的 `Page<T>` 做的事，第 5 篇会讲），但自己写一个也就二十行代码。关键是这个类是**全局复用**的——所有需要分页的接口都返回 `ApiResponse<PageResult<T>>`。

## HTTP 状态码：不是 200 就够了

当前的 `PostController` 有一个不规范的细节：创建文章返回 200，删除文章返回 200。HTTP 状态码是 API 契约的一部分，用对了能让前端少写很多判断逻辑。

几个关键的状态码选择：

| 场景         | 状态码           | 含义                 | Controller 返回方式                    |
| ------------ | ---------------- | -------------------- | -------------------------------------- |
| 查询成功     | 200 OK           | 请求成功，有响应体   | `ApiResponse.success(data)`            |
| 创建成功     | 201 Created      | 资源已创建           | `ResponseEntity.status(201).body(...)` |
| 删除成功     | 204 No Content   | 操作成功，无响应体   | `ResponseEntity.noContent().build()`   |
| 资源不存在   | 404 Not Found    | 请求的资源不存在     | `ResponseEntity.notFound().build()`    |
| 参数校验失败 | 400 Bad Request  | 客户端请求格式有误   | 全局异常处理（第 7 篇）                |
| 未认证       | 401 Unauthorized | 缺少或无效的身份凭证 | Spring Security（第 9 篇）             |

这里有一个设计选择：是用 HTTP 状态码来表达业务结果，还是把所有接口都返回 200 然后在响应体的 `code` 字段里区分？

两种风格都有人用。RESTful 的正统做法是**充分利用 HTTP 状态码**——创建返回 201，未找到返回 404。`ApiResponse` 里的 `code` 字段和 HTTP 状态码保持一致，作为对响应的补充描述。另一种做法是所有响应都返回 200，用 `code: 200/400/404/500` 来区分——这在某些前端框架（比如早期的 axios 拦截器）里处理起来更方便，因为它不需要针对不同状态码写不同的 catch 逻辑。

两种都不是"错的"。对于 CMS 项目，我们选择 RESTful 的正统做法。

## CMS 演进：完善 PostController

现在把上述设计落地到代码里。先升级 `PostService`，支持按状态筛选和按时间排序：

```java
@Service
public class PostService {

    private final List<Post> posts = new ArrayList<>();
    private long nextId = 3;

    @PostConstruct
    public void init() {
        posts.add(new Post(1L, "深入理解 Java 内存模型", "java-memory-model",
            "JMM 定义了线程如何与内存交互...", PostStatus.PUBLISHED,
            1L, 1520, LocalDateTime.of(2026, 3, 15, 10, 0)));
        posts.add(new Post(2L, "Spring Boot 3 新特性详解", "spring-boot-3-features",
            "Spring Boot 3 带来了诸多改进...", PostStatus.DRAFT,
            2L, 0, LocalDateTime.of(2026, 3, 20, 14, 30)));
    }

    /**
     * 分页查询，支持按状态筛选和按时间排序
     */
    public PageResult<Post> findAll(int page, int size,
                                     PostStatus status, String sort) {
        List<Post> filtered = posts.stream()
            .filter(p -> status == null || p.getStatus() == status)
            .sorted((a, b) -> "desc".equals(sort)
                ? b.getCreatedAt().compareTo(a.getCreatedAt())
                : a.getCreatedAt().compareTo(b.getCreatedAt()))
            .collect(Collectors.toList());

        long total = filtered.size();
        int totalPages = (int) Math.ceil((double) total / size);
        int from = Math.max(0, (page - 1) * size);
        int to = Math.min(filtered.size(), from + size);

        return new PageResult<>(
            filtered.subList(from, to), page, size, total
        );
    }

    public Optional<Post> findById(Long id) {
        return posts.stream()
            .filter(p -> p.getId().equals(id))
            .findFirst();
    }

    public Post create(String title, String content, String slug) {
        String finalSlug = (slug != null && !slug.isBlank())
            ? slug : generateSlug(title);

        boolean slugExists = posts.stream()
            .anyMatch(p -> p.getSlug().equals(finalSlug));
        if (slugExists) {
            throw new IllegalArgumentException(
                "Slug '" + finalSlug + "' already exists");
        }

        Post post = new Post(
            nextId++, title, finalSlug, content,
            PostStatus.DRAFT, 1L, 0, LocalDateTime.now()
        );
        posts.add(post);
        return post;
    }

    public Optional<Post> update(Long id, String title, String content,
                                  String slug, PostStatus status) {
        return findById(id).map(existing -> {
            if (title != null) existing.setTitle(title);
            if (content != null) existing.setContent(content);
            if (slug != null) existing.setSlug(slug);
            if (status != null) existing.setStatus(status);
            return existing;
        });
    }

    public boolean delete(Long id) {
        return posts.removeIf(p -> p.getId().equals(id));
    }

    private String generateSlug(String title) {
        return title.toLowerCase()
            .replaceAll("[^a-z0-9\\u4e00-\\u9fa5]+", "-")
            .replaceAll("^-|-$", "");
    }
}
```

注意 `findAll` 的变化：新增了 `status` 和 `sort` 参数。`status` 为 null 时不过滤（返回所有状态），`sort` 决定按创建时间正序还是倒序排列。这是查询参数的典型用法——可选的、用于筛选和排序的参数。

然后是重写后的 `PostController`：

```java
@RestController
@RequestMapping("/api/posts")
public class PostController {

    private final PostService postService;

    public PostController(PostService postService) {
        this.postService = postService;
    }

    /**
     * 分页查询文章
     * GET /api/posts?page=1&size=10&status=PUBLISHED&sort=createdAt,desc
     */
    @GetMapping
    public ApiResponse<PageResult<Post>> getPosts(
            @RequestParam(defaultValue = "1") int page,
            @RequestParam(defaultValue = "10") int size,
            @RequestParam(required = false) PostStatus status,
            @RequestParam(defaultValue = "desc") String sort) {
        return ApiResponse.success(
            postService.findAll(page, size, status, sort)
        );
    }

    /**
     * 获取单篇文章
     * GET /api/posts/42
     */
    @GetMapping("/{id}")
    public ResponseEntity<ApiResponse<Post>> getPost(
            @PathVariable Long id) {
        return postService.findById(id)
            .map(post -> ResponseEntity.ok(
                ApiResponse.success(post)))
            .orElse(ResponseEntity.notFound().build());
    }

    /**
     * 创建文章，返回 201 Created
     * POST /api/posts
     */
    @PostMapping
    public ResponseEntity<ApiResponse<Post>> createPost(
            @RequestBody @Valid CreatePostRequest request) {
        Post post = postService.create(
            request.getTitle(),
            request.getContent(),
            request.getSlug()
        );
        return ResponseEntity
            .status(HttpStatus.CREATED)
            .body(ApiResponse.success("Post created", post));
    }

    /**
     * 更新文章
     * PUT /api/posts/42
     */
    @PutMapping("/{id}")
    public ResponseEntity<ApiResponse<Post>> updatePost(
            @PathVariable Long id,
            @RequestBody @Valid UpdatePostRequest request) {
        return postService.update(
            id, request.getTitle(), request.getContent(),
            request.getSlug(), request.getStatus()
        ).map(post -> ResponseEntity.ok(
            ApiResponse.success("Post updated", post)))
         .orElse(ResponseEntity.notFound().build());
    }

    /**
     * 删除文章，返回 204 No Content
     * DELETE /api/posts/42
     */
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deletePost(@PathVariable Long id) {
        return postService.delete(id)
            ? ResponseEntity.noContent().build()
            : ResponseEntity.notFound().build();
    }
}
```

和上一篇的 Controller 对比，变化体现在三个方面：

**响应结构统一了。** 所有接口都返回 `ApiResponse<T>` 包装，前端拿到的每个响应都有 `code`、`message`、`data`、`timestamp`。

**分页接口增强了。** 从简单的 `subList` 升级为 `PageResult`，支持 `status` 筛选和 `sort` 排序。前端可以通过 `?status=PUBLISHED&sort=desc` 只看已发布的文章并按时间倒序。

**状态码更准确了。** 创建返回 201，删除返回 204，查不到返回 404。不再是所有接口一律 200。

有一个细节值得注意：`deletePost` 返回 `ResponseEntity<Void>` 而不是 `ApiResponse<Void>`。204 No Content 规范上不应该有响应体，所以不需要包装。这也是为什么 `ApiResponse` 提供了 `noContent()` 方法——虽然这里没用到，但在某些前端框架需要统一解析响应体的场景下，你也可以让删除接口返回 200 + `ApiResponse.noContent()`，而不是 204。

## Tag API：资源嵌套设计

CMS 需要标签功能。一篇文章可以有多个标签，一个标签下有多篇文章。这是经典的多对多关系。

先定义 `Tag` 实体：

```java
public class Tag {
    private Long id;
    private String name;
    private String slug;
    private String color;

    // 构造函数、getter、setter 省略
}
```

`TagController` 提供 Tag 的 CRUD：

```java
@RestController
@RequestMapping("/api/tags")
public class TagController {

    private final TagService tagService;

    public TagController(TagService tagService) {
        this.tagService = tagService;
    }

    @GetMapping
    public ApiResponse<List<Tag>> getAllTags() {
        return ApiResponse.success(tagService.findAll());
    }

    @PostMapping
    public ResponseEntity<ApiResponse<Tag>> createTag(
            @RequestBody @Valid CreateTagRequest request) {
        Tag tag = tagService.create(
            request.getName(), request.getSlug(), request.getColor()
        );
        return ResponseEntity
            .status(HttpStatus.CREATED)
            .body(ApiResponse.success("Tag created", tag));
    }
}
```

资源嵌套出现在"获取某篇文章的标签"这个场景。有两种设计方式：

```
方案 A：GET /api/posts/42/tags        —— 嵌套在文章资源下
方案 B：GET /api/tags?postId=42       —— 通过查询参数关联
```

方案 A 更 RESTful——标签是文章的子资源，URL 层级关系清晰表达了所有权。方案 B 更灵活——前端不需要知道嵌套关系，只需要在一个 Tag 列表接口上加过滤条件。

实际项目中两种都用。对于"某篇文章有哪些标签"这种强关联查询，方案 A 更合适。对于"查询所有标签，同时支持按文章过滤"这种弱关联查询，方案 B 更合适。CMS 项目里我们两种都支持：

```java
// TagController 中新增：按文章 ID 查询标签
@GetMapping
public ApiResponse<List<Tag>> getTags(
        @RequestParam(required = false) Long postId) {
    if (postId != null) {
        return ApiResponse.success(
            tagService.findByPostId(postId));
    }
    return ApiResponse.success(tagService.findAll());
}

// PostController 中新增：获取文章的标签
@GetMapping("/{postId}/tags")
public ResponseEntity<ApiResponse<List<Tag>>> getPostTags(
        @PathVariable Long postId) {
    return postService.findById(postId)
        .map(post -> ResponseEntity.ok(
            ApiResponse.success(tagService.findByPostId(postId))))
        .orElse(ResponseEntity.notFound().build());
}
```

`/api/posts/{postId}/tags` 走路径参数，语义上表示"这篇文章的标签"。`/api/tags?postId=42` 走查询参数，语义上表示"筛选标签列表"。两者功能重叠但语义不同，前端根据场景选择。

## Express vs Spring Boot 对比

| 概念     | Express                     | Spring Boot                                     |
| -------- | --------------------------- | ----------------------------------------------- |
| 路径参数 | `req.params.id`             | `@PathVariable Long id`                         |
| 查询参数 | `req.query.page`            | `@RequestParam(defaultValue="1") int page`      |
| 类型转换 | 手动 `parseInt()`           | 框架自动转换，类型不匹配返回 400                |
| 参数可选 | 手动 `\|\| defaultValue`    | `@RequestParam(required=false)`                 |
| 枚举参数 | 手动校验字符串值            | `PostStatus status` 自动反序列化                |
| 统一响应 | 自定义中间件或辅助函数      | 泛型类 `ApiResponse<T>` + 编译期检查            |
| 分页     | 手动拼装 offset/limit/count | 自定义 `PageResult<T>` 或 Spring Data `Page<T>` |
| 状态码   | `res.status(201).json()`    | `ResponseEntity.status(CREATED).body()`         |
| 响应控制 | 副作用调用 `res.xxx()`      | 返回值 `ResponseEntity<T>`                      |

有一个差异在表格里没有展开，但值得注意：Spring Boot 的 `@RequestParam` 直接接收枚举类型。请求 `?status=PUBLISHED` 会自动映射到 `PostStatus.PUBLISHED`。如果客户端传了一个不存在的枚举值（`?status=INVALID`），框架返回 400。在 Express 里你需要手动做这个校验。

## Trade-off：RESTful 规范的边界

给 API 套上 RESTful 规范之后，接口的可预测性和前端对接体验确实提升了。但这套规范不是万能的。

**过度设计风险。** 对于一个内部管理后台的 CRUD，严格区分 200/201/204/404 的意义不大——前端反正都要处理接口调用的结果，多一个状态码只是多一个判断分支。RESTful 规范的价值在开放 API（对外提供给第三方调用的接口）上体现得最明显，因为第三方开发者需要通过 URL 和状态码来理解你的 API 行为，而不是靠看你的文档。

**统一响应格式的代价。** `ApiResponse<T>` 意味着每个响应都多了一层嵌套。对于流式响应（Server-Sent Events）、文件下载、WebSocket 这些非 JSON 场景，`ApiResponse` 完全不适用。一个实际项目里，RESTful JSON API 只是接口的一部分，还有可能需要 gRPC、GraphQL、SSE 等多种协议。不要试图用一个 `ApiResponse` 统管所有场景。

**分页设计的分歧。** 这篇用的 page-based 分页（`page=1&size=10`），另一种常见的方式是 cursor-based 分页（`cursor=abc123&limit=10`）。前者简单直观，适合中小规模数据；后者性能更好（不需要 `COUNT(*)`），适合大数据量和不稳定排序场景（比如按热度排序的动态 feed）。CMS 的文章列表用 page-based 就够了，但如果要做评论的无限滚动，cursor-based 会是更好的选择。

## 回顾

上一篇我们把业务逻辑从 Controller 抽到了 Service 层，CMS 有了 Controller-Service 两层结构。这篇在这两层之上加了 RESTful 规范：

- 路径参数标识资源（`/api/posts/42`），查询参数用于筛选和分页（`?status=PUBLISHED&page=1`）
- `ApiResponse<T>` 统一了所有接口的响应结构，前端解析逻辑只需写一套
- `PageResult<T>` 封装了分页元信息，列表接口从返回裸数组升级为返回完整的分页对象
- HTTP 状态码从"一律 200"升级为语义化使用（201 Created、204 No Content、404 Not Found）
- Tag 的 CRUD 引入了资源嵌套设计（`/api/posts/{id}/tags`）和查询参数关联（`/api/tags?postId=42`）

CMS 的 API 层现在有了规范的 URL 设计和响应格式。但有一个根本性的问题还没有解决：数据还是存在内存的 `List` 里。下一篇我们会引入 Spring Data JPA，把数据持久化到 MySQL，用 Repository 模式替代手动的 `List` 操作。
