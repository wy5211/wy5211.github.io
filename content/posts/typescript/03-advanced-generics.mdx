## 从一个真实的痛点开始

你接手了一个 API 客户端的封装工作，后端的响应结构是这样的：

```typescript
type ApiResponse<T> = {
  code: number;
  message: string;
  data: T;
};

// 实际用起来
type UserDetailResponse = ApiResponse<{
  user: {
    id: string;
    name: string;
    profile: {
      avatar: string;
      bio: string;
    };
    orders: Array<{
      id: string;
      amount: number;
      createdAt: string;
    }>;
  };
}>;
```

需求来了：单元测试里需要一个 mock 数据，大部分字段可以不填，但类型还要保持正确。你第一反应是用 `Partial`：

```typescript
type MockResponse = Partial<UserDetailResponse>;
// 结果：code, message, data 都变成了可选的
// 但 data.user.profile.avatar 依然是必填的
```

`Partial` 只作用于第一层。你想要的是**所有叶子节点都变成可选的**，嵌套多深都行。这就是 `DeepPartial` 要解决的问题。

在解决这个问题之前，我们先回顾几个泛型的进阶技巧，它们是构建复杂类型工具的基础。

## 多个泛型参数：一个不够用的时候

初学泛型时，大多数场景一个 `<T>` 就够了。但有些工具函数天然涉及两种不同的类型，强行只用一个泛型反而会把类型搞得更复杂。

看一个 `pluck` 函数——从对象数组中提取某个字段：

```typescript
function pluck<T, K extends keyof T>(items: T[], key: K): T[K][] {
  return items.map((item) => item[key]);
}

const users = [
  { id: "usr_001", name: "张三", role: "admin" },
  { id: "usr_002", name: "李四", role: "editor" },
  { id: "usr_003", name: "王五", role: "viewer" },
];

const ids = pluck(users, "id"); // string[]
const roles = pluck(users, "role"); // string[]

// 传了不存在的 key，编译器直接报错
// pluck(users, 'age');  // Error
```

这里 `T` 是对象的类型，`K` 是字段的 key。它们各司其职：`K extends keyof T` 确保 key 一定存在于对象上，`T[K]` 取出对应字段的类型作为返回值。如果只有一个泛型参数，你没办法在保证类型安全的前提下表达"从 T 中取 K 这个字段的类型"这个关系。

再看看我们系列主线里的 API 客户端场景：

```typescript
async function request<TResponse, TBody = void>(
  url: string,
  body?: TBody
): Promise<TResponse> {
  /* ... */
}

// TBody 默认是 void，不传 body 时不需要指定
const users = await request<User[]>("/users");
// 需要传 body 时，两个泛型都派上用场
const newUser = await request<User, CreateUserDTO>("/users", { name: "张三" });
```

`TResponse` 和 `TBody` 是两个完全独立的维度，分开声明比合并成一个对象泛型更清晰。

## 泛型约束：extends 不是继承

`extends` 在泛型里的含义和在 class 里不同。泛型里的 `extends` 是约束，意思是"T 必须满足某种条件"。

```typescript
function mergeDeep<T extends object>(target: T, source: Partial<T>): T {
  const result = { ...target };
  for (const key in source) {
    const targetVal = result[key];
    const sourceVal = source[key];
    if (
      typeof targetVal === "object" &&
      targetVal !== null &&
      typeof sourceVal === "object" &&
      sourceVal !== null
    ) {
      result[key] = mergeDeep(targetVal, sourceVal);
    } else if (sourceVal !== undefined) {
      result[key] = sourceVal as T[Extract<keyof T, string>];
    }
  }
  return result;
}

const defaults = {
  pagination: { page: 1, pageSize: 20 },
  filters: { status: "active", region: "cn" },
};

// source 的类型被约束为 Partial<typeof defaults>，写错字段名会报错
const config = mergeDeep(defaults, {
  pagination: { pageSize: 50 },
});
```

没有 `T extends object` 这个约束，`mergeDeep(42, {})` 这种无意义的调用也能通过类型检查。约束把错误扼杀在编译期。

约束也可以更灵活，不只是 `object`。比如只接受带有 `id` 字段的对象：

```typescript
function findById<T extends { id: string }>(
  items: T[],
  id: string
): T | undefined {
  return items.find((item) => item.id === id);
}
```

## 默认泛型参数：减少调用处的噪音

有些泛型参数在绝大多数场景下都是同一个值，每次都手动传一遍很烦。默认泛型参数解决这个问题：

```typescript
// 不用默认值 —— 调用时必须写
function createCache<T>(): Map<string, T> {
  /* ... */
}
const cache1 = createCache<User>(); // 每次都要写泛型

// 用默认值 —— 简单场景不用写
function createCache<T = unknown>(): Map<string, T> {
  /* ... */
}
const cache2 = createCache(); // Map<string, unknown>
const cache3 = createCache<User>(); // Map<string, User>
```

默认值放在最右边。多泛型参数时，有默认值的参数必须排在没有默认值的后面：

```typescript
function request<TResponse, TBody = void, TQuery = void>() // ...
{
  /* ... */
}

// 三种调用方式都合法
request<User[]>();
request<User[], CreateUserDTO>("/users", body);
request<User[], void, PaginationQuery>("/users", undefined, query);
```

## 递归泛型：DeepPartial 的实现

回到文章开头的问题。`DeepPartial` 需要递归地遍历类型结构，把每一层的属性都变成可选。核心思路和递归函数一样：有一个基准情况（base case），有一个递归情况（recursive case）。

```typescript
type DeepPartial<T> = T extends object
  ? {
      [K in keyof T]?: T[K] extends object ? DeepPartial<T[K]> : T[K];
    }
  : T;
```

拆解一下这个类型：

- `T extends object`：如果 T 是对象，进入映射类型
- `[K in keyof T]?`：每一层的属性都变成可选
- `T[K] extends object`：如果属性的值还是对象，继续递归
- 否则直接返回 `T[K]` 本身（基准情况）

用在我们的 API 响应上：

```typescript
type MockResponse = DeepPartial<UserDetailResponse>;
// 现在整个嵌套结构的所有叶子节点都是可选的
const mock: MockResponse = {
  code: 200,
  data: {
    user: {
      name: "测试用户",
      // profile、orders 全都可以不填
    },
  },
};
```

同理可以写 `DeepReadonly`：

```typescript
type DeepReadonly<T> = T extends object
  ? {
      readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
    }
  : T;

type FrozenConfig = DeepReadonly<{
  database: { host: string; port: number };
  cache: { ttl: number; maxSize: number };
}>;
// FrozenConfig.database.host 是 readonly 的，所有嵌套层都是
```

这里有一个需要注意的地方：TypeScript 对递归类型的深度有限制，大约 50 层。对于正常业务数据来说完全够用，但如果你有一个超过 50 层嵌套的类型（极其罕见），TypeScript 会报递归过深的错误。

实际项目中，用 `ts-toolbelt` 或 `type-fest` 这类库提供的实现更稳妥，它们处理了数组、Date、函数等边界情况。上面的实现是一个教学版本，帮你理解原理。

## 类型安全的路径访问

在 API 客户端里，经常需要用字符串路径访问嵌套数据。比如 `get(response, 'data.user.profile.avatar')`。lodash 的 `get` 函数能做到运行时安全，但类型上返回的是 `any`。

用泛型配合模板字面量类型（我们在 02 篇学过的）和条件类型（01 篇的内容），可以做到编译期类型安全：

```typescript
type PathValue<
  T,
  Path extends string,
> = Path extends `${infer Key}.${infer Rest}`
  ? Key extends keyof T
    ? PathValue<T[Key], Rest>
    : never
  : Path extends keyof T
    ? T[Path]
    : never;

type User = {
  id: string;
  name: string;
  profile: {
    avatar: string;
    bio: string;
  };
  orders: Array<{ id: string; amount: number }>;
};

// 编译期推导出正确类型
type A = PathValue<User, "id">; // string
type B = PathValue<User, "profile.avatar">; // string
type C = PathValue<User, "name">; // string

// 路径不存在，返回 never
type D = PathValue<User, "profile.foo">; // never
type E = PathValue<User, "address.city">; // never
```

`PathValue` 的工作原理：

1. `Path extends '${infer Key}.${infer Rest}'` —— 用模板字面量匹配，把路径拆成第一段 `Key` 和剩余部分 `Rest`
2. 如果 `Key` 是 `T` 的属性，就递归处理 `T[Key]` 和 `Rest`
3. 如果路径里没有 `.` 了（基准情况），直接从 `T` 中取对应属性

这就是前三篇文章知识的综合运用：泛型负责类型参数传递，条件类型负责分支判断，模板字面量负责字符串解析。

## API 响应类型变换

实际项目中，后端返回的 JSON 和前端使用的数据类型之间经常有差异。比如后端的时间字段都是 ISO 字符串，前端想统一转成 `Date` 对象：

```typescript
type StringToDate<T> = T extends string
  ? Date
  : T extends object
    ? { [K in keyof T]: StringToDate<T[K]> }
    : T;

// 后端返回的类型
type ApiUser = {
  id: string;
  name: string;
  createdAt: string;
  updatedAt: string;
  profile: {
    birthday: string;
  };
};

// 前端使用的类型 —— 所有 string 变成了 Date
type DomainUser = StringToDate<ApiUser>;
// {
//   id: Date;    // 注意：id 也会变成 Date
//   name: Date;
//   createdAt: Date;
//   updatedAt: Date;
//   profile: { birthday: Date };
// }
```

这个实现太粗暴了——它把所有 string 都变成了 Date。更精确的做法是只转换特定字段名：

```typescript
type DateFields = "createdAt" | "updatedAt" | "birthday" | "deletedAt";

type TransformDates<T> = T extends object
  ? {
      [K in keyof T]: K extends DateFields
        ? Date
        : T[K] extends object
          ? TransformDates<T[K]>
          : T[K];
    }
  : T;

type DomainUser = TransformDates<ApiUser>;
// {
//   id: string;
//   name: string;
//   createdAt: Date;      // 只有日期字段被转换
//   updatedAt: Date;
//   profile: { birthday: Date };
// }
```

另一个常见需求是去除响应中的 `null` 字段，得到一个更确定的类型：

```typescript
type NonNullableDeep<T> = T extends null | undefined
  ? never
  : T extends object
    ? { [K in keyof T]: NonNullableDeep<T[K]> }
    : T;

type RawResponse = {
  user: {
    name: string | null;
    email: string | null;
    phone: string;
  };
};

type CleanResponse = NonNullableDeep<RawResponse>;
// {
//   user: {
//     name: string;
//     email: string;
//     phone: string;
//   }
// }
```

这些类型变换工具组合起来，就能在 API 客户端层自动完成"后端类型"到"前端类型"的转换，调用方拿到的永远是干净、精确的类型。

## 泛型复杂度的取舍

泛型是强大的，但不是免费的。每多一层泛型嵌套，类型检查的时间就会增加，代码的可读性也会下降。

一个内部只用了一次的工具函数，加三个泛型参数可能是过度设计：

```typescript
// 过度设计 —— 一个只在一处使用的私有函数
function formatItem<
  T extends { id: string; createdAt: string },
  K extends keyof T,
>(item: T, fields: K[]): Pick<T, K> & { formattedAt: string } {
  // ...
}

// 实际上直接写可能更清晰
function formatUserItem(item: ApiUser, fields: (keyof ApiUser)[]) {
  // ...
}
```

泛型的价值在于**复用**和**类型安全的自动化**。如果一个类型工具要在多个地方使用，或者它能让调用方省去大量的类型标注，那投入精力设计泛型是值得的。我们的 `DeepPartial`、`PathValue`、`TransformDates` 都属于这一类——写一次，到处用，每次用都能自动推导出正确的类型。

反过来，如果一个函数只在一处调用，参数类型和返回类型都确定，硬编码类型比泛型更直观。读代码的人不需要在脑子里"展开"泛型就能理解函数签名，维护成本更低。

判断标准很简单：如果去掉泛型，调用方需要手动标注的类型变多了，或者类型安全性下降了，那就该用泛型。否则，保持简单。
