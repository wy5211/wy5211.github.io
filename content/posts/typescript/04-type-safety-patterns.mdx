线上出了一个 bug：用户投诉自己的订单被删了。排查下来发现，后端日志里 `deleteUser` 接口收到的参数是一个 orderId，格式是 `ord_a1x9k2m`。但 TypeScript 编译没有任何报错——因为 userId 和 orderId 都是 `string`，对编译器来说它们完全一样。

这就是 TypeScript 结构化类型系统的盲区：它只关心值的"形状"，不关心值的"语义"。`string` 就是 `string`，不管这个字符串里存的是用户 ID 还是订单 ID。

前三篇文章我们一直在造工具——条件类型、模板字面量、高级泛型。它们返回的类型越来越精确，但基础类型的粒度不够，再精确也是白搭。这篇来解决这个"底层类型太粗"的问题。

## 品牌类型：给 string 贴标签

解决问题的思路很直接：能不能让编译器区分"都是 string 但语义不同"的值？

TypeScript 本身没有"名义类型"（nominal type）的概念——不像 Java 里 `class UserId extends String`，两个类即使结构相同也是不同类型。但我们可以用交叉类型模拟这个行为：

```typescript
// 品牌类型的核心：string + 一个不可能在运行时存在的属性
type UserId = string & { readonly __brand: "UserId" };
type OrderId = string & { readonly __brand: "OrderId" };
```

为什么这能工作？`string & { readonly __brand: 'UserId' }` 要求一个值既要是 string，又要有一个 `__brand` 属性。正常的 string 字面量不满足第二个条件，所以你写 `const id: UserId = "abc"` 会报错。只有通过专门的创建函数生成的值才能通过类型检查。

这个"通过交叉类型给类型加标签"的技巧就叫品牌类型（Branded Types）。名字来自它给一个通用类型打上了独特的"品牌"标识。

创建品牌类型的值需要一个工厂函数，顺带做运行时校验：

```typescript
function createUserId(value: string): UserId {
  if (!/^usr_[a-z0-9]{7}$/.test(value)) {
    throw new Error(`Invalid UserId: ${value}`);
  }
  return value as UserId;
}

function createOrderId(value: string): OrderId {
  if (!/^ord_[a-z0-9]{7}$/.test(value)) {
    throw new Error(`Invalid OrderId: ${value}`);
  }
  return value as OrderId;
}
```

现在回到开头的 bug 场景：

```typescript
function deleteUser(id: UserId): Promise<void> {
  /* ... */
}

const userId = createUserId("usr_a1x9k2m");
const orderId = createOrderId("ord_b3x7n4p");

deleteUser(orderId); // 编译错误：OrderId 不能赋值给 UserId
```

`as UserId` 断言绕过了类型检查，但这是一种"有意识的越权"——你写 `as` 的时候知道自己在做什么。而直接传参混了，编译器能帮你拦住。

在我们的 API 客户端里，路由参数的类型可以升级为品牌类型：

```typescript
const apiRoutes = {
  'GET /users/:userId/orders/:orderId': {
    params: { userId: UserId; orderId: OrderId }
    response: Order
  }
} as const
```

这样调用 `client.get('/users/:userId/orders/:orderId', { userId, orderId })` 时，参数类型的混淆会在编译期就被发现。

## 类型谓词：运行时验证 + 编译期收窄

品牌类型的创建函数用了 `as` 断言，它告诉编译器"相信我，这个值就是目标类型"。这在函数内部是合理的——我们刚做完校验。但在其他地方收到一个 unknown 的值时，我们需要更安全的方式：类型谓词（Type Predicate）。

```typescript
// 类型谓词：value is UserId
// 如果函数返回 true，TypeScript 会把 value 收窄为 UserId
function isUserId(value: unknown): value is UserId {
  return typeof value === "string" && /^usr_[a-z0-9]{7}$/.test(value);
}
```

类型谓词的语法是 `value is Type`。它的作用是：当函数返回 `true` 时，调用方代码块中 `value` 的类型会被收窄为 `Type`。这和 `typeof`、`instanceof` 的行为一致，只是你自定义了收窄的条件。

在 API 客户端里，类型谓词非常适合验证后端返回的数据：

```typescript
interface User {
  id: UserId;
  name: string;
  email: string;
  role: "admin" | "user";
}

function isUser(value: unknown): value is User {
  if (typeof value !== "object" || value === null) return false;
  const obj = value as Record<string, unknown>;
  return (
    isUserId(obj.id) &&
    typeof obj.name === "string" &&
    typeof obj.email === "string" &&
    (obj.role === "admin" || obj.role === "user")
  );
}

// 使用
const data: unknown = await response.json();
if (isUser(data)) {
  // data 被收窄为 User，可以安全访问 data.id, data.name 等
  console.log(data.id); // UserId
}
```

类型谓词的一个关键点是：返回 `false` 时不会报错，只是不做类型收窄。如果你需要在数据无效时直接中断流程，应该用断言函数。

## 断言函数：不通过就抛异常

断言函数（Assertion Functions）的语法是 `asserts value is Type`。和类型谓词不同，它如果返回了（没有抛异常），TypeScript 就认为 `value` 一定是 `Type`：

```typescript
function assertUserId(value: unknown): asserts value is UserId {
  if (typeof value !== "string" || !/^usr_[a-z0-9]{7}$/.test(value)) {
    throw new Error(`Expected UserId, got ${typeof value}: ${value}`);
  }
}
```

使用时不需要 `if` 判断：

```typescript
const raw: unknown = JSON.parse(responseBody);
assertUserId(raw.id);
// 执行到这里，raw.id 已经是 UserId 类型
// 如果 raw.id 不合法，函数已经抛异常了，不会走到这里
```

类型谓词和断言函数的选择取决于你的错误处理策略：能恢复用谓词，不能恢复用断言。在 API 客户端里，通常用断言——如果后端返回的数据格式都不对，后面的逻辑也没法继续。

## 工具类型的实现原理

TypeScript 内置了一批工具类型：`Partial<T>`、`Pick<T, K>`、`Omit<T, K>`、`Record<K, V>` 等。你平时可能直接用，但不太清楚它们是怎么实现的。

它们的核心都是**映射类型**（Mapped Types）：

```typescript
// Partial 的实现
type Partial<T> = {
  [P in keyof T]?: T[P];
};
```

`[P in keyof T]` 遍历 `T` 的所有属性名，`?:` 让每个属性变成可选。就这么简单。理解了这个模式，其他工具类型一看就懂。

`Pick<T, K>` 从 `T` 中选取部分属性：

```typescript
type Pick<T, K extends keyof T> = {
  [P in K]: T[P];
};
```

`K extends keyof T` 约束 K 必须是 T 的属性名之一。然后只遍历 K 中的属性。

`Omit<T, K>` 排除部分属性。它的实现用到了 `Exclude` 条件类型：

```typescript
type Omit<T, K extends keyof T> = Pick<T, Exclude<keyof T, K>>;
```

先从 `keyof T` 中排除掉 K，再用 Pick 选取剩余属性。两层组合。

`Record<K, V>` 构造一个键值对类型：

```typescript
type Record<K extends string | number | symbol, V> = {
  [P in K]: V;
};
```

和前面几个不同，Record 不操作已有类型，而是从零构造一个新的对象类型。

`Readonly<T>` 把所有属性变成只读：

```typescript
type Readonly<T> = {
  readonly [P in keyof T]: T[P];
};
```

映射类型里还可以加 `readonly`、`?` 修饰符，也可以用 `-` 前缀移除修饰符（`-readonly`、`-?`），这是后续高级模式的基础。

映射类型本身不难，难的是灵活组合。比如我们给 API 客户端定义响应类型时，经常需要"从完整类型中提取部分字段，而且全部可选"——这就是 `Partial<Pick<T, K>>`。

## Key Remapping：字段名自动转换

TypeScript 4.1 引入了映射类型中的 `as` 子句，允许在遍历属性时重命名 key。这解决了一个非常实际的问题：后端返回 snake_case，前端用 camelCase。

```typescript
// 基础的 key remapping 语法
type Keys = "hello_world" | "foo_bar";
type Mapped = {
  [K in Keys as K extends `${infer A}_${infer B}`
    ? `${A}${Capitalize<B>}`
    : K]: string;
};
// 结果：{ helloWorld: string; fooBar: string }
```

`as` 后面是一个类型表达式。遍历到 `hello_world` 时，模板字面量类型把它匹配成 `hello` 和 `world`，然后用 `Capitalize` 把 `world` 变成 `World`，拼接成 `helloWorld`。上一篇学的模板字面量类型在这里派上了用场。

但要处理任意的 snake_case 字符串，单次匹配不够。比如 `created_at` 没问题，但 `user_avatar_url` 需要处理多个下划线。这就需要递归：

```typescript
type SnakeToCamel<S extends string> = S extends `${infer Head}_${infer Tail}`
  ? `${Head}${Capitalize<SnakeToCamel<Tail>>}`
  : S;

// 'user_avatar_url' → Head='user', Tail='avatar_url'
// 递归：'avatar_url' → Head='avatar', Tail='url'
// 递归：'url' → 不匹配，返回 'url'
// 拼接：'user' + 'Avatar' + 'Url' = 'userAvatarUrl'
```

结合映射类型，就能实现整个对象类型的字段名转换：

```typescript
type CamelCaseKeys<T> = {
  [K in keyof T as K extends string ? SnakeToCamel<K> : K]: T[K];
};
```

现在测试一下：

```typescript
// 后端返回的类型（snake_case）
type BackendUser = {
  user_id: string;
  user_name: string;
  email_address: string;
  created_at: string;
  avatar_url: string | null;
};

// 自动转换为 camelCase
type FrontendUser = CamelCaseKeys<BackendUser>;
// 等价于：
// {
//   userId: string
//   userName: string
//   emailAddress: string
//   createdAt: string
//   avatarUrl: string | null
// }
```

这样在 API 客户端里，后端的 snake_case 响应类型可以自动映射为前端的 camelCase 类型，不需要手写两套类型定义。

## 组合起来：类型安全的 API 响应

把品牌类型和字段名转换组合起来，就能给 API 客户端构建一个完整的类型安全层。

先定义品牌类型的通用工具：

```typescript
type Brand<T, B extends string> = T & { readonly __brand: B };
type UserId = Brand<string, "UserId">;
type OrderId = Brand<string, "OrderId">;
```

后端返回的原始类型：

```typescript
type BackendOrderResponse = {
  order_id: string;
  user_id: string;
  total_amount: number;
  status: "pending" | "paid" | "shipped";
  created_at: string;
  updated_at: string;
};
```

我们需要的转换是：字段名 camelCase + ID 字段升级为品牌类型：

```typescript
type BrandedCamelCase<T> = {
  [K in keyof T as K extends string
    ? SnakeToCamel<K>
    : K]: K extends `${string}_id`
    ? Brand<T[K], Capitalize<SnakeToCamel<K>>>
    : T[K];
};

type Order = BrandedCamelCase<BackendOrderResponse>;
// {
//   orderId: OrderId       // Brand<string, 'OrderId'>
//   userId: UserId         // Brand<string, 'UserId'>
//   totalAmount: number
//   status: 'pending' | 'paid' | 'shipped'
//   createdAt: string
//   updatedAt: string
// }
```

`K extends \`${string}\_id\``判断字段名是否以`\_id`结尾，如果是，就把值类型从`string` 升级为对应的品牌类型。`Capitalize<SnakeToCamel<K>>`会把`order_id`变成`OrderId`，作为品牌标识。

运行时的转换函数也很简单：

```typescript
function snakeToCamel(str: string): string {
  return str.replace(/_([a-z])/g, (_, c) => c.toUpperCase());
}

function transformResponse<T>(data: Record<string, unknown>): T {
  const result: Record<string, unknown> = {};
  for (const [key, value] of Object.entries(data)) {
    result[snakeToCamel(key)] = value;
  }
  return result as T;
}
```

类型层面的 `BrandedCamelCase<T>` 和运行时的 `transformResponse` 配合使用：编译时保证类型正确，运行时完成实际转换。

## 什么时候用品牌类型

品牌类型的核心价值是防止"类型相同但语义不同"的值被混淆。这种 bug 在生产环境里虽然不频繁，但一旦出现后果很严重——删错用户、扣错金额、发错通知。

适合使用品牌类型的场景：

- ID 类字段（userId、orderId、productId 等）。ID 混淆是最常见的，也是后果最严重的。
- 金额（不能把人民币金额当美元金额传给支付接口）。
- 邮箱地址、手机号等有明确格式的字符串。
- 文件路径（防止把文件路径当 URL 传）。

不适合使用的场景：

- 普通的 string 字段（name、title、description）。给它们加品牌类型只会增加代码量，不会防止任何实际 bug。
- 内部工具函数的参数。如果函数只在你自己的模块里用，参数混淆的风险很低。
- 一次性脚本。不值得花时间设计类型。

品牌类型不是免费的午餐。每加一个品牌类型，你就多了一个创建函数、一个类型谓词、一套 `as` 断言。在边界处（比如从 localStorage 读数据、从 URL 参数解析）都需要做类型转换。这增加了代码量和认知负担。关键是找到平衡点——对那些"混了会出大问题"的字段用品牌类型，其余的保持普通类型。
