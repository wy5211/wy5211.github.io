# 条件类型与 infer：TypeScript 的 if-else

你在给团队封装一个统一的 API 调用层。后端的接口返回格式是统一的，外面一层信封，里面才是真正的数据：

```typescript
// 后端返回的实际结构
{
  code: 0,
  message: "ok",
  data: {
    userId: "usr_a1x9k2m",
    email: "zhangsan@example.com",
    avatar: "https://cdn.example.com/avatars/a1x9k2m.webp"
  }
}
```

你的 API 客户端拿到这个响应后，应该把 `data` 里的东西交给业务代码用。业务代码不关心 `code` 和 `message`——那是客户端统一处理的。问题来了：你怎么写一个类型工具，把 `ApiResponse<User>` 里的 `User` 自动提取出来？

泛型做不到这件事。泛型只是"参数化类型"的容器，它本身没有分支能力。你需要的是类型层面的 if-else——条件类型。

## 条件类型：类型层面的分支

JavaScript 里你会写 `typeof x === 'string' ? doA() : doB()` 来做分支判断。TypeScript 的条件类型做的事情几乎一样，只不过判断发生在类型层面：

```typescript
type IsString<T> = T extends string ? true : false;

type A = IsString<"hello">; // true
type B = IsString<42>; // false
```

`T extends U ? X : Y` 这段代码的含义是：如果类型 `T` 可以赋值给类型 `U`，那么整个条件类型的结果是 `X`，否则是 `Y`。这和运行时的三元表达式是一一对应的。

为什么需要这个东西？回到我们开头的问题。API 响应有两种可能：成功时 `data` 里有东西，失败时 `data` 可能是 `null`。你想根据泛型参数来决定类型行为，但泛型本身不会"做判断"——它只是个占位符。条件类型补上了这个能力缺口。

用一个最简单的例子来感受一下：

```typescript
type UnwrapApiResponse<T> = T extends { data: infer D } ? D : T;

type RawUser = {
  code: number;
  message: string;
  data: { userId: string; email: string };
};

type User = UnwrapApiResponse<RawUser>;
// { userId: string; email: string }
```

这里出现了一个新面孔：`infer`。

## infer：在条件分支里"捕获"类型

`infer` 的作用是声明一个类型变量，让 TypeScript 在模式匹配的过程中自动推断它的具体类型。上面的例子里，`T extends { data: infer D }` 的意思是："如果 `T` 是一个包含 `data` 字段的对象，把 `data` 字段的类型记为 `D`"。后面的 `? D` 就直接使用了这个推断出来的类型。

关键点在于：你不需要事先知道 `D` 是什么，TypeScript 会根据传入的实际类型来推断。这就像正则表达式里的捕获组——你定义一个模式，引擎帮你把匹配到的部分提取出来。

让我们看几个更实用的例子，都围绕 API 客户端这个场景。

### 从 Promise 里提取内部类型

API 客户端的每个方法都返回 `Promise`。你经常需要拿到 Promise 内部的类型来做后续的类型操作：

```typescript
type Awaited<T> = T extends Promise<infer Inner> ? Inner : T;

type UserPromise = Promise<{
  userId: string;
  email: string;
}>;

type User = Awaited<UserPromise>;
// { userId: string; email: string }

type PlainString = Awaited<string>;
// string（不是 Promise，原样返回）
```

这个类型工具的逻辑很清晰：传入的类型如果是 `Promise<某个东西>`，就把那个"东西"提取出来；如果不是 Promise，原样返回。TypeScript 标准库里已经内置了 `Awaited`，但理解它的实现原理比直接用它重要得多——因为同样的模式可以应用在任何"容器类型"上。

### 从函数签名里提取返回值和参数

API 客户端里你还会遇到函数类型。比如你想从一个请求处理函数里提取它的参数类型和返回值类型：

```typescript
type GetReturn<T> = T extends (...args: any[]) => infer R ? R : never;
type GetParams<T> = T extends (first: infer P, ...args: any[]) => any
  ? P
  : never;

type FetchUsers = (page: number, pageSize: number) => Promise<User[]>;

type Response = GetReturn<FetchUsers>;
// Promise<User[]>

type FirstArg = GetParams<FetchUsers>;
// number
```

`GetReturn` 的模式匹配规则是：如果 `T` 是一个函数（不管参数是什么，用 `any[]` 忽略掉），就把返回值类型记为 `R`。`GetParams` 类似，但它只关心第一个参数。

这两个工具类型组合起来能做的事情很多。比如你有一批请求函数，想统一提取它们的返回类型做成一个联合类型——在后面的文章里会用到这个技巧。

### 多个 infer 同时使用

一个条件分支里可以放多个 `infer`，各自捕获不同位置的类型：

```typescript
type FunctionParts<T> = T extends (
  first: infer F,
  ...rest: infer R
) => infer Return
  ? { first: F; rest: R; return: Return }
  : never;

type Handler = (id: string, name: string, age: number) => boolean;

type Parts = FunctionParts<Handler>;
// { first: string; rest: [string, number]; return: boolean }
```

这里 `F` 捕获第一个参数，`R` 捕获剩余参数组成的元组，`Return` 捕获返回值。三个 `infer` 各司其职，互不干扰。

有一点需要注意：`infer` 只能在 `extends` 后面的模式里使用，不能在其他地方出现。它不是一个通用的类型变量声明机制，而是专门为条件类型的模式匹配设计的。

## 分布式条件类型：联合类型会自动展开

条件类型有一个行为很容易让人踩坑。假设你写了一个把类型包装成数组的工具：

```typescript
type ToArray<T> = T extends any ? T[] : never;

type Result = ToArray<string | number>;
```

你觉得 `Result` 是什么？是 `(string | number)[]` 吗？不是。实际结果是 `string[] | number[]`。

这就是**分布式条件类型**。规则是：当条件类型作用于一个"裸类型参数"（就是直接用泛型参数 `T` 本身，没有被包裹在元组或其他类型里），并且传入的实际类型是联合类型时，TypeScript 会把联合类型拆开，对每个成员分别计算，最后再把结果合并成联合类型。

等价于：

```typescript
type Result = ToArray<string> | ToArray<number>;
// string[] | number[]
```

这在很多时候是你想要的行为。但有时候你确实需要把联合类型当作一个整体来处理。阻止分布的方法是用元组包一层：

```typescript
type ToArrayNonDist<T> = [T] extends [any] ? T[] : never;

type Result = ToArrayNonDist<string | number>;
// (string | number)[]
```

把 `T` 放进元组 `[T]` 之后，它就不再是"裸类型参数"了，分布式行为不会触发。这个技巧在写工具类型时经常用到。

为什么默认要分布式？因为大多数场景下分布式的结果更精确。`ToArray<string | number>` 返回 `string[] | number[]`，意味着"要么是 string 数组，要么是 number 数组"，这比 `(string | number)[]`（"一个可能同时包含 string 和 number 的数组"）语义更严格。默认更严格的语义是一个合理的设计选择。

## 递归条件类型：处理嵌套的 Promise

回到 `Awaited` 那个例子。如果传入的类型是 `Promise<Promise<User>>` 呢？我们之前写的简单版本只能剥一层：

```typescript
// 简单版本，只能剥一层
type SimpleAwaited<T> = T extends Promise<infer Inner> ? Inner : T;

type R = SimpleAwaited<Promise<Promise<User>>>;
// Promise<User>，不是 User
```

在真实场景里，多层嵌套的 Promise 并不少见。比如一个中间件先包了一层 Promise，API 客户端又包了一层。要彻底剥干净，需要递归：

```typescript
type DeepAwaited<T> = T extends Promise<infer Inner> ? DeepAwaited<Inner> : T;

type R = DeepAwaited<Promise<Promise<User>>>;
// User

type R2 = DeepAwaited<Promise<Promise<Promise<{ orderId: string }>>>>;
// { orderId: string }
```

逻辑是这样的：如果 `T` 是 `Promise<某个东西>`，就把那个东西取出来，再递归调用 `DeepAwaited`。如果不是 Promise 了，就到达递归终点，直接返回 `T`。

TypeScript 支持这种递归条件类型，但有一个深度限制——通常在 45 到 50 层左右。如果你的类型嵌套超过了这个深度，编译器会报 "Type instantiation is excessively deep and possibly infinite" 的错误。在正常业务代码里几乎不可能达到这个限制，但在写极端的类型体操时需要注意。

递归条件类型是 TypeScript 类型系统里非常强大的能力。很多看似复杂的问题——深层嵌套对象的处理、路径类型的解析、模板字面量类型的反复匹配——都可以用递归优雅地解决。后面几篇文章会反复用到这个模式。

## 实战：构建 API 响应解包类型

现在把前面学到的东西组合起来，为我们的 API 客户端写一个完整的响应解包类型。

后端的接口返回格式并不总是完美的。有的接口用 `{ code, message, data }` 包装，有的接口直接返回数据，还有的接口在 data 里嵌套了一层分页结构。我们需要一个能处理这些差异的类型工具。

先定义基础结构：

```typescript
interface User {
  userId: string;
  email: string;
  avatar: string;
}

interface PaginatedResponse<T> {
  code: number;
  message: string;
  data: {
    items: T[];
    total: number;
    page: number;
    pageSize: number;
  };
}

interface PlainResponse<T> {
  code: number;
  message: string;
  data: T;
}
```

然后写解包类型：

```typescript
// 第一层：解包外层信封
type UnwrapEnvelope<T> = T extends { data: infer D } ? D : T;

// 第二层：如果是分页结构，提取 items 数组的元素类型
type ExtractPageItem<T> = T extends { items: (infer I)[] } ? I : T;

// 组合起来
type ApiResult<T> = ExtractPageItem<UnwrapEnvelope<T>>;
```

用实际类型验证一下：

```typescript
type UserResult = ApiResult<PlainResponse<User>>;
// User —— 普通接口，解包信封后直接拿到 User

type UserListResult = ApiResult<PaginatedResponse<User>>;
// User —— 分页接口，解包信封后拿到 { items, total, page, pageSize }，
//         再提取 items 的元素类型，得到 User
```

这两个类型工具各做一件事，然后通过组合完成整个解包流程。这种"小工具组合"的思路比写一个巨大的条件类型更清晰——每个工具的职责单一，出问题了也好排查。

再来看一个更复杂的场景：有的接口返回的数据本身就是异步的，你需要同时处理 Promise 嵌套和信封结构。把 `DeepAwaited` 加进来：

```typescript
type SafeApiResult<T> = ApiResult<DeepAwaited<T>>;

type AsyncUser = SafeApiResult<Promise<PlainResponse<User>>>;
// User —— 先剥掉 Promise，再解包信封

type AsyncUserList = SafeApiResult<Promise<PaginatedResponse<User>>>;
// User —— 先剥掉 Promise，再解包信封，再提取分页元素
```

四个类型工具（`DeepAwaited`、`UnwrapEnvelope`、`ExtractPageItem`、`ApiResult`），每个不超过两行，组合起来能处理 API 响应的各种变形。这就是条件类型的实用价值——你写的不是一行复杂的"万能公式"，而是几个简洁、可组合的小工具。

## 条件类型的代价

条件类型确实强大，但它不是万能的，使用时需要权衡几个问题。

第一是**类型计算开销**。条件类型（尤其是递归的）会在编译时执行类型计算。TypeScript 编译器对类型实例化的深度有硬性限制，超过就会直接报错。在一个大型项目里，如果到处都是复杂的条件类型，编译速度会受到明显影响。你可以在 `tsconfig.json` 里设置 `"noErrorTruncation": true` 来看到完整的错误信息，方便定位是哪个类型导致了过深的实例化。

第二是**可读性**。`T extends (infer U)[] ? DeepAwaited<U> : never` 这种代码对作者来说很清晰，但团队里的其他人可能要盯着看一分钟才能理解。复杂的条件类型最好加上注释说明意图，或者拆成命名清晰的多个小类型。

第三是**调试困难**。当条件类型的结果不符合预期时，你没有办法像调试运行时代码那样 `console.log`。只能靠 IDE 的悬浮提示或者 `type X = ...` 这种辅助类型来逐步排查。TypeScript 4.7+ 提供了 `--generateTrace` 选项，可以输出编译器的类型检查轨迹，但输出内容非常冗长，只适合作为最后的手段。

我的建议是：条件类型用在真正需要类型层面分支的场景——从容器类型中提取内部类型、根据输入类型决定输出类型、处理联合类型的分发。如果用简单的泛型约束就能解决的问题，不要为了"看起来高级"而引入条件类型。工具类型应该服务于类型安全的目标，而不是成为炫技的载体。

下一篇我们会进入另一个领域：模板字面量类型。如果说条件类型给了你在类型层面做判断的能力，那模板字面量类型就给了你在类型层面操作字符串的能力——我们会用它来解决 API 路由参数提取的问题。
