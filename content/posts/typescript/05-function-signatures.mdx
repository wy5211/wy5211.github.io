## 从一个 API 客户端说起

假设你在写一个 HTTP API 客户端，核心方法是 `get`。真实场景里，这个方法的调用方式非常多样：`get('/users')` 获取列表返回 `User[]`，`get('/users/:id', { id: '42' })` 获取单个资源返回 `User`，`get('/users/:id/posts', { id: '42' }, { page: 1 })` 获取子资源返回 `Post[]`。

同一个函数名，三套参数结构，三种返回类型。用单一类型签名怎么写？

最直接的想法是用联合类型，把参数声明为 `string | { id: string } | { id: string; page: number }`。但问题立刻暴露了：返回类型变成了 `Promise<unknown>`，你丢失了参数和返回值之间的对应关系。调用 `get('/users')` 和 `get('/users/:id', { id: '42' })` 在类型层面毫无区别，TypeScript 不知道前者返回数组、后者返回单个对象。用 `any` 更不用说了，直接放弃类型检查。

这就是函数重载要解决的问题：**让同一个函数名根据不同的参数组合，对应不同的返回类型**。

## 函数重载的基本结构

函数重载的写法是：上面放多个签名声明（只有类型，没有实现），最下面放一个实现签名（包含函数体）。以 API 客户端的 `get` 方法为例：

```typescript
class ApiClient {
  // 重载签名 1：无参数，返回列表
  get<T>(path: string): Promise<T[]>;

  // 重载签名 2：有路径参数，返回单个对象
  get<T>(path: string, params: Record<string, string>): Promise<T>;

  // 实现签名：必须兼容上面所有重载
  get<T>(path: string, params?: Record<string, string>): Promise<T | T[]> {
    const url = params ? resolvePath(path, params) : path;
    return fetch(url).then((res) => res.json());
  }
}
```

调用时，TypeScript 会按声明顺序从上往下匹配。`client.get<User>('/users')` 匹配第一个签名，返回类型推导为 `User[]`；`client.get<User>('/users/:id', { id: '42' })` 匹配第二个签名，推导为 `User`。

这里有个关键细节：**重载签名的排列顺序是从具体到宽泛**。如果把宽泛的签名放在前面，TypeScript 会优先匹配它，后面的具体签名就永远不会生效。这跟 `switch` 语句不写 `break` 是类似的道理——匹配到了就不会继续往下走。

## 什么时候用重载，什么时候用联合类型

判断标准很简单：**不同的输入是否对应不同的输出**。

输出类型相同，用联合类型就够了。比如一个 `formatDate` 函数，不管传 `Date` 对象、时间戳数字、还是 ISO 字符串，返回的都是 `string`，实现内部统一转成 `Date` 再格式化就行。这种情况下写重载是过度设计，徒增签名数量，对调用者没有额外帮助。

但输出类型随输入变化时，就需要重载。比如 API 客户端的 `get` 方法，传不同的参数意味着请求不同的资源、拿到不同结构的数据，这种对应关系只有重载能表达。

还有一种实用的判断依据：如果你发现自己写出了 `Promise<A | B | C>` 这样的返回类型，而实际上调用者能根据传入的参数判断出具体是哪一种，那大概率应该用重载把这个联合类型拆开。

## 重载与泛型的组合

真实项目里，API 客户端通常不会只写死几个路由，而是需要一个通用的 endpoint 定义机制。这时重载和泛型就要一起出场了。

思路是定义一个路由映射表（`RouteMap`），把所有端点的路径、参数结构、返回类型集中在一个接口里。然后通过泛型参数 `P` 代表路径字符串，用条件类型 `Extract<RouteMap[P], ...>` 从表中提取对应的类型信息。有参数的路由走一个重载签名，无参数的路由走另一个重载签名，两个签名都从 `RouteMap` 中自动推导参数和返回类型。

```typescript
interface RouteMap {
  "/users": { response: User[] };
  "/users/:id": { params: { id: string }; response: User };
}

class TypedClient {
  get<P extends string>(
    path: P,
    params: Extract<RouteMap[P], { params: any }>["params"]
  ): Promise<Extract<RouteMap[P], { params: any }>["response"]>;
  get<P extends string>(
    path: P
  ): Promise<Extract<RouteMap[P], { params?: never }>["response"]>;
  // 实现签名...
}
```

这个模式的好处是**增量维护**：每新增一个路由，只需要在 `RouteMap` 里加一条定义，类型自动推导，不需要手写新的重载签名。`RouteMap` 本身就成了一个活文档，任何端点的类型信息都能直接查到。

## 装饰器：在运行时保留类型信息

函数重载解决的是编译期的类型精确性问题。但 TypeScript 有一个根本性的限制：**类型擦除**。所有类型信息在编译后都不存在了，运行时拿不到。

这在某些场景下是真实的问题。比如你想在 API 请求发出之前自动校验请求体的结构，校验规则应该从类型定义中派生出来，但运行时根本不知道你的接口定义长什么样。又比如你想根据方法的参数类型自动生成 OpenAPI 文档，同样拿不到类型信息。

装饰器就是用来填补这个鸿沟的。它在声明时给函数、类、属性附加元数据，这些元数据在运行时可以读取和消费。

先看一个请求日志装饰器的例子：

```typescript
function log(target: Function, context: ClassMethodDecoratorContext) {
  return function (this: any, ...args: any[]) {
    console.log(
      `[${new Date().toISOString()}] Calling ${String(context.name)}`
    );
    return (target as Function).apply(this, args);
  };
}

class ApiClient {
  @log
  async get(path: string) {
    return fetch(path);
  }
}
```

注意这里的语法。TypeScript 5.0 引入了新的装饰器标准（Stage 3 proposal），和之前的 `experimentalDecorators` 有本质区别。新语法中，装饰器函数的签名是 `(target, context) => replacement | void`，其中 `context` 是一个包含 `name`、`kind`、`access` 等信息的上下文对象。如果你用的是旧版 `experimentalDecorators`（很多 Angular 和 NestJS 项目仍在用），签名是 `(target, propertyKey, descriptor)` 三参数形式。

实际项目中要注意版本一致性。新项目建议用 TypeScript 5.0+ 的标准装饰器，旧项目迁移时需要仔细确认 API 差异，两者不能混用。

## 实战：为 API 客户端添加校验和日志

回到 API 客户端的场景。我们构建两个装饰器：`@validate` 在请求发出前校验请求体，`@log` 记录请求耗时。

```typescript
function validate(schema: Record<string, string>) {
  return function (target: Function, context: ClassMethodDecoratorContext) {
    return function (this: any, ...args: any[]) {
      const body = args.find((a) => typeof a === "object" && a !== null);
      if (body) {
        for (const [key, expectedType] of Object.entries(schema)) {
          if (!(key in body)) throw new Error(`Missing required field: ${key}`);
          if (typeof body[key] !== expectedType)
            throw new Error(
              `Field ${key} should be ${expectedType}, got ${typeof body[key]}`
            );
        }
      }
      return (target as Function).apply(this, args);
    };
  };
}
```

`@log` 装饰器在之前的基础上增加响应时间记录：在方法调用前记录 `performance.now()`，如果返回值是 Promise，则在 `.then()` 中计算耗时并输出日志。

两个装饰器叠加使用时，执行顺序是从下往上的：先执行 `@validate`（最靠近方法），再执行 `@log`。也就是说 `@log` 包裹的是已经经过校验的方法，它记录的时间包含了校验耗时。如果你想让日志记录的是纯网络请求时间，把装饰器顺序反过来即可。

```typescript
class UserService {
  @log
  @validate({ name: "string", email: "string" })
  async createUser(body: CreateUserBody): Promise<User> {
    const res = await fetch("/api/users", {
      method: "POST",
      body: JSON.stringify(body),
    });
    return res.json();
  }
}
```

这里有一个取舍值得思考：装饰器的校验逻辑是运行时的，而你的 TypeScript 类型已经保证了编译期的类型安全，为什么还需要运行时校验？因为 API 的输入不只有你的前端代码。数据可能来自第三方 Webhook、用户表单提交、或者被 `as any` 绕过的类型检查。编译期类型保护的是"正常路径"——代码本身写得对不对；运行时校验保护的是"异常路径"——外部输入是否可信。两者不是替代关系，而是互补关系。

## 权衡与取舍

函数重载和装饰器各有适用边界，用错了会增加不必要的复杂度。

重载最大的风险是**签名膨胀**。我见过一些库的重载签名超过 10 个，IDE 的自动补全弹出一整屏，开发者根本不知道该用哪个。经验上，一个函数的重载签名控制在 3 到 5 个以内比较合理。如果超过了，说明这个函数承担了太多职责，考虑拆分成多个命名明确的独立方法，比如 `getList`、`getById`、`getWithQuery`。拆分之后每个方法的签名都简单清晰，IDE 补全也会友好得多。

另外，重载只在类型层面起作用，运行时的实现只有一个。这意味着你的实现签名必须兼容所有重载签名，有时候不得不在实现里写一些类型断言或条件判断，这本身就是一种维护成本——每次修改重载签名，都要同步检查实现签名是否还能兼容。

装饰器的主要成本在于**运行时开销和认知负担**。每次调用被装饰的方法，都会多经过一层函数包装。在绝大多数业务场景下这个开销可以忽略不计，但在高频调用的热路径上（比如每秒处理数千个请求的网关中间件），多层装饰器嵌套会有可观的性能损耗。更重要的是，装饰器让控制流变得隐式。读代码的人看到一个 `@log`，需要跳转到装饰器定义才能知道它到底做了什么、在什么条件下会抛异常。这对团队的代码阅读能力有要求，也增加了调试时追踪调用链的难度。

两者的结合点可以这样理解：**重载解决"这个函数能怎么被调用"的问题，装饰器解决"调用时自动发生什么行为"的问题**。它们作用于不同维度，可以独立使用，也可以组合。但在实际项目中，我建议优先用重载来改善类型表达——它的成本几乎为零，收益是直接的类型安全提升。只在确实需要在运行时附加行为（校验、日志、权限检查、缓存）时才引入装饰器，并且用注释清楚地说明每个装饰器的作用。类型系统的价值在于让你更早地发现问题，而装饰器的价值在于让你在运行时自动处理横切关注点——搞清楚你到底需要哪一个，再去用对应的工具。
