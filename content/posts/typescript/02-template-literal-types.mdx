## 从一个真实问题开始

假设你在给团队封装统一的 API 调用层。后端给了你 30 多个接口，每个接口都有对应的路由路径，其中不少带路径参数：

```
GET  /users
GET  /users/:id
POST /users/:id/avatar
GET  /users/:userId/orders/:orderId
PUT  /orders/:orderId/status
```

你需要在调用这些接口时传入正确的参数。`/users/:id` 需要传 `{ id: string }`，`/users/:userId/orders/:orderId` 需要传 `{ userId: string; orderId: string }`。

30 多个接口，每个都手写参数类型？那不仅繁琐，而且后端加个参数你这边就忘了更新，类型和实际接口就对不上了。

能不能让 TypeScript 从路由字符串里**自动提取**出参数类型？

这就是模板字面量类型要解决的问题。

## 先回顾：模板字面量类型是什么

模板字面量类型的语法和 JavaScript 的模板字符串一模一样，只是它出现在类型位置：

```typescript
type Greeting = `hello ${string}`;
// "hello world" | "hello ts" | "hello 任意字符串"
```

上篇我们用条件类型和 `infer` 提取了 API 响应里的类型。这篇我们把同样的思路用在字符串上 -- 从 API 路由中提取参数类型。条件类型负责"判断"，模板字面量负责"拆解字符串"，两者配合就能完成字符串级别的类型操作。

## 联合类型的笛卡尔积

模板字面量类型和联合类型结合时，会产生笛卡尔积。这个特性非常有用，但它也是最容易被忽略的一点。

```typescript
type Event = "user" | "order";
type Action = "login" | "logout" | "create";
type EventName = `${Event}:${Action}`;
// 结果：5 种组合
// "user:login" | "user:logout" | "user:create"
// "order:login" | "order:logout" | "order:create"
```

这不只是语法糖。在实际项目中，你可以用它来约束事件名称、CSS 类名、路由路径等字符串，确保不会拼错：

```typescript
type Method = "GET" | "POST" | "PUT" | "DELETE";
type Resource = "users" | "orders" | "products";
type ApiRoute = `${Method} /${Resource}`;
// "GET /users" | "GET /orders" | "GET /products"
// "POST /users" | "POST /orders" | "POST /products"
// ...共 12 种
```

注意，如果联合类型的成员太多（比如超过几百个），TypeScript 会做内部截断来避免类型爆炸。后面我们会聊到这个性能问题。

## 内置字符串操作类型

TypeScript 提供了四个内置的字符串变换类型，它们都作用于类型层面：

```typescript
type Uppercase<S extends string> = ...
type Lowercase<S extends string> = ...
type Capitalize<S extends string> = ...
type Uncapitalize<S extends string> = ...
```

单独看没什么感觉，但和模板字面量组合起来就有实际用途了。比如后端的 API 返回 snake_case 字段名，前端用的是 camelCase，你可以做一个自动转换：

```typescript
type SnakeToCamel<S extends string> = S extends `${infer Head}_${infer Tail}`
  ? `${Head}${Capitalize<SnakeToCamel<Tail>>}`
  : S;

type Result = SnakeToCamel<"user_avatar_url">;
// "userAvatarUrl"
```

这里用到了递归条件类型（上篇讲过的 `infer`），每次匹配到一个下划线，就把后面的部分首字母大写再递归处理。字符串模式匹配 + 递归，这两个工具组合起来能处理大部分字符串变换。

## 字符串模式匹配与 infer

模板字面量类型最强大的能力是**模式匹配**。你可以在条件类型中用模板字面量来"拆解"一个字符串类型：

```typescript
// 提取冒号后面的部分
type AfterColon<S extends string> = S extends `${string}:${infer Rest}`
  ? Rest
  : never;

type A = AfterColon<"user:login">; // "login"
type B = AfterColon<"no-colon">; // never
```

`${string}:${infer Rest}` 这个模式的意思是：如果 S 能匹配到"任意字符串 + 冒号 + 某个值"，就把那个值捕获到 Rest 里。

把这个思路用到路由参数上。一个路由 `/users/:id` 的参数格式是 `:参数名`，我们可以这样提取：

```typescript
// 提取单个 :param
type ExtractParam<S extends string> =
  S extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParam<Rest>
    : S extends `${string}:${infer Param}`
      ? Param
      : never;

type P1 = ExtractParam<"/users/:id">;
// "id"

type P2 = ExtractParam<"/users/:userId/orders/:orderId">;
// "userId" | "orderId"
```

逻辑是递归地查找 `/` 分隔的每一段，如果某一段以 `:` 开头，就提取参数名。这已经能覆盖大部分路由场景了。

## 处理可选参数

实际的路由里还有可选参数，比如 `/orders/:orderId/items/:itemId?`。这里的 `:itemId?` 表示 itemId 是可选的。提取时需要区分必需参数和可选参数：

```typescript
type ExtractParam<S extends string> =
  S extends `${string}:${infer Param}/${infer Rest}`
    ? Param extends `${infer Name}?`
      ? Name | ExtractParam<Rest>
      : Param | ExtractParam<Rest>
    : S extends `${string}:${infer Param}`
      ? Param extends `${infer Name}?`
        ? Name
        : Param
      : never;
```

差别在于：如果参数名以 `?` 结尾，就把 `?` 去掉后再收集。但这里有个问题 -- 我们丢失了"这个参数是可选的"这个信息。要做到参数可选性也能正确反映到类型上，需要一个更精细的类型。

## 构建 RouteParams 类型

现在我们把前面的思路组合起来，构建一个完整的 `RouteParams` 类型。它接收路由字符串，返回对应的参数对象类型：

```typescript
type ParamToObject<T extends string> = T extends `${infer Name}?`
  ? { [K in Name]?: string }
  : { [K in T]: string };

type RouteParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? ParamToObject<Param> & RouteParams<Rest>
    : T extends `${string}:${infer Param}`
      ? ParamToObject<Param>
      : {};
```

`ParamToObject` 把参数名转成属性，可选参数用 `?` 修饰。`RouteParams` 递归遍历路由的每一段，把参数拼接成对象类型。来验证一下：

```typescript
type R1 = RouteParams<"/users">;
// {}

type R2 = RouteParams<"/users/:id">;
// { id: string }

type R3 = RouteParams<"/users/:userId/orders/:orderId">;
// { userId: string } & { orderId: string }
// 等价于 { userId: string; orderId: string }

type R4 = RouteParams<"/orders/:orderId/items/:itemId?">;
// { orderId: string } & { itemId?: string }
// 等价于 { orderId: string; itemId?: string }
```

没有参数的路由返回空对象 `{}`，单参数返回单属性对象，多参数通过交叉类型（`&`）合并。可选参数也能正确标记。

这就是条件类型 + 模板字面量 + infer 三者配合的典型模式。上篇学的条件类型和 infer 是基础能力，模板字面量把它们的应用范围从"类型本身"扩展到了"字符串内容"。

## 类型安全的路由导航函数

有了 `RouteParams`，我们就能构建一个在调用时就检查参数的路由导航函数：

```typescript
function navigate<Route extends string>(
  route: Route,
  params: RouteParams<Route>
): string {
  let result = route;
  // 运行时替换 :param 占位符
  for (const [key, value] of Object.entries(params)) {
    if (value !== undefined) {
      result = result.replace(`:${key}`, String(value));
    }
  }
  return result;
}

// 编译通过
navigate("/users/:userId/orders/:orderId", {
  userId: "usr_a1x9k2m",
  orderId: "ord_b3y7n4p",
});
// => "/users/usr_a1x9k2m/orders/ord_b3y7n4p"

// 编译报错：缺少 orderId
navigate("/users/:userId/orders/:orderId", {
  userId: "usr_a1x9k2m",
});

// 编译报错：多了个 extra 参数不会报错（交叉类型特性），
// 但传错参数名会报错
navigate("/users/:userId/orders/:orderId", {
  userId: "usr_a1x9k2m",
  order_id: "ord_b3y7n4p", // 类型错误
});
```

`navigate` 的第一个参数是路由字符串，第二个参数的类型由 `RouteParams<Route>` 自动推导。你传了什么路由，TypeScript 就要求你传什么参数。路由改了，参数类型跟着变，不需要手动同步。

运行时部分就是一个简单的字符串替换。类型安全在编译期保证，运行时逻辑保持简单。

## 与条件类型的协作关系

回看整个实现，`RouteParams` 用到了三个上篇讲过的概念：

- **条件类型** `T extends U ? X : Y` -- 判断字符串是否匹配某个模式
- **infer** -- 在模式匹配中捕获参数名
- **递归** -- 处理多段路由字符串

模板字面量在这里的角色是**模式描述**。`${string}:${infer Param}/${infer Rest}` 不是普通的泛型约束，它描述的是一个字符串结构模式。TypeScript 在条件类型中看到模板字面量，会尝试对字符串类型进行模式匹配，匹配上了就把对应位置的值通过 infer 捕获出来。

所以本质上，模板字面量类型让条件类型的"判断"能力从类型层面下沉到了字符串内容层面。你能判断的不只是"这个类型是不是 string"，而是"这个字符串是不是以 `/users/:` 开头"。

## 需要注意的限制

模板字面量类型虽然强大，但有两个实际使用中会碰到的限制。

**第一是联合类型的笛卡尔积爆炸。** 当你在模板字面量中引用的联合类型成员很多时，TypeScript 会生成所有组合。两个 10 成员的联合类型做模板拼接，会产生 100 个成员。如果嵌套了几层，数量会指数级增长。TypeScript 内部对类型实例化数量有上限（默认约 500 万），超出就会报错 `Type instantiation is excessively deep and possibly infinite`。实际项目中，如果一个路由联合类型有几百条，配合多层模板字面量操作，就可能触发这个限制。

**第二是字符串长度限制。** TypeScript 对模板字面量类型中的字符串有长度限制（大约 10 万个字符）。这个限制很少在日常开发中碰到，但如果你试图用模板字面量操作一个非常长的字符串字面量类型（比如把一整个 JSON 文件内容当作字符串类型），就会遇到 `String literal too long` 的错误。

遇到这些限制时，通常的应对方式是：拆分大的联合类型，用递归条件类型替代多层笛卡尔积，或者在类型复杂度过高时适当放宽类型精度。

模板字面量类型的定位很明确：它适合处理结构化的、有固定模式的字符串（路由、事件名、CSS 选择器），不适合处理自由格式的长文本。在这个范围内使用，它的表现非常出色。
