## 前言

前几篇文章里，我们用条件类型、模板字面量类型和泛型构建了不少类型工具。但有一个问题一直悬而未决：这些工具返回的类型，本质还是 `string`、`number` 这些基础类型。你的 API 客户端里，`userId` 和 `orderId` 都是 `string`，传混了编译器完全不管——直到线上出了 bug 你才发现。

这不是 TypeScript 的缺陷，而是它的设计选择。TypeScript 用的是**结构化类型系统**，只看"长什么样"，不看"叫什么名字"。本文要解决的就是这个问题：怎么让编译器区分"语义不同但底层类型相同"的值。

我们会从品牌类型入手，给基础类型加上"身份标识"；然后引入类型谓词和断言函数，在运行时验证类型、在编译时收窄类型；最后用可辨识联合把所有知识串起来，构建一个类型安全的 ID 系统。

## string 是 string，但这不是我们想要的

先看一个真实的坑。假设你在给团队封装 API 客户端，后端有用户和订单两种资源：

```typescript
// 两个"都是" string 的类型
type UserId = string;
type OrderId = string;

// 查询用户
async function getUser(id: UserId) {
  /* ... */
}

// 查询订单
async function getOrder(id: OrderId) {
  /* ... */
}

const orderId: OrderId = "ord_8x9k2m";

// 传混了——编译器不报错，但逻辑上完全错误
getUser(orderId); // 编译通过，但 orderId 不是用户 ID
```

在 Java 或 C# 里，这种情况不会发生，因为它们用的是**名义类型系统**（Nominal Typing）——类型名不同就不兼容。但 TypeScript 用的是**结构化类型系统**（Structural Typing），只看结构不看名字。`type UserId = string` 和 `type OrderId = string` 在编译器眼里完全一样。

怎么破？用**品牌类型**（Branded Types）给 string 加上一个编译期的"身份标识"。

## 品牌类型：给 string 一个身份标识

品牌类型的核心思路很简单：通过交叉类型，在基础类型上叠加一个独一无二的"品牌"属性，让编译器区分它们。

```typescript
// Brand 接受一个原始类型 T 和一个品牌名 B
// 通过交叉类型 T & { __brand: B } 让两个"长相相同"的类型变得不同
type Brand<T, B extends string> = T & { readonly __brand: B };

// 现在 UserId 和 OrderId 虽然底层都是 string，
// 但它们的品牌标识不同，编译器会把它们视为不同类型
type UserId = Brand<string, "UserId">;
type OrderId = Brand<string, "OrderId">;
```

光有类型定义还不够。`__brand` 属性在运行时并不存在（它是编译期的幻象），所以你需要一个**构造函数**来创建品牌类型的值：

```typescript
// 构造函数：验证格式 + 类型断言
// 这是创建品牌类型值的唯一入口，所有值都经过校验
function createUserId(id: string): UserId {
  if (!id.startsWith("usr_")) {
    throw new Error(`Invalid UserId format: ${id}`);
  }
  // 类型断言：告诉编译器"这个 string 是 UserId"
  // 因为前面已经做了格式校验，所以断言是安全的
  return id as UserId;
}

function createOrderId(id: string): OrderId {
  if (!id.startsWith("ord_")) {
    throw new Error(`Invalid OrderId format: ${id}`);
  }
  return id as OrderId;
}
```

现在试试看：

```typescript
const userId = createUserId("usr_a1x9k2m");
const orderId = createOrderId("ord_8x9k2m");

// 正确使用
getUser(userId); // 编译通过
getOrder(orderId); // 编译通过

// 传混了——编译器直接报错
getUser(orderId);
// ❌ Argument of type 'OrderId' is not assignable to parameter of type 'UserId'
```

编译器终于帮我们抓住了这个错误。

### 品牌类型工厂

如果你有十几个不同的 ID 类型，一个一个写构造函数太烦了。可以抽象出一个工厂：

```typescript
// 通用的 ID 品牌类型：底层是 string，品牌名为 B
type EntityId<B extends string> = Brand<string, B>;

// 通用构造函数：接受前缀验证规则
function createEntityId<B extends string>(
  prefix: string,
  value: string
): EntityId<B> {
  if (!value.startsWith(prefix)) {
    throw new Error(`ID must start with "${prefix}", got: ${value}`);
  }
  return value as EntityId<B>;
}

// 用法：每种 ID 只需定义类型和对应的构造函数
type UserId = EntityId<"UserId">;
type OrderId = EntityId<"OrderId">;
type ProductId = EntityId<"ProductId">;

const asUserId = (v: string) => createEntityId<"UserId">("usr_", v);
const asOrderId = (v: string) => createEntityId<"OrderId">("ord_", v);
const asProductId = (v: string) => createEntityId<"ProductId">("prd_", v);
```

## 运行时验证：创建时校验，编译时检查

品牌类型只在编译期生效，运行时它就是普通的 `string`。这意味着从外部接收的数据（API 响应、用户输入）不会自动变成品牌类型。你需要**运行时验证**来确保数据符合类型要求。

还记得上一篇提到的类型谓词 `value is T` 吗？它就是用来做这件事的。

### 类型谓词：`arg is Type`

`value is Type` 语法让 TypeScript 知道：当这个函数返回 `true` 时，参数的类型是 `Type`。这叫做**类型谓词**（Type Predicate），也叫自定义类型守卫。

```typescript
// 普通的校验函数返回 boolean，TypeScript 不会自动收窄类型
function checkUserId(value: string): boolean {
  return value.startsWith("usr_");
}

const id: string = "usr_123";
if (checkUserId(id)) {
  // id 的类型还是 string，TypeScript 不知道它是 UserId
}

// 类型谓词版本：返回值类型是 value is UserId
// 当函数返回 true 时，TypeScript 会把 value 收窄为 UserId
function isUserId(value: string): value is UserId {
  return value.startsWith("usr_");
}

if (isUserId(id)) {
  // id 的类型在这里被收窄为 UserId
  getUser(id); // 编译通过
}
```

这个区别看似微小，但影响巨大。类型谓词让你把运行时的检查结果"传递"给编译器，让编译器在后续代码中给出更精确的类型提示。

### 断言函数：`asserts`

类型谓词返回 `true/false`，调用方需要自己写 `if` 判断。有时候你更想要一种"如果条件不满足就直接抛错"的行为——这就是**断言函数**（Assertion Functions）。

```typescript
// asserts 语法：如果函数正常返回，TypeScript 就认为断言成立
// 如果验证失败，函数必须抛出错误（否则类型系统就不可信了）
function assertUserId(value: string): asserts value is UserId {
  if (!value.startsWith("usr_")) {
    throw new Error(`Expected UserId, got: ${value}`);
  }
  // 函数正常返回时，TypeScript 会将 value 收窄为 UserId
}

// 使用
function processUser(id: string) {
  assertUserId(id);
  // assertUserId 之后，id 的类型已经被收窄为 UserId
  // 不需要 if 判断，不满足条件时函数已经抛出了
  getUser(id);
}
```

类型谓词和断言函数是同一枚硬币的两面：前者让你灵活处理"可能不是这个类型"的情况，后者让你在"必须是这个类型"的场景下更简洁。

### 为品牌类型选择验证策略

什么时候用类型谓词，什么时候用断言函数？取决于数据来源：

- **数据来自外部**（API 响应、localStorage、用户输入）：用类型谓词，因为你需要处理无效数据
- **程序内部传递**：用断言函数，如果数据不对说明是 bug，应该尽早暴露

```typescript
// 场景 1：从 localStorage 恢复会话
// 数据可能被篡改，需要优雅处理
function loadSession(): { userId?: UserId; token?: string } {
  const raw = localStorage.getItem("session");
  if (!raw) return {};

  const data = JSON.parse(raw) as { userId: string; token: string };

  // 用类型谓词：localStorage 中的数据可能无效
  if (isUserId(data.userId)) {
    return { userId: data.userId, token: data.token };
  }
  return {}; // 无效数据，返回空对象
}

// 场景 2：内部函数之间传递 ID
// 如果传了错误的类型，说明有 bug，应该立刻报错
function deleteUser(userId: UserId) {
  assertUserId(userId as string); // 防御性检查
  // userId 在这里被确认为有效的 UserId
}
```

## 可辨识联合：用 type 字段区分类型

品牌类型解决了"两个 string 怎么区分"的问题。但还有一种更常见的场景：你有多种不同的数据结构，需要根据某个字段来区分它们。

比如 API 响应可能是成功或失败，你要根据 `status` 字段走不同逻辑。**可辨识联合**（Discriminated Unions）就是干这个的：

```typescript
// 每个成员都有一个共同的字面量类型属性（"判别属性"）
// 这里 status 就是判别属性，TypeScript 通过它区分联合成员
type ApiResponse<T> =
  | { status: 200; data: T }
  | { status: 400; error: string }
  | { status: 404; error: string }
  | { status: 500; error: string };

// 类型守卫：根据 status 的值收窄类型
function isSuccess<T>(res: ApiResponse<T>): res is { status: 200; data: T } {
  return res.status === 200;
}

function isClientError(
  res: ApiResponse<any>
): res is { status: 400 | 404; error: string } {
  return res.status === 400 || res.status === 404;
}

// 使用：TypeScript 在每个分支里都能精确知道 res 的类型
async function fetchUser(id: UserId): Promise<User> {
  const res = await api.get<ApiResponse<User>>(`/users/${id}`);

  if (isSuccess(res)) {
    return res.data; // res.data 类型正确
  }

  if (isClientError(res)) {
    throw new UserFacingError(res.error); // res.error 类型正确
  }

  throw new InternalError(res.error); // 500 错误
}
```

可辨识联合和类型谓词是天生的搭档。判别属性让 TypeScript 能自动收窄类型（如果你用 `if (res.status === 200)` TypeScript 也能收窄），而类型谓词让收窄逻辑更语义化、更可复用。

## 把它们串起来：类型安全的 ID 系统

现在把品牌类型、类型谓词、断言函数和可辨识联合组合起来，为我们的 API 客户端构建一个完整的类型安全 ID 系统。

先定义各种品牌类型：

```typescript
// 通用品牌类型工具
type Brand<T, B extends string> = T & { readonly __brand: B };
type Id<B extends string> = Brand<string, B>;

// 各种 ID 类型
type UserId = Id<"UserId">;
type OrderId = Id<"OrderId">;
type ProductId = Id<"ProductId">;

// 货币类型：底层是 number，但 USD 和 EUR 不能混用
type USD = Brand<number, "USD">;
type EUR = Brand<number, "EUR">;
```

然后为每种 ID 提供构造函数、类型谓词和断言函数：

```typescript
// 构造函数：创建值时做格式校验
function createUserId(id: string): UserId {
  if (!id.startsWith("usr_")) {
    throw new Error(`Invalid UserId: ${id}`);
  }
  return id as UserId;
}

// 类型谓词：检查一个值是否是有效的 UserId
// 用于处理外部数据（API 响应、localStorage 等）
function isUserId(value: unknown): value is UserId {
  return typeof value === "string" && value.startsWith("usr_");
}

// 断言函数：断言值必须是 UserId
// 用于内部函数参数验证
function assertUserId(value: unknown): asserts value is UserId {
  if (!isUserId(value)) {
    throw new TypeError(`Expected UserId, got ${typeof value}: ${value}`);
  }
}
```

有了这些基础设施，你的 API 客户端就能获得编译期的类型安全保障：

```typescript
// 类型安全的仓储接口
interface Repository<T, ID extends Brand<string, any>> {
  findById(id: ID): Promise<T | null>;
  delete(id: ID): Promise<void>;
}

class UserRepository implements Repository<User, UserId> {
  private store = new Map<UserId, User>();

  async findById(id: UserId): Promise<User | null> {
    return this.store.get(id) ?? null;
  }

  async delete(id: UserId): Promise<void> {
    this.store.delete(id);
  }
}

const userRepo = new UserRepository();
const orderId = createOrderId("ord_8x9k2m");

// 编译错误：OrderId 不能赋值给 UserId
userRepo.findById(orderId);
// ❌ Argument of type 'OrderId' is not assignable to parameter of type 'UserId'
```

品牌类型用在数字上同样威力巨大。比如防止货币混淆：

```typescript
function createUSD(value: number): USD {
  if (value < 0) throw new Error("USD cannot be negative");
  return value as USD;
}

// 只有同种货币才能相加，不同货币混用会编译报错
function addUSD(a: USD, b: USD): USD {
  return createUSD((a as number) + (b as number));
}

function usdToEur(amount: USD, rate: number): EUR {
  return createEUR((amount as number) * rate);
}

const price = createUSD(99.99);
const tax = createUSD(8.99);
const total = addUSD(price, tax); // USD + USD = USD

// 欧元价格不能直接加到美元价格上
const euroPrice = createEUR(85);
addUSD(price, euroPrice);
// ❌ Argument of type 'EUR' is not assignable to parameter of type 'USD'
```

## 几个容易踩的坑

**用 `as` 绕过验证**。品牌类型的类型断言 `as UserId` 随时可以用，但这等于绕过了所有安全检查。始终通过构造函数创建品牌类型的值，而不是直接用 `as`：

```typescript
// 危险：绕过了前缀校验，任何 string 都能变成 UserId
const id = "totally_not_a_user_id" as UserId;

// 安全：通过构造函数，格式不合法会立刻报错
const id = createUserId("usr_a1x9k2m");
```

**序列化丢失品牌**。`JSON.stringify` 只保留运行时的值，品牌标识是编译期信息，序列化后就会丢失。反序列化时需要重新走验证流程：

```typescript
const userId = createUserId("usr_a1x9k2m");

// 序列化后就是一个普通 string
const json = JSON.stringify(userId);
// '"usr_a1x9k2m"'

// 反序列化后需要重新验证，才能恢复品牌类型
const parsed: unknown = JSON.parse(json);
if (isUserId(parsed)) {
  // parsed 在这里又是 UserId 了
}
```

**不要过度品牌化**。品牌类型增加了一些开发成本（构造函数、验证逻辑），只在真正容易混淆的地方使用。比如区分不同的 ID 类型很有价值，但把 `FirstName` 和 `LastName` 也品牌化就过度了——它们通常在同一个对象里，类型系统已经通过字段名区分了它们。

## 什么时候用品牌类型，什么时候用可辨识联合

这两个工具解决的问题有重叠，但适用场景不同：

**品牌类型**适合"底层类型相同但语义不同"的简单值：`UserId` vs `OrderId`、`USD` vs `EUR`。它轻量、直接，不会改变数据结构。

**可辨识联合**适合"完全不同的数据结构需要统一处理"的场景：API 响应的成功/失败、不同类型的通知消息。它把多种类型打包成一个联合类型，通过判别属性来分发。

在实际项目中，它们经常配合使用。比如你的 API 客户端用品牌类型保证 ID 的类型安全，同时用可辨识联合处理不同类型的 API 响应。
