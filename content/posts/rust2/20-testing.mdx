## 为什么需要测试？

在构建分布式任务处理系统时，我们实现了任务分发、并发处理、进度追踪等功能。这些代码的复杂度随着功能增加而提高，手动测试变得不可行。

更重要的是，重构或添加新功能时，如何确保不破坏现有功能？手动测试不仅耗时，而且容易遗漏边界情况。自动化测试解决了这个问题——它是一套可重复运行的验证程序，能在代码变更时快速发现问题。

Rust 内置了强大的测试框架，不需要额外安装任何依赖。本章将学习如何编写和组织测试，确保代码质量。

## 单元测试

单元测试针对代码的最小单元（函数、方法）进行测试。在 Rust 中，单元测试通常放在 `#[cfg(test)]` 模块中，与被测试代码在同一个文件。

```rust
// src/lib.rs

/// 计算两个数的和
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

/// 除法运算，除数为零时 panic
pub fn divide(a: f64, b: f64) -> f64 {
    if b == 0.0 {
        panic!("除数不能为零");
    }
    a / b
}

#[cfg(test)]
mod tests {
    use super::*;

    // 基本测试
    #[test]
    fn test_add() {
        assert_eq!(add(2, 3), 5);
        assert_eq!(add(-1, 1), 0);
    }

    // 测试预期 panic
    #[test]
    #[should_panic(expected = "除数不能为零")]
    fn test_divide_by_zero() {
        divide(10.0, 0.0);
    }

    // 正常的除法测试
    #[test]
    fn test_divide_normal() {
        assert_eq!(divide(10.0, 2.0), 5.0);
        assert_eq!(divide(7.0, 2.0), 3.5);
    }
}
```

运行测试：`cargo test`

测试函数用 `#[test]` 标注，测试失败时会 panic。`#[should_panic]` 标注测试预期会 panic，还可以检查 panic 的消息内容。

### 测试辅助函数

测试中可能有重复的设置逻辑，可以提取为辅助函数。辅助函数不需要 `#[test]` 标注。

```rust
pub struct Task {
    pub id: u32,
    pub name: String,
    pub priority: u32,
}

impl Task {
    pub fn new(id: u32, name: String, priority: u32) -> Self {
        Self { id, name, priority }
    }

    pub fn is_high_priority(&self) -> bool {
        self.priority >= 5
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    // 辅助函数：创建测试用的任务
    fn create_test_task(id: u32, priority: u32) -> Task {
        Task::new(
            id,
            format!("测试任务 {}", id),
            priority
        )
    }

    #[test]
    fn test_high_priority() {
        let task = create_test_task(1, 8);
        assert!(task.is_high_priority());
    }

    #[test]
    fn test_normal_priority() {
        let task = create_test_task(2, 3);
        assert!(!task.is_high_priority());
    }

    #[test]
    fn test_boundary_priority() {
        let task = create_test_task(3, 5);
        assert!(task.is_high_priority());  // 5 算高优先级
    }
}
```

### 使用 Result 的测试

测试函数也可以返回 `Result<T, E>`，这样可以用 `?` 运算符简化错误处理。

```rust
pub fn parse_task_id(input: &str) -> Result<u32, String> {
    input.parse().map_err(|_| "无效的任务 ID".to_string())
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_parse_valid() -> Result<(), String> {
        let id = parse_task_id("123")?;
        assert_eq!(id, 123);
        Ok(())
    }

    #[test]
    fn test_parse_invalid() {
        assert!(parse_task_id("abc").is_err());
        assert_eq!(parse_task_id("abc").unwrap_err(), "无效的任务 ID");
    }
}
```

### 断言宏

Rust 提供了几个断言宏：

```rust
#[cfg(test)]
mod tests {
    // assert_eq!：判断相等
    #[test]
    fn test_equal() {
        assert_eq!(2 + 2, 4);
    }

    // assert_ne!：判断不等
    #[test]
    fn test_not_equal() {
        assert_ne!(2 + 2, 5);
    }

    // assert!：判断条件为 true
    #[test]
    fn test_condition() {
        let result = 10;
        assert!(result > 5);
    }

    // 自定义错误消息
    #[test]
    fn test_with_message() {
        let value = 3;
        assert!(value < 10, "值 {} 应该小于 10", value);
    }
}
```

## 控制测试执行

### 过滤测试

可以只运行名字匹配某个模式的测试：

```bash
# 只运行名字包含 "add" 的测试
cargo test add

# 运行某个模块的所有测试
cargo test tests::test_add
```

### 忽略测试

有些测试运行时间很长，可以暂时忽略：

```rust
#[test]
#[ignore]
fn expensive_test() {
    // 耗时很长的测试
}

// 运行被忽略的测试
// cargo test -- --ignored
```

### 显示输出

默认情况下，测试中的 `println!` 输出不会显示。用 `--nocapture` 可以看到输出：

```bash
cargo test -- --nocapture
```

## 集成测试

集成测试放在 `tests/` 目录下，测试整个库的公开 API。每个 `tests/` 下的文件都是一个独立的 crate，只能使用库的公开接口。

```
my_project/
├── Cargo.toml
├── src/
│   └── lib.rs
└── tests/
    └── integration_test.rs
```

`tests/integration_test.rs`:

```rust
use my_project::{add, divide, Task};

#[test]
fn test_basic_operations() {
    assert_eq!(add(2, 3), 5);
    assert_eq!(divide(10.0, 2.0), 5.0);
}

#[test]
fn test_task_operations() {
    let task = Task::new(1, "集成测试任务".to_string(), 8);
    assert!(task.is_high_priority());
}

#[test]
fn test_combined_operations() {
    let sum = add(2, 3);
    let result = divide(sum as f64, 5.0);
    assert_eq!(result, 1.0);
}
```

运行集成测试：`cargo test`

集成测试的子模块可以共享代码：

```
tests/
├── common/
│   └── mod.rs      # 共享辅助代码
└── integration_test.rs
```

`tests/common/mod.rs`:

```rust
pub fn setup_test_tasks() -> Vec<Task> {
    vec![
        Task::new(1, "任务 A".to_string(), 3),
        Task::new(2, "任务 B".to_string(), 8),
    ]
}
```

`tests/integration_test.rs`:

```rust
mod common;

use my_project::Task;

#[test]
fn test_with_common_setup() {
    let tasks = common::setup_test_tasks();
    assert_eq!(tasks.len(), 2);
    assert!(tasks[1].is_high_priority());
}
```

## 文档测试

Rust 支持在文档注释中编写测试，这既是示例又是测试：

````rust
/// 将两个数相加。
///
/// # Examples
///
/// ```
/// use my_project::add;
///
/// let result = add(2, 3);
/// assert_eq!(result, 5);
/// ```
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

/// 创建一个新任务。
///
/// # Examples
///
/// ```
/// use my_project::Task;
///
/// let task = Task::new(1, "测试任务".to_string(), 5);
/// assert!(task.is_high_priority());
/// ```
pub struct Task {
    pub id: u32,
    pub name: String,
    pub priority: u32,
}
````

运行文档测试：`cargo test --doc`

文档测试的好处是确保示例代码与实际 API 同步，防止文档过时。

## 实战：为任务系统编写测试

现在让我们为之前构建的任务处理系统编写完整的测试。

```rust
// src/lib.rs

#[derive(Debug, Clone, PartialEq)]
pub enum TaskStatus {
    Pending,
    InProgress,
    Completed,
    Failed,
}

#[derive(Debug, Clone)]
pub struct Task {
    pub id: u32,
    pub name: String,
    pub priority: u32,
    pub status: TaskStatus,
}

impl Task {
    pub fn new(id: u32, name: String, priority: u32) -> Self {
        Self {
            id,
            name,
            priority,
            status: TaskStatus::Pending,
        }
    }

    pub fn is_high_priority(&self) -> bool {
        self.priority >= 5
    }

    pub fn start(&mut self) {
        if self.status == TaskStatus::Pending {
            self.status = TaskStatus::InProgress;
        }
    }

    pub fn complete(&mut self) {
        if self.status == TaskStatus::InProgress {
            self.status = TaskStatus::Completed;
        }
    }

    pub fn fail(&mut self) {
        if self.status == TaskStatus::InProgress {
            self.status = TaskStatus::Failed;
        }
    }
}

pub struct TaskQueue {
    tasks: Vec<Task>,
}

impl TaskQueue {
    pub fn new() -> Self {
        Self { tasks: Vec::new() }
    }

    pub fn add(&mut self, task: Task) {
        self.tasks.push(task);
    }

    pub fn get_high_priority_tasks(&self) -> Vec<&Task> {
        self.tasks.iter()
            .filter(|task| task.is_high_priority())
            .collect()
    }

    pub fn get_pending_count(&self) -> usize {
        self.tasks.iter()
            .filter(|task| task.status == TaskStatus::Pending)
            .count()
    }

    pub fn find_by_id(&self, id: u32) -> Option<&Task> {
        self.tasks.iter()
            .find(|task| task.id == id)
    }

    pub fn total_priority(&self) -> u32 {
        self.tasks.iter()
            .map(|task| task.priority)
            .sum()
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    // 辅助函数：创建测试任务
    fn create_task(id: u32, priority: u32) -> Task {
        Task::new(id, format!("任务 {}", id), priority)
    }

    mod task_tests {
        use super::*;

        #[test]
        fn test_high_priority() {
            let task = create_task(1, 8);
            assert!(task.is_high_priority());
        }

        #[test]
        fn test_normal_priority() {
            let task = create_task(1, 3);
            assert!(!task.is_high_priority());
        }

        #[test]
        fn test_boundary_case() {
            let task = create_task(1, 5);
            assert!(task.is_high_priority());
        }

        #[test]
        fn test_status_transitions() {
            let mut task = create_task(1, 5);

            // 初始状态
            assert_eq!(task.status, TaskStatus::Pending);

            // 开始任务
            task.start();
            assert_eq!(task.status, TaskStatus::InProgress);

            // 完成任务
            task.complete();
            assert_eq!(task.status, TaskStatus::Completed);
        }

        #[test]
        fn test_fail_transition() {
            let mut task = create_task(1, 5);
            task.start();
            task.fail();
            assert_eq!(task.status, TaskStatus::Failed);
        }

        #[test]
        #[should_panic(expected = "assertion failed")]
        fn test_invalid_transition() {
            let mut task = create_task(1, 5);
            // 尝试从未开始状态完成任务（应该失败）
            task.complete();
        }
    }

    mod task_queue_tests {
        use super::*;

        #[test]
        fn test_add_tasks() {
            let mut queue = TaskQueue::new();
            queue.add(create_task(1, 5));
            queue.add(create_task(2, 3));

            assert_eq!(queue.get_pending_count(), 2);
        }

        #[test]
        fn test_high_priority_filter() {
            let mut queue = TaskQueue::new();
            queue.add(create_task(1, 8));
            queue.add(create_task(2, 3));
            queue.add(create_task(3, 6));

            let high_priority = queue.get_high_priority_tasks();
            assert_eq!(high_priority.len(), 2);
        }

        #[test]
        fn test_find_by_id() {
            let mut queue = TaskQueue::new();
            queue.add(create_task(1, 5));
            queue.add(create_task(2, 3));

            assert!(queue.find_by_id(1).is_some());
            assert!(queue.find_by_id(2).is_some());
            assert!(queue.find_by_id(99).is_none());
        }

        #[test]
        fn test_total_priority() {
            let mut queue = TaskQueue::new();
            queue.add(create_task(1, 5));
            queue.add(create_task(2, 3));
            queue.add(create_task(3, 7));

            assert_eq!(queue.total_priority(), 15);
        }

        #[test]
        fn test_empty_queue() {
            let queue = TaskQueue::new();
            assert_eq!(queue.get_pending_count(), 0);
            assert_eq!(queue.get_high_priority_tasks().len(), 0);
            assert_eq!(queue.total_priority(), 0);
        }
    }
}
```

## 测试组织

随着项目增长，测试也需要良好组织：

1. **单元测试**：放在 `src/` 目录下，用 `#[cfg(test)]` 模块
2. **集成测试**：放在 `tests/` 目录，测试公开 API
3. **文档测试**：放在文档注释中，确保示例正确

```
project/
├── Cargo.toml
├── src/
│   ├── lib.rs
│   ├── task.rs
│   └── queue.rs
└── tests/
    ├── common/
    │   └── mod.rs
    ├── task_tests.rs
    └── integration_tests.rs
```

## 测试最佳实践

1. **测试命名清晰**：`test_` 前缀 + 描述行为
2. **测试公开 API**：集成测试只测试公开接口
3. **测试边界条件**：空输入、零、最大值等
4. **使用 `Result` 测试**：避免 `unwrap` 在测试中 panic
5. **保持测试独立**：每个测试应该独立运行
6. **运行 `cargo test` 作为 CI 的一部分**
7. **测试应该快速**：慢测试用 `#[ignore]` 标注

## 权衡与考虑

测试的权衡：

**优势：**

- 提高代码质量和可靠性
- 支持重构和快速迭代
- 作为代码的使用文档
- 早期发现 bug

**挑战：**

- 编写测试需要时间
- 维护测试有成本
- 测试不能保证没有 bug
- 可能给代码变更带来阻力

**建议：**

- 为核心逻辑编写测试
- 为公开 API 编写集成测试
- 保持测试简洁易读
- 定期审查和更新测试
- 但不要追求 100% 覆盖率
