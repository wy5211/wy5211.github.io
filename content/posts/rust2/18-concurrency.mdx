## 为什么并发编程这么难？

在构建分布式任务处理系统时，我们需要同时处理多个任务：多个 Worker 并行执行、主线程分发任务、监听进度更新。这些操作如果串行执行，系统效率会极其低下。

但并发编程是出了名的困难：数据竞争、死锁、竞态条件无处不在。大多数语言依赖开发者小心处理这些问题，结果就是各种难以复现的 bug。

Rust 选择了一条不同的路：**通过所有权系统在编译期防止数据竞争**。这被称为"无畏并发"——如果代码能编译通过，就不会有数据竞争。Rust 通过 `Send` 和 `Sync` 两个 marker trait 实现了这一点，编译器会阻止不安全的并发操作。

## 创建线程

Rust 的 `std::thread` 模块提供了创建线程的 API。新线程会与主线程并发执行。

```rust
use std::thread;
use std::time::Duration;

fn main() {
    // 创建新线程，打印数字 1-5
    let handle = thread::spawn(|| {
        for i in 1..=5 {
            println!("Worker 线程: {}", i);
            thread::sleep(Duration::from_millis(10));
        }
    });

    // 主线程同时打印字母 A-C
    for c in ['A', 'B', 'C'] {
        println!("主线程: {}", c);
        thread::sleep(Duration::from_millis(10));
    }

    // 等待 Worker 线程完成
    handle.join().unwrap();
    println!("所有线程完成");
}
```

`thread::spawn` 返回一个 `JoinHandle`，调用其 `join()` 方法会等待线程结束。如果不调用 `join()`，主线程可能会在线程完成前就退出，导致线程被强制终止。

### move 闭包与所有权

线程默认使用闭包捕获环境变量，但需要 `move` 关键字来转移所有权。这是因为线程可能比创建它的作用域活得更久。

```rust
use std::thread;

fn main() {
    let task_data = vec![1, 2, 3, 4, 5];

    // 错误：编译器不允许在线程中使用外部变量的引用
    // let handle = thread::spawn(|| {
    //     println!("任务数据: {:?}", task_data);
    // });

    // 正确：使用 move 转移所有权
    let handle = thread::spawn(move || {
        println!("Worker 处理数据: {:?}", task_data);
        // task_data 的所有权已转移到线程
    });

    handle.join().unwrap();
    // task_data 不再可用，所有权已转移
}
```

`move` 闭包强制获取捕获变量的所有权，确保数据在线程使用期间一直有效。这是 Rust 防止数据竞争的第一道防线。

## 消息传递

Rust 的并发哲学是："不要通过共享内存来通信，而要通过通信来共享内存"。channel（通道）是实现这一理念的工具——一个线程发送数据，另一个线程接收数据。

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    // 创建 channel，返回发送者（tx）和接收者（rx）
    let (tx, rx) = mpsc::channel();

    // 创建新线程，发送任务
    thread::spawn(move || {
        let task = String::from("处理数据文件");
        tx.send(task).unwrap();
        // task 的所有权已转移到 channel，不再可用
    });

    // 主线程接收任务
    let received = rx.recv().unwrap();
    println!("收到任务: {}", received);
}
```

`send()` 方法会转移数据的所有权，确保发送后不再访问该数据。`recv()` 会阻塞直到收到消息，或当发送者关闭时返回错误。

### 发送多个值

channel 的接收者可以作为迭代器使用，持续接收消息直到发送者关闭。

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let tasks = vec![
            String::from("导入数据"),
            String::from("验证数据"),
            String::from("生成报表"),
        ];

        for task in tasks {
            tx.send(task).unwrap();
            thread::sleep(Duration::from_millis(100));
        }
        // tx 离开作用域，channel 关闭
    });

    // rx 作为迭代器，自动接收所有消息
    for received in rx {
        println!("处理任务: {}", received);
    }

    println!("所有任务完成");
}
```

### 多发送者

`mpsc` 是 "multiple producer, single consumer"（多生产者，单消费者）的缩写。我们可以通过 `clone()` 创建多个发送者。

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();

    // 创建多个发送者
    let tx1 = tx.clone();
    let tx2 = tx.clone();

    // Worker 1
    thread::spawn(move || {
        let tasks = vec!["Worker 1: 任务 A", "Worker 1: 任务 B"];
        for task in tasks {
            tx1.send(task).unwrap();
            thread::sleep(Duration::from_millis(100));
        }
    });

    // Worker 2
    thread::spawn(move || {
        let tasks = vec!["Worker 2: 任务 X", "Worker 2: 任务 Y"];
        for task in tasks {
            tx2.send(task).unwrap();
            thread::sleep(Duration::from_millis(150));
        }
    });

    // 主线程接收所有消息
    for received in rx {
        println!("收到: {}", received);
    }
}
```

### 非阻塞接收

`recv()` 会阻塞等待消息，而 `try_recv()` 会立即返回，适合需要轮询的场景。

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        thread::sleep(Duration::from_secs(1));
        tx.send("延迟消息".to_string()).unwrap();
    });

    // 非阻塞轮询
    loop {
        match rx.try_recv() {
            Ok(msg) => {
                println!("收到: {}", msg);
                break;
            }
            Err(mpsc::TryRecvError::Empty) => {
                println!("等待消息...");
                thread::sleep(Duration::from_millis(200));
            }
            Err(mpsc::TryRecvError::Disconnected) => {
                println!("发送者已断开");
                break;
            }
        }
    }
}
```

## 共享状态并发

虽然消息传递是首选方案，但有时我们需要共享可变状态。Rust 通过 `Mutex<T>`（互斥锁）和 `Arc<T>`（原子引用计数）来实现安全的多线程共享状态。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    // Arc<Mutex<T>> 是多线程共享可变数据的标准组合
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    // 创建多个线程，每个都增加计数器
    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            // 获取锁，修改数据
            let mut num = counter.lock().unwrap();
            *num += 1;
            // 离开作用域，锁自动释放
        });
        handles.push(handle);
    }

    // 等待所有线程完成
    for handle in handles {
        handle.join().unwrap();
    }

    println!("最终计数: {}", *counter.lock().unwrap());
}
```

`Mutex<T>` 确保同一时刻只有一个线程可以访问数据。`lock()` 方法返回一个 `MutexGuard`，它实现了 `Drop` trait，离开作用域时自动释放锁。

### Mutex 的工作原理

Mutex 通过内部的锁机制保证互斥访问。当一个线程获取锁后，其他线程必须等待。

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

fn main() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));

    let handle1 = {
        let data = Arc::clone(&data);
        thread::spawn(move || {
            let mut data = data.lock().unwrap();
            println!("线程 1 获取锁");
            data.push(4);
            thread::sleep(Duration::from_secs(1));  // 持有锁 1 秒
            println!("线程 1 释放锁");
        })
    };

    let handle2 = {
        let data = Arc::clone(&data);
        thread::spawn(move || {
            thread::sleep(Duration::from_millis(100));  // 稍等片刻
            println!("线程 2 尝试获取锁...");
            let mut data = data.lock().unwrap();  // 阻塞直到线程 1 释放锁
            println!("线程 2 获取锁");
            data.push(5);
        })
    };

    handle1.join().unwrap();
    handle2.join().unwrap();

    println!("最终数据: {:?}", *data.lock().unwrap());
}
```

### 避免死锁

死锁是并发编程中的常见问题，通常由不当的锁顺序引起。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let lock1 = Arc::new(Mutex::new(0));
    let lock2 = Arc::new(Mutex::new(0));

    // 危险：可能死锁
    // 线程 1 持有 lock1 等待 lock2
    // 线程 2 持有 lock2 等待 lock1
    let handle1 = {
        let lock1 = Arc::clone(&lock1);
        let lock2 = Arc::clone(&lock2);
        thread::spawn(move || {
            let _l1 = lock1.lock().unwrap();
            thread::sleep(std::time::Duration::from_millis(50));
            let _l2 = lock2.lock().unwrap();  // 可能死锁
        })
    };

    let handle2 = {
        let lock1 = Arc::clone(&lock1);
        let lock2 = Arc::clone(&lock2);
        thread::spawn(move || {
            let _l2 = lock2.lock().unwrap();
            thread::sleep(std::time::Duration::from_millis(50));
            let _l1 = lock1.lock().unwrap();  // 可能死锁
        })
    };

    // 解决方案：总是以相同的顺序获取锁
    handle1.join().ok();
    handle2.join().ok();
}
```

避免死锁的最佳实践：

- 总是以相同的顺序获取多个锁
- 尽量减少持锁时间
- 使用 `try_lock()` 而非 `lock()` 来避免阻塞

## 实战：多线程任务处理系统

现在让我们用并发原语构建一个实际的任务处理系统。系统包含一个主线程分发任务，多个 Worker 并行处理任务。

```rust
use std::sync::mpsc;
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

#[derive(Debug)]
struct Task {
    id: u32,
    name: String,
    duration_ms: u64,
}

#[derive(Debug)]
struct TaskResult {
    task_id: u32,
    worker_id: u32,
    status: String,
}

fn main() {
    let (task_sender, task_receiver) = mpsc::channel::<Task>();
    let (result_sender, result_receiver) = mpsc::channel::<TaskResult>();

    // 共享的任务接收器，用 Arc + Mutex 包裹
    let task_receiver = Arc::new(Mutex::new(task_receiver));

    let mut worker_handles = vec![];

    // 创建 3 个 Worker
    for worker_id in 1..=3 {
        let task_receiver = Arc::clone(&task_receiver);
        let result_sender = result_sender.clone();

        let handle = thread::spawn(move || {
            loop {
                // 获取任务
                let task = {
                    let receiver = task_receiver.lock().unwrap();
                    receiver.recv()
                };

                match task {
                    Ok(task) => {
                        println!("Worker {} 开始处理任务: {}",
                            worker_id, task.name);

                        // 模拟任务处理
                        thread::sleep(Duration::from_millis(task.duration_ms));

                        // 发送结果
                        result_sender.send(TaskResult {
                            task_id: task.id,
                            worker_id,
                            status: format!("完成 (耗时 {}ms)", task.duration_ms),
                        }).unwrap();
                    }
                    Err(_) => {
                        // channel 关闭，没有更多任务
                        println!("Worker {} 退出", worker_id);
                        break;
                    }
                }
            }
        });

        worker_handles.push(handle);
    }

    // 主线程发送任务
    let tasks = vec![
        Task { id: 1, name: String::from("数据导入"), duration_ms: 100 },
        Task { id: 2, name: String::from("数据验证"), duration_ms: 150 },
        Task { id: 3, name: String::from("生成报表"), duration_ms: 200 },
        Task { id: 4, name: String::from("发送邮件"), duration_ms: 80 },
        Task { id: 5, name: String::from("更新缓存"), duration_ms: 120 },
    ];

    for task in tasks {
        task_sender.send(task).unwrap();
        println!("主线程分发任务: {}", task.id);
        thread::sleep(Duration::from_millis(50));  // 模拟任务到达间隔
    }

    // 关闭任务发送端，Workers 会在处理完所有任务后退出
    drop(task_sender);

    // 收集结果
    let mut results = vec![];
    for result in result_receiver {
        println!("收到结果: 任务 {} 由 Worker {} 处理 - {}",
            result.task_id, result.worker_id, result.status);
        results.push(result);
    }

    // 等待所有 Worker 完成
    for handle in worker_handles {
        handle.join().unwrap();
    }

    println!("\n所有任务完成，共处理 {} 个任务", results.len());
}
```

这个例子展示了并发编程的核心模式：

- channel 用于线程间通信
- Arc + Mutex 用于共享状态
- 多个 Worker 并行处理任务
- 主线程协调和收集结果

## Send 和 Sync trait

Rust 通过两个 marker trait 实现并发安全的编译时检查：

- `Send`：类型可以在线程间转移所有权
- `Sync`：类型的引用可以在线程间共享

几乎所有类型都自动实现 Send，除了包含 `Rc` 的类型。几乎所有类型都自动实现 Sync，除了包含 `Rc`、`RefCell` 的类型。

```rust
use std::rc::Rc;
use std::sync::Arc;
use std::thread;

fn main() {
    // Rc 不是 Send，不能跨线程
    let data = Rc::new(5);
    // thread::spawn(move || {
    //     println!("{}", data);  // 编译错误
    // });

    // Arc 是 Send，可以跨线程
    let data = Arc::new(5);
    let data2 = Arc::clone(&data);
    thread::spawn(move || {
        println!("{}", data2);  // 正确
    }).join().unwrap();
}
```

## 权衡与考虑

并发编程的权衡：

**优势：**

- 充分利用多核 CPU
- 提高系统吞吐量
- 响应式用户体验

**挑战：**

- 复杂度增加
- 调试困难
- 死锁、竞态条件风险

**Rust 的优势：**

- 编译期防止数据竞争
- 明确的所有权语义
- 强大的类型系统保证

**选择建议：**

- 优先使用 channel 而非共享状态
- 使用 Arc + Mutex 时注意锁的粒度
- 考虑使用线程池避免频繁创建销毁线程
- 对于异步 I/O，考虑使用 async/await
