## 问题引入

假设你正在构建一个分布式任务处理系统。系统需要接收来自客户端的任务请求，将它们存储在内存中，然后分发给多个 worker 节点处理。在这个过程中，你会遇到几个数据存储问题：

1. 任务到达的顺序很重要，需要按顺序存储和访问
2. 需要通过任务 ID 快速查找特定任务
3. Worker 节点集合需要自动去重，避免重复注册
4. 需要高效地统计每个 worker 处理的任务数量

Rust 提供了三个核心集合类型来解决这些问题：`Vec`、`HashMap` 和 `HashSet`。让我们通过任务处理系统的实际场景来学习它们。

## Vec：动态数组

### 为什么需要 Vec

在任务处理系统中，最先到达的任务应该先被处理（FIFO，先进先出）。我们需要一个能动态增长、按索引访问的数据结构。数组长度固定，不适合存储数量未知的任务。`Vec<T>`（vector）就是为此设计的动态数组。

### 创建和操作 Vec

```rust
// 任务队列结构体
struct TaskQueue {
    tasks: Vec<Task>,
}

struct Task {
    id: u64,
    name: String,
    payload: Vec<u8>,
}

impl TaskQueue {
    // 创建空任务队列
    fn new() -> Self {
        Self {
            tasks: Vec::new(),
        }
    }

    // 添加任务到队列末尾
    fn push(&mut self, task: Task) {
        self.tasks.push(task);
    }

    // 从队列头部取出任务（FIFO）
    fn pop_front(&mut self) -> Option<Task> {
        if self.tasks.is_empty() {
            None
        } else {
            // remove(0) 会移动所有元素，效率较低
            // 后续文章会介绍更高效的 VecDeque
            Some(self.tasks.remove(0))
        }
    }

    // 查看指定位置的任务
    fn get(&self, index: usize) -> Option<&Task> {
        self.tasks.get(index)
    }
}
```

### 安全访问元素

访问 Vec 元素时，索引可能越界。Rust 提供了两种方式：

```rust
impl TaskQueue {
    fn access_examples(&self) {
        // 方式一：直接索引，越界会 panic
        // let task = &self.tasks[0];  // 如果为空，程序崩溃

        // 方式二：get 方法，返回 Option，安全
        match self.tasks.get(0) {
            Some(task) => println!("首个任务: {}", task.id),
            None => println!("队列为空"),
        }

        // 方式三：使用 if let 简化模式匹配
        if let Some(task) = self.tasks.get(0) {
            println!("任务 ID: {}", task.id);
        }
    }
}
```

**为什么推荐使用 `get`**：直接索引（`v[0]`）在越界时会 panic，导致程序崩溃。`get` 方法返回 `Option<&T>`，强制你处理"索引越界"这种情况。

### 遍历 Vec

```rust
impl TaskQueue {
    // 不可变遍历：只读取任务
    fn list_all(&self) {
        for (i, task) in self.tasks.iter().enumerate() {
            println!("任务 {}: {}", i, task.name);
        }
    }

    // 可变遍历：修改任务状态
    fn mark_all_in_progress(&mut self) {
        for task in &mut self.tasks {
            // 假设 Task 有 status 字段
            // task.status = TaskStatus::InProgress;
        }
    }
}
```

### Vec 的所有权规则

```rust
fn ownership_example() {
    let mut queue = TaskQueue::new();

    // 创建任务并转移所有权到 Vec 中
    let task = Task {
        id: 1,
        name: String::from("处理数据"),
        payload: vec![1, 2, 3],
    };
    queue.push(task);  // task 的所有权被移动到 queue.tasks
    // task.name  // 错误：task 已失效

    // 获取引用不会转移所有权
    if let Some(task_ref) = queue.get(0) {
        println!("任务: {}", task_ref.name);
    }  // task_ref 离开作用域，queue 仍然拥有 task

    // 消耗性遍历会转移所有权
    for task in queue.tasks {
        println!("处理任务: {}", task.name);
        // task 在这里被 drop
    }  // queue.tasks 现在为空
}
```

## HashMap：键值对存储

### 为什么需要 HashMap

在任务系统中，每个任务都有唯一 ID。客户端可能随时查询某个任务的状态，如果用 Vec 遍历查找，时间复杂度是 O(n)。`HashMap<K, V>` 使用哈希表实现，查找的平均时间复杂度是 O(1)，适合通过键快速查找值。

### 创建和使用 HashMap

```rust
use std::collections::HashMap;

struct TaskStore {
    // 通过任务 ID 快速查找任务
    tasks: HashMap<u64, Task>,
    // 统计每个 worker 处理的任务数
    worker_task_counts: HashMap<String, usize>,
}

impl TaskStore {
    fn new() -> Self {
        Self {
            tasks: HashMap::new(),
            worker_task_counts: HashMap::new(),
        }
    }

    // 添加任务，如果 ID 已存在则覆盖
    fn insert(&mut self, task: Task) {
        self.tasks.insert(task.id, task);
    }

    // 通过 ID 查找任务
    fn get(&self, id: u64) -> Option<&Task> {
        self.tasks.get(&id)
    }

    // 删除任务
    fn remove(&mut self, id: u64) -> Option<Task> {
        self.tasks.remove(&id)
    }
}
```

### entry API：处理键存在或不存在的逻辑

任务统计是一个常见场景：如果 worker 已存在，增加计数；如果不存在，创建新条目。`entry` API 优雅地处理这种"存在则更新，不存在则插入"的模式：

```rust
impl TaskStore {
    // 记录 worker 完成任务
    fn record_task_completion(&mut self, worker_id: String) {
        // entry 返回 Entry 枚举，or_insert 在键不存在时插入默认值
        let count = self.worker_task_counts
            .entry(worker_id)
            .or_insert(0);  // 键不存在时插入 0，返回可变引用
        *count += 1;
    }

    // 更复杂的 entry 用法：使用 or_insert_with 延迟计算
    fn get_or_create_task(&mut self, id: u64, name: String) -> &mut Task {
        self.tasks
            .entry(id)
            .or_insert_with(|| Task {
                id,
                name,
                payload: Vec::new(),
            })
    }
}
```

### HashMap 遍历

```rust
impl TaskStore {
    fn print_worker_stats(&self) {
        println!("Worker 统计:");
        for (worker_id, count) in &self.worker_task_counts {
            println!("  {} 完成了 {} 个任务", worker_id, count);
        }
    }
}
```

**注意**：HashMap 的遍历顺序是不确定的。如果需要有序遍历，可以使用 `BTreeMap`（基于红黑树，按键排序）。

### HashMap 键的要求

HashMap 的键类型必须实现 `Eq` 和 `Hash` trait：

```rust
// 基本类型已实现 Eq 和 Hash，可以直接用
let map: HashMap<i32, String> = HashMap::new();
let map: HashMap<String, Task> = HashMap::new();

// 自定义类型作为键需要 derive
#[derive(Hash, Eq, PartialEq)]
struct TaskId(u64);

let map: HashMap<TaskId, Task> = HashMap::new();
```

## HashSet：去重集合

### 为什么需要 HashSet

在分布式系统中，worker 节点可能重复注册（网络重试、重启等原因）。我们需要存储唯一的 worker 集合，自动去重。`HashSet<T>` 就是 `HashMap<K, ()>` 的简化版，只存储键，适合去重和成员检查。

### 使用 HashSet 管理 worker

```rust
use std::collections::HashSet;

struct WorkerRegistry {
    // 存储活跃 worker 的 ID，自动去重
    workers: HashSet<String>,
}

impl WorkerRegistry {
    fn new() -> Self {
        Self {
            workers: HashSet::new(),
        }
    }

    // 注册 worker，如果已存在则忽略
    fn register(&mut self, worker_id: String) -> bool {
        // insert 返回 true 如果插入成功（键不存在）
        self.workers.insert(worker_id)
    }

    // 注销 worker
    fn unregister(&mut self, worker_id: &str) -> bool {
        // remove 返回 true 如果键存在并删除
        self.workers.remove(worker_id)
    }

    // 检查 worker 是否已注册
    fn is_registered(&self, worker_id: &str) -> bool {
        self.workers.contains(worker_id)
    }

    // 获取活跃 worker 数量
    fn active_count(&self) -> usize {
        self.workers.len()
    }
}
```

### 集合运算

HashSet 支持并集、交集、差集等集合运算，在权限管理、标签匹配等场景很有用：

```rust
impl WorkerRegistry {
    // 查找能处理特定任务的 worker（假设 worker 有技能标签）
    fn find_qualified_workers(
        &self,
        required_skills: &HashSet<String>,
        worker_skills: &HashMap<String, HashSet<String>>
    ) -> HashSet<String> {
        let mut qualified = HashSet::new();

        for worker in &self.workers {
            if let Some(skills) = worker_skills.get(worker) {
                // 检查 worker 是否拥有所有必需技能
                if required_skills.is_subset(skills) {
                    qualified.insert(worker.clone());
                }
            }
        }

        qualified
    }
}
```

### 创建 HashSet 的便捷方法

```rust
// 从 Vec 创建，自动去重
let task_ids: HashSet<u64> = vec![1, 2, 3, 2, 1].into_iter().collect();
// 结果: {1, 2, 3}

// 使用 extend 批量插入
let mut set = HashSet::new();
set.extend(vec![1, 2, 3, 2]);
```

## 综合案例：任务调度器

让我们把三个集合结合起来，实现一个简单的任务调度器：

```rust
use std::collections::{HashMap, HashSet};

struct TaskScheduler {
    // 任务队列：按顺序存储待处理任务
    pending: Vec<Task>,
    // 任务存储：通过 ID 查找任务
    tasks: HashMap<u64, Task>,
    // worker 注册表：去重的 worker 集合
    workers: HashSet<String>,
    // worker 负载：记录每个 worker 的任务数
    worker_loads: HashMap<String, usize>,
}

#[derive(Clone)]
struct Task {
    id: u64,
    name: String,
    payload: Vec<u8>,
}

impl TaskScheduler {
    fn new() -> Self {
        Self {
            pending: Vec::new(),
            tasks: HashMap::new(),
            workers: HashSet::new(),
            worker_loads: HashMap::new(),
        }
    }

    // 提交任务
    fn submit(&mut self, task: Task) {
        self.tasks.insert(task.id, task.clone());
        self.pending.push(task);
    }

    // 注册 worker
    fn register_worker(&mut self, worker_id: String) {
        if self.workers.insert(worker_id.clone()) {
            self.worker_loads.insert(worker_id, 0);
        }
    }

    // 分配任务给负载最低的 worker
    fn assign_task(&mut self) -> Option<(String, Task)> {
        if self.pending.is_empty() || self.workers.is_empty() {
            return None;
        }

        // 找到负载最低的 worker
        let worker_id = self.worker_loads
            .iter()
            .min_by_key(|(_, &load)| load)
            .map(|(id, _)| id.clone())?;

        // 取出任务
        let task = self.pending.remove(0);

        // 更新负载
        *self.worker_loads.get_mut(&worker_id)? += 1;

        Some((worker_id, task))
    }

    // 查询任务状态
    fn get_task(&self, id: u64) -> Option<&Task> {
        self.tasks.get(&id)
    }

    // 打印状态
    fn print_status(&self) {
        println!("待处理任务: {}", self.pending.len());
        println!("活跃 worker: {}", self.workers.len());
        println!("Worker 负载:");
        for (worker, load) in &self.worker_loads {
            println!("  {}: {} 个任务", worker, load);
        }
    }
}

fn main() {
    let mut scheduler = TaskScheduler::new();

    // 注册 worker
    scheduler.register_worker("worker-1".to_string());
    scheduler.register_worker("worker-2".to_string());
    scheduler.register_worker("worker-1".to_string());  // 重复注册，会被忽略

    // 提交任务
    for i in 1..=5 {
        scheduler.submit(Task {
            id: i,
            name: format!("任务-{}", i),
            payload: vec![],
        });
    }

    scheduler.print_status();

    // 分配任务
    while let Some((worker, task)) = scheduler.assign_task() {
        println!("分配 {} 给 {}", task.name, worker);
    }

    scheduler.print_status();
}
```

## 常见陷阱

### 1. Vec 的 remove(0) 性能问题

`Vec::remove(0)` 会移动所有元素，时间复杂度 O(n)。如果频繁在头部操作，考虑使用 `VecDeque`。

### 2. HashMap 遍历顺序不确定

不要依赖 HashMap 的遍历顺序，它是不确定的。需要有序时用 `BTreeMap`。

### 3. 迭代时修改集合

```rust
// 错误：在迭代时插入元素
for task in &queue.tasks {
    queue.tasks.push(new_task);  // 编译错误
}

// 正确：使用 retain 过滤
queue.tasks.retain(|task| task.is_active());
```

## Trade-off 分析

### Vec vs HashMap

**Vec 优势**：

- 内存紧凑，缓存友好
- 保持插入顺序
- 按索引访问 O(1)

**Vec 劣势**：

- 查找 O(n)（除非排序后二分查找）
- 头部插入/删除 O(n)

**HashMap 优势**：

- 查找、插入、删除平均 O(1)
- 灵活的键类型

**HashMap 劣势**：

- 内存开销较大
- 不保持顺序
- 性能依赖哈希函数质量

### HashSet vs Vec+去重

**HashSet 优势**：

- 插入时自动去重
- 成员检查 O(1)

**HashSet 劣势**：

- 不保持插入顺序
- 内存开销比 Vec 大

选择建议：

- 需要顺序、元素较少：用 Vec
- 需要快速查找、键值对：用 HashMap
- 需要去重、成员检查：用 HashSet

## 总结

Rust 的三个核心集合各有所长：

- **Vec\<T\>**：动态数组，适合有序数据、队列场景
- **HashMap\<K, V\>**：哈希表，适合快速查找、计数场景
- **HashSet\<T\>**：哈希集合，适合去重、成员检查

在分布式任务处理系统中，我们使用 Vec 存储任务队列，HashMap 存储任务索引和 worker 负载，HashSet 管理 worker 注册表。这三个集合组合使用，可以高效地处理任务调度、状态查询和负载均衡等需求。
