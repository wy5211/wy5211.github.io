上一篇我们聊了 Hooks——在特定时间点自动执行的脚本。Hook 的局限在于它是"被动"的：只能在预设的触发点运行，不能按需调用。

如果你想让 Claude 按特定流程做事——比如"用 TDD 方式开发这个功能"、"帮我按团队规范做代码审查"、"以特定格式生成文档"——这就需要 Skills 了。

Skill 本质上是一份预制的指令模板。你定义好"这个 Skill 做什么、怎么做、遵循什么规范"，之后只需要一句话就能触发 Claude 执行整套流程。和 Hooks 的区别是：Hooks 像闹钟，到点自动响；Skills 像食谱，你想做这道菜的时候才翻开来用。

## Skill 的结构

一个 Skill 由两个部分组成：

1. **frontmatter**：元数据——Skill 的名字、描述、触发条件
2. **指令内容**：Markdown 格式的详细指令，告诉 Claude 这个 Skill 要怎么做

文件放在 `.claude/skills/` 目录下。比如一个简单的代码审查 Skill：

```markdown
---
name: code-review
description: 按照团队规范审查代码变更
---

# 代码审查 Skill

## 审查流程

1. 读取当前 git diff，了解变更内容
2. 逐文件检查以下维度：
   - 代码逻辑正确性
   - 是否有安全风险（SQL 注入、XSS 等）
   - 命名是否清晰、是否遵循项目命名规范
   - 是否有可以提取的重复代码
3. 对每个问题给出具体位置和修改建议
4. 按严重程度分级：critical / warning / suggestion

## 输出格式

对每个发现的问题，按以下格式输出：

- 文件路径和行号
- 问题描述
- 修改建议（附代码片段）
- 严重程度
```

这就是一个完整的 Skill。当你在 Claude Code 里输入 `/code-review` 时，Claude 会读取这个文件，按照里面定义的流程执行代码审查。

## 社区热门 Skills 实测

社区已经有大量现成的 Skills 可以直接用。这里推荐几个我实际用过、确实能提升效率的。

### Superpowers——最火的开发工作流

Superpowers 是目前社区最受欢迎的 Claude Code Skill（GitHub 上 star 数最多的之一）。它提供了一套完整的开发工作流：从需求分析到 Plan 到 TDD 实现到代码审查。

没有 Superpowers 的时候，你让 Claude "加个功能"，它直接写代码。有了 Superpowers，它会先分析需求、制定 Plan、写测试用例、再实现代码——整个过程更接近一个有经验的工程师的工作方式。

适合谁：经常让 Claude 从零开发新功能的人。如果你主要是用 Claude 做小修改和 bug 修复，Superpowers 可能有点重了。

### Context7——解决"训练数据过时"问题

Claude 的训练数据有截止日期。这意味着当你让它用最新版本的框架（比如 Next.js 15、Tailwind CSS 4）时，它可能会用到已经废弃的 API。

Context7 解决的就是这个问题——它在 Claude 回答之前，先去拉取最新的官方文档和代码示例，注入到上下文里。这样 Claude 就能基于最新文档来回答，而不是凭训练数据猜测。

实际体验：让 Claude 用 Next.js 15 的 Server Actions 写一个表单提交功能。没有 Context7 时，它大概率会用 Pages Router 的 API Routes 方式；有了 Context7，它会拉取最新的 App Router 文档，正确使用 Server Actions。

适合谁：经常用新版本框架和库的开发者。如果你的项目技术栈比较稳定，Context7 的价值就小一些。

### Task Master——从 PRD 到任务实现

Task Master 把一个大的需求拆解成可管理的小任务，然后逐步实现。你给它一份 PRD（产品需求文档），它帮你：

1. 拆分成独立的开发任务
2. 确定任务依赖关系和执行顺序
3. 逐个任务实现，每完成一个自动验证
4. 跟踪整体进度

适合谁：接到一个较大的需求，不知道从哪里下手的时候。Task Master 帮你把大象切成一口能吃的小块。

### Deep Research——多步自动研究

当你需要做一个技术调研——比如"比较 React Query 和 SWR，看哪个更适合我们的项目"——Deep Research 会自动进行多步搜索：搜索官方文档、搜索社区评价、搜索性能对比，然后综合所有信息给出一份带引用的报告。

适合谁：经常需要做技术选型和技术调研的人。

### Code Review——自动 PR 审查

不同于你手动让 Claude "看看这段代码"，Code Review Skill 有一套完整的审查框架：安全性检查、性能检查、代码风格检查、可维护性评估。每个维度都有具体的检查项和评分标准。

适合谁：团队里没有专门的 Code Reviewer，或者想在做人工 Review 之前先用 AI 过一遍。

## 一站式工具包：everything-claude-code（ECC）

如果你觉得一个个挑 Skill 太麻烦，有一个选择是直接装 everything-claude-code（简称 ECC）。

ECC 是一位 Anthropic 黑客松获奖者整理的配置集合，包含 136+ 个 Skills、Agents、Hooks 和 MCP 配置，覆盖了：

- **代码审查**：自动检查代码质量、安全漏洞
- **安全扫描**：检测常见的安全问题
- **测试覆盖**：自动生成和补充测试用例
- **构建错误修复**：分析构建错误并自动修复
- **Sprint 追踪**：项目管理集成
- **自动修复**：发现问题后自动尝试修复
- **行为验证**：验证代码变更是否符合预期

**适合谁**：不想花时间一个个挑、想快速拥有一套完整配置的开发者。特别是刚从其他 AI 编程工具转到 Claude Code 的人，ECC 能帮你快速上手。

**不适合谁**：已经有一套自己定制配置的人。如果你已经花时间调教好了自己的 Skills 和 Hooks，ECC 里的配置可能会和你现有的冲突。而且 136+ 个配置全部加载会占用不少上下文空间。

**安装方式**：从 GitHub（arabicapp/everything-claude-code）的 Releases 页面下载，支持 Windows、macOS、Linux。如果你的团队主要用中文，还有中文版：everything-claude-code-zh（xu-xiang 维护）。

## Skill 的发现与安装

怎么找到适合自己的 Skill？有几个主要渠道。

### claudeskills.info

目前最全的社区 Skill 目录，收录了 658+ 个 Skill。按类别浏览——开发、效率、设计、测试、安全等——每个 Skill 有简介、评分和安装方式。适合"我想找一个做 XX 的 Skill"的场景。

### awesome-claude-skills（GitHub）

GitHub 上的精选列表，收录了 1000+ 个 Skill。按功能分类，每个都有 GitHub 链接和简要说明。适合按功能浏览发现新工具。

### npx skills add

社区形成的一键安装标准。大部分 Skill 都支持通过这个命令安装：

```bash
npx skills add <skill-name>
```

它会自动把 Skill 文件放到 `.claude/skills/` 目录下，配置好描述和触发条件。

### 判断一个 Skill 值不值得装

不是所有 Skill 都靠谱。安装之前看这几点：

- **更新频率**：最后一次更新是什么时候？超过 3 个月没更新的 Skill 可能已经过时了
- **Star 数和使用量**：虽然是参考，但大量用户在用的 Skill 通常更稳定
- **实际测试**：装了之后跑一次，看效果是否符合预期。不好用的直接删掉，不要留着占上下文空间

**一个重要的建议：装 8-12 个精选 Skill 就够了。** 装太多反而有问题——每个 Skill 都会在对话中占用上下文，Skill 太多会让 Claude 的响应变慢，甚至产生冲突。聚焦在解决你最常见需求的几个 Skill 上。

## 自己写一个 Skill

社区 Skill 虽多，但不可能完全匹配你的需求。比如我是一个博客写作者，我想要一个 Skill 专门帮我按特定格式创建博客系列——从计划书到第一篇文章。这个需求太小众，社区里不会有现成的。

那就自己写一个。

### 分析需求

首先想清楚这个 Skill 要解决什么问题：

- 我经常需要创建新的博客系列
- 每个系列有固定的结构：plan.md + N 篇文章
- 文章有固定的 frontmatter 格式
- 写完要跑 build 验证

这些都是重复性操作，但每次的内容不同。适合做成 Skill。

### 编写 Skill 文件

```markdown
---
name: blog-series-writer
description: 创建和撰写博客系列文章，从计划书到完整文章
---

# 博客系列写作 Skill

## 前置条件

确认以下信息：

- 系列主题
- 文章数量
- 文章类型（A 技术 / B 健康 / C 心理 / D 投资 / E 生活）
- 目标读者

## 创建流程

### 第一步：创建目录和计划书

1. 在 content/posts/{category}/ 下创建新目录
2. 创建 plan.md，包含：
   - 背景（为什么写这个主题）
   - 博客目录结构（所有文章的文件名）
   - 阶段划分
   - 每篇文章大纲（学习目标、核心问题、内容要点）
   - 系列节奏规划

### 第二步：逐篇撰写

对每篇文章：

1. 先读 plan.md 中本篇的大纲
2. 如果不是第一篇，读上一篇确保连贯性
3. 按以下格式创建文件：

frontmatter 格式：
```

title: "序号 文章标题"
date: "YYYY-MM-DD"
summary: "一句话总结（100字以内）"
tags: ["标签1", "标签2"]
category: "{category}"
cover: ""
draft: false
series: "{系列名称}"
seriesOrder: N

````

### 第三步：构建验证

所有文章写完后，运行：
```bash
cd ~/Desktop/code/wy5211.github.io && npm run build
````

修复所有错误直到构建通过。

## 写作要求

- 开头用一个真实问题抓住读者，不要用"本文将介绍..."
- 每个知识点先说为什么需要关注它
- 用一个贯穿全文的真实场景
- 结尾做 trade-off 分析
- 代码占比不超过 40%
- 不用玩具示例

````

这就是一个完整的 Skill。它告诉 Claude 要做什么、怎么做、遵循什么规范。之后我只需要说"用 blog-series-writer 创建一个关于 Redis 的技术系列"，Claude 就会按照这个流程执行。

### 测试和迭代

Skill 写完后不要期望一次就完美。我的建议是：

1. 先用一个简单的系列测试，看 Claude 的执行效果
2. 记录不符合预期的部分
3. 回到 Skill 文件，把缺失的指令补充上
4. 再测一次，直到效果稳定

我自己的 blog-series-writer Skill 大概迭代了 3 轮才稳定下来。第一版没有"构建验证"这一步，导致好几次文章格式有问题但没被发现。加上之后就好了。

## Skill 的进阶用法

### 参数化

你可以在 Skill 的指令里留一些"变量"，让调用时传入。比如：

```markdown
当用户调用此 Skill 时，询问以下参数：
- series_name：系列名称
- article_count：文章数量（默认 8）
- target_audience：目标读者描述
````

这样同一个 Skill 可以适应不同的需求，不用每次都改 Skill 文件。

### 组合使用

多个 Skill 可以配合。比如你先用 Task Master 把大需求拆成小任务，然后用 Code Review Skill 审查每个任务的输出。两个 Skill 各司其职，比一个全能 Skill 更灵活。

关键是要理解每个 Skill 的边界——它擅长做什么、不擅长做什么。不要试图让一个 Skill 解决所有问题，而是让每个 Skill 专注做好一件事。

## Skill vs Hook：什么时候用哪个

最后再明确一下两者的边界：

| 场景                 | 用 Hook | 用 Skill |
| -------------------- | ------- | -------- |
| 每次编辑后都要格式化 | ✅      | ❌       |
| 提交前都要跑检查     | ✅      | ❌       |
| 按特定流程写代码     | ❌      | ✅       |
| 按特定格式审查代码   | ❌      | ✅       |
| 生成特定格式的文档   | ❌      | ✅       |
| 自动记录操作日志     | ✅      | ❌       |

简单判断：**如果操作需要在特定时间点自动执行，用 Hook；如果操作需要按需调用且有复杂流程，用 Skill。**

下一篇我们聊 MCP——Skills 和 Hooks 扩展的是 Claude 的"能力"，而 MCP 扩展的是 Claude 能触及的"世界"。接上 MCP 服务器后，Claude 就不再局限于本地文件和终端了。
