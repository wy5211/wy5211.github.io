这个系列写了七篇，从 CLAUDE.md 到 Workflow，每个功能都拆开讲了。但功能是零散的，你需要的是一个完整的日常工作流——什么时候用什么、怎么配合使用、什么时候不该用。

这篇是系列的收尾，我会还原一个真实的开发日，把所有功能串起来。然后说说社区里有哪些好用的周边工具，以及我踩过的 10 个最常见的坑。

## 一个真实的开发日

### 早上 9:00：新会话启动

打开终端，启动 Claude Code。

它自动做了这些事：

- 加载项目根目录的 `CLAUDE.md`，了解项目的技术栈和约定
- 加载 `.claude/` 下的记忆文件，恢复之前积累的项目经验
- `SessionStart` Hook 触发，自动 `git pull` 拉取最新代码

你不需要做任何手动操作。三分钟前你还什么都没开始，现在 Claude 已经完全了解了你的项目状态。

### 上午 9:30：新功能开发

接到一个需求："给博客文章加一个字数统计和预估阅读时间"。

这个需求涉及多个文件（文章渲染逻辑、组件、类型定义），而且需要决定实现方式（构建时计算还是运行时计算）。所以进入 Plan 模式。

Plan 的结果：构建时计算更合理（文章内容是静态的，没必要运行时算），需要改三个文件。确认方案后退出 Plan 模式，Claude 按计划实现。

每次 Claude 编辑文件后，`PostEdit` Hook 自动跑 Prettier 格式化。你看到的代码永远是整齐的。

### 中午 12:00：提交代码

功能开发完成。Claude 执行 `git commit` 时，`PreCommit` Hook 触发，自动跑 TypeScript 类型检查和 ESLint。

检查通过，commit 成功。如果检查失败，Claude 会自动看到错误信息并修复，然后重新 commit——你不需要介入。

提交完成后，让 Claude 通过 GitHub MCP Server 创建一个 PR，自动填写 PR 描述。然后去吃午饭。

### 下午 2:00：代码审查

回来后 PR 已经创建好了。让 Claude 用 Workflow 编排多个 Agent 做代码审查：

- Agent 1：Bug 检查（逻辑错误、边界条件）
- Agent 2：安全审查（XSS、注入）
- Agent 3：性能分析（不必要的计算、缓存策略）
- Agent 4：可维护性（命名、复杂度）

四个 Agent 并行审查，各自给出发现。然后对每个发现做对抗验证，过滤掉误报。最终汇总成一份审查报告。

根据审查结果，Claude 修复了两个有效问题。修复后 Hook 自动跑检查，确认没有新问题。

### 下午 4:00：Bug 修复

收到一个线上 bug 报告：某篇文章的系列导航显示错误。

这个任务很简单——单文件 bug 修复，不需要 Plan 模式。直接告诉 Claude 错误现象，它定位问题、修复、跑 build 验证。5 分钟搞定。

如果你装了 Sentry MCP，甚至可以直接说"看看最近有没有相关的线上错误"，Claude 会从 Sentry 拉取错误日志帮你分析。

### 下午 5:30：收工

一天的工作结束。做几件事：

1. 让 Claude 总结今天改了什么，更新记忆文件（方便明天继续）
2. `git push` 推送到远程
3. `Stop` Hook 触发桌面通知，确认任务完成

明天再打开 Claude Code 时，记忆和 CLAUDE.md 会自动加载，无缝衔接。

## 社区周边工具

除了 Claude Code 本身的功能，社区还涌现了一批周边工具，能进一步提升使用体验。

### Claude HUD——实时状态栏

Claude HUD 是一个状态栏工具，实时显示 Claude Code 的运行状态——上下文使用量、当前在执行什么工具、Agent 进度等。

没有 HUD 的时候，你让 Claude 执行一个耗时任务，只能盯着终端看它一个字一个字地输出。有了 HUD，你在状态栏就能看到进度——"正在审查第 3/10 个文件"，不用一直盯着屏幕。

安装方式：

```bash
claude install claude-hud
```

或者通过 Skill 安装：

```bash
npx skills add claude-hud
```

### everything-claude-code（ECC）——一站式配置包

在 [第 5 篇](/posts/claude-code-tips/05-skills-extend-capabilities) 里我们详细介绍过 ECC。这里再提一次是因为它作为"开箱即用"的配置包，对新手特别友好。

136+ 个 Skills、Agents、Hooks、MCP 配置打包在一起，从代码审查到安全扫描到测试覆盖都覆盖了。如果你不想一个个配置，ECC 是最快的方式。

从 GitHub（arabicapp/everything-claude-code）下载，有中文版（everything-claude-code-zh）。

### Skill 发现目录

找 Skill 的两个主要入口：

- **claudeskills.info**：658+ 个社区 Skill 的目录网站，按类别浏览，有评分和简介
- **awesome-claude-skills**（GitHub ComposioHQ/awesome-claude-skills）：1000+ 个 Skill 的精选列表

找 MCP 服务器的入口：

- **mcpmarket.com**：MCP 服务器排行榜，按热度排序

### npx skills add——一键安装

大部分社区 Skill 都支持通过 `npx skills add <name>` 一键安装。它会自动把 Skill 文件放到 `.claude/skills/` 目录下，省去手动下载和配置的步骤。

### Ruflo——多 Agent 编排工具

如果你觉得 [第 7 篇](/posts/claude-code-tips/07-workflow-multi-agent) 里讲的 Workflow 手写起来比较复杂，Ruflo 提供了一个更友好的多 Agent 编排工具。它帮你管理 Agent 的启动、通信和结果汇总，适合经常需要多 Agent 协作的复杂场景。

## 10 个最常见的坑

用了三个月 Claude Code，这些是我遇到最多的坑。

### 1. CLAUDE.md 写太长

超过 200 行的 CLAUDE.md Claude 会开始"选择性忽略"。保持精炼，核心约定直接写，详细规范用文件链接引用。

### 2. 上下文溢出导致"失忆"

长对话（超过 1-2 小时）中 Claude 会忘记早期说过的话。解决方案：用 `/compact` 压缩上下文，或者在关键节点保存记忆。参考 [第 2 篇](/posts/claude-code-tips/02-memory-context-management)。

### 3. 不该用 Plan 模式的时候用了

Plan 模式有成本——Claude 需要花时间分析、写 Plan、等你确认。对于简单的 bug 修复和单文件改动，直接开干更快。参考 [第 3 篇](/posts/claude-code-tips/03-plan-mode-deep-dive)。

### 4. Hook 配置错误导致无限循环

如果 `PostEdit` Hook 本身会修改文件（比如格式化后文件变了），可能触发新的编辑事件，导致无限循环。解决方案：确保 Hook 是幂等的——格式化一个已经格式化过的文件不应该产生任何变更。

### 5. MCP 服务器超时没处理

首次用 npx 启动 MCP 服务器时需要下载依赖，可能超时。解决方案：提前手动运行一次 `npx -y @xxx/server-xxx` 让依赖缓存到本地。参考 [第 6 篇](/posts/claude-code-tips/06-mcp-connect-world)。

### 6. Workflow 的 token 消耗失控

多 Agent 编排的 token 消耗是单 Agent 的 N 倍。设置 token 预算上限，不要让 Workflow 无限制地运行。参考 [第 7 篇](/posts/claude-code-tips/07-workflow-multi-agent)。

### 7. 在错误的项目层级写记忆

记忆是分项目隔离的——在项目 A 里保存的记忆不会出现在项目 B 的对话里。如果你有一条跨项目通用的经验（比如"我喜欢用函数式写法"），应该写进全局 CLAUDE.md（`~/.claude/CLAUDE.md`）而不是某个项目的记忆里。

### 8. 让 Claude 做它不擅长的事

Claude 不擅长精确数值计算、严格排序、和大范围的代码追踪。如果你让它"帮我计算这个函数的精确执行时间"或者"找出项目中所有使用了这个变量的地方"，结果可能不准确。这种时候用传统工具（profiler、grep）更靠谱。

### 9. 不看 Claude 的输出直接信任

Claude 生成的代码大多数时候是对的，但不是每次都对。特别是涉及业务逻辑的部分，一定要自己审查。把它当成一个很能干但偶尔犯错的同事——信任但要验证。

### 10. 忽略权限配置

`.claude/settings.json` 里的权限配置（`permissions` 字段）能减少大量的权限确认弹窗。比如你可以允许 Claude 自由运行 `npm run build` 和 `git diff`，而不需要每次都手动确认。用 [第 4 篇](/posts/claude-code-tips/04-hooks-automation) 提到的 `fewer-permission-prompts` Skill 可以自动分析你的对话历史，推荐合适的权限配置。

## 不同场景的最佳实践速查

### 新项目启动

1. 写一份精炼的 CLAUDE.md（技术栈、目录结构、构建命令）
2. 在全局 CLAUDE.md 里写好个人偏好
3. 配置 3-5 个最常用的 MCP 服务器
4. 配置 PostEdit Hook（自动格式化）和 PreCommit Hook（类型检查 + lint）
5. 根据项目需要装 2-3 个 Skill

### 日常开发

1. Plan 模式设计 → 确认方案 → 实现代码 → Hook 自动检查 → 测试
2. 每隔 1-2 小时用 `/compact` 压缩上下文
3. 关键决定保存到记忆里

### Bug 修复

1. 直接对话，不需要 Plan 模式
2. 给足错误上下文：错误信息、复现步骤、相关文件
3. 修复后跑 build 验证

### 代码审查

1. 用 Workflow 编排多 Agent 并行审查
2. 对审查结果做对抗验证
3. 按严重程度排列修复优先级

### 大规模重构

1. Plan 模式 + Worktree（在独立分支上工作，不影响主分支）
2. 如果重构范围很大，考虑拆成多个会话，每个会话处理一个子模块
3. 用记忆记录重构进度和约定

## 成本控制实操

### 用 /compact 压缩上下文

`/compact` 是最直接的成本控制手段。它把当前对话压缩成摘要，释放大量上下文空间。压缩后的上下文约是原来的 10-20%。

什么时候该用：对话超过 30 轮、或者 Claude 开始重复之前说过的话。

### 合理选择模型

不是所有任务都需要最强模型。一个实用的分级：

- **Opus**：架构决策、复杂重构、需要深度推理的任务
- **Sonnet**：日常开发、代码生成、bug 修复（大部分时候够用）
- **Haiku**：格式化、简单搜索、文本处理

在 Workflow 中可以为不同的 Agent 指定不同的模型——审查用 Sonnet，架构决策用 Opus，格式化用 Haiku。

### 用 Worktree 隔离实验

大规模修改前创建 Worktree，在独立分支上实验。如果结果不满意，直接丢弃分支，不会有任何损失。这避免了在主分支上反复修改、回滚带来的额外 token 消耗。

## Claude Code 的能力边界

最后说说 Claude Code 擅长什么、不擅长什么。了解边界，才能用好工具。

### 擅长

- 代码生成与重构——这是它的核心能力，远超手动编写
- 文件操作——批量创建、修改、移动文件
- 搜索与分析——在代码库中找模式、分析依赖关系
- 流程编排——通过 Hooks/Skills/Workflow 自动化工作流
- 文档生成——从代码生成文档、从需求生成 Plan

### 不擅长

- 精确数值计算——涉及浮点数、精确排序的任务
- 大型项目全局理解——超过几百个文件的项目，Claude 很难把握全貌
- 需要 GUI 的操作——它只能操作终端和文件，不能帮你点按钮
- 实时协作——它是异步的，不是实时协作工具
- 重复性很高的机械操作——虽然 Hooks 能自动化一部分，但有些操作还是需要脚本或专用工具

### 和 Cursor/Copilot 的对比

这三个工具各有所长，不是替代关系：

- **Claude Code**：擅长终端工作流、批量操作、复杂重构。适合"给一个任务，让它端到端完成"的场景。
- **Cursor**：擅长编辑器内的实时补全和对话。适合"边写代码边问"的场景。
- **Copilot**：擅长行级补全和简单函数生成。适合"写到一半让它补全"的场景。

我的用法是：日常开发用 Cursor 做实时补全和小改动，遇到较大的任务（重构、新功能、批量操作）切换到 Claude Code。Copilot 用得少了，因为 Cursor 的补全已经够用。

## 系列回顾

八篇文章，我们从一个"不好用的 Claude Code"走到了一个"高效的 AI 开发工作流"：

1. **CLAUDE.md**——让 Claude 理解你的项目
2. **记忆系统**——让关键信息跨对话传递
3. **Plan 模式**——想清楚再动手
4. **Hooks**——把重复操作自动化
5. **Skills 生态**——装上社区能力，或自己造
6. **MCP 生态**——连上 GitHub、数据库、浏览器
7. **Workflow**——多 Agent 协作处理大任务
8. **这篇**——串成完整工作流

这些功能不是堆砌的，它们有清晰的层次关系：CLAUDE.md 和记忆是地基，Plan、Hooks、Skills 是日常工具，MCP 和 Workflow 是进阶能力。你不需要一次全部用上——先从 CLAUDE.md 开始，逐步加上你需要的功能。

用好 Claude Code 的核心不是记住所有功能，而是建立一个适合自己的工作流。希望这个系列能帮你少走一些弯路。
