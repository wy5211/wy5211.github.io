上一篇我们聊了用 Claude Code 做代码重构、测试和调试。但到目前为止，我们讨论的都是"个人使用"的场景。实际工作中，代码不是你一个人写的——你在一个团队里，需要提交代码、创建 PR、做 code review、保持团队规范一致。

这篇文章聊聊怎么把 Claude Code 融入 Git 工作流和团队协作。

## /commit：让 commit message 不再是痛苦

写 commit message 看起来是一件小事，但实际上很影响团队的代码可读性。好的 commit message 能让人在 `git log` 里快速理解每次改动的目的和影响。差的 commit message（"fix bug"、"update"、"wip"）让人完全摸不着头脑。

Claude Code 的 `/commit` 命令能帮你自动生成规范的 commit message。它的工作流程是：

1. 并行运行 `git status`、`git diff`、`git log`，了解当前变更和历史提交风格
2. 分析所有变更，理解改动的目的和影响范围
3. 参考仓库已有的提交风格，生成 commit message
4. 暂存相关文件并创建提交

关键是第三步——它会**参考你仓库已有的提交风格**。如果你的团队习惯用 Conventional Commits（`feat: xxx`、`fix: xxx`），它生成的 message 也会是这个格式。如果你的团队习惯用中文描述，它也会用中文。

```
你: /commit
```

就这么简单。Claude Code 会分析变更，生成 message，然后让你确认。你可以修改 message 再提交，也可以直接确认。

### 什么时候该自己写 commit message

/commit 不是万能的。以下场景自己写更好：

**重大的架构变更**。这种 commit 的 message 本身就是文档，需要你仔细措辞，说明为什么做这个变更、影响范围是什么、迁移策略是什么。

**涉及多个不相关改动的 commit**。你在一个分支上改了三件不相关的事，应该拆成三个 commit，每个 commit 对应一个改动。/commit 可能会把所有改动混成一个 commit——这时候你需要手动拆分。

## /review-pr：从分析到创建

创建 PR 也是一个繁琐但重要的流程。你需要写一个清晰的标题、详细的描述、列出改动的文件、说明测试计划。Claude Code 的 `/review-pr` 能帮你自动化这个过程。

它会：

1. 分析当前分支相对于主分支的所有 commit
2. 检查远程仓库的状态
3. 生成 PR 标题（不超过 70 个字符）和描述
4. 推送分支并创建 PR

PR 描述会自动包含 summary（改动概要）和 test plan（测试计划），格式规范且完整。

```
你: /review-pr
```

它会分析你的改动，生成一个像这样的 PR：

```
标题：fix: 修复任务筛选查询的 SQL 注入风险

描述：
## Summary
- 将 task.service.ts 中的字符串拼接查询改为参数化查询
- 为 QueryTaskDto 添加输入校验，防止非法字符
- 补充边界情况的单元测试

## Test plan
- [ ] 验证包含特殊字符的 project name 不再触发 SQL 错误
- [ ] 运行 `npm test` 确认所有现有测试通过
- [ ] 手动测试筛选接口的正常和异常场景
```

这个描述比大多数手写的 PR 描述都要清晰和完整。

## PR 审查：让 AI 做第一轮

code review 是团队保证代码质量的关键环节。但人工 review 有一个痛点：reviewer 的时间有限，可能只关注逻辑正确性，忽略了一些模式性的问题——类型安全、错误处理一致性、命名规范等。

让 Claude Code 做第一轮 review 是一个实用的策略。它不会替代人工 review，但能帮你快速发现那些"模式化的问题"，让人工 review 聚焦在更重要的业务逻辑上。

你可以这样用：

```
审查 PR #142 的变更，重点检查：
1. 类型安全：有没有不安全的类型断言
2. 错误处理：异常是否被正确捕获和传播
3. 性能：有没有 N+1 查询或不必要的循环
4. 安全：用户输入是否有校验
```

Claude Code 会读取 PR 的 diff，按你指定的维度检查，输出一份问题清单。你可以把这份清单作为 review 评论的基础，或者先自己处理这些问题再让同事 review。

这样做的好处：你的同事收到的 PR 质量更高了（明显的低级问题已经被你处理掉了），review 的效率也更高了（reviewer 可以聚焦在业务逻辑和架构决策上）。

## CLAUDE.md 团队规范

团队协作中最容易被忽略的一个问题：**CLAUDE.md 不应该是某一个人写的个人笔记，应该是团队共识的产物**。

如果每个人的 CLAUDE.md 都不一样，Claude Code 在不同人的机器上行为不一致——张三的 Claude Code 生成的代码遵循一套规范，李四的 Claude Code 生成的代码遵循另一套规范。这和"没有规范"差不多。

### 团队 CLAUDE.md 应该写什么

**技术栈和框架版本**。"我们用 NestJS 10 + TypeORM 0.3 + PostgreSQL 15"。这些信息决定了 Claude Code 生成的代码能用到哪些 API。

**代码规范**。命名规则、文件组织、错误处理方式、DTO 定义方式——上一篇聊过的那些。这些规范应该在 project 级别的 CLAUDE.md 中，提交到版本控制。

**项目结构**。目录组织、模块划分、入口文件——让 Claude Code 知道去哪里找代码。

**禁止事项**。有时候"不能做什么"比"应该做什么"更重要。"不允许使用 any 类型"、"不允许直接操作 DOM"、"不允许在 Service 中导入 Controller"。

### 团队 CLAUDE.md 不应该写什么

**个人偏好**。"我喜欢用 tabs 而不是 spaces"——这是个人偏好，放 user 级别的 CLAUDE.md，不提交到仓库。

**临时信息**。"这个 bug 我正在修"——这是会话级别的信息，放 Memory 里，不放 CLAUDE.md。

**过于具体的实现细节**。"UserService 的 register 方法先查邮箱再用 bcrypt 加密密码"——这种信息 Claude Code 通过读代码就能知道，不需要写在 CLAUDE.md 里占上下文空间。

### 保持 CLAUDE.md 的更新

项目在演进，CLAUDE.md 也应该跟着更新。几个实用建议：

- **重大技术变更时更新**。比如升级了 NestJS 版本、引入了新的 ORM、改了错误处理方式——这些变更应该立刻反映到 CLAUDE.md
- **定期 review**。每个月花 10 分钟看一遍 CLAUDE.md，删除过时的信息
- **让团队贡献**。任何人发现 CLAUDE.md 和实际代码不一致，都应该提 PR 修正

## settings.json 分层与团队共享

Claude Code 的 settings.json 也有分层机制：user → project → local。

项目级别的 settings.json（`.claude/settings.json`）应该提交到版本控制，包含团队共享的配置：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "npx prettier --write \"$FILE_PATH\" 2>/dev/null || true"
      }
    ]
  },
  "mcpServers": {
    "taskflow-db": {
      "command": "npx",
      "args": ["@anthropic/mcp-postgres", "${TASKFLOW_DATABASE_URL}"]
    }
  }
}
```

这样团队成员 clone 项目后，自动就有了统一的 Hook 和 MCP 配置。

个人配置（`~/.claude/settings.json`）放你自己的偏好——权限模式、默认模型、个人 MCP Server 等。

## 团队推广的常见阻力

在团队中推广 Claude Code，你可能会遇到几种阻力。

**安全顾虑**。"Claude Code 会不会把我们的代码发送到 Anthropic 的服务器？"——是的，它会。如果你公司的代码有严格的保密要求（金融、军工等），需要评估合规风险。对于大多数互联网公司来说，API 调用的数据不会用于训练模型，风险是可控的。

**质量担忧**。"AI 写的代码质量不行，不如自己写。"——这个问题确实存在，但更多是"使用方式"的问题而不是工具本身的问题。上一篇我们聊过，AI 生成的代码需要 review。如果你能建立"AI 生成 → 人工 review"的流程，效率比纯人工写高得多。

**习惯改变**。"我习惯了现在的开发方式，不想学新工具。"——这个阻力最难克服。最好的方式不是"培训"，而是"示范"。你在团队里用 Claude Code 高效完成了一些任务，让同事看到实际的效率提升——他们自然会有兴趣尝试。

**成本问题**。"Max 订阅 $100/月，5 个人就是 $500/月。"——算一笔账：如果每个人每天因为 Claude Code 省 1 小时（保守估计），一个月就是 20 小时。时薪 $50 的话，$1000 的价值 vs $100 的成本。ROI 很高。

## 一个实用的团队推广路径

如果你想在团队中推广 Claude Code，建议分三步走：

**第一步：个人先用起来**。你自己先用 Claude Code 完成日常任务，积累经验，验证它在你的项目中确实有用。

**第二步：建立团队规范**。写好 project 级别的 CLAUDE.md 和 settings.json，提交到版本控制。这样其他人上手就有统一的配置。

**第三步：邀请试用**。邀请 1-2 个感兴趣的同事试用，收集反馈，调整团队规范。等更多人看到实际效果后，自然会扩散。

不要试图一步到位地推广到整个团队。工具的推广是自下而上的，不是自上而下的。
