上一篇我们用 Skills 和 Hooks 把 TaskFlow 团队的重复性工作自动化了。但随着你用 Claude Code 越来越多，一个新问题出现了：对话进行到第 30 轮，你发现它开始"忘记"前面讨论过的内容。你之前说过"用 Redis 做缓存"，到了第 35 轮它又问"要不要用 Redis 做缓存？"

这不是 bug。这是上下文窗口的物理限制。这篇文章聊怎么管理上下文，以及怎么用 Memory 系统让重要信息跨会话保留。

## 上下文窗口是怎么工作的

Claude Code 有 200K tokens 的上下文窗口。听起来很多，但你得知道 tokens 是怎么被消耗的。

一个 TypeScript 文件平均每行消耗大约 15-20 tokens。一个 200 行的文件大约消耗 3000-4000 tokens。TaskFlow 项目有 127 个文件，如果你让它读了一半——上下文就快满了。

但上下文不只是你让它读的文件。每次对话中，以下内容都会占据上下文：

- **系统提示**：Claude Code 的内置指令，大约占 5000 tokens
- **CLAUDE.md**：你的项目指令，取决于你写了多少
- **对话历史**：你说的每一句话、它回复的每一个字、执行的每一个工具调用及其结果
- **工具读取的文件**：它用 Read 工具读过的所有文件内容
- **搜索结果**：Grep 和 Glob 的匹配结果

对话轮次越多，对话历史占的上下文越大。到了某个临界点，Claude Code 会自动压缩早期的对话内容——但压缩意味着信息损失。这就是为什么到了第 30 轮它开始"忘记"前面讨论的内容。

理解了这个机制，你就知道该怎么管理上下文了：**减少不必要的上下文消耗，让有限的窗口留给真正重要的信息**。

## CLAUDE.md 的分层设计

上一篇介绍了 CLAUDE.md 的基本用法。这里展开说说它的分层机制，因为这对大项目的上下文管理至关重要。

Claude Code 支持四个层级的 CLAUDE.md，按优先级从高到低：

**managed policy**（组织管理员设置）：整个组织的统一规范，任何人不能覆盖。适合放安全策略、合规要求。

**user**（`~/.claude/CLAUDE.md`）：你个人的全局偏好，在你所有项目中生效。适合放你个人的编码习惯——比如"我偏好函数式风格"、"用 pnpm 而不是 npm"。

**project**（`./CLAUDE.md` 或 `./.claude/CLAUDE.md`）：当前项目的规范，提交到版本控制，团队成员共享。适合放技术栈、代码规范、项目结构。

**local**（`./CLAUDE.local.md`）：当前项目的个人配置，不提交到 Git。适合放本地的开发环境信息——比如"我本地的数据库端口是 5433 而不是 5432"。

这些文件不是互相覆盖的关系，而是**叠加关系**。Claude Code 会同时读取所有层级的 CLAUDE.md。如果不同层级有冲突，优先级高的覆盖优先级低的。

这意味着什么？你不需要在一个巨大的 CLAUDE.md 里写所有东西。把通用规范放在 user 级别，项目特定规范放在 project 级别，本地特殊配置放在 local 级别。每层只写自己该写的内容。

### .claude/rules/ 目录：路径匹配的规则

大项目里还有一个更精细的工具：`.claude/rules/` 目录。你可以在这里放多个规则文件，每个文件指定它作用的路径范围。

比如 TaskFlow 项目里，你可以在 `.claude/rules/` 下放这些文件：

```
.claude/rules/
├── task-module.md      # paths: ["src/modules/task/**"]
├── notification.md     # paths: ["src/modules/notification/**"]
└── database.md         # paths: ["src/**/*.entity.ts", "src/**/migration/**"]
```

`task-module.md` 的内容只会在 Claude Code 操作 `src/modules/task/` 下的文件时加载。这样每个模块的规则只在该模块的上下文中生效，不会浪费其他模块的上下文空间。

规则文件支持 YAML frontmatter 来指定路径匹配：

```markdown
---
paths:
  - "src/modules/task/**"
  - "src/modules/task/*.ts"
---

# Task 模块规则

- Task 状态变更必须通过 TaskStatusService，不直接修改 status 字段
- 任务查询必须走 TaskQueryBuilder，不直接写 SQL
- 任务删除是软删除（设置 deletedAt），不允许物理删除
```

这在大型项目中特别有用。想想看：如果你的 CLAUDE.md 有 200 行规则，其中 50 行是关于 task 模块的、30 行是关于 notification 模块的——当你在改 task 模块的代码时，notification 的规则白白占了上下文空间。用 rules 目录，每个模块的规则只在需要时加载。

## Memory 系统：跨会话的记忆

CLAUDE.md 解决的是"项目规范"的问题——它是静态的、手写的、对所有会话都一样。但有些信息是动态的、在对话中产生的——比如"上次我们讨论过把认证从 JWT 换成 Session"、"张三负责 notification 模块"、"v2.3 的 deadline 是下周五"。

这些信息如果不记录下来，下次开新会话就丢了。Claude Code 的 Memory 系统就是解决这个问题。

Memory 是 Claude Code 自己写的持久化笔记，存储在 `~/.claude/projects/<project>/memory/` 目录下。它有两个组成部分：

- **MEMORY.md**：索引文件，每次会话自动加载到上下文中（前 200 行）
- **具体的记忆文件**：每个记忆是一个独立的 Markdown 文件

### Memory 的四种类型

**user**：关于你的信息——你的角色、技术背景、偏好。比如"你是后端负责人，熟悉 NestJS 和 PostgreSQL，不擅长前端"。

**feedback**：你给过 Claude Code 的反馈——什么做法你喜欢、什么做法你不喜欢。比如"不要每次都总结你做了什么，我知道"。

**project**：项目相关的动态信息——谁在做什么、deadline、技术决策。比如"团队决定 v2.3 不引入 Redis，用内存缓存先顶一下"。

**reference**：外部资源的指针——哪里能找到什么信息。比如"API 文档在 Notion 的 Engineering 空间里"。

Claude Code 会在对话过程中自动创建和更新 Memory。你也可以手动告诉它"记住这个"——它会把信息存入 Memory，下次会话自动加载。

### Memory 的使用场景

几个特别有用的场景：

**跨会话延续工作**。你今天用 Claude Code 讨论了"要不要把 NotificationService 拆分"，结论是"先不拆，等 v2.4 再说"。这个结论会被存入 project 类型的 Memory。明天你开新会话，说"上次我们讨论过 NotificationService 的拆分"，它会自动回忆起昨天的结论。

**记住团队信息**。你告诉它"李四是前端负责人，负责 task 模块的 UI 部分"，它存入 user 类型 Memory。以后你说"task 模块的前端问题问谁"，它能回答"李四"。

**积累你的偏好**。你纠正过它几次"代码注释不要每行都写，只在关键位置写"，它会把这个偏好存入 feedback 类型 Memory。以后生成代码时自动遵循。

### Memory 不是万能的

Memory 有一个物理限制：MEMORY.md 的前 200 行会被加载到每次会话的上下文中。超过 200 行的内容会被截断，不会自动加载。

这意味着 Memory 应该写**精炼的、真正需要跨会话保留的信息**。不要把所有的对话笔记都存进去——大部分对话内容只在那次会话中有用，不需要跨会话保留。

一个实用建议：定期检查 Memory 目录，删除过时的记忆。如果 Memory 膨胀到 200 行以上，最重要的信息可能被截断了。

## @ 引用策略：精准提供上下文

@ 引用是控制上下文消耗的最直接手段。上一篇提到过基本用法，这里展开说说策略。

**引用具体文件，不要引用目录**。`@src/modules/task/task.service.ts` 只加载一个文件的内容。`@src/modules/task/` 会加载整个目录下所有文件——可能几十个文件，瞬间塞满上下文。

**用 Grep 精确定位，不要让它自己找**。如果你知道要找的代码大概在哪个位置，用 Grep 先定位再引用，比让它自己搜索要快得多。

```
# 低效：让它自己找
帮我看看 task 模块里的错误处理是怎么写的

# 高效：先定位再引用
@grep "catch" src/modules/task/task.service.ts
这个 catch 块里的错误处理方式，在所有 Service 里统一成这样
```

**只在需要时引用，不要预防性引用**。有些人习惯在对话开始时引用十几个文件"以防万一"。这些文件白白占用了上下文空间。只在当前任务确实需要某个文件时才引用它。

## /clear：什么时候该重新开始

有时候最好的上下文管理就是"清掉重来"。`/clear` 命令会清空当前对话的历史，释放上下文窗口。

几个应该用 /clear 的场景：

**对话跑偏了**。你和 Claude Code 讨论了 20 轮，方向越走越偏，最后发现最初的思路是对的。这时候与其继续纠偏，不如 /clear 重新开始。

**切换任务**。你先用 Claude Code 讨论 bug 修复，现在要开始做新功能开发。两个任务的上下文混在一起，互相干扰。/clear 之后，新任务的上下文是干净的。

**上下文过大导致质量下降**。你发现 Claude Code 开始重复之前说过的内容、忽略你最新的指令——这说明上下文快满了。/clear 重置上下文，用 CLAUDE.md 和 Memory 保留关键信息就够了。

注意 /clear 只清空对话历史，不清空 CLAUDE.md 和 Memory。所以清空之后，Claude Code 仍然知道项目规范和之前积累的重要信息。

## 一个实用的上下文管理习惯

总结一下，长时间使用 Claude Code 的实用习惯：

**开头**：检查 CLAUDE.md 是否包含了当前任务需要的项目信息。如果没有，先补充。

**过程中**：精准 @ 引用，不要过度加载文件。用 Grep 定位而不是让它自己找。

**切换任务**：用 /clear 清空对话历史，而不是在同一个对话里切换。

**结束前**：如果有需要跨会话保留的信息，告诉它"记住这个"。

**定期**：检查 Memory 目录，清理过时信息，确保重要记忆不被截断。

这些习惯单个看起来都是小事，但它们能让你的上下文利用率提升很多——花在"纠正它忘记的内容"上的时间会大幅减少。
