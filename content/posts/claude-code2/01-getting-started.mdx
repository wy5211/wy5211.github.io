你用过 GitHub Copilot 吧？写函数的时候它会自动补全，速度很快，但经常补出来的代码你还得改半天。你试过 Cursor 吗？整个编辑器都是 AI 驱动的，体验不错，但遇到跨文件的重构还是力不从心。

Claude Code 走的是一条完全不同的路。它不是一个"更聪明的代码补全"，而是一个**能在终端里跑命令、读代码、改文件的 AI 助手**。区别在哪？Copilot 只能看到你当前正在编辑的那个文件，Cursor 能多看几个文件，而 Claude Code 能看到你的整个项目——所有文件、Git 历史、目录结构——然后自己决定该读哪些、该改哪些、该跑什么命令。

这篇文章带你从安装到第一次高效使用。读完之后你会知道：Claude Code 到底和其他工具差在哪、CLAUDE.md 为什么是你最重要的配置文件、权限模式怎么选。

## Claude Code 和其他工具的本质区别

很多人把 Claude Code 和 Copilot、Cursor 放在一起比较，觉得它们是同类产品。其实不是。它们解决的是完全不同的问题。

Copilot 解决的是"写代码时的卡顿"——你写一半，它帮你补完。它像一个很快的打字员，但只能看到你当前屏幕上的内容。

Cursor 解决的是"编辑器内的 AI 辅助"——它在 Copilot 的基础上增加了多文件上下文和 Chat 功能。它像一个稍微聪明一点的编辑器插件。

Claude Code 解决的是"复杂开发任务的自动化"——你告诉它"重构认证模块支持 OAuth 2.0"，它会自己去读代码、分析依赖、设计方案、修改文件、跑测试。它更像一个能自己动手的初级开发者。

这个区别决定了使用场景。如果你只是想快速补全一个函数，Copilot 或 Cursor 的 inline 补全就够了。但如果你要做跨文件的修改、分析陌生代码库、自动化重复性工作——Claude Code 的优势就体现出来了。

什么时候该用 Claude Code，什么时候用 IDE 插件就够了？一个简单的判断标准：**这个任务需要读 3 个以上的文件吗？需要跑命令吗？需要理解项目整体结构吗？** 如果都是，用 Claude Code。

## 安装与认证

安装很简单，前提是你有 Node.js 22+：

```bash
npm install -g @anthropic-ai/claude-code
```

装完后在终端输入 `claude` 就能启动。第一次启动会引导你认证——你有两个选择：

**用 Anthropic 账号直接登录**（推荐）。如果你有 Claude Max 订阅（$100/月 或 $200/月），直接登录就能用，不需要单独的 API Key，也没有 token 计费。这是目前性价比最高的方式。

**用 API Key 认证**。如果你没有 Max 订阅，可以去 [console.anthropic.com](https://console.anthropic.com) 创建 API Key，按 token 用量付费。这种方式的好处是灵活，坏处是用多了费用可能超过 Max 订阅。

```bash
# API Key 方式认证
export ANTHROPIC_API_KEY="sk-ant-xxx"
```

一个建议：如果你每天使用 Claude Code 超过 2 小时，Max 订阅几乎肯定比按量付费便宜。

## 第一次启动：感受它的不同

装好之后，在你自己的某个项目目录里输入 `claude` 启动：

```bash
cd /path/to/your/project
claude
```

启动后你会看到一个交互式界面。试试这个请求：

```
帮我分析这个项目的整体结构，告诉我用了什么技术栈、代码是怎么组织的
```

注意观察它会做什么：它不会直接给你一个笼统的回答，而是会先用 Glob 工具扫描目录结构、用 Read 工具读 package.json 和关键配置文件、用 Grep 工具搜索特定模式——**它在主动探索你的项目**，而不是等你把信息喂给它。

这就是 Claude Code 和 ChatGPT 网页版的根本区别。在 ChatGPT 里，你得自己把代码复制粘贴进去。在 Claude Code 里，你只需要说"帮我看看这个项目"，它自己会去读。

## CLAUDE.md：你最重要的配置文件

Claude Code 启动时会自动读取项目根目录下的 `CLAUDE.md` 文件。这个文件告诉它"这个项目的基本信息、技术栈、代码规范、特殊约定"。你可以把它理解为**给 AI 的项目说明书**。

没有 CLAUDE.md 会怎样？Claude Code 仍然能工作，但它不知道你的项目用什么框架、代码风格是什么、有哪些特殊约定。每次对话你都得重复这些信息——"我们用的是 NestJS"、"错误处理统一用 RpcException"、"文件命名用 kebab-case"。

有了 CLAUDE.md，你只需要写一次，以后每次对话它都会自动遵循。

一个实用的 CLAUDE.md 长什么样？这里有一个简洁但有效的例子：

```markdown
# TaskFlow API

## 技术栈

- NestJS 10 + TypeORM + PostgreSQL
- TypeScript 5.x，strict 模式
- 测试框架：Jest

## 代码规范

- 文件命名：kebab-case（task.service.ts）
- DTO 用 class 而不是 interface
- 所有 API 返回统一格式：{ code, message, data }
- 错误处理统一用自定义 RpcException

## 项目结构

src/
├── modules/ # 业务模块（task、user、notification...）
├── common/ # 公共工具（过滤器、守卫、拦截器）
├── config/ # 配置模块
└── main.ts # 入口文件

## 注意事项

- 数据库迁移只用 TypeORM migration，不用 sync
- 敏感配置走环境变量，不硬编码
```

几个关键点：**写它真正需要知道的，不写废话**。它不需要知道"这个项目很棒"或者"请认真工作"。它需要知道的是技术栈、代码规范、项目结构、特殊约定——这些会影响它生成代码质量的信息。

CLAUDE.md 支持分层配置，优先级从高到低：

| 层级           | 文件位置                               | 作用范围               |
| -------------- | -------------------------------------- | ---------------------- |
| managed policy | 组织管理员设置                         | 整个组织，不可覆盖     |
| user           | `~/.claude/CLAUDE.md`                  | 你所有的项目           |
| project        | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 当前项目，团队共享     |
| local          | `./CLAUDE.local.md`                    | 当前项目，不提交到 Git |

project 层级的 CLAUDE.md 应该提交到版本控制，让团队成员共享。local 层级放你个人的偏好（比如"我喜欢用 pnpm 而不是 npm"），不提交。

还可以用 `.claude/rules/` 目录放路径匹配的规则文件——比如 `src/modules/task/rule.md` 只在操作 task 模块文件时生效。这在大型项目中很有用，避免一个巨大的 CLAUDE.md。

## 权限模型：6 种模式怎么选

Claude Code 默认不会随便执行可能有危险的操作。每次它要跑命令、写文件，都会先问你一声。这是 `default` 权限模式的行为。

但每次都确认很烦，尤其是你信任的项目里。Claude Code 提供了 6 种权限模式：

**default**：大部分操作自动允许，但写文件、跑 shell 命令时会问你。适合日常开发。

**plan**：只读模式，不会执行任何修改操作。适合代码审查、学习探索。

**acceptEdits**：自动允许文件编辑，但跑命令还是会问你。适合频繁改代码但不常跑命令的场景。

**auto**：自动允许大部分操作，包括命令执行。但危险操作（rm -rf、git push --force）还是会确认。适合个人项目快速开发。

**dontAsk**：从不提示确认。设计给 CI/CD 场景用的，你平时不会用到。

**bypassPermissions**：跳过所有权限检查。慎用。

大多数情况下，`default` 就够了。如果你觉得确认太多影响效率，可以切换到 `acceptEdits`。千万不要直接用 `bypassPermissions`——除非你在跑自动化脚本。

权限模式可以在 settings.json 中设置，也可以通过 `/permissions` 命令交互式配置。还可以设置 allow/deny 规则，比如"自动允许 git status 和 git diff，但 rm 命令必须确认"。

## 基本工作流体验

现在来体验一下 Claude Code 的完整工作流。我们用一个非常简单的场景：在一个空目录里搭建一个 Express API 项目。

创建一个空目录，启动 Claude Code，然后说：

```
帮我用 Express + TypeScript 搭建一个任务管理 API 的基础骨架。
要求：
- POST /tasks 创建任务（title、description、priority 字段）
- GET /tasks 查询任务列表，支持 ?priority=high 筛选
- 用内存数组存储，先不接数据库
- 代码风格遵循标准 TypeScript 规范
```

观察它的执行过程：它会先创建 package.json，再安装依赖，然后创建项目结构、写路由和控制器代码、配置 TypeScript。整个过程你可能只需要确认几次权限，其他都是自动完成的。

搭完之后你可以继续让它改进：

```
给任务加一个 status 字段，支持 pending/in_progress/done 三种状态。
加一个 PATCH /tasks/:id/status 接口来更新状态。
状态从 pending 变成 in_progress 时，自动设置 startedAt 时间戳。
```

注意这次它怎么工作的：它先读取了之前创建的代码，理解了现有的结构，然后在现有代码基础上修改——而不是从头重写。这就是 Claude Code 的"上下文连续性"，它记得自己之前做过什么。

## 一个常见的问题

很多人第一次用 Claude Code 会问："我该怎么告诉它要做什么？"

答案比你想的简单：**像和一个靠谱的同事说话一样**。你不需要写结构化的 Prompt，不需要列出步骤。用自然语言描述你的需求，补充必要的约束，它就能理解。

比如：

```
# 不需要这样（太结构化了）
任务：修改用户注册接口
步骤：
1. 打开 src/modules/user/user.service.ts
2. 找到 register 方法
3. 添加邮箱格式校验
4. ...

# 这样就好（自然语言 + 约束）
用户注册接口缺邮箱格式校验，加上。
用 class-validator 的 isEmail 装饰器，校验失败返回 400。
参考 src/modules/user/dto/create-user.dto.ts 里现有的校验风格。
```

关键在于：告诉它**做什么**和**有什么约束**，不要告诉它**怎么做**。它比你更知道代码应该怎么组织。

## 什么时候 Claude Code 不是最佳选择

说了这么多好处，也得说说它不擅长什么。

**简单的一行修改**：你只需要改一个变量名，自己动手比启动 Claude Code、描述需求、等它执行要快得多。

**需要精确控制每一行代码的场景**：如果你已经完全想好了每一步该怎么做，直接写可能比让 AI 猜你的意图更快。

**实时性要求极高的场景**：Claude Code 需要几秒到几十秒的响应时间，不适合需要即时反馈的场景（比如调试时的快速试错）。

一个实用的经验法则：**如果这个任务你自己做不超过 5 分钟，直接做。超过 5 分钟的，让 Claude Code 帮你**。

Claude Code 最大的价值不是帮你省下 5 分钟的小事，而是帮你处理那些"自己做要半小时以上"的任务——分析陌生代码库、跨文件重构、批量生成测试、搭建项目骨架。这些才是它真正发光的地方。
