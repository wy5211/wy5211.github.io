上一篇我们通过 MCP 让 Claude Code 能直接查询 TaskFlow 的数据库。这篇文章回到代码层面，聊一个更实际的问题：**代码质量**。

TaskFlow 有一个出了名的"问题模块"——NotificationService，400 多行代码，混合了邮件、推送、站内信、Slack、Webhook 五种通知渠道的逻辑。测试覆盖率 22%。每次改这里面的代码都像拆炸弹——改了 A 渠道，B 渠道莫名其妙坏了。

用 Claude Code 来处理这类问题，效率确实很高，但有一个前提：**你得用对方法**。直接说"帮我重构这个文件"然后盲目信任它的输出，是灾难的开始。

## 用 Claude Code 做安全重构

重构的难点不在于"怎么改"，而在于"改了之后不能出 bug"。Claude Code 擅长生成代码，但它不知道你的业务规则的全部边界情况。所以安全重构的核心是：**让它理解现有逻辑，小步修改，每步验证**。

### 第一步：用 Plan Mode 分析

别直接让它动手。先用 Plan Mode 让它分析现有代码：

```
进入 Plan Mode。

分析 @src/modules/notification/notification.service.ts，
告诉我：
1. 这个文件有哪些方法，每个方法大概做什么
2. 五种通知渠道的逻辑是怎么组织的
3. 有没有明显的代码坏味道（重复逻辑、过长方法、职责混乱）
4. 哪些逻辑是各渠道独立的，哪些是共享的
先不改代码。
```

Claude Code 会读代码、分析结构，然后给你一份报告。关键信息包括：哪些逻辑可以安全提取、哪些逻辑有隐式依赖（比如邮件模板和站内信模板共享某个格式化函数）、哪些地方有潜在的 bug。

### 第二步：确认重构方案

看完分析报告后，你和 Claude Code 讨论出重构方案。比如：

```
根据你的分析，我决定按渠道拆分。
但有几个约束：
1. 先只拆邮件和站内信，其他三个渠道不动
2. 拆分后原来的 NotificationService 变成调度器
3. 共享逻辑（模板格式化、发送队列）提取到公共模块
4. 每拆一个渠道，先补测试再改代码
```

### 第三步：小步执行，每步验证

确认方案后退出 Plan Mode，让它分步执行。关键是**不要让它一口气改完所有文件**——分步来，每步都能验证。

```
先做第一步：为邮件通知逻辑补测试。
只针对 sendEmailNotification 方法写测试，
覆盖正常发送、发送失败、模板渲染失败三个场景。
```

测试写完之后，跑一遍确认通过。然后下一步：

```
测试通过了。现在把邮件通知逻辑提取到 EmailNotificationService。
基于刚才写的测试来验证提取后功能不变。
```

每一步都基于上一步的测试来验证。这种方式下，即使 Claude Code 某步改错了，测试会立刻告诉你，你可以在错误扩散之前修复。

### 重构的节奏

一次改多少合适？这取决于模块的复杂度和你的测试覆盖。

**测试覆盖率高的模块**（>60%）：可以一次改大一点。测试是你的安全网，改错了测试会报错。

**测试覆盖率低的模块**（不足 30%）：像 TaskFlow 的 NotificationService，必须先补测试再改代码。每改一点补一点测试，不要大刀阔斧。

**业务逻辑复杂的模块**：小步前进。一次只改一个方法，验证通过再改下一个。复杂业务逻辑是 AI 最容易出错的地方。

一个经验法则：**每次让 Claude Code 改动的文件数不超过 3 个**。超过 3 个，review 起来很困难，出了问题也不好定位。

## 测试生成：不是"跑一下就行"

Claude Code 生成测试的能力很强，但生成的测试质量参差不齐。问题通常出在：它生成的测试能通过，但测的不是真正的业务逻辑。

### 常见的测试质量问题

**只测 happy path**。"用户注册成功"的场景测了，但"邮箱已被注册"、"密码格式不合法"、"数据库连接失败"这些异常场景没测。

**测试和实现耦合太紧**。测试直接检查返回值是不是某个固定字符串——你重构了实现（没改行为），测试就挂了。好的测试应该检查行为，不检查实现细节。

**用假数据**。测试数据是 `{ id: 1, name: 'test' }` 这种没有业务含义的东西，无法覆盖真实的边界情况。

### 怎么让它生成有质量的测试

关键在于你在 Prompt 中提供的上下文：

```
为 TaskService.create 方法生成测试。
这个方法的业务规则：
- title 不能为空且不超过 200 字符
- priority 必须是 low/medium/high 之一
- 如果 assigneeId 指定的用户不存在，抛 NOT_FOUND 异常
- 创建成功后自动设置 status 为 pending、createdAt 为当前时间
- 同一个项目中 title 不能重复（区分大小写）

覆盖正常场景和所有异常场景。
用 TaskFlow 项目中真实的测试风格（参考 @src/modules/user/user.service.spec.ts）。
```

注意这个 Prompt 的特点：你告诉了它**具体的业务规则**和**异常场景**，而不是泛泛地说"帮我写测试"。它知道了业务规则，才能生成覆盖边界情况的测试。

### 测试生成的正确流程

1. 让它先读代码，理解业务逻辑
2. 你补充它可能不知道的业务规则
3. 让它列出测试场景清单（先不写代码）
4. 你审阅清单，补充遗漏的场景
5. 让它按清单生成测试代码
6. 跑测试，确认通过

这个流程比直接说"帮我写测试"多花 2 分钟，但测试质量会好很多。

## 调试问题排查

Claude Code 在调试方面能帮很大的忙，但前提是你得正确地描述问题。

### 有效的问题描述

差的描述：

```
任务列表接口报错了，帮我看看
```

它不知道报什么错、在什么情况下报错、错误信息是什么。只能猜。

好的描述：

```
GET /tasks?project=proj_billing&priority=high 接口返回 500。
错误日志：
Error: column "proj_billing" does not exist
  at Query.format (node_modules/pg/lib/query.js:102:16)
  at TaskService.findAll (src/modules/task/task.service.ts:47:22)

看起来是 project 参数被当成了 SQL 值而不是参数化查询。
相关代码在 @src/modules/task/task.service.ts 第 40-55 行。
```

这个描述包含了四个关键信息：**复现步骤**（哪个接口、什么参数）、**错误信息**（完整的堆栈）、**你的初步分析**（看起来是 SQL 注入问题）、**相关代码位置**。

Claude Code 拿到这些信息后，可以直接定位到 `task.service.ts` 第 47 行，发现问题——确实是参数化查询写错了。修复方案很直接。

### 让它分析日志

如果你的应用有日志文件，可以直接让 Claude Code 读：

```
应用日志在 logs/app.log。
帮我分析最近 100 行错误日志，看看有没有模式——
是同一个接口反复出错，还是分散在不同接口？
```

它能从大量日志中识别出模式——比如"90% 的错误都来自 Webhook 投递超时"，或者"每天下午 2-3 点错误率明显升高"。这些模式你自己翻日志可能要半小时才能发现。

### 验证修复方案

AI 给的修复方案不一定对。特别是涉及业务逻辑的 bug——它可能修了表面症状，但没解决根本原因。

一个必须养成的习惯：**让它给出修复方案后，不要直接应用，先问它"你怎么确认这个修复是正确的"**。

```
你建议把 query builder 的 where 条件从字符串拼接改成参数化查询。
这个修复怎么验证？
```

它应该能告诉你：可以构造一个包含特殊字符的 project name 来测试，修复前会报 SQL 错误，修复后正常返回。你按照它的建议测试一遍，确认修复有效再提交。

## AI 生成的代码必须 review

这篇文章反复强调一个原则，值得再重复一遍：**AI 生成的代码和同事写的代码一样，都需要 review**。

甚至应该更严格。因为同事写的代码如果有问题，review 的时候你能看出来——它毕竟是人写的，逻辑通常你能理解。AI 生成的代码有时候"看起来很对"——语法没问题、类型没问题、测试也通过了——但可能有微妙的逻辑错误。比如边界条件处理不对、并发场景下有竞态条件、错误处理遗漏了某种情况。

几个必须检查的点：

- **业务逻辑是否正确**：AI 不知道你的业务规则，它可能"合理但不符合业务"地实现了功能
- **错误处理是否完整**：它可能处理了正常路径和主要异常，但遗漏了边界情况
- **是否有安全问题**：SQL 注入、XSS、敏感信息泄露——AI 不一定知道你的安全要求
- **是否和现有代码风格一致**：命名、注释、组织方式是否和项目其他部分匹配

如果你发现 Claude Code 经常在某类问题上出错，把它记录到 Memory 或 CLAUDE.md 里——"TypeORM 查询必须用参数化，不允许字符串拼接"。这样它下次会自动遵循。

Claude Code 是一个很强的"第一版生成器"——它能在几分钟内生成一个可工作的初版。但"可工作"和"生产就绪"之间还有一段距离，这段距离需要你自己来走。
