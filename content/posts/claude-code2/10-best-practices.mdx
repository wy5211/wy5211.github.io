前面 9 篇文章，我们拆着讲了 Claude Code 的各个能力：怎么写 Prompt、怎么用 Plan Mode、怎么让 Agent 分工、怎么自动化工作流、怎么管理上下文、怎么接 MCP、怎么保证代码质量、怎么融入 Git 工作流。

但这可能会给你一种错觉：以为日常开发中你会频繁切换这些功能。实际情况不是这样的。大多数时候你只需要用其中两三个功能，而且用得很自然——就像你开车不需要每次都想着"现在该踩离合了"。

这篇文章用两个部分来收尾：先用一个模拟的开发日展示真实的工作节奏，再聊聊 Claude Code 的能力边界——知道它不擅长什么，和知道它擅长什么一样重要。

## 一个真实的开发日

假设你在 TaskFlow 团队，这是一个典型的工作日。

### 早上：Review PR

打开电脑，先看看昨天同事提交的 PR。

```
审查 PR #156 的变更，重点检查类型安全和错误处理。
```

Claude Code 读取 PR diff，按你指定的维度检查，输出问题清单。你花 5 分钟扫一遍，发现两个问题：一个类型断言不安全，一个 catch 块吞掉了异常。在 PR 里评论指出，让同事修改。

这个场景没有用什么高级功能——就是一次普通的对话请求。Claude Code 擅长这种"模式化检查"，帮你快速过滤掉低级问题，让你把精力放在更有价值的 review 上。

### 上午：修一个 bug

收到一个用户反馈：任务列表按标签筛选时，有些标签下的任务漏掉了。你让 Claude Code 先分析原因。

```
进入 Plan Mode。
分析任务列表的标签筛选逻辑，@src/modules/task/task.service.ts。
用户反馈：选择 "bug" 标签筛选时，结果比预期少。
先不改代码，告诉我可能的原因。
```

它读了代码后发现：标签筛选用的是 `WHERE tags @> ARRAY['bug']`（PostgreSQL 数组包含操作符），但有些任务的 tags 字段存的是 `"Bug"` 而不是 `"bug"`——大小写不一致。

退出 Plan Mode，让它修复：

```
按你的分析修复。把标签匹配改成大小写不敏感。
同时检查一下数据里有多少标签存在大小写不一致的情况。
```

它修复了代码，顺便通过数据库查询发现 12 个标签存在大小写不一致。你决定同时做一次数据清洗。

这个场景用了 Plan Mode（先分析再动手）和 MCP（查询数据库）。Plan Mode 在这里的价值是：如果直接说"修复标签筛选"，它可能会改代码但不会发现数据不一致的问题。先分析发现了根因（大小写不一致），修复就更有针对性。

### 下午：重构 NotificationService

这是一个大活。NotificationService 有 400 行，你想把它按渠道拆分。但这个重构影响面大，需要小心处理。

```
进入 Plan Mode。
分析 @src/modules/notification/notification.service.ts，
设计一个按渠道拆分的重构方案。
约束：先只拆邮件和站内信，其他不动。
先补测试再改代码。
```

Plan Mode 分析完后给出了拆分方案。你审阅了方案，做了一些调整，确认后退出 Plan Mode。

然后分步执行：

```
按方案第一步：为 sendEmailNotification 补测试。
覆盖正常发送、模板渲染失败、SMTP 连接超时三个场景。
```

测试写完，跑通过。继续：

```
测试通过了。提取 EmailNotificationService。
保持 sendEmailNotification 的行为不变。
```

改完跑测试，通过。继续站内信的拆分。整个过程用了将近 2 小时，但你每一步都有测试验证，心里很踏实。

这个场景综合用了 Plan Mode + 分步执行 + 测试驱动。大重构就该这么做——急不得。

### 下班前：提交代码

一天的改动包括：bug 修复（标签筛选）、数据清洗、NotificationService 部分重构。你想分成两个 commit：bug 修复一个，重构一个。

```
帮我分两次提交。第一次提交标签筛选的修复（task.service.ts 和 migration）。
第二次提交 NotificationService 的重构。
```

Claude Code 分析 diff，分别生成两个 commit message，你确认后提交。最后创建 PR。

### 回顾：什么任务用什么策略

回顾这一天，不同类型的任务用了不同的策略：

| 任务类型  | 用的策略             | 为什么                         |
| --------- | -------------------- | ------------------------------ |
| PR review | 直接对话             | 范围明确，不需要规划           |
| 修 bug    | Plan Mode + MCP      | 先分析根因再修复，需要查数据库 |
| 大重构    | Plan Mode + 分步执行 | 方向不能错，需要逐步验证       |
| 提交代码  | /commit              | 自动化，省时间                 |

你不会每天用 Plan Mode，也不会每次都启动 Agent。但遇到对的任务时，你知道该用什么策略。

## Claude Code 的能力边界

用了一个月之后，你可能会有一种错觉：Claude Code 无所不能。是时候泼盆冷水了。

### 它擅长的

**代码生成和修改**。这是它最强的能力。给它明确的需求和足够的上下文，它能生成质量不错的代码。尤其是有明确模式的工作——CRUD 操作、DTO 定义、测试用例、数据迁移脚本。

**代码分析和理解**。它能快速阅读大量代码，找出模式、发现不一致、总结结构。用 Explore Agent 分析一个陌生的代码库，比你自己一个个文件翻快得多。

**重复性工作的自动化**。生成 commit message、格式化代码、跑 lint 和测试、按固定模板做 code review——这些事它不厌其烦，而且做得比你细心。

### 它不擅长的

**架构决策**。它可以说出"微服务和单体架构的优劣"，但要不要拆微服务——这个决策需要考虑团队规模、业务阶段、运维能力、成本预算，这些上下文它通常不具备。架构决策需要人来拍板，Claude Code 可以帮你分析利弊，但不能替你决策。

**业务逻辑设计**。它不知道你的业务规则。比如"任务从 pending 变成 done 的时候，是否需要通知项目的所有关注者"、"Webhook 投递失败超过 3 次之后该怎么处理"——这些需要你和产品经理确认，不是 AI 能猜的。

**跨系统的集成调试**。当 bug 涉及多个系统的交互（前端 + 后端 + 第三方 API + 数据库），Claude Code 能帮你分析单个系统的代码，但它无法运行整个系统来复现问题。这类调试还是需要你自己的经验和直觉。

**评估代码的长期影响**。它能告诉你"这段代码现在能跑"，但很难评估"半年后这段代码会不会成为技术债"。长期影响需要人对项目 Roadmap 的理解。

## 三个最危险的误区

### 误区一：不验证就提交

Claude Code 生成了代码，看起来没问题，你直接提交了。等到 code review 的时候才发现——类型断言不安全、错误处理有遗漏、边界情况没覆盖。

这种事发生的频率比你想象的高。因为 AI 生成的代码有一个特点：**它"看起来很对"**。语法正确、结构清晰、命名规范——这些表面特征让你放松警惕，忽略了微妙的逻辑错误。

原则：**AI 生成的代码和同事写的代码一样，都需要 review 和测试**。甚至应该更严格。

### 误区二：Prompt 写太细，变成"人工编译器"

有些人在 Prompt 里写了 20 行指令，精确到每一行代码该怎么写。这不叫"用 AI"，这叫"用自然语言写代码"。如果你已经知道每一步该怎么做，直接自己写可能更快。

原则：**告诉它"做什么"和"有什么约束"，不要告诉它"怎么做"**。

### 误区三：过度依赖，失去独立思考能力

用了一段时间 Claude Code 之后，你可能会发现自己越来越不愿意自己写代码了。简单的方法也想让它生成，能自己想出来的 bug 也让它帮忙查。

这个趋势很危险。工具应该增强你的能力，而不是替代你的思考。如果你发现自己写代码的速度变慢了、对项目的理解变浅了——是时候刻意减少依赖了。

一个建议：每周至少有一天完全不用 Claude Code。强迫自己手动写代码、手动 debug，保持肌肉记忆。

## 效率的真实差距

聊了这么多理论和技巧，最后说点实际的。Claude Code 到底能帮你省多少时间？

坦率地说，这取决于你怎么用它。如果你只是用它来"帮我写一个函数"，那它就是一个稍快一点的代码补全工具，可能省 10-20% 的时间。

如果你按这个系列讲的方式用——Plan Mode 规划复杂任务、Agent 探索陌生代码库、Skill 封装重复流程、Memory 跨会话保留上下文——它能帮你省 40-60% 的"重复劳动时间"。注意是重复劳动，不是思考时间。设计架构、理解业务需求、做技术决策——这些事 AI 帮不了你，也不应该帮你做。

最真实的效率提升往往不是来自某个炫酷的功能，而是来自**微习惯的积累**。比如：

- 每次改完代码让它跑一下 lint 和测试，省掉了"部署后发现 lint 报错"的时间
- 每次提交让它生成 commit message，省掉了"想半天措辞"的时间
- 每次接手新模块让它先用 Explore Agent 分析结构，省掉了"翻半天代码搞不懂"的时间
- CLAUDE.md 里写好代码规范，省掉了每次对话都要重复"我们用 RpcException"的时间

这些微习惯单个看起来不值一提，但一天积累下来，可能就是 1-2 个小时。

## 系列回顾

花了 10 篇文章聊 Claude Code，用一个知识地图来收尾。

```
基础篇（01-03）—— 建立直觉
  01 快速入门：安装配置、CLAUDE.md、权限模型
  02 高效对话：Prompt 技巧、@引用、沟通陷阱
  03 Plan Mode：复杂任务先规划再执行

核心篇（04-06）—— 深入进阶
  04 Agent System：三种内置 Agent、自定义 Agent、worktree 隔离
  05 Skills 与 Hooks：自动化工作流、25+ Hook 事件
  06 上下文与 Memory：CLAUDE.md 分层、rules 目录、Memory 类型

进阶篇（07-09）—— 工作流集成
  07 MCP Servers：外部工具接入、数据库查询、安全考量
  08 代码质量：安全重构、测试生成、调试排查
  09 Git 工作流：/commit、/review-pr、团队规范

实战篇（10）—— 综合运用
  10 最佳实践：日常工作流、能力边界、常见误区
```

工具的价值不在于你学了多少功能，而在于你形成了稳定的工作流。学了 10 个功能但每天只用其中 2 个，比学了 2 个功能但每天都在用要好得多。找到你自己的"高频功能组合"，把它变成习惯——这就是从"会用"到"玩得溜"的距离。
