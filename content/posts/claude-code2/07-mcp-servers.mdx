上一篇我们解决了"内部上下文管理"的问题——怎么让 Claude Code 在长时间对话中不迷路。但还有一个维度的问题没解决：**外部信息**。

Claude Code 能读写你项目里的文件、能跑终端命令、能搜索代码。但它不知道你的 PostgreSQL 数据库里存了什么数据，不知道 Jira 上有哪些 open 的 ticket，不知道 Grafana 上的监控指标，不知道内部 API 文档长什么样。

它被关在你的项目目录里，看不到外面的世界。

MCP（Model Context Protocol）就是打开这扇门的钥匙。它是一个标准化协议，让 Claude Code 能连接外部工具和数据源——数据库、API、文档系统、监控平台——然后在对话中直接使用这些外部能力。

## MCP 是什么

MCP 的全称是 Model Context Protocol，翻译过来就是"模型上下文协议"。它做的事情很简单：**定义一个标准接口，让 AI 助手能调用外部工具**。

没有 MCP 的时候，如果你想问 Claude Code"数据库里有多少条未完成的任务"，你得先自己在终端里跑 `SELECT COUNT(*) FROM tasks WHERE status != 'done';`，然后把结果粘贴给它。

有了 MCP 之后，你可以直接说"查一下数据库里有多少条未完成的任务"。Claude Code 会通过 MCP 调用数据库工具，自己查询，自己返回结果。

这看起来只是省了一步粘贴操作，但实际上价值远不止于此。当 Claude Code 能直接访问数据库时，它可以做的事情发生了质变：

- "查一下最近一周创建的任务，按项目分组统计数量"
- "找出所有 assignee 为空的任务，告诉我哪些项目缺人"
- "对比一下任务表和用户表的关联关系，看有没有孤立的数据"

这些操作需要理解数据库 schema、构造查询语句、分析查询结果。如果每次都由你来执行 SQL 再粘贴结果，效率极低。有了 MCP，Claude Code 可以自主完成整个流程。

## MCP Server 的类型

MCP Server 是提供外部能力的程序，它通过标准化的协议和 Claude Code 通信。根据通信方式的不同，有四种类型：

**stdio**：最常用的类型。Claude Code 启动 Server 进程，通过标准输入输出通信。适合本地工具——数据库客户端、文件系统操作、本地 API。

**HTTP**：通过 HTTP 请求通信。适合远程服务——内部 API、云服务、第三方平台。

**SSE（Server-Sent Events）**：基于 HTTP 的双向通信。适合需要服务端主动推送消息的场景。

**WebSocket**：全双工通信。适合需要实时双向通信的场景。

大多数情况下你用 stdio 就够了——它是本地运行的工具，配置简单，延迟低。

## 配置 MCP Server

### 用命令行添加

最简单的方式是用 `claude mcp add` 命令：

```bash
# 添加一个 PostgreSQL MCP Server
claude mcp add postgres -- npx @anthropic/mcp-postgres "postgresql://user:pass@localhost:5432/taskflow"

# 添加一个文件系统 MCP Server
claude mcp add filesystem -- npx @anthropic/mcp-filesystem /path/to/docs

# 查看已添加的 MCP Server
claude mcp list
```

### 用 settings.json 配置

更灵活的方式是在 `settings.json` 中配置。支持项目级别（`.claude/settings.json`）和用户级别（`~/.claude/settings.json`）：

```json
{
  "mcpServers": {
    "taskflow-db": {
      "command": "npx",
      "args": [
        "@anthropic/mcp-postgres",
        "postgresql://user:pass@localhost:5432/taskflow"
      ]
    },
    "internal-docs": {
      "command": "npx",
      "args": ["@anthropic/mcp-filesystem", "/path/to/internal/docs"]
    },
    "jira": {
      "type": "http",
      "url": "https://jira.example.com/mcp"
    }
  }
}
```

### Per-Agent 配置

MCP Server 还可以配置给特定的 Agent 使用。在自定义 Agent 的定义文件中：

```markdown
---
name: db-analyst
description: 分析数据库数据
mcpServers:
  - taskflow-db
tools:
  - Read
  - Grep
---
```

这样 db-analyst Agent 就能直接查询数据库，而其他 Agent 不能。这是一个有用的安全隔离——不是所有 Agent 都需要访问数据库。

## 工具命名规则

MCP Server 提供的工具会以 `mcp__<server>__<tool>` 的格式命名。比如 PostgreSQL MCP Server 提供的 `query` 工具会被命名为 `mcp__taskflow-db__query`。

你不需要记住这些名字。当 Claude Code 需要查询数据库时，它会自动选择正确的工具。但如果你在对话中想明确指定用某个工具，可以这样：

```
用 mcp__taskflow-db__query 查询任务表中状态为 in_progress 的任务数量
```

## 实战：为 TaskFlow 配置数据库 MCP

让我们把 TaskFlow 的 PostgreSQL 数据库接入 Claude Code。

### 第一步：安装 PostgreSQL MCP Server

```bash
npm install -g @anthropic/mcp-postgres
```

### 第二步：配置连接

在 TaskFlow 项目的 `.claude/settings.json` 中添加：

```json
{
  "mcpServers": {
    "taskflow-db": {
      "command": "npx",
      "args": [
        "@anthropic/mcp-postgres",
        "postgresql://taskflow_user:your_password@localhost:5432/taskflow_dev"
      ]
    }
  }
}
```

注意：数据库连接字符串包含密码，不应该提交到版本控制。更好的做法是把连接字符串放在环境变量里：

```json
{
  "mcpServers": {
    "taskflow-db": {
      "command": "npx",
      "args": ["@anthropic/mcp-postgres", "${TASKFLOW_DATABASE_URL}"]
    }
  }
}
```

然后在 `.claude/settings.local.json`（不提交到 Git）中设置环境变量。

### 第三步：使用

配置好后，重启 Claude Code，你就可以直接在对话中查询数据库了：

```
查一下 taskflow-db 里有多少条任务，按状态分组统计
```

Claude Code 会调用 MCP 工具查询数据库，返回类似这样的结果：

```
状态分布：
- pending: 342 条
- in_progress: 128 条
- done: 1,847 条
- cancelled: 56 条

总计：2,373 条任务
```

你现在可以基于真实数据做更深入的分析：

```
找出所有 in_progress 状态但超过 7 天没有更新的任务，
按项目分组，告诉我哪些项目可能有阻塞问题。
```

这种查询结合了数据库访问和业务分析能力——Claude Code 不只是执行 SQL，它还能理解结果背后的业务含义。

## VS Code 中的内置 MCP Server

如果你用 VS Code 的 Claude Code 扩展，有一个内置的 MCP Server：`ide`。

它提供了几个实用的工具：

- 读取 VS Code 的诊断信息（错误、警告）
- 执行 Jupyter notebook 单元格
- 获取当前打开的文件信息

这意味着在 VS Code 中，Claude Code 可以直接看到编辑器中的错误和警告：

```
当前项目有哪些 TypeScript 错误？
帮我分析一下原因并修复
```

它会调用 `ide` MCP Server 获取诊断信息，然后分析错误原因，再修复代码。整个过程不需要你手动复制错误信息。

## MCP 的安全考量

把外部工具接入 Claude Code 带来了便利，也带来了风险。

**权限控制**：不是所有 MCP 工具都应该允许写操作。数据库 MCP Server 如果支持写操作（INSERT、UPDATE、DELETE），Claude Code 可能会在不经意间修改数据。尽量给 MCP Server 配置只读权限，写操作由你手动执行。

```json
{
  "mcpServers": {
    "taskflow-db": {
      "command": "npx",
      "args": [
        "@anthropic/mcp-postgres",
        "${TASKFLOW_DATABASE_URL}",
        "--readonly"
      ]
    }
  }
}
```

**敏感信息**：MCP Server 的配置中可能包含密码、Token 等敏感信息。这些信息不要提交到版本控制。用环境变量或 local 级别的配置文件来存储。

**网络暴露**：HTTP 类型的 MCP Server 意味着 Claude Code 会向外部服务发送请求。确保这些服务的认证和授权机制是可靠的。

**工具审计**：定期检查已配置的 MCP Server，移除不再使用的。每个 MCP Server 都是一个潜在的攻击面。

## MCP 的适用边界

MCP 很强大，但不是什么都需要接。

**值得接的**：

- 数据库（用于数据分析和问题排查）
- 内部文档系统（让 Claude Code 查阅 API 文档、设计文档）
- 监控系统（让它看 Grafana 上的错误率、延迟数据）
- Issue 跟踪系统（让它了解当前有哪些 bug 和需求）

**不需要接的**：

- 你已经能通过项目文件获取的信息
- 使用频率很低的外部系统
- 安全要求极高、不适合 AI 访问的系统

一个简单的判断：**如果你每周至少需要在对话中引用 3 次某个外部系统的数据，就值得通过 MCP 接入**。偶尔用一次的，手动查询粘贴结果就行了。

MCP 的核心价值不是让你少粘贴几次结果，而是让 Claude Code 能**自主地、连续地**使用外部信息来完成复杂任务——比如"分析数据库中的任务数据，找出异常模式，然后去代码里找到对应的 bug"。这种需要外部信息和代码分析结合的任务，手动操作效率极低，有了 MCP 才真正可行。
