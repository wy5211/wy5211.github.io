上一篇你装好了 Claude Code，跑通了基本命令。但你很快会碰到一个让人沮丧的事：你说"帮我改一下这个接口"，它改了，但改的不是你想要的地方。你说"不对，不是那个意思"，它又改了，还是不对。来回十轮，你花了 20 分钟在"纠正它的理解"上，还不如自己直接改。

这不是 Claude Code 的问题。问题在于你和它"说话"的方式。

Claude Code 不是搜索引擎——你丢一个关键词过去，它不会"理解你的意图"。它也不是人类同事——你不能指望它通过"暗示"或"上下文"猜出你没说出来的需求。它是一个很聪明的工具，但你需要用它能理解的方式沟通。

这篇文章聊一件事：怎么描述需求，让 Claude Code **第一次**就做对。

## 先搞清楚它怎么理解你的项目

在聊怎么写 Prompt 之前，得先理解 Claude Code 的上下文机制。它和网页版 Claude 有本质区别。

网页版 Claude：你每发一条消息，它只看到你手动粘贴的内容。对话结束，什么都忘了。

Claude Code：它在你的项目目录里运行，能看到你的整个项目。每次收到你的指令时，它会从这些来源获取信息：

- **CLAUDE.md**：项目级指令——技术栈、代码规范、特殊约定
- **Memory**：跨会话的持久记忆——比如"上次我们讨论过把认证从 JWT 换成 Session"
- **对话历史**：你在这次会话中说过的所有话、它做过的所有操作
- **@ 引用的文件**：你明确指出"看这个文件"的内容
- **工具读取的文件**：它用 Read、Grep、Glob 自己读到的代码

理解了这个机制，一个关键原则就清晰了：**不要重复它已经知道的信息，但一定要补充它不知道的上下文**。

比如你的 CLAUDE.md 里已经写了"错误处理统一用 RpcException"，那你在对话中就不需要再说"记得用 RpcException"。但如果你要改的模块有一个特殊的约定（比如"支付模块的错误码用 3xxx 开头"），而 CLAUDE.md 里没写——你就得在 Prompt 里补充。

## 有效 Prompt 的核心：三件事

一个有效的 Prompt 不需要长篇大论。你只需要说清楚三件事：**做什么、有什么约束、看哪里**。

"做什么"是目标——你希望它完成什么。"有什么约束"是边界——不能用什么方式、必须符合什么规范。"看哪里"是上下文——相关的文件、代码、文档。

用一个真实场景来说明。假设你要给任务列表接口加筛选功能。

### 只说了"做什么"

```
帮我加个筛选功能
```

三个字，信息量为零。加在哪里？筛选什么字段？筛选条件怎么组合？返回格式变不变？Claude Code 只能猜，猜错了你就得再来一轮。

### 说清楚了"做什么"和"看哪里"

```
给任务列表接口加筛选功能。
参考 src/modules/task/task.service.ts 里的 findAll 方法。
```

好一些了，至少它知道改哪个文件。但"筛选功能"还是很模糊——按什么筛？支持多条件组合吗？分页怎么办？

### 三件事都说了

```
给任务列表接口（GET /tasks）加上筛选和排序功能。
筛选条件：status（精确匹配）、priority（精确匹配）、
          assigneeId（精确匹配）、createdAt（范围查询，支持 from/to）。
排序：支持按 createdAt、updatedAt、priority 排序，默认 createdAt desc。
多个条件取交集。
参考 src/modules/task/task.service.ts 里的 findAll 方法，
以及 src/modules/task/dto/query-task.dto.ts 里现有的 DTO 定义。
保持和现有代码一致的风格。
```

这个 Prompt 清楚地告诉了 Claude Code：做什么（加筛选排序）、约束是什么（交集逻辑、默认排序、代码风格一致）、看哪里（service 和 DTO 文件）。它第一次就能给出正确的实现。

## @ 引用：精准投喂上下文

@ 引用是 Claude Code 提供的上下文标注语法。你可以用 `@` 加文件路径或目录路径，让它优先关注这些文件。

为什么要用 @ 引用？因为 Claude Code 虽然能看到整个项目，但在 100+ 文件的项目里，它不一定会自动找到你关心的那个文件。你明确指出来，它就不需要浪费时间搜索。

```bash
# 引用具体文件
参考 @src/modules/task/task.service.ts 里的错误处理方式

# 引用目录
分析 @src/modules/notification/ 下的所有文件，总结通知模块的架构

# 引用搜索结果
我搜了一下，@grep "RpcException" src/ 告诉我哪些地方用了这个异常
```

@ 引用不是必须的。如果你的项目很小（十几个文件），Claude Code 通常能自己找到需要的代码。但项目一大，@ 引用就能帮你省掉"它读了半天不相关文件"的时间。

一个容易犯的错误：**@ 引用太多文件反而会降低质量**。你一口气引用了 20 个文件，Claude Code 的上下文窗口会被塞满，关键信息反而被淹没了。只引用和当前任务直接相关的 2-5 个文件。

## 分步指令 vs 一步到位

有时候你的需求很复杂，涉及多个步骤。这时候有两种策略：一口气说完，或者分步指导。

**什么时候该一步到位**：步骤之间有强依赖，且整体方案你心里有数。

```
重构 NotificationService：
1. 把邮件通知逻辑提取到 EmailNotificationService
2. 把推送通知逻辑提取到 PushNotificationService
3. 把站内信逻辑提取到 InAppNotificationService
4. NotificationService 变成一个调度器，根据通知类型分发
5. 补上对应的单元测试
```

这种情况下，你一次性说明完整的重构方案，Claude Code 能理解步骤之间的依赖关系，按正确顺序执行。

**什么时候该分步**：你不确定每一步该怎么做，需要看上一步的结果再决定下一步。

```
先分析一下 NotificationService，告诉我它现在有哪些通知渠道、
各自的逻辑有多复杂。先不改代码。
```

等它分析完了，你看到结果后再决定怎么拆分。这样每一步都有依据，不会出现"改了三步发现方向错了"的情况。

一个实用的经验：**如果你自己都不确定怎么做，先让它分析，别急着让它动手**。

## 常见的沟通陷阱

用了一段时间 Claude Code 之后，你会发现有几个坑特别容易踩。

**太模糊**。"帮我优化一下"——优化什么？性能？可读性？安全性？代码体积？"加个功能"——什么功能？业务规则是什么？边界条件有哪些？

**太冗长**。有些人写了 20 行 Prompt，精确到每一行代码该怎么写。这不叫"用 AI"，这叫"用自然语言写代码"。如果你已经知道每一步该怎么做，直接自己写可能更快。

**自相矛盾**。"用 TypeScript 严格模式，但类型定义可以用 any"——严格模式和 any 本身就是矛盾的。Claude Code 会纠结于满足哪个约束。

**当搜索引擎用**。"React useEffect 怎么用？"——这类通用编程问题，搜索引擎或官方文档更快更准。Claude Code 擅长的是结合你的项目上下文回答问题，而不是替代通用参考文档。

## 让它问你，而不是猜你

Claude Code 有一个内置能力叫 AskUserQuestion。当它不确定你的意图时，它会主动问你——比如"你希望用 Redis 还是内存缓存？"、"这个接口需要认证吗？"。

但默认情况下，它不一定会主动问。有些时候它更倾向于"猜一个最可能的方案"然后执行。如果你希望它多问你、少猜，可以在 CLAUDE.md 或 Prompt 里明确说：

```
如果有不确定的地方，先问我，不要自己猜。
```

反过来，如果你希望它尽量自己做决定、减少来回确认：

```
按照你的最佳判断来做，不需要每一步都问我。
```

两种方式各有适用场景。需要精确控制结果时用前者，需要快速推进时用后者。

## 代码规范怎么传达

一个常见的问题：项目有一套代码规范（命名规则、错误处理方式、注释风格等），每次对话都要重复一遍吗？

不需要。这就是 CLAUDE.md 的作用。把代码规范写进 CLAUDE.md，它每次启动都会自动读取。

但有个细节：**CLAUDE.md 里的规范要具体，不要泛泛而谈**。

```markdown
# 不好的规范（太笼统）

## 代码风格

请遵循良好的编码规范，保持代码整洁。

# 好的规范（具体可执行）

## 代码风格

- 文件命名：kebab-case（task.service.ts）
- DTO 用 class + class-validator，不用 interface
- 错误处理统一抛 RpcException，格式：new RpcException({ code: 'XXX', message: '...' })
- 数据库查询返回实体，Service 层转换为 DTO
- Controller 不写业务逻辑，只做参数校验和调用 Service
```

前者 Claude Code 看了等于没看——"良好的编码规范"对不同人意味着不同东西。后者每一条都是具体的、可执行的规则，它能精确地遵循。

如果你发现 Claude Code 生成的代码风格和你的预期不一致，不要在对话中一遍遍纠正——把规则加到 CLAUDE.md 里，一劳永逸。

## 一个实用技巧：给它看"对的代码"

Claude Code 是一个很强的"模仿者"。如果你给它看一段符合你预期的代码，然后说"按照这个风格来实现"，效果通常比写一堆文字描述要好得多。

```
按照 src/modules/user/user.service.ts 的风格，
实现 TaskService 的 create 方法。
注意看 user.service.ts 里的错误处理方式和 DTO 转换模式。
```

它读了 user.service.ts 之后，会自动提取出代码风格特征——错误处理怎么写、DTO 怎么转换、返回值怎么组织——然后在实现 TaskService 时模仿这些特征。比你用文字描述"先校验参数，然后查数据库，然后转换 DTO"要精准得多。

这个技巧在团队里尤其有用。新人加入团队时，让他给 Claude Code 看几个"标准实现"，然后说"按这个风格来"，生成的代码就能基本符合团队规范。
