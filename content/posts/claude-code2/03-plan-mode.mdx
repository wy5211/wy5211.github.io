上一篇我们聊了怎么和 Claude Code 高效沟通。但沟通再好，也架不住一个场景：你让它做一件复杂的事，它理解了你的意思，然后直接动手改了——改了 30 个文件，改完你一看，方向完全错了。

这就是为什么 Claude Code 有 Plan Mode。

## 一次惨痛的教训

假设你的项目有一个 NotificationService，400 多行代码，混合了 5 种通知渠道（邮件、推送、站内信、Slack、Webhook）。你跟 Claude Code 说：

```
重构 NotificationService，把不同渠道的逻辑拆分开。
```

如果没有 Plan Mode，它可能会怎么做？直接开始改——拆文件、移动方法、改 import。改了 30 个文件之后你一看：它把邮件通知拆出去了，但忘记处理邮件模板的共享逻辑；Webhook 的重试机制被丢弃了； Slack 通知的 @mention 功能丢了。

为什么？因为它没有先理解整体架构，就急匆匆地开始改代码。就像一个装修工人，没看户型图就开始拆墙。

Plan Mode 解决的就是这个问题。开启 Plan Mode 后，Claude Code **只能读代码和搜索，不能修改任何文件**。它先探索代码、分析依赖、设计方案，等你确认了方案之后，再退出 Plan Mode 执行修改。

## Plan Mode 的本质

Plan Mode 本质上就是：**先做只读探索和方案设计，你确认后再写代码**。

进入 Plan Mode 的方式很简单——在对话中按 `Shift+Tab`，或者直接说"进入 Plan Mode"。

进入之后，Claude Code 的行为会有几个变化：

- 它会**先花时间读代码**，用 Glob 扫描相关文件、用 Read 读关键文件、用 Grep 搜索依赖关系
- 它会把分析和方案**写成一份计划文档**，列出要做的事情、涉及的文件、潜在风险
- 你可以**审阅这份计划**，提出修改意见、补充约束、调整方向
- 确认方案后，退出 Plan Mode，它才会开始执行修改

核心价值在哪？**让你在它动手之前就能发现问题**。方向错了、方案有漏洞、遗漏了边界情况——这些在规划阶段发现和改掉的成本，远低于代码改完之后再修复。

## 什么时候用 Plan Mode

不是所有任务都需要 Plan Mode。用多了反而浪费时间。

**应该用 Plan Mode 的场景**：

改动范围大——涉及 5 个以上文件的修改。比如"重构认证模块"、"把错误处理统一成一种方式"。这类任务一旦方向错了，返工成本很高。

涉及架构决策——"引入 Redis 缓存"、"拆分微服务"。这类任务有多种方案可选，需要先分析利弊再决定。

你不确定怎么做——你自己心里也没底，需要先探索代码再决定方案。这时候让 Claude Code 先分析，你看了分析结果再决策。

**不需要用 Plan Mode 的场景**：

单文件修改——"给这个方法加个参数"、"修复这个 typo"。改动范围小，方向不容易错，直接做就行。

明确的小需求——"加一个按优先级排序的接口"。你知道要做什么、怎么做，不需要规划。

探索性任务——"帮我看看这个模块是怎么工作的"。这只是读代码分析，不涉及修改，本身就不需要"执行"阶段。

一个简单的判断标准：**如果改错了，返工成本超过 10 分钟，就用 Plan Mode**。

## 一个完整的工作流程

用一个真实场景来走一遍 Plan Mode 的完整流程：给任务列表加全文搜索功能。

### 第一步：进入 Plan Mode，提出需求

```
进入 Plan Mode。

我想给任务列表（GET /tasks）加全文搜索功能，用户输入关键词，
搜索任务的 title 和 description 字段。
先分析现有代码，再设计实现方案。
```

### 第二步：Claude Code 探索代码

进入 Plan Mode 后，它会先读代码。它会做的事情可能包括：

- 读 `task.service.ts`，了解现有的查询逻辑
- 读 `task.entity.ts`，了解数据模型和索引
- 检查 `package.json`，看有没有已经安装的搜索相关依赖
- 搜索项目中是否有其他全文搜索的实现可以参考
- 检查数据库配置，了解用的是 PostgreSQL 还是 MySQL

这个过程可能需要十几秒到半分钟，取决于项目大小。

### 第三步：它给出分析和方案

探索完之后，Claude Code 会给你一份分析报告和实施方案。大致内容可能是：

**现状分析**：

- 当前任务列表用的是 TypeORM 的 `find` 方法，支持精确匹配筛选
- PostgreSQL 数据库，支持原生全文搜索（`tsvector` + `tsquery`）
- 没有安装额外的搜索依赖（如 Elasticsearch）

**方案建议**：

- 方案 A：用 PostgreSQL 原生全文搜索。优点是零额外依赖，性能足够。缺点是搜索能力有限（不支持模糊匹配、同义词）。
- 方案 B：接入 Elasticsearch。优点是搜索能力强，支持模糊匹配、高亮、聚合。缺点是引入新组件，运维成本增加。

**推荐**：方案 A。原因：当前任务量级（万级），PostgreSQL 原生搜索完全够用。如果未来任务量增长到百万级，再迁移到 Elasticsearch。

**改动清单**：

1. 在 Task 实体上添加全文搜索索引列（tsvector）
2. 在 task.service.ts 中添加 search 方法
3. 在 query-task.dto.ts 中添加 keyword 参数
4. 写一个数据库迁移脚本

**风险点**：

- 需要一个数据库迁移来添加索引列，迁移过程需要锁表（数据量大时注意）
- 已有数据的 tsvector 列需要回填

### 第四步：你审阅方案，给出反馈

这一步很关键。你看了方案之后，可能想调整：

```
方案 A 可以。但有几个补充：
1. 搜索结果按相关度排序，不是按时间
2. keyword 为空时不走搜索逻辑，避免不必要的索引查询
3. 数据回填写成一个独立的迁移脚本，不和索引创建放在一起
```

Claude Code 会根据你的反馈修改方案，然后给出更新后的版本。

### 第五步：确认方案，退出 Plan Mode 执行

方案确认后，你退出 Plan Mode（再按一次 `Shift+Tab`），Claude Code 开始按照方案执行修改。

这时候你观察它的执行过程，会发现它很有条理：先创建迁移脚本，再修改实体，再修改 Service，最后更新 DTO。每一步都和方案对应，不会漏掉什么，也不会多做什么。

## 怎么给有效反馈

Plan Mode 中你给反馈的质量，直接决定了最终方案的质量。几个原则：

**具体指出问题，不要泛泛否定**。"方案不对"——它不知道哪里不对。"方案 A 的回填策略有问题，数据量大了会锁表太久"——它知道该怎么改。

**补充它不知道的上下文**。Claude Code 不知道你的生产环境数据量、不知道你的运维能力、不知道你的发布节奏。这些信息你不说，它就得猜。

**区分"必须改"和"建议"**。"这个方案里，数据库迁移必须分两步执行（先加列，再建索引）。但搜索结果的高亮功能可以先不做，后面再加。"这样它知道哪些是硬性约束，哪些可以灵活处理。

## Plan Mode 的局限

Plan Mode 不是万能的。

**它不能替代你的判断**。Claude Code 能分析代码结构、评估技术方案的优劣，但架构决策最终需要人来拍板。它可能推荐方案 A，但如果你的团队对方案 B 更熟悉、维护成本更低——你应该选 B。

**规划质量取决于代码的可读性**。如果代码写得很乱、命名很随意、没有注释——Claude Code 的分析质量也会下降。Garbage in, garbage out。

**不是所有复杂任务都适合规划**。有些任务需要"边做边想"——你改了一步，发现了新的信息，然后调整下一步的方向。这类任务强制先规划反而会限制灵活性。

## 什么时候该放弃 Plan Mode

如果你发现自己在 Plan Mode 里来回讨论了 5 轮还没确定方案，可能说明这个任务不适合用 Plan Mode。这时候果断退出，直接动手做——先做一个最小可用的版本，再迭代改进。

Plan Mode 是一个工具，不是教条。它的目的是减少返工成本，如果规划本身的成本已经超过了返工成本，就不用规划了。
