认证（Authentication）和授权（Authorization）是全栈应用安全的基石。一个设计良好的认证授权系统，不仅能保护用户数据安全，还能提升用户体验。本文将深入探讨现代全栈应用中的最佳实践方案。

## 认证 vs 授权：概念澄清

在开始之前，先明确两个概念的区别：

- **认证（Authentication）**：验证"你是谁"
- **授权（Authorization）**：验证"你能做什么"

```
认证：用户身份验证 → 确认用户身份
授权：权限验证 → 确认访问权限
```

## 一、认证方案选型

### 1.1 Session-Cookie 认证

传统的 Session 方案，适合服务端渲染应用。

**工作流程：**

```mermaid
用户登录 → 服务器创建 Session → 返回 Session ID (Cookie) → 后续请求携带 Cookie → 服务器验证 Session
```

**优点：**

- 实现简单，浏览器自动管理 Cookie
- 服务端可随时撤销 Session
- 相对安全（Session ID 不包含敏感信息）

**缺点：**

- 不适合分布式系统（需要 Session 共享）
- CSRF 攻击风险
- 跨域配置复杂

**实现示例（NestJS）：**

```typescript
// auth.controller.ts
@Controller("auth")
export class AuthController {
  constructor(private authService: AuthService) {}

  @Post("login")
  async login(
    @Body() loginDto: LoginDto,
    @Session() session: Record<string, any>
  ) {
    const user = await this.authService.validateUser(
      loginDto.email,
      loginDto.password
    );

    // 存储用户信息到 Session
    session.userId = user.id;
    session.userRole = user.role;

    return { message: "登录成功", user };
  }

  @Post("logout")
  async logout(@Session() session: Record<string, any>) {
    session.destroy();
    return { message: "登出成功" };
  }
}
```

### 1.2 JWT 认证

现代前后端分离应用的主流方案。

**工作流程：**

```
用户登录 → 服务器签发 JWT → 客户端存储 Token → 请求携带 Token → 服务器验证签名
```

**JWT 结构：**

```
Header.Payload.Signature
```

- **Header**：算法类型（如 HS256、RS256）
- **Payload**：用户信息（userId、role、exp 等）
- **Signature**：防篡改签名

**优点：**

- 无状态，易于扩展
- 跨域友好
- 适合微服务架构

**缺点：**

- 无法主动撤销（需要黑名单机制）
- Token 泄露风险高
- Payload 不加密，不能存敏感信息

**实现示例（NestJS + Passport）：**

```typescript
// jwt.strategy.ts
@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor(configService: ConfigService) {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: configService.get("JWT_SECRET"),
    });
  }

  async validate(payload: any) {
    return { userId: payload.sub, email: payload.email, role: payload.role };
  }
}

// auth.service.ts
@Injectable()
export class AuthService {
  constructor(
    private jwtService: JwtService,
    private usersService: UsersService
  ) {}

  async login(user: any) {
    const payload = {
      sub: user.id,
      email: user.email,
      role: user.role,
    };

    return {
      access_token: this.jwtService.sign(payload),
      refresh_token: this.jwtService.sign(payload, {
        expiresIn: "7d",
      }),
    };
  }
}
```

### 1.3 OAuth2 第三方登录

让用户通过 Google、GitHub、微信等平台登录。

**OAuth2 授权码流程：**

```
1. 用户点击"使用 GitHub 登录"
2. 重定向到 GitHub 授权页面
3. 用户同意授权
4. GitHub 回调并返回授权码 (code)
5. 服务器用 code 换取 access_token
6. 使用 token 获取用户信息
```

**实现示例（Passport-GitHub2）：**

```typescript
// github.strategy.ts
@Injectable()
export class GitHubStrategy extends PassportStrategy(Strategy, "github") {
  constructor(configService: ConfigService) {
    super({
      clientID: configService.get("GITHUB_CLIENT_ID"),
      clientSecret: configService.get("GITHUB_CLIENT_SECRET"),
      callbackURL: "http://localhost:3000/auth/github/callback",
      scope: ["user:email"],
    });
  }

  async validate(
    accessToken: string,
    refreshToken: string,
    profile: any,
    done: VerifyCallback
  ) {
    const { id, displayName, emails, username } = profile;

    let user = await this.usersService.findByGithubId(id);

    if (!user) {
      user = await this.usersService.create({
        githubId: id,
        email: emails[0].value,
        name: displayName || username,
      });
    }

    done(null, user);
  }
}

// auth.controller.ts
@Controller("auth")
export class AuthController {
  @Get("github")
  @UseGuards(AuthGuard("github"))
  async githubLogin() {
    // Passport 自动重定向
  }

  @Get("github/callback")
  @UseGuards(AuthGuard("github"))
  async githubCallback(@Req() req, @Res() res) {
    const tokens = await this.authService.login(req.user);

    // 方案1：设置 HttpOnly Cookie
    res.cookie("access_token", tokens.access_token, {
      httpOnly: true,
      secure: true,
      sameSite: "strict",
    });

    // 方案2：重定向到前端并携带 token
    res.redirect(
      `https://myapp.com/auth/callback?token=${tokens.access_token}`
    );
  }
}
```

## 二、Token 安全最佳实践

### 2.1 双 Token 机制（Access + Refresh）

```
Access Token：短期有效（15-30分钟）
Refresh Token：长期有效（7-30天）
```

**实现示例：**

```typescript
// auth.service.ts
async refreshTokens(userId: string, refreshToken: string) {
  // 验证 refresh token
  const payload = this.jwtService.verify(refreshToken);

  if (payload.sub !== userId) {
    throw new UnauthorizedException('Invalid refresh token');
  }

  // 检查 refresh token 是否在黑名单中
  const isBlacklisted = await this.redisService.get(
    `blacklist:${refreshToken}`,
  );

  if (isBlacklisted) {
    throw new UnauthorizedException('Token revoked');
  }

  // 签发新的 access token
  const user = await this.usersService.findById(userId);

  return {
    access_token: this.jwtService.sign({
      sub: user.id,
      email: user.email,
      role: user.role,
    }),
    // 可选：同时刷新 refresh token
    refresh_token: this.jwtService.sign(
      { sub: user.id },
      { expiresIn: '7d' },
    ),
  };
}
```

### 2.2 Token 存储方案对比

| 存储方式        | XSS 风险 | CSRF 风险 | 推荐度   |
| --------------- | -------- | --------- | -------- |
| LocalStorage    | 高       | 无        | ⭐⭐     |
| SessionStorage  | 高       | 无        | ⭐⭐     |
| HttpOnly Cookie | 无       | 高        | ⭐⭐⭐⭐ |
| Memory（内存）  | 无       | 无        | ⭐⭐⭐   |

**推荐方案：HttpOnly Cookie + CSRF 保护**

```typescript
// 登录时设置 Cookie
@Post('login')
async login(@Body() loginDto: LoginDto, @Res() res: Response) {
  const tokens = await this.authService.login(loginDto);

  res.cookie('access_token', tokens.access_token, {
    httpOnly: true,  // 防止 XSS
    secure: true,    // 仅 HTTPS
    sameSite: 'strict',  // 防止 CSRF
    maxAge: 15 * 60 * 1000,  // 15 分钟
  });

  res.cookie('refresh_token', tokens.refresh_token, {
    httpOnly: true,
    secure: true,
    sameSite: 'strict',
    maxAge: 7 * 24 * 60 * 60 * 1000,  // 7 天
  });

  res.json({ message: '登录成功' });
}
```

### 2.3 Token 撤销机制

JWT 的无状态特性导致难以主动撤销，常用方案：

**方案1：黑名单机制（Redis）**

```typescript
// auth.service.ts
async logout(userId: string, token: string) {
  // 将 token 加入黑名单
  const ttl = this.getTTL(token);  // 剩余有效时间
  await this.redisService.set(
    `blacklist:${token}`,
    '1',
    'EX',
    ttl,
  );

  // 可选：清除用户所有 token（强登）
  await this.redisService.set(
    `user:${userId}:invalidated_at`,
    Date.now(),
  );
}

// jwt.strategy.ts
async validate(payload: any, token: string) {
  // 检查黑名单
  const isBlacklisted = await this.redisService.get(
    `blacklist:${token}`,
  );

  if (isBlacklisted) {
    throw new UnauthorizedException('Token revoked');
  }

  // 检查是否在登出时间之后签发
  const invalidatedAt = await this.redisService.get(
    `user:${payload.sub}:invalidated_at`,
  );

  if (invalidatedAt && payload.iat * 1000 &lt; parseInt(invalidatedAt)) {
    throw new UnauthorizedException('Token expired');
  }

  return { userId: payload.sub, role: payload.role };
}
```

**方案2：短期 Token + Refresh 机制**

```
- Access Token 有效期短（5-15分钟）
- 每次刷新时检查用户状态
- 发现异常立即拒绝刷新
```

## 三、授权系统设计

### 3.1 RBAC（基于角色的访问控制）

最常用的授权模型，适合大多数应用。

**数据模型：**

```sql
-- 用户表
CREATE TABLE users (
  id VARCHAR(36) PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password VARCHAR(255) NOT NULL,
  role_id VARCHAR(36) REFERENCES roles(id)
);

-- 角色表
CREATE TABLE roles (
  id VARCHAR(36) PRIMARY KEY,
  name VARCHAR(50) UNIQUE NOT NULL,  -- admin, user, moderator
  description TEXT
);

-- 权限表
CREATE TABLE permissions (
  id VARCHAR(36) PRIMARY KEY,
  name VARCHAR(100) UNIQUE NOT NULL,  -- user:create, post:delete
  resource VARCHAR(50) NOT NULL,      -- user, post, comment
  action VARCHAR(20) NOT NULL         -- create, read, update, delete
);

-- 角色-权限关联表
CREATE TABLE role_permissions (
  role_id VARCHAR(36) REFERENCES roles(id),
  permission_id VARCHAR(36) REFERENCES permissions(id),
  PRIMARY KEY (role_id, permission_id)
);
```

**实现示例（NestJS）：**

```typescript
// roles.decorator.ts
export const ROLES_KEY = "roles";
export const Roles = (...roles: string[]) => SetMetadata(ROLES_KEY, roles);

// roles.guard.ts
@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.getAllAndOverride<string[]>(
      ROLES_KEY,
      [context.getHandler(), context.getClass()]
    );

    if (!requiredRoles) {
      return true;
    }

    const { user } = context.switchToHttp().getRequest();

    return requiredRoles.some((role) => user.role === role);
  }
}

// 使用示例
@Controller("admin")
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles("admin")
export class AdminController {
  @Get("users")
  async getAllUsers() {
    // 只有 admin 角色可以访问
  }
}
```

### 3.2 ABAC（基于属性的访问控制）

更灵活的授权模型，适合复杂场景。

**核心概念：**

```
用户属性：部门、职级、年龄
资源属性：所有者、部门、敏感级别
环境属性：时间、IP、设备
操作属性：读取、编辑、删除
```

**策略示例：**

```typescript
// 策略：用户只能编辑自己部门且非敏感的文档
{
  "effect": "allow",
  "action": "edit",
  "resource": "document",
  "conditions": {
    "user.department === resource.department",
    "resource.sensitivityLevel &lt; 3",
    "user.role !== 'guest'"
  }
}
```

**实现示例：**

```typescript
// casl-ability.factory.ts
import { Ability, AbilityBuilder, AbilityClass } from '@casl/ability';

type Actions = 'manage' | 'create' | 'read' | 'update' | 'delete';
type Subjects = 'Article' | 'Comment' | 'User' | 'all';

export type AppAbility = Ability&lt;[Actions, Subjects]&gt;;

@Injectable()
export class CaslAbilityFactory {
  createForUser(user: User) {
    const { can, cannot, build } = new AbilityBuilder&lt;Ability&lt;[Actions, Subjects]&gt;&gt;(
      Ability as AbilityClass<AppAbility>,
    );

    if (user.role === 'admin') {
      // 管理员可以做任何事
      can('manage', 'all');
    } else {
      // 普通用户可以读取所有文章
      can('read', 'Article');

      // 可以创建文章
      can('create', 'Article');

      // 可以更新/删除自己的文章
      can(['update', 'delete'], 'Article', { authorId: user.id });

      // 不能删除已被置顶的文章
      cannot('delete', 'Article', { isPinned: true });
    }

    return build();
  }
}

// policies.guard.ts
@Injectable()
export class PoliciesGuard implements CanActivate {
  constructor(
    private caslAbilityFactory: CaslAbilityFactory,
    private reflector: Reflector,
  ) {}

  async canActivate(context: ExecutionContext): Promise&lt;boolean&gt; {
    const policyHandlers =
      this.reflector.get&lt;PolicyHandler[]&gt;(
        CHECK_POLICIES_KEY,
        context.getHandler(),
      ) || [];

    const { user } = context.switchToHttp().getRequest();
    const ability = this.caslAbilityFactory.createForUser(user);

    return policyHandlers.every((handler) =>
      this.execPolicyHandler(handler, ability),
    );
  }

  private execPolicyHandler(handler: PolicyHandler, ability: AppAbility) {
    if (typeof handler === 'function') {
      return handler(ability);
    }
    return handler.handle(ability);
  }
}

// 使用示例
@Controller('articles')
export class ArticleController {
  @Post()
  @UseGuards(JwtAuthGuard, PoliciesGuard)
  @CheckPolicies((ability: AppAbility) =>
    ability.can('create', 'Article')
  )
  create(@Body() createArticleDto: CreateArticleDto) {
    // 创建文章
  }
}
```

```typescript
@UseGuards(RolesGuard)

  }

async getArticles(userId: string, role: string) {

  if (role !== 'admin') {
    // 非管理员只能看公开文章或自己的文章
    query.where('article.isPublic = :isPublic', { isPublic: true })
         .orWhere('article.authorId = :userId', { userId });
  }

  return query.getMany();
}
```

### 4.3 Token 自动刷新

```typescript
// lib/apiClient.ts
class ApiClient {
  private isRefreshing = false;
  private refreshSubscribers: Array&lt;(token: string) => void&gt; = [];

  async request(url: string, options: RequestInit = {}) {
    try {
      const response = await fetch(url, {
        ...options,
        credentials: 'include',
      });

      if (response.status === 401) {
        // Token 过期，尝试刷新
        return this.handleTokenExpired(url, options);
      }

      return response;
    } catch (error) {
      throw error;
    }
  }

  private async handleTokenExpired(url: string, options: RequestInit) {
    if (!this.isRefreshing) {
      this.isRefreshing = true;

      try {
        // 刷新 token
        await fetch('/api/auth/refresh', {
          method: 'POST',
          credentials: 'include',
        });

        // 重试原请求
        this.isRefreshing = false;
        return this.request(url, options);
      } catch (error) {
        // 刷新失败，跳转登录
        window.location.href = '/login';
        throw error;
      }
    } else {
      // 等待刷新完成
      return new Promise((resolve) => {
        this.refreshSubscribers.push(() => {
          resolve(this.request(url, options));
        });
      });
    }
  }
}

export const apiClient = new ApiClient();
```

```typescript
import * as csurf from 'csurf';

app.use(csurf({ cookie: true }));

// 在登录响应中返回 CSRF token
@Post('login')
async login(@Req() req, @Res() res) {
  const tokens = await this.authService.login(loginDto);

  res.json({
    ...tokens,
    csrfToken: req.csrfToken(),  // 返回给前端
  });
}

// 前端每次请求携带 CSRF token
fetch('/api/data', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': csrfToken,
  },
  credentials: 'include',
});
```

### 6.2 NextAuth.js 集成（推荐）

```typescript
// lib/auth.ts
import NextAuth from "next-auth";
import CredentialsProvider from "next-auth/providers/credentials";
import GitHubProvider from "next-auth/providers/github";

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [
    CredentialsProvider({
      name: "credentials",
      credentials: {
        email: { label: "Email", type: "email" },
        password: { label: "Password", type: "password" },
      },
      async authorize(credentials) {
        const user = await verifyCredentials(
          credentials.email,
          credentials.password
        );

        if (user) {
          return { id: user.id, email: user.email, role: user.role };
        }

        return null;
      },
    }),
    GitHubProvider({
      clientId: process.env.GITHUB_CLIENT_ID!,
      clientSecret: process.env.GITHUB_CLIENT_SECRET!,
    }),
  ],
  callbacks: {
    async jwt({ token, user }) {
      if (user) {
        token.role = user.role;
      }
      return token;
    },
    async session({ session, token }) {
      session.user.role = token.role;
      return session;
    },
  },
  pages: {
    signIn: "/login",
  },
});

// middleware.ts
export { auth as middleware } from "@/lib/auth";

export const config = {
  matcher: ["/dashboard/:path*", "/admin/:path*"],
};
```

## 七、总结

### 认证方案选择建议

| 场景           | 推荐方案              |
| -------------- | --------------------- |
| 传统服务端渲染 | Session-Cookie        |
| 前后端分离     | JWT + HttpOnly Cookie |
| 微服务架构     | JWT + 网关统一认证    |
| 第三方登录     | OAuth2 + JWT          |
| 企业内部系统   | SSO + LDAP/AD         |

### 授权模型选择建议

| 场景                     | 推荐模型        |
| ------------------------ | --------------- |
| 简单应用（2-3个角色）    | 硬编码角色检查  |
| 中型应用（多角色多权限） | RBAC            |
| 复杂业务规则             | ABAC            |
| 细粒度资源控制           | CASL/自定义策略 |

### 安全检查清单

- [ ] 密码使用 bcrypt/scrypt 加密（cost factor &gt;= 12）
- [ ] JWT 密钥足够复杂（&gt;= 256 bits）
- [ ] 使用 HTTPS
- [ ] 实施 CORS 策略
- [ ] 设置 Content Security Policy
- [ ] 实施 CSRF 保护（如使用 Cookie）
- [ ] 登录限流（防暴力破解）
- [ ] 敏感操作二次验证
- [ ] 日志记录和监控
- [ ] 定期安全审计
