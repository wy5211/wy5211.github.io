# 全栈项目架构：从单体到微服务的演进

架构不是一开始就设计出来的，而是随业务发展演进的。本文讲解从单体到微服务的演进路径，帮你做出合理的架构决策。

## 一、单体架构

### 1.1 什么是单体

```
┌─────────────────────────────────┐
│          Monolith App           │
│  ┌─────────┐  ┌──────────────┐ │
│  │   API   │  │   Frontend   │ │
│  │  Layer  │  │   (SSR/SPA)  │ │
│  └────┬────┘  └──────────────┘ │
│       │                         │
│  ┌────┴────────────────────┐   │
│  │      Business Logic     │   │
│  └────────────┬────────────┘   │
│               │                 │
│  ┌────────────┴────────────┐   │
│  │       Database          │   │
│  └─────────────────────────┘   │
└─────────────────────────────────┘
```

**优点：**

- 开发简单，部署方便
- 事务管理容易
- 调试方便
- 适合小团队快速迭代

**缺点：**

- 代码耦合，维护困难
- 扩展受限，只能整体扩容
- 技术栈单一
- 部署风险高

### 1.2 何时使用单体

- 项目初期，业务不明确
- 团队规模 &lt; 10 人
- 用户量 &lt; 10 万
- 快速验证 MVP

### 1.3 单体最佳实践

```javascript
// 分层架构
src/
├── controllers/    # API 层
├── services/       # 业务逻辑
├── repositories/   # 数据访问
├── models/         # 数据模型
├── middlewares/    # 中间件
└── utils/          # 工具函数

// 模块化
// ❌ 错误：按技术分层
src/
├── controllers/
│   ├── userController.js
│   ├── orderController.js
│   └── productController.js

// ✅ 正确：按业务领域分层（DDD）
src/
├── modules/
│   ├── user/
│   │   ├── controller.js
│   │   ├── service.js
│   │   ├── repository.js
│   │   └── model.js
│   ├── order/
│   └── product/
```

## 二、模块化单体

### 2.1 什么是模块化单体

在单体架构中，按业务领域划分模块，模块间通过明确接口通信。

```
┌─────────────────────────────────────┐
│          Modular Monolith           │
│  ┌─────────┐ ┌─────────┐ ┌────────┐│
│  │  User   │ │  Order  │ │Product ││
│  │ Module  │ │ Module  │ │ Module ││
│  └────┬────┘ └────┬────┘ └───┬────┘│
│       │           │          │      │
│       └───────────┼──────────┘      │
│                   │                 │
│            ┌──────┴──────┐          │
│            │   Database  │          │
│            └─────────────┘          │
└─────────────────────────────────────┘
```

### 2.2 实现方式

```javascript
// 模块定义
// modules/user/index.js
class UserModule {
  constructor() {
    this.repository = new UserRepository();
    this.service = new UserService(this.repository);
    this.controller = new UserController(this.service);
  }

  getRouter() {
    return this.controller.router;
  }

  // 暴露给其他模块的接口
  async getUserById(id) {
    return this.service.findById(id);
  }
}

// 应用入口
// app.js
const userModule = new UserModule();
const orderModule = new OrderModule(userModule);

app.use("/users", userModule.getRouter());
app.use("/orders", orderModule.getRouter());
```

### 2.3 模块间通信

```javascript
// ❌ 错误：直接访问其他模块内部
class OrderService {
  async createOrder(userId) {
    // 直接调用 user 模块的 repository
    const user = await userRepository.findById(userId);
  }
}

// ✅ 正确：通过模块公开的接口
class OrderService {
  constructor(userModule) {
    this.userModule = userModule;
  }

  async createOrder(userId) {
    const user = await this.userModule.getUserById(userId);
  }
}
```

## 三、微服务架构

### 3.1 什么是微服务

```
┌──────────┐   ┌──────────┐   ┌──────────┐
│   User   │   │  Order   │   │ Product  │
│ Service  │   │ Service  │   │ Service  │
└────┬─────┘   └────┬─────┘   └────┬─────┘
     │              │              │
┌────┴────┐   ┌────┴────┐   ┌────┴────┐
│User DB  │   │Order DB │   │Prod DB  │
└─────────┘   └─────────┘   └─────────┘
```

**特点：**

- 每个服务独立部署
- 服务间通过 API 通信
- 每个服务有自己的数据库
- 技术栈可以不同

### 3.2 何时拆分微服务

- 业务边界清晰
- 团队规模 &gt; 20 人
- 需要独立扩展某些模块
- 不同模块有不同的技术需求

### 3.3 服务拆分策略

**按业务能力拆分：**

```
- 用户服务（User Service）
- 订单服务（Order Service）
- 商品服务（Product Service）
- 支付服务（Payment Service）
- 通知服务（Notification Service）
```

**按子域拆分（DDD）：**

```
- 用户子域：认证、权限、用户信息
- 订单子域：购物车、订单、物流
- 商品子域：商品、库存、价格
```

### 3.4 服务通信

**同步通信（REST/gRPC）：**

```javascript
// REST
const response = await fetch("http://user-service/api/users/123");
const user = await response.json();

// gRPC
const user = await userServiceClient.getUser({ id: "123" });
```

**异步通信（消息队列）：**

```javascript
// 发布事件
await messageQueue.publish("order.created", {
  orderId: "123",
  userId: "456",
  amount: 99.99,
});

// 订阅事件
messageQueue.subscribe("order.created", async (event) => {
  // 更新用户积分
  await updateUserPoints(event.userId, event.amount);
});
```

## 四、数据一致性

### 4.1 分布式事务问题

```javascript
// ❌ 问题：跨服务事务
async function placeOrder(orderData) {
  // 1. 创建订单
  await orderService.createOrder(orderData);

  // 2. 扣减库存（失败怎么办？）
  await productService.decreaseStock(orderData.productId);

  // 3. 扣款（失败怎么办？）
  await paymentService.charge(orderData.userId, orderData.amount);
}
```

### 4.2 Saga 模式

```javascript
// 编排式 Saga
class OrderSaga {
  async execute(orderData) {
    const steps = [
      { action: "createOrder", compensate: "cancelOrder" },
      { action: "reserveStock", compensate: "releaseStock" },
      { action: "chargePayment", compensate: "refundPayment" },
    ];

    const executed = [];

    for (const step of steps) {
      try {
        await this[step.action](orderData);
        executed.push(step);
      } catch (error) {
        // 补偿已执行的步骤
        for (const s of executed.reverse()) {
          await this[s.compensate](orderData);
        }
        throw error;
      }
    }
  }
}
```

### 4.3 最终一致性

```javascript
// 事件驱动
class OrderService {
  async createOrder(orderData) {
    const order = await this.repository.create(orderData)

    // 发布事件
    await this.eventBus.publish('OrderCreated', {
      orderId: order.id,
      items: order.items
    })

    return order
  }
}

// 库存服务监听事件
@EventHandler('OrderCreated')
async function handleOrderCreated(event) {
  for (const item of event.items) {
    await inventoryService.decrease(item.productId, item.quantity)
  }
}
```

## 五、API 网关

### 5.1 为什么需要网关

- 统一入口
- 认证授权
- 限流熔断
- 负载均衡
- 日志监控

### 5.2 实现方案

```nginx
# Nginx 配置
upstream user_service {
  server user-service:3001;
}

upstream order_service {
  server order-service:3002;
}

server {
  listen 80;

  location /api/users {
    proxy_pass http://user_service;
  }

  location /api/orders {
    proxy_pass http://order_service;
  }
}
```

**使用 Kong / APISIX：**

```yaml
# Kong 配置
routes:
  - name: user-service
    paths:
      - /api/users
    service: user-service
    plugins:
      - name: jwt
      - name: rate-limiting
        config:
          minute: 100
```

## 六、服务发现

### 6.1 为什么需要服务发现

微服务动态扩缩容，IP 不固定，需要自动发现服务实例。

### 6.2 实现方案

**Consul：**

```javascript
// 注册服务
await consul.agent.service.register({
  name: "user-service",
  address: "192.168.1.100",
  port: 3001,
});

// 发现服务
const services = await consul.catalog.service.nodes("user-service");
// [{ address: '192.168.1.100', port: 3001 }, ...]
```

**Kubernetes：**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service
  ports:
    - port: 80
      targetPort: 3001
```

## 八、监控与日志

### 8.1 分布式追踪

```javascript
// OpenTelemetry
const { trace } = require("@opentelemetry/api");

const tracer = trace.getTracer("user-service");

async function getUser(id) {
  const span = tracer.startSpan("getUser");

  try {
    const user = await userRepository.findById(id);
    span.end();
    return user;
  } catch (error) {
    span.recordException(error);
    span.end();
    throw error;
  }
}
```

## 九、演进路线

```
Stage 1: 单体应用
  ↓ (用户量增长，团队扩大)
Stage 2: 模块化单体
  ↓ (性能瓶颈，独立扩展需求)
Stage 3: 前后端分离 + 服务拆分
  ↓ (业务复杂度增加)
Stage 4: 微服务 + 容器化
  ↓ (大规模分布式)
Stage 5: 云原生 + Service Mesh
```

## 十、架构决策清单

### 拆分前考虑

- [ ] 业务边界是否清晰？
- [ ] 团队是否足够大（每个服务至少 2-3 人维护）？
- [ ] 运维能力是否足够？
- [ ] 是否真的需要独立扩展？

### 拆分策略

- [ ] 从最独立的模块开始
- [ ] 先拆分读多写少的服务
- [ ] 数据库最后拆分
- [ ] 保持向后兼容

### 拆分后

- [ ] 服务间通信方案
- [ ] 数据一致性方案
- [ ] 监控和日志
- [ ] 故障处理

## 总结

架构演进是渐进的过程。

**核心原则：**

1. **不要过早微服务** - 从单体开始
2. **按需演进** - 有问题再拆
3. **模块化先行** - 模块化单体是过渡
4. **数据最后拆** - 数据一致性最难处理

**记住：**

> 微服务不是目的，解决问题才是。

大多数项目，模块化单体就够了。不要为了微服务而微服务。
