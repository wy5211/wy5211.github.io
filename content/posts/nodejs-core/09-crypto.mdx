日志中包含用户的手机号、邮箱、IP 地址。直接明文存储，一旦数据库泄露就是一场灾难。日志查询 API 没有任何认证，谁都能调用。日志文件存了一周，你不确定中间有没有被篡改过。

Node.js 的 `crypto` 模块能解决这些问题。但加密不是简单地调个 API——选错算法、用错模式、管理不好密钥，都可能让加密形同虚设。

## 哈希函数：数据指纹

哈希函数把任意长度的数据映射成固定长度的"指纹"。同样的输入永远得到同样的输出，从输出无法反推输入。

```js
import { createHash } from "node:crypto";

const hash = createHash("sha256").update("hello").digest("hex");
// '2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824'
```

不同算法的输出长度和安全级别：

| 算法    | 输出长度 | 安全状态                 |
| ------- | -------- | ------------------------ |
| MD5     | 128 位   | 已破解，不要用于安全场景 |
| SHA-1   | 160 位   | 已破解，不要用于安全场景 |
| SHA-256 | 256 位   | 安全，当前推荐           |
| SHA-512 | 512 位   | 安全，更长的输出         |

MD5 和 SHA-1 已经被证明存在碰撞攻击——你可以找到两个不同的输入产生相同的哈希值。它们唯一合适的用途是**非安全场景的完整性校验**（比如检查文件下载是否完整）。任何涉及安全的地方（密码存储、数字签名、数据校验），用 SHA-256 或更高。

### 流式哈希：大文件校验

对于大文件，不需要一次性读入内存。`createHash` 本身就是一个 Transform Stream，可以直接和文件流连接：

```js
import { createHash } from "node:crypto";
import { createReadStream } from "node:fs";
import { pipeline } from "node:stream/promises";

async function fileHash(filePath, algorithm = "sha256") {
  const hash = createHash(algorithm);
  await pipeline(createReadStream(filePath), hash);
  return hash.digest("hex");
}

// 校验日志文件是否被篡改
const hash = await fileHash("logs/2026-03-15.log");
console.log(`SHA-256: ${hash}`);
```

这个用法利用了 [第 2 篇](/posts/nodejs-core/02-stream) 讲的 Stream 管线——不管文件多大，内存占用始终很小。

## AES 对称加密：保护敏感字段

日志中的手机号、邮箱需要加密存储。对称加密的意思是：加密和解密用同一个密钥。

Node.js 推荐用 AES-GCM 模式。GCM（Galois/Counter Mode）不仅加密数据，还同时生成认证标签（authentication tag），能检测密文是否被篡改：

```js
import { createCipheriv, createDecipheriv, randomBytes } from "node:crypto";

function encrypt(plaintext, key) {
  // GCM 需要 12 字节的随机 IV
  const iv = randomBytes(12);

  const cipher = createCipheriv("aes-256-gcm", key, iv);
  const encrypted = Buffer.concat([
    cipher.update(plaintext, "utf-8"),
    cipher.final(),
  ]);
  // authTag 用于验证密文完整性
  const authTag = cipher.getAuthTag();

  // IV 和 authTag 需要和密文一起存储（它们不保密，但不能丢）
  return Buffer.concat([iv, authTag, encrypted]);
}

function decrypt(combined, key) {
  const iv = combined.subarray(0, 12);
  const authTag = combined.subarray(12, 28);
  const encrypted = combined.subarray(28);

  const decipher = createDecipheriv("aes-256-gcm", key, iv);
  decipher.setAuthTag(authTag);

  return decipher.update(encrypted) + decipher.final("utf-8");
}
```

为什么推荐 GCM 而不是 CBC？CBC 只加密不认证——攻击者可以在不知道密钥的情况下修改密文（padding oracle attack）。GCM 的认证标签能防止这种情况。如果你还在用 CBC，建议迁移到 GCM。

密钥必须是 32 字节（AES-256）。生成密钥：

```js
const key = randomBytes(32); // 每次运行都不同，需要持久化保存
```

**密钥管理是加密系统中最难的部分**。密钥不能硬编码在代码里（代码进了 Git 就泄露了），不能放在环境变量里（`ps aux` 能看到）。生产环境通常用专门的密钥管理服务（AWS KMS、HashiCorp Vault），或者至少用一个独立的密钥文件（权限设为 600）。

## HMAC：API 签名认证

日志查询 API 需要认证。一种轻量级的方式是 HMAC 签名——客户端和服务端共享一个密钥，客户端用密钥对请求签名，服务端验证签名。

```js
import { createHmac } from "node:crypto";

// 客户端：生成签名
function signRequest(method, path, timestamp, body, secretKey) {
  const payload = `${method}\n${path}\n${timestamp}\n${body}`;
  return createHmac("sha256", secretKey).update(payload).digest("hex");
}

// 使用
const timestamp = Date.now().toString();
const body = JSON.stringify({ level: "ERROR" });
const signature = signRequest(
  "GET",
  "/api/logs",
  timestamp,
  body,
  "your-secret-key"
);

// 发送请求时带上签名
// headers: { 'X-Timestamp': timestamp, 'X-Signature': signature }
```

```js
// 服务端：验证签名
function verifyRequest(req, res, secretKey) {
  const timestamp = req.headers['x-timestamp']
  const signature = req.headers['x-signature']

  if (!timestamp || !signature) {
    res.writeHead(401, { 'Content-Type': 'application/json' })
    res.end(JSON.stringify({ error: 'Missing authentication headers' }))
    return false
  }

  // 防止重放攻击：时间戳超过 5 分钟拒绝
  if (Date.now() - parseInt(timestamp) > 5 * 60_000) {
    res.writeHead(401, { 'Content-Type': 'application/json' })
    res.end(JSON.stringify({ error: 'Request expired' }))
    return false
  }

  // 重新计算签名并对比
  const payload = `${req.method}\n${new URL(req.url, 'http://localhost').pathname}\n${timestamp}\n`
  const expected = createHmac('sha256', secretKey).update(payload).digest('hex')

  // 使用 timingSafeEqual 防止时序攻击
  const { timingSafeEqual } = await import('node:crypto')
  const sigBuf = Buffer.from(signature, 'hex')
  const expectedBuf = Buffer.from(expected, 'hex')

  if (sigBuf.length !== expectedBuf.length || !timingSafeEqual(sigBuf, expectedBuf)) {
    res.writeHead(401, { 'Content-Type': 'application/json' })
    res.end(JSON.stringify({ error: 'Invalid signature' }))
    return false
  }

  return true
}
```

两个安全细节：

1. **时间戳检查**防止重放攻击——攻击者截获一个合法请求后，不能无限次重放。
2. **`timingSafeEqual`** 防止时序攻击——普通的 `===` 比较在第一个不匹配的字符就返回，攻击者可以通过响应时间推断出签名的正确前缀。`timingSafeEqual` 无论是否匹配都花同样的时间。

## crypto.randomBytes：安全随机数

永远不要用 `Math.random()` 生成安全相关的随机数——它是伪随机的，理论上可以被预测。

```js
import { randomBytes, randomUUID } from "node:crypto";

// 生成安全的随机字节
const token = randomBytes(32).toString("hex"); // 64 字符的随机字符串

// 生成 UUID v4
const id = randomUUID(); // '1b9d6bcd-bbfd-4b2d-9b5d-ab8dfbbd4bed'
```

`crypto.randomBytes()` 使用操作系统的密码学安全随机数生成器（`/dev/urandom` 或 `CryptGenRandom`），不可预测。

## 什么时候用什么

日志服务的三层安全对应三种不同的加密场景：

| 场景                 | 技术             | 特点                       |
| -------------------- | ---------------- | -------------------------- |
| 存储加密（敏感字段） | AES-GCM 对称加密 | 加解密快，需要安全管理密钥 |
| 传输认证（API 签名） | HMAC 签名        | 轻量级，不需要加密整个请求 |
| 文件校验（防篡改）   | SHA-256 哈希     | 单向不可逆，验证数据完整性 |

加密是双刃剑。加密后的数据不能被搜索——你无法在加密后的手机号上做 `WHERE phone = '13800138000'`。如果需要搜索加密字段，通常的方案是：对搜索条件做同样的加密，然后用加密后的值做匹配（确定性加密），或者使用可搜索加密（Searchable Encryption，实现复杂度高得多）。
