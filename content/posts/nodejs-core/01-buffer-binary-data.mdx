你正在做一个日志处理服务，上游系统发来的日志不是 JSON 文本，而是一串二进制数据。你用 `fs.readFile` 读出来一看，得到的是一堆 `<Buffer 00 00 00 1a 45 52 52 ...>` 这种东西。怎么把它变成可读的日志内容？

在前端开发中，我们几乎不需要直接操作二进制数据。最多处理一下 `FormData`，或者在 Canvas 上画个图。但在 Node.js 的世界里，文件读写、网络通信、加密解密，全都是二进制操作。不理解 Buffer，你连一个完整的 HTTP 请求体都读不明白。

## 为什么 Node.js 需要 Buffer

JavaScript 的字符串（V8 引擎中的 `v8::String`）使用 UTF-16 编码，每个字符占 2 或 4 字节。这个设计对处理文本没问题，但二进制数据中有大量 0x00 之类的无效 UTF-16 序列——V8 遇到这些数据会直接报错或者丢信息。

更关键的问题是性能。JavaScript 字符串是**不可变的**（immutable），每次拼接、截取都会创建新的字符串对象，V8 的垃圾回收器要频繁清理这些临时对象。当你从网络连接中持续接收数据时，这种开销是不可接受的。

Buffer 就是为了解决这个问题而生的：它是一块**固定大小的原始内存区域**，直接操作字节，不走 V8 的字符串机制。

```
JavaScript 字符串                    Buffer
┌──────────────┐                   ┌──────────────┐
│ V8 管理的堆  │                   │  堆外内存     │
│ UTF-16 编码  │                   │  原始字节     │
│ 不可变       │                   │  可修改       │
│ GC 管理      │                   │  手动/池化    │
└──────────────┘                   └──────────────┘
```

## Buffer 的内存模型

### 堆外内存

Buffer 的数据存储在 V8 堆之外的内存中。这意味着：

- Buffer 的大小不受 V8 堆内存限制（`--max-old-space-size` 不影响它）
- Buffer 的分配和释放不走 V8 的 GC，而是由 Node.js 自己管理
- 你可以创建很大的 Buffer（几百 MB），不会触发 V8 的垃圾回收

这也是为什么 Node.js 能处理大文件——文件数据存在 Buffer 里，不占 V8 的堆空间。

### 8KB 池化机制

当你创建小于 4KB 的 Buffer 时，Node.js 不会每次都向操作系统申请内存，而是从一个预分配的 8KB 内存池中切一块出来：

```js
// 创建 100 字节的 Buffer
const buf = Buffer.alloc(100);
// 实际上从 8KB 池中切了 100 字节，池中还剩 8192 - 100 = 8092 字节

// 再创建 200 字节的 Buffer
const buf2 = Buffer.alloc(200);
// 从同一个池中再切 200 字节
```

这个设计减少了频繁的系统调用（`malloc`/`free`）。池化只在创建小于 4KB 的 Buffer 时生效，大于等于 4KB 的 Buffer 直接走 `malloc`。

### alloc vs allocUnsafe

创建 Buffer 有三种方式，选错会有安全风险：

```js
// 方式 1：alloc —— 填充零，安全但稍慢
const safe = Buffer.alloc(256);
// 所有字节都是 0x00

// 方式 2：allocUnsafe —— 不填充，快但可能泄露旧数据
const fast = Buffer.allocUnsafe(256);
// 内存中可能残留上一次使用的数据（比如密码、密钥等）

// 方式 3：from —— 从已有数据创建
const fromString = Buffer.from("hello");
const fromArray = Buffer.from([1, 2, 3, 4]);
```

`allocUnsafe` 为什么快？因为它跳过了内存清零的步骤。但这也意味着 Buffer 中可能包含上一次使用的敏感数据。在处理用户密码、API 密钥等场景下，必须用 `alloc`。

一个容易犯的错：用 `Buffer.allocUnsafe` 创建后忘记手动填充，直接把数据发给了客户端。对方可能从你的 Buffer 残留数据中读到其他用户的信息。

## Buffer 与浏览器 ArrayBuffer 的关系

如果你写过前端，可能见过 `ArrayBuffer` 和 `Uint8Array`。它们和 Buffer 是什么关系？

Buffer 本质上就是 `Uint8Array` 的子类，但有一些额外的 Node.js 特有能力：

```js
buf instanceof Uint8Array; // true
buf instanceof Buffer; // true
```

两者的关键区别：

| 特性       | Buffer            | ArrayBuffer + TypedArray |
| ---------- | ----------------- | ------------------------ |
| 所在环境   | Node.js           | 浏览器 + Node.js         |
| 内存管理   | 池化 + 堆外       | 直接分配                 |
| 字符串转换 | 内置 `toString()` | 需要手动用 TextDecoder   |
| 拼接       | `Buffer.concat()` | 需要手动拷贝             |

需要互转时，用 `Buffer.from()` 和 `buffer.buffer`：

```js
// ArrayBuffer → Buffer
const ab = new ArrayBuffer(16);
const buf = Buffer.from(ab);

// Buffer → ArrayBuffer（注意要用 byteOffset 和 byteLength 限定范围）
const view = new Uint8Array(buf.buffer, buf.byteOffset, buf.byteLength);
```

注意 `buf.buffer` 返回的是整个内存池（可能是 8KB），不是 Buffer 自身的数据。所以要用 `byteOffset` 和 `byteLength` 来限定范围，否则 `Uint8Array` 会包含池中其他数据。

## 字符编码处理

Buffer 的 `toString()` 支持多种编码：

```js
const buf = Buffer.from("你好世界", "utf-8");

buf.toString("utf-8"); // '你好世界'
buf.toString("hex"); // 'e4bda0e5a5bde4b896e7界的...'
buf.toString("base64"); // '5L2g5aW95LiW55WM...'
buf.toString("ascii"); // 乱码——ascii 不支持中文
```

UTF-8 中，英文字符占 1 字节，中文字符占 3 字节。这直接影响了 Buffer 的长度计算：

```js
Buffer.byteLength("hello"); // 5
Buffer.byteLength("你好"); // 6（两个中文字符，每个 3 字节）
Buffer.byteLength("hello你好"); // 11
```

一个常见的 bug 是用 `string.length` 来预估 Buffer 大小——这在包含多字节字符时会出错。永远用 `Buffer.byteLength()` 来计算。

## 大端序与小端序

处理多字节整数时，字节序是一个绕不开的问题。假设你要把整数 `0x01020304` 存入 Buffer，有两种方式：

```
大端序（Big Endian，网络字节序）：
┌───┬───┬───┬───┐
│01 │02 │03 │04 │  高位字节在前
└───┴───┴───┴───┘

小端序（Little Endian，x86 架构）：
┌───┬───┬───┬───┐
│04 │03 │02 │01 │  低位字节在前
└───┴───┴───┴───┘
```

网络协议统一使用大端序。Buffer 的读写方法默认也是大端序：

```js
const buf = Buffer.alloc(4);
buf.writeUInt32BE(0x01020304, 0); // 大端序写入
buf[0]; // 0x01
buf[3]; // 0x04

const value = buf.readUInt32BE(0); // 0x01020304
```

如果数据来自本机（x86 架构），可能需要用 `writeUInt32LE` / `readUInt32LE`。读写两端的字节序必须一致，否则数据就是乱的。

## 实战：解析自定义二进制日志协议

回到开头的问题。上游系统发来的日志用的是自定义的二进制协议，帧格式如下：

```
┌──────────┬──────────┬──────────────┬──────────────────┐
│ 4 bytes  │ 1 byte   │ 8 bytes      │ N bytes          │
│ 总长度   │ 日志级别 │ 时间戳(ms)   │ 日志内容(UTF-8)  │
│ (BE)     │          │ (BE)         │                  │
└──────────┴──────────┴──────────────┴──────────────────┘
```

比如一条 ERROR 日志可能是这样的字节序列：

```
00 00 00 3A  03  00 00 01 7F 3A 8C 90 00  {"traceId":"trace_a1x9k2m","message":"Connection timeout..."}
```

解析代码：

```js
function parseLogFrame(data) {
  if (data.length < 13) {
    throw new Error(
      `帧长度不足：至少需要 13 字节头部，实际 ${data.length} 字节`
    );
  }

  let offset = 0;

  // 4 字节总长度（大端序）
  const totalLength = data.readUInt32BE(offset);
  offset += 4;

  if (data.length < totalLength) {
    throw new Error(
      `帧不完整：期望 ${totalLength} 字节，实际 ${data.length} 字节`
    );
  }

  // 1 字节日志级别
  const levelMap = { 0: "DEBUG", 1: "INFO", 2: "WARN", 3: "ERROR", 4: "FATAL" };
  const level = data[offset];
  offset += 1;

  // 8 字节时间戳（大端序，毫秒级 Unix 时间戳）
  const timestamp = Number(data.readBigUInt64BE(offset));
  offset += 8;

  // 剩余字节为日志内容（UTF-8）
  const content = data.subarray(offset, totalLength).toString("utf-8");

  return {
    level: levelMap[level] ?? `UNKNOWN(${level})`,
    timestamp: new Date(timestamp).toISOString(),
    content: JSON.parse(content),
  };
}
```

注意几个细节：

- 用 `readBigUInt64BE` 而不是 `readUInt32BE`，因为 8 字节时间戳可能超过 `Number.MAX_SAFE_INTEGER`
- 用 `subarray` 而不是 `slice`——`subarray` 返回的是原 Buffer 的视图，不拷贝数据，性能更好
- 先检查总长度再读取内容，避免帧不完整时读到垃圾数据

### 处理 TCP 粘包

在实际的网络传输中，你不能假设一次 `data` 事件就收到一个完整的帧。TCP 是流式协议，可能把两个帧粘在一起发过来，也可能把一个帧拆成两半。

处理方式是维护一个接收缓冲区，每次收到数据后尝试从中提取完整的帧：

```js
class FrameReader {
  constructor(onFrame) {
    this.buffer = Buffer.alloc(0);
    this.onFrame = onFrame;
  }

  feed(data) {
    // 拼接到缓冲区
    this.buffer = Buffer.concat([this.buffer, data]);

    // 循环提取完整帧
    while (this.buffer.length >= 4) {
      const frameLength = this.buffer.readUInt32BE(0);

      if (this.buffer.length < frameLength) {
        // 数据不够，等下次 feed
        break;
      }

      // 提取完整帧
      const frame = this.buffer.subarray(0, frameLength);
      this.buffer = this.buffer.subarray(frameLength);

      try {
        this.onFrame(parseLogFrame(frame));
      } catch (err) {
        console.error("帧解析失败:", err.message);
      }
    }
  }
}
```

每次调用 `Buffer.concat` 都会创建新的 Buffer——如果数据量很大，频繁 concat 会有性能问题。更高效的实现可以用链表结构管理缓冲区块，不过对于大多数场景，concat 已经够用了。TCP 粘包的完整处理我们会在 [第 7 篇：网络编程](/posts/nodejs-core/07-tcp-udp-network-programming) 中深入讨论。

## 什么时候用 Buffer，什么时候用字符串

日常开发中，一个实用的判断标准：

**用 Buffer 的场景**：文件读写、网络通信、加密解密、图片/音视频处理、自定义协议解析。这些场景本质上是处理"字节"而不是"字符"。

**用字符串的场景**：配置文件解析、模板渲染、日志文本格式化、API 响应构造。这些场景处理的是"文本"。

**两者混合的场景**：HTTP 请求体可能是 JSON 字符串，但传输层是二进制的 TCP 连接。Node.js 的 `http` 模块帮你做了这层转换——你用字符串处理请求体，底层用 Buffer 做网络传输。

理解 Buffer 不是为了每天手写二进制协议解析，而是让你在遇到"数据不对"、"编码乱码"、"内存暴涨"这类问题时，知道该往哪个方向排查。下一篇我们会讲 Stream——当你需要处理几百 MB 甚至几个 GB 的数据时，光靠 Buffer 就不够了，需要用流式的方式一段一段地处理。
