前三篇建立了 Node.js 的核心认知，代码都是单文件示例。现在我们开始构建日志处理服务，第一个要解决的问题就是：日志存哪里、怎么存、存满了怎么办。

`fs` 模块是 Node.js 和文件系统交互的桥梁，但它有三套 API 风格、文件描述符要手动关闭、跨平台路径要处理……光一个"把日志写入文件"就能踩一排坑。

## 三套 API 风格

Node.js 的 `fs` 模块提供了三种调用方式，新手很容易搞混：

```js
import fs from "node:fs";
import fsPromises from "node:fs/promises";

// 1. 同步 API —— 阻塞事件循环，只在启动脚本或 CLI 工具中用
const data = fs.readFileSync("config.json", "utf-8");

// 2. 回调 API —— 老风格，callback hell
fs.readFile("config.json", "utf-8", (err, data) => {
  if (err) throw err;
  console.log(data);
});

// 3. Promise API —— 现代写法，推荐使用
const data = await fsPromises.readFile("config.json", "utf-8");
```

什么时候用哪种？基本原则：**在事件循环中的代码永远用 Promise API**。同步 API 会阻塞事件循环（[第 3 篇](/posts/nodejs-core/03-event-loop-deep-dive)讲过），导致所有请求卡住。同步 API 只适合在程序启动时读配置文件这类只执行一次的场景。

## 文件描述符：你打开的门忘了关

每次调用 `fs.open()` 都会从操作系统获得一个文件描述符（fd）。fd 是有限的系统资源，不关闭就会泄露：

```js
import { open, close } from "node:fs/promises";

// 每次循环都打开文件但从不关闭 —— fd 泄露
for (let i = 0; i < 10000; i++) {
  const fd = await open("log.txt", "r");
  const stat = await fd.stat();
  // 忘了 close(fd)
}
// 报错：EMFILE: too many open files
```

用 `using` 关键字（Node.js 20+）可以自动管理 fd 的生命周期：

```js
// Node.js 20+ 支持 using 声明，离开作用域自动调用 [Symbol.asyncDispose]
{
  using fd = await open("log.txt", "r");
  const stat = await fd.stat();
} // 这里自动 close
```

如果不支持 `using`，用 `try/finally`：

```js
const fd = await open("log.txt", "r");
try {
  const stat = await fd.stat();
} finally {
  await fd.close();
}
```

实际上，大多数时候你不需要手动管理 fd。`fsPromises.readFile()` 和 `createReadStream()` 会在内部打开和关闭文件。只有在你需要精细控制文件操作（比如读写交替、文件锁）时才需要手动 `open`。

## 实战：日志存储层

日志服务的存储需求：按日期分目录存储，支持流式追加写入，超过 100MB 自动轮转。

### 按日期分目录存储

```js
import { mkdir, appendFile } from "node:fs/promises";
import { join, dirname } from "node:path";
import { fileURLToPath } from "node:url";

const __dirname = dirname(fileURLToPath(import.meta.url));

class LogStore {
  constructor(baseDir = join(__dirname, "../logs")) {
    this.baseDir = baseDir;
  }

  async _ensureDir(filePath) {
    await mkdir(dirname(filePath), { recursive: true });
  }

  getLogPath(date = new Date()) {
    const y = date.getFullYear();
    const m = String(date.getMonth() + 1).padStart(2, "0");
    const d = String(date.getDate()).padStart(2, "0");
    return join(this.baseDir, `${y}-${m}`, `${d}.log`);
  }
}
```

注意 `mkdir` 的 `{ recursive: true }` 选项——没有这个选项，如果父目录不存在就会报错。日志服务按日期分目录，每天的目录都需要自动创建。

### 流式追加写入

对于高频写入的日志，每次都 `appendFile` 打开再关闭文件太浪费了。用 `createWriteStream` 的 append 模式保持文件打开：

```js
import { createWriteStream, createReadStream, stat } from "node:fs";

class LogWriter {
  constructor(filePath) {
    this.filePath = filePath;
    this.stream = null;
    this.bytesWritten = 0;
  }

  async start() {
    // 确保目录存在
    await mkdir(dirname(this.filePath), { recursive: true });

    // 获取已有文件大小（用于日志轮转判断）
    try {
      const info = await stat(this.filePath);
      this.bytesWritten = info.size;
    } catch {
      this.bytesWritten = 0;
    }

    // append 模式：如果文件存在则追加，不存在则创建
    this.stream = createWriteStream(this.filePath, { flags: "a" });

    // 处理写入错误
    this.stream.on("error", (err) => {
      console.error(`日志写入失败 [${this.filePath}]:`, err.message);
    });
  }

  write(logLine) {
    const data = logLine + "\n";
    this.bytesWritten += Buffer.byteLength(data, "utf-8");
    return this.stream.write(data);
  }

  async stop() {
    if (this.stream) {
      // end() 会先写入缓冲区中的数据，然后关闭文件
      await new Promise((resolve, reject) => {
        this.stream.end(resolve);
        this.stream.on("error", reject);
      });
    }
  }
}
```

### 日志轮转

日志文件超过 100MB 时，需要分割并压缩归档：

```js
import { createReadStream } from "node:fs";
import { createGzip } from "node:zlib";
import { pipeline } from "node:stream/promises";
import { rename, unlink } from "node:fs/promises";

class LogRotator {
  constructor(maxSize = 100 * 1024 * 1024) {
    this.maxSize = maxSize;
  }

  async shouldRotate(writer) {
    return writer.bytesWritten >= this.maxSize;
  }

  async rotate(filePath) {
    const timestamp = new Date().toISOString().replace(/[:.]/g, "-");
    const archivedPath = filePath.replace(".log", `-${timestamp}.log.gz`);

    // 关闭当前写入流
    // 压缩归档
    await pipeline(
      createReadStream(filePath),
      createGzip(),
      createWriteStream(archivedPath)
    );

    // 删除原文件，下次写入会自动创建新文件
    await unlink(filePath);

    return archivedPath;
  }
}
```

把写入和轮转组合起来：

```js
const store = new LogStore();
const writer = new LogWriter(store.getLogPath());
const rotator = new LogRotator();

await writer.start();

// 定期检查是否需要轮转
setInterval(async () => {
  if (await rotator.shouldRotate(writer)) {
    await writer.stop();
    const archived = await rotator.rotate(writer.filePath);
    console.log(`日志已轮转: ${archived}`);
    await writer.start(); // 重新开始写入新文件
  }
}, 60_000); // 每分钟检查一次
```

## fs.watch vs fs.watchFile

日志服务可能需要监听配置文件变化，实现热更新。Node.js 提供了两种文件监听方式：

```js
import { watch, watchFile } from "node:fs";

// fs.watch —— 基于 OS 事件（inotify/FSEvents），效率高
watch("config.json", (eventType, filename) => {
  console.log(`${filename} 发生了 ${eventType} 事件`);
});

// fs.watchFile —— 轮询文件状态，跨平台一致但有延迟
watchFile("config.json", { interval: 2000 }, (curr, prev) => {
  if (curr.mtime !== prev.mtime) {
    console.log("文件被修改了");
  }
});
```

| 特性         | `fs.watch`                    | `fs.watchFile`     |
| ------------ | ----------------------------- | ------------------ |
| 实现方式     | OS 原生事件                   | 轮询 stat          |
| 性能         | 高（事件驱动）                | 低（定时轮询）     |
| 跨平台一致性 | 差（事件名不同）              | 好                 |
| 事件准确性   | 可能重复触发或丢失            | 可靠（基于 mtime） |
| 文件名回调   | macOS 上 filename 可能为 null | 始终有值           |

实际项目中，`fs.watch` 的跨平台问题比较头疼。如果只是监听配置文件变化，`fs.watchFile` 虽然效率低一些但更可靠。对于高频率的文件监听场景（比如监听整个日志目录），考虑用 `chokidar` 这个第三方库，它封装了各平台的差异。

## path 模块：跨平台路径处理

一个容易被忽略的问题：路径分隔符在 Windows 和 Unix 上不同。`join`、`resolve`、`dirname`、`extname` 这些方法帮你处理了跨平台差异：

```js
import { join, resolve, dirname, extname } from "node:path";

// join —— 拼接路径，自动处理分隔符
join("logs", "2026-03", "15.log");
// Unix: 'logs/2026-03/15.log'
// Windows: 'logs\\2026-03\\15.log'

// resolve —— 解析为绝对路径
resolve("logs", "15.log");
// '/Users/you/project/logs/15.log'

// dirname / extname —— 提取目录和扩展名
dirname("/logs/2026-03/15.log"); // '/logs/2026-03'
extname("/logs/2026-03/15.log"); // '.log'
```

永远用 `path.join()` 而不是字符串拼接来构建路径。`'logs' + '/' + file` 在 Windows 上会产生混合分隔符的路径，某些 API 可能处理不了。
