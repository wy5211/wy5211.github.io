你的项目里一半用 `require()`，一半用 `import`。有些 npm 包只支持 CJS，有些只支持 ESM。你把 `__dirname` 拷到一个 ESM 文件里，一运行就报 `ReferenceError: __dirname is not defined`。更崩溃的是，同一个包在不同文件中 `import` 出来的是不同实例——这就是双包地狱。

Node.js 的模块系统是你日常写代码接触最多的基础设施，但大多数人对它的理解停留在"能用就行"。今天我们把这块搞清楚。

## CommonJS：运行时加载

CJS 是 Node.js 最老的模块系统。`require()` 的行为看起来简单，但背后有不少机制值得了解。

### require 的执行流程

当你写 `require('./parser')` 时，Node.js 做了这些事：

```js
// 简化版的 require 实现
function require(id) {
  // 1. 检查缓存
  if (Module._cache[id]) return Module._cache[id].exports;

  // 2. 解析文件路径（处理目录下的 index.js、package.json 中的 main 字段等）
  const filename = Module._resolveFilename(id);

  // 3. 创建模块实例
  const module = new Module(filename);

  // 4. 放入缓存
  Module._cache[id] = module;

  // 5. 用模块包装函数执行文件内容
  const wrapper = `
    (function(exports, require, module, __filename, __dirname) {
      ${fs.readFileSync(filename, "utf-8")}
    })
  `;
  wrapper.call(
    module.exports,
    module.exports,
    require,
    module,
    filename,
    dirname
  );

  return module.exports;
}
```

关键点在**第 1 步和第 4 步**：模块在第一次 `require` 后就被缓存了，后续的 `require` 直接返回缓存。这就是为什么模块级别的代码只执行一次。

这个包装函数也解释了为什么每个 CJS 文件都有 `exports`、`require`、`module`、`__filename`、`__dirname` 这五个变量——它们不是全局变量，而是包装函数的参数。

### module.exports vs exports

一个高频坑：

```js
// 错误：exports 被重新赋值，不再指向 module.exports
exports = {
  parse() {
    /* ... */
  },
};

// 正确：给 module.exports 赋新值
module.exports = {
  parse() {
    /* ... */
  },
};
```

原因：包装函数开头有 `var exports = module.exports`。`exports` 只是 `module.exports` 的引用。当你给 `exports` 赋新值时，只是让 `exports` 指向了新对象，但 `module.exports` 还是指向原来的空对象。`require` 返回的是 `module.exports`，所以你赋给 `exports` 的东西被丢弃了。

安全用法：只往 `exports` 上添加属性，不重新赋值。如果需要导出一个完整的对象或类，用 `module.exports`。

## ESM：编译时静态分析

ESM（ECMAScript Modules）的设计思路和 CJS 完全不同。

### 静态分析

`import` 语句在代码执行之前就被解析了。这意味着：

```js
// CJS：可以条件加载
if (condition) {
  const parser = require("./parser");
}

// ESM：import 必须在顶层，不能条件加载
import parser from "./parser.js"; // OK
if (condition) {
  import parser from "./parser.js"; // SyntaxError!
}
```

静态分析让 V8 可以在编译阶段就确定模块的依赖关系，好处是：

- **Tree Shaking**：打包工具可以删除未使用的导出
- **更快的启动**：不需要运行时解析依赖链
- **更好的类型推断**：TypeScript 可以更准确地推断类型

需要条件加载时，用动态 `import()`：

```js
if (condition) {
  const { parse } = await import("./parser.js");
}
```

### 只读绑定

ESM 的 `import` 创建的是对原始导出的**只读绑定**，而不是值的拷贝：

```js
// counter.js
export let count = 0;
export function increment() {
  count++;
}

// main.js
import { count, increment } from "./counter.js";
console.log(count); // 0
increment();
console.log(count); // 1 — 你拿到的是最新的值
```

CJS 不这样：

```js
// counter.js
let count = 0;
module.exports = { count, increment: () => count++ };

// main.js
const { count, increment } = require("./counter.js");
console.log(count); // 0
increment();
console.log(count); // 还是 0 — 你拿到的是导出时的拷贝
```

### **dirname 和 **filename 在 ESM 中不存在

ESM 没有 CJS 的包装函数，所以 `__dirname` 和 `__filename` 不可用。替代方案：

```js
import { fileURLToPath } from "node:url";
import { dirname } from "node:path";

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
```

`import.meta.url` 返回的是 `file://` 协议的 URL（如 `file:///Users/you/project/src/parser.js`），需要用 `fileURLToPath` 转换成本地路径。

## CJS 和 ESM 的互操作

Node.js 允许 CJS 和 ESM 混用，但有一些规则：

**ESM 可以 import CJS**：没问题。`import pkg from 'express'` 会把 `module.exports` 作为默认导入。

**CJS 不能 require ESM**（同步方式不行）：因为 ESM 是异步加载的，`require()` 是同步的。替代方案：

```js
// CJS 中动态加载 ESM
const { parse } = await import("./parser.mjs");
```

注意这要求外层函数是 `async` 的。

**文件扩展名**：ESM 中 `import` 必须写完整的文件扩展名（包括 `.js`），而 CJS 的 `require` 可以省略：

```js
// CJS
const parser = require("./parser"); // OK，自动尝试 .js、.json、.node

// ESM
import parser from "./parser"; // Error! 必须写扩展名
import parser from "./parser.js"; // OK
```

## 双包陷阱

当一个 npm 包同时提供 CJS 和 ESM 两种格式时，可能出现"双包陷阱"：CJS 和 ESM 各自缓存了一份模块实例，导致同一个包被加载了两次。

```js
// cjs-user.js
const pkg = require("some-pkg");
pkg.setData("hello");

// esm-user.js
import pkg from "some-pkg";
console.log(pkg.getData()); // undefined — 这是另一个实例！
```

如果你的包有状态（比如单例、缓存），双包陷阱会导致状态不一致。解决方案：

1. **只提供 ESM**：在 `package.json` 中设置 `"type": "module"`。简单粗暴，但 CJS 用户需要用 `await import()` 动态加载。
2. **只提供 CJS**：传统方式，ESM 用户通过默认导入使用。
3. **用 exports 条件导出**：指定不同环境使用哪个入口文件，避免重复加载。

### exports 条件导出

```json
{
  "name": "log-parser",
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.js"
    }
  }
}
```

这样 `import` 会走 ESM 入口，`require` 会走 CJS 入口。但仍然可能有双包陷阱——如果你的 ESM 和 CJS 代码互相引用，可能形成两套独立的模块图。

最安全的做法是：用 CJS 编写，然后用 `esbuild` 或 `rollup` 打包出一个 ESM 的 wrapper，wrapper 内部用动态 `import()` 加载 CJS 入口。这样不管从哪个入口进来，最终都使用同一个模块实例。

## 循环依赖

CJS 和 ESM 处理循环依赖的方式不同。

CJS 的处理方式是"拿到什么就用什么"：

```js
// a.js
const b = require("./b.js");
console.log("a 拿到的 b:", b.value); // undefined
module.exports = { value: "from a" };

// b.js
const a = require("./a.js");
console.log("b 拿到的 a:", a.value); // 'from a'
module.exports = { value: "from b" };
```

执行 `require('./a.js')` 时：a 开始执行，require b，b 开始执行，require a，此时 a 的 `module.exports` 还是空对象（因为 a 还没执行完），所以 b 拿到的是 `{}`。然后 b 执行完导出 `{ value: 'from b' }`，回到 a 继续执行，a 拿到的 b 是完整的 `{ value: 'from b' }`。

ESM 的处理方式是"只读绑定，但可能还没赋值"：

```js
// a.js
import { value as bValue } from "./b.js";
console.log("a 拿到的 b:", bValue); // ReferenceError: Cannot access 'value' before initialization

// b.js
import { value as aValue } from "./a.js";
console.log("b 拿到的 a:", aValue);
export const value = "from b";
```

ESM 在模块完全加载之前不允许访问导出——如果两个模块互相依赖且都试图在加载阶段访问对方的导出，会抛出 `ReferenceError`。

实践中，循环依赖通常是架构问题的信号。如果 a 和 b 互相依赖，说明它们的职责划分有问题，应该抽取一个公共模块 c 让两者都依赖 c。
