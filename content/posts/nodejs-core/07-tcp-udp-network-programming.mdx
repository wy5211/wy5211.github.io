HTTP 很好，但不是所有通信都需要 HTTP。日志采集端只需要往服务器推数据，不需要请求-响应模型。一个自定义的 TCP 协议比 HTTP 更轻量、更快。服务发现和心跳检测只需要定期发一个 UDP 包，TCP 的连接建立和维护都显得多余。

[上一篇](/posts/nodejs-core/06-http-https-core-mechanisms)我们用 HTTP 构建了查询 API。这篇往下一层，直接操作 TCP 和 UDP。

## TCP vs UDP：选哪个

一句话总结：TCP 是打电话（先建立连接，保证对方听到），UDP 是寄明信片（直接发出去，不保证对方收到）。

| 特性     | TCP                  | UDP                      |
| -------- | -------------------- | ------------------------ |
| 连接     | 面向连接（三次握手） | 无连接                   |
| 可靠性   | 保证送达、按序到达   | 不保证送达、不保证顺序   |
| 速度     | 相对慢（协议开销大） | 快（几乎没有协议开销）   |
| 数据边界 | 流式（无边界）       | 数据报（每条消息有边界） |
| 适用场景 | 文件传输、API 请求   | 实时通信、服务发现、DNS  |

日志采集选 TCP 是因为日志不能丢——每条日志都很重要，需要保证送达。服务发现选 UDP 是因为丢一两个心跳包无所谓，下次还会再发。

## TCP 服务端：日志采集协议

用 `net` 模块实现一个 TCP 日志采集服务，使用我们在 [第 1 篇](/posts/nodejs-core/01-buffer-binary-data) 设计的二进制帧格式：

```
[4字节总长度][1字节级别][8字节时间戳][N字节日志内容]
```

### 基本服务端

```js
import { createServer } from "node:net";

const clients = new Set();

const server = createServer((socket) => {
  const clientId = `${socket.remoteAddress}:${socket.remotePort}`;
  console.log(`采集端连接: ${clientId}`);
  clients.add(socket);

  // 每个连接维护自己的缓冲区（处理 TCP 粘包）
  const buffers = [];

  socket.on("data", (data) => {
    buffers.push(data);
    // 尝试从缓冲区中提取完整帧
    processBuffer();
  });

  socket.on("error", (err) => {
    console.error(`连接异常 [${clientId}]:`, err.message);
  });

  socket.on("close", () => {
    console.log(`采集端断开: ${clientId}`);
    clients.delete(socket);
  });

  function processBuffer() {
    // 合并缓冲区
    const combined = Buffer.concat(buffers);
    buffers.length = 0;

    let offset = 0;
    while (offset + 4 <= combined.length) {
      const frameLength = combined.readUInt32BE(offset);

      if (offset + frameLength > combined.length) {
        // 帧不完整，把剩余数据放回缓冲区
        buffers.push(combined.subarray(offset));
        return;
      }

      const frame = combined.subarray(offset, offset + frameLength);
      offset += frameLength;

      // 处理完整帧
      handleLogFrame(frame);
    }

    // 处理剩余的不完整数据
    if (offset < combined.length) {
      buffers.push(combined.subarray(offset));
    }
  }
});

function handleLogFrame(frame) {
  // 复用第 1 篇的解析逻辑
  const level = frame[4];
  const levelMap = { 0: "DEBUG", 1: "INFO", 2: "WARN", 3: "ERROR", 4: "FATAL" };
  const timestamp = Number(frame.readBigUInt64BE(5));
  const content = frame.subarray(13).toString("utf-8");

  console.log(
    `[${levelMap[level]}] ${new Date(timestamp).toISOString()} ${content}`
  );
}

server.listen(9090, () => {
  console.log("TCP 日志采集服务已启动: 0.0.0.0:9090");
});
```

注意这里处理 TCP 粘包的方式和 [第 1 篇](/posts/nodejs-core/01-buffer-binary-data) 的 `FrameReader` 类似，但这里是基于 `data` 事件的流式处理。每次 `data` 事件收到的数据可能包含多个完整帧，也可能只包含半个帧。

### TCP 客户端

```js
import { connect } from "node:net";

const client = connect(9090, "127.0.0.1", () => {
  console.log("已连接到采集服务");

  // 构造一条日志帧
  const content = JSON.stringify({
    traceId: "trace_a1x9k2m",
    service: "payment-gateway",
    message: "Connection timeout to database",
  });
  const contentBuf = Buffer.from(content, "utf-8");

  const frame = Buffer.alloc(13 + contentBuf.length);
  frame.writeUInt32BE(frame.length, 0); // 总长度
  frame[4] = 3; // ERROR 级别
  frame.writeBigUInt64BE(BigInt(Date.now()), 5); // 时间戳
  contentBuf.copy(frame, 13); // 日志内容

  client.write(frame);
});

client.on("end", () => {
  console.log("连接已断开");
});
```

### Socket 事件处理

TCP 连接有多个需要处理的事件。忽略任何一个都可能导致资源泄露：

```js
socket.on("data", handleData); // 收到数据
socket.on("error", handleError); // 连接出错
socket.on("close", handleClose); // 连接关闭
socket.on("timeout", handleTimeout); // 超时（需要先设置 socket.setTimeout()）
```

`error` 事件是必须监听的。如果不监听，Node.js 会直接抛出未捕获异常导致进程崩溃。`close` 事件用于清理资源——从客户端集合中移除、释放关联的内存等。

## UDP：服务发现心跳

UDP 没有连接的概念，每次发送都是独立的。这让它非常适合服务发现和心跳场景。

```js
import { createSocket } from "node:dgram";

// 服务端：接收心跳，维护服务注册表
const registry = new Map(); // serviceName → { address, port, lastSeen }

const server = createSocket("udp4");

server.on("message", (msg, rinfo) => {
  const payload = JSON.parse(msg.toString());
  const key = `${payload.service}@${rinfo.address}:${rinfo.port}`;

  registry.set(key, {
    service: payload.service,
    address: rinfo.address,
    port: payload.port,
    lastSeen: Date.now(),
  });

  console.log(`心跳: ${key}`);
});

// 定期清理过期服务（30 秒没心跳的认为下线）
setInterval(() => {
  const now = Date.now();
  for (const [key, entry] of registry) {
    if (now - entry.lastSeen > 30_000) {
      registry.delete(key);
      console.log(`服务下线: ${key}`);
    }
  }
}, 10_000);

server.bind(9999, () => {
  console.log("UDP 心跳服务已启动: 0.0.0.0:9999");
});

// 客户端：定期发送心跳
const client = createSocket("udp4");
const heartbeat = Buffer.from(
  JSON.stringify({
    service: "log-collector",
    port: 9090,
  })
);

setInterval(() => {
  client.send(heartbeat, 9999, "127.0.0.1");
}, 5000);
```

UDP 代码比 TCP 简单得多——没有连接建立、没有粘包问题（每条消息有边界）、没有流的概念。但代价是你不能确定消息是否送达。如果可靠性很重要，要么在应用层实现确认机制，要么用 TCP。

## TCP Keep-Alive

TCP 连接如果长时间没有数据传输，中间的路由器可能会静默断开连接（NAT 超时）。开启 TCP Keep-Alive 可以定期发送探测包保持连接活跃：

```js
socket.setKeepAlive(true, 10000); // 启用 Keep-Alive，初始探测间隔 10 秒
```

Keep-Alive 和应用层心跳是两个层面的东西：Keep-Alive 保证 TCP 连接不被中间设备断开，应用层心跳（如上面的 UDP 心跳）保证服务确实还在正常工作。生产环境两者通常都需要。

HTTP 底层就是 TCP。[上一篇](/posts/nodejs-core/06-http-https-core-mechanisms) 的 `http.createServer` 内部就是创建了一个 TCP 服务器，然后解析 HTTP 协议。理解了 TCP，你就能理解 HTTP 的 Keep-Alive、连接池、超时等机制到底在做什么。
