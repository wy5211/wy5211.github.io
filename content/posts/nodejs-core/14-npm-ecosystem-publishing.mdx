日志服务的二进制协议解析器写得不错，支付团队和数据团队都想复用。你把代码拷过去，但每次修复 bug 都要同步三份代码。是时候把它抽成独立的 npm 包了。

但发 npm 包不是 `npm publish` 一条命令的事。`package.json` 里十几个字段，每个都有讲究。`exports`、`main`、`types` 之间的关系让人困惑。还有 CJS 和 ESM 的兼容问题——[第 4 篇](/posts/nodejs-core/04-module-system-esm-cjs) 讲的双包地狱，发包时会直接面对。

## package.json：那些容易被忽视的字段

一个 npm 包的 `package.json` 和一个应用的 `package.json` 侧重点不同。应用更关注 `scripts`、`dependencies`，而包更关注入口声明和发布控制。

```json
{
  "name": "@yourorg/log-parser",
  "version": "1.2.0",
  "description": "Binary log protocol parser for the log processing service",
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "import": { "types": "./dist/index.d.ts", "default": "./dist/index.js" },
      "require": {
        "types": "./dist/index.d.cts",
        "default": "./dist/index.cjs"
      }
    }
  },
  "files": ["dist", "README.md"],
  "sideEffects": false,
  "engines": { "node": ">=18.0.0" },
  "peerDependencies": {},
  "dependencies": {},
  "devDependencies": {
    "typescript": "^5.0.0",
    "vitest": "^2.0.0"
  }
}
```

几个关键决策点：

**`exports` vs `main`**：`main` 是 Node.js 传统的入口字段，只支持一个文件。`exports` 是现代方式，支持条件导出——可以根据调用方用的是 `import` 还是 `require` 提供不同的文件。两者同时存在是为了向后兼容，新包建议用 `exports` 为主。

**`files`**：控制发布到 npm 时包含哪些文件。不声明 `files` 的话，默认会上传整个项目目录（包括源码、测试、配置文件）。声明 `"files": ["dist"]` 后，只上传 `dist/` 目录。`.gitignore` 不影响发布——即使文件被 gitignore 了，只要在 `files` 里声明就会被包含。

**`sideEffects: false`**：告诉打包工具（Webpack、Vite）这个包没有副作用，可以安全地进行 tree shaking。对于纯工具库，一定要加。

## 依赖分类：dependencies 的三种

npm 有三种依赖声明，搞混了会导致包体积膨胀或者运行时报错：

```json
{
  "dependencies": {
    // 运行时必需的包。用户安装你的包时，这些会被自动安装
    "encoding": "^0.1.13"
  },
  "devDependencies": {
    // 开发时需要的工具。不会被传递给用户
    "typescript": "^5.0.0",
    "vitest": "^2.0.0"
  },
  "peerDependencies": {
    // 期望用户自己安装的包。不会自动安装，缺少时会警告
    "node-fetch": "^3.0.0"
  }
}
```

一个常见的错误：把 TypeScript 编译后的运行时依赖放进了 `devDependencies`。如果你的包在运行时用到了某个库，它必须放在 `dependencies` 里，不管你在源码中是通过什么方式引入的。`devDependencies` 里的东西只在开发你的包时才需要——测试框架、编译器、linter 这些。

`peerDependencies` 适合什么场景？比如你的包是一个日志中间件，需要配合 Express 使用。你不想把 Express 作为 `dependencies`（用户的 Express 版本可能和你锁定的不一样），也不想作为 `devDependencies`（运行时需要）。`peerDependencies` 告诉用户："你需要自己安装 Express"，但不指定具体版本。

## 版本号：语义化版本不是建议，是约定

npm 包的版本号遵循 SemVer（语义化版本）：`MAJOR.MINOR.PATCH`。

- `1.0.0` → `1.0.1`：修 bug，向后兼容。`~1.0.0` 和 `^1.0.0` 都会更新到这个版本。
- `1.0.0` → `1.1.0`：加新功能，向后兼容。`^1.0.0` 会更新，`~1.0.0` 不会。
- `1.0.0` → `2.0.0`：破坏性变更。`^1.0.0` 和 `~1.0.0` 都不会更新。

版本范围符号：

| 范围            | 含义                     | 例子                           |
| --------------- | ------------------------ | ------------------------------ |
| `^1.2.3`        | 兼容 1.x.x（不升 major） | `1.2.3` → `1.9.9` ✓，`2.0.0` ✗ |
| `~1.2.3`        | 兼容 1.2.x（不升 minor） | `1.2.3` → `1.2.9` ✓，`1.3.0` ✗ |
| `>=1.2.3`       | 大于等于                 | `1.2.3` → 任何更高版本         |
| `1.2.3 - 1.5.0` | 范围                     | `1.2.3` ≤ x ≤ `1.5.0`          |

实际开发中一个容易犯的错：不遵守 SemVer。比如在 `1.3.0` 里删除了一个公开函数——这是破坏性变更，应该升到 `2.0.0`。不遵守 SemVer 的后果是用户不敢升级你的包，最后只能锁死在一个旧版本上。

## pnpm：为什么不是 npm 或 yarn

npm 和 yarn（v1）安装依赖时，每个包的文件会被完整复制到 `node_modules` 里。如果你的项目有 100 个包，它们都依赖 lodash，lodash 会被复制 100 次。磁盘占用巨大，安装速度慢。

pnpm 用**内容寻址存储**解决这个问题：所有包存在全局的一个 store 里（`~/.pnpm-store/`），`node_modules` 中只创建硬链接。100 个包依赖 lodash，磁盘上只有一份 lodash 的实际数据。

```
npm 的 node_modules:
├── package-a/
│   └── node_modules/
│       └── lodash/ (完整副本 1)
├── package-b/
│   └── node_modules/
│       └── lodash/ (完整副本 2)

pnpm 的 node_modules:
├── .pnpm/
│   └── lodash@4.17.21/  (唯一的一份实际数据)
├── package-a → (符号链接到 .pnpm)
└── package-b → (符号链接到 .pnpm)
```

还有一个好处：pnpm 严格的依赖树结构防止了"幽灵依赖"——你在代码里 `import` 了一个没有在 `package.json` 里声明的包，npm 允许这样做（因为它可能被某个依赖安装了），但 pnpm 会直接报错。这避免了因为某个依赖悄悄删除了子依赖导致你的项目突然崩溃。

## 发布流程：npm pack 先预览

在 `npm publish` 之前，先用 `npm pack` 预览将要发布的内容：

```bash
npm pack --dry-run
```

这会列出将要被打包的所有文件和总大小。检查一下有没有意外包含的大文件（日志、测试数据、编译产物）。

如果需要排除特定文件，创建 `.npmignore`（和 `.gitignore` 语法相同）。但如果你已经声明了 `files` 字段，`.npmignore` 就不需要了——`files` 是白名单，更明确。

发布时还有一个选择：公开包还是私有包。

```bash
# 公开包（任何人都能安装）
npm publish

# 私有包（需要付费的 npm 账号，只有组织成员能安装）
npm publish --access restricted
```

`@scope/package-name` 格式的包默认是私有的，发布公开包需要显式 `--access public`。非 scoped 包只能是公开的。

## 什么时候用什么

| 场景             | 建议                                |
| ---------------- | ----------------------------------- |
| 新项目选包管理器 | pnpm。安装快、省磁盘、依赖树严格    |
| 包的入口声明     | 用 `exports`，不用 `main`           |
| 运行时依赖       | `dependencies`，不要放错            |
| 工具链依赖       | `devDependencies`                   |
| 期望用户自己装的 | `peerDependencies`                  |
| 发布前           | `npm pack --dry-run` 检查内容       |
| 版本号           | 严格遵守 SemVer，破坏性变更升 major |

npm 包是 Node.js 生态的基石。理解了包的发布和管理，你不仅能共享自己的代码，还能更好地理解别人的包是怎么工作的。日志服务的所有功能模块都已经就绪，最后一件事：怎么把它部署到生产环境并保持稳定运行。
