日志存好了（[第 5 篇](/posts/nodejs-core/05-filesystem)），现在要让其他服务能查询这些日志。你可能会直接上 Express——`app.get('/api/logs', handler)` 一行搞定。但 Express 的 `req` 和 `res` 到底包装了什么？`app.listen(3000)` 背后发生了什么？

理解 Node.js 原生的 `http` 模块，不是让你弃用框架，而是让你在框架遇到问题时知道怎么排查。

## http.createServer 做了什么

```js
import { createServer } from "node:http";

const server = createServer((req, res) => {
  res.writeHead(200, { "Content-Type": "text/plain" });
  res.end("hello\n");
});

server.listen(3000);
```

这段代码做了三件事：创建一个 TCP 服务器（[第 7 篇](/posts/nodejs-core/07-tcp-udp)会讲底层细节），监听端口 3000，每当有 HTTP 请求到达时调用回调函数。

回调函数的两个参数 `req` 和 `res` 是理解 HTTP 模块的关键。

## req：请求对象

`req` 是 `http.IncomingMessage` 的实例，它本身就是一个 Readable Stream。这意味着你可以用 Stream 的方式处理请求体：

```js
import { createServer } from "node:http";

const server = createServer(async (req, res) => {
  // 请求行信息
  console.log(req.method); // 'GET'
  console.log(req.url); // '/api/logs?level=ERROR&service=payment-gateway'
  console.log(req.headers); // { 'content-type': 'application/json', 'user-agent': '...' }

  // 解析 URL 查询参数
  const url = new URL(req.url, `http://${req.headers.host}`);
  const level = url.searchParams.get("level"); // 'ERROR'
  const service = url.searchParams.get("service"); // 'payment-gateway'

  // 读取请求体（req 是 Readable Stream）
  const chunks = [];
  for await (const chunk of req) {
    chunks.push(chunk);
  }
  const body = Buffer.concat(chunks).toString("utf-8");
});
```

注意 `req.url` 包含了完整的路径和查询字符串，但不包含协议和域名。用 `new URL()` 解析是最干净的方式。

对于 JSON 请求体，处理模式是固定的：

```js
async function readJsonBody(req) {
  const chunks = [];
  for await (const chunk of req) {
    chunks.push(chunk);
  }
  const raw = Buffer.concat(chunks).toString("utf-8");
  return JSON.parse(raw);
}
```

## res：响应对象

`res` 是 `http.ServerResponse` 的实例。它是一个 Writable Stream，你可以往里面写数据：

```js
res.writeHead(200, {
  "Content-Type": "application/json",
  "X-Request-Id": "req_a1x9k2m",
});
res.write('{"status":"ok"}');
res.end();
```

`res.writeHead()` 设置状态码和响应头。`res.write()` 写入响应体。`res.end()` 结束响应——调用 `end()` 后不能再写数据。

也可以用更简洁的方式：

```js
res.statusCode = 200;
res.setHeader("Content-Type", "application/json");
res.end(JSON.stringify({ status: "ok" }));
```

两种方式都可以，区别是 `writeHead` 一次性设置状态码和所有响应头，而 `setHeader` 可以多次调用分别设置。

## 实战：日志查询 API

现在用原生 `http` 模块构建一个简单的日志查询接口。

### 路由分发

原生 `http` 模块没有路由功能，需要自己实现简单的路由匹配：

```js
function createRouter() {
  const routes = [];

  function add(method, pattern, handler) {
    routes.push({ method, pattern, handler });
  }

  function resolve(req, res) {
    const url = new URL(req.url, `http://${req.headers.host}`);
    const pathname = url.pathname;

    for (const route of routes) {
      if (route.method === req.method && route.pattern === pathname) {
        return route.handler(req, res, url);
      }
    }

    // 404
    res.writeHead(404, { "Content-Type": "application/json" });
    res.end(JSON.stringify({ error: "Not Found", path: pathname }));
  }

  return { add, resolve };
}
```

### 日志查询接口

```js
import { createReadStream, readdir, stat } from "node:fs/promises";
import { join } from "node:path";
import { createServer } from "node:http";
import { pipeline } from "node:stream/promises";
import { Transform } from "node:stream";
import { createGzip } from "node:zlib";

const router = createRouter();

// GET /api/logs?level=ERROR&from=2026-03-15&to=2026-03-16&limit=100
router.add("GET", "/api/logs", async (req, res, url) => {
  const level = url.searchParams.get("level");
  const from = url.searchParams.get("from");
  const to = url.searchParams.get("to");
  const limit = parseInt(url.searchParams.get("limit") ?? "100", 10);

  // 根据日期范围找到对应的日志文件
  const logDir = join(process.cwd(), "logs");
  const files = await readdir(logDir);
  const matchedFiles = files
    .filter((f) => f.endsWith(".log"))
    .filter((f) => {
      if (from && f < from) return false;
      if (to && f > to) return false;
      return true;
    })
    .sort();

  // 流式读取 + 过滤 + 限制数量
  let count = 0;
  const logFilter = new Transform({
    transform(chunk, enc, cb) {
      if (count >= limit) {
        cb();
        return;
      }
      const line = chunk.toString("utf-8").trim();
      if (!line) return cb();
      if (level && !line.includes(`"level":"${level}"`)) return cb();

      count++;
      this.push(line + "\n");
      cb();
    },
  });

  // 支持 gzip 压缩响应
  const acceptEncoding = req.headers["accept-encoding"] ?? "";
  if (acceptEncoding.includes("gzip")) {
    res.writeHead(200, {
      "Content-Type": "text/plain; charset=utf-8",
      "Content-Encoding": "gzip",
    });
    // 注意：需要依次处理每个文件
    for (const file of matchedFiles) {
      if (count >= limit) break;
      await pipeline(
        createReadStream(join(logDir, file)),
        logFilter,
        createGzip(),
        res
      );
    }
  } else {
    res.writeHead(200, { "Content-Type": "text/plain; charset=utf-8" });
    for (const file of matchedFiles) {
      if (count >= limit) break;
      await pipeline(createReadStream(join(logDir, file)), logFilter, res);
    }
  }
});

const server = createServer((req, res) => {
  router.resolve(req, res);
});

server.listen(3000, () => {
  console.log("日志查询 API 已启动: http://localhost:3000");
});
```

这段代码展示了几个关键点：

- 用 Transform 流做日志过滤和数量限制，内存占用可控
- 根据 `Accept-Encoding` 头决定是否 gzip 压缩响应
- 用 `pipeline` 连接文件读取、过滤、压缩、响应

这就是为什么理解 Stream（[第 2 篇](/posts/nodejs-core/02-stream)）很重要——HTTP 的请求体和响应体本质上都是 Stream。

## HTTP 状态码的正确使用

一个常见问题：所有错误都返回 500。HTTP 状态码是有语义的，用错了会让客户端很难处理：

```
2xx 成功
  200 OK          — 请求成功
  201 Created     — 资源创建成功（POST）
  204 No Content  — 成功但无返回内容（DELETE）

4xx 客户端错误
  400 Bad Request           — 请求参数错误
  401 Unauthorized          — 未认证
  403 Forbidden             — 已认证但无权限
  404 Not Found             — 资源不存在
  429 Too Many Requests     — 请求频率过高

5xx 服务端错误
  500 Internal Server Error  — 服务端未知错误
  502 Bad Gateway            — 上游服务不可用
  503 Service Unavailable    — 服务暂时不可用
```

## HTTPS 配置

生产环境必须用 HTTPS。Node.js 配置 HTTPS 需要证书文件：

```js
import { createSecureServer } from "node:https";
import { readFileSync } from "node:fs";

const server = createSecureServer(
  {
    key: readFileSync("cert/key.pem"),
    cert: readFileSync("cert/cert.pem"),
    // 控制 TLS 版本和加密套件
    minVersion: "TLSv1.2",
    ciphers: "TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256",
  },
  (req, res) => {
    res.end("Hello over HTTPS");
  }
);

server.listen(3443);
```

开发环境用自签名证书就行（`openssl req -x509 -newkey rsa:2048 -nodes -keyout key.pem -out cert.pem -days 365`），生产环境用 Let's Encrypt 或公司内部的 CA。

## Keep-Alive：连接复用

HTTP/1.1 默认开启 Keep-Alive。这意味着一个 TCP 连接可以处理多个 HTTP 请求，省去了每次请求的 TCP 三次握手开销：

```
无 Keep-Alive：
客户端 → [TCP 握手] → 请求1 → [TCP 断开]
客户端 → [TCP 握手] → 请求2 → [TCP 断开]
客户端 → [TCP 握手] → 请求3 → [TCP 断开]

有 Keep-Alive：
客户端 → [TCP 握手] → 请求1 → 请求2 → 请求3 → [TCP 断开]
```

Node.js 的 HTTP 服务器默认启用 Keep-Alive。你可以通过 `req.headers['connection']` 查看客户端是否支持。`server.keepAliveTimeout` 控制空闲连接的超时时间（默认 5 秒）。

但 Keep-Alive 也有代价：每个空闲连接都占用一个文件描述符。如果你的服务有大量长连接的客户端（比如移动端），需要适当缩短 `keepAliveTimeout`，避免 fd 耗尽。

原生 `http` 模块适合理解底层机制和小型服务。当你的 API 接口超过 10 个、需要中间件、路由分组、参数校验时，Express 或 Fastify 会更高效。但理解了 `http` 模块，你用框架时就知道它在做什么，出问题也知道怎么排查。
