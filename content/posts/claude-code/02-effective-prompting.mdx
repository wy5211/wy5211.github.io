## 前言

上一篇你装好了 Claude Code，配好了 CLAUDE.md，跑通了第一个示例。但很快你会发现一个让人沮丧的事实：同样的工具，别人一句话就能让它干对，你来回对话十轮还是没搞对。

不是你不够聪明，也不是 Claude Code 不够好。问题在于你和它"说话"的方式。Claude Code 不是搜索引擎，你丢给它一个关键词然后期待它理解你的意图——这种方式对 ChatGPT 也许够用，但对 Claude Code 来说远远不够。

这篇文章就聊一件事：怎么写 Prompt，让 Claude Code **第一次**就给出你想要的结果。

## Claude Code 是怎么理解你的项目的

在聊怎么写 Prompt 之前，先理解 Claude Code 的上下文机制。它和网页版的 Claude 有本质区别。

网页版 Claude：你每发一条消息，它只看到你手动粘贴的内容。对话结束，上下文清空。

Claude Code：它能看到你的整个项目。你的 CLAUDE.md、Memory 目录、之前对话的历史、甚至它自己读过的文件——这些都构成它的"理解"。

具体来说，Claude Code 在每次收到你的指令时，会从这几个来源获取上下文：

- **CLAUDE.md**：项目级的全局指令，告诉它"这个项目用什么技术栈、遵循什么规范"
- **Memory**：跨会话的持久记忆，比如"上次我们讨论过把认证模块从 JWT 换成 Session"
- **当前对话历史**：你在这次会话中说过的所有话、它做过的所有操作
- **你 @ 引用的文件**：你明确指出的"看这个文件"
- **它自己读到的文件**：根据你的指令，它用 Read、Grep、Glob 工具读到的代码

理解了这个机制，你就知道该怎么和它沟通了：**不要重复它已经知道的信息，但一定要补充它不知道的上下文**。

## 有效 Prompt 的四要素

一个有效的 Prompt 不需要长篇大论，但需要包含四个关键要素。我们用一个真实场景来说明——"给课程列表接口加分页查询"。

### 差的 Prompt

```
帮我加分页
```

三个字，信息量为零。Claude Code 不知道：加在哪里？用什么方式？分页参数怎么定义？返回格式是什么？

### 好的 Prompt

```
给 @src/modules/course/course.controller.ts 中的
GET /courses 接口加分页查询。

要求：
- 使用 query 参数 page（默认 1）和 pageSize（默认 20）
- pageSize 最大不超过 100
- 返回格式为 { items: Course[], total: number, page: number, pageSize: number }
- 参考 @src/common/dto/pagination.dto.ts 中已有的分页 DTO
```

这个 Prompt 包含了四个要素：

| 要素       | 说明         | 示例                                     |
| ---------- | ------------ | ---------------------------------------- |
| **目标**   | 做什么       | 给 GET /courses 接口加分页查询           |
| **上下文** | 在哪做       | @src/modules/course/course.controller.ts |
| **约束**   | 有什么限制   | pageSize 最大 100、参考已有 DTO          |
| **示例**   | 期望什么格式 | 返回格式明确给出                         |

目标和上下文决定了 Claude Code 去**哪里**做什么，约束避免了它做多余的事（比如它可能自作主张给所有接口都加排序），示例确保输出符合你的预期。

### 什么时候需要给示例

不是每个 Prompt 都需要示例。如果你要的是标准的、常见的做法（比如"添加一个 CRUD 接口"），不需要示例——Claude Code 知道怎么做。但如果你有特殊的格式要求（比如特定的响应结构、自定义的错误码），给一个示例能省去大量来回沟通。

一个简单的判断标准：如果你觉得"按常规做法"就能满足你的需求，不需要示例。如果你心里有一个明确的期望输出，但这个期望不是行业通用的——给示例。

## @ 引用：精确制导 vs 让它自己找

Claude Code 有能力自己搜索代码库（用 Grep、Glob 等工具），但让它自己找有两个问题：**慢**（每多搜索一次就多消耗 token 和时间）和**可能找错**（项目大了它不一定能精确找到你想要的文件）。

用 @ 引用文件和目录，等于给它精确坐标。

```
# 模糊：让它自己找
"修改用户注册的逻辑"  ← 它要去搜索"注册"、"register"、"signup"等关键词，
                      可能找到好几个文件，不确定改哪个

# 精确：直接告诉它
"修改 @src/modules/auth/auth.service.ts 中的 register 方法"
```

@ 引用支持几种写法：

- `@src/file.ts`：引用单个文件
- `@src/modules/auth/*.ts`：引用目录下所有 ts 文件
- `@src/common/dto/`：引用整个目录

有一个容易忽略的技巧：你可以引用**多个文件**来提供完整的上下文。

```
"给课程模块添加类似用户模块的分页功能。
参考 @src/modules/user/user.service.ts 中的分页实现，
修改 @src/modules/course/course.service.ts"
```

这样 Claude Code 既能看到"目标代码"（course.service.ts），也能看到"参考代码"（user.service.ts），不需要它去猜你的意图。

## 分步指令 vs 一步到位

有些任务适合一次说完，有些适合拆成几步。怎么判断？

**适合一步到位的场景**：改动范围明确、不涉及架构决策。比如"给 UserService 添加一个 findByEmail 方法"、"给 Course entity 添加一个 description 字段"。

**适合分步的场景**：涉及多文件修改、需要你中途确认方向。比如"重构认证模块"——如果你一次性说完，Claude Code 可能改了 30 个文件，结果方向完全错了。

分步不是让你每次只说一句话，而是把大任务拆成合理的阶段：

```
# 第一轮：让它先分析
"分析 @src/modules/auth/ 目录下的认证实现，
 列出当前的问题和改进方向，先不要改代码"

# 看完分析结果后...

# 第二轮：让它按方案执行
"按你刚才分析的第二种方案重构，
 先改 auth.service.ts 和 auth.controller.ts，
 保持向后兼容"
```

这样你能在每一步确认方向正确再继续，避免大面积返工。

## 三个常见的沟通陷阱

### 陷阱一：过于模糊

```
# 模糊
"优化一下性能"
"改进一下代码"
"修一下那个 bug"
```

Claude Code 不知道你说的"性能"是指接口响应时间还是包体积，不知道"改进"是指重构还是加功能，不知道"那个 bug"是哪个 bug。它只能猜，而猜错的结果就是你要花更多轮次来纠正。

**修正**：说清楚具体要做什么、在哪做、期望的结果是什么。

### 陷阱二：过于冗长

和模糊相反，有些人的 Prompt 写得像需求文档：

```
"我们团队经过讨论，决定对课程列表接口进行优化。
首先，我们需要添加分页功能，page 参数默认值为 1，
pageSize 参数默认值为 20。其次，我们要支持按课程
名称模糊搜索，搜索参数名为 keyword。然后，我们
还想支持按创建时间排序，sort 参数接受 'asc' 和
'desc' 两个值..."
```

这段话说了 150 个字，但有效信息可能只要 50 个字。冗长的 Prompt 的问题是：Claude Code 可能在大量文字中遗漏关键约束，而且浪费 token。

**修正**：用结构化的格式代替长篇叙述。关键信息用列表或表格呈现。

### 陷阱三：自相矛盾

这个陷阱最隐蔽。你可能在一个 Prompt 里前后说了不一致的要求：

```
"给课程接口加分页，返回所有课程数据。
pageSize 默认 20，但也要支持返回全部数据。"
```

"返回所有课程数据"和"分页"本身就是矛盾的——如果返回全部，分页就没有意义。Claude Code 遇到矛盾时要么选一个执行（通常是你说的第一个），要么要求你澄清，要么两边都做但互相冲突。

**修正**：发 Prompt 之前先想清楚你要什么，不要把"我还没想清楚"的问题抛给它。

## 代码规范：写在 CLAUDE.md 里，别每次重复

如果你发现自己在每次对话中都在说同样的话——"用 TypeScript 严格模式"、"变量用 camelCase"、"不要用 any"——这些就该写进 CLAUDE.md。

CLAUDE.md 是**项目级的持久指令**，每次对话 Claude Code 都会自动读取。你把通用规范放在那里，Prompt 里就只需要说任务本身，不用重复规范。

```
# 不需要每次都说
"创建一个用户 DTO，用 TypeScript 严格模式，
 变量用 camelCase，不要用 any"

# CLAUDE.md 已经定义了这些规范，Prompt 只需要说
"创建一个用户 DTO，包含 email、password、name 字段"
```

但 CLAUDE.md 也不是什么都能放。它适合放**通用的、不变的规范**（代码风格、技术栈、目录结构），不适合放**具体的、一次性的指令**（"这个 PR 要改什么"）。后者在对话中说就好。

## 让它"问你"而不是"猜你"

有时候你不确定该怎么做，希望 Claude Code 先问你一些问题再动手。这时候可以在 Prompt 里明确告诉它：

```
"我需要给课程模块加一个评分功能，
 但我还没想清楚数据结构。
 你先问我要几个关键问题，
 确认清楚再开始设计。"
```

Claude Code 有一个 AskUserQuestion 工具，可以在执行前向你确认。但你需要在 Prompt 里**主动触发**这个行为——如果你不说"先问清楚"，它很可能默认你已经想好了，直接开始做。

这个技巧在以下场景特别有用：

- 你对需求只有模糊的想法，需要讨论才能明确
- 涉及架构决策，有多种可行方案
- 你不确定当前的代码结构，需要它先分析再提建议
