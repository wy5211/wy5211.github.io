# Claude Code 进阶指南

## 背景

你装了 Claude Code，会用它写函数、改 bug、跑命令。但你发现别人用同样的工具，效率是你的三倍——他们能让 Claude Code 自动规划任务、分工协作、重构代码、写测试，而你还在一行一行地告诉它该干什么。

差距不在于工具，在于你**怎么用**它。Claude Code 不只是一个"聪明的终端"，它有 Plan Mode、Agent System、Skills 生态、Memory 系统一整套能力。这个系列就是帮你把这些能力串起来，从一个"会用"的用户变成一个"玩得溜"的用户。

### 技术栈选择

| 技术        | 版本   | 为什么选它                                                 |
| ----------- | ------ | ---------------------------------------------------------- |
| Claude Code | Latest | Anthropic 官方 CLI，200K 上下文、Agent System、Skills 生态 |
| Node.js     | 22 LTS | Claude Code 的运行时                                       |

> 本系列不教"什么是 AI 编程助手"，而是面向已经在用 Claude Code 的开发者，帮你把它的进阶功能用起来。如果你还没用过 Claude Code，建议先看[官方文档](https://docs.anthropic.com/en/docs/claude-code)跑通第一个示例。

## 主线案例：接手一个团队的 NestJS API 项目

整个系列围绕一个真实的开发场景展开：你刚加入一个团队，接手了一个运行了半年的 NestJS 后端 API 项目。代码库有 50+ 文件，没有文档，测试覆盖率不到 20%，技术债务不少。你需要用 Claude Code 快速理解项目、修复问题、重构代码、补齐测试。

**案例场景**：一个在线教育平台的 API——课程管理、用户注册、订单支付、视频播放记录。

**案例数据示例**（贯穿全系列使用）：

```
项目：edu-platform-api（NestJS + TypeORM + PostgreSQL）
文件数：86 个 TypeScript 文件
测试覆盖率：18%
已知问题：
  - 课程列表接口在数据量大时响应超过 3 秒
  - 用户注册没有做邮箱格式校验
  - OrderService 里复制粘贴了大量重复逻辑
  - 没有统一错误处理，有的接口返回 500 有的返回 200 + error 字段
```

## 基础篇：建立直觉（01-03）

目标：用 3 篇文章帮你掌握 Claude Code 的核心交互方式，从"会用"到"用好"。

---

### 01. Claude Code 快速入门

**核心问题**：你装了 Claude Code，能跑通基本命令。但和 GitHub Copilot 比有什么区别？什么时候该用 Claude Code，什么时候用 IDE 插件就够了？CLAUDE.md 到底怎么写？

**主线案例**：用 Claude Code 在一个空目录里搭建一个 Express API 项目——从初始化到运行，体验 Claude Code 的完整工作流。

**内容要点**：

- Claude Code vs GitHub Copilot vs Cursor：定位差异和使用场景
- 安装与基本配置：权限模式、API Key
- CLAUDE.md 文件：项目级指令、全局配置、Memory 目录
- 基本工作流：给它一段需求，看它怎么读代码、写代码、跑命令
- 什么时候该用 Claude Code 而不是 IDE 插件
- 权限模型：自动允许、逐次确认、信任项目的边界

**与前篇关联**：系列第一篇，无前篇。

---

### 02. 高效对话：怎么和 Claude Code 说话它才听得懂

**核心问题**：同样是用 Claude Code，有人一句话就能让它干对，有人来回对话十轮还是没搞对。问题出在哪？怎么写 Prompt 才能让它第一次就给出你想要的结果？

**主线案例**：对比"好的 Prompt"和"差的 Prompt"在同一任务上的效果差异——实现一个分页查询接口，展示不同描述方式导致的质量差距。

**内容要点**：

- Claude Code 的上下文理解机制：它怎么理解你的项目
- 有效 Prompt 的结构：目标 + 约束 + 上下文 + 示例
- @ 引用文件和目录：精确提供上下文而不是让它自己找
- 分步指令 vs 一步到位：什么时候该拆、什么时候不该拆
- 常见的沟通陷阱：过于模糊、过于冗长、自相矛盾
- 代码规范传达：通过 CLAUDE.md 还是每次对话都重复
- 如何让它"问你"而不是"猜你"：利用 AskUserQuestion

**与前篇关联**：上一篇搭好了项目，这一篇优化你与 Claude Code 的沟通方式。

---

### 03. Plan Mode：复杂任务先想清楚再做

**核心问题**：你让 Claude Code "重构认证模块"，它二话不说就开始改代码，改了 30 个文件，结果方向完全错了。复杂任务不能直接动手，得先规划。Plan Mode 就是干这个的。

**主线案例**：用 Plan Mode 规划一个"给课程列表接口加缓存"的功能——先探索代码、设计缓存方案、列出改动文件、评估风险，全部确认后再动手。

**内容要点**：

- Plan Mode 的本质：先读代码想方案，你确认后再执行
- 什么时候该用 Plan Mode：改动范围大、涉及多模块、架构级变更
- 什么时候不该用：单文件修改、明确的小需求
- Plan Mode 的工作流程：探索 → 设计 → 评审 → 执行
- 如何在 Plan Mode 中提供有效反馈
- Plan Mode 的局限：不是所有复杂任务都适合规划

**与前篇关联**：上一篇学会了怎么和它说话，这一篇学会让它先想清楚再动手。

### 04. Agent System：让 AI 自己分工协作

**核心问题**：你接手了一个 86 个文件的项目，想快速搞清楚整体架构和模块依赖。手动一个个文件看？太慢了。Claude Code 的 Agent System 可以帮你自动探索代码库、搜索关键词、分析依赖关系。

**主线案例**：用 Agent System 分析 edu-platform-api 项目——Explore Agent 摸清架构，General Purpose Agent 修复已知问题，理解不同 Agent 的适用场景。

**内容要点**：

- Agent System 是什么：Claude Code 内置的"子任务分发"机制
- 三种内置 Agent：Explore（快速搜索）、Plan（设计方案）、General Purpose（通用执行）
- 什么时候该用 Agent 而不是直接对话：搜索类任务、多步骤任务、并行任务
- Explore Agent 的使用：搜索文件、查找定义、理解代码结构
- Agent 的隔离模式：worktree 隔离，不污染你的工作分支
- 什么时候不该用 Agent：简单任务用 Agent 反而更慢

**与前篇关联**：上一篇用 Plan Mode 规划了任务，这一篇用 Agent System 执行多步骤的探索和修改。

---

### 05. Skills：发现、使用与自定义开发

**核心问题**：你每次写博客都要手动排版、每次提交都要想 commit message、每次写文章都要遵循一套规范……这些重复性工作能不能自动化？Skills 就是 Claude Code 的"插件系统"。

**主线案例**：为 edu-platform-api 团队创建自定义 Skills——代码审查 Skill、数据库迁移检查 Skill、API 文档生成 Skill，让团队协作更规范。

**内容要点**：

- Skills 是什么：预定义的指令模板，让 Claude Code 按特定流程工作
- 发现和使用已有 Skills：社区 Skills、官方推荐
- 自定义 Skill 的结构：SKILL.md 的编写规范
- Skill 设计原则：什么时候该做成 Skill、什么时候不需要
- 实战：从零创建一个 Skill——从需求分析到测试验证
- Skill 的局限：不是所有重复工作都适合做成 Skill

**与前篇关联**：上一篇用 Agent 处理了一次性任务，这一篇把重复性工作封装成可复用的 Skill。

### 06. 代码重构与测试驱动

**核心问题**：接手的项目里，OrderService 复制粘贴了大量重复逻辑，测试覆盖率只有 18%。你想重构，但怕改出 bug；你想补测试，但不知道从哪下手。Claude Code 能帮你安全地做这两件事。

**主线案例**：用 Claude Code 重构 edu-platform-api 的 OrderService——先用 Plan Mode 分析重构方案，再小步重构、同步补测试，确保功能不变。

**内容要点**：

- 用 Claude Code 做安全重构：先理解再动手、小步提交、跑测试验证
- 重构前必做：分析影响范围、确认测试覆盖、准备回滚方案
- 测试驱动开发：先写测试再改代码，让 Claude Code 帮你生成测试
- 常见重构场景：提取公共逻辑、拆分大函数、统一错误处理
- 重构的节奏：一次改多少合适、怎么分批提交
- 什么时候不该让 AI 重构：业务逻辑复杂的模块需要人把关

**与前篇关联**：上一篇封装了自动化 Skill，这一篇用 Claude Code 处理代码层面的问题。

### 07. 调试与问题排查

**核心问题**：线上接口返回 500，错误日志只有一堆堆栈信息。你把日志贴给 Claude Code，它给了一个"可能的修复方案"但根本不对。怎么描述问题才能让它给出有用的诊断？

**主线案例**：排查 edu-platform-api 的生产问题——课程列表接口响应慢、用户注册偶发失败、视频播放记录丢失。

**内容要点**：

- 描述问题的有效方式：错误信息 + 复现步骤 + 相关代码 + 环境
- 让 Claude Code 读日志：提供日志文件，让它分析模式
- 让 Claude Code 读堆栈：从错误堆栈定位问题代码
- 常见调试场景：接口报错、性能问题、数据不一致
- 什么时候该给 AI 看代码、什么时候不该：敏感信息、配置文件
- 验证修复方案：AI 给的方案不一定对，怎么验证

**与前篇关联**：上一篇重构完代码，这一篇处理代码运行时的问题。

---

### 08. 上下文管理与性能优化

**核心问题**：对话进行到第 30 轮，Claude Code 开始"忘记"前面的内容，回答质量明显下降。86 个文件的项目，它经常找不到正确的代码。上下文管理是 Claude Code 高效使用的核心能力。

**主线案例**：在 edu-platform-api 这样规模的项目中，优化上下文使用——只加载相关文件、合理使用 Memory、避免上下文膨胀。

**内容要点**：

- 上下文窗口的工作机制：200K tokens 看似很多，但代码文件很占空间
- @ 引用的策略：引用具体文件 vs 引用目录 vs 引用 Grep 结果
- Memory 系统：跨会话记住项目上下文
- CLAUDE.md 的分层设计：全局配置 vs 项目配置 vs 目录配置
- 什么时候该 /clear 重新开始：对话跑偏、上下文过大
- Token 使用优化：只提供必要信息，避免冗余

**与前篇关联**：前几篇在单次对话中完成任务，这一篇解决长时间、大项目的上下文管理问题。

---

### 09. Git 工作流与团队协作

**核心问题**：你用 Claude Code 写代码很舒服，但一到 Git 操作就头疼——commit message 写不好、PR review 漏掉问题、多人协作时 CLAUDE.md 规范不统一。怎么把 Claude Code 融入团队工作流？

**主线案例**：为 edu-platform-api 团队搭建 Claude Code 协作规范——统一的 CLAUDE.md、规范的 Git 工作流、PR 审查流程。

**内容要点**：

- Git 集成：用 Claude Code 生成 commit message、创建 PR、review 代码
- CLAUDE.md 团队规范：写什么、不写什么、怎么保持更新
- PR 审查：让 Claude Code 做第一轮 code review，人工复核
- Hooks 集成：pre-commit 检查、commit message 规范
- 多人协作：不同角色的权限设置、共享 Memory
- Claude Code 在团队推广的常见阻力：安全顾虑、质量担忧、习惯改变

**与前篇关联**：前几篇是个人使用技巧，这一篇扩展到团队场景。

### 10. 综合实战与最佳实践

**核心问题**：你学了一堆技巧，但怎么在日常开发中形成稳定的工作流？哪些功能该天天用、哪些偶尔用、哪些不需要用？Claude Code 不是万能的，知道它的边界同样重要。

**主线案例**：模拟一个完整的开发日——早上查看 PR review，上午修 bug，下午重构模块、补测试，下班前提交。全部用 Claude Code 辅助，展示真实的工作节奏。

**内容要点**：

- 日常工作流模板：不同类型任务的 Claude Code 使用策略
- Claude Code 的能力边界：它擅长什么、不擅长什么
- 常见误区：过度依赖 AI、不验证就提交、忽略安全风险
- 效率对比：有 Claude Code vs 没有 Claude Code 的真实差距
- 持续学习：Claude Code 更新快，怎么跟上新功能
- 系列回顾：从入门到精通的知识地图

**与前篇关联**：前 9 篇掌握了各个功能模块，这一篇综合运用，形成完整的工作流。
