上篇用 Agent 和 Skills 自动化了工作流。但接手的项目里，OrderService 有大量重复代码，测试覆盖率只有 18%。

你想重构，但怕改出 bug；你想补测试，但不知道从哪下手。

## 安全重构的原则

Claude Code 可以帮你重构，但你必须遵守这些原则：

**1. 先理解再动手**

```bash
# 先让 Claude Code 分析代码
> 分析 @src/order/order.service.ts 的代码结构，
> 列出重复模式，不要改代码

📊 代码分析：
- 发现 5 处重复的查询逻辑
- 发现 3 处重复的验证逻辑
- 发现 2 处重复的错误处理
```

**2. 小步提交**

```bash
# 第一步：提取公共查询
> 提取 findOrderById 和 findOrdersByUser 到 BaseRepository
> 只改这两个方法，其他不动

# 验证通过后，再下一步
> 提取 validateOrderStatus 到 BaseValidator
```

**3. 每步都要测试**

```bash
> 重构完成后，运行相关测试
> npm test src/order/order.service.test.ts
```

## 测试驱动开发

### 让 Claude Code 生成测试

```bash
> 为 @src/order/order.service.ts 写单元测试
> �盖以下场景：
> - 创建订单成功
> - 创建订单失败（库存不足）
> - 查询订单列表
> - 更新订单状态
```

Claude Code 会生成：

```typescript
describe("OrderService", () => {
  it("should create order successfully", async () => {
    // 测试代码
  });

  it("should fail when insufficient stock", async () => {
    // 测试代码
  });
});
```

### 重构流程

```
1. 先写测试（即使代码还不存在）
2. 运行测试（会失败）
3. 写代码让测试通过
4. 重构代码（测试仍然通过）
```

## 重构的节奏

**一次改多少合适？**

```
❌ 一次重构整个模块（风险大）
✅ 一次改一个类，提交一次（风险小）
```

**判断标准**：**如果这次重构出问题，能快速回滚吗？如果不能，拆小点。**

## 什么时候不该让 AI 重构

- 业务逻辑复杂的模块（AI 可能理解错）
- 涉及安全的代码（加密、权限）
- 性能关键的代码（AI 的优化可能适得其反）

这些需要你自己把关，Claude Code 可以辅助，但不能替代你的判断。
