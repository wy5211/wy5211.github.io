## 前言

上一篇我们用 Agent System 处理了一次性的复杂任务——探索代码、设计方案、修复 bug。但有一类问题 Agent 解决不了：**重复性工作**。

你每次提交代码都要想 commit message 的格式，每次写博客都要遵循同一套排版规范，每次做代码审查都要按同一个标准打分。这些事情模式固定、重复度高，但每次都要你手动告诉 Claude Code 该怎么做。

Skills 就是解决这个问题的。它本质上是**预定义的指令模板**——你把一套流程写好，以后只需要触发它，Claude Code 就会按你定义的步骤执行。

## Skills 是什么：让 Claude Code 按你的规矩干活

先看一个具体场景。假设你的团队规定所有 API 端点必须包含请求验证、错误处理和 Swagger 文档。没有 Skill 的时候，每次新建端点你都要重复这些要求：

```
你: "创建一个课程列表接口，要加 class-validator 验证，
     错误处理统一用 NestJS 的 HttpException，
     别忘了加 @ApiTags 和 @ApiOperation 装饰器"

你: "再创建一个用户详情接口，要求一样..."

你: "再来一个订单查询接口..."
```

有了 Skill，你只需要定义一次规范，之后直接触发：

```
你: /create-endpoint "课程列表"
你: /create-endpoint "用户详情"
你: /create-endpoint "订单查询"
```

这就是 Skill 和 CLAUDE.md 的关键区别。CLAUDE.md 是**通用规范**——它告诉 Claude Code "我们的项目用 TypeScript、用 NestJS、遵循 Airbnb 代码风格"。Skill 是**特定流程**——它告诉 Claude Code "创建 API 端点时，按这 5 步走：1. 创建 DTO 2. 实现 Service 3. 写 Controller 4. 加验证 5. 补测试"。

用一个类比：CLAUDE.md 是员工手册，Skill 是 SOP 操作流程。

## 发现和使用已有 Skills

在你决定自己写 Skill 之前，先看看有没有人已经做过了。Claude Code 有一个活跃的 Skills 生态。

### 从哪里找 Skills

最直接的方式是在 Claude Code 里使用 `/find-skills`，它会帮你搜索匹配你需求的 Skill。你也可以浏览社区仓库，看看别人分享了什么。

官方维护的 Skills 仓库里有几个常用的：`commit`（自动生成规范的 commit message）、`review-pr`（PR 代码审查）、`test-runner`（运行和分析测试）。这些是经过官方验证的，质量有保障。

社区贡献的 Skills 覆盖面更广——Docker 配置生成、Kubernetes 部署、GraphQL Schema 构建、数据库迁移辅助等。选择社区 Skill 时，重点看三个指标：最近是否还在更新、使用量多少、有没有清晰的文档和示例。

### 安装和使用

安装 Skill 就是把它放到你的 `.claude/skills/` 目录下。可以全局安装（所有项目可用），也可以只装在某个项目里（仅该项目可用）。

安装完成后，在对话中输入 `/skill-name` 就能触发。Skill 的指令会被注入到 Claude Code 的上下文中，它会按照 SKILL.md 里定义的流程执行。

一个值得注意的点：同时安装太多 Skill 不一定好事。Skill 的指令会占用上下文窗口，装了 20 个 Skill，每个都有一大段指令，留给实际代码的空间就少了。只装你经常用的那几个。

## 自定义 Skill：从零开始

社区 Skill 不一定符合你团队的具体需求。比如你们的代码审查标准、你们的部署流程、你们的测试规范——这些都需要自定义。好在写一个 Skill 不难，它就是一个 Markdown 文件。

### Skill 的文件结构

每个 Skill 的核心是一个 `SKILL.md` 文件，放在 `.claude/skills/<skill-name>/` 目录下：

```
.claude/skills/
└── code-review/
    └── SKILL.md
```

SKILL.md 的内容就是给 Claude Code 的指令。你不需要学什么特殊的语法，用自然的语言把流程描述清楚就行。一个好的 SKILL.md 通常包含这些部分：

- **什么时候用这个 Skill**——帮 Claude Code 判断是否该触发
- **工作流程**——按步骤描述要做什么
- **代码规范**——具体的格式要求和约束
- **示例**——展示输入和期望的输出

### 实战：为 edu-platform-api 创建代码审查 Skill

假设你的团队对代码审查有以下要求：

1. 检查是否有类型安全的问题（`any`、`as` 滥用）
2. 检查错误处理是否完善
3. 检查是否有必要的测试覆盖
4. 检查命名是否符合团队规范

对应的 SKILL.md 大概长这样：

```markdown
---
name: code-review
description: 审查 TypeScript 代码变更，按团队规范检查质量
---

# 代码审查

## 何时使用

当需要审查 PR 或代码变更时。

## 审查流程

### 1. 理解变更范围

用 git diff 查看变更内容，理解改了什么、为什么改。

### 2. 类型安全检查

重点检查：

- 是否使用了 `any` 类型（不允许）
- 是否有 `as` 类型断言（需要注释说明原因）
- 函数返回值类型是否明确标注
- 泛型参数是否有合理的约束

### 3. 错误处理检查

- 数据库操作是否有 try-catch
- 外部 API 调用是否有超时和重试
- 错误信息是否对用户友好（不暴露内部细节）
- 是否有统一的错误响应格式

### 4. 测试覆盖检查

- 新增的业务逻辑是否有对应的测试
- 测试是否覆盖了边界情况（空值、零值、异常输入）
- 测试描述是否清晰（"should do X when Y" 格式）

### 5. 输出审查报告

按以下格式输出：

## 审查结果

### 类型安全

- [列出问题或"无问题"]

### 错误处理

- [列出问题或"无问题"]

### 测试覆盖

- [列出缺失的测试或"覆盖充分"]

### 改进建议

- [可选的优化建议]
```

写好之后，在项目中就可以用 `/code-review` 触发审查了。Claude Code 会按你定义的 5 步流程执行，最后输出一份结构化的审查报告。

### 实战：数据库迁移检查 Skill

团队中经常有人忘记在改完 Entity 后运行迁移。可以写一个 Skill 来检查：

```markdown
---
name: check-migration
description: 检查 Entity 变更是否生成了对应的迁移文件
---

# 数据库迁移检查

## 工作流程

1. 运行 `git diff --name-only` 获取变更的文件列表
2. 筛选出 Entity 文件（`*.entity.ts`）
3. 检查是否有对应的迁移文件被创建或修改
4. 如果 Entity 有变更但没有迁移文件，输出警告
5. 如果有迁移文件，检查文件名是否包含时间戳且排序正确

## 警告格式

⚠️ Entity `src/modules/course/course.entity.ts` 有变更，
但未找到对应的迁移文件。
请运行 `npm run migration:generate` 生成迁移。
```

这个 Skill 很简单，但它解决了一个真实问题：团队成员经常忘记生成迁移文件，直到部署到测试环境才发现数据库结构没更新。

## Skill 设计原则：什么时候该做 Skill

不是所有重复性工作都适合做成 Skill。有几个判断标准。

**适合做成 Skill 的场景**：流程固定、步骤明确、需要反复执行。代码审查、创建特定类型的文件、部署检查、commit message 格式化——这些都符合。

**不适合做成 Skill 的场景**：

- **一次性任务**——你只做一次的事情，不值得花时间封装
- **变化频繁的流程**——如果规范每个月都在改，维护 Skill 本身就成了负担，不如直接写在 CLAUDE.md 里
- **太简单的操作**——如果一行命令就能搞定（比如 `npm run test`），不需要包装成 Skill
- **需要大量人工判断的任务**——Skill 的指令是文本，太模糊的判断标准会让 Claude Code 的输出不稳定

一个实用的判断方法：如果你在过去一周里，对 Claude Code 说过三次以上同样的话，那这段话就应该变成 Skill。

### Skill vs CLAUDE.md 怎么选

简单来说：**通用的规范放 CLAUDE.md，具体的流程放 Skill**。

"我们用 TypeScript 严格模式"——这是规范，放 CLAUDE.md。
"创建 API 端点时按这 5 步走"——这是流程，放 Skill。

"变量用 camelCase 命名"——规范，CLAUDE.md。
"提交 PR 前检查类型安全和错误处理"——流程，Skill。

两者配合使用效果最好：CLAUDE.md 定义"我们的标准"，Skill 定义"怎么达到标准"。

## Skill 的局限

Skill 不是万能的。有几个需要注意的地方：

**上下文占用**：每个 Skill 的指令都会占用上下文窗口。如果你的 SKILL.md 写了 2000 行，那每次触发都会消耗大量 token。保持 Skill 精简，只写必要的流程和约束。

**输出不稳定**：Skill 说到底是自然语言指令，Claude Code 对它的理解和执行可能有波动。特别是模糊的指令（"代码要写得清晰"）远不如具体的指令（"函数不超过 30 行"）可靠。

**不能执行实际操作**：Skill 只是告诉 Claude Code 该做什么，真正执行还是要靠 Claude Code 的能力。如果你的流程需要调用外部 API、操作数据库等，Skill 本身做不了，需要配合其他工具。

**维护成本**：Skill 写好之后不是一劳永逸的。项目技术栈变了、团队规范改了，Skill 也要跟着更新。如果一个 Skill 长期没人维护，它可能已经和项目现状脱节了。
