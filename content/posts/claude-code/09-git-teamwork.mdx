## 前言

前几篇文章聊的都是你一个人用 Claude Code 的场景——怎么写 Prompt、怎么用 Agent、怎么管上下文。但软件开发从来不是一个人的事。你写好了代码，要提交、要创建 PR、要让人 review、要合并到主分支。这个过程中 Claude Code 能帮什么忙？

更关键的问题是：当团队里不止你一个人用 Claude Code，怎么保证大家用它写出来的代码风格一致、质量可控？怎么避免有人用它生成了不安全的代码直接提交？

这篇文章把视角从"个人效率工具"切换到"团队协作平台"，看看怎么把 Claude Code 融入团队的 Git 工作流。

## Commit Message：让它帮你写，但别让它直接提交

写 commit message 是一件典型的小事但烦人的事。你知道这次改了什么，但要把改动浓缩成一行简洁的描述——`feat(auth): add OAuth2 refresh token support`——每次都要想半天。

Claude Code 擅长这个。它会分析你当前的 `git diff`，自动生成符合 Conventional Commits 规范的 commit message：

```
你: "帮我提交这次的改动"

Claude Code 分析变更后生成:

feat(course): add pagination to course list endpoint

- Add page and pageSize query parameters
- Implement offset-based pagination with default page size 20
- Return total count for client-side pagination controls

Co-Authored-By: Claude <noreply@anthropic.com>
```

它做的不仅仅是拼一行文字。它会读取你 CLAUDE.md 里定义的 commit 规范（如果你写了的话），按照团队的约定格式化输出。它还会区分 feat/fix/refactor/docs 这些前缀，根据改动内容自动选择。

但有一个原则：**让它生成 commit message，但不要让它自动提交**。你应该在提交前看一眼它生成的内容，确认描述准确、没有遗漏敏感信息。`Co-Authored-By` 标记也很重要——它让团队知道这段代码有 AI 参与，保持透明。

## PR 审查：让它做第一轮筛选

PR 审查是最耗时的团队协作环节之一。Claude Code 可以做第一轮自动审查，帮你筛掉明显的问题，让人工 review 聚焦在真正需要人判断的地方。

使用方式很简单：告诉它审查哪个 PR 或哪些文件的变更。

```
你: "审查当前分支相对于 main 的所有变更，
     重点关注类型安全和错误处理"
```

Claude Code 会读取 `git diff main...HEAD`，逐文件检查，输出一份结构化的审查报告。关键在于你要告诉它**审查的重点**——如果你不指定，它会面面俱到地检查所有方面，报告可能很长而且包含很多无关紧要的"建议"。

实际操作中，一个好的策略是分层审查：

```
# 第一轮：让 Claude Code 做自动检查
你: "审查这个 PR，只关注以下三类问题：
     1. 类型安全问题（any、不安全的类型断言）
     2. 错误处理遗漏（缺少 try-catch 的异步操作）
     3. 安全问题（硬编码的密钥、未验证的用户输入）"

# 第二轮：人工审查 Claude Code 标记出的问题
# 以及 Claude Code 无法判断的架构和业务逻辑
```

这样 Claude Code 做的是"过滤"而不是"替代"——它帮你省掉看几十个文件里有没有低级错误的时间，但最终的架构判断、业务逻辑审核仍然需要人来完成。

## Hooks：在提交时自动检查

Claude Code 支持 Hooks 机制——你可以在特定事件（比如提交代码、运行命令）发生时自动执行检查。

最常见的用法是 pre-commit hook：在每次 `git commit` 之前自动跑一遍 lint 和类型检查。如果检查不通过，提交会被阻止。

```json
// .claude/settings.json 中的 Hooks 配置
{
  "hooks": {
    "pre-commit": {
      "command": "npx tsc --noEmit && npm run lint",
      "description": "运行类型检查和 lint"
    }
  }
}
```

还有 commit message 的 hook：防止提交不规范的 message。

```json
{
  "hooks": {
    "commit-msg": {
      "command": "node scripts/validate-commit-msg.js",
      "description": "验证 commit message 格式"
    }
  }
}
```

Hooks 的价值在于**强制执行**。没有 hook 的时候，规范只靠自觉——有人忘了跑 lint 就直接提交了。有了 hook，不符合规范的代码根本提交不上去。

但要注意 hook 的粒度。如果 pre-commit hook 跑太多检查（lint、类型检查、单元测试、格式化……），每次提交都要等很久，开发者会倾向于跳过 hook（`git commit --no-verify`）。保持 hook 快速且聚焦——只检查最关键的几项。

## CLAUDE.md 的团队规范

前几篇文章我们主要从个人角度聊 CLAUDE.md——它告诉 Claude Code "这个项目用什么技术栈"。在团队场景下，CLAUDE.md 是**团队共识的载体**。

一个团队级的 CLAUDE.md 应该包含什么？有一个简单的判断标准：**如果一个规范需要你每次对话都重复告诉 Claude Code，它就该写进 CLAUDE.md**。

典型内容包括：

- 技术栈和版本（TypeScript 5.x、NestJS 10.x、PostgreSQL 16）
- 代码风格约定（命名规范、文件组织、导入顺序）
- Git 工作流（分支命名、commit message 格式、PR 流程）
- 测试要求（覆盖率标准、测试框架、mock 策略）
- 禁止事项（不允许用 `any`、不允许跳过 lint）

不该放什么？**一次性的指令**（"这个 PR 改了认证模块"）和**频繁变化的规范**（"当前 Sprint 的目标"）。这些在对话中说就好。

CLAUDE.md 的维护和代码一样重要。如果规范变了但 CLAUDE.md 没更新，Claude Code 还是按旧规范生成代码，结果和新规范冲突。建议在每次架构决策变更时，同步更新 CLAUDE.md——把它当成项目的"活文档"。

## 多人协作：权限和 Memory

团队里不同角色对 Claude Code 的需求不同。后端开发者需要它能操作数据库命令，前端开发者不需要；Tech Lead 可能需要更高的权限来执行部署操作， junior 开发者应该被限制在只读和开发分支内。

Claude Code 的权限模式可以按角色区分：

- **开发者日常使用**：允许读写代码、运行测试和 lint，不允许 `git push`、`npm publish` 等影响外部环境的操作
- **代码审查**：只读模式，可以读代码、写评论，但不能修改文件
- **Tech Lead**：完全权限，包括部署和发布

Memory 系统在团队场景下也有用处。项目级的 Memory 目录可以提交到 Git 仓库，所有团队成员共享。比如"认证模块决定使用 JWT 而不是 Session"、"数据库索引策略改用复合索引"这类决策记录，写在 Memory 里比写在某个人的对话历史中有用得多——因为它对所有团队成员可见。

## 团队推广的真实阻力

把 Claude Code 引入团队不是技术问题，是人的问题。你会遇到三种典型阻力。

**安全顾虑**。"它会把我们的代码发送到外部服务器吗？会不会泄露商业机密？"这种担心很合理。应对方式是：让安全团队审核 Claude Code 的数据处理方式，明确哪些代码可以交给 AI 处理、哪些不能（比如密钥文件、用户数据）。CLAUDE.md 里可以加一条规则："不要读取或引用 .env 文件和包含密钥的配置"。

**质量担忧**。"它生成的代码能靠谱吗？不会引入 bug 吧？"应对方式是：不要让 AI 代码直接进主分支。要求所有 AI 辅助的代码必须经过人工 review、必须有对应的测试。把 Claude Code 定位为"帮你写初稿"的工具，而不是"替代你思考"的工具。

**习惯改变**。"我已经有自己的工作流了，不想学新工具。"这是最常见也最难应对的。不要试图一步到位。从一个痛点切入——比如"每次写 commit message 都很烦"，让 Claude Code 先解决这一个问题。等团队感受到价值后，再逐步扩展到 code review、重构、测试生成等场景。
