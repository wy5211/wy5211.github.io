上篇我们掌握了重构和测试。但个人用得很爽，不代表团队能顺畅协作。

commit message 写不好、PR review 漏问题、CLAUDE.md 规范不统一——这些都是团队协作的痛点。

## Git 集成

### 自动生成 commit message

```bash
> 根据当前的代码变更生成 commit message
> 修复了订单查询的 bug，添加了日志
```

Claude Code 会生成：

```
fix: 修复订单查询性能问题

问题：
- 课程列表接口响应慢（>3s）
- 未使用数据库索引

修改：
- 在 course.title 字段添加索引
- 优化查询条件，避免 N+1 问题
- 添加性能日志

测试：已验证响应时间 < 200ms

Refs: #123
```

### 创建 PR

```bash
> 创建 PR，分支是 feature/add-course-cache
> 标题：优化课程列表接口性能
> 描述：见 commit message
```

## CLAUDE.md 团队规范

### 写什么、不写什么

**应该写**：

- 项目概述和技术栈
- 代码风格和命名约定
- 目录结构和文件组织
- 常见任务的工作流

**不应该写**：

- 具体的一次性指令（在对话中说就行）
- 变化的规范（这会让 CLAUDE.md 很难维护）
- 敏感信息（API Key、密码）

### 保持更新的方法

- 重大架构变更时更新
- 每季度 review 一次
- 有新的约定时补充

## 能力边界

### Claude Code 擅长的

- 理解项目结构和代码
- 按规范生成代码
- 分析问题并给出方案
- 自动化重复性工作

### Claude Code 不擅长的

- 业务逻辑的判断（需要你把关）
- 架构决策（需要团队讨论）
- 代码审查的最终决策

## 日常使用工作流

```bash
# 早上：查看 PR review
> 审查 @pr/123 的代码变更

# 上午：修 bug
> 修复 @src/auth/login.ts 中的 token 验证问题

# 下午：重构模块
> 用 Plan Mode 重构 OrderService

# 下班前：提交代码
> 生成 commit message 并提交
```
