上篇我们学会了怎么和 Claude Code 有效沟通。但如果你给它一个复杂任务——"重构认证模块支持 OAuth 2.0"，它可能会二话不说就开始改代码。

结果呢？改了 30 个文件，方向全错了，回滚都要半天。

复杂任务不能直接动手，得先规划。Plan Mode 就是干这个的。

## Plan Mode 的本质

Plan Mode 和直接对话的区别：

```
直接对话模式：
你说："重构认证模块"
  ↓
Claude Code：开始改代码
  ↓
结果：改了 30 个文件，方向错了

Plan Mode：
你说："重构认证模块"
  ↓
Claude Code：先探索代码、设计方案
  ↓
你确认后：才开始动手
  ↓
结果：按计划执行，不会走偏
```

## 什么时候该用 Plan Mode

**适合用**：

- 重构大型模块（10+ 文件）
- 涉及架构变更
- 不熟悉的老代码
- 多人协作的项目

**不适合用**：

- 修复简单 bug
- 添加小功能
- 单文件修改
- 你很熟悉的代码

判断标准：**如果这个任务让你自己写都要先想清楚，那就用 Plan Mode**。

## Plan Mode 工作流程

### 1. 探索：理解现有代码

```bash
# 进入 Plan Mode
claude plan

# 给它任务
规划：给课程列表接口添加 Redis 缓存，解决响应慢的问题
```

Claude Code 会先探索代码：

```
🔍 正在探索课程模块...
✓ 找到课程列表接口：src/modules/course/course.controller.ts
✓ 找到服务层：src/modules/course/course.service.ts
✓ 分析数据库查询：使用了 TypeORM 的 findMany
✓ 查看日志：没有找到性能日志
```

### 2. 分析：评估影响和风险

```bash
📊 影响范围分析：
需要修改的文件：
- src/modules/course/course.service.ts（添加缓存逻辑）
- src/modules/course/course.controller.ts（添加缓存失效）
- src/config/redis.config.ts（Redis 配置）

⚠️ 潜在风险：
1. 缓存一致性问题：课程更新后缓存要失效
2. Redis 单点故障：需要高可用方案
3. 内存占用：缓存数据不能太大

💡 缓解措施：
- 使用合理的 TTL（5 分钟）
- 实现缓存预热机制
- 监控 Redis 内存使用
```

### 3. 规划：设计实施步骤

```bash
📋 实施计划：

第一步：Redis 基础设施（1 天）
  1.1 安装 Redis 依赖
  1.2 创建 Redis 配置
  1.3 实现 Redis 连接测试

第二步：缓存层实现（2 天）
  2.1 在 course.service.ts 添加缓存方法
  2.2 实现缓存键生成策略
  2.3 实现缓存失效逻辑

第三步：测试验证（1 天）
  3.1 单元测试：缓存逻辑
  3.2 集成测试：端到端验证
  3.3 性能测试：对比缓存前后

总预计：4 天
```

### 4. 确认：你批准后再执行

```bash
❓ 是否确认此计划？
  /confirm - 开始执行
  /modify - 修改计划
  /save - 保存计划到文件
  /abort - 取消
```

## Plan Mode 的价值

你可能会想：这比我直接写代码慢多了。

但计算一下：

```
直接改代码：
- 写代码：2 天
- 发现方向错了：回滚 1 天
- 重新实现：2 天
- 总计：5 天 + 压力

Plan Mode：
- 规划：半天
- 写代码：2 天
- 验证通过：半天
- 总计：3 天 + 心安
```

更重要的是，Plan Mode 避免了"改完发现方向不对"的风险。

## 在 Claude Code 中使用 Plan Mode

```bash
# 方式 1：对话中进入
> 我想重构认证模块
> 先用 Plan Mode 规划一下

# 方式 2：直接启动
claude plan "重构认证模块，添加 OAuth 2.0 支持"

# 方式 3：从文件加载
claude plan --load migration-plan.md
```

## 常见误区

### 误区 1：所有任务都用 Plan Mode

```
❌ 用 Plan Mode 修复一个 bug
✅ 直接对话："修复 @src/auth/login.ts 中的类型错误"
```

Plan Mode 的探索和规划是有成本的。简单任务直接做反而更快。

### 误区 2：不看计划直接确认

```
❌ Claude Code 给了计划，你直接 /confirm
✅ 仔细审查计划，调整不合理的地方再确认
```

Plan Mode 是辅助工具，最终决策权在你。

### 误区 3：规划太细

```
❌ "第一步：创建文件，第二步：写函数，第三步：写测试"
✅ "第一阶段：实现基础缓存（1 天），第二阶段：集成测试（半天）"
```

规划要有合理的粒度，太细反而失去灵活性。
