## 前言

前面 9 篇文章，我们拆着讲了 Claude Code 的各个功能：怎么写 Prompt、怎么用 Plan Mode、怎么让 Agent 分工、怎么封装 Skill、怎么管上下文、怎么融入 Git 工作流。但你可能读完之后有一个感觉："道理我都懂，但实际工作的时候，我到底该怎么用？"

这篇就是回答这个问题的。我们不学新功能了，而是把前面的知识串成一个**可以每天用的操作手册**。同时聊一个经常被忽略的话题：Claude Code 不擅长什么，以及哪些"坑"你需要刻意避开。

## 日常工作流：一个真实的开发日

用一个典型的开发日来展示 Claude Code 在不同任务中的使用策略。假设你在做 edu-platform-api 项目，这是一个在线教育平台的后端。

### 早上：查看 PR review

```
你: "审查 PR #142 的变更，重点检查类型安全和错误处理"
```

这里你不需要打开 GitHub 页面一个个文件看。Claude Code 会拉取 PR 的 diff，按你指定的维度检查。这个场景用的是第 9 篇讲的 PR 审查技巧。

### 上午：修一个 bug

课程列表接口在大数据量下响应超过 3 秒。这是一个需要先分析再修复的任务。

```
# 第一轮：分析问题（Plan Mode）
你: "进入 Plan Mode，分析课程列表接口慢的原因"

Claude Code 读取代码后发现：
1. 查询没有使用索引
2. 每次 N+1 加载了讲师信息和标签
3. 没有分页，一次返回全部数据

# 第二轮：执行修复
你: "按你分析的方案修复，先加分页和索引优化"
```

Plan Mode 是处理 bug 的好习惯——先搞清楚原因再动手，而不是上来就改。这是第 3 篇的内容。

### 下午：重构 OrderService

OrderService 里有一堆重复逻辑，测试覆盖率不到 20%。这是一个大改动，需要小心处理。

```
# 用 Plan Mode 先规划重构方案
你: "分析 @src/modules/order/order.service.ts，
     找出重复逻辑，设计重构方案。先不改代码。"

# 确认方案后，分步执行
你: "先提取公共的 calculateTotal 方法，补上测试，再提交"
```

重构用 Plan Mode + 分步执行是最安全的方式。每一步做完跑一下测试，确认没破坏现有功能再继续。这是第 6 篇讲的代码重构策略。

### 下班前：提交代码

```
你: "帮我提交这次的改动"
```

Claude Code 分析 diff，生成 Conventional Commits 格式的 message。你确认后提交。这是第 9 篇讲的 Git 集成。

### 回顾：什么任务用什么策略

| 任务类型    | 推荐策略                   | 为什么                   |
| ----------- | -------------------------- | ------------------------ |
| 修小 bug    | 直接说清楚问题，一步到位   | 范围小，不需要规划       |
| 修复杂 bug  | Plan Mode 分析 + 分步修复  | 先定位原因再动手         |
| 新功能开发  | Plan Mode 规划 + 分步实现  | 涉及多文件，需要确认方向 |
| 重构        | Plan Mode + 每步跑测试     | 安全第一，小步前进       |
| 补测试      | 直接说明测试范围           | 范围明确，不需要规划     |
| Code Review | 指定审查重点               | 避免面面俱到的长报告     |
| 提交代码    | 让它生成 message，人工确认 | 快速但保持掌控           |

## Claude Code 的能力边界

知道它擅长什么和不擅长什么，比多学三个技巧有用得多。

### 它擅长的

**代码生成和修改**。这是 Claude Code 最强的能力。给它明确的需求和上下文，它能生成质量不错的代码——尤其是样板代码、CRUD 操作、DTO 定义、测试用例这类有明确模式的工作。

**代码分析和理解**。它能快速阅读大量代码，找出模式、发现不一致、总结结构。用 Explore Agent 分析一个陌生的代码库，比你自己一个个文件翻快得多。

**重构建议**。它擅长识别代码中的坏味道（重复逻辑、过长函数、类型不安全），并给出具体的改进建议。Plan Mode 下它会先分析再给方案，质量通常不错。

**文档和测试生成**。写文档和写测试是开发者最不愿意做但又必须做的事。Claude Code 不介意做这些，而且做得还不错。

### 它不擅长的

**架构决策**。它可以说出"微服务和单体架构的优劣"，但要不要拆微服务——这个决策需要考虑团队规模、业务阶段、运维能力、成本预算，这些上下文它通常不具备。架构决策需要人来拍板，Claude Code 可以帮你分析利弊。

**业务逻辑设计**。它不知道你的业务规则。比如"课程退款的流程是什么"、"什么情况下允许学生免费重修"——这些需要你和产品经理确认，而不是让 AI 猜。

**跨系统的集成调试**。当 bug 涉及多个系统的交互（前端 + 后端 + 第三方 API + 数据库），Claude Code 能帮你分析单个系统的代码，但它无法运行整个系统来复现问题。这类调试还是需要你自己的经验和直觉。

**评估代码的"长期影响"**。它能告诉你"这段代码现在能跑"，但很难评估"半年后这段代码会不会成为技术债"。长期影响需要人对项目 Roadmap 的理解。

## 三个最危险的误区

### 误区一：不验证就提交

Claude Code 生成了代码，看起来没问题，你直接提交了。等到 code review 的时候才发现——类型断言不安全、错误处理有遗漏、边界情况没覆盖。

**原则：AI 生成的代码和同事写的代码一样，都需要 review 和测试。** 甚至应该更严格——因为它"看起来很对"但实际上可能有微妙的逻辑错误。

一个实用的习惯：让 Claude Code 自己生成测试，然后跑一遍测试看看是否通过。测试通过不代表代码完美，但能帮你挡住最明显的问题。

### 误区二：把 Prompt 写得太细，变成"人工编译器"

有些人在 Prompt 里写了 20 行指令，精确到每一行代码该怎么写。这不叫"用 AI"，这叫"用自然语言写代码"。如果你已经知道每一步该怎么做，直接自己写可能更快。

**原则：告诉它"做什么"和"有什么约束"，不要告诉它"怎么做"。** 比如"实现一个用户注册接口，用 bcrypt 加密密码"就够了，不需要说"先创建一个 AuthService 类，然后在里面写一个 register 方法，方法签名是 async register(dto: CreateUserDto)"。

### 误区三：过度依赖，失去独立思考能力

用了一段时间 Claude Code 之后，你可能会发现自己越来越不愿意自己写代码了。简单的方法也想让它生成，能自己想出来的 bug 也让它帮忙查。

这个趋势很危险。工具应该增强你的能力，而不是替代你的思考。如果你发现自己写代码的速度变慢了、对项目的理解变浅了——是时候刻意减少对它的依赖了。

一个建议：每周至少有一天完全不用 Claude Code。强迫自己手动写代码、手动 debug，保持肌肉记忆。

## 有 vs 没有：真实的效率差距

聊了这么多理论和技巧，最后说点实际的。Claude Code 到底能帮你省多少时间？

坦率地说，这取决于你怎么用它。如果你只是用它来"帮我写一个函数"，那它就是一个稍快一点的代码补全工具，可能省 10-20% 的时间。

如果你按这个系列讲的方式用——Plan Mode 规划复杂任务、Agent 探索陌生代码库、Skill 封装重复流程、上下文管理保持高效对话——它能帮你省 40-60% 的"重复劳动时间"。注意是重复劳动，不是思考时间。设计架构、理解业务需求、做技术决策——这些事 AI 帮不了你，也不应该帮你做。

最真实的效率提升往往不是来自某个炫酷的功能，而是来自**微习惯的积累**。比如：

- 每次改完代码让它跑一下 lint 和测试，省掉了"部署后发现 lint 报错"的时间
- 每次提交让它生成 commit message，省掉了"想半天措辞"的时间
- 每次接手新模块让它先用 Explore Agent 分析结构，省掉了"翻半天代码搞不懂"的时间

这些微习惯单个看起来不值一提，但一天积累下来，可能就是 1-2 个小时。

## 持续学习：跟上一个更新很快的工具

Claude Code 更新频率很高，新功能、新 Agent 类型、新 Skill 生态都在持续迭代。怎么跟上？

最简单的方式：定期看官方的 changelog 和文档更新。Claude Code 的更新日志通常会列出新增的功能和变更的 API，花 10 分钟扫一遍就够了。

另一个方式：关注社区。GitHub 上的 Skills 仓库、Discord 里的讨论、Twitter 上的分享——这些地方经常有高手的用法和技巧，比官方文档更贴近实际使用场景。

最后，回到这个系列的核心观点：**工具的价值不在于你学了多少功能，而在于你形成了稳定的工作流。** 学了 10 个功能但每天只用其中 2 个，比学了 2 个功能但每天都在用要好得多。

```

  01 快速入门：安装配置、CLAUDE.md、权限模型
  02 高效对话：Prompt 四要素、@ 引用、沟通陷阱

核心篇（03-05）—— 深入进阶
  03 Plan Mode：复杂任务先规划再执行
  04 Agent System：Explore、Plan、General Purpose 三种 Agent
  05 Skills：发现、使用与自定义开发

实战篇（06-08）—— 代码层面
  06 代码重构与测试：安全重构、TDD、分步验证
  07 调试与问题排查：描述问题、分析日志、验证方案
  08 上下文管理：Token 优化、Memory、CLAUDE.md 分层

团队篇（09-10）—— 协作与综合
  09 Git 工作流：commit、PR review、Hooks、团队推广
  10 综合实战：日常工作流、能力边界、常见误区
```

如果你是从第 1 篇跟到这里的，你现在应该能做到：

- 写出让 Claude Code 第一次就理解的 Prompt
- 用 Plan Mode 安全地处理复杂任务
- 用 Agent System 自动化代码探索和修改
- 封装 Skill 让重复性工作一键执行
- 安全地重构代码、补齐测试
- 高效排查问题、管理大项目的上下文
- 把 Claude Code 融入团队的 Git 工作流
- 清楚地知道 Claude Code 能做什么、不能做什么

这就是从"会用"到"玩得溜"的完整路径。
