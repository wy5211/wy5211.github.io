## 前言

在上一篇文章中，我们学习了如何在 Monorepo 中组织 tRPC 项目，实现了代码共享和统一构建。今天我们来聊一个同样重要的话题：测试。

测试的目的不是为了追求覆盖率数字，而是为了给你改代码的信心。当你重构业务逻辑或添加新功能时，有测试保驾护航，就能快速发现问题。tRPC 的类型系统天然适合测试，因为 Mock 数据必须符合类型定义，否则编译器就会报错。

对于我们的协作任务管理平台，测试重点包括：

- 任务 CRUD 操作是否正确
- 权限控制是否生效
- 实时通知是否可靠触发
- 边界情况是否妥善处理

今天我们将学习如何为 tRPC 应用编写全面的测试。

## 为什么要测试 tRPC 应用

在传统 API 开发中，测试往往是个痛点。你需要构造各种 HTTP 请求，处理 JSON 序列化，还要手动确保 Mock 数据格式正确。一旦 API 接口变化，测试可能还能通过，但实际运行时却因为类型不匹配而崩溃。

tRPC 改变了这一切。因为前后端共享类型，测试时构造的 Mock 数据必须在编译时就符合类型要求。这意味着：

**类型安全的测试**：如果你 Mock 的数据结构不对，TypeScript 会直接报错，不用等测试运行才发现问题。

**可预测的 API**：tRPC 的输入输出都有明确的类型定义，测试时不需要猜测 API 返回什么。

**易于 Mock**：清晰的数据结构让 Mock 变得简单，不需要手动构造复杂的 HTTP 响应。

**集成测试友好**：tRPC 本质上还是 HTTP，可以用标准的 HTTP 测试工具。

接下来我们看看具体怎么实现。

## 项目配置

### 安装测试依赖

首先安装 Vitest 和相关测试库。Vitest 是 Vite 原生的测试框架，速度更快，配置更简单。

```bash
# 安装 Vitest
npm install -D vitest @vitejs/plugin-react

# 安装测试工具库
npm install -D @testing-library/react @testing-library/jest-dom
npm install -D @testing-library/user-event

# 安装测试环境
npm install -D happy-dom
```

### 配置 Vitest

创建 Vitest 配置文件：

```typescript
// vitest.config.ts
import { defineConfig } from "vitest/config";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  test: {
    globals: true, // 使用全局 API，不需要 import
    environment: "happy-dom", // 轻量级的 DOM 环境
    setupFiles: ["./src/test/setup.ts"], // 测试前的初始化文件
  },
});
```

这个配置做了几件事：启用全局 API 让测试代码更简洁，使用 happy-dom 作为浏览器环境（比 jsdom 快得多），指定测试初始化文件。

### 测试环境设置

在测试开始前，我们需要做一些准备工作：

```typescript
// src/test/setup.ts
import { expect, afterEach } from "vitest";
import { cleanup } from "@testing-library/react";
import * as matchers from "@testing-library/jest-dom/matchers";

// 扩展 Jest 的 matchers，让断言更丰富
expect.extend(matchers);

// 每个测试后自动清理 DOM
afterEach(() => {
  cleanup();
});
```

这个文件会在每个测试前执行，扩展了断言功能并自动清理测试产生的 DOM，避免测试相互干扰。

## 单元测试

单元测试关注单个函数或组件的行为，测试范围小、运行快、定位问题容易。在 tRPC 应用中，单元测试主要用于测试 Procedure 的业务逻辑。

### 测试工具函数

在写测试之前，我们需要一些辅助工具：

```typescript
// src/test/utils.ts
import { Context } from "@/trpc/context";

// 创建测试用的 tRPC Caller
export function createTestCaller(ctx: Context) {
  const { t } = require("@/trpc");
  return t.createCallerFactory()(ctx);
}

// Mock 数据生成器
export function mockUser(overrides = {}) {
  return {
    id: "test-user-id",
    name: "Test User",
    email: "test@example.com",
    role: "USER",
    ...overrides,
  };
}

export function mockTask(overrides = {}) {
  return {
    id: "test-task-id",
    title: "Test Task",
    description: "Test description",
    status: "TODO",
    priority: "MEDIUM",
    assigneeId: "test-user-id",
    ...overrides,
  };
}
```

这些工具函数让我们能快速创建测试所需的上下文和 Mock 数据。`createTestCaller` 创建一个可以直接调用 Procedure 的函数，不需要经过 HTTP 层。

### 测试任务列表查询

让我们从最简单的开始：测试获取任务列表的 Procedure。

```typescript
// src/trpc/routers/tasks.test.ts
import { describe, it, expect, beforeEach } from "vitest";
import { PrismaClient } from "@prisma/client";
import { createTestCaller, mockUser, mockTask } from "@/test/utils";

describe("tasks.list", () => {
  let prisma: PrismaClient;
  let caller: ReturnType<typeof createTestCaller>;

  beforeEach(async () => {
    // 每个测试前创建新的数据库连接
    prisma = new PrismaClient({
      datasources: {
        db: {
          url: process.env.DATABASE_URL_TEST,
        },
      },
    });

    // 清空测试数据
    await prisma.task.deleteMany();

    // 创建测试用户和任务
    const user = await prisma.user.create({
      data: mockUser(),
    });

    await prisma.task.createMany({
      data: [
        mockTask({ title: "Task 1", userId: user.id }),
        mockTask({ title: "Task 2", userId: user.id }),
        mockTask({ title: "Task 3", userId: user.id }),
      ],
    });

    // 创建测试调用器
    caller = createTestCaller({
      user: await prisma.user.findUnique({
        where: { id: user.id },
      }),
      prisma,
    });
  });

  it("应该返回任务列表", async () => {
    const result = await caller.tasks.list.query({
      page: 1,
      limit: 10,
    });

    expect(result.tasks).toHaveLength(3);
    expect(result.tasks[0].title).toBe("Task 1");
    expect(result.pagination.total).toBe(3);
  });

  it("应该支持分页", async () => {
    const result = await caller.tasks.list.query({
      page: 1,
      limit: 2,
    });

    expect(result.tasks).toHaveLength(2);
    expect(result.pagination.totalPages).toBe(2);
  });

  it("应该按状态过滤", async () => {
    await prisma.task.update({
      where: { title: "Task 1" },
      data: { status: "DONE" },
    });

    const result = await caller.tasks.list.query({
      page: 1,
      limit: 10,
      status: "DONE",
    });

    expect(result.tasks).toHaveLength(1);
    expect(result.tasks[0].status).toBe("DONE");
  });
});
```

这段测试展示了几个要点：使用 `beforeEach` 确保每个测试独立运行，测试正常的查询、分页和过滤功能。因为使用了 tRPC 的类型系统，`caller.tasks.list.query` 的参数和返回值都有完整的类型提示，写测试时不容易出错。

### 测试权限控制

权限控制是测试的重点，因为这是安全相关的重要逻辑。

```typescript
describe("tasks.update", () => {
  beforeEach(async () => {
    // 创建两个用户
    owner = await prisma.user.create({
      data: mockUser({ id: "owner-id", email: "owner@example.com" }),
    });

    other = await prisma.user.create({
      data: mockUser({ id: "other-id", email: "other@example.com" }),
    });

    // owner 创建一个任务
    task = await prisma.task.create({
      data: mockTask({
        userId: owner.id,
        title: "Owner's Task",
      }),
    });
  });

  it("任务创建者可以更新任务", async () => {
    const caller = createTestCaller({ user: owner, prisma });

    const result = await caller.tasks.update.mutate({
      id: task.id,
      title: "Updated Title",
    });

    expect(result.title).toBe("Updated Title");
  });

  it("其他用户不能更新别人的任务", async () => {
    const caller = createTestCaller({ user: other, prisma });

    await expect(
      caller.tasks.update.mutate({
        id: task.id,
        title: "Hacked Title",
      })
    ).rejects.toThrow("FORBIDDEN");
  });

  it("未登录用户不能更新任务", async () => {
    const caller = createTestCaller({ user: null, prisma });

    await expect(
      caller.tasks.update.mutate({
        id: task.id,
        title: "Anonymous Title",
      })
    ).rejects.toThrow("UNAUTHORIZED");
  });
});
```

这些测试确保了只有任务创建者才能修改任务，其他人和未登录用户都会被拒绝。注意这里我们测试的是错误情况，使用 `rejects.toThrow` 来验证是否抛出预期的错误。

### 测试输入验证

tRPC 使用 Zod 进行输入验证，我们需要测试无效输入是否能被正确拒绝。

```typescript
describe("tasks.create - 输入验证", () => {
  it("应该拒绝空标题", async () => {
    const caller = createTestCaller({ user: mockUser(), prisma });

    await expect(
      caller.tasks.create.mutate({
        title: "", // 空标题应该被拒绝
        description: "Valid description",
      })
    ).rejects.toThrow();
  });

  it("应该拒绝过长的标题", async () => {
    const caller = createTestCaller({ user: mockUser(), prisma });

    await expect(
      caller.tasks.create.mutate({
        title: "a".repeat(201), // 假设限制 200 字符
        description: "Valid description",
      })
    ).rejects.toThrow();
  });

  it("应该拒绝无效的优先级", async () => {
    const caller = createTestCaller({ user: mockUser(), prisma });

    await expect(
      caller.tasks.create.mutate({
        title: "Valid Title",
        priority: "INVALID_PRIORITY", // 枚举之外的值
      })
    ).rejects.toThrow();
  });
});
```

这些测试确保了 Zod schema 定义的验证规则确实生效。当你修改 schema 时，这些测试会帮你确保没有意外放宽限制。

## 集成测试

单元测试关注单个 Procedure，而集成测试则测试多个组件协同工作的场景。对于我们的协作任务管理平台，典型的集成测试场景包括：创建任务后，其他用户能看到更新；任务状态变更后，订阅者收到通知。

### 设置 HTTP 测试服务器

集成测试需要通过真实的 HTTP 调用，所以我们需要启动一个测试服务器。

```typescript
// src/test/api/server.ts
import { createTRPCProxyClient, httpBatchLink } from "@trpc/client";
import { AppRouter } from "@/trpc";
import { createExpressServer } from "./express";

let testServer: any;

export async function setupTestServer() {
  // 清空测试数据库
  await prisma.task.deleteMany();
  await prisma.user.deleteMany();

  // 创建测试用户
  const user = await prisma.user.create({
    data: mockUser(),
  });

  // 启动 HTTP 服务器
  testServer = createExpressServer().listen(0); // 端口 0 表示随机分配
  const port = testServer.address().port;

  // 创建 tRPC 客户端
  const client = createTRPCProxyClient<AppRouter>({
    links: [
      httpBatchLink({
        url: `http://localhost:${port}/trpc`,
      }),
    ],
  });

  return { client, user, port };
}

export async function teardownTestServer() {
  if (testServer) {
    testServer.close();
  }
}
```

这个辅助函数启动一个真实的 Express 服务器，监听随机端口，并返回一个 tRPC 客户端。这样我们就能通过 HTTP 调用 API，测试完整的请求响应流程。

### 测试完整的用户流程

让我们测试一个典型场景：用户创建任务后，其他用户能看到更新。

```typescript
// src/test/integration/task-flow.test.ts
import { describe, it, expect, beforeAll, afterAll } from "vitest";
import { setupTestServer, teardownTestServer } from "../api/server";
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient({
  datasources: {
    db: {
      url: process.env.DATABASE_URL_TEST,
    },
  },
});

describe("任务创建和查看流程", () => {
  let client: any;
  let user: any;

  beforeAll(async () => {
    const setup = await setupTestServer();
    client = setup.client;
    user = setup.user;
  });

  afterAll(async () => {
    await teardownTestServer();
  });

  it("用户创建任务后，其他用户能看到", async () => {
    // 用户 A 创建任务
    const created = await client.tasks.create.mutate({
      title: "Integration Test Task",
      description: "Testing the full flow",
      priority: "HIGH",
    });

    expect(created).toHaveProperty("id");
    expect(created.title).toBe("Integration Test Task");

    // 用户 B 查询任务列表
    const list = await client.tasks.list.query({
      page: 1,
      limit: 10,
    });

    expect(list.tasks).toContainEqual(
      expect.objectContaining({
        id: created.id,
        title: "Integration Test Task",
      })
    );
  });

  it("任务状态更新后，查询能反映最新状态", async () => {
    const task = await client.tasks.create.mutate({
      title: "Status Test Task",
      priority: "MEDIUM",
    });

    // 更新状态
    await client.tasks.update.mutate({
      id: task.id,
      status: "IN_PROGRESS",
    });

    // 查询任务详情
    const updated = await client.tasks.byId.query({
      id: task.id,
    });

    expect(updated.status).toBe("IN_PROGRESS");
  });
});
```

这个测试涵盖了创建和查看的完整流程，验证了数据能正确地保存和检索。因为是通过 HTTP 调用，也测试了序列化、反序列化的过程。

### 测试并发操作

协作平台的核心是多用户同时操作，我们需要测试并发场景。

```typescript
describe("并发操作", () => {
  it("多个用户同时更新任务，应该只有一个成功", async () => {
    const { client, user } = await setupTestServer();

    const task = await client.tasks.create.mutate({
      title: "Concurrency Test",
      priority: "MEDIUM",
    });

    // 模拟两个用户同时更新
    const update1 = client.tasks.update.mutate({
      id: task.id,
      title: "Update 1",
    });

    const update2 = client.tasks.update.mutate({
      id: task.id,
      title: "Update 2",
    });

    // 等待两个操作完成
    const results = await Promise.allSettled([update1, update2]);

    // 应该至少有一个失败
    const successCount = results.filter((r) => r.status === "fulfilled").length;
    const failCount = results.filter((r) => r.status === "rejected").length;

    expect(successCount + failCount).toBe(2);
    expect(Math.min(successCount, failCount)).toBeGreaterThanOrEqual(1);
  });
});
```

这个测试模拟了两个用户同时修改同一个任务的场景。在实际应用中，你可能需要使用乐观锁或悲观锁来处理并发冲突。

## React 组件测试

测试 React 组件时，我们通常关注组件的渲染逻辑和用户交互。Testing Library 提供了一套工具，让测试更接近用户真实操作。

### Mock tRPC Hooks

在测试中使用 tRPC 时，我们需要 Mock tRPC 的 React Hooks。

```typescript
// src/test/mocks/trpc.ts
import { jest } from "@jest/globals";

export const mockTrpc = {
  tasks: {
    list: {
      useQuery: jest.fn(),
    },
    create: {
      useMutation: jest.fn(),
    },
    update: {
      useMutation: jest.fn(),
    },
  },
};

export function mockQuery(data: any, isLoading = false) {
  return {
    data,
    isLoading,
    error: null,
    refetch: jest.fn(),
  };
}

export function mockMutation() {
  return {
    mutate: jest.fn(),
    mutateAsync: jest.fn(),
    isLoading: false,
    error: null,
  };
}
```

这些 Mock 函数让我们能控制 tRPC Hooks 的返回值，测试组件在不同状态下的表现。

### 测试任务列表组件

```typescript
// src/components/TaskList.test.tsx
import { describe, it, expect, vi } from "vitest";
import { render, screen, waitFor } from "@testing-library/react";
import { TaskList } from "./TaskList";
import { mockTrpc, mockQuery } from "@/test/mocks/trpc";

vi.mock("@/lib/trpc", () => ({
  trpc: mockTrpc,
}));

describe("TaskList", () => {
  it("应该显示加载状态", () => {
    mockTrpc.tasks.list.useQuery.mockReturnValue(
      mockQuery(undefined, true)
    );

    render(<TaskList />);

    expect(screen.getByText("加载中...")).toBeInTheDocument();
  });

  it("应该显示任务列表", async () => {
    const mockTasks = [
      { id: "1", title: "Task 1", status: "TODO" },
      { id: "2", title: "Task 2", status: "DONE" },
    ];

    mockTrpc.tasks.list.useQuery.mockReturnValue(
      mockQuery({ tasks: mockTasks, pagination: { total: 2 } })
    );

    render(<TaskList />);

    await waitFor(() => {
      expect(screen.getByText("Task 1")).toBeInTheDocument();
      expect(screen.getByText("Task 2")).toBeInTheDocument();
    });
  });

  it("应该显示错误状态", () => {
    mockTrpc.tasks.list.useQuery.mockReturnValue({
      data: undefined,
      isLoading: false,
      error: new Error("加载失败"),
    });

    render(<TaskList />);

    expect(screen.getByText(/加载失败/)).toBeInTheDocument();
  });

  it("点击任务应该导航到详情页", async () => {
    const mockTasks = [
      { id: "1", title: "Clickable Task", status: "TODO" },
    ];

    mockTrpc.tasks.list.useQuery.mockReturnValue(
      mockQuery({ tasks: mockTasks })
    );

    const user = userEvent.setup();
    render(<TaskList />);

    const taskLink = screen.getByText("Clickable Task");
    await user.click(taskLink);

    expect(mockRouter.push).toHaveBeenCalledWith("/tasks/1");
  });
});
```

这些测试覆盖了组件的主要状态：加载、成功、错误，以及用户交互。注意我们使用了 `waitFor` 来等待异步更新，这是 Testing Library 的最佳实践。

### 测试创建任务表单

```typescript
// src/components/CreateTaskForm.test.tsx
import { describe, it, expect, vi } from "vitest";
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
```

这个测试验证了表单的提交流程、客户端验证和加载状态。使用 `userEvent` 模拟真实用户操作，比直接触发事件更可靠。

## E2E 测试

单元测试和集成测试关注代码逻辑，而 E2E 测试则模拟真实用户在浏览器中的操作。Playwright 是目前最流行的 E2E 测试框架之一。

### 配置 Playwright

```bash
npm install -D @playwright/test
npx playwright install  # 安装浏览器
```

```typescript
// playwright.config.ts
import { defineConfig, devices } from "@playwright/test";

export default defineConfig({
  testDir: "./e2e",

  // 失败时重试（CI 环境重试次数多些）
  retries: process.env.CI ? 2 : 0,
```

```typescript
// e2e/task-management.spec.ts
```

## 测试最佳实践

### 使用测试数据库

永远不要在开发或生产数据库上运行测试。应该使用独立的测试数据库，并在每个测试前清空数据。

```typescript
// 使用环境变量指定测试数据库
const testPrisma = new PrismaClient({
  datasources: {
    db: {
      url: process.env.DATABASE_URL_TEST,
    },
  },
});

// 在测试套件开始前连接
beforeAll(async () => {
  await testPrisma.$connect();
});

// 在测试套件结束后断开
afterAll(async () => {
  await testPrisma.$disconnect();
});

// 每个测试前清空数据
beforeEach(async () => {
  await testPrisma.task.deleteMany();
  await testPrisma.user.deleteMany();
});
```

### Factory 模式简化数据创建

当测试需要大量数据时，手动构造会很繁琐。Factory 模式能帮我们快速生成测试数据。

```typescript
// src/test/factories/task.factory.ts
export class TaskFactory {
  static async create(prisma: PrismaClient, overrides = {}) {
    return await prisma.task.create({
      data: {
        title: "Factory Task",
        description: "Generated by factory",
        status: "TODO",
        priority: "MEDIUM",
        ...overrides,
      },
    });
  }

  static async createMany(prisma: PrismaClient, count: number) {
    const tasks = Array.from({ length: count }, (_, i) => ({
      title: `Task ${i}`,
      description: `Description ${i}`,
      status: "TODO",
      priority: "MEDIUM",
    }));

    return await prisma.task.createMany({ data: tasks });
  }
}

// 使用
const task = await TaskFactory.create(prisma, {
  title: "Custom Task",
  status: "IN_PROGRESS",
});

await TaskFactory.createMany(prisma, 10); // 创建 10 个任务
```

### 测试覆盖率要有意义

追求 100% 的覆盖率没有意义，应该关注关键业务逻辑。对于我们的任务管理平台，以下是需要重点测试的：

**核心业务逻辑**：任务的创建、更新、删除流程

**权限控制**：用户只能操作自己的资源

**边界情况**：空列表、无效输入、并发冲突

**错误处理**：数据库错误、网络失败

不要为了覆盖率而测试琐碎的细节，比如 getter 方法或简单的字符串拼接。

### CI/CD 集成

测试的价值在于每次代码变更时自动运行，这是 CI/CD 的用武之地。

```yaml
# .github/workflows/test.yml
name: Test

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "18"

      - name: Install pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 8

      - name: Install dependencies
        run: pnpm install

      - name: Run tests
        run: pnpm test

      - name: Run E2E tests
        run: pnpm test:e2e

      - name: Upload coverage
        uses: codecov/codecov-action@v3
        if: always()
        with:
          files: ./coverage/lcov.info
```

这个 GitHub Actions 配置在每次 push 和 PR 时运行测试，并上传覆盖率报告。设置 `if: always()` 确保即使测试失败也会上传报告，方便分析。

## 总结与权衡

测试是有成本的。写测试需要时间，维护测试也需要时间。当需求变更时，测试代码也要跟着改。那为什么我们还要写测试？

因为测试带来的收益更大：

**重构信心**：有测试的情况下，你可以大胆重构代码，不用担心破坏现有功能

**文档作用**：测试用例本身就是最好的文档，展示了代码应该如何使用

**快速反馈**：测试能在几秒钟内发现问题，比手动测试快得多

**设计驱动**：难以测试的代码通常设计也有问题，写测试会促使你写出更易维护的代码

当然，测试不是越多越好。关键是要找到平衡点：

**单元测试**：运行快、定位准，适合测试业务逻辑。但无法测试组件集成

**集成测试**：测试多个组件协同工作，但运行慢、难定位

**E2E 测试**：最接近真实场景，但最脆弱、维护成本高

我的建议是：大部分测试应该是单元测试，少量集成测试验证关键流程，极少量 E2E 测试覆盖最核心的用户路径。对于我们的协作任务管理平台，可以这样分配：

- 单元测试：每个 Procedure 的业务逻辑、输入验证、权限控制
- 集成测试：创建任务、更新状态、分配用户的完整流程
- E2E 测试：登录、创建任务、退出登录这一条主流程

这样既能保证代码质量，又不会过度增加测试负担。
