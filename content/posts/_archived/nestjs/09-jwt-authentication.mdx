## 认证方案选型的纠结

博客平台要做用户系统了。打开技术选型文档，发现光是"用户登录"这一件事就有这么多方案：

- **Session + Cookie**：传统方案，服务端存会话数据
- **JWT（JSON Web Token）**：无状态方案，Token 里带着用户信息
- **OAuth2**：第三方登录，"用 GitHub 登录"
- **OIDC**：OAuth2 的上层协议，增加身份层

先搞清楚这些方案的本质区别。

**Session 方案**：用户登录后，服务端生成一个 session_id，存到 Redis 或内存里，把 session_id 写到浏览器的 Cookie 里。下次请求带着 Cookie，服务端查 session_id 就知道是谁。

问题在于"有状态"——服务端要存所有在线用户的会话数据。用户量大了，session 存储会成为瓶颈。而且要做分布式部署的话，所有实例都要共享 session 存储。

**JWT 方案**：用户登录后，服务端生成一个 Token（本质上是签过名的 JSON 字符串），发给客户端。客户端把 Token 存在 localStorage 或 Cookie 里，每次请求带上 Token，服务端验证签名就知道是谁。

好处是"无状态"——服务端不需要存任何会话数据，Token 自身就携带了用户信息。做分布式部署天然没问题。

博客平台是一个典型的 API 服务，前后端分离，未来可能还要做移动端。JWT 的无状态特性非常适合这个场景。所以选 JWT。

但 JWT 不是银弹。它的安全边界在哪里，后面会细说。

## JWT 到底是什么

先看一个真实的 JWT 长什么样。zhang_wei 登录博客平台后，服务端返回的 Token 是这样的：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOiJ1c3JfYTF4OWsybSIsImVtYWlsIjoiemhhbmdfd2VpQGV4YW1wbGUuY29tIiwibmFtZSI6InpoYW5nX3dlaSIsImlhdCI6MTczNTU2MTYwMCwiZXhwIjoxNzM1NTcwMjAwfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

这个字符串分三段，用 `.` 分隔：

**第一段 Header（头部）**，base64 解码后：

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

声明了签名算法是 HS256（HMAC-SHA256）。

**第二段 Payload（负载）**，base64 解码后：

```json
{
  "userId": "usr_a1x9k2m",
  "email": "zhang_wei@example.com",
  "name": "zhang_wei",
  "iat": 1735561600,
  "exp": 1735570200
}
```

这就是用户信息。`iat` 是签发时间，`exp` 是过期时间。注意 Payload 只是 base64 编码，不是加密——任何人都能解码看到里面的内容。所以不要在 Payload 里放密码等敏感信息。

**第三段 Signature（签名）**：

```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```

签名的作用是防篡改。服务端用密钥对 Header + Payload 做签名，客户端带着 Token 来请求时，服务端重新算一遍签名，如果一致说明 Token 没被改过。没有密钥的人没法伪造签名（假设密钥足够长且没有被泄露）。

所以 JWT 的安全依赖两件事：**HTTPS 传输**（防止中间人截获 Token）和**密钥保密**（防止伪造 Token）。

## 为什么需要 Refresh Token

JWT 有一个根本问题：无法主动撤销。

Token 签发后，在过期之前都是有效的。即使用户点了"退出登录"，Token 本身还是能用的——服务端没有存储来标记这个 Token 已失效。

如果把 Token 过期时间设得很长（比如 30 天），Token 泄露的风险就很大。设得很短（比如 5 分钟），用户动不动就要重新登录，体验极差。

解决方案是**双 Token 机制**：

- **Access Token**：短期有效（15 分钟），用于 API 认证
- **Refresh Token**：长期有效（7 天），只用于刷新 Access Token

Access Token 过期后，前端用 Refresh Token 换一对新的 Token，用户无感知。Refresh Token 过期了，才需要重新登录。

而且 Refresh Token 存在数据库里，用户退出登录时可以直接删除数据库里的记录，实现 Token 立即失效。这就是 Refresh Token 比 Access Token 更安全的原因——它是"有状态"的。

## 密码存储：为什么不存明文

在实现注册之前，先说密码存储。

数据库里存明文密码是最低级的安全事故。但很多人不知道，连 MD5、SHA256 这些"哈希"也不够——因为它们算得太快了。一台现代 GPU 一秒钟能算几十亿次 MD5，攻击者用彩虹表（预先算好的哈希对照表）几乎瞬间就能反查出明文。

**bcrypt** 是专门为密码存储设计的算法。它有两个关键特性：

1. **加盐（Salt）**。每个密码都自动生成一个随机盐值，同样的密码存出来的哈希不同。`P@ssw0rd` 存两次，得到两个完全不同的哈希值，彩虹表失效。
2. **慢（可调节）**。bcrypt 故意设计得很慢，默认 cost factor 是 10，意味着要跑 2^10 = 1024 轮。攻击者即使拿到了哈希值，暴力破解也非常慢。

```typescript
// 加密
import * as bcrypt from "bcrypt";

const hashedPassword = await bcrypt.hash("P@ssw0rd", 10);
// → $2b$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy

// 验证
const isMatch = await bcrypt.compare("P@ssw0rd", hashedPassword);
// → true
```

`bcrypt.hash` 的第二个参数 `10` 就是 cost factor。数字越大越安全但越慢。10 是一个合理的默认值，一般不需要调整。

## 用户注册

注册流程：验证参数 → 检查邮箱唯一性 → 密码哈希 → 存入数据库 → 生成 Token。

```typescript
// src/auth/auth.service.ts
import { Injectable } from "@nestjs/common";
import { JwtService } from "@nestjs/jwt";
import { UsersService } from "../users/users.service";
import { BusinessException } from "../common/exceptions/business.exception";
import { ErrorCodes } from "../common/constants/error-codes";
import * as bcrypt from "bcrypt";

@Injectable()
export class AuthService {
  constructor(
    private usersService: UsersService,
    private jwtService: JwtService
  ) {}

  async register(email: string, name: string, password: string) {
    // 检查邮箱唯一性
    const existing = await this.usersService.findByEmail(email);
    if (existing) {
      throw new BusinessException(
        ErrorCodes.EMAIL_ALREADY_EXISTS.code,
        `该邮箱 ${email} 已被注册`
      );
    }

    // 密码哈希
    const hashedPassword = await bcrypt.hash(password, 10);

    // 创建用户
    const user = await this.usersService.create({
      email,
      name,
      password: hashedPassword,
    });

    // 生成 Token 对
    const tokens = this.generateTokens(user.id, user.email, user.role);
    return { user, ...tokens };
  }

  private generateTokens(userId: string, email: string, role: string) {
    const payload = { userId, email, role };

    const accessToken = this.jwtService.sign(payload, {
      expiresIn: "15m",
    });

    const refreshToken = this.jwtService.sign(payload, {
      expiresIn: "7d",
    });

    return { accessToken, refreshToken };
  }
}
```

注意复用了上一篇的 `BusinessException` 和错误码。注册失败时，前端拿到的错误格式是统一的：

```json
{
  "code": 40004,
  "message": "该邮箱 zhang_wei@example.com 已被注册",
  "detail": {},
  "timestamp": "2026-03-30T10:30:00.000Z",
  "path": "/auth/register"
}
```

## 用户登录

登录流程：查用户 → 验证密码 → 生成 Token。注意一点：错误信息不要说"密码错误"，而要说"邮箱或密码错误"。否则攻击者可以通过枚举用户名来判断哪些邮箱已经注册了。

```typescript
async login(email: string, password: string) {
  const user = await this.usersService.findByEmail(email);

  // 用户不存在时也验证密码，防止通过响应时间差异判断用户是否存在
  if (!user) {
    await bcrypt.compare(password, '$2b$10$dummydummydummydummydummydummydummydummydummydummyd');
    throw new BusinessException(
      ErrorCodes.INVALID_CREDENTIALS.code,
      ErrorCodes.INVALID_CREDENTIALS.message,
    );
  }

  const isPasswordValid = await bcrypt.compare(password, user.password);
  if (!isPasswordValid) {
    throw new BusinessException(
      ErrorCodes.INVALID_CREDENTIALS.code,
      ErrorCodes.INVALID_CREDENTIALS.message,
    );
  }

  const tokens = this.generateTokens(user.id, user.email, user.role);
  return { user, ...tokens };
}
```

这里有一个安全细节：当用户不存在时，我们仍然调用了 `bcrypt.compare` 来验证密码（虽然是对一个假的哈希值做验证）。这是因为 `bcrypt.compare` 的执行时间和密码是否正确有关——如果用户不存在直接返回错误，响应时间会比密码错误时短，攻击者可以通过时间差异判断用户是否存在。

## Passport + JWT 策略

Token 生成了，但 API 怎么验证每个请求带的 Token 是否有效？

NestJS 的做法是通过 Passport 的策略模式。Passport 是一个认证中间件库，它把"认证方式"抽象成"策略"。JWT 认证就是其中一个策略。

```typescript
// src/auth/strategies/jwt.strategy.ts
import { Injectable } from "@nestjs/common";
import { PassportStrategy } from "@nestjs/passport";
import { ExtractJwt, Strategy } from "passport-jwt";
import { ConfigService } from "@nestjs/config";

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor(configService: ConfigService) {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: configService.get<string>("JWT_SECRET"),
    });
  }

  async validate(payload: { userId: string; email: string; role: string }) {
    return { userId: payload.userId, email: payload.email, role: payload.role };
  }
}
```

`ExtractJwt.fromAuthHeaderAsBearerToken()` 告诉 Passport 从 `Authorization: Bearer <token>` 头里提取 Token。

`validate` 方法在 Token 验证通过后被调用，返回值会挂到 `request.user` 上。这样 Controller 里就能通过 `request.user` 拿到当前用户信息。

Guard 就是把 Passport 策略包了一层：

```typescript
// src/auth/guards/jwt-auth.guard.ts
import { Injectable } from "@nestjs/common";
import { AuthGuard } from "@nestjs/passport";

@Injectable()
export class JwtAuthGuard extends AuthGuard("jwt") {}
```

用起来就是一行装饰器：

```typescript
@UseGuards(JwtAuthGuard)
@Get('me')
async getCurrentUser() { /* ... */ }
```

没有 Token 或 Token 过期的请求，Guard 直接返回 401，不会进入 Controller。

## 自定义装饰器 @CurrentUser()

每个需要认证的接口都要从 `request.user` 里取用户信息，写多了很烦。用一个自定义参数装饰器简化：

```typescript
// src/auth/decorators/current-user.decorator.ts
import { createParamDecorator, ExecutionContext } from "@nestjs/common";

export const CurrentUser = createParamDecorator(
  (data: string | undefined, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    const user = request.user;
    return data ? user?.[data] : user;
  }
);
```

使用时可以取整个 user 对象，也可以只取某个字段：

```typescript
// 取整个 user
@Get('me')
getCurrentUser(@CurrentUser() user: { userId: string; email: string; role: string }) {
  return user;
}

// 只取 userId
@Post('posts')
createPost(
  @CurrentUser('userId') userId: string,
  @Body() dto: CreatePostDto,
) {
  return this.postsService.create(userId, dto);
}
```

zhang_wei 创建文章时，`userId` 自动从 Token 里提取，不需要前端传。

## 全局认证 + @Public() 白名单

大多数接口都需要认证，只有少数（注册、登录、文章列表）是公开的。与其在每个 Controller 上加 `@UseGuards(JwtAuthGuard)`，不如全局启用认证，用 `@Public()` 标记白名单。

先定义 `@Public()` 装饰器：

```typescript
// src/auth/decorators/public.decorator.ts
import { SetMetadata } from "@nestjs/common";

export const IS_PUBLIC_KEY = "isPublic";
export const Public = () => SetMetadata(IS_PUBLIC_KEY, true);
```

然后写一个全局 Guard，检查当前路由有没有 `@Public()` 标记：

```typescript
// src/auth/guards/guard.decorator.ts
import { Injectable, ExecutionContext } from "@nestjs/common";
import { Reflector } from "@nestjs/core";
import { AuthGuard } from "@nestjs/passport";
import { IS_PUBLIC_KEY } from "../decorators/public.decorator";

@Injectable()
export class JwtAuthGuard extends AuthGuard("jwt") {
  constructor(private reflector: Reflector) {
    super();
  }

  canActivate(context: ExecutionContext) {
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);

    if (isPublic) {
      return true;
    }

    return super.canActivate(context);
  }
}
```

`Reflector` 从路由的元数据中读取 `@Public()` 标记。如果标记了，跳过认证。

在 `app.module.ts` 里注册为全局 Guard：

```typescript
import { Module } from "@nestjs/common";
import { APP_GUARD } from "@nestjs/core";
import { JwtAuthGuard } from "./auth/guards/guard.decorator";

@Module({
  providers: [
    {
      provide: APP_GUARD,
      useClass: JwtAuthGuard,
    },
  ],
})
export class AppModule {}
```

这样所有接口默认需要认证，公开接口加上 `@Public()` 就行：

```typescript
@Public()
@Post('register')
async register(@Body() dto: RegisterDto) { /* ... */ }

@Public()
@Post('login')
async login(@Body() dto: LoginDto) { /* ... */ }

@Public()
@Get('posts')
async findAll(@Query() query: QueryPostsDto) { /* ... */ }

// 以下接口都需要认证，不用加 @UseGuards
@Get('me')
async getCurrentUser(@CurrentUser() user) { /* ... */ }

@Post('posts')
async createPost(@CurrentUser('userId') userId: string, @Body() dto: CreatePostDto) { /* ... */ }
```

## Token 刷新

Access Token 过期后，前端用 Refresh Token 换一对新的。刷新接口本身需要认证，但用的是 Refresh Token 而不是 Access Token，所以需要单独的 Guard：

```typescript
// src/auth/strategies/jwt-refresh.strategy.ts
import { Injectable } from "@nestjs/common";
import { PassportStrategy } from "@nestjs/passport";
import { ExtractJwt, Strategy } from "passport-jwt";
import { ConfigService } from "@nestjs/config";

@Injectable()
export class JwtRefreshStrategy extends PassportStrategy(
  Strategy,
  "jwt-refresh"
) {
  constructor(configService: ConfigService) {
    super({
      jwtFromRequest: ExtractJwt.fromBodyField("refreshToken"),
      ignoreExpiration: false,
      secretOrKey: configService.get<string>("JWT_REFRESH_SECRET"),
    });
  }

  async validate(payload: any) {
    return payload;
  }
}
```

注意这里用了不同的密钥 `JWT_REFRESH_SECRET`，而且从请求体而不是 Header 里提取 Token。Refresh Token 和 Access Token 用不同的密钥签名，即使一个泄露了另一个还是安全的。

```typescript
// src/auth/guards/jwt-refresh.guard.ts
import { Injectable } from "@nestjs/common";
import { AuthGuard } from "@nestjs/passport";

@Injectable()
export class JwtRefreshGuard extends AuthGuard("jwt-refresh") {}
```

刷新 Token 的逻辑：

```typescript
// src/auth/auth.service.ts
async refreshTokens(refreshToken: string) {
  try {
    const payload = this.jwtService.verify(refreshToken, {
      secret: this.configService.get<string>('JWT_REFRESH_SECRET'),
    });

    // 检查 Refresh Token 是否在数据库中（是否被撤销）
    const storedToken = await this.prisma.refreshToken.findUnique({
      where: { token: refreshToken },
    });

    if (!storedToken) {
      throw new BusinessException(
        ErrorCodes.TOKEN_EXPIRED.code,
        ErrorCodes.TOKEN_EXPIRED.message,
      );
    }

    // 生成新的 Token 对
    const tokens = this.generateTokens(payload.userId, payload.email, payload.role);

    // 删除旧的 Refresh Token，存入新的（Rotation）
    await this.prisma.refreshToken.delete({ where: { token: refreshToken } });
    await this.prisma.refreshToken.create({
      data: {
        token: tokens.refreshToken,
        userId: payload.userId,
        expiresAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000),
      },
    });

    return tokens;
  } catch {
    throw new BusinessException(
      ErrorCodes.TOKEN_EXPIRED.code,
      'Refresh Token 无效或已过期',
    );
  }
}
```

每次刷新时执行 **Token Rotation**：旧的 Refresh Token 删除，生成一个新的。这样即使 Refresh Token 泄露，被攻击者用了一次之后就会失效（因为已经被删除了）。

## 退出登录

退出登录就是把数据库里的 Refresh Token 删掉：

```typescript
async logout(userId: string, refreshToken: string) {
  await this.prisma.refreshToken.deleteMany({
    where: { userId, token: refreshToken },
  });
}
```

Refresh Token 删除后，即使 Access Token 还没过期，Refresh Token 也没法用了。Access Token 最坏的情况是再活 15 分钟就自动失效。

如果需要"立即让所有 Token 失效"（比如用户改了密码），可以删掉该用户的所有 Refresh Token：

```typescript
async revokeAllTokens(userId: string) {
  await this.prisma.refreshToken.deleteMany({
    where: { userId },
  });
}
```

Access Token 没法主动撤销（因为它是无状态的），但 15 分钟后自然过期。这是一个可接受的权衡——为了无状态的好处，接受了短暂的不可撤销窗口。

## 环境变量和密钥管理

JWT 的密钥必须是长且随机的字符串。不要硬编码在代码里，放在 `.env` 文件中：

```env
JWT_SECRET=a-random-string-at-least-32-characters-long-for-hs256
JWT_REFRESH_SECRET=another-random-string-at-least-32-characters-long
```

通过 `ConfigModule` 注入：

```typescript
// src/auth/auth.module.ts
import { Module } from "@nestjs/common";
import { JwtModule } from "@nestjs/jwt";
import { PassportModule } from "@nestjs/passport";
import { ConfigModule, ConfigService } from "@nestjs/config";

@Module({
  imports: [
    PassportModule,
    JwtModule.registerAsync({
      imports: [ConfigModule],
      inject: [ConfigService],
      useFactory: (configService: ConfigService) => ({
        secret: configService.get<string>("JWT_SECRET"),
        signOptions: {
          expiresIn: configService.get<string>("JWT_EXPIRES_IN", "15m"),
        },
      }),
    }),
  ],
  providers: [AuthService, JwtStrategy, JwtRefreshStrategy],
  controllers: [AuthController],
  exports: [AuthService],
})
export class AuthModule {}
```

`registerAsync` 让我们可以从 `ConfigService` 读配置，而不是硬编码。这比直接写 `JwtModule.register({ secret: 'xxx' })` 好得多。

```
→ 200
```

```
    "author": { "name": "zhang_wei" }
  }
}
```

**4. Access Token 过期后刷新**

```
POST /auth/refresh
{ "refreshToken": "eyJhbG..." }

→ 200
{
  "code": 200,
  "data": {
    "accessToken": "new_eyJhbG...",
    "refreshToken": "new_eyJhbG..."
  }
}
```

**5. 退出登录**

```
POST /auth/logout
Authorization: Bearer eyJhbG...
{ "refreshToken": "eyJhbG..." }

→ 200 { "code": 200, "data": null }
```

每一步的错误响应都遵循上一篇建立的统一格式。

## JWT 的安全边界

最后聊聊 JWT 的安全。很多人对 JWT 有误解，以为"用了 JWT 就是安全的"。不是的。

**JWT 做到的**：

- 防篡改（没有密钥无法伪造 Token）
- 无状态（服务端不需要存会话）
- 分布式友好（任何节点都能验证 Token）

**JWT 做不到的**：

- 防窃取（如果 Token 被截获，攻击者可以冒用）
- 防重放（截获的 Token 可以反复使用，直到过期）
- 主动撤销（Access Token 签发后无法立即失效）

所以实践中要注意：

1. **必须用 HTTPS**。HTTP 传输下 Token 就是明文，中间人可以直接截获。
2. **Access Token 过期时间要短**。15 分钟是常见选择，缩小泄露后的利用窗口。
3. **Refresh Token 要存储在 HttpOnly Cookie 里**（而不是 localStorage）。HttpOnly Cookie 无法被 JavaScript 读取，XSS 攻击偷不走。
4. **敏感操作需要二次验证**。比如改密码、删除账号，应该要求重新输入密码，而不是只看 Token。
5. **密钥要足够长且定期轮换**。HS256 要求密钥至少 32 个字符。

## 小结

这一篇实现了完整的用户认证流程：

1. **注册**：邮箱验证 + bcrypt 密码哈希 + 生成 Token 对
2. **登录**：查用户 → 验证密码 → 生成 Token 对，错误信息模糊化防止枚举
3. **JWT 策略**：Passport + passport-jwt，从 Authorization Header 提取 Token 并验证
4. **双 Token 机制**：Access Token 短期有效，Refresh Token 长期有效 + Token Rotation
5. **全局认证 + @Public()**：默认所有接口需要认证，白名单用装饰器标记
6. **@CurrentUser() 装饰器**：简化 Controller 中获取当前用户信息的代码
7. **退出登录**：删除数据库中的 Refresh Token，Access Token 自然过期
