该接数据库了。

## 选型：为什么是 TypeORM 而不是 Prisma

打开 NestJS 的文档，你会发现它对 TypeORM 有一等公民级别的支持——`@nestjs/typeorm` 是官方维护的包。这不是偶然。TypeORM 和 NestJS 的设计哲学高度一致：都用装饰器声明结构，都依赖 IoC 容器管理生命周期。

但这不是唯一的理由。先看看主流选择：

**TypeORM**（我们选的）：装饰器风格定义 Entity，和 NestJS 的 `@Module`、`@Injectable` 一脉相承。关系映射用 `@OneToMany`、`@ManyToOne` 这些装饰器，写起来像在描述业务关系而不是写 SQL。NestJS 的 `TypeOrmModule.forFeature([Article])` 可以直接把 Repository 注入到 Service 里，不需要自己封装。

**Prisma**：用 `.prisma` 文件定义 Schema，生成类型安全的 Client。查询 API 很优雅（`prisma.article.findMany({ where: { published: true } })`），但它的 Schema 文件和 TypeScript 代码是割裂的——你要在两个文件之间跳来跳去。Prisma 更适合从零开始的项目，或者你不在意多一层抽象的情况。

**Sequelize**：老牌 ORM，基于 Promise 的 API，但 TypeScript 支持较弱，装饰器支持也是后加的。生态在萎缩，新项目不太推荐。

选型的关键不是"哪个更好"，而是"哪个更适合你的项目"。如果你已经在用 NestJS 的装饰器体系，TypeORM 的学习成本最低，集成也最自然。如果你更看重类型安全和查询 API 的优雅度，Prisma 是很好的选择——但那需要另外一套工具链。

本系列选 TypeORM，还有一个实际原因：当你在 NestJS 里写 `@Entity()`、`@Column()`、`@OneToMany()` 时，你会发现它们的思维模型和 `@Controller()`、`@Get()`、`@Injectable()` 是一致的——都在用装饰器声明"这个类是什么、做什么"。认知负担更小。

## 第一个 Entity：User

Entity 是 TypeORM 对数据库表的 TypeScript 映射。你用装饰器描述字段和类型，TypeORM 负责生成对应的 DDL 和查询。

先从用户表开始。我们的博客平台需要一个基础的用户模型：

```typescript
// src/users/entities/user.entity.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
} from "typeorm";

@Entity("users")
export class User {
  @PrimaryGeneratedColumn("uuid")
  id: string;

  @Column({ type: "varchar", length: 50, unique: true })
  username: string;

  @Column({ type: "varchar", length: 100, unique: true })
  email: string;

  @Column({ type: "varchar", length: 255 })
  password: string;

  @Column({ type: "varchar", length: 50, nullable: true })
  nickname: string;

  @Column({ type: "text", nullable: true })
  bio: string;

  @Column({ type: "varchar", length: 500, nullable: true })
  avatar: string;

  @Column({
    type: "enum",
    enum: ["reader", "author", "admin"],
    default: "reader",
  })
  role: "reader" | "author" | "admin";

  @CreateDateColumn({ type: "timestamptz" })
  createdAt: Date;

  @UpdateDateColumn({ type: "timestamptz" })
  updatedAt: Date;
}
```

逐个说几个关键装饰器：

`@PrimaryGeneratedColumn('uuid')` 生成 UUID 主键。为什么不用自增 ID？UUID 天然分布式友好，不同服务之间不会冲突。在博客平台这种规模，自增 ID 也没问题，但用 UUID 是个好习惯。

`@Column` 的 `unique: true` 会生成唯一约束。`username` 和 `email` 加了唯一约束，数据库层面保证不会出现重复。

`@Column` 的 `nullable: true` 表示允许为空。`nickname`、`bio`、`avatar` 不是必填项——用户注册时可以不填，后续再补全。

`@CreateDateColumn` 和 `@UpdateDateColumn` 自动管理时间戳。插入时自动填充 `createdAt`，每次更新时自动刷新 `updatedAt`。你不需要在代码里手动设置。

注意 `@Entity('users')` 括号里的参数——它指定了数据库表名。如果不传，TypeORM 会用类名做表名（`User` → `user`）。显式指定表名是个好习惯，避免将来类名和表名不一致时出问题。

## Article Entity：文章表

```typescript
// src/articles/entities/article.entity.ts
import {

  JoinTable,

import { User } from "../../users/entities/user.entity";

@Entity("articles")
export class Article {
  @PrimaryGeneratedColumn("uuid")
  id: string;

  @Column({ type: "varchar", length: 200 })

  @Column({ type: "text" })
  content: string;

  @Column({ type: "varchar", length: 500, nullable: true })
  excerpt: string;

  @Column({ type: "varchar", length: 500, nullable: true })
  coverImage: string;

  @Column({ type: "varchar", length: 200, unique: true, nullable: true })
  slug: string;

  @Column({ type: "boolean", default: false })
  published: boolean;

  @Column({ type: "int", default: 0 })

  @Column({ type: "timestamptz", nullable: true })

```

现在 Article 还没有关联关系。我们先看一下关联设计，再回头补充。

## 关系映射：四张表怎么连

博客平台有四张核心表：User、Article、Comment、Tag。它们的关系如下：

```
User ──1:N──> Article    一个用户可以写多篇文章
User ──1:N──> Comment    一个用户可以写多条评论
Article ──1:N──> Comment  一篇文章可以有多条评论
Article ──N:M──> Tag      一篇文章可以有多个标签
```

### 为什么是 @ManyToOne，不是 @OneToOne

一篇文章属于一个作者——这是 `1:N` 关系。但在 TypeORM 里，你可以在"多"的那一侧用 `@ManyToOne`，也可以在"1"的那一侧用 `@OneToMany`。两者描述的是同一个关系，只是视角不同。

我们在 Article 这一侧声明 `@ManyToOne`：

```typescript
// article.entity.ts 里补充
@ManyToOne(() => User, { onDelete: 'CASCADE' })
@JoinColumn({ name: 'authorId' })
author: User;
```

`onDelete: 'CASCADE'` 表示删除用户时，该用户的所有文章也一起删除。这是一个需要慎重考虑的设计决策。有些场景下你更希望"删除用户后文章保留"（把作者标记为"已注销"），那就不用 CASCADE。

`@JoinColumn` 指定了外键列名。不传的话 TypeORM 会自动生成（类似 `authorId`），但显式指定更可控。

如果你需要在 User 那一侧也能访问到文章列表，可以加 `@OneToMany`：

```typescript
// user.entity.ts 里补充
import { Article } from '../../articles/entities/article.entity';

@OneToMany(() => Article, (article) => article.author)
articles: Article[];
```

TypeORM 的双向关系需要两侧都声明。`@OneToMany` 的第二个参数是一个函数，告诉 TypeORM "Article 的哪个字段指向 User"——就是 `article.author`。

### Tag：多对多关系

文章和标签是多对多：一篇文章可以有多个标签，一个标签可以属于多篇文章。数据库层面需要一个中间表（`article_tags`）来关联。

```typescript
// tag.entity.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  ManyToMany,
} from "typeorm";
import { Article } from "../../articles/entities/article.entity";

@Entity("tags")
export class Tag {
  @PrimaryGeneratedColumn("uuid")
  id: string;

  @Column({ type: "varchar", length: 50, unique: true })
  name: string;

  @Column({ type: "varchar", length: 50, unique: true })
  slug: string;

  @ManyToMany(() => Article, (article) => article.tags)
  articles: Article[];

  @CreateDateColumn({ type: "timestamptz" })
  createdAt: Date;
}
```

在 Article 那一侧：

```typescript
// article.entity.ts 里补充
import { Tag } from '../../tags/entities/tag.entity';

@ManyToMany(() => Tag, (tag) => tag.articles)
@JoinTable({
  name: 'article_tags',
  joinColumn: { name: 'articleId', referencedColumnName: 'id' },
  inverseJoinColumn: { name: 'tagId', referencedColumnName: 'id' },
})
tags: Tag[];
```

`@JoinTable` 放在关系的"拥有方"（owning side）。对于多对多关系，TypeORM 要求你在其中一侧声明 `@JoinTable`，它会自动创建中间表。`name` 指定中间表名，`joinColumn` 和 `inverseJoinColumn` 指定外键列名。

### Comment：带自引用的评论

评论需要支持回复——评论下面可以嵌套评论。这通过自引用关系实现：

```typescript
// comments/entities/comment.entity.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  ManyToOne,
  JoinColumn,
} from "typeorm";
import { User } from "../../users/entities/user.entity";
import { Article } from "../../articles/entities/article.entity";

@Entity("comments")
export class Comment {
  @PrimaryGeneratedColumn("uuid")
  id: string;

  @Column({ type: "text" })
  content: string;

  @ManyToOne(() => User, { onDelete: "CASCADE" })
  @JoinColumn({ name: "authorId" })
  author: User;

  @ManyToOne(() => Article, { onDelete: "CASCADE" })
  @JoinColumn({ name: "articleId" })
  article: Article;

  // 自引用：父评论
  @ManyToOne(() => Comment, { nullable: true, onDelete: "CASCADE" })
  @JoinColumn({ name: "parentId" })
  parent: Comment | null;

  @CreateDateColumn({ type: "timestamptz" })
  createdAt: Date;
}
```

`parent` 指向同表的另一条评论。`nullable: true` 表示顶级评论没有 parent。删除父评论时 `CASCADE` 会级联删除所有子评论。

## TypeORM 在 NestJS 中的配置

Entity 定义好了，现在要把 TypeORM 接入 NestJS。这通过 `TypeOrmModule.forRootAsync()` 实现：

```typescript
// app.module.ts
import { Module } from "@nestjs/common";
import { ConfigModule, ConfigService } from "@nestjs/config";
import { TypeOrmModule } from "@nestjs/typeorm";
import { User } from "./users/entities/user.entity";
import { Article } from "./articles/entities/article.entity";
import { Comment } from "./comments/entities/comment.entity";
import { Tag } from "./tags/entities/tag.entity";

@Module({
  imports: [
    ConfigModule.forRoot({ isGlobal: true }),
    TypeOrmModule.forRootAsync({
      imports: [ConfigModule],
      inject: [ConfigService],
      useFactory: (configService: ConfigService) => ({
        type: "postgres",
        host: configService.get("DB_HOST", "localhost"),
        port: configService.get<number>("DB_PORT", 5432),
        username: configService.get("DB_USERNAME", "blog_dev"),
        password: configService.get("DB_PASSWORD", "dev_secret_2026"),
        database: configService.get("DB_DATABASE", "blog_platform"),
        entities: [User, Article, Comment, Tag],
        synchronize: false, // 永远不要在生产环境设为 true
      }),
    }),
    // 其他模块...
  ],
})
export class AppModule {}
```

`.env` 文件里放实际的配置：

```env
DB_HOST=localhost
DB_PORT=5432
DB_USERNAME=blog_dev
DB_PASSWORD=dev_secret_2026
DB_DATABASE=blog_platform
```

关于 `synchronize: false`：这个选项如果设为 `true`，TypeORM 会在每次启动时自动同步 Entity 和数据库表结构——开发阶段很方便，但生产环境绝对不能用，因为它可能 DROP 列导致数据丢失。正确的方式是用 Migration 管理数据库变更。

## Migration：数据库版本控制

Migration 是一种"数据库的 git"。每次修改 Entity 后，你生成一个 Migration 文件，里面是对应的 SQL（CREATE TABLE、ALTER TABLE 之类），然后在数据库上执行。这个文件纳入版本控制，团队成员 pull 下来后执行同样的 Migration，大家的数据库结构就保持一致。

先安装 TypeORM 的 CLI：

```bash
pnpm add -D typeorm-ts-node-commonjs
```

在 `package.json` 里配置：

```json
{
  "typeorm": {
    "dataSource": "src/data-source.ts"
  }
}
```

创建数据源文件：

```typescript
// src/data-source.ts
import { DataSource } from "typeorm";
import { User } from "./users/entities/user.entity";
import { Article } from "./articles/entities/article.entity";
import { Comment } from "./comments/entities/comment.entity";
import { Tag } from "./tags/entities/tag.entity";

export const AppDataSource = new DataSource({
  type: "postgres",
  host: process.env.DB_HOST || "localhost",
  port: parseInt(process.env.DB_PORT || "5432"),
  username: process.env.DB_USERNAME || "blog_dev",
  password: process.env.DB_PASSWORD || "dev_secret_2026",
  database: process.env.DB_DATABASE || "blog_platform",
  entities: [User, Article, Comment, Tag],
  migrations: ["src/migrations/*.ts"],
  synchronize: false,
});
```

生成第一个 Migration：

```bash
npx typeorm migration:generate -n init
```

TypeORM 会对比 Entity 定义和当前数据库状态，生成一个 Migration 文件：

```typescript
// src/migrations/1700000000000-init.ts
export class init1700000000000 implements MigrationInterface {
  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(`
      CREATE TABLE "users" (
        "id" uuid PRIMARY KEY DEFAULT uuid_generate_v4(),
        "username" varchar(50) NOT NULL,
        "email" varchar(100) NOT NULL,
        "password" varchar(255) NOT NULL,
        "nickname" varchar(50),
        "bio" text,
        "avatar" varchar(500),
        "role" enum('reader','author','admin') NOT NULL DEFAULT 'reader',
        "createdAt" timestamptz NOT NULL DEFAULT now(),
        "updatedAt" timestamptz NOT NULL DEFAULT now(),
        CONSTRAINT "uq_users_username" UNIQUE ("username"),
        CONSTRAINT "uq_users_email" UNIQUE ("email")
      )
    `);
    // ... 其他表的 CREATE TABLE
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(`DROP TABLE "users"`);
    // ... 其他表的 DROP TABLE
  }
}
```

执行 Migration：

```bash
npx typeorm migration:run
```

以后每次改了 Entity，流程是：

```bash
# 1. 修改 Entity 定义
# 2. 生成新的 Migration
npx typeorm migration:generate -n add-article-cover
# 3. 执行 Migration
npx typeorm migration:run
```

`down` 方法是回滚用的——`npx typeorm migration:revert` 会执行最近一次 Migration 的 `down`，把数据库回退到上一个版本。这就是 Migration 和 `synchronize: true` 的本质区别：Migration 是可控的、可回滚的、纳入版本控制的。

## Seeding：填充初始数据

空数据库不利于开发。你需要一些测试数据——几个用户、几篇文章、一些标签——让你在开发时能看到真实效果。

TypeORM 没有内置的 seeding 功能，但写一个很简单：

```typescript
// src/seed.ts
import { AppDataSource } from "./data-source";
import { User } from "./users/entities/user.entity";
import { Article } from "./articles/entities/article.entity";
import { Tag } from "./tags/entities/tag.entity";

async function seed() {
  await AppDataSource.initialize();

  const userRepo = AppDataSource.getRepository(User);
  const articleRepo = AppDataSource.getRepository(Article);
  const tagRepo = AppDataSource.getRepository(Tag);

  // 清空现有数据（仅开发环境）
  await AppDataSource.synchronize(true);

  // 创建用户
  const zhangWei = await userRepo.save({
    username: "zhang_wei",
    email: "zhangwei@example.com",
    password: "hashed_password_placeholder",
    nickname: "张威",
    bio: "前端工程师，最近在学 NestJS",
    role: "author",
  });

  // 创建标签
  const nodejsTag = await tagRepo.save({ name: "Node.js", slug: "nodejs" });
  const nestjsTag = await tagRepo.save({ name: "NestJS", slug: "nestjs" });
  const archTag = await tagRepo.save({
    name: "后端架构",
    slug: "backend-arch",
  });

  // 创建文章
  await articleRepo.save({
    title: "从 Express 迁移到 NestJS 的踩坑记录",
    content: "（文章正文省略）...",
    excerpt: "记录了从 Express 迁移到 NestJS 过程中遇到的坑和解决方案",
    slug: "express-to-nestjs-pitfalls",
    published: true,
    views: 3842,
    publishedAt: new Date("2025-12-15T08:30:00Z"),
    author: zhangWei,
    tags: [nodejsTag, nestjsTag, archTag],
  });

  await articleRepo.save({
    title: "NestJS 依赖注入到底怎么工作",
    content: "（文章正文省略）...",
    excerpt: "从源码角度理解 NestJS 的 IoC 容器",
    slug: "nestjs-di-deep-dive",
    published: true,
    views: 2156,
    publishedAt: new Date("2026-01-08T10:00:00Z"),
    author: zhangWei,
    tags: [nestjsTag, archTag],
  });

  console.log("Seed 完成");
  await AppDataSource.destroy();
}

seed().catch((e) => {
  console.error(e);
  process.exit(1);
});
```

在 `package.json` 里加个脚本：

```json
{
  "scripts": {
    "seed": "ts-node src/seed.ts"
  }
}
```

```bash
pnpm seed
```

注意 `AppDataSource.synchronize(true)` 只应该在 seeding 脚本里用，它会自动同步表结构（相当于先 DROP 再 CREATE），确保 seeding 时表结构是最新的。正式代码里绝对不要用。

## 在模块里注册 Entity

配置好全局的 TypeORM 后，还需要在每个功能模块里声明它用到哪些 Entity，这样 TypeORM 才会把对应的 Repository 注入到 Service：

```typescript
// articles.module.ts
import { Module } from "@nestjs/common";
import { TypeOrmModule } from "@nestjs/typeorm";
import { Article } from "./entities/article.entity";
import { ArticlesController } from "./articles.controller";
import { ArticlesService } from "./articles.service";

@Module({
  imports: [TypeOrmModule.forFeature([Article])],
  controllers: [ArticlesController],
  providers: [ArticlesService],
  exports: [ArticlesService],
})
export class ArticlesModule {}
```

`TypeOrmModule.forFeature([Article])` 做了两件事：一是告诉 TypeORM 这个模块需要 Article 的 Repository，二是让你在 Service 里可以通过 `@InjectRepository(Article)` 注入它。

到这里，数据库接好了。四张表、关联关系、Migration、初始数据都有了。下一篇，我们基于这些 Entity 实现完整的 CRUD 接口——创建草稿、发布文章、分页查询、软删除。

## 当前数据模型总览

```
users
├── id (uuid, PK)
├── username (varchar, unique)
├── email (varchar, unique)
├── password (varchar)
├── nickname (varchar, nullable)
├── bio (text, nullable)
├── avatar (varchar, nullable)
├── role (enum: reader/author/admin)
├── createdAt (timestamptz)
└── updatedAt (timestamptz)

articles
├── id (uuid, PK)
├── title (varchar)
├── content (text)
├── excerpt (varchar, nullable)
├── coverImage (varchar, nullable)
├── slug (varchar, unique, nullable)
├── published (boolean)
├── views (int)
├── authorId (uuid, FK → users.id)
├── publishedAt (timestamptz, nullable)
├── createdAt (timestamptz)
└── updatedAt (timestamptz)

comments
├── id (uuid, PK)
├── content (text)
├── authorId (uuid, FK → users.id)

├── id (uuid, PK)
├── name (varchar, unique)
├── slug (varchar, unique)

├── articleId (uuid, FK → articles.id)
└── tagId (uuid, FK → tags.id)
```

四张表、四种关系：`1:N`（用户-文章、用户-评论、文章-评论）、`N:M`（文章-标签）、自引用（评论-评论）。这是博客平台最核心的数据骨架。
