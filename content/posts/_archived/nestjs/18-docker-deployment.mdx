## "在我机器上能跑"的代价

博客平台在本地开发环境运行得好好的。你 `pnpm run start:dev`，应用跑在 3000 端口，PostgreSQL 在 5432，Redis 在 6379，一切正常。

然后你要部署到测试服务器。运维给你开了一台 Ubuntu 机器，你 SSH 上去，发现：

- Node 版本是 16，你的项目要求 20
- 没装 pnpm
- PostgreSQL 版本不匹配，某个查询语法报错
- Redis 的 `appendonly` 配置和本地不一样，导致内存行为不同
- 环境变量里的 `DATABASE_URL` 少了一个参数

你在服务器上折腾了半天，终于跑起来了。但一周后你换了一台服务器部署预发布环境，同样的步骤又要走一遍，而且这次又遇到了新的问题。

这不是什么罕见场景。每个后端开发者都经历过"环境不一致"的痛苦。Docker 解决的核心问题就是：**把你运行代码所需的整个环境打包在一起**，包括操作系统、运行时、依赖、配置，确保在任何机器上都一样。

## .dockerignore：不要把垃圾打进镜像

在写 Dockerfile 之前，先创建 `.dockerignore` 文件。它的作用类似 `.gitignore`，告诉 Docker 构建时哪些文件不需要复制到镜像里。

```
# .dockerignore
node_modules
dist
.git
.env
.env.local
.env.*.local
*.md
.vscode
.idea
coverage
test
*.log
npm-debug.log*
pnpm-debug.log*
.DS_Store
docker-compose*.yml
Dockerfile
nginx
```

为什么这很重要：如果不排除 `node_modules`，Docker 会把你本地的所有依赖（包括开发依赖、平台相关的二进制文件）复制到镜像里，导致镜像体积膨胀，而且可能和目标平台不兼容。如果不排除 `.env`，你本地的数据库密码和 JWT 密钥会被打包进镜像，这是严重的安全问题。

`node_modules`、`dist`、`.git` 这三个目录加起来通常有几百 MB，排除之后 Docker 的构建上下文（build context）会小很多，传输到 Docker daemon 的速度也快很多。

## 多阶段 Dockerfile：构建和运行分开

一个基本原则：构建阶段和运行阶段应该分开。构建需要完整的 Node 环境（编译 TypeScript），运行只需要编译后的 JS 文件和最小化的依赖。

```dockerfile
# Dockerfile

# ========== 构建阶段 ==========
FROM node:20-alpine AS builder

WORKDIR /app

# 安装 pnpm
RUN corepack enable && corepack prepare pnpm@9 --activate

# 先复制依赖文件，利用 Docker 缓存层
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile

# 再复制源代码和 Prisma schema
COPY prisma ./prisma/
COPY tsconfig.json tsconfig.build.json nest-cli.json ./
COPY src ./src/

# 生成 Prisma Client + 编译 TypeScript
RUN pnpm run prisma:generate
RUN pnpm run build

# ========== 运行阶段 ==========
FROM node:20-alpine AS production

# 创建非 root 用户
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup

WORKDIR /app

# 安装 pnpm（只用于运行 prisma migrate deploy）
RUN corepack enable && corepack prepare pnpm@9 --activate

# 复制依赖文件并只安装生产依赖
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile --prod

# 从构建阶段复制产物
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules/.prisma ./node_modules/.prisma
COPY --from=builder /app/prisma ./prisma

# 设置文件所有权
RUN chown -R appuser:appgroup /app

# 切换到非 root 用户
USER appuser

# 环境变量
ENV NODE_ENV=production

EXPOSE 3000

# 入口：先跑数据库迁移，再启动应用
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/main.js"]
```

逐段解释一下这个 Dockerfile 的设计决策。

**为什么用 Alpine 而不是标准 Node 镜像**：`node:20-alpine` 约 130MB，`node:20` 约 1GB。Alpine 基于 musl libc 而不是 glibc，体积小得多。NestJS 项目是纯 JavaScript/TypeScript，不依赖原生模块，Alpine 完全够用。

**为什么要分两个阶段**：构建阶段安装了所有依赖（包括 devDependencies 里的 TypeScript、ESLint 等），编译完成后只需要运行时依赖。分阶段让最终镜像不包含任何编译工具，镜像更小、攻击面更小。

**为什么要创建非 root 用户**：容器默认以 root 运行。如果应用有漏洞被攻破，攻击者获得的就是 root 权限。用 `appuser` 运行应用是一个基本的安全实践。

**COPY 的顺序为什么重要**：Docker 按层缓存。`package.json` 和 `pnpm-lock.yaml` 变化的频率远低于源代码，所以先复制它们并安装依赖，这样代码改了不需要重新安装依赖。如果你 `COPY . .` 在最前面，改一行代码就会导致依赖重新安装，构建时间翻好几倍。

**CMD 里的 `prisma migrate deploy`**：确保应用启动前数据库 schema 是最新的。这比在 CI/CD 流水线里单独跑迁移更可靠——你不需要担心"先部署代码还是先跑迁移"的顺序问题。不过这也意味着每次容器重启都会检查迁移（Prisma 会跳过已经执行过的），开销很小。

构建一次看看最终镜像有多大：

```bash
docker build -t blog-api .
docker images blog-api
# REPOSITORY   TAG       SIZE
# blog-api     latest    ~180MB
```

180MB 左右是合理的。如果超过了 300MB，检查一下是不是 `node_modules` 没排除，或者没用 `--prod` 安装。

## Docker Compose：一键启动完整环境

博客平台不只一个 Node 应用，还需要 PostgreSQL、Redis、Nginx。Docker Compose 把这些服务编排在一起：

```yaml
# docker-compose.yml

services:
  # PostgreSQL 数据库
  postgres:
    image: postgres:16-alpine
    container_name: blog-postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-blog}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?请设置 POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB:-blog_platform}
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U blog"]
      interval: 10s
      timeout: 5s
      retries: 5

  # Redis 缓存 + 消息队列
  redis:
    image: redis:7-alpine
    container_name: blog-redis
    restart: unless-stopped
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5

  # NestJS 应用
  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: blog-app
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      NODE_ENV: production
      DATABASE_URL: postgresql://${POSTGRES_USER:-blog}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB:-blog_platform}
      REDIS_HOST: redis
      REDIS_PORT: 6379
      JWT_SECRET: ${JWT_SECRET:?请设置 JWT_SECRET}
      JWT_EXPIRES_IN: ${JWT_EXPIRES_IN:-15m}
      JWT_REFRESH_SECRET: ${JWT_REFRESH_SECRET:?请设置 JWT_REFRESH_SECRET}
      JWT_REFRESH_EXPIRES_IN: ${JWT_REFRESH_EXPIRES_IN:-7d}
      PORT: 3000
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://localhost:3000/health || exit 1"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  # Nginx 反向代理
  nginx:
    image: nginx:alpine
    container_name: blog-nginx
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
    depends_on:
      app:
        condition: service_healthy

volumes:
  postgres_data:
  redis_data:
```

几个关键配置解释：

**`depends_on` + `condition: service_healthy`**：app 服务会等到 PostgreSQL 和 Redis 的健康检查通过后才启动。Nginx 会等到 app 的健康检查通过后才启动。这避免了"应用启动了但数据库还没就绪"的竞态问题。

**Redis 的 `command` 配置**：`appendonly yes` 开启 AOF 持久化，容器重启后数据不丢。`maxmemory 256mb` 限制内存使用，`allkeys-lru` 在内存满时淘汰最近最少使用的 key。博客平台的缓存数据丢失了可以重建，所以用 LRU 淘汰策略是合理的。

**环境变量的 `${VAR:?msg}` 语法**：如果环境变量没设置，Docker Compose 启动时报错并显示提示信息，而不是用一个空值默默运行。这是防止配置遗漏的手段。对比 `${VAR:-default}`，后者在变量未设置时使用默认值。

**`restart: unless-stopped`**：容器崩溃或服务器重启后自动恢复。比 `always` 更合理——如果你手动停了它（`docker compose stop`），重启后不会自动拉起来。

**app 的 `healthcheck`**：用 `wget` 请求 `/health` 端点。`start_period: 40s` 给应用足够的启动时间（包括数据库迁移），在这个期间内失败不计入重试次数。

环境变量放在 `.env` 文件里（这个文件不要提交到 Git）：

```env
# .env（生产环境）
POSTGRES_USER=blog
POSTGRES_PASSWORD=kB8mR2xL9pQ3wT7n
POSTGRES_DB=blog_platform
JWT_SECRET=a1f9e3c7b2d4m6k8p0q2s4t6u8v0w2x4y6z8
JWT_REFRESH_SECRET=b2g0f4d8h2j6l0n4p8r0s4t8v0x2z4b6d8f0
```

同时提供一个 `.env.example` 提交到 Git，告诉团队成员需要配置哪些变量：

```env
# .env.example
POSTGRES_USER=blog
POSTGRES_PASSWORD=请设置一个强密码
POSTGRES_DB=blog_platform
JWT_SECRET=请设置一个随机字符串
JWT_REFRESH_SECRET=请设置另一个随机字符串
```

## Nginx 反向代理：为什么不直接暴露 3000 端口

你当然可以让用户直接访问 `http://your-server:3000`。但生产环境有几个问题要解决：

- 静态文件（头像、封面图）需要缓存策略
- HTTPS 需要证书管理和 SSL 终止
- 需要请求头转发（X-Real-IP、X-Forwarded-Proto 等）
- 需要限制请求体大小（文件上传）
- 需要 gzip 压缩

Nginx 作为反向代理处理这些事情：

```nginx
# nginx/conf.d/default.conf

upstream nestjs_app {
    server app:3000;
}

server {
    listen 80;
    server_name your-domain.com;

    # 限制请求体大小（文件上传需要）
    client_max_body_size 20m;

    # gzip 压缩
    gzip on;
    gzip_types text/plain application/json application/javascript text/css;
    gzip_min_length 1024;

    # 代理到 NestJS 应用
    location / {
        proxy_pass http://nestjs_app;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;

        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    # 健康检查端点不走日志
    location /health {
        proxy_pass http://nestjs_app;
        access_log off;
    }

    # WebSocket 代理（实时评论用）
    location /socket.io/ {
        proxy_pass http://nestjs_app;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_read_timeout 86400;
    }
}
```

`proxy_set_header X-Real-IP` 和 `X-Forwarded-For` 很重要——NestJS 应用看到的客户端 IP 是 Nginx 容器的内网 IP，如果不转发真实 IP，日志里记录的全是 `172.x.x.x`，没有意义。

WebSocket 的 `proxy_read_timeout 86400` 把超时设为 24 小时。默认 60 秒的超时会让长连接断开，导致实时推送失效。

`client_max_body_size 20m` 和博客平台的文件上传配合。NestJS 里如果用 `@nestjs/serve-static` 提供静态文件服务，Nginx 层面不需要额外配置缓存——但如果你用 CDN 或对象存储（S3/OSS），静态文件根本不经过 Nginx。

如果需要 HTTPS，在 Nginx 里加上 SSL 配置（证书可以用 Let's Encrypt 免费申请）：

```nginx
server {
    listen 443 ssl http2;
    server_name your-domain.com;

    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    # ... 其余配置和上面一样
}
```

## 健康检查端点：让运维知道服务还活着

前面 Docker Compose 里引用了 `/health` 端点。这个端点需要在 NestJS 里实现：

```typescript
// src/health/health.controller.ts
import { Controller, Get } from "@nestjs/common";
import { ApiTags, ApiOperation } from "@nestjs/swagger";
import { Public } from "../auth/decorators/public.decorator";
import { PrismaService } from "../database/prisma.service";
import { Redis } from "ioredis";

@ApiTags("系统")
@Controller("health")
export class HealthController {
  constructor(
    private prisma: PrismaService,
    private redis: Redis
  ) {}

  @Get()
  @Public()
  @ApiOperation({ summary: "健康检查" })
  async check() {
    const start = Date.now();

    try {
      await this.prisma.$queryRaw`SELECT 1`;
      const dbStatus = "ok";
      const dbLatency = Date.now() - start;

      await this.redis.ping();
      const redisStatus = "ok";
      const redisLatency = Date.now() - start - dbLatency;

      return {
        status: "ok",
        timestamp: new Date().toISOString(),
        uptime: process.uptime(),
        services: {
          database: { status: dbStatus, latency: `${dbLatency}ms` },
          redis: { status: redisStatus, latency: `${redisLatency}ms` },
        },
      };
    } catch (error) {
      return {
        status: "error",
        timestamp: new Date().toISOString(),
        error: error.message,
      };
    }
  }
}
```

这个端点做两件事：检查数据库和 Redis 是否可达，返回各自的延迟。返回的 JSON 结构固定，Docker 的 healthcheck、Kubernetes 的 liveness probe、监控系统的告警规则都可以解析它。

Docker 的 healthcheck 配置是：

```yaml
healthcheck:
  test: ["CMD-SHELL", "wget -qO- http://localhost:3000/health || exit 1"]
  interval: 30s
  timeout: 10s
  retries: 3
  start_period: 40s
```

`start_period` 很关键：应用启动时要跑数据库迁移，这需要一些时间。如果 `start_period` 太短，健康检查会在迁移还没完成时就判定应用不健康，导致容器反复重启。

## 生产安全：CORS、Helmet、Rate Limiting

容器化解决的是环境一致性问题，应用本身的安全加固还需要在代码里做。

在 `main.ts` 里加上安全中间件：

```typescript
// src/main.ts
import helmet from "helmet";
import rateLimit from "express-rate-limit";
import compression from "compression";

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // Helmet：设置安全相关的 HTTP 头
  app.use(helmet());

  // CORS：只允许指定域名访问
  app.enableCors({
    origin: process.env.ALLOWED_ORIGINS?.split(",") || [
      "http://localhost:3001",
    ],
    credentials: true,
    methods: ["GET", "POST", "PUT", "PATCH", "DELETE"],
    allowedHeaders: ["Content-Type", "Authorization"],
  });

  // Rate Limiting：防止暴力攻击
  app.use(
    rateLimit({
      windowMs: 15 * 60 * 1000, // 15 分钟
      max: 100, // 每个 IP 最多 100 个请求
      standardHeaders: true,
      legacyHeaders: false,
      message: { message: "请求过于频繁，请稍后再试" },
    })
  );

  // 压缩响应
  app.use(compression());

  // 生产环境关闭 Swagger
  if (process.env.NODE_ENV !== "production") {
    const config = new DocumentBuilder()
      .setTitle("技术博客平台 API")
      .setVersion("1.0")
      .addBearerAuth({ type: "http", scheme: "bearer" }, "access-token")
      .build();
    const document = SwaggerModule.createDocument(app, config);
    SwaggerModule.setup("api-docs", app, document);
  }

  await app.listen(process.env.PORT || 3000);
}
bootstrap();
```

安装依赖：

```bash
pnpm add helmet express-rate-limit compression
pnpm add -D @types/compression
```

这三层安全措施各有侧重：

**Helmet** 自动设置了十几个安全相关的 HTTP 响应头。比如 `X-Content-Type-Options: nosniff` 防止浏览器猜测 MIME 类型、`X-Frame-Options: DENY` 防止被 iframe 嵌套用于点击劫持。一行代码加上，不需要自己逐个配置。

**CORS** 限制哪些前端域名可以调用你的 API。`origin: ['http://localhost:3001']` 在开发环境够用，生产环境改成你实际的前端域名。不要用 `origin: '*'`，这会允许任何网站跨域请求你的 API。

**Rate Limiting** 限制单个 IP 的请求频率。15 分钟 100 次对正常用户完全够用，但能挡住大部分暴力破解和 DDoS 攻击。登录接口可以单独设置更严格的限制（15 分钟 5 次）：

```typescript
// 登录接口单独限流
app.use(
  "/auth/login",
  rateLimit({
    windowMs: 15 * 60 * 1000,
    max: 5,
    message: { message: "登录尝试过多，请 15 分钟后再试" },
  })
);
```

## 日志管理：console.log 不够用

开发阶段 `console.log` 很方便，但生产环境有几个问题：没有时间戳、没有日志级别、不能按模块过滤、不能输出到文件、容器重启后日志就没了。

NestJS 内置的 Logger 比原生 console 好一些，但在容器化环境里，推荐让日志输出到 stdout/stderr，由 Docker 的日志驱动统一收集。

```typescript
// main.ts 里配置 Logger
async function bootstrap() {
  const app = await NestFactory.create(AppModule, {
    logger:
      process.env.NODE_ENV === "production"
        ? ["error", "warn", "log"]
        : ["error", "warn", "log", "debug", "verbose"],
  });
  // ...
}
```

生产环境只输出 error、warn、log 三个级别，减少日志量。debug 和 verbose 只在开发环境开启。

Docker 默认用 `json-file` 日志驱动。可以在 `docker-compose.yml` 里配置日志轮转，防止日志文件无限增长：

```yaml
services:
  app:
    build: .
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
  postgres:
    image: postgres:16-alpine
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "2"
```

这样每个日志文件最大 10MB，最多保留 3 个文件，不会撑爆磁盘。

查看日志的常用命令：

```bash
# 查看所有服务日志
docker compose logs -f

# 只看 app 的日志
docker compose logs -f app

# 看最近 100 行
docker compose logs --tail 100 app

# 看指定时间之后的日志
docker compose logs --since 30m app
```

如果后续上了 ELK（Elasticsearch + Logstash + Kibana）或 Loki 这样的日志系统，改一下 Docker 的日志驱动就行，应用代码不需要改。这也是容器化的好处——基础设施的关注点从应用里剥离出来了。

## 部署流程：从本地到生产

有了 Dockerfile 和 docker-compose.yml，部署流程就很简单了：

```bash
# 1. 克隆代码
git clone your-repo.git
cd blog-api

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env，填入生产环境的数据库密码、JWT 密钥等
```

更新部署：

```bash
# 拉取最新代码
git pull origin main

# 重新构建并启动（只会重建有变化的层）
docker compose up -d --build

# 清理旧的镜像
docker image prune -f
```

`--build` 参数让 Docker Compose 在启动前重新构建镜像。因为 Dockerfile 里利用了层缓存，只有 package.json 变了才会重新安装依赖，只有源代码变了才会重新编译。日常改代码的部署通常只需要几十秒。

如果需要零停机部署（blue-green 部署），可以启动两份 app 容器，Nginx 通过 upstream 轮询，逐步切换流量。但这对小团队来说可能过度设计了——Docker Compose 的 `restart` 策略加上几秒的重启时间，对大多数博客平台已经够用。

## 系列回顾

这是 NestJS 渐进式学习系列的最后一篇。从第 01 篇到第 18 篇，我们围绕一个技术博客平台，从零构建了一个完整的后端 API。回顾一下整个旅程：

**基础篇（01-03）**——理解了 NestJS 的设计哲学，搭建了项目骨架，搞清楚了模块化架构和控制器路由。这三个概念是 NestJS 的地基，后面所有内容都建立在这个基础之上。

**核心篇（04-08）**——接入了 PostgreSQL + Prisma 实现数据持久化，完成了文章的 CRUD 操作，用 class-validator 做了数据验证，用拦截器和异常过滤器统一了响应格式。到第 08 篇结束时，博客平台已经有了一个结构清晰、格式统一的 API 骨架。

**进阶篇（09-15）**——这是系列最长的部分，也是离生产环境最近的。实现了 JWT 认证和角色授权，解决了"用户是谁"和"用户能做什么"两个核心安全问题。文件上传和大文件切片处理了用户内容管理的需求。BullMQ 异步任务队列让耗时操作不阻塞请求。Redis 缓存加速了高频查询。WebSocket 实现了实时评论推送。

**部署篇（16-18）**——为核心模块写了单元测试和 E2E 测试，用 Swagger 自动生成了 API 文档，最后用 Docker 多阶段构建加 Compose 编排完成了容器化部署。

18 篇文章下来，我们构建的东西具体包括：

- 一个模块化的 NestJS 项目，有清晰的数据模型（User、Post、Comment、Tag）
- 完整的 JWT 认证流程（注册、登录、刷新 token、退出）
- 基于角色的访问控制（reader、author、admin）
- 文章全生命周期管理（草稿、发布、编辑、软删除）
- 文件上传（头像、封面图）和大文件切片上传
- 基于 BullMQ 的异步任务（邮件通知、摘要生成）
- 基于 Redis 的多级缓存策略
- 基于 WebSocket 的实时评论推送
- 覆盖核心路径的测试体系
- 自动生成的 Swagger API 文档
- 容器化的生产部署方案（Docker + Nginx + PostgreSQL + Redis）

你拥有的不只是零散的知识点，而是一个**从需求分析到架构设计到编码实现到测试到部署**的完整实践。这个实践可以迁移到任何 NestJS 项目上。

继续深入的方向有很多：微服务架构（NestJS 原生支持 TCP、Redis、RabbitMQ 等传输层）、GraphQL 替代 REST、Kubernetes 编排替代 Docker Compose、CI/CD 流水线自动化测试和部署。这些都是在当前基础上的自然延伸。
