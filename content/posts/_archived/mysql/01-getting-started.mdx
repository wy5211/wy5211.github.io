你决定用 Next.js 做一个带博客功能的个人网站。页面写好了，但数据往哪存？

有人推荐 MongoDB，说文档型数据库和 JavaScript 天然契合。但现实是，大多数公司的后端都用 MySQL，面试也考 MySQL，你想趁这个机会把 MySQL 学了。

问题是——你以前从来没碰过数据库。连 MySQL 怎么安装都不知道，更别说建表、写查询了。

别慌。这篇文章带你从零开始：用 Docker 启动 MySQL，创建数据库，建第一张表，跑通第一个查询。整个过程不超过 30 分钟。

## 用 Docker 启动 MySQL

安装 MySQL 最烦的是什么？版本冲突、配置文件找不到、卸载不干净。Docker 完美解决了这些问题——一行命令启动，不用就删掉，系统干干净净。

先确认你装了 Docker：

```bash
docker --version
# Docker version 24.x 就行
```

然后一行命令启动 MySQL 8.0：

```bash
docker run -d \
  --name mysql-blog \
  -e MYSQL_ROOT_PASSWORD=dev_password \
  -e MYSQL_DATABASE=blog_db \
  -p 3306:3306 \
  mysql:8.0
```

逐行解释一下：

- `-d`：后台运行，不占用当前终端
- `--name mysql-blog`：给容器起个名字，后面好管理
- `-e MYSQL_ROOT_PASSWORD=dev_password`：设置 root 密码（开发环境用简单密码就行）
- `-e MYSQL_DATABASE=blog_db`：启动时自动创建一个叫 `blog_db` 的数据库
- `-p 3306:3306`：把容器内的 3306 端口映射到本机（MySQL 默认端口）
- `mysql:8.0`：使用 MySQL 8.0 镜像

等个十几秒，MySQL 就跑起来了。可以用这个命令确认状态：

```bash
docker ps
# 看到 mysql-blog 的 STATUS 是 Up 就说明启动成功了
```

**如果端口被占用怎么办？** 说明你本机已经有 MySQL 在运行。把 `-p 3306:3306` 改成 `-p 3307:3306`，用 3307 端口连接就行。

## 连上 MySQL 看看

启动了 MySQL，怎么连上去？有两种方式：命令行和图形工具。

### 命令行连接

```bash
docker exec -it mysql-blog mysql -u root -pdev_password blog_db
```

- `docker exec -it`：在容器里执行交互式命令
- `mysql -u root -pdev_password`：用 root 用户连接，密码紧跟 `-p` 后面（**注意 `-p` 和密码之间没有空格**，这是个经典坑）
- `blog_db`：直接连到我们创建的数据库

连上之后你会看到 `mysql>` 提示符。试试这几个命令：

```sql
-- 看看有哪些数据库
SHOW DATABASES;

-- 确认当前在哪个数据库
SELECT DATABASE();

-- 看看当前数据库有哪些表（当然是空的）
SHOW TABLES;
```

### 用 DBeaver 连接（推荐）

命令行能用，但日常开发还是图形工具更方便。推荐 DBeaver——免费、跨平台、支持几乎所有数据库。

下载安装就不说了，连接配置填这些：

| 字段   | 值           |
| ------ | ------------ |
| 主机   | localhost    |
| 端口   | 3306         |
| 用户名 | root         |
| 密码   | dev_password |
| 数据库 | blog_db      |

连上后你会看到左侧树形结构里有 `blog_db` 数据库。以后写 SQL 可以直接在 DBeaver 的 SQL 编辑器里跑，比命令行舒服多了。

## SQL 语言分类：30 秒搞懂

SQL 命令虽然多，但按用途分就四类，知道名字就行：

| 分类 | 全称              | 干什么   | 常用命令               |
| ---- | ----------------- | -------- | ---------------------- |
| DDL  | Data Definition   | 定义结构 | CREATE、ALTER、DROP    |
| DML  | Data Manipulation | 操作数据 | INSERT、UPDATE、DELETE |
| DQL  | Data Query        | 查询数据 | SELECT                 |
| DCL  | Data Control      | 控制权限 | GRANT、REVOKE          |

不需要死记。你只需要知道：**建表用 DDL，增删改用 DML，查询用 DQL，权限用 DCL**。前三个天天用，最后一个开发环境基本碰不到。

## 建第一张表：用户表

现在正式开始。博客系统的第一张表——用户表。一个用户注册后，需要存这些信息：

```
用户 ID、用户名、邮箱、密码哈希、头像 URL、个人简介、VIP 等级、注册时间
```

写成 SQL：

```sql
CREATE TABLE users (
  id INT AUTO_INCREMENT PRIMARY KEY,
  username VARCHAR(50) NOT NULL UNIQUE,
  email VARCHAR(255) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  avatar_url VARCHAR(500),
  bio TEXT,
  vip_level TINYINT DEFAULT 0,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

逐行看看这张表的设计思路：

- `id INT AUTO_INCREMENT PRIMARY KEY`：自增主键，每条记录自动分配唯一编号。大部分表都需要这么一个字段
- `username VARCHAR(50) NOT NULL UNIQUE`：用户名，最长 50 字符，不能为空，不能重复（UNIQUE）
- `password_hash VARCHAR(255) NOT NULL`：存的是密码的哈希值，**永远不要明文存密码**。这很重要，后面会讲怎么做
- `avatar_url VARCHAR(500)`：头像链接，可以为空（用户不一定要传头像）
- `bio TEXT`：个人简介，用 TEXT 因为长度不确定
- `vip_level TINYINT DEFAULT 0`：VIP 等级，默认 0 表示普通用户
- `created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`：注册时间，自动填入当前时间

这里面出现了几种数据类型：INT、VARCHAR、TEXT、TINYINT、TIMESTAMP。下一篇会详细讲怎么选数据类型，现在先知道有这些就行。

## 第一次增删改查

表建好了，来跑一遍 CRUD（Create、Read、Update、Delete）——这是你以后最常写的四种操作。

### 插入数据（Create）

```sql
INSERT INTO users (username, email, password_hash, avatar_url, bio)
VALUES (
  'wang_dev',
  'wang@example.com',
  '$2b$10$dummyHashForDev',
  'https://cdn.example.com/avatars/wang_dev.jpg',
  '全栈独立开发者，热爱前端和开源'
);
```

注意几点：

- `id` 不用写，AUTO_INCREMENT 会自动分配
- `created_at` 不用写，DEFAULT CURRENT_TIMESTAMP 会自动填入
- 字符串用单引号 `'`，不是双引号

### 查询数据（Read）

```sql
-- 查所有用户
SELECT * FROM users;

-- 只查需要的列（推荐，比 SELECT * 性能好）
SELECT username, email, vip_level, created_at
FROM users;

-- 条件查询：按用户名查找
SELECT * FROM users WHERE username = 'wang_dev';
```

### 更新数据（Update）

```sql
UPDATE users
SET bio = '独立开发者，专注前端和全栈开发'
WHERE username = 'wang_dev';
```

**重要提醒**：UPDATE 一定要写 WHERE！不写 WHERE 的话，所有行都会被更新。这是个新手经典翻车现场。

养成习惯：先写 `UPDATE users SET ... WHERE id = 1`，确认 WHERE 条件对了，再填 SET 的内容。

### 删除数据（Delete）

```sql
DELETE FROM users WHERE username = 'wang_dev';
```

同样，DELETE 不写 WHERE 会删掉所有数据。生产环境中，通常用"软删除"——不真的删，而是标记为已删除：

```sql
-- 给表加一个 deleted_at 字段
ALTER TABLE users ADD COLUMN deleted_at TIMESTAMP NULL;

-- "删除"实际上是更新这个字段
UPDATE users SET deleted_at = CURRENT_TIMESTAMP WHERE username = 'wang_dev';

-- 查询时过滤掉已删除的
SELECT * FROM users WHERE deleted_at IS NULL;
```

软删除在 [第 3 篇：增删改查实战](/posts/mysql/03-crud-operations) 会详细讲。

## 再插入几条测试数据

为了让后面的查询有数据可玩，再插入几条用户：

```sql
INSERT INTO users (username, email, password_hash, bio, vip_level) VALUES
  ('frontend_lee', 'lee@example.com', '$2b$10$hash1', 'React 重度用户', 1),
  ('vue_fan', 'vue@example.com', '$2b$10$hash2', 'Vue 全家桶选手', 2),

```

用一条 INSERT 插入多行数据，比写多条 INSERT 效率高得多。这是批量操作的基本技巧。

## 命名规范：写别人能看懂的 SQL

既然开始写 SQL 了，先养成好习惯：

| 规范         | 正确                | 错误                | 原因                        |
| ------------ | ------------------- | ------------------- | --------------------------- |
| 表名用复数   | users、articles     | user、article       | 一张表存多条记录            |
| 字段用下划线 | created_at、user_id | createdAt、userId   | SQL 的惯例是 snake_case     |
| 别用保留字   | name、desc、order   | —                   | 这些是 SQL 关键字，容易冲突 |
| 主键叫 id    | id                  | user_id（在用户表） | 简洁且通用                  |

其中 `desc` 是个特别常见的坑——你可能想给文章表加一个 `desc` 字段存描述，但 `DESC` 是 MySQL 关键字（`DESCRIBE` 的缩写）。如果一定要用，得加反引号 `` `desc` ``。不如直接叫 `description` 或 `summary`，省心。

## 什么时候该用什么工具

聊几个实际开发中你会遇到的取舍：

**Docker vs 本地安装**：学习阶段用 Docker，不污染系统。如果你在公司开发，后端同事通常已经帮你搞定了数据库环境，你只需要连上去就行。

**命令行 vs DBeaver**：学习阶段两种都用用。命令行让你理解每条 SQL 在做什么，DBeaver 让你更快地浏览数据和表结构。日常开发 DBeaver 更方便，但排查问题时命令行更灵活。

**SELECT \* vs 指定列**：平时随手查数据用 `SELECT *` 没问题，但在代码里（比如 API 接口的 SQL）要写具体的列名。原因有两个：一是性能，不需要的列不查；二是如果表结构变了（比如加了个字段），`SELECT *` 的结果也会变，可能导致代码出错。
