前面 7 篇，我们一直在一张张"已经建好的表"上操作——增删改查、JOIN、聚合、索引、事务。但你有没有想过：**这些表为什么长这样？**

为什么文章和标签要用中间表而不是在文章表里加个 `tag_ids` 字段？为什么标签表里要冗余一个 `article_count`？评论的回复功能为什么用 `parent_id` 而不是单独建一张回复表？

这篇来回答这些问题。不是讲教科书里的三范式（说实话前端项目不太需要），而是从实际需求出发，讲清楚每个设计决策背后的思考。

## 关系类型：一对一、一对多、多对多

数据库设计的第一步是搞清楚数据之间的关系。博客系统里有这几种：

### 一对多

**用户和文章**：一个用户可以发多篇文章，一篇文章只属于一个用户。

```sql
-- 在"多"的一方（articles）加一个指向"一"的一方（users）的字段
CREATE TABLE articles (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,  -- 指向 users.id
  ...
);
```

**判断标准**：A 拥有多个 B，B 只属于一个 A → 在 B 表里加 `a_id` 字段。

这是最常见的关型。用户-文章、文章-评论、用户-项目都是一对多。

### 多对多

**文章和标签**：一篇文章可以有多个标签，一个标签下有多篇文章。

多对多不能通过在任一方加字段解决——你在文章表里加 `tag_id`，一篇文章只能有一个标签；在标签表里加 `article_id`，一个标签只能属于一篇文章。

**解决方案：中间表**。

```sql
-- 中间表只存两个 ID 的对应关系
CREATE TABLE article_tags (
  article_id INT NOT NULL,
  tag_id INT NOT NULL,
  PRIMARY KEY (article_id, tag_id)
);
```

联合主键 `(article_id, tag_id)` 保证了一篇文章不会重复关联同一个标签。

**判断标准**：A 有多个 B，B 也属于多个 A → 建中间表。

前端项目里常见的多对多：用户-角色、文章-标签、学生-课程。

### 一对一

**用户和用户设置**：一个用户只有一套设置，一套设置只属于一个用户。

```sql
-- 方案一：在用户表里直接加字段（设置少的时候）
ALTER TABLE users ADD COLUMN theme VARCHAR(20) DEFAULT 'light';

-- 方案二：单独建一张表（设置多的时候）
CREATE TABLE user_settings (
  user_id INT PRIMARY KEY,  -- 既是主键也是外键
  theme VARCHAR(20) DEFAULT 'light',
  language VARCHAR(10) DEFAULT 'zh-CN',
  email_notification TINYINT DEFAULT 1
);
```

一对一关系在前端项目里不太常见。大部分时候直接把字段加在主表里就行，除非字段特别多或者有些字段很少用到（避免主表太宽）。

## 外键约束：要不要用

建表时可以加外键约束：

```sql
CREATE TABLE articles (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  ...
  FOREIGN KEY (user_id) REFERENCES users(id)
    ON DELETE CASCADE  -- 用户被删了，文章也跟着删
);
```

外键约束保证数据一致性——不可能存在 `user_id = 999` 的文章，如果 users 表里没有 id=999 的用户。

**但实际开发中有两种观点**：

- **用外键**：数据库层面保证一致性，不需要应用层做检查
- **不用外键**：外键影响性能（每次写入都要检查），而且级联删除可能导致意外数据丢失

**前端项目的建议**：开发阶段不用外键，用应用层代码保证一致性。原因很简单——外键的级联操作（CASCADE）是隐式的，你可能不知道删一个用户会连带删掉他的所有文章。显式地在代码里处理更可控。

## 范式的直觉：什么时候该冗余

数据库设计有个"三范式"的理论，核心思想是"不要重复存数据"。比如不应该在文章表里存作者的用户名，因为用户改名后要改两处。

但生产环境经常需要"反范式"——故意冗余一些数据，用空间换时间。

博客系统里的例子：

### 冗余：标签表的 article_count

```sql
-- tags 表里有 article_count 字段
-- 每次文章发布/删除时更新它
UPDATE tags SET article_count = article_count + 1 WHERE id = 1;
```

严格按范式，不应该存这个值——每次需要时 `SELECT COUNT(*) FROM article_tags WHERE tag_id = 1` 就行。但标签列表页要显示每个标签的文章数，如果每次都 COUNT，标签多了要查 N 次。直接冗余一个计数字段，查询时直接读就行。

**什么时候该冗余**：某个值经常需要读取，但计算它需要 JOIN 或 COUNT，而且更新频率不高。

**什么时候不该冗余**：数据经常变化，或者冗余后一致性的维护成本很高。

### 不冗余：文章的作者信息

文章列表显示作者用户名，不需要在文章表里存 `author_name`——因为用户名可能改，而且通过 JOIN 查也很简单：

```sql
SELECT a.title, u.username FROM articles a JOIN users u ON a.user_id = u.id;
```

这种场景 JOIN 就够了，没必要冗余。

## JSON 字段：什么时候用，什么时候不用

[第 2 篇](/posts/mysql/02-table-design) 里项目表用了 JSON 存灵活数据：

```sql
-- 适合用 JSON 的场景
tech_stack JSON,    -- 技术栈组合，每个项目不一样
links JSON,         -- 链接集合，字段不固定
```

**适合用 JSON 的情况**：

- 数据结构因记录而异（每个项目的链接字段不一样）
- 字段可能增减，不想频繁 ALTER TABLE
- 不需要对 JSON 内部的字段做条件查询或排序

**不适合用 JSON 的情况**：

- 每条记录都有这个字段（比如文章标题，每篇都有）
- 需要对这个字段建索引或做 WHERE 条件
- 需要用这个字段做 JOIN 关联

比如评论内容、文章标题、用户邮箱——这些应该用普通列，不要放进 JSON。

## 软删除策略

[第 3 篇](/posts/mysql/03-crud-operations) 介绍了软删除：用 `deleted_at` 字段标记删除，而不是真的 DELETE。

软删除的设计要点：

```sql
-- 加 deleted_at 字段
ALTER TABLE articles ADD COLUMN deleted_at TIMESTAMP NULL DEFAULT NULL;

-- 查询时过滤
SELECT * FROM articles WHERE deleted_at IS NULL;

-- "删除"操作
UPDATE articles SET deleted_at = CURRENT_TIMESTAMP WHERE id = 1;

-- 真正删除（比如定期清理 30 天前的软删除数据）
DELETE FROM articles WHERE deleted_at IS NOT NULL AND deleted_at < DATE_SUB(NOW(), INTERVAL 30 DAY);
```

**哪些表适合软删除**：核心业务数据——文章、用户、评论。删了可能要恢复。

**哪些表不需要软删除**：关联表（article_tags）、配置表、临时数据。删了就删了，不需要恢复。

## 时间戳设计

几乎每张表都需要 `created_at` 和 `updated_at`：

```sql
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

几个注意事项：

- `created_at` 不要用 `ON UPDATE`，只在插入时自动填入
- `updated_at` 加上 `ON UPDATE CURRENT_TIMESTAMP`，每次 UPDATE 自动刷新
- 如果你在事务里手动设置了 `updated_at`，MySQL 不会再自动更新（以你手动设置的为准）
- 查询"最近更新"时用 `ORDER BY updated_at DESC`

## 从设计稿到表结构

最后讲一个实际的工作流程。产品经理给了你一个页面设计稿，怎么推导出表结构？

以"项目展示页"为例：

**第一步：找出实体（名词）**

设计稿里有：项目、截图、技术标签。→ 三张表：projects、project_screenshots、tags（已有）。

**第二步：找出关系（动词）**

项目有多个截图 → 一对多（在 screenshots 表加 project_id）
项目有多个标签，标签属于多个项目 → 多对多（建中间表 project_tags）

**第三步：确定每个实体的属性**

项目：标题、描述、状态、排序、链接...
截图：图片 URL、排序、说明...

**第四步：选数据类型，建表**

```sql
CREATE TABLE projects (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  title VARCHAR(200) NOT NULL,
  description TEXT,
  tech_stack JSON,
  links JSON,
  status ENUM('draft', 'published', 'archived') DEFAULT 'draft',
  sort_order INT DEFAULT 0,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE project_screenshots (
  id INT AUTO_INCREMENT PRIMARY KEY,
  project_id INT NOT NULL,
  image_url VARCHAR(500) NOT NULL,
  caption VARCHAR(200),
  sort_order INT DEFAULT 0
);
```

**第五步：加索引**

```sql
CREATE INDEX idx_project_user ON projects(user_id);
CREATE INDEX idx_screenshot_project ON project_screenshots(project_id);
```

这个流程可以套用到任何页面：找实体 → 找关系 → 定属性 → 选类型 → 建表加索引。

## 博客系统完整表结构回顾

整个系列的 8 张表设计一览：

```
users             用户表
  ↓ 一对多
articles          文章表
  ↓ 多对多（通过 article_tags）
tags              标签表

articles          文章表
  ↓ 一对多
comments          评论表（parent_id 自引用实现回复）

users             用户表
  ↓ 一对多
projects          项目展示表

projects          项目展示表
  ↓ 一对多
project_screenshots 项目截图表
```

设计回顾：

- **users** 是核心，通过 user_id 关联到 articles、comments、projects
- **articles 和 tags** 通过中间表 article_tags 实现多对多
- **comments** 用 parent_id 自引用实现嵌套回复，避免了多张评论表
- **projects** 的 tech_stack 和 links 用 JSON 存灵活数据
- 每张表都有 `id`、`created_at`，需要更新的表有 `updated_at`

这个结构支撑了一个完整的个人博客 + 作品集网站的所有功能。它不是"教科书式"的完美设计，而是前端项目够用、好用、好维护的设计。

## 设计决策速查

| 问题                  | 建议                                       |
| --------------------- | ------------------------------------------ |
| 一对多还是多对多      | 想想"A 能有多个 B 吗？B 能属于多个 A 吗？" |
| 要不要外键约束        | 前端项目建议不用，用代码保证一致性         |
| 要不要冗余字段        | 读取频繁、计算成本高的考虑冗余             |
| 用 JSON 还是拆表      | 结构灵活多变用 JSON，需要查询排序用独立列  |
| 要不要软删除          | 核心数据用软删除，关联数据不需要           |
| created_at/updated_at | 每张表都加上                               |
| 索引怎么加            | 先有慢查询再加，用 EXPLAIN 验证            |

MySQL 的学习不是背语法，而是在实际项目中解决问题。希望这个系列帮你建立了对数据库的直觉，以后碰到数据库相关的问题，知道从哪个方向思考。
