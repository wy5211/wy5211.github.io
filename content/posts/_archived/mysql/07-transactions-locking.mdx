上一篇我们用索引解决了查询慢的问题。但想象一个场景：MacBook Pro 库存只剩最后 1 件，两个用户同时点击"立即购买"。两个请求几乎同时到达后端，各自检查库存发现都够，各自扣减库存——结果库存变成了 -1。

这不是查询优化能解决的问题，而是**并发写入时的数据一致性**问题。事务和锁就是 MySQL 用来解决这类问题的机制。

主线场景：电商系统的下单流程。用户点击"下单"后，系统需要依次完成三件事：扣减库存 → 创建订单 → 记录支付流水。这三步要么全部成功，要么全部回滚。中间任何一步失败，之前做的操作都要撤销。

## 事务：要么全做，要么全不做

### ACID 四个特性

用下单场景来理解 ACID：

```sql
START TRANSACTION;

-- 第 1 步：扣减库存
UPDATE products SET stock = stock - 1 WHERE id = 101 AND stock >= 1;

-- 第 2 步：创建订单
INSERT INTO orders (user_id, total_amount, status) VALUES (1, 14999.00, 2);

-- 第 3 步：记录支付流水
INSERT INTO payments (order_id, amount, channel) VALUES (LAST_INSERT_ID(), 14999.00, 'alipay');

COMMIT;
```

**原子性（Atomicity）**：三步操作是一个不可分割的整体。如果第 3 步插入支付流水时失败了（比如支付宝接口超时），前两步的操作也会被撤销——库存恢复，订单消失。用 `ROLLBACK` 替代 `COMMIT` 就能做到。

**一致性（Consistency）**：事务执行前后，数据库从一个合法状态变成另一个合法状态。库存不会变成负数，订单金额和支付金额必须一致。一致性是由原子性 + 应用层的约束（比如 `stock >= 1`）共同保证的。

**隔离性（Isolation）**：两个用户同时下单，各自的事务互不干扰。但"互不干扰"有不同的程度——这就是隔离级别的概念。

**持久性（Durability）**：一旦 COMMIT，数据就不会丢。即使 MySQL 崩溃、服务器断电，重启后数据仍然在。InnoDB 通过 redo log 实现这一点——数据修改先写入 redo log（顺序写，很快），再写入数据文件（随机写，较慢）。COMMIT 时保证 redo log 已刷盘，崩溃后可以靠 redo log 恢复。

### 事务控制语句

```sql
-- 开启事务
START TRANSACTION;  -- 或 BEGIN

-- 回滚到事务开始前的状态
ROLLBACK;

-- 提交事务，所有修改永久生效
COMMIT;

-- 保存点：可以在事务中设一个"书签"
SAVEPOINT after_stock_update;
-- 如果后续操作失败，可以只回滚到这个保存点
ROLLBACK TO after_stock_update;
-- 而不是回滚整个事务
```

SAVEPOINT 在复杂事务中有用——比如下单流程中，如果支付失败，你可能想保留扣库存和创建订单（改状态为"待支付"），只回滚支付那一步。不过实际开发中更常见的做法是把事务拆小，而不是用一个超大事务加 SAVEPOINT。

## 隔离级别：并发时能看到什么

MySQL 支持四种隔离级别，从低到高：READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE。InnoDB 默认使用 **REPEATABLE READ**。

用两个并发会话来演示它们的区别：

### READ UNCOMMITTED：能读到别人还没提交的数据

```
会话 A                              会话 B
BEGIN;
UPDATE products SET stock = 0 WHERE id = 101;
                                    BEGIN;
                                    SELECT stock FROM products WHERE id = 101;
                                    -- 结果：0（但会话 A 还没 COMMIT！）
ROLLBACK;  -- 会话 A 回滚了，库存应该是 1
```

会话 B 读到了会话 A 还没提交的数据（stock = 0），这叫**脏读**。如果会话 B 基于这个"0"做了业务决策（比如告诉用户"已售罄"），而会话 A 回滚了库存恢复成 1，会话 B 的决策就是基于错误数据做的。这个隔离级别几乎没人用。

### READ COMMITTED：只能读到已提交的数据

```
会话 A                              会话 B
BEGIN;
SELECT stock FROM products;          BEGIN;
-- stock = 1
                                    SELECT stock FROM products;
UPDATE products SET stock = 0;       -- 结果：1（读已提交的值）
COMMIT;
                                    SELECT stock FROM products;
                                    -- 结果：0（A 已提交，B 能看到了）
```

脏读解决了，但同一个事务里两次 SELECT 同一行数据结果不同——这叫**不可重复读**。如果你在事务里需要基于一致性快照做计算（比如统计库存总额），两次读取结果不一致会导致计算错误。

### REPEATABLE READ：同一事务内读取结果一致（MySQL 默认）

```
会话 A                              会话 B
BEGIN;
SELECT stock FROM products;          BEGIN;
-- stock = 1
                                    SELECT stock FROM products;
UPDATE products SET stock = 0;       -- 结果：1（和第一次一样）
COMMIT;
                                    SELECT stock FROM products;
                                    -- 结果：1（仍然一样！即使 A 已提交）
                                    COMMIT;
                                    BEGIN;
                                    SELECT stock FROM products;
                                    -- 结果：0（新事务才能看到最新值）
```

InnoDB 通过 **MVCC（多版本并发控制）** 实现了可重复读：事务开始时拍一个"快照"，后续读取都基于这个快照。这解决了不可重复读的问题。

但 REPEATABLE READ 在理论上仍然存在**幻读**的可能——事务 A 读了某范围的数据，事务 B 在这个范围内插入了新行，事务 A 再次读取会多出"幻影"行。InnoDB 通过**间隙锁（Gap Lock）** 在 REPEATABLE READ 级别下也防止了幻读。

### SERIALIZABLE：完全串行化

所有事务排队执行，并发性能最差，一般不用。

| 隔离级别         | 脏读 | 不可重复读 | 幻读           | 性能 |
| ---------------- | ---- | ---------- | -------------- | ---- |
| READ UNCOMMITTED | 可能 | 可能       | 可能           | 最高 |
| READ COMMITTED   | 不会 | 可能       | 可能           | 高   |
| REPEATABLE READ  | 不会 | 不会       | 不会（InnoDB） | 中   |
| SERIALIZABLE     | 不会 | 不会       | 不会           | 最低 |

实际选择：绝大多数场景用默认的 REPEATABLE READ 就够了。少数需要"读最新已提交数据"的场景（比如库存实时查询），可以切换到 READ COMMITTED。

## 锁：InnoDB 怎么防止冲突

### 行锁：锁住特定行

两个事务同时修改同一行数据，InnoDB 会用行锁让后到的事务等待：

```sql
-- 事务 A
BEGIN;
UPDATE orders SET status = 3 WHERE id = 1001;  -- 给 id=1001 这一行加锁
-- 还没 COMMIT...

-- 事务 B（另一个连接）
BEGIN;
UPDATE orders SET status = 5 WHERE id = 1001;  -- 被阻塞！等事务 A 释放锁
```

InnoDB 的行锁是加在**索引**上的，不是加在数据行上。这意味着如果你 UPDATE 时没有命中索引（比如 `WHERE total_amount = 14999` 而 total_amount 没有索引），InnoDB 无法精确定位到行，只能锁住整张表——这就是所谓的"没有索引行锁变表锁"。

### 间隙锁：防止幻读

REPEATABLE READ 级别下，InnoDB 除了锁住已有的行，还会锁住行之间的"间隙"：

```sql
-- 事务 A
BEGIN;
SELECT * FROM orders WHERE id BETWEEN 1000 AND 1020 FOR UPDATE;
-- 锁住 id=1000 到 id=1020 的所有行，以及它们之间的间隙

-- 事务 B
INSERT INTO orders (id, ...) VALUES (1005, ...);
-- 被阻塞！1005 落在 1000-1020 的间隙内
```

间隙锁的目的是防止事务 A 在范围查询期间，事务 B 在范围内插入新行导致幻读。这是一个比较高级的特性，大多数时候你不需要主动关心它——但在排查死锁时，间隙锁经常是"嫌疑人"。

### 死锁：两个事务互相等对方

```sql
-- 事务 A                    -- 事务 B
BEGIN;                       BEGIN;
UPDATE orders SET status=3   UPDATE orders SET status=5
  WHERE id=1001;               WHERE id=1002;
  -- 锁住 1001                 -- 锁住 1002

UPDATE orders SET status=5   UPDATE orders SET status=3
  WHERE id=1002;               WHERE id=1001;
  -- 等待 B 释放 1002          -- 等待 A 释放 1001
  -- 死锁！                    -- 死锁！
```

事务 A 等事务 B 释放 1002 的锁，事务 B 等事务 A 释放 1001 的锁。InnoDB 检测到死锁后，会自动回滚其中一个事务（通常是修改数据量少的那个），让另一个继续执行。

排查死锁的方法：

```sql
-- 查看最近一次死锁信息
SHOW ENGINE INNODB STATUS;
-- 在 LATEST DETECTED DEADLOCK 部分能看到两个事务各持有什么锁、等什么锁
```

预防死锁的经验：

- **按固定顺序操作**：比如总是先更新订单再更新库存，不要一个事务先订单后库存、另一个先库存后订单。
- **保持事务简短**：事务越短，持锁时间越短，冲突概率越低。
- **给查询加合适的索引**：避免行锁升级成表锁。

## 乐观锁 vs 悲观锁：两种并发策略

回到库存扣减的问题。两种常见的解决方案：

### 悲观锁：先锁再改

```sql
BEGIN;
-- 先锁住这行，其他事务不能修改
SELECT stock FROM products WHERE id = 101 FOR UPDATE;
-- stock = 1，确认库存够

UPDATE products SET stock = stock - 1 WHERE id = 101;
INSERT INTO orders (...) VALUES (...);
COMMIT;  -- 释放锁
```

`FOR UPDATE` 在读取时就加上了排他锁，其他事务想修改同一行必须等待。这叫"悲观"——假定冲突一定会发生，提前锁住。

### 乐观锁：先改再检查

```sql
-- products 表有一个 version 字段
ALTER TABLE products ADD COLUMN version INT UNSIGNED DEFAULT 0;

-- 不加锁，直接更新，带上版本号条件
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = 101 AND stock >= 1 AND version = 5;
-- 如果影响行数为 0，说明被其他事务改过了，需要重试
```

乐观锁不加数据库锁，而是通过版本号（或时间戳）在 UPDATE 时检查数据是否被改过。如果 `ROW_COUNT()` 返回 0，说明 version 已经变了——有其他事务先修改了，当前操作需要重试。

### 怎么选

**悲观锁**适合写冲突频繁的场景（比如秒杀——大量请求抢同一个商品）。冲突确实会发生，提前加锁更直接。

**乐观锁**适合读多写少的场景（比如编辑个人资料——两个人同时编辑同一份资料的概率很低）。不加锁提高了读性能，只在更新时做检查。

在应用层，乐观锁的重试逻辑需要自己实现（检测到冲突后重新读取 → 重新计算 → 再次尝试更新），代码复杂度比悲观锁高。但乐观锁不占用数据库锁资源，在并发量大的系统中扩展性更好。

## Trade-off：事务与锁的取舍

**事务大小**：事务太大（一次操作几千行、持续几秒）会长时间持锁，阻塞其他事务。事务太小（每个操作独立一个事务）又无法保证一致性。实际建议：一个事务只包含逻辑上必须原子的操作，不要在事务里做网络请求、文件读写等慢操作。

**隔离级别**：REPEATABLE READ 是安全的选择，适合绝大多数场景。READ COMMITTED 在某些高并发场景下性能稍好（锁的持有时间更短），但需要你自行处理不可重复读的问题。不要为了性能降级到 READ UNCOMMITTED。

**悲观锁 vs 乐观锁**：写冲突多用悲观锁（数据库层面直接排队），写冲突少用乐观锁（应用层重试）。如果不确定，先从乐观锁开始（实现简单，不加数据库锁），遇到性能问题再评估是否需要切换。

**避免大事务中的用户交互**：事务开启后不要等用户输入。事务应该在代码里自动完成，从 BEGIN 到 COMMIT 尽可能在几十毫秒内结束。如果需要用户确认，先在事务外完成用户交互，准备好所有数据后，再开启事务一次性提交。
