# MySQL 渐进式学习计划

## 背景

你是一个后端开发者，项目要存数据了。领导说用 MySQL，你打开终端敲了几个 SELECT，能查数据了。但真要设计表结构、优化慢查询、处理并发……才发现自己只会皮毛。网上的教程要么太浅（教你 CRUD 就完了），要么太厚（1000 页的书看着就困）。

这个系列就是为你准备的。我们不会把 MySQL 手册翻译一遍，而是从一个**电商订单系统**的真实场景出发，每解决一个数据库问题，你就自然掌握了一个 MySQL 特性。

### 技术栈选择

| 技术    | 版本 | 为什么选它                                                     |
| ------- | ---- | -------------------------------------------------------------- |
| MySQL   | 8.0  | 全球使用最广的开源关系型数据库，CTE、窗口函数、JSON 等特性齐全 |
| Docker  | -    | 一行命令启动数据库，开发环境和生产环境一致                     |
| DBeaver | -    | 免费开源的数据库管理工具，跨平台，比 MySQL Workbench 更轻量    |

> 如果你对 PostgreSQL 更感兴趣，它在 JSON 支持、扩展性方面更强。本系列选择 MySQL，是因为它在国内企业中使用更广泛，面试和工作中遇到的机会更多。MySQL 的 SQL 语法和核心概念（索引、事务、锁）与其他关系型数据库通用，学会了可以平滑迁移。

### 01. MySQL 快速入门

**核心问题**：项目要用 MySQL，但你只会 SELECT \* FROM table。Docker 启动一个实例、建库建表、跑几个查询——最快多久能上手？

**主线案例**：用 Docker 启动 MySQL，创建一个简单的任务管理数据库，完成基本的增删改查，理解 MySQL 的基本概念。

**内容要点**：

- MySQL vs PostgreSQL vs MongoDB：选型的关键考量
- Docker 一行命令启动 MySQL
- 数据库和表的基本操作：CREATE / SHOW / DROP
- SQL 语言分类：DDL、DML、DQL、DCL 各管什么
- 第一个查询：INSERT、SELECT、UPDATE、DELETE
- 客户端工具：DBeaver / MySQL Shell 的使用
- 命名规范和注释：写别人能看懂的 SQL

**与前篇关联**：系列第一篇，无前篇。

### 02. 表设计与数据类型

**核心问题**：你设计了第一张表，用 VARCHAR 存金额、用 INT 存手机号、没有设主键。上线后数据一多就出问题——金额精度丢了、手机号存不下、查询慢到离谱。表设计到底该怎么搞？

**主线案例**：为一个内容发布平台设计用户表——涉及字符串、数字、日期、JSON、枚举等多种数据类型，展示选对数据类型的重要性。

**内容要点**：

- 整数类型：TINYINT 到 BIGINT，选哪个合适（用够就行，别浪费）
- 浮点与精确：为什么金额必须用 DECIMAL 而不是 FLOAT
- 字符串类型：CHAR vs VARCHAR vs TEXT，长度怎么选
- 日期时间：DATE、DATETIME、TIMESTAMP 的区别和坑
- JSON 类型：MySQL 8.0 的 JSON 支持能做什么
- 枚举与集合：ENUM 的便利和隐患
- 主键设计：自增 ID vs UUID，各适合什么场景
- NULL 值：为什么 NOT NULL 通常是更好的选择
- 字符集：为什么一定要用 utf8mb4

**与前篇关联**：上一篇用了最简单的 CREATE TABLE，这一篇深入每种数据类型的选型。

### 03. 增删改查实战

**核心问题**：CRUD 谁不会写？但"批量插入时遇到重复怎么处理"、"UPDATE 忘了写 WHERE 删了全表"、"分页查询在数据量大时越来越慢"——这些坑你踩过吗？

**主线案例**：实现一个文章管理系统的完整 CRUD——批量导入文章、多条件筛选文章列表、分页、软删除。

**内容要点**：

- INSERT 的高阶用法：批量插入、INSERT ON DUPLICATE KEY UPDATE、INSERT SELECT
- SELECT 的过滤技巧：比较运算符、LIKE 模糊匹配、IN / BETWEEN
- WHERE 条件组合：AND / OR 的优先级，什么时候必须加括号
- ORDER BY 排序：多字段排序、NULL 值排序
- LIMIT 分页：OFFSET 在大数据量时的问题
- UPDATE 的陷阱：忘了 WHERE 会怎样、批量更新的安全写法
- DELETE vs TRUNCATE：两种删除方式的区别
- CASE 表达式：在查询中做条件判断

**与前篇关联**：上一篇设计好了表结构，这一篇对表进行完整的增删改查操作。

## 核心篇：电商系统案例（04-07）

目标：引入电商系统作为贯穿案例，深入多表查询、聚合统计、索引设计和事务处理。从这一篇开始，所有文章都在构建同一个电商系统的数据库。

### 04. 高级查询与 JOIN

**核心问题**：你的电商系统有用户表、商品表、订单表、订单明细表……要查"最近 30 天每个用户的消费总额"，一条 SELECT 搞不定。多表关联怎么写？LEFT JOIN 和 INNER JOIN 什么时候用哪个？

**主线案例**：为电商系统实现多表关联查询——订单详情页（订单+用户+商品+物流）、用户消费记录、未下单的用户列表。

**内容要点**：

- JOIN 的本质：为什么多表查询需要 JOIN
- INNER JOIN：只返回两表都有的数据
- LEFT JOIN：保留左表所有数据，右表没有就填 NULL
- RIGHT JOIN 和 CROSS JOIN：什么时候会用到
- 自连接：用一张表和自己做关联（商品分类树）
- 多表 JOIN：三张、四张表关联的写法和注意事项
- UNION 与 UNION ALL：合并查询结果的区别
- NULL 值处理：COALESCE、IFNULL、NULLIF
- JOIN 的性能考量：关联表太多会怎样

**与前篇关联**：上一篇的单表 CRUD 搞定了，这一篇处理多表关联查询。

### 05. 聚合、窗口函数与子查询

**核心问题**：运营要一份报表——"月度销售趋势、每个品类的销售额排名、用户累计消费金额"。COUNT + GROUP BY 能搞定前两个，但"累计消费"这种需要"看前面的行"的需求怎么实现？

**主线案例**：为电商系统实现运营报表——日/月销售趋势、品类销售排名、用户消费排行榜、连续下单用户识别。

**内容要点**：

- 聚合函数：COUNT、SUM、AVG、MAX、MIN
- GROUP BY 分组：分组后只能查聚合列和分组列
- HAVING 过滤分组：HAVING 和 WHERE 的区别
- 窗口函数（MySQL 8.0）：ROW_NUMBER、RANK、DENSE_RANK
- LAG / LEAD：访问前一行/后一行的数据
- SUM OVER：计算累计值、移动平均
- CTE（公用表表达式）：WITH 子句让复杂查询更可读
- 递归 CTE：树形结构的查询（商品分类、组织架构）
- 子查询 vs JOIN vs CTE：什么时候该用哪个

**与前篇关联**：上一篇用 JOIN 关联了多张表，这一篇在关联结果上做聚合和统计。

### 06. 索引：从原理到实战

**核心问题**：电商系统上线后，订单表有了百万级数据，"按用户 ID 查订单"要 3 秒。你听人说加索引能解决，但加了一堆索引后写入变慢了。索引到底该怎么加？

**主线案例**：诊断和优化电商系统的查询性能——用 EXPLAIN 分析慢查询，设计合理的索引策略，避免索引的常见陷阱。

**内容要点**：

- 为什么需要索引：从全表扫描到 B+ 树查找
- B+ 树索引的原理：为什么 MySQL 选择 B+ 树
- 索引类型：主键索引、唯一索引、普通索引、全文索引
- 复合索引与最左前缀原则：索引列的顺序怎么选
- 覆盖索引：不回表就能拿到结果
- EXPLAIN 执行计划：type、key、rows、Extra 各代表什么
- 索引失效的常见场景：函数运算、类型转换、LIKE '%xx'、OR 条件
- 前缀索引：长文本字段怎么加索引
- 索引不是越多越好：写入代价和维护成本

**与前篇关联**：上一篇的聚合查询在数据量大时会变慢，这一篇用索引解决性能问题。

### 07. 事务与锁机制

**核心问题**：用户下单时，要扣库存、创建订单、记录支付——这三步必须同时成功或同时失败。中间网络断了怎么办？两个人同时买最后一件商品怎么办？

**主线案例**：电商系统中的事务场景——下单流程（扣库存+创建订单+记录流水）、库存超卖问题、退款流程。

**内容要点**：

- 事务的 ACID 特性：为什么需要原子性、一致性、隔离性、持久性
- 事务控制语句：BEGIN、COMMIT、ROLLBACK、SAVEPOINT
- 四种隔离级别：READ UNCOMMITTED 到 SERIALIZABLE，MySQL 默认用哪个
- 脏读、不可重复读、幻读：每种隔离级别解决了什么问题
- 锁的类型：共享锁、排他锁、意向锁
- 行锁、间隙锁、临键锁：InnoDB 的锁机制
- 死锁：怎么产生、怎么排查、怎么避免
- 乐观锁 vs 悲观锁：SELECT FOR UPDATE 和版本号方案

**与前篇关联**：上一篇的查询优化解决了读性能，这一篇处理并发写入的数据一致性问题。

### 08. 查询优化实战

**核心问题**：你给电商系统的订单列表加了索引，但还是慢。EXPLAIN 看了半天，发现走了索引但还是扫描了几万行。查询优化不只是加索引这么简单。

**主线案例**：诊断电商系统的真实慢查询——订单列表的分页优化、商品搜索的全文索引、统计报表的物化视图方案。

**内容要点**：

- EXPLAIN 深入解读：每个字段的含义和优化方向
- 慢查询日志：怎么开启、怎么分析
- 分页优化：LIMIT OFFSET 在深分页时的问题和替代方案
- 全文索引：LIKE '%关键词%' 的性能杀手和解决方案
- 子查询优化：什么时候改用 JOIN、什么时候用 EXISTS
- 避免 SELECT \*：只查需要的列
- 批量操作优化：一条一条 INSERT vs 批量 INSERT
- 分区表：按时间分区的大表查询优化

**与前篇关联**：上一篇理解了索引原理，这一篇把索引和查询优化结合起来解决实际问题。

### 09. 数据库设计原则

**核心问题**：你的电商系统跑了半年，订单表加了十几个字段，商品表和 SKU 表分不清，每次改需求都要改表结构。当初的设计怎么就扛不住了？

**主线案例**：重新设计电商系统的数据库——从需求分析到 ER 图，从三范式到适度的反范式，处理真实业务中的设计决策。

**内容要点**：

- 数据库设计三范式：1NF、2NF、3NF 分别解决了什么问题
- 范式与性能的权衡：为什么生产环境经常需要适度反范式
- ER 图设计：实体、属性、关系的建模方法
- 一对一、一对多、多对多：关系怎么用外键表达
- 外键约束：要不要用外键？生产环境的两种观点
- 商品模型设计：SPU 和 SKU 的拆分策略
- 订单状态机：用数据库还是用代码管理状态
- 大表拆分：什么时候该考虑垂直拆分和水平拆分

**与前篇关联**：前面几篇在已有表结构上做查询和优化，这一篇回到设计层面，思考怎么从一开始就设计好。

### 10. 用户管理与安全

**核心问题**：你的数据库用的是 root 账号，密码是 123456，所有服务共用一个数据库账号。安全团队审计的时候差点把你开了。MySQL 的权限管理怎么做才规范？

**主线案例**：为电商系统设计数据库账号体系——应用账号（读写分离）、只读账号（报表）、运维账号（DBA）、审计日志。

**内容要点**：

- 用户管理：CREATE USER、ALTER USER、DROP USER
- 权限管理：GRANT / REVOKE，全局权限 vs 数据库权限 vs 表权限
- 角色（MySQL 8.0）：用角色简化权限管理
- 密码策略：密码复杂度、过期时间、登录失败锁定
- 连接安全：限制访问 IP、SSL 连接
- SQL 注入防护：参数化查询、最小权限原则
- 审计：MySQL Enterprise Audit、开源替代方案
- 安全最佳实践清单

**与前篇关联**：上一篇设计好了数据库结构，这一篇确保只有对的人才能访问对的数据。

### 11. 备份恢复与主从复制

**核心问题**：运维不小心执行了 DROP TABLE orders，线上数据全没了。没有备份，只能跑路。怎么设计一个靠谱的备份策略？数据量大了之后，单机扛不住，怎么用主从复制分担读压力？

**主线案例**：为电商系统搭建数据安全保障——自动备份脚本、基于 binlog 的 point-in-time 恢复、主从复制架构、读写分离方案。

**内容要点**：

- 备份类型：逻辑备份（mysqldump）vs 物理备份（XtraBackup）
- mysqldump 的使用：全库备份、单表备份、压缩备份、恢复
- binlog（二进制日志）：开启、格式选择、基于 binlog 的数据恢复
- 备份策略设计：全量 + 增量、保留周期、异地备份
- 主从复制原理：binlog → relay log → SQL thread
- GTID 复制：为什么推荐用 GTID 而不是 binlog position
- 读写分离：应用层实现 vs 中间件（ProxySQL）
- 高可用方案：主从切换、MHA、InnoDB Cluster

**与前篇关联**：上一篇确保了数据安全，这一篇确保数据不丢、服务不断。

### 12. 性能调优综合实战

**核心问题**：电商大促活动来了，QPS 从 100 涨到 10000。数据库 CPU 100%、连接数打满、查询超时。怎么从配置、SQL、架构三个层面综合调优？

**主线案例**：对电商系统进行全面性能诊断和调优——慢查询治理、连接池配置、InnoDB 参数调优、监控告警搭建。

**内容要点**：

- 性能分析工具：Performance Schema、SHOW PROFILE、Slow Query Log
- InnoDB 核心参数：buffer_pool_size、log_file_size、flush_log_at_trx_commit
- 连接管理：max_connections、连接池、连接泄漏排查
- 表结构优化：字段类型选择、大字段分离、分区表
- SQL 优化技巧汇总：前面的知识在这里综合运用
- 监控指标：QPS/TPS、连接数、缓冲池命中率、慢查询数量
- 监控工具：Prometheus + Grafana、Percona PMM
- 系列回顾：技术选型总结、最佳实践清单、进阶方向

**与前篇关联**：前面 11 篇掌握了 MySQL 的各个方面，这一篇综合运用，应对真实生产场景。
