# MySQL 渐进式学习计划（前端开发者版）

## 背景

你是一个前端开发者。平时写 React/Vue，调后端接口，日子过得不错。但总有些时刻让你不踏实——

联调时后端说"数据没问题，你自己查一下数据库"，你连怎么连数据库都不知道。想做个全栈 side project，数据库设计全靠猜，字段类型随便选，上线后查一条数据要 5 秒。看后端同事讨论"索引走了没""事务回滚了"，你完全插不上话。

这个系列就是为你准备的。不讲 DBA 那些备份恢复、主从复制、性能调优的东西——那些交给运维。我们只讲**前端开发者真正用得上的 MySQL 知识**：能自己查数据、能设计表结构、能看懂慢查询、能和后端顺畅沟通。

### 技术栈选择

### 01. MySQL 快速上手

**核心问题**：你想做一个全栈项目，后端选了 MySQL，但以前从来没碰过数据库。Docker 启动一个实例、建库建表、跑几个查询——最快多久能上手？

**主线案例**：用 Docker 启动 MySQL，为个人博客系统创建数据库和第一张表（用户表），完成基本的连接和查询。

**内容要点**：

- Docker 一行命令启动 MySQL（不用装，不污染系统）
- 数据库基本操作：CREATE / SHOW / USE / DROP
- SQL 语言分类速览：DDL、DML、DQL、DCL 各管什么（30 秒搞懂就行）
- 第一个查询：INSERT 一条数据、SELECT 查出来
- 客户端工具：DBeaver 的安装和基本使用
- 命名规范：写别人能看懂的 SQL（下划线命名、别用保留字）

**砍掉的内容**（前端用不到）：

- MySQL vs PostgreSQL vs MongoDB 的详细对比（选型是后端的事）
- MySQL Shell 的使用（DBeaver 够用了）
- DCL 的详细讲解（权限管理后面用不到）

**与前篇关联**：系列第一篇，无前篇。

### 02. 表设计与数据类型

**核心问题**：你给博客设计了一张文章表，用 VARCHAR 存发布时间、用 TEXT 存标题、忘了加主键。跑起来发现查询结果排序乱的、标题没法建索引、重复数据一堆。表设计到底该怎么搞？

**主线案例**：为博客系统设计核心表——用户表、文章表、标签表、评论表，涉及前端项目中最常见的数据类型选型。

**内容要点**：

- 整数类型：INT 和 BIGINT 就够了，不用记 TINYINT 到 BIGINT 全家桶
- 为什么金额必须用 DECIMAL（FLOAT 会丢精度，你猜怎么发现的）
- 字符串类型：VARCHAR vs TEXT，日常用哪个
- 日期时间：DATETIME 和 TIMESTAMP 怎么选（时区坑）
- JSON 类型：MySQL 8.0 的 JSON 支持，存灵活结构的好帮手
- 主键设计：自增 ID vs UUID，前端项目用哪个更方便
- NOT NULL：为什么字段尽量设为 NOT NULL
- 字符集：永远用 utf8mb4（支持 emoji，不需要理由）

**砍掉的内容**：

- CHAR vs VARCHAR 的详细对比（前端项目几乎用不到 CHAR）
- ENUM 类型的讲解（坑太多，不如用 VARCHAR 或 JSON）
- BLOB 类型（前端用不到）

**与前篇关联**：上一篇建了数据库和第一张表，这篇深入每种数据类型的选型，把博客系统的核心表设计完。

### 03. 增删改查实战

**核心问题**：CRUD 谁不会写？但"批量导入文章时遇到重复标题怎么处理"、"UPDATE 忘了写 WHERE 更新了全表"、"分页查询数据一多就巨慢"——这些坑你踩过吗？

**主线案例**：实现博客系统的完整 CRUD——批量导入文章、多条件筛选文章列表、分页查询、软删除。

**内容要点**：

- INSERT 的高阶用法：批量插入、INSERT ... ON DUPLICATE KEY UPDATE（导入文章时遇到重复标题怎么办）
- SELECT 的过滤技巧：WHERE 条件组合、LIKE 模糊搜索、IN / BETWEEN
- ORDER BY 排序：按发布时间倒序、多字段排序
- LIMIT 分页：基本分页写法（深分页的问题留到索引篇讲）
- UPDATE 的陷阱：永远先写 WHERE 再写 SET
- 软删除 vs 硬删除：文章删了要不要真的删
- CASE 表达式：在查询里做条件判断

**砍掉的内容**：

- DELETE vs TRUNCATE 的详细对比（前端项目几乎不需要 TRUNCATE）
- INSERT SELECT（不常用）
- 子查询（放到 JOIN 篇讲）

**与前篇关联**：上一篇设计好了表结构，这一篇对表进行完整的增删改查操作。

### 04. 多表查询与 JOIN

**核心问题**：你的博客有文章表、标签表、评论表、用户表……要查"某篇文章的所有评论及评论者信息"，一条 SELECT 搞不定。多表关联怎么写？LEFT JOIN 和 INNER JOIN 什么时候用哪个？

**主线案例**：为博客系统实现多表关联查询——文章详情页（文章+标签+评论+评论者）、某用户的所有文章及评论数、没有评论的文章列表。

**内容要点**：

- 为什么需要 JOIN：从两个独立的表到关联查询
- INNER JOIN：两边都有的才返回（有评论的文章）
- LEFT JOIN：左边的全保留（所有文章，包括没评论的）
- 自连接：用一张表和自己关联（评论的回复关系）
- 多表 JOIN：三张、四张表关联的写法
- UNION：合并多个查询结果
- NULL 值处理：COALESCE、IFNULL
- JOIN 性能直觉：关联表不是越多越好

**砍掉的内容**：

- RIGHT JOIN（换成 LEFT JOIN 就行）
- CROSS JOIN（前端项目几乎用不到）
- USING 关键字（用 ON 更明确）

**与前篇关联**：上一篇搞定了单表 CRUD，这一篇处理多表关联查询——真正开始有"数据库"的感觉。

### 05. 聚合与窗口函数

**核心问题**：你想给博客加一个仪表盘——"每月发布了多少篇文章"、"每个标签下有多少文章"、"阅读量 TOP 10 的文章"、"用户连续发文天数"。COUNT + GROUP BY 能搞定前两个，但"连续发文天数"这种需要"看前面的行"的需求怎么实现？

**主线案例**：为博客系统实现数据仪表盘——月度发文趋势、标签热度排名、文章阅读排行榜、用户活跃度统计。

**内容要点**：

- 聚合函数：COUNT、SUM、AVG、MAX、MIN（日常够用）
- GROUP BY 分组：按标签分组统计文章数
- HAVING：过滤分组结果（HAVING 和 WHERE 的区别）
- 窗口函数（MySQL 8.0）：ROW_NUMBER、RANK、DENSE_RANK（排名）
- LAG / LEAD：访问前一行的数据（计算日环比增长）
- SUM OVER：累计值（累计发文数）
- CTE（WITH 子句）：让复杂查询更可读

**砍掉的内容**：

- 递归 CTE（树形结构查询，前端项目很少用）
- 子查询 vs JOIN vs CTE 的详细对比（太学术）
- 所有窗口函数的完整列表（只讲最常用的几个）

**与前篇关联**：上一篇用 JOIN 关联了多张表，这一篇在关联结果上做聚合和统计，给博客加上数据看板功能。

### 06. 索引：让你的查询快起来

**核心问题**：博客文章过千篇后，按标题搜索要 2 秒，按标签筛选要 1.5 秒。你加了个索引，搜索变快了，但写入文章变慢了。索引到底该怎么加？加多了好还是加少了好？

**主线案例**：诊断博客系统的查询性能——用 EXPLAIN 分析慢查询，为常用查询场景设计合理的索引。

**内容要点**：

- 为什么需要索引：从全表扫描到索引查找（类比书的目录）
- 索引类型：主键索引、唯一索引、普通索引（只讲这三种）
- 复合索引与最左前缀原则：索引列的顺序怎么选
- EXPLAIN 执行计划：重点看 type、key、rows、Extra 这几个字段
- 索引失效的常见场景：LIKE '%关键词'、函数运算、类型转换
- 索引不是越多越好：写入代价的理解

**砍掉的内容**：

- B+ 树索引的底层原理（前端不需要理解数据结构细节）
- 覆盖索引的深入讲解（知道概念就行）
- 前缀索引（长文本场景较少）
- 全文索引（用 Elasticsearch 更好）

**与前篇关联**：上一篇的聚合查询在数据量大时会变慢，这一篇用索引解决性能问题。

### 07. 事务基础

**核心问题**：用户发布文章时，要往文章表插一条数据、往文章-标签关联表插多条数据、更新标签的文章计数——这三步必须同时成功。中间第二步失败了怎么办？文章插进去了但标签没关联上，数据就乱了。

**主线案例**：博客系统中的事务场景——发布文章（文章+标签关联+计数更新）、批量删除文章及其关联数据、用户注册（检查重名+插入用户+初始化配置）。

**内容要点**：

- 事务解决什么问题：要么全成功，要么全回滚
- 事务控制语句：BEGIN、COMMIT、ROLLBACK
- 四种隔离级别：只需要知道 MySQL 默认 REPEATABLE READ，能防止大部分问题
- 脏读和幻读：用博客场景理解（两个人同时编辑同一篇文章）
- 乐观锁：用版本号处理并发编辑（适合前端场景）

**砍掉的内容**：

- 锁的详细分类（共享锁、排他锁、意向锁、间隙锁、临键锁——DBA 才需要知道）
- 死锁的排查和处理（前端项目并发量不大）
- 悲观锁 SELECT FOR UPDATE（前端项目几乎用不到）
- SAVEPOINT（不常用）

**与前篇关联**：上一篇的查询优化解决了读性能，这一篇处理多步写入的数据一致性问题。

### 08. 数据库设计实践

**核心问题**：你的博客系统跑了半年，文章表加了十几个字段，标签和分类傻傻分不清，评论嵌套用 JSON 存的查不出来。当初怎么就设计成这样了？如果重来一次，应该怎么设计？

**主线案例**：回顾和重构博客系统的数据库设计——从第一篇的简单设计到现在的完整设计，思考每一步为什么要这样做。

**内容要点**：

- 一对一、一对多、多对多：博客里的每种关系怎么设计（用户-文章是一对多，文章-标签是多对多）
- 主键和外键：要不要用外键约束（前端项目的建议）
- 范式与反范式：什么时候该冗余字段（文章表里冗余评论数，避免每次 COUNT）
- 软删除策略：deleted_at 字段的设计
- JSON 字段的使用时机：什么时候用 JSON、什么时候该拆表
- 时间戳设计：created_at 和 updated_at 的自动化
- 从原型到表结构：拿到设计稿后怎么推导出表结构

**砍掉的内容**：

- 数据库设计三范式的学术讲解（太理论）
- ER 图的详细画法（工具自动生成）
- SPU/SKU 电商模型（不适用）
- 分库分表、垂直拆分、水平拆分（前端项目规模用不到）

**与前篇关联**：前面 7 篇在已有表结构上操作，这一篇回到设计层面，思考怎么从一开始就设计好。同时是对整个系列的回顾——从第 1 篇的简单设计到现在的完整设计，看清楚每一步的演进。
