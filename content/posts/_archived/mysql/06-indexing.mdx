你的博客文章过了 500 篇。某天打开文章列表页，加载要 1.5 秒。你打开浏览器 DevTools 一看，API 响应慢。后端同事（或者你自己）一看 SQL：

```sql
SELECT id, title, summary, created_at
FROM articles
WHERE status = 'published' AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 10;
```

500 条数据而已，为什么要这么久？答案是没有索引。数据库在找数据时，只能从第一行开始逐行扫描——500 行还行，5 万行就受不了了。

索引就是解决这个问题。理解索引，你就掌握了 SQL 性能优化最关键的一环。

## 索引是什么

你手边有一本书，想找"事务"这个章节。两种方式：

1. 从第一页翻到最后一页，逐页看标题——**全表扫描**
2. 翻到书的目录，找到"事务"在第几页，直接翻过去——**索引查找**

索引就是数据库的"目录"。它对某个字段（或几个字段）的值建了一个排序好的数据结构，让查询不用扫描全部数据就能定位到目标。

## 怎么加索引

MySQL 有几种索引类型，前端项目只需要关注这三种：

### 主键索引

你建表时写的 `PRIMARY KEY` 就是主键索引。每张表只有一个主键，它自动就是索引，不需要额外操作。

### 唯一索引

UNIQUE 约束的字段自动创建唯一索引。比如文章的 `slug` 字段：

```sql
-- 建表时已经用 UNIQUE 定义了，自动有唯一索引
slug VARCHAR(200) NOT NULL UNIQUE
```

唯一索引保证值不重复，同时加速查询。

### 普通索引

大部分时候你要加的就是普通索引。用 CREATE INDEX 添加：

```sql
-- 给文章的 status 字段加索引
CREATE INDEX idx_status ON articles(status);

-- 给文章的 created_at 字段加索引（用于排序）
CREATE INDEX idx_created_at ON articles(created_at);

-- 给评论的 article_id 加索引（用于关联查询）
CREATE INDEX idx_article_id ON comments(article_id);
```

索引名用 `idx_` 开头加字段名，这是 SQL 的命名惯例。

## 复合索引：一个索引覆盖多个字段

上面的慢查询有两个条件：`status = 'published'` 和 `ORDER BY created_at DESC`。分别给 status 和 created_at 加索引，不如建一个复合索引：

```sql
CREATE INDEX idx_status_created ON articles(status, created_at);
```

复合索引遵循**最左前缀原则**——索引 `(status, created_at)` 可以用于：

- `WHERE status = 'published'`——只用到索引的第一列
- `WHERE status = 'published' ORDER BY created_at DESC`——两列都用到了
- `ORDER BY created_at WHERE status = 'published'`——两列都用到了

但**不能用于**：

- `WHERE created_at > '2025-03-01'`——跳过了第一列 status，索引用不上
- `ORDER BY created_at`——同样跳过了 status

**列的顺序很重要**：把最常用于条件过滤的列放前面，用于排序的放后面。

## EXPLAIN：看 MySQL 怎么执行你的查询

加了索引，但查询还是慢？用 EXPLAIN 看看 MySQL 实际是怎么执行的：

```sql
EXPLAIN SELECT id, title, summary, created_at
FROM articles
WHERE status = 'published' AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 10;
```

MySQL 会返回一张执行计划表，重点关注这几个字段：

| 字段  | 看什么                                                  |
| ----- | ------------------------------------------------------- |
| type  | 访问类型，从好到差：const > eq_ref > ref > range > ALL  |
| key   | 实际用了哪个索引，NULL 表示没走索引                     |
| rows  | 预估扫描的行数，越小越好                                |
| Extra | 额外信息，出现 Using filesort 或 Using temporary 要注意 |

最关键的判断：

- **type 是 ALL**：全表扫描，最差的情况，说明索引没生效
- **type 是 ref**：走了索引，正常情况
- **key 是 NULL**：没有使用任何索引
- **Extra 里有 Using filesort**：额外的排序操作，说明排序没有用上索引

```sql
-- 没加索引时的 EXPLAIN 结果
-- type: ALL, key: NULL, rows: 500
-- → 全表扫描，扫描了 500 行

-- 加了 idx_status_created 索引后
-- type: ref, key: idx_status_created, rows: 50
-- → 走了索引，只扫描了 50 行
```

`rows` 不一定精确，是 MySQL 的估算值，但够用来判断方向了。

## 索引失效的常见场景

索引加了但查询还是慢？可能是索引"失效"了——MySQL 放弃了索引，改用全表扫描。几种常见原因：

### LIKE '%关键词'

```sql
-- 索引失效：以通配符开头
SELECT * FROM articles WHERE title LIKE '%Next.js%';

-- 索引可用：不以通配符开头
SELECT * FROM articles WHERE title LIKE 'Next.js%';
```

LIKE 以 `%` 开头时，MySQL 没法利用索引的排序特性，只能逐行匹配。`LIKE 'Next%'` 不以 `%` 开头，可以用索引。

对于搜索功能，几百篇文章用 `LIKE '%关键词%'` 问题不大。文章多了就该考虑全文索引或 Elasticsearch。

### 在索引列上用函数

```sql
-- 索引失效：对 created_at 用了函数
SELECT * FROM articles
WHERE DATE_FORMAT(created_at, '%Y-%m') = '2025-03';

-- 索引可用：改为范围查询
SELECT * FROM articles
WHERE created_at >= '2025-03-01' AND created_at < '2025-04-01';
```

在索引列上使用函数（DATE_FORMAT、YEAR、MONTH 等），MySQL 要对每行计算函数值，没法用索引。改用范围查询可以走索引。

### 隐式类型转换

```sql
-- 索引失效：status 是 ENUM，和字符串比较时可能隐式转换
SELECT * FROM articles WHERE status = 1;

-- 索引可用：类型一致
SELECT * FROM articles WHERE status = 'published';
```

字段类型和查询值的类型要一致。ENUM 类型和字符串比较要用引号。

### OR 条件

```sql
-- 可能让索引失效：OR 两边的列不都有索引
SELECT * FROM articles WHERE title = 'xxx' OR content = 'yyy';

-- 改写为 UNION ALL，让每个条件单独走索引
SELECT * FROM articles WHERE title = 'xxx'
UNION ALL
SELECT * FROM articles WHERE content = 'yyy';
```

OR 条件要求两边的列都有索引才能都走。如果只有一边有索引，MySQL 可能放弃索引直接全表扫描。

## 索引不是越多越好

你可能想"那我给每个字段都加索引不就好了？"不行。索引有代价：

1. **写入变慢**：每次 INSERT、UPDATE、DELETE 都要同步更新索引。索引越多，写入越慢
2. **占存储空间**：索引本身也要占磁盘空间
3. **优化器困惑**：MySQL 的查询优化器要在多个索引中选一个，索引太多反而可能选错

**加索引的原则**：只为实际会慢的查询加索引。怎么判断？看 EXPLAIN。先有慢查询，再加索引解决。不要提前给所有字段加索引。

## 给博客系统加索引

根据前面的分析，博客系统需要的索引不多：

```sql
-- 文章表：status + created_at 的复合索引（列表页查询）
CREATE INDEX idx_status_created ON articles(status, created_at);

-- 文章表：user_id 索引（查某用户的所有文章）
CREATE INDEX idx_user_id ON articles(user_id);

-- 评论表：article_id 索引（查某文章的所有评论）
CREATE INDEX idx_article_id ON comments(article_id);

-- 评论表：user_id 索引（查某用户的所有评论）
CREATE INDEX idx_user_id ON comments(user_id);

-- 文章标签关联表：tag_id 索引（查某标签下的所有文章）
CREATE INDEX idx_tag_id ON article_tags(tag_id);
```

就这 5 个索引。覆盖了博客系统的主要查询场景。外键关联的字段（article_id、user_id）通常需要加索引，因为 JOIN 查询要用到。

## 查看和删除索引

```sql
-- 查看某张表的所有索引
SHOW INDEX FROM articles;

-- 删除索引
DROP INDEX idx_status_created ON articles;
```

如果你发现某个索引从来没被用到（EXPLAIN 的 key 从来不是它），可以删掉，减少写入开销。

## 索引使用清单

把这篇的要点整理成一张决策表：

| 场景                   | 怎么做                                 |
| ---------------------- | -------------------------------------- |
| 查询慢                 | 先用 EXPLAIN 看原因                    |
| type 是 ALL            | 考虑给 WHERE/ORDER BY 的字段加索引     |
| 多个条件 + 排序        | 用复合索引，过滤字段在前，排序字段在后 |
| LIKE '%xxx%' 慢        | 没法走普通索引，数据量大考虑全文搜索   |
| 在索引列上用了函数     | 改写查询，避免对列使用函数             |
| 加了很多索引但写入变慢 | 删掉不常用的索引                       |
| 拿不准要不要加索引     | 先加，用 EXPLAIN 验证效果，不好再删    |

现在你能让查询快起来了。但还有一个问题：用户发布一篇文章时，要往 articles 表插一条数据、往 article_tags 表插多条数据、更新 tags 表的 article_count——如果中间某步失败了，数据就不一致了。下一篇来搞定事务，确保多步操作"要么全成功，要么全失败"。
