上一篇我们花了整篇文章讨论数据类型的选型，最终设计出了一张用户表。你可能觉得"不就建个表嘛，至于这么讲究？"——至于。因为表结构定下来之后，你要在上面跑成千上万次增删改查，类型选错了，每次操作都在付出代价。

今天我们在这张表上做真正的事。不是"INSERT 一条、SELECT 一下"的入门操作——而是你在日常开发中真正会遇到的需求：批量导入 1000 篇文章、多条件筛选文章列表、处理重复数据、分页查询在深分页时的优化、用软删除代替物理删除。

主线场景：你的内容平台刚上线，运营同学给了你一个 Excel，里面有 500 篇历史文章要导入系统。导入完成后，还需要一个文章管理后台——按分类筛选、按时间排序、分页展示、支持搜索和软删除。

## INSERT：不只是插入一条那么简单

### 批量插入：快几十倍不是夸张

运营给了 500 篇文章，你打算怎么导入？一条一条 INSERT？

```sql
-- 方式一：循环单条插入（慢）
INSERT INTO articles (title, author_id, category, content, status)
VALUES ('第一篇', 1, '技术', '内容...', 1);

INSERT INTO articles (title, author_id, category, content, status)
VALUES ('第二篇', 1, '技术', '内容...', 1);

-- 500 次 INSERT，每次都要解析 SQL、获取锁、写 binlog、刷盘
```

```sql
-- 方式二：批量插入（快）
INSERT INTO articles (title, author_id, category, content, status) VALUES
('深入理解 MySQL 索引', 1, '技术', '内容...', 1),
('React 18 新特性解析', 2, '前端', '内容...', 1),
('Docker 容器网络详解', 3, '运维', '内容...', 1),
-- ... 一次插入 100-500 条
('Go 并发编程实战', 1, '技术', '内容...', 1);
```

为什么批量插入快这么多？因为每次执行 SQL 语句，MySQL 都要走一遍"解析 SQL → 优化查询 → 获取锁 → 执行 → 写 binlog → 释放锁"的流程。批量插入只走一遍流程，数据是一条一条追加到数据文件里的。在实际测试中，1000 条数据，单条插入可能要 10 秒，批量插入只需要 0.3 秒。

但批量插入也有上限：MySQL 的 `max_allowed_packet` 参数限制了单个 SQL 语句的大小（默认 4MB）。如果你一条 INSERT 塞 10000 条记录，每条内容 1KB，加起来就 10MB，超过了限制会报错。一般建议每次批量插入 500-1000 条。

### 遇到重复数据怎么办

500 篇文章里可能有重复的——运营从不同来源整理的，有些标题重复了。直接插入会报唯一键冲突：

```sql
INSERT INTO articles (title, author_id, category, content, status)
VALUES ('深入理解 MySQL 索引', 1, '技术', '内容...', 1);
-- 第二次执行报错：Duplicate entry '深入理解 MySQL 索引' for key 'idx_title'
```

三种处理方式，适用于不同场景：

```sql
-- 方式一：INSERT IGNORE —— 遇到重复就跳过
INSERT IGNORE INTO articles (title, author_id, category, content, status)
VALUES ('深入理解 MySQL 索引', 1, '技术', '更新后的内容...', 1);
-- 已存在，静默跳过，不报错也不更新

-- 方式二：ON DUPLICATE KEY UPDATE —— 遇到重复就更新
INSERT INTO articles (title, author_id, category, content, status)
VALUES ('深入理解 MySQL 索引', 1, '技术', '更新后的内容...', 1)
ON DUPLICATE KEY UPDATE
  content = VALUES(content),
  updated_at = CURRENT_TIMESTAMP;
-- 已存在，更新内容和更新时间

-- 方式三：REPLACE —— 遇到重复就删掉旧的插入新的
REPLACE INTO articles (title, author_id, category, content, status)
VALUES ('深入理解 MySQL 索引', 1, '技术', '新内容...', 1);
-- 先 DELETE 再 INSERT，自增 ID 会变
```

**导入历史数据时用 INSERT IGNORE**（保留已有的、跳过重复的）。**实时更新数据时用 ON DUPLICATE KEY UPDATE**（有就更新，没有就插入，即 upsert）。**REPLACE 慎用**——它是先删后插，会导致自增 ID 变化，如果有其他表通过 ID 关联这条记录，关联关系就断了。

### 从查询结果插入

有时候你需要从一张表汇总数据到另一张表。比如统计每个作者的文章数：

```sql
-- 先创建统计表
CREATE TABLE author_stats (
  author_id     BIGINT UNSIGNED PRIMARY KEY COMMENT '作者ID',
  article_count INT UNSIGNED DEFAULT 0 COMMENT '文章数',
  latest_publish_time TIMESTAMP NULL COMMENT '最近发布时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 从文章表聚合后直接插入统计表
INSERT INTO author_stats (author_id, article_count, latest_publish_time)
SELECT
  author_id,
  COUNT(*) AS article_count,
  MAX(created_at) AS latest_publish_time
FROM articles
WHERE status = 1
GROUP BY author_id;
```

`INSERT ... SELECT` 的用途不限于同一数据库——你可以跨表、跨库（需要 FEDERATED 引擎或 ETL 工具）做数据迁移和同步。日常开发中最常见的场景就是定期把业务表的统计数据汇总到报表表。

## SELECT：筛选、排序、分页

数据导入完了，现在来写管理后台的查询接口。

### 多条件组合筛选

管理后台的筛选条件往往是"有就加上，没有就忽略"：

```sql
-- 需求：按分类、状态、关键词筛选文章列表
SELECT id, title, category, status, created_at
FROM articles
WHERE category = '技术'              -- 有分类参数时
  AND status = 1                      -- 有状态参数时
  AND title LIKE '%MySQL%'            -- 有关键词时
ORDER BY created_at DESC
LIMIT 20;
```

`LIKE '%MySQL%'` 里的 `%` 是通配符——前面的 `%` 表示"MySQL 前面可以有任意字符"，后面的 `%` 表示"MySQL 后面可以有任意字符"。注意，**以 `%` 开头的 LIKE 无法使用索引**，因为 MySQL 不知道从 B+ 树的哪个位置开始查找。如果你的搜索功能是核心需求，后面讲索引和全文检索时会专门讲怎么做。

多个 `AND` / `OR` 组合时，注意优先级。`AND` 的优先级高于 `OR`，这会导致不符合预期的结果：

```sql
-- 需求：查"技术分类的已发布文章"或"前端分类的所有文章"
SELECT * FROM articles
WHERE category = '技术' AND status = 1
   OR category = '前端';

-- MySQL 理解为：(category = '技术' AND status = 1) OR category = '前端'
-- 符合预期

-- 但如果你写反了：
SELECT * FROM articles
WHERE category = '技术'
   OR category = '前端' AND status = 1;

-- MySQL 理解为：category = '技术' OR (category = '前端' AND status = 1)
-- 技术分类的文章不管什么状态都出来了——不是你想要的
```

**养成习惯：混合使用 AND 和 OR 时，始终用括号明确优先级。**

### 分页：深分页是个隐藏的性能杀手

管理后台的分页查询，前几页通常没问题。但翻到第 1000 页时：

```sql
-- 第 1 页：很快
SELECT id, title, created_at FROM articles
ORDER BY created_at DESC LIMIT 20 OFFSET 0;

-- 第 1000 页：可能很慢
SELECT id, title, created_at FROM articles
ORDER BY created_at DESC LIMIT 20 OFFSET 19980;
```

为什么慢？因为 `OFFSET 19980` 的意思是"跳过前 19980 条记录"。MySQL 不是真的跳过——它需要先扫描这 19980 条记录，然后丢弃它们，只返回后面的 20 条。数据量小的时候感觉不到，数据量一上去，第 N 页的响应时间会随 N 线性增长。

一个简单的优化方案是**游标分页**（也叫 keyset pagination）：

```sql
-- 第 1 页
SELECT id, title, created_at FROM articles
ORDER BY created_at DESC LIMIT 21;  -- 多取 1 条判断是否有下一页

-- 第 2 页：用上一页最后一条记录的时间作为游标
SELECT id, title, created_at FROM articles
WHERE created_at < '2025-03-14 08:30:00'  -- 上一页最后一条的 created_at
ORDER BY created_at DESC LIMIT 21;
```

游标分页利用了索引的有序性——`WHERE created_at < 某个值` 可以直接通过 B+ 树定位到起始位置，不需要扫描和跳过前面的记录。无论翻到第几页，性能都一样。

但游标分页有个限制：不支持跳页。用户不能直接跳到第 50 页，只能"下一页""上一页"地翻。对于管理后台和移动端无限滚动这种场景够用了，如果业务确实需要跳页，OFFSET 分页仍然是必要的——后续讲查询优化时会介绍深分页的更多优化手段。

## UPDATE：一行忘写 WHERE 的代价

文章管理后台有个"批量下架"功能，你写的 SQL 是这样的：

```sql
-- 需求：下架所有"待审核"状态的文章
UPDATE articles SET status = 0 WHERE status = 2;
```

如果你忘了写 `WHERE status = 2`，会怎样？

```sql
UPDATE articles SET status = 0;
-- 所有文章都被下架了。所有。
```

这不是在吓你。这种事故在真实的生产环境中发生过无数次。预防措施有几个：

**1. 先 SELECT 验证，再 UPDATE**

```sql
-- 先看看会影响到哪些记录
SELECT id, title, status FROM articles WHERE status = 2;

-- 确认无误后再更新
UPDATE articles SET status = 0 WHERE status = 2;
```

**2. 加 LIMIT 限制影响范围**

```sql
UPDATE articles SET status = 0 WHERE status = 2 LIMIT 100;
-- 最多影响 100 条，防止误操作波及全表
```

**3. 在事务里操作**

```sql
START TRANSACTION;
UPDATE articles SET status = 0 WHERE status = 2;
-- 检查影响行数
SELECT ROW_COUNT();  -- 返回受影响的行数
-- 确认无误后提交
COMMIT;
-- 发现不对，回滚
-- ROLLBACK;
```

### CASE 表达式：在 UPDATE 里做条件判断

有时候不同条件的更新逻辑不一样——比如批量调整文章状态：已发布的保持不变，待审核的改为草稿，草稿的打上"待完善"标记。一条 UPDATE 搞定：

```sql
UPDATE articles
SET status = CASE
    WHEN status = 2 THEN 0        -- 待审核 → 草稿
    WHEN status = 0 THEN 3        -- 草稿 → 待完善
    ELSE status                    -- 其他不变
END
WHERE status IN (0, 2);
```

`CASE` 表达式在 SELECT 里也很有用——比如把状态码转成中文显示：

```sql
SELECT
  title,
  CASE status
    WHEN 0 THEN '草稿'
    WHEN 1 THEN '已发布'
    WHEN 2 THEN '待审核'
    WHEN 3 THEN '待完善'
    ELSE '未知'
  END AS status_text,
  created_at
FROM articles
ORDER BY created_at DESC LIMIT 10;
```

## DELETE：为什么生产环境很少用物理删除

物理删除（DELETE）是从数据库里永久移除数据。但在大多数生产系统中，你看到的不是 DELETE，而是"软删除"——给表加一个 `deleted_at` 字段，删除时只更新这个字段：

```sql
-- 软删除：不是真的删，而是标记为"已删除"
ALTER TABLE articles ADD COLUMN deleted_at TIMESTAMP NULL COMMENT '软删除时间';

UPDATE articles SET deleted_at = CURRENT_TIMESTAMP WHERE id = 42;

-- 查询时过滤掉已删除的记录
SELECT id, title, status FROM articles WHERE deleted_at IS NULL;
```

为什么偏好软删除？

**数据可恢复**。误删了怎么办？物理删除后只能从备份恢复（如果有备份的话，恢复时间可能以小时计）。软删除只需要把 `deleted_at` 设回 NULL。

**审计需要**。很多业务场景需要知道"谁在什么时候删了什么"。软删除天然保留了这些信息。

**关联数据不受影响**。一篇文章可能有评论、点赞、收藏。物理删除文章后，关联数据要么级联删除（丢失用户数据），要么变成孤儿记录（指向一个不存在的文章 ID）。软删除不产生这个问题。

软删除的代价是查询条件里永远要加 `WHERE deleted_at IS NULL`——容易忘，忘了就会查出已删除的数据。解决方案是在应用层封装（ORM 通常支持全局 scope），或者用视图：

```sql
CREATE VIEW active_articles AS
SELECT * FROM articles WHERE deleted_at IS NULL;

-- 之后查询视图就行
SELECT * FROM active_articles WHERE category = '技术';
```

### TRUNCATE vs DELETE

清空整张表的数据有两个选择：

```sql
DELETE FROM temp_import_log;   -- 逐行删除，可回滚（在事务中），不重置自增 ID
TRUNCATE TABLE temp_import_log; -- 直接清空，不可回滚，重置自增 ID
```

TRUNCATE 的本质是 DROP TABLE + CREATE TABLE，所以快得多，但不走事务。生产环境中，清空临时表、日志表用 TRUNCATE；需要条件删除或可能回滚的场景用 DELETE。

## 一个完整的文章管理场景

把前面的知识串起来，模拟一次完整的内容导入和管理流程：

```sql
-- 1. 批量导入历史文章（遇到重复标题就跳过）
INSERT IGNORE INTO articles (title, author_id, category, summary, status) VALUES
('深入理解 MySQL 索引', 1, '技术', '从 B+ 树原理到索引设计实战', 1),
('React 18 新特性解析', 2, '前端', 'Concurrent 模式和 Suspense 详解', 1),
('Docker 网络模式对比', 3, '运维', 'bridge、host、overlay 三种网络模式', 1),
('Go 并发编程实战', 1, '技术', 'goroutine 和 channel 的正确用法', 1),
('前端性能优化指南', 2, '前端', '从加载到渲染的全链路优化', 1);

-- 2. 管理后台查询：按分类筛选，按时间倒序，分页
SELECT id, title, category, status, created_at
FROM articles
WHERE category = '技术'
  AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- 3. 搜索文章
SELECT id, title, category
FROM articles
WHERE title LIKE '%MySQL%'
  AND deleted_at IS NULL
  AND status = 1;

-- 4. 批量下架待审核文章
UPDATE articles
SET status = 0, updated_at = CURRENT_TIMESTAMP
WHERE status = 2
  AND deleted_at IS NULL;

-- 5. 软删除指定文章
UPDATE articles SET deleted_at = CURRENT_TIMESTAMP WHERE id = 42;

-- 6. 统计各分类文章数
SELECT
  category,
  COUNT(*) AS total,
  SUM(CASE WHEN status = 1 THEN 1 ELSE 0 END) AS published,
  SUM(CASE WHEN status = 0 THEN 1 ELSE 0 END) AS draft
FROM articles
WHERE deleted_at IS NULL
GROUP BY category;
```

注意最后那个统计查询——`SUM(CASE WHEN ... THEN 1 ELSE 0 END)` 是一种常见的"条件计数"技巧，比写多条 WHERE 子查询高效得多。

## Trade-off：CRUD 中的取舍

**批量插入 vs 单条插入**：批量插入快几十倍，但单条 SQL 过大可能超过 `max_allowed_packet`。日常建议每次 500-1000 条。如果单条数据很大（比如含长文本），减少每次的批量数。

**INSERT IGNORE vs ON DUPLICATE KEY UPDATE**：IGNORE 遇到重复就跳过，什么都不改；upsert 遇到重复就更新。导入历史数据用 IGNORE（保留旧数据优先），实时同步用 upsert（保证数据最新）。

**OFFSET 分页 vs 游标分页**：OFFSET 支持跳页但深分页慢，游标分页性能稳定但不支持跳页。管理后台需要跳页时用 OFFSET，移动端和 API 无限滚动时用游标。

**物理删除 vs 软删除**：物理删除节省存储、查询简单；软删除可恢复、可审计、不破坏关联。生产环境几乎一律软删除，只对临时数据和日志表用物理删除。

**SELECT \* vs 指定列**：`SELECT *` 写起来方便，但会传输不需要的列（包括 TEXT 类型的大字段），浪费 I/O 和网络带宽，也无法利用覆盖索引。养成习惯只查需要的列。

前两篇我们分别搞定了"建表"和"增删改查"，但所有操作都局限在一张表内。真实的内容平台不可能只有一张文章表——用户表、分类表、评论表、标签表……查"某个用户的文章列表及其评论数"需要跨多张表关联。下一篇我们来解决多表查询这个核心难题。
