前两篇我们搞定了建表和增删改查，但所有操作都在一张表里。真实系统不可能只有一张表——用户表、商品表、订单表、订单明细表、分类表……查"张三最近买了什么、每件商品叫什么、属于哪个品类"，需要跨四五张表关联。

这就是 JOIN 的用武之地。但 JOIN 不止一种：INNER JOIN、LEFT JOIN、RIGHT JOIN、自连接……用错了不是报错，而是结果"看起来对但其实少了数据"——这种 bug 比报错更难排查。

从这篇开始，我们的主线场景从内容平台切换到一个**电商订单系统**——这个系统会贯穿后续所有文章。先建好表结构：

```sql
CREATE DATABASE shop CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE shop;

CREATE TABLE users (
  id       BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(50) NOT NULL,
  city     VARCHAR(50) DEFAULT '北京'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE categories (
  id   BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL,
  parent_id BIGINT UNSIGNED DEFAULT NULL COMMENT '父分类ID，NULL表示顶级分类'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE products (
  id          BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  category_id BIGINT UNSIGNED NOT NULL,
  name        VARCHAR(200) NOT NULL,
  price       DECIMAL(10, 2) NOT NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE orders (
  id           BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  user_id      BIGINT UNSIGNED NOT NULL,
  total_amount DECIMAL(10, 2) NOT NULL,
  status       TINYINT UNSIGNED DEFAULT 1 COMMENT '1待支付 2已支付 3已发货 4已完成 5已取消',
  created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE order_items (
  id         BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  order_id   BIGINT UNSIGNED NOT NULL,
  product_id BIGINT UNSIGNED NOT NULL,
  quantity   INT UNSIGNED NOT NULL DEFAULT 1,
  price      DECIMAL(10, 2) NOT NULL COMMENT '购买时单价'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

这五张表构成了一个简化版的电商数据模型：用户下单生成订单，订单包含多条明细，每条明细对应一个商品，商品属于某个分类。

## INNER JOIN：两边都有的才匹配

运营问了一个问题："最近 30 天每个用户下了多少单？"——只关心**有订单的用户**，没下过单的不需要。

```sql
SELECT
  u.username,
  COUNT(o.id) AS order_count,
  SUM(o.total_amount) AS total_spent
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.created_at >= DATE_SUB(CURRENT_DATE, INTERVAL 30 DAY)
GROUP BY u.id, u.username;
```

INNER JOIN 的逻辑是：**只返回两张表中能匹配上的行**。如果某个用户没下过单，`users` 表里他的那一行在 `orders` 表里找不到匹配，结果集中就不会出现这个用户。

ON 后面是关联条件——`u.id = o.user_id` 告诉 MySQL "users 表的 id 等于 orders 表的 user_id 时，这两行是关联的"。这个条件等价于 WHERE 里的等值判断，但语义更清晰：ON 说的是"怎么关联"，WHERE 说的是"怎么过滤"。

## LEFT JOIN：保留左表所有数据

运营又问："给我一份完整的用户列表，包括那些从没下过单的。"

INNER JOIN 做不到——它天然排除了没下过单的用户。这时候用 LEFT JOIN：

```sql
SELECT
  u.username,
  u.city,
  COUNT(o.id) AS order_count,
  COALESCE(SUM(o.total_amount), 0) AS total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.username, u.city
ORDER BY order_count ASC;
```

LEFT JOIN 的逻辑是：**左表（FROM 后面的表）的所有行都保留**，右表中能匹配的就关联，匹配不上的字段填 NULL。所以没下过单的用户也会出现在结果里，只是 `order_count` 为 0，`total_spent` 为 0。

注意 `COALESCE(SUM(o.total_amount), 0)`——当一个用户没有订单时，`SUM(NULL)` 的结果是 NULL，COALESCE 把它转成 0。如果不做这个处理，前端显示的金额可能是空白或者"null"。

### LEFT JOIN 的一个经典坑

"查所有从未下单的用户"——你需要的是 LEFT JOIN 后右表为 NULL 的那些行：

```sql
SELECT u.username, u.city
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;
```

`WHERE o.id IS NULL` 过滤出"右表匹配不到"的行，即从未下单的用户。这个模式在实际开发中很常用——查"没有 X 的所有 Y"。

**但如果你把过滤条件写错了位置，结果完全不同：**

```sql
-- 错误写法：把右表条件放在 WHERE 里
SELECT u.username
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 2;
-- 这实际上变成了 INNER JOIN 的效果！
-- 因为 o.status = 2 会过滤掉 o.status 为 NULL 的行（即未下单的用户）
```

规则是：**ON 里放关联条件，WHERE 里放过滤条件**。如果你想在 LEFT JOIN 时过滤右表的数据但不影响左表的完整性，把条件放在 ON 里：

```sql
-- 正确：左表用户全部保留，只关联已支付的订单
SELECT u.username, COUNT(o.id) AS paid_orders
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 2
GROUP BY u.id, u.username;
-- 未下单用户 paid_orders = 0（不是被过滤掉）
```

## 多表 JOIN：关联三四张表

运营要一个"订单详情页"——一个订单里包含用户信息、每件商品的名称和分类、总金额。这需要同时关联四张表：

```sql
SELECT
  o.id AS order_id,
  u.username,
  u.city,
  p.name AS product_name,
  c.name AS category_name,
  oi.quantity,
  oi.price AS unit_price,
  oi.quantity * oi.price AS subtotal
FROM orders o
INNER JOIN users u ON o.user_id = u.id
INNER JOIN order_items oi ON o.id = oi.order_id
INNER JOIN products p ON oi.product_id = p.id
INNER JOIN categories c ON p.category_id = c.id
WHERE o.id = 1001;
```

执行顺序是：从 `orders` 开始，找到用户 → 找到订单明细 → 找到商品 → 找到分类。每次 JOIN 都是在前一次的结果上继续关联。MySQL 的优化器会自动选择最优的关联顺序，不需要你手动调整。

多表 JOIN 的性能考量：**关联的表越多，中间结果集可能越大**。如果订单有 5 条明细，JOIN products 后变成 5 行，JOIN categories 后还是 5 行（因为 product 和 category 是一对一）。但如果某个 JOIN 产生一对多或多多关联，行数会膨胀。实际开发中，超过 5 张表的 JOIN 通常意味着查询设计有问题——可能需要拆分子查询或使用 CTE。

## 自连接：一张表和自己做关联

分类表有个 `parent_id` 字段，形成了一个树形结构——"电子产品"下有"手机"和"电脑"，"手机"下有"苹果"和"安卓"。查"每个子分类的父分类名称"：

```sql
SELECT
  child.name AS category,
  parent.name AS parent_category
FROM categories child
LEFT JOIN categories parent ON child.parent_id = parent.id;
-- 电子产品 | NULL（顶级分类，无父分类）
-- 手机     | 电子产品
-- 电脑     | 电子产品
-- 苹果     | 手机
```

自连接的诀窍是**给同一张表取两个不同的别名**，然后在 ON 里建立父子关系。LEFT JOIN 确保顶级分类（parent_id 为 NULL）也出现在结果里。

## UNION：合并多个查询的结果

运营要一份"用户活动列表"，包含下单记录和退款记录，按时间倒序排列。这两类记录存在不同的表里，但需要合并展示：

```sql
SELECT '下单' AS type, o.id AS ref_id, u.username, o.total_amount AS amount, o.created_at
FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE o.status IN (2, 3, 4)

UNION ALL

SELECT '退款' AS type, r.id AS ref_id, u.username, r.refund_amount AS amount, r.created_at
FROM refunds r
INNER JOIN users u ON r.user_id = u.id
WHERE r.status = 1

ORDER BY created_at DESC
LIMIT 50;
```

`UNION ALL` 把两个查询的结果纵向拼接在一起。`UNION`（不加 ALL）也会拼接，但会**去重**——去掉完全相同的行。如果确定两个查询结果不会重复，用 UNION ALL 更快（省去了去重排序的步骤）。

使用 UNION 的前提是两个查询的列数和列类型必须一致。上面的例子里两个查询都是 5 列，类型也对应：字符串、整数、字符串、小数、时间戳。

## NULL 值处理：COALESCE、IFNULL、NULLIF

多表关联时，NULL 值处理是高频操作。三个函数各有用途：

```sql
-- COALESCE：返回第一个非 NULL 的值
SELECT COALESCE(nickname, username, '匿名') AS display_name FROM users;
-- nickname 有值就用 nickname，否则看 username，都没有就显示"匿名"

-- IFNULL：MySQL 特有，只接受两个参数
SELECT IFNULL(phone, '未填写') AS phone_display FROM users;

-- NULLIF：两个值相等时返回 NULL，否则返回第一个值
SELECT NULLIF(email, '') AS email FROM users;
-- 如果 email 是空字符串，返回 NULL（方便后续 IS NULL 判断）
```

日常用得最多的是 COALESCE，因为它支持多个参数、是 SQL 标准（在 PostgreSQL、Oracle 里也通用）。IFNULL 是 MySQL 特有的简写，功能等价于 `COALESCE(x, y)`。

## Trade-off：JOIN 的选择和限制

**INNER JOIN vs LEFT JOIN**：INNER JOIN 只返回匹配的行，结果集更小、语义更精确。LEFT JOIN 保留所有左表数据，适合"完整列表"的场景。当你不确定该用哪个时，问自己一个问题："左表的数据有没有可能全部出现在结果里？"如果可能，用 LEFT JOIN；如果只需要关联上的，用 INNER JOIN。

**JOIN vs 子查询**：某些场景下子查询和 JOIN 可以互相替代。比如"下单金额超过 500 的用户"：

```sql
-- 用 JOIN
SELECT DISTINCT u.username
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.total_amount > 500;

-- 用子查询
SELECT username FROM users
WHERE id IN (SELECT user_id FROM orders WHERE total_amount > 500);
```

在 MySQL 8.0 中，优化器通常会把这两种写法优化成相同的执行计划。但 JOIN 的可读性通常更好——关联关系一目了然。子查询更适合"存在性判断"（EXISTS）或"取最大/最小的一条"这种场景。

**关联表的数量的上限**：没有硬性限制，但经验上超过 5 张表就要考虑拆分查询。一是性能（中间结果集膨胀），二是可读性（同事看到 7 张表的 JOIN 会崩溃）。
