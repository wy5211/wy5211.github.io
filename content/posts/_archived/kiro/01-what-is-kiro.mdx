我最近半年一直在用 Cursor 写代码。说实话体验不错——对着 Chat 面板说"帮我写个用户注册接口"，它就能生成一段能跑的代码。但时间长了，一个问题越来越明显：**AI 生成的代码越多，我对项目的掌控感就越弱**。

我不知道它改了哪些文件，不知道它为什么选择这种实现方式，也不知道这段代码会不会和昨天写的模块冲突。每次生成都像拆盲盒。

后来我看到 AWS 发布了 Kiro，宣传语写着"Bring engineering rigor to agentic development"——给智能体开发带来工程严谨性。这句话戳中了我。

这个系列就是我在实际项目中使用 Kiro 的完整记录。不是搬运文档，不是功能罗列，而是一个开发者从 Cursor 切到 Kiro 的真实体验。第一篇先回答最基本的问题：Kiro 是什么，它和现有的 AI IDE 到底有什么不一样。

## 先看看 AI IDE 是怎么一步步走到今天的

理解 Kiro 之前，得先看看 AI 编程工具的进化路线。

**第一代：代码补全。** GitHub Copilot 是典型代表。你写个函数名，它帮你补全函数体；你写个注释，它帮你生成代码。本质上是高级版的自动完成——AI 在你光标后面接着写，你决定接不接受。

**第二代：对话式编程。** Cursor 把这个体验往前推了一步。你可以选中一段代码，直接问"这段有没有 bug"；或者对着 Chat 面板说"把这个函数改成异步的"，它直接帮你改。从"补全"变成了"对话"，AI 的角色从被动变成了半主动。

**第三代：智能体开发。** 这一代的核心变化是：AI 不再只是响应你的指令，而是能自己规划任务、读写多个文件、执行命令。Windsurf 的 Cascade、Cursor 的 Agent 模式都属于这一代。

但三代工具都有一个共同的短板：**AI 干活很快，但你缺乏对过程的掌控**。它改了十个文件，你得一个一个看。它做了个架构决策，你可能要到出了 bug 才发现。它生成了五百行代码，你和它之间的理解差距越来越大。

这就是 Kiro 想解决的问题。

## 用同一个需求，对比 Cursor 和 Kiro

光说概念没感觉。我们用一个具体场景来看差异。

假设你正在开发一个习惯追踪 App——叫 HabitFlow。现在需要实现**用户注册 API**：接收邮箱和密码，校验格式，检查邮箱是否已存在，密码加密存储到数据库。

### 用 Cursor 怎么做

打开 Chat 面板，输入需求：

> 帮我写一个用户注册 API，用 Hono 框架，邮箱和密码校验，密码 bcrypt 加密，存到 SQLite。

Cursor 的 Agent 模式会直接开始干活：创建路由文件、写校验逻辑、配置数据库连接。速度快，但过程是黑的——它觉得该怎么做就怎么做。如果项目里已经有类似的模块，它可能不会复用，而是重新生成一套。你得仔细 review 每一步，或者事后修补。

### 用 Kiro 怎么做

在 Kiro 里，同样这个需求，它会先走一个**三阶段流程**：

**第一阶段：需求（Requirements）**。Kiro 先分析你的需求，生成一份 requirements 文档——注册接口需要处理哪些边界情况？邮箱格式校验用什么规则？密码强度有什么要求？重复邮箱返回什么状态码？

**第二阶段：设计（Design）**。基于需求文档，Kiro 生成设计方案——路由结构怎么组织、校验逻辑放在哪一层、错误处理用什么模式、和现有代码怎么衔接。

**第三阶段：任务（Tasks）**。把设计方案拆成具体的编码任务，按依赖关系排序，然后逐个执行。

注意区别：Cursor 是**先写代码再想对不对**，Kiro 是**先想清楚再写代码**。

你可能会说，这不是更慢吗？确实，对于一行代码就能搞定的事，三阶段流程是杀鸡用牛刀。但对于一个涉及多个模块、多种边界条件的功能来说，先理清思路再动手，反而省去了后续反复修改的时间。

这就是 Kiro 的核心定位：**Spec-driven development（规格驱动开发）**。

## Kiro 的核心差异化：四把钥匙

除了 Spec 驱动开发，Kiro 还有几个独特的功能。先混个脸熟，后面文章会挨个深入。

### Specs：规格驱动

上面说的需求→设计→任务三阶段就是 Spec。Kiro 支持两种：

- **Feature Spec**：用于新功能开发，走完整三阶段
- **Bugfix Spec**：用于修 bug，分析原因→设计方案→修复任务

Spec 的本质是给 AI 的输出加了一层"设计评审"。AI 不能上来就写代码，得先把思路摊开给你看，你确认了它才动手。

### Steering：引导系统

你有没有这种经历：每次新开一个 Chat，都要重复告诉 AI"用 TypeScript"、"用 Hono 框架"、"错误处理统一返回这个格式"？Steering 就是解决这个问题的。

你可以在项目里创建 `.kiro/steering/` 目录，把项目规范写成 markdown 文件放进去。Kiro 每次干活前会自动读取这些文件，就像给 AI 装了一个"项目记忆"。

### Hooks：自动化钩子

Hooks 让你在特定事件触发时自动执行操作。比如文件保存时自动跑 lint，创建组件时自动生成测试文件，代码提交前检查是否符合团队规范。

### Powers：能力包

Powers 是可插拔的能力扩展。比如 Supabase Power 让 Kiro 直接操作数据库，AWS Power 让它管理云资源。按需安装，用的时候加载，不用的时候不占资源。

## 和主流 AI IDE 的对比

说了这么多 Kiro 的特点，来一张对比表更直观：

| 维度       | Copilot      | Cursor       | Windsurf   | Kiro                  |
| ---------- | ------------ | ------------ | ---------- | --------------------- |
| 核心能力   | 代码补全     | 对话+Agent   | 连续工作流 | Spec驱动开发          |
| AI 定位    | 辅助工具     | 编码助手     | 编码助手   | 工程协作者            |
| 结构化流程 | 无           | 无           | 无         | 需求→设计→任务        |
| 项目记忆   | 无有限       | .cursorrules | 无         | Steering 文件体系     |
| 扩展机制   | VS Code 插件 | VS Code 插件 | 有限       | Powers + Skills + MCP |
| 底层架构   | VS Code 插件 | VS Code Fork | 自研       | VS Code 架构          |
| 背后公司   | GitHub/MS    | Cursor       | Codeium    | AWS                   |

注意 Kiro 的定位不是"更好的 Cursor"，而是**不同类型的工具**。Cursor 追求快速编码，Kiro 追求可控的工程化开发。这两个目标有时候是矛盾的。

## Kiro 支持什么模型

Kiro 支持多种模型切换：

- **Claude Opus 4.7 / 4.6 / 4.5**：推理能力强，适合复杂的 Spec 设计
- **Claude Sonnet 4.6 / 4.5 / 4.0**：速度和质量的平衡点，日常首选
- **DeepSeek 3.2**：性价比高
- **Qwen3 Coder**：中文场景友好
- **MiniMax 2.1**：AWS 生态内的选择

默认用 Auto 模式，Kiro 会根据任务复杂度自动选择模型。你也可以手动指定。

## 适合谁用

说了这么多，Kiro 到底适合你吗？我分几种情况：

**适合切换到 Kiro 的场景**：

- 你在做一个有一定复杂度的项目（不是写个脚本就跑的那种）
- 你对 AI 生成的代码总是不放心，需要 review 但又觉得 review 成本很高
- 你在团队里工作，需要让 AI 遵循团队规范
- 你用的是 AWS 生态，Kiro 和 AWS 服务有更好的集成

**暂时不需要切换的场景**：

- 你只是偶尔写写小脚本、做做 side project
- 你对 Cursor 的 Agent 模式已经很顺手，没有掌控感的问题
- 你的项目比较简单，AI 生成代码基本不用怎么改
- 你对学习新工具有抵触——Kiro 有自己的概念体系，需要时间适应

## Trade-off 总结

**适用场景**：中大型项目、需要结构化开发流程、团队协作、对 AI 输出质量有要求。

**不适用场景**：快速原型验证、脚本工具开发、项目复杂度低、已经对 Cursor 非常满意且没有痛点。

**对比替代方案**：Cursor 适合快速迭代，追求编码速度；Copilot 适合轻量辅助，不想改变现有工作流；Kiro 适合需要工程严谨性的场景，代价是学习成本更高、单次交互流程更长。

**实践建议**：如果你对"AI 写代码但我不知道它在干嘛"这个问题有共鸣，值得花一个周末试试 Kiro。下一篇我们会从安装配置开始，创建 HabitFlow 项目，用 Kiro 生成第一个 API 接口。
