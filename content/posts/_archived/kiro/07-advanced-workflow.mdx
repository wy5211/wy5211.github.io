六篇文章走下来，HabitFlow 已经从一个空项目变成了功能完整的习惯追踪应用：Spec 驱动开发了打卡功能，Steering 让 AI 记住了项目规范，Chat 和 Agent 搞定了连续打卡统计，Powers 和 MCP 接通了 Supabase 数据库。

但说实话，前面的场景都偏"理想"——需求明确，改动范围清晰，AI 执行也顺利。**真实项目不是这样的**。你会遇到莫名其妙的 bug、慢到离谱的查询、团队成员各自为战写出来的五花八门的代码。

这篇用 HabitFlow 遭遇的一个真实问题，把 Kiro 的能力整合成完整工作流，然后诚实地说说它的边界在哪。

## 从一个 bug 开始：并发打卡导致数据不一致

HabitFlow 上线两周后，有用户反馈：打卡记录里出现了同一天重复打卡的数据。更奇怪的是，连续天数统计时而正确时而错误。

排查这个 bug 的过程，恰好能展示 Kiro 在调试场景下的完整工作流。

### 第一步：用 Vibe 模式定位问题

我没有急着让 AI 改代码，先开了一个 Vibe Session：

```
用户反馈同一天出现重复打卡记录，streak 统计也时而错误。
checkins 表有 id, habit_id, checked_date, created_at 字段。
当前打卡接口逻辑是：先查当天是否已打卡，没有就插入。
会不会是并发请求导致的竞态条件？
```

Kiro 分析了项目代码和数据库 schema，确认了问题根因：**两个请求几乎同时到达，都查到"当天没有打卡记录"，于是都执行了插入操作**。经典的 TOCTOU（Time-of-check to time-of-use）竞态条件。

streak 计算的错误也是连锁反应——重复记录导致日期连续性判断出错。

### 第二步：用 Bugfix Spec 结构化修复

问题定位了，该修了。切到 Spec Session，选择 Bugfix Spec：

```
修复并发打卡导致重复记录的问题：
1. 在 checkins 表添加 (habit_id, checked_date) 的唯一约束
2. 打卡接口使用 INSERT ... ON CONFLICT DO NOTHING
3. 返回值区分"新建打卡"和"今天已打卡"
4. 修复 streak 计算逻辑，先去重再计算连续天数
```

Bugfix Spec 会走"分析→设计→任务"三阶段，但侧重点和 Feature Spec 不同——它先确认 bug 根因，再设计修复方案，最后拆成任务执行。

Kiro 生成的任务列表：

1. 添加数据库唯一约束的 migration
2. 修改打卡 service 使用 upsert 逻辑
3. 修复 streak 计算函数的去重逻辑
4. 补充并发场景的集成测试

在 Autopilot 模式下执行，Kiro 依次完成四个任务，Diagnostics 自动跑了测试，全部通过。整个过程不到 5 分钟。

### 第三步：验证

我手动创建了一个 Checkpoint，然后用 Kiro 的终端模拟并发请求：

```bash
curl -X POST http://localhost:3000/api/checkins -d '{"habit_id": 1}' &
curl -X POST http://localhost:3000/api/checkins -d '{"habit_id": 1}' &
```

两次请求都返回了成功，但数据库只插入了一条记录——唯一约束生效了。streak 计算也恢复了正确。

这个调试流程的要点是：**先 Vibe 定位问题，再 Spec 结构化修复，最后手动验证**。不是所有步骤都需要 AI 参与，知道什么时候自己动手同样重要。

## 性能优化：让 Kiro 帮你找慢查询

并发 bug 修完没多久，又出现了新问题：习惯列表页的加载越来越慢。数据量大了之后，streak 的实时计算成了性能瓶颈。

这次我让 Kiro 直接分析慢查询。在 Chat 里说：

```
Habits 列表接口响应时间超过 2 秒。
每个习惯都要实时计算 streak，目前是 N+1 查询。
帮我分析优化方案。
```

Kiro 做了两件事：一是检查了当前的查询逻辑（确实每个习惯都单独查一次 checkins 表算 streak），二是提出了优化方案——用一条 SQL 窗口函数批量计算所有习惯的 streak，减少数据库往返。

修复后列表接口响应时间从 2 秒降到了 80 毫秒。

但这里有个教训：**Kiro 能帮你优化代码，但不能替你发现性能问题**。你得先知道"慢了"，才能让 AI 去查为什么慢。性能监控和告警还是要靠自己的工具链。

## 团队协作：把个人经验变成团队标准

HabitFlow 如果从个人项目变成团队项目，前面的配置需要做什么调整？

### Steering 文件纳入版本控制

`.kiro/steering/` 下的文件应该提交到 Git 仓库。新成员 clone 项目后，Kiro 自动读取 Steering 文件，立刻知道项目的技术栈、编码规范、目录结构——相当于一份可执行的入职文档。

团队实践中我发现一个有用的做法：在 `tech.md` 里加一节"常见错误"：

```markdown
## Common Pitfalls

- 不要在 service 层直接操作数据库，必须通过 repository 层
- checkins 表有唯一约束，插入时用 upsert 不要用 insert
- streak 计算必须先去重，参考 commit abc123 的修复
```

这些是用血泪换来的经验，写进 Steering 比写在 wiki 里有效得多——因为 AI 每次都会读到。

### Hooks 做团队质量门禁

团队项目的 Hooks 配置也要进 Git。我为 HabitFlow 团队加了一个 Hook：**PR 提交前自动检查 Steering 遵循情况**。虽然不能做完整的代码审查，但能抓住最常见的违规——比如直接 throw 而不是用 Result 类型、路由文件放错了目录。

### Skills 封装团队工作流

第 6 篇讲的 create-table Skill 在团队里价值更大。一个人定义好工作流，所有团队成员创建新表时 Kiro 都按同一套流程走——migration、类型定义、CRUD 路由、测试，一个不少。

## 企业版：当团队规模再大一点

Kiro 有企业版，通过 AWS Console 管理，支持 IAM Identity Center、Okta、Microsoft Entra ID 等身份认证。当前支持 us-east-1 和 eu-central-1 两个区域。

企业版的核心价值不在 IDE 本身，而在**治理**：管理员可以统一推送 Steering 文件到团队成员（前面提到的 MDM 推送），可以监控使用情况，可以设置权限边界。对于有合规要求的公司——金融、医疗——这些是硬需求。

此外还有 VPC Endpoints 支持，确保代码和模型交互不经过公网。CLI 工具（当前版本 2.3.0）可以集成到 CI/CD 流水线里，让 Kiro 的能力在自动化环境中也能用。

但坦白说，如果你是十人以下的小团队，免费版完全够用。企业版解决的是规模化之后的治理问题，不是开发效率问题。

## Kiro 的能力边界

六篇文章写下来，该说说 Kiro **做不到什么**了。

**擅长的事**：结构化的功能开发、有明确规范的代码生成、需要读写多个文件的联动改动、数据库操作和外部服务对接。Spec 驱动这个思路，在需求明确的场景下确实能减少返工。

**不擅长的事**：需要反复试错的探索性工作（调 CSS 样式、调 prompt）、完全开放式的架构设计（AI 的设计品味有限）、涉及运行时状态的调试（内存泄漏、性能 profiling，AI 看不到运行时数据）、大规模重构（改动超过 20 个文件时容易丢失上下文）。

**一个更本质的问题**：Kiro 的 Spec 流程给了你"掌控感"，但掌控感不等于掌控。你审查 AI 生成的 Spec 和代码，审查质量取决于你对项目的理解深度。如果你自己也不确定该怎么设计，审查就变成了走过场。

## 实践指南：从个人到团队的路径

最后整理一份实操建议，按项目阶段分：

**个人项目起步阶段**：先装好 Kiro，写好三个基础 Steering 文件，用 Vibe 模式探索需求，用 Spec 模式写代码。别急着配 Hooks 和 Skills，先用起来再说。

**项目成型阶段**：加上自定义 Steering 文件（把你重复纠正 AI 的事情写进去），配一两个最关键的 Hooks（保存时 lint、提交前测试），开始用 Checkpoints 保护关键节点。

**团队协作阶段**：所有 Steering 文件和 Hooks 配置进 Git，封装 2-3 个最常用的 Skills，考虑企业版的治理能力是否需要。

**一个不变的底线**：不管在哪个阶段，**你仍然是最重要的质量保障**。Kiro 能让你更快地写代码、更有结构地做开发，但它不能替你思考。Spec 驱动开发的核心不是 AI 生成的规格文档，而是你在审查和修改那份文档时投入的思考。
