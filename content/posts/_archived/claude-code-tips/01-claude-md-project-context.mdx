你有没有过这种体验——让 Claude Code 帮你加个功能，它写出来的代码风格完全不像你的项目？明明你用 TypeScript，它给你写 JavaScript；明明你习惯用函数组件，它给你写 class。更离谱的是，有时候它连你用的是什么框架都搞不清楚。

这不是 Claude 能力不行。是你没给它足够的上下文。

Claude Code 每次启动新会话时，会自动读取项目根目录的 `CLAUDE.md` 文件。这个文件就是 Claude 理解你项目的唯一入口——写好了，它就像一个熟悉你项目的老同事；写差了，它就是一个刚入职还在摸索的新人。

## CLAUDE.md 不是 README

很多人第一次听说 CLAUDE.md 的时候，觉得它就是 README 的另一个版本。不是。

README 是给人看的，重点在"这个项目是什么、怎么跑起来"。CLAUDE.md 是给 Claude 看的，重点在"这个项目怎么开发、有哪些约定"。两者的受众和目的完全不同。

拿我自己来说，我之前把 README 的内容原封不动搬进 CLAUDE.md：

```markdown
# 我的博客

基于 Next.js 的个人博客，使用 MDX 写文章。

## 开发

npm run dev
npm run build
```

然后让 Claude 帮我加一个新的博客系列。它创建了一个 `content/blog/new-series/` 目录，文件用 `.md` 格式，frontmatter 里写了 `layout: post`——没一个是对的。

为什么？因为 README 里根本没告诉 Claude：博客文章放在 `content/posts/` 下、必须用 `.mdx` 格式、frontmatter 要包含 `series` 和 `seriesOrder` 字段。Claude 只能猜，而它的猜测基于训练数据里最通用的模式，不一定符合你的项目。

## 一份好的 CLAUDE.md 长什么样

经过反复迭代，我总结出一个实用的三层结构。不要求每层都写，但核心信息不能缺。

### 第一层：项目概览（必须写）

用 2-3 句话告诉 Claude 这个项目是什么、技术栈是什么。不是给人类看的项目介绍，而是帮 Claude 快速定位的技术摘要。

```markdown
个人博客项目，使用 Next.js 15 + Tailwind CSS 4 + MDX。
文章以系列形式组织，每个系列是一个子目录。
构建命令：npm run build（构建会检查所有文章的 frontmatter 格式）。
```

这三句话已经告诉 Claude：框架是什么、内容怎么组织的、构建怎么跑。足够让它不犯方向性错误。

### 第二层：关键约定（按需写）

这里写你的项目里"和常规做法不一样"的地方。Claude 会基于训练数据里的常见模式来写代码，如果你的项目有特殊约定，必须显式说明。

```markdown
## 文章格式

- 文章路径：content/posts/{category}/XX-kebab-case-title.mdx
- frontmatter 必须包含：title, date, summary, tags, category, cover, draft, series, seriesOrder
- 系列文章必须有 plan.md 文件描述整体规划
- 文章中禁止手动添加"上一篇/下一篇"链接（系统自动生成）

## 代码风格

- 使用 TypeScript，不使用 any
- 函数组件 + hooks，不使用 class 组件
- 使用 Tailwind CSS，不写内联 style
- API 路由放在 src/app/api/ 下，使用 Next.js Route Handlers
```

这些约定看似简单，但它们直接决定了 Claude 生成的代码能不能用。我之前没写"不使用 class 组件"这条，Claude 有大概 30% 的概率会生成 class 组件，因为它的训练数据里 class 组件的样本很多。

### 第三层：常见任务指引（建议写）

告诉 Claude 你最常让它做的事情应该怎么做。这相当于给新同事一份"常见操作手册"。

```markdown
## 常见任务

### 新增博客系列

1. 在 content/posts/ 下创建新目录
2. 创建 plan.md 描述系列规划
3. 每篇文章遵循 frontmatter 格式要求
4. 写完后运行 npm run build 验证

### 修改现有功能

1. 先找到相关文件（用 grep 或文件搜索）
2. 理解现有实现再动手
3. 修改后运行 npm run build 确认没有破坏其他功能
```

这一层的价值在于减少来回沟通。没有这些指引，你可能会遇到这样的对话循环：让 Claude 加功能 → 它忘了跑 build → 你提醒它 → 它跑 build 发现报错 → 它修复 → 再跑 → 又报错……有了"写完后运行 build"这条指引，它会自动在完成操作后验证，省掉好几轮来回。

## 项目级 vs 全局级 CLAUDE.md

Claude Code 支持两种级别的配置：

- **项目级**：放在项目根目录的 `CLAUDE.md`，只对这个项目生效
- **全局级**：放在 `~/.claude/CLAUDE.md`，对你所有项目生效

全局级适合写什么？写你和项目无关的个人偏好：

```markdown
## 我的偏好

- 提交信息用英文
- 代码注释用中文
- 优先使用函数式写法
- 解释技术概念时用类比，先说为什么再说怎么做
```

项目级适合写什么？写和这个项目相关的具体约定：目录结构、文件格式、构建命令、测试方式等。

一个常见的错误是把项目特有的配置写在全局级里——比如把"文章必须用 MDX 格式"写在全局 CLAUDE.md 里。这会导致 Claude 在其他项目里也试图用 MDX 格式，反而添乱。

## 多少行算合适

CLAUDE.md 不是越长越好。它会占用上下文窗口的空间，写太多反而会让 Claude 忽略重要的内容。

实际测试下来，**100-200 行是一个好的区间**。少于 50 行，信息不够；超过 200 行，Claude 开始"选择性忽略"后面的内容。

如果你的项目确实有很多约定要写，考虑分层组织：核心约定写在 CLAUDE.md 里，详细的编码规范用链接引用到单独的文件。Claude 可以通过读取文件来获取更多信息，但 CLAUDE.md 本身应该精炼。

```markdown
## 详细规范

- 编码规范：参考 .eslintrc.js 和 tsconfig.json
- 组件规范：参考 src/components/README.md
- API 设计：参考 src/app/api/README.md
```

这样 CLAUDE.md 保持精炼，但 Claude 知道去哪里找详细规范。

## 一份真实的 CLAUDE.md

作为参考，这是我博客项目的 CLAUDE.md（简化版）：

```markdown
# 个人博客

Next.js 15 静态博客，使用 Tailwind CSS 4 + MDX。文章以系列形式组织在 content/posts/ 下。

## 构建

- 开发：npm run dev
- 构建：npm run build（必须通过才能提交）
- 文章 frontmatter 格式错误会导致构建失败

## 文章结构

- 路径：content/posts/{category}/XX-kebab-case-title.mdx
- 分类映射在 src/lib/category-map.ts
- 文章组件在 src/components/PostList.tsx
- 每个系列有 plan.md 文件

## frontmatter 格式

必须字段：title, date, summary（100字以内）, tags, category, series, seriesOrder, draft
禁止手动添加"上一篇/下一篇"导航链接（系统自动处理）

## 写作规则

- 文章开头先说明引入该技术解决了什么问题
- 面向有基础但非精通的读者
- 代码注释要详细，标注每一步的作用
- 不用玩具示例，用真实场景

## 代码风格

- TypeScript strict mode，不使用 any
- 函数组件 + hooks
- Tailwind CSS，不用内联 style
- 文件命名：kebab-case
```

30 行左右，覆盖了最关键的信息。每次新建文章，Claude 都能正确地创建文件、填写 frontmatter、跑 build 验证。

## 进阶：让 CLAUDE.md 随项目演进

CLAUDE.md 不是写一次就完事的。随着项目发展，你需要更新它。

一个实用的做法：每次你发现 Claude 犯了重复性的错误，就往 CLAUDE.md 里加一条约定。比如我发现 Claude 多次在创建新文章时忘记加 `seriesOrder` 字段，就在 frontmatter 格式部分加粗了这个要求。

另一个做法：让 Claude 自己帮你维护 CLAUDE.md。当你觉得 Claude 的表现变差了，可以直接问它："你觉得 CLAUDE.md 里缺了什么信息，导致你刚才犯了那个错误？"Claude 通常能给出很好的自我诊断。

## 什么时候该花时间优化 CLAUDE.md

不是所有项目都需要精心维护 CLAUDE.md。判断标准很简单：

- **你打算长期用 Claude Code 开发的项目**：值得花时间写好，它是你的效率倍增器
- **一次性项目或短期项目**：简单写个技术栈和构建命令就够了
- **开源项目或团队项目**：更值得写好，因为它也能帮助团队其他成员使用 Claude Code

CLAUDE.md 写得好不好，直接决定了你后续使用 Claude Code 的体验上限。花 30 分钟把 CLAUDE.md 写对，能省下之后无数次的"不对，再改改"。
