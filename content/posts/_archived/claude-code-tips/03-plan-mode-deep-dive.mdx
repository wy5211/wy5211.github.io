前两篇我们解决了两个基础问题：让 Claude 理解你的项目（CLAUDE.md），以及让关键信息跨对话传递（记忆系统）。有了这两个基础，Claude 已经不会犯"方向性错误"了。

但还有一个常见的效率杀手：Claude 一上来就写代码，写了一半发现方向不对，推翻重来。或者更惨的情况——它写完了，你也看着挺像回事，合并之后才发现方案选错了。

Plan 模式就是解决这个问题的。它强迫 Claude 先想清楚再动手。

## 没有 Plan 的世界是什么样的

看一个真实的例子。我让 Claude 帮我的博客加一个"相关文章推荐"功能：

> 给博客文章底部加一个"相关文章推荐"模块，显示同系列的其他文章。

Claude 直接开始写代码了。它创建了一个 `RelatedPosts.tsx` 组件，从 `category-map.ts` 读取分类信息，按分类匹配相关文章。

看起来没问题？但它的实现方式有个根本性缺陷：它是按分类匹配的，而我的博客里"同分类"不等于"相关"。比如 Python 分类下有 25 篇文章，涵盖基础语法、Web 开发、数据处理、异步编程等完全不同的主题。用户在看一篇 Python 异步编程的文章时，看到"Python 列表推导式"作为相关推荐，没有意义。

如果 Claude 先问一句"相关文章是按什么规则推荐的？按分类、按标签、还是按内容相似度？"，我就能及时纠正方向。

这就是 Plan 模式的价值——**让 Claude 在动手之前先说清楚它打算怎么做，给你一个纠正的机会**。

## 怎么进入 Plan 模式

有几种方式：

**1. 在 prompt 里加 "plan"**

在任务描述中加上 "先做 plan" 或 "用 plan 模式"，Claude Code 会自动进入 Plan 模式。

**2. 用 EnterPlanMode 工具**

Claude Code 内置了 `EnterPlanMode` 工具。你可以在对话过程中随时让 Claude 进入 Plan 模式，即使它已经开始写代码了。

**3. 手动触发**

在 Claude Code 的命令行界面中，可以用快捷键手动切换到 Plan 模式。

进入 Plan 模式后，Claude 只能做只读操作——读文件、搜索代码、分析结构，但不能写文件或执行命令。它会把分析结果写成一份计划文档，等你确认后才开始执行。

## 什么时候该用 Plan 模式

Plan 模式不是所有任务都需要的。用对了是效率倍增器，用错了是浪费时间。

### 需要 Plan 的场景

**新功能开发**：你要加一个从未有过的功能。Plan 能帮 Claude 理解现有架构、确定插入点、规划改动范围。

比如"给博客加一个 RSS 订阅功能"——Claude 需要先了解你的路由结构、数据层、构建流程，才能决定 RSS feed 放在哪里、用什么方式生成。

**架构重构**：涉及多个模块的改动。Plan 能帮你看清依赖关系、确定改动顺序、评估风险。

比如"把状态管理从 Context 迁移到 Zustand"——Claude 需要找出所有使用 Context 的组件、规划迁移顺序、考虑中间状态的兼容性。

**不确定需求的任务**：你自己也不太清楚具体要怎么做，需要 Claude 帮你分析可行性。

比如"想优化博客的加载速度"——Claude 需要先分析当前的性能瓶颈、确定优化方向、给出优先级排序。

### 不需要 Plan 的场景

**简单 bug 修复**：错误信息已经明确了问题位置和原因，直接改就行。

**单文件改动**：只改一个文件，影响范围明确。

**重复性任务**：之前做过类似的事情，模式已经确定。比如"用同样的方式再加一篇文章"。

判断标准很简单：**如果你能清楚地说出"Claude 应该改哪几个文件、每个文件大概改什么"，就不需要 Plan。如果你说不太清楚，就需要。**

## Plan 模式的正确打开方式

进入 Plan 模式只是第一步。怎么引导 Claude 做出好的 Plan，才是关键。

### 给足够的问题上下文

差的 prompt：

> 帮我重构博客的分类系统。

好的 prompt：

> 博客目前用 category-map.ts 硬编码分类颜色和名称。随着分类越来越多，维护这个文件变得很麻烦。我想改成每个分类有自己的配置文件，或者从文章的 frontmatter 自动生成分类列表。帮我分析一下这两种方案的优劣，然后 Plan 出重构步骤。

区别在于：好的 prompt 说清了现状、问题、和可选方案。Claude 不需要从零开始猜测你的需求，可以直接聚焦在方案对比和实施细节上。

### 善用 Plan 中的交互

Plan 模式不是 Claude 单方面的输出。你可以利用 Plan 中的交互工具来引导方向。

**AskUserQuestion**：Claude 会在遇到不确定的地方主动问你。比如它可能会问"分类配置文件放在哪里？是 content/posts/{category}/config.json 还是单独的 src/config/categories/ 目录？"。这种时候不要偷懒，认真回答——你的回答直接影响 Plan 的质量。

**EnterPlanMode**：如果你在普通对话中发现 Claude 的方向不对，可以随时让它切换到 Plan 模式。"先停一下，别直接写代码，先做个 Plan 看看方案对不对。"

### Plan 文件的结构

Claude 生成的 Plan 文件通常包含这些部分：

```markdown
# 重构计划：博客分类系统

## 现状分析

- 当前方案：category-map.ts 硬编码所有分类
- 问题：新增分类需要手动修改代码，容易遗漏

## 方案对比

### 方案 A：每个分类独立配置文件

- 优点：新增分类只需加文件，不影响其他分类
- 缺点：需要额外的文件发现机制

### 方案 B：从文章 frontmatter 自动生成

- 优点：零维护，分类跟着文章走
- 缺点：无法自定义分类的显示名称和颜色

## 推荐方案

方案 A，因为...

## 实施步骤

1. 创建分类配置的 TypeScript 类型
2. 实现分类文件的发现和加载
3. 替换 category-map.ts 的硬编码逻辑
4. 迁移现有分类数据
5. 更新组件使用新的分类系统

## 风险点

- 构建时分类文件加载失败的降级策略
- 现有 URL 路径不能变（SEO 影响）
```

这个结构的好处是：现状分析让你确认 Claude 理对了问题，方案对比让你有选择的余地，实施步骤让你对改动范围有预期，风险点让你提前考虑边界情况。

### Plan 的粒度控制

一个常见的错误是 Plan 写得太细——把每个文件的每一行改动都列出来，变成了伪代码。这样的 Plan 有两个问题：一是 Claude 把时间花在了 Plan 上而不是实现上，二是太细的 Plan 反而失去了灵活性——实现过程中发现某个步骤需要调整时，整个 Plan 就得重写。

另一个常见错误是 Plan 写得太泛——"先改数据层，再改组件层，最后测试"。这种 Plan 等于没做，因为它没有提供任何超出常识的信息。

好的粒度是：**说清楚改什么文件、为什么改、改完期望什么效果，但不需要说具体怎么改每一行。**

## 一个实战对比

让我用同一个任务对比一下有 Plan 和没有 Plan 的区别。

任务：给博客加暗黑模式支持。

**没有 Plan 的流程**：

1. 我说"加暗黑模式"
2. Claude 创建了 `ThemeContext.tsx`，用 React Context 管理主题状态
3. 我看了一眼发现，我的项目用了 Next.js App Router，没有全局 Context Provider 的位置
4. 我告诉 Claude 这个问题
5. Claude 改用 middleware 方案
6. 又发现 Tailwind CSS 的暗黑模式配置没开
7. 又改配置
8. 来回 4 轮才搞定

**有 Plan 的流程**：

1. 我说"加暗黑模式，先做 Plan"
2. Claude 读了项目结构，发现用的是 App Router
3. Plan 里列了：middleware 方案 vs CSS 变量方案 vs next-themes 库的对比
4. 我选了 next-themes（Plan 里的推荐方案）
5. Plan 列出需要改的文件：tailwind.config.ts、layout.tsx、ThemeToggle 组件
6. 确认后一次性改完

4 轮 vs 1 轮，而且有 Plan 的版本方案选得更合理（Claude 在 Plan 阶段就有时间比较多个方案）。

## Plan 模式的边界

Plan 模式不是万能的。有几种情况它帮不上忙：

- 任务本身就是探索性的，没有明确的终点（比如"看看这个项目有什么可以优化的"）
- 你对要做什么还完全没有想法，需要 Claude 帮你发散思维
- 时间紧迫的小改动，Plan 的成本超过了收益

这些时候直接对话反而更高效。Plan 模式的价值在于**减少不确定性带来的返工**，当不确定性很低的时候，Plan 的价值也就低了。

下一篇我们聊 Hooks——Plan 解决了"想清楚再干"的问题，Hooks 解决的是"干了之后自动检查"的问题。两个配合使用，Claude Code 的出错率会大幅下降。
