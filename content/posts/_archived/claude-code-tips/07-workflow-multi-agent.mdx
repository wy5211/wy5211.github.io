到这一篇为止，我们已经把 Claude Code 的单会话能力推到了极限——CLAUDE.md 给上下文、记忆系统传经验、Plan 模式想方案、Hooks 自动化检查、Skills 扩展能力、MCP 连接外部服务。

但有些任务单会话搞不定。比如你需要同时审查一个 PR 里的所有文件，每个文件从不同维度（安全性、性能、可维护性）检查。单会话里 Claude 只能一个一个文件看，非常慢。或者你需要在一个不确定大小的代码库里搜索所有潜在的 bug，搜一轮可能不够，需要反复搜直到确认找全了。

这就是 Workflow 和多 Agent 编排的价值——让多个 Claude 实例同时工作，或者按照特定逻辑循环执行，直到任务完成。

## 从单 Agent 到多 Agent

先搞清楚一个前提：**不是所有任务都需要多 Agent。**

单 Agent（普通对话模式）已经能处理大部分日常开发任务：写代码、改 bug、重构、搜索文件。只有在以下场景才需要多 Agent：

- **大批量任务**：需要对 10+ 个文件执行同样的操作
- **多维度分析**：同一个代码变更需要从不同角度审查
- **不确定范围的搜索**：不知道问题有多少个，需要反复搜索直到确认找全
- **需要独立视角**：同一个问题需要多个独立的方案进行比较

判断标准：如果你发现自己在对话里反复说"再看下一个文件"或者"再搜一轮"，就是时候考虑 Workflow 了。

## Workflow 的三种核心模式

Claude Code 的 Workflow 支持三种编排模式，每种解决不同类型的问题。

### Pipeline——流水线

Pipeline 的逻辑是：把一组任务排成流水线，每个任务依次通过多个阶段处理。阶段之间不需要同步——任务 A 可以在阶段 3，任务 B 还在阶段 1。

适合场景：每个任务需要经过一系列处理步骤，但任务之间互不依赖。

一个实际例子：你有一批文件需要格式化（阶段 1）→ 类型检查（阶段 2）→ lint 修复（阶段 3）。每个文件独立处理，不需要等其他文件完成。

Pipeline 的优势是速度——因为任务之间不用等，总的处理时间接近最慢的那个任务，而不是所有任务时间的总和。

### Parallel——并行屏障

Parallel 的逻辑是：同时启动多个任务，等所有任务完成后汇总结果。这是一个"屏障"——所有并行任务必须在汇总前完成。

适合场景：需要多个独立视角的结果放在一起比较或合并。

一个实际例子：代码审查时，让三个 Agent 分别从安全性、性能、可维护性三个角度审查同一段代码，然后汇总三个审查结果。这种场景下，汇总阶段需要所有审查结果都在手，所以必须用 Parallel。

### Loop-until-dry——循环搜索

Loop-until-dry 的逻辑是：反复执行搜索任务，直到连续 N 轮没有发现新问题为止。

适合场景：不确定问题有多少个，需要穷尽式搜索。

一个实际例子：在一个大型代码库中搜索所有潜在的 XSS 漏洞。第一轮搜索找到了 5 个，第二轮找到了 3 个新的，第三轮没有找到新的——说明搜索已经"干"了，可以停止了。

## 实战：用 Workflow 做一次多维度代码审查

让我用一个完整的例子展示 Workflow 怎么用。假设你有一个包含 12 个文件变更的 PR，需要做全面的代码审查。

### 设计审查维度

先确定审查维度。我通常用这四个：

1. **Bug 检查**：逻辑错误、边界条件、空指针风险
2. **安全审查**：注入漏洞、敏感数据暴露、权限问题
3. **性能分析**：不必要的循环、N+1 查询、内存泄漏
4. **可维护性**：命名清晰度、代码重复、复杂度过高

每个维度由一个独立的 Agent 负责，这样它们互不影响，能给出更聚焦的审查结果。

### 编排 Agent

用 Parallel 模式启动四个审查 Agent：

```
Workflow：
1. 启动 4 个并行 Agent，每个负责一个审查维度
2. 等 4 个 Agent 都完成
3. 对每个发现的问题做对抗验证（adversarial verify）
4. 汇总最终结果
```

每个审查 Agent 收到的指令类似这样：

> 你是一个代码审查专家，专注安全审查。请审查以下文件变更，
> 找出所有潜在的安全问题：SQL 注入、XSS、CSRF、敏感数据暴露、
> 不安全的 API 调用。对每个发现，给出文件位置、问题描述和修复建议。

四个 Agent 并行工作，各自只关注自己的维度。因为是并行的，总时间取决于最慢的那个 Agent，而不是四个加起来。

### 对抗验证（Adversarial Verify）

审查结果出来了，但怎么确定 Agent 的发现是真实的问题而不是误报？

这里用到"对抗验证"——给每个发现再派一个 Agent，让它尝试反驳这个发现。如果大多数验证 Agent 认为这个发现站不住脚，就把它过滤掉。

这是 Workflow 中非常有用的质量保障模式。单次审查可能产生误报，但对抗验证能把误报率降到很低。

### 汇总结果

最后，把通过验证的发现按严重程度排序，输出一份审查报告。

整个过程你只需要一个指令，Workflow 自动编排多个 Agent 协作完成。比手动逐文件审查快得多，覆盖面也更全。

## 成本控制

多 Agent 的 token 消耗是单 Agent 的 N 倍——如果你同时启动 4 个审查 Agent 加 4 个验证 Agent，就是 8 倍的消耗。这不是开玩笑的数字。

几个控制成本的方法：

### 按需选择模型

不是所有 Agent 都需要最强模型。审查 Agent 用 Sonnet 就够了（它擅长代码理解），只有需要复杂推理的任务（比如架构决策）才用 Opus。在 Workflow 中可以为不同阶段指定不同的模型。

### 控制并行数量

Claude Code 对并行 Agent 有数量上限（通常是 16 个）。但你不应该每次都用满。3-5 个并行 Agent 对大多数场景已经足够了。更多不一定更好——Agent 之间的结果汇总和去重也是成本。

### 设置 token 预算

Workflow 支持 token 预算设置。你可以设置总预算，当消耗接近上限时自动停止。这避免了 Workflow 因为某个 Agent 跑飞了而消耗大量 token 的情况。

### Loop-until-dry 的提前终止

循环搜索时，不要设"无限循环直到找完"。设一个最大循环次数（比如 5 次），或者设置"连续 2 轮没有新发现就停止"。这样即使有遗漏，也不会无限循环下去。

## 什么时候 Workflow 不值得

Workflow 虽然强大，但有额外的开销：

- **启动成本**：每个 Agent 都需要独立初始化，大约 200-500ms
- **上下文成本**：每个 Agent 都要加载 CLAUDE.md 和记忆，这些是重复的
- **汇总成本**：多个 Agent 的结果需要合并和去重，这本身也需要 token

所以对于简单的任务，Workflow 反而比单 Agent 慢。

一个简单的判断：如果你的任务能在 3-5 轮对话内完成，用单 Agent。如果需要 10 轮以上，或者需要并行处理多个独立子任务，用 Workflow。

## Workflow 和前面内容的配合

这一篇是系列的高潮——它把前面学到的所有能力组合在了一起：

- **CLAUDE.md**（第 1 篇）：每个 Agent 都会自动加载项目上下文，确保它们理解你的代码规范

- **Plan 模式**（第 3 篇）：Workflow 本身就是一种更高级的 Plan——把任务拆解成可执行的步骤
- **Hooks**（第 4 篇）：Agent 编辑文件后，Hook 自动跑格式化和检查
- **Skills**（第 5 篇）：Agent 可以调用特定的 Skill，比如用 Code Review Skill 做审查
