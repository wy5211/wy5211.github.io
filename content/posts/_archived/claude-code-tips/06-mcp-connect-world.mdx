前几篇聊的都是 Claude Code 在本地的能力——CLAUDE.md 给它上下文、记忆系统传递经验、Plan 模式帮它想清楚、Hooks 自动化检查、Skills 扩展技能。但所有这些能力都有一个限制：Claude 只能操作本地文件和终端。

如果你需要它查数据库、调 API、读 Figma 设计稿、管理 GitHub Issues 呢？这些操作需要连接外部服务，而 MCP（Model Context Protocol）就是 Claude Code 连接外部世界的桥梁。

## MCP 是什么

简单说，MCP 是一种协议，让 Claude Code 能通过"服务器"连接到外部工具和数据源。每个 MCP 服务器封装了一组特定的能力——比如 GitHub MCP 服务器让 Claude 能操作 Issues 和 PR，数据库 MCP 服务器让 Claude 能执行 SQL 查询。

你可以把 MCP 理解成"给 Claude 开外挂"。没有 MCP，Claude 是一个只能看到你本地文件的助手；有了 MCP，它可以访问 GitHub、数据库、浏览器、设计工具等一切有 MCP 服务器的服务。

如果你读过我的 [AI 全栈系列中关于 MCP 的文章](/posts/ai-fullstack/04-mcp-protocol)，那里讲的是怎么**开发**一个 MCP 服务器。这篇的角度不同——我们关注的是怎么**使用和配置**现成的 MCP 服务器。

## 配置 MCP 服务器

所有 MCP 服务器都配置在 `.claude/settings.json` 的 `mcpServers` 字段里（项目级）或 `~/.claude/settings.json`（全局级）。

一个基本的配置长这样：

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxxxxxxxxxx"
      }
    }
  }
}
```

每个 MCP 服务器有三个关键配置：

- `command`：启动服务器的命令
- `args`：命令参数
- `env`：环境变量（通常用来传 API Token）

配置好后重启 Claude Code，新的 MCP 服务器就会自动加载。你可以通过 `/mcp` 命令查看当前加载的所有 MCP 服务器及其状态。

## 热门 MCP 服务器实战测评

以下是社区最受欢迎的几个 MCP 服务器，每个我都实际配置和测试过。

### GitHub MCP Server——最值得装的

如果只装一个 MCP 服务器，装这个。

GitHub MCP Server 提供了 51 个工具，覆盖了完整的 GitHub 工作流：创建和管理 Issues、创建和审查 PR、搜索代码、管理分支、查看 CI 状态。

没有它的时候，Claude 让你"去 GitHub 上创建一个 Issue"，你得手动操作。有了它，Claude 可以直接帮你：

- 搜索项目里的所有 open issues，看有没有和你相关的工作
- 创建 PR 并自动填写描述
- 审查 PR 的代码变更并给出意见
- 搜索代码仓库中的特定实现模式

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "你的Token"
      }
    }
  }
}
```

Token 需要在 GitHub Settings → Developer Settings → Personal Access Tokens 里创建，建议用 Fine-grained token，只给必要的权限。

### Supabase / PostgreSQL MCP——让 Claude 直接查数据

如果你用 Supabase 或 PostgreSQL，这个 MCP 服务器让 Claude 能直接查询数据库。不需要你手动导出 SQL 结果再粘贴给 Claude——它能自己查、自己分析。

实际用法：我在开发博客的搜索功能时，需要了解文章数据的分布情况（每个分类有多少篇、标签使用频率等）。以前我会手动跑 SQL、把结果粘贴给 Claude。有了数据库 MCP，我直接说"帮我分析一下文章数据的分布"，Claude 会自己写 SQL、执行、分析结果。

```json
{
  "mcpServers": {
    "supabase": {
      "command": "npx",
      "args": ["-y", "@supabase/mcp-server-supabase"],
      "env": {
        "SUPABASE_ACCESS_TOKEN": "你的Token",
        "SUPABASE_PROJECT_ID": "你的项目ID"
      }
    }
  }
}
```

如果是自建的 PostgreSQL，用 PostgreSQL MCP：

```json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://user:pass@localhost:5432/mydb"
      ]
    }
  }
}
```

安全提示：不要把数据库连接字符串硬编码在配置文件里——特别是如果这个文件会被提交到 Git。用环境变量或者 `.env` 文件来管理敏感信息。

### Playwright MCP——浏览器自动化

Playwright MCP 让 Claude 能控制浏览器——打开网页、截图、点击元素、填写表单、提取内容。

最有用的场景是测试：让 Claude 自动打开你的应用、执行一系列操作、验证页面表现。比手动测试快很多，也比写自动化测试脚本灵活——你可以用自然语言描述测试步骤。

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["-y", "@playwright/mcp"]
    }
  }
}
```

另一个常见用法：让 Claude 打开一个网页、读取内容、然后分析或总结。比如"打开这个 API 文档，帮我梳理一下接口列表和参数说明"。

### Figma MCP——设计稿直接用

前端开发者会喜欢这个。Figma MCP 让 Claude 能直接读取设计稿——组件结构、颜色值、间距、字体大小等信息都能获取。

以前的工作流：设计师发 Figma 链接 → 你打开 Figma → 手动标注各种属性 → 写 CSS。有了 Figma MCP：直接把 Figma 链接给 Claude，它能读取设计参数然后生成对应的组件代码。

```json
{
  "mcpServers": {
    "figma": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-figma"],
      "env": {
        "FIGMA_ACCESS_TOKEN": "你的Token"
      }
    }
  }
}
```

### Sentry MCP——线上错误直通车

Sentry MCP 让 Claude 能直接查看线上错误——错误堆栈、影响用户数、发生频率、关联的 release。不需要你在 Sentry 网页上翻来翻去再复制错误信息给 Claude。

典型用法："看看最近 24 小时有什么新错误"。Claude 会从 Sentry 拉取数据、分析堆栈、定位到代码中的问题位置、甚至直接给出修复建议。

```json
{
  "mcpServers": {
    "sentry": {
      "command": "npx",
      "args": ["-y", "@sentry/mcp-server"],
      "env": {
        "SENTRY_AUTH_TOKEN": "你的Token"
      }
    }
  }
}
```

### 其他值得关注的服务器

- **Slack MCP**：读取频道消息，让 Claude 了解团队讨论的上下文。适合让 Claude 帮你总结讨论内容、提取行动项。
- **Linear MCP**：项目管理集成。让 Claude 能查看和更新任务状态、创建新任务。
- **Notion MCP**：读写 Notion 页面和数据库。适合把 Notion 作为知识库或文档中心的团队。
- **AWS MCP**：AWS 服务集成。查看 EC2 实例状态、S3 存储桶内容、CloudWatch 日志等。
- **Apidog MCP**：API 规范管理。让 Claude 能读取 API 定义、生成接口调用代码。

## MCP 服务器的发现与选型

### mcpmarket.com

MCP Market 是一个 MCP 服务器的排行榜网站，按 GitHub star 数和安装量排序。每个服务器有功能介绍、配置示例和用户评价。想找某类服务器的 MCP 时，先来这里看看。

### modelcontextprotocol/servers（GitHub）

官方的参考实现仓库。这里的服务器质量有保障，但不一定是功能最丰富的。适合作为"标准选择"。

### 选型标准

选 MCP 服务器时看这几条：

1. **活跃度**：最近有没有更新？MCP 协议迭代很快，不活跃的服务器可能已经不兼容了
2. **安全性**：Token 和敏感数据怎么处理？有没有明文存储的风险？
3. **功能覆盖度**：你需要的功能它支持吗？有些服务器只实现了只读操作，不支持写入

### 装多少合适

和 Skills 一样，不是越多越好。每个 MCP 服务器在启动时都要初始化，加载太多会拖慢 Claude Code 的启动速度。建议只装你日常开发中真正需要的 3-5 个。

我自己的配置是：GitHub（必备）+ Playwright（测试常用）+ 数据库（开发时查数据），三个覆盖了大部分需求。

## 安全注意事项

MCP 服务器能连接外部服务，意味着安全问题不能忽视。

### Token 管理

最基本的原则：**不要把 API Token 硬编码在配置文件里**。特别是如果 `.claude/settings.json` 会被提交到 Git。

用环境变量代替：

```json
{
  "env": {
    "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
  }
}
```

或者在 shell profile 里 export 环境变量，MCP 服务器启动时会继承。

### 权限最小化

给 MCP 服务器的 Token 权限要最小化。比如 GitHub Token，如果你只需要读 Issues 和 PR，就不要给写权限。PostgreSQL 连接字符串，如果只是查数据，就用只读用户。

### 不可信的 MCP 服务器

不要随意安装来源不明的 MCP 服务器。MCP 服务器有完整的工具执行权限，一个恶意的服务器理论上可以读取你的文件、执行任意命令。只使用有信誉的、开源的、社区验证过的服务器。

## 踩坑经验

配置 MCP 服务器的过程中，我遇到过几个常见问题。

### 超时

有些 MCP 服务器启动比较慢（特别是用 npx 首次安装时），可能会超时。解决方案：第一次配置时手动跑一次 `npx -y @xxx/server-xxx`，让它先把包下载好。后续启动就会快很多。

### 重连

网络波动时 MCP 服务器可能断连。Claude Code 会尝试自动重连，但如果重连失败，你需要重启 Claude Code 或者用 `/mcp` 命令手动重连。

### 多服务器冲突

同时装了很多 MCP 服务器时，可能会出现工具名称冲突——两个服务器都提供了同名工具。这种情况下 Claude 可能会调错服务器。解决方案：检查是否有重复的工具名称，必要时调整服务器配置。

## 自己搭建 MCP 服务器

如果现成的 MCP 服务器都不满足你的需求，你可以自己搭建一个。比如你们公司有内部的 API 平台、内部的 CI/CD 系统，想让 Claude 也能操作这些系统。

开发 MCP 服务器的详细教程可以参考我的 [AI 全栈系列第 4 篇](/posts/ai-fullstack/04-mcp-protocol)，那里有完整的代码示例。这里只说几个要点：

- 用 TypeScript 或 Python 开发都行，官方都有 SDK
- 核心是定义"工具"——工具名、参数 schema、执行函数
- 可以用 stdio 传输（本地运行）或 SSE 传输（远程服务）
- 开发时用 `npx @modelcontextprotocol/inspector` 测试，它提供了一个可视化的调试界面

接上 MCP 服务器后，Claude Code 已经能做很多事情了。但如果你需要让它同时做多件事——比如并行审查 10 个文件、或者循环搜索直到找到所有问题——就需要 Workflow 了。下一篇我们聊多 Agent 编排。
