上一篇我们聊了 CLAUDE.md——给 Claude 一份"工作手册"，让它理解你的项目。但你可能很快会发现一个新问题：CLAUDE.md 是静态的，它能告诉 Claude "这个项目用 TypeScript"，但没法告诉 Claude "上次你说用 React Query 替代 SWR，我同意了"。

这种"上次对话里说好的事情"该怎么传递？这就涉及到 Claude Code 的另一个核心机制——记忆系统。

## 先搞清楚：Claude 没有真正的"记忆"

说 Claude Code 有"记忆"其实是用词不当。它不是像人类一样把经验存在大脑里随时提取。Claude Code 的"记忆"本质上是：**在每次新对话开始时，自动把某些信息注入到上下文窗口里**。

就像你在新同事的工位上贴了一张便利贴——看起来像是他"记住"了，其实只是你每次都给他看同一张纸条。效果类似，但机制完全不同。

理解这一点很重要，因为它决定了你怎么使用记忆：

- 记忆不是无限的——它占用上下文窗口的空间
- 记忆不是随时更新的——它只在对话开始时注入
- 记忆不是智能的——它不会根据当前任务自动选择该注入什么

## 两种记忆机制

Claude Code 有两种记忆方式，适合不同的场景。

### Auto-memory：让 Claude 自己记

Auto-memory 是 Claude Code 内置的能力。当你在对话中说"记住这个"或者"以后都这样做"，Claude 会自动把这条信息保存到项目的记忆文件里。下次新对话开始时，它会自动读取。

记忆文件存放在 `.claude/` 目录下，每个记忆是一个独立的 `.md` 文件。你可以直接去看 Claude 记住了什么，也可以手动编辑或删除。

Auto-memory 适合记什么？

- 项目特定的偏好："这个项目的 API 错误码统一用小写"
- 临时性的约定："本周重构时先用函数式写法"
- Claude 不容易从代码推断的信息："用户要求所有中文注释用全角标点"

不适合记什么？

- CLAUDE.md 里已经有的信息——会重复占用上下文
- 一次性的指令——"今天只改文件 A"这种不需要跨对话保留
- 很长的技术文档——记忆应该简短精炼

### 手动记忆：用 /remember 命令

除了让 Claude 自动记，你也可以直接用 `/remember` 命令手动添加记忆。这在你想精确控制记什么的时候很有用。

比如你发现 Claude 每次都在犯同一个错误——给博客文章加手动的"上一篇/下一篇"链接。你可以直接 `/remember` 写一条：

```
博客文章中禁止手动添加"上一篇/下一篇"导航链接，系统已自动处理。
```

这比在对话中反复提醒高效得多。

## 记忆 vs CLAUDE.md：什么时候用哪个

这是最容易混淆的地方。我总结了一个简单的判断标准：

| 信息类型               | 放在哪里       | 为什么                     |
| ---------------------- | -------------- | -------------------------- |
| 技术栈、目录结构       | CLAUDE.md      | 长期不变，所有对话都需要   |
| 编码规范、文件格式     | CLAUDE.md      | 长期不变，所有对话都需要   |
| 个人偏好（语言、风格） | 全局 CLAUDE.md | 跨项目的个人习惯           |
| 项目偏好的演进         | 记忆           | 会变化、需要跨对话传递     |
| 某次对话中达成的约定   | 记忆           | CLAUDE.md 不会每次对话都改 |
| 踩坑经验               | 记忆           | "上次这样做出了问题"       |

简单说：**CLAUDE.md 存"规章制度"，记忆存"工作经验"。**

一个实际的例子：我的 CLAUDE.md 里写了"文章必须用 MDX 格式"，这是长期不变的规章制度。但记忆里有一条"用户偏好：写完文章后自动跑 build 验证"，这是从多次对话中总结出来的工作习惯，可能会随着工作流变化而更新。

## 上下文窗口的"经济学"

记忆和 CLAUDE.md 都会占用上下文窗口。上下文窗口是有限的资源——当信息量接近上限时，Claude 会开始"忘记"早期的对话内容，这就是所谓的"失忆"。

理解上下文窗口的使用成本，能帮你做出更好的决策。

### 什么占得多

- 大段代码（特别是粘贴整个文件）
- 长篇的 Claude 输出（让 Claude 解释一大段代码）
- 大量的工具调用结果（比如搜索了整个项目）
- 多个 MCP 服务器返回的大数据

### 什么占得少

- 简短的指令
- CLAUDE.md（通常几十行，占比不大）
- 记忆文件（每条通常很短）
- 单个文件的修改

### 一个实用的估算方法

Claude Code 的上下文窗口大约是 200k tokens。一个大致的换算：

- 1 行代码 ≈ 3-5 tokens
- 1 段中文 ≈ 2-3 tokens/字
- 一个典型文件 ≈ 500-2000 tokens
- CLAUDE.md（100 行）≈ 500-1000 tokens
- 全部记忆文件 ≈ 500-3000 tokens

也就是说，CLAUDE.md 和记忆加起来通常只占上下文的 1-2%，非常划算。真正吃空间的是大段代码和工具输出。

### 长对话中的策略

当你在一个对话里工作了很长时间（比如 2 小时以上），上下文会逐渐被填满。这时候你有几个选择：

**1. 用 `/compact` 命令压缩上下文**

这个命令会让 Claude 总结当前对话的关键信息，然后用总结替代原始对话内容。效果是释放大量上下文空间，但代价是丢失了对话的细节。

什么时候用：你刚完成了一轮大的改动，接下来要开始新的任务，之前的对话细节不太需要了。

**2. 开新对话，利用记忆传递上下文**

如果下一个任务和之前的方向差别较大，直接开新对话可能更好。记忆和 CLAUDE.md 会自动注入新对话，你只需要简单说明当前要做什么。

什么时候用：之前的对话已经很长了（几十轮交互），新任务方向不同。

**3. 在关键节点主动保存记忆**

当你感觉到对话快要很长时，可以在关键节点主动让 Claude 保存记忆。比如"记住我们刚决定了用 React Query 替代 SWR，API 层已经改完了，接下来要改组件层"。

什么时候用：你知道这个任务要跨多个对话才能完成。

## 一个真实的场景

让我还原一个我曾经遇到的情况，来说明记忆系统怎么用。

我在用 Claude Code 重构博客的分类系统。对话进行了大约一个半小时，改了十几个文件。然后我发现 Claude 开始"忘记"之前讨论过的方案——它开始用旧的命名方式创建新文件，和半小时前的约定矛盾了。

原因很简单：上下文已经很长了，早期的对话内容被挤出了窗口。

如果当时我用对了记忆系统，流程应该是这样的：

1. **讨论完方案后**，让 Claude 记住关键决定："记住：分类系统用 category-map.ts 统一管理，不在组件里硬编码分类名"
2. **完成一个阶段后**，用 `/compact` 压缩，释放空间给下一阶段
3. **如果需要跨天继续**，在记忆里保存当前进度："分类系统重构进行到一半，API 层已完成，组件层待改"

这样即使开新对话，Claude 也能从记忆里获取关键上下文，不会从零开始。

## 记忆的维护

和 CLAUDE.md 一样，记忆也需要维护。我建议每隔一两周清理一次记忆文件：

- 删掉已经过时的记忆（比如"本周"开头的记忆）
- 把反复出现的记忆合并、精简
- 把应该长期保留的记忆迁移到 CLAUDE.md 里

你可以在 `.claude/` 目录下直接看到所有记忆文件，像管理代码一样管理它们。有些团队甚至会把这些文件纳入 git 管理，这样团队成员可以共享项目经验。

## 不要过度依赖记忆

最后一个提醒：记忆是补充，不是万能药。如果一条信息对所有对话都重要，它应该进 CLAUDE.md 而不是记忆。记忆应该只存"从经验中总结出来的、可能变化的"信息。

一个判断方法：如果你发现同一条记忆存在了超过一个月没有被淘汰，考虑把它迁移到 CLAUDE.md 里——它可能就是一个长期的约定，而不只是临时经验。
