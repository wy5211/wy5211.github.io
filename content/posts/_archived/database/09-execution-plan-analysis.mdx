上一篇讲了几个查询优化的套路。但很多时候你不知道查询慢在哪里——加了索引没用、改了写法反而更慢、同样的查询有时快有时慢。这时候你需要看数据库的执行计划。

`EXPLAIN ANALYZE` 是 PostgreSQL 最有用的诊断工具。它不只告诉你数据库"打算怎么执行"（计划），还会告诉你"实际执行了多久"（真实耗时）。这两个信息合在一起，就能帮你定位慢查询的根因。

## EXPLAIN vs EXPLAIN ANALYZE

先区分两个命令：

`EXPLAIN` 只显示执行计划，不真正执行查询。它告诉你数据库打算用什么策略，每个步骤预估的成本是多少。但它不会告诉你实际花了多少时间——因为查询根本没跑。

`EXPLAIN ANALYZE` 真正执行查询，并返回实际执行时间和行数。这才是你需要用的。

```sql
EXPLAIN ANALYZE
SELECT o.id, o.order_number, o.total, u.full_name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'paid'
ORDER BY o.created_at DESC
LIMIT 20;
```

输出大概是这样：

```
Limit  (cost=0.71..458.23 rows=20 width=72) (actual time=0.032..0.156 ms rows=20 loops=1)
  ->  Nested Loop  (cost=0.71..15234.56 rows=678 width=72) (actual time=0.031..0.148 ms rows=20 loops=1)
        ->  Index Scan using idx_orders_status_created on orders o  (cost=0.42..12456.78 rows=678 width=52) (actual time=0.021..0.089 ms rows=20 loops=1)
              Index Cond: (status = 'paid'::text)
        ->  Index Scan using users_pkey on users u  (cost=0.29..4.12 rows=1 width=28) (actual time=0.003..0.003 ms rows=1 loops=20)
              Index Cond: (id = o.user_id)
Planning Time: 0.124 ms
Execution Time: 0.234 ms
```

第一眼看过去可能有点懵，但其实结构很清晰。每一行代表一个执行步骤，缩进表示步骤之间的嵌套关系（像一棵树）。每个步骤有两个关键信息：`cost` 是预估成本，`actual time` 是实际耗时。**看 actual time，忽略 cost**——预估成本和实际耗时经常对不上。

## 三种扫描方式

执行计划里最常见的操作是"扫描"——数据库怎么从表里读数据。

**Seq Scan（顺序扫描）**：从第一行开始，逐行扫描整张表。没有 WHERE 条件的全表查询就是这个。如果在 WHERE 条件里有过滤但没有合适的索引，也会退化为 Seq Scan。

```
Seq Scan on orders  (cost=0.00..15432.00 rows=1000000 width=64)
  Filter: (status = 'paid')
```

这个输出告诉你：数据库在 orders 表上做顺序扫描，然后在每一行上执行 `status = 'paid'` 的过滤。如果 orders 表有 100 万行，但只有 10 万行是 `paid` 状态，那 90 万行的扫描是浪费的。

Seq Scan 不一定是坏事。如果你的查询需要返回表中大部分行（比如超过 30%），全表扫描可能比索引扫描更快——因为索引扫描需要先读索引再回表读数据，两次 I/O 的开销可能比一次全表扫描还大。数据库的优化器会根据统计信息自动选择。

**Index Scan（索引扫描）**：通过索引定位到满足条件的行，然后回表读取完整的行数据。

```
Index Scan using idx_orders_status_created on orders  (cost=0.42..4567.89 rows=100000 width=52)
  Index Cond: (status = 'paid')
```

`using idx_orders_status_created` 告诉你数据库用了哪个索引。`rows=100000` 是预估返回的行数。如果你的 WHERE 条件能精确匹配索引列，而且返回的行数远小于总行数，Index Scan 通常比 Seq Scan 快得多。

**Index Only Scan（只读索引扫描）**：只需要的数据都在索引里，不需要回表。

```
Index Only Scan using idx_orders_status_created on orders  (cost=0.42..2345.67 rows=100000 width=20)
  Index Cond: (status = 'paid')
```

Index Only Scan 是最理想的——数据库只需要读索引，不需要读表数据，I/O 量最少。但前提是你 SELECT 的列全部包含在索引里。比如你的索引是 `(status, created_at)`，查询 `SELECT status, created_at FROM orders WHERE status = 'paid'` 就可以走 Index Only Scan。但如果你还 SELECT 了 `total`，数据库就得回表去读 `total` 列，退化为 Index Scan。

想让更多查询走 Index Only Scan，可以考虑把需要的列都加到索引里——但要注意索引会变大，写入会更慢。

## 三种 JOIN 策略

当查询涉及 JOIN 时，数据库有三种执行策略可选。选哪种取决于表的大小、是否有索引、数据分布。

**Nested Loop（嵌套循环）**：对外表的每一行，在内表里查找匹配的行。就是两层循环。

```
Nested Loop  (actual time=0.031..0.148 ms rows=20 loops=1)
  ->  Index Scan on orders (actual time=0.021..0.089 ms rows=20 loops=1)
  ->  Index Scan on users (actual time=0.003..0.003 ms rows=1 loops=20)
```

注意内层扫描的 `loops=20`——外层返回了 20 行，所以内层执行了 20 次。如果内层有索引，每次执行都很快（0.003ms），总开销就是 20 × 0.003 = 0.06ms，可以接受。

Nested Loop 适合外表小、内表有索引的场景。如果外表有 10 万行，内表没有索引，Nested Loop 就会很慢——要做 10 万次全表扫描。

**Hash Join（哈希连接）**：先扫描较小的表，在内存里建一个哈希表，然后扫描较大的表，用哈希表来查找匹配。

```
Hash Join  (actual time=12.345..89.012 ms rows=500000 loops=1)
  Hash Cond: (o.user_id = u.id)
  ->  Seq Scan on orders (actual time=0.012..34.567 ms rows=1000000 loops=1)
  ->  Hash  (actual time=0.123..0.456 ms rows=50000 loops=1)
        ->  Seq Scan on users (actual time=0.011..0.345 ms rows=50000 loops=1)
```

Hash Join 的两个阶段很清楚：先建 Hash（扫描 users 表，50ms），再做 Join（扫描 orders 表，34ms）。总共 84ms。

Hash Join 适合两个表都比较大、没有合适索引的场景。它的优势是只需要扫描两张表各一次，不需要反复查找。但代价是需要内存来存哈希表——如果小表太大，内存放不下，就要溢出到磁盘，性能会断崖式下降。PostgreSQL 的 `work_mem` 参数控制每个操作能用的最大内存。

**Merge Join（归并连接）**：要求两个表都按 JOIN 的列排好序。然后像合并两个有序数组一样，各取一行比较，匹配的输出，不匹配的移动指针。

```
Merge Join  (actual time=5.678..45.123 ms rows=500000 loops=1)
  Merge Cond: (o.user_id = u.id)
  ->  Index Scan using idx_orders_user_id on orders (actual time=0.012..20.345 ms rows=1000000 loops=1)
  ->  Sort  (actual time=0.123..2.456 ms rows=50000 loops=1)
        Sort Key: u.id
        ->  Seq Scan on users (actual time=0.011..0.567 ms rows=50000 loops=1)
```

Merge Join 在两个表都已经有序（通过索引或排序）的情况下最快——只需要各遍历一次。如果其中一个表没有按 JOIN 列排序，优化器需要先排序，排序的开销可能抵消 Merge Join 的优势。

PostgreSQL 的优化器会自动在这三种策略中选择它认为最快的。大多数情况下它的选择是对的。但如果统计信息过时，它可能选错。

## 优化器选错方案怎么办

执行计划里最让人困惑的情况是：你明明建了索引，查询还是走了 Seq Scan。

这通常是因为统计信息过时了。PostgreSQL 的查询优化器根据统计信息（每列有多少不同值、数据的分布情况、表有多少行）来选择执行策略。这些统计信息通过 `ANALYZE` 命令收集，PostgreSQL 默认会在数据变化超过一定阈值后自动执行 `ANALYZE`。但如果你刚导入了一大堆数据、刚建了一张新表、或者刚跑了大批量的 UPDATE/DELETE，统计信息可能还没更新，优化器就会做出错误的判断。

比如你刚给 orders 表导入了 100 万条数据，然后查询 `WHERE status = 'paid'`。优化器根据旧统计信息认为 `status = 'paid'` 会返回 80% 的行，选择 Seq Scan。但实际上新数据里只有 10% 是 `paid` 状态，用 Index Scan 会快得多。

手动更新统计信息：

```sql
ANALYZE orders;
```

`ANALYZE` 不会锁表，可以随时执行。如果你不确定哪张表的统计信息过时了，可以更新所有表：

```sql
ANALYZE;
```

另一个常见原因是**参数类型不匹配**。假设 `user_id` 列的类型是 `INTEGER`，但你的查询写的是：

```sql
SELECT * FROM orders WHERE user_id = '42';  -- 字符串 '42'，不是整数 42
```

PostgreSQL 不会自动把 `'42'` 转成 `42` 来匹配索引——它会认为你在比较 `INTEGER` 和 `TEXT`，类型不匹配，索引不能用。解决办法是确保参数类型正确，或者显式转换：

```sql
SELECT * FROM orders WHERE user_id = '42'::INTEGER;
```

这在 ORM 里也可能发生。如果 Prisma 的 schema 里 `userId` 是 `Int` 类型，但前端传过来的参数是字符串，Prisma 通常会处理好类型转换。但如果你用了 `$queryRaw` 或 `$executeRaw` 写原生 SQL，就需要自己注意类型匹配。

## pg_stat_statements：找出最慢的 SQL

你不可能每次都手动给每个查询加 EXPLAIN ANALYZE。当系统变慢了，你首先想知道的是："哪些 SQL 最慢？"

PostgreSQL 的 `pg_stat_statements` 扩展帮你做这件事。它记录数据库执行过的每条 SQL 的统计信息：执行次数、总耗时、平均耗时、最快/最慢一次的耗时。

```sql
-- 启用扩展
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 查看最慢的 10 条 SQL（按平均耗时排序）
SELECT query,
       calls,
       total_exec_time / 1000 AS total_ms,
       mean_exec_time / 1000 AS avg_ms,
       max_exec_time / 1000 AS max_ms,
       rows
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
```

这个查询返回的是已经执行过的 SQL 的统计信息，不是执行计划。它能帮你快速定位"哪条 SQL 该优化"，但具体怎么优化还需要对那条 SQL 加 EXPLAIN ANALYZE 看执行计划。

`pg_stat_statements` 还有一个有用的功能：看"总耗时最高的 SQL"。一条 SQL 平均只执行 50ms，但如果每秒执行 100 次，它占用的总数据库时间可能比一条执行 2 秒但每小时只执行一次的 SQL 还多。

```sql
-- 按总耗时排序（找出最浪费数据库时间的 SQL）
SELECT query,
       calls,
       total_exec_time / 1000 AS total_ms,
       mean_exec_time / 1000 AS avg_ms
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

在 Docker 环境里启用 `pg_stat_statements`，需要在 `shared_preload_libraries` 配置里加上这个扩展，然后重启 PostgreSQL：

```yaml
# docker-compose.yml
postgres:
  image: postgres:16-alpine
  command:
    - "postgres"
    - "-c"
    - "shared_preload_libraries=pg_stat_statements"
    - "-c"
    - "pg_stat_statements.track=all"
```

## 小结

读执行计划的核心是关注 `actual time` 而不是 `cost`。看数据库用了什么扫描方式（Seq Scan 说明可能缺索引，Index Only Scan 是最理想的），用了什么 JOIN 策略（Nested Loop 适合小外表，Hash Join 适合大表），每个步骤实际返回了多少行（和预估差很多说明统计信息过时了）。

当你发现查询走了 Seq Scan 但觉得应该走 Index Scan，先检查：索引存在吗？WHERE 条件里的列和索引列匹配吗？参数类型匹配吗？如果都匹配但还是走 Seq Scan，跑一下 `ANALYZE` 更新统计信息。

`pg_stat_statements` 是你的全局监控工具——它告诉你哪些 SQL 最该优化。EXPLAIN ANALYZE 是你的局部诊断工具——它告诉你某条 SQL 具体慢在哪里。两个配合使用，就能系统地解决性能问题。
