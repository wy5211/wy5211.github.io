上一篇我们聊了为什么需要数据库——文件系统扛不住并发、查不了条件、保证不了一致性。现在你知道该用数据库了，但怎么和它"对话"？这就是 SQL 要做的事。

你打开一个电商项目的代码库，看到几十个 `.sql` 文件，里面有 `CREATE TABLE`、`INSERT INTO`、`SELECT ... JOIN ... GROUP BY`。你大概能猜到每一句在做什么，但如果让你自己写一个"查过去 30 天消费超过 5000 元的用户，同时显示他们买过的商品类别"的查询，你会怎么写？更关键的是，你写出来的查询，数据库到底是怎么执行的？

这篇文章我们就围绕一个真实的小型电商系统，把 SQL 从里到外讲清楚。不只是语法，更是理解它的思维方式。

## SQL 是一种声明式语言

大多数人第一次接触编程，学的是命令式语言——你告诉计算机"先做这个，再做那个"。比如 JavaScript 里你要遍历一个数组找最大值，你得写循环、写比较、写变量。

SQL 不是这样。SQL 是声明式的：你只描述"我要什么"，至于"怎么做"，交给数据库的查询优化器去决定。

举个具体的例子。假设你想找出价格最高的商品，用命令式思维，你得这样想：

1. 先把所有商品取出来
2. 逐个比较价格
3. 记住最大的那个
4. 返回它

用 SQL，你只需要写：

```sql
SELECT name, price
FROM products
ORDER BY price DESC
LIMIT 1;
```

"给我价格最高的那个商品的名字和价格。"就这样。数据库自己去决定是用索引扫描还是全表扫描，是用归并排序还是快速排序。你不需要关心。

这种思维方式的转变，是写好 SQL 的第一步。你不再是在"编程"，而是在"描述需求"。

## 建房子：DDL/DML/DCL/TCL 各管什么

SQL 不是一个单一的东西，它更像是一套工具箱。我们用"建房子"来类比。

**DDL（Data Definition Language）是打地基、砌墙。** 它定义数据库的结构——表长什么样、列是什么类型、约束是什么。你只在搭建和修改结构时才会用到它。

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    full_name VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

这段 SQL 在做三件事：定义一张叫 `users` 的表，规定它有哪些列，以及每列的约束（`UNIQUE` 保证邮箱不重复，`NOT NULL` 保证字段不能为空，`PRIMARY KEY` 保证每行有唯一标识）。这就像画建筑图纸——一旦定下来，不轻易改动。

**DML（Data Manipulation Language）是往房子里搬东西。** 插入数据、修改数据、删除数据，都是 DML 的活。

```sql
-- 新注册一个用户
INSERT INTO users (email, password_hash, full_name)
VALUES ('zhang_wei@example.com', 'hashed_$2b$12$...', '张伟');

-- 用户修改了姓名
UPDATE users
SET full_name = '张伟华'
WHERE id = 1;
```

这里有个新手常犯的错：写 `UPDATE` 的时候忘记加 `WHERE` 条件。结果就是——整张表都被更新了。没有 `WHERE` 的 `UPDATE` 和 `DELETE` 就像没有目标的喷火器，烧到什么算什么。

**DCL（Data Control Language）是配钥匙。** 它决定谁能进哪间房间。`GRANT` 给权限，`REVOKE` 收权限。

```sql
-- 给客服角色只读访问订单表的权限
GRANT SELECT ON orders TO customer_service;
```

**TCL（Transaction Control Language）是保险箱的开关。** `COMMIT` 确认操作，`ROLLBACK` 撤销操作。上一篇讲 ACID 时提过的事务，就是通过 TCL 控制的。

```sql
BEGIN;
UPDATE products SET stock = stock - 1 WHERE id = 42;
INSERT INTO order_items (order_id, product_id, quantity, unit_price)
VALUES (1001, 42, 1, 299.00);
COMMIT;
```

如果 `INSERT` 失败了，`COMMIT` 之前的一切操作都会被撤销，库存不会白白扣掉。这就是事务的意义。

## 书写顺序和执行顺序不是一回事

很多人写 SQL 是从 `SELECT` 开始写的，因为这是你最关心的部分——"我要查什么"。但数据库不是从 `SELECT` 开始执行的。

拿一个具体的电商查询来说。运营想看"2025 年 3 月每个月的销售额，只看已付款的订单，按月份降序排列，只要销售额超过 10000 的月份"：

```sql
SELECT
    DATE_TRUNC('month', created_at) AS month,
    SUM(total) AS revenue
FROM orders
WHERE status = 'paid'
  AND created_at >= '2025-03-01'
  AND created_at < '2025-04-01'
GROUP BY DATE_TRUNC('month', created_at)
HAVING SUM(total) > 10000
ORDER BY revenue DESC;
```

这段 SQL 你是从上往下写的，但数据库的执行顺序是这样的：

1. **FROM orders** -- 先确定数据源，找到 orders 表
2. **WHERE** -- 过滤行，只保留已付款且在 3 月份的订单
3. **GROUP BY** -- 按月份分组
4. **HAVING** -- 过滤分组，只保留销售额超过 10000 的月份
5. **SELECT** -- 计算每组的汇总值
6. **ORDER BY** -- 按销售额排序

你注意到 `SELECT` 排在第 5 位了吗？这意味着你在 `WHERE` 里不能引用 `SELECT` 里定义的别名。下面这个写法会报错：

```sql
-- 这是错的：WHERE 阶段还不知道 revenue 是什么
SELECT SUM(total) AS revenue
FROM orders
WHERE revenue > 10000;
```

正确的做法是用 `HAVING`，或者把查询包在子查询里。理解了这个执行顺序，很多"为什么这段 SQL 报错"的问题就迎刃而解了。

## JOIN 的本质：基于条件组合行

JOIN 大概是 SQL 里被误解最多的概念。很多人把 JOIN 想象成"把两个表合并到一起"，但实际上 JOIN 做的事情更精确——它基于某个条件，从两个表中各取一行，把它们拼成一行。

想象一下你的电商系统里有这样两张表的数据：

**users 表**：

| id  | email                 | full_name |
| --- | --------------------- | --------- |
| 1   | zhang_wei@example.com | 张伟      |
| 2   | li_na@example.com     | 李娜      |

**orders 表**：

| id  | user_id | order_number      | total   |
| --- | ------- | ----------------- | ------- |
| 101 | 1       | ORD-20250315-a8f3 | 2499.00 |
| 102 | 1       | ORD-20250322-c7d1 | 599.00  |
| 103 | 3       | ORD-20250328-e4b2 | 1299.00 |

注意 user_id = 3 的用户在 users 表里根本不存在（可能已经被删除了），而李娜（id = 2）还没有下过订单。

现在运营想看"每个用户的订单情况"。如果你用 INNER JOIN：

```sql
SELECT u.full_name, o.order_number, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
```

结果只有张伟的两条订单。李娜没有订单所以被过滤掉了，order 103 因为用户不存在也被过滤掉了。INNER JOIN 只保留两边都能匹配上的行。

但运营可能还需要看到"哪些用户没有下过订单"来做营销推送。这时候就得用 LEFT JOIN：

```sql
SELECT u.full_name, o.order_number, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
```

现在李娜也会出现在结果里，只是她的 order_number 和 total 都是 NULL。LEFT JOIN 保证左表（users）的每一行都出现在结果中，哪怕右表（orders）没有匹配的行。

那 order 103 呢？那个 user_id = 3 的订单。如果你需要找出"哪些订单关联的用户已经不存在了"（数据清理场景），你需要 RIGHT JOIN 或者反过来写 LEFT JOIN：

```sql
SELECT o.order_number, o.total, u.full_name
FROM orders o
LEFT JOIN users u ON o.user_id = u.id
WHERE u.id IS NULL;
```

这里 `WHERE u.id IS NULL` 是个常用技巧——LEFT JOIN 之后右表没有匹配的行会是 NULL，所以反过来就能找到"孤儿数据"。

实际开发中，我建议你默认用 LEFT JOIN，只在明确需要排除不匹配的行时才用 INNER JOIN。原因很简单：LEFT JOIN 不会丢数据，而 INNER JOIN 可能会在你不注意的时候把行"吃掉"。

## 子查询 vs CTE：什么时候用哪个

当你的查询变得复杂，你可能会想到"先查出一个中间结果，再基于这个结果继续查"。这就是子查询和 CTE 发挥作用的地方。

假设运营需要一份报表：每个用户的总消费金额，以及他们属于哪个消费等级（普通用户 < 1000，银卡用户 1000-5000，金卡用户 > 5000）。

用子查询可以这么写：

```sql
SELECT
    u.full_name,
    u.email,
    user_stats.total_spent,
    CASE
        WHEN user_stats.total_spent < 1000 THEN '普通用户'
        WHEN user_stats.total_spent < 5000 THEN '银卡用户'
        ELSE '金卡用户'
    END AS level
FROM users u
LEFT JOIN (
    SELECT user_id, SUM(total) AS total_spent
    FROM orders
    WHERE status = 'paid'
    GROUP BY user_id
) user_stats ON u.id = user_stats.user_id
WHERE user_stats.total_spent > 5000;
```

能跑，但嵌套了一层又一层，读起来费劲。同样的逻辑，用 CTE（Common Table Expression，公用表表达式）来写会清晰很多：

```sql
WITH user_spending AS (
    SELECT user_id, SUM(total) AS total_spent
    FROM orders
    WHERE status = 'paid'
    GROUP BY user_id
)
SELECT
    u.full_name,
    u.email,
    us.total_spent,
    CASE
        WHEN us.total_spent < 1000 THEN '普通用户'
        WHEN us.total_spent < 5000 THEN '银卡用户'
        ELSE '金卡用户'
    END AS level
FROM users u
LEFT JOIN user_spending us ON u.id = us.user_id
WHERE us.total_spent > 5000;
```

CTE 用 `WITH ... AS (...)` 的语法，把中间结果取了个名字 `user_spending`，然后在下面的查询里直接引用。好处是显而易见的：每一步做了什么一目了然，像在讲故事一样。

什么时候用子查询，什么时候用 CTE？我的建议是：

- 只需要一层嵌套、逻辑简单的，子查询就行
- 需要多步处理、逻辑有层次的，用 CTE
- CTE 还有一个子查询做不到的好处：可以引用多次。如果你需要基于同一个中间结果做两次不同的计算，CTE 让你只算一次

不过有一点要注意：在 PostgreSQL 里，CTE 默认是一个"优化围栏"——数据库不会把 CTE 里的逻辑"推进"到外层查询里做全局优化。在数据量不大的时候无所谓，但如果 CTE 里涉及几百万行数据，你可能需要加上 `NOT MATERIALIZED` 关键字告诉优化器可以内联处理。

## SQL 注入：为什么拼接字符串写 SQL 是危险的

我们前面写的所有 SQL 都是"静态"的——值直接写在语句里。但在真实的应用里，数据来自用户输入：搜索框里的关键词、表单里的邮箱地址、URL 里的参数。

很多人（尤其是刚入门的后端开发者）会这样做：

```sql
-- 这是最危险的做法，千万别这么写
SELECT * FROM users
WHERE email = '' OR '1'='1';
```

如果用户的邮箱输入框里填的是 `' OR '1'='1`，拼接出来的 SQL 就是上面这样。注意 `WHERE` 后面的条件变成了 `email = '' OR '1'='1'`——永远为真。这意味着这条查询会返回所有用户的数据。

这就是 SQL 注入。2019 年，一个叫 "Breach of Capital One" 的事件就是因为 SQL 注入漏洞导致超过 1 亿条用户数据泄露。这不是教科书里的假设，而是真实发生过的灾难。

正确做法是使用参数化查询。不同的编程语言有不同的写法，但核心思想一样：SQL 结构和数据分离，数据库引擎不会把参数里的内容当作 SQL 指令来执行。

```sql
-- 参数化查询（伪代码展示思路）
PREPARE find_user AS
    SELECT id, email, full_name
    FROM users
    WHERE email = $1;

EXECUTE find_user('zhang_wei@example.com');
```

在实际开发中，你几乎不会直接写 `PREPARE` 和 `EXECUTE`。ORM 框架（比如 TypeORM、Prisma）和查询构建器（比如 Knex.js）会帮你处理好参数化。但理解背后的原理很重要——你不应该信任任何来自用户的输入，永远不要用字符串拼接来构造 SQL。

## 什么时候用 SQL 的什么能力

聊了这么多，最后回到一个实际的问题：在你的项目里，SQL 应该怎么用？

**SQL 最擅长的事情是集合操作**。当你需要"从一组数据里筛选、聚合、关联"的时候，SQL 是效率最高的工具。一个写得好的 SQL 查询，可能比你用代码循环处理几百条记录快几十倍。因为数据库引擎在底层做了索引优化、并行计算、内存管理，这些你手动写代码很难做到同样好。

**SQL 不擅长的事情是复杂的业务流程控制**。如果你发现自己在 SQL 里写了大量的 `CASE WHEN` 嵌套，或者在存储过程里写了上百行的流程逻辑，那可能是时候把一部分逻辑移到应用层了。SQL 适合声明"要什么"，不适合描述"一步步怎么做"。

**CTE vs 子查询的取舍**：CTE 让代码更可读，但在某些极端性能场景下，直接写子查询（甚至不用子查询，而是用 JOIN 一步到位）反而更快。如果你在处理百万级数据，记得用 `EXPLAIN ANALYZE` 看看执行计划，不要只凭直觉优化。

**JOIN 的选择**：前面说了默认用 LEFT JOIN 更安全，但 LEFT JOIN 有一个容易被忽视的代价——如果右表有多条匹配的行，左表的行会被"膨胀"。比如一个用户有 100 个订单，LEFT JOIN 之后这个用户在结果里出现 100 次。如果你只是想统计订单数量，应该用子查询或 CTE 先聚合，再 JOIN。

**关于 `SELECT *`**：开发阶段用 `SELECT *` 没问题，快速验证数据。但上线前一定要改成明确列出需要的列。原因不只是"网络传输多余数据"这么简单——如果你的应用层代码依赖列的顺序，一旦表结构变更（加了一列、改了列的位置），代码就悄悄坏了。

SQL 是一门"入门容易精通难"的语言。基础的增删改查半天就能学会，但写出高效、安全、可维护的查询，需要你对执行原理有深入的理解。后面我们会专门讲 PostgreSQL 的特性，到那时候你会发现，SQL 能做的事情远比你现在想象的要多。
