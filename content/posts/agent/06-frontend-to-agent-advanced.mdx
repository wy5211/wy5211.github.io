# 前端工程师的 Agent 进阶之路（下）：进阶与实战

在上一篇文章中，我们学会了基础的 Agent 开发：调用 LLM API、实现工具调用、构建简单的对话应用。这让我们能够快速上手，但当需求变复杂时，会发现很多挑战。

比如，当 Agent 需要完成一个多步骤的任务时，如何让它自主规划？当工具调用失败时，如何重试？当需要同时使用多个工具时，如何协调？

这些问题需要更高级的架构和技巧。让我们继续完善代码仓库分析 Agent，学习如何构建真正的生产级 Agent。

## 从简单到复杂：为什么需要更强大的架构

先看一个场景。假设我们想让 Agent 做这样的事情："分析 vuejs/vue 这个仓库，重点关注 TypeScript 使用情况、测试覆盖率、以及代码质量。"

这个请求包含多个步骤：

1. 获取仓库基本信息
2. 分析代码结构（文件列表、语言分布）
3. 检查 TypeScript 配置和使用情况
4. 查看测试相关文件和配置
5. 综合以上信息生成报告

用入门阶段的方式，我们需要把这些步骤都硬编码在逻辑中。不够灵活，难以扩展。

更好的方式是让 Agent 自己规划任务。这就是 ReAct（Reasoning + Acting）模式：Agent 先思考需要做什么，然后逐步执行。

## 任务规划：让 Agent 学会分解问题

实现任务规划的核心是给 LLM 一个"思考"的机会。看个例子：

```typescript
// lib/planner.ts
import { openai } from "@ai-sdk/openai";
import { generateText } from "ai";

interface Plan {
  steps: Array<{
    description: string;
    tools: string[];
    dependencies: string[];
  }>;
}

async function createPlan(goal: string): Promise<Plan> {
  const { text } = await generateText({
    model: openai("gpt-4o-mini"),
    system: `你是一个任务规划专家。根据用户的目标，生成一个详细的执行计划。

规划原则：
1. 将复杂目标分解为可管理的小步骤
2. 明确每一步需要使用的工具
3. 标注步骤之间的依赖关系
4. 估算每一步的难度

以 JSON 格式输出计划。`,
    prompt: `目标：${goal}

请生成执行计划，格式如下：
{
  "steps": [
    {
      "description": "步骤描述",
      "tools": ["工具1", "工具2"],
      "dependencies": ["依赖的前置步骤"]
    }
  ]
}`,
  });

  try {
    return JSON.parse(text);
  } catch {
    // 如果解析失败，返回一个默认计划
    return {
      steps: [
        {
          description: goal,
          tools: [],
          dependencies: [],
        },
      ],
    };
  }
}
```

这个规划器能让 Agent 在执行前先思考。但要注意，规划不是一成不变的。Agent 需要在执行过程中根据实际情况调整。

这就引出了下一个概念：动态重规划。

## 动态重规划：适应变化的能力

Agent 在执行过程中可能会遇到各种意外：工具调用失败、获取的数据不符合预期、发现新的需要关注的点。这时就需要调整计划。

```typescript
// lib/executor.ts
interface ExecutionContext {
  plan: Plan;
  currentStep: number;
  results: Map<string, any>;
  errors: Array<{ step: number; error: string }>;
}

class AgentExecutor {
  private tools: Map<string, Function>;

  constructor(tools: Record<string, Function>) {
    this.tools = new Map(Object.entries(tools));
  }

  async execute(goal: string): Promise<string> {
    // 创建初始计划
    let plan = await createPlan(goal);
    let context: ExecutionContext = {
      plan,
      currentStep: 0,
      results: new Map(),
      errors: [],
    };

    // 执行计划，最多重试 3 次
    for (let attempt = 0; attempt < 3; attempt++) {
      try {
        const result = await this.executePlan(context);

        // 检查是否需要重规划
        if (result.needsReplan) {
          plan = await this.replan(goal, context);
          context.plan = plan;
          context.currentStep = 0;
          continue;
        }

        return result.finalAnswer;
      } catch (error) {
        context.errors.push({
          step: context.currentStep,
          error: error instanceof Error ? error.message : "未知错误",
        });

        // 尝试恢复
        const recovered = await this.tryRecover(context);
        if (!recovered) {
          throw error;
        }
      }
    }

    throw new Error("执行失败，超过最大重试次数");
  }

  private async executePlan(
    context: ExecutionContext
  ): Promise<{ finalAnswer: string; needsReplan: boolean }> {
    const { plan } = context;

    for (let i = 0; i < plan.steps.length; i++) {
      context.currentStep = i;
      const step = plan.steps[i];

      // 检查依赖是否满足
      if (!this.checkDependencies(step, context)) {
        continue;
      }

      // 执行步骤
      const stepResult = await this.executeStep(step, context);
      context.results.set(`step-${i}`, stepResult);

      // 检查是否需要调整计划
      if (stepResult.requiresReplan) {
        return { finalAnswer: "", needsReplan: true };
      }
    }

    // 生成最终答案
    const finalAnswer = await this.generateFinalAnswer(context);
    return { finalAnswer, needsReplan: false };
  }

  private async executeStep(
    step: Plan["steps"][0],
    context: ExecutionContext
  ): Promise<any> {
    // 执行该步骤需要的工具
    const results = [];
    for (const toolName of step.tools) {
      const tool = this.tools.get(toolName);
      if (tool) {
        const result = await tool(context);
        results.push(result);
      }
    }

    return {
      description: step.description,
      results,
      requiresReplan: false, // 可以根据结果判断是否需要重规划
    };
  }

  private async replan(goal: string, context: ExecutionContext): Promise<Plan> {
    const { text } = await generateText({
      model: openai("gpt-4o-mini"),
      system: "你是一个任务规划专家。根据执行情况和当前状态，调整执行计划。",
      prompt: `原始目标：${goal}

已完成的步骤：
${Array.from(context.results.entries())
  .map(([key, value]) => `${key}: ${JSON.stringify(value)}`)
  .join("\n")}

遇到的错误：
${context.errors.map((e) => `步骤${e.step}: ${e.error}`).join("\n")}

请调整计划并输出新的 JSON 格式计划。`,
    });

    try {
      return JSON.parse(text);
    } catch {
      return context.plan; // 保持原计划
    }
  }

  // ... 其他辅助方法
}
```

这个执行器实现了动态规划能力。Agent 在执行过程中遇到问题时，可以重新评估和调整计划。这是构建可靠 Agent 的关键。

## 工具链设计：模块化和可复用

随着 Agent 功能增加，工具会越来越多。如何组织这些工具，让它们易于管理和复用？

一个好的做法是将工具按功能分组，每个工具专注于单一职责。看我们的代码仓库分析 Agent 的工具设计：

```typescript
// lib/tools/analysis-tools.ts
export const analysisTools = {
  // 代码质量检测
  async detectCodeQuality(files: any[]) {
    // 检测是否有测试文件
    const hasTests = files.some(
      (f) =>
        f.name.includes(".test.") ||
        f.name.includes(".spec.") ||
        f.name === "__tests__"
    );

    // 检测是否有配置文件
    const hasConfig = files.some(
      (f) =>
        f.name.includes(".eslintrc") ||
        f.name.includes(".prettierrc") ||
        f.name === "tsconfig.json"
    );

    return {
      hasTests,
      hasConfig,
      score: (hasTests ? 50 : 0) + (hasConfig ? 50 : 0),
    };
  },

  // TypeScript 使用分析
  async analyzeTSUsage(files: any[]) {
    const tsFiles = files.filter(
      (f) => f.name.endsWith(".ts") || f.name.endsWith(".tsx")
    );
    const jsFiles = files.filter(
      (f) => f.name.endsWith(".js") || f.name.endsWith(".jsx")
    );

    const total = tsFiles.length + jsFiles.length;
    const tsPercentage = total > 0 ? (tsFiles.length / total) * 100 : 0;

    return {
      tsFiles: tsFiles.length,
      jsFiles: jsFiles.length,
      tsPercentage,
      isTypeScriptProject: tsPercentage > 50,
    };
  },

  // 依赖分析
  async analyzeDependencies(owner: string, repo: string) {
    try {
      const response = await fetch(
        `https://api.github.com/repos/${owner}/${repo}/contents/package.json`
      );

      if (!response.ok) {
        return { error: "无法读取 package.json" };
      }

      const data = await response.json();
      const content = atob(data.content);
      const pkg = JSON.parse(content);

      return {
        dependencies: Object.keys(pkg.dependencies || {}).length,
        devDependencies: Object.keys(pkg.devDependencies || {}).length,
        hasTypeScript: !!pkg.devDependencies?.typescript,
      };
    } catch {
      return { error: "解析失败" };
    }
  },
};
```

这种模块化设计有几个好处：

首先是职责清晰。每个工具做一件事，容易理解和测试。其次是易于扩展。添加新功能时，只需要添加新工具，不影响现有代码。最后是可复用。这些工具可以在不同 Agent 之间共享。

## 错误处理：让 Agent 更健壮

在真实环境中，API 调用会失败，网络会中断，数据会缺失。Agent 必须能优雅地处理这些情况。

```typescript
// lib/resilient-agent.ts
interface RetryConfig {
  maxRetries: number;
  initialDelay: number;
  maxDelay: number;
  backoffMultiplier: number;
}

class ResilientAgent {
  private retryConfig: RetryConfig = {
    maxRetries: 3,
    initialDelay: 1000,
    maxDelay: 10000,
    backoffMultiplier: 2,
  };

  async executeWithRetry<T>(fn: () => Promise<T>, context: string): Promise<T> {
    let lastError: Error | null = null;
    let delay = this.retryConfig.initialDelay;

    for (let attempt = 0; attempt < this.retryConfig.maxRetries; attempt++) {
      try {
        return await fn();
      } catch (error) {
        lastError = error instanceof Error ? error : new Error(String(error));

        // 判断是否值得重试
        if (!this.isRetryable(error)) {
          throw lastError;
        }

        console.warn(
          `[${context}] 尝试 ${attempt + 1}/${this.retryConfig.maxRetries} 失败: ${lastError.message}`
        );

        // 等待后重试
        await this.sleep(delay);
        delay = Math.min(
          delay * this.retryConfig.backoffMultiplier,
          this.retryConfig.maxDelay
        );
      }
    }

    throw lastError;
  }

  private isRetryable(error: any): boolean {
    // 429 Too Many Requests 可以重试
    if (error?.status === 429) return true;
    // 5xx 服务器错误可以重试
    if (error?.status >= 500) return true;
    // 网络错误可以重试
    if (error?.code === "ECONNRESET") return true;
    // 其他错误不重试
    return false;
  }

  private sleep(ms: number): Promise<void> {
    return new Promise((resolve) => setTimeout(resolve, ms));
  }

  // 带降级的执行
  async executeWithFallback<T>(
    primary: () => Promise<T>,
    fallback: () => Promise<T>,
    context: string
  ): Promise<T> {
    try {
      return await primary();
    } catch (error) {
      console.warn(`[${context}] 主方法失败，使用降级方案: ${error}`);
      return await fallback();
    }
  }
}
```

这个可恢复的 Agent 实现了几个关键模式：

指数退避重试，避免在服务压力大时持续请求。智能重试判断，只对可恢复的错误重试，不浪费时间。降级策略，当主方法失败时，使用备用方案保证功能可用。

## 实战：增强版代码仓库分析 Agent

让我们用学到的技巧，构建一个更强大的代码仓库分析 Agent。

```typescript
// app/api/analyze-advanced/route.ts
import { openai } from "@ai-sdk/openai";
import { generateText, tool } from "ai";
import { z } from "zod";
import { githubTools } from "@/lib/tools/github-tools";
import { analysisTools } from "@/lib/tools/analysis-tools";
import { ResilientAgent } from "@/lib/resilient-agent";

const resilient = new ResilientAgent();

export async function POST(req: Request) {
  const { repoUrl, focus } = await req.json();

  // 解析仓库 URL
  const match = repoUrl.match(/github\.com\/([^/]+)\/([^/]+)/);
  if (!match) {
    return Response.json({ error: "无效的 GitHub URL" }, { status: 400 });
  }

  const [, owner, repo] = match;

  // 定义工具
  const tools = {
    getRepoInfo: tool({
      description: "获取 GitHub 仓库的基本信息",
      parameters: z.object({
        owner: z.string(),
        repo: z.string(),
      }),
      execute: async ({ owner, repo }) => {
        return resilient.executeWithRetry(
          () => githubTools.getRepoInfo(owner, repo),
          "getRepoInfo"
        );
      },
    }),

    getCodeStructure: tool({
      description: "获取仓库的代码结构（文件和目录列表）",
      parameters: z.object({
        owner: z.string(),
        repo: z.string(),
        path: z.string().optional().default(""),
      }),
      execute: async ({ owner, repo, path }) => {
        return resilient.executeWithRetry(
          () => githubTools.getCodeStructure(owner, repo, path),
          "getCodeStructure"
        );
      },
    }),

    getLanguageStats: tool({
      description: "获取仓库的编程语言统计",
      parameters: z.object({
        owner: z.string(),
        repo: z.string(),
      }),
      execute: async ({ owner, repo }) => {
        return resilient.executeWithRetry(
          () => githubTools.getLanguageStats(owner, repo),
          "getLanguageStats"
        );
      },
    }),

    analyzeCodeQuality: tool({
      description: "分析代码质量（测试、配置文件等）",
      parameters: z.object({
        files: z.array(z.any()),
      }),
      execute: async ({ files }) => {
        return await analysisTools.detectCodeQuality(files);
      },
    }),

    analyzeTypeScriptUsage: tool({
      description: "分析 TypeScript 的使用情况",
      parameters: z.object({
        files: z.array(z.any()),
      }),
      execute: async ({ files }) => {
        return await analysisTools.analyzeTSUsage(files);
      },
    }),
  };

  // 构建分析提示
  const focusArea = focus || "整体情况";
  const systemPrompt = `你是一个专业的代码仓库分析专家。你能够深入分析 GitHub 仓库，提供有价值的见解。

分析时关注：
1. 项目的目的和价值（基于描述和 README）
2. 代码质量和工程实践
3. 技术栈和架构选择
4. 社区活跃度和健康状况
5. ${focusArea}

提供具体、有数据支持的分析。`;

  try {
    const { text } = await generateText({
      model: openai("gpt-4o"),
      system: systemPrompt,
      tools,
      toolChoice: "auto",
      maxSteps: 10, // 允许多步骤执行
      prompt: `请深入分析这个 GitHub 仓库：${repoUrl}

关注重点：${focusArea}

请提供一个全面的分析报告，包括：
1. 项目概览
2. 代码质量评估
3. 技术栈分析
4. 社区活跃度
5. 改进建议`,
    });

    return Response.json({
      analysis: text,
      repo: `${owner}/${repo}`,
      timestamp: new Date().toISOString(),
    });
  } catch (error) {
    console.error("分析失败:", error);
    return Response.json({ error: "分析失败，请稍后重试" }, { status: 500 });
  }
}
```

这个增强版的 Agent 有几个改进：

使用了多步骤执行（`maxSteps: 10`），让 Agent 能够自主决定需要调用哪些工具、按什么顺序调用。加入了错误处理和重试机制，提高稳定性。支持用户指定关注点，分析更有针对性。使用了更强的模型（`gpt-4o`），分析质量更好。

## 进阶阶段的权衡

进入进阶阶段后，我们的 Agent 变得更强大，但也引入了复杂度。

任务规划让 Agent 能处理复杂场景，但规划本身可能出错，而且增加了 LLM 调用次数。错误处理提高了稳定性，但代码量增加，调试难度提高。工具模块化让代码可复用，但过度模块化会增加抽象层次。

关键是要根据实际需求平衡。不是所有 Agent 都需要复杂的规划，不是所有工具调用都需要重试。
