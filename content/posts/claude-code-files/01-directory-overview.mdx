用了很久 Claude Code，你大概率从没认真打开过 `~/.claude/` 这个目录。

它就躺在你的家目录下，Claude Code 每次启动都会读它。里面密密麻麻一堆文件和文件夹——`settings.json`、`CLAUDE.md`、`commands/`、`agents/`、`skills/`、`hooks/`——每个名字都见过，却说不清它们各自干嘛、什么时候被读取、彼此怎么协作。

这个系列要做的就是一次"目录考古"：以文件系统为线索，把 Claude Code 的目录结构、高频配置文件的作用、加载与合并机制、扩展体系，一次性讲透。这是第 1 篇，我们先建立一张完整的地图——搞清楚 Claude Code 到底有"几个家"，每个家装了什么。

## 为什么 Claude Code 要分两个目录

很多新手以为 Claude Code 把所有配置都塞在一个地方。其实它有**两个家**：

- **全局家**：`~/.claude/`——跨所有项目共享的配置、记忆、扩展
- **项目家**：`<你的项目>/.claude/`——只对当前项目生效的配置和扩展

为什么要分两层？因为有些东西是你个人的偏好（比如"我永远用 pnpm，不要用 npm"），有些东西是某个项目的约定（比如"这个项目用 yarn"）。如果你把所有配置都塞全局，换一个项目就冲突；如果都塞项目，换个机器又要重新配一遍。

分层的好处是：**个人偏好走全局，项目约定走项目，团队共享走项目且进 git，纯本地的不进 git。**

接下来我们逐个看这两个家里都有什么。

## 全局家：`~/.claude/`

打开你的终端，执行 `ls ~/.claude/`，你会看到类似这样的内容：

```bash
$ ls ~/.claude/
CLAUDE.md          # 你的全局记忆（跨项目通用指令）
settings.json      # 全局设置（权限、模型、hooks 等）
commands/          # 你个人的斜杠命令
agents/            # 你个人的子代理
skills/            # 你个人的技能
hooks/             # （部分场景的 hooks 配置目录）
projects/          # 所有项目的会话历史
memory/            # 跨会话的持久记忆
shell-snapshots/   # shell 环境快照（冷门）
statsig/           # 灰度实验配置（冷门）
plugins/           # 安装的插件
keybindings.json   # 键位绑定
todos/             # 任务记录
```

第一次看到这一堆文件，谁都头大。但如果你按"使用频率"分个档，立刻就清晰了。

### 🔥 高频：天天在用的核心文件

这些是你日常每个会话都在跑的核心，必须搞懂，本系列后续会单独成篇细讲：

| 文件/目录       | 作用                                           | 本系列第几篇 |
| --------------- | ---------------------------------------------- | ------------ |
| `settings.json` | 全局设置：权限规则、默认模型、hooks 注册       | 第 4 篇      |
| `CLAUDE.md`     | 全局记忆：跨所有项目生效的指令（你的个人偏好） | 第 3 篇      |
| `commands/`     | 你的个人斜杠命令，任何项目都能用               | 第 5 篇      |
| `agents/`       | 你的个人子代理，比如"代码审查专家"             | 第 7 篇      |
| `skills/`       | 你的个人技能，按需自动激活                     | 第 6 篇      |

举个例子：你在 `~/.claude/commands/` 下放一个 `ship.md`，那么在**任何项目**里输入 `/ship` 都能触发这个命令。这就是全局家的威力——配置一次，到处生效。

### ⚡ 中频：偶尔会用到

| 文件/目录   | 作用                                                                                  |
| ----------- | ------------------------------------------------------------------------------------- |
| `projects/` | 所有项目的会话历史，每个项目一个子目录，存着转录文件。`--resume` 找回会话就是从这里读 |
| `memory/`   | 跨会话的持久记忆，Claude 自己写入的"长期记忆"（区别于 CLAUDE.md 这个你手写的记忆）    |
| `todos/`    | 任务列表记录，对应会话里的 todo 功能                                                  |

`projects/` 和 `memory/` 我们在第 10 篇（上下文与会话）会专门讲。这里只需要知道：**你的会话历史和 Claude 的长期记忆都存在全局家**。

### ❄ 冷门：知道存在就行，几乎不用碰

| 文件/目录          | 作用                                                         |
| ------------------ | ------------------------------------------------------------ |
| `shell-snapshots/` | shell 环境快照，用来保证 Bash 工具能正确继承你的 PATH 和别名 |
| `statsig/`         | 灰度实验配置缓存，Claude Code 用来决定给你开放哪些实验性功能 |
| `plugins/`         | 通过 `/plugin` 安装的插件存放处                              |
| `keybindings.json` | 自定义快捷键绑定                                             |

本系列**故意不细讲这些**。它们是 Claude Code 的内部实现细节，普通用户一辈子可能都不需要手动改。`shell-snapshots` 解决的是"为什么我的 alias 在 Bash 工具里不生效"这种问题，`statsig` 是产品团队做灰度用的。你只要知道它们在那里、不要手动删就行。

> 💡 一个判断标准：如果一个文件/目录你不知道是干嘛的，且不在"高频"清单里——大概率就是冷门的内部机制，忽略它。

## 项目家：`<项目>/.claude/`

进入任何一个用 Claude Code 的项目，它的根目录下通常有个 `.claude/` 文件夹：

```bash
$ ls .claude/
settings.json          # 团队共享的项目设置（进 git）
settings.local.json    # 你的本地设置（不进 git）
CLAUDE.md              # 项目记忆（项目约定，通常进 git）
commands/              # 项目专属命令
agents/                # 项目专属子代理
skills/                # 项目专属技能
hooks/                 # 项目专属 hooks（或写在 settings.json 里）
```

注意三个关键点：

**1. 大部分内容跟全局家长得一样。**`commands/`、`agents/`、`skills/`、`CLAUDE.md`、`settings.json` 这些名字你在全局家也见过。区别在于**作用域**：项目家里的只对当前项目生效。

**2. 多了一个 `settings.local.json`。**这是项目家里最特殊的一个文件，下一节单独讲。

**3. `CLAUDE.md` 在项目里通常放在项目根目录（`./CLAUDE.md`），而不是 `.claude/` 里。**两种位置 Claude Code 都会读，但约定俗成放在根目录更显眼，方便团队成员一眼看到。

### `settings.json` vs `settings.local.json`：哪个进 git？

这是项目家里最容易踩坑的一对文件。规则很简单：

```bash
# 项目根目录的 .claude/
settings.json        # ✅ 进 git，团队所有人共享
settings.local.json  # ❌ 不进 git，只对你自己生效
```

为什么要有两个？因为有些设置是**团队约定**（比如"这个项目所有人都不允许直接 push 到 main"），有些是**个人偏好**（比如"我个人习惯让 Claude 自动跑测试，别人不一定喜欢"）。

- 团队约定 → 写进 `settings.json`，提交到 git，所有人 clone 下来就生效
- 个人偏好 → 写进 `settings.local.json`，自动被 `.gitignore` 忽略（Claude Code 会帮你加），只影响你自己

举个真实场景：你们团队规定，在这个项目里，Claude 不允许删除任何文件（防止误删）。这条规则应该写进 `settings.json`：

```json
// .claude/settings.json —— 进 git，团队共享
{
  "permissions": {
    "deny": ["Bash(rm -rf *)", "Bash(git push --force)"]
  }
}
```

而你自己想给某些命令免确认，属于个人偏好，写进 `settings.local.json`：

```json
// .claude/settings.local.json —— 不进 git，只对你生效
{
  "permissions": {
    "allow": ["Bash(npm run test)", "Bash(npm run lint)"]
  }
}
```

这两个文件会在第 4 篇（权限系统）详细讲，这里先建立这个直觉：**共享进 git，本地不进 git。**

## 两层目录的职责对比表

把全局家和项目家放在一起对比，职责分工就很清楚了：

| 配置项                         | 全局家 `~/.claude/` | 项目家 `.claude/`      | 该不该进 git        |
| ------------------------------ | ------------------- | ---------------------- | ------------------- |
| 设置 `settings.json`           | 你的个人偏好/权限   | 团队共享的设置         | 项目设置进 git      |
| 本地设置 `settings.local.json` | —                   | 你对这个项目的个人偏好 | ❌ 不进 git         |
| 记忆 `CLAUDE.md`               | 跨项目的个人指令    | 这个项目的约定         | 项目级通常进 git    |
| 命令 `commands/`               | 你的通用命令        | 项目专属命令           | 项目级进 git        |
| 子代理 `agents/`               | 你的通用代理        | 项目专属代理           | 项目级进 git        |
| 技能 `skills/`                 | 你的通用技能        | 项目专属技能           | 项目级进 git        |
| 会话历史 `projects/`           | 所有项目都在这      | —                      | ❌ 不进 git（个人） |
| 记忆 `memory/`                 | Claude 的长期记忆   | —                      | ❌ 不进 git         |

几个判断原则：

1. **跨项目通用的 → 全局家**。比如"我喜欢用 gitmoji 风格的 commit"，跟项目无关，放全局。
2. **跟某个项目绑定的 → 项目家**。比如"这个项目用 pnpm，禁止 npm"，放项目。
3. **团队都要遵守的 → 项目家且进 git**。让 clone 下来的人自动继承。
4. **只你自己要的、带敏感信息的 → `settings.local.json`**。比如本地的 API key、个人路径。

## 哪些绝对不能进 git

这个要单独强调，因为泄露了很麻烦。在项目根目录的 `.gitignore` 里，至少要确保以下内容不进 git：

```gitignore
# .gitignore

# Claude Code 本地设置（可能含本地路径、密钥）
.claude/settings.local.json

# 整个全局家本来就不在项目里，无需担心
```

实际上 Claude Code 在你创建 `settings.local.json` 时会自动帮你加这条 gitignore。但如果你是手动建的，记得自己加。

另外，**绝对不要在 `CLAUDE.md` 或 `settings.json`（进 git 的那些）里写密钥**。需要密钥时，用环境变量，并通过 `settings.local.json` 引用环境变量名（而不是值）。

## 实战：给你的第一个项目搭一套合理的目录结构

假设你刚接手一个新项目，想用 Claude Code 把它配置好。照着这个流程来：

**第 1 步：建项目家，先确定共享 vs 本地。**

```bash
cd your-project
mkdir -p .claude/commands .claude/agents

# 团队共享（进 git）
touch .claude/settings.json
touch CLAUDE.md           # 放根目录，更显眼

# 你的本地（不进 git）
touch .claude/settings.local.json
```

**第 2 步：在 `CLAUDE.md` 写最核心的项目约定。**

```markdown
# 项目约定

- 包管理器：pnpm（禁止使用 npm/yarn）
- 测试：vitest，运行 `pnpm test`
- 代码风格：遵循 .eslintrc，提交前跑 `pnpm lint`
- 提交规范：conventional commits
```

**第 3 步：在 `.claude/settings.json` 写团队硬规则。**

```json
{
  "permissions": {
    "deny": ["Bash(rm -rf *)", "Bash(git push --force main)"]
  }
}
```

**第 4 步：把团队该有的命令、代理放进 `.claude/`。**

比如 `/ship` 命令、`code-reviewer` 代理——这些我们在第 5、7 篇会讲怎么写。

**第 5 步：把项目提交到 git，让同事 clone 后自动继承。**

```bash
git add .claude/settings.json CLAUDE.md .claude/commands/ .claude/agents/
git commit -m "chore: 配置 Claude Code 项目家"
```

同事 clone 下来，立刻就拥有了相同的权限规则、命令和代理——不需要任何人手动配。

## 系列衔接：这些文件，谁先加载？

到这里你已经有了完整的目录地图，知道每个家里装了什么。但马上会冒出一个新问题：

> 我在全局 `~/.claude/settings.json` 和项目 `.claude/settings.json` 里都写了规则，到底谁说了算？

同样的，`CLAUDE.md` 全局和项目都有，怎么合并？`commands/`、`agents/`、`skills/` 全局和项目都有，是覆盖还是叠加？

这就是下一篇文章的主题——**配置加载机制：合并、覆盖、优先级**。搞清楚它，你才能精准控制"改一个配置到底生效在哪一层"，避免"明明配了却不生效"的诡异 bug。

记住这一篇的核心结论就够了：**Claude Code 有两个家，全局家管个人偏好，项目家管项目约定；共享的进 git，本地的进 `settings.local.json`。**
