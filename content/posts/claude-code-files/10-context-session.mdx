这是系列的最后一篇。

前九篇我们逐一拆解了 Claude Code 的高频机制——目录结构、加载合并、CLAUDE.md、权限、commands、skills、agents、MCP、hooks。每一个都是独立的零件。

但零件不会自己跑。这一篇我们要把它们装成一台完整的机器，看清它们怎么协作，以及一个每个人都遇到过的问题：

> 会话聊了几百轮后，Claude 开始"忘事"——忘了早期的约定、忘了项目背景、重复问已经讨论过的事。这是怎么回事？怎么治？

答案的核心是一个词：**上下文**。

## 所有东西都在抢同一个上下文窗口

Claude Code 每次会话有一个**上下文窗口**（token 上限）。这个窗口不是无限大的，里面要塞进所有这些内容：

```
上下文窗口（固定大小）
├── CLAUDE.md 内容（用户级 + 项目级，启动就加载）
├── 已激活 skill 的 description（始终占位）
├── 匹配触发的 SKILL.md 正文（按需加载）
├── 系统提示词（Claude Code 自己的指令）
├── 对话历史（你说的话 + Claude 的回复）
├── 工具调用结果（读的文件内容、命令输出、MCP 返回）
└── 当前任务的相关信息
```

**所有这些挤在同一个窗口里。**当窗口满了，最早的内容会被"挤出去"或被压缩——这就是 Claude"忘事"的根源。

理解了这一点，前面九篇的所有机制都有了统一的解释：

- **CLAUDE.md 要精简** → 因为它一直占着窗口，太长就稀释核心指令
- **skill 用渐进式披露** → 为了按需加载，不浪费窗口
- **agent 用独立上下文** → 把中间过程隔离出去，不污染主窗口
- **权限规则要聚焦** → 不必把所有规则都塞进窗口（settings 是规则层，不进对话上下文，但相关的会让 Claude 读到）

每一个设计都在为"省上下文"或"用好上下文"服务。

## 上下文压缩：何时触发、保留了什么、丢了什么

聊久了上下文满了怎么办？Claude Code 会触发**上下文压缩（compaction）**——把早期的对话摘要成一段更短的总结，腾出空间。

### 压缩何时触发

通常是上下文用量达到某个阈值时（比如接近上限的某个比例），Claude Code 自动触发压缩。你也可以手动用 `/compact` 触发。

### 压缩保留了什么、丢了什么

压缩不是等比缩小，而是**有选择地保留**：

| 内容                   | 压缩后                    |
| ---------------------- | ------------------------- |
| 核心约定（CLAUDE.md）  | ✅ 保留（本来就单独注入） |
| 关键决策、结论         | ✅ 保留（摘要里）         |
| 任务目标               | ✅ 保留                   |
| 早期对话的细节、过程   | ❌ 丢失（被摘要替代）     |
| 工具调用的大量中间输出 | ❌ 丢失（被摘要替代）     |
| 未完成的 todo          | ✅ 保留                   |

**问题就在这里：压缩是"有损"的。**摘要可能漏掉一些早期细节，比如"我们三天前讨论过用 A 方案不是 B 方案"，如果这个决策没被显式记下来，压缩后 Claude 可能重新考虑 B 方案。

这就是"忘事"的本质——**不是 Claude 故意忘，是压缩把细节摘要掉了。**

## memory 文件系统：跨会话的持久记忆

第 1 篇提到 `~/.claude/memory/` 这个目录，现在讲清楚它。

CLAUDE.md 是**你手写**的记忆，每次都在。但有些事你不方便写进 CLAUDE.md（比如"这个用户喜欢简洁回答""这个 bug 的根因是 X"），Claude Code 自己会把这些**观察到的、值得长期记住的事**写进 memory 文件系统。

memory 是**跨会话的持久记忆**：

- **手动写入**：你让 Claude"记住这个项目用 pnpm"，它会写进 memory
- **自动写入**：Claude 在工作中发现某些值得记的事，会自己写入 memory

下次会话启动，memory 的内容会被读进上下文（通过摘要或按需），Claude 就"记得"之前的事了。

**memory vs CLAUDE.md 的区别：**

| 维度   | CLAUDE.md          | memory                       |
| ------ | ------------------ | ---------------------------- |
| 谁写   | 你手写             | Claude 写（偶尔你让它写）    |
| 内容   | 项目约定、个人偏好 | 工作中的观察、事实、用户偏好 |
| 加载   | 每次全量注入       | 按相关性按需召回             |
| 进 git | 项目级通常进       | 不进 git，个人化             |

简单说：**CLAUDE.md 是"规则手册"，memory 是"工作笔记"。**前者你写，后者 Claude 写。

## 会话恢复：找回之前的会话

会话关了再开，怎么接续之前的工作？Claude Code 提供两个方式：

### `--continue`：继续最近的会话

```bash
claude --continue
```

恢复当前项目最近的那个会话，带着完整上下文。适合"我刚关了终端，想接着干"。

### `--resume`：选择恢复哪个会话

```bash
claude --resume
```

列出当前项目的历史会话，你选一个恢复。适合"我想回到三天前那个会话"。

会话历史存在 `~/.claude/projects/<项目名>/` 下（第 1 篇讲过）。每次会话的转录都在那里，`--resume` 就是从这些转录里恢复。

**重要：会话恢复能找回上下文，但如果会话太长被压缩过，恢复的也是压缩后的状态。**所以重要决策最好显式记进 CLAUDE.md 或 memory，别只依赖会话历史。

## 管理上下文的实用技巧

既然上下文是稀缺资源，怎么用好它？几个实用建议：

### 1. 长会话适时开新会话

一个会话聊了几百轮，上下文里堆了大量历史，Claude 注意力分散，开始忘事、跑偏。**这时候开新会话**，把关键结论写进 CLAUDE.md 或 memory，新会话从干净的上下文开始。

判断信号：Claude 开始重复问已讨论过的问题、忽略近期的指令、回答质量下降。这些是"上下文太满"的信号。

### 2. 用 `/clear` 清空当前上下文

不想开新会话，但当前上下文太乱，可以用 `/clear` 清空对话历史（保留 CLAUDE.md 和配置）。适合"换个任务，不要被上一个任务的历史干扰"。

### 3. 用子代理隔离大任务

回顾第 7 篇——大任务派给子代理，中间过程不污染主对话。这是保护主上下文最有效的手段。审查大代码库、做大量探索，都该用子代理。

### 4. CLAUDE.md 保持精简

CLAUDE.md 不是越全越好，是越准越好。定期 review，删掉过时的、矛盾的、太啰嗦的。核心约定放最前面，详细的拆到 `@import` 的子文件或 skill 里。

### 5. 关键决策显式记录

重要的决定（"用 A 方案""这个 bug 的根因""这个模块的设计意图"），显式让 Claude 写进 memory 或 CLAUDE.md。别指望靠对话历史记住——压缩一来可能就丢了。

## 系列收尾：一条完整的数据流

现在我们把前九篇串起来，看一次完整的会话从启动到结束，所有机制怎么协作：

```
1. 会话启动
   │
   ├─ 加载配置（第2篇）
   │   全局 → 项目 → 本地，按字段合并
   │
   ├─ 加载 CLAUDE.md（第3篇）
   │   用户级 + 项目级，注入上下文
   │
   └─ 加载所有 skill 的 description（第6篇）
       轻量占位，等待触发
   │
   ▼
2. 你输入任务 / 触发命令（第5篇）
   │
   ├─ 命令展开（$ARGUMENTS、!shell、@file）
   │
   └─ 任务匹配 skill → 加载 SKILL.md 正文（第6篇）
   │
   ▼
3. Claude 决定调用工具
   │
   ├─ 经过权限检查（第4篇）
   │   deny > ask > allow，更具体的规则优先
   │
   ├─ PreToolUse hook 先跑（第9篇）
   │   可能拦截、修改
   │
   ├─ 工具执行
   │   ├─ 内置工具（Read/Write/Bash）
   │   ├─ MCP 工具（第8篇，mcp__server__tool）
   │   └─ 派发子代理（第7篇，独立上下文，只回结论）
   │
   └─ PostToolUse hook 后跑（第9篇）
       自动格式化、跑检查
   │
   ▼
4. 响应结束 → Stop hook（第9篇）
   │   可能跑测试、最终验证
   │
   ▼
5. 上下文累积
   │
   ├─ 接近上限 → 自动压缩（这一篇）
   │   摘要早期内容，腾出空间
   │
   └─ 关键事实 → 写入 memory（这一篇）
       跨会话持久化
   │
   ▼
6. 会话结束
   │
   └─ 转录存到 ~/.claude/projects/（第1篇）
       下次 --resume 可恢复
```

**这就是 Claude Code 作为一个系统的完整运转方式。**每一个文件、每一个配置、每一个扩展，都在这条数据流里有自己的位置。理解了这条流，你就理解了 Claude Code 的全部。

## 结尾：日常使用的三个 trade-off

整个系列讲完，最后给你三个日常使用中的核心权衡，作为收尾：

### trade-off 1：什么时候信任自动压缩，什么时候主动开新会话

- **信任自动压缩**：单次任务、连续工作、不想丢上下文。压缩能让你继续干。
- **主动开新会话**：任务切换、上下文明显太满、压缩后 Claude 开始跑偏。开新会话 + 记录关键结论到 memory，比硬撑着长会话更高效。

### trade-off 2：什么时候用 CLAUDE.md，什么时候用 skill

- **CLAUDE.md**：始终生效的核心约定、项目背景。短小精悍，放最关键的。
- **skill**：按场景激活的专业知识。内容多、只在特定场景用、不想一直占上下文。

判断标准：这条规则是不是"每次都要生效"？是 → CLAUDE.md；否 → skill。

### trade-off 3：什么时候直接做，什么时候派子代理

- **直接做**：小任务、强依赖当前上下文、需要来回讨论。
- **派子代理**：大任务、独立可并行、产生大量中间过程。

派子代理有开销，但保护主上下文。任务够大就值得。

## 系列结语

十篇文章，从"打开 `~/.claude/` 目录被吓到"，到"理解整个系统如何运转"。我们走过的路径：

**地图 → 加载 → 记忆 → 权限 → 命令 → 技能 → 代理 → MCP → hooks → 上下文。**

每一个高频机制都单独成篇细讲，冷门机制（worktrees 细节、session 转录、statsig 等）刻意不写。因为前者是你每次会话都在用的核心，后者是内部实现，普通用户不需要碰。

记住这一篇的核心洞察就够了：**Claude Code 是一个文件系统驱动的系统，所有配置、记忆、扩展、自动化都落在文件上；而这些机制最终都服务于同一件事——在一个有限的上下文窗口里，让 Claude 高效、安全、可控地工作。**

理解了文件，就理解了 Claude Code。希望这个系列帮你从"会用"进阶到"懂原理"。
