扩展三件套讲到最后一站：**agents（子代理）**。

前两篇的 command 和 skill 都在**主对话上下文**里工作——加载它们的指令会占用你当前会话的上下文窗口。agent 不一样，它有**自己独立的上下文**。这一个区别，决定了 agent 适合干 command 和 skill 干不了的活。

## 子代理解决什么问题

设想这个场景：你让 Claude 审查一个超大代码库，几千个文件。Claude 开始一个个读，读了三分之一，主对话上下文爆了，它开始"忘事"——忘了最初的审查标准，忘了前面发现的问题。

或者这个场景：你让 Claude 同时做三件独立的事——审查代码、写测试、查文档。三件事的中间过程全堆在主对话里，互相干扰，上下文很快乱成一锅粥。

**子代理就是解决这两个问题的：**

1. **上下文隔离**——子代理有独立的上下文窗口，干完活只把结论返回主对话，中间过程不污染主上下文。
2. **专业分工**——可以定义专门的子代理，比如"代码审查专家""测试工程师""文档查找员"，各司其职。

主对话像项目经理，子代理像被派出去的专员——专员在自己工位上干活，干完回来汇报结论，项目经理的桌面不会被专员的草稿纸堆满。

## agent 文件结构

agent 也是 Markdown 文件，放在 `agents/` 目录下，文件名即 agent 名：

```
~/.claude/agents/
├── code-reviewer.md     →  code-reviewer 代理
├── test-writer.md       →  test-writer 代理
└── doc-finder.md        →  doc-finder 代理
```

一个 agent 文件长这样：

```markdown
## <!-- ~/.claude/agents/code-reviewer.md -->

name: code-reviewer
description: 代码审查专家。审查代码的正确性、安全性、可维护性，只读不改。
tools: Read, Grep, Glob, Bash
model: sonnet

---

你是一个资深代码审查专家。

## 你的职责

- 审查给定代码的正确性、安全性、可维护性
- 按严重级别输出问题：CRITICAL / HIGH / MEDIUM / LOW
- 只审查，不修改代码

## 审查重点

1. 安全漏洞（注入、XSS、密钥泄露）
2. 逻辑错误和边界情况
3. 性能问题（N+1 查询、无界循环）
4. 可维护性（命名、复杂度、重复）

## 输出格式

按级别分组列出问题，每个问题包含：位置、描述、建议修复方案。
```

### frontmatter 字段

| 字段          | 作用                                 |
| ------------- | ------------------------------------ |
| `name`        | 代理名，文件名一致                   |
| `description` | 描述，主对话靠它判断何时派这个代理   |
| `tools`       | **工具白名单**——这个代理能用哪些工具 |
| `model`       | 用哪个模型（sonnet / opus / haiku）  |

其中 `tools` 是 agent 区别于 skill/command 的第二个关键特性（第一个是独立上下文）。

## 工具隔离：最小权限原则

子代理默认能用所有工具，但你可以用 `tools` 字段做**白名单限制**，只给它需要的工具。这是最小权限原则——只给干活必须的工具，不给多余的。

```markdown
# 一个只读的代码审查代理

tools: Read, Grep, Glob

# 只能读、搜，不能写、不能执行命令
```

```markdown
# 一个测试编写代理

tools: Read, Write, Edit, Bash

# 能读写测试文件、能跑测试
```

工具白名单的好处：

1. **安全**——审查代理没有 Write 权限，绝对不会误改代码。
2. **可控**——给一个查文档的代理 Bash 权限没必要，限制了就不会乱跑命令。
3. **明确职责**——工具集本身就是职责的声明。

**写 agent 时养成习惯：先想清楚它需要哪些工具，写到白名单里。**不要让它拥有比职责更多的能力。

## 上下文隔离：只把结论带回来

这是 agent 最有价值的特性。子代理工作时的全过程——读文件、思考、试错——都在它自己的上下文里，主对话看不到也存不下。

主对话只收到子代理**最终的汇报**（一段文字）。看个对比：

**不用子代理（在主对话里直接做）：**

```
主对话上下文：
  [你的指令]
  [Claude 读文件1的内容]      ← 占用
  [Claude 读文件2的内容]      ← 占用
  [Claude 读文件3的内容]      ← 占用
  [Claude 的思考过程]         ← 占用
  [Claude 的结论]             ← 占用

  → 主对话上下文被中间过程塞满
```

**用子代理：**

```
主对话上下文：
  [你的指令]
  [派发给 code-reviewer]     ← 很小
  [code-reviewer 的结论]     ← 只占结论

子代理独立上下文：
  [读文件1、2、3]
  [思考、试错]
  [生成结论]                 → 只把结论交回主对话

  → 主对话上下文干净，只多了结论
```

**差别巨大。**审查 100 个文件，不用子代理主对话早爆了；用子代理，主对话只多了一段审查结论。这就是为什么"大任务分块"要用子代理。

## 何时用子代理，何时不用

子代理有好处，但也有开销（启动子代理、加载上下文都要时间和 token）。不是所有任务都该用。

### 该用子代理

- **大任务分块**——审查整个代码库、迁移大量文件、批量改写。主对话装不下，分给子代理一块块处理。
- **专业分工**——审查、测试、查文档分给不同子代理，互不干扰。
- **保护主上下文**——任务会产生大量中间过程（读一堆文件、长思考），但你只关心结论。
- **并行处理**——多个独立任务同时派给多个子代理。

### 不该用子代理

- **简单任务**——改个 typo、读一个文件、跑个命令。派子代理的开销比直接做还大。
- **需要主对话完整上下文的任务**——任务依赖你和 Claude 之前讨论的细节，子代理看不到主对话历史。
- **强交互任务**——需要反复来回讨论的，子代理是"派出去单干"的模式，不适合。

**判断标准：任务够大、够独立、产生大量中间过程 → 用子代理；小而简单、强依赖上下文 → 直接在主对话做。**

## 内置 agent 类型 vs 自定义 agent

Claude Code 有一些内置的通用 agent 类型（比如专门做探索的 `Explore`、通用任务 `general-purpose`），你不需要自己定义就能用。但大多数专业场景，自定义 agent 更合适——你能精确控制它的角色、工具、模型。

自定义 agent 的核心是**系统提示词设计**。一个好的 agent 提示词要包含：

1. **角色**——你是谁（"资深代码审查专家"）
2. **职责边界**——做什么、不做什么（"只审查，不修改"）
3. **工作方式**——怎么做（"按严重级别输出"）
4. **输出格式**——汇报长什么样（"位置 + 描述 + 修复建议"）

把这四点写清楚，agent 就能稳定输出你要的结果。

## 编排模式：并行、串行、主从

多个子代理怎么配合？三种常见模式：

### 1. 并行 fan-out（扇出）

主对话同时派多个子代理干独立的活，等全部回来汇总。

```
主对话 ──┬──→ reviewer 审查代码
         ├──→ test-writer 写测试
         └──→ doc-finder 查文档
              ↓ 全部完成后
         主对话汇总三份结论
```

适合：多个**互不依赖**的任务，并行省时间。

### 2. 串行 pipeline（流水线）

子代理一个接一个，前一个的输出是后一个的输入。

```
主对话 → analyzer 分析需求 → planner 规划 → coder 实现 → reviewer 审查
```

适合：有先后依赖的任务，像流水线。

### 3. 主从 review

主对话做完主要工作，派一个 reviewer 子代理做独立审查，避免"自己审自己"的盲区。

```
主对话写代码 → 派 reviewer 独立审查 → 主对话根据反馈修改
```

适合：需要独立视角把关的场景。

Claude Code 会自动选择编排模式——你只要在指令里说清任务，它自己决定要不要派子代理、派几个、怎么编排。你定义好 agent，剩下的交给编排。

## 实战：定义一个只读代码审查 agent

照这个模板做：

**第 1 步：建 agent 文件。**

```markdown
## <!-- .claude/agents/security-reviewer.md -->

name: security-reviewer
description: 安全审查专家。审查代码的安全漏洞，包括注入、XSS、密钥泄露、权限问题。只读，不修改代码。
tools: Read, Grep, Glob
model: sonnet

---

你是资深安全审查专家。

## 职责

审查给定代码的安全问题，**只审查不修改**。

## 检查清单

1. **注入**——SQL 注入、命令注入、XSS
2. **认证授权**——越权、缺失鉴权、session 问题
3. **密钥**——硬编码密钥、密钥泄露到日志、密钥进前端
4. **输入验证**——未验证的用户输入、信任外部数据
5. **依赖**——已知漏洞的依赖、不安全的第三方调用

## 输出格式

按严重程度（CRITICAL / HIGH / MEDIUM）分组，每个问题：

- 位置：文件:行号
- 类型：什么漏洞
- 描述：为什么是漏洞
- 建议：怎么修

没有发现问题时，明确说"未发现安全问题"。
```

**第 2 步：注意 tools 白名单。**

只给了 `Read, Grep, Glob`——只能读和搜，**没有 Write、Edit、Bash**。这样它绝对不会改代码、不会跑命令，纯审查。

**第 3 步：在会话里用。**

跟 Claude 说"用 security-reviewer 审查 src/auth/ 下的认证代码"。主对话会派这个子代理去审，审查完把安全报告带回来，主对话上下文干干净净。

**第 4 步：进 git 共享。**

```bash
git add .claude/agents/security-reviewer.md
git commit -m "feat: 添加安全审查 agent"
```

团队所有人都能派这个安全审查员。
