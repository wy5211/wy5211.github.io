前两篇我们讲了目录地图和加载机制。从这一篇开始，我们逐个深入最高频的配置文件。

第一个就是 **CLAUDE.md**——Claude Code 里你最该花时间打磨的文件。它本质上是给 Claude 看的"项目说明书"和"个人偏好手册"，写好了 Claude 像熟悉你项目的老同事，写差了它就是个每次都要重新摸索的新人。

但很多人有个困惑：

> 我明明在 CLAUDE.md 里写了"用函数组件""TypeScript 严格模式"，Claude 为什么偶尔还是写出 class 组件、忽略严格模式？

这一篇就来回答这个核心问题，顺便把 CLAUDE.md 的分层、加载、`@import`、写法原则一次讲透。

## CLAUDE.md 的多个层级

CLAUDE.md 不止一个文件，它在文件系统里有多个"驻点"，Claude Code 会全部加载：

```
~/.claude/CLAUDE.md          用户级    你的个人偏好，跨所有项目生效
./CLAUDE.md                  项目级    这个项目的约定，通常进 git
./src/CLAUDE.md              子目录级  进入 src 目录工作时额外加载
./packages/api/CLAUDE.md     更深的子目录级  当工作涉及该目录时加载
```

### 用户级：跨项目的个人偏好

`~/.claude/CLAUDE.md` 是你的"个人 DNA"。所有项目都会读到它。适合写**跟具体项目无关、但你永远希望 Claude 遵守**的东西：

```markdown
# 我的偏好

- 我习惯用 gitmoji 风格的 commit message
- 代码注释用中文
- 回答我问题时，先给结论再给细节
- 默认用 pnpm，除非项目明确指定其他
```

注意：这些是你的**个人**偏好，不是某个项目的约定。换个项目也成立。

### 项目级：这个项目的约定

`./CLAUDE.md`（项目根目录）是最高频、最重要的一个。它告诉 Claude"这个项目怎么开发"。这就是上一篇地图里说的，团队共享、通常进 git 的那个文件：

```markdown
# 项目约定

## 技术栈

- Next.js 15 App Router + TypeScript
- Tailwind CSS + shadcn/ui
- Prisma + PostgreSQL

## 代码规范

- 用函数组件，禁止 class 组件
- TypeScript 严格模式，禁止 any
- 文件命名用 kebab-case

## 常用命令

- 开发：pnpm dev
- 测试：pnpm test
- 提交前必跑：pnpm lint && pnpm typecheck
```

### 子目录级：进入该目录才加载

这是很多人忽略的层级。你可以在项目的子目录里也放 `CLAUDE.md`，Claude Code 在**工作涉及该目录**时会自动加载它。适合写"只在这个子模块适用"的特殊约定。

比如一个 monorepo，`packages/api/` 是后端，`packages/web/` 是前端：

```markdown
<!-- packages/api/CLAUDE.md -->

# API 包约定

- 用 tRPC 而非 REST
- 所有数据库访问必须经过 service 层，禁止直接调 Prisma
- 错误统一用 AppError 抛出
```

```markdown
<!-- packages/web/CLAUDE.md -->

# Web 包约定

- 组件用 server component 为默认，需要时再加 "use client"
- 样式优先用 Tailwind utility，自定义样式抽到 .module.css
```

这样 Claude 在改 API 代码时拿到 API 约定，改 Web 代码时拿到 Web 约定，互不污染。**这是大型项目避免上下文稀释的关键技巧。**

## 加载时机与上下文占用

理解 CLAUDE.md 为啥"偶尔失效"，必须先懂它的加载机制。

**会话启动时**，Claude Code 会读取用户级和项目级的 CLAUDE.md，把内容注入到上下文里。子目录级的 CLAUDE.md 在你工作涉及对应目录时按需注入。

这意味着 CLAUDE.md 的内容**从一开始就占着上下文窗口**。这一条很重要，它引出了两个后果：

1. **CLAUDE.md 越长，留给实际任务的上下文越少。**一个 5000 字的 CLAUDE.md，等于每次会话都先扣掉 5000 字的额度。
2. **写太长会"稀释"核心指令。**当上下文里塞了 50 条规则，Claude 注意力分散，最重要的那几条反而容易被忽略。这就是你写的"用函数组件"偶尔失效的原因——它被淹没在其他规则里了。

所以 CLAUDE.md 的核心原则是：**精简、聚焦、可执行。**

## `@import`：拆分大型 CLAUDE.md

当项目变大，单个 CLAUDE.md 写不下，或者你想把通用规则抽出来复用，就用 `@import` 语法引用其他文件。

```markdown
<!-- ./CLAUDE.md -->

# 项目约定

@docs/conventions/coding-style.md
@docs/conventions/git-workflow.md
@docs/conventions/testing.md
```

Claude Code 看到 `@import` 会把被引用文件的内容也加载进来。这样你可以：

- 把详细的代码规范拆成单独文件，CLAUDE.md 只保留索引
- 在多个项目间共享同一份规范（比如公司级的编码规范放一个仓库，各项目 `@import`）
- 子目录 CLAUDE.md 也能 `@import` 公共文件

**用法要点：**

- 路径相对于当前 CLAUDE.md 所在目录
- 可以引用 `.md`、`.txt` 等文本文件
- 也可以引用整个目录（会加载目录下所有支持的文件）

## 写法原则：为什么 Claude 偶尔不遵守

回到开头的问题。Claude 偶尔不遵守 CLAUDE.md，除了上下文稀释，还有几个常见原因：

### 1. 指令太模糊，不够可执行

```markdown
# ❌ 模糊

代码要写得好一点。
注意代码质量。

# ✅ 可执行

所有函数必须有显式的参数和返回类型标注。
单文件不超过 400 行，超过就拆分。
所有 export 的函数必须有 JSDoc 注释。
```

模糊的指令等于没指令。Claude 需要**具体的、可判断对错的**规则。

### 2. 指令互相矛盾

```markdown
# ❌ 矛盾

- 优先用 server component
- 所有组件都要支持交互，默认加 "use client"
```

这两条同时存在，Claude 不知道听谁的。检查你的 CLAUDE.md 有没有自相矛盾的规则。

### 3. 指令写在了错误的层级

```markdown
# ❌ "禁止用 npm" 写在用户级 ~/.claude/CLAUDE.md

# 但有个老项目必须用 npm
```

把项目特定的规则写到全局，会污染其他项目。规则要放对层级——通用的归全局，项目特定的归项目。

### 4. 写得太长，核心被稀释

一个 300 行的 CLAUDE.md 里，最重要的 10 条规则可能被淹没。**控制长度，把最关键的约定放在最前面。**Claude 对上下文开头和结尾的内容注意力更强。

## 高频写法模板

一份实用的项目 CLAUDE.md，通常包含这几个部分：

```markdown
# 项目名

## 项目概览

2-3 句话说清这是什么、技术栈是什么。帮 Claude 快速定位。

## 代码规范

这个项目的编码约定。具体到可执行的程度。

## 目录结构

关键目录的职责说明，避免 Claude 把东西放错地方。

- content/posts/ —— 博客文章，用 .mdx
- src/components/ —— React 组件
- src/lib/ —— 工具函数

## 常用命令

开发、测试、构建、部署的命令。Claude 会优先用这些。

## 避坑清单

你踩过的、希望 Claude 避免的坑。这是最实用的部分。

- ❌ 不要用 npm，用 pnpm
- ❌ 不要直接改 prisma schema，先在 migrations 里规划
- ❌ 提交前必须跑 pnpm lint，不要跳过
```

**最实用的"避坑清单"放在显眼位置。**因为这些是你血泪总结的，Claude 一旦违反代价最大。

## CLAUDE.md vs settings：软规则 vs 硬规则

这是理解 CLAUDE.md 本质的关键。很多人混淆了"记忆"和"规则"：

| 维度     | CLAUDE.md（记忆）          | settings.json（规则）    |
| -------- | -------------------------- | ------------------------ |
| 本质     | 提示词层，给 Claude 的建议 | 强制层，工具调用的硬约束 |
| 强制力   | 软的，Claude **可能**违反  | 硬的，工具**无法**绕过   |
| 例子     | "用函数组件"               | "禁止 rm -rf"            |
| 违反后果 | 代码不符合你的风格         | 命令根本执行不了         |

**用一句话区分：**CLAUDE.md 管"Claude 应该怎么做"，settings 管"Claude 能不能做"。

所以涉及到**安全、不可逆、团队强制**的规则，不要写进 CLAUDE.md，要写进 settings 的 `permissions`。比如"禁止 force push"——写进 CLAUDE.md 的话 Claude 偶尔会忽略（软规则），写进 `permissions.deny` 的话它根本执行不了（硬规则）。

下一篇我们就专门讲 settings 的权限系统——allow / deny / ask 三态机制。

## 实战：给你的项目写一份高信噪比的 CLAUDE.md

照这个流程做，你的 CLAUDE.md 信噪比会立刻提升：

**第 1 步：写一个 3 句话的项目概览。**

```markdown
# 我的博客

基于 Next.js 14 App Router 的个人博客，用 MDX 写文章，部署在 Vercel。
```

**第 2 步：列出目录结构。**

```markdown
## 目录结构

- content/posts/<series>/ —— 文章，按系列分目录，用 .mdx
- src/components/ —— 组件
- src/lib/ —— 工具
```

**第 3 步：写避坑清单（最重要）。**

```markdown
## 避坑

- ❌ 不要创建 .md 文件，文章必须是 .mdx
- ❌ frontmatter 必须有 series 和 seriesOrder 字段
- ❌ 包管理用 pnpm，不要用 npm/yarn
```

**第 4 步：删掉所有模糊、矛盾、过时的内容。**

每隔一段时间 review 一遍 CLAUDE.md，删掉不再适用的规则。CLAUDE.md 不是越长越好，是越准越好。
