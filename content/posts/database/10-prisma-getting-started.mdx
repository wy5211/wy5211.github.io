前 9 篇我们一直在写原生 SQL。你大概已经能熟练地写 SELECT、JOIN、子查询了。但回到实际的 TypeScript 项目里，原生 SQL 有几个让人头疼的问题。

第一个是拼字符串。你要把 JavaScript 变量塞进 SQL 语句里，一不小心就犯了 SQL 注入的错误（第 2 篇讲过）。即使用了参数化查询，写起来也啰嗦：

```typescript
// 原生 SQL 的写法
const result = await client.query(
  "SELECT * FROM users WHERE email = $1 AND is_active = $2",
  [email, true]
);
// result.rows[0].full_name —— 没有类型提示，不知道有哪些字段
```

第二个是没有类型提示。你查出来一个 `result.rows`，它的类型是 `any[]`。`full_name` 还是 `fullName`？`created_at` 是 string 还是 Date？全靠记忆或者翻数据库文档。

第三个是模型和数据库的同步。你在代码里加了一个字段，得去数据库里 ALTER TABLE 加一列。时间长了，代码里的字段和数据库里的列经常对不上。

Prisma 就是来解决这些问题的。它是 TypeScript 生态里最友好的 ORM，核心卖点是类型安全——你定义了 Schema，它自动生成类型，你在代码里写查询时就有完整的自动补全和类型检查。

## 为什么选 Prisma 而不是 TypeORM

Node.js 生态里 ORM 选择不少。TypeORM 历史更长，用装饰器（decorator）定义模型，和 NestJS 配合得很好。Drizzle 更轻量，SQL-like 的 API，性能更好。Sequelize 年代最久，但类型支持比较弱。

Prisma 的优势在于开发体验。它的 Schema 文件是独立于代码的（不侵入你的 TypeScript 类），自动生成的类型推断非常完整（不像有些 ORM 需要 `as unknown as SomeType`），而且 Prisma Studio 提供了一个可视化的数据管理界面，开发时查看数据很方便。

Prisma 的劣势是：不支持多个数据库之间的跨库事务（TypeORM 可以），动态查询的能力比原生 SQL 弱（复杂查询有时需要降级到 `$queryRaw`），生成的 SQL 有时不一定是最优的（特别是复杂 JOIN）。

如果你做一个新项目，团队用 TypeScript，Prisma 是很好的默认选择。如果你的项目已经在用 TypeORM 或者 Drizzle，而且没什么大问题，没必要迁移。

## Schema 定义：把 SQL 翻译成 Prisma

Prisma 用一个 `schema.prisma` 文件来定义数据模型。你可以把它理解为"用 Prisma 的语法重新写一遍 CREATE TABLE"。

把我们电商系统的核心表翻译成 Prisma Schema：

```prisma
// schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id           Int       @id @default(autoincrement())
  email        String    @unique
  passwordHash String    @map("password_hash")
  fullName     String?   @map("full_name")
  phone        String?
  isVerified   Boolean   @default(false) @map("is_verified")
  isActive     Boolean   @default(true) @map("is_active")
  createdAt    DateTime  @default(now()) @map("created_at")
  updatedAt    DateTime  @updatedAt @map("updated_at")

  orders Order[]

  @@map("users")
}

model Product {
  id          Int       @id @default(autoincrement())
  name        String
  description String?
  price       Decimal   @db.Decimal(10, 2)
  stock       Int       @default(0)
  categoryId  Int?      @map("category_id")
  attributes  Json      @default("{}")
  isActive    Boolean   @default(true) @map("is_active")
  createdAt   DateTime  @default(now()) @map("created_at")
  updatedAt   DateTime  @updatedAt @map("updated_at")

  category   Category?  @relation(fields: [categoryId], references: [id])
  orderItems OrderItem[]

  @@map("products")
}

model Order {
  id           Int      @id @default(autoincrement())
  userId       Int      @map("user_id")
  orderNumber  String   @unique @map("order_number")
  status       String   @default("pending")
  total        Decimal  @db.Decimal(10, 2)
  createdAt    DateTime @default(now()) @map("created_at")

  user       User        @relation(fields: [userId], references: [id])
  orderItems OrderItem[]

  @@map("orders")
}

model OrderItem {
  id         Int     @id @default(autoincrement())
  orderId    Int     @map("order_id")
  productId  Int     @map("product_id")
  quantity   Int
  unitPrice  Decimal @db.Decimal(10, 2) @map("unit_price")

  order   Order   @relation(fields: [orderId], references: [id])
  product Product @relation(fields: [productId], references: [id])

  @@unique([orderId, productId])
  @@map("order_items")
}

model Category {
  id        Int       @id @default(autoincrement())
  name      String
  slug      String    @unique
  parentId  Int?      @map("parent_id")
  createdAt DateTime  @default(now()) @map("created_at")

  parent   Category?  @relation("CategoryTree", fields: [parentId], references: [id])
  children Category[] @relation("CategoryTree")
  products Product[]

  @@map("categories")
}
```

几个要点。

`@map("password_hash")` 把 Prisma 的驼峰命名 `passwordHash` 映射到数据库的蛇形命名 `password_hash`。这样你在代码里写 `user.passwordHash`，数据库里存的是 `password_hash` 列，两边都舒服。`@@map("users")` 做的事情类似，把 Prisma model 名 `User` 映射到数据库表名 `users`。

`@relation` 定义模型之间的关系。`User` 和 `Order` 是一对多，所以 `Order` 里有 `user User @relation(fields: [userId], references: [id])`，`User` 里有 `orders Order[]`。Prisma 会根据这两个 `@relation` 自动建立外键。

`@updatedAt` 自动维护 `updatedAt` 字段——每次执行 `update()` 操作时，Prisma 自动把时间戳设为当前时间。不需要像原生 SQL 那样写触发器。

`Decimal` 类型对应数据库的 `NUMERIC`。Prisma 会在 TypeScript 里生成一个特殊的 `Decimal` 类型（来自 `@prisma/client/runtime/library`），在代码里你可以把它当字符串用，也可以转成 number。不要用 `Float` 存金额——上一篇讲过原因。

## Prisma Migrate：用代码管理数据库变更

有了 Schema 之后，你需要把它同步到数据库。Prisma Migrate 就是做这件事的——它对比 Schema 和当前数据库状态的差异，生成 SQL 迁移文件。

```bash
# 第一次：创建数据库并应用所有迁移
npx prisma migrate dev --name init

# 之后每次修改 Schema，运行：
npx prisma migrate dev --name add_phone_to_users
```

`migrate dev` 会做三件事：生成迁移 SQL 文件、应用到开发数据库、重新生成 Prisma Client 的类型。

迁移文件长这样：

```sql
-- migrations/20250320_add_phone_to_users/migration.sql
ALTER TABLE "users" ADD COLUMN "phone" TEXT;
```

这些迁移文件应该提交到 Git 里。团队成员 pull 代码后，运行 `npx prisma migrate dev` 就能自动把数据库更新到最新状态。部署到生产环境时，用 `npx prisma migrate deploy`——它只应用还没执行的迁移文件，不会生成新的。

Prisma Migrate 的优势是迁移历史是可追溯的——每个迁移文件都有时间戳和名称，你可以清楚地看到数据库经历了哪些变更。但也有局限：它不支持复杂的数据迁移（比如"把 A 列的值拆到 B 列和 C 列"），这种需要手写 SQL 然后用 `npx prisma migrate resolve` 标记为已执行。

## CRUD 操作：类型安全的数据库访问

Prisma Client 生成好之后，你就可以在代码里用它来操作数据库了。每个 model 都有一套完整的 CRUD 方法。

```typescript
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

// Create：创建用户
const user = await prisma.user.create({
  data: {
    email: "zhang_wei@example.com",
    passwordHash: "$2b$12$...",
    fullName: "张伟",
  },
});
// user 的类型是 User，所有字段都有自动补全

// Read：按 ID 查找
const found = await prisma.user.findUnique({
  where: { id: 1 },
});

// Read：按条件查找
const users = await prisma.user.findMany({
  where: {
    isVerified: true,
    createdAt: { gte: new Date("2025-01-01") },
  },
  orderBy: { createdAt: "desc" },
  take: 20, // LIMIT
});

// Update
const updated = await prisma.user.update({
  where: { id: 1 },
  data: { fullName: "张伟华" },
});

// Delete
await prisma.user.delete({
  where: { id: 1 },
});
```

注意 `where` 和 `data` 里的字段名都是 TypeScript 风格的驼峰命名——Prisma 自动帮你映射到数据库的蛇形命名。`createdAt` 在数据库里是 `created_at`，你不需要关心这个映射。

类型安全体现在几个方面。`findUnique` 返回的类型是 `User | null`（因为可能找不到），你必须处理 null 的情况。`findMany` 返回的是 `User[]`。`create` 的 `data` 参数类型是根据 model 的字段定义生成的——如果你漏了一个 `NOT NULL` 字段，TypeScript 会直接报错。

这就是 Prisma 的核心价值：**在编译阶段就能发现大部分数据访问的错误**，而不是等到运行时才发现 SQL 拼错了或者字段名写错了。

## 关联查询：include 和 select

上一篇文章讲过 N+1 查询的问题。Prisma 的 `include` 就是为了解决这个问题。

```typescript
// 查询订单列表，同时带上用户信息和订单项
const orders = await prisma.order.findMany({
  where: { status: "paid" },
  orderBy: { createdAt: "desc" },
  take: 20,
  include: {
    user: {
      select: { id: true, fullName: true, email: true },
    },
    orderItems: {
      include: {
        product: { select: { name: true, price: true } },
      },
    },
  },
});
```

`include` 告诉 Prisma："查订单的时候，把关联的用户和订单项也一起查出来。"Prisma 在底层会生成带 JOIN 的 SQL，一次查询搞定，不会出现 N+1 问题。

`select` 是更精细的控制——你指定要哪些字段，Prisma 只查这些字段。这在返回给前端的 API 里特别有用，避免把 `passwordHash` 这类敏感字段也带出来。

`include` 和 `select` 不能同时用在同一个关系上（你不可能既说"带上所有字段"又说"只要某几个字段"），但可以嵌套——`include` 一个关系，然后在那个关系里 `select` 特定字段，像上面那个例子一样。

## Prisma Studio：可视化的数据管理

开发时经常需要看数据库里有什么数据。psql 命令行能看，但不直观。Prisma 自带的 Studio 是一个 Web 界面的数据浏览器：

```bash
npx prisma studio
```

它会启动一个本地服务，浏览器打开后就能看到所有表的数据，支持筛选、排序、新增、编辑、删除。不需要额外安装 pgAdmin 或 DBeaver。

Prisma Studio 不是生产工具——它没有权限控制、没有审计日志、不支持多用户同时操作。但作为开发时的数据查看工具，它足够了。

## 小结

Prisma 把前 9 篇学的 SQL 知识包装成了类型安全的 TypeScript API。Schema 定义是起点——你在这里定义模型、关系、字段类型。Migrate 帮你管理数据库变更。Prisma Client 提供类型安全的 CRUD 操作，编译阶段就能发现错误。

但 Prisma 不是万能的。复杂的查询（多表 JOIN、窗口函数、递归 CTE）用 Prisma 的 API 写起来可能比原生 SQL 还别扭。遇到这种情况，用 `$queryRaw` 直接写 SQL，或者用 `queryRaw` 做只读查询。
