上一篇讲了备份恢复。这篇是这个系列的最后一篇——数据库跑起来之后，怎么知道它跑得好不好？出了问题怎么快速定位？流量涨了之后怎么扩展？

监控不是锦上添花，是生产环境的必需品。没有监控的数据库就像没有仪表盘的汽车——你能开，但不知道油量还剩多少、发动机温度多高、刹车片磨到什么程度了。

## 关键监控指标

数据库需要关注的指标不多，但每一个都很关键。

**连接数**：当前有多少客户端连接到数据库。PostgreSQL 的 `max_connections` 默认是 100。如果连接数经常接近上限，说明应用层可能没有正确使用连接池，或者有连接泄漏。

```sql
SELECT count(*) FROM pg_stat_activity;
```

注意 `pg_stat_activity` 里包含所有状态的连接。你可能更关心"活跃连接"（正在执行查询的）而不是"空闲连接"：

```sql
SELECT state, count(*)
FROM pg_stat_activity
WHERE datname = 'ecommerce'
GROUP BY state;
```

`active` 是正在执行查询的连接，`idle` 是空闲等待的连接，`idle in transaction` 是开了事务但没在执行查询的连接（如果这种连接很多，说明有代码忘了提交事务）。

**QPS（Queries Per Second）**：每秒执行的查询数量。这个指标帮你了解数据库的负载水平。

```sql
-- 通过 pg_stat_database 的统计信息推算
SELECT
  xact_commit + xact_rollback AS total_queries,
  -- 需要间隔一段时间再查一次，用差值除以时间间隔得到 QPS
  numbackends AS active_connections
FROM pg_stat_database
WHERE datname = 'ecommerce';
```

更实用的方式是隔 10 秒查两次，计算差值。

**慢查询数量**：执行时间超过阈值的查询数量。第 9 篇讲的 `pg_stat_statements` 可以按平均耗时排序找出最慢的 SQL。另一个方法是配置 `log_min_duration_statement`，让 PostgreSQL 自动把慢查询记录到日志里。

```yaml
# docker-compose.yml
postgres:
  command:
    - "postgres"
    - "-c"
    - "log_min_duration_statement=500" # 超过 500ms 的查询记录到日志
```

**缓存命中率**：PostgreSQL 的共享缓冲区（shared_buffers）命中率。如果命中率低于 99%，说明你的数据量超过了内存能缓存的量，查询需要更多地读磁盘，性能会下降。

```sql
SELECT
  round(
    blks_hit::numeric / nullif(blks_hit + blks_read, 0) * 100, 2
  ) AS cache_hit_ratio_percent
FROM pg_stat_database
WHERE datname = 'ecommerce';
```

`blks_hit` 是从内存缓冲区读到的块数，`blks_read` 是从磁盘读到的块数。命中率 99% 意味着 99% 的数据请求可以直接从内存满足，只有 1% 需要读磁盘。

新启动的数据库命中率会很低（因为缓冲区是空的），运行一段时间后会逐渐升高到 99%+。如果运行了很久命中率还是很低，可能是 `shared_buffers` 设置太小了。

**复制延迟**：如果你有主从复制（后面会讲），需要监控从库和主库之间的数据同步延迟。

```sql
-- 在从库上执行
SELECT
  client_addr,
  state,
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  (EXTRACT(EPOCH FROM (now() - reply_time)) * 1000)::integer AS lag_ms
FROM pg_stat_replication;
```

`lag_ms` 是从库落后主库多少毫秒。如果延迟持续增长，说明从库跟不上主库的写入速度，可能需要优化从库的硬件或减少主库的写入压力。

## PostgreSQL 配置调优

第 3 篇介绍了几个基础配置（shared_buffers、work_mem）。这里聊几个生产环境常用的调优参数。

**effective_cache_size**：告诉优化器系统大概有多少缓存可用。不是实际分配内存，只是一个"提示"。

```sql
-- 查看当前值
SHOW effective_cache_size;
-- 默认 4GB，建议设为物理内存的 50-75%
```

这个值影响优化器选择索引扫描还是顺序扫描。如果你告诉优化器"系统有很多缓存"，它会更倾向于用索引扫描（因为估计数据已经在缓存里了）。设置合理可以避免优化器做出错误的选择。

**random_page_cost**：从磁盘随机读一个页面的"成本"。默认值是 4.0，意思是优化器认为随机读磁盘比顺序读磁盘贵 4 倍。

如果你的数据库运行在 SSD 上（现在大多数云服务器都是），随机读和顺序读的性能差距没那么大。可以把这个值调低到 1.1-1.5，让优化器更倾向于使用索引：

```sql
SET random_page_cost = 1.1;  -- SSD 环境
```

**maintenance_work_mem**：维护操作（VACUUM、CREATE INDEX、ALTER TABLE）能用的内存。默认 64MB。

```sql
SET maintenance_work_mem = '256MB';
```

如果你要在一个大表上建索引或做 VACUUM，调大这个参数能显著加速。这个参数不影响日常查询性能。

**checkpoint_timeout 和 max_wal_size**：控制 WAL 检查点的频率。检查点是 PostgreSQL 把内存中的修改刷写到磁盘的时机。检查点越频繁，崩溃恢复越快（需要回放的 WAL 越少），但正常的 I/O 开销越大（每次检查点都要大量写磁盘）。

```sql
-- 默认值通常够用
SHOW checkpoint_timeout;  -- 默认 5 分钟
SHOW max_wal_size;         -- 默认 1GB
```

除非你观察到明显的 I/O 峰值（检查点时磁盘利用率突然飙升），否则不需要调整这两个参数。

## 连接池：PgBouncer

第 3 篇已经介绍了 PgBouncer。这里补充几个生产环境的注意事项。

PgBouncer 有三种池模式：session、transaction、statement。

**transaction 模式**是生产环境最常用的。一个客户端连接在事务开始时从池里取连接，事务结束后归还。大部分时间里你的连接处于空闲状态（等应用代码处理逻辑、等客户端下一次请求），transaction 模式让多个客户端可以共享少量的数据库连接。

但 transaction 模式有一个限制：**不能用 PostgreSQL 的 server-side prepared statements 和 advisory locks**。因为这些状态绑定在数据库连接上，连接被归还到池里后下一个使用者不知道有这些状态。如果你的应用大量使用了 prepared statements，可能需要用 session 模式（性能差一些但没有限制），或者在应用层禁用 prepared statements。

`DEFAULT_POOL_SIZE` 的设置取决于 PostgreSQL 的 `max_connections` 和你的 CPU 核心数。一般设为 CPU 核心数的 2-3 倍。如果你的服务器是 8 核，设 20 就够了。`MAX_CLIENT_CONN` 可以设大一些（200-500），因为 PgBouncer 到客户端的连接很轻量。

```yaml
# docker-compose.yml
pgbouncer:
  image: edoburu/pgbouncer
  environment:
    DATABASES: "ecommerce = host=postgres port=5432 dbname=ecommerce"
    POOL_MODE: transaction
    MAX_CLIENT_CONN: 500
    DEFAULT_POOL_SIZE: 20
    RESERVE_POOL_SIZE: 5
    RESERVE_POOL_TIMEOUT: 3
```

`RESERVE_POOL_SIZE` 和 `RESERVE_POOL_TIMEOUT` 是应对突发流量的——当连接池满了，新请求会等待。如果设置了 reserve pool，PgBouncer 会额外保留几个连接，在突发流量时释放出来，最多等 `RESERVE_POOL_TIMEOUT` 秒。

## 什么时候需要分库分表

当数据量和流量增长到单机 PostgreSQL 扛不住的时候，你有几个选项。

**读写分离**是最简单的一步。设置一个主库（处理写操作）和一个或多个从库（处理读操作）。PostgreSQL 原生支持流复制（streaming replication），从库自动从主库同步数据。

读写分离的架构大概是：

```
应用 → PgBouncer → 主库（写）
                  → 从库1（读）
                  → 从库2（读）
```

应用层需要在代码里区分"读操作走从库，写操作走主库"。大多数 ORM（包括 Prisma）支持配置读写分离。延迟方面，流复制的延迟通常在毫秒级别，对于大多数业务可以接受。但要注意：写入后立即读取（比如"创建订单后返回订单详情"）需要走主库，否则可能从库还没同步到最新数据。

**分库**是把不同的业务数据拆到不同的数据库实例。比如用户数据放一个库，订单数据放另一个库，商品数据放第三个库。每个库独立扩容。

分库的代价是跨库查询变得困难——你不能在一个 SQL 里 JOIN 两个不同库的表。应用层需要做数据聚合。所以分库的前提是：业务模块之间没有频繁的跨库查询。

**分表**是在同一个数据库里把一张大表拆成多张小表。比如 orders 表按月拆分成 orders_202501、orders_202502、...。查询时根据时间范围路由到对应的表。

分表能解决单表数据量过大的问题（索引太大、查询太慢），但应用层需要处理路由逻辑，而且跨月查询（比如"查过去 3 个月的订单"）需要合并多个表的结果。

**什么时候需要分库分表？** 一个实用的判断标准：

- 单表数据量超过 5000 万行 → 考虑分表
- 单库 QPS 超过 5000 → 考虑读写分离
- 单库数据量超过 100GB → 考虑分库
- 连接数经常打满 → 先检查应用层是否正确使用连接池，再考虑分库

大多数中小项目，读写分离 + 适当的索引优化就够用了。分库分表是最后的手段——它带来的复杂度是指数级的，不到万不得已不要做。

## 高可用架构

最后聊一下高可用——当数据库服务器挂了，怎么保证服务不中断。

**最基础的方案：自动故障转移**。两台 PostgreSQL 服务器，一台主库一台从库。主库挂了，自动把从库提升为主库。工具可以用 Patroni（Kubernetes 环境）或 repmgr（传统环境）。

**更成熟的方案：云数据库**。AWS RDS、阿里云 RDS、Supabase 这些托管服务已经内置了自动故障转移、备份、监控。你不需要自己搭建主从复制、不需要手动切换、不需要担心 WAL 日志的管理。对于大多数团队，用云数据库是最务实的选择。

**最大的数据库不在数据库里**。如果流量真的非常大，很多数据其实不需要实时查询——历史订单可以归档到数据仓库，行为日志可以用 MongoDB + TTL 自动清理，搜索走 Elasticsearch。核心 PostgreSQL 只存"热数据"——最近 3 个月的订单、活跃用户、在售商品。这样即使不分区不分库，单机 PostgreSQL 也能扛住相当大的流量。

## 小结

数据库监控的核心指标：连接数、QPS、慢查询数量、缓存命中率、复制延迟。这些指标帮你快速判断数据库是否健康。

PostgreSQL 的配置调优从几个关键参数开始：shared_buffers（内存的 25%）、work_mem（16-32MB）、effective_cache_size（内存的 50-75%）、random_page_cost（SSD 设为 1.1）。调优之前先用 EXPLAIN ANALYZE 确认瓶颈在哪里，不要凭直觉改参数。

连接池（PgBouncer）的 transaction 模式是生产环境的标准配置。读写分离是扩展读能力的第一步。分库分表是最后的手段——带来的复杂度远大于收益，除非你真的遇到了单机无法解决的问题。
