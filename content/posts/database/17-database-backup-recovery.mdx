前 16 篇我们搭建了完整的数据库架构：PostgreSQL 存核心业务数据，MongoDB 存行为日志，Redis 做缓存，Elasticsearch 做搜索。功能都有了，但如果明天服务器硬盘坏了，所有数据都没了——你的电商系统就真的完了。

备份是数据安全的最后一道防线。你可能觉得"云数据库有自动备份，不用担心"。确实，AWS RDS、Supabase 这些托管服务会自动做备份。但"有备份"和"备份能用"是两回事——如果你从来没验证过恢复流程，备份可能已经坏了而你不知道。

## 备份的 3-2-1 原则

备份领域有一个广泛认可的原则叫 3-2-1：**3 份副本、2 种存储介质、1 份异地**。

3 份副本：原始数据 + 2 份备份。不是"1 份就够了"——如果备份文件本身损坏了怎么办？

2 种存储介质：不要把所有备份都放在同一块硬盘上。硬盘坏了两个一起坏。一份放服务器本地磁盘，一份放对象存储（S3、OSS）。

1 份异地：本地备份不能防机房级别的事故（火灾、断电、网络中断）。一份备份应该存到不同的地区或云服务上。

这个原则听起来很基础，但很多团队做不到。最常见的遗漏是"没有异地备份"——备份文件存在数据库同一台机器上，机器挂了备份也没了。

## PostgreSQL 备份

PostgreSQL 有两种备份方式：逻辑备份和物理备份。

**逻辑备份**（pg_dump）：把数据库内容导出为 SQL 文件。类似你手动执行的 `CREATE TABLE` + `INSERT INTO` 语句。

```bash
# 导出整个数据库
pg_dump -h localhost -U admin -d ecommerce > backup_20250320.sql

# 只导出特定表
pg_dump -h localhost -U admin -d ecommerce -t orders -t users > partial_backup.sql

# 导出为自定义格式（支持并行恢复）
pg_dump -h localhost -U admin -d ecommerce -Fc > backup_20250320.dump
```

`-Fc` 格式的优势是恢复时可以用 `-j` 参数并行执行，比纯 SQL 文件快很多。

恢复：

```bash
# 从 SQL 文件恢复
psql -h localhost -U admin -d ecommerce < backup_20250320.sql

# 从自定义格式恢复（4 个并行任务）
pg_restore -h localhost -U admin -d ecommerce -j 4 backup_20250320.dump
```

逻辑备份的优点是灵活——你可以选择导出哪些表、可以查看和编辑 SQL 文件、可以跨版本恢复（从 PG 15 备份恢复到 PG 16）。缺点是速度慢（大库可能要几个小时）和一致性——如果你在备份期间有数据在写入，`pg_dump` 默认不保证一致性。加 `--serializable-deferrable` 可以保证一致性，但会更慢。

**物理备份**（pg_basebackup）：直接复制 PostgreSQL 的数据目录文件。不需要导出/导入 SQL，直接文件级别的复制。

```bash
pg_basebackup -h localhost -U admin -D /var/backups/pg_base -Ft -z -P
```

`-Ft` 输出 tar 格式，`-z` 压缩，`-P` 显示进度。

物理备份比逻辑备份快得多——大库几十 GB 的数据，pg_basebackup 几分钟就搞定，pg_dump 可能要几小时。而且物理备份包含了所有的配置文件、WAL 日志、索引，恢复后数据库的状态和备份时完全一致。

缺点是不灵活——你不能选择只恢复某张表，恢复的 PostgreSQL 版本必须和备份时一致（大版本升级后旧的物理备份不能直接用）。

选择建议：**日常定时备份用逻辑备份（灵活、可审查），大版本升级前做一次物理备份（快速、完整）**。

定时备份可以用 crontab：

```bash
# 每天凌晨 2 点做逻辑备份
0 2 * * * pg_dump -h localhost -U admin -d ecommerce -Fc > /backups/daily/dump_$(date +\%Y\%m\%d).dump

# 每周日做物理备份
0 3 * * 0 pg_basebackup -h localhost -U admin -D /backups/weekly/base_$(date +\%Y\%m\%d) -Ft -z
```

别忘了清理旧备份——不然磁盘会被撑满：

```bash
# 只保留最近 30 天的日常备份
find /backups/daily -name "*.dump" -mtime +30 -delete
```

## MongoDB 备份

MongoDB 有官方的备份工具 `mongodump`。

```bash
# 导出整个数据库
mongodump --uri "mongodb://localhost:27017/ecommerce" --out /backups/mongo/$(date +%Y%m%d)

# 只导出特定集合
mongodump --uri "mongodb://localhost:27017/ecommerce" --collection user_events --out /backups/mongo/$(date +%Y%m%d)
```

恢复：

```bash
mongorestore --uri "mongodb://localhost:27017/ecommerce" --drop /backups/mongo/20250320/ecommerce
```

`--drop` 表示恢复前先删除已有的集合。不加这个参数的话，mongorestore 会合并数据而不是覆盖。

对于日志数据（user_events），你可能不需要备份全部历史——备份最近 7 天的就行，更早的数据反正有 TTL 索引会自动清理。

```bash
# 只备份最近 7 天的日志
mongodump --uri "mongodb://localhost:27017/ecommerce" \
  --collection user_events \
  --query '{"createdAt": {"$gte": {"$date": "2025-03-13T00:00:00Z"}}}' \
  --out /backups/mongo/$(date +%Y%m%d)
```

`--query` 参数让 mongodump 只导出满足条件的文档。对于几千万条日志的集合，这能大幅减少备份时间和存储空间。

## Redis 持久化

Redis 的数据存在内存里，默认重启后数据就丢了。但它提供两种持久化方式。

**RDB（快照）**：定期把内存中的数据以二进制形式保存到磁盘。

```bash
# 手动触发一次快照
redis-cli BGSAVE
```

RDB 生成一个紧凑的二进制文件（`dump.rdb`），恢复时直接加载这个文件。优点是文件小、恢复快。缺点是两次快照之间的数据会丢失——如果你配置了每小时一次快照，最坏情况会丢失 1 小时的数据。

**AOF（Append Only File）**：把每条写命令追加到日志文件里。

```
SET product:42 '{"name":"Sony WH-1000XM5",...}'
EXPIRE product:42 600
DEL product:42
```

恢复时重放这些命令就能重建数据。AOF 的数据安全性比 RDB 高——最多丢失 1 秒的写入（默认每秒 fsync 一次）。缺点是文件比 RDB 大，恢复速度比 RDB 慢。

实际项目中两者可以同时开启：

```
# redis.conf
save 900 1        # 900 秒内至少 1 个 key 变化就触发 RDB
save 300 10       # 300 秒内至少 10 个 key 变化
save 60 10000     # 60 秒内至少 10000 个 key 变化

appendonly yes    # 开启 AOF
appendfsync everysec  # 每秒 fsync
```

Redis 重启时会优先用 AOF 恢复（因为 AOF 数据更完整），RDB 作为 AOF 损坏时的后备方案。

对于电商系统里的缓存数据（商品详情、热门列表），Redis 持久化的优先级不高——即使 Redis 重启数据丢了，也可以从 PostgreSQL 重新加载。但如果你用 Redis 存了不能丢失的数据（比如分布式锁的状态、限流计数器），就需要开启 AOF 确保数据安全。

## 恢复演练：没有验证过的备份等于没有备份

这是最重要但最容易被忽略的一步。

很多人做了备份就觉得安全了，直到需要恢复的时候发现：备份文件损坏了、恢复命令参数写错了、恢复后的数据不一致、恢复时间太长超过了业务可接受的停机时间。

**定期做恢复演练**。至少每季度一次，在测试环境里完整地恢复一次备份，验证：

1. 备份文件是否完整（没有损坏）
2. 恢复流程是否正确（命令能跑通）
3. 恢复后的数据是否一致（抽查几条关键数据）
4. 恢复需要多长时间（评估 RTO——业务可接受的停机时间）

演练步骤：

```bash
# 1. 在测试环境创建一个空数据库
createdb -h testhost -U admin ecommerce_test

# 2. 从备份恢复
pg_restore -h testhost -U admin -d ecommerce_test -j 4 /backups/daily/dump_20250320.dump

# 3. 验证数据一致性
psql -h testhost -U admin -d ecommerce_test -c "
  SELECT 'users' AS table_name, COUNT(*) FROM users
  UNION ALL
  SELECT 'orders', COUNT(*) FROM orders
  UNION ALL
  SELECT 'products', COUNT(*) FROM products
"

# 4. 记录恢复时间
# 开始: 10:00:00
# 完成: 10:15:32
# RTO: 15 分 32 秒
```

把恢复时间记录下来，和业务方确认这个时间是否可接受。如果不可接受，就需要优化恢复流程（比如用物理备份替代逻辑备份，或者用 standby 服务器做热备切换）。

## 小结

备份的核心原则是 3-2-1：3 份副本、2 种介质、1 份异地。PostgreSQL 用 pg_dump 做逻辑备份（灵活）或 pg_basebackup 做物理备份（快速）。MongoDB 用 mongodump，可以通过 `--query` 只备份需要的数据。Redis 同时开启 RDB 和 AOF，RDB 做定期快照，AOF 记录每次写入。

但最重要的是：**定期做恢复演练**。没有验证过的备份等于没有备份。你知道备份能恢复，和"希望"备份能恢复，是完全不同的两件事。

最后一篇我们讲监控与调优——怎么知道数据库运行得是否健康，出了问题怎么快速定位。
