你大概做过这样的事：项目刚启动，懒得搭数据库，直接把用户数据写进一个 `users.json` 文件。注册的时候 `JSON.parse` 读出来，追加一条，再 `JSON.stringify` 写回去。几十个用户的时候一切正常，团队还觉得这方案挺清爽。

然后某天运营搞了个拉新活动，一夜涌入两千人。你的服务器日志里开始出现这种错误：

```
Error: ENOENT: no such file or directory, open './data/users.json'
```

或者更隐蔽的——用户投诉说"我明明注册成功了，刷新页面又要我重新注册"。打开文件一看，数据写了一半，JSON 格式直接坏了。

这不是 bug，这是你选错了存储方案。文件系统从来就不是为并发读写设计的，它只是一个"存东西的地方"，而不是一个"管理数据的系统"。这两者之间的差距，就是数据库存在的理由。

## 文件存数据到底差在哪里

很多人觉得"文件能存数据，为什么还要数据库"，这个想法很自然，但忽略了一个关键问题：**当读写同时发生的时候，谁来做协调？**

用一个 Node.js 的例子来看这个问题。假设你的用户注册接口是这样写的：

```javascript
const fs = require("fs");
const path = "./data/users.json";

// 极其简化版的文件存储用户注册
function registerUser(email, passwordHash) {
  const raw = fs.readFileSync(path, "utf-8");
  const users = JSON.parse(raw);

  const newUser = {
    id: `usr_${Date.now()}_${Math.random().toString(36).slice(2, 8)}`,
    email,
    password_hash: passwordHash,
    created_at: new Date().toISOString(),
  };

  users.push(newUser);
  fs.writeFileSync(path, JSON.stringify(users, null, 2));
  return newUser;
}
```

这段代码在单线程、低并发的场景下跑得很好。但注意 `readFileSync` 到 `writeFileSync` 之间有一段时间窗口——两个请求同时进来，都读到了同一份数据，各自追加一条，然后后者覆盖前者。第一个用户的数据就这样被无声地丢掉了。

这就是**并发写入冲突**。文件系统不会帮你锁资源、不会帮你排队、不会告诉你"有人正在写，你等一下"。

除了并发，文件存储还有几个让你头疼的问题：

**查询能力为零。** 你想找"2025 年 3 月注册的所有邮箱以 `@gmail.com` 结尾的用户"，用文件就得把整个 JSON 读进内存，写一个 filter 遍历全部数据。用户量到十万级的时候，这个操作可能要好几秒。

**没有数据校验。** 你可以在代码里手动检查 `email` 是否重复、`age` 是否合法，但这些检查分散在各个接口里，漏掉一个就是线上事故。数据库可以声明式地定义约束，从根源上杜绝非法数据。

**崩溃恢复不可靠。** `writeFileSync` 写到一半服务器断电了，文件内容截断，JSON 解析直接抛异常。你之前存的所有用户数据都可能因为一个半截文件而无法读取。

这些问题单个看起来都不算致命，但叠加在一起，当你的项目从"自己玩"变成"给别人用"的时候，它们会成为反复出现的噩梦。

## 数据库解决了什么问题

数据库和文件系统的根本区别不在于"存得更安全"或"写得更快"，而在于它提供了一套**完整的数据管理机制**。

**并发控制。** 数据库通过锁机制和 MVCC（多版本并发控制）来协调同时发生的读写操作。一千个用户同时注册，数据库能保证每个注册都正确完成，不会互相覆盖。

**结构化查询。** 你不需要把所有数据加载到内存再过滤，只需要告诉数据库"我要什么条件的数据"，它自己会找到最快的路径返回结果。背后是 B+ 树索引、查询优化器这些机制在支撑。

**事务保证。** 这是数据库最有价值的能力之一，后面会详细展开。

**数据完整性约束。** 你可以声明"邮箱必须唯一"、"年龄必须在 0 到 150 之间"、"订单状态只能是这几种"，数据库会替你守住这些规则，不管哪个接口、哪段代码来写数据。

**崩溃恢复。** 数据库通过 WAL（Write-Ahead Log，预写日志）机制保证：即使服务器在写入过程中断电，重启后也能把数据恢复到一致的状态，不会出现半截数据。

所以，数据库不是一个"更好的文件"，它是一个**数据管理系统**。它关心的不只是"把数据存下来"，还有"数据怎么被安全地读写、怎么保持一致、怎么快速找到"。

## 关系型 vs NoSQL：不是谁更好，是场景不同

很多人一上来就问"PostgreSQL 和 MongoDB 哪个好"，这个问题没法简单回答，因为它们解决的是不同的问题。

**关系型数据库**（PostgreSQL、MySQL）的核心思想是把数据组织成表格，表和表之间通过外键建立关系。它的强项是事务和数据一致性。

想象一个电商系统：用户下了一单，包含三件商品。你需要同时完成这几件事：创建订单记录、扣减三件商品的库存、扣除用户账户余额、生成一条物流记录。这几步要么全部成功，要么全部失败——不可能出现"钱扣了但订单没创建"的情况。这就是事务，关系型数据库在这方面做得非常成熟。

**NoSQL 数据库**是一个统称，包含几种完全不同的东西。别被这个名字误导，NoSQL 不是"不用 SQL"，而是"Not Only SQL"——用不同的数据模型来处理关系型数据库不擅长的场景。

举几个实际场景：

**MongoDB**（文档型）适合存结构不固定的数据。比如一个内容管理系统，文章可能有文字、图片、视频、投票等不同类型的附加内容，每篇文章的结构都不一样。硬塞进关系型表里要做很多关联表和类型字段，用 MongoDB 直接存成文档就清爽得多。

**Redis**（键值型）适合需要极致速度的场景。比如电商首页的秒杀倒计时、用户的登录 session、热门商品的缓存——这些数据访问频率极高，但可以容忍短暂的不一致。Redis 把数据放在内存里，读写速度是磁盘数据库的几十倍。

**Cassandra**（列族型）适合海量数据的写入。比如物联网设备每秒产生上万条传感器数据，或者社交平台的用户行为日志。这类数据量太大，单机数据库扛不住，需要分布在多台机器上，Cassandra 天生就是为这种场景设计的。

用一个简化的对比来看它们各自的适用场景：

| 场景                  | 推荐选择      | 核心原因                   |
| --------------------- | ------------- | -------------------------- |
| 电商订单、金融交易    | PostgreSQL    | 强事务、强一致性、复杂查询 |
| 内容管理、用户画像    | MongoDB       | 灵活 schema、嵌套文档      |
| 缓存、排行榜、session | Redis         | 内存级速度、丰富数据结构   |
| 传感器数据、行为日志  | Cassandra     | 分布式写入、水平扩展       |
| 商品搜索、日志分析    | Elasticsearch | 全文检索、聚合分析         |

这里有个常见的误区：很多团队一上来就想搞"多数据库混合架构"，觉得这样显得专业。但实际上，对于一个日活几千的小项目，PostgreSQL 一个数据库就够用了。过早引入多种数据库只会增加运维复杂度，而不会带来实际收益。

## ACID：不是要你背概念，而是理解为什么需要它

ACID 这四个字母你可能在面试里背过无数遍，但放到真实场景里，它的意义其实很直观。

假设你在做一个转账功能：用户 `zhang_wei` 给 `li_na` 转账 500 元。用关系型数据库写出来是这样的：

```sql
BEGIN;

-- 从 zhang_wei 的账户扣除 500 元
UPDATE accounts
SET balance = balance - 500
WHERE user_id = 'usr_a1x9k2m'
  AND balance >= 500;

-- 向 li_na 的账户增加 500 元
UPDATE accounts
SET balance = balance + 500
WHERE user_id = 'usr_b3y7n4p';

COMMIT;
```

这段 SQL 的 `BEGIN` 和 `COMMIT` 之间就是一个事务。ACID 说的就是这个事务的四个特性：

**原子性（Atomicity）**——要么两个 UPDATE 都成功，要么都不成功。不会出现"扣了 zhang_wei 的钱但没到 li_na 账上"的情况。如果第二步因为任何原因失败了（比如 li_na 的账户被冻结），数据库会自动回滚第一步的操作，zhang_wei 的钱原封不动。

**一致性（Consistency）**——事务执行前后，数据库都处于合法状态。比如你在表上定义了 `CHECK (balance >= 0)` 约束，那么任何事务都不能让余额变成负数。如果转账后余额为负，整个事务会被拒绝。

**隔离性（Isolation）**——如果 zhang_wei 同时给两个人转账，两个事务不会互相干扰。每个事务看到的都是一致的"快照"。否则可能出现：两个事务都读到余额是 1000 元，各自扣除 500，结果余额变成了 0 而不是 500。

**持久性（Durability）**——一旦 `COMMIT` 成功，即使服务器下一秒断电，转账结果也不会丢。数据库会把修改记录到日志文件里，重启后通过日志恢复数据。

你可能觉得"这不就是基本要求吗"，但想想前面用 JSON 文件的方案——原子性没有（写一半崩溃就完蛋），隔离性没有（并发覆盖），一致性没有（没有校验），持久性也靠不住。ACID 看起来是基本要求，但实现起来并不简单，这正是数据库的价值所在。

## CAP 定理：双十一抢购帮你理解分布式取舍

如果你的服务只需要跑在一台机器上，前面说的 ACID 足够了。但现实是，当用户量到了一定规模，你需要把数据库部署在多台机器上——这就是分布式数据库。一旦数据分散到多台机器，问题就复杂了。

CAP 定理说的是：一个分布式数据系统，**一致性（Consistency）**、**可用性（Availability）**、**分区容错性（Partition tolerance）** 这三个特性，你最多只能同时满足两个。

听起来很学术，但用"双十一抢购"的场景来理解就直观了。

假设你有三台数据库服务器，存着同一个商品的库存数量。双十一零点，全国用户疯狂点击"立即购买"。

**一致性优先（CP）**意味着：每次扣库存前，三台服务器必须先同步确认"当前库存是多少"。如果其中一台因为网络故障联系不上（这就是"分区"），系统宁愿让用户等一等甚至报错，也不允许在数据不一致的情况下继续卖。结果是：用户可能看到"系统繁忙请稍后再试"，但不会出现超卖。

**可用性优先（AP）**意味着：不管三台服务器之间能不能通信，只要有任何一台能响应，就先给用户处理。结果是：用户体验流畅，秒杀成功率高，但可能出现短暂的不一致——比如库存显示还有 10 件，实际三台服务器的数据还没对齐，最后卖出去 15 件。

**分区容错性（P）**基本是不可妥协的——在分布式环境下，网络故障是必然发生的，你必须能容忍它。所以真正的选择是在 C 和 A 之间。

现实中的做法通常是**最终一致性**：系统优先保证可用性（AP），允许短暂的不一致，但在几秒或几十秒内把数据同步到一致的状态。你刷微博的时候，热门博文的点赞数可能刷新一次变一下，过一会儿才稳定——这就是最终一致性的表现。

对于大多数 Web 应用来说，这种短暂的不一致是完全可以接受的。但对于银行转账这类场景，任何不一致都不行，所以金融系统会倾向于强一致性。

理解 CAP 不是为了选边站，而是让你在做架构决策时清楚自己放弃了什么、得到了什么。没有任何方案是完美的，每个选择都是一个 trade-off。

## 你的项目该用什么数据库

聊了这么多理论，回到一个最实际的问题：我现在要做一个项目，该选什么数据库？

这里有一个简单的判断方法。先问自己三个问题：

**第一，你的数据之间有没有复杂的关系？** 如果你的核心业务涉及"用户-订单-商品-支付"这种多实体关联，而且经常需要跨实体查询（比如"查某个用户的所有已完成订单中包含哪些商品"），关系型数据库是自然的选择。

**第二，你对数据一致性有多高的要求？** 如果是电商交易、账户余额这类"错一点都不行"的数据，必须用支持 ACID 事务的数据库。如果是日志记录、浏览历史这类"丢了或重复几条无所谓"的数据，MongoDB 或 Cassandra 都可以。

**第三，你的数据量级和访问模式是什么？** 十万级用户以内、常规读写模式，单机 PostgreSQL 绰绰有余。千万级以上或者需要缓存热点数据，再考虑引入 Redis。如果需要全文搜索（比如站内搜索），加一个 Elasticsearch。

基于这些判断，给几个具体建议：

如果你刚开始做项目，不知道该选什么，**选 PostgreSQL**。它支持 JSON 字段、全文搜索、数组类型，覆盖了大部分 NoSQL 的使用场景，同时提供完善的 ACID 事务。等你的项目真的遇到了 PostgreSQL 解决不了的问题（比如需要内存级缓存速度），再引入 Redis 也不迟。

如果你在做的是内容管理系统、博客平台这类结构灵活的场景，**MongoDB** 会让你少写很多关联查询，开发速度快很多。但要注意，MongoDB 对事务的支持虽然在 4.0 之后有了，但复杂度和性能跟 PostgreSQL 比还是有差距。

**Redis** 不应该作为主数据库来用。它的数据默认存在内存里，虽然可以配置持久化，但可靠性不如磁盘数据库。Redis 的正确用法是作为 PostgreSQL 或 MongoDB 前面的缓存层——先查缓存，缓存没有再查主库。

很多成熟的产品最终都会走向**多数据库组合**的架构：PostgreSQL 存核心业务数据，Redis 做缓存和实时功能，Elasticsearch 负责搜索。但这不是一开始就需要的东西，而是随着业务增长逐步引入的。过早地追求"完美的数据库架构"，只会增加不必要的复杂度。

## 小结

回到开头那个 `users.json` 的问题。文件存储的问题不在于它"不能存数据"，而在于它只管存，不管数据怎么被安全地读写。数据库提供的并发控制、事务保证、查询能力、数据校验、崩溃恢复，这些才是让数据从"存下来"变成"可靠地用起来"的关键。

关系型数据库和 NoSQL 之间不是"谁替代谁"的关系。它们各自擅长不同的场景：PostgreSQL 在事务和一致性上无可替代，MongoDB 在灵活文档上更顺手，Redis 在速度上碾压一切。理解它们各自的优势和局限，比纠结"哪个更好"有意义得多。

而 ACID 和 CAP 这些看起来枯燥的概念，实际上是在回答一个很朴素的问题：当你的系统变得更复杂、用户变得更多、故障开始发生的时候，你的数据还能不能被信任？数据库的存在，就是为了让这个答案始终是"能"。
