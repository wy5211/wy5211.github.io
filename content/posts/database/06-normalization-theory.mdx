上一篇我们用外键把 users、orders、products 连了起来。但有个问题没讨论：这些表里的字段该怎么组织？

你可能见过这样的 orders 表——为了"查询方便"，把用户信息也塞进了订单表：

```
orders 表
├── id
├── user_id
├── user_email          ← 冗余
├── user_full_name      ← 冗余
├── order_number
├── total
├── items               ← 一个字段存多个商品
└── created_at
```

运营说"我需要快速查订单列表，不想每次都 JOIN users 表"，于是你把用户邮箱和姓名复制了一份到 orders 表。产品说"一个订单可能有多个商品，用一个字段存就行"，于是你把商品列表存成了一个 JSON 数组。

一开始没问题。然后用户改了名字——你得同时更新 users 表和 orders 表里所有相关记录。忘了更新？数据就不一致了。一个订单的商品列表存在一个 JSON 字段里，你想统计"上个月卖了多少台 iPhone"，得把所有订单的 JSON 解析一遍再过滤，性能很差。

这就是**数据冗余**带来的两个经典问题：更新异常和查询困难。范式理论就是为了解决这些问题而存在的。但范式做到什么程度合适？完全规范化和完全冗余之间，有一个你需要根据业务场景来决定的灰色地带。

## 第一范式：每个字段只存一个值

第一范式（1NF）的要求很简单：表的每个字段必须包含原子值，不能在一个字段里存多个值。

用电商订单来举例。如果订单表是这样设计的：

```
| id | user_id | items                  | total   |
|----|---------|------------------------|---------|
| 1  | 42      | iPhone 15, AirPods     | 9798.00 |
| 2  | 43      | MacBook Pro, iPhone 15 | 22998.00|
```

`items` 字段里存了多个商品名称，用逗号分隔。这违反了 1NF。问题很明显：你想查"包含 iPhone 15 的订单"怎么办？用 `LIKE '%iPhone 15%'`？那如果有一个商品叫"iPhone 15 保护壳"也会匹配到。你想统计每个商品卖了多少件？你得用字符串函数把逗号分隔的值拆开再数。

满足 1NF 的做法是把这个字段拆开——每个商品一行：

```sql
-- 满足 1NF 的订单项表
CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER NOT NULL REFERENCES orders(id),
    product_name VARCHAR(255) NOT NULL,
    quantity INTEGER NOT NULL,
    unit_price NUMERIC(10, 2) NOT NULL
);
```

现在每个字段都是原子值，你可以精确地查询、统计、筛选。

你可能觉得 1NF 太显而易见了，谁会在一个字段里存多个值？实际上这种设计在新手项目中很常见，特别是用 MongoDB 习惯了的开发者——MongoDB 里把商品列表存成数组是很自然的做法，但搬到关系型数据库里就不合适了。

PostgreSQL 的数组类型（`TEXT[]`、`INTEGER[]`）看起来也违反了 1NF，但实际上 PostgreSQL 的数组类型是"原子的"——数据库能对数组元素建立索引（GIN 索引），能用 `ANY()` 操作符查询，能判断包含关系。所以用数组类型不算违反 1NF，前提是你需要用到数组的查询能力，而不是把数组当成字符串来存。

## 第二范式：消除部分依赖

第二范式（2NF）在 1NF 的基础上加了一个要求：非主键字段必须完全依赖于整个主键，不能只依赖主键的一部分。

这个要求主要影响**组合主键**的表。如果主键只有一个字段（比如自增 ID），那它天然满足 2NF。

假设你的 order_items 表用 `(order_id, product_id)` 作为组合主键：

```sql
CREATE TABLE order_items (
    order_id INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    product_name VARCHAR(255),     -- 依赖 product_id，不依赖 order_id
    unit_price NUMERIC(10, 2),     -- 依赖 product_id，不依赖 order_id
    quantity INTEGER NOT NULL,      -- 依赖 order_id 和 product_id
    PRIMARY KEY (order_id, product_id)
);
```

`product_name` 和 `unit_price` 只依赖于 `product_id`，不依赖于 `order_id`——同一件商品在不同订单里名字和单价是一样的（至少当前是）。这就是"部分依赖"。

部分依赖会导致什么问题？假设 AirPods 的价格从 1299 涨到了 1499，你得更新所有包含 AirPods 的订单项记录。漏了一条？数据就不一致了。

满足 2NF 的做法是把依赖 `product_id` 的字段拆到 products 表里：

```sql
-- products 表存商品信息
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    price NUMERIC(10, 2) NOT NULL
);

-- order_items 只存订单和商品的关系
CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER NOT NULL REFERENCES orders(id),
    product_id INTEGER NOT NULL REFERENCES products(id),
    quantity INTEGER NOT NULL,
    unit_price NUMERIC(10, 2) NOT NULL  -- 这是购买时的快照价格，不是当前价格
);
```

等等，`unit_price` 还是冗余的啊？没错——但这里的冗余是故意的。订单项里的 `unit_price` 是"用户购买时的价格"，它不应该跟着 products 表的价格变动。这种"历史快照"性质的冗余不属于范式要消除的范畴，因为它记录的是特定时刻的数据。

## 第三范式：消除传递依赖

第三范式（3NF）在 2NF 的基础上加了一个要求：非主键字段不能依赖于其他非主键字段。

回到 orders 表。假设你这样设计：

```sql
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id),
    user_email VARCHAR(255),       -- 通过 user_id 可以从 users 表得到
    user_full_name VARCHAR(100),   -- 通过 user_id 可以从 users 表得到
    order_number VARCHAR(50) UNIQUE NOT NULL,
    total NUMERIC(10, 2) NOT NULL
);
```

`user_email` 和 `user_full_name` 依赖于 `user_id`，而 `user_id` 是非主键字段。这就是传递依赖：`user_email → user_id → id`。

传递依赖的问题是更新异常。用户改了邮箱，你得同时更新 users 表和所有 orders 表里的 `user_email`。如果用户改了两次邮箱（先改成 A，又改成 B），你更新的时候可能漏掉了某些订单记录，导致同一个用户在不同订单里显示不同的邮箱。

满足 3NF 的做法：删掉 orders 表里的 `user_email` 和 `user_full_name`，需要的时候 JOIN users 表。

```sql
SELECT o.order_number, o.total, u.email, u.full_name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.id = 42;
```

## 大多数项目做到 3NF 就够了

你可能听说过 BCNF（Boyce-Codd 范式）、4NF、5NF 乃至 6NF。但在实际项目中，做到 3NF 就够了。原因很简单：3NF 之后的问题在实际业务中极少遇到。

BCNF 解决的是"一个表里有多个候选键"的场景——比如"学生选课表"里，(学生, 老师) 和 (学生, 课程) 都能唯一确定一行。这种设计本身就有问题，应该拆表而不是追求更高范式。

4NF 和 5NF 解决的是多值依赖和连接依赖，这些问题在正常业务设计中几乎不会出现。如果你在设计表的时候遇到了这些问题，大概率是表的设计本身就不对。

所以实用建议是：**确保每张表满足 3NF，不要为了追求更高范式而过度设计**。

## 什么时候该反范式

3NF 保证了一致性，但付出了查询性能的代价——每次需要关联数据就得 JOIN，JOIN 多了查询就慢。

电商的订单列表页就是一个典型场景。用户打开"我的订单"，需要看到：

```
订单号 | 商品名称 | 状态 | 总金额 | 下单时间
```

如果完全规范化，你需要 JOIN orders、order_items、products 三张表。用户量大的情况下，这个查询可能要几百毫秒。

一个常见的优化是把商品名称冗余到 order_items 表里：

```sql
CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER NOT NULL REFERENCES orders(id),
    product_id INTEGER NOT NULL REFERENCES products(id),
    product_name VARCHAR(255) NOT NULL,  -- 冗余：购买时的商品名
    quantity INTEGER NOT NULL,
    unit_price NUMERIC(10, 2) NOT NULL    -- 冗余：购买时的价格
);
```

这样查询订单列表只需要 JOIN orders 和 order_items 两张表，而且不需要 JOIN products 表。代价是：如果商品改名了，历史订单里的商品名不会更新——但这是合理的，历史记录本来就应该保持购买时的状态。

什么时候该反范式？有几个判断标准：

**读多写少的场景适合反范式**。报表查询、统计面板、商品列表——这些数据读的频率远高于写的频率。冗余一份能省掉每次查询的 JOIN，值得。

**需要记录历史快照的场景适合反范式**。订单项里的商品名和价格、发货记录里的收货地址——这些是"购买时刻"的数据，不应该跟着源数据的变化而变化。

**数据一致性要求极高的场景不适合反范式**。银行账户余额、库存数量——这些数据只能有一个"真相来源"，冗余了就要保证同步，一旦同步出了问题就是线上事故。

## 物化视图：反范式的数据库级方案

如果你需要一张"预先计算好的汇总表"，但不想在应用层维护它的同步，PostgreSQL 的物化视图是一个好选择。

假设运营需要一份"每日销售汇总"报表：

```sql
CREATE MATERIALIZED VIEW daily_sales AS
SELECT
    DATE(created_at) AS sale_date,
    COUNT(*) AS order_count,
    SUM(total) AS revenue,
    AVG(total) AS avg_order_value
FROM orders
WHERE status IN ('paid', 'shipped', 'delivered')
GROUP BY DATE(created_at);
```

物化视图和普通视图的区别在于：普通视图只是一个保存的查询，每次查询时重新执行；物化视图把查询结果**实际存下来**，查询时直接读取预计算的结果，速度很快。

代价是：物化视图的数据是"快照"，不会自动更新。你需要在数据变更后手动刷新：

```sql
REFRESH MATERIALIZED VIEW daily_sales;
```

物化视图有"快"和"旧"两个属性。它快是因为数据已经算好了，旧是因为不会实时更新。这和缓存的概念很像——用数据的"稍微旧一点"换取查询的"快很多"。

你可以用定时任务（crontab）来定期刷新物化视图。比如每 5 分钟刷新一次，对于运营报表来说足够了。

```sql
-- 并发刷新（PostgreSQL 9.4+），刷新期间物化视图仍然可查
REFRESH MATERIALIZED VIEW CONCURRENTLY daily_sales;
```

`CONCURRENTLY` 关键字很重要——普通的 `REFRESH` 会锁住物化视图，刷新期间查询会等待。加了 `CONCURRENTLY` 后，PostgreSQL 会在后台构建新数据，构建完成后再替换旧数据，查询不会被打断。使用 `CONCURRENTLY` 需要物化视图有至少一个唯一索引。

## 冗余数据的同步：应用层 vs 触发器

决定反范式之后，下一个问题是：冗余数据怎么保持同步？

比如你在 orders 表里冗余了 `user_email`（虽然 3NF 不建议这么做，但假设你有充分的理由）。当用户改了邮箱，你需要同时更新 users 表和 orders 表里的所有相关记录。

**方案一：应用层同步**。在"修改邮箱"的业务代码里，同时执行两条 SQL：

```sql
-- 先更新 users 表
UPDATE users SET email = 'new@example.com' WHERE id = 42;

-- 再更新 orders 表里的冗余字段
UPDATE orders SET user_email = 'new@example.com' WHERE user_id = 42;
```

优点是逻辑清晰，开发者能一眼看懂"改邮箱"会影响哪些表。缺点是如果以后有新的地方也需要这个冗余字段（比如加了一张 user_logs 表），你得找到所有更新邮箱的代码路径加上新逻辑，容易遗漏。

**方案二：数据库触发器**。在 users 表上建一个触发器，邮箱更新后自动同步到 orders 表：

```sql
CREATE OR REPLACE FUNCTION sync_user_email_to_orders()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.email <> OLD.email THEN
        UPDATE orders SET user_email = NEW.email WHERE user_id = NEW.id;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_sync_user_email
AFTER UPDATE ON users
FOR EACH ROW
EXECUTE FUNCTION sync_user_email_to_orders();
```

优点是无论从哪里更新邮箱（API、管理后台、手动 SQL），触发器都会自动同步，不会遗漏。缺点是触发器的逻辑藏在数据库里，应用开发者不一定知道它的存在，调试的时候可能困惑"我没更新 orders 表，它怎么变了"。

两种方案各有适用场景。我的建议是：**如果冗余字段只在一两个业务操作里更新，用应用层同步——简单直接。如果冗余字段可能被多种操作更新（比如用户名、邮箱这种基础信息），用触发器——不容易遗漏**。

但有一个原则不变：触发器只适合维护简单的冗余同步，不适合放复杂的业务逻辑。如果你发现触发器里写了 50 行代码，多半是设计出了问题。

## 小结

范式不是教条，是工具。1NF 告诉你"一个字段只存一件事"，2NF 告诉你"别在组合主键的表里塞不相关的字段"，3NF 告诉你"能通过外键查到的数据就别冗余"。这三个规则能帮你避免大多数设计问题。

反范式也不是偷懒。它是当你发现规范化导致了明显的性能瓶颈时，有意识地引入冗余来换取速度。冗余了就要考虑同步——用应用层代码同步还是用触发器，取决于更新路径的复杂度。

物化视图是反范式的一种特殊形式——数据库帮你预计算并缓存结果。适合读多写少的报表和统计场景。

下一篇我们开始讲索引——这是让查询变快的最直接手段。没有索引的数据库就像一本没有目录的百科全书，能找到答案，但得从头翻到尾。
