## 前言

你的查询慢吗？添加索引可能让速度提升 100 倍！

但索引不是万能的：滥用索引会降低写入性能，占用大量存储空间。什么时候该建索引？哪种索引类型最合适？

本文将深入讲解索引的工作原理，帮助你合理设计索引，大幅提升查询性能。

---

## 核心概念

### 什么是索引？

**索引**是数据库中用于加速查询的数据结构，类似于书的目录。

**索引优势**：

- ✅ 大幅提升查询速度（10-1000 倍）
- ✅ 加速 ORDER BY、GROUP BY
- ✅ 强制唯一性（UNIQUE 索引）

**索引代价**：

- ❌ 占用存储空间
- ❌ 降低写入性能（INSERT、UPDATE、DELETE）
- ❌ 需要维护成本

**类比**：

- 无索引：从第一页翻到最后一页找内容
- 有索引：直接查看目录，跳转到对应页码

---

## 索引数据结构

### B-Tree 索引（默认）

PostgreSQL 默认的索引类型，适用于大部分场景。

```sql
-- 创建 B-Tree 索引
CREATE INDEX idx_users_email ON users(email);

-- 查看执行计划
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'user@example.com';

-- 输出：Index Scan using idx_users_email on users（使用索引）
```

**B-Tree 特点**：

- ✅ 平衡树结构
- ✅ 时间复杂度：O(log n)
- ✅ 支持 =、<、>、<=、>=、BETWEEN
- ✅ 适合单列、范围查询

**适用场景**：

- 等值查询：WHERE email = '...'
- 范围查询：WHERE created_at > '...'
- 排序：ORDER BY created_at

---

### Hash 索引

只支持等值查询，但速度更快。

```sql
-- 创建 Hash 索引
CREATE INDEX idx_users_hash_email ON users USING HASH (email);

-- 只支持 = 操作
SELECT * FROM users WHERE email = 'user@example.com';  -- ✅ 使用索引
SELECT * FROM users WHERE email LIKE 'user%';          -- ❌ 不使用索引
```

**Hash 索引特点**：

- ✅ 等值查询极快
- ❌ 不支持范围查询
- ❌ 不支持排序

---

### GIN 索引（倒排索引）

适用于 JSON、数组、全文搜索。

```sql
-- JSON 字段索引
CREATE INDEX idx_users_preferences ON users USING GIN (preferences);

-- 数组索引
CREATE INDEX idx_products_tags ON products USING GIN (tags);

-- 查询 JSON 字段
SELECT * FROM users WHERE preferences @> '{"theme": "dark"}';

-- 查询数组
SELECT * FROM products WHERE '手机' = ANY(tags);
```

**GIN 索引特点**：

- ✅ 适合多值类型（JSON、数组）
- ✅ 全文搜索
- ❌ 占用空间大
- ❌ 写入性能差

---

### GiST 索引

适用于地理数据、范围查询。

```sql
-- 安装 PostGIS 扩展
CREATE EXTENSION postgis;

-- 创建地理数据表
CREATE TABLE locations (
  id UUID PRIMARY KEY,
  name VARCHAR(255),
  geom GEOMETRY(Point, 4326)
);

-- 创建 GiST 索引
CREATE INDEX idx_locations_geom ON locations USING GIST (geom);

-- 地理查询（附近 1km 的点）
SELECT * FROM locations
WHERE ST_DWithin(geom, ST_SetSRID(ST_MakePoint(116.4074, 39.9042), 4326), 1000);
```

---

### BRIN 索引（块范围索引）

适用于大数据表，占用空间小。

```sql
-- 创建 BRIN 索引（适合时间序列数据）
CREATE INDEX idx_orders_created ON orders USING BRIN (created_at);

-- 适合查询：WHERE created_at > '2024-01-01'
```

**BRIN 索引特点**：

- ✅ 占用空间极小
- ✅ 适合大数据表（亿级）
- ❌ 只适合有序数据

---

## 索引类型

### 1. 单列索引

```sql
-- 单列索引
CREATE INDEX idx_users_email ON users(email);

-- 查询
SELECT * FROM users WHERE email = 'user@example.com';
```

---

### 2. 复合索引（多列索引）

```sql
-- 复合索引（注意列顺序）
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- ✅ 使用索引
SELECT * FROM orders WHERE user_id = 'user-1' AND status = 'paid';
SELECT * FROM orders WHERE user_id = 'user-1';

-- ❌ 不使用索引（跳过了第一列）
SELECT * FROM orders WHERE status = 'paid';
```

**复合索引列顺序原则**：

- ✅ 高选择性列在前（区分度高的列）
- ✅ 频繁查询的列在前
- ✅ 范围查询列放在后面

---

### 3. 唯一索引

```sql
-- 唯一索引（自动创建 UNIQUE 约束）
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- 等价于
ALTER TABLE users ADD CONSTRAINT unique_email UNIQUE (email);
```

---

### 4. 部分索引（WHERE 条件）

```sql
-- 部分索引（只索引活跃用户）
CREATE INDEX idx_users_active ON users(email) WHERE is_active = TRUE;

-- ✅ 使用索引
SELECT * FROM users WHERE is_active = TRUE AND email = 'user@example.com';

-- ❌ 不使用索引
SELECT * FROM users WHERE email = 'user@example.com';
```

**部分索引优势**：

- ✅ 减少索引大小
- ✅ 提升查询性能
- ✅ 适合有频繁过滤条件的场景

---

### 5. 表达式索引

```sql
-- 表达式索引
CREATE INDEX idx_users_lower_email ON users(LOWER(email));
CREATE INDEX idx_orders_date ON orders(DATE(created_at));

-- ✅ 使用索引
SELECT * FROM users WHERE LOWER(email) = 'USER@EXAMPLE.COM';
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-15';
```

---

## 索引设计原则

### ✅ 应该建索引的场景

```sql
-- 1. 频繁作为 WHERE 条件的列
CREATE INDEX idx_users_email ON users(email);

-- 2. JOIN 连接的列（外键）
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 3. ORDER BY、GROUP BY 的列
CREATE INDEX idx_orders_created ON orders(created_at);

-- 4. UNIQUE 约束（自动建索引）
ALTER TABLE users ADD CONSTRAINT unique_email UNIQUE (email);
```

---

### ❌ 不应该建索引的场景

```sql
-- 1. 很少查询的列
-- ❌ CREATE INDEX idx_users_bio ON users(bio);

-- 2. 数据量小的表（< 1000 行）
-- ❌ CREATE INDEX idx_config ON config(key);

-- 3. 频繁更新的列（写入性能差）
-- ❌ CREATE INDEX idx_logs_created ON logs(created_at);

-- 4. 区分度低的列（性别、状态）
-- ❌ CREATE INDEX idx_users_gender ON users(gender);
```

## 索引实战案例

### 案例 1：用户登录

```sql
-- 需求：根据邮箱查询用户
-- ✅ 创建索引
CREATE INDEX idx_users_email ON users(email);

-- 查询（使用索引）
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'user@example.com';
```

---

### 案例 2：订单列表

```sql
-- 需求：查询用户的已支付订单，按创建时间排序
-- ✅ 创建复合索引
CREATE INDEX idx_orders_user_status_created ON orders(user_id, status, created_at DESC);

-- 查询（使用索引）
SELECT * FROM orders
WHERE user_id = 'user-1' AND status = 'paid'
ORDER BY created_at DESC
LIMIT 20;
```

---

### 案例 3：全文搜索

```sql
-- 需求：搜索商品名称和描述
-- ✅ 创建 GIN 索引
CREATE INDEX idx_products_name ON products USING GIN(to_tsvector('chinese', name));
CREATE INDEX idx_products_description ON products USING GIN(to_tsvector('chinese', description));

-- 全文搜索
SELECT * FROM products
WHERE to_tsvector('chinese', name || ' ' || description) @@ to_tsquery('chinese', 'iPhone & 5G');
```

## 索引维护

### 查看索引

```sql
-- 查看表的所有索引
SELECT
  indexname,
  indexdef
FROM pg_indexes
WHERE tablename = 'users';

-- 查看索引使用情况
SELECT
  schemaname,
  tablename,
  indexname,
  idx_scan as index_scans,
  idx_tup_read as tuples_read,
  idx_tup_fetch as tuples_fetched
FROM pg_stat_user_indexes
WHERE tablename = 'users'
ORDER BY idx_scan ASC;

-- 查找未使用的索引
SELECT
  schemaname,
  tablename,
  indexname
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexname NOT LIKE '%_pkey';
```

---

### 重建索引

```sql
-- 重建表的所有索引
REINDEX TABLE users;

-- 重建单个索引
REINDEX INDEX idx_users_email;

-- 并发重建（不锁表）
CREATE INDEX CONCURRENTLY idx_users_email_new ON users(email);
DROP INDEX idx_users_email;
ALTER INDEX idx_users_email_new RENAME TO idx_users_email;
```

---

### 删除索引

```sql
-- 删除索引
DROP INDEX idx_users_email;

-- 删除索引（如果存在）
DROP INDEX IF EXISTS idx_users_email;
```

## 最佳实践

### 1. 索引设计原则

- ✅ 外键默认建索引
- ✅ 频繁查询的字段建索引
- ✅ 复合索引注意列顺序
- ✅ 使用部分索引减少大小
- ❌ 避免过多索引（影响写入）
- ❌ 避免重复索引

---

### 2. 索引优化

```sql
-- ✅ 使用 EXPLAIN ANALYZE 分析查询
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'user@example.com';

-- ✅ 定期检查未使用的索引
SELECT * FROM pg_stat_user_indexes WHERE idx_scan = 0;

-- ✅ 定期重建索引
REINDEX TABLE users;

-- ✅ 使用 CONCURRENTLY 并发重建
CREATE INDEX CONCURRENTLY idx_users_email ON users(email);
```

---

### 3. 监控索引

```sql
-- 查看索引大小
SELECT
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexrelid)) as size
FROM pg_stat_user_indexes
ORDER BY pg_relation_size(indexrelid) DESC;

-- 查看索引使用率
SELECT
  tablename,
  indexname,
  idx_scan,
  idx_tup_read,
  idx_tup_fetch
FROM pg_stat_user_indexes
WHERE idx_scan = 0;
```

## 常见问题

### Q1: 为什么查询没使用索引？

**可能原因**：

- ❌ 索引列被函数包裹：`WHERE LOWER(email) = '...'`
- ❌ 隐式类型转换：`WHERE email = 123`（email 是字符串）
- ❌ 数据量太小，直接全表扫描更快
- ❌ 使用了 OR、NOT IN 等
- ❌ 统计信息过时

**解决方案**：

```sql
-- 1. 使用表达式索引
CREATE INDEX idx_users_lower_email ON users(LOWER(email));

-- 2. 避免类型转换
WHERE email = '123'  -- 而不是 123

-- 3. 更新统计信息
ANALYZE users;
```

---

### Q2: 复合索引的列顺序重要吗？

**非常重要**！

```sql
-- 索引：(user_id, status, created_at)

-- ✅ 使用索引
WHERE user_id = '...'
WHERE user_id = '...' AND status = '...'
WHERE user_id = '...' AND status = '...' ORDER BY created_at

-- ❌ 不使用索引（跳过了第一列）
WHERE status = '...'
WHERE created_at > '...'
```

**原则**：高选择性、高频查询的列放前面。

---

### Q3: 索引越多越好吗？

**不是**！

**索引的代价**：

- ❌ 占用存储空间
- ❌ 降低写入性能（INSERT、UPDATE、DELETE）
- ❌ 需要维护成本

**建议**：

- ✅ 只为必要的列建索引
- ✅ 定期删除未使用的索引
- ✅ 复合索引减少索引数量

---

### Q4: 如何判断索引是否有效？

```sql
-- 使用 EXPLAIN ANALYZE
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'user@example.com';

-- 查看输出
-- Index Scan using idx_users_email on users (cost=0.42..8.44 rows=1 width=200)
```

**关键指标**：

- `Index Scan`：使用索引
- `Seq Scan`：全表扫描（慢）
- `cost`：成本估算（越低越好）

---

### Q5: 什么时候用 GIN 索引？

**适用场景**：

- ✅ JSON 字段查询
- ✅ 数组查询
- ✅ 全文搜索
- ✅ 范围查询（range types）

**不适用**：

- ❌ 单列等值查询（用 B-Tree）
- ❌ 数据频繁更新
