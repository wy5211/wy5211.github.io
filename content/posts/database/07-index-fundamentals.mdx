上一篇聊了范式和反范式。现在假设你的电商系统已经上线，表设计得也不错，规范化做得挺好。但运营反馈了一个问题：订单列表页打开要 3 秒，用户管理后台查一个邮箱要 5 秒。你打开数据库一看，users 表 50 万行，orders 表 200 万行——没有索引。

你给 orders 表的 `user_id` 加了一个索引，再查，30 毫秒。快了 100 倍。

索引就是这么一个东西——它不改变你的数据，不改变你的查询逻辑，只是让数据库更快地找到数据。但索引也不是免费的，每次写入都要维护它，建多了反而拖慢写入速度。什么时候该建、建什么类型的、怎么建才有效，这些是需要认真对待的。

## 索引的本质：有序的数据结构

没有索引的时候，数据库要找一条记录只能全表扫描——从第一行开始，一行一行地看，直到找到目标。如果 users 表有 50 万行，你要找邮箱是 `zhang_wei@example.com` 的用户，最坏情况下要看 50 万行。

索引的思路是：**把某个列的值排序，建一个查找结构**。PostgreSQL 默认的索引类型是 B-Tree，你可以把它想象成一棵多层有序的树。

拿 users 表的 email 列来举例。如果给 email 建了索引，数据库会维护一个类似这样的结构：

```
                        [M]
                    /        \
              [F]                [S]
            /    \            /      \
        [alice]  [li_na]   [tom]   [zhang_wei]
```

这不是二叉树——B-Tree 的每个节点可以有很多子节点。实际上一棵三层 B-Tree 大概能存几百万条记录。查找的时候从根节点开始，比较目标值和节点值，决定往左还是往右，通常 3-4 次比较就能找到目标。

50 万行全表扫描 vs 3-4 次 B-Tree 查找，这就是 100 倍性能差异的来源。

B-Tree 之所以快，是因为它是有序的。有序意味着数据库可以用二分查找，而不是线性扫描。这也是为什么 B-Tree 索引不仅加速等值查询（`WHERE email = 'xxx'`），还加速范围查询（`WHERE created_at BETWEEN '2025-03-01' AND '2025-03-31'`）和排序（`ORDER BY created_at DESC`）。

## 复合索引和最左前缀

实际业务中，你经常需要同时按多个条件查询。比如运营后台的订单列表：

```sql
SELECT * FROM orders
WHERE user_id = 42 AND status = 'paid'
ORDER BY created_at DESC
LIMIT 20;
```

这里涉及三个列：`user_id`、`status`、`created_at`。你是建三个单独的索引，还是建一个复合索引？

复合索引是把多个列的值组合成一个有序结构。假设你建了 `CREATE INDEX idx_orders_user_status ON orders(user_id, status)`，数据库会先按 `user_id` 排序，`user_id` 相同的再按 `status` 排序：

```
user_id=1, status='cancelled'
user_id=1, status='paid'
user_id=1, status='paid'
user_id=1, status='shipped'
user_id=2, status='pending'
user_id=2, status='paid'
...
```

这个结构有一个重要的特性：**最左前缀原则**。索引 `(user_id, status)` 能用于 `WHERE user_id = ?` 的查询，也能用于 `WHERE user_id = ? AND status = ?` 的查询，但不能用于只有 `WHERE status = ?` 的查询。

原因是索引是先按 `user_id` 排序的。如果你只查 `status = 'paid'`，数据库得遍历所有 `user_id` 的分组才能找到匹配的行——这和全表扫描没太大区别。

所以复合索引的列顺序很重要。把区分度高的列放左边。`user_id` 有几万个不同值，`status` 只有 5 个值（pending、paid、shipped、delivered、cancelled）。把 `user_id` 放左边，索引的第一层就能把数据分到几万个桶里，效率高得多。

那上面的查询还涉及 `ORDER BY created_at DESC` 怎么办？你可以把 `created_at` 也加到索引里：

```sql
CREATE INDEX idx_orders_user_status_created
ON orders(user_id, status, created_at DESC);
```

这样查询 `WHERE user_id = 42 AND status = 'paid' ORDER BY created_at DESC` 时，数据库可以直接从索引里按顺序读出结果，不需要额外的排序操作（filesort）。

## 部分索引：只索引需要的数据

电商系统里有一个典型的慢查询：用户登录后查自己的订单列表。这个查询几乎每次都会带 `WHERE user_id = ?` 条件，而且大部分订单的状态不是 `cancelled`。

如果你给 `user_id` 建一个普通索引，这个索引会包含所有订单——包括已取消的。但用户看自己的订单列表时，通常不关心已取消的订单（或者只需要统计数量）。已取消的订单占了索引空间但不怎么被查询，这是一种浪费。

部分索引让你只索引满足条件的数据：

```sql
-- 只索引非取消状态的订单
CREATE INDEX idx_orders_active ON orders(user_id)
WHERE status != 'cancelled';
```

这个索引只包含 `status != 'cancelled'` 的订单行。好处是明显的：索引体积更小，查询非取消订单时更快（因为要扫描的索引条目更少）。维护成本也更低——取消一个订单时不需要更新这个索引。

部分索引的另一个用途是处理"倾斜数据"。假设你有一个 `is_active` 布尔字段，99% 的记录是 `TRUE`。你不需要给 `is_active` 建一个包含所有记录的索引，只需要建一个只索引 `FALSE` 记录的部分索引：

```sql
CREATE INDEX idx_products_inactive ON products(id)
WHERE is_active = FALSE;
```

查"未上架的商品"时走这个小索引就够了。

## 表达式索引：解决函数导致索引失效的问题

假设用户搜索商品时经常用小写输入，你的查询是：

```sql
SELECT * FROM products WHERE LOWER(name) = 'iphone 15';
```

即使你给 `name` 建了索引，这个查询也不会走索引。原因是：索引里存的是原始值 `'iPhone 15'`，但你的查询条件是 `LOWER(name)` 的结果 `'iphone 15'`。数据库不知道 `LOWER('iPhone 15')` 等于 `'iphone 15'`，所以只能全表扫描，对每一行都执行 `LOWER(name)` 再比较。

解决方案是建一个表达式索引：

```sql
CREATE INDEX idx_products_lower_name ON products(LOWER(name));
```

这个索引存的是 `LOWER(name)` 的结果，而不是原始值。现在查询 `WHERE LOWER(name) = 'iphone 15'` 就能直接走索引了。

类似的情况还有：用 `COALESCE` 处理 NULL 值查询、用日期函数做范围查询、用字符串函数做模式匹配。只要 WHERE 条件里对列使用了函数，普通索引就可能失效，这时候就需要表达式索引。

```sql
-- 按月查询订单
CREATE INDEX idx_orders_created_month
ON orders(DATE_TRUNC('month', created_at));

-- 查询时能走索引
SELECT * FROM orders
WHERE DATE_TRUNC('month', created_at) = '2025-03-01';
```

## GIN 索引：JSONB 和数组的专用索引

上一篇我们聊到 products 表有一个 `attributes` JSONB 字段，存商品的自定义属性。你可能想查询"所有存储量为 256GB 的手机"：

```sql
SELECT name, price, attributes
FROM products
WHERE attributes->>'storage' = '256GB';
```

B-Tree 索引帮不上忙——它索引的是整个 JSONB 值，不是内部的某个字段。你需要 GIN（Generalized Inverted Index）索引：

```sql
CREATE INDEX idx_products_attributes ON products USING GIN (attributes);
```

GIN 索引会把 JSONB 文档中的每个键值对都提取出来，建立倒排索引。`{"storage": "256GB", "color": "黑色"}` 会被拆成两个条目：`storage:256GB` 和 `color:黑色`。查询 `WHERE attributes->>'storage' = '256GB'` 时，GIN 索引可以直接定位到包含这个键值对的文档。

GIN 索引也支持数组类型。如果你有一个 `tags TEXT[]` 字段，建了 GIN 索引后，`WHERE '手机' = ANY(tags)` 查询就能走索引。

```sql
CREATE INDEX idx_products_tags ON products USING GIN (tags);

-- 走索引
SELECT name FROM products WHERE '降噪' = ANY(tags);
```

GIN 索引的查询速度很快，但写入比 B-Tree 慢——因为每次更新 JSONB 字段都需要重新提取键值对并更新倒排索引。所以只在你确实需要查询 JSONB 内部字段的时候才加 GIN 索引。

## 索引的代价：不是免费的

索引加速了查询，但拖慢了写入。每次 INSERT、UPDATE、DELETE，数据库不仅要修改表数据，还要同步更新所有相关的索引。

一个 INSERT 操作如果没有索引，只需要写一次表数据。如果有 5 个索引，就要写 6 次——1 次表 + 5 次索引。索引越多，写入越慢。

索引还占磁盘空间。一个 B-Tree 索引的大小大约是它索引的列的数据量的 1.2-1.5 倍。如果你给 orders 表的 6 个列各建了一个索引，索引占用的空间可能比表数据本身还大。

所以索引的设计是一个 trade-off。我的建议：

**主键索引必须有**，这是最基本的。**外键索引必须有**，上一篇已经解释了原因。**频繁出现在 WHERE 条件中的列，建索引**。**出现在 ORDER BY 中的列，如果同时有 WHERE 条件，考虑复合索引**。**很少查询的列，不建索引**——比如 `admin_notes` 这种只在管理后台偶尔看的字段。

一个实用的检查方法：用 `EXPLAIN ANALYZE` 看你的查询走了什么扫描方式。如果是 `Seq Scan`（全表扫描），而且这张表的数据量在持续增长，那就该考虑加索引了。如果已经是 `Index Scan` 或 `Index Only Scan`，说明索引已经生效了。

## 小结

索引的本质是有序的数据结构。B-Tree 索引通过保持有序来支持快速查找，复合索引通过最左前缀原则支持多条件查询，部分索引通过只索引需要的行来减少维护成本，表达式索引解决了函数导致索引失效的问题，GIN 索引是 JSONB 和数组字段的专用武器。

但索引不是越多越好。每个索引都是写入时的负担和磁盘上的开销。你需要根据实际的查询模式来决定建什么索引，而不是"所有列都建一个以防万一"。
