上一篇我们为电商系统设计了几张独立的表——users、products、orders。但现实里这些表不是孤立存在的：订单属于某个用户，订单包含多个商品，商品属于某个分类。这些"关系"才是关系型数据库的灵魂。

你可能遇到过这种场景：运营让你导出一份"过去 30 天每个用户的订单详情"，你写了四五个 JOIN 把 users、orders、order_items、products 连起来，结果查了 30 秒还没出来。DBA 看了一眼说"你的外键没加索引"。你可能还遇到过删除用户时直接报错"因为有订单引用了这个用户"，不知道该 CASCADE 还是 RESTRICT。

这篇文章我们就用电商系统把这些关系搞清楚。

## 一对多：最常见的关系

电商系统里到处都是一对多关系。一个用户有多个订单，一个分类有多个商品，一个订单有多个订单项。

一对多关系的实现方式很直接：在"多"的那张表里加一个外键列，指向"一"的那张表的主键。

```sql
-- 一个用户有多个订单
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id),
    order_number VARCHAR(50) UNIQUE NOT NULL,
    total NUMERIC(10, 2) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

`user_id` 就是外键。它告诉数据库："这个订单属于哪个用户"。`REFERENCES users(id)` 建立了约束——你不能插入一个 `user_id` 在 users 表里不存在的订单。

这里有一个很多人忽略的事情：**外键列必须加索引**。PostgreSQL 不会自动为外键列创建索引，但每次你删除或更新 users 表的记录时，数据库都要去 orders 表里检查有没有引用这条记录。如果没有索引，这个检查就是全表扫描——当 orders 表有百万行的时候，删一个用户可能要好几秒。

```sql
-- 手动为外键创建索引
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

JOIN 查询也需要这个索引。当你写 `SELECT * FROM users u JOIN orders o ON u.id = o.user_id` 的时候，如果没有索引，数据库会扫描 orders 表的每一行来匹配 users 表的记录。加了索引之后，数据库可以直接通过索引找到匹配的行，速度快几个数量级。

## 外键的 ON DELETE：删用户时怎么办

假设运营要删除一个违规用户。这个用户有 50 个订单、200 条操作日志、30 条收货地址。你一执行 `DELETE FROM users WHERE id = 42`，数据库直接报错：

```
ERROR: update or delete on table "users" violates foreign key constraint
"orders_user_id_fkey" on table "orders"
```

这是数据库在保护你——如果没有外键约束，你删除了用户但忘了处理他的订单，那些订单就变成了"孤儿数据"，`user_id` 指向一个不存在的用户，查询的时候 LEFT JOIN 出来全是 NULL。

怎么处理？外键约束提供了几种策略：

**CASCADE**：删用户时，连带删除他的所有订单。适合"用户没了，数据也没意义"的场景。用户的收货地址、购物车、收藏夹——这些跟着用户走的数据，删用户时一起删掉是合理的。

```sql
CREATE TABLE user_addresses (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    province VARCHAR(50) NOT NULL,
    city VARCHAR(50) NOT NULL,
    address VARCHAR(255) NOT NULL
);
```

**RESTRICT**（默认）：有订单的用户不能删。适合"数据有独立价值"的场景。订单属于用户，但订单本身也有财务价值——你不能因为用户被删了就把订单记录也删了。

```sql
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    total NUMERIC(10, 2) NOT NULL
);
```

**SET NULL**：删用户时，把订单的 `user_id` 设为 NULL。适合"关系可以断裂但数据要保留"的场景。比如商品和分类的关系——分类删了，商品还在，只是暂时没有分类。

```sql
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    category_id INTEGER REFERENCES categories(id) ON DELETE SET NULL
);
```

选择哪种策略取决于业务需求。我的习惯是：**对用户私有数据用 CASCADE（地址、偏好设置），对有独立业务价值的数据用 RESTRICT（订单、评价），对可选关联用 SET NULL（分类、标签）**。

实际项目里，很多团队会采用"软删除"来绕开这个问题：不真删用户，而是在 users 表加一个 `is_deleted BOOLEAN DEFAULT FALSE`，查询时过滤掉已删除的。这样既保留了数据（用户可能回来），又避免了外键约束的冲突。但这会导致"活跃用户"查询总是带一个 `WHERE is_deleted = FALSE` 条件，时间长了会有点烦。

## 多对多：中间表

订单和商品是多对多关系——一个订单包含多个商品，一个商品出现在多个订单里。关系型数据库不能直接表达多对多，需要一张中间表来拆成两个一对多。

```sql
-- 中间表：order_items
CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
    product_id INTEGER NOT NULL REFERENCES products(id) ON DELETE RESTRICT,
    quantity INTEGER NOT NULL CHECK (quantity > 0),
    unit_price NUMERIC(10, 2) NOT NULL,
    PRIMARY KEY (order_id, product_id)
);

CREATE INDEX idx_order_items_product ON order_items(product_id);
```

注意几个设计决策。`ON DELETE CASCADE` 在 order_id 上——订单删了，订单项跟着删，这没问题。但 `ON DELETE RESTRICT` 在 product_id 上——如果这个商品被某个订单引用了，就不能删商品。这是合理的：商品被买过，订单记录需要保留商品信息，不能让商品被删掉导致订单项找不到对应的商品。

`unit_price` 冗余存了购买时的单价。为什么不在查询时 JOIN products 表取最新价格？因为价格会变。用户去年买的时候是 2499，现在降到了 1999，订单记录里应该显示购买时的价格 2499，而不是当前价格。这种"记录历史快照"的冗余是合理的，我们在下一篇讲范式的时候会详细讨论。

`PRIMARY KEY (order_id, product_id)` 是组合主键，确保同一个订单里不会重复添加同一个商品。如果用户买了两件同样的商品，应该把 `quantity` 设为 2，而不是插入两条记录。

## JOIN：理解它而不是背它

JOIN 的语法很简单，但很多人只记住了"INNER JOIN 返回两边都有的，LEFT JOIN 返回左边所有的"，不理解背后的逻辑。

JOIN 的本质是：**从左表取一行，去右表找所有满足条件的行，把它们拼在一起**。就这么简单。

用电商数据来演示。假设 users 表和 orders 表的数据是这样的：

**users**：

| id  | email                 | full_name |
| --- | --------------------- | --------- |
| 1   | zhang_wei@example.com | 张伟      |
| 2   | li_na@example.com     | 李娜      |
| 3   | wang_fang@example.com | 王芳      |

**orders**：

| id  | user_id | order_number      | total   |
| --- | ------- | ----------------- | ------- |
| 101 | 1       | ORD-20250315-a8f3 | 2499.00 |
| 102 | 1       | ORD-20250322-c7d1 | 599.00  |
| 103 | 4       | ORD-20250328-e4b2 | 1299.00 |

注意 user_id = 4 的订单引用了一个不存在的用户（可能被删了），而李娜（id = 2）和王芳（id = 3）还没有下过订单。

INNER JOIN 从 users 取第一行（张伟），去 orders 里找 `user_id = 1` 的行，找到两条，拼成两行结果。然后取第二行（李娜），找 `user_id = 2`，没找到，跳过。第三行（王芳），同样没找到，跳过。最终结果是两行。

```sql
SELECT u.full_name, o.order_number, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
```

| full_name | order_number      | total   |
| --------- | ----------------- | ------- |
| 张伟      | ORD-20250315-a8f3 | 2499.00 |
| 张伟      | ORD-20250322-c7d1 | 599.00  |

LEFT JOIN 的逻辑一样，只是如果右表没有匹配的行，会用 NULL 填充：

```sql
SELECT u.full_name, o.order_number, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
```

| full_name | order_number      | total   |
| --------- | ----------------- | ------- |
| 张伟      | ORD-20250315-a8f3 | 2499.00 |
| 张伟      | ORD-20250322-c7d1 | 599.00  |
| 李娜      | NULL              | NULL    |
| 王芳      | NULL              | NULL    |

LEFT JOIN 有一个容易忽视的问题：如果右表有多条匹配的行，左表的行会被"膨胀"。张伟有 2 个订单，所以在结果里出现了 2 次。如果你只是想统计每个用户的订单数量，应该先聚合再 JOIN，或者用子查询。

```sql
-- 先聚合再 JOIN，避免行膨胀
SELECT u.full_name, COALESCE(order_stats.order_count, 0) AS order_count
FROM users u
LEFT JOIN (
    SELECT user_id, COUNT(*) AS order_count
    FROM orders
    GROUP BY user_id
) order_stats ON u.id = order_stats.user_id;
```

`COALESCE` 把 NULL 转成 0，这样没有订单的用户也会显示 0 而不是 NULL。

RIGHT JOIN 和 FULL JOIN 在实际中用得不多。RIGHT JOIN 就是反过来写 LEFT JOIN，FULL JOIN 是两边都保留。你可能在数据对账的时候用得到 FULL JOIN——比较两张表，找出只在左边有的、只在右边有的、两边都有的记录。

## 自关联：商品分类的树形结构

有一种特殊的关系是"表和自己关联"。最典型的例子是商品分类——数码是一个顶级分类，手机和电脑是数码的子分类，iPhone 和 MacBook 又是手机的子分类。

```sql
CREATE TABLE categories (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    parent_id INTEGER REFERENCES categories(id) ON DELETE SET NULL,
    sort_order INTEGER DEFAULT 0
);
```

`parent_id` 指向同一张表的 `id`，这就是自关联。顶级分类的 `parent_id` 是 NULL。

查询分类树需要递归，PostgreSQL 的递归 CTE 正好做这件事：

```sql
WITH RECURSIVE category_tree AS (
    -- 基础查询：找出所有顶级分类
    SELECT id, name, parent_id, name AS path, 0 AS level
    FROM categories
    WHERE parent_id IS NULL

    UNION ALL

    -- 递归查询：找子分类
    SELECT c.id, c.name, c.parent_id,
           ct.path || ' > ' || c.name,
           ct.level + 1
    FROM categories c
    INNER JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT * FROM category_tree ORDER BY path;
```

结果大概是这样：

| id  | name   | path                 | level |
| --- | ------ | -------------------- | ----- |
| 1   | 数码   | 数码                 | 0     |
| 2   | 手机   | 数码 > 手机          | 1     |
| 4   | iPhone | 数码 > 手机 > iPhone | 2     |
| 3   | 电脑   | 数码 > 电脑          | 1     |
| 5   | 服装   | 服装                 | 0     |

递归 CTE 的工作方式是：先执行基础查询（UNION ALL 上面的部分），得到顶级分类。然后把这些结果"喂"给下面的递归查询，找到它们的子分类。再把子分类的结果"喂"回去，继续找下一层。直到某一次递归没有返回新行，递归结束。

递归 CTE 的性能在层级不深的时候（几十层以内）完全没问题。但如果你有一棵几千层深的树（比如评论的嵌套回复），可能需要考虑用"闭包表"或"路径枚举"（在每条记录里存完整的路径，如 `1/2/4`）来优化。

## JOIN 的性能：小表驱动大表

写 JOIN 查询时，表的顺序会影响性能吗？

对于 INNER JOIN，PostgreSQL 的查询优化器会自动决定先扫描哪张表。你写 `FROM users JOIN orders` 和 `FROM orders JOIN users`，优化器会根据统计信息选择更高效的执行顺序。所以你不需要纠结 JOIN 的书写顺序。

但对于 LEFT JOIN，顺序是有语义的——左表必须全部保留。如果你不小心把大表放左边，可能产生大量结果行。

真正影响性能的是**索引**。确保 JOIN 条件中的列（通常是外键列）有索引。没有索引的 JOIN 就是笛卡尔积——左表每一行都和右表每一行比较。两张一万行的表，没有索引的 JOIN 可能要做 1 亿次比较。

另一个常见问题是**过度 JOIN**。运营要一份报表，你写了 6 个 JOIN 把 users、orders、order_items、products、categories、coupons 全连起来。6 个表的笛卡尔积空间非常大，即使有索引，优化器也不一定选对执行计划。如果报表查询太复杂，考虑用 CTE 分步处理，或者用物化视图预计算。

## 小结

关系设计的关键不是记住哪种关系用什么语法，而是理解每种关系背后的业务含义。一对多是最常见的，外键加索引是铁律。多对多需要中间表，中间表里往往要存一些"快照数据"（比如购买时的价格）。ON DELETE 策略的选择取决于数据的生命周期——跟着父记录走的数据用 CASCADE，有独立价值的用 RESTRICT。

JOIN 的本质是从一张表取一行，去另一张表找匹配的行。理解了这个机制，LEFT JOIN 的 NULL 填充、行膨胀问题、自关联的递归查询，就都不是难理解的概念了。
