# 数据库渐进式学习计划

## 背景

你是一个做 Node.js 全栈的开发者，项目里天天要和数据库打交道。但你可能遇到过这些场景：查询慢了不知道怎么优化、表设计一改就要改一堆代码、并发下单时库存出现超卖、缓存策略怎么设计心里没底……

市面上的数据库教程要么是翻译官方文档，要么用学生成绩表这种玩具例子，读完之后遇到真实业务还是不会。

这个系列就是为你准备的。我们不把每种数据库的 API 翻译一遍，而是从"为什么需要它"出发，带你逐步构建一个**电商平台的数据库层**。每解决一个真实问题，你就自然掌握了一个数据库技能。

### 技术栈

| 技术             | 版本 | 为什么选它                                         |
| ---------------- | ---- | -------------------------------------------------- |
| PostgreSQL       | 16   | 功能最全的开源关系型数据库，JSON 支持好，扩展性强  |
| Prisma           | 5.x  | TypeScript 生态最友好的 ORM，类型安全、开发体验好  |
| MongoDB          | 7    | 最流行的文档数据库，适合日志、内容管理等灵活场景   |
| Redis            | 7.x  | 高性能缓存，电商场景必备（库存、排行榜、分布式锁） |
| Elasticsearch    | 8.x  | PostgreSQL 全文搜索不够用时的终极方案              |
| Docker + Compose | -    | 一键启动所有数据库服务，统一开发环境               |

> 本系列选择 PostgreSQL 而非 MySQL，是因为 PostgreSQL 的 JSONB、数组类型、CTE 等特性在现代 Web 开发中更实用。如果你团队用 MySQL，大部分 SQL 知识和设计原则是通用的。

## 基础篇：独立小案例（01-03）

目标：用 3 篇文章帮你建立对数据库的直觉，每篇一个独立场景。

---

### 01. 为什么需要数据库

**核心问题**：你的项目数据存在 JSON 文件里，用户量一多就出问题——并发写入冲突、查询慢、数据丢失。为什么不能继续用文件存数据？

**主线案例**：对比"用 JSON 文件存用户数据"和"用 PostgreSQL 存用户数据"，通过真实痛点引出数据库的核心价值。

**内容要点**：

- 从文件系统出发：为什么 JSON 文件存数据扛不住（并发、查询、一致性）
- 数据库的本质：不是"存数据的工具"，而是"管理数据的系统"
- 关系型 vs NoSQL：不是谁更好，是场景不同（电商订单 vs 用户行为日志）
- ACID 不是背概念：用一个转账场景说明为什么事务很重要
- CAP 定理：不要被吓到，用"双十一抢购"来理解一致性 vs 可用性
- 选型决策：你的项目该用什么数据库？一个简单的判断方法

**与前篇关联**：系列第一篇，无前篇。

---

### 02. SQL 是怎样一种语言

**核心问题**：你打开一个项目的数据库，看到一堆 SQL 语句，发现它既像英语又不像英语。增删改查、联合查询、分组排序……SQL 到底是怎么工作的？

**主线案例**：为一个小型电商系统编写 SQL——创建用户表、插入商品数据、查询订单、统计销售额。用电商场景覆盖 SQL 的核心用法。

**内容要点**：

- SQL 不是"编程语言"，而是"声明式查询语言"：你告诉数据库要什么，而不是怎么做
- SQL 的执行顺序和书写顺序不一样：为什么 WHERE 写在 SELECT 前面，但执行时 SELECT 先
- DDL/DML/DCL/TCL 各管什么：用"建房子"类比
- JOIN 的本质：不是"合并两个表"，而是"基于条件组合行"
- 子查询 vs CTE：什么时候该用哪个，用一个"找出消费最高的用户"的例子
- SQL 注入：为什么拼接字符串写 SQL 是危险的

**与前篇关联**：上一篇理解了"为什么需要数据库"，这一篇学习"怎么和数据库对话"。

---

### 03. 搭建 PostgreSQL 开发环境

**核心问题**：团队决定用 PostgreSQL，但安装配置一通折腾——版本不对、端口冲突、pg_hba.conf 连不上。怎么快速搭一个靠谱的开发环境？

**主线案例**：用 Docker Compose 一键启动 PostgreSQL + pgAdmin，创建电商数据库，导入初始数据。

**内容要点**：

- 为什么选 PostgreSQL 而不是 MySQL（JSONB、CTE、扩展性）
- Docker Compose 方案：为什么比直接装在本机好（环境一致、随时重建）
- psql 客户端：最常用的几个命令（\l、\dt、\d、\conninfo）
- 角色和权限：超级用户 vs 普通用户，为什么开发时别用超级用户
- 连接池：为什么不能每个请求都创建一个新连接
- 常用配置：shared_buffers、work_mem 这些参数到底什么意思

**与前篇关联**：上一篇学了 SQL 语法，但还没地方执行。这一篇搭好环境，后面就可以开始建表了。

## 核心篇：电商案例起步（04-09）

目标：引入电商平台作为贯穿案例，从建表到优化，建立完整的数据建模能力。

### 04. 为电商系统设计表结构

**核心问题**：你要给电商平台设计数据库。用户名用 VARCHAR(50) 还是 VARCHAR(255)？价格用 FLOAT 还是 DECIMAL？主键用自增 ID 还是 UUID？这些选择会影响后续所有的开发。

**主线案例**：为电商平台设计 users、products、orders 三张核心表，逐一讨论字段选型背后的考量。

**内容要点**：

- 命名规范：蛇形命名法、单数表名，为什么不用驼峰
- 主键设计：UUID vs 自增 ID，各适合什么场景（分布式 vs 单体）
- 数据类型选择不是"随便选"：价格为什么用 NUMERIC(10,2) 而不是 FLOAT
- JSONB 是 PostgreSQL 的杀手级特性：用户偏好设置、商品扩展属性
- 约束不是"锦上添花"：CHECK 约束防止脏数据、EXCLUDE 约束防重复
- TOAST 机制：为什么 PostgreSQL 没有 TEXT 长度限制

**与前篇关联**：上一篇搭好了 PostgreSQL 环境，这一篇开始在里面建第一张表。

---

### 05. 表之间的关系：外键与 JOIN

**核心问题**：订单属于某个用户，订单包含多个商品，商品属于某个分类。这些关系怎么在数据库里表达？外键加错了会导致什么问题？

**主线案例**：为电商系统设计完整的关系——用户-订单（一对多）、订单-商品（多对多）、用户-个人资料（一对一）、评论-评论（自关联）。

**内容要点**：

- 一对一、一对多、多对多：用电商场景解释三种关系的区别
- 外键的 ON DELETE 策略：CASCADE（删用户连带删订单）vs RESTRICT（有订单不能删用户）
- 为什么外键列要加索引：不加索引的 JOIN 是全表扫描
- JOIN 的四种类型：INNER JOIN、LEFT JOIN、RIGHT JOIN、FULL JOIN 各适合什么场景
- JOIN 的性能：为什么小表驱动大表很重要
- 自关联：商品分类的树形结构、评论的嵌套回复

**与前篇关联**：上一篇设计了独立的表结构，这一篇用外键把它们串联起来。

### 06. 范式与反范式：设计上的取舍

**核心问题**：你的订单表里直接存了用户邮箱和用户名，改个用户名要更新几百条订单。但每次查订单都要 JOIN 用户表，性能又跟不上。规范化到底要做到什么程度？

**主线案例**：对比"完全规范化的电商数据库"和"适当反范式的电商数据库"，分析各自在查询性能和写入一致性上的 trade-off。

**内容要点**：

- 1NF/2NF/3NF 不是背定义，而是解决具体问题：用订单表演进来说明
- 为什么大多数系统做到 3NF 就够了：BCNF 在实际中很少遇到
- 反范式不是"偷懒"：冗余字段、预计算字段、物化视图都是合理的优化手段
- 物化视图 vs 普通视图：为什么物化视图有"快"和"旧"两个属性
- 触发器维护冗余字段：自动同步 vs 应用层同步的取舍
- 什么时候该反范式：读多写少的统计表、报表查询

**与前篇关联**：上一篇建立了表关系，这一篇讨论关系的"粒度"——该拆多细，什么时候该合。

### 07. 索引：让查询快 100 倍的秘密

**核心问题**：你的订单查询从 10ms 涨到了 3 秒，加了 WHERE 条件反而更慢。DBA 说"加个索引就好了"——但索引到底在干什么？为什么不是所有列都加索引？

**主线案例**：为电商系统设计索引策略——用户邮箱查询、订单状态筛选、商品搜索、JSONB 字段查询。

**内容要点**：

- 索引的本质：B-Tree 不是二叉树，是一个多层有序结构
- 为什么索引能让查询快：全表扫描 vs 索引扫描，用"查字典"类比
- 复合索引的最左前缀原则：为什么 idx(user_id, status) 查 status 用不上
- 部分索引：只索引"活跃用户"的邮箱，减少索引体积
- 表达式索引：为什么 LOWER(email) 查询需要建表达式索引
- GIN 索引：JSONB 和数组字段的专用索引
- 索引的代价：每次 INSERT/UPDATE/DELETE 都要维护索引

**与前篇关联**：上一篇完成了表设计，这一篇让表查询变快。

### 08. SQL 查询优化实战

**核心问题**：你写了 WHERE 条件、加了索引，查询还是慢。EXPLAIN 的结果看不懂，不知道问题出在哪。有哪些常见的 SQL 优化套路？

**主线案例**：优化电商系统的几个典型慢查询——订单列表深分页、用户消费统计、商品搜索过滤。

**内容要点**：

- EXPLAIN ANALYZE：不是看 cost，是看 actual time 和 rows
- 深分页问题：OFFSET 10000 为什么慢，keyset pagination 怎么解决
- N+1 查询：先查 100 个订单，再逐个查用户信息——怎么一次搞定
- EXISTS vs IN：不是 EXISTS 一定快，要看子查询的数据量
- UNION ALL vs UNION：不需要去重时别用 UNION
- 批量操作：单条 INSERT vs 批量 INSERT vs COPY，性能差 10 倍
- CTE 的性能陷阱：PostgreSQL 12 之前的 CTE 是"优化围栏"

**与前篇关联**：上一篇理解了索引原理，这一篇把索引用到实际的慢查询优化中。

---

### 09. 执行计划：慢查询诊断终极武器

**核心问题**：你加了索引但查询还是走全表扫描，EXPLAIN 的结果里一堆 Seq Scan、Hash Join、Nested Loop 看不明白。执行计划到底在告诉你什么？

**主线案例**：解读电商系统的真实执行计划——为什么这个 JOIN 用了 Hash Join 而不是 Nested Loop，为什么统计信息过时会导致优化器选错方案。

**内容要点**：

- 执行计划的三种扫描：Seq Scan（全表）、Index Scan（索引）、Index Only Scan（只读索引）
- 三种 JOIN 策略：Nested Loop（小表）、Hash Join（大表）、Merge Join（已排序）
- 为什么优化器会"选错"：统计信息过时、参数类型不匹配
- ANALYZE 的重要性：为什么定期更新统计信息能让查询变快
- pg_stat_statements：找出最慢的 10 条 SQL
- auto_explain：自动记录慢查询的执行计划

**与前篇关联**：上一篇学了优化技巧，这一篇教你验证优化效果——看懂执行计划。

## ORM 篇：用 Prisma 串联（10-12）

### 10. 用 Prisma 代替手写 SQL

**核心问题**：你写 SQL 时经常拼错字段名、忘记处理 null、类型转换出错。有没有一种方式既保留 SQL 的灵活性，又给 TypeScript 的类型安全？

**主线案例**：用 Prisma Schema 定义电商系统的数据模型，实现用户注册、创建订单、查询商品等基本操作。

**内容要点**：

- Prisma vs TypeORM vs Drizzle：选型的关键考量
- Schema 定义：把 SQL 的 CREATE TABLE 翻译成 Prisma Schema
- Prisma Migrate：用代码管理数据库变更，告别手动改表
- CRUD 操作：create、findUnique、findMany、update、delete
- 类型安全：为什么 Prisma 能自动推断返回类型
- Prisma Studio：可视化的数据管理界面

**与前篇关联**：前 9 篇都在写原生 SQL，这一篇开始用 ORM 提升开发效率。电商案例的表结构直接翻译成 Prisma Schema。

---

### 11. Prisma 高级查询与性能优化

**核心问题**：基础 CRUD 够用了，但业务越来越复杂——按多条件筛选订单、统计用户消费、分页查询商品列表。Prisma 的高级查询怎么写？性能要注意什么？

**主线案例**：实现电商系统的几个复杂查询——订单管理后台的多条件筛选、用户消费统计面板、商品列表的游标分页。

**内容要点**：

- include vs select：include 带关联数据，select 只查需要的字段
- N+1 查询的 Prisma 版本：include 一次搞定 vs 循环查询
- 聚合查询：aggregate 和 groupBy 的用法和局限
- 游标分页 vs OFFSET 分页：为什么大数据量要用游标
- 原生 SQL：$queryRaw 和 $executeRaw，什么时候必须降级
- JSON 字段查询：path 过滤、equals 匹配
- 原子操作：increment、decrement，为什么库存扣减不能先读后写

**与前篇关联**：上一篇学会了 Prisma 基础用法，这一篇处理真实业务中的复杂查询。

---

### 12. 事务与并发：防止库存超卖

**核心问题**：双十一秒杀，100 个用户同时抢 10 件商品。怎么保证不会多卖？事务的隔离级别该怎么选？乐观锁和悲观锁哪个适合你的场景？

**主线案例**：实现电商系统的高频下单场景——创建订单 + 扣减库存 + 记录流水，全部在一个事务中完成。

**内容要点**：

- 事务的 ACID：用"银行转账"来理解原子性和一致性
- 四种隔离级别：Read Committed（PostgreSQL 默认）够用吗？什么时候要 Serializable
- 脏读/不可重复读/幻读：用并发下单场景演示三种问题
- 悲观锁：SELECT FOR UPDATE，适合高冲突场景（秒杀）
- 乐观锁：版本号机制，适合低冲突场景（编辑文档）
- Prisma 事务：$transaction 的两种用法和隔离级别设置
- 重试机制：并发冲突不是错误，是正常现象——怎么优雅地重试

**与前篇关联**：上一篇的查询都是"读"操作，这一篇处理"写"操作中最棘手的并发问题。

### 13. MongoDB：当关系型数据库不够用时

**核心问题**：电商系统要记录用户操作日志，日志结构经常变——今天加了个字段，明天又改了格式。每次改都要 ALTER TABLE，太麻烦了。有没有更灵活的方案？

**主线案例**：用 MongoDB 存储电商系统的用户行为日志——浏览记录、搜索历史、点击流数据。

**内容要点**：

- 文档模型 vs 关系模型：什么时候该用 MongoDB（日志、内容、配置）
- Schema 灵活性：MongoDB 不是"没有结构"，是"动态结构"
- 嵌入 vs 引用：订单明细嵌入订单文档 vs 引用商品文档，怎么选
- CRUD 操作：insertOne、findOne、updateOne、deleteOne
- 索引：MongoDB 也要建索引，不建索引的查询也是全表扫描
- 什么时候不该用 MongoDB：需要事务、需要 JOIN、数据结构稳定

**与前篇关联**：前 12 篇都用 PostgreSQL，这一篇引入 MongoDB 处理"灵活结构"的数据。

---

### 14. MongoDB 聚合管道：数据统计分析

**核心问题**：运营想要一份报表——每日销售额、用户消费排行、商品销售 Top 10。这些统计查询在 SQL 里用 GROUP BY，MongoDB 里怎么实现？

**主线案例**：为电商系统实现数据分析功能——用户消费统计、商品销售排行、日/周/月营收趋势。

**内容要点**：

- 聚合管道 vs SQL GROUP BY：概念对比，用 SQL 思维理解 MongoDB
- $match → $group → $sort → $project：管道的基本套路
- $lookup：MongoDB 的 JOIN，为什么能不用就不用
- $unwind：展开数组，为什么聚合前经常要先展开
- $facet：一次查询返回多种维度的统计结果
- 性能优化：$match 放最前面、合理使用索引

**与前篇关联**：上一篇学会了 MongoDB 基础操作，这一篇处理复杂的数据分析。

### 15. Redis 缓存策略：让接口快起来

**核心问题**：电商首页的商品列表每次都查数据库，QPS 一高数据库就扛不住。加了缓存后，库存更新了但页面还是旧数据。缓存到底该怎么设计？

**主线案例**：为电商系统设计缓存层——商品详情缓存、热门商品列表缓存、库存缓存、分布式锁防超卖。

**内容要点**：

- 缓存的本质：用空间换时间，但不是所有数据都该缓存
- 五种数据结构的实际用法：String（缓存对象）、Hash（用户属性）、List（最新消息）、Set（标签）、Sorted Set（排行榜）
- Cache Aside 模式：最常用的缓存策略，为什么不是 Read Through
- 三大缓存问题：穿透（查询不存在的数据）、击穿（热点 key 过期）、雪崩（大量 key 同时过期）
- 分布式锁：Redis SET NX EX + Lua 脚本释放锁
- 什么时候不该用缓存：写频繁的数据（库存）、强一致性数据（余额）

**与前篇关联**：上一篇用 MongoDB 做了日志存储，这一篇用 Redis 做缓存加速查询。

### 16. Elasticsearch：商品搜索的正确姿势

**核心问题**：用 PostgreSQL 的 LIKE 搜索商品，"降噪耳机"搜不到"主动降噪无线耳机"，中文分词一塌糊涂。怎么做一个像淘宝一样的搜索功能？

**主线案例**：为电商系统实现商品全文搜索——支持中文分词、模糊匹配、搜索高亮、多维度筛选。

**内容要点**：

- 为什么 PostgreSQL 的全文搜索不够用：分词精度、相关性排序、性能
- 倒排索引：ES 快的核心，一句话解释清楚
- Mapping 设计：text vs keyword，为什么搜索字段用 text、筛选字段用 keyword
- 查询 DSL：match、term、bool、multi_match 各适合什么场景
- 与 PostgreSQL 的数据同步：应用层双写 vs CDC（Debezium）
- 什么时候该上 ES：数据量 > 100万、需要中文分词、需要相关性排序

**与前篇关联**：上一篇用 Redis 缓存加速了查询，这一篇解决"搜索"这个特殊的查询场景。

### 17. 备份与恢复：数据安全的最后一道防线

**核心问题**：生产环境的 PostgreSQL 挂了，数据全丢了。你发现上次备份是三个月前做的，而且从来没验证过能不能恢复。备份策略到底该怎么设计？

**主线案例**：为电商系统设计完整的备份方案——PostgreSQL 定时备份、MongoDB 日志备份、Redis 持久化配置，以及一次真实的恢复演练。

**内容要点**：

- 备份的 3-2-1 原则：3 份副本、2 种介质、1 份异地
- pg_dump vs pg_basebackup：逻辑备份 vs 物理备份，什么时候用哪个
- MongoDB mongodump：为什么 MongoDB 备份要特别注意 Oplog
- Redis RDB vs AOF：快照 vs 追加日志，两者可以同时开
- 恢复演练：没有验证过的备份等于没有备份
- 自动化：crontab + 备份脚本 + 监控告警

**与前篇关联**：前面 16 篇构建了完整的数据库层，这一篇确保数据不会丢。

### 18. 监控与调优：生产环境的数据库运维

**核心问题**：电商系统上线了，但数据库的 QPS、慢查询、缓存命中率你一无所知。出了问题只能重启。怎么建立一套可观测的数据库监控体系？

**主线案例**：为电商系统建立数据库监控——PostgreSQL 慢查询追踪、连接池监控、缓存命中率监控，以及常见的性能调优方案。

**内容要点**：

- 关键监控指标：连接数、QPS、慢查询数量、缓存命中率、复制延迟
- pg_stat_statements：找出最慢的 SQL，按执行时间排序
- PostgreSQL 配置调优：shared_buffers、work_mem、random_page_cost
- 连接池：PgBouncer 的 transaction 模式，为什么比 session 模式省连接
- 分库分表：什么时候需要分，什么时候读写分离就够了
- 高可用架构：主从复制、哨兵模式、集群模式，各适合什么规模

**与前篇关联**：上一篇确保了数据安全，这一篇确保数据库跑得稳、跑得快。
