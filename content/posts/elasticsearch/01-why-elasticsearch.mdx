你做了一个电商网站，商品表里有 50 万条数据。用户在搜索框输入"机械键盘"，你写下这条 SQL：

```sql
SELECT * FROM products WHERE name LIKE '%机械键盘%';
```

结果只返回了 3 条——商品名包含"机械键盘"连续四个字的只有 3 条。但数据库里明明还有"机械 游戏键盘 红轴 87键"、"Cherry MX 机械式键盘 青轴"这些商品，它们没被搜出来。

你试了 `LIKE '%机械%' AND LIKE '%键盘%'`，能多搜到一些，但还是漏了"游戏键盘"这种表述。更别说用户输入"手机"搜不到"智能手机"，输入"笔记本"搜不到"笔记本电脑 ThinkPad"。

这就是传统数据库做搜索的天花板。不是你 SQL 写得不好，是数据库的索引结构从根本上就不适合做全文搜索。

## 数据库为什么搜不好

MySQL 和 PostgreSQL 默认用 B+ 树做索引。B+ 树擅长的是**精确匹配和范围查询**——查 id=123、查价格在 100-500 之间的商品，这些操作 B+ 树非常快。但全文搜索的逻辑完全不同：你要找的是"包含某个词的所有文档"，而不是"等于某个值的那条记录"。

用 `LIKE '%关键词%'` 做搜索有两个致命问题：

**第一，没法用索引。** `%关键词%` 前面有通配符，B+ 树没法利用索引的有序性，只能全表扫描。50 万条数据全扫一遍，接口 P99 直接飙到几秒。

**第二，搜不"懂"你的意思。** 用户搜"机械键盘"，你想搜出所有跟机械键盘相关的商品，不管商品名里有没有完整的"机械键盘"四个字。但数据库不懂"机械键盘"可以拆成"机械"和"键盘"两个词，也不懂"游戏键盘"和"机械键盘"有关系。

PostgreSQL 有全文索引（`tsvector` + `tsquery`），MySQL 也有 `FULLTEXT INDEX`，能解决一部分问题。但它们在中文分词、相关度排序、多字段加权搜索、聚合筛选这些能力上，和专业的搜索引擎比差距很大。

## 倒排索引：搜索引擎的核心

Elasticsearch 能搜得好，核心原因是它用了一种完全不同的索引结构——**倒排索引（Inverted Index）**。

先看正排索引（就是数据库的方式）：

```
文档 ID → 内容
1 → "Cherry MX 机械式键盘 青轴"
2 → "罗技 G502 游戏鼠标"
3 → "达尔优 机械游戏键盘 红轴 87键"
```

正排索引是从文档找词，你得把每个文档的内容扫一遍才知道包不包含某个词。

倒排索引反过来，**从词找文档**：

```
词 → 文档 ID 列表
机械    → [1, 3]
键盘    → [1, 3]
游戏    → [2, 3]
鼠标    → [2]
罗技    → [2]
Cherry  → [1]
红轴    → [3]
青轴    → [1]
```

用户搜"机械键盘"，分词成"机械"和"键盘"两个词。查倒排索引：`机械` 在文档 [1, 3]，`键盘` 在文档 [1, 3]。取交集，得到 [1, 3]——这两条商品都命中了。

注意，这个过程**不需要扫描任何文档的内容**，只需要查两次索引然后取交集。50 万条数据也好，500 万条也好，查询速度都在毫秒级。

这就是 Elasticsearch 搜索快的根本原因。当然实际的倒排索引比这个复杂得多——还要处理词频（TF）、文档频率（IDF）、位置信息（用于短语搜索）等等，这些共同决定了搜索结果的**相关度排序**。

### 相关度排序：为什么第一条是你最想要的

数据库 `ORDER BY` 只能按某个字段的值排序，做不到"按相关性排序"。什么叫相关性？用户搜"机械键盘"，商品 A 叫"机械键盘 红轴"，商品 B 叫"机械游戏键盘 87键"，商品 A 应该排在前面，因为它的标题更贴近搜索词。

Elasticsearch 用 **BM25 算法** 计算相关度得分。简单说就是考虑三个因素：

- **词频（TF）**：搜索词在文档里出现次数越多，得分越高
- **文档频率（IDF）**：搜索词在所有文档里越罕见，得分越高（"机械"比"键盘"更罕见，所以"机械"的匹配权重更高）
- **字段长度**：标题越短匹配越精准（"机械键盘"出现在 5 个字的标题里，比出现在 50 个字的描述里权重更高）

这样搜出来的结果天然就是按"最相关"排序的，不需要你写任何排序逻辑。

## Elasticsearch 是什么

了解了核心原理，来看看 Elasticsearch 的整体架构。它用几个关键概念组织数据：

- **文档（Document）**：最小数据单元，类似数据库的一行记录，JSON 格式。一条商品信息就是一个文档。
- **索引（Index）**：文档的集合，类似数据库的表。一个"products"索引存放所有商品文档。
- **字段（Field）**：文档里的一个键值对，类似数据库的列。商品有 title、price、brand 等字段。
- **节点（Node）**：运行中的 ES 实例，一台机器可以跑一个或多个节点。
- **集群（Cluster）**：多个节点组成，对外提供统一的搜索服务。
- **分片（Shard）**：索引被切分成多个分片，分布在不同节点上，实现水平扩展。

这些概念不需要一次记全。现在你只需要知道：**文档存在索引里，索引存在节点上，节点组成集群**。后面的文章会逐个展开。

## 跑通第一个搜索

原理讲完了，动手试试。我们从 Docker 启动一个 ES 实例，索引一条商品文档，然后搜索它。

### 用 Docker 启动 Elasticsearch

```bash
docker network create es-net

docker run -d \
  --name elasticsearch \
  --net es-net \
  -p 9200:9200 \
  -e "discovery.type=single-node" \
  -e "xpack.security.enabled=false" \
  -e "ES_JAVA_OPTS=-Xms512m -Xmx512m" \
  docker.elastic.co/elasticsearch/elasticsearch:8.12.0
```

几个关键参数：

- `discovery.type=single-node`：单节点模式，开发环境够用
- `xpack.security.enabled=false`：关闭安全认证，方便调试（生产环境千万别这么干）
- `ES_JAVA_OPTS`：JVM 堆内存，512m 对本地开发足够

等个 30 秒让它启动完，验证一下：

```bash
curl http://localhost:9200

# 返回类似：
# {
#   "name" : "elasticsearch",
#   "cluster_name" : "docker-cluster",
#   "version" : { "number" : "8.12.0" ... },
#   "tagline" : "You Know, for Search"
# }
```

看到 "You Know, for Search" 就说明启动成功了。

### 索引一条商品文档

Elasticsearch 的 API 是 RESTful 风格，用 HTTP 请求操作数据。索引一条商品就是往 `/{索引名}/_doc/{文档id}` 发一个 PUT 或 POST 请求：

```bash
curl -X PUT "http://localhost:9200/products/_doc/1" -H 'Content-Type: application/json' -d '
{
  "name": "Cherry MX 机械式键盘 青轴",
  "brand": "Cherry",
  "category": "外设",
  "price": 899,
  "description": "德国原厂机械键盘，MX青轴，段落感强烈，适合长时间打字",
  "stock": 156,
  "tags": ["机械键盘", "青轴", "办公"],
  "createdAt": "2025-12-01T10:30:00Z"
}'
```

返回结果：

```json
{
  "_index": "products",
  "_id": "1",
  "result": "created",
  "_version": 1
}
```

这里有个细节：我们没提前创建 `products` 这个索引，也没定义任何字段类型。ES 自动帮我们做了——这就是 **dynamic mapping**，它根据第一次插入的 JSON 推断字段类型：字符串默认当成 text + keyword 双类型，数字当成 long 或 float，日期尝试自动解析。

开发阶段这很方便，但生产环境不建议依赖 dynamic mapping——推断的类型可能不是你想要的。下一篇我们会专门讲怎么手动设计 Mapping。

### 搜索：见证差异

现在来搜索"机械键盘"：

```bash
curl -X GET "http://localhost:9200/products/_search" -H 'Content-Type: application/json' -d '
{
  "query": {
    "match": {
      "name": "机械键盘"
    }
  }
}'
```

注意看返回结果——**文档 1 和文档 3 都被搜到了**，而它们的商品名里没有完整的"机械键盘"四个连续字。ES 把"机械键盘"拆成了"机械"和"键盘"两个词，分别匹配，找到包含这两个词的文档。

更关键的是，结果按相关度排序了。`_score` 字段就是 BM25 算法算出的得分：

```json
{
  "hits": {
    "total": { "value": 2 },
    "hits": [
      {
        "_id": "3",
        "_score": 0.754,
        "_source": { "name": "达尔优 机械游戏键盘 红轴 87键" }
      },
      {
        "_id": "1",
        "_score": 0.538,
        "_source": { "name": "Cherry MX 机械式键盘 青轴" }
      }
    ]
  }
}
```

文档 3（"机械游戏键盘"）排在文档 1（"机械式键盘"）前面。因为"机械"和"键盘"在文档 3 的标题里靠得更近，BM25 认为它更相关。

这就是数据库 `LIKE` 做不到的事。

再试一个搜索，搜"游戏"：

```bash
curl -X GET "http://localhost:9200/products/_search" -H 'Content-Type: application/json' -d '
{
  "query": {
    "match": {
      "name": "游戏"
    }
  }
}'
```

文档 2（"游戏鼠标"）和文档 3（"游戏键盘"）都被搜到了。如果这是数据库的 `LIKE '%游戏%'`，结果一样。但如果用户搜"游戏 键盘"呢？

```bash
curl -X GET "http://localhost:9200/products/_search" -H 'Content-Type: application/json' -d '
{
  "query": {
    "match": {
      "name": "游戏 键盘"
    }
  }
}'
```

三条文档都出来了。因为 `match` 默认是 OR 逻辑——包含"游戏"或"键盘"任意一个就算匹配。文档 3 同时包含两个词，得分最高排在第一。

这就是 ES 的查询 DSL。不是 SQL 那种声明式的语法，而是用 JSON 描述查询意图。看起来啰嗦，但好处是可以精确控制搜索行为——OR 还是 AND、权重怎么分配、要不要模糊匹配，全在这个 JSON 里控制。

## 什么时候该用 Elasticsearch

ES 不是所有场景都需要。很多项目的搜索需求简单到数据库就能搞定。

**该用 ES 的场景：**

- 你的搜索需要中文分词（搜"手机"找到"智能手机"），数据库的 `LIKE` 和全文索引做不好这件事
- 搜索结果需要按相关度排序，而不是按时间或价格
- 搜索条件复杂——全文搜索 + 价格区间 + 品牌筛选 + 分类聚合，这些组合查询在数据库里要多表 JOIN，性能很差
- 数据量大（百万级以上），数据库全文搜索开始扛不住

**不需要 ES 的场景：**

- 数据量几万条以内，搜索场景简单，PostgreSQL 的 `tsvector` + `tsquery` 足够了
- 你只需要精确匹配和简单的排序，不需要相关度评分
- 团队没有 ES 运维经验，引入 ES 意味着多维护一套基础设施
- 项目是内部工具或管理后台，搜索不是核心功能

**和轻量级搜索引擎比呢？** Meilisearch 和 Typesense 是近几年出现的轻量搜索方案，开箱即用，配置简单，对中小项目更友好。它们的问题是不支持 ES 那么复杂的聚合、自定义分词器、集群扩展。如果你的需求是"快速给项目加个搜索框"，Meilisearch 可能比 ES 更合适。但如果你的需求是电商搜索、日志分析、大规模数据处理，ES 仍然是第一选择。

选择的核心逻辑：**用能解决问题的最简单方案**。数据库搜索够用就别上 ES，ES 够用就别上更重的方案。
