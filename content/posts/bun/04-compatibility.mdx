前两篇我们看到了 Bun 的性能优势和工具链体验。如果你和我一样，看完之后的第一反应是："那我能不能把现有项目直接跑在 Bun 上？"

这篇文章就来做这件事。我拿一个真实的 Express + MySQL 项目，试着用 Bun 跑起来，看看哪些地方顺畅，哪些地方踩坑。

## Bun 的兼容策略

在开始之前，先理解 Bun 的 Node.js 兼容层是怎么工作的。

Bun 没有 fork Node.js 的代码，也没有重新实现 Node.js 的全部 API。它的策略是**渐进式兼容**：

1. **标准库重写**：用 Zig 重新实现了 Node.js 的核心模块（`fs`、`path`、`http` 等），保持 API 签名一致但底层实现不同
2. **模块解析兼容**：支持 `node_modules` 查找逻辑、`package.json` 的 `exports` 字段、条件导出
3. **运行时 polyfill**：对一些 Node.js 特有的全局对象（`process`、`Buffer`、`__dirname` 等）提供兼容实现

这意味着大部分"正常的"Node.js 代码可以直接跑。但"不正常"的代码——依赖了 Node.js 特定实现细节、用了冷门 API、或者依赖原生 C++ 模块的代码——可能会出问题。

## 兼容性现状：逐模块扫描

### ✅ 完善的模块

这些模块的 API 兼容性已经很好，大部分项目可以直接用：

**`fs` 文件系统**

```typescript
import { readFileSync, writeFileSync, mkdirSync, readdirSync } from "fs";
import { join } from "path";

// 这些操作在 Bun 中都能正常工作
const config = readFileSync(join(process.cwd(), "config.json"), "utf-8");
writeFileSync("./output.txt", "Hello from Bun");
mkdirSync("./data", { recursive: true });
const files = readdirSync("./src");
```

Bun 的 `fs` 模块是用 Zig 原生实现的，性能和 Node.js 相当甚至更快。同步和异步 API 都支持，包括 `fs.promises`。

**`path` 路径处理**

完全兼容。`join`、`resolve`、`dirname`、`basename`、`extname` 等常用函数行为一致。

**`crypto` 加密**

```typescript
import { createHash, randomBytes, createHmac } from "crypto";

const hash = createHash("sha256").update("hello").digest("hex");
const token = randomBytes(32).toString("hex");
const hmac = createHmac("sha256", "secret").update("data").digest("base64");
```

常用的加密算法都支持。不过一些冷门的算法（比如 `scrypt` 的某些参数组合）可能行为有差异。

**`stream`、`buffer`、`events`**

这三个基础模块兼容性很好。`Buffer.from()`、`EventEmitter`、`Readable`/`Writable` 流都可以正常使用。

**`url`、`querystring`**

URL 解析和查询字符串处理完全兼容。

### ⚠️ 部分支持的模块

这些模块的**核心 API** 可以工作，但一些边缘场景或高级功能可能有差异：

**`http` / `https`**

基础 HTTP 服务器和客户端功能没问题。但一些高级场景需要注意：

```typescript
import http from "http";

// ✅ 能工作
const server = http.createServer((req, res) => {
  res.writeHead(200);
  res.end("ok");
});
server.listen(3000);

// ⚠️ 部分场景有差异
// - HTTP Agent 的连接池行为可能不完全一致
// - Server-Sent Events 的某些边缘情况处理不同
// - HTTP/2 支持有限
```

如果你用的是 Express、Koa 这类基于 `http` 模块的框架，基本都能跑。但如果你直接用了 `http` 模块的一些底层特性（比如 raw socket 操作、自定义 Agent），可能需要调试。

**`child_process`**

```typescript
import { execSync, spawn } from "child_process";

// ✅ 能工作
execSync("echo hello");

// ⚠️ 一些差异
// - spawn 的某些 stdio 配置可能行为不同
// - 跨平台的 shell 行为可能有微妙差异
// - 大量子进程并发场景下性能特征不同
```

**`net` / `tls`**

TCP 服务器和 TLS 连接基本能用，但一些高级特性（比如自定义 TLS 证书选项、特定的 socket 事件）可能有差异。

**`dns`**

基础 DNS 查询工作正常，但 `dns.promises` 的某些方法和自定义 resolver 支持有限。

### ❌ 尚未完善的模块

这些模块要么还没实现，要么实现程度不够用在生产环境：

**`cluster`**

Bun 目前不支持 Node.js 的 `cluster` 模块。如果你的项目用 `cluster` 做多进程，在 Bun 里需要改用其他方案（比如用操作系统的进程管理工具，或者用 `child_process` 手动 fork）。

**`dgram`（UDP）**

UDP 相关的 API 支持有限。如果你的项目涉及 UDP 通信（比如某些实时游戏、DNS 服务），暂时不能完全依赖 Bun。

**`worker_threads` 的部分特性**

基础的 Worker 功能支持，但 `SharedArrayBuffer` 的一些高级用法、`MessageChannel` 的某些场景可能有差异。

**`vm` 模块**

`vm.runInContext`、`vm.Script` 等沙箱执行 API 支持有限。如果你的项目用 `vm` 模块做代码沙箱（比如在线代码编辑器、Serverless 平台），需要仔细测试。

## 实战：迁移一个 Express 项目

说了这么多模块级别的兼容性，来看看实际迁移是什么体验。

### 项目概况

一个典型的 Express API 服务：

- Express 4.x
- mysql2 做数据库连接
- jsonwebtoken 做 JWT 认证
- bcrypt 做密码哈希
- morgan 做日志
- cors 做跨域
- dotenv 加载环境变量

### 第一步：直接跑

```bash
bun run src/index.ts
```

结果：

```
✅ Express 启动成功
✅ 路由注册正常
✅ JSON 解析中间件工作
✅ CORS 中间件工作
✅ morgan 日志输出正常
✅ dotenv 不需要了，Bun 自动加载 .env
```

出乎意料的顺利。大部分 Express 中间件都能直接跑，因为它们本质上就是操作 `req`/`res` 对象的函数，不依赖 Node.js 的底层实现细节。

### 第二步：数据库连接

```typescript
import mysql from "mysql2/promise";

const pool = mysql.createPool({
  host: process.env.DB_HOST,
  user: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  database: process.env.DB_NAME,
});
```

```
✅ 连接池创建成功
✅ 查询执行正常
✅ 事务处理正常
⚠️ 连接池的某些超时配置行为略有不同
```

`mysql2` 使用的是纯 JavaScript 实现（不需要原生 C++ 绑定），所以在 Bun 上运行没问题。但如果你用的是 `pg`（PostgreSQL）的某些原生版本（`pg-native`），可能会有兼容性问题。

### 第三步：JWT 和密码哈希

```typescript
import jwt from "jsonwebtoken";
import bcrypt from "bcrypt";
```

```
✅ jsonwebtoken 完全兼容
⚠️ bcrypt 有坑
```

`bcrypt` 是个常见的坑点。它依赖 `node-gyp` 编译的原生 C++ 模块。在 Bun 上，一些原生模块可以工作，但 `bcrypt` 在某些平台上可能编译失败。

解决方案是换用纯 JavaScript 实现：

```typescript
// 替换 bcrypt 为 bcryptjs（纯 JS 实现，无原生依赖）
import bcrypt from "bcryptjs"; // 注意是 bcryptjs，不是 bcrypt

const hash = await bcrypt.hash("password123", 10);
const match = await bcrypt.compare("password123", hash);
```

`bcryptjs` 的 API 和 `bcrypt` 完全一样，只是纯 JavaScript 实现，性能稍慢一点（哈希操作慢约 30%），但在大多数场景下可以接受。

### 第四步：启动性能对比

迁移完成后，同一个项目用两个运行时启动：

```
Node.js 冷启动：    1.2s
Bun 冷启动：        0.3s

Node.js 热启动：    0.8s
Bun 热启动：        0.15s
```

Bun 快了大约 4-5 倍。对于 API 服务来说，冷启动速度影响的是自动扩容和 Serverless 场景的响应时间。

## 原生模块（napi）兼容性

这是迁移中最需要注意的部分。

Node.js 的原生模块有两种：

1. **node-gyp（老的绑定方式）**：用 C++ 写的绑定，编译成 `.node` 文件
2. **Node-API / napi（新的绑定方式）**：ABI 稳定的 C 接口

Bun 支持 Node-API（napi），这意味着用 napi 编写的原生模块理论上可以在 Bun 上运行。但 node-gyp 编译的老模块支持有限。

常见的原生模块兼容情况：

| 模块                | 类型     | Bun 兼容性               |
| ------------------- | -------- | ------------------------ |
| `sharp`（图像处理） | napi     | ✅ 支持                  |
| `bcrypt`            | node-gyp | ⚠️ 部分平台有问题        |
| `canvas`            | node-gyp | ⚠️ 部分平台有问题        |
| `better-sqlite3`    | napi     | ✅ 支持                  |
| `argon2`            | napi     | ✅ 支持                  |
| `sqlite3`           | node-gyp | ⚠️ 建议用 better-sqlite3 |
| `node-postgres`     | 纯 JS    | ✅ 支持                  |

经验法则：如果一个包有纯 JavaScript 替代品，优先用纯 JS 版本，避免原生模块的兼容性问题。

## 迁移检查清单

如果你打算迁移一个现有项目，按这个顺序检查：

**第一步：列出原生依赖**

```bash
# 找出项目中的原生模块
npm ls | grep -i "node-gyp\|\.node"
```

如果有原生模块，查一下它是否用了 napi。不是 napi 的，找纯 JS 替代品。

**第二步：跑测试**

```bash
# 先把测试跑起来
bun test
```

如果你用的是 Jest，需要把 `jest.mock` 改成 `mock.module`（上一篇讲过的差异）。测试通过是兼容性的最好证明。

**第三步：启动应用**

```bash
bun run src/index.ts
```

观察日志，看有没有运行时警告或错误。Bun 会在遇到不支持的 API 时打印警告信息。

**第四步：性能验证**

用 `wrk` 或 `ab` 对两个运行时做压力测试，确认迁移后性能没有退化。

**第五步：灰度上线**

如果一切正常，建议在灰度环境先跑一段时间再全面切换。Bun 的兼容性还在快速迭代，某些边缘场景可能在新版本中行为有变化。

## 一个务实的判断

Bun 的 Node.js 兼容性已经足够支撑**大部分常见的 API 项目**（Express/Koa + 数据库 + 中间件），但如果你有这些情况，建议暂缓迁移：

- 项目依赖大量原生 C++ 模块
- 用了 `cluster` 做多进程部署
- 有 UDP 通信需求
- 用了 `vm` 模块做代码沙箱
- 项目的测试覆盖率低，无法通过测试验证兼容性

下一篇我们换个角度，看看 Bun 不只是"兼容 Node.js"——它还有自己的原生 API，比如内置的 `fetch`、`WebSocket` 和 `Bun.serve()`。
