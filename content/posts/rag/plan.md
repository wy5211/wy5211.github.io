# RAG 深入：从原理到生产级检索增强生成

## 背景

你在 [LangChain 系列的 05、06 篇](/posts/langchain/05-rag-document-loading) 中已经用 LangChain 实现了一个基础 RAG 系统——文档加载、切分、向量化、检索、生成。但当你把它上线后，发现：

- 用户问"怎么配置 CI/CD"，检索到的却是"CICD 会议纪要"——关键词干扰
- 文档更新了，但检索结果还是旧的——向量没更新
- 回答质量时好时坏，不知道怎么衡量和优化

这个系列不教你用 LangChain（那是上一个系列的事），而是深入 RAG 技术本身的每个环节——Embedding 的原理、向量数据库的选型、Chunking 策略的选择、高级检索技巧、效果评估方法。理解了这些，不管用什么框架，你都能构建出高质量的 RAG 系统。

### 技术栈选择

| 技术              | 版本                                 | 为什么选它                     |
| ----------------- | ------------------------------------ | ------------------------------ |
| TypeScript        | 5.x                                  | 全栈统一语言，读者已有基础     |
| OpenAI API        | text-embedding-3-small / gpt-4o-mini | 默认 Embedding 和生成模型      |

> 本系列不绑定特定框架。代码示例用原生 API 和轻量库演示原理，你可以用 LangChain、LlamaIndex 或自己封装来落地。

### 01. RAG 架构原理：从 Naive RAG 到 Advanced RAG

**核心问题**：LLM 的训练数据有截止日期，而且它不知道你公司的内部文档。RAG 到底是怎么把外部知识"喂"给 LLM 的？为什么同一个 RAG 框架，有的系统回答很准，有的经常胡说？

**主线案例**：团队新人问"我们为什么用 PostgreSQL 不用 MongoDB？"——对比有 RAG 和没 RAG 时 LLM 的回答差异。

**内容要点**：

- RAG 解决的核心问题：知识时效性、私有数据、幻觉
- RAG vs Fine-tuning vs 长上下文：什么时候用哪个
- Naive RAG 的完整流程（5 步）和每步的潜在问题
- Advanced RAG 的改进：检索前、检索中、检索后的优化
- Modular RAG：可插拔的模块化设计
- RAG 的适用场景和局限

## 核心篇：深入三个关键环节（02-04）

---

### 02. Embedding 模型深入：文本如何变成向量

**核心问题**：你已经知道 Embedding 把文本变成向量，但为什么同一个模型，有时候检索很准，有时候完全不相关？embedding-3-small 和 embedding-3-large 差在哪？开源模型能替代 OpenAI 吗？

**主线案例**：对比不同 Embedding 模型在知识库文档上的检索效果——同样的查询，不同模型返回的结果差异有多大。

**内容要点**：

- Embedding 的直觉理解：语义相近 → 向量距离近
- 余弦相似度、欧氏距离、点积——三种距离度量怎么选
- OpenAI text-embedding-3-small vs 3-large vs ada-002 对比
- 开源 Embedding 模型：BGE、E5、GTE（什么时候值得自己部署）
- 多语言支持：中文文档用什么模型效果好
- Matryoshka Representation Learning：可变维度 Embedding
- 踩坑：长文本截断、领域词汇 OOV、多语言混合

**与前篇关联**：上一篇讲了 RAG 的整体架构，这一篇深入第一个环节——Embedding。它是整个检索链的基础。

---

### 03. 向量数据库实战：选型、索引与生产优化

**核心问题**：你已经能把文档变成向量了，但存到哪？Chroma 够用吗？Pinecone 太贵？Qdrant 和 Milvus 怎么选？10 万条文档时检索还有多快？

**主线案例**：为知识库选型向量数据库——从内存向量库到 Chroma 到 Qdrant，对比性能、成本、运维复杂度。

**内容要点**：

- 向量数据库 vs 传统数据库 + 向量插件的取舍
- 主流向量数据库对比：Chroma / Qdrant / Milvus / Pinecone / Weaviate
- HNSW 索引原理（为什么向量检索能做到毫秒级）
- 内存索引 vs 磁盘索引的 trade-off
- 元数据过滤 + 向量检索的组合查询
- 知识库的向量更新策略（全量重建 vs 增量更新）
- 生产环境：监控、备份、扩容

**与前篇关联**：上一篇讲了 Embedding 模型的选择，这一篇解决向量存到哪、怎么查的问题。

### 04. 文档处理与 Chunking 策略：切对了才搜得准

**核心问题**：你把文档按 500 字符切了，但用户问"支付服务宕机的根因是什么"，检索到的是"6月12日故障复盘"的前半段——根因在后半段。切分策略直接决定检索质量，但很多人低估了它的重要性。

**主线案例**：处理知识库中的四种文档（Markdown API 文档、PDF 架构设计、会议纪要、故障复盘），对比不同切分策略的检索效果。

**内容要点**：

- Chunking 为什么重要：切分粒度对检索质量的量化影响
- 固定长度切分 vs 递归字符切分 vs 语义切分
- 按文档结构切分（Markdown 标题、代码块、表格）
- 语义切分（Semantic Chunking）：用 Embedding 相似度找切分点
- 父子文档策略：小片段检索，返回大片段上下文
- 元数据管理：每段保留什么信息
- 实战：处理表格、代码块、嵌套列表等难点

**与前篇关联**：前两篇解决了 Embedding 和存储问题，这一篇回到数据源头——如果切得不好，再好的模型和数据库也救不回来。

## 进阶篇：从能用变好用（05）

---

### 05. 高级检索策略：混合检索、重排序与 Query 改写

**核心问题**：你的 RAG 系统能用了，但检索准确率只有 60%。用户问"CI/CD 配置"，结果返回了"CICD 年度会议总结"。纯向量检索有盲区——怎么补上？

**主线案例**：把知识库的检索准确率从 60% 提升到 85%——一步步加混合检索、Query 改写、重排序。

**内容要点**：

- 纯向量检索的盲区：精确匹配失败、关键词干扰、领域术语
- 混合检索（Hybrid Search）：向量 + BM25 关键词检索
  - 为什么需要两者结合
  - 分数归一化（向量分数和关键词分数不在同一尺度）
- Query 改写：让用户的问题更适合检索
  - Query Expansion：扩展为多个子查询
  - HyDE（Hypothetical Document Embedding）：让 LLM 先猜答案，用答案去搜
  - Step-back Prompting：退一步问更宽泛的问题
- 重排序（Reranking）：二次筛选提升精度
  - Cross-encoder vs Bi-encoder 的区别
  - Cohere Rerank API 的使用
  - 什么时候需要重排序（检索结果 > 20 条时效果明显）
- 实战：对比各策略的检索质量提升

**与前篇关联**：前三篇搭好了 RAG 基础设施，这一篇在上面叠加高级策略，从 Naive RAG 升级到 Advanced RAG。

## 评估篇：量化改进（06）

---

### 06. RAG 评估与优化：量化你的检索质量

**核心问题**：产品经理问"RAG 系统效果怎么样"，你说"感觉还行"。这不行。你需要数据：检索准确率多少、回答质量如何、哪些查询容易失败、改进后提升了多少。

**主线案例**：为知识库构建评估体系——构建测试集、跑评估、发现薄弱环节、针对性优化、验证提升。

**内容要点**：

- 为什么 RAG 需要专门的评估（不是普通搜索的评估，也不是 LLM 评估）
- 检索质量评估
  - 命中率（Hit Rate）、MRR、nDCG
  - 如何构建测试集（人工标注 vs LLM 生成）
- 生成质量评估（RAGAS 框架）
  - Faithfulness（忠实度）：回答是否基于检索文档
  - Answer Relevance（相关性）：回答是否切题
  - Context Precision（精确度）：检索结果中相关内容的比例
  - Context Recall（召回率）：回答需要的所有信息是否都被检索到
- 端到端评估：从用户问题到最终回答的完整评估
- 从评估到优化的闭环：发现问题 → 定位原因 → 优化 → 重新评估
- 常见问题模式：检索不到、检索到但不相关、相关但 LLM 没用好

**与前篇关联**：前面 5 篇构建了完整的 RAG 系统，这一篇给它加上"仪表盘"——用数据驱动持续优化。
