上一篇你学会了把文本变成向量。接下来的问题：这些向量存到哪？怎么查？1000 条向量时随便存，100 万条时还能毫秒级检索吗？

这一篇解决向量数据库的选型和优化问题。

## 向量数据库在 RAG 中的角色

向量数据库只做一件事：**存向量，查最近邻**。给它一个查询向量，返回数据库中距离最近的 N 个向量。

```
查询向量：[0.023, -0.041, 0.087, ...]
         ↓
向量数据库：从 100 万条向量中找到距离最近的 5 条
         ↓
返回：[文档片段 A（距离 0.12）, 文档片段 B（距离 0.15）, ...]
```

你可以用任何数据库存向量（PostgreSQL 的 pgvector 插件），但专用向量数据库在检索性能上有数量级的优势。

## 主流向量数据库对比

| 数据库       | 类型            | 适合规模  | 部署方式     | 特色                          |
| ------------ | --------------- | --------- | ------------ | ----------------------------- |
| **Chroma**   | 嵌入式          | < 100 万  | 本地/Docker  | 最简单，开发调试首选          |
| **Qdrant**   | 独立服务        | < 1000 万 | Docker/云    | Rust 写的，性能好，过滤查询强 |
| **Milvus**   | 分布式          | 无上限    | Kubernetes   | 企业级，支持百亿级            |
| **Pinecone** | 全托管          | < 几百万  | 云服务       | 零运维，按量付费              |
| **Weaviate** | 独立服务        | < 1000 万 | Docker/云    | 内置多模态，GraphQL API       |
| **pgvector** | PostgreSQL 插件 | < 100 万  | 已有 PG 就行 | 不引入新组件，但性能一般      |

### 选型决策树

```
你的文档量有多大？
  │
  ├─ < 10 万条
  │   → Chroma（开发阶段）或 pgvector（已有 PostgreSQL）
  │
  ├─ 10 万 ~ 100 万条
  │   → Qdrant（自部署）或 Pinecone（不想运维）
  │
  └─ > 100 万条
      → Milvus（分布式）或 Pinecone（全托管）
```

对于我们的知识库场景（50 人团队，2000 篇文档，约 2 万个片段），Chroma 或 Qdrant 都够用。下面用这两个做对比演示。

## HNSW 索引：为什么向量检索这么快

100 万条向量，每条 1536 维。如果逐个计算距离，一次检索需要 100 万次 × 1536 维的乘法——太慢了。

HNSW（Hierarchical Navigable Small World）是向量数据库最常用的索引算法。它的核心思想：**不要和所有向量比较，只和"邻居"比较。**

```
原理类比：找北京最近的餐厅

暴力搜索：列出全国所有餐厅，逐一算距离 → 太慢
HNSW 方法：
  第 1 层（粗筛）：只有省份 → "最近的是河北省"
  第 2 层（细化）：河北省的城市 → "最近的是北京市"
  第 3 层（精确）：北京市的区 → "朝阳区"
  第 4 层（落地）：朝阳区的街道 → "三里屯附近"
```

每一层只和少量节点比较，逐层缩小范围。时间复杂度从 O(N) 降到 O(log N)。

```typescript
// Qdrant 创建 HNSW 索引的配置
const collectionConfig = {
  vectors: {
    size: 1536, // 向量维度（和 Embedding 模型一致）
    distance: "Cosine", // 距离度量
  },
  hnsw_config: {
    m: 16, // 每个节点的邻居数（越大越精确，但越占内存）
    ef_construct: 100, // 构建索引时的搜索宽度（越大越精确，构建越慢）
  },
};
```

两个关键参数：

- **m**：每个节点在图中连接多少个邻居。m 越大，图越稠密，检索越精确，但内存占用越高。默认 16 是个好的起点。
- **ef_construct**：构建索引时的努力程度。值越大，索引质量越好，但构建时间越长。这是一次性成本，建好之后不影响检索速度。

## Chroma：开发调试首选

Chroma 是最简单的选择，Python/JS 都支持，开箱即用：

```typescript
import { ChromaClient, OpenAIEmbeddingFunction } from "chromadb";

const client = new ChromaClient({ path: "http://localhost:8000" });

// 创建集合（如果不存在）
const collection = await client.getOrCreateCollection({
  name: "knowledge_base",
  embeddingFunction: new OpenAIEmbeddingFunction({
    openai_api_key: process.env.OPENAI_API_KEY,
    modelName: "text-embedding-3-small",
  }),
});

// 添加文档
await collection.add({
  ids: ["doc_001", "doc_002"],
  documents: [
    "经过团队评估，考虑到关系型查询需求占 80% 以上，我们选择 PostgreSQL 作为主数据库...",
    "根因分析：Redis 连接池耗尽导致支付服务雪崩。解决方案：增加连接池上限 + 实施熔断...",
  ],
  metadatas: [
    { source: "ADR-2025-012", type: "architecture", author: "chen_fei" },
    { source: "INC-2025-047", type: "incident", severity: "P1" },
  ],
});

// 查询
const results = await collection.query({
  queryTexts: ["为什么选择 PostgreSQL"],
  nResults: 4,
  where: { type: "architecture" }, // 元数据过滤
});

results.documents[0].forEach((doc: string, i: number) => {
  console.log(`结果 ${i + 1}:`, doc.slice(0, 80) + "...");
  console.log(`来源:`, results.metadatas[0][i].source);
});
```

Chroma 的 Docker 启动：

```bash
docker run -d -p 8000:8000 -v chroma-data:/chroma/chroma chromadb/chroma
```

## Qdrant：生产环境推荐

Qdrant 用 Rust 写的，性能比 Chroma 好很多，支持更复杂的过滤查询：

```typescript
import { QdrantClient } from "@qdrant/js-client-rest";

const qdrant = new QdrantClient({ url: "http://localhost:6333" });

// 创建集合
await qdrant.createCollection("knowledge_base", {
  vectors: { size: 1536, distance: "Cosine" },
});

// 添加向量（注意：Qdrant 需要你自己先算好向量）
await qdrant.upsert("knowledge_base", {
  points: [
    {
      id: "doc_001",
      vector: embeddingVector, // 你用 Embedding API 算出来的向量
      payload: {
        content: "经过团队评估，考虑到关系型查询需求...",
        source: "ADR-2025-012",
        type: "architecture",
        author: "chen_fei",
        createdAt: "2025-08-15",
      },
    },
  ],
});

// 带元数据过滤的检索
const results = await qdrant.search("knowledge_base", {
  vector: queryVector,
  limit: 5,
  filter: {
    must: [
      { key: "type", match: { value: "architecture" } },
      { key: "createdAt", range: { gte: "2025-01-01" } },
    ],
  },
});
```

元数据过滤在实际项目中非常有用。比如用户只想看"今年发布的架构决策"，你可以用过滤条件 `type: "architecture" AND createdAt >= "2025-01-01"`，在向量检索之前先缩小候选范围，既快又准。

## 向量更新的策略

文档会更新——API 改了、架构决策修订了、故障复盘补充了。向量怎么同步？

### 全量重建

最简单但最慢：删掉整个集合，重新 Embedding 所有文档。

适合：文档更新不频繁（每天/每周一次）、文档量不大（< 10 万）。

### 增量更新

只更新变化的文档。每个文档有唯一 ID，更新时 upsert 同一个 ID。

```typescript
// 文档更新时，用同一个 ID 重新 upsert
await qdrant.upsert("knowledge_base", {
  points: [
    {
      id: "doc_001", // 和创建时的 ID 一样
      vector: newEmbeddingVector, // 新的向量
      payload: {
        content: updatedContent,
        source: "ADR-2025-012",
        version: 2, // 标记版本
        updatedAt: "2025-11-01",
      },
    },
  ],
});
```

### 生产实践

```
文档变更 → 触发 Webhook
  → 提取变更的文档 ID
  → 重新 Embedding
  → Upsert 到向量数据库
  → 更新版本号和更新时间
```

建议给每个文档加一个 `contentHash` 字段。Embedding 前先算 hash，如果和数据库中的一致，说明内容没变，跳过。避免文档没改却重复调用 Embedding API。

## 这一篇你学到了什么

- **Chroma** 适合开发和中小规模，**Qdrant** 适合生产环境，**Milvus** 适合超大规模
- **HNSW 索引**用多层图结构实现近似最近邻搜索，把复杂度从 O(N) 降到 O(log N)
- **元数据过滤**是向量数据库的杀手级功能——先过滤再检索，精准又高效
- 向量更新策略：小规模全量重建，大规模增量更新 + contentHash 去重
