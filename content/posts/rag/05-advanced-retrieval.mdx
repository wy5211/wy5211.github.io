前四篇搭好了 RAG 基础设施：Embedding 模型、向量数据库、Chunking 策略。现在你的 RAG 系统能跑了，但检索准确率只有 60%。

用户问"CI/CD 怎么配"，检索结果第一名是"CICD 年度总结会议"——因为"CICD"这个词在两篇文档中都出现了，向量模型觉得它们语义相近。但用户要的是配置教程，不是会议纪要。

纯向量检索有盲区。这一篇介绍三种高级策略来补上它。

## 纯向量检索的三个盲区

盲区 1：**关键词精确匹配不行**

"PostgreSQL 16.4 的 pgvector 扩展"——向量模型理解了"PostgreSQL"和"pgvector"的语义，但"16.4"这个版本号被忽略了。你可能检索到 PostgreSQL 15 的文档。

盲区 2：**领域术语干扰**

"CI/CD"在向量空间中离"会议总结"比离"配置教程"更近——因为向量模型看到"CI/CD"出现在很多不同类型的文档中，它的语义表示变得模糊。

盲区 3：**用户表达 ≠ 文档表达**

用户说"部署怎么搞"，文档写"Production Deployment Guide"。人一眼就能看出是在说同一件事，但向量模型可能不这么认为。

## 策略 1：混合检索（Hybrid Search）

把向量检索和关键词检索（BM25）结合，各取所长：

```
用户查询："PostgreSQL 连接池配置"

向量检索（擅长语义理解）→
  ✅ 找到"数据库连接管理最佳实践"
  ❌ 漏掉"PostgreSQL 16.4 连接池参数说明"（因为"16.4"不参与语义）

关键词检索 BM25（擅长精确匹配）→
  ✅ 精确命中"PostgreSQL 连接池参数"
  ❌ 漏掉"数据库连接管理"（因为没有"PostgreSQL"这个关键词）

混合结果 → 合并两者，去重后排序
```

### BM25 是什么

BM25 是经典的关键词检索算法。它基于词频（TF）和逆文档频率（IDF）计算相关性：

- 一个词在一篇文档中出现次数越多，越相关（TF）
- 一个词在整个文档库中越罕见，区分度越高（IDF）
- "PostgreSQL"在所有文档中只出现 3 次 → 区分度高 → 权重大
- "配置"在所有文档中出现 500 次 → 区分度低 → 权重小

### 分数归一化

混合检索有个技术问题：向量检索的分数（余弦相似度 0~1）和 BM25 的分数（无固定范围）不在同一个尺度上。直接相加不行。

```typescript
// Min-Max 归一化：把两组分数都映射到 [0, 1]
function normalizeScores(scores: number[]): number[] {
  const min = Math.min(...scores);
  const max = Math.max(...scores);
  if (max === min) return scores.map(() => 1);
  return scores.map((s) => (s - min) / (max - min));
}

// 混合检索
async function hybridSearch(query: string, topK = 5) {
  // 1. 向量检索
  const vectorResults = await vectorDB.search(
    await embed(query),
    topK * 3 // 多取一些候选
  );

  // 2. BM25 关键词检索
  const bm25Results = await bm25Index.search(query, topK * 3);

  // 3. 归一化分数
  const vectorScores = normalizeScores(vectorResults.map((r) => r.score));
  const bm25Scores = normalizeScores(bm25Results.map((r) => r.score));

  // 4. 合并（权重可调）
  const alpha = 0.7; // 向量检索权重
  const merged = new Map<string, { content: string; score: number }>();

  vectorResults.forEach((r, i) => {
    merged.set(r.id, { content: r.content, score: alpha * vectorScores[i] });
  });

  bm25Results.forEach((r, i) => {
    const existing = merged.get(r.id);
    const bm25Score = (1 - alpha) * bm25Scores[i];
    if (existing) {
      existing.score += bm25Score; // 同一篇文档，分数相加
    } else {
      merged.set(r.id, { content: r.content, score: bm25Score });
    }
  });

  // 5. 按合并分数排序
  return [...merged.values()].sort((a, b) => b.score - a.score).slice(0, topK);
}
```

`alpha` 参数控制两者权重：0.7 表示 70% 向量 + 30% 关键词。经验上 0.6-0.8 效果最好，需要根据你的数据集调整。

## 策略 2：Query 改写

用户的问题通常不适合直接拿去检索。口语化、模糊、信息不足。

### Query Expansion（查询扩展）

把一个查询扩展为多个子查询，分别检索后合并：

```typescript
async function expandQuery(query: string): Promise<string[]> {
  const prompt = `用户在技术知识库中搜索以下问题。请生成 3 个不同角度的搜索查询，帮助找到相关文档。

用户问题：${query}

搜索查询（每行一个，不要编号）：`;

  const response = await llm.chat(prompt);
  return response.split("\n").filter((l) => l.trim());

  // 示例：
  // 用户问题："部署怎么搞"
  // 扩展为：
  // "Production deployment guide"
  // "部署流程和注意事项"
  // "CI/CD 自动化部署配置"
}

// 分别检索每个扩展查询，合并去重
const allQueries = await expandQuery("部署怎么搞");
const allResults = await Promise.all(
  allQueries.map((q) => vectorDB.search(await embed(q), topK: 3))
);
const merged = deduplicate(allResults.flat());
```

### HyDE（Hypothetical Document Embedding）

HyDE 的思路很巧妙：**让 LLM 先猜一个答案，用答案去检索。**

```
用户问："部署怎么搞？"

步骤 1：让 LLM 猜答案（不需要准确）
LLM 猜测："部署通常需要以下步骤：1. 构建项目 2. 构建 Docker 镜像
3. 推送到镜像仓库 4. 在服务器上拉取并运行..."

步骤 2：用这个猜测答案去做向量检索
→ 因为猜测答案的措辞更接近文档的表达方式
→ 检索效果比直接用"部署怎么搞"好得多
```

```typescript
async function hydeSearch(query: string) {
  // 1. 让 LLM 生成假设性答案
  const hypotheticalAnswer = await llm.chat(
    `请简要回答以下问题（不需要完全准确，给出你的最佳猜测）：\n${query}`
  );

  // 2. 用假设答案的 Embedding 去检索
  const vector = await embed(hypotheticalAnswer);
  return vectorDB.search(vector, topK: 5);
}
```

HyDE 为什么有效？因为 LLM 的猜测答案在措辞和术语上更接近文档的表达（"部署流程" vs "部署怎么搞"），用这种更正式的文本去检索，命中率自然更高。

### Step-back Prompting

让用户的问题退一步，变得更宽泛：

```
原始问题："支付服务宕机时 Redis 连接池从 200 调到 1000 的具体步骤"
退一步的问题："支付服务宕机的故障处理流程"

原始问题太具体，可能检索不到（文档里没有这段话）
退一步后更宽泛，能检索到故障复盘文档
```

```typescript
async function stepBackQuery(query: string): Promise<string> {
  const prompt = `你是一个搜索优化专家。请将以下具体问题改写为一个更宽泛、更通用的问题，帮助检索到更多相关文档。

具体问题：${query}
宽泛问题：`;

  return await llm.chat(prompt);
}

```

## 策略 3：重排序（Reranking）

初次检索用快速但粗糙的方法（向量 + BM25）找到 20 条候选，再用精确但慢的模型重新排序，挑出最好的 5 条。

### 为什么需要重排序

向量检索（Bi-encoder）和 BM25 都是**独立编码**——查询和文档分别变成向量/词频，然后比较。快，但不够精确。

重排序用 **Cross-encoder**——把查询和每个候选文档一起输入模型，直接判断相关性。慢，但精确得多。

```
Bi-encoder（初次检索）：
  查询 → 向量 A
  文档 → 向量 B
  比较 A 和 B 的距离 → 快但不精确

Cross-encoder（重排序）：
  [查询 + 文档1] → 相关性分数 0.95
  [查询 + 文档2] → 相关性分数 0.72
  [查询 + 文档3] → 相关性分数 0.31
  → 慢但精确
```

### 用 Cohere Rerank API

Cohere 提供了开箱即用的重排序服务：

```typescript
async function rerank(
  query: string,
  documents: string[],
  topN = 5
): Promise<{ index: number; relevanceScore: number }[]> {
  const response = await fetch("https://api.cohere.com/v2/rerank", {
    method: "POST",
    headers: {
      Authorization: `Bearer ${process.env.COHERE_API_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      model: "rerank-v3.5",
      query,
      documents,
      top_n: topN,
    }),
  });

  const data = await response.json();
  return data.results.map((r: any) => ({
    index: r.index,
    relevanceScore: r.relevance_score,
  }));
}

// 使用
const candidates = await hybridSearch("CI/CD 配置", 20); // 初次检索 20 条
const docs = candidates.map((c) => c.content);
const reranked = await rerank("CI/CD 配置教程", docs, 5); // 重排序取 5 条
```

### 什么时候需要重排序

- 检索结果 > 10 条时效果明显（候选越多，排序的价值越大）
- 文档类型多样时（技术文档、会议纪要混在一起，需要更精确的筛选）
- 初次检索准确率不够高时的第一优先优化手段

如果初次检索只有 3-5 条结果，重排序的价值不大。

## 实战：对比各策略的检索质量

把三种策略叠加到我们的知识库上：

```
基线（纯向量检索）：
  准确率：~60%
  问题：关键词匹配失败，领域术语干扰

+ 混合检索（向量 + BM25，alpha=0.7）：
  准确率：~75%
  提升：精确匹配问题解决

+ Query 改写（Query Expansion）：
  准确率：~80%
  提升：口语化查询效果改善

+ 重排序（Cohere Rerank）：
  准确率：~85%
  提升：候选集排序更精确
```

每种策略独立有提升，叠加后效果更明显。但也要注意：策略越多，延迟越高、成本越高。生产环境需要在准确率和延迟之间做取舍。

## 这一篇你学到了什么

- **混合检索**（向量 + BM25）补上关键词精确匹配的短板，是最实用的改进
- **Query 改写**（Expansion / HyDE / Step-back）让用户的口语化问题变成高质量查询
- **重排序**用 Cross-encoder 对候选结果二次排序，精度提升显著
- 每种策略独立有收益，但叠加越多延迟越高——需要根据场景取舍
- Qdrant 内置混合检索支持，比自己实现更高效
