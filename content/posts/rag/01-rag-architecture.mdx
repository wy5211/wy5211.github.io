团队新来的同事问："我们为什么用 PostgreSQL 不用 MongoDB？"

你把这个问题丢给 ChatGPT，它给你一段通用的对比——"PostgreSQL 适合关系型数据，MongoDB 适合文档型数据"。听起来没错，但完全不是你们团队的决策逻辑。你们的架构师陈飞写了一篇架构决策记录（ADR-2025-012），详细说明了评估过程和结论。ChatGPT 不知道这篇文档的存在。

**这就是 RAG 要解决的核心问题：让 LLM 基于你的私有数据回答问题，而不是只靠它训练时的知识。**

## LLM 的三个先天缺陷

LLM 很强大，但有三件事它做不到：

**1. 知识有截止日期**

GPT-4o 的训练数据截止到某个时间点。之后发生的事、新增的文档，它一概不知。

**2. 不知道你的私有数据**

你的 API 文档、架构决策记录、故障复盘——这些永远不会出现在 LLM 的训练数据里。

**3. 会编造答案**

LLM 的设计目标是"生成合理的文本"，不是"生成准确的文本"。当你问它不知道的事情时，它更倾向于编造一个看起来合理的答案，而不是说"我不知道"。

RAG 的思路很直接：既然 LLM 不知道，那我在提问时把相关知识一起给它，让它基于这些知识回答。

## RAG 的完整流程

一个最基础的 RAG 系统分五步：

```
用户提问："为什么用 PostgreSQL 不用 MongoDB？"
  │
  ▼
① 文档处理：把 ADR-2025-012 等文档切成小片段
  │
  ▼
② 向量化：用 Embedding 模型把每个片段变成数字向量
  │
  ▼
③ 索引存储：把向量存进向量数据库
  │
  ▼
④ 检索：把用户问题也变成向量，在数据库中找最相似的片段
  │
  ▼
⑤ 生成：把检索到的片段拼进 prompt，让 LLM 基于这些内容回答
```

对应到代码，核心就是这几行（伪代码）：

```typescript
// ④ 把用户问题向量化，检索相关文档
const queryVector = await embed("为什么用 PostgreSQL 不用 MongoDB？");
const relevantDocs = await vectorDB.search(queryVector, topK: 4);

// ⑤ 拼接 prompt，让 LLM 回答
const prompt = `
基于以下文档内容回答用户问题。如果文档中没有相关信息，请说"我不知道"。

文档内容：
${relevantDocs.map(d => d.content).join("\n\n")}

用户问题：为什么用 PostgreSQL 不用 MongoDB？
`;

const answer = await llm.chat(prompt);
```

这就是 Naive RAG——能用，但问题很多。

## RAG vs Fine-tuning vs 长上下文

给 LLM 补充知识有三种方案，RAG 只是其中之一：

| 方案            | 原理                      | 优点                         | 缺点                            | 适合场景                   |
| --------------- | ------------------------- | ---------------------------- | ------------------------------- | -------------------------- |
| **RAG**         | 检索相关文档，拼进 prompt | 知识实时更新、成本低、可溯源 | 检索可能不准、prompt 占用 token | 知识库问答、客服、文档搜索 |
| **Fine-tuning** | 用你的数据微调模型        | 模型内化了知识、推理快       | 训练成本高、知识不实时、难溯源  | 特定风格/格式的任务        |
| **长上下文**    | 把所有内容塞进 prompt     | 最简单、无信息损失           | 成本极高、有上限、噪声多        | 文档总结、小规模分析       |

一个常见的误区是"Fine-tuning 能让模型学会新知识"。实际上，Fine-tuning 更适合调整模型的行为和风格（如"用我们公司的语气写邮件"），而不是注入新的事实知识。要注入事实，RAG 更可靠。

实际项目中，三者经常组合使用。比如：RAG 提供知识检索，Fine-tuning 让模型学会用你们的术语和格式回答。

## Naive RAG 的五步，每步都有坑

基础 RAG 看起来简单，但每一步都可能出问题：

**① 文档处理的问题**

把一篇 2000 字的文档按 500 字符切分，关键信息可能正好被切断。比如根因分析在前半段的最后一段，解决方案在后半段的第一段。

**② 向量化的问题**

不同 Embedding 模型对语义的理解不同。有些模型对技术术语不敏感，"PostgreSQL" 和 "PG" 被当成完全无关的词。

**③ 索引存储的问题**

向量数据库选型不当——小规模用 Chroma 够了，但到了百万级文档就需要考虑分布式方案。文档更新后向量没同步，检索到的是过期信息。

**④ 检索的问题**

这是最大的瓶颈。用户说"CI/CD 怎么配"，纯向量检索可能返回"CICD 年度总结会议"而不是"GitHub Actions 配置指南"。语义相近，但意图完全不同。

**⑤ 生成的问题**

检索到了正确的文档，但 LLM 没好好利用——它可能无视检索内容，继续用自己的知识编造答案。或者检索到了 5 段内容，LLM 只看了前 2 段就回答了。

## Advanced RAG：三阶段的改进

针对 Naive RAG 的问题，业界总结出了 Advanced RAG 的改进思路。按环节分为三组：

### 检索前优化（Pre-retrieval）

目标：让查询更好，让文档更易检索。

```
用户原始问题："CI/CD 怎么配"
  │
  ├─ Query 改写 → "如何配置 GitHub Actions CI/CD 流水线"
  ├─ Query 扩展 → 同时搜索 "CI/CD 配置" + "GitHub Actions 教程" + "自动化部署"
  └─ HyDE → 让 LLM 先猜一个答案，用答案去检索（后面文章会详细讲）
```

同时，对文档端也做优化——调整 Chunking 策略、添加摘要、提取关键词作为元数据。

### 检索中优化（Retrieval）

目标：找得更准。

```
单一向量检索
  │
  ├─ 混合检索：向量 + BM25 关键词，各有所长
  ├─ 多路检索：同时用多种策略检索，合并去重
  └─ 父子文档：用小片段精准匹配，返回大片段给 LLM
```

混合检索是最实用的改进。向量检索擅长语义理解（"持续集成" ≈ "CI"），BM25 擅长精确匹配（"PostgreSQL" 必须精确出现）。两者结合能覆盖各自的盲区。

### 检索后优化（Post-retrieval）

目标：给 LLM 更好的输入。

```
原始检索结果（Top 10）
  │
  ├─ 重排序（Reranking）：用更精确的模型对结果二次排序
  ├─ 压缩：去掉与问题无关的段落，只保留关键信息
  └─ 去重：相似度过高的片段只保留一个
```

重排序的效果很显著。初次检索用快速但粗糙的模型找 20 条候选，重排序用精确但慢的模型从中挑出最好的 5 条。计算成本只增加了少量，但准确率大幅提升。

## Modular RAG：可插拔的架构

Advanced RAG 的改进思路催生了一种新架构——Modular RAG。每个环节都是一个可替换的模块：

```
                  ┌─────────────┐
Query ──────────►│ Query Engine │─── 改写/扩展/分解
                  └──────┬──────┘
                         │
                  ┌──────▼──────┐
                  │  Retriever   │─── 向量/BM25/混合/多路
                  └──────┬──────┘
                         │
                  ┌──────▼──────┐
                  │   Reranker   │─── Cross-encoder/Cohere
                  └──────┬──────┘
                         │
                  ┌──────▼──────┐
                  │  Generator   │─── Prompt 组装 + LLM
                  └──────┬──────┘
                         │
                       Answer
```

每个模块都可以独立替换和升级。Retriever 从 Chroma 换成 Qdrant，不影响其他模块。Reranker 从不用到用，只是加一个模块。

这种架构的好处是：**你可以像搭积木一样，根据实际效果逐步改进，而不是一次性搞一个复杂系统。**

## RAG 不是万能的

RAG 适合做"基于已知事实的回答"，但不适合：

**不适合的场景：**

- 需要推理和综合判断的问题（"我们该不该迁移到微服务？"——这需要综合很多信息做判断）
- 需要实时数据的场景（"现在有多少用户在线？"——RAG 处理的是静态文档）
- 创意生成类任务（"帮我写一篇产品文案"——不需要检索）

**适合的场景：**

- 知识库问答（"我们的 API 限流策略是什么？"）
- 文档搜索（"有没有关于数据库选型的设计文档？"）
- 客服问答（"企业版支持多少人同时在线？"）
- 合规查询（"GDPR 对数据导出有什么要求？"）
