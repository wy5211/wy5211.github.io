上一篇我们聊了 SDD 的方法论——Spec 层、Steering 层、Hook 层三层循环，以及 SDD 和 Claude Code Plan 模式的哲学差异。现在问题变成了：怎么写出一份好的 Spec？

Spec 写太细变成伪代码，Agent 失去灵活性；写太泛等于没写，Agent 自由发挥可能跑偏。这篇聊两个核心工具：EARS 记法（写 Requirements）和 GIVEN-WHEN-THEN（写验收标准），帮你找到"不多不少"的粒度。

## 从一个模糊的需求开始

假设产品经理说了一句："加一个用户注册功能"。

如果直接丢给 Claude Code，它大概率会这样干：创建一个注册页面、加一个 API 路由、连数据库、写几个字段验证——看起来像是那么回事，但细节可能全是错的。邮箱要不要验证？密码有什么要求？用户名能改吗？注册完跳哪？

Claude Code 的 Plan 模式能在一定程度上解决这个问题——Claude 会问你一些问题。但问题在于，Claude 问什么取决于它"想到"问什么。它可能忘了问"用户名是否可以包含中文"这种细节。

Kiro 的 SDD 换了一个思路：**不是让 AI 来问你，而是你在 Spec 里把所有细节说清楚**。EARS 记法就是帮你说清楚的工具。

## EARS 记法

EARS 是四个关键词的首字母：

- **Event（事件）**：WHEN 什么发生了
- **Condition（条件）**：AND/IF 什么前提满足
- **Action（动作）**：THEN 系统做什么
- **Response（响应）**：AND/ELSE 额外的响应

基本模板：

```
WHEN [事件发生]
AND [条件满足]
THEN [系统执行动作]
AND [产生响应]
```

用 EARS 写"用户注册"的需求：

```
WHEN 用户点击注册按钮
AND 所有必填字段已填写
AND 邮箱格式合法
AND 密码 >= 8 位且包含大小写和数字
AND 邮箱未被注册
THEN 创建用户账号
AND 发送验证邮件
AND 跳转到"验证邮箱"提示页

WHEN 用户点击注册按钮
AND 邮箱已被注册
THEN 显示"该邮箱已注册"
AND 提供"去登录"链接
AND 不创建重复账号

WHEN 用户点击注册按钮
AND 有必填字段未填写
THEN 高亮未填写的字段
AND 在字段旁显示具体错误信息
AND 禁用注册按钮直到所有错误修正
```

看到了吗？一个"加用户注册"的模糊需求，用 EARS 拆成了三个具体场景：正常注册、邮箱已存在、字段验证失败。每个场景都有明确的触发条件、系统行为和用户反馈。

这样的 Spec 交给 Kiro 的 Agent，它不需要猜——每个场景该怎么做都写得清清楚楚。

### EARS 的常见错误

**错误 1：把实现细节写进 Event**

```
WHEN 用户用鼠标点击 class="btn-register" 的按钮
AND 使用 React useState 管理表单状态
```

这是伪代码，不是需求。EARS 描述的是用户行为和系统响应，不关心用什么技术实现。

正确写法：

```
WHEN 用户提交注册表单
```

**错误 2：遗漏边界情况**

只写了正常流程，忘了错误场景。好的 Spec 应该覆盖：正常路径、错误路径、边界情况。

一个检查清单：

- 输入为空时怎么办？
- 输入非法时怎么办？
- 网络超时怎么办？
- 重复操作怎么办？（比如连续点两次注册按钮）

**错误 3：写得太泛**

```
WHEN 用户注册
THEN 注册成功
```

这不叫需求，这叫愿望。EARS 的价值在于结构化地描述每个场景的具体行为——"注册成功"之后发生了什么？跳转？提示？发邮件？都得写清楚。

## GIVEN-WHEN-THEN 验收标准

EARS 写的是需求（系统应该做什么），GIVEN-WHEN-THEN 写的是验收标准（怎么验证做对了）。

模板：

```
GIVEN [前置条件]
AND [额外上下文]
WHEN [触发事件]
THEN [期望结果]
AND [副作用]
```

看起来和 EARS 很像？区别在于：EARS 是写给 AI 看的需求说明，GIVEN-WHEN-THEN 是写给验证者的测试用例。一个说"做什么"，一个说"怎么验证做对了"。

用注册功能的例子：

```
# 正常注册
GIVEN 注册页面已加载
AND 数据库中没有 "test@example.com" 这个邮箱
WHEN 用户输入邮箱 "test@example.com" 和密码 "Pass1234"
AND 点击注册按钮
THEN 数据库中创建了一条新用户记录
AND 用户看到"验证邮件已发送"的提示
AND 用户收到验证邮件

# 邮箱已存在
GIVEN 注册页面已加载
AND 数据库中已存在 "test@example.com" 这个邮箱
WHEN 用户输入邮箱 "test@example.com" 和密码 "Pass1234"
AND 点击注册按钮
THEN 数据库中没有新增用户记录
AND 用户看到"该邮箱已注册"的提示
AND 提示中包含"去登录"的链接

# 密码不符合要求
GIVEN 注册页面已加载
WHEN 用户输入密码 "123"（少于 8 位）
AND 点击注册按钮
THEN 注册按钮被禁用
AND 密码输入框下方显示"密码至少 8 位，需包含大小写字母和数字"
```

GIVEN-WHEN-THEN 的好处是：每个场景都是一个可执行的测试用例。Agent 实现完功能后，你可以逐条验证。

## Spec 的三层结构

把 EARS 和 GIVEN-WHEN-THEN 组合起来，一份完整的 Spec 有三层：

### 第一层：Requirements（WHAT）——用 EARS

```markdown
## Requirements

WHEN 用户提交注册表单
AND 所有字段验证通过
THEN 创建账号并发送验证邮件

WHEN 用户提交注册表单
AND 邮箱已存在
THEN 提示"邮箱已注册"并提供登录链接

WHEN 用户提交注册表单
AND 字段验证失败
THEN 高亮错误字段并显示具体错误信息
```

### 第二层：Design（HOW）——架构决策

```markdown
## Design

- 使用 Next.js Server Actions 处理表单提交（参考 Steering Files tech.md）
- 密码用 bcrypt 哈希后存储
- 验证邮件通过 Resend API 发送
- 表单验证使用 zod schema
- 前端实时验证 + 后端二次验证
```

Design 层不需要写代码，但要说清楚技术决策和理由。这样 Agent 不需要在实现时做技术选型——你已经替它决定了。

### 第三层：Tasks（IMPLEMENTATION）——拆分 + GIVEN-WHEN-THEN

```markdown
## Tasks

### Task 1: 创建注册表单组件

- 邮箱输入框 + 密码输入框 + 注册按钮
- 实时验证：邮箱格式、密码强度
- GIVEN 表单已渲染 WHEN 页面加载完成 THEN 所有字段为空 AND 注册按钮显示但可点击

### Task 2: 创建 Server Action

- 接收邮箱和密码
- 验证邮箱唯一性
- 哈希密码并存储
- 发送验证邮件
- GIVEN 数据库无此邮箱 WHEN 调用 register(email, password) THEN 返回 { success: true } AND 数据库新增记录

### Task 3: 错误处理 UI

- 显示服务器返回的错误信息
- 重复提交保护（按钮 disabled 直到响应）
- GIVEN 注册请求进行中 WHEN 用户再次点击注册 THEN 按钮保持禁用 AND 不发送重复请求
```

每层各司其职：Requirements 说做什么、Design 说怎么做、Tasks 说做哪些。Agent 只需要按 Tasks 逐个实现，不需要做任何猜测。

## 粒度控制：怎么写不多不少

### 写太多（过度规格化）

```
WHEN 用户在邮箱输入框中按下键盘 A 键
THEN 输入框显示字符 "a"
AND 光标移动到下一个位置
AND 输入框边框保持灰色（未验证状态）
```

这种级别的 Spec 是在描述浏览器行为，不是需求。Agent 不需要你告诉它"输入框能打字"。

### 写太少（规格不够）

```
加一个注册功能
```

这不是 Spec，这是一个 todo。Agent 不知道邮箱要验证、密码有强度要求、重复邮箱要提示——它能做的只有猜。

### 刚好

```
WHEN 用户提交注册表单
AND 邮箱格式合法 AND 密码 >= 8 位（含大小写和数字）
AND 邮箱未被注册
THEN 创建账号 AND 发送验证邮件 AND 跳转验证提示页
```

描述了用户行为、验证规则、系统响应和后续动作。不多不少——没有实现细节，也没有遗漏关键行为。

一个实用的粒度标准：**如果换一个人（不是 AI）来实现，他看完 Spec 能不能直接开工而不再来问你问题？** 如果能，粒度刚好。如果他还需要追问细节，规格不够。如果他觉得你在教他写代码，规格过细。

## 对照 Claude Code 的 Plan 文件

在 [Claude Code 系列第三篇](/posts/claude-code-tips/03-plan-mode-deep-dive) 里，我展示过 Claude Code Plan 文件的结构：

```markdown
# 重构计划

## 现状分析

## 方案对比

## 推荐方案

## 实施步骤

## 风险点
```

两种格式各有侧重：

- **Claude Code Plan**：偏向"方案分析"——现状是什么、方案 A vs 方案 B、推荐哪个。适合不确定怎么做的任务，需要 AI 帮你分析。
- **Kiro Spec**：偏向"需求规格"——做什么、验收标准是什么。适合你已经知道要做什么、只需要 AI 帮你执行的任务。

实际使用中，两者可以互补。用 Claude Code Plan 来做方案探索，确定方案后再用 Kiro Spec 来规格化执行。这不是"二选一"，而是"先用哪个后用哪个"。

## Spec 模板

最后分享一个我常用的 Spec 模板，你可以直接复制修改：

```markdown
# {功能名称} Spec

## 背景

一句话说明为什么做这个功能。

## Requirements

WHEN {事件}
AND {条件}
THEN {动作}
AND {响应}

WHEN {事件}（错误场景）
AND {条件}
THEN {错误处理}
AND {用户反馈}

## Design

- 技术方案：{方案描述}
- 数据结构：{数据模型}
- API 设计：{接口定义}

## Tasks

### Task 1: {任务名}

- {具体步骤}
- 验收：GIVEN ... WHEN ... THEN ...

### Task 2: {任务名}

- {具体步骤}
- 验收：GIVEN ... WHEN ... THEN ...

## 边界情况

- {空输入、超时、并发、安全等}
```

用这个模板，一份 Spec 大约 30-60 行，写一次大概 15 分钟。比起"直接让 AI 干、来回改好几轮"的总时间，往往更快。
