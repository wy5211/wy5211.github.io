在 [Claude Code 系列第六篇](/posts/claude-code-tips/06-mcp-connect-world) 里，我详细讲过 Claude Code 的 MCP 生态——GitHub、Supabase、Playwright、Figma、Sentry，每个都是独立的 MCP 服务器，需要在 `settings.json` 里配置、管理 Token、处理超时和重连。

Kiro 不需要这些——至少在 AWS 生态内不需要。作为 Amazon 的产品，Kiro 对 AWS 服务的集成是原生的：不需要装 MCP 服务器、不需要配 Token、不需要处理连接问题。

如果你用 AWS，这是选 Kiro 最强的理由。如果你不用 AWS，这篇对你意义不大——可以直接跳到下一篇。

## 原生 vs 插件：两种集成模式

Claude Code 的 MCP 是"插件模式"——每个外部服务是一个独立的插件，你按需安装和配置。灵活，但有配置成本。

Kiro 对 AWS 的集成是"原生模式"——AWS 服务是内置的，不需要额外配置。开箱即用，但只限于 AWS 生态。

```
Claude Code：安装 GitHub MCP → 配置 Token → 连接 → 使用
Kiro + AWS：打开 Kiro → 使用（AWS 凭证自动从环境读取）
```

少了一整个"安装配置"的步骤。

## Kiro 支持的 AWS 集成

### Amazon Bedrock——模型平台

Kiro 默认使用 Amazon Bedrock 作为模型平台。你可以在 Kiro 里直接切换不同的模型：

- Claude 3.5 Sonnet（Anthropic）
- Amazon Nova（Amazon 自研）
- 其他 Bedrock 支持的模型

在 Claude Code 里，模型选择是通过参数指定的。在 Kiro 里，有一个可视化的模型选择器（`Ctrl+B`），切换更直观。

Bedrock 的另一个优势是企业级安全——你的代码和 prompt 不会离开 AWS 的网络，不经过公网。对于有合规要求的企业，这一点很重要。

### AWS CDK——基础设施即代码

如果你用 AWS CDK（Cloud Development Kit）管理基础设施，Kiro 能直接生成和修改 CDK 代码。它会读取你的 AWS 账户信息，生成符合最佳实践的 CDK Stack。

实际用法：你说"创建一个 S3 + CloudFront 的静态网站托管"，Kiro 直接生成完整的 CDK Stack，包括 Bucket 配置、Distribution 设置、Origin Access Control 等。不需要你手动查 CDK 文档。

在 Claude Code 里做同样的事，需要装 Context7 Skill 来拉最新的 CDK 文档，然后让 Claude 生成代码——多了一步，而且 CDK 的 API 更新频繁，训练数据可能过时。

### AWS Lambda——Serverless 函数

Kiro 的 Hooks 可以直接触发 Lambda 函数。比如你的 `onSave` Hook 不是跑本地命令，而是调用一个 Lambda 函数来做更复杂的处理。

```yaml
# .kiro/hooks/lambda-scan.yaml
trigger: onSave
condition: path_matches("src/**/*.py")
actions:
  - run: aws lambda invoke --function-name code-scanner {{file}}
  - notify: "安全扫描完成"
```

这在 Claude Code 里也能做（通过 Shell 命令调用 AWS CLI），但 Kiro 的集成更自然——Lambda 是 Kiro 生态的一部分，不是外部工具。

### Amazon S3——存储

Kiro 可以直接读写 S3 存储桶。比如你的项目模板存在 S3 上，Kiro 创建新项目时可以直接从 S3 拉取，不需要手动下载。

### Strands Agents Framework——AWS 的 Agent 框架

Strands 是 AWS 推出的轻量级 Agent 框架，Kiro 对它有原生支持。用 `@tool` 装饰器定义 Agent 能力，然后用 AgentCore Runtime 在 AWS 上运行。

这意味着你可以在 Kiro 里开发 Agent，然后一键部署到 AWS——从开发到部署的全流程在同一个工具里完成。

## 非 AWS 项目的集成方案

Kiro 不是只能用 AWS。它也支持 MCP 服务器——和 Claude Code 一样，可以连接 GitHub、数据库、浏览器等服务。

但这里有一个重要的区别：**MCP 在 Kiro 里是"二等公民"，在 Claude Code 里是"一等公民"**。

- Claude Code 的 MCP 是核心扩展机制，生态完善、社区活跃
- Kiro 的 MCP 支持是后来加的，配置方式类似但社区支持较少

如果你的项目不用 AWS，Kiro 的 MCP 体验不如 Claude Code。原因有三：

1. **配置复杂度**：Kiro 的 MCP 配置和 AWS 原生集成相比多了一层间接性
2. **社区支持**：Claude Code 社区有大量 MCP 使用经验和踩坑指南，Kiro 的 MCP 文档较少
3. **工具发现**：Claude Code 有 mcpmarket.com 排行榜来发现 MCP 服务器，Kiro 没有等价的发现渠道

## 实战对比：同一任务两种方式

任务：创建一个 API 接口，从 DynamoDB 读取数据并返回 JSON。

### Claude Code + MCP

```json
// .claude/settings.json
{
  "mcpServers": {
    "aws": {
      "command": "npx",
      "args": ["-y", "@aws/mcp-server"],
      "env": {
        "AWS_REGION": "us-east-1",
        "AWS_ACCESS_KEY_ID": "${AWS_ACCESS_KEY_ID}",
        "AWS_SECRET_ACCESS_KEY": "${AWS_SECRET_ACCESS_KEY}"
      }
    }
  }
}
```

配置好后，Claude 通过 MCP 调用 AWS SDK 来操作 DynamoDB。但它对 AWS 最佳实践的理解有限——它可能生成一个直接查询 DynamoDB 的 Lambda，而不是建议用 API Gateway + Lambda 的标准模式。

### Kiro + AWS 原生

不需要额外配置。直接说"创建一个从 DynamoDB 读取数据的 API"，Kiro 会：

1. 读取你的 AWS 凭证（自动从环境变量或 AWS config 获取）
2. 了解你的 DynamoDB 表结构（可以直接查询）
3. 生成符合 AWS 最佳实践的代码（API Gateway + Lambda + IAM Role）
4. 甚至可以生成对应的 CDK 基础设施代码

整个过程中你不需要配置任何 Token、不需要安装任何 MCP 服务器。Kiro 对 AWS 的理解是"原生"的——它知道 API Gateway + Lambda 是标准模式，不需要你教。

## 安全性对比

AWS 集成在安全性上 Kiro 也有优势。

**Claude Code + MCP**：AWS 凭证需要在配置文件里引用。如果配置文件被提交到 Git（即使用了环境变量引用），仍有泄露风险。而且 MCP 服务器是第三方进程，你需要注意它的安全审计。

**Kiro + AWS 原生**：凭证通过 AWS 标准方式管理（IAM Role、环境变量、AWS SSO）。Kiro 本身运行在 AWS 环境中，不需要把凭证传给第三方 MCP 服务器。对于有安全合规要求的企业，这是重要的考量。

## 定价角度

AWS 集成还影响成本结构。

- **Claude Code**：按 API 调用计费。每个 MCP 工具调用都消耗 token
- **Kiro**：$20/月（Pro 计划），包含 1000 credits。AWS 集成的操作消耗的 credits 和本地操作一样——不会因为用了 AWS 服务就额外收费

如果你的项目重度依赖 AWS 服务，Kiro 的固定月费可能比 Claude Code 的按量计费更划算。但如果只是偶尔用 AWS，Claude Code 的灵活性更好。

## 什么时候 Kiro 的 AWS 集成是决定性优势

- 你的团队已经在 AWS 上运行所有基础设施
- 你需要生成和维护大量 CDK / CloudFormation 代码
- 你的项目使用 Lambda、DynamoDB、S3、API Gateway 等 AWS 服务
- 你有合规要求，需要代码和 prompt 不离开 AWS 网络
- 你希望从开发到部署在同一个工具里完成

## 什么时候这个优势不重要

- 你不用 AWS（用 GCP、Azure 或自建基础设施）
- 你只用 GitHub 和数据库这类通用服务（Claude Code 的 MCP 生态更丰富）
- 你是个人开发者，不需要企业级安全和合规

Kiro 的 AWS 原生集成是它的"主场优势"——就像 Mac 对苹果生态的优化一样，原生集成的体验是插件方案无法完全复制的。但这个优势只在你用 AWS 时才成立。
