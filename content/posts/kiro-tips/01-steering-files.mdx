如果你用过 Claude Code，一定对 CLAUDE.md 不陌生——一个文件搞定项目上下文，写好之后 Claude 就像熟悉你项目的老同事。我在 [Claude Code 系列第一篇](/posts/claude-code-tips/01-claude-md-project-context) 里详细聊过怎么写好它。

Kiro 也有类似的东西，叫 Steering Files。但它不是一个文件，而是一整个目录。`.kiro/steering/` 下面有 `product.md`、`structure.md`、`tech.md`，甚至还能加自定义文件。

你的第一反应可能是：有必要搞这么复杂吗？一个 CLAUDE.md 不就够了吗？

答案是：取决于项目大小和团队规模。个人小项目，CLAUDE.md 确实够用。但当你在一个 10 人团队、多个模块、前后端分离的大型项目里工作时，一个文件的局限性就暴露了——不同角色的开发者需要不同的上下文，前端不想看后端的数据库规范，后端不想看 CSS 命名约定，但它们全挤在同一个文件里。

## Steering Files 和 CLAUDE.md 的核心区别

先看结构对比：

**Claude Code**：一个文件搞定一切

```
项目根目录/
├── CLAUDE.md          ← 所有项目上下文都在这里
└── ...
```

**Kiro**：按关注点分文件

```
项目根目录/
├── .kiro/
│   └── steering/
│       ├── product.md    ← 产品愿景和用户故事
│       ├── structure.md  ← 目录结构和命名约定
│       ├── tech.md       ← 技术栈和编码规范
│       └── ui-ux.md      ← （可选）UI/UX 规范
└── ...
```

本质区别不是文件数量，而是**关注点分离**。CLAUDE.md 把所有信息混在一起，Steering Files 把不同维度的信息拆开。

这带来几个好处：

- **精准加载**：Kiro 可以根据当前任务只加载相关的 Steering File。你在改前端样式时，不需要加载后端的数据库规范
- **独立维护**：产品经理可以维护 `product.md`，技术负责人可以维护 `tech.md`，各管各的不冲突
- **渐进式**：你可以从最简单的配置开始，随着项目复杂度增加逐步添加更多 Steering File

## 每个 Steering File 该写什么

### product.md——产品愿景和用户故事

这个文件回答的问题是：**我们在做什么、为谁做、解决什么问题**。

Claude Code 的 CLAUDE.md 通常不写这些——它是给 AI 看的技术文档，不需要产品背景。但 Kiro 的 SDD（规格驱动开发）流程需要产品上下文来生成更准确的需求和测试用例。

```markdown
# 产品愿景

## 产品定位

面向独立开发者的个人博客平台，支持 MDX 文章、系列管理、自动分类。

## 目标用户

- 独立开发者，有一定的前端基础
- 希望用 Markdown 写技术博客
- 关注 SEO 和加载性能

## 核心功能

- MDX 文章写作和管理
- 系列文章自动组织
- 自动生成分类和标签页面
- 响应式设计，移动端友好

## 约束

- 纯静态站点，无服务端
- 构建时间不超过 30 秒（当前 660+ 篇文章）
```

这个文件不需要经常改。项目启动时写好，只有大的功能变更时才更新。

### structure.md——目录结构和命名约定

这个文件回答的问题是：**代码怎么组织的、文件放哪里、怎么命名**。

这部分在 CLAUDE.md 里通常会写，但往往和其他约定混在一起。单独拿出来后更清晰：

```markdown
## 目录结构

- content/posts/{category}/ 文章目录，每个分类一个子目录
- src/components/ React 组件
- src/lib/ 工具函数和配置
- src/app/ Next.js App Router 页面

## 命名约定

- 文章文件：XX-kebab-case-title.mdx（XX 为序号）
- 组件文件：PascalCase.tsx
- 工具函数：camelCase.ts
- 目录名：kebab-case

## 禁止事项

- 不要手动添加"上一篇/下一篇"链接
- 不要在文章中使用 .md 后缀，必须用 .mdx
- 不要在组件中硬编码分类名，用 category-map.ts
```

这个文件对新加入团队的成员特别有用——他们能快速了解项目组织方式，而不是靠猜。

### tech.md——技术栈和编码规范

这个文件回答的问题是：**用什么技术、怎么写代码**。

和 CLAUDE.md 里的"代码风格"部分对应，但可以写得更详细：

```markdown
## 技术栈

- Next.js 15 (App Router)
- TypeScript 5 (strict mode)
- Tailwind CSS 4
- MDX for content

## 编码规范

- 函数组件 + hooks，不使用 class 组件
- 状态管理：Zustand（全局状态），React Query（服务端状态）
- API 路由使用 Next.js Route Handlers
- 类型定义放在 src/types/ 下

## 测试

- 运行：npm run build（构建时会验证所有文章格式）
- 构建失败 = 文章 frontmatter 格式错误

## 性能要求

- 首屏加载 < 1s
- Lighthouse Performance > 90
- 构建时间 < 30s
```

### 自定义 Steering File

除了三个标准文件，你还可以创建自定义的 Steering File。比如 `ui-ux.md`（设计规范）或者 `api.md`（API 设计规范）。命名随意，放在 `.kiro/steering/` 下就行。

```markdown
<!-- .kiro/steering/ui-ux.md -->

# UI/UX 规范

## 设计系统

- 颜色：使用 Tailwind 默认色板
- 间距：遵循 Tailwind 的 4px 基数
- 字体：系统字体栈，代码用 JetBrains Mono

## 响应式

- 移动优先
- 断点：sm(640px)、md(768px)、lg(1024px)
- 不支持 IE（不需要 polyfill）

## 暗黑模式

- 必须支持
- 使用 CSS 变量方案
- 图片在暗黑模式下降低亮度
```

## 把一个 CLAUDE.md 拆解成 Steering Files

来看一个实际的例子。这是我之前用 Claude Code 时的 CLAUDE.md（简化版）：

```markdown
# 个人博客

Next.js 15 静态博客，使用 Tailwind CSS 4 + MDX。

## 构建

npm run dev / npm run build

## 文章结构

- 路径：content/posts/{category}/XX-kebab-case-title.mdx
- frontmatter 必须包含：title, date, summary, tags, category, series, seriesOrder

## 代码风格

- TypeScript strict mode
- 函数组件 + hooks
- Tailwind CSS，不用内联 style
- 文件命名：kebab-case
```

30 行，对一个小项目来说完全够用。但如果我要把它拆成 Steering Files：

**product.md**（取产品定位部分）：

```markdown
个人博客平台。面向独立开发者，支持 MDX 文章写作和系列管理。
纯静态站点，关注加载性能和 SEO。
```

**structure.md**（取目录结构和命名部分）：

```markdown
文章路径：content/posts/{category}/XX-kebab-case-title.mdx
frontmatter 必须包含：title, date, summary, tags, category, series, seriesOrder
文件命名：kebab-case
```

**tech.md**（取技术栈和编码规范部分）：

```markdown
Next.js 15 + TypeScript strict + Tailwind CSS 4 + MDX
函数组件 + hooks，不用 class 组件
构建：npm run dev / npm run build
```

拆完之后你会发现：对于一个小项目，拆不拆其实差别不大。Steering Files 的价值在大型项目中才真正体现。

## 什么时候 Steering Files 比 CLAUDE.md 更好

### 团队协作

CLAUDE.md 是给 AI 看的，Steering Files 可以给人也看。产品经理维护 `product.md`，技术负责人维护 `tech.md`，设计师维护 `ui-ux.md`。每个人只关心自己那部分，不会互相干扰。

在 Claude Code 里，所有约定都挤在一个文件里，一个人改了可能影响其他人。特别是团队大了之后，CLAUDE.md 会越来越长，超过 200 行之后 Claude 开始"选择性忽略"——我在 [Claude Code 系列第一篇](/posts/claude-code-tips/01-claude-md-project-context) 里详细聊过这个问题。

### 大型项目

项目越大，上下文越多。CLAUDE.md 一个文件装不下所有的约定，而 Steering Files 可以无限扩展——每个模块、每个子系统都可以有自己的规范文件。

Kiro 的另一个优势是**按需加载**。当你在改前端时，Kiro 可以只加载 `structure.md` 和 `ui-ux.md`，不加载 `tech.md` 里的后端规范。这比把所有信息都塞进 CLAUDE.md 更高效。

### 合规要求

企业项目通常有合规要求——代码规范、安全规范、测试规范都要文档化。Steering Files 天然适合这种场景，每个规范一个文件，版本控制和审计追踪都方便。

## 什么时候 CLAUDE.md 够用

说了这么多 Steering Files 的好处，但我不是在说 CLAUDE.md 不好。

- **个人项目**：你一个人开发，不需要分工，一个文件够了
- **小型项目**：10 个文件以内的项目，约定不多，拆开反而增加管理成本
- **快速原型**：你只是验证一个想法，没空写详细规范
- **你主要用 Claude Code**：那就用 CLAUDE.md，没必要为了用 Kiro 的配置格式而切换工具

Steering Files 是一个更好的组织方式，但不是必须的。工具是为了效率服务的，不是为了"正确"而存在的。

## 两种配置可以共存

如果你同时在用 Kiro 和 Claude Code（后面会聊到这种场景），你可以两种配置都保留。CLAUDE.md 放在项目根目录给 Claude Code 用，`.kiro/steering/` 给 Kiro 用。

甚至可以这样做：维护 Steering Files 作为"源文件"，然后写个小脚本从 Steering Files 生成 CLAUDE.md。这样你只需要维护一套规范，两个工具都能用。

```bash
# 生成 CLAUDE.md 的简单脚本
cat .kiro/steering/product.md > CLAUDE.md
echo "" >> CLAUDE.md
cat .kiro/steering/structure.md >> CLAUDE.md
echo "" >> CLAUDE.md
cat .kiro/steering/tech.md >> CLAUDE.md
```

这样两边的信息始终保持同步，不需要手动维护两份。

## 进阶：Steering Files 的层级

Kiro 还支持 Steering Files 的层级——你可以有全局级、项目级、模块级三个层级：

- **全局级**（`~/.kiro/steering/`）：个人偏好，所有项目共享
- **项目级**（`.kiro/steering/`）：项目特定规范
- **模块级**（`.kiro/steering/modules/{module}/`）：子模块特定规范

这和 Claude Code 的全局/项目级 CLAUDE.md 类似，但多了一个模块级。在大型项目里，这个额外的层级很实用——支付模块和用户模块的技术规范可能完全不同。
