在 [Claude Code 系列第七篇](/posts/claude-code-tips/07-workflow-multi-agent) 里，我讲过 Claude Code 的 Workflow——Pipeline、Parallel、Loop-until-dry 三种编排模式，加上对抗验证等质量保障手段。Claude Code 的多 Agent 是"临时编排"——你定义任务，Claude 动态创建 Agent 执行。

Kiro 的多 Agent 是"预定义分工"——Subagent 是固定的专业化角色（前端、后端、DevOps、安全、测试），它们在各自的领域内工作。

这两种模型代表了不同的设计哲学：**编排式（Claude Code）vs 组织式（Kiro）**。

## 两种多 Agent 模型

### Claude Code：编排式

```
你定义任务 → 动态创建 Agent → Agent 执行 → 汇总结果
```

每个 Workflow 是临时构建的——你需要定义 Agent 的角色、输入、输出。灵活，但每次都要设计编排方案。

这像一个项目经理临时组建团队——每次任务来了，根据需要找不同的人来干。团队组成不固定，但可以精确匹配任务需求。

### Kiro：组织式

```
任务进入 → 分配给专业 Subagent → Subagent 在其领域内执行 → 结果汇总
```

Subagent 是预定义的角色——Frontend Agent、Backend Agent、Security Agent 等。任务进来后自动分配给对应的 Subagent。

这像一个常设部门的公司——前端部、后端部、安全部，各司其职。任务来了按领域分配，不需要每次重新组建团队。

## 用同一个任务对比

任务：一个包含前端和后端变更的 PR，需要做多维度代码审查。

### Claude Code 的做法

用 Parallel 模式启动 4 个临时 Agent：

```
Workflow:
parallel([
  Agent("审查 bug — 逻辑错误、边界条件"),
  Agent("审查安全 — XSS、注入、权限"),
  Agent("审查性能 — 不必要计算、N+1 查询"),
  Agent("审查可维护性 — 命名、复杂度")
])
```

每个 Agent 独立审查，结果汇总后做对抗验证。整个过程你需要手动定义审查维度和 Agent 数量。

优势：完全自由——你可以定义任意维度、任意数量的审查 Agent。

### Kiro 的做法

触发代码审查后，Kiro 自动分配给对应的 Subagent：

```
Security Agent → 审查安全相关代码
Frontend Agent → 审查前端组件变更
Backend Agent → 审查 API 和数据库变更
Testing Agent → 检查测试覆盖
```

每个 Subagent 在自己的领域内做专业审查，结果自动汇总到 Agent Monitor Dashboard。

优势：不需要手动定义——Subagent 的分工是预定义的，开箱即用。

## Agent Monitor Dashboard

Kiro 有一个 Claude Code 没有的功能：**Agent Monitor Dashboard**（`Ctrl+G` 打开）。

这个 Dashboard 实时显示：

- 当前有哪些 Subagent 在工作
- 每个 Subagent 在做什么任务
- 进度百分比和预估完成时间
- 资源使用情况（CPU、内存、token）
- 关键的"干预点"——你可以在 Agent 执行到某个步骤时介入

在 Claude Code 里，多 Agent 执行时你只能看文字输出一个字一个字地蹦——不知道整体进度、不知道哪个 Agent 慢、不知道是不是卡住了。Kiro 的 Dashboard 解决了这个问题——多 Agent 协作变得可视化。

这对大型任务特别有用。比如你在做一次大规模重构，5 个 Subagent 同时在改不同的模块。Dashboard 让你一眼看到哪些完成了、哪些还在跑、哪个可能出问题了。

## Checkpoint 系统

Kiro 另一个特色功能是 Checkpoint——在 Agent 执行的关键节点自动创建检查点，你可以随时回滚。

```
Checkpoint 1: 初始状态
Checkpoint 2: 前端组件修改完成
Checkpoint 3: 后端 API 修改完成
Checkpoint 4: 测试通过
← 当前位置
```

如果发现第 3 步的方向错了，你可以回滚到 Checkpoint 2，从头开始后端部分。

Claude Code 也有类似的机制——Worktree（在独立分支上工作）。但 Worktree 是手动创建和管理的，而 Kiro 的 Checkpoint 是自动创建的——Agent 每完成一个主要步骤就自动存一个检查点。

## 成本对比

多 Agent 的成本是一个需要认真考虑的问题。

**Claude Code Workflow**：

- 每个 Agent 都消耗 token
- 总消耗 = Agent 数量 × 单 Agent 消耗
- 没有固定月费，按用量计费
- 可以在 Workflow 中为不同 Agent 选择不同模型来控制成本

**Kiro Subagent**：

- 包含在月费里（$20/月 Pro 计划，1000 credits）
- 多 Agent 操作消耗的 credits 比单 Agent 多，但有上限
- 超出月度 credits 后按 $0.02/credit 计费

对于偶尔使用多 Agent的场景，Claude Code 的按量计费更划算。对于频繁使用多 Agent 的场景（比如大型企业项目），Kiro 的固定月费可能更经济。

## 各自适合什么

### 选 Claude Code Workflow 的场景

- **一次性复杂任务**：临时需要多 Agent 并行处理，不是常态
- **灵活维度**：每次任务的审查维度不同，需要自定义编排
- **成本敏感**：用得不多，不想付固定月费
- **非 AWS 技术栈**：项目不依赖 AWS 服务

### 选 Kiro 多 Agent 的场景

- **常态化多 Agent**：团队日常就需要多 Agent 协作，预定义分工更高效
- **AWS 项目**：Subagent 对 AWS 服务的理解是原生的
- **团队协作**：Dashboard 和 Checkpoint 让多 Agent 的管理更透明
- **需要审计**：Checkpoint 系统提供了完整的执行记录

### 两者配合使用

一个实际的做法是：用 Kiro 的 Subagent 做日常的结构化任务（代码审查、测试生成），用 Claude Code 的 Workflow 做一次性的复杂编排（大规模重构、多维度分析）。

Kiro 负责"稳定的日常"，Claude Code 负责"灵活的特殊情况"。

## Kiro 多 Agent 的局限

### 跨领域协作

Subagent 的专业化是优势也是局限。Frontend Agent 不了解后端逻辑，Backend Agent 不关心前端表现。如果你需要一个同时理解前后端的 Agent 来做全局优化，Kiro 的分工模型反而碍事。

Claude Code 没有这个问题——一个通用 Agent 可以同时处理前后端，虽然不如专业化 Agent 在单一领域精准，但全局视野更好。

### 自定义 Subagent

目前 Kiro 的 Subagent 角色是预定义的，你不能轻易创建新的 Subagent。如果你需要一个"数据库性能优化"或"国际化"的专用 Agent，目前没有好的方式。

Claude Code 在这方面更灵活——你可以创建任何角色的临时 Agent，只需要在 prompt 里定义它的职责。

### 生态成熟度

Claude Code 的 Workflow 有丰富的社区模式和实践分享（Pipeline、Parallel、Loop-until-dry、对抗验证等）。Kiro 的多 Agent 架构相对较新，社区最佳实践还在积累中。

## Agent 编排的最佳实践（通用）

不管你用 Kiro 还是 Claude Code，多 Agent 编排都有一些通用的最佳实践：

1. **不要为了用多 Agent 而用多 Agent**——单 Agent 能搞定的任务不要拆
2. **明确每个 Agent 的职责边界**——避免多个 Agent 做重复的事
3. **设置成本上限**——多 Agent 的消耗是 N 倍，必须有预算控制
4. **验证结果**——多 Agent 的输出需要汇总和去重，不能直接信任
5. **保留回滚能力**——无论是 Kiro 的 Checkpoint 还是 Claude Code 的 Worktree
