这个系列写了七篇，每篇都在对比 Kiro 和 Claude Code 的对应功能。从 Steering Files 到 SDD、从 Hooks 到 Skills、从 AWS 集成到多 Agent 架构，两个工具在每一个维度上都有不同的设计取舍。

这篇是收尾。我不会说"Kiro 更好"或者"Claude Code 更好"——因为答案取决于你是谁、在做什么项目、在什么团队里。我会给你一个实用的选择框架：什么场景用什么工具，两者怎么配合。

## 按场景选择

### 场景 1：大型企业项目（10 人以上团队）

**推荐：Kiro**

原因：

- **Steering Files 的分文件管理**比 CLAUDE.md 的单文件更适合团队——产品、技术、设计各管各的
- **SDD 的规格驱动**确保团队所有人理解一致——Spec 是团队沟通的工具
- **预定义 Subagent** 让不同角色的开发者使用各自领域的专业化 Agent
- **AWS 原生集成**如果团队用 AWS，省去大量 MCP 配置成本
- **固定月费**对团队预算更可控

### 场景 2：个人开发者或小团队（1-3 人）

**推荐：Claude Code**

原因：

- **CLAUDE.md 够用**——个人项目不需要分文件管理上下文
- **迭代式开发**更适合——不需要每次都写 Spec，直接上手更快
- **丰富的社区生态**——658+ Skills 随用随装，开箱即用
- **按量计费**——用多少付多少，不需要固定月费
- **80.9% SWE-bench**——综合代码能力目前业界最强

### 场景 3：快速原型和 MVP

**推荐：Claude Code**

原因：

- **不需要 SDD**——快速验证想法时，写 Spec 的时间可能比写代码还长
- **直接上手**——说一句"帮我实现 XXX"，Claude 就开始干
- **Plan 模式够用**——如果需要想清楚，Plan 模式比完整的 SDD 轻量得多

### 场景 4：需要规格化的需求开发

**推荐：Kiro**

原因：

- **SDD 方法论天然适合**——需求明确、需要文档化、需要验收标准
- **EARS + GIVEN-WHEN-THEN**——比 Claude Code Plan 文件更结构化
- **Spec 是活文档**——开发完成后 Spec 变成了需求文档和测试用例

### 场景 5：AWS 重度用户

**推荐：Kiro**

原因：

- **原生集成**不需要配置 MCP 服务器、不需要管理 Token
- **CDK 支持**直接生成基础设施代码
- **Bedrock 模型**在企业环境下更安全
- **固定月费**对重度用户更划算

### 场景 6：调试和代码分析

**推荐：Claude Code**

原因：

- **80.9% SWE-bench**——代码理解和 bug 定位能力更强
- **终端原生**——调试工具链（gdb、strace、日志）配合更自然
- **Sentry MCP**——直接查线上错误并分析
- **快速迭代**——调试需要快速试错，SDD 的结构化流程太重

### 场景 7：开源项目和非 AWS 技术栈

**推荐：Claude Code**

原因：

- **MCP 生态**对 GitHub、数据库、设计工具的支持更完善
- **社区生态**有大量 Skill 和最佳实践分享
- **不绑定云厂商**——GCP、Azure、自建基础设施都一样用

## 两者配合使用的策略

不是"二选一"。两个工具可以配合使用，各取所长。

### 策略 1：Kiro 规划 + Claude Code 执行

用 Kiro 的 SDD 写 Spec（规划），然后用 Claude Code 来实现（执行）。

流程：

1. 在 Kiro 里用 SDD 方法论分析需求、写 Spec
2. 把 Spec 导出为 Markdown
3. 在 Claude Code 里读取 Spec，按规格实现

这样你既有 Kiro 的结构化规划，又有 Claude Code 的灵活执行。而且 Steering Files 和 CLAUDE.md 可以共存——它们放在不同的目录，互不干扰。

### 策略 2：Kiro 做结构化任务 + Claude Code 做探索性任务

日常开发用 Kiro：

- 新功能开发（用 SDD 规划 → Subagent 实现）
- 代码审查（用 Security Agent + Testing Agent）
- AWS 相关工作（原生集成）

遇到特殊情况切 Claude Code：

- 紧急 bug 修复（不需要写 Spec，直接上）
- 技术调研和原型验证（需要快速试错）
- 复杂的多 Agent 编排（Workflow 比 Subagent 更灵活）

### 策略 3：配置共享

两边可以共享一部分配置：

- **代码规范**：在 Steering Files 的 `tech.md` 里写好，然后生成一份 CLAUDE.md 给 Claude Code 用
- **Hooks**：两边的 Hook 可以并行工作——Kiro 处理文件级自动化（onSave），Claude Code 处理会话级自动化（SessionStart、Stop）
- **记忆**：Kiro 的 Spec 历史和 Claude Code 的记忆系统可以互补——一个记录需求决策，一个记录开发经验

## 定价对比

**Kiro**：

- Free：50 credits/月 + 500 试用 credits
- Pro：$20/月，1000 credits
- Team/Enterprise：定制报价

**Claude Code**：

- 按用量计费（基于模型和 token 消耗）
- 大约 $0.01-0.05/次交互（取决于任务复杂度和模型选择）
- 重度使用大约 $50-200/月

**简单的成本估算**：

- 如果你每天使用 AI 编程工具不到 1 小时 → Claude Code 按量计费更便宜
- 如果你每天使用 3 小时以上 → Kiro 的 $20 固定月费更划算
- 如果你的团队 10 人都用 → Kiro Team 方案有量价优势

## 各自的能力边界

### Kiro 擅长

- 结构化的功能开发（SDD 流程）
- AWS 原生集成（从开发到部署）
- 团队协作（Steering Files + Spec 共享）
- 大型项目的代码一致性和质量保障
- 可视化的多 Agent 监控和回滚

### Kiro 不擅长

- 快速原型和探索性开发（SDD 的前期投入大）
- 非 AWS 技术栈的深度集成
- 灵活的多 Agent 编排（Subagent 角色固定）
- 社区生态（还在早期阶段）
- 一次性小任务（杀鸡用牛刀）

### Claude Code 擅长

- 快速上手和迭代开发
- 调试和代码分析（SWE-bench 80.9%）
- 灵活的多 Agent Workflow 编排
- 丰富的社区生态（Skills + MCP）
- 个人开发者和小团队
- 非 AWS 技术栈

### Claude Code 不擅长

- 团队级的一致性保障（CLAUDE.md 单文件管理有局限）
- AWS 原生集成（需要 MCP 配置）
- 需求规格化和文档化
- 固定预算（按量计费不可预测）
- 可视化的 Agent 监控

## 系列回顾

八篇文章，我们从一个问题开始——"Kiro 和 Claude Code 到底有什么不同"——然后逐层拆解：

1. **Steering Files vs CLAUDE.md**——多文件分关注点 vs 单文件搞定
2. **SDD vs Plan 模式**——规格驱动 vs 迭代驱动
3. **Spec 编写**——EARS + GIVEN-WHEN-THEN 的实战技巧
4. **Hooks 对比**——YAML 声明式 vs JSON 命令式
5. **能力扩展**——Powers/Subagents/Skills vs Skills/MCP
6. **AWS 集成**——原生 vs 插件
7. **多 Agent**——预定义分工 vs 临时编排
8. **这篇**——选择框架与配合策略

两个工具不是竞争关系，而是互补关系。Kiro 像一个严谨的项目经理——先规划再执行，确保质量和一致性。Claude Code 像一个灵活的高级工程师——快速上手、聪明地解决问题、善于调试和分析。

最好的策略不是选一个，而是根据任务特点选择合适的工具。希望这个系列能帮你建立自己的判断框架。
