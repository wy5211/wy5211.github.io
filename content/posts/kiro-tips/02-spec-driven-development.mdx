上一篇我们聊了 Steering Files——Kiro 版的 CLAUDE.md，多文件、分关注点、适合大型项目。但 Steering Files 只是 Kiro 的配置系统，真正让 Kiro 和 Claude Code 产生本质区别的是 SDD（Spec-Driven Development，规格驱动开发）。

Claude Code 的 Plan 模式是"先想再干"——进入 Plan 模式，Claude 分析项目、列出方案、等你确认，然后动手。这个流程在 [Claude Code 系列第三篇](/posts/claude-code-tips/03-plan-mode-deep-dive) 里详细聊过。

Kiro 的 SDD 更进一步：不只是"想"，而是要把想法写成一份结构化的规格文档（Spec），然后 Agent 严格按照规格来实现。Spec 变成了开发流程的一等公民——它不是一次性的思考过程，而是一份持续维护的活文档。

## SDD 的三层控制循环

Kiro 用三层循环来管理开发过程：

```
Spec 层（WHAT）   → 你要做什么？需求是什么？
Steering 层（HOW） → 怎么做？用什么技术、遵循什么规范？
Hook 层（WHEN）   → 什么时候自动检查？什么时机触发？
```

上一篇讲的 Steering Files 就是 Steering 层——它们定义了"怎么做"。这篇聚焦 Spec 层——定义"做什么"。

Hook 层我们后面再展开。现在只需要记住：三层循环让 Kiro 的开发过程比 Claude Code 更结构化。Claude Code 的 Plan 模式只覆盖了"想清楚再干"这一个环节，而 Kiro 把"想清楚"、"按规范干"、"干完自动检查"串成了一个完整的闭环。

## SDD 的完整流程

一个完整的 SDD 流程分三个阶段：

### Requirements（需求）——定义要做什么

用自然语言描述需求，但要遵循 EARS 记法（下一篇会详细讲）。关键是用结构化的方式描述"在什么情况下、发生了什么、系统应该怎么响应"。

比如一个博客搜索功能的需求：

```
WHEN 用户在搜索框输入关键词
AND 关键词长度 >= 2 个字符
THEN 搜索文章标题和摘要
AND 实时显示匹配结果（debounce 300ms）
AND 高亮匹配的关键词
```

### Design（设计）——定义怎么实现

描述技术方案——用什么架构、什么数据结构、什么算法。不需要写代码，但要说清楚技术决策。

```
搜索方案：
- 前端实时搜索（无服务端）
- 使用 FlexSearch 建立搜索索引
- 构建时生成索引文件（搜索内容是静态的）
- debounce 300ms 减少不必要的搜索
```

### Tasks（任务）——定义实现步骤

把需求和设计拆成具体的实现任务，每个任务有明确的验收标准。

```
Task 1: 集成 FlexSearch
  - 安装 flexsearch 依赖
  - 创建搜索索引构建脚本
  - 在构建流程中加入索引生成步骤

Task 2: 搜索组件
  - 创建 SearchBar 组件
  - 实现输入 debounce
  - 实现搜索结果展示和关键词高亮

Task 3: 构建集成
  - 在 next.config 中添加索引生成步骤
  - 确保构建时间增加 < 5s
```

## 和 Claude Code Plan 模式的对比

用同一个任务对比一下两种方式。

任务：给博客加一个"相关文章推荐"功能。

**Claude Code Plan 模式**：

1. 我说"先做 Plan"
2. Claude 读了项目结构，分析了现有分类系统
3. 写了一份 Plan：按标签匹配、改三个文件、加一个组件
4. 我确认，Claude 开始实现
5. 实现过程中发现标签数据不够丰富，调整方案
6. 前后大概 2 轮 Plan + 3 轮实现

**Kiro SDD**：

1. 我写了一份 Spec（Requirements + Design + Tasks）
2. Kiro 读取 Steering Files 了解项目规范
3. Agent 按规格实现——严格按照我写的验收标准
4. Hook 自动检查格式和类型
5. 实现完成，对比 Spec 验收

两者最大的区别在于**灵活性 vs 确定性**：

- Claude Code 的 Plan 是一个**讨论过程**——Claude 提方案、你确认或修改、来回几轮。灵活，但结果取决于讨论质量
- Kiro 的 SDD 是一个**执行过程**——你写好规格、Agent 按规格执行。确定性强，但前期投入更多

这没有绝对的优劣——就像瀑布流和敏捷不是谁替代谁的关系，而是各自适合不同的场景。

## 什么任务适合 SDD

### 适合 SDD 的场景

**明确需求的新功能**：你知道要做什么、怎么做，只是需要 Agent 帮你实现。SDD 能确保 Agent 严格按照你的意图来，不会自由发挥跑偏。

**大型团队协作**：Spec 可以作为团队沟通的工具——产品经理写 Requirements，架构师写 Design，开发拆 Tasks。每个人都能看到完整的计划。

**需要审计追踪的项目**：Spec 是一份活文档，记录了"当初为什么这样设计"。这在合规审查时很有用。

**重复性的开发模式**：如果你的团队经常做类似的功能（比如 CRUD 接口），可以做成 Spec 模板，每次只需要填内容。

### 不适合 SDD 的场景

**简单的 bug 修复**：修一个 typo 或者改一个 CSS 值，写 Spec 的时间比修 bug 本身还长。

**探索性任务**：你不确定要做什么，需要先试试看。SDD 要求先想清楚，但有些问题就是需要动手试才能想清楚。

**紧急修复**：线上挂了，需要立刻修。没时间写 Spec。

**一次性脚本**：写完就扔的工具脚本，不值得规格化。

一个简单的判断标准：**如果你能在 5 分钟内清楚说出需求和技术方案，SDD 的价值就不大。如果你需要 30 分钟以上来理清思路，SDD 能帮你把这个思考过程结构化。**

## SDD 的一个实际案例

让我用一个真实的例子走一遍 SDD 的流程。

需求：给博客加暗黑模式支持。

### Claude Code 的处理方式

我说"加暗黑模式"，Claude 进 Plan 模式，分析了项目：

- 发现用的是 Next.js App Router
- 列了三个方案：CSS 变量 / next-themes 库 / Tailwind dark mode
- 推荐了 next-themes
- Plan 列出需要改的文件

来回大概 4 轮搞定。快，但方案的质量取决于 Claude 的判断——如果它没想到 next-themes 这个库，可能给你一个更笨的方案。

### Kiro SDD 的处理方式

我先写一份 Spec：

```markdown
# 暗黑模式 Spec

## Requirements

WHEN 用户点击主题切换按钮
AND 当前为亮色模式
THEN 切换为暗色模式
AND 保存用户偏好到 localStorage
AND 下次访问时自动应用上次选择的主题

WHEN 系统偏好为暗色（prefers-color-scheme: dark）
AND 用户没有手动选择过主题
THEN 默认使用暗色模式

## Design

使用 next-themes 库管理主题状态：

- 原因：Next.js App Router 兼容性好、支持 SSR、自动处理 flash
- 主题值：light / dark / system
- 存储方式：localStorage + cookie（SSR 时不闪烁）

组件结构：

- ThemeProvider 包裹 layout.tsx
- ThemeToggle 组件放在导航栏
- 使用 Tailwind CSS dark: 变体

## Tasks

Task 1: 安装 next-themes，配置 ThemeProvider

- 验收：npm run build 通过

Task 2: 创建 ThemeToggle 组件

- 验收：点击切换亮/暗模式，图标随主题变化

Task 3: 给所有页面添加暗色样式

- 验收：所有页面在暗色模式下可读、对比度足够
- 验收：代码块在暗色模式下语法高亮正常

Task 4: 处理 SSR 闪烁

- 验收：刷新页面时不会先闪亮色再切暗色
```

然后把这份 Spec 交给 Kiro。Agent 读取 Steering Files（了解项目的技术栈和编码规范），按 Tasks 逐个实现。每个 Task 完成后自动检查验收标准。

关键差异：**在 SDD 里，是我决定用 next-themes 而不是 CSS 变量**。在 Claude Code Plan 模式里，是 Claude 推荐 next-themes。前者我有完全的控制权，后者我依赖 Claude 的判断。

哪个更好？取决于你对技术方案的把握程度。如果你很清楚该怎么做，SDD 让你的意图精确传达。如果你不确定，Claude Code 的讨论式 Plan 可能更适合——它帮你想，而不是你告诉它。

## SDD 的代价

SDD 不是免费的午餐。它的代价是前期投入更多时间。

写一份完整的 Spec（Requirements + Design + Tasks）大约需要 15-30 分钟。对于简单任务，这个时间比直接让 Agent 干活还长。

Kiro 社区里有一个常见的吐槽："我用 Kiro 修一个 CSS bug，它生成了 16 条验收标准。杀鸡用了牛刀。"

这不是 Kiro 的问题，是使用方式的问题。SDD 是一把重型武器——用它打蚊子当然不合适，但用它拆房子就很趁手。
