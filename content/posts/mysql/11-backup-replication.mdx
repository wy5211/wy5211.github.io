## 凌晨三点的那个电话

周五晚上十一点，你的电商系统一切正常，准备回家过周末。凌晨三点手机响了——运维同事说，他在做数据库维护时不小心执行了 `DROP TABLE orders`。线上订单数据全部消失，客服群里已经炸了。

你打开电脑，冷静下来问了自己三个问题：

1. 有没有备份？最近一次备份是什么时候？
2. 从备份到误操作之间，还有多少数据没被覆盖？
3. 数据库能不能在恢复期间继续对外提供服务？

如果任何一个问题的答案是否定的，那就不是周末加班的问题了，而是职业生涯的问题。

上一篇我们聊了用户管理与安全，通过权限控制和账号体系防止"不该操作的人做了不该做的事"。但安全做得再好，也防不住有权限的人犯糊涂——备份才是最后一道防线。而当我们聊到"恢复期间还能不能继续服务"时，就自然引出了主从复制和高可用的需求。这一篇，我们把这两件事一起解决。

贯穿全文的案例是：为电商系统 `shop` 数据库搭建一套完整的数据安全保障体系——从备份脚本到 point-in-time 恢复，再到主从复制架构和读写分离方案。

## 一、备份类型：选对工具比什么都重要

MySQL 的备份方案可以按两个维度分类：**备份方式**（逻辑 vs 物理）和**备份范围**（全量 vs 增量 vs 差异）。

### 逻辑备份 vs 物理备份

逻辑备份导出的是 SQL 语句或 CSV 文本，物理备份直接复制磁盘上的数据文件。两者的核心区别在于恢复速度和对生产环境的影响。

| 维度     | 逻辑备份（mysqldump）    | 物理备份（XtraBackup）                      |
| -------- | ------------------------ | ------------------------------------------- |
| 备份内容 | SQL 语句或文本文件       | InnoDB 数据文件副本                         |
| 备份速度 | 慢（需要生成 SQL）       | 快（文件级别复制）                          |
| 恢复速度 | 慢（逐条执行 SQL）       | 快（直接拷贝文件）                          |
| 跨平台   | 可以                     | 不可以                                      |
| 可读性   | 文本，可以编辑和审查     | 二进制，不可读                              |
| 适用场景 | 小中型数据库、跨平台迁移 | 大型数据库、快速恢复                        |
| 工具     | mysqldump、mydumper      | Percona XtraBackup、MySQL Enterprise Backup |

对于电商系统 `shop` 数据库（假设几十 GB 级别），逻辑备份完全够用，而且可读、可审查的优势在出问题时非常有价值。如果数据量到了 TB 级别，就需要考虑物理备份了。

### 全量、增量与差异

这三者的区别决定了你的备份策略怎么设计：

- **全量备份**：备份整个数据库。恢复最简单，但耗时最长、占用空间最大。
- **增量备份**：只备份自上次备份以来变化的数据。节省空间，但恢复时需要依次回放所有增量。
- **差异备份**：备份自上次全量备份以来变化的数据。恢复比增量简单（只需全量 + 最新差异），但占用的空间比增量多。

对于我们的电商系统，推荐的是"全量 + binlog"的组合——每周一次全量逻辑备份，配合 binlog 实现任意时间点的恢复。这比传统的"全量 + 增量"方案更灵活，因为 binlog 是 MySQL 原生支持的，不需要额外的增量备份工具。

## 二、mysqldump：最常用的备份工具

`mysqldump` 是 MySQL 自带的逻辑备份工具，几乎所有 MySQL 环境都能直接使用。

### 全库备份与单表备份

最基本的用法：

```bash
# 备份整个 shop 数据库
mysqldump -uroot -p shop > /backup/shop_backup.sql

# 备份所有数据库（用于灾备）
mysqldump -uroot -p --all-databases > /backup/all_databases.sql

# 只备份订单表和用户表（出问题时只恢复关键表）
mysqldump -uroot -p shop users orders > /backup/shop_critical_tables.sql
```

这几个命令足够应对大部分场景。但要注意，默认参数下 `mysqldump` 会对表加锁，在备份期间写入会被阻塞。对于线上环境，必须加上 `--single-transaction` 参数。

### 生产环境推荐参数

```bash
# 生产环境备份的标准参数组合
mysqldump -uroot -p \
  --single-transaction \   # InnoDB 表不加锁，利用 MVCC 生成一致性快照
  --routines \             # 导出存储过程和函数
  --triggers \             # 导出触发器
  --events \               # 导出定时事件
  --quick \                # 逐行导出，避免大表撑爆内存
  --set-gtid-purged=OFF \  # 导入到其他实例时不保留 GTID 信息
  shop > /backup/shop_backup.sql
```

参数解读：

- `--single-transaction` 是最关键的一个参数。它利用 InnoDB 的 MVCC 机制，在备份开始时创建一个一致性快照，备份过程中不锁表，业务可以正常读写。
- `--quick` 防止 `mysqldump` 一次性把整张表读入内存。对于百万级大表，不加这个参数可能会导致 OOM。
- `--set-gtid-purged=OFF` 是当你需要把备份导入到另一个 MySQL 实例（比如从库或者测试环境）时必须加的，否则会因为 GTID 冲突而导入失败。

### 压缩备份

数据库备份文件通常很大，压缩可以节省 70%-90% 的存储空间：

```bash
# 直接管道压缩，不生成中间文件
mysqldump -uroot -p --single-transaction shop | gzip > /backup/shop_$(date +%Y%m%d).sql.gz

# 恢复时解压管道导入
gunzip < /backup/shop_20260330.sql.gz | mysql -uroot -p shop
```

### 恢复数据

恢复操作是备份的价值所在，但恢复前有几个注意事项：

```bash
# 1. 恢复到测试环境验证（不要直接恢复到生产库）
mysql -uroot -p shop_test < /backup/shop_backup.sql

# 2. 如果需要恢复到生产库，先确认备份文件的完整性
head -n 20 /backup/shop_backup.sql
# 应该看到 "MySQL dump 10.13" 开头的注释

# 3. 恢复生产数据
mysql -uroot -p shop < /backup/shop_backup.sql

# 4. 也可以在 MySQL 客户端内用 source 命令恢复
mysql -uroot -p
mysql> USE shop;
mysql> SOURCE /backup/shop_backup.sql;
```

## 三、binlog：实现任意时间点恢复的关键

`mysqldump` 的全量备份只能恢复到备份那个时刻的数据。如果备份是凌晨 2 点做的，而误操作发生在下午 3 点，中间 13 个小时的数据怎么办？答案就是 binlog（二进制日志）。

### 什么是 binlog

binlog 记录了所有对数据库数据进行修改的操作（INSERT、UPDATE、DELETE、DDL 等）。它最初是为复制设计的，但同时也是增量备份和 point-in-time 恢复的基础。

### 开启和配置 binlog

```ini
# /etc/mysql/my.cnf
[mysqld]
log-bin=mysql-bin                    # 开启 binlog，文件前缀为 mysql-bin
binlog_format=ROW                    # 推荐使用 ROW 格式（后面解释为什么）
expire_logs_days=7                   # binlog 保留 7 天（根据恢复需求调整）
max_binlog_size=256M                 # 单个 binlog 文件最大 256M，超出自动滚动
sync_binlog=1                        # 每次事务提交都刷盘（保证不丢数据，但有性能开销）
```

关于 `binlog_format` 的选择，有三种格式：

| 格式      | 记录内容                  | 优点                 | 缺点                                           |
| --------- | ------------------------- | -------------------- | ---------------------------------------------- |
| STATEMENT | 记录 SQL 语句原文         | 日志文件小           | 不确定性函数（NOW()）可能导致主从不一致        |
| ROW       | 记录每行数据的变化        | 最安全，主从一定一致 | 日志文件大（批量 UPDATE 会产生大量行变更记录） |
| MIXED     | 自动选择 STATEMENT 或 ROW | 折中方案             | 行为不可预测                                   |

生产环境推荐使用 **ROW** 格式。虽然日志文件更大，但恢复的可靠性远比磁盘空间重要。而且 ROW 格式可以通过 `mysqlbinlog` 工具精确地看到每行数据的变化，排查问题时非常方便。

### 查看 binlog 内容

```sql
-- 查看当前正在写入的 binlog 文件和位置
SHOW MASTER STATUS;
-- 结果示例：
-- File: mysql-bin.000003
-- Position: 154
-- Binlog_Do_DB:
-- Binlog_Ignore_DB:

-- 查看所有 binlog 文件列表
SHOW BINARY LOGS;

-- 查看最新 binlog 文件中的事件（默认显示前 20 条）
SHOW BINLOG EVENTS IN 'mysql-bin.000003' LIMIT 10;
```

也可以用命令行工具查看更详细的内容：

```bash
# 查看 binlog 的 SQL 内容（STATEMENT 格式可读性好）
mysqlbinlog /var/lib/mysql/mysql-bin.000003

# ROW 格式需要加 -v 参数才能看到具体的行变更
mysqlbinlog -v /var/lib/mysql/mysql-bin.000003

# 加 -vv 可以看到更详细的字段变更（实际数据值）
mysqlbinlog -vv /var/lib/mysql/mysql-bin.000003
```

### 基于 binlog 的 point-in-time 恢复

这是整个备份体系中最重要的能力：恢复到误操作前的那一刻。

假设场景：运维在 `2026-03-30 15:30:00` 误删了 `orders` 表，全量备份是 `2026-03-30 02:00:00` 做的。恢复步骤如下：

```bash
# 第一步：恢复全量备份到备份时刻（02:00）
mysql -uroot -p shop < /backup/shop_20260330.sql.gz

# 第二步：从 binlog 中找到误操作的精确位置
# 查看 02:00 到误操作之间的 binlog 事件
mysqlbinlog --start-datetime="2026-03-30 02:00:00" \
            --stop-datetime="2026-03-30 15:30:00" \
            /var/lib/mysql/mysql-bin.000003

# 第三步：只恢复到误操作之前的数据
mysqlbinlog --start-datetime="2026-03-30 02:00:00" \
            --stop-datetime="2026-03-30 15:29:59" \
            /var/lib/mysql/mysql-bin.000003 | mysql -uroot -p shop
```

也可以用 position 精确控制恢复范围：

```bash
# 先用 SHOW BINLOG EVENTS 找到误操作的起始 position
# 假设误操作从 position 4387 开始

# 恢复到 position 4387 之前的所有数据
mysqlbinlog --start-position=154 \
            --stop-position=4387 \
            /var/lib/mysql/mysql-bin.000003 | mysql -uroot -p shop
```

> **关键提醒**：恢复前务必先在测试环境验证。binlog 恢复是不可逆的，一旦回放错误的日志，数据会越搞越乱。

## 四、备份策略设计：不只是"定期备份"

有了备份工具和恢复方法，还需要一套完整的策略把它们组织起来。对于电商系统，我们设计如下备份方案：

### 推荐的备份节奏

```
周日 02:00  全量逻辑备份（mysqldump + gzip）
周一~周六   binlog 实时记录（保留 7 天）
每周        全量备份通过 rsync 同步到异地服务器
每月        在测试环境做一次恢复演练
```

### 自动备份脚本

把备份流程写成脚本，交给 cron 定时执行：

```bash
#!/bin/bash
# shop_backup.sh —— 电商系统自动备份脚本
# 用法：加入 crontab，每天凌晨 2 点执行
# crontab -e → 0 2 * * 0 /scripts/shop_backup.sh

set -euo pipefail  # 遇错即停、未定义变量报错、管道错误传递

# ===== 配置区 =====
BACKUP_DIR="/data/backup/mysql"
MYSQL_USER="backup_user"           # 上一篇创建的专用备份账号
MYSQL_PASSWORD="BackupPass123!"    # 生产环境建议用 mysql_config_editor
MYSQL_HOST="192.168.1.100"         # 主库地址
DB_NAME="shop"
RETENTION_DAYS=30                  # 本地备份保留 30 天

# ===== 创建备份目录 =====
mkdir -p "$BACKUP_DIR"
DATE_TAG=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/${DB_NAME}_${DATE_TAG}.sql.gz"

# ===== 执行全量备份 =====
echo "[$(date)] 开始备份 $DB_NAME ..."
mysqldump -h"$MYSQL_HOST" -u"$MYSQL_USER" -p"$MYSQL_PASSWORD" \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --quick \
  --set-gtid-purged=OFF \
  "$DB_NAME" | gzip > "$BACKUP_FILE"

# ===== 校验备份文件 =====
if [ $? -eq 0 ] && [ -s "$BACKUP_FILE" ]; then
  FILE_SIZE=$(du -h "$BACKUP_FILE" | cut -f1)
  echo "[$(date)] 备份成功: $BACKUP_FILE ($FILE_SIZE)"
  echo "[$(date)] 备份成功: $BACKUP_FILE ($FILE_SIZE)" >> "$BACKUP_DIR/backup.log"
else
  echo "[$(date)] 备份失败！" | tee -a "$BACKUP_DIR/error.log"
  exit 1
fi

# ===== 清理过期备份 =====
DELETED=$(find "$BACKUP_DIR" -name "*.sql.gz" -mtime +$RETENTION_DAYS -delete -print | wc -l)
echo "[$(date)] 清理了 $DELETED 个过期备份文件"

# ===== 异地同步（可选） =====
# rsync -az "$BACKUP_DIR/" user@remote-server:/data/backup/mysql/
```

### 3-2-1 备份原则

业界广泛认可的备份安全标准是 **3-2-1 原则**：

- **3** 份数据副本：1 份生产数据 + 2 份备份
- **2** 种不同介质：本地磁盘 + 对象存储（OSS/S3）或异地服务器
- **1** 份异地备份：防止单机房故障导致数据全部丢失

对于电商系统，这意味着：

1. 本地磁盘保留 30 天的全量备份（快速恢复）
2. 对象存储（如阿里云 OSS 或 AWS S3）保留 180 天的备份（长期归档）
3. 异地机房同步一份最近 7 天的备份（容灾）

### 备份加密

备份文件包含完整的业务数据，传输和存储时必须加密：

```bash
# 备份时通过管道加密
mysqldump -uroot -p --single-transaction shop \
  | gzip \
  | openssl enc -e -aes-256-cbc -salt -pass pass:YourEncryptionKey \
  > /backup/shop_encrypted.sql.gz.enc

# 恢复时反向解密
openssl enc -d -aes-256-cbc -pass pass:YourEncryptionKey \
  -in /backup/shop_encrypted.sql.gz.enc \
  | gunzip | mysql -uroot -p shop
```

## 五、主从复制：从单机到分布式

备份解决了"数据不丢"的问题，但它无法解决另一个问题：**恢复期间数据库不可用**。全量恢复几十 GB 的数据可能需要几十分钟甚至几个小时，这段时间电商系统完全无法下单。

主从复制（Replication）就是解决这个问题的关键技术：把数据实时同步到另一台服务器，主库挂了从库顶上，业务几乎无感知。

### 复制原理：binlog 是核心

主从复制的整个流程围绕 binlog 展开：

```
                     主库 (Master)
                       │
          ┌────────────┴────────────┐
          │  1. 应用写入数据         │
          │  2. 数据写入 InnoDB      │
          │  3. 变更记录到 binlog    │
          └────────────┬────────────┘
                       │ binlog
                       ▼
                  从库 IO 线程
                       │
          ┌────────────┴────────────┐
          │  4. 连接主库，拉取 binlog │
          │  5. 写入本地 relay log   │
          └────────────┬────────────┘
                       │ relay log
                       ▼
                 从库 SQL 线程
                       │
          ┌────────────┴────────────┐
          │  6. 读取 relay log       │
          │  7. 重放 SQL，写入从库    │
          └─────────────────────────┘
```

理解这个流程的关键点：

- **IO 线程**只负责搬运 binlog，不执行 SQL，所以它很轻量。
- **SQL 线程**负责回放 SQL，如果主库写入压力大，从库的 SQL 线程可能跟不上，这就是**复制延迟**。
- 复制是**异步**的：主库写入成功就返回给客户端了，不等从库同步完成。这意味着极端情况下从库可能丢数据。

### 主库配置

```ini
# /etc/mysql/conf.d/master.cnf —— 主库配置
[mysqld]
server-id=1                          # 每个实例的唯一 ID，主从不能相同
log-bin=mysql-bin                    # 开启 binlog
binlog_format=ROW                    # ROW 格式，保证主从数据一致
expire_logs_days=7                   # binlog 保留 7 天
sync_binlog=1                        # 每次事务提交都刷盘
read_only=OFF                        # 主库可读可写
```

在主库上创建复制专用账号：

```sql
-- 创建复制专用账号（上一篇的安全原则：最小权限）
CREATE USER 'repl_user'@'192.168.1.%' IDENTIFIED BY 'ReplPass123!';

-- 只授予复制所需的权限
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'192.168.1.%';

FLUSH PRIVILEGES;

-- 查看主库的 binlog 状态（配置从库时需要这两个值）
SHOW MASTER STATUS;
-- File: mysql-bin.000001, Position: 154
```

### 从库配置与启动复制

```ini
# /etc/mysql/conf.d/slave.cnf —— 从库配置
[mysqld]
server-id=2                          # 与主库不同
relay-log=relay-bin                  # 中继日志前缀
read_only=ON                         # 从库只读，防止误写入
log-bin=mysql-bin                    # 从库也开 binlog（方便级联复制和故障切换）
binlog_format=ROW
```

在从库上执行以下命令启动复制：

```sql
-- 告诉从库去哪里找主库的 binlog
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',        -- 主库 IP
  MASTER_USER='repl_user',            -- 复制账号
  MASTER_PASSWORD='ReplPass123!',     -- 复制密码
  MASTER_LOG_FILE='mysql-bin.000001', -- 主库的 binlog 文件名
  MASTER_LOG_POS=154;                 -- 主库的 binlog 位置

-- 启动复制
START SLAVE;

-- 检查复制状态（关键命令，出问题时第一个执行的）
SHOW SLAVE STATUS\G
```

`SHOW SLAVE STATUS` 中需要重点关注的字段：

```
Slave_IO_Running: Yes         -- IO 线程是否正常（连接主库、拉取 binlog）
Slave_SQL_Running: Yes        -- SQL 线程是否正常（回放 relay log）
Seconds_Behind_Master: 0      -- 复制延迟（秒数），0 表示完全同步
Last_Error:                   -- 最后一次错误信息（正常应该为空）
```

如果 `Slave_IO_Running` 是 `No`，通常是网络问题或复制账号权限问题。如果 `Slave_SQL_Running` 是 `No`，通常是从库上执行了冲突的 SQL（比如有人直接在从库写了数据）。

## 六、GTID 复制：为什么比传统方式更好

前面配置从库时，我们需要手动指定 `MASTER_LOG_FILE` 和 `MASTER_LOG_POS`，这叫**基于 binlog position 的复制**。这种方式有个明显的痛点：当主库故障需要把从库提升为新主库时，新主库的 binlog 位置和旧主库不同，重新配置其他从库时需要手动计算位置偏移，容易出错。

**GTID（Global Transaction Identifier，全局事务标识）** 解决了这个问题。每个事务在提交时会被分配一个全局唯一的 ID，格式为 `source_id:transaction_id`（例如 `3E11FA47-71CA-11E1-9E33-C80AA9429562:23`）。配置复制时不再需要指定 binlog 位置，MySQL 会自动根据 GTID 找到需要同步的起点。

### GTID 配置

主库和从库都需要开启 GTID 相关参数：

```ini
# 主库和从库都需要添加以下配置
[mysqld]
server-id=1                          # 主库为 1，从库为 2、3...
log-bin=mysql-bin
binlog_format=ROW
gtid-mode=ON                         # 启用 GTID
enforce-gtid-consistency=ON          # 强制 GTID 一致性（不允许创建不支持 GTID 的语句）
```

> `enforce-gtid-consistency=ON` 很重要。开启后，MySQL 会拒绝执行不支持 GTID 的语句，比如 `CREATE TABLE ... SELECT`（这种语句在 binlog 中会被拆成两个事务，无法用单一 GTID 标识）。遇到这个报错时，需要把语句拆成先 `CREATE TABLE` 再 `INSERT ... SELECT`。

### 用 GTID 启动复制

配置从库时，不再需要指定 binlog 文件和位置：

```sql
-- GTID 方式配置复制（注意最后一行的区别）
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_USER='repl_user',
  MASTER_PASSWORD='ReplPass123!',
  MASTER_AUTO_POSITION=1;            -- 自动根据 GTID 找到同步起点

START SLAVE;
```

`MASTER_AUTO_POSITION=1` 告诉从库：不需要手动指定位置，你自己根据 GTID 集合找到该从哪里开始同步。这在故障切换时非常方便——新主库的 GTID 集合自动包含了所有已提交的事务，其他从库连上来就能继续同步。

### GTID 方便故障切换的具体例子

假设主库 A 挂了，需要把从库 B 提升为新主库：

**传统方式**：需要先在 B 上执行 `SHOW MASTER STATUS` 获取 binlog 位置，然后在其他从库 C 上执行 `CHANGE MASTER TO` 并手动填入 B 的 binlog 文件名和位置。如果算错了位置，从库就会漏数据或者报错。

**GTID 方式**：直接把 C 的复制目标切换到 B，用 `MASTER_AUTO_POSITION=1`，MySQL 自动处理一切。

```sql
-- 从库 C 切换到新主库 B（GTID 方式）
STOP SLAVE;
CHANGE MASTER TO
  MASTER_HOST='192.168.1.101',       -- 新主库 B 的 IP
  MASTER_USER='repl_user',
  MASTER_PASSWORD='ReplPass123!',
  MASTER_AUTO_POSITION=1;
START SLAVE;
```

## 七、读写分离：让从库分担查询压力

主从复制搭建好之后，主库处理写操作，从库是主库的实时副本。如果不利用从库，那复制就只起到了备份的作用。读写分离的核心思想是：**写操作走主库，读操作走从库**，从而把读压力分散到多台从库上。

### 读写分离的架构

```
                    应用程序
                       │
              ┌────────┴────────┐
              │   写操作（INSERT │
              │   UPDATE/DELETE）│
              └────────┬────────┘
                       ▼
                   主库 (Master)
                  读 + 写
                       │
                 binlog 同步
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
        从库1        从库2        从库3
        只读         只读         只读
          ▲            ▲            ▲
          └────────────┼────────────┘
              ┌────────┴────────┐
              │   读操作（SELECT）│
              └─────────────────┘
```

### 实现方式一：应用层分离

最简单的方式是在应用代码中根据 SQL 类型选择不同的数据源：

```javascript
// Node.js 应用层读写分离示例
const masterPool = createPool({ host: '192.168.1.100', user: 'app_user', ... });
const slavePool  = createPool({ host: '192.168.1.101', user: 'app_user', ... });

function query(sql, params) {
  // 写操作走主库
  if (/^(INSERT|UPDATE|DELETE|CREATE|ALTER|DROP)/i.test(sql)) {
    return masterPool.execute(sql, params);
  }
  // 读操作走从库
  return slavePool.execute(sql, params);
}

// 对于"刚写入后立刻读取"的场景（比如下单后查看订单详情），
// 必须走主库，否则可能读到从库的旧数据
queryWithMaster(sql, params) {
  return masterPool.execute(sql, params);
}
```

应用层分离的优点是简单直接，不需要额外部署中间件。缺点是：

- 与业务代码耦合，每个需要数据库的项目都要处理
- 无法动态感知从库的健康状态和延迟
- 多个从库的负载均衡需要自己实现

### 实现方式二：中间件（ProxySQL）

ProxySQL 是目前最流行的 MySQL 读写分离中间件，它作为应用和 MySQL 之间的代理，自动将写请求路由到主库、读请求路由到从库。

```
应用程序 → ProxySQL → 主库（写）
                    → 从库1（读）
                    → 从库2（读）
```

ProxySQL 的核心配置思路：

```sql
-- 1. 在 ProxySQL 中注册主库和从库
INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight)
VALUES
  (10, '192.168.1.100', 3306, 1),   -- hostgroup 10 = 写组（主库）
  (20, '192.168.1.101', 3306, 1),   -- hostgroup 20 = 读组（从库1）
  (20, '192.168.1.102', 3306, 1);   -- hostgroup 20 = 读组（从库2）

-- 2. 配置路由规则：写操作走 hostgroup 10，读操作走 hostgroup 20
INSERT INTO mysql_query_rules (rule_id, active, match_digest, destination_hostgroup, apply)
VALUES
  (1, 1, '^SELECT.*FOR UPDATE', 10, 1),  -- SELECT FOR UPDATE 是写操作，走主库
  (2, 1, '^SELECT', 20, 1);               -- 其他 SELECT 走从库
-- 不匹配任何规则的默认走主库（写操作）

-- 3. 加载配置使其生效
LOAD MYSQL SERVERS TO RUNTIME;
LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL SERVERS TO DISK;
SAVE MYSQL QUERY RULES TO DISK;
```

ProxySQL 还能自动监控从库的复制延迟，如果某个从库延迟过大，自动将其从读组中剔除，等延迟恢复后再加回来。这是应用层分离做不到的。

### 读写分离的注意事项

读写分离引入了一个新问题：**主从延迟导致的数据不一致**。主库写入后，数据通过异步复制同步到从库，中间有几毫秒到几秒的延迟。如果用户刚下单就查询订单详情，而请求被路由到了从库，就会看到"订单不存在"。

解决方法：

1. 关键读操作（刚写入后立即读取）强制走主库
2. 在 ProxySQL 中配置延迟阈值，延迟超过阈值的从库不接收读请求
3. 对于一致性要求极高的场景，考虑半同步复制（主库至少等一个从库确认收到 binlog 才算提交成功）

## 八、高可用方案：主库挂了怎么办

读写分离解决了读压力的问题，但主库仍然是单点——主库挂了，写操作就全部中断。高可用方案要解决的就是：**主库故障时自动切换，业务几乎无感知**。

### 半同步复制：在性能和安全之间找平衡

前面提到默认的异步复制可能丢数据。半同步复制要求主库在提交事务时，至少等待一个从库确认收到了对应的 binlog 事件（注意是收到，不是执行完毕）。

```sql
-- 主库上安装半同步复制插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';

-- 从库上安装半同步复制插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 主库开启半同步
SET GLOBAL rpl_semi_sync_master_enabled = ON;
SET GLOBAL rpl_semi_sync_master_timeout = 1000;  -- 等待超时 1 秒，超时降级为异步

-- 从库开启半同步
SET GLOBAL rpl_semi_sync_slave_enabled = ON;
```

半同步复制的权衡：正常情况下多了一轮网络往返（主库等从库确认），写延迟增加 1-2ms。但如果所有从库都不可用，超过 `timeout` 时间后会自动降级为异步复制，不会完全阻塞写入。对于电商系统，这个权衡是值得的。

### MHA（Master High Availability）

MHA 是业界最成熟的主从切换工具之一。它的核心工作流程：

1. 监控主库状态（通过心跳检测）
2. 主库故障时，从所有从库中选出一个数据最新的作为新主库
3. 将其他从库的复制目标切换到新主库
4. 处理主库故障期间可能丢失的 binlog
5. 通知应用层切换数据库连接

MHA 的架构中有一个 Manager 节点和多个 MySQL 节点。Manager 负责监控和切换决策，通常部署在一台独立的服务器上。

```
           MHA Manager（监控 + 切换决策）
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
  主库       从库1       从库2
 (Master)   (Slave)     (Slave)
```

MHA 切换通常在 10-30 秒内完成，期间写操作会短暂中断。对于大多数电商场景，这个中断时间是可以接受的。

### InnoDB Cluster（MySQL 官方方案）

如果你使用 MySQL 8.0，还可以选择 MySQL 官方的 InnoDB Cluster 方案。它基于 Group Replication（组复制），多个节点组成一个集群，任何一个节点都可以处理读写请求。

```
       MySQL Router（自动路由）
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
  节点1     节点2     节点3
  读写      读写      读写
    └─────────┼─────────┘
         Group Replication
         （Paxos 协议共识）
```

InnoDB Cluster 的优势：

- 多主模式：所有节点都可以写入（也可以配置为单主模式）
- 自动故障检测和切换：某个节点挂了，集群自动剔除
- 强一致性：基于 Paxos 协议保证数据一致，不会丢数据

InnoDB Cluster 的代价：

- 至少需要 3 个节点（Paxos 协议要求多数派）
- 写入性能不如单机（每次写入需要集群内达成共识）
- 架构更复杂，排查问题难度更大

### 方案选型建议

| 场景                       | 推荐方案          | 理由                                       |
| -------------------------- | ----------------- | ------------------------------------------ |
| 中小型电商，预算有限       | 主从 + MHA        | 成熟稳定，运维简单                         |
| 大型电商，数据一致性要求高 | InnoDB Cluster    | 官方方案，自动化程度高                     |
| 云上环境                   | 云厂商 RDS 高可用 | 阿里云、AWS RDS 自带主从切换，无需自己维护 |

## 九、完整方案：电商系统的数据安全保障

把前面所有内容整合起来，电商系统 `shop` 数据库的完整安全保障方案如下：

```
日常运行
├── 主库（192.168.1.100）：处理写操作
├── 从库1（192.168.1.101）：处理读查询
├── 从库2（192.168.1.102）：处理读查询 + 备份专用
└── ProxySQL（192.168.1.200）：读写分离代理

数据保护
├── 每周日 02:00：全量逻辑备份（mysqldump → gzip → 本地 + 对象存储）
├── 实时：binlog 记录（ROW 格式，保留 7 天）
├── 异地：每日同步备份文件到异地机房
└── 每月：在测试环境做一次恢复演练

高可用
├── 半同步复制：防止主库故障丢数据
├── MHA 监控：主库自动切换
└── 告警：复制延迟超过 5 秒、主库不可达时立即通知
```

## 权衡与选择

这一篇覆盖了大量内容，最后做几个关键决策的权衡总结。

**逻辑备份 vs 物理备份**：如果数据库在几十 GB 以内，逻辑备份（mysqldump）是更好的选择——它可读、可审查、跨平台，代价只是备份和恢复速度慢一些。TB 级别以上的数据才需要考虑 XtraBackup 这样的物理备份方案。

**binlog position vs GTID**：新项目没有理由不用 GTID。GTID 让故障切换和从库管理变得简单，代价只是需要确保所有 SQL 语句兼容 GTID 模式（`enforce-gtid-consistency` 会帮你检查）。唯一需要 GTID 的是主从复制场景，如果你只有单机没有从库，GTID 对你来说没有实际价值。

**异步 vs 半同步复制**：异步复制性能最好但可能丢数据，全同步复制最安全但性能最差。半同步复制是一个合理的折中——正常情况下只增加 1-2ms 的写延迟，但能保证数据至少到达一个从库。对于电商订单这种关键数据，这个代价是值得的。

**应用层 vs 中间件读写分离**：项目早期用应用层分离就够了，简单直接。当从库数量超过 2 个、或者需要自动剔除延迟从库时，再引入 ProxySQL。不要过早引入中间件——多一层代理就多一个故障点。

**自建高可用 vs 云数据库**：如果你的团队有专职 DBA，自建主从 + MHA 完全可控。如果没有，直接用云厂商的 RDS 高可用版本——他们帮你处理了复制、切换、备份、监控，你只需要关注业务逻辑。这不是技术能力的问题，而是资源分配的决策。
