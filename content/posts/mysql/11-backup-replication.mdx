## 下午三点的那条 SQL

周三下午三点，运维同事在生产环境的 `shop` 数据库上执行了一条 UPDATE，忘了写 WHERE。

```sql
-- 他想修改某个用户的 VIP 等级
UPDATE users SET vip_level = 5;
-- 42 万条记录全部被改成 vip_level = 5
```

客服群里五分钟内涌入了上百条消息——用户发现自己的会员等级全变了，积分体系乱套了。运营总监打电话过来问：能恢复吗？多快？

你打开电脑，脑子里快速过了一遍三个问题：

1. 最近一次全量备份是什么时候？——昨晚凌晨两点，`mysqldump` 自动跑的。
2. 从备份到现在，中间 13 个小时的数据怎么补？——binlog 开着，应该能通过回放日志补回来。
3. 恢复期间数据库能不能继续用？——这是单机，恢复要停库，不能。

前两个问题的答案让你松了口气，第三个问题让你开始认真考虑是不是该搭主从复制了。

上一篇我们聊了用户管理与安全，通过权限控制和账号体系防止"不该操作的人做了不该做的事"。但权限做得再好，也防不住有权限的人犯糊涂——备份才是最后一道防线。而当我们意识到"恢复期间不能服务"是个问题时，就自然需要主从复制了。

这一篇，我们用电商系统 `shop` 数据库的这次事故为线索，先解决"数据怎么恢复"，再解决"怎么避免单点故障"。

## 一、先选对工具：逻辑备份还是物理备份

动手恢复之前，先搞清楚 MySQL 有哪些备份方式。选错工具，恢复的时候就知道痛了。

MySQL 的备份按**备份内容**分为两种：

- **逻辑备份**：导出 SQL 语句或文本文件。代表工具是 `mysqldump`（MySQL 自带）和 `mydumper`（多线程导出）。
- **物理备份**：直接复制 InnoDB 的数据文件。代表工具是 Percona XtraBackup。

两者的核心区别：

| 维度     | 逻辑备份（mysqldump） | 物理备份（XtraBackup） |
| -------- | --------------------- | ---------------------- |
| 备份内容 | SQL 语句文本          | InnoDB 数据文件副本    |
| 备份速度 | 慢，需要生成 SQL      | 快，文件级别复制       |
| 恢复速度 | 慢，逐条执行 SQL      | 快，直接拷贝文件       |
| 可读性   | 文本，能打开看和编辑  | 二进制，不可读         |
| 跨平台   | 可以，SQL 是通用的    | 不可以，依赖文件格式   |
| 适用场景 | 几十 GB 以内的数据库  | TB 级别的大型数据库    |

对于我们的 `shop` 数据库（假设几十 GB），逻辑备份完全够用。而且它的可读性在出问题时特别有价值——你能打开备份文件确认数据是否正确，物理备份做不到这一点。

只有当数据量到了 TB 级别，逻辑备份的"慢"才真正成为问题，这时候才需要考虑 XtraBackup。

---

## 二、用 mysqldump 做全量备份和恢复

`mysqldump` 是 MySQL 自带的逻辑备份工具，不需要额外安装，几乎所有 MySQL 环境都能直接用。

### 最基本的备份命令

```bash
# 备份整个 shop 数据库
mysqldump -uroot -p shop > /backup/shop_backup.sql

# 备份所有数据库（用于灾备）
mysqldump -uroot -p --all-databases > /backup/all_databases.sql

# 只备份关键表（出问题时只恢复特定表）
mysqldump -uroot -p shop users orders > /backup/shop_critical_tables.sql
```

这几个命令能跑，但直接用在生产环境会踩坑。默认参数下 `mysqldump` 会对表加读锁，备份期间写入操作全部阻塞。对于 7x24 小时运行的电商系统，这是不可接受的。

### 生产环境的参数组合

```bash
mysqldump -uroot -p \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --quick \
  --set-gtid-purged=OFF \
  shop > /backup/shop_backup.sql
```

逐个解释这些参数为什么重要：

- `--single-transaction` 是最关键的一个。它利用 InnoDB 的 MVCC 机制，在备份开始时创建一致性快照，备份过程中不锁表，业务正常读写。**只有 InnoDB 引擎支持这个参数**，MyISAM 表还是会锁。
- `--routines` 导出存储过程和函数，`--triggers` 导出触发器，`--events` 导出定时事件。不加这些，备份恢复后你会发现存储过程丢了。
- `--quick` 让 `mysqldump` 逐行从服务器读取数据，而不是把整张表加载到内存。百万级大表不加这个参数，备份进程可能直接 OOM。
- `--set-gtid-purged=OFF` 在需要把备份导入到另一个 MySQL 实例（从库、测试环境）时必须加，否则会因为 GTID 冲突导入失败。

### 压缩备份节省空间

数据库备份文件通常很大，压缩可以节省 70%-90% 的存储空间。直接用管道压缩，不生成中间文件：

```bash
# 备份时直接压缩
mysqldump -uroot -p --single-transaction shop | gzip > /backup/shop_$(date +%Y%m%d).sql.gz

# 恢复时解压并导入
gunzip < /backup/shop_20260330.sql.gz | mysql -uroot -p shop
```

### 恢复数据

备份的价值在于恢复。但恢复前有几件事比"敲命令"更重要：

1. 先在测试环境验证备份文件的完整性，不要直接在生产库操作。
2. 确认备份文件的开头是 `MySQL dump 10.13` 之类的注释——如果文件是空的或者被截断了，越早发现越好。

```bash
# 恢复到测试库验证
mysql -uroot -p shop_test < /backup/shop_backup.sql

# 确认无误后恢复生产数据
mysql -uroot -p shop < /backup/shop_backup.sql
```

也可以在 MySQL 客户端内用 `SOURCE /backup/shop_backup.sql` 命令恢复，效果一样。

回到我们的场景：凌晨两点的全量备份恢复后，数据回到了昨晚的状态。但今天上午 9 点到下午 3 点之间产生的订单、用户注册、商品修改——这些数据还在吗？

这就是 binlog 登场的时候。

## 三、binlog：补上备份到误操作之间的数据

全量备份只能恢复到备份那个时刻。我们凌晨 2 点做的备份，误操作发生在下午 3 点，中间 13 个小时的数据需要靠 binlog 补回来。

### binlog 是什么

binlog（二进制日志）记录了所有对数据库数据进行修改的操作——INSERT、UPDATE、DELETE、DDL 语句都会被记录。SELECT 不会被记录，因为它不修改数据。

binlog 最初是为主从复制设计的，但它同时也是增量备份和 point-in-time 恢复的基础。**如果没开 binlog，备份到误操作之间的数据就真的丢了。**

### 确认 binlog 已开启

```ini
# /etc/mysql/my.cnf
[mysqld]
log-bin=mysql-bin
binlog_format=ROW
expire_logs_days=7
max_binlog_size=256M
sync_binlog=1
```

关于 `binlog_format`，有三种选择：

| 格式      | 记录内容                  | 优点                 | 缺点                                            |
| --------- | ------------------------- | -------------------- | ----------------------------------------------- |
| STATEMENT | SQL 语句原文              | 日志文件小           | 不确定性函数（NOW()、UUID()）可能导致主从不一致 |
| ROW       | 每行数据的变化            | 最安全，主从一定一致 | 批量操作时日志文件大                            |
| MIXED     | 自动选择 STATEMENT 或 ROW | 折中                 | 行为不可预测，排查问题时不知道用了哪种格式      |

生产环境推荐 **ROW** 格式。日志文件虽然大一些，但恢复的可靠性远比磁盘空间重要。而且 ROW 格式可以用 `mysqlbinlog -v` 精确看到每行数据的变化，排查问题时非常方便。

`sync_binlog=1` 表示每次事务提交都把 binlog 刷到磁盘。这是最安全的设置，确保主机断电不丢 binlog，但会牺牲一些写入性能。如果你能接受断电后丢失一秒以内的事务，可以设为 0 或更大的值。

### 查看 binlog 内容

```sql
-- 查看当前正在写入的 binlog 文件和位置
SHOW MASTER STATUS;
-- 结果示例：
-- File: mysql-bin.000003
-- Position: 154

-- 查看所有 binlog 文件列表
SHOW BINARY LOGS;

-- 查看最新 binlog 中的事件
SHOW BINLOG EVENTS IN 'mysql-bin.000003' LIMIT 10;
```

命令行工具 `mysqlbinlog` 可以看到更详细的内容：

```bash
# ROW 格式需要加 -v 才能看到具体的行变更
mysqlbinlog -v /var/lib/mysql/mysql-bin.000003
```

### 基于 binlog 的 point-in-time 恢复

这是整个备份体系的核心能力——恢复到误操作之前的那一刻。

我们的场景：全量备份是凌晨 2:00 做的，误操作发生在下午 3:30。恢复步骤：

```bash
# 第一步：恢复全量备份（回到凌晨 2:00 的状态）
gunzip < /backup/shop_20260330.sql.gz | mysql -uroot -p shop

# 第二步：回放 binlog，从备份时刻到误操作之前
# 用时间范围精确控制，只恢复到 15:29:59
mysqlbinlog --start-datetime="2026-03-30 02:00:00" \
            --stop-datetime="2026-03-30 15:29:59" \
            /var/lib/mysql/mysql-bin.000003 \
            /var/lib/mysql/mysql-bin.000004 \
            | mysql -uroot -p shop
```

也可以用 position 精确控制，比时间更准确（因为时间精度只到秒）：

```bash
# 先用 SHOW BINLOG EVENTS 找到误操作的起始 position
# 假设误操作从 position 4387 开始

mysqlbinlog --start-position=154 \
            --stop-position=4387 \
            /var/lib/mysql/mysql-bin.000003 | mysql -uroot -p shop
```

恢复完成后，42 万条 `users` 记录回到了误操作之前的状态，中间 13 个小时的订单、注册等数据也完整保留。

这个过程有一个关键前提：**binlog 必须在误操作之前就已经开启**。如果等到出事了才发现 binlog 没开，那中间的数据就只能从业务日志或其他数据源手动补了。

## 四、备份策略：自动化、定期、异地

手动敲 `mysqldump` 命令不是长久之计。你需要一套自动化的备份策略，确保"该备份的时候一定备份了，该保留的一定没被清理掉"。

### 备份节奏设计

对于 `shop` 数据库，推荐的方案是"全量 + binlog"：

```
周日 02:00    全量逻辑备份（mysqldump + gzip）
每天          binlog 实时记录（保留 7 天，配合全量可恢复到任意秒）
每周          全量备份同步到异地服务器
每月          在测试环境做一次恢复演练
```

为什么推荐"全量 + binlog"而不是"全量 + 增量"？因为 binlog 是 MySQL 原生支持的，不需要额外的增量备份工具。增量备份（比如 XtraBackup 的增量）恢复时要依次回放每个增量文件，而 binlog 可以直接用时间或 position 定位，更灵活。

### 自动备份脚本

把备份流程写成脚本，交给 cron 定时执行：

```bash
#!/bin/bash
# shop_backup.sh
# 加入 crontab：0 2 * * 0 /scripts/shop_backup.sh

set -euo pipefail

BACKUP_DIR="/data/backup/mysql"
MYSQL_USER="backup_user"           # 用上一篇创建的专用备份账号
MYSQL_PASSWORD="BackupPass123!"    # 生产环境建议用 mysql_config_editor 存密码
MYSQL_HOST="192.168.1.100"
DB_NAME="shop"
RETENTION_DAYS=30

mkdir -p "$BACKUP_DIR"
DATE_TAG=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/${DB_NAME}_${DATE_TAG}.sql.gz"

# 执行全量备份
mysqldump -h"$MYSQL_HOST" -u"$MYSQL_USER" -p"$MYSQL_PASSWORD" \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --quick \
  --set-gtid-purged=OFF \
  "$DB_NAME" | gzip > "$BACKUP_FILE"

# 校验备份文件
if [ $? -eq 0 ] && [ -s "$BACKUP_FILE" ]; then
  FILE_SIZE=$(du -h "$BACKUP_FILE" | cut -f1)
  echo "[$(date)] 备份成功: $BACKUP_FILE ($FILE_SIZE)" >> "$BACKUP_DIR/backup.log"
else
  echo "[$(date)] 备份失败！" | tee -a "$BACKUP_DIR/error.log"
  exit 1
fi

# 清理过期备份
DELETED=$(find "$BACKUP_DIR" -name "*.sql.gz" -mtime +$RETENTION_DAYS -delete -print | wc -l)
echo "[$(date)] 清理了 $DELETED 个过期备份文件" >> "$BACKUP_DIR/backup.log"

# 异地同步（取消注释启用）
# rsync -az "$BACKUP_DIR/" user@remote-server:/data/backup/mysql/
```

注意脚本里用了上一篇创建的 `backup_user` 账号——它只有 `SELECT` 和 `LOCK TABLES` 权限，即使脚本泄漏了，攻击者也无法通过这个账号修改数据。

### 3-2-1 备份原则

业界广泛认可的备份安全标准：

- **3** 份数据副本：1 份生产数据 + 2 份备份
- **2** 种不同介质：本地磁盘 + 对象存储或异地服务器
- **1** 份异地备份：防止单机房故障导致数据全部丢失

对于电商系统：

1. 本地磁盘保留 30 天的全量备份，恢复速度快
2. 对象存储（阿里云 OSS 或 AWS S3）保留 180 天，长期归档
3. 异地机房同步最近 7 天的备份，用于容灾

### 备份文件要加密

备份文件包含完整的业务数据，传输和存储时必须加密。可以在管道中用 `openssl enc -e -aes-256-cbc` 加密，恢复时用 `openssl enc -d` 解密后导入。密钥不要硬编码在脚本里，用环境变量或密钥管理服务存储。

## 五、主从复制：不再依赖单台服务器

备份解决了"数据不丢"的问题，但回到我们场景中的第三个问题：恢复期间数据库不能提供服务。全量恢复几十 GB 的数据可能需要几十分钟甚至几个小时，这段时间电商系统完全无法下单。

主从复制就是把数据实时同步到另一台服务器。主库挂了从库顶上，业务几乎无感知。即使不挂，从库也能分担读压力。

### 复制的核心流程

主从复制围绕 binlog 展开，整个过程由三个组件完成：

```
主库 (Master)
  │
  │  1. 应用写入数据
  │  2. 变更记录到 binlog
  │
  ▼ binlog 传输
从库 IO 线程
  │
  │  3. 连接主库，拉取 binlog
  │  4. 写入本地 relay log（中继日志）
  │
  ▼ relay log 回放
从库 SQL 线程
  │
  │  5. 读取 relay log
  │  6. 重放 SQL，写入从库数据
  │
  ▼
从库数据与主库一致
```

理解这个流程的几个关键点：

- **IO 线程**只负责搬运 binlog，不执行 SQL，所以它很轻量。它断开后可以自动重连，从上次断开的位置继续拉取。
- **SQL 线程**负责回放 SQL，这是真正干活的。如果主库写入压力大，SQL 线程可能跟不上，这就是**复制延迟**——从库的数据比主库旧。
- 复制是**异步**的：主库写入成功就返回给客户端了，不等从库同步完成。这意味着主库突然宕机时，从库可能还没收到最新的 binlog，这部分数据就丢了。

### 主库配置

```ini
# /etc/mysql/conf.d/master.cnf
[mysqld]
server-id=1                  # 集群内唯一 ID，不能和从库重复
log-bin=mysql-bin            # 开启 binlog
binlog_format=ROW
expire_logs_days=7
sync_binlog=1
```

在主库上创建复制专用账号（遵循上一篇的最小权限原则）：

```sql
-- 只授予复制所需权限，不给其他权限
CREATE USER 'repl_user'@'192.168.1.%' IDENTIFIED BY 'ReplPass123!';
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'192.168.1.%';
FLUSH PRIVILEGES;

-- 记录当前 binlog 位置（配置从库时需要）
SHOW MASTER STATUS;
-- File: mysql-bin.000001, Position: 154
```

### 从库配置与启动复制

```ini
# /etc/mysql/conf.d/slave.cnf
[mysqld]
server-id=2                  # 与主库不同
relay-log=relay-bin          # 中继日志前缀
read_only=ON                 # 从库只读，防止误写入
log-bin=mysql-bin            # 从库也开 binlog（方便级联复制和故障切换）
binlog_format=ROW
```

`read_only=ON` 很重要。它防止业务代码或运维人员直接在从库写入数据。从库写入数据会导致 SQL 线程报错、复制中断——排查起来很麻烦。

在从库上启动复制：

```sql
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_USER='repl_user',
  MASTER_PASSWORD='ReplPass123!',
  MASTER_LOG_FILE='mysql-bin.000001',
  MASTER_LOG_POS=154;

START SLAVE;

-- 检查复制状态（出问题时第一个执行的命令）
SHOW SLAVE STATUS\G
```

`SHOW SLAVE STATUS` 输出很长，重点关注这几个字段：

```
Slave_IO_Running: Yes         -- IO 线程正常（连接主库、拉取 binlog）
Slave_SQL_Running: Yes        -- SQL 线程正常（回放 relay log）
Seconds_Behind_Master: 0      -- 复制延迟秒数，0 表示完全同步
Last_Error:                   -- 最后一次错误（正常为空）
```

如果 `Slave_IO_Running` 是 No，通常是网络不通或复制账号权限问题。如果 `Slave_SQL_Running` 是 No，通常是有人直接在从库写了数据导致 SQL 冲突。

## 六、GTID 复制：不再手动算 binlog 位置

前面配置从库时，我们手动指定了 `MASTER_LOG_FILE` 和 `MASTER_LOG_POS`。这叫**基于 binlog position 的复制**。

这种方式的痛点在于故障切换。假设主库 A 挂了，需要把从库 B 提升为新主库。B 的 binlog 位置和 A 不同，你需要先在 B 上执行 `SHOW MASTER STATUS` 获取新位置，然后在其他从库 C 上手动填入——算错了就会漏数据或者报错。

**GTID（Global Transaction Identifier）** 解决了这个问题。每个事务在提交时会被分配一个全局唯一的 ID，格式为 `server_uuid:transaction_id`（比如 `3E11FA47-71CA-11E1-9E33-C80AA9429562:23`）。配置复制时不需要指定 binlog 位置，MySQL 会根据 GTID 自动找到同步起点。

### GTID 配置

主库和从库都需要添加：

```ini
[mysqld]
gtid-mode=ON
enforce-gtid-consistency=ON   # 强制 GTID 一致性
```

`enforce-gtid-consistency=ON` 会拒绝执行不支持 GTID 的语句，比如 `CREATE TABLE ... SELECT`（这种语句在 binlog 中会被拆成两个事务，无法用单一 GTID 标识）。遇到这个报错时，把语句拆成先 `CREATE TABLE` 再 `INSERT ... SELECT` 就行。

### 用 GTID 启动复制

配置从库时不再需要指定 binlog 文件和位置：

```sql
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_USER='repl_user',
  MASTER_PASSWORD='ReplPass123!',
  MASTER_AUTO_POSITION=1;      -- 自动根据 GTID 找同步起点

START SLAVE;
```

`MASTER_AUTO_POSITION=1` 是 GTID 复制的核心。从库告诉主库"我已经有哪些 GTID"，主库自动把缺失的事务发过来。

故障切换时的对比更明显。假设主库 A 挂了，从库 B 提升为新主库，从库 C 需要切换到 B：

```sql
-- GTID 方式：不需要查 B 的 binlog 位置
STOP SLAVE;
CHANGE MASTER TO
  MASTER_HOST='192.168.1.101',    -- 新主库 B 的 IP
  MASTER_USER='repl_user',
  MASTER_PASSWORD='ReplPass123!',
  MASTER_AUTO_POSITION=1;
START SLAVE;
```

新项目建议直接用 GTID，省去手动管理 binlog 位置的麻烦。唯一的前提是你的 SQL 语句必须兼容 GTID 模式，`enforce-gtid-consistency` 会帮你检查。

## 七、读写分离：让从库真正发挥作用

主从复制搭建好之后，如果不利用从库，那它只是个热备。读写分离的核心思想：**写操作走主库，读操作走从库**，把读压力分散到多台机器上。

电商系统的读写比通常在 8:1 到 10:1 之间——用户浏览商品、搜索、看订单详情都是读，只有下单、支付、修改信息是写。这意味着一个主库加两三个从库，就能把读压力降到原来的四分之一。

### 架构图

```
应用程序
   │
   ├── 写操作（INSERT/UPDATE/DELETE）
   │
   ▼
主库 (Master)  ── 读写
   │
   │ binlog 同步
   │
   ├── 从库1（只读）──┐
   ├── 从库2（只读）──┤  读操作（SELECT）
   └── 从库3（只读）──┘
```

### 实现方式一：应用层路由

最简单的方式是在应用代码中根据 SQL 类型选择数据源：

```javascript
const masterPool = createPool({ host: '192.168.1.100', ... });
const slavePool  = createPool({ host: '192.168.1.101', ... });

function query(sql, params) {
  if (/^(INSERT|UPDATE|DELETE|CREATE|ALTER|DROP)/i.test(sql)) {
    return masterPool.execute(sql, params);
  }
  return slavePool.execute(sql, params);
}
```

应用层分离简单直接，不需要额外部署中间件。但有个容易忽略的问题：**刚写入后立刻读取**。用户下单后刷新页面看订单详情，如果这个 SELECT 走了从库，而从库还没同步到最新数据，用户就会看到"订单不存在"。

解决方法是提供一个 `queryWithMaster` 方法，关键场景强制走主库：

```javascript
// 下单后查看订单、支付后查看支付结果——必须走主库
function queryWithMaster(sql, params) {
  return masterPool.execute(sql, params);
}
```

应用层分离的缺点：每个项目都要处理路由逻辑，无法动态感知从库的健康状态和延迟，多个从库的负载均衡需要自己实现。

### 实现方式二：中间件（ProxySQL）

当从库数量超过两个、或者需要自动剔除延迟从库时，引入 ProxySQL 这样的中间件更合适。它作为应用和 MySQL 之间的代理，自动将写请求路由到主库、读请求路由到从库：

```
应用程序 → ProxySQL（代理层）→ 主库（写）
                             → 从库1（读）
                             → 从库2（读）
```

ProxySQL 的核心配置思路是定义"写组"（hostgroup 10，指向主库）和"读组"（hostgroup 20，指向从库），然后用路由规则把 SQL 分发到对应组。比如 `^SELECT.*FOR UPDATE` 走写组，普通 `^SELECT` 走读组。配置完成后用 `LOAD ... TO RUNTIME` 和 `SAVE ... TO DISK` 使其生效并持久化

ProxySQL 能自动监控从库的复制延迟。如果某个从库延迟超过阈值，自动将其从读组中剔除，等延迟恢复后再加回来。这是应用层分离做不到的。

### 读写分离的数据一致性代价

读写分离引入了一个不可避免的代价：**从库数据比主库旧**。异步复制意味着从库的延迟通常是毫秒级到秒级，但对用户来说，"刚提交的数据查不到"就是 bug。

应对方法：

1. 关键读操作（下单后看订单、支付后看结果）强制走主库
2. 在 ProxySQL 中配置延迟阈值，延迟过大的从库不接收读请求
3. 一致性要求极高的场景，考虑半同步复制

## 八、高可用：主库挂了怎么办

读写分离解决了读压力，但主库仍然是单点——主库挂了，所有写操作中断。高可用方案要解决的就是主库故障时的自动切换。

### 半同步复制：减少数据丢失风险

默认的异步复制可能丢数据——主库写入成功就返回了，还没来得及发 binlog 给从库就宕机了。

半同步复制要求主库提交事务时，至少等一个从库确认**收到**了对应的 binlog 事件（收到不是执行完毕）：

```sql
-- 主库安装半同步插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = ON;
SET GLOBAL rpl_semi_sync_master_timeout = 1000;  -- 超时 1 秒后降级为异步

-- 从库安装半同步插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
SET GLOBAL rpl_semi_sync_slave_enabled = ON;
```

半同步复制的权衡：正常情况下写延迟增加 1-2ms（多了一轮网络往返）。如果所有从库都不可用，超过 `timeout` 时间后会自动降级为异步复制，不会完全阻塞写入。对于电商订单这种关键数据，这个代价是值得的。

### MHA：自动故障切换

MHA（Master High Availability）是业界最成熟的主从切换工具之一。它的核心工作流程：

1. 监控主库状态（心跳检测）
2. 主库故障时，从所有从库中选出数据最新的作为新主库
3. 将其他从库的复制目标切换到新主库
4. 尝试从故障主库保存未发送的 binlog
5. 通知应用层切换数据库连接

```
MHA Manager（监控 + 切换决策）
     │
     ├── 主库 (Master)
     ├── 从库1 (Slave)
     └── 从库2 (Slave)
```

MHA 切换通常在 10-30 秒内完成，期间写操作会短暂中断。对于大多数电商场景，这个中断时间可以接受。

### InnoDB Cluster：MySQL 官方方案

MySQL 8.0 提供的官方高可用方案，基于 Group Replication（组复制），多个节点组成集群：

```
MySQL Router（自动路由）
     │
     ├── 节点1（读写）
     ├── 节点2（读写）
     └── 节点3（读写）
         └── Group Replication（Paxos 协议共识）
```

InnoDB Cluster 的优势是强一致性（基于 Paxos 协议），不会丢数据；自动故障检测和切换；支持多主写入。代价是至少需要 3 个节点，写入性能不如单机（每次写入需要集群内达成共识），架构更复杂。

### 方案选型

| 场景                   | 推荐方案          | 理由                         |
| ---------------------- | ----------------- | ---------------------------- |
| 中小型电商，预算有限   | 主从 + MHA        | 成熟稳定，运维简单           |
| 大型电商，一致性要求高 | InnoDB Cluster    | 官方方案，自动化程度高       |
| 云上环境               | 云厂商 RDS 高可用 | 阿里云、AWS RDS 自带主从切换 |

## 权衡与选择

这一篇覆盖了大量内容，最后梳理几个关键决策的取舍。

**逻辑备份还是物理备份。** 几十 GB 以内的数据库，逻辑备份（mysqldump）是更好的选择——可读、可审查、跨平台，代价只是速度慢一些。TB 级别以上才需要 XtraBackup。很多人过早引入物理备份方案，实际上是在用复杂度换一个还没遇到的问题。

**binlog position 还是 GTID。** 新项目没有理由不用 GTID。它让故障切换和从库管理变得简单，代价只是需要确保 SQL 兼容 GTID 模式。唯一不需要 GTID 的场景是单机没有从库——但如果你在考虑备份和主从复制，说明你迟早会有从库。

**异步复制还是半同步复制。** 异步性能最好但可能丢数据，全同步最安全但性能最差。半同步是合理的折中——正常情况只增加 1-2ms 写延迟，但保证数据至少到达一个从库。电商订单这种数据值得这个代价，但用户行为日志可能不值得。

**应用层分离还是中间件。** 项目早期用应用层分离够了，简单直接。当从库超过 2 个、需要自动剔除延迟从库时再引入 ProxySQL。不要过早引入中间件——多一层代理就多一个故障点，多一份运维成本。

**自建高可用还是云数据库。** 有专职 DBA 的团队，自建主从 + MHA 完全可控。没有的话，直接用云厂商的 RDS 高可用版本——他们帮你处理了复制、切换、备份、监控，你只需要关注业务。这不是技术能力的问题，而是资源投入的决策。

到目前为止，我们已经能保证数据不丢（备份）、服务不断（主从复制和高可用）。下一篇要解决的是另一个层面的问题——当电商大促到来，QPS 从 100 涨到 10000 时，怎么让数据库扛住压力。那篇会把整个系列讲过的索引、查询优化、配置调优综合起来，打一场性能攻坚战。
