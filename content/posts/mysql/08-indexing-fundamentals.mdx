## 前言

索引是数据库性能优化的核心。一个恰当的索引可以让查询速度提升成百上千倍。

本文将深入讲解索引的原理、类型和使用方法，帮助你建立索引设计的基础知识。

## 什么是索引

索引是对数据库表中一列或多列的值进行排序的一种结构，类似书的目录。

```
没有索引：逐页查找内容（全表扫描）
有索引：通过目录快速定位

类比：
- 书的目录 → 索引
- 页码 → 数据地址
- 内容 → 表数据
```

## 为什么需要索引

### 性能对比

```sql
-- 创建测试表（1000万条数据）
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(50),
  email VARCHAR(100),
  age TINYINT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 插入1000万条数据...
```

**没有索引**：

```sql
SELECT * FROM users WHERE email = 'user@example.com';
-- 耗时：~5秒（全表扫描）
-- 扫描行数：10000000
```

**有索引**：

```sql
CREATE INDEX idx_email ON users(email);
SELECT * FROM users WHERE email = 'user@example.com';
-- 耗时：~0.001秒
-- 扫描行数：1
```

**性能提升：5000倍！**

## 索引类型

### 1. 主键索引（PRIMARY KEY）

```sql
-- 主键自动创建唯一索引
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,  -- 主键索引
  username VARCHAR(50)
);

-- 添加主键
ALTER TABLE users ADD PRIMARY KEY (id);
```

### 2. 唯一索引（UNIQUE）

```sql
-- 创建表时定义
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  username VARCHAR(50) UNIQUE,  -- 唯一索引
  email VARCHAR(100),
  UNIQUE KEY uk_email (email)   -- 另一种语法
);

-- 添加唯一索引
CREATE UNIQUE INDEX idx_username ON users(username);
ALTER TABLE users ADD UNIQUE INDEX idx_email (email);

-- 特点：
-- - 保证数据唯一性
-- - 查询效率高
-- - 可以有多个 NULL（除 B+Tree 外）
```

### 3. 普通索引（INDEX）

```sql
-- 创建表时定义
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  username VARCHAR(50),
  age INT,
  INDEX idx_age (age)
);

-- 添加普通索引
CREATE INDEX idx_age ON users(age);
ALTER TABLE users ADD INDEX idx_age (age);

-- 特点：
-- - 无唯一性约束
-- - 可以重复
-- - 用于加速查询
```

### 4. 全文索引（FULLTEXT）

```sql
-- 创建表时定义
CREATE TABLE articles (
  id BIGINT PRIMARY KEY,
  title VARCHAR(200),
  content TEXT,
  FULLTEXT INDEX ft_title_content (title, content)
) ENGINE=InnoDB;

-- 添加全文索引
CREATE FULLTEXT INDEX ft_content ON articles(content);

-- 使用全文搜索
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('MySQL 教程' IN NATURAL LANGUAGE MODE);

-- 特点：
-- - 用于文本搜索
-- - 支持 MATCH AGAINST
-- - 只有 InnoDB 和 MyISAM 支持
```

### 5. 空间索引（SPATIAL）

```sql
-- 用于地理空间数据
CREATE TABLE locations (
  id BIGINT PRIMARY KEY,
  name VARCHAR(100),
  coordinates POINT NOT NULL,
  SPATIAL INDEX idx_coordinates (coordinates)
);

-- 特点：
-- - 用于地理空间数据
-- - 支持 GEOMETRY 类型
```

## B+Tree 索引原理

MySQL InnoDB 默认使用 B+Tree 索引结构。

### B+Tree 特点

```
              [根节点]
             /    |    \
        [分支] [分支] [分支]
         /  \    /  \    /  \
     [叶子][叶子][叶子][叶子][叶子][叶子]
      ↓
    [数据]

特点：
1. 所有数据都在叶子节点
2. 叶子节点形成链表（支持范围查询）
3. 查询时间复杂度：O(log n)
4. 层高通常3-4层（1000万数据只需3-4次I/O）
```

### 为什么是 B+Tree

| 结构   | 查询     | 范围查询 | 磁盘I/O | MySQL支持  |
| ------ | -------- | -------- | ------- | ---------- |
| Hash   | O(1)     | ❌       | 少      | MEMORY引擎 |
| B+Tree | O(log n) | ✅       | 少      | ✅         |
| 二叉树 | O(log n) | ✅       | 多      | ❌         |

## 索引管理

### 创建索引

```sql
-- 方式1：CREATE INDEX
CREATE INDEX idx_age ON users(age);
CREATE UNIQUE INDEX idx_email ON users(email);
CREATE FULLTEXT INDEX ft_content ON articles(content);

-- 方式2：ALTER TABLE
ALTER TABLE users ADD INDEX idx_age (age);
ALTER TABLE users ADD UNIQUE INDEX idx_email (email);
ALTER TABLE users ADD FULLTEXT INDEX ft_content (content);

-- 复合索引
CREATE INDEX idx_age_status ON users(age, status);

-- 方式3：创建表时定义
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  username VARCHAR(50),
  age INT,
  INDEX idx_age (age)
);
```

### 查看索引

```sql
-- 查看表的所有索引
SHOW INDEX FROM users;

-- 查看索引信息
SELECT
  INDEX_NAME,
  COLUMN_NAME,
  SEQ_IN_INDEX,
  INDEX_TYPE
FROM information_schema.STATISTICS
WHERE TABLE_SCHEMA = 'shop' AND TABLE_NAME = 'users';

-- 查看表的创建语句（包含索引）
SHOW CREATE TABLE users;
```

### 删除索引

```sql
-- 方式1：DROP INDEX
DROP INDEX idx_age ON users;

-- 方式2：ALTER TABLE
ALTER TABLE users DROP INDEX idx_age;

-- 删除主键
ALTER TABLE users DROP PRIMARY KEY;
```

## 索引设计原则

### 1. 选择合适的字段

```sql
-- ✅ 适合建索引的字段：
-- - WHERE 条件字段
CREATE INDEX idx_email ON users(email);

-- - JOIN 关联字段
CREATE INDEX idx_user_id ON orders(user_id);

-- - ORDER BY 排序字段
CREATE INDEX idx_created_at ON orders(created_at);

-- - GROUP BY 分组字段
CREATE INDEX idx_status ON orders(status);

-- ❌ 不适合建索引的字段：
-- - 频繁更新的字段
-- - 区分度低的字段（gender、status）
-- - 很少查询的字段
-- - TEXT/BLOB 大字段
```

### 2. 最左前缀原则（复合索引）

```sql
-- 复合索引：(a, b, c)
CREATE INDEX idx_abc ON table(a, b, c);

-- ✅ 可以使用索引：
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3
WHERE a = 1 AND b IN (2, 3) AND c = 4

-- ❌ 不能使用索引：
WHERE b = 2                    -- a 缺失
WHERE c = 3                    -- a、b 缺失
WHERE b = 2 AND c = 3          -- a 缺失

-- 原理：B+Tree 按索引字段顺序构建
```

### 3. 区分度高的字段优先

```sql
-- ✅ 区分度高：适合建索引
SELECT COUNT(DISTINCT email) / COUNT(*) FROM users;
-- 结果：接近 1.0 → 区分度高

-- ❌ 区分度低：不适合建索引
SELECT COUNT(DISTINCT status) / COUNT(*) FROM users;
-- 结果：0.01 → 区分度低（只有2-3个值）

-- 示例：email 适合，gender 不适合
CREATE INDEX idx_email ON users(email);       -- ✅ 好
CREATE INDEX idx_gender ON users(gender);     -- ❌ 差
```

### 4. 索引不是越多越好

```sql
-- ⚠️ 索引的代价：
-- 1. 占用磁盘空间
-- 2. 降低 INSERT/UPDATE/DELETE 性能
-- 3. 维护成本（索引也需要更新）

-- 建议：
-- - 单表不超过 5 个索引
-- - 复合索引字段不超过 5 个
-- - 根据实际查询需求创建
```

## EXPLAIN 分析索引

使用 EXPLAIN 查看查询是否使用索引。

```sql
-- 基本用法
EXPLAIN SELECT * FROM users WHERE email = 'user@example.com';

-- 关键列：
-- type：访问类型（ALL、index、range、ref、eq_ref、const）
-- possible_keys：可能使用的索引
-- key：实际使用的索引
-- rows：预计扫描的行数
-- Extra：额外信息

-- type 类型（从差到好）：
-- ALL     - 全表扫描（最差）
-- index   - 索引扫描
-- range   - 范围扫描（>、<、BETWEEN、IN）
-- ref     - 索引查找
-- eq_ref  - 唯一索引查找
-- const   - 主键或唯一索引查找（最好）

-- 示例：
EXPLAIN SELECT * FROM users WHERE id = 1;
-- type: const（最优）

EXPLAIN SELECT * FROM users WHERE email = 'user@example.com';
-- type: ref（使用了索引）

EXPLAIN SELECT * FROM users WHERE age > 25;
-- type: range（范围查询）

EXPLAIN SELECT * FROM users WHERE username LIKE '%user%';
-- type: ALL（全表扫描，索引失效）
```

## 实战案例

### 为订单表创建索引

```sql
-- 订单表
CREATE TABLE orders (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT UNSIGNED NOT NULL,
  order_no VARCHAR(50) NOT NULL UNIQUE,
  total_amount DECIMAL(10, 2) NOT NULL,
  status VARCHAR(20) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;

-- 索引设计：
-- 1. user_id：经常关联查询
CREATE INDEX idx_user_id ON orders(user_id);

-- 2. (status, created_at)：复合索引，支持常用查询
CREATE INDEX idx_status_created ON orders(status, created_at);

-- 3. 查询验证
EXPLAIN SELECT * FROM orders
WHERE user_id = 1 AND status = 'completed'
ORDER BY created_at DESC;

-- 结果：应该使用 idx_status_created 或 idx_user_id
```

## 最佳实践

### 1. 为 WHERE 字段建索引

```sql
-- ✅ 推荐
CREATE INDEX idx_email ON users(email);

-- ✅ 推荐
CREATE INDEX idx_status_date ON orders(status, created_at);
```

### 2. 覆盖索引优化

```sql
-- 创建覆盖索引
CREATE INDEX idx_status_created_total ON orders(status, created_at, total_amount);

-- 查询只访问索引，不回表
SELECT status, created_at, total_amount
FROM orders
WHERE status = 'completed'
ORDER BY created_at DESC;
```

### 3. 避免 SELECT \*

```sql
-- ❌ 避免：无法使用覆盖索引
SELECT * FROM orders WHERE status = 'completed';

-- ✅ 推荐：只查询需要的字段
SELECT id, order_no, total_amount
FROM orders
WHERE status = 'completed';
```

## 常见问题

### Q1: 为什么查询没有使用索引？

**A:** 检查索引失效场景。

```sql
-- 索引失效场景：
-- 1. 使用函数
WHERE LOWER(email) = 'user@example.com'  -- ❌
WHERE email = 'user@example.com'         -- ✅

-- 2. 类型转换
WHERE phone = 13800138000               -- ❌（phone是字符串）
WHERE phone = '13800138000'             -- ✅

-- 3. LIKE 前缀通配符
WHERE email LIKE '%@example.com'        -- ❌
WHERE email LIKE 'user%@example.com'    -- ✅

-- 4. OR 条件（部分情况）
WHERE id = 1 OR age = 25                -- ❌
```

### Q2: 如何查看索引使用情况？

**A:** 使用 EXPLAIN 和系统表。

```sql
-- EXPLAIN
EXPLAIN SELECT * FROM users WHERE email = 'user@example.com';

-- 查看索引统计
SELECT
  TABLE_NAME,
  INDEX_NAME,
  SEQ_IN_INDEX,
  COLUMN_NAME
FROM information_schema.STATISTICS
WHERE TABLE_SCHEMA = 'shop';
```
