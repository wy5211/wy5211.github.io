上一篇我们用 Docker 启动了 MySQL，创建了一个任务表，完成了基本的增删改查。建表语句是照着写的，你可能没太留意每个字段为什么用那个类型——比如 `priority` 为什么是 `TINYINT` 而不是 `INT`，`status` 为什么用 `ENUM` 而不是 `VARCHAR`。

今天我们来认真回答这个问题。不是背数据类型表，而是从一个真实场景出发：你的团队要做一个内容发布平台，用户可以注册、发布文章、评论互动。你负责设计数据库表结构。

第一版建出来可能是这样——`VARCHAR` 存金额、`INT` 存手机号、没有设主键——上线后数据一多就出问题：金额精度丢了、手机号存不下、查询慢到离谱。这些问题的根源都是数据类型选错了。

## 整数：够用就行，别浪费

用户表里有好几个整数类型的字段：用户 ID、年龄、状态、文章数。它们的值域差别很大——用户 ID 可能到几千万，年龄不超过 150，状态通常只有 0-9 几个值。

MySQL 的整数类型从小到大有 TINYINT（1 字节）、SMALLINT（2 字节）、MEDIUMINT（3 字节）、INT（4 字节）、BIGINT（8 字节）。选多大不是看"够不够大"，而是看"用最小的能不能装下"。

```sql
CREATE TABLE users (
  id         BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
  age        TINYINT UNSIGNED DEFAULT 0 COMMENT '年龄',
  status     TINYINT UNSIGNED DEFAULT 1 COMMENT '状态：1正常 0禁用 2待验证',
  post_count INT UNSIGNED DEFAULT 0 COMMENT '文章数',
  follower_count INT UNSIGNED DEFAULT 0 COMMENT '粉丝数'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

为什么这么选：

- **用户 ID 用 BIGINT UNSIGNED**：几千万用户听起来多，但如果你做的是微博那种量级的产品，INT 的 42 亿上限也不是不可能触达。BIGINT 多占 4 字节，换来的是永远不会因为 ID 溢出而不得不做数据迁移。`UNSIGNED` 表示不接受负数，把正数范围翻倍。
- **年龄用 TINYINT UNSIGNED**：0-255 完全覆盖人类年龄，只需要 1 字节。如果用 INT，每个用户多浪费 3 字节，1000 万用户就是 30MB 的纯浪费——看起来不多，但 MySQL 加载索引页时是按页（默认 16KB）算的，越紧凑的索引，一页能装越多的数据，磁盘 I/O 就越少。
- **状态用 TINYINT UNSIGNED**：状态码不会超过 255，TINYINT 绰绰有余。
- **文章数和粉丝数用 INT UNSIGNED**：单用户最大 42 亿，够用了。如果你做的是超级大 V 平台，再考虑 BIGINT。

一个常见的新手错误是给所有整数字段都用 INT，甚至 BIGINT。"反正也占不了多少空间"——这个想法在小规模时没问题，但当你的表有几千万行、有十几个索引时，每个字段多出来的字节会乘以行数再乘以索引数，累积效应非常可观。

## 金额：FLOAT 和 DOUBLE 的精度陷阱

内容平台可能涉及付费专栏、打赏、会员订阅，这些都涉及金额。

先看一个实验：

```sql
CREATE TABLE price_test (
  f_price FLOAT,
  d_price DOUBLE,
  dec_price DECIMAL(10, 2)
);

INSERT INTO price_test VALUES (199.99, 199.99, 199.99);

SELECT f_price, d_price, dec_price FROM price_test;
-- f_price:   199.99     （看起来对，但内部存储有误差）
-- d_price:   199.99     （同上）
-- dec_price: 199.99     （精确值）
```

看起来都一样？来做一个运算就露馅了：

```sql
SELECT f_price * 1000000 FROM price_test;
-- 199989992.94793796  ← 偏了！

SELECT dec_price * 1000000 FROM price_test;
-- 199990000.00       ← 精确
```

**FLOAT 和 DOUBLE 使用 IEEE 754 浮点数标准**，某些十进制小数无法用二进制精确表示（就像十进制无法精确表示 1/3 一样）。这种误差在单价 19.99 元的商品上看起来可以忽略，但当你对百万笔订单做 `SUM()` 聚合时，误差会累加到可感知的程度——财务对账时就出大问题。

所以金额相关的字段，**永远用 DECIMAL**：

```sql
DECIMAL(10, 2)  -- 总共 10 位数字，其中 2 位小数
-- 最大值：99999999.99（接近 1 亿，对大多数场景够用）
-- 超大金额场景用 DECIMAL(12, 2) 或更大
```

那 FLOAT/DOUBLE 就没用了吗？不是。存储地理坐标（经纬度）、科学计算的近似值时，FLOAT 完全够用——这些场景不需要"一分钱都不差"的精度。

## 字符串：不是越长越好

用户名、邮箱、手机号、文章标题、文章内容——都是字符串，但适合的类型完全不同。

### 固定长度 vs 变长

**CHAR** 是固定长度，**VARCHAR** 是变长。区别在于 CHAR 会在末尾补空格到指定长度，而 VARCHAR 只存储实际内容加 1-2 字节的长度前缀。

```sql
-- 固定长度的字符串用 CHAR
phone CHAR(11)        -- 手机号固定 11 位
id_card CHAR(18)      -- 身份证号固定 18 位
password_hash CHAR(64) -- SHA256 哈希固定 64 字符

-- 长度不固定的字符串用 VARCHAR
username VARCHAR(50)   -- 用户名长度不一
email VARCHAR(100)     -- 邮箱长度不一
title VARCHAR(200)     -- 文章标题长度不一
```

CHAR 在查询时有微小的性能优势（不需要计算偏移量），但这个优势在现代硬件上几乎可以忽略。选择 CHAR 的真正理由是：**数据本身就是固定长度的**。手机号用 VARCHAR 也能存，但用 CHAR 语义更明确，存储更紧凑。

### VARCHAR 长度设多大

一个常见的疑问：VARCHAR(50) 和 VARCHAR(500) 性能差多少？

存储上差异不大——VARCHAR 只占实际内容的空间。但有两个地方会有影响：**内存中的临时表和排序缓冲区**。MySQL 在做 `ORDER BY` 或 `GROUP BY` 时可能创建内存临时表，此时 VARCHAR 字段会按定义的最大长度分配内存。VARCHAR(500) 意味着每行多预留 450 字节的内存。

实际建议：

```sql
username VARCHAR(50)    -- 够用，大多数平台限制用户名 20-50 字符
email VARCHAR(100)      -- 够用，最长的邮箱也很少超过 80 字符
title VARCHAR(200)      -- 够用，文章标题一般 100 字符以内
summary VARCHAR(500)    -- 摘要，500 字符足够
content TEXT            -- 正文长度不可预期，用 TEXT
```

**不要用 VARCHAR 存长文本**。行大小有 65535 字节的限制（所有字段加在一起），长文本用 TEXT/MEDIUMTEXT/LONGTEXT，它们的数据存在行外，不占行大小。

## 日期时间：DATETIME 和 TIMESTAMP 的坑

用户注册时间、文章发布时间、最后登录时间——看起来都是"时间"，但选错类型会遇到奇怪的 bug。

MySQL 有四种日期时间类型，最常用的两个：

| 类型      | 存储   | 范围         | 时区                               |
| --------- | ------ | ------------ | ---------------------------------- |
| DATETIME  | 8 字节 | 1000-9999 年 | 不转换（存什么就是什么）           |
| TIMESTAMP | 4 字节 | 1970-2038 年 | 自动转换（存 UTC，读时转当前时区） |

TIMESTAMP 有两个关键行为：存储时把当前时区的时间转成 UTC 存进去，读取时再从 UTC 转成当前时区。DATETIME 则完全不做转换。

这意味着什么？假设你的服务器在北京时间 2025-03-15 10:00:00 写入一条记录：

```sql
-- 用 TIMESTAMP 存
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
-- 存入的是 UTC 时间 2025-03-15 02:00:00
-- 在北京时间读取显示 2025-03-15 10:00:00
-- 在东京时间读取显示 2025-03-15 11:00:00

-- 用 DATETIME 存
event_start DATETIME
-- 存入的就是 2025-03-15 10:00:00
-- 在任何时区读取都是 2025-03-15 10:00:00
```

选择建议：

- **记录数据的创建/更新时间** → 用 `TIMESTAMP`。自动填入当前时间，自动更新，自动处理时区。`DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` 一行搞定。
- **记录固定时间点** → 用 `DATETIME`。比如"活动开始时间"是 2025-03-20 14:00，不管你从哪个时区访问，这个活动都是在这个时间开始，不应该被转换。

TIMESTAMP 有一个需要注意的限制：**2038 年问题**。4 字节的 TIMESTAMP 最大只能表示到 2038-01-19 03:14:07 UTC。如果你的系统需要存储未来的时间（比如会员到期时间、定期订阅的下次扣费时间），可能超出范围。MySQL 8.0.28+ 已经在内部做了优化，但如果你用的是旧版本，要特别注意。

我们的用户表这样设计时间字段：

```sql
created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
last_login_at TIMESTAMP NULL COMMENT '最后登录时间',
birthday      DATE COMMENT '生日'
```

`last_login_at` 用了 `TIMESTAMP NULL` 而不是 `DEFAULT CURRENT_TIMESTAMP`，因为用户注册时尚未登录，这个字段应该为空而不是填入注册时间。`TIMESTAMP` 默认是 `NOT NULL` 的，必须显式声明 `NULL` 才允许存 NULL 值。

## JSON：灵活但不万能

MySQL 8.0 的 JSON 支持已经很成熟了。在内容平台中，JSON 适合存什么？

商品属性是个好场景：不同品类的商品属性差异很大——手机有"屏幕尺寸"、"处理器"，衣服有"材质"、"尺码"。如果用关系表做，要建很多字段或关联表；用 JSON 就很自然：

```sql
CREATE TABLE products (
  id          BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  name        VARCHAR(200) NOT NULL COMMENT '商品名称',
  price       DECIMAL(10, 2) NOT NULL COMMENT '价格',
  attributes  JSON COMMENT '商品属性（不同品类不同结构）',
  tags        JSON COMMENT '标签列表'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

INSERT INTO products (name, price, attributes, tags) VALUES
('MacBook Pro 14寸', 14999.00,
 '{"cpu": "M3 Pro", "memory": "18GB", "storage": "512GB", "screen": "14.2寸"}',
 '["笔记本电脑", "Apple", "专业级"]');

-- 查询 JSON 内部字段
SELECT
  name,
  price,
  attributes->>'$.cpu' AS cpu,
  attributes->>'$.memory' AS memory
FROM products;
-- MacBook Pro 14寸 | 14999.00 | M3 Pro | 18GB
```

`->>` 运算符提取 JSON 字段并返回字符串类型（去掉引号），`->` 返回 JSON 类型（带引号）。日常查询用 `->>` 更方便。

JSON 的坑在于**不能直接建索引**。如果你需要按 `attributes->>'$.cpu'` 过滤，MySQL 会做全表扫描，每行都解析 JSON 然后对比。解决方案是建一个**虚拟列**：

```sql
ALTER TABLE products
  ADD COLUMN cpu VARCHAR(50) AS (attributes->>'$.cpu') STORED,
  ADD INDEX idx_cpu (cpu);
```

现在 `WHERE cpu = 'M3 Pro'` 就能走索引了。

但要注意：JSON 不适合存"需要频繁查询和关联"的数据。如果你的业务逻辑经常需要 `JOIN` JSON 内部的数据，那应该把它们拆成正经的关系表。JSON 的定位是"存那些偶尔查一下、结构不太固定的附加信息"。

## ENUM：方便但有隐患

状态字段怎么存？ENUM 是一个看起来很优雅的方案：

```sql
status ENUM('active', 'inactive', 'banned') DEFAULT 'active'
```

它限制了可选值，写入无效数据会报错（严格模式下）或变成空字符串（非严格模式），存储只占 1-2 字节（存的是索引值，不是字符串）。

但 ENUM 有一个硬伤：**新增选项需要 ALTER TABLE**。如果你的系统已经上线，`ALTER TABLE` 在大表上会锁表（MySQL 8.0 虽然改善了，但仍然有风险）。

对于状态这种"选项相对固定、很少变化"的场景，ENUM 是合适的。如果状态码可能频繁增加，用 TINYINT + COMMENT 注释更好：

```sql
status TINYINT UNSIGNED DEFAULT 1 COMMENT '状态：1正常 0禁用 2待验证 3封禁'
```

选项存在代码里（枚举类或常量），数据库只存数字。新增状态不需要改表结构。

## 字符集：为什么一定是 utf8mb4

这不是一个"锦上添花"的选择，而是一个"不选就可能出事"的必要设置。

MySQL 的 `utf8` 字符集最多支持 3 字节的 UTF-8 字符。但完整的 UTF-8 最多 4 字节，涵盖所有 Unicode 字符——包括 emoji 和部分生僻字（如"𠮷"）。MySQL 的 `utf8` 不是真正的 UTF-8，`utf8mb4` 才是。

```sql
-- utf8：无法存储 emoji
INSERT INTO users (username, bio) VALUES ('zhang_wei', '今天天气真好 ☀️');
-- 可能报错：Incorrect string value 或 emoji 被截断

-- utf8mb4：正常存储
INSERT INTO users (username, bio) VALUES ('zhang_wei', '今天天气真好 ☀️');
-- 存储成功
```

utf8mb4 比 utf8 多占一点存储空间（对于纯 ASCII 字符完全一样，对于中文每个字符多占 1 字节），但换来的是不用担心任何字符存不进去。**新项目没有任何理由用 utf8，直接 utf8mb4**。

建库和建表时都要指定：

```sql
CREATE DATABASE content_platform
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

CREATE TABLE users (...) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

`COLLATE utf8mb4_unicode_ci` 中的 `ci` 表示 case-insensitive（不区分大小写），意味着 `WHERE username = 'ZhangWei'` 能匹配到 `zhangwei`。如果你需要区分大小写，用 `utf8mb4_bin`。

## NOT NULL 和 DEFAULT：养成好习惯

数据库里最让人头疼的数据问题之一就是 NULL。NULL 不是 0，不是空字符串，它表示"未知"。而"未知"在 SQL 中的行为处处是坑：

```sql
-- NULL 参与运算，结果一定是 NULL
SELECT NULL + 1;          -- NULL
SELECT NULL = NULL;       -- NULL（不是 TRUE！）
SELECT NULL <> 1;         -- NULL（不是 TRUE！）

-- 查询 NULL 必须用 IS NULL / IS NOT NULL
SELECT * FROM users WHERE phone IS NULL;      -- 正确
SELECT * FROM users WHERE phone = NULL;       -- 永远返回空结果
```

**建议：字段尽量设为 NOT NULL，给一个合理的 DEFAULT 值。** 这样可以避免大量 `IS NULL` / `COALESCE` 的判断，也能让索引更高效（MySQL 处理 NULL 值的索引有一些额外的开销）。

```sql
username VARCHAR(50) NOT NULL DEFAULT '',       -- 不允许 NULL，默认空字符串
status TINYINT UNSIGNED NOT NULL DEFAULT 1,     -- 默认正常状态
created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
```

真正需要 NULL 的场景是"暂时没有值，且没有合理的默认值"，比如用户的头像 URL（注册时还没上传）、用户的生日（隐私信息，可能选择不填）。

## 完整的用户表设计

把上面讨论的所有决策汇总到一张表里：

```sql
CREATE TABLE users (
  id              BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID',
  username        VARCHAR(50) NOT NULL UNIQUE COMMENT '用户名',
  email           VARCHAR(100) NOT NULL UNIQUE COMMENT '邮箱',
  password_hash   CHAR(64) NOT NULL COMMENT '密码哈希（SHA256）',
  nickname        VARCHAR(50) DEFAULT '' COMMENT '昵称',
  avatar          VARCHAR(500) COMMENT '头像URL',
  bio             VARCHAR(500) DEFAULT '' COMMENT '个人简介',
  birthday        DATE COMMENT '生日',
  status          TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '状态：1正常 0禁用 2待验证',
  post_count      INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '文章数',
  follower_count  INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '粉丝数',
  last_login_at   TIMESTAMP NULL COMMENT '最后登录时间',
  last_login_ip   VARCHAR(45) COMMENT '登录IP（支持IPv6）',
  created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
  updated_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

  INDEX idx_email (email),
  INDEX idx_status (status),
  INDEX idx_created (created_at DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表';
```

每个字段的选择都对应了前面讨论的原则：

- 整数字段选最小的够用类型
- 金额不会出现在用户表，但内容平台的打赏/专栏费用表会用 DECIMAL
- 时间字段区分了 TIMESTAMP（自动记录）和 DATE（生日）
- 可选字段允许 NULL，核心字段 NOT NULL + DEFAULT
- 索引建在常用的查询字段上（邮箱、状态、注册时间倒序）

## Trade-off：表设计中的取舍

**够用 vs 预留**：字段类型选"够用"的，但 ID 选 BIGINT 预留空间。原因是字段类型改起来成本较低（ALTER TABLE），但 ID 溢出意味着整张表的数据要迁移。对不可逆的选择多预留，对可逆的选择够用就行。

**范式 vs 性能**：用户表里的 `post_count` 和 `follower_count` 是冗余字段——它们可以从文章表和关注表里 COUNT 出来。冗余的代价是更新时要同步，好处是查询用户列表时不需要 JOIN 和聚合。对于高频读取、低频更新的统计数据，冗余是值得的。

**ENUM vs TINYINT**：ENUM 约束更强、存储更紧凑，但扩展需要 ALTER TABLE。TINYINT 灵活但约束在应用层。状态码变化少的场景用 ENUM，变化频繁的场景用 TINYINT。

**TIMESTAMP vs DATETIME**：TIMESTAMP 自动处理时区、4 字节存储，但有 2038 限制。DATETIME 不转换时区、8 字节存储，范围更大。创建/更新时间用 TIMESTAMP，业务固定时间用 DATETIME。

上一篇我们用最简单的建表语句完成了第一次 CRUD。这篇我们深入了每个字段背后的选型逻辑——这不是"背下来就行"的知识，而是在实际设计表时会反复面对的决策。下一篇，我们在这张设计好的表上，做真正有挑战的增删改查操作——批量导入、多条件筛选、分页、软删除，这些才是日常开发中高频遇到的场景。
