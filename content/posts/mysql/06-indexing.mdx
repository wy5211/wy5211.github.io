## 一条慢查询引发的问题

周五下午，运维群里弹出一条告警：电商系统的"我的订单"接口响应时间从 200ms 飙升到了 3 秒。你登录数据库一看，订单表已经积累了 150 万条数据，而开发时只用了不到 1 万条测试数据，当时怎么写都很快。

你用 EXPLAIN 分析那条按用户 ID 查订单的 SQL，发现 `type: ALL`——MySQL 在对 150 万行做全表扫描。解决方案很明确：加索引。但问题来了——给哪些字段加？加几个？会不会影响写入性能？加了之后真的能命中吗？

这就是索引设计的核心矛盾：**读得快和写得快不可兼得**。接下来，我们围绕这个电商系统的真实场景，从索引的原理出发，一步步建立索引设计的系统认知。

> 上一篇我们用聚合函数、窗口函数和子查询完成了运营报表的统计需求。当数据量小时，GROUP BY 和窗口函数跑得飞快；但数据量一大，这些查询就会成为系统的性能瓶颈。索引，就是解决"数据量大之后查询变慢"这一问题的核心武器。

## 从全表扫描到 B+ 树查找

要理解索引为什么有效，先得理解没有索引时 MySQL 是怎么查数据的。

假设你要在订单表里找用户 `usr_a1x9k2m` 的所有订单。没有索引的情况下，MySQL 只能从第一行开始，逐行对比 `user_id` 字段，直到找完所有匹配的行。这就是**全表扫描**（Full Table Scan）。150 万行数据，即使每行只花 0.01 毫秒，加起来也要 15 秒。

索引的本质是一棵排序后的数据结构。MySQL InnoDB 引擎默认使用 **B+ 树**作为索引结构。你可以把它想象成一棵"多路平衡搜索树"：

```
                    [根节点]
                 /      |      \
           [分支节点] [分支节点] [分支节点]
            /    \      /    \      /    \
        [叶子] [叶子] [叶子] [叶子] [叶子] [叶子]
          |________________________________|
              （叶子节点通过双向链表串联）
```

B+ 树有几个关键特性让它特别适合数据库索引：

- **所有数据都存在叶子节点**，非叶子节点只存键值，用于导航。这意味着同一层的节点能容纳更多键值，树的高度就更矮。
- **叶子节点通过链表串联**，做范围查询（比如 `created_at > '2025-03-01'`）时，找到起点后顺着链表扫就行，不需要回到上层节点。
- **树的高度通常只有 3-4 层**。1000 万条数据，查找一条记录最多只需 3-4 次磁盘 I/O，这就是索引能将查询从秒级降到毫秒级的根本原因。

为什么不用其他数据结构？Hash 表查询是 O(1)，但不支持范围查询；二叉搜索树在最坏情况下会退化成链表；红黑树虽然平衡，但高度远高于 B+ 树，磁盘 I/O 次数多。B+ 根树在磁盘 I/O 和查询效率之间取得了最佳平衡。

## 索引的四种主要类型

MySQL 中常用的索引有四种，各有其适用场景。

### 主键索引

每张 InnoDB 表都有且只有一个聚簇索引（Clustered Index），通常就是主键索引。聚簇索引的叶子节点直接存储整行数据，所以通过主键查数据是最快的。

```sql
-- 主键索引在建表时自动创建
CREATE TABLE orders (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,  -- 这就是主键索引
  user_id BIGINT UNSIGNED NOT NULL,
  order_no VARCHAR(50) NOT NULL,
  total_amount DECIMAL(10, 2) NOT NULL,
  status VARCHAR(20) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB;
```

### 唯一索引

唯一索引保证列值不重复，但允许有多个 NULL 值。除了保证数据一致性，唯一索引在等值查询时也和主键索引一样高效。

```sql
-- 订单号不允许重复，适合用唯一索引
CREATE UNIQUE INDEX uk_order_no ON orders(order_no);
```

### 普通索引

普通索引没有唯一性约束，是最常用的索引类型，纯粹用于加速查询。

```sql
-- 按用户 ID 查订单是最频繁的操作，建立普通索引
CREATE INDEX idx_user_id ON orders(user_id);
```

### 全文索引

当需要对文本内容做关键词搜索时，`LIKE '%关键词%'` 无法走普通索引，这时就需要全文索引。

```sql
-- 为商品描述添加全文索引，支持关键词搜索
CREATE FULLTEXT INDEX ft_product_desc ON products(name, description);

-- 使用 MATCH AGAINST 进行全文搜索
SELECT * FROM products
WHERE MATCH(name, description) AGAINST('MacBook Pro' IN NATURAL LANGUAGE MODE);
```

## 复合索引与最左前缀原则

单列索引解决不了所有问题。比如"查询某个用户已完成的订单，按创建时间倒序"这个需求，同时涉及 `user_id`、`status`、`created_at` 三个字段。给每个字段各建一个索引？不如建一个**复合索引**。

```sql
-- 复合索引：按 user_id, status, created_at 的顺序建立
CREATE INDEX idx_user_status_created
ON orders(user_id, status, created_at);
```

复合索引遵循**最左前缀原则**：索引按字段定义的顺序排列，查询时必须从最左边的列开始使用。

```sql
-- 能命中 idx_user_status_created 索引的查询：
SELECT * FROM orders WHERE user_id = 1001;
-- 用到 user_id

SELECT * FROM orders
WHERE user_id = 1001 AND status = 'completed';
-- 用到 user_id + status

SELECT * FROM orders
WHERE user_id = 1001 AND status = 'completed' AND created_at > '2025-03-01';
-- 用到全部三列

-- 无法命中索引的查询：
SELECT * FROM orders WHERE status = 'completed';
-- 跳过了 user_id，最左前缀断裂

SELECT * FROM orders WHERE created_at > '2025-03-01';
-- 跳过了 user_id 和 status，完全无法使用
```

那复合索引的列顺序该怎么选？一个实用原则：**区分度高的列放左边**。`user_id` 有几十万个不同值，区分度接近 1.0；`status` 只有"待支付、已支付、已发货、已完成"几个值，区分度很低。把 `user_id` 放在最左边，能让索引在最开始就大幅缩小扫描范围。

## 覆盖索引：不回表就能拿到结果

普通索引（也叫二级索引）的叶子节点存的是索引列的值和对应的主键，不存整行数据。通过普通索引查到主键后，还需要再回聚簇索引查一次完整行，这叫**回表**。如果一次查询要回表很多次，性能会很差。

**覆盖索引**的思路是：把查询需要的所有字段都放进索引里，这样 MySQL 直接从索引中就能拿到结果，不需要回表。

```sql
-- 运营后台需要展示"已完成订单的金额和创建时间"
-- 如果只有 idx_user_status_created，查询时需要回表拿 total_amount

-- 建立覆盖索引，把 total_amount 也加进去
CREATE INDEX idx_status_created_total
ON orders(status, created_at, total_amount);

-- 这条查询只访问索引，不回表
SELECT status, created_at, total_amount
FROM orders
WHERE status = 'completed'
ORDER BY created_at DESC
LIMIT 10;
```

用 EXPLAIN 验证，如果 `Extra` 列显示 `Using index`，说明命中了覆盖索引。

覆盖索引的代价是索引体积变大，占用更多磁盘空间和内存。所以只对高频查询场景使用覆盖索引，不要无节制地往索引里加字段。

## 用 EXPLAIN 诊断查询

光加索引不够，你得验证索引是否真的被用上了。`EXPLAIN` 是 MySQL 提供的查询诊断工具，它会告诉你优化器打算怎么执行这条 SQL。

以电商系统最常见的"我的订单"查询为例：

```sql
EXPLAIN SELECT id, order_no, total_amount, status, created_at
FROM orders
WHERE user_id = 1001 AND status = 'completed'
ORDER BY created_at DESC
LIMIT 20;
```

重点关注以下几列：

| 列名      | 含义           | 关注点                                                                                                                                   |
| --------- | -------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **type**  | 访问类型       | 从差到好：ALL（全表扫描）< index（索引扫描）< range（范围扫描）< ref（索引等值查找）< eq_ref（唯一索引查找）< const（主键/唯一索引等值） |
| **key**   | 实际使用的索引 | 如果为 NULL，说明没走索引                                                                                                                |
| **rows**  | 预估扫描行数   | 越小越好，如果和实际返回行数差距大，说明有优化空间                                                                                       |
| **Extra** | 额外信息       | `Using index` 表示覆盖索引；`Using filesort` 表示额外排序；`Using where` 表示在存储引擎返回后做了过滤                                    |

下面看一个真实的诊断过程。运维报告"订单列表"接口慢，我们先看当前状态：

```sql
-- 订单表目前只有主键索引，没有为查询条件建索引
EXPLAIN SELECT * FROM orders WHERE user_id = 1001;
-- type: ALL（全表扫描）
-- key: NULL（没有可用索引）
-- rows: 1500000（扫描全部 150 万行）
-- Extra: Using where
```

全表扫描 150 万行，难怪要 3 秒。加上索引后再看：

```sql
-- 添加复合索引
CREATE INDEX idx_user_status_created
ON orders(user_id, status, created_at);

-- 再次分析
EXPLAIN SELECT id, order_no, total_amount, status, created_at
FROM orders
WHERE user_id = 1001 AND status = 'completed'
ORDER BY created_at DESC
LIMIT 20;
-- type: ref（索引等值查找）
-- key: idx_user_status_created（命中复合索引）
-- rows: 85（预估只扫描 85 行）
-- Extra: Backward index scan; Using where
```

从扫描 150 万行降到 85 行，查询时间从 3 秒降到 2 毫秒。这就是索引的力量。

## 索引失效的常见场景

加了索引不代表一定能命中。以下几种写法会导致索引失效，在实际开发中非常常见。

### 对索引列使用函数

```sql
-- 假设 email 上有索引
CREATE INDEX idx_email ON users(email);

-- 对索引列调用函数，索引失效
SELECT * FROM users WHERE LOWER(email) = 'zhang_wei@example.com';
-- type: ALL（全表扫描）

-- 改为不使用函数的方式，索引生效
SELECT * FROM users WHERE email = 'zhang_wei@example.com';
-- type: ref
```

如果业务确实需要忽略大小写查询，MySQL 8.0 提供了函数索引作为解决方案：

```sql
-- 对函数结果建索引（MySQL 8.0+）
CREATE INDEX idx_lower_email ON users((LOWER(email)));

-- 现在可以用函数查询并命中索引了
SELECT * FROM users WHERE LOWER(email) = 'zhang_wei@example.com';
-- type: ref
```

### 隐式类型转换

```sql
-- phone 字段是 VARCHAR 类型，建有索引
CREATE INDEX idx_phone ON users(phone);

-- 传入数字，MySQL 会隐式将 phone 转为数字再做比较，索引失效
SELECT * FROM users WHERE phone = 13800138000;
-- type: ALL

-- 传入字符串，索引生效
SELECT * FROM users WHERE phone = '13800138000';
-- type: ref
```

### LIKE 以通配符开头

```sql
-- 假设 username 上有索引
CREATE INDEX idx_username ON users(username);

-- 通配符在最前面，索引失效
SELECT * FROM users WHERE username LIKE '%zhang%';
-- type: ALL

-- 通配符在后面，索引生效（前缀匹配）
SELECT * FROM users WHERE username LIKE 'zhang%';
-- type: range

-- 如果需要任意位置匹配，考虑全文索引
```

### OR 连接的列只有一个有索引

```sql
-- user_id 有索引，但 status 没有单独的索引
SELECT * FROM orders WHERE user_id = 1001 OR status = 'completed';
-- type: ALL（因为 OR 只要有一个条件没索引，整体就退化为全表扫描）

-- 解决方案：给 status 也加上索引，或者改用 UNION
SELECT * FROM orders WHERE user_id = 1001
UNION ALL
SELECT * FROM orders WHERE status = 'completed' AND user_id != 1001;
```

这些场景有一个共同规律：**只要对索引列做了"计算"或"转换"，优化器就无法利用索引的有序性**。保持查询条件与索引列的原始形态一致，是避免索引失效的最基本原则。

## 前缀索引：长文本字段的索引方案

电商系统的用户表里，`email`、手机号、收货地址等字段都是字符串类型，有些 URL 字段甚至长达 500 个字符。给整个字段建索引，索引体积会非常大。**前缀索引**允许你只对字段的前 N 个字符建立索引。

```sql
-- 对 email 的前 12 个字符建索引
CREATE INDEX idx_email_prefix ON users(email(12));

-- 对 URL 的前 20 个字符建索引
CREATE INDEX idx_url_prefix ON users(url(20));
```

前缀长度怎么选？太短了区分度不够，太长了索引体积大。可以用以下 SQL 找到合适的截断点：

```sql
-- 计算不同前缀长度下的区分度
SELECT
  COUNT(DISTINCT LEFT(email, 6))  / COUNT(*) AS ratio_6,
  COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) AS ratio_10,
  COUNT(DISTINCT LEFT(email, 15)) / COUNT(*) AS ratio_15,
  COUNT(DISTINCT LEFT(email, 20)) / COUNT(*) AS ratio_20
FROM users;
-- 选择区分度接近完整值（1.0）的最短长度
```

比如结果是 `ratio_6=0.42, ratio_10=0.85, ratio_15=0.98, ratio_20=0.99`，那选 15 个字符就够了。

前缀索引有两个限制需要注意：

- **不支持覆盖索引**，因为索引中只存了前 N 个字符，不是完整值。
- **不支持 ORDER BY 和 GROUP BY**，排序需要完整的字段值。

对于必须用完整值做精确匹配的场景，可以用 Hash 字段替代前缀索引：

```sql
-- 添加一个 Hash 字段存储 email 的 MD5 值
ALTER TABLE users ADD COLUMN email_hash CHAR(32) NOT NULL;
CREATE INDEX idx_email_hash ON users(email_hash);

-- 写入时同时计算 Hash 值
UPDATE users SET email_hash = MD5(email);

-- 查询时用 Hash 值匹配
SELECT * FROM users WHERE email_hash = MD5('zhang_wei@example.com');
```

## 索引不是越多越好

理解了索引的威力后，很容易走向另一个极端：给所有字段都加索引。但索引是有代价的。

每次 INSERT、UPDATE、DELETE 操作，MySQL 不只要修改数据行，还要同步更新所有相关的索引。假设一张表有 8 个索引，每插入一行数据就要写 8 棵 B+ 树。在电商大促期间，订单表每秒写入数千行，如果索引过多，写入性能会显著下降。

此外，索引本身也占磁盘空间。一个包含多个 VARCHAR 列的复合索引，体积可能比数据本身还大。更多的索引意味着更多的内存消耗，而 InnoDB 的缓冲池（Buffer Pool）是有限的——索引占多了，热数据就缓存不下，反而拖慢了查询。

一个实用的经验法则：

- 单表索引数量控制在 5 个以内
- 单个复合索引的列数不超过 4-5 个
- 区分度低于 0.1 的字段（比如性别、状态）通常不值得单独建索引
- 优先为 WHERE 条件中高频出现的字段建索引，其次考虑 ORDER BY 和 GROUP BY

可以用 MySQL 8.0 的 performance_schema 查看索引的实际使用情况，找出那些创建了但从未被命中的"僵尸索引"：

```sql
SELECT
  OBJECT_NAME AS table_name,
  INDEX_NAME,
  COUNT_FETCH AS read_count,
  COUNT_INSERT AS insert_count
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE OBJECT_SCHEMA = 'shop'
  AND INDEX_NAME IS NOT NULL
ORDER BY COUNT_FETCH ASC;
-- read_count 为 0 的索引就是从未被使用的，可以考虑删除
```

MySQL 8.0 还提供了**不可见索引**功能，让你安全地测试删除索引的影响：

```sql
-- 先把索引设为不可见，优化器不会使用它，但数据变更仍会维护它
ALTER INDEX idx_age ON users INVISIBLE;

-- 观察几天，如果没有任何性能问题，再真正删除
DROP INDEX idx_age ON users;

-- 如果有问题，随时恢复可见
ALTER INDEX idx_age ON users VISIBLE;
```

## 权衡：索引设计是在读写之间找平衡

回到开头那个问题：订单表的慢查询，我们通过一个 `(user_id, status, created_at)` 的复合索引解决了，查询从 3 秒降到了 2 毫秒。但这个索引不是免费午餐——每次新订单写入时，MySQL 都要额外维护这棵 B+ 树，而且它占了额外的磁盘和内存空间。

索引设计的本质就是在**读性能**和**写性能**之间做取舍。没有万能的索引方案，只有针对具体业务场景的权衡。以下是几个关键判断依据：

- **读多写少**的表（如订单查询、商品浏览），可以大胆加索引，优先保障查询性能
- **写多读少**的表（如日志表、消息队列表），索引要精简，优先保障写入吞吐量
- **读写均衡**的表，用复合索引替代多个单列索引，用覆盖索引减少回表次数，用不可见索引做灰度验证

掌握了索引原理和这些实战策略，你就具备了诊断和优化大部分慢查询的能力。下一篇我们将进入 MySQL 另一个核心话题——事务与锁，看看在并发场景下如何保证数据的一致性。
