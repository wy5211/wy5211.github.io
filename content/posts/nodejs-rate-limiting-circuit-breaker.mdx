在高并发场景下，限流和熔断是保护后端服务的两道重要防线。本文将深入讲解如何在 Node.js 应用中实现这两大核心机制。

## 为什么需要限流和熔断？

### 限流的必要性

想象一下这个场景：你的 API 突然收到 10 倍于平时的请求量，可能是：

- 爬虫疯狂抓取
- 某个客户端出现 bug，陷入死循环调用
- 营销活动带来真实流量高峰
- 恶意攻击

没有限流的话，服务器会迅速过载，导致：

- 响应变慢，正常用户体验变差
- 数据库连接池耗尽
- 内存溢出，服务崩溃
- 连锁反应影响整个微服务系统

### 熔断的必要性

在微服务架构中，服务 A 调用服务 B，B 调用服务 C。如果 C 挂了：

1. A 的请求会一直等待 B 的响应
2. B 的请求会一直等待 C 的响应
3. 线程/资源被耗尽
4. 雪崩效应：整个系统瘫痪

熔断器的作用就是：

- 快速失败：当下游服务不可用时，立即返回错误，而不是等待超时
- 自动恢复：定期尝试恢复，而不是需要人工重启
- 降级处理：返回兜底数据，保证核心功能可用

## 限流算法详解

### 1. 固定窗口算法

最简单的限流方式：在固定时间窗口内限制请求数量。

```javascript
class FixedWindowLimiter {
  constructor(limit, windowMs) {
    this.limit = limit;
    this.windowMs = windowMs;
    this.count = 0;
    this.windowStart = Date.now();
  }

  tryAcquire() {
    const now = Date.now();

    // 检查是否需要重置窗口
    if (now - this.windowStart >= this.windowMs) {
      this.windowStart = now;
      this.count = 0;
    }

    // 检查是否超过限制
    if (this.count < this.limit) {
      this.count++;
      return true;
    }

    return false;
  }
}

// 使用示例：每分钟最多 100 次请求
const limiter = new FixedWindowLimiter(100, 60000);

if (limiter.tryAcquire()) {
  // 处理请求
} else {
  // 返回 429 Too Many Requests
}
```

**优点**：实现简单，内存占用小

**缺点**：存在边界突发问题。例如在 00:59 来了 100 个请求，01:01 又来了 100 个请求，虽然每个窗口都没超限，但在 2 秒内处理了 200 个请求。

### 2. 滑动窗口算法

解决固定窗口的边界问题，更平滑地控制请求速率。

```javascript
class SlidingWindowLimiter {
  constructor(limit, windowMs) {
    this.limit = limit;
    this.windowMs = windowMs;
    this.requests = [];
  }

  tryAcquire() {
    const now = Date.now();
    const windowStart = now - this.windowMs;

    // 移除窗口外的请求
    this.requests = this.requests.filter((time) => time > windowStart);

    // 检查是否超过限制
    if (this.requests.length < this.limit) {
      this.requests.push(now);
      return true;
    }

    return false;
  }
}

// 使用示例
const limiter = new SlidingWindowLimiter(100, 60000);
```

**优点**：更精确，无边界突发问题

**缺点**：需要存储所有请求时间戳，内存占用较高

### 3. 滑动窗口对数（生产推荐）

滑动窗口的优化版本，通过采样减少内存占用。

```javascript
class SlidingWindowLogLimiter {
  constructor(limit, windowMs) {
    this.limit = limit;
    this.windowMs = windowMs;
    this.log = new Map(); // 使用 Map 存储
  }

  tryAcquire(key = "default") {
    const now = Date.now();
    const windowStart = now - this.windowMs;

    if (!this.log.has(key)) {
      this.log.set(key, []);
    }

    const requests = this.log.get(key);

    // 移除过期请求
    while (requests.length && requests[0] <= windowStart) {
      requests.shift();
    }

    if (requests.length < this.limit) {
      requests.push(now);
      return true;
    }

    return false;
  }
}
```

### 4. 令牌桶算法（推荐）

最常用的限流算法，允许一定的突发流量。

**原理**：

- 系统以固定速率往桶里放令牌
- 请求到达时，从桶里取一个令牌
- 桶满了令牌就丢弃
- 桶空了就拒绝请求

```javascript
class TokenBucketLimiter {
  constructor(capacity, fillRate) {
    this.capacity = capacity; // 桶容量
    this.fillRate = fillRate; // 每秒填充的令牌数
    this.tokens = capacity; // 当前令牌数
    this.lastFillTime = Date.now();
  }

  tryAcquire(tokens = 1) {
    this._refill();

    if (this.tokens >= tokens) {
      this.tokens -= tokens;
      return true;
    }

    return false;
  }

  _refill() {
    const now = Date.now();
    const elapsed = (now - this.lastFillTime) / 1000;

    // 计算应该补充的令牌数
    const tokensToAdd = elapsed * this.fillRate;
    this.tokens = Math.min(this.capacity, this.tokens + tokensToAdd);
    this.lastFillTime = now;
  }
}

// 使用示例：容量 100，每秒补充 10 个令牌
const limiter = new TokenBucketLimiter(100, 10);
```

**优点**：

- 允许突发流量（桶里有令牌时）
- 实现相对简单
- 可以应对短期流量高峰

### 5. 漏桶算法

将请求平滑地输出，适合保护下游服务。

**原理**：

- 请求像水一样流入桶中
- 桶底有个洞，以固定速率漏水（处理请求）
- 桶满了就溢出（拒绝请求）

```javascript
class LeakyBucketLimiter {
  constructor(capacity, leakRate) {
    this.capacity = capacity; // 桶容量
    this.leakRate = leakRate; // 每秒漏出的请求数
    this.queue = []; // 请求队列
    this.lastLeakTime = Date.now();
  }

  tryAcquire() {
    this._leak();

    if (this.queue.length < this.capacity) {
      this.queue.push(Date.now());
      return true;
    }

    return false;
  }

  _leak() {
    const now = Date.now();
    const elapsed = (now - this.lastLeakTime) / 1000;
    const leakCount = Math.floor(elapsed * this.leakRate);

    // 移除已处理的请求
    for (let i = 0; i < leakCount && this.queue.length > 0; i++) {
      this.queue.shift();
    }

    this.lastLeakTime = now;
  }
}

// 使用示例：桶容量 100，每秒处理 10 个请求
const limiter = new LeakyBucketLimiter(100, 10);
```

**优点**：输出速率恒定，保护下游服务

**缺点**：无法应对突发流量

## Express 中实现限流

### 使用 express-rate-limit（推荐）

```javascript
const express = require("express");
const rateLimit = require("express-rate-limit");

const app = express();

// 基础限流：每个 IP 每分钟最多 100 次请求
const basicLimiter = rateLimit({
  windowMs: 60 * 1000, // 1 分钟
  max: 100, // 最多 100 次请求
  message: {
    error: "请求过于频繁，请稍后再试",
  },
  standardHeaders: true, // 返回 RateLimit-* 头信息
  legacyHeaders: false,
});

app.use(basicLimiter);

// 针对特定路由的严格限流
const strictLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 分钟
  max: 5, // 最多 5 次请求
  skipSuccessfulRequests: false,
  keyGenerator: (req) => {
    // 使用用户 ID 而不是 IP（需登录）
    return req.user?.id || req.ip;
  },
});

// 登录接口使用严格限流
app.post("/api/login", strictLimiter, (req, res) => {
  // 登录逻辑
});

// API 限流：根据用户等级
const apiLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: (req) => {
    // VIP 用户每分钟 1000 次，普通用户 100 次
    return req.user?.isVIP ? 1000 : 100;
  },
});

app.use("/api", apiLimiter);

app.listen(3000);
```

### 分布式限流（使用 Redis）

单机限流在多实例部署时无效，需要使用 Redis 实现分布式限流。

```javascript
const express = require("express");
const Redis = require("ioredis");
const redis = new Redis();

const app = express();

class RedisRateLimiter {
  constructor(options) {
    this.redis = options.redis;
    this.windowMs = options.windowMs;
    this.max = options.max;
    this.prefix = options.prefix || "ratelimit:";
  }

  async tryAcquire(key) {
    const redisKey = this.prefix + key;
    const now = Date.now();
    const windowStart = now - this.windowMs;

    // 使用 Lua 脚本保证原子性
    const luaScript = `
      local key = KEYS[1]
      local now = tonumber(ARGV[1])
      local windowStart = tonumber(ARGV[2])
      local max = tonumber(ARGV[3])

      -- 移除窗口外的请求
      redis.call('ZREMRANGEBYSCORE', key, 0, windowStart)

      -- 获取当前窗口内的请求数
      local count = redis.call('ZCARD', key)

      if count < max then
        -- 添加当前请求
        redis.call('ZADD', key, now, now .. '-' .. math.random())
        redis.call('PEXPIRE', key, ARGV[4])
        return {1, max - count - 1}
      else
        return {0, 0}
      end
    `;

    const result = await this.redis.eval(
      luaScript,
      1,
      redisKey,
      now,
      windowStart,
      this.max,
      this.windowMs
    );

    return {
      allowed: result[0] === 1,
      remaining: result[1],
    };
  }
}

const limiter = new RedisRateLimiter({
  redis,
  windowMs: 60000,
  max: 100,
});

app.use(async (req, res, next) => {
  const key = req.ip;
  const result = await limiter.tryAcquire(key);

  // 设置响应头
  res.setHeader("X-RateLimit-Remaining", result.remaining);

  if (!result.allowed) {
    return res.status(429).json({
      error: "请求过于频繁，请稍后再试",
    });
  }

  next();
});

app.listen(3000);
```

## 熔断器实现

### 熔断器的三种状态

1. **关闭（Closed）**：正常状态，请求正常通过
2. **打开（Open）**：熔断状态，请求直接失败，不调用下游服务
3. **半开（Half-Open）**：尝试恢复状态，允许少量请求通过测试

### 手写熔断器

```javascript
class CircuitBreaker {
  constructor(options) {
    this.failureThreshold = options.failureThreshold || 5;
    this.successThreshold = options.successThreshold || 2;
    this.timeout = options.timeout || 60000; // 熔断持续时间

    this.failureCount = 0;
    this.successCount = 0;
    this.state = "CLOSED"; // CLOSED, OPEN, HALF_OPEN
    this.nextAttempt = Date.now();
  }

  async call(fn) {
    if (this.state === "OPEN") {
      if (Date.now() < this.nextAttempt) {
        throw new Error("熔断器处于打开状态");
      }
      // 进入半开状态
      this.state = "HALF_OPEN";
    }

    try {
      const result = await fn();

      this.onSuccess();
      return result;
    } catch (error) {
      this.onFailure();
      throw error;
    }
  }

  onSuccess() {
    this.failureCount = 0;

    if (this.state === "HALF_OPEN") {
      this.successCount++;

      if (this.successCount >= this.successThreshold) {
        // 恢复正常
        this.state = "CLOSED";
        this.successCount = 0;
      }
    }
  }

  onFailure() {
    this.failureCount++;

    if (this.state === "HALF_OPEN") {
      // 半开状态下失败，立即打开
      this.state = "OPEN";
      this.nextAttempt = Date.now() + this.timeout;
      this.successCount = 0;
    } else if (this.failureCount >= this.failureThreshold) {
      // 失败次数达到阈值，打开熔断器
      this.state = "OPEN";
      this.nextAttempt = Date.now() + this.timeout;
    }
  }

  getState() {
    return {
      state: this.state,
      failureCount: this.failureCount,
      successCount: this.successCount,
      nextAttempt: this.nextAttempt,
    };
  }
}

// 使用示例
const circuitBreaker = new CircuitBreaker({
  failureThreshold: 5,
  successThreshold: 2,
  timeout: 30000,
});

async function callExternalService() {
  try {
    const result = await circuitBreaker.call(async () => {
      const response = await fetch("https://api.example.com/data");
      if (!response.ok) {
        throw new Error("请求失败");
      }
      return response.json();
    });
    return result;
  } catch (error) {
    // 熔断器打开或服务调用失败
    console.error("服务调用失败:", error.message);

    // 降级处理
    return getCachedData();
  }
}
```

async function fetchWithRetry(url, options, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const response = await fetch(url, options);

      if (response.status === 429) {
        // 读取 Retry-After 头
        const retryAfter = response.headers.get("Retry-After") || 60;
        await sleep(retryAfter * 1000);
        continue;
      }

      return response;
    } catch (error) {
      if (i === maxRetries - 1) throw error;
      await sleep(Math.pow(2, i) * 1000); // 指数退避
    }
  }
}

## 常见问题

### Q1: 限流和熔断有什么区别？

**限流**：防止系统过载，主动拒绝超出限制的请求
**熔断**：当下游服务故障时，快速失败，避免级联故障

### Q2: 分布式限流如何实现？

使用 Redis + Lua 脚本保证原子性，或者使用专门的限流服务如：

- Redis 7.0 的 Rate Limiter
- Envoy 的限流过滤器
- Kong 的 Rate Limiting 插件

### Q3: 熔断器打开后如何恢复？

熔断器会在 resetTimeout 后进入半开状态，允许少量请求通过。如果成功，恢复正常；如果失败，继续熔断。

### Q4: 如何选择限流粒度？

- **IP 级别**：防止单个 IP 滥用
- **用户级别**：公平分配资源
- **接口级别**：关键接口更严格
- **全局限别**：保护整体系统

## 总结

限流和熔断是构建高可用系统的重要手段：

1. **限流**保护系统不被突发流量击垮
2. **熔断**防止故障在微服务间级联传播
3. **降级**保证核心功能在故障时仍可用
4. **监控**及时发现问题并告警

记住：**宁可快速失败，也不要让整个系统崩溃**。
