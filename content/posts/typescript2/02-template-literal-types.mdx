上一篇我们学了条件类型和 infer，能在类型层面做 if-else 判断。但条件类型处理的是"类型和类型之间的关系"——T 是否是 U 的子类型。有一类问题它解决不了：**字符串内部的结构**。

比如你的项目有一个事件系统，事件名都是 `"module:action"` 格式——`"user:login"`、`"order:created"`、`"payment:failed"`。你想让 `emit` 和 `on` 的事件名严格匹配，参数类型也自动关联。但事件名是字符串，TypeScript 怎么知道 `"user:login"` 对应的参数是 `{ userId: string; timestamp: Date }`？

这就是模板字面量类型要解决的问题。

## 模板字面量类型：字符串的"类型化"

模板字面量类型的语法和 JavaScript 的模板字符串一模一样，只不过它用在 `type` 定义里：

```typescript
type Greeting = `Hello, ${string}`;
// 这个类型接受任何以 "Hello, " 开头的字符串
const a: Greeting = "Hello, World"; // 正确
const b: Greeting = "Hello, 张三"; // 正确
const c: Greeting = "Hi, World"; // 错误：不以 "Hello, " 开头
```

`${string}` 是一个占位符，表示"任意字符串"。你也可以用更具体的类型：

```typescript
type EventName = `${string}:${string}`;
// 接受任何包含一个冒号的字符串
const a: EventName = "user:login"; // 正确
const b: EventName = "order:created"; // 正确
const c: EventName = "user-login"; // 错误：没有冒号
```

到这里你可能觉得：这有什么用？`string` 加个前缀还是 `string`，TypeScript 又不能帮你检查前缀对不对。

别急。模板字面量类型的威力不在单独使用，而在和**联合类型**、**条件类型**、**映射类型**组合。

## 联合类型 + 模板字面量：自动展开

当 `${}` 里面放的是联合类型时，TypeScript 会自动展开所有组合——就像笛卡尔积：

```typescript
type Module = "user" | "order" | "payment";
type Action = "created" | "updated" | "deleted";
type EventName = `${Module}:${Action}`;

// 自动展开为 9 种组合：
// "user:created" | "user:updated" | "user:deleted" |
// "order:created" | "order:updated" | "order:deleted" |
// "payment:created" | "payment:updated" | "payment:deleted"
```

这不是运行时计算的，是编译期展开的。你传入 `"user:login"`，编译器直接报错——因为 `login` 不在 `Action` 联合类型里。

这个特性让 TypeScript 能做到一件很强的事：**把自由度的字符串收窄为有限的、可枚举的集合**。

## 内置字符串工具类型

TypeScript 提供了四个内置的字符串操作类型，专门配合模板字面量使用：

```typescript
type S = "hello world";

type U = Uppercase<S>; // "HELLO WORLD"
type L = Lowercase<S>; // "hello world"
type C = Capitalize<S>; // "Hello world" — 首字母大写
type UC = Uncapitalize<S>; // "hello world" — 首字母小写
```

单独看没什么用。配合模板字面量类型，它们能做类型级别的字符串变换：

```typescript
type EventName = "user:login" | "order:created" | "payment:failed";

// 把事件名转换为对应的 handler 函数名
// "user:login" → "handleUserLogin"
type HandlerName<T extends string> = T extends `${infer Module}:${infer Action}`
  ? `handle${Capitalize<Module>}${Capitalize<Action>}`
  : never;

type Handlers = HandlerName<EventName>;
// "handleUserLogin" | "handleOrderCreated" | "handlePaymentFailed"
```

注意这里的模式：先用 `infer` 从字符串中提取出模块名和动作名，再用 `Capitalize` 变换大小写，最后用模板字面量拼接成新的字符串类型。**条件类型 + infer + 模板字面量**这个组合，是处理字符串类型问题的标准套路。

## 字符串模式匹配：从 URL 中提取参数

模板字面量类型最实用的场景之一是从 URL 路径中提取参数。比如你的 API 有这些路由：

```
GET /users
GET /users/:userId
GET /users/:userId/orders/:orderId
```

你想让 `:userId` 和 `:orderId` 自动提取为参数对象类型：

```typescript
// 从路径中提取参数名
type ExtractParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParams<Rest>
    : T extends `${string}:${infer Param}`
      ? Param
      : never;

type Params1 = ExtractParams<"/users">;
// never — 没有参数

type Params2 = ExtractParams<"/users/:userId">;
// "userId"

type Params3 = ExtractParams<"/users/:userId/orders/:orderId">;
// "userId" | "orderId"
```

这段类型定义用了递归条件类型。逻辑是：如果路径中有 `:参数名/剩余部分`，提取参数名，然后继续处理剩余部分。如果没有剩余部分了，提取最后一个参数名。如果没有 `:`，说明没有参数，返回 never。

有了参数名列表，再把它转换为参数对象类型就简单了：

```typescript
type ParamsToObject<T extends string> = Record<ExtractParams<T>, string>;

type UserParams = ParamsToObject<"/users/:userId">;
// { userId: string }

type OrderParams = ParamsToObject<"/users/:userId/orders/:orderId">;
// { userId: string; orderId: string }
```

现在你定义 API 路由时，参数类型是自动推导的——不需要手写 `{ userId: string }`。

## 实战：类型安全的事件系统

把上面的知识组合起来，实现一个完整的事件系统。这个事件系统的目标是：**事件名和参数类型一一对应，传错事件名或参数，编译器直接报错**。

```typescript
// 第一步：定义事件映射表
type EventMap = {
  "user:login": { userId: string; ip: string };
  "user:logout": { userId: string };
  "order:created": { orderId: string; totalAmount: number };
  "order:shipped": { orderId: string; trackingNumber: string };
  "payment:failed": { orderId: string; reason: string; retryCount: number };
};

// 第二步：实现类型安全的事件发射器
class TypedEmitter<Events extends Record<string, any>> {
  private handlers = new Map<string, Set<Function>>();

  on<K extends string & keyof Events>(
    event: K,
    handler: (data: Events[K]) => void
  ): () => void {
    if (!this.handlers.has(event)) {
      this.handlers.set(event, new Set());
    }
    this.handlers.get(event)!.add(handler);
    return () => this.handlers.get(event)?.delete(handler);
  }

  emit<K extends string & keyof Events>(event: K, data: Events[K]): void {
    this.handlers.get(event)?.forEach((fn) => fn(data));
  }
}
```

使用的时候：

```typescript
const emitter = new TypedEmitter<EventMap>();

// 正确：事件名和参数类型匹配
emitter.on("user:login", ({ userId, ip }) => {
  console.log(`${userId} logged in from ${ip}`);
});

emitter.emit("order:created", {
  orderId: "ord_7n3j5k",
  totalAmount: 12900,
});

// 错误：事件名拼写错误——编译器报错
emitter.on("user:logn", ({ userId, ip }) => {}); // "user:logn" 不在 EventMap 中

// 错误：参数不完整——编译器报错
emitter.emit("order:created", { orderId: "ord_7n3j5k" });
// 缺少 totalAmount
```

这个事件系统的关键在于泛型参数 `Events extends Record<string, any>`。`on` 和 `emit` 都通过 `K extends string & keyof Events` 约束事件名，然后通过 `Events[K]` 获取对应的参数类型。事件名传错了，K 不在 Events 的 key 中，报错。参数传错了，类型和 `Events[K]` 不匹配，报错。

全程零运行时开销——所有检查都在编译期完成。

## 模板字面量类型的局限性

模板字面量类型很强，但有几个限制你得知道。

**只能匹配固定模式**。它擅长匹配已知的字符串结构（`"module:action"`、`"/users/:id"`），但不擅长处理完全自由的字符串。如果你的路由路径是运行时动态拼接的（比如用户自定义的 URL 模式），模板字面量类型帮不了你。

**展开后可能产生巨大的联合类型**。如果你的 Module 有 10 个值、Action 有 10 个值，展开后就是 100 个字符串字面量的联合类型。再和其他类型组合，类型检查的性能可能会下降。

**infer 提取的类型是精确的**。`ExtractParams<"/users/:userId">` 提取的是字面量 `"userId"`，不是宽泛的 `string`。这通常是你想要的，但有时候你需要更宽泛的类型——比如你想要所有以 `/users/` 开头的路径都返回同样的参数类型。这种情况下你需要用条件类型做额外的判断。

## 什么时候用模板字面量类型

适合的场景：

- 事件名、动作名、状态名等有固定格式的字符串
- URL 路径和路由参数的提取
- CSS 属性名、HTML 属性名等有命名约定的字符串
- 任何"字符串中编码了类型信息"的场景

不适合的场景：

- 完全自由的字符串输入（用户名、搜索关键词）
- 运行时动态生成的字符串
- 没有结构化模式的字符串

一个实用的判断标准：**如果你能用正则表达式描述这个字符串的格式，那它大概率适合用模板字面量类型**。上一篇学了条件类型能判断"是什么"，这一篇让条件类型能处理"字符串里面是什么"。下一篇我们把这两种能力用泛型串联起来，处理更复杂的类型问题。
