# LLM Prompt Engineering 最佳实践：从入门到精通

Prompt Engineering（提示词工程）是与大语言模型（LLM）交互的核心技能。精心设计的 Prompt 可以显著提升模型输出质量、准确性和实用性。本文将系统讲解 Prompt Engineering 的最佳实践、常见模式和实战技巧。

## 一、Prompt Engineering 核心原则

### 1. 清晰明确（Clarity）

**原则**：避免歧义，让模型明确理解你的意图。

```markdown
❌ 糟糕的 Prompt：
写一个函数

✅ 优秀的 Prompt：
用 TypeScript 编写一个函数，功能是：

- 输入：字符串数组
- 输出：去重后的字符串数组，按字母升序排列
- 要求：处理空数组、null、undefined 边界情况
```

### 2. 提供上下文（Context）

**原则**：给模型足够的背景信息，帮助其理解任务场景。

```markdown
❌ 缺少上下文：
优化这段代码

✅ 提供完整上下文：
这是一个 Node.js API 的性能关键路径，每秒处理 1000+ 请求。
当前代码存在性能瓶颈，请优化：

[代码片段]

约束条件：

- 不能引入新的依赖库
- 保持代码可读性
- 内存占用不超过当前水平的 1.5 倍
```

### 3. 分步指导（Step-by-Step）

**原则**：复杂任务拆解为多个步骤，引导模型逐步推理。

```markdown
请按以下步骤分析这个技术方案：

1. **架构评估**：分析方案的分层设计、模块划分
2. **性能分析**：识别潜在的性能瓶颈
3. **风险评估**：列出可能的技术风险
4. **改进建议**：提供具体的优化建议

技术方案详情：
[方案描述]
```

## 二、高级 Prompt 设计模式

### 1. 角色扮演模式（Role Prompting）

赋予模型特定角色，使其输出更专业、更贴合场景。

```markdown
你是一位拥有 10 年经验的后端架构师，专注于高并发系统设计。
请从以下角度评估我的系统架构：

- 可扩展性
- 可维护性
- 成本效益
- 技术选型合理性

系统架构描述：
[架构详情]
```

### 2. Few-Shot Learning（少样本学习）

通过示例教模型如何完成任务。

```markdown
从用户评论中提取情感和关键信息：

示例 1：
评论："这个产品太棒了！物流很快，质量也很好。"
输出：{"sentiment": "positive", "keywords": ["产品好", "物流快", "质量好"]}

示例 2：
评论："价格有点贵，但功能还可以。"
输出：{"sentiment": "neutral", "keywords": ["价格贵", "功能还可以"]}

现在请处理：
评论："客服态度很差，产品用了一周就坏了，不推荐购买。"
```

### 3. Chain-of-Thought（思维链）

引导模型展示推理过程，提高复杂任务的准确性。

```markdown
请一步步推理并回答：

问题：如果一个数组有 100 个元素，二分查找最多需要比较多少次？

让我们一步步思考：

1. 二分查找每次将搜索范围缩小一半
2. 初始范围：100 个元素
3. 第一次比较后：50 个元素
4. 第二次比较后：25 个元素
5. ...继续计算直到剩余 1 个元素
6. 计算总共需要多少次比较

请给出详细的推理过程和最终答案。
```

### 4. 结构化输出（Structured Output）

明确指定输出格式，便于程序处理。

````markdown
分析以下代码的时间复杂度和空间复杂度，以 JSON 格式输出：

```typescript
function fibonacci(n: number): number {
  if (n &lt;= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}
```
````

输出格式：
{
"timeComplexity": "Big O 表示法",
"spaceComplexity": "Big O 表示法",
"explanation": "详细解释",
"optimization": "优化建议"
}

````

## 三、实战技巧与最佳实践

### 1. 使用分隔符清晰界定输入

```markdown
请总结以下文章的核心观点：

---文章开始---
[文章内容]
---文章结束---

要求：
- 提炼 3-5 个核心观点
- 每个观点不超过 50 字
````

### 2. 明确约束条件

````markdown
为 React 组件编写单元测试：

组件代码：

```tsx
function Button({ onClick, disabled, children }) {
  return (
    &lt;button onClick={onClick} disabled={disabled}&gt;
      {children}
    &lt;/button&gt;
  );
}
```
````

约束条件：

- 使用 Vitest + React Testing Library
- 测试覆盖率要求：100%
- 必须测试的场景：
  1. 正常点击
  2. disabled 状态下点击无响应
  3. children 正确渲染
- 不要使用 any 类型
- 测试代码必须有清晰的描述

````

### 3. 迭代优化策略

```markdown
初始 Prompt：
"写一个 API 文档"

优化后：
"为以下 REST API 编写文档：

API 端点：POST /api/users

请求体：
{
  "name": "string (必填，2-50 字符)",
  "email": "string (必填，有效邮箱格式)",
  "age": "number (可选，18-120)"
}

文档要求：
1. 包含请求示例（curl 命令）
2. 列出所有可能的响应状态码
3. 提供成功和失败的响应示例
4. 说明错误处理机制
5. 使用 Markdown 格式"
````

### 4. 温度参数调优

```python
# 创意性任务：温度设置较高（0.7-0.9）
response = openai.ChatCompletion.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "写一个科幻故事开头"}],
    temperature=0.8  # 更有创意
)

# 精确性任务：温度设置较低（0-0.3）
response = openai.ChatCompletion.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "将以下文本翻译为英文"}],
    temperature=0.2  # 更准确
)
```

## 四、常见场景 Prompt 模板

### 1. 代码审查

````markdown
请对以下代码进行全面审查：

```typescript
[代码片段];
```
````

审查维度：

1. **代码质量**：可读性、命名规范、代码风格
2. **潜在 Bug**：逻辑错误、边界情况、类型安全
3. **性能问题**：时间/空间复杂度、优化建议
4. **安全性**：SQL 注入、XSS、敏感信息泄露
5. **最佳实践**：设计模式、SOLID 原则

请按以下格式输出：

- 🔴 严重问题（必须修复）
- 🟡 建议改进（推荐修复）
- 🟢 优秀实践（值得保留）

````

### 2. 技术方案设计

```markdown
请设计一个 [系统名称] 的技术方案：

业务需求：
[需求描述]

技术约束：
- 编程语言：TypeScript
- 框架：NestJS + React
- 数据库：PostgreSQL
- 部署环境：Kubernetes

请提供：
1. 系统架构图（用文字描述）
2. 技术选型理由
3. 数据库表设计
4. API 接口设计
5. 关键技术难点及解决方案
6. 性能优化策略
7. 监控和告警方案
````

### 3. Bug 排查

```markdown
帮我排查以下 Bug：

问题描述：
[Bug 现象]

复现步骤：

1. [步骤1]
2. [步骤2]
3. [步骤3]

期望行为：
[期望结果]

实际行为：
[实际结果]

环境信息：

- Node.js 版本：v20.10.0
- 操作系统：macOS 14.0
- 相关依赖版本：[package.json]

请分析：

1. 可能的原因（按可能性排序）
2. 调试建议
3. 修复方案
```

### 4. 学习路径规划

```markdown
我想学习 [技术方向]，请为我制定一个 3 个月的学习计划：

当前水平：
[描述你的基础]

学习目标：
[描述你的目标]

可用时间：

- 工作日：每天 2 小时
- 周末：每天 4 小时

请提供：

1. 每周学习主题
2. 推荐学习资源（文档、视频、书籍）
3. 实战项目建议
4. 里程碑检查点
5. 常见坑点提醒
```

## 五、Prompt Engineering 进阶技巧

### 1. 负面提示（Negative Prompting）

明确告诉模型**不要**做什么。

```markdown
为初学者解释 React Hooks：

要求：

- 使用简单易懂的语言
- 提供代码示例
- ❌ 不要使用复杂的术语
- ❌ 不要假设读者有函数式编程背景
- ❌ 不要超过 500 字
```

### 2. 自我反思（Self-Reflection）

让模型检查和改进自己的输出。

```markdown
请完成以下任务：

1. 编写一个 TypeScript 函数，实现数组扁平化
2. 写完后，请自我检查：
   - 类型定义是否准确？
   - 是否处理了所有边界情况？
   - 代码是否符合最佳实践？
3. 如果发现问题，请给出改进版本
```

### 3. 多轮对话策略

```markdown
第一轮：
"我想开发一个博客系统，请列出核心功能模块。"

第二轮：
"针对你提到的'评论系统'，请详细设计数据库表结构。"

第三轮：
"考虑到高并发场景，这个表结构有什么优化建议？"
```

## 七、常见错误与避坑指南

### 错误 1：Prompt 过于简短

```markdown
❌ "优化代码"
✅ "优化以下代码的执行速度，当前运行时间为 2 秒，目标在 500ms 以内。约束：不能改变函数签名，不能引入新依赖。"
```

### 错误 2：忽略输出格式

```markdown
❌ "列出微服务的优缺点"
✅ "以表格形式列出微服务架构的优缺点，包含三列：维度、优点、缺点。维度包括：扩展性、维护性、部署、性能、团队协作。"
```

### 错误 3：一次性要求过多

```markdown
❌ "设计整个系统，包括架构、数据库、API、前端、测试、部署"

✅ 分步进行：
第一步："设计系统架构，包括模块划分和技术选型"
第二步："基于上述架构，设计数据库表结构"
第三步："设计核心 API 接口"
```

## 八、总结

Prompt Engineering 是一门实践性很强的技能，核心在于：

1. **清晰表达**：明确、具体、无歧义
2. **提供上下文**：给模型足够的背景信息
3. **结构化思维**：复杂任务拆解为简单步骤
4. **迭代优化**：不断测试和改进 Prompt
5. **善用模式**：掌握常见的设计模式（Few-Shot、CoT 等）

**实践建议**：

- 建立 Prompt 库，积累优秀模板
- 使用版本控制管理 Prompt
- 进行 A/B 测试，量化 Prompt 效果
- 关注模型更新，及时调整策略

掌握 Prompt Engineering，你将能更高效地利用 LLM 构建强大的 AI 应用！
