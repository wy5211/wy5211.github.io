上一篇我们把 Compose 配置完善到了生产级。现在来看看镜像本身有什么问题。

```bash
docker images order-api
# REPOSITORY   TAG   IMAGE ID       SIZE
# order-api    latest abc123def456   468MB
```

468MB。一个 Node.js API 项目，核心代码不到 1MB，镜像却有 468MB。每次部署要上传 468MB 的镜像，拉取也要几分钟。如果有 10 个类似的微服务，光镜像就占了近 5GB。

这篇我们一步步优化，每一步都对比镜像大小，让你看到每个优化手段的实际效果。

## 分析现状：镜像为什么这么大

先看看镜像的每一层有多大：

```bash
docker history order-api:latest
# IMAGE          CREATED       SIZE    COMMENT
# abc123def456   2 hours ago   468MB
# def456abc123   2 hours ago   298MB   COPY . .
# 789abc123def   2 hours ago   156MB   RUN npm install
# 123def456abc   2 hours ago   0B      EXPOSE 3000
# 456abc123def   2 hours ago   0B      WORKDIR /app
# 789def456abc   2 weeks ago   14MB    CMD ["node", "src/index.js"]
```

最大的是 `COPY . .`（298MB）和 `RUN npm install`（156MB）。298MB 里包含了 `node_modules`、测试文件、`.git` 目录、文档——一大堆运行时不需要的东西。156MB 是安装的依赖，里面包含了开发依赖（测试框架、代码检查工具等）。

## 第一步：.dockerignore 精简构建上下文

创建 `.dockerignore` 文件，排除不需要的东西：

```
node_modules
.git
.gitignore
.env
.env.local
npm-debug.log
coverage
.DS_Store
tests
*.md
.vscode
docker-compose*.yml
Dockerfile*
```

重新构建：

```bash
docker build -t order-api:step1 .
```

```
order-api:latest    468MB  (优化前)
order-api:step1     468MB  (加 .dockerignore)
```

镜像大小没变？对，`.dockerignore` 减少的是构建上下文的传输量（从 310MB 降到 8MB），构建速度快了很多，但 `COPY . .` 复制的文件少了，`node_modules` 本来就没被复制进来（因为 Dockerfile 里先 `COPY package.json` 再 `RUN npm install`），所以最终镜像大小没变。

不过构建速度的提升也是实打实的收益——从几十秒缩短到几秒。

## 第二步：不安装开发依赖

看看 `package.json` 里的依赖结构：

```json
{
  "dependencies": {
    "express": "^4.18.2",
    "pg": "^8.11.3",
    "ioredis": "^5.3.2",
    "zod": "^3.22.4"
  },
  "devDependencies": {
    "jest": "^29.7.0",
    "eslint": "^8.50.0",
    "nodemon": "^3.0.1",
    "typescript": "^5.2.2",
    "@types/express": "^4.17.21",
    "@types/pg": "^8.10.9"
  }
}
```

生产环境不需要 Jest、ESLint、Nodemon、TypeScript 编译器。修改 Dockerfile，只安装运行时依赖：

```dockerfile
FROM node:20-slim
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 3000
CMD ["node", "src/index.js"]
```

`--omit=dev`（或者旧版的 `--only=production`）告诉 npm 只安装 `dependencies` 里的包，跳过 `devDependencies`。

```
order-api:latest    468MB
order-api:step2     312MB  (不装开发依赖)
```

减少了 156MB。Jest、TypeScript、ESLint 加起来还真不小。

## 第三步：换 Alpine 基础镜像

```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 3000
CMD ["node", "src/index.js"]
```

```
order-api:latest    468MB
order-api:step3     185MB  (Alpine 基础镜像)
```

一步减了 127MB。Alpine 用的是 musl libc（约 2MB）而不是 Debian 的 glibc（约 30MB），系统工具也更精简。

## 第四步：多阶段构建

如果项目需要编译（比如 TypeScript），编译工具应该留在构建阶段，不进入最终镜像。

```dockerfile
# 构建阶段——包含所有工具
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build    # TypeScript 编译，产出 dist/ 目录

# 运行阶段——只保留运行时需要的东西
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev
COPY --from=builder /app/dist ./dist
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

关键变化：`COPY --from=builder /app/dist ./dist` 只从构建阶段复制编译产物。TypeScript 编译器、源代码、开发依赖全部留在构建阶段，不会进入最终镜像。

```
order-api:latest    468MB
order-api:step4     152MB  (多阶段构建)
```

如果你的项目不需要编译（纯 JavaScript），多阶段构建的收益会小一些，但仍然有价值——构建阶段的 `node_modules`（包含开发依赖，约 150MB）不会进入最终镜像。

## 第五步：合并 RUN 指令

每条 `RUN` 产生一个新层。虽然现在层的数量对运行时影响不大（Docker 会合并只读层），但减少层数有助于减小镜像的元数据大小。

更重要的是，有些 `RUN` 会留下中间产物。比如 `apk add` 会留下缓存，`npm install` 会留下 `.npm` 缓存。如果不在同一条 `RUN` 里清理，缓存会永远留在那一层里。

```dockerfile
FROM node:20-alpine
WORKDIR /app

# 在一条 RUN 里完成：安装依赖 + 清理缓存
COPY package.json package-lock.json ./
RUN npm ci --omit=dev && npm cache clean --force

COPY . .

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

```
order-api:latest    468MB
order-api:step5     138MB  (合并 RUN + 清理缓存)
```

又减了 14MB。单个项目可能不明显，但 `npm cache clean --force` 清掉的是 npm 的下载缓存，依赖越多节省越多。

## 优化效果汇总

| 步骤     | 操作          | 镜像大小  | 减少                |
| -------- | ------------- | --------- | ------------------- |
| 初始     | node:20-slim  | 468MB     | -                   |
| 1        | .dockerignore | 468MB     | 0MB（构建速度提升） |
| 2        | --omit=dev    | 312MB     | 156MB               |
| 3        | Alpine        | 185MB     | 127MB               |
| 4        | 多阶段构建    | 152MB     | 33MB                |
| 5        | 清理缓存      | 138MB     | 14MB                |
| **总计** |               | **138MB** | **330MB（-70%）**   |

从 468MB 到 138MB，缩小了 70%。构建速度从 45 秒降到 12 秒（利用缓存时只需 2-3 秒）。

## Alpine 的坑

Alpine 很小，但它有一个著名的坑：**glibc 兼容性问题**。

Alpine 用的 C 标准库是 musl libc，不是大多数 Linux 发行版用的 glibc。某些 npm 包（尤其是包含原生模块的包，比如 `bcrypt`、`canvas`、`sharp`）在编译时可能链接了 glibc，在 Alpine 上运行会报 `Error: shared library not found`。

解决办法：

```dockerfile
# 方案一：在 Alpine 里重新编译原生模块
RUN apk add --no-cache python3 make g++
RUN npm rebuild

# 方案二：用 node:20-slim 代替 Alpine（兼容性更好，但体积大 50MB 左右）
FROM node:20-slim
```

方案二更省心。如果你不确定项目里的所有依赖是否兼容 Alpine，先用 `node:20-slim`。体积虽然比 Alpine 大一些，但兼容性有保障。

实际上，大多数纯 JavaScript 项目（Express、Koa、Fastify 等）在 Alpine 上完全没问题。只有依赖了原生模块（C/C++ 编写的 Node.js 扩展）时才需要考虑兼容性。

## distroless：极致安全的选择

Google 提供了一组叫 distroless 的基础镜像，比 Alpine 更极端——没有包管理器、没有 shell、没有文本编辑器，只有运行时和你的应用。

```dockerfile
# 构建阶段
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev
COPY . .

# 运行阶段——distroless
FROM gcr.io/distroless/nodejs20-debian12
COPY --from=builder /app /app
WORKDIR /app
EXPOSE 3000
CMD ["dist/index.js"]
```

```
order-api:step5    138MB  (Alpine)
order-api:distroless  82MB  (distroless)
```

更小，也更安全——攻击者即使攻入了容器，也没有 shell 可以用，没有 `curl`、`wget` 可以下载东西。

但代价也很明显：**调试困难**。没有 shell 意味着你不能 `docker exec -it xxx /bin/sh` 进入容器。出了问题只能看日志，或者临时用 Alpine 镜像替换来排查。

distroless 适合对安全性要求极高的生产环境。对于大多数项目，Alpine 是更好的平衡点——够小、够安全、还能调试。

## 镜像体积不是唯一指标

优化到这一步，有必要泼一盆冷水：**镜像体积不是追求的目标，稳定性和可调试性同样重要。**

一个 138MB 的 Alpine 镜像和一个 312MB 的 slim 镜像，在生产环境中的差异其实不大——存储成本可以忽略，拉取时间差几秒钟。但如果你为了缩小 50MB 而引入了 Alpine 的兼容性问题，或者用了 distroless 导致无法调试，那就得不偿失了。

实际的建议：

- **大多数项目**：`node:20-alpine` + `--omit=dev` + 多阶段构建，做到 150MB 左右就很好了。
- **有原生模块的项目**：`node:20-slim` 代替 Alpine，多 50MB 换来更好的兼容性。
- **安全敏感场景**：考虑 distroless，但要有调试预案。

## 用 dive 分析镜像层

如果你想深入分析镜像每一层的内容，可以用 [dive](https://github.com/wagoodman/dive) 工具：

```bash
# 安装 dive（macOS）
brew install dive

# 分析镜像
dive order-api:latest
```

dive 会显示每一层的文件变化，帮你找到哪些层占用了大量空间、哪些文件不该被打进去。它的界面分为左右两栏：左边是当前层的文件列表，右边是上一层的文件列表，高亮显示新增和删除的文件。

## 回顾

这篇我们对订单 API 的 Dockerfile 做了五步优化，从 468MB 缩小到 138MB：

- **.dockerignore**：排除不需要的文件，加快构建速度。
- **--omit=dev**：不安装开发依赖，节省约 156MB。
- **Alpine 基础镜像**：换用更小的系统，节省约 127MB。
- **多阶段构建**：编译工具留在构建阶段，节省约 33MB。
- **清理缓存**：在同一条 RUN 里清理 npm 缓存，节省约 14MB。

镜像够小够快了，但还有一个关键问题：容器里跑的是 root 用户，镜像里可能有已知漏洞。下一篇来处理安全和监控。
