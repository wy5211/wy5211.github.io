前几篇我们分别解决了 Docker 的几个核心问题：打包镜像（Dockerfile）、数据持久化（Volume）、容器通信（网络）。但到目前为止，每次启动订单 API 的完整技术栈，你都需要手动执行这一堆命令：

```bash
# 创建网络
docker network create order-network

# 创建 Volume
docker volume create order-pgdata

# 启动 PostgreSQL
docker run -d --name order-db \
  --network order-network \
  -v order-pgdata:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=order_secret_2025 \
  postgres:16

# 启动 API
docker run -d --name order-api \
  --network order-network \
  -p 3000:3000 \
  -e DATABASE_URL=postgres://postgres:order_secret_2025@order-db:5432/postgres \
  order-api:latest

# 启动 Nginx
docker run -d --name nginx \
  --network order-network \
  -p 80:80 \
  -v $(pwd)/nginx.conf:/etc/nginx/nginx.conf:ro \
  nginx:alpine
```

六条命令，参数又长又容易敲错。而且这只是三个服务，如果有 Redis、消息队列、监控……命令会更多。

更麻烦的是团队协作。新同事加入项目，你告诉他"先启动 PostgreSQL，再启动 API，注意网络要一致，Volume 要挂对"——口口相传，总有人会漏掉某个参数。

Docker Compose 就是来解决这个问题的。

## 从命令式到声明式

Docker Compose 的核心思路是：把所有服务的配置写在一个 YAML 文件里，然后一条命令启动全部服务。

```yaml
# docker-compose.yml
services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://postgres:order_secret_2025@db:5432/postgres
    depends_on:
      - db

  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD: order_secret_2025

volumes:
  pgdata:
```

启动全部服务：

```bash
docker compose up -d
```

一条命令。Docker Compose 会自动创建网络、创建 Volume、按依赖顺序启动所有服务。

停止并删除所有容器：

```bash
docker compose down
```

这就是"声明式"的威力——你告诉 Docker 你要什么（三个服务、什么配置），而不是一步步告诉它怎么做。

## docker-compose.yml 结构解析

一个 Compose 文件由三个顶层元素组成：`services`（服务）、`volumes`（数据卷）、`networks`（网络）。

**`services`** 定义了你要运行的容器。每个服务对应一个（或多个）容器。上面的例子中定义了两个服务：`api` 和 `db`。

`api` 服务：

- `build: .`：从当前目录的 Dockerfile 构建镜像，而不是用现成的镜像。
- `ports: ["3000:3000"]`：端口映射，和 `docker run -p` 一样。
- `environment`：环境变量，和 `docker run -e` 一样。
- `depends_on: [db]`：依赖关系——先启动 `db`，再启动 `api`。

`db` 服务：

- `image: postgres:16`：使用 Docker Hub 上的 PostgreSQL 16 镜像。
- `volumes: [pgdata:/var/lib/postgresql/data]`：挂载数据卷，和 `docker run -v` 一样。
- `environment`：设置 PostgreSQL 的超级用户密码。

**`volumes`** 声明数据卷。这里声明了一个叫 `pgdata` 的 Volume。如果不在这里声明，Docker 也会自动创建，但显式声明的好处是名称确定、便于管理。

**`networks`** 我们没有显式声明，但 Compose 默认会创建一个以项目名命名的 bridge 网络。所有服务都在这个网络里，可以通过服务名互相访问。所以 API 里用 `db:5432` 连接数据库，而不是 IP 地址。

## build 和 image 的区别

`build` 和 `image` 是两种不同的方式指定服务使用的镜像：

- `image: postgres:16`：直接使用 Docker Hub 上已有的镜像。
- `build: .`：从当前目录的 Dockerfile 构建镜像。

对于你自己的项目用 `build`，对于第三方服务（数据库、缓存等）用 `image`。

你也可以同时指定两者：

```yaml
services:
  api:
    build: .
    image: order-api:latest
```

这样 Compose 会先构建镜像并打上 `order-api:latest` 标签。好处是你可以在其他地方引用这个镜像（比如推送到仓库）。

`build` 还支持指定 Dockerfile 路径和构建参数：

```yaml
services:
  api:
    build:
      context: .
      dockerfile: Dockerfile.prod
      args:
        - NODE_ENV=production
```

## depends_on 的局限

`depends_on` 只保证启动顺序，不保证服务就绪。

```yaml
services:
  api:
    depends_on:
      - db
```

这表示"先启动 db，再启动 api"。但 db 启动了不代表 PostgreSQL 可以接受连接了——PostgreSQL 启动后还要做初始化（创建系统表、恢复 WAL 日志等），可能需要几秒钟。如果 API 在这几秒内尝试连接，仍然会失败。

解决方案有两个：

**方案一：在应用层面加重试逻辑。** 这是最推荐的做法。大多数数据库连接池都支持连接重试：

```javascript
// Node.js 的 pg 连接池自带重试
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  connectionTimeoutMillis: 10000, // 连接超时 10 秒
});
```

**方案二：用 healthcheck + depends_on condition。** 这是 Compose 的更高级用法：

```yaml
services:
  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5

  api:
    depends_on:
      db:
        condition: service_healthy
```

这样 Compose 会等 PostgreSQL 的健康检查通过后才启动 API。不过 healthcheck 的配置增加了复杂度，对于大多数项目，应用层面的重试就够了。

## 常用 Compose 命令

```bash
# 启动所有服务（-d 后台运行）
docker compose up -d

# 启动时重新构建镜像
docker compose up -d --build

# 查看服务状态
docker compose ps

# 查看某个服务的日志
docker compose logs api

# 实时跟踪日志
docker compose logs -f

# 停止所有服务（不删除容器）
docker compose stop

# 重新启动
docker compose start

# 停止并删除容器和网络（Volume 保留）
docker compose down

# 停止并删除容器、网络和 Volume（数据全没了，慎用）
docker compose down -v

# 重启某个服务
docker compose restart api

# 查看某个服务的配置（验证 Compose 文件是否正确）
docker compose config
```

`docker compose config` 特别有用。它会解析并验证你的 `docker-compose.yml` 文件，展示最终生效的配置。如果文件有语法错误，它会告诉你哪里有问题。

## 开发体验优化：Bind Mount 挂载源代码

在开发时，你希望改了代码后不用重新构建镜像就能看到效果。用 Bind Mount 把源代码目录挂载到容器里：

```yaml
services:
  api:
    build: .
    ports:
      - "3000:3000"
    volumes:
      - .:/app # 把当前目录挂载到 /app
      - /app/node_modules # 排除 node_modules（用容器内的）
    environment:
      - DATABASE_URL=postgres://postgres:order_secret_2025@db:5432/postgres
      - NODE_ENV=development
    depends_on:
      - db
```

这里有一个技巧：先挂载 `.` 到 `/app`（覆盖容器内的所有文件），再用匿名 Volume 挂载 `/app/node_modules`（恢复容器内构建好的 node_modules）。如果不这样做，宿主机的 `node_modules` 会覆盖容器内的，而宿主机的 node_modules 可能是 macOS 编译的，在 Linux 容器里跑不了。

配合 `nodemon` 之类的热重载工具，改了代码保存后 API 自动重启：

```bash
# package.json
{
  "scripts": {
    "dev": "nodemon src/index.js"
  }
}
```

```yaml
services:
  api:
    # ...
    command: npm run dev # 覆盖 Dockerfile 里的 CMD
```

## 用 .env 文件管理配置

把密码和配置写死在 `docker-compose.yml` 里不是好习惯——特别是你要把配置文件提交到 Git 仓库的时候。

用 `.env` 文件管理敏感配置：

```bash
# .env
POSTGRES_PASSWORD=order_secret_2025
DATABASE_URL=postgres://postgres:order_secret_2025@db:5432/postgres
API_PORT=3000
```

```yaml
# docker-compose.yml
services:
  api:
    ports:
      - "${API_PORT}:3000"
    environment:
      - DATABASE_URL=${DATABASE_URL}
    depends_on:
      - db

  db:
    environment:
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
```

Compose 会自动读取同目录下的 `.env` 文件。记得把 `.env` 加入 `.gitignore`，不要把密码提交到仓库。

## 项目命名

Docker Compose 默认用当前目录名作为项目名。项目名会影响容器名、网络名和 Volume 名。

```bash
# 默认项目名是目录名，比如 "order-api"
docker compose up -d

# 容器名会是 order-api-api-1、order-api-db-1
# 网络名会是 order-api_default
# Volume 名会是 order-api_pgdata

# 自定义项目名
docker compose -p myproject up -d
```

如果你在同一台机器上要跑多个相同项目的实例（比如多套测试环境），用 `-p` 指定不同的项目名来隔离它们。

## 回顾

这篇我们用 Docker Compose 把前几篇学的所有东西整合在了一起：

- **声明式配置**：把所有服务的定义写在 `docker-compose.yml` 里，一条 `docker compose up -d` 启动全部。
- **自动网络**：Compose 自动创建网络，服务之间用名字通信。
- **自动 Volume**：在 `volumes` 部分声明，数据持久化。
- **依赖管理**：`depends_on` 控制启动顺序，配合 healthcheck 等待服务就绪。
- **开发体验**：Bind Mount 挂载源代码 + nodemon 热重载。
- **配置分离**：`.env` 文件管理敏感配置。

现在的 `docker-compose.yml` 已经能满足日常开发了。但生产环境还需要更多东西：健康检查、资源限制、日志配置、服务扩展。下一篇我们把这些生产级特性加上去。
