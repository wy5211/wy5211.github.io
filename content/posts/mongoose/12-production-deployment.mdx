在上一篇文章中，我们为 CMS 系统编写了完整的测试用例，确保数据层逻辑的正确性。有了测试作为安全网，我们可以更自信地将应用部署到生产环境。但生产环境不同于开发环境，需要考虑稳定性、安全性、可观测性等多方面因素。

想象这样一个场景：你的 CMS 系统开发完成，准备上线。选择哪个 MongoDB 服务商？连接字符串怎么配置？数据库连接池要多大？如何监控系统运行状态？出问题时怎么快速恢复？这些都是生产部署必须面对的问题。

MongoDB Atlas 是官方推荐的云数据库服务，提供自动备份、监控告警、自动扩缩容等功能，是生产部署的首选。让我们看看如何将 CMS 系统部署到生产环境。

## MongoDB Atlas 部署

MongoDB Atlas 提供免费版和付费版，免费版适合开发测试，付费版提供更多功能和更高性能。创建一个 Atlas 集群很简单：

在 Atlas 控制台创建集群后，会得到一个连接字符串，格式如下：

```typescript
mongodb+srv://username:password@cluster0.example.net/cms?retryWrites=true&w=majority
```

这个连接字符串包含了所有必要的信息：协议、用户名、密码、主机地址、数据库名，以及一些连接选项。

在生产环境中，应该将连接字符串存储在环境变量中，而不是硬编码在代码里：

```typescript
// .env.production
MONGODB_URI=mongodb+srv://username:password@cluster0.example.net/cms?retryWrites=true&w=majority
NODE_ENV=production
```

然后在代码中读取环境变量：

```typescript
const uri = process.env.MONGODB_URI;

if (!uri) {
  throw new Error("MONGODB_URI 环境变量未定义");
}

await mongoose.connect(uri);
```

这样敏感信息不会暴露在代码仓库中，也更安全。

## 连接池优化

数据库连接是有限资源，合理配置连接池对应用性能很重要。连接池太大浪费资源，太小会导致请求等待。对于典型的 Web 应用，以下配置是合理的起点：

```typescript
await mongoose.connect(uri, {
  maxPoolSize: 10, // 最大连接数，默认 100
  minPoolSize: 5, // 最小连接数，默认 0
  socketTimeoutMS: 45000, // Socket 超时时间，默认 45000ms
  serverSelectionTimeoutMS: 5000, // 服务器选择超时，默认 30000ms
  heartbeatFrequencyMS: 10000, // 心跳频率，默认 10000ms
  retryWrites: true, // 重试写入操作
  retryReads: true, // 重试读取操作
});
```

maxPoolSize 应该根据应用的并发请求量来设置。对于 Node.js 应用，10 个连接通常足够，因为 Node.js 是单线程的，同一时间处理的请求数有限。设置太大反而会浪费数据库资源。

minPoolSize 确保连接池中最少保持的连接数，设置为正数可以避免应用启动时需要频繁建立连接。

## 连接事件监听

生产环境需要监控数据库连接状态，及时发现问题：

```typescript
mongoose.connection.on("connected", () => {
  console.log("MongoDB 已连接");
});

mongoose.connection.on("error", (err) => {
  console.error("MongoDB 连接错误:", err);
});

mongoose.connection.on("disconnected", () => {
  console.warn("MongoDB 已断开");
});

// 监控连接池状态
setInterval(() => {
  const state = mongoose.connection.readyState;
  const states = ["disconnected", "connected", "connecting", "disconnecting"];
  console.log(`连接状态: ${states[state]}`);
}, 60000); // 每分钟检查一次
```

readyState 表示连接的当前状态：0 表示断开，1 表示已连接，2 表示连接中，3 表示断开中。定期检查这个状态可以帮助你了解应用是否正常连接到数据库。

## 优雅关闭

当应用需要重启或关闭时，应该先关闭数据库连接，确保正在进行的数据操作完成：

```typescript
async function gracefulShutdown(signal: string) {
  console.log(`收到 ${signal} 信号，准备关闭...`);

  try {
    // 停止接受新的 HTTP 请求（假设你用的是 Express）
    // server.close()

    // 关闭数据库连接
    await mongoose.connection.close();
    console.log("MongoDB 连接已关闭");

    process.exit(0);
  } catch (error) {
    console.error("关闭时出错:", error);
    process.exit(1);
  }
}

// 监听系统信号
process.on("SIGTERM", () => gracefulShutdown("SIGTERM"));
process.on("SIGINT", () => gracefulShutdown("SIGINT"));
```

SIGTERM 信号通常由容器编排系统（如 Kubernetes）或进程管理器（如 PM2）发送，SIGINT 信号则是用户按 Ctrl+C 时触发。处理这些信号可以让应用优雅地关闭，而不是突然终止。

## 日志策略

生产环境需要良好的日志记录，帮助排查问题和监控系统运行状态。可以使用 Winston 或 Pino 等日志库：

```typescript
import winston from "winston";

const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || "info",
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  transports: [
    new winston.transports.File({ filename: "error.log", level: "error" }),
    new winston.transports.File({ filename: "combined.log" }),
  ],
});

if (process.env.NODE_ENV !== "production") {
  logger.add(
    new winston.transports.Console({
      format: winston.format.simple(),
    })
  );
}

// 记录慢查询
mongoose.set("debug", (collectionName, method, query, doc) => {
  logger.info(`${collectionName}.${method}`, { query, doc });
});
```

生产环境建议将日志输出到文件或日志服务，而不是控制台，方便后续分析和检索。

## 监控与告警

MongoDB Atlas 提供了丰富的监控指标，可以在控制台实时查看：

- 连接数：当前活跃连接数，接近 maxPoolSize 时需要扩容
- 操作吞吐量：每秒的读写操作数，评估系统负载
- 查询性能：慢查询的执行时间，优化索引设计的依据
- 存储使用：磁盘占用情况，避免空间不足
- CPU 使用率：数据库实例的 CPU 负载

除了 Atlas 控制台，还可以使用 APM（Application Performance Monitoring）工具，如 DataDog、New Relic，来监控应用的整体性能。

设置告警规则很重要，可以在问题影响到用户前及时发现：

```typescript
// Atlas 告警建议配置
// 1. 连接数超过 maxPoolSize 的 80%
// 2. 慢查询超过 1 秒
// 3. CPU 使用率超过 80%
// 4. 存储使用率超过 85%
// 5. 副本集节点掉线
```

## 备份与恢复

数据备份是生产环境的最后一道防线。MongoDB Atlas 提供自动备份功能：

- 连续备份：每分钟备份一次操作日志，可以恢复到任意时间点
- 快照备份：每天定期备份完整数据，保留指定天数
- 按需备份：在重要操作前手动创建快照

恢复数据时，可以选择恢复到新集群（不影响生产）或覆盖现有集群（需谨慎）。

```bash
# Atlas 建议备份策略
# 1. 启用连续云备份，保留 7 天
# 2. 每日凌晨自动快照，保留 30 天
# 3. 发布重大版本前手动快照
```

## 安全最佳实践

生产环境的安全性不容忽视：

1. 网络隔离：在 Atlas 中配置 IP 白名单，只允许应用服务器的 IP 访问数据库

2. 认证授权：为应用创建专用的数据库用户，只授予必要的权限

3. TLS 加密：确保连接使用 TLS 协议加密，防止数据在传输过程中被窃听

4. 审计日志：启用 Atlas 的审计日志功能，记录所有数据库操作，便于事后追踪

```typescript
// 生产环境连接字符串安全配置
mongodb+srv://app_user:password@cluster0.example.net/cms?\
  ssl=true&\
  authSource=admin&\
  retryWrites=true&\
  w=majority
```

## 性能优化建议

生产环境的性能优化是个持续的过程，这里有几个实用建议：

1. 使用索引：为常用查询字段创建合适的索引，定期检查慢查询日志

2. 投影优化：只查询需要的字段，减少网络传输和内存占用

3. 连接池调优：根据实际负载调整连接池大小，监控连接等待时间

4. 读写分离：对于读多写少的应用，使用读写分离减轻主节点压力

```typescript
// 读写分离配置
await mongoose.connect(uri, {
  readPreference: "secondaryPreferred", // 优先读从节点
  readConcern: { level: "majority" }, // 读取已确认的数据
});
```
