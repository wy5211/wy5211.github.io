凌晨两点，监控报警把你叫醒——数据库连接数爆了。登录一看，PostgreSQL 的 100 个连接全部占满，新请求全部报 "too many connections"。你排查了一圈发现，Next.js 开发环境的热重载每次都重新执行模块级代码，`new PrismaClient()` 被调了几十次，每次都创建新的连接池，旧的从没释放。

前八篇我们掌握了 Prisma 的核心能力——Schema 设计、CRUD、关联查询、迁移、事务、类型安全、性能优化。但 Prisma 从来不是孤立运行的，它必须嵌入到你的项目框架里。这篇解决的就是集成问题：Next.js 全栈架构和 Hono API 分离架构下，Prisma 各自怎么接、连接怎么管、错误怎么处理。

## 两种架构，两种思路

你的博客系统有两种常见的部署方式。

第一种是 Next.js 全栈。Server Components 直接查数据库，Server Actions 处理表单提交，ISR 做页面级缓存。好处是开发快、代码少、SSR 天然支持 SEO。坏处是前后端耦合在同一个项目里，如果将来需要给移动端提供 API，还得另外写一层。

第二种是 Hono API 分离。后端用 Hono 搭建独立的 REST API，前端用 Next.js 或任意框架消费 API。好处是前后端独立部署、独立扩展，一套 API 可以同时服务 Web、App、小程序。坏处是多了一层网络通信，架构复杂度更高。

两种架构对 Prisma 的使用方式差异很大。Next.js 全栈里 Prisma 直接跑在 Server Components 和 Server Actions 里，组件和数据库之间没有任何中间层；Hono API 里 Prisma 封装在路由处理器中，通过 HTTP 暴露出去，前端通过 `fetch` 消费。

选择哪种架构不是一个纯粹的技术决策，它取决于你的团队结构、产品规划和部署环境。如果你的博客只有一个 Web 端，而且团队规模在 3 人以内，Next.js 全栈的效率碾压 API 分离。如果你的产品计划同时出 Web 和 App，或者前后端由不同的人负责，API 分离就是必须的。接下来分别展开两种架构的实现细节。

## Next.js App Router 集成

Next.js App Router 是目前 Prisma 最常见的宿主框架。Server Components 让你可以在组件里直接查数据库，Server Actions 让你可以在服务端处理表单提交，ISR 让你可以做页面级缓存。三个能力组合起来，开发效率非常高。但连接管理是第一个要搞对的事，搞不对就会出现开头那个"连接数爆满"的惨剧。

### Prisma 单例：解决开发环境连接泄漏

开头的报警场景不是虚构的，这是 Next.js + Prisma 最经典的坑。Next.js 开发模式下，Fast Refresh 会反复重新执行模块级代码。如果你在文件顶层直接 `new PrismaClient()`，每次热重载都会创建一个新实例，每个实例维护自己的连接池，旧的不会释放。跑了几次热重载之后，连接池数量就会翻倍增长，很快打满数据库的连接上限。

解决办法是利用 `globalThis` 做单例缓存——它在 Node.js 进程的生命周期内不会被重置，即使模块被热重载了，缓存的实例还在。

```typescript
// lib/prisma.ts
import { PrismaClient } from "@prisma/client";

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined;
};

export const prisma =
  globalForPrisma.prisma ??
  new PrismaClient({
    log: process.env.NODE_ENV === "development" ? ["query"] : [],
  });

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}
```

逻辑很直白：如果 `globalThis` 上已经挂了 PrismaClient 实例就复用，否则创建新的。开发环境下把实例挂到 `globalThis` 上，下次热重载时就能复用，不管 Fast Refresh 触发多少次，PrismaClient 只有一个。生产环境不需要这层缓存——Vercel 的 Serverless 函数每次冷启动都是新进程，模块级变量天然是单例。

`log: ["query"]` 在开发环境下打印每条 SQL，方便你观察 Prisma 实际发给数据库的查询。生产环境关掉，避免日志量爆炸。第8篇详细讲过查询日志的使用方式，这里不再重复。

有一点值得注意：这个单例模式不是 Prisma 独有的，几乎所有在 Next.js 开发环境下使用数据库连接的库都需要它——Drizzle 用 `postgres-js` 时也要做同样的事。核心思路都是一样的：开发环境用 `globalThis` 缓存，生产环境不需要。你可以把这段代码当作模板，任何需要管理长连接的资源都可以套用这个模式。

### Server Components 里直接查文章列表

App Router 的 Server Components 是 async 函数，可以直接 `await` 数据库查询。不需要 API 路由，不需要序列化反序列化，查出来的数据直接渲染成 HTML。这是 Next.js 全栈架构最大的卖点——组件就是数据获取的地方，数据获取和渲染在同一个作用域内完成。

对比传统的方式：以前你需要写一个 API 路由 `/api/posts`，在里面查数据库、序列化成 JSON 返回；前端组件用 `fetch("/api/posts")` 拿到数据、反序列化、再渲染。中间有两次序列化/反序列化，还有一个网络往返。Server Components 把这些全部省掉了——数据在服务端直接查出来、直接渲染成 HTML，发送给客户端的是完整的 HTML 而不是 JSON。客户端拿到 HTML 就能直接展示，首屏速度更快，SEO 也更好。

列表页的核心查询如下。`revalidate = 300` 让页面做 ISR（Incremental Static Regeneration），首次访问时渲染并缓存，300 秒内的后续访问直接返回缓存，300 秒后的下一次访问触发后台重新渲染。对博客列表页来说，5 分钟的延迟完全可接受。

```typescript
// app/posts/page.tsx
export const revalidate = 300; // ISR：每 5 分钟重新生成

export default async function PostsPage() {
  const posts = await prisma.post.findMany({
    where: { status: "PUBLISHED" },
    select: {
      id: true,
      title: true,
      slug: true,
      excerpt: true,
      publishedAt: true,
      author: { select: { name: true } },
      _count: { select: { comments: true } },
    },
    orderBy: { publishedAt: "desc" },
    take: 20,
  });
  // 渲染 JSX...
}
```

`select` 只取了渲染需要的字段——文章列表不需要 `content`（可能好几 KB），查了也是浪费带宽和内存。`_count` 用第4篇讲过的方式统计评论数，避免 N+1 查询。整个查询的返回类型由 TypeScript 自动推断，你选了 `id`、`title`、`slug`，类型里就只有这几个字段，精确到字段级别。

这也是第7篇讲的"查询决定类型"的核心优势——Prisma 的类型系统会根据你的 `select` 和 `include` 推导出精确的返回类型。改了查询，类型自动变，不用手动同步。在 Server Components 里这个优势尤其明显，因为组件的 props 类型直接从查询结果推导，从数据库到 UI 的类型链路是完全闭合的。

### 文章详情页：关联查询 + ISR

详情页需要文章内容、作者信息、标签、评论列表。用第4篇学过的 `include` 组装关联数据。详情页和列表页有一个重要区别：列表页用 `select` 精确选取字段，详情页用 `include` 加载关联。原因很简单——列表页只需要平铺的字段，详情页需要嵌套的关联数据（作者对象、标签数组、评论数组）。`select` 和 `include` 不能同时用在一个查询里（Prisma 的设计约束），所以当你需要关联数据时，`include` 是唯一的选择。

```typescript
// app/posts/[slug]/page.tsx
export const revalidate = 300;

export default async function PostDetailPage({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;

  const post = await prisma.post.findUnique({
    where: { slug },
    include: {
      author: { select: { name: true, avatarUrl: true } },
      tags: { select: { tag: { select: { name: true, slug: true } } } },
      comments: {
        where: { parentId: null },
        select: {
          id: true,
          content: true,
          createdAt: true,
          author: { select: { name: true, avatarUrl: true } },
        },
        orderBy: { createdAt: "desc" },
        take: 50,
      },
    },
  });

  if (!post) notFound();
  // 渲染 JSX...
}
```

`include` 一次查询带上了作者、标签和评论。评论通过 `where: { parentId: null }` 只查顶层评论，嵌套回复可以单独加载。`take: 50` 防止热门文章拖慢查询。如果评论量很大（几百条以上），建议像第8篇提到的那样拆分查询——文章和评论分开查，评论做分页，避免一次查询返回过多数据。

一个常见的问题是：为什么不用一个大的 JOIN 把所有数据一次查出来？因为那样会产生笛卡尔积——一篇有 10 条评论和 5 个标签的文章，JOIN 结果是 50 行。Prisma 的 `include` 内部会拆分成多次查询来避免这个问题，你不需要手动处理。

还有一种选择是把详情页设为动态渲染——每次请求都查数据库，不做缓存。导出 `export const dynamic = "force-dynamic"` 就行。适合评论量大、需要实时的场景。但大多数博客的评论频率没那么高，ISR 的 5 分钟延迟完全可以接受，还能大幅减轻数据库压力。

### Server Actions：创建文章

Server Actions 是 Next.js 处理数据变更的推荐方式。它在服务端执行，不需要手动写 API 路由，也不需要 `fetch` 调用。你可以把它理解为一个"自动处理 HTTP 请求的函数"——Next.js 帮你处理了请求解析、响应序列化、错误处理这些样板代码，你只需要关注业务逻辑。

结合第7篇的 Zod 验证和第3篇的嵌套创建，一个完整的"创建文章"Server Action 如下。这段代码里的类型安全链路是完整的：Zod Schema 校验运行时数据（第7篇），Prisma 的 `data` 参数受 `UserCreateInput` 约束（编译时），`select` 控制返回字段（编译时），TypeScript 贯穿全程。

```typescript
// app/actions.ts
"use server";
import { prisma } from "@/lib/prisma";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
import { z } from "zod";

const createPostSchema = z.object({
  title: z.string().min(1).max(255),
  content: z.string().min(1),
  tagIds: z.array(z.number().int().positive()).optional(),
});

export async function createPost(formData: FormData) {
  const parsed = createPostSchema.safeParse({
    title: formData.get("title"),
    content: formData.get("content"),
    tagIds: formData
      .getAll("tagId")
      .map(Number)
      .filter((n) => !isNaN(n)),
  });
  if (!parsed.success) return { error: parsed.error.flatten().fieldErrors };

  const slug = parsed.data.title
    .toLowerCase()
    .replace(/\s+/g, "-")
    .replace(/[^\w-]/g, "")
    .slice(0, 100);

  const post = await prisma.post.create({
    data: {
      title: parsed.data.title,
      slug,
      content: parsed.data.content,
      status: "DRAFT",
      authorId: 1,
      tags: parsed.data.tagIds
        ? {
            create: parsed.data.tagIds.map((id) => ({
              tag: { connect: { id } },
            })),
          }
        : undefined,
    },
  });

  revalidatePath("/posts"); // ISR 缓存失效
  redirect(`/posts/${post.slug}`);
}
```

`safeParse` 校验输入，失败时返回字段级错误信息。`tags` 用嵌套 `create` 一次性建立文章和标签的关联，不需要先查再插。`revalidatePath("/posts")` 让列表页的 ISR 缓存失效，下次访问时重新渲染。

一个容易踩的坑：`redirect` 必须在 `revalidatePath` 之后调用。Next.js 的 `redirect` 会抛出一个特殊的错误来中断执行流程，如果你把它放在前面，缓存失效逻辑根本不会执行。这是 Next.js 文档里提到过的已知行为，坑过不少人。

还有一点：Server Actions 目前没有原生的"全局中间件"机制。如果你需要在每个 Action 前做认证检查、限流或者日志记录，你只能在每个 Action 函数开头手动调用。这和 Hono 的中间件形成了鲜明对比——Hono 里一行 `app.use("*", authMiddleware)` 就能拦截所有路由。如果你的项目有大量需要认证的操作，Server Actions 的重复代码会很快积累。一个缓解办法是把认证检查封装成高阶函数，用 `withAuth(async () => { ... })` 包装每个 Action。

## Hono API 集成

如果你的博客需要同时服务 Web 端和移动端，或者前后端团队独立工作，把 Prisma 封装在 Hono API 里是更灵活的选择。Hono 是一个新兴的 Web 框架，体积小、速度快、API 设计直觉。它最早为 Cloudflare Workers 设计，后来扩展到支持所有 JavaScript 运行时。在 TypeScript 生态里，Hono 已经成为 Express 的主要替代方案之一，特别适合构建轻量级的 API 服务。

和 Next.js 全栈相比，Hono API 的优势在于关注点分离。你的 API 代码不包含任何 UI 逻辑，纯粹处理数据——接收请求、校验参数、查询数据库、返回 JSON。这使得 API 可以独立测试、独立部署、独立扩展。当你的移动端团队需要接口时，他们只需要对接 API 文档，不需要理解你的前端代码。

### 基本设置

Hono 比 Express 轻量得多，中间件机制也更直觉。它的路由定义方式类似 Koa——链式调用，支持路径参数、中间件组合和类型推断。基本设置只需要几行代码：初始化 PrismaClient、创建 Hono 实例、挂载 CORS 中间件。`cors` 只对 `/api/*` 路径生效，生产环境把 `origin` 改成实际域名。

Hono 的另一个优势是运行时无关。同样的代码可以跑在 Node.js、Bun、Deno、Cloudflare Workers 上，不需要改任何逻辑。这意味着如果你的项目将来要从 Node.js 迁移到 Edge，Hono 这层不用动，只需要处理 Prisma 的 Edge 兼容性问题。

```typescript
// src/index.ts
import { Hono } from "hono";
import { cors } from "hono/cors";
import { PrismaClient } from "@prisma/client";

export const prisma = new PrismaClient();
const app = new Hono();
app.use("/api/*", cors({ origin: ["http://localhost:3000"] }));
```

### 文章 CRUD API

读取接口的写法。列表接口用 `Promise.all` 并行执行 `findMany` 和 `count`，分页参数做了边界处理防止全表扫描——`page` 至少为 1，`limit` 限制在 1-50 之间。

```typescript
// GET /api/posts - 文章列表（带分页）
posts.get("/api/posts", async (c) => {
  const page = Math.max(1, Number(c.req.query("page")) || 1);
  const limit = Math.min(50, Math.max(1, Number(c.req.query("limit")) || 10));

  const [items, total] = await Promise.all([
    prisma.post.findMany({
      where: { status: "PUBLISHED" },
      select: {
        id: true,
        title: true,
        slug: true,
        excerpt: true,
        author: { select: { name: true } },
      },
      orderBy: { publishedAt: "desc" },
      skip: (page - 1) * limit,
      take: limit,
    }),
    prisma.post.count({ where: { status: "PUBLISHED" } }),
  ]);

  return c.json({
    data: items,
    pagination: { page, limit, total, totalPages: Math.ceil(total / limit) },
  });
});
```

详情接口的 `include` 只选必要字段——`author` 只取 `name` 和 `avatarUrl`，避免把密码哈希泄露到响应里。通过 `id` 参数查文章，`isNaN` 校验防止非法输入。

分页接口的响应格式值得说一下。`pagination` 对象里包含 `page`、`limit`、`total`、`totalPages` 四个字段，前端拿到后可以直接渲染分页组件。`totalPages` 通过 `Math.ceil(total / limit)` 计算，避免前端重复计算。这种响应格式在第7篇的 API 响应类型里也用过，保持一致性对前后端协作很重要。

写入操作的模式相同：Zod 校验输入，Prisma 执行操作，错误统一处理。创建接口用了第7篇学的 `zValidator` 中间件——请求体先过校验，不合法直接返回 400 和字段级错误，合法数据才进入业务逻辑。核心代码如下。

```typescript
// POST /api/posts - 创建文章（含 Zod 验证）
const createPostSchema = z.object({
  title: z.string().min(1).max(255),
  content: z.string().min(1),
  tagIds: z.array(z.number().int().positive()).optional(),
});

posts.post(
  "/api/posts",
  zValidator("json", createPostSchema, (result, c) => {
    if (!result.success) {
      return c.json(
        {
          errors: result.error.issues.map((i) => ({
            field: i.path.join("."),
            message: i.message,
          })),
        },
        400
      );
    }
  }),
  async (c) => {
    const body = c.req.valid("json");
    try {
      const post = await prisma.post.create({
        data: {
          title: body.title,
          content: body.content,
          status: "DRAFT",
          authorId: 1,
          tags: body.tagIds
            ? {
                create: body.tagIds.map((id) => ({ tag: { connect: { id } } })),
              }
            : undefined,
        },
      });
      return c.json({ data: post }, 201);
    } catch (error) {
      return handlePrismaError(error, c);
    }
  }
);
```

更新（PUT）和删除（DELETE）的逻辑结构完全一样——校验参数、执行 Prisma 操作、try-catch 包裹。更新用 `prisma.post.update({ where: { id }, data: { ... } })`，删除用 `prisma.post.delete({ where: { id } })`。两者都包在 try-catch 里，错误交给下面讲的 `handlePrismaError` 统一处理。

### Prisma 错误码映射成 HTTP 状态码

Prisma 抛出的错误是 `PrismaClientKnownRequestError`，带一个 `code` 字段。把这些错误码映射成合适的 HTTP 状态码，前端就能根据状态码做对应处理。这是 API 架构里非常重要的一层——没有它，所有 Prisma 错误都会变成 500，前端根本无法区分"资源不存在"和"服务器炸了"。

Prisma 的错误码体系设计得比较完善，覆盖了从唯一约束冲突到连接超时的各种场景。日常开发中最常见的就那么几个：P2002（唯一约束冲突）、P2025（记录不存在）、P2003（外键约束失败）。其他的比如 P2014（必需关联缺失）、P2004（约束失败）偶尔也会遇到，但频率低得多。

```typescript
// src/errors.ts
import { Prisma } from "@prisma/client";
import type { Context } from "hono";

export function handlePrismaError(error: unknown, c: Context) {
  if (error instanceof Prisma.PrismaClientKnownRequestError) {
    const map: Record<string, { status: number; msg: string }> = {
      P2002: { status: 409, msg: "唯一约束冲突" }, // slug 重复等
      P2025: { status: 404, msg: "资源不存在" }, // 更新/删除不存在的记录
      P2003: { status: 400, msg: "关联资源不存在" }, // 外键约束失败
    };
    const entry = map[error.code];
    if (entry) return c.json({ error: entry.msg }, entry.status);
    return c.json({ error: "数据库操作失败" }, 500);
  }
  return c.json({ error: "服务器内部错误" }, 500);
}
```

P2002 发生在唯一约束冲突时——比如两篇文章的 slug 相同，前端收到 409 后可以提示用户修改标题再试。P2025 发生在你尝试更新或删除一个不存在的记录时。P2003 发生在外键约束失败时——比如创建文章时关联了一个不存在的标签 ID。错误消息里不暴露数据库表名和列名，避免给攻击者提供信息。

把错误处理抽成独立函数有两个好处：一是错误响应格式统一，前端只需要一套处理逻辑；二是将来 Prisma 加了新的错误码，只需要改这一个地方。如果你想做得更完善，还可以加上日志记录——在 `console.error` 里把完整的错误信息写进去，方便排查。

还有一个容易忽略的点：错误消息的措辞。上面的代码里用的是"唯一约束冲突"而不是 "Unique constraint failed on the fields: `slug`"。后者的错误消息暴露了数据库的字段名，如果直接返回给前端，攻击者就能知道你的数据库结构。错误消息要做到两点：对前端足够友好（能用来显示用户提示），对攻击者足够模糊（不泄露内部实现细节）。完整的错误信息应该只在服务端日志里记录。

## Edge 兼容方案

第8篇提到过 Prisma 在 Serverless 环境下的连接数问题，Edge 环境更极端。Cloudflare Workers、Vercel Edge Functions 这些运行时不支持 Node.js 的 TCP 连接，而 Prisma 的 Query Engine 是 Rust 编译的二进制文件，依赖 Node.js API。直接 `import { PrismaClient } from "@prisma/client"` 在 Edge 上跑不起来。你会得到类似 "Query engine library for current platform could not be found" 的错误。

这不是 Prisma 的 bug，而是 Edge 运行时和 Node.js 运行时之间的根本差异。Edge 运行时被设计为轻量、快速启动的执行环境，它不支持完整的 Node.js API——没有 `child_process`、没有文件系统、没有原生 TCP socket。Prisma 的 Query Engine 恰好依赖这些能力。

### 两种应对方案

第一种是用 `@prisma/client/edge` 导入路径。它使用一个 WebAssembly 编译的轻量级 Query Engine，不依赖 Node.js 的 `child_process` 和文件系统 API。初始化方式和标准导入一样，只是改了导入路径。但功能有折衷——比如交互式事务（`$transaction` 的回调模式）在某些 Edge 运行时上不可用，第6篇讲过的部分高级事务特性用不了。

第二种，也是更推荐的方案，是 Prisma Accelerate。它是一个托管的连接池代理服务，你的 Edge 函数通过 HTTPS 协议和 Accelerate 通信，Accelerate 再用 TCP 连到你的数据库。

```typescript
import { PrismaClient } from "@prisma/client/edge";
import { withAccelerate } from "@prisma/extension-accelerate";

const prisma = new PrismaClient({
  datasourceUrl: process.env.ACCELERATE_URL, // prisma://accelerate.prisma-data.net/?api_key=xxx
}).$extends(withAccelerate());
```

`prisma://` 协议替代了直连的 `postgresql://`。Accelerate 在全球边缘节点维护共享连接池，不管你有多少个 Edge 函数实例在跑，到数据库的连接数都是可控的。它还内置了查询缓存，重复的查询可以直接从边缘节点返回。局限在于它是付费服务，免费额度有限。如果你的项目预算紧张，可以考虑用 HTTP 驱动直连支持 HTTP 协议的数据库服务（如 Neon 的 serverless driver），绕过 Prisma 的连接管理。

### 为什么 Prisma 在 Edge 上比较麻烦

根本原因是 Prisma 的架构设计。Prisma Client 内嵌了一个 Query Engine——一个用 Rust 编译的二进制文件，负责把 Prisma 的查询 API 翻译成具体数据库的 SQL。这个二进制文件在运行时通过 `child_process` 启动，和你的应用进程通过 IPC 通信。Edge 运行时没有 `child_process`，也没有文件系统来存放这个二进制，所以标准的 Prisma Client 跑不起来。

WASM 版本去掉了对 Node.js API 的依赖，但在功能和性能上都有折衷——启动稍慢、不支持所有数据库特性、内存占用受限。如果你的项目必须跑在 Edge 上而且数据库操作比较复杂，可能需要重新评估技术选型——Drizzle 的 Edge 兼容性比 Prisma 好得多，因为它没有内嵌 Query Engine 的设计，直接用 JavaScript 生成 SQL，在任何 JavaScript 运行时上都能跑。

总结一下 Edge 场景的选型：如果你的项目只在 Node.js 环境下运行（Vercel Serverless、Railway、Fly.io、AWS Lambda），Prisma 用起来没有任何问题，不需要考虑 Edge 兼容性。如果你明确要部署到 Cloudflare Workers 或 Vercel Edge Functions，而且查询不复杂（基础 CRUD），Prisma 的 Edge 方案够用。如果你在 Edge 上要做复杂查询、事务、全文搜索，Drizzle 是更省心的选择。

## Trade-off 分析

两种架构没有绝对的优劣，只有适合不适合。决策的关键变量有三个：你的客户端数量、团队分工方式、部署环境。

**Next.js 全栈（Server Components + Server Actions）的开发效率确实高。** 一个 `async` 组件里直接 `await prisma.post.findMany()`，不需要写 API 路由、不需要序列化、不需要在两端定义类型。Server Actions 处理表单提交也是一行 `revalidatePath` 就搞定缓存失效。对于个人博客或小团队项目来说，这是最快出活的方案。

但全栈方案的代价是耦合。你的数据获取逻辑和 UI 组件混在一起，时间久了组件会越来越大。你需要主动把查询逻辑抽到独立的 `lib/queries.ts` 里，保持组件只负责渲染。Server Actions 也没有中间件层——认证、限流、日志这些横切关注点只能在每个 action 里重复写，不如 Hono 的中间件机制优雅。更关键的是，如果你将来要给 iOS/Android 提供接口，还是得另起一套 API。

**Hono API 分离的架构更灵活但更复杂。** 前后端独立部署意味着可以独立扩展——API 扛不住了加机器，前端不用动。一套 API 同时服务多个客户端，类型定义共享一份（参考第7篇的 Monorepo 方案）。Hono 的中间件机制让认证、限流、日志这些横切关注点有统一的处理位置。

代价是架构复杂度。你需要维护两个项目的构建和部署流程，前后端之间的接口变更需要同步更新类型定义。数据多了一次网络往返——前端先请求 HTML，再 fetch API 拿数据，比 Server Components 的直接查数据库多了一跳。如果你用 SSR，这一跳发生在服务端（Next.js 服务端 fetch Hono API），用户感知不到延迟，但服务端多了一层网络开销。

**Edge 兼容性是 Prisma 的短板。** 如果你的项目需要跑在 Cloudflare Workers 上，Prisma 的 Edge 方案要么依赖 Accelerate 付费服务，要么接受 WASM 版本的功能和性能折衷。如果你的部署目标是 Edge 并且对性能敏感，Drizzle 可能是更好的选择。如果你的项目部署在 Node.js 环境下（Vercel Serverless、Railway、Fly.io），Prisma 没有任何问题。

**连接管理是两种架构都要面对的问题。** Next.js 全栈要处理热重载时的连接泄漏（用 `globalThis` 单例），Hono API 要处理 Serverless 环境下的连接数爆炸（减小连接池或用代理）。第8篇讲过的 `connection_limit` 参数调优在两种架构里都用得上。如果你用的是 Vercel + Supabase，建议把 `connection_limit` 设为 2-3，Supabase 自带的 PgBouncer 会帮你管理上游连接。如果你用的是直连数据库，需要自己评估并发量和数据库连接上限之间的关系。

**我的建议：** 如果项目只有 Web 端、团队小、追求开发速度，用 Next.js 全栈。如果项目需要多端（Web + App）、团队有前后端分工、或者 API 需要独立扩展，用 Hono API 分离。不要同时用两种架构——同一个数据流不要一半走 Server Component 一半走 `fetch('/api/...')`，那样既多了一层复杂度，又丢失了类型推断。

第7篇我们实现了端到端类型安全，第8篇优化了查询性能，这篇解决了框架集成的问题。下一篇是系列最后一篇——把前面九篇的所有知识整合到一个完整的博客项目中，从 Monorepo 架构到部署上线，跑通整个链路。
