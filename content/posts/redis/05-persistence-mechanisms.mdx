## 前言

Redis 是一个内存数据库，所有数据都存储在内存中。这意味着一旦服务器重启或发生故障，所有数据都会丢失。

为了解决这个问题，Redis 提供了**持久化**机制，可以将内存中的数据保存到磁盘，在重启后恢复。

本文将带你深入了解：

- 🎯 Redis 持久化的两种方式：RDB 和 AOF
- 📊 两种方式的原理、优缺点和使用场景
- 💡 混合持久化机制（Redis 4.0+）
- 🔧 持久化配置和最佳实践
- 🛠️ 如何选择合适的持久化方案

---

## 持久化方式总览

Redis 提供了两种持久化方式：

| 特性           | RDB              | AOF              |
| -------------- | ---------------- | ---------------- |
| **全称**       | Redis Database   | Append Only File |
| **原理**       | 定时保存内存快照 | 记录每个写命令   |
| **文件大小**   | 小               | 大               |
| **恢复速度**   | 快               | 慢               |
| **数据完整性** | 可能丢失最近数据 | 更好             |
| **性能影响**   | fork 时有影响    | 每次写入都有影响 |
| **适用场景**   | 备份、灾难恢复   | 数据完整性要求高 |

**选择建议**：

- 只用于缓存 → 不持久化
- 可以接受少量数据丢失 → RDB
- 不能接受数据丢失 → AOF
- **推荐配置** → 混合持久化（Redis 4.0+）

## RDB 持久化

### 什么是 RDB？

RDB（Redis Database）是 Redis 默认的持久化方式。它通过**定时生成内存快照**的方式，将某个时刻的数据保存到磁盘。

### 工作原理

```
1. Redis 主进程 fork 子进程
2. 子进程将内存数据写入临时文件
3. 完成后用临时文件替换旧的 RDB 文件
4. 主进程继续处理命令
```

```bash
# RDB 文件位置
/var/lib/redis/dump.rdb

# 文件名可配置
dbfilename dump.rdb
dir /var/lib/redis
```

### 配置方式

在 `redis.conf` 中配置：

```bash
# RDB 配置
# save <seconds> <changes>
# 表示在 <seconds> 秒内如果有 <changes> 个 key 变化，则保存

save 900 1     # 900 秒（15 分钟）内至少 1 个 key 变化
save 300 10    # 300 秒（5 分钟）内至少 10 个 key 变化
save 60 10000  # 60 秒（1 分钟）内至少 10000 个 key 变化

# 如果不想使用 RDB，可以注释掉所有 save 行
# save ""

# RDB 文件名
dbfilename dump.rdb

# RDB 文件目录
dir /var/lib/redis

# 是否压缩 RDB 文件
rdbcompression yes

# RDB 文件是否校验
rdbchecksum yes
```

### 手动触发 RDB

```bash
# 生成 RDB 快照（阻塞）
redis-cli SAVE

# 生成 RDB 快照（异步，推荐）
redis-cli BGSAVE

# 查看最后一次 RDB 保存时间
redis-cli LASTSAVE
```

### RDB 的优缺点

#### 优点

✅ **紧凑性好**：RDB 文件是压缩的二进制文件，占用空间小

✅ **恢复速度快**：直接将文件加载到内存，不需要执行命令

✅ **对性能影响小**：fork 子进程进行持久化，主进程继续处理请求

✅ **适合备份**：RDB 文件非常适合用于灾难恢复

#### 缺点

❌ **可能丢失数据**：如果 Redis 异常终止，会丢失最后一次快照之后的数据

❌ **fork 开销**：fork 子进程时会有短暂的性能影响（数据量大时更明显）

❌ **不适合实时持久化**：无法做到秒级持久化

### RDB 使用场景

1. **备份和灾难恢复**

```bash
# 定期备份 RDB 文件
crontab -e
# 每天凌晨 2 点备份
0 2 * * * cp /var/lib/redis/dump.rdb /backup/dump-$(date +\%Y\%m\%d).rdb
```

2. **不需要持久化的缓存场景**

```bash
# 关闭 RDB 和 AOF，纯缓存使用
save ""
appendonly no
```

3. **数据可以丢失的场景**

```bash
# 配置较长的保存间隔
save 3600 1    # 1 小时
save 7200 10   # 2 小时
```

## AOF 持久化

### 什么是 AOF？

AOF（Append Only File）通过**记录每个写命令**来实现持久化。Redis 执行每个写命令后，将命令追加到 AOF 文件。

### 工作原理

```
1. Redis 执行写命令
2. 将命令追加到 AOF 缓冲区
3. 根据策略将缓冲区内容写入文件
4. Redis 重启时重新执行 AOF 文件中的命令
```

### 配置方式

在 `redis.conf` 中配置：

```bash
# 启用 AOF
appendonly yes

# AOF 文件名
appendfilename "appendonly.aof"

# AOF 文件目录
dir /var/lib/redis

# AOF 同步策略
appendfsync everysec

# 其他选项：
# appendfsync always    # 每次写命令都同步（最安全，但最慢）
# appendfsync everysec  # 每秒同步一次（推荐）
# appendfsync no        # 由操作系统决定（最快，但可能丢失最多数据）

# AOF 重写配置
no-appendfsync-on-rewrite no   # 重写时是否不同步
auto-aof-rewrite-percentage 100 # AOF 文件大小比上次重写增长的百分比
auto-aof-rewrite-min-size 64mb  # 触发重写的 AOF 文件最小大小

# AOF 加载时是否忽略最后可能损坏的命令
aof-load-truncated yes
```

### AOF 同步策略

| 策略         | 数据安全性 | 性能 | 推荐场景                 |
| ------------ | ---------- | ---- | ------------------------ |
| **always**   | 最高       | 最低 | 不能接受任何数据丢失     |
| **everysec** | 高         | 高   | **推荐**，平衡安全和性能 |
| **no**       | 低         | 最高 | 可以接受数据丢失         |

### AOF 重写

随着时间推移，AOF 文件会越来越大。AOF 重写可以压缩文件：

```bash
# 手动触发 AOF 重写（异步）
redis-cli BGREWRITEAOF

# 配置自动重写
auto-aof-rewrite-percentage 100  # 文件大小是上次重写的 2 倍时触发
auto-aof-rewrite-min-size 64mb   # 文件至少 64MB 才触发重写
```

**重写过程**：

```
1. Redis fork 子进程
2. 子进程读取当前内存数据，生成优化的 AOF 命令
3. 主进程继续处理命令，将新命令追加到缓冲区
4. 子进程完成后，合并缓冲区和新文件
5. 替换旧的 AOF 文件
```

### AOF 的优缺点

#### 优点

✅ **数据完整性高**：最多丢失 1 秒数据（everysec 模式）

✅ **可读性强**：AOF 文件是文本格式，可以查看和修改

✅ **容错性好**：即使 AOF 文件末尾损坏，也可以恢复

#### 缺点

❌ **文件大**：AOF 文件通常比 RDB 文件大

❌ **恢复慢**：需要重新执行所有命令

❌ **性能影响**：每次写命令都有开销（everysec 模式）

### AOF 使用场景

1. **不能接受数据丢失的场景**

```bash
appendonly yes
appendfsync everysec
```

2. **需要查看命令历史的场景**

```bash
# 查看 AOF 文件
tail -f /var/lib/redis/appendonly.aof
```

3. **数据审计场景**

AOF 文件记录了所有写命令，可以用于审计和分析。

## 混合持久化

### 什么是混合持久化？

Redis 4.0+ 引入了混合持久化，结合了 RDB 和 AOF 的优势：

- **RDB**：用于基础快照
- **AOF**：用于增量更新

### 工作原理

```
1. AOF 重写时，先生成 RDB 格式的文件
2. 将重写期间的新命令以 AOF 格式追加
3. 最终文件：RDB（开头）+ AOF（增量）
```

### 配置方式

在 `redis.conf` 中配置：

```bash
# 启用 AOF
appendonly yes

# 启用混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes
```

### 混合持久化的优势

✅ **文件小**：RDB 格式紧凑，占用空间小

✅ **恢复快**：RDB 部分加载快，AOF 部分只包含增量

✅ **数据完整**：结合了两种方式的优势

## 持久化选择指南

### 决策流程

```
                     Redis 数据需要持久化？
                           │
         ┌────────YES──────┴──────NO──────┐
         │                              │
         ▼                              ▼
   能接受数据丢失？              纯缓存场景
         │                        不持久化
    ┌────YES───┴────NO────┐
    │                    │
    ▼                    ▼
  使用 RDB            使用 AOF
（或关闭持久化）    （或混合持久化）
```

### 配置建议

#### 场景 1：纯缓存（数据可以丢失）

```bash
# redis.conf
save ""
appendonly no
```

#### 场景 2：可以接受少量数据丢失

```bash
# redis.conf
save 900 1
save 300 10
save 60 10000
appendonly no
```

#### 场景 3：不能接受数据丢失（推荐）

```bash
# redis.conf
save ""
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes  # Redis 4.0+
```

#### 场景 4：最高数据安全性

```bash
# redis.conf
save ""
appendonly yes
appendfsync always
aof-use-rdb-preamble no
```

## 最佳实践

### 1. 生产环境配置

```bash
# redis.conf

# 禁用 RDB
save ""

# 启用 AOF
appendonly yes
appendfilename "appendonly.aof"

# 每秒同步
appendfsync everysec

# 启用混合持久化
aof-use-rdb-preamble yes

# 自动重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

### 2. 定期备份

```bash
# 备份脚本
#!/bin/bash
DATE=$(date +%Y%m%d)
BACKUP_DIR="/backup/redis"
REDIS_DIR="/var/lib/redis"

# 备份 RDB
cp $REDIS_DIR/dump.rdb $BACKUP_DIR/dump-$DATE.rdb

# 备份 AOF
cp $REDIS_DIR/appendonly.aof $BACKUP_DIR/appendonly-$DATE.aof

# 压缩
gzip $BACKUP_DIR/dump-$DATE.rdb
gzip $BACKUP_DIR/appendonly-$DATE.aof

# 删除 7 天前的备份
find $BACKUP_DIR -name "*.gz" -mtime +7 -delete
```

### 3. 监控持久化

```typescript
// 持久化监控脚本
import Redis from "ioredis";

const redis = new Redis();

async function monitorPersistence() {
  const info = await redis.info("persistence");

  console.log("RDB 最后保存时间:", info.match(/rdb_last_save_time:(\d+)/)?.[1]);
  console.log("AOF 文件大小:", info.match(/aof_current_size:(\d+)/)?.[1]);
  console.log("AOF 重写状态:", info.match(/aof_rewrite_in_progress:(\d)/)?.[1]);
}

setInterval(monitorPersistence, 60000); // 每分钟检查
```

## 常见问题

### Q1: RDB 和 AOF 可以同时启用吗？

**A**: 可以，但通常不推荐：

- 同时启用会影响性能
- Redis 重启时会优先使用 AOF 文件
- 推荐使用混合持久化（Redis 4.0+）

### Q2: 如何从持久化文件恢复数据？

**A**:

```bash
# 1. 停止 Redis
redis-cli shutdown

# 2. 备份当前文件
cp /var/lib/redis/dump.rdb /var/lib/redis/dump.rdb.backup

# 3. 复制备份文件
cp /backup/dump-20260323.rdb /var/lib/redis/dump.rdb

# 4. 启动 Redis
redis-server /etc/redis/redis.conf
```

### Q3: AOF 文件损坏怎么办？

**A**:

```bash
# 1. 备份损坏的文件
cp appendonly.aof appendonly.aof.corrupted

# 2. 尝试修复
redis-cli --fix-aof appendonly.aof.corrupted

# 3. 或使用 redis-check-aof 工具
redis-check-aof --fix appendonly.aof
```

## 总结

本文我们深入学习了 Redis 的持久化机制：

✅ **RDB 持久化**

- 定时生成内存快照
- 文件小、恢复快
- 可能丢失数据

✅ **AOF 持久化**

- 记录每个写命令
- 数据完整性好
- 文件大、恢复慢

✅ **混合持久化**

- 结合 RDB 和 AOF 优势
- Redis 4.0+ 推荐配置

✅ **持久化选择**

- 纯缓存 → 不持久化
- 可接受数据丢失 → RDB
- 不能接受数据丢失 → AOF
- **生产环境推荐** → 混合持久化
