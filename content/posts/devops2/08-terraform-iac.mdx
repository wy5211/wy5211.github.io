前 7 篇我们把 FlowAPI 从部署到监控的完整链路搭了一遍。但有个问题一直没提：那些基础设施——VPC、子网、安全组、EC2 实例、RDS 数据库——都是怎么来的？

答案很尴尬：手动在 AWS 控制台上点出来的。

你花了两个下午在控制台上点来点去，好不容易把环境搭好了。然后有一天你需要搭一套测试环境，或者换个 AWS region 部署，又或者团队来了新人需要搭一套开发环境。你会发现：

**你搭不出第二套一样的环境。**

你忘了当时安全组开了哪些端口，忘了 RDS 用的是哪个子网组，忘了 VPC 的 CIDR 块是 `10.0.0.0/16` 还是 `10.1.0.0/16`。控制台上能看到的只有最终结果，看不到"为什么这么配"的决策过程。

这就是手动管理基础设施的三大问题：

- **不可重复**：换个人搭，结果不一样；自己搭第二遍，也不一样
- **不可审计**：谁在什么时候改了什么配置，没有记录
- **不可回滚**：改错了，不知道怎么恢复

解决方案其实很直观：**用代码来描述基础设施**。代码可以版本控制、可以 code review、可以重复执行。这就是 Infrastructure as Code（IaC），而 Terraform 是目前最主流的 IaC 工具。

## Terraform 是什么

Terraform 是 HashiCorp 开源的基础设施管理工具。你用一种叫 HCL（HashiCorp Configuration Language）的语言描述你想要什么基础设施，Terraform 帮你去云厂商的 API 上创建。

一个最简单的例子——你想要一台 EC2 实例：

```hcl
resource "aws_instance" "flowapi" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.small"
}
```

写完这段配置，运行 `terraform apply`，Terraform 就会去 AWS 创建这台实例。改了 `instance_type` 再 apply 一次，它会帮你升级配置。把这段代码删了再 apply，它会帮你销毁实例。

整个逻辑就这么简单：**你描述期望状态，Terraform 负责达成**。

Terraform 支持几乎所有主流云厂商——AWS、Azure、GCP、阿里云、腾讯云——通过 provider 机制。换云厂商只需要换 provider，配置语法是一样的。

## 核心概念

在动手写配置之前，先搞清楚 Terraform 的几个核心概念。

**Provider**：云厂商的插件。你要用 AWS 就装 AWS provider，要用阿里云就装阿里云 provider。Terraform 通过 provider 调用云厂商的 API。

**Resource**：一个具体的资源，比如一台 EC2 实例、一个 VPC、一个 RDS 数据库。resource 是 Terraform 配置的核心，你绝大部分时间都在写 resource。

**Variable**：变量。把可变的值（比如实例类型、区域、环境名）提取成变量，方便复用和切换环境。

**Output**：输出值。资源创建完之后，你想知道一些信息（比如实例的公网 IP、数据库的连接地址），用 output 输出。

**State**：状态文件。Terraform 把当前基础设施的实际状态记录在一个文件里（`terraform.tfstate`）。每次 apply 之前，它会对比配置文件和 state 的差异，算出需要做什么操作。这个文件极其重要，后面专门讲。

**Module**：模块。把一组相关的 resource 打包成一个可复用的单元，比如"一个完整的 VPC 模块"或"一个标准的 Web 服务模块"。

概念关系大概是这样：

```
Provider (AWS)
├── Resource: VPC
│   ├── Resource: 子网 (public)
│   ├── Resource: 子网 (private)
│   └── Resource: 安全组
├── Resource: EC2 实例
└── Resource: RDS 数据库

Variable → 注入到 Resource
Resource → 输出到 Output
State    → 记录所有 Resource 的实际状态
```

## 用 Terraform 搭建 FlowAPI 基础设施

接下来我们用 Terraform 创建 FlowAPI 需要的完整 AWS 基础设施：VPC、子网、安全组、EC2 实例、RDS 数据库。

先创建一个项目目录，结构如下：

```
terraform-flowapi/
├── main.tf          # 主配置，provider 和资源定义
├── variables.tf     # 变量定义
└── outputs.tf       # 输出值定义
```

### 定义变量

先在 `variables.tf` 里把可变的值抽出来：

```hcl
# variables.tf

# AWS 区域，默认用东京，国内访问延迟相对低
variable "aws_region" {
  description = "AWS 区域"
  type        = string
  default     = "ap-northeast-1"
}

# 环境名，用于给资源打标签和区分命名
# dev / staging / production
variable "environment" {
  description = "环境名称"
  type        = string
  default     = "dev"
}

# 项目名，用于资源命名前缀
variable "project_name" {
  description = "项目名称"
  type        = string
  default     = "flowapi"
}

# EC2 实例类型
# dev 环境用 t3.small 省钱，production 用 t3.medium
variable "instance_type" {
  description = "EC2 实例类型"
  type        = string
  default     = "t3.small"
}

# RDS 数据库实例类型
variable "db_instance_class" {
  description = "RDS 实例类型"
  type        = string
  default     = "db.t3.micro"
}

# 数据库用户名和密码
# 密码不要硬编码，后面通过环境变量传入
variable "db_username" {
  description = "数据库管理员用户名"
  type        = string
  default     = "flowapi_admin"
}

variable "db_password" {
  description = "数据库管理员密码"
  type        = string
  sensitive   = true
  # 没有默认值，必须在 apply 时传入或设置环境变量 TF_VAR_db_password
}
```

把密码这类敏感信息从代码里分离出来是基本操作。`sensitive = true` 标记这个变量是敏感的，Terraform 在输出日志里会自动遮盖它的值。实际使用时通过环境变量 `TF_VAR_db_password` 传入，不要写在代码里。

### 配置 Provider 和网络

`main.tf` 是主配置文件，内容最多，我们分块来看。先是 provider 和网络部分：

```hcl
# main.tf

# 告诉 Terraform 我们要用 AWS，并指定区域
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  # 后面讲到 remote state 的时候会在这里配置 backend
}

provider "aws" {
  region = var.aws_region
}

# ============================================================
# 网络基础设施：VPC、子网、Internet Gateway
# ============================================================

# VPC：虚拟私有云，相当于一个隔离的网络环境
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  # 这个 CIDR 块提供 65536 个 IP 地址，足够用
  enable_dns_support   = true
  enable_dns_hostnames = true
  # 开启 DNS 支持，RDS 需要通过 DNS 主机名连接

  tags = {
    Name        = "${var.project_name}-vpc-${var.environment}"
    Environment = var.environment
  }
}

# 公有子网：用于放需要公网访问的资源（比如 NAT Gateway、负载均衡器）
resource "aws_subnet" "public" {
  count                   = 2
  # count = 2 表示创建两个公有子网，分布在不同可用区
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.${count.index}.0/24"
  availability_zone       = "${var.aws_region}${count.index == 0 ? "a" : "c"}"
  # ap-northeast-1a 和 ap-northeast-1c，两个可用区做高可用
  map_public_ip_on_launch = true
  # 自动分配公网 IP

  tags = {
    Name = "${var.project_name}-public-${count.index}"
  }
}

# 私有子网：用于放后端服务和数据库，不直接暴露在公网
resource "aws_subnet" "private" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 10}.0/24"
  # 10.0.10.0/24 和 10.0.11.0/24，和公有子网错开
  availability_zone = "${var.aws_region}${count.index == 0 ? "a" : "c"}"

  tags = {
    Name = "${var.project_name}-private-${count.index}"
  }
}

# Internet Gateway：让 VPC 内的资源能访问公网
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${var.project_name}-igw"
  }
}

# 公有子网的路由表：指向 Internet Gateway
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    # 所有外部流量都走 Internet Gateway
    gateway_id = aws_internet_gateway.main.id
  }

  tags = {
    Name = "${var.project_name}-public-rt"
  }
}

# 把公有子网关联到公有路由表
resource "aws_route_table_association" "public" {
  count          = 2
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}
```

这块配置做的事和在 AWS 控制台上手动创建 VPC 是一模一样的，只不过现在用代码描述了。好处是：这段代码可以反复执行，每次结果都一样；可以提交到 Git，有人改了什么一目了然。

### 安全组

安全组是云上的防火墙，控制哪些流量能进出资源。

```hcl
# ============================================================
# 安全组
# ============================================================

# Web 服务器安全组：只允许 HTTP/HTTPS 和 SSH 入站
resource "aws_security_group" "web" {
  name        = "${var.project_name}-web-sg"
  description = "Security group for FlowAPI web servers"
  vpc_id      = aws_vpc.main.id

  # 允许 HTTP 入站
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    # 对所有 IP 开放，因为这是 Web 服务
  }

  # 允许 HTTPS 入站
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # 允许 SSH 入站（仅限你的 IP，不要对全网开放）
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["你的IP/32"]
    # 重要：不要用 0.0.0.0/0，等于把 SSH 暴露给全世界
  }

  # 所有出站流量都允许
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "${var.project_name}-web-sg"
  }
}

# 数据库安全组：只允许 Web 服务器访问
resource "aws_security_group" "db" {
  name        = "${var.project_name}-db-sg"
  description = "Security group for FlowAPI database"
  vpc_id      = aws_vpc.main.id

  # 只允许来自 Web 安全组的资源访问 3306 端口
  ingress {
    from_port       = 3306
    to_port         = 3306
    protocol        = "tcp"
    security_groups = [aws_security_group.web.id]
    # 注意这里不是 cidr_blocks，而是引用了 web 安全组
    # 只有属于 web 安全组的实例才能连数据库
  }

  # 不需要出站规则，数据库不需要主动访问外部

  tags = {
    Name = "${var.project_name}-db-sg"
  }
}
```

安全组的核心原则是最小权限：Web 服务只开 80/443，数据库只允许 Web 服务器访问 3306。这和第一篇讲服务器安全加固时的思路是一致的。

### EC2 实例

```hcl
# ============================================================
# EC2 实例：运行 FlowAPI
# ============================================================

# 查询最新的 Amazon Linux 2023 AMI
# 不要硬编码 AMI ID，不同 region 的 AMI ID 不一样
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }
}

resource "aws_instance" "flowapi" {
  ami                    = data.aws_ami.amazon_linux.id
  instance_type          = var.instance_type
  subnet_id              = aws_subnet.public[0].id
  # 放在第一个公有子网，因为需要公网访问
  vpc_security_group_ids = [aws_security_group.web.id]

  # 给实例打一个 Name 标签，在 AWS 控制台上能看清楚
  tags = {
    Name        = "${var.project_name}-server-${var.environment}"
    Environment = var.environment
  }
}

# 分配一个弹性 IP，这样重启实例后 IP 不会变
resource "aws_eip" "flowapi" {
  instance = aws_instance.flowapi.id
  domain   = "vpc"

  tags = {
    Name = "${var.project_name}-eip"
  }
}
```

`data "aws_ami"` 是个 data source——它不创建资源，而是从 AWS 查询已有的数据。这里用来动态获取最新的 Amazon Linux AMI ID，避免硬编码。

### RDS 数据库

```hcl
# ============================================================
# RDS 数据库：MySQL
# ============================================================

# 先创建数据库子网组，RDS 要求指定子网
resource "aws_db_subnet_group" "main" {
  name       = "${var.project_name}-db-subnet"
  subnet_ids = [aws_subnet.private[0].id, aws_subnet.private[1].id]
  # 数据库放在私有子网，用两个子网做高可用

  tags = {
    Name = "${var.project_name}-db-subnet-group"
  }
}

resource "aws_db_instance" "flowapi" {
  identifier     = "${var.project_name}-db-${var.environment}"
  engine         = "mysql"
  engine_version = "8.0"
  instance_class = var.db_instance_class

  # 存储配置
  allocated_storage     = 20
  max_allocated_storage = 100
  # 开启存储自动扩展，最多扩到 100GB
  storage_type          = "gp3"

  # 数据库配置
  db_name  = "flowapi"
  username = var.db_username
  password = var.db_password

  # 网络配置
  db_subnet_group_name   = aws_db_subnet_group.main.name
  vpc_security_group_ids = [aws_security_group.db.id]
  # 关联数据库安全组，只有 Web 服务器能访问

  # 生产环境建议开启多可用区部署
  multi_az = var.environment == "production" ? true : false
  # dev 环境省钱用单可用区，production 用多可用区

  # 其他配置
  skip_final_snapshot       = var.environment == "production" ? false : true
  # 生产环境删除时创建最终快照，dev 环境跳过
  final_snapshot_identifier = "${var.project_name}-final-snapshot"

  # 自动备份保留天数
  backup_retention_period = var.environment == "production" ? 7 : 1

  tags = {
    Name        = "${var.project_name}-db-${var.environment}"
    Environment = var.environment
  }
}
```

这里用了 `var.environment == "production" ? true : false` 这样的三元表达式来区分环境。同一份代码，通过变量切换就能创建不同规格的环境。

### 输出值

创建完资源后，我们需要知道一些关键信息，比如服务器的公网 IP、数据库的连接地址。

```hcl
# outputs.tf

# EC2 实例的公网 IP，用于 SSH 登录和配置 DNS
output "server_public_ip" {
  description = "FlowAPI 服务器的公网 IP"
  value       = aws_eip.flowapi.public_ip
}

# RDS 数据库的连接地址
# 应用代码里的数据库连接字符串要用这个值
output "db_endpoint" {
  description = "RDS 数据库连接地址"
  value       = aws_db_instance.flowapi.endpoint
}

# VPC ID，其他配置可能会引用
output "vpc_id" {
  description = "VPC ID"
  value       = aws_vpc.main.id
}
```

## Terraform 的核心工作流

配置文件写完了，接下来看看怎么用。Terraform 的日常工作流就三步：

### 第一步：init

```bash
terraform init
```

这个命令做两件事：下载配置中声明的 provider（AWS provider），初始化 backend（存储 state 的地方）。每个新项目只需要执行一次。

### 第二步：plan

```bash
terraform plan
```

plan 是 Terraform 最有价值的功能之一。它会对比你的配置文件和当前 state，告诉你接下来要做什么操作，但**不会真正执行**。输出大概长这样：

```
Terraform will perform the following actions:

  + aws_vpc.main
      will be created

  + aws_subnet.public[0]
      will be created

  + aws_subnet.public[1]
      will be created

  + aws_instance.flowapi
      will be created

  + aws_db_instance.flowapi
      will be created

Plan: 12 to add, 0 to change, 0 to destroy.
```

`+` 表示新增，`~` 表示修改，`-` 表示删除。养成先 plan 再 apply 的习惯，能避免很多意外。

### 第三步：apply

```bash
terraform apply
```

apply 会先执行一遍 plan，让你确认，然后真正去调用 AWS API 创建资源。首次 apply 大概需要 5-10 分钟，因为 RDS 创建比较慢。

你也可以传变量值：

```bash
# 通过命令行传参
terraform apply -var="environment=staging" -var="instance_type=t3.medium"

# 或者通过环境变量（密码等敏感信息推荐这种方式）
export TF_VAR_db_password="your-secure-password"
terraform apply
```

### 销毁资源

开发或测试时经常需要清理环境：

```bash
terraform destroy
```

一条命令把所有资源全部销毁。手动在控制台上删资源容易漏删（尤其是 EIP、EBS 卷这些），Terraform 不会漏。

## State 管理：最重要也最容易出问题的地方

前面提到，Terraform 用一个 state 文件记录当前基础设施的实际状态。这个文件默认存在本地（`terraform.tfstate`），在实际项目中有严重问题。

### 为什么不能把 state 存在本地

**团队协作冲突**。你 apply 了一次，state 文件更新了。同事也 apply 了一次，他的本地 state 文件和你不同步。下次你们各自 apply 的时候，Terraform 会基于各自的 state 做出不同的决策，可能重复创建资源，也可能删掉对方创建的资源。

**state 里包含敏感信息**。数据库密码、私有密钥这些会以明文存在 state 里。把 state 提交到 Git 等于把密钥公开了。

**本地丢失就完了**。如果不小心删了 state 文件，Terraform 就不知道当前基础设施的状态了。资源明明还在 AWS 上跑着，但 Terraform 认为什么都没有，下次 apply 会重新创建所有资源——然后你就有了两套重复的基础设施。

### Remote State：把 state 存到 S3

解决方案是把 state 存到远程的共享存储里。AWS 上通常用 S3 存 state 文件，用 DynamoDB 做状态锁防止并发冲突。

先手动创建一个 S3 bucket 和 DynamoDB 表（是的，管理 Terraform state 的基础设施本身需要手动创建，这是一个 bootstrap 过程）：

```bash
# 创建 S3 bucket 存 state 文件
aws s3api create-bucket \
  --bucket flowapi-terraform-state \
  --region ap-northeast-1 \
  --create-bucket-configuration LocationConstraint=ap-northeast-1

# 开启版本控制，state 文件损坏了可以回滚
aws s3api put-bucket-versioning \
  --bucket flowapi-terraform-state \
  --versioning-configuration Status=Enabled

# 开启加密，保护 state 里的敏感信息
aws s3api put-bucket-encryption \
  --bucket flowapi-terraform-state \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "AES256"
      }
    }]
  }'

# 创建 DynamoDB 表做状态锁
# 防止两个人同时 apply 导致 state 损坏
aws dynamodb create-table \
  --table-name flowapi-terraform-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

然后在 `main.tf` 的 `terraform` 块里配置 backend：

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  # 配置 remote state
  backend "s3" {
    bucket         = "flowapi-terraform-state"
    key            = "flowapi/terraform.tfstate"
    # 每个项目用不同的 key，同一个 bucket 可以存多个项目的 state
    region         = "ap-northeast-1"
    encrypt        = true
    # 使用 S3 服务端加密

    dynamodb_table = "flowapi-terraform-lock"
    # 用 DynamoDB 做状态锁
  }
}
```

配置完 backend 后，重新执行 `terraform init`，Terraform 会问你是否要把本地 state 迁移到 S3——选 Yes。

这样的好处是：

- 团队所有人共享同一个 state 文件，不会冲突
- state 文件有版本控制，搞坏了可以回滚
- DynamoDB 锁保证了同一时间只有一个人在操作
- state 文件加密存储，不会泄露敏感信息

## 变量文件和环境隔离

实际项目中通常有多个环境（dev、staging、production）。与其每次手动传变量，不如用 `.tfvars` 文件管理不同环境的配置。

创建 `dev.tfvars`：

```hcl
# dev.tfvars
aws_region      = "ap-northeast-1"
environment     = "dev"
instance_type   = "t3.small"
db_instance_class = "db.t3.micro"
```

创建 `production.tfvars`：

```hcl
# production.tfvars
aws_region      = "ap-northeast-1"
environment     = "production"
instance_type   = "t3.medium"
db_instance_class = "db.t3.medium"
```

使用时指定 tfvars 文件：

```bash
# 创建 dev 环境
terraform apply -var-file=dev.tfvars

# 创建 production 环境
terraform apply -var-file=production.tfvars
```

同一套代码，不同的参数，就能创建不同规格的环境。这在手动操作时代是做不到的。

## 模块化：避免复制粘贴

当项目变大，把所有资源写在一个 `main.tf` 里会很臃肿。Terraform 的 module 机制可以把一组资源封装成可复用的模块。

比如把 VPC 相关的配置提取成模块：

```
terraform-flowapi/
├── main.tf
├── variables.tf
├── outputs.tf
├── dev.tfvars
├── production.tfvars
└── modules/
    └── vpc/
        ├── main.tf      # VPC、子网、路由表的 resource
        ├── variables.tf  # CIDR 块、可用区等变量
        └── outputs.tf    # VPC ID、子网 ID 等输出
```

在根目录的 `main.tf` 里引用模块：

```hcl
# 使用自定义 VPC 模块
module "vpc" {
  source = "./modules/vpc"

  project_name = var.project_name
  environment  = var.environment
  vpc_cidr     = "10.0.0.0/16"
  azs          = ["ap-northeast-1a", "ap-northeast-1c"]
}

# 引用模块输出的值
resource "aws_instance" "flowapi" {
  subnet_id = module.vpc.public_subnet_ids[0]
  # ...
}
```

更进一步，你可以用社区维护的模块，不用自己写。比如 [terraform-aws-modules/vpc/aws](https://registry.terraform.io/modules/terraform-aws-modules/vpc/aws) 是一个广泛使用的 VPC 模块，支持完整的 VPC 配置：

```hcl
# 使用社区 VPC 模块
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"

  name = "${var.project_name}-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["ap-northeast-1a", "ap-northeast-1c"]
  public_subnets  = ["10.0.0.0/24", "10.0.1.0/24"]
  private_subnets = ["10.0.10.0/24", "10.0.11.0/24"]

  enable_nat_gateway = true
  # 自动创建 NAT Gateway，让私有子网也能访问公网
}
```

社区模块帮你处理了大量细节（NAT Gateway、路由表关联、DHCP 配置等），比自己写省很多精力。但代价是你需要理解模块的接口和默认行为，出了问题也要能看懂模块内部的代码。

## Terraform vs Pulumi

聊到 IaC 就绕不开 Pulumi。两者解决同一个问题，但思路不同。

**Terraform** 用 HCL 描述基础设施。HCL 是一种声明式语言，你写的是"我要什么"，不是"怎么做"。学习曲线平缓，语法简洁，但缺乏编程语言的灵活性（循环、条件判断写起来比较别扭）。

**Pulumi** 用通用编程语言（TypeScript、Python、Go）描述基础设施。你可以用熟悉的语言写 if/else、for 循环、函数封装，甚至可以写单元测试。对开发者更友好，但要求团队有编程基础。

一个简单的对比：

```hcl
# Terraform：用 count 和循环创建多个子网
resource "aws_subnet" "private" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 10}.0/24"
  availability_zone = "${var.aws_region}${count.index == 0 ? "a" : "c"}"
}
```

```typescript
// Pulumi：用 TypeScript 的 for 循环创建多个子网
const subnets = ["ap-northeast-1a", "ap-northeast-1c"].map(
  (az, index) =>
    new aws.ec2.Subnet(`private-${index}`, {
      vpcId: vpc.id,
      cidrBlock: `10.0.${index + 10}.0/24`,
      availabilityZone: az,
    })
);
```

怎么选？

- 团队主要是 DevOps / 运维背景，选 Terraform。生态成熟、文档完善、社区资源丰富，AWS 官方也大量使用 Terraform。
- 团队主要是开发者，觉得 HCL 限制太多，选 Pulumi。用熟悉的语言写基础设施，学习成本低。
- 已有大量 Terraform 配置在生产环境跑着，没必要迁移。两者可以共存。

## Trade-off

Terraform 把基础设施从手动操作变成了代码管理，解决了不可重复、不可审计、不可回滚的问题。但它不是银弹。

**适合的场景**：需要管理大量云资源的团队，尤其是多环境（dev/staging/production）需要保持一致的场景。代码化的基础设施让 code review 成为可能，新成员加入后看代码就能理解整个架构。

**需要注意的代价**：Terraform 的学习曲线不在语法（HCL 很简单），在于你需要理解 AWS 的网络模型、IAM 策略、RDS 配置等云服务知识。Terraform 只是帮你用代码表达这些配置，不会帮你设计架构。另外，state 管理是一个持续的运维负担——state 损坏、state 和实际资源不同步（drift）这类问题总会出现，你需要知道怎么排查和修复。

**不适合的场景**：如果你只有一台服务器、一个数据库，手动在控制台上点几下就够了，上 Terraform 反而增加了复杂度。过度工程化和工程化不足一样糟糕。经验法则是：当你需要第二次搭建相同的基础设施时，就该考虑 Terraform 了。
