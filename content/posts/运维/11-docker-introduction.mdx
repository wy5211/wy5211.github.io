## 前言

在软件开发的世界里，"在我机器上能跑"是经典的悲剧。环境不一致导致的问题困扰了无数开发者。Docker 的出现彻底改变了这一局面，它让应用及其运行环境打包在一起，真正实现了"一次构建，到处运行"。

**为什么需要 Docker？**

想象一下这些场景：

- 开发环境用 macOS，测试环境是 Linux，生产环境是 Windows
- 不同项目依赖不同版本的 Node.js、Python
- 新同事入职需要花费一整天配置开发环境
- 应用升级时，依赖库版本冲突导致服务崩溃

Docker 通过容器技术完美解决了这些问题，让应用像集装箱一样标准化、可移植。

## 容器 vs 虚拟机

### 传统虚拟机架构

```
┌─────────────────────────────────────────┐
│          应用 A         应用 B         │
├─────────────────────────────────────────┤
│           Guest OS      Guest OS       │
├─────────────────────────────────────────┤
│             Hypervisor                 │
├─────────────────────────────────────────┤
│              Host OS                   │
├─────────────────────────────────────────┤
│             硬件层                     │
└─────────────────────────────────────────┘
```

**虚拟机特点**：

- ✅ 完全隔离，运行完整操作系统
- ✅ 安全性高
- ❌ 资源占用大（每个 VM 几 GB）
- ❌ 启动慢（分钟级）
- ❌ 性能损耗明显

### Docker 容器架构

```
┌─────────────────────────────────────────┐
│      应用 A    |    应用 B              │
├─────────────────────────────────────────┤
│     引擎层（Docker Engine）            │
├─────────────────────────────────────────┤
│              Host OS                   │
├─────────────────────────────────────────┤
│             硬件层                     │
└─────────────────────────────────────────┘
```

**容器特点**：

- ✅ 轻量级（MB 级别）
- ✅ 秒级启动
- ✅ 性能接近原生
- ✅ 共享宿主机内核
- ❌ 隔离性稍弱（但足够安全）

### 性能对比

| 指标     | 虚拟机   | Docker 容器 |
| -------- | -------- | ----------- |
| 启动时间 | 分钟级   | 秒级        |
| 镜像大小 | GB 级    | MB 级       |
| 性能损耗 | 5%-20%   | `<2%`       |
| 单机容量 | 几十个   | 几百上千    |
| 隔离性   | 完全隔离 | 进程级隔离  |

## Docker 三大核心概念

### 1. 镜像（Image）

**镜像是什么？**

镜像是一个只读的模板，包含了运行应用所需的一切：

- 应用代码
- 运行时环境（Node.js、Python 等）
- 系统库
- 配置文件
- 依赖项

**类比理解**：

- 镜像就像**菜谱**
- 定义了所有原材料和步骤
- 可以根据菜谱制作出无数份相同的菜肴

**镜像特点**：

- 📦 **分层存储**：每一层都是只读的
- 🔄 **可复用**：多个镜像可共享基础层
- 🚀 **快速构建**：修改只需重建变更层

**示例**：

```bash
# 查看 nginx 镜像的分层
docker history nginx:latest

# 输出：
# IMAGE          CREATED      CREATED BY                                      SIZE
# 8f336738...    2 weeks ago  /bin/sh -c #(nop)  CMD ["nginx" "-g" "daemon...  0B
# <missing>      2 weeks ago  /bin/sh -c #(nop)  STOPSIGNAL SIGQUIT           0B
# <missing>      2 weeks ago  /bin/sh -c #(nop)  EXPOSE 80                   0B
# ... 更多层 ...
```

### 2. 容器（Container）

**容器是什么？**

容器是从镜像创建的运行实例，可以理解为：

- 镜像是**类**（Class）
- 容器是**对象**（Object）

**类比理解**：

- 容器就像**根据菜谱做出的菜**
- 同一个镜像可以创建多个容器
- 每个容器都是独立运行的

**容器特点**：

- ✅ **可写**：容器顶层是可写的
- ✅ **隔离**：每个容器有独立的文件系统、网络、进程空间
- ✅ **临时**：容器删除后，其中的更改会丢失（除非使用卷）

**生命周期**：

```
Created → Running → Paused → Stopped → Deleted
```

**示例**：

```bash
# 从镜像创建并运行容器
docker run -d --name my-nginx -p 8080:80 nginx

# 查看运行中的容器
docker ps

# 停止容器
docker stop my-nginx

# 启动已停止的容器
docker start my-nginx

# 删除容器
docker rm my-nginx
```

### 3. 仓库（Registry）

**仓库是什么？**

仓库用于存储和分发镜像，就像：

- GitHub 存储代码
- Docker Hub 存储镜像

**仓库类型**：

- 🌐 **公共仓库**：Docker Hub、阿里云镜像仓库
- 🔒 **私有仓库**：企业自建 Registry、Harbor

**镜像命名规范**：

```
[仓库地址/][用户名/]镜像名[:标签]

示例：
nginx                          # 官方镜像
nginx:1.24                     # 指定标签
nginx:alpine                   # 使用 alpine 版本
myregistry.com/myapp:latest   # 私有仓库
username/myapp:v1.0           # 用户镜像
```

## Docker 核心架构

### 组件关系图

```
┌─────────────────────────────────────────────┐
│           Docker Client (CLI)               │
│      docker build, pull, run, ps...         │
└─────────────────┬───────────────────────────┘
                  │ REST API
                  ▼
┌─────────────────────────────────────────────┐
│         Docker Daemon (dockerd)             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │  Images  │  │Containers│  │Networks  │  │
│  └──────────┘  └──────────┘  └──────────┘  │
│  ┌──────────┐  ┌──────────┐                │
│  │Volumes   │  │ Builder  │                │
│  └──────────┘  └──────────┘                │
└─────────────────┬───────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────┐
│          containerd + runc                 │
│     (容器运行时，负责创建容器)              │
└─────────────────┬───────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────┐
│              Linux Kernel                  │
│   Namespace (隔离) + Cgroups (资源限制)    │
└─────────────────────────────────────────────┘
```

### 关键组件说明

**1. Docker Client（客户端）**

- 用户与 Docker 交互的主要界面
- 通过 CLI 或 API 发送命令
- 可以连接到本地或远程 Daemon

**2. Docker Daemon（守护进程）**

- `dockerd`，后台服务
- 监听 Docker API 请求
- 管理镜像、容器、网络、卷

**3. containerd（容器守护进程）**

- 行业标准的容器运行时
- 管理容器生命周期
- 拉取和推送镜像

**4. runc（容器运行时）**

- 底层容器运行时 CLI 工具
- 实际创建和运行容器
- 遵循 OCI 规范

## Linux 底层技术

Docker 容器的隔离和资源限制依赖于 Linux 内核的两大核心技术。

### 1. Namespace（命名空间）

**作用**：实现资源隔离

**六种 Namespace**：

| Namespace      | 隔离内容     | 说明                         |
| -------------- | ------------ | ---------------------------- |
| PID Namespace  | 进程 ID      | 容器内拥有独立 PID 1         |
| NET Namespace  | 网络栈       | 独立网络设备、IP 地址、路由  |
| MNT Namespace  | 挂载点       | 独立文件系统挂载点           |
| UTS Namespace  | 主机名和域名 | 容器可设置自己的 hostname    |
| IPC Namespace  | 进程间通信   | 隔离消息队列、信号量等       |
| USER Namespace | 用户和用户组 | 容器内可拥有独立的 root 用户 |

**示例**：

```bash
# 查看容器的 Namespace
docker inspect my-nginx | grep -A 10 "Namespace"

# 输出示例：
# "Namespace": [
#     {
#         "Type": "pid",
#         "Path": "/proc/12345/ns/pid"
#     },
#     ...
# ]
```

### 2. Cgroups（控制组）

**作用**：实现资源限制和监控

**可限制的资源**：

- 💻 **CPU**：限制使用哪些 CPU 核、使用比例
- 🧠 **内存**：限制内存使用量、交换空间
- 💾 **磁盘 I/O**：限制读写速度
- 📊 **网络**：限制网络带宽

**示例**：

```bash
# 限制容器最多使用 512MB 内存和 1 个 CPU 核心
docker run -d \
  --memory="512m" \
  --cpus="1.0" \
  nginx

# 查看容器的资源使用情况
docker stats my-nginx
```

## Docker 工作流程

### 典型使用流程

```
1. docker pull nginx
   └─→ 从 Docker Hub 下载 nginx 镜像

2. docker run -d -p 8080:80 --name my-nginx nginx
   ├─→ 创建容器可写层
   ├─→ 配置网络（桥接、端口映射）
   ├─→ 设置 Namespace 和 Cgroups
   └─→ 启动容器进程

3. docker ps
   └─→ 查看运行中的容器

4. docker logs my-nginx
   └─→ 查看容器日志

5. docker stop my-nginx
   └─→ 停止容器（发送 SIGTERM 信号）

6. docker rm my-nginx
   └─→ 删除容器
```

### 镜像分层原理

**UnionFS（联合文件系统）**：

```
┌─────────────────────┐  ← 容器可写层
│   /app/config.js    │
├─────────────────────┤
│   /app/server.js    │  ← 镜像层（只读）
├─────────────────────┤
│   /node_modules/    │  ← 镜像层（只读）
├─────────────────────┤
│   node:18-alpine    │  ← 基础镜像（只读）
└─────────────────────┘
```

**优势**：

- 📦 **共享基础层**：多个容器共享同一基础镜像
- ⚡ **快速构建**：只重建变更的层
- 💾 **节省空间**：相同层只存储一次

## 实战演练

### 运行第一个容器

```bash
# 运行 hello-world 镜像
docker run hello-world

# 输出：
# Hello from Docker!
# This message shows that your installation appears to be working correctly.

# 底层发生了什么：
# 1. Docker Client 调用 Docker Daemon
# 2. Daemon 在本地查找 hello-world 镜像
# 3. 未找到，从 Docker Hub 拉取
# 4. 创建容器并运行
# 5. 输出信息后退出
```

### 运行 Nginx 容器

```bash
# 后台运行 Nginx，映射端口
docker run -d --name my-nginx -p 8080:80 nginx:latest

# -d: 后台运行（detached）
# --name: 指定容器名称
# -p: 端口映射（宿主机:容器）

# 访问 http://localhost:8080
curl http://localhost:8080

# 查看容器详情
docker inspect my-nginx

# 查看容器日志
docker logs my-nginx

# 实时跟踪日志
docker logs -f my-nginx

# 停止容器
docker stop my-nginx

# 删除容器
docker rm my-nginx
```

### 交互式运行容器

```bash
# 运行 Ubuntu 容器并进入
docker run -it ubuntu:22.04 /bin/bash

# -i: 交互式（保持 STDIN 开启）
# -t: 分配伪终端（TTY）

# 现在你进入了容器内部
root@abc123:/# pwd
/
root@abc123:/# ls
bin  boot  dev  etc  home  lib  ...
root@abc123:/# exit

# 容器退出后会被停止
```

## 最佳实践

### 1. 镜像选择

**选择官方镜像**：

```bash
# ✅ 推荐：使用官方镜像
docker pull nginx
docker pull node
docker pull postgres

# ❌ 避免：使用不明来源的镜像
docker pull random-user/nginx-custom
```

**使用特定版本标签**：

```bash
# ✅ 推荐：指定版本
docker pull nginx:1.24
docker pull node:18-alpine

# ❌ 避免：使用 latest 标签（生产环境）
docker pull nginx:latest  # 可能导致版本不一致
```

### 2. 容器命名

```bash
# ✅ 推荐：使用有意义的名称
docker run -d --name web-server nginx
docker run -d --name api-gateway nginx
docker run -d --name cache-db redis

# ❌ 避免：让 Docker 自动生成名称
docker run -d nginx  # 可能生成 boring_ganguly
```

### 3. 资源限制

```bash
# ✅ 推荐：设置资源限制
docker run -d \
  --name my-app \
  --memory="512m" \
  --cpus="1.5" \
  --restart=unless-stopped \
  myapp:1.0

# 防止容器占用过多资源导致宿主机崩溃
```

### 4. 数据持久化

```bash
# ✅ 推荐：使用卷管理重要数据
docker run -d \
  --name mysql \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  mysql:8.0

# ❌ 避免：将数据存储在容器内
docker run -d mysql:8.0  # 容器删除后数据丢失
```

## 常见问题

### Q1: Docker 和虚拟机哪个更好？

**答**：两者各有适用场景。

**选择 Docker**：

- 微服务架构
- CI/CD 流水线
- 快速部署和扩缩容
- 资源受限的环境

**选择虚拟机**：

- 需要运行完整操作系统
- 强隔离要求（如运行不受信任的代码）
- 运行不同内核的应用

### Q2: 容器是虚拟机吗？

**答**：不是。

- **虚拟机**：模拟完整硬件，运行完整操作系统
- **容器**：共享宿主机内核，只是进程级别的隔离

### Q3: 为什么容器启动这么快？

**答**：

1. **不需要启动操作系统**：容器只是宿主机上的一个进程
2. **镜像分层缓存**：常用的层已经存在
3. **轻量级**：镜像通常只有几十到几百 MB

### Q4: Docker 只能用于 Linux 应用吗？

**答**：不是。

虽然 Docker 最初是为 Linux 设计的，但现在：

- **macOS**：通过 Docker Desktop 运行（内部使用 Linux 虚拟机）
- **Windows**：通过 WSL2 运行（真正的 Linux 内核）
- 可以容器化任何平台的应用

### Q5: 容器安全吗？

**答**：相对安全，但需要注意：

1. **不要以 root 运行容器**（使用 USER 指令）
2. **扫描镜像漏洞**（使用 docker scan）
3. **限制资源**（防止 DoS 攻击）
4. **使用私有仓库**（避免恶意镜像）
5. **及时更新基础镜像**（修复已知漏洞）

## 总结

本篇文章我们深入理解了 Docker 的核心概念：

**核心要点**：

1. 🐳 **Docker 三大核心**：镜像（模板）、容器（实例）、仓库（存储）
2. 🏗️ **架构组成**：Client → Daemon → containerd → runc → Kernel
3. 🔒 **底层技术**：Namespace（隔离）+ Cgroups（限制）
4. ⚡ **优势**：轻量、快速、可移植、一致

**关键对比**：

- 容器 vs 虚拟机：容器共享内核，虚拟机独立内核
- 镜像 vs 容器：镜像是只读模板，容器是运行实例
