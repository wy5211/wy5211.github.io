## 前言

Kubernetes（K8s）已经成为容器编排的事实标准。无论你是在云原生环境部署微服务，还是管理复杂的分布式系统，K8s 都是不可绕开的技术。

但说实话，K8s 的学习曲线确实陡峭。概念多、配置复杂、调试困难——很多开发者在入门阶段就放弃了。

本文的目标很简单：**用最实用的方式，带你从零开始理解 K8s，并完成一次真正的部署**。不讲空洞的理论，只讲你马上能用到的知识。

---

## 一、K8s 是什么？为什么需要它？

### 1.1 从 Docker 到 K8s

如果你用过 Docker，一定知道它很方便：把应用打包成镜像，一条命令就能运行。但当你有 **几十个容器** 需要管理时，问题就来了：

- **容器挂了怎么办？** 需要自动重启
- **流量太多怎么办？** 需要自动扩容
- **如何更新应用？** 需要滚动发布
- **多个容器如何通信？** 需要服务发现

Docker Compose 能解决一部分问题，但它只能管理单机上的容器。跨多台服务器？无能为力。

**K8s 的定位：跨集群的容器编排系统。** 它帮你管理成百上千个容器，处理扩容、故障恢复、负载均衡、滚动更新等工作。

### 1.2 K8s 的核心能力

| 能力         | 说明                               |
| ------------ | ---------------------------------- |
| **自动调度** | 决定容器运行在哪台机器上           |
| **自动恢复** | 容器挂了自动重启，节点挂了自动迁移 |
| **水平扩缩** | 根据负载自动增减容器数量           |
| **服务发现** | 容器之间如何找到彼此               |
| **负载均衡** | 流量均匀分配到多个容器             |
| **滚动更新** | 零停机发布新版本                   |
| **配置管理** | 统一管理环境变量、密钥             |

## 二、核心概念速览（别被吓到）

K8s 有很多概念，但初学者只需要掌握这几个：

### 2.1 Node（节点）

**Node = 一台物理机或虚拟机**

K8s 集群由多个 Node 组成。Node 分两种：

- **Master Node**：控制平面，负责调度和管理
- **Worker Node**：真正运行容器的工作节点

### 2.2 Pod（最小调度单位）

**Pod = 一个或多个紧密相关的容器**

Pod 是 K8s 调度的最小单位。一个 Pod 里的容器共享网络和存储，总是部署在同一台机器上。

> 为什么不是直接调度容器？因为有时候多个容器需要紧密协作（比如主容器 + 日志收集 sidecar）。

### 2.3 Deployment（控制器）

**Deployment = 管理 Pod 的副本数量和更新策略**

你很少直接创建 Pod，而是创建 Deployment。它会：

- 确保指定数量的 Pod 一直运行
- 支持滚动更新和回滚
- 记录历史版本

```yaml
# 一个简单的 Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3 # 运行 3 个副本
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: nginx:latest
          ports:
            - containerPort: 80
```

### 2.4 Service（服务发现）

**Service = 为一组 Pod 提供稳定的访问入口**

Pod 的 IP 是动态的，每次重启都会变。Service 提供一个稳定的 IP 或域名，让其他服务能访问你。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 80
  type: ClusterIP # 集群内部访问
```

### 2.5 ConfigMap & Secret（配置管理）

- **ConfigMap**：存储非敏感配置（环境变量、配置文件）
- **Secret**：存储敏感信息（密码、证书）

```yaml
# ConfigMap 示例
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DATABASE_URL: "postgres://db:5432/mydb"
  LOG_LEVEL: "info"
```

## 三、环境搭建：选一个适合你的

### 3.1 本地开发：Minikube / Kind / k3d

**最简单的选择：Minikube**

```bash
# macOS
brew install minikube

# 启动集群
minikube start

# 查看状态
minikube status
```

**更轻量的选择：k3d（基于 k3s）**

```bash
# 安装 k3d
brew install k3d

# 创建集群
k3d cluster create my-cluster
```

### 3.2 云服务：托管 K8s

如果你有云账号，可以直接用托管服务：

- **GKE**（Google Kubernetes Engine）
- **EKS**（Amazon Elastic Kubernetes Service）
- **AKS**（Azure Kubernetes Service）
- **阿里云 ACK** / **腾讯云 TKE**

托管服务省去了 master 节点的运维，但需要付费。

### 3.3 验证安装

```bash
# 查看 kubectl 版本
kubectl version

# 查看集群信息
kubectl cluster-info

# 查看节点
kubectl get nodes
```

## 四、实战：部署一个 Web 应用

现在来点真实的。我们部署一个 Nginx 应用，包含 Deployment、Service、ConfigMap。

### 4.1 创建 Deployment

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
          resources:
            requests:
              memory: "64Mi"
              cpu: "100m"
            limits:
              memory: "128Mi"
              cpu: "200m"
```

```bash
# 应用配置
kubectl apply -f deployment.yaml

# 查看 Pod 状态
kubectl get pods

# 查看 Deployment 状态
kubectl get deployments
```

### 4.2 创建 Service

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30080 # 节点端口（30000-32767）
  type: NodePort # 可以从集群外部访问
```

```bash
kubectl apply -f service.yaml

# 查看 Service
kubectl get services

# 本地访问（minikube）
minikube service web-service
```

### 4.3 添加 ConfigMap

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-config
data:
  ENVIRONMENT: "production"
  LOG_LEVEL: "info"
```

修改 Deployment 使用 ConfigMap：

```yaml
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      envFrom:
        - configMapRef:
            name: web-config
```

```bash
kubectl apply -f configmap.yaml
kubectl apply -f deployment.yaml
```

### 4.4 验证部署

```bash
# 查看 Pod 的环境变量
kubectl exec -it <pod-name> -- env | grep ENVIRONMENT

# 查看日志
kubectl logs <pod-name>

# 进入容器
kubectl exec -it <pod-name> -- /bin/bash
```

## 五、常用命令速查表

| 命令                                           | 说明                     |
| ---------------------------------------------- | ------------------------ |
| `kubectl get pods`                             | 查看所有 Pod             |
| `kubectl get deployments`                      | 查看所有 Deployment      |
| `kubectl get services`                         | 查看所有 Service         |
| `kubectl describe pod <name>`                  | 查看详细信息（包括事件） |
| `kubectl logs <pod-name>`                      | 查看日志                 |

| `kubectl scale deployment <name> --replicas=5` | 扩容                     |
| `kubectl rollout undo deployment/<name>`       | 回滚                     |

## 六、常见问题与调试技巧

### 6.1 Pod 一直是 Pending

**可能原因：**

- 资源不足（CPU/内存）
- 节点不够
- 持久卷未绑定

**调试方法：**

```bash
kubectl describe pod <pod-name>
# 查看 Events 部分
```

### 6.2 ImagePullBackOff

**原因：** 镜像拉取失败

**解决：**

```bash
# 检查镜像名是否正确
# 检查是否有权限访问私有镜像
kubectl describe pod <pod-name>
```

### 6.3 CrashLoopBackOff

**原因：** 容器启动后立即崩溃

**调试方法：**

```bash
# 查看日志
kubectl logs <pod-name> --previous

# 查看退出码
kubectl describe pod <pod-name>
```

### 6.4 Service 无法访问

**检查清单：**

1. Service 的 selector 是否与 Pod 的 label 匹配？
2. Pod 是否正常运行？
3. 端口配置是否正确？

```bash
# 检查 Endpoints
kubectl get endpoints <service-name>
```

## 七、进阶方向

当你熟悉了基础操作后，可以探索这些方向：

### 7.1 Helm：K8s 的包管理器

Helm 让你用 Chart（类似 npm 包）部署复杂应用：

```bash
# 安装 Helm
brew install helm

# 添加仓库
helm repo add bitnami https://charts.bitnami.com/bitnami

# 安装应用
helm install my-nginx bitnami/nginx
```

### 7.2 Ingress：更智能的路由

Service 只能做 4 层负载均衡，Ingress 可以做 7 层（域名、路径）：

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web-service
                port:
                  number: 80
```

### 7.3 监控：Prometheus + Grafana

K8s 的监控标配：

- **Prometheus**：采集指标
- **Grafana**：可视化展示

### 7.4 日志：ELK / Loki

集中收集和查询日志。

## 八、学习建议

### 8.1 不要一开始就上生产

先用本地环境（Minikube/k3d）玩熟悉，再去碰云服务。生产环境的 K8s 配置复杂得多（RBAC、NetworkPolicy、LimitRange...）。

### 8.2 多看官方文档

K8s 官方文档质量很高：[https://kubernetes.io/docs/](https://kubernetes.io/docs/)

### 8.3 动手实践

只看不练是学不会的。给自己设定一个小目标：

- 部署一个简单的 Web 应用
- 实现滚动更新
- 配置 Ingress
- 搭建监控系统

---

## 总结

K8s 确实复杂，但它的核心逻辑是清晰的：

1. **Deployment** 管理 Pod 的数量和更新
2. **Service** 提供稳定的访问入口
3. **ConfigMap/Secret** 管理配置

掌握这三个概念，你就能完成 80% 的日常工作。

剩下的 20%（网络、存储、安全、调度策略...）可以在实践中逐步学习。

**关键建议：** 不要试图一次学会所有东西。先用起来，遇到问题再深入。
