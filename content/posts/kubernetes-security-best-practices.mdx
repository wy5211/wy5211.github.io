Kubernetes 已成为云原生时代的事实标准，但随着集群规模扩大和业务复杂度提升，安全问题也日益突出。一个配置不当的 K8s 集群可能成为攻击者的后花园。

本文将系统讲解 Kubernetes 安全的最佳实践，从身份认证到网络隔离，从权限管理到密钥保护，帮助你构建纵深防御的安全体系。

## 一、Kubernetes 安全模型概览

### 1.1 安全边界划分

Kubernetes 安全可以分为三个层级：

```
┌─────────────────────────────────────────────┐
│           集群基础设施层                      │
│  (API Server、etcd、kubelet 安全)            │
├─────────────────────────────────────────────┤
│           控制平面安全                        │
│  (RBAC、Admission Control、Audit)            │
├─────────────────────────────────────────────┤
│           工作负载安全                        │
│  (Pod Security、Network Policy、Secret)      │
└─────────────────────────────────────────────┘
```

### 1.2 攻击面分析

<Callout type="warning">
  常见的 K8s 安全风险： - 过度宽松的 RBAC 权限 - 容器以 root 用户运行 -
  未配置网络隔离策略 - Secret 明文存储在 etcd - 使用存在漏洞的镜像 - API Server
  暴露在公网
</Callout>

## 二、RBAC：精细化权限管理

RBAC（Role-Based Access Control）是 K8s 权限管理的核心。

### 2.1 核心概念

```yaml
# Role - 命名空间级别权限
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "watch", "list"]
---
# ClusterRole - 集群级别权限
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: secret-reader
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get"]
```

### 2.2 最佳实践

**原则 1：最小权限原则（Least Privilege）**

```yaml
# ❌ 错误示例：给开发者 cluster-admin 权限
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: developers-admin
subjects:
  - kind: Group
    name: developers
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: cluster-admin # 危险！
  apiGroup: rbac.authorization.k8s.io

---
# ✅ 正确示例：只给开发环境特定权限
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developers-dev-access
  namespace: development
subjects:
  - kind: Group
    name: developers
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer-role
  apiGroup: rbac.authorization.k8s.io
```

**原则 2：避免使用通配符**

```yaml
# ❌ 危险：通配符授予所有权限
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]

# ✅ 安全：明确指定权限
rules:
- apiGroups: [""]
  resources: ["pods", "services"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "create", "update"]
```

### 2.3 RBAC 审计工具

使用 `kubectl auth can-i` 检查权限：

```bash
# 检查用户是否有权限删除 pod
kubectl auth can-i delete pods --as=alice --namespace=production

# 检查 ServiceAccount 的权限
kubectl auth can-i list secrets --as=system:serviceaccount:default:my-sa

# 列出用户的所有权限
kubectl auth can-i --list --as=alice
```

<Callout type="info">
  推荐工具： - **rbac-tool**：可视化 RBAC 权限树 -
  **kubectl-who-can**：查看谁有特定权限 - **rbac-lookup**：搜索用户/组的权限绑定
</Callout>

## 三、Network Policy：网络隔离

默认情况下，K8s 集群内所有 Pod 可以互相通信，这在生产环境是危险的。

### 3.1 基础网络策略

```yaml
# 默认拒绝所有入站流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: sensitive-apps
spec:
  podSelector: {} # 选择所有 Pod
  policyTypes:
    - Ingress
```

```yaml
# 只允许特定标签的 Pod 访问
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api-server
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              role: frontend
        - namespaceSelector:
            matchLabels:
              name: istio-system
      ports:
        - protocol: TCP
          port: 8080
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: database
      ports:
        - protocol: TCP
          port: 5432
    - to: # 允许 DNS 查询
        - namespaceSelector: {}
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
```

### 3.2 多层防御策略

```yaml
# 数据库层：只接受来自应用层的连接
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: database-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: database
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              tier: backend
      ports:
        - protocol: TCP
          port: 5432
  egress: [] # 数据库不需要主动外连
```

### 3.3 Network Policy 实践要点

| 场景     | 推荐策略                           |
| -------- | ---------------------------------- |
| 前端应用 | 允许来自 Ingress，允许访问后端     |
| 后端 API | 允许来自前端，允许访问数据库和缓存 |
| 数据库   | 只允许来自后端，禁止主动外连       |
| 监控组件 | 允许访问所有 Pod 的 metrics 端口   |

<Callout type="warning">
  注意：Network Policy 需要网络插件支持（Calico、Cilium、Weave 等），flannel
  默认不支持。
</Callout>

## 四、Pod Security Standards

Kubernetes 1.25+ 弃用了 PodSecurityPolicy，改用 Pod Security Standards（PSS）。

### 4.1 三种安全级别

| 级别           | 描述               | 适用场景          |
| -------------- | ------------------ | ----------------- |
| **Privileged** | 不限制             | 系统/基础设施组件 |
| **Baseline**   | 禁止明显危险的配置 | 大多数应用        |
| **Restricted** | 严格限制           | 安全敏感的应用    |

### 4.2 启用 PSS 准入控制

```yaml
# Namespace 标签启用 PSS
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

### 4.3 符合 Restricted 级别的 Pod 配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: myapp:1.0
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop:
            - ALL
      resources:
        limits:
          memory: "512Mi"
          cpu: "500m"
```

### 4.4 安全上下文配置清单

```yaml
# 完整的安全上下文配置
securityContext:
  # Pod 级别
  runAsNonRoot: true # 禁止以 root 运行
  runAsUser: 1000 # 指定运行用户
  runAsGroup: 1000 # 指定运行组
  fsGroup: 1000 # 卷文件所属组
  seccompProfile:
    type: RuntimeDefault # 启用 seccomp

  # 容器级别
  allowPrivilegeEscalation: false # 禁止提权
  readOnlyRootFilesystem: true # 只读根文件系统
  capabilities:
    drop:
      - ALL # 删除所有 capabilities
```

## 五、Secret 管理最佳实践

### 5.1 Secret 加密存储

默认 Secret 以 base64 编码存储在 etcd，需要启用加密：

```yaml
# encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - aescbc:
          keys:
            - name: key1
              secret: <BASE64_ENCODED_SECRET>
      - identity: {} # 作为后备
```

API Server 启动参数：

```bash
kube-apiserver \
  --encryption-provider-config=/etc/kubernetes/encryption-config.yaml \
  ...
```

### 5.2 使用外部密钥管理

**方案 1：Sealed Secrets**

```bash
# 安装 Sealed Secrets Controller
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.24.0/controller.yaml

# 加密 Secret
kubeseal --format=yaml &lt; secret.yaml &gt; sealed-secret.yaml
```

**方案 2：External Secrets Operator**

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: database-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secretsmanager
    kind: ClusterSecretStore
  target:
    name: db-credentials
  data:
    - secretKey: password
      remoteRef:
        key: production/database
        property: password
```

### 5.3 Secret 使用规范

```yaml
# ✅ 正确：从 Secret 挂载
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
    - name: app
      image: myapp:1.0
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
      volumeMounts:
        - name: certs
          mountPath: /etc/certs
          readOnly: true
  volumes:
    - name: certs
      secret:
        secretName: tls-certs
```

<Callout type="info">
  Secret 管理建议： - 不要将 Secret 提交到 Git - 使用 RBAC 限制 Secret 访问 -
  定期轮换密钥 - 启用 etcd 加密 - 考虑使用外部密钥管理系统（Vault、AWS Secrets
  Manager）
</Callout>

## 六、容器镜像安全

### 6.1 使用最小化基础镜像

```dockerfile
# ❌ 不推荐：使用完整系统镜像
FROM ubuntu:22.04

# ✅ 推荐：使用 distroless 或 alpine
FROM gcr.io/distroless/static:latest
# 或
FROM alpine:3.19
```

### 6.2 镜像安全扫描

**使用 Trivy 扫描镜像：**

```bash
# 扫描本地镜像
trivy image myapp:1.0

# 扫描并输出 JSON 报告
trivy image --format json --output report.json myapp:1.0

# 只显示 HIGH 和 CRITICAL 漏洞
trivy image --severity HIGH,CRITICAL myapp:1.0
```

**在 CI/CD 中集成扫描：**

```yaml
# GitHub Actions 示例
name: Security Scan
on: [push]
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build image
        run: docker build -t myapp:${{ github.sha }} .
      - name: Run Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:${{ github.sha }}
          format: "table"
          exit-code: "1"
          severity: "CRITICAL,HIGH"
```

### 6.3 镜像签名验证

使用 Cosign 签名镜像：

```bash
# 生成密钥对
cosign generate-key-pair

# 签名镜像
cosign sign --key cosign.key myregistry.io/myapp:1.0

# 验证签名
cosign verify --key cosign.pub myregistry.io/myapp:1.0
```

在 K8s 中强制验证（使用 Kyverno）：

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signatures
spec:
  validationFailureAction: enforce
  background: false
  rules:
    - name: verify-signature
      match:
        any:
          - resources:
              kinds:
                - Pod
      verifyImages:
        - imageReferences:
            - "myregistry.io/*"
          attestors:
            - entries:
                - keys:
                    publicKeys: |-
                      -----BEGIN PUBLIC KEY-----
                      ...
                      -----END PUBLIC KEY-----
```

## 七、API Server 安全加固

### 7.1 审计日志

```yaml
# audit-policy.yaml
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
  # 记录 Secret 的所有操作
  - level: Metadata
    resources:
      - group: ""
        resources: ["secrets"]

  # 记录对 Pod 的修改操作
  - level: RequestResponse
    resources:
      - group: ""
        resources: ["pods"]
    verbs: ["create", "update", "patch", "delete"]

  # 忽略系统命名空间的只读请求
  - level: None
    namespaces: ["kube-system"]
    verbs: ["get", "watch", "list"]
```

### 7.2 API Server 安全配置

```bash
kube-apiserver \
  # 禁用匿名访问
  --anonymous-auth=false \
  # 启用准入控制器
  --enable-admission-plugins=NodeRestriction,PodSecurity,LimitRanger,ServiceAccount \
  # 审计配置
  --audit-policy-file=/etc/kubernetes/audit-policy.yaml \
  --audit-log-path=/var/log/kubernetes/audit.log \
  --audit-log-maxage=30 \
  # 加密配置
  --encryption-provider-config=/etc/kubernetes/encryption-config.yaml \
  # 限制请求速率
  --max-requests-inflight=400 \
  --max-mutating-requests-inflight=200
```

### 7.3 kubelet 安全

```yaml
# /var/lib/kubelet/config.yaml
authentication:
  anonymous:
    enabled: false
  webhook:
    enabled: true
authorization:
  mode: Webhook
readOnlyPort: 0 # 禁用只读端口
```

## 八、运行时安全

### 8.1 使用 Falco 检测异常行为

```yaml
# Falco 规则示例
- rule: Shell Spawned in Container
  desc: Detect shell spawned in a container
  condition: >
    spawned_process and container and
    (proc.name in (shell_binaries))
  output: >
    Shell spawned in container (user=%user.name container_id=%container.id
    image=%container.image.repository process=%proc.name)
  priority: WARNING
  tags: [container, shell]
```

### 8.2 使用 OPA/Gatekeeper 强制策略

```yaml
# 禁止容器以 root 运行
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sPSPPrivilegedContainer
metadata:
  name: deny-privileged-containers
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
    namespaces:
      - production
  parameters:
    exemptImages:
      - "trusted-registry.io/*"
```

## 九、安全检查清单

### 9.1 集群级别

- [ ] API Server 禁用匿名访问
- [ ] etcd 启用 TLS 加密
- [ ] 启用审计日志
- [ ] Secret 启用加密存储
- [ ] 定期备份 etcd
- [ ] 禁用不安全的 API（如 v1alpha1）

### 9.2 工作负载级别

- [ ] Pod 配置 securityContext
- [ ] 禁止特权容器
- [ ] 禁止以 root 运行
- [ ] 配置资源限制
- [ ] 使用 Network Policy

### 9.3 访问控制

- [ ] 启用 RBAC
- [ ] 避免使用 cluster-admin
- [ ] 定期审计权限绑定
- [ ] 使用 Service Account 而非用户证书

### 9.4 镜像安全

- [ ] 使用私有镜像仓库
- [ ] 启用镜像签名验证
- [ ] CI/CD 中集成漏洞扫描
- [ ] 定期更新基础镜像

## 十、总结

Kubernetes 安全是一个系统工程，需要从多个维度构建防御体系：

1. **身份认证**：严格管理访问权限，遵循最小权限原则
2. **网络隔离**：使用 Network Policy 限制 Pod 间通信
3. **工作负载安全**：配置 Pod Security Standards
4. **数据保护**：加密 Secret，使用外部密钥管理
5. **镜像安全**：扫描漏洞，签名验证
6. **运行时防护**：监控异常行为，强制安全策略

<Callout type="info">
  **推荐工具**： - **kube-bench**：CIS 安全基准检查 -
  **kube-hunter**：集群渗透测试 - **Polaris**：配置最佳实践检查 -
  **Falco**：运行时安全监控 - **Trivy**：镜像漏洞扫描
</Callout>

安全不是一次性的工作，而是持续的过程。定期审计、更新策略、培训团队，才能构建真正安全的 Kubernetes 环境。
