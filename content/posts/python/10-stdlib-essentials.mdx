你在 Java 里要读写一个文件，需要 `Files.readString(Path.of("config.json"))`——`Path.of` 来自 `java.nio.file`，`Files.readString` 来自同一个包。要解析 JSON，得引入 Jackson 或 Gson——`java.base` 里没有 JSON 支持。要做 HTTP 请求，得用 `HttpClient`（Java 11 才有）或者引入 OkHttp。Java 的标准库像一个工具箱，每个工具都在自己的格子里，但有些常用工具你得自己买。

Python 的标准库有个外号叫"batteries included"——自带电池。你不需要安装任何包就能操作文件系统、解析 JSON、处理日期时间、发送 HTTP 请求、操作正则表达式、压缩文件。Python 的标准库不像 Java 那样"每个功能一个包"，而是把常用功能都内置了。这篇我们用 Java 标准库做参照，看 Python 标准库的核心模块怎么用，哪些比 Java 更方便，哪些不如 Java。

## Java vs Python 标准库对比

| 功能        | Java 标准库                           | Python 标准库                          |
| ----------- | ------------------------------------- | -------------------------------------- |
| 路径操作    | `java.nio.file.Path`                  | `pathlib.Path`                         |
| 日志        | `java.util.logging`（通常用 SLF4J）   | `logging`                              |
| 日期时间    | `java.time`（LocalDateTime 等）       | `datetime` + `zoneinfo`                |
| JSON        | 无（需 Jackson/Gson）                 | `json`                                 |
| HTTP 客户端 | `java.net.http.HttpClient`（Java 11） | `urllib.request`（内置，但不如 httpx） |
| 正则表达式  | `java.util.regex`                     | `re`                                   |
| 集合增强    | 无（用 Guava）                        | `collections`（Counter, deque 等）     |
| 迭代工具    | Stream API                            | `itertools`                            |
| 数据压缩    | `java.util.zip`                       | `zipfile` / `tarfile`                  |
| 配置解析    | `java.util.Properties`                | `configparser` / `tomllib`             |
| 命令行参数  | 无（用 Apache CLI）                   | `argparse`                             |
| CSV         | 无（用 OpenCSV）                      | `csv`                                  |

最明显的差距在 JSON 和命令行参数——Java 标准库没有内置 JSON 解析器，没有命令行参数解析器，这迫使每个 Java 项目都引入 Jackson 和 Apache CLI。Python 的标准库把这些常用功能都包含在内了。

但这不意味着 Python 标准库在所有方面都更强。Java 的 `java.time` API 是 Joda-Time 的作者 Stephen Colebourne 设计的，比 Python 的 `datetime` 更严谨；Java 的 `java.util.concurrent` 提供了完整的并发工具，Python 需要依赖 `threading` + `multiprocessing` 两个模块加第三方库。标准库的比较不是"谁全面"，而是"各自在哪些方面做得更好"。

## pathlib：面向对象的路径操作

Java 的 `Path` 类是 `java.nio.file` 包的核心，Python 的 `pathlib.Path` 功能等价。两者都是面向对象的路径操作，不再需要拼接字符串。

```python
from pathlib import Path


# 构造路径 — / 运算符拼接（Python 独有，Java 用 Path.resolve("child")）
config_path = Path("src") / "config" / "settings.toml"
logs_dir = Path("logs") / "app.log"

# 路径属性
print(config_path.name)      # settings.toml
print(config_path.stem)      # settings
print(config_path.suffix)    # .toml
print(config_path.parent)    # src/config

# 判断存在
if config_path.exists():
    content = config_path.read_text(encoding="utf-8")

# 创建目录（parents=True 相当于 Java 的 Files.createDirectories）
output_dir = Path("output") / "reports"
output_dir.mkdir(parents=True, exist_ok=True)

# 写入文件
(output_dir / "report.txt").write_text("报告内容", encoding="utf-8")

# 遍历目录（rglob 递归，glob 不递归）
python_files = list(Path("src").rglob("*.py"))
```

等价的 Java 写法：

```java
// Java — java.nio.file.Path
var configPath = Path.of("src", "config", "settings.toml");
var logsDir = Path.of("logs", "app.log");

// 路径属性
System.out.println(configPath.getFileName());   // settings.toml
configPath.getParent();  // src/config
// 注意：Java 没有直接获取 stem（去掉扩展名）的方法，需要自己写

// 判断存在 + 读取
if (Files.exists(configPath)) {
    String content = Files.readString(configPath);
}

// 创建目录
var outputDir = Path.of("output", "reports");
Files.createDirectories(outputDir);

// 写入文件
Files.writeString(outputDir.resolve("report.txt"), "报告内容");

// 遍历目录
try (var files = Files.walk(Path.of("src"))) {
    files.filter(p -> p.toString().endsWith(".py"))
         .toList();
}
```

Python 的 `/` 运算符拼接路径比 Java 的 `Path.of("a", "b")` 或 `resolve()` 更自然——它重载了除法运算符，让路径拼接看起来像文件系统的目录结构。`read_text()` 和 `write_text()` 是 `Path` 对象的方法，Java 的 `Files.readString()` 是静态方法——Python 的风格更面向对象。

Python 的 `rglob("*.py")` 比 Java 的 `Files.walk().filter()` 更简洁，但 Java 的 `Files.walk()` 返回 Stream，可以用 `filter`、`map`、`collect` 做链式操作——Java 的 Stream API 在复杂场景下更灵活。

## logging：为什么 Python 不需要 SLF4J

Java 的日志生态是 SLF4J（接口）+ Logback/Log4j（实现）的组合——SLF4J 是门面，Logback 是引擎。这个架构解决了"不同库用不同日志框架"的问题：所有库都面向 SLF4J 编程，运行时选择具体的实现。Python 的 logging 模块本身就是门面加实现的组合体——它既定义了日志接口，也提供了完整的实现，不需要额外的框架。

```python
import logging

# 基本配置 — 相当于 Logback 的 <configuration>
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)

# 获取 logger — 相当于 SLF4J 的 LoggerFactory.getLogger()
logger = logging.getLogger(__name__)  # __name__ 自动取当前模块名


class ArticleService:
    def __init__(self) -> None:
        self.logger = logging.getLogger(f"{__name__}.{self.__class__.__name__}")

    def create(self, title: str, content: str) -> dict:
        self.logger.info("Creating article: %s", title)
        self.logger.debug("Content length: %d", len(content))
        # ...
        return {"id": 1, "title": title}
```

对比 Java 的 SLF4J + Logback：

```java
// Java — SLF4J
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class ArticleService {
    private static final Logger logger =
        LoggerFactory.getLogger(ArticleService.class);

    public Article create(String title, String content) {
        logger.info("Creating article: {}", title);
        logger.debug("Content length: {}", content.length());
        // ...
    }
}
```

两者非常相似：`logging.getLogger(__name__)` 对应 `LoggerFactory.getLogger()`，`logger.info()` 对应 `logger.info()`。区别在于 Python 用 `%s` 做占位符（`%` 风格），Java 用 `{}`（SLF4J 风格）。Python 的 `%s` 是 logging 模块自己处理的——它只在日志级别启用时才格式化字符串，和 SLF4J 的 `{}` 一样避免了不必要的字符串拼接。

### logging 的层级结构

logging 模块用层级结构控制日志输出——每个 logger 有名字，用 `.` 分隔表示父子关系。`techblog.services.article_service` 是 `techblog.services` 的子 logger，也是 `techblog` 的子 logger。子 logger 的日志会向上传播给父 logger 的 handler。

```python
# 按模块配置日志级别
logging.getLogger("techblog").setLevel(logging.WARNING)
logging.getLogger("techblog.services").setLevel(logging.INFO)
logging.getLogger("techblog.api").setLevel(logging.DEBUG)

# techblog.services 的日志 >= INFO 才输出
# techblog.api 的日志 >= DEBUG 才输出
# techblog.core 的日志 >= WARNING 才输出（继承 techblog 的配置）
```

这和 Logback 的 logger 层级完全对应：

```xml
<!-- Logback — 等价的配置 -->
<configuration>
    <logger name="techblog" level="WARN"/>
    <logger name="techblog.services" level="INFO"/>
    <logger name="techblog.api" level="DEBUG"/>
</configuration>
```

Python 的 logging 可以用代码配置（像上面那样），也可以用 `dictConfig` 或 `fileConfig` 加载配置文件。在实际项目中，推荐用 `dictConfig`——和 Logback 的 XML 配置类似，但用 Python 字典：

```python
import logging.config

LOGGING_CONFIG = {
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {
        "standard": {
            "format": "%(asctime)s [%(levelname)s] %(name)s: %(message)s",
        },
    },
    "handlers": {
        "console": {
            "class": "logging.StreamHandler",
            "formatter": "standard",
            "level": "INFO",
        },
        "file": {
            "class": "logging.handlers.RotatingFileHandler",
            "filename": "logs/app.log",
            "maxBytes": 10_000_000,  # 10MB
            "backupCount": 5,
            "formatter": "standard",
            "level": "DEBUG",
        },
    },
    "root": {
        "handlers": ["console", "file"],
        "level": "DEBUG",
    },
}

logging.config.dictConfig(LOGGING_CONFIG)
```

| 维度       | SLF4J + Logback                     | Python logging                            |
| ---------- | ----------------------------------- | ----------------------------------------- |
| 门面/实现  | 分离（SLF4J 是门面）                | 统一（logging 既是门面也是实现）          |
| 配置方式   | XML / Groovy                        | 代码 / dictConfig / fileConfig            |
| 日志级别   | TRACE / DEBUG / INFO / WARN / ERROR | DEBUG / INFO / WARNING / ERROR / CRITICAL |
| 占位符     | `{}`                                | `%s`（logging 内置格式化）                |
| 参数化日志 | `logger.info("msg {}", x)`          | `logger.info("msg %s", x)`                |
| 结构化日志 | Logstash encoder 等插件             | 需要第三方库（structlog）                 |
| 异步日志   | Logback AsyncAppender               | `QueueHandler` + `QueueListener`（内置）  |

Python logging 最大的不足是缺少结构化日志——它默认输出文本行，不像 Logback 那样有成熟的 JSON encoder。生产环境通常需要引入 `structlog` 库来补充这个能力。

## datetime vs java.time

Python 的 `datetime` 模块是标准库里最常被吐槽的——API 设计古老，时区处理不直观。Java 8 引入的 `java.time` API（JSR 310）是业界公认最好的日期时间库之一，设计比 Python 更合理。

```python
from datetime import datetime, date, timedelta, timezone
from zoneinfo import ZoneInfo  # Python 3.9+

# 当前时间
now = datetime.now(timezone.utc)          # 带时区 — 推荐写法
today = date.today()

# 创建指定时间
dt = datetime(2026, 3, 30, 10, 30, tzinfo=ZoneInfo("Asia/Shanghai"))

# 时间运算
tomorrow = now + timedelta(days=1)
next_week = now + timedelta(weeks=1)

# 时区转换
dt_utc = dt.astimezone(timezone.utc)
dt_ny = dt.astimezone(ZoneInfo("America/New_York"))

# 格式化
dt.strftime("%Y-%m-%d %H:%M:%S")    # 2026-03-30 10:30:00
dt.isoformat()                        # 2026-03-30T10:30:00+08:00

# 解析
datetime.fromisoformat("2026-03-30T10:30:00")
datetime.strptime("2026-03-30", "%Y-%m-%d")
```

等价的 Java 写法：

```java
// Java — java.time
var now = LocalDateTime.now(ZoneId.of("Asia/Shanghai"));
var today = LocalDate.now();

// 创建指定时间
var dt = LocalDateTime.of(2026, 3, 30, 10, 30);

// 时间运算
var tomorrow = now.plusDays(1);
var nextWeek = now.plusWeeks(1);

// 时区转换
var dtUtc = dt.atZone(ZoneId.of("Asia/Shanghai"))
               .withZoneSameInstant(ZoneId.of("UTC"));

// 格式化
dt.format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));

// 解析
LocalDateTime.parse("2026-03-30T10:30:00");
LocalDate.parse("2026-03-30");
```

两者的 API 风格不同但功能覆盖基本一致。Java 的 `plusDays()` / `plusWeeks()` 比 Python 的 `timedelta(days=1)` 更明确——你一眼就知道这是"加 1 天"。Python 的 `timedelta` 更灵活——`timedelta(days=1, hours=2, minutes=30)` 一次性加多个时间单位，Java 需要链式调用 `.plusDays(1).plusHours(2).plusMinutes(30)`。

Python 的一个坑是 `datetime.now()` 不带时区——它返回的是"naive datetime"，没有时区信息。Java 的 `LocalDateTime.now()` 也是如此，但 Java 的类型系统帮你区分了 `LocalDateTime`（无时区）和 `ZonedDateTime`（有时区）。Python 的 `datetime` 类型既可以是 naive 也可以是 aware，运行时才能判断——这是 Python 的老问题，类型系统帮不了你。

```python
# Python 的坑 — naive vs aware
naive = datetime.now()                        # 无时区 — 危险
aware = datetime.now(timezone.utc)            # 有时区 — 安全

# naive 和 aware 不能直接比较或运算
# naive < aware  → TypeError
```

| 维度     | `java.time`                         | `datetime`                                  |
| -------- | ----------------------------------- | ------------------------------------------- |
| 无时区   | `LocalDateTime`（类型明确）         | `datetime.now()`（naive，类型不区分）       |
| 有时区   | `ZonedDateTime`（类型明确）         | `datetime(tzinfo=...)`（aware，类型不区分） |
| 时间差   | `Duration` / `Period`               | `timedelta`（统一处理）                     |
| 格式化   | `DateTimeFormatter`（线程安全）     | `strftime`（实例方法）                      |
| 解析     | `LocalDate.parse()`（ISO 默认支持） | `fromisoformat()`（Python 3.7+）            |
| 时区数据 | JDK 内置                            | 需要安装 `tzdata`（Linux 上）               |

`java.time` 的设计更严谨——无时区和有时区是不同的类型，编译器帮你防止混淆。Python 需要靠约定（永远用 `timezone.utc`）来避免 naive datetime 的问题。

## collections：Java 没有的便捷数据结构

`collections` 模块提供了几个 Java 标准库里没有的直接等价物的数据结构：

```python
from collections import Counter, defaultdict, deque

# Counter — 计数器（统计频率）
word_counts = Counter(["python", "java", "python", "go", "python", "java"])
print(word_counts)        # Counter({'python': 3, 'java': 2, 'go': 1})
print(word_counts.most_common(2))  # [('python', 3), ('java', 2)]

# defaultdict — 默认值字典（Java 没有）
# 访问不存在的 key 时自动创建默认值，而不是抛 KeyError
user_articles = defaultdict(list)
user_articles["alice"].append("Python 入门")
user_articles["alice"].append("Java 实战")
user_articles["bob"].append("Go 并发")
# user_articles["bob"] 不会报 KeyError，返回 []

# deque — 双端队列（Java 的 ArrayDeque）
recent_logs = deque(maxlen=100)  # 固定长度，超出自动丢弃最老的
recent_logs.append("log1")
recent_logs.append("log2")
recent_logs.appendleft("log0")   # 从左边插入
recent_logs.pop()                 # 从右边弹出
```

Java 没有直接等价 `Counter` 的类——你需要用 `Map<String, Long>` + `merge()` 或 Stream 的 `Collectors.counting()`。`defaultdict` 的功能可以用 `Map.getOrDefault()` 或 `computeIfAbsent()` 模拟，但没有 Python 那么简洁。`deque` 在 Java 里是 `ArrayDeque`，功能基本一致。

## json：Java 标准库的空白

Python 的 `json` 模块是标准库的一部分——不需要安装任何包就能序列化和反序列化 JSON。Java 标准库没有内置 JSON 支持，你必须引入 Jackson 或 Gson。

```python
import json

# 序列化
data = {"title": "Python 入门", "tags": ["python", "tutorial"], "views": 1024}
json_str = json.dumps(data, indent=2, ensure_ascii=False)

# 反序列化
loaded = json.loads(json_str)

# 文件读写
Path("data.json").write_text(json.dumps(data, indent=2), encoding="utf-8")
data = json.loads(Path("data.json").read_text(encoding="utf-8"))

# 自定义序列化（处理 datetime 等类型）
def json_serializer(obj):
    if isinstance(obj, datetime):
        return obj.isoformat()
    raise TypeError(f"Object of type {type(obj)} is not JSON serializable")

json_str = json.dumps({"created_at": datetime.now()}, default=json_serializer)
```

等价的 Java 写法需要 Jackson：

```java
// Java — 需要引入 Jackson
ObjectMapper mapper = new ObjectMapper();

// 序列化
var data = Map.of("title", "Python 入门", "views", 1024);
String json = mapper.writerWithDefaultPrettyPrinter().writeValueAsString(data);

// 反序列化
Map<String, Object> loaded = mapper.readValue(json, Map.class);
```

Python 的 `json` 模块功能简单直接——`dumps` 序列化，`loads` 反序列化。但它不支持 JSON Schema 验证、注解驱动序列化、循环引用处理等高级功能。这些场景需要用 Pydantic（上一篇讲过）或者 `orjson`（更快的第三方 JSON 库）。

Java 的 Jackson 功能远比 Python 的 `json` 模块强大——注解驱动的序列化、多态类型处理、自定义序列化器、流式 API 等。但 Jackson 是第三方库，不是标准库。这就是两个语言的 trade-off：Python 标准库提供了 80% 场景够用的 JSON 支持，Java 标准库没有但提供了更强大的第三方选择。

## itertools：函数式数据处理

`itertools` 提供了一组用于高效操作的迭代器工具。Java 的 Stream API 覆盖了大部分相同的功能，但 `itertools` 的一些组合操作在 Java 里需要更多代码：

```python
from itertools import chain, groupby, islice, product, combinations

# chain — 连接多个可迭代对象
all_ids = chain([1, 2, 3], [4, 5], [6, 7, 8])
# Java: Stream.of(list1, list2, list3).flatMap(Collection::stream)

# groupby — 分组（注意：需要先排序）
transactions = sorted(transactions, key=lambda t: t["date"][:7])
for month, group in groupby(transactions, key=lambda t: t["date"][:7]):
    amounts = [t["amount"] for t in group]
    print(f"{month}: {sum(amounts)}")
# Java: transactions.stream().collect(Collectors.groupingBy(...))

# product — 笛卡尔积
colors = ["red", "blue"]
sizes = ["S", "M", "L"]
variants = product(colors, sizes)
# Java: 没有直接等价物，需要嵌套循环

# combinations — 组合（不关心顺序）
team = ["Alice", "Bob", "Charlie"]
pairs = combinations(team, 2)
# Java: 没有直接等价物

# islice — 切片迭代器（不加载全部数据）
first_100 = islice(huge_data_source, 100)
# Java: hugeDataStream.limit(100)
```

`itertools` 的设计哲学是"惰性计算"——它返回的是迭代器，不是列表。只有在需要时才计算下一个元素，适合处理大数据集。Java 的 Stream API 也是惰性的，两者的思路一样。`itertools` 在组合数学（`product`、`combinations`、`permutations`）方面比 Stream API 更丰富，但 Stream API 在通用数据处理（`map`、`filter`、`reduce`、`collect`）方面更直观。

## Trade-off：batteries included 的得与失

Python 的标准库确实比 Java 的更"全"——JSON、CSV、命令行参数、配置解析、邮件发送、压缩文件，这些 Java 需要第三方库的功能，Python 都内置了。对于写脚本、做数据处理、快速原型来说，Python 标准库的覆盖面让开发效率很高——你不需要在 `pom.xml` 里加依赖，不需要管理版本冲突，`import` 就能用。

但这种"大而全"也有代价。Python 标准库里有些模块的设计已经过时了——`datetime` 的 naive/aware 区分不如 `java.time` 严谨，`logging` 的结构化日志不如 Logback 成熟，`urllib.request` 的 API 远不如 httpx 直观。标准库里的模块一旦发布就很难做破坏性改动（Python 有严格的向后兼容承诺），所以过时的设计只能靠新的第三方库来弥补。

Java 的策略相反——标准库保持精简和稳定，把创新留给第三方生态。Jackson 比 `java.base` 里可能存在的 JSON 实现更好，Guava 比 `java.util` 里的工具方法更丰富，Spring 的日志抽象比 `java.util.logging` 更实用。代价是每个 Java 项目都有一个长长的依赖列表和偶尔的版本冲突。

实际使用中的建议：Python 的 `pathlib`、`json`、`re`、`collections`、`itertools` 直接用标准库就行，质量足够好。`datetime` 建议始终带时区。`logging` 在简单场景下够用，生产环境考虑 `structlog`。HTTP 请求直接用第三方 `httpx`，别用 `urllib.request`。
