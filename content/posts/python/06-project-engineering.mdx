你用 Maven 创建了一个 Java 项目，目录结构是固定的——`src/main/java` 放源码，`src/main/resources` 放配置文件，`src/test/java` 放测试，`pom.xml` 管一切。任何一个 Java 开发者打开别人的项目，都不需要猜代码在哪。现在你打开一个 Python 项目，发现每个人的目录结构都不一样：有人用 `src/`，有人不用；有人用 `requirements.txt`，有人用 `setup.py`，还有人用 `pyproject.toml`；有人用 black 格式化，有人用 autopep8，还有人什么都不用。

这不是 Python 社区的混乱，而是 Python 工程化标准还在快速演进。好消息是，2024 年之后基本尘埃落定：`pyproject.toml` 成为统一配置文件，`ruff` 一统格式化和 lint，`src layout` 成为推荐的项目结构。这篇我们用 Java 的 Maven 体系做参照，把 Python 的工程化体系理清楚。

## Java vs Python 工程化全景

| 概念       | Java                          | Python                         |
| ---------- | ----------------------------- | ------------------------------ |
| 项目元数据 | `pom.xml` / `build.gradle`    | `pyproject.toml`               |
| 目录规范   | Maven 标准目录结构            | src layout（社区约定，非强制） |
| 依赖管理   | Maven Central + 依赖范围      | PyPI + dependency-groups       |
| 代码格式化 | Google Java Format / Spotless | ruff format                    |
| 代码检查   | Checkstyle / PMD / SpotBugs   | ruff check / mypy              |
| 构建工具   | Maven / Gradle                | uv / poetry                    |
| 预提交检查 | Maven 插件 / Spotless apply   | pre-commit hooks               |
| 锁文件     | 无（依赖版本在 pom 中固定）   | `uv.lock` / `poetry.lock`      |

最大的区别在第一行和最后一行：Java 的 `pom.xml` 既是元数据又是构建脚本，Python 的 `pyproject.toml` 只是元数据，构建逻辑由工具（uv/poetry）读取它来执行。Java 不需要锁文件，因为 `pom.xml` 里已经锁定了版本；Python 需要，因为 PyPI 上的包更新更频繁，依赖解析更复杂。

## 项目结构：没有强制标准，但有社区共识

Java 的标准目录结构是 Maven 规定的，你在 `pom.xml` 里声明 `src/main/java`，Maven 就知道去哪里找源码。Python 没有这种"官方强制"的目录标准，但 PyPA（Python Packaging Authority）推荐的 src layout 已经成为事实标准。

```
techblog/
├── pyproject.toml          # 统一配置文件（相当于 pom.xml）
├── uv.lock                 # 锁文件（确保依赖版本一致）
├── README.md
├── .gitignore
├── .pre-commit-config.yaml # Git 提交前自动检查
├── src/
│   └── techblog/
│       ├── __init__.py     # 包标识（__version__、__all__）
│       ├── __main__.py     # python -m techblog 入口
│       ├── core/
│       │   ├── __init__.py
│       │   ├── config.py
│       │   └── exceptions.py
│       ├── models/
│       │   ├── __init__.py
│       │   └── article.py
│       └── services/
│           ├── __init__.py
│           └── article_service.py
├── tests/
│   ├── conftest.py         # pytest 共享 fixture
│   ├── test_models/
│   │   └── test_article.py
│   └── test_services/
│       └── test_article_service.py
└── scripts/
    └── seed_data.py
```

对比 Java Maven 标准结构：

```
techblog/
├── pom.xml
├── src/
│   ├── main/
│   │   ├── java/com/example/techblog/
│   │   │   ├── config/
│   │   │   ├── model/
│   │   │   ├── service/
│   │   │   └── TechblogApplication.java
│   │   └── resources/
│   │       └── application.yml
│   └── test/
│       └── java/com/example/techblog/
│           ├── model/
│           └── service/
```

两者思路一样：源码和测试分开，按功能模块组织。区别在于：

- Java 的包名是反向域名（`com.example.techblog`），Python 直接用项目名（`techblog`）
- Java 的 `resources/` 对应 Python 的各种配置文件（`.env`、`pyproject.toml` 中的配置）
- Python 多了 `__init__.py`——每个目录下都必须有这个文件，告诉 Python "这是一个包"
- Java 的 `src/main` 和 `src/test` 是 Maven 强制的，Python 的 `src/` 和 `tests/` 是社区约定

Python 的 `__init__.py` 常被误解为空文件。它至少应该做两件事：

```python
# src/techblog/__init__.py
__version__ = "1.0.0"

# 控制公开 API — from techblog import * 时只导出这些
__all__ = ["ArticleService", "create_app"]

# 提供便捷导入
from techblog.services.article_service import ArticleService
from techblog.core.config import create_app
```

这和 Java 的"每个 `public class` 都在单独的文件中"的思路不同。Python 允许一个文件里放多个类，`__init__.py` 用 `__all__` 来控制哪些是对外公开的——类似 Java 的 `public` vs package-private，但粒度更灵活。

## pyproject.toml：一个文件替代 pom.xml

Java 的 `pom.xml` 同时承载三件事：项目元数据（`groupId`、`artifactId`）、依赖声明（`<dependencies>`）、构建插件（`<plugins>`）。Python 的 `pyproject.toml` 承载的项目元数据和依赖声明类似，但构建插件变成了各个工具各自的配置段。

一个完整的 `pyproject.toml` 看起来像这样：

```toml
[project]
name = "techblog"
version = "0.1.0"
description = "A tech blog built with Python"
requires-python = ">=3.12"
dependencies = [
    "fastapi>=0.115.0",
    "uvicorn[standard]>=0.34.0",
    "pydantic>=2.0.0",
    "pydantic-settings>=2.0.0",
    "sqlalchemy>=2.0.0",
]

# 开发依赖 — 相当于 Maven 的 <scope>test</scope>
[dependency-groups]
dev = [
    "pytest>=8.0.0",
    "pytest-asyncio>=0.24.0",
    "ruff>=0.9.0",
    "mypy>=1.14.0",
    "pre-commit>=4.0.0",
]

# 项目入口 — 相当于 Maven 的 mainClass
[project.scripts]
techblog = "techblog.__main__:main"

# 构建系统 — 声明用什么工具打包
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

# ---- 以下各 [tool.xxx] 段相当于 Maven 的 <plugins> ----

[tool.ruff]
target-version = "py312"
line-length = 88

[tool.ruff.lint]
select = ["E", "W", "F", "I", "N", "UP", "B", "SIM", "RUF"]
ignore = ["E501"]

[tool.ruff.lint.isort]
known-first-party = ["techblog"]

[tool.ruff.format]
quote-style = "double"
indent-style = "space"

[tool.mypy]
python_version = "3.12"
strict = true

[tool.pytest.ini_options]
testpaths = ["tests"]
asyncio_mode = "auto"
```

和 `pom.xml` 的关键区别：

| 维度     | pom.xml                                          | pyproject.toml                           |
| -------- | ------------------------------------------------ | ---------------------------------------- |
| 格式     | XML                                              | TOML（更简洁）                           |
| 依赖范围 | `<scope>compile</scope>` / `<scope>test</scope>` | `[dependencies]` / `[dependency-groups]` |
| 构建插件 | `<build><plugins><plugin>`                       | `[tool.ruff]` / `[tool.mypy]` 等独立段   |
| 版本锁定 | 依赖版本直接写在 pom 中                          | `uv.lock` 锁文件单独管理                 |
| 多模块   | `<modules><module>`                              | workspace（较新特性）                    |
| 仓库地址 | `<repositories>`                                 | 默认 PyPI，私有源需配置 `[tool.uv]`      |

TOML 比 XML 可读性好得多。一个包含 ruff、mypy、pytest 配置的 `pyproject.toml` 大约 60 行，等价的 `pom.xml` 要 200 行以上——光 XML 的闭合标签就占了不少行数。

## uv：Python 的 Maven

Java 有 Maven 和 Gradle 两个主流构建工具，Python 现在有 uv 和 poetry。uv 是 Astral 团队（也是 ruff 的开发者）用 Rust 写的新一代工具，速度快 10-100 倍，已经成了 2025 年新项目的首选。

```bash
# 创建项目（相当于 mvn archetype:generate）
uv init techblog
cd techblog

# 添加依赖（相当于在 pom.xml 中加 <dependency>）
uv add fastapi uvicorn
uv add --dev pytest ruff mypy    # 开发依赖

# 安装全部依赖（相当于 mvn install）
uv sync

# 运行（相当于 mvn exec:java）
uv run python -m techblog

# 运行测试（相当于 mvn test）
uv run pytest

# 格式化 + 检查（相当于 mvn spotless:check）
uv run ruff format .
uv run ruff check --fix .
```

对比 Maven 命令：

| 操作      | Maven                    | uv                     |
| --------- | ------------------------ | ---------------------- |
| 创建项目  | `mvn archetype:generate` | `uv init`              |
| 添加依赖  | 手动编辑 pom.xml         | `uv add <package>`     |
| 安装依赖  | `mvn install`            | `uv sync`              |
| 编译+运行 | `mvn compile exec:java`  | `uv run python -m xxx` |
| 运行测试  | `mvn test`               | `uv run pytest`        |
| 代码检查  | `mvn checkstyle:check`   | `uv run ruff check .`  |
| 打包      | `mvn package`            | `uv build`             |

一个重要的差异：Maven 不需要"虚拟环境"，因为每个项目用自己 JDK 和本地仓库。Python 需要 `uv` 自动创建的 `.venv` 目录来隔离依赖——这是 Python 没有 JDK 内置包管理器的结果。上一篇我们讲过这个区别，这里不再赘述。

## ruff：一个工具替代 Checkstyle + PMD + SpotBugs

Java 的代码质量工具通常是三个独立的：Checkstyle 管格式规范，PMD 管代码逻辑，SpotBugs 管潜在 bug。Python 曾经也是这样：flake8 做 lint，black 做格式化，isort 管导入排序，pyupgrade 管语法升级——四五个工具，四五种配置格式。

ruff 把这些全部统一了。它用 Rust 实现，速度快 10-100 倍，配置全写在 `pyproject.toml` 的 `[tool.ruff]` 段里。

```bash
# 代码检查（替代 flake8 + isort + pyupgrade + flake8-bugbear）
uv run ruff check .

# 自动修复能修的问题
uv run ruff check --fix .

# 代码格式化（替代 black）
uv run ruff format .

# 查看某个规则的说明
uv run ruff rule E501
```

ruff 的规则集对应关系：

| 旧工具（Java 也有类似工具） | ruff 规则集   | Java 对应物               |
| --------------------------- | ------------- | ------------------------- |
| flake8（基础 lint）         | `E`, `W`      | Checkstyle 基础规则       |
| pyflakes（未使用变量）      | `F`           | IDE 的 unused warning     |
| isort（导入排序）           | `I`           | IDE 自动 organize imports |
| pyupgrade（语法升级）       | `UP`          | IDE 的 quick-fix          |
| flake8-bugbear（常见陷阱）  | `B`           | PMD                       |
| flake8-simplify（简化写法） | `SIM`         | PMD / IntelliJ 警告       |
| pep8-naming（命名规范）     | `N`           | Checkstyle naming 检查    |
| black（格式化）             | `ruff format` | Google Java Format        |

Java 的 Checkstyle 配置通常是一个独立的 XML 文件：

```xml
<!-- checkstyle.xml -->
<module name="Checker">
  <module name="LineLength">
    <property name="max" value="120"/>
  </module>
  <module name="TreeWalker">
    <module name="NamingConvention"/>
    <module name="UnusedImports"/>
  </module>
</module>
```

ruff 的等价配置直接写在 `pyproject.toml` 里，不需要额外文件：

```toml
[tool.ruff]
line-length = 88
[tool.ruff.lint]
select = ["E", "W", "F", "I", "N", "UP", "B", "SIM"]
```

ruff 还有一个 Java 工具链没有的优势：`--fix` 可以自动修复很多问题。Java 的 Checkstyle 只能报错，你要么手动改，要么用 Spotless 的 `apply` 功能（但 Spotless 的自动修复能力远不如 ruff 全面）。

## pre-commit：Java Maven 插件的 Python 版本

Java 里你用 Maven 的 `maven-checkstyle-plugin` 在 `verify` 阶段跑检查，确保不合规的代码不能发布。Python 的 pre-commit 在 `git commit` 时自动跑检查，更早拦截问题。

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.9.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v5.0.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
        args: ["--maxkb=500"]
```

```bash
# 安装 Git 钩子
uv run pre-commit install

# 首次对已有代码运行一次
uv run pre-commit run --all-files

# 之后每次 git commit 自动触发
git commit -m "feat: add article service"
# pre-commit 自动跑 ruff check 和 ruff format
```

Java 没有等价的 pre-commit 工具。Spring Boot 项目通常在 CI 的 Maven 构建阶段做检查，或者依赖 IDE 的 save actions。pre-commit 的好处是问题在本地就被拦截了，不需要等 CI 报红。

## 虚拟环境：Python 独有的概念

Java 不需要虚拟环境。你安装了 JDK 17，所有项目都用它；Maven 的本地仓库 `~/.m2/repository` 全局共享，不同项目通过 `pom.xml` 声明各自的依赖版本。Python 不行——系统级 Python 解释器只有一个，如果你直接 `pip install fastapi`，所有项目都会受影响。所以 Python 用虚拟环境（virtual environment）做隔离。

```bash
# uv 自动创建和管理 .venv，你不需要手动操作
uv venv                    # 创建虚拟环境（通常 uv sync 会自动做）
uv sync                    # 在虚拟环境中安装依赖
uv run python -m techblog  # 在虚拟环境中运行
```

虚拟环境本质上是一个独立的 Python 解释器副本 + 一个独立的 `site-packages` 目录。`.venv/bin/python` 是虚拟环境的解释器，它和系统 Python 共享标准库，但第三方包是隔离的。

| 维度     | Java                     | Python                                         |
| -------- | ------------------------ | ---------------------------------------------- |
| 解释器   | 每个 JDK 实例完全独立    | 共享系统 Python，虚拟环境隔离包                |
| 全局仓库 | `~/.m2/repository`       | `~/.cache/uv`（uv 的包缓存）                   |
| 隔离方式 | 每个项目独立 JDK（可选） | `.venv` 目录                                   |
| 激活方式 | 不需要                   | `source .venv/bin/activate`（可选，uv 不需要） |

uv 的 `uv run` 命令让你完全不需要手动激活虚拟环境——它自动检测 `.venv` 并在其中执行命令。这是 uv 比 `pip + venv` 方便的核心原因之一。

## Trade-off：Python 工程化的设计取舍

Python 的工程化标准没有 Maven 那样"一个规范管一切"的权威。`pyproject.toml` 是 PEP 621 定义的标准，但它只规定了项目元数据的格式，不规定目录结构——src layout 是社区推荐，不是强制。这和 Java 不同：Maven 的标准目录结构是强制的，你不按它的规则来，Maven 就找不到你的代码。

这种"弱规范"带来了灵活性，也带来了混乱。Java 开发者打开任何 Maven 项目，目录结构都一样；Python 开发者打开不同项目，可能看到 flat layout、src layout、甚至没有 layout。好处是你可以根据项目规模选择合适的方式——脚本级别的小项目不需要 src layout，但团队协作的工程化项目应该用。坏处是新手上路的学习成本更高，需要自己判断什么时候该用什么结构。

ruff 的统一是近年来最好的变化。Java 的 Checkstyle + PMD + SpotBugs + Google Java Format 分属不同项目，配置分散，版本兼容性需要手动管理。ruff 把所有这些能力集中到一个工具里，配置统一写在 `pyproject.toml`。但 ruff 的规则集还在快速迭代，新版本可能引入新规则导致现有代码报错——建议在 `pyproject.toml` 里锁定 ruff 的版本（`uv add --dev ruff==0.9.0`），升级时再统一更新。
