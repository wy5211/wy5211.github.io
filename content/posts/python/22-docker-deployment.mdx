# Docker 部署与生产优化

## 前言

在前面 21 篇文章中，我们用 FastAPI 构建了一个完整的博客系统 API，涵盖认证、缓存、WebSocket、后台任务等功能。但到目前为止，所有内容都运行在开发环境中。要把这个应用真正推向生产，我们需要解决一系列问题：环境一致性、进程管理、反向代理、HTTPS、日志收集、健康检查...

Docker 是解决这些问题的标准方案。如果你有 Java 背景，可以把它理解为"更轻量的 JVM"——一次构建，处处运行；如果你来自 Node.js 生态，Docker 的作用类似于 PM2 + 环境隔离的终极形态。

**本文将带你：**

1. 编写生产级多阶段 Dockerfile
2. 使用 Docker Compose 编排完整的服务栈
3. 配置 Gunicorn + Uvicorn 生产服务器
4. 设置 Nginx 反向代理与安全策略
5. 实现健康检查、日志管理和环境变量管理

## 核心概念

### 容器化部署架构总览

```
                    ┌──────────────┐
                    │   客户端      │
                    └──────┬───────┘
                           │ HTTPS
                    ┌──────▼───────┐
                    │    Nginx     │  反向代理 + 静态文件 + SSL
                    │   :443/:80   │
                    └──┬───────┬───┘
                       │       │
              ┌────────▼┐  ┌──▼────────┐
              │  Web    │  │  Worker   │
              │  FastAPI│  │  Celery   │
              │  :8000  │  │           │
              └──┬──┬───┘  └──┬──┬────┘
                 │  │         │  │
        ┌────────┘  │    ┌────┘  └────────┐
        │           │    │                 │
   ┌────▼────┐ ┌───▼────▼──┐  ┌────────────▼──┐
   │PostgreSQL│ │   Redis   │  │   RabbitMQ    │
   │  :5432  │ │  :6379    │  │   :5672       │
   └─────────┘ └───────────┘  └───────────────┘
```

### Docker vs 传统部署

| 维度       | 传统部署                        | Docker 部署                       |
| ---------- | ------------------------------- | --------------------------------- |
| 环境一致性 | 依赖系统环境，"在我机器上能跑"  | 镜像打包一切，完全一致            |
| 部署速度   | 手动配置，容易出错              | `docker compose up` 一键启动      |
| 扩缩容     | 需要额外配置                    | `docker compose up --scale web=3` |
| 隔离性     | 进程共享系统资源                | 容器级别隔离                      |
| 回滚       | 复杂，依赖运维经验              | 切换镜像版本即可                  |
| 类比       | 直接运行 `.jar` / `node app.js` | 类似 Kubernetes Pod 的本地版本    |

### 多阶段构建概念

多阶段构建（Multi-stage Build）是 Docker 生产部署的核心技巧。编译型语言（Go、Rust、Java）在构建阶段需要编译器和工具链，但运行时不需要。Python 虽然是解释型语言，但同样适用这个模式：

```
阶段 1: builder —— 安装依赖（pip install）
阶段 2: runtime —— 只复制依赖产物 + 源代码（更小的镜像）
```

## 实战演练

### 示例 1：多阶段 Dockerfile

```dockerfile
# =============================================
# 博客系统 API - 生产级 Dockerfile
# =============================================

# ---------- 阶段 1: 构建阶段 ----------
FROM python:3.12-slim AS builder

# 设置环境变量（避免 Python 生成 .pyc，保证日志实时输出）
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

# 安装系统构建依赖（编译某些 Python 包需要，如 psycopg2）
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# 创建并切换工作目录
WORKDIR /build

# 先复制依赖文件（利用 Docker 缓存层，依赖不变时无需重新安装）
COPY pyproject.toml ./

# 安装依赖到指定目录（不安装项目本身）
RUN pip install --no-cache-dir --prefix=/install -r <(python -c "import tomllib; f=open('pyproject.toml','rb'); d=tomllib.load(f); print('\n'.join(d['project']['dependencies']))")

# ---------- 阶段 2: 运行阶段 ----------
FROM python:3.12-slim AS runtime

# 设置标签
LABEL maintainer="blog-team" \
      description="Blog API Production Image" \
      version="1.0"

# 创建非 root 用户（安全最佳实践）
RUN groupadd -r appuser && useradd -r -g appuser -d /app -s /sbin/nologin appuser

# 安装运行时系统依赖（只需要 PostgreSQL 客户端库，不需要 dev 包）
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 \
    curl \
    && rm -rf /var/lib/apt/lists/*

# 设置环境变量
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PATH="/opt/venv/bin:$PATH" \
    # Pydantic 设置
    APP_ENV=production \
    # Uvicorn 设置
    UVICORN_HOST=0.0.0.0 \
    UVICORN_PORT=8000

# 从构建阶段复制已安装的 Python 依赖
COPY --from=builder /install /opt/venv

# 创建应用目录并设置权限
WORKDIR /app
RUN mkdir -p /app/logs /app/uploads && chown -R appuser:appuser /app

# 复制应用源代码
COPY --chown=appuser:appuser . .

# 切换到非 root 用户
USER appuser

# 暴露端口
EXPOSE 8000

# 健康检查（每 30 秒检查一次，超时 5 秒，连续 3 次失败标记为不健康）
HEALTHCHECK --interval=30s --timeout=5s --start-period=40s --retries=3 \
    CMD curl -f http://localhost:8000/api/health || exit 1

# 启动命令（Gunicorn + Uvicorn Worker）
CMD ["gunicorn", "app.main:app", \
     "--bind", "0.0.0.0:8000", \
     "--workers", "4", \
     "--worker-class", "uvicorn.workers.UvicornWorker", \
     "--timeout", "120", \
     "--keep-alive", "5", \
     "--access-logfile", "-", \
     "--error-logfile", "-", \
     "--log-level", "info"]
```

如果你使用 `uv` 作为包管理器，构建阶段可以这样简化：

```dockerfile
# ---------- 阶段 1: 构建阶段（使用 uv）----------
FROM python:3.12-slim AS builder

RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# 安装 uv（极速包管理器）
COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv

WORKDIR /build

COPY pyproject.toml uv.lock ./

# 使用 uv 安装依赖到虚拟环境
RUN uv sync --frozen --no-dev --no-install-project

# ---------- 阶段 2: 运行阶段 ----------
FROM python:3.12-slim AS runtime

RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 curl \
    && rm -rf /var/lib/apt/lists/*

RUN groupadd -r appuser && useradd -r -g appuser -d /app -s /sbin/nologin appuser

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \

# 从构建阶段复制虚拟环境
COPY --from=builder /build/.venv /opt/venv
```

### 示例 2：Gunicorn 生产配置文件

```python
# gunicorn.conf.py
"""
Gunicorn 生产配置

类比 Java：相当于 Spring Boot 内嵌 Tomcat 的线程池配置
类比 Node.js：相当于 cluster 模式的 worker 数量配置
"""
import multiprocessing
import os

# ========== 服务器绑定 ==========
bind = f"0.0.0.0:{os.getenv('PORT', '8000')}"

# ========== Worker 配置 ==========
# Worker 数量公式：(2 x CPU 核心数) + 1
# 这是因为 Python 有 GIL，CPU 密集型任务需要多进程
# 但 FastAPI 主要是 I/O 密集型（异步），实际可以适当减少
workers = int(os.getenv("WEB_CONCURRENCY", str(multiprocessing.cpu_count() * 2 + 1)))

# Worker 类：使用 Uvicorn 的 ASGI Worker
worker_class = "uvicorn.workers.UvicornWorker"

# 每个 Worker 的线程数（Uvicorn Worker 下一般保持 1，由 asyncio 处理并发）
threads = 1

# Worker 启动超时
timeout = int(os.getenv("GUNICORN_TIMEOUT", "120"))

# Keep-Alive 超时（秒）
# Nginx 和 Uvicorn 之间的长连接保持时间
keepalive = 5

# 最大请求数：Worker 处理这么多请求后自动重启（防止内存泄漏）
max_requests = int(os.getenv("MAX_REQUESTS", "5000"))
max_requests_jitter = 500  # 添加随机偏移，避免所有 Worker 同时重启

# ========== 日志配置 ==========
# 输出到 stdout/stderr（Docker 会自动收集）
accesslog = "-"       # 访问日志输出到 stdout
errorlog = "-"        # 错误日志输出到 stderr
loglevel = os.getenv("LOG_LEVEL", "info")

# 日志格式（包含请求耗时）
access_log_format = '%(h)s %(l)s %(u)s %(t)s "%(r)s" %(s)s %(b)s "%(f)s" "%(a)s" %(D)s'

# ========== 安全配置 ==========
limit_request_line = 8190       # 请求行最大大小
limit_request_fields = 100      # 请求头最大数量
limit_request_field_size = 8190 # 单个请求头最大大小

# ========== 进程命名 ==========
proc_name = "blog_api"

# ========== 预加载应用 ==========
# 启动时预加载应用代码（节省内存，但热更新需要重启）
preload_app = True

# ========== 钩子函数 ==========

def on_starting(server):
    """Worker 启动前的回调"""
    print(f"[Gunicorn] 启动 {workers} 个 Uvicorn Worker...")

def post_fork(server, worker):
    """Worker fork 后的回调"""
    server.log.info(f"[Gunicorn] Worker {worker.pid} 已启动")

def worker_int(worker):
    """Worker 被中断时的回调"""
    worker.log.info(f"[Gunicorn] Worker {worker.pid} 收到中断信号")

def worker_abort(worker):
    """Worker 异常退出时的回调"""
    worker.log.info(f"[Gunicorn] Worker {worker.pid} 异常退出")
```

### 示例 3：FastAPI 生产入口与配置

```python
"""
app/main.py - 生产环境入口
"""
import logging
import os
from contextlib import asynccontextmanager

from fastapi import FastAPI
from fastapi.middleware.gzip import GZipMiddleware

# ========== 日志配置 ==========

def setup_logging() -> None:
    """配置生产环境日志"""
    log_level = os.getenv("LOG_LEVEL", "INFO")
    log_format = (
        "%(asctime)s | %(levelname)-8s | %(name)s | %(message)s"
    )

    logging.basicConfig(
        level=getattr(logging, log_level),
        format=log_format,
        datefmt="%Y-%m-%d %H:%M:%S",
    )

    # 降低第三方库的日志级别
    logging.getLogger("uvicorn.access").setLevel(logging.WARNING)
    logging.getLogger("sqlalchemy.engine").setLevel(logging.WARNING)


# ========== 应用配置 ==========

@asynccontextmanager
async def lifespan(app: FastAPI):
    """应用生命周期管理"""
    setup_logging()
    logging.getLogger(__name__).info("博客系统 API 启动中...")

    # 启动时初始化连接池
    from app.core.database import engine
    from app.core.redis import redis_client

    logging.getLogger(__name__).info("[DB] 数据库连接池已就绪")
    logging.getLogger(__name__).info("[Redis] Redis 连接已就绪")

    yield

    # 关闭时清理连接
    await engine.dispose()
    await redis_client.close()
    logging.getLogger(__name__).info("博客系统 API 已关闭")


app = FastAPI(
    title="博客系统 API",
    docs_url="/api/docs" if os.getenv("APP_ENV") == "development" else None,
    redoc_url="/api/redoc" if os.getenv("APP_ENV") == "development" else None,
    lifespan=lifespan,
)


# ========== 中间件 ==========

# Gzip 压缩（响应体 > 1KB 时启用）
app.add_middleware(GZipMiddleware, minimum_size=1000)


# ========== 健康检查端点 ==========

@app.get("/api/health")
async def health_check():
    """
    健康检查端点

    Docker HEALTHCHECK 和负载均衡器会调用此端点
    """
    return {"status": "healthy", "version": "1.0.0"}


@app.get("/api/ready")
async def readiness_check():
    """
    就绪检查端点

    检查所有依赖服务是否可用
    """
    checks = {}

    # 检查数据库连接
    try:
        from app.core.database import async_session
        from sqlalchemy import text

        async with async_session() as session:
            await session.execute(text("SELECT 1"))
        checks["database"] = "ok"
    except Exception as e:
        checks["database"] = f"error: {e}"

    # 检查 Redis 连接
    try:
        from app.core.redis import redis_client
        await redis_client.ping()
        checks["redis"] = "ok"
    except Exception as e:
        checks["redis"] = f"error: {e}"

    all_ok = all(v == "ok" for v in checks.values())
    return {
        "status": "ready" if all_ok else "not_ready",
        "checks": checks,
    }


# ========== 注册路由 ==========

from app.api.v1.router import api_router

app.include_router(api_router, prefix="/api/v1")
```

### 示例 4：环境变量管理（Pydantic Settings）

```python
"""
app/core/config.py - 生产环境配置
使用 Pydantic Settings 管理环境变量，自动验证和类型转换
"""
from functools import lru_cache
from pydantic import Field, field_validator, PostgresDsn, RedisDsn
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    """应用配置（从环境变量读取）"""

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        case_sensitive=False,
    )

    # ========== 应用配置 ==========
    app_env: str = Field(default="production", alias="APP_ENV")
    app_name: str = Field(default="Blog API", alias="APP_NAME")
    debug: bool = Field(default=False, alias="DEBUG")
    secret_key: str = Field(..., alias="SECRET_KEY")  # 必填
    api_v1_prefix: str = "/api/v1"

    # ========== 数据库 ==========
    database_url: PostgresDsn = Field(..., alias="DATABASE_URL")
    database_pool_size: int = Field(default=20, alias="DB_POOL_SIZE")
    database_max_overflow: int = Field(default=10, alias="DB_MAX_OVERFLOW")

    # ========== Redis ==========
    redis_url: RedisDsn = Field(default="redis://redis:6379/0", alias="REDIS_URL")

    # ========== JWT ==========
    jwt_algorithm: str = Field(default="HS256", alias="JWT_ALGORITHM")
    jwt_access_token_expire_minutes: int = Field(default=30, alias="JWT_ACCESS_EXPIRE")
    jwt_refresh_token_expire_days: int = Field(default=7, alias="JWT_REFRESH_EXPIRE")

    # ========== Celery ==========
    celery_broker_url: str = Field(default="redis://redis:6379/1", alias="CELERY_BROKER_URL")
    celery_result_backend: str = Field(default="redis://redis:6379/2", alias="CELERY_RESULT_BACKEND")

    # ========== RabbitMQ（生产环境推荐）==========
    rabbitmq_url: str = Field(default="", alias="RABBITMQ_URL")

    # ========== 文件上传 ==========
    upload_dir: str = Field(default="/app/uploads", alias="UPLOAD_DIR")
    max_upload_size: int = Field(default=10 * 1024 * 1024, alias="MAX_UPLOAD_SIZE")  # 10MB

    # ========== CORS ==========
    cors_origins: list[str] = Field(default=["https://myblog.com"], alias="CORS_ORIGINS")

    # ========== 日志 ==========
    log_level: str = Field(default="INFO", alias="LOG_LEVEL")

    @field_validator("cors_origins", mode="before")
    @classmethod
    def parse_cors_origins(cls, v):
        """解析 CORS 来源（支持逗号分隔的字符串）"""
        if isinstance(v, str):
            return [origin.strip() for origin in v.split(",")]
        return v

    @property
    def is_production(self) -> bool:
        return self.app_env == "production"

    @property
    def is_development(self) -> bool:
        return self.app_env == "development"


@lru_cache
def get_settings() -> Settings:
    """获取配置单例（lru_cache 确保只读取一次）"""
    return Settings()
```

### 示例 5：Docker Compose 完整编排

```yaml
# docker-compose.yml
# 博客系统 - 完整服务编排
#
# 启动：docker compose up -d
# 查看日志：docker compose logs -f
# 停止：docker compose down
# 重建：docker compose up -d --build

version: "3.9"

services:
  # ========== Web 应用 ==========
  web:
    build:
      context: .
      dockerfile: Dockerfile
      target: runtime
    container_name: blog_api_web
    restart: unless-stopped
    env_file:
      - .env.production
    environment:
      - APP_ENV=production
    ports:
      - "8000:8000" # 仅用于调试，生产环境通过 Nginx 代理
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - uploads:/app/uploads # 持久化上传文件
      - logs:/app/logs # 持久化日志
    networks:
      - blog_network
    deploy:
      resources:
        limits:
          cpus: "2.0"
          memory: 1G
        reservations:
          cpus: "0.5"
          memory: 256M
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/health"]
      interval: 30s
      timeout: 5s
      start_period: 40s
      retries: 3

  # ========== Celery Worker ==========
  worker:
    build:
      context: .
      dockerfile: Dockerfile
      target: runtime
    container_name: blog_api_worker
    restart: unless-stopped
    command: >
      celery -A app.worker.celery_app worker
      --loglevel=info
      --concurrency=4
      --max-tasks-per-child=1000
      --without-heartbeat
    env_file:
      - .env.production
    environment:
      - APP_ENV=production
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - uploads:/app/uploads
    networks:
      - blog_network
    deploy:
      resources:
        limits:
          cpus: "2.0"
          memory: 1G

  # ========== Celery Beat（定时任务调度器）==========
  beat:
    build:
      context: .
      dockerfile: Dockerfile
      target: runtime
    container_name: blog_api_beat
    restart: unless-stopped
    command: >
      celery -A app.worker.celery_app beat
      --loglevel=info
      --schedule=/app/celerybeat-schedule
    env_file:
      - .env.production
    environment:
      - APP_ENV=production
    depends_on:
      redis:
        condition: service_healthy
    networks:
      - blog_network
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 256M

  # ========== PostgreSQL 数据库 ==========
  postgres:
    image: postgres:16-alpine
    container_name: blog_postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: blog_user
      POSTGRES_PASSWORD: blog_pass
      POSTGRES_DB: blog_db
      # 优化配置
      POSTGRES_INITDB_ARGS: "--encoding=UTF-8 --lc-collate=C --lc-ctype=C"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      # 初始化脚本（首次启动时执行）
      - ./docker/init-db.sql:/docker-entrypoint-initdb.d/init.sql:ro
```

### 示例 7：日志管理与结构化日志

```python
"""
app/core/logging_config.py - 生产环境日志配置
"""
import logging
import logging.config
import os
import sys
from datetime import datetime, timezone


# JSON 格式日志（便于 ELK/Loki 等日志系统收集）
class JSONFormatter(logging.Formatter):
    """结构化 JSON 日志格式化器"""

    def format(self, record: logging.LogRecord) -> str:
        import json

        log_entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
            "module": record.module,
            "function": record.funcName,
            "line": record.lineno,
        }

        # 附加异常信息
        if record.exc_info:
            log_entry["exception"] = self.formatException(record.exc_info)

        # 附加额外字段
        if hasattr(record, "request_id"):
            log_entry["request_id"] = record.request_id
        if hasattr(record, "user_id"):
            log_entry["user_id"] = record.user_id
        if hasattr(record, "duration_ms"):
            log_entry["duration_ms"] = record.duration_ms

        return json.dumps(log_entry, ensure_ascii=False)


def setup_logging(log_level: str = "INFO") -> None:
    """配置生产环境日志"""

    config = {
        "version": 1,
        "disable_existing_loggers": False,

        "formatters": {
            # 标准文本格式（开发环境）
            "standard": {
                "format": (
                    "%(asctime)s | %(levelname)-8s | %(name)s | "
                    "%(funcName)s:%(lineno)d | %(message)s"
                ),
                "datefmt": "%Y-%m-%d %H:%M:%S",
            },
            # JSON 格式（生产环境，方便日志系统收集）
            "json": {
                "()": JSONFormatter,
            },
        },

        "handlers": {
            # 控制台输出（Docker 收集 stdout）
            "console": {
                "class": "logging.StreamHandler",
                "stream": sys.stdout,
                "formatter": "json" if os.getenv("APP_ENV") == "production" else "standard",
            },
        },

        "loggers": {
            # 应用日志
            "app": {
                "level": log_level,
                "handlers": ["console"],
                "propagate": False,
            },
            # 第三方库日志（降低级别）
            "uvicorn": {"level": "WARNING"},
            "uvicorn.access": {"level": "WARNING"},
            "sqlalchemy": {"level": "WARNING"},
            "celery": {"level": "INFO"},
            "httpx": {"level": "WARNING"},
            # 根日志
            "": {
                "level": "WARNING",
                "handlers": ["console"],
            },
        },
    }

    logging.config.dictConfig(config)


# ========== 使用示例 ==========

logger = logging.getLogger("app.api.posts")

# 普通日志
logger.info("文章创建成功", extra={"user_id": 42, "post_id": 100})

# 带耗时的日志
logger.info(
    "查询文章列表",
    extra={"duration_ms": 23, "request_id": "req-abc123"},
)

# 错误日志（自动包含堆栈）
try:
    result = 1 / 0
except ZeroDivisionError:
    logger.exception("计算错误")
```

```dockerfile
# 镜像优化要点总结：

# 1. 使用 slim 镜像而非完整镜像
#    好: python:3.12-slim (约 150MB)
#    差: python:3.12 (约 1GB)

#    先 COPY 依赖文件，再 COPY 源代码
#    依赖不变时无需重新安装

#    好: RUN apt-get update && apt-get install -y pkg && rm -rf /var/lib/apt/lists/*
#    差: RUN apt-get update
#         RUN apt-get install -y pkg

# 5. 使用 .dockerignore（减小构建上下文）

# 6. 非特权用户运行
#    RUN groupadd -r appuser && useradd -r -g appuser appuser
#    USER appuser

# 7. 固定基础镜像版本（避免意外更新导致的问题）

#    差: python:3.12-slim
```

## 常见问题

**Q: Docker Compose 适合生产环境吗？**

A: Docker Compose 适合中小型项目的单机部署。如果需要多机部署、自动扩缩容、滚动更新等高级功能，应该使用 Kubernetes。但即使是 Kubernetes，Docker Compose 的配置思路（服务定义、网络、卷）也是相通的。对于大多数博客系统来说，Docker Compose 完全够用。

**Q: Gunicorn 的 Worker 数量怎么确定？**

A: 官方推荐公式是 `(2 * CPU核心数) + 1`。但 FastAPI 是异步框架，I/O 密集型场景下每个 Worker 可以处理大量并发连接，所以实际可以根据负载测试调整。建议设置为环境变量 `WEB_CONCURRENCY`，方便在 K8s 中根据 Pod 资源动态调整。

**Q: Uvicorn Worker 和 Uvicorn 直接启动有什么区别？**

A: `uvicorn app.main:app` 启动单个进程，适合开发环境。Gunicorn + Uvicorn Worker 启动多个进程，Gunicorn 负责进程管理（启动、重启、负载均衡），Uvicorn Worker 负责 ASGI 请求处理。类比 Java：Uvicorn 相当于内嵌 Tomcat，Gunicorn 相当于 Apache + Tomcat 的组合。

**Q: 如何实现零停机部署？**

A: Docker Compose 的 `docker compose up -d --build` 会自动滚动更新（先启动新容器，健康检查通过后再停止旧容器）。更完善的方案是蓝绿部署或金丝雀发布，需要配合 Nginx upstream 的权重切换或 Kubernetes 的 Rolling Update。

**Q: 容器内如何收集日志？**

A: 生产环境推荐 JSON 格式日志输出到 stdout/stderr，由 Docker 的日志驱动收集。可以搭配 ELK（Elasticsearch + Logstash + Kibana）或 Grafana Loki 进行日志聚合和查询。Docker 自带 `json-file` 日志驱动，配合 `max-size` 和 `max-file` 选项防止磁盘写满。

**Q: 如何管理 SSL 证书？**

A: 推荐使用 Let's Encrypt + Certbot 自动申请和续期。Docker Compose 中可以添加一个 `certbot` 服务，通过定时任务（cron）自动续期。也可以使用 Nginx Proxy Manager 等工具简化 SSL 管理。
