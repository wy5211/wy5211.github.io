# Docker 部署

22 篇文章的代码都跑在本地开发环境。要把 CMS 部署到生产，你需要：一个可以重复构建的运行环境、一个能管理多个服务依赖的编排方案、一套环境变量管理机制。

Docker 解决的是"在我机器上能跑"的问题。但 Python 的 Docker 化和 Java 有一些本质区别，值得单独拿出来讲。

## Dockerfile：多阶段构建

CMS 的 Dockerfile 需要做两件事：安装依赖和复制代码。Python 用多阶段构建把这两步分开：

```dockerfile
# ========== 阶段 1：安装依赖 ==========
FROM python:3.12-slim AS builder

ENV PYTHONDONTWRITEBYTECODE=1

RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential libpq-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /build

# 先复制依赖文件，利用 Docker 层缓存
COPY pyproject.toml ./
RUN pip install --no-cache-dir --prefix=/install .

# ========== 阶段 2：运行 ==========
FROM python:3.12-slim AS runtime

# 创建非 root 用户
RUN groupadd -r appuser && useradd -r -g appuser appuser

# 只安装运行时依赖（不需要 build-essential）
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 curl \
    && rm -rf /var/lib/apt/lists/*

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

# 从构建阶段复制已安装的依赖
COPY --from=builder /install /usr/local

WORKDIR /app
COPY --chown=appuser:appuser . .

USER appuser
EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=5s --start-period=30s --retries=3 \
    CMD curl -f http://localhost:8000/api/health || exit 1

# Gunicorn + Uvicorn Worker
CMD ["gunicorn", "app.main:app", \
     "--bind", "0.0.0.0:8000", \
     "--workers", "4", \
     "--worker-class", "uvicorn.workers.UvicornWorker", \
     "--timeout", "120"]
```

同样的 CMS 项目用 Spring Boot 写 Dockerfile：

```dockerfile
# Spring Boot 多阶段构建
# ========== 阶段 1：编译 ==========
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline  # 先下载依赖，利用缓存
COPY src ./src
RUN mvn package -DskipTests

# ========== 阶段 2：运行 ==========
FROM eclipse-temurin:21-jre-alpine
RUN addgroup -S appuser && adduser -S appuser -G appuser
WORKDIR /app
COPY --from=builder /build/target/cms-0.0.1-SNAPSHOT.jar app.jar
USER appuser
EXPOSE 8080
HEALTHCHECK CMD wget -qO- http://localhost:8080/actuator/health || exit 1
ENTRYPOINT ["java", "-jar", "app.jar"]
```

关键差异：

| 维度         | Python Dockerfile          | Java/Spring Boot Dockerfile            |
| ------------ | -------------------------- | -------------------------------------- |
| 构建阶段     | pip install（安装依赖）    | mvn package（编译 + 打包）             |
| 构建时间     | 10-30 秒                   | 1-5 分钟（下载依赖 + 编译）            |
| 构建阶段镜像 | python:3.12-slim（~150MB） | maven:3.9（~500MB）                    |
| 运行时镜像   | python:3.12-slim（~150MB） | eclipse-temurin:21-jre-alpine（~80MB） |
| 最终镜像大小 | ~200-350MB                 | ~180-250MB                             |

Python 的构建阶段更快，但运行时镜像更大（需要完整的 Python 解释器和标准库）。Java 的构建阶段很慢（Maven/Gradle 下载依赖 + 编译），但运行时只需要 JRE，镜像反而更小。这是编译型语言和解释型语言在 Docker 化上的典型差异。

## 镜像大小对比：三种技术栈

用同样复杂的 CMS 项目分别 Docker 化，最终镜像大小：

| 技术栈             | 基础镜像                      | 最终镜像大小 | 说明                          |
| ------------------ | ----------------------------- | ------------ | ----------------------------- |
| Python (FastAPI)   | python:3.12-slim              | 200-350MB    | Python 解释器 + pip 依赖      |
| Java (Spring Boot) | eclipse-temurin:21-jre-alpine | 180-250MB    | JRE + fat JAR                 |
| Node.js (Express)  | node:20-alpine                | 150-250MB    | Node.js 运行时 + node_modules |

Python 镜像通常最大，因为 Python 解释器和标准库占了不少空间。但 350MB 在今天不算大——一个 Java Spring Boot 项目加上各种依赖的 fat JAR 轻轻松松 100MB+，加上 JRE 也不比 Python 小多少。Node.js 的 `node_modules` 同样可以膨胀到几百 MB。

真正的镜像优化手段对所有语言都适用：使用 `alpine` 变体、多阶段构建、`.dockerignore` 排除无关文件、清理包管理器缓存。

## Docker Compose：编排 CMS 全套服务

CMS 不只是 FastAPI 应用本身，还有 PostgreSQL、Redis、Nginx。用 Docker Compose 编排：

```yaml
# docker-compose.yml
services:
  # ========== Web 应用 ==========
  web:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: cms_web
    restart: unless-stopped
    env_file: .env.production
    ports:
      - "8000:8000"
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    volumes:
      - uploads:/app/uploads
    networks:
      - cms_network

  # ========== Celery Worker ==========
  worker:
    build: .
    container_name: cms_worker
    restart: unless-stopped
    command: celery -A app.worker.celery_app worker --loglevel=info --concurrency=4
    env_file: .env.production
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - cms_network

  # ========== Celery Beat（定时任务）==========
  beat:
    build: .
    container_name: cms_beat
    restart: unless-stopped
    command: celery -A app.worker.celery_app beat --loglevel=info
    env_file: .env.production
    depends_on:
      redis:
        condition: service_healthy
    networks:
      - cms_network

  # ========== PostgreSQL ==========
  postgres:
    image: postgres:16-alpine
    container_name: cms_postgres
    restart: unless-stopped
    environment:
      POSTGRES_USER: cms_user
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: cms_db
    volumes:
      - postgres_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    networks:
      - cms_network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U cms_user -d cms_db"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ========== Redis ==========
  redis:
    image: redis:7-alpine
    container_name: cms_redis
    restart: unless-stopped
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redis_data:/data
    ports:
      - "6379:6379"
    networks:
      - cms_network
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ========== Nginx ==========
  nginx:
    image: nginx:1.25-alpine
    container_name: cms_nginx
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./docker/nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - uploads:/usr/share/nginx/html/uploads:ro
    depends_on:
      web:
        condition: service_healthy
    networks:
      - cms_network

volumes:
  postgres_data:
  redis_data:
  uploads:

networks:
  cms_network:
    driver: bridge
```

Spring Boot 版本的 Docker Compose 结构完全相同，只是 `web` 和 `worker` 服务的构建方式不同。Java 项目不需要 `worker` 和 `beat`（后台任务用 `@Async` 或消息队列），但结构大同小异。

## 环境变量管理

FastAPI 用 Pydantic Settings 管理环境变量，和 Spring Boot 的 `application.yml` + `@Value` 思路类似，但配置源不同：

```python
# app/core/config.py
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    database_url: str
    redis_url: str = "redis://redis:6379/0"
    secret_key: str
    debug: bool = False

    class Config:
        env_file = ".env"

settings = Settings()
```

```yaml
# Spring Boot application.yml
spring:
  datasource:
    url: ${DATABASE_URL}
  data:
    redis:
      url: ${REDIS_URL:redis://redis:6379/0}
app:
  security:
    jwt:
      secret: ${SECRET_KEY}
```

Spring Boot 的 `application.yml` 可以定义默认值和类型，通过 `${}` 注入环境变量覆盖。Python 的 Pydantic Settings 从 `.env` 文件和环境变量读取，字段定义本身就是类型和默认值。两者都能做到"开发环境用默认值，生产环境用环境变量覆盖"。

## 三种技术栈对比总结

学了 Python/FastAPI，和 Java/Spring Boot、Node.js/Express 相比，各自的定位很清晰：

| 维度               | Java / Spring Boot           | Python / FastAPI           | Node.js / Express         |
| ------------------ | ---------------------------- | -------------------------- | ------------------------- |
| **开发效率**       | 中（样板代码多，IDE 辅助强） | 高（语法简洁，开发速度快） | 高（JS 全栈，生态丰富）   |
| **运行性能**       | 高（JIT 编译，多线程）       | 中（GIL 限制，异步弥补）   | 中（单线程事件循环）      |
| **类型安全**       | 强（编译期检查）             | 中（类型注解 + mypy 可选） | 中（TypeScript 可选）     |
| **并发模型**       | 线程池（同步为主）           | asyncio（异步为主）        | 事件循环（异步为主）      |
| **Web 框架成熟度** | 极高（Spring 全家桶）        | 高（FastAPI 生态快速成长） | 高（Koa/Nest 等多种选择） |
| **ORM**            | Hibernate / JPA              | SQLAlchemy                 | Prisma / TypeORM          |
| **后台任务**       | @Async（轻量）               | BackgroundTasks + Celery   | 无标准方案（Bull/BullMQ） |
| **缓存**           | @Cacheable（注解式）         | redis-py（手动式）         | ioredis（手动式）         |
| **WebSocket**      | STOMP（消息代理）            | 原生 WebSocket（手动管理） | ws / socket.io            |
| **部署镜像**       | 180-250MB                    | 200-350MB                  | 150-250MB                 |
| **招聘市场**       | 企业后端主流                 | AI/数据/Web 兼顾           | 全栈 + 前端               |

**什么时候选 Java/Spring Boot**：大型企业系统、需要强类型安全和编译期保障、团队有 Java 背景、对运行性能有极致要求。Spring Boot 的注解体系在标准场景下效率极高，但学习曲线陡峭，"Spring 生态"本身就是一个需要投入时间学习的体系。

**什么时候选 Python/FastAPI**：快速原型开发、AI/ML 集成（Python 在数据科学领域无可替代）、中小型 API 服务、团队追求开发效率。FastAPI 的异步性能不弱，redis-py 的手动缓存和 Celery 的任务队列给了你完全的掌控力，代价是需要写更多代码。

**什么时候选 Node.js**：全栈开发（前后端同一语言）、实时应用（聊天、协作编辑）、I/O 密集型服务。JavaScript 的异步模型和 Python 的 asyncio 类似，但 Node.js 在前端集成方面有天然优势。

没有最好的技术栈，只有最适合当前场景的选择。理解了这三种技术栈各自的设计哲学和 trade-off，你就能在面对不同项目需求时做出合理的判断。

## 对比总结

| 维度            | Spring Boot              | FastAPI                          |
| --------------- | ------------------------ | -------------------------------- |
| Dockerfile 构建 | Maven/Gradle 编译 + 打包 | pip install 依赖                 |
| 构建时间        | 较长（1-5 分钟）         | 较短（10-30 秒）                 |
| 运行时          | JRE（无需编译器）        | Python 解释器                    |
| 进程管理        | 内嵌 Tomcat，单进程即可  | Gunicorn + Uvicorn Worker 多进程 |
| 后台任务        | @Async + 线程池          | BackgroundTasks + Celery         |
| 定时任务        | @Scheduled               | Celery Beat                      |
| 环境变量        | application.yml + ${}    | Pydantic Settings + .env         |

本系列到此完结。22 篇文章覆盖了从语言基础到生产部署的完整路径。无论你最终选择哪种技术栈，理解了这些概念和 trade-off，都能让你在不同的语言和框架之间自由切换。
