## 一个真实的问题

CMS 的文章创建接口有一个棘手的验证需求：标题不能包含敏感词，内容长度要根据文章类型动态调整，标签数量不能超过 5 个，发布时间如果指定了就不能是过去的时间。这些规则不是简单的"非空"、"最大长度"能覆盖的。

在 Java 中，你会在 DTO 字段上堆注解：`@NotNull`、`@Size(max=200)`、`@Pattern(regexp="...")`，然后写一个 `Validator` 类处理跨字段逻辑。Pydantic v2 的做法不同 -- 它把验证逻辑写成装饰器方法，嵌在模型定义内部。

本文从 CMS 文章创建的具体场景出发，对比 Pydantic v2 和 Bean Validation 在复杂验证上的差异。

---

## Field 验证器：内置约束

Pydantic 的 `Field` 和 Java 的 Bean Validation 注解是一一对应的：

```python
class PostCreate(BaseModel):
    title: str = Field(..., min_length=1, max_length=200)
    content: str = Field(..., min_length=1)
    is_published: bool = Field(default=False)
    tag_ids: list[int] = Field(default_factory=list)
```

对应 Java Bean Validation：

```java
public class PostCreateDTO {
    @NotBlank
    @Size(max = 200)
    private String title;

    @NotBlank
    private String content;

    private boolean isPublished = false;

    @NotEmpty
    private List<Long> tagIds = new ArrayList<>();
}
```

映射关系如下：

| Pydantic Field     | Bean Validation             | 作用     |
| ------------------ | --------------------------- | -------- |
| `...` (Ellipsis)   | `@NotNull` / `@NotBlank`    | 必填     |
| `min_length=1`     | `@Size(min=1)`              | 最小长度 |
| `max_length=200`   | `@Size(max=200)`            | 最大长度 |
| `ge=0, le=100`     | `@Min(0) @Max(100)`         | 数值范围 |
| `pattern=r"^\d+$"` | `@Pattern(regexp="^\\d+$")` | 正则匹配 |
| `EmailStr`         | `@Email`                    | 邮箱格式 |
| `HttpUrl`          | `@URL`                      | URL 格式 |

语法不同，能力等价。但 Pydantic v2 有一个显著优势：它的验证引擎用 Rust 实现，比 Bean Validation 快 5-50 倍。在大量数据验证场景下（批量导入、API 网关），这个差异是可感知的。

## 自定义验证器：@field_validator vs 自定义注解

当内置约束不够用时，需要自定义验证逻辑。这是两种方案差异最大的地方。

### 单字段验证

CMS 要求标题不能包含敏感词，密码必须满足复杂度要求：

```python
from pydantic import BaseModel, Field, field_validator
import re

class PostCreate(BaseModel):
    title: str = Field(..., min_length=1, max_length=200)
    content: str = Field(..., min_length=1)
    post_type: str = Field(default="article", pattern=r"^(article|page|draft)$")
    tag_ids: list[int] = Field(default_factory=list)

    @field_validator("title")
    @classmethod
    def validate_title(cls, v: str) -> str:
        """标题不能包含敏感词"""
        forbidden = ["广告", "代写", "刷单"]
        for word in forbidden:
            if word in v:
                raise ValueError(f"标题包含禁止词: {word}")
        return v.strip()

    @field_validator("content")
    @classmethod
    def validate_content_length(cls, v: str, info) -> str:
        """根据 post_type 动态调整内容长度要求"""
        post_type = info.data.get("post_type", "article")
        limits = {"article": 100, "page": 50, "draft": 10}
        min_len = limits.get(post_type, 10)
        if len(v) < min_len:
            raise ValueError(f"{post_type} 类型的内容至少 {min_len} 字")
        return v
```

在 Java 中，同样的逻辑需要写自定义注解 + `ConstraintValidator`：

```java
// 1. 定义注解
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = NoForbiddenWordsValidator.class)
public @interface NoForbiddenWords {
    String message() default "包含禁止词";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// 2. 实现 Validator
public class NoForbiddenWordsValidator implements ConstraintValidator<NoForbiddenWords, String> {
    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        List<String> forbidden = List.of("广告", "代写", "刷单");
        return forbidden.stream().noneMatch(value::contains);
    }
}

// 3. 使用
@NoForbiddenWords
private String title;
```

三步走：定义注解、实现验证器、标注字段。Pydantic 的 `@field_validator` 只需要一步 -- 直接在模型内部写方法。对于 CMS 这种验证规则经常变化的场景，Pydantic 的维护成本明显更低。

`info.data` 参数是一个有用的细节：它允许访问同一模型中其他字段的已验证值。上面的例子中，`content` 的验证依赖 `post_type` 的值，通过 `info.data.get("post_type")` 拿到。Bean Validation 要实现类似的跨字段依赖，需要 `@ClassValidator` 或手写 Service 层校验。

## 跨字段验证：@model_validator vs @Valid + Service

CMS 的注册表单需要验证两次密码一致，文章创建需要验证标签数量不超过 5 个。这类跨字段逻辑，Pydantic 用 `@model_validator` 处理：

```python
from pydantic import BaseModel, field_validator, model_validator

class ArticleCreate(BaseModel):
    title: str
    content: str
    tag_ids: list[int] = Field(default_factory=list)
    scheduled_at: datetime | None = None
    is_published: bool = False

    @model_validator(mode="after")
    def validate_article_rules(self) -> "ArticleCreate":
        # 规则 1：未发布文章不能指定发布时间
        if self.scheduled_at and not self.is_published:
            raise ValueError("未发布文章不能指定发布时间")

        # 规则 2：标签数量上限
        if len(self.tag_ids) > 5:
            raise ValueError("一篇文章最多关联 5 个标签")

        # 规则 3：指定发布时间不能是过去
        if self.scheduled_at and self.scheduled_at < datetime.now(timezone.utc):
            raise ValueError("发布时间不能是过去")

        return self
```

`mode="after"` 表示在所有字段单独验证通过后，再执行模型级别的验证。如果改成 `mode="before"`，则会在字段验证之前执行，适合做数据预处理。

Java 中的等价实现通常放在 Service 层，而不是 DTO 的注解里。Bean Validation 的 `@ClassValidator`（Hibernate Validator 6+）功能有限，复杂的跨字段逻辑一般还是手写：

```java
@Service
public class ArticleService {
    public void create(ArticleCreateDTO dto) {
        if (dto.getScheduledAt() != null && !dto.isPublished()) {
            throw new ValidationException("未发布文章不能指定发布时间");
        }
        if (dto.getTagIds().size() > 5) {
            throw new ValidationException("最多关联 5 个标签");
        }
        // ... 业务逻辑
    }
}
```

两种方案各有取舍：Pydantic 把验证逻辑集中在模型定义中，数据约束一目了然；Java 把验证分散在 DTO 注解和 Service 层，更灵活但代码更分散。

## 嵌套模型验证

CMS 的文章详情接口需要返回嵌套结构：文章包含作者信息、标签列表、评论列表。Pydantic 的嵌套验证是自动递归的：

```python
from pydantic import BaseModel, ConfigDict
from datetime import datetime

class AuthorSchema(BaseModel):
    id: int
    username: str
    avatar_url: str | None = None
    model_config = ConfigDict(from_attributes=True)

class TagSchema(BaseModel):
    id: int
    name: str
    model_config = ConfigDict(from_attributes=True)

class CommentSchema(BaseModel):
    id: int
    content: str
    created_at: datetime
    user: AuthorSchema | None = None
    model_config = ConfigDict(from_attributes=True)

class PostDetailSchema(BaseModel):
    id: int
    title: str
    content: str
    author: AuthorSchema
    tags: list[TagSchema] = []
    comments: list[CommentSchema] = []
    created_at: datetime
    model_config = ConfigDict(from_attributes=True)
```

`from_attributes=True` 是 Pydantic v2 的关键配置（v1 叫 `orm_mode`），允许直接从 SQLAlchemy ORM 对象创建 Pydantic 模型，不需要手动转换字段。在 Java 中，这对应 `@Valid` 注解的级联验证：

```java
public class PostDetailVO {
    @NotNull
    private AuthorVO author;

    @NotEmpty
    @Valid  // 级联验证：验证每个 TagVO
    private List<TagVO> tags;

    @Valid
    private List<CommentVO> comments;
}
```

两者的区别在于：Pydantic 的嵌套验证是默认行为，不需要显式声明；Java 需要 `@Valid` 注解触发级联验证。反过来，Java 的 `@Valid` 可以在任意层级使用，而 Pydantic 只需要模型定义正确，验证就是自动的。

## 条件验证

CMS 有一个需求：文章类型为"置顶"时，必须填写推荐理由。Pydantic v2 用 `model_validator` 的 `mode="before"` 实现条件验证：

```python
class PostUpdate(BaseModel):
    title: str | None = None
    content: str | None = None
    is_pinned: bool = False
    pin_reason: str | None = None

    @model_validator(mode="after")
    def validate_pin_reason(self) -> "PostUpdate":
        if self.is_pinned and not self.pin_reason:
            raise ValueError("置顶文章必须填写推荐理由")
        if not self.is_pinned and self.pin_reason:
            self.pin_reason = None  # 非置顶时自动清除
        return self
```

`exclude_unset=True` 在部分更新场景中很重要。`PostUpdate` 的所有字段都是可选的（用于 PATCH 接口），但用户只传了哪些字段、哪些字段保持不变，需要区分清楚：

```python
# 只更新用户实际传入的字段
update_data = post_data.model_dump(exclude_unset=True)
for field, value in update_data.items():
    setattr(post, field, value)
```

Java 中的等价做法是 `@Patch` + 手动判空：

```java
public void update(Long id, PostUpdateDTO dto) {
    Post post = repository.findById(id).orElseThrow();
    if (dto.getTitle() != null) post.setTitle(dto.getTitle());
    if (dto.getContent() != null) post.setContent(dto.getContent());
    // ... 逐字段判空
}
```

Pydantic 的 `model_dump(exclude_unset=True)` 自动过滤未设置的字段，比 Java 的逐字段判空更干净。

## 统一错误格式

Bean Validation 的默认错误格式是 `{"field": "message"}`，Pydantic v2 的默认格式是 `{"loc": ["field"], "msg": "message", "type": "error_type"}`。在 CMS 中，我们需要统一错误格式：

```python
from fastapi import Request
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError

async def validation_handler(request: Request, exc: RequestValidationError):
    errors = [
        {
            "field": ".".join(str(loc) for loc in err["loc"][1:]),  # 跳过 "body"
            "message": err["msg"],
        }
        for err in exc.errors()
    ]
    return JSONResponse(status_code=422, content={"code": 422, "details": errors})

# app/main.py
app.add_exception_handler(RequestValidationError, validation_handler)
```

这样前端收到的错误格式就是 `{"code": 422, "details": [{"field": "title", "message": "..."}]}`，和 Bean Validation 的 `MethodArgumentNotValidException` 处理后的格式对齐。

## 对比总结

| 维度       | Bean Validation (Java)           | Pydantic v2 (Python)                 |
| ---------- | -------------------------------- | ------------------------------------ |
| 基础验证   | 字段注解 `@NotNull` `@Size`      | Field 参数 `min_length` `max_length` |
| 自定义验证 | 自定义注解 + ConstraintValidator | `@field_validator` 装饰器            |
| 跨字段验证 | Service 层手写或 ClassValidator  | `@model_validator` 装饰器            |
| 嵌套验证   | `@Valid` 级联触发                | 自动递归，无需声明                   |
| 部分更新   | 逐字段判 null                    | `model_dump(exclude_unset=True)`     |
| 验证位置   | DTO 注解 + Service 层混合        | 全部集中在模型定义中                 |
| 性能       | 纯 Java 反射                     | Rust 内核（快 5-50 倍）              |

---

## Trade-off

Pydantic v2 的优势在于**验证逻辑的集中度**：所有数据约束都在模型定义中，不用在 DTO 注解和 Service 层之间跳来跳去。`@field_validator` 和 `@model_validator` 让自定义验证的编写成本远低于 Java 的 `ConstraintValidator`。

代价是 Pydantic 不具备 Bean Validation 的集成深度。Bean Validation 与 Spring MVC、JPA、Swagger/OpenAPI 天然集成，一个 `@Valid` 注解就能触发整个验证链。Pydantic 的集成范围主要在 FastAPI 生态内，离开 FastAPI 需要手动调用 `model_validate()`。

对于 CMS 这种 FastAPI 全栈项目，Pydantic 是最自然的选择。
