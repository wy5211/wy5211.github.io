在 Spring Boot 系列第 5 篇中，我们用 Spring Data JPA 给 CMS 加上了 MySQL 持久化。JPA 的使用体验很"Java"：写一个 Entity 类加 `@Entity` 注解，写一个接口继承 `JpaRepository`，框架帮你生成 SQL、管理连接池、处理事务。你几乎不需要碰 SQL 本身。

SQLAlchemy 是 Python 生态里做同样事情的库，但设计哲学和 JPA 有本质区别。JPA 是一套规范，Spring Data JPA 在规范之上提供了一层极简的抽象——`save()`、`findById()`、`findAll()` 就够了，SQL 完全藏起来。SQLAlchemy 更像是"ORM + SQL Builder"的组合体——它给你 ORM 的便利，同时保留对 SQL 的完全控制。你可以在 ORM 查询和原生 SQL 之间自由切换，不需要绕过框架。

这篇用 SQLAlchemy 2.0 定义 CMS 的四个核心模型：User、Post、Tag、Comment。

## 安装

```bash
uv add sqlalchemy asyncpg aiosqlite alembic
```

`asyncpg` 是 PostgreSQL 的异步驱动，`aiosqlite` 是 SQLite 的异步驱动（开发环境用），`alembic` 是数据库迁移工具。FastAPI 是异步框架，搭配异步驱动能最大化性能。

## 声明式映射：Mapped + mapped_column

SQLAlchemy 2.0 引入了全新的类型注解语法，替代了旧版的 `Column()` 写法。对比一下：

```python
# 1.x 旧写法（不推荐）
class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    username = Column(String(50), unique=True)

# 2.0 新写法
class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)
    username: Mapped[str] = mapped_column(String(50), unique=True)
```

`Mapped[int]` 告诉类型检查器"这个属性是 int 类型"，`mapped_column()` 告诉 SQLAlchemy"这是一个数据库列"。两者配合，既满足了 IDE 的类型推断，又满足了 ORM 的列定义。这比 JPA 的 `private Long id;` + `@Id @GeneratedValue` 写法更紧凑，信息密度更高。

来看 CMS 的完整模型定义。

## CMS 数据模型

### 基类与混入

```python
# app/models/base.py
from datetime import datetime
from sqlalchemy import DateTime, func
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column


class Base(DeclarativeBase):
    """所有模型的基类"""
    pass


class TimestampMixin:
    """时间戳混入：自动管理 created_at"""
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True),
        server_default=func.now(),
    )
```

`TimestampMixin` 用 Python 的混入（Mixin）模式实现时间戳复用。JPA 没有直接的等价物——Java 不支持多继承，你通常用 `@MappedSuperclass` 或 AOP 来实现类似效果。Python 的混入机制更轻量，不需要额外的注解或配置。

### User 模型

```python
# app/models/user.py
from sqlalchemy import String, Integer
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.models.base import Base, TimestampMixin


class User(Base, TimestampMixin):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    username: Mapped[str] = mapped_column(String(50), unique=True, index=True)
    email: Mapped[str] = mapped_column(String(200), unique=True, index=True)
    password_hash: Mapped[str] = mapped_column(String(200))
    role: Mapped[str] = mapped_column(String(20), default="author")
    avatar: Mapped[str | None] = mapped_column(String(500), default=None)

    # 关系
    posts: Mapped[list["Post"]] = relationship(back_populates="author", lazy="selectin")
    comments: Mapped[list["Comment"]] = relationship(back_populates="author", lazy="selectin")

    def __repr__(self) -> str:
        return f"<User(id={self.id}, username='{self.username}')>"
```

对比 Spring Boot 系列里的 JPA Entity：

```java
// JPA
@Entity
@Table(name = "users")
public class User {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true, nullable = false)
    private String username;

    @OneToMany(mappedBy = "author", fetch = FetchType.EAGER)
    private List<Post> posts = new ArrayList<>();
}
```

两者的映射关系几乎是一一对应的。几个差异点：

**`Mapped[str | None]`** 表示可空字段，等价于 JPA 的 `@Column(nullable = true)`。SQLAlchemy 2.0 用 Python 的 `| None`（Union 类型）来表达可空，比 JPA 的 `nullable = false/true` 更符合 Python 的类型系统。

**`lazy="selectin"`** 是 SQLAlchemy 的加载策略。JPA 用 `FetchType.EAGER` / `FetchType.LAZY` 做同样的事，但 JPA 的默认是 LAZY，SQLAlchemy 的默认是 `select`（类似 LAZY 的行为）。`selectin` 会用一条 `WHERE id IN (...)` 查询预加载关联数据，避免 N+1 问题。在 JPA 里你需要手动配置 `@EntityGraph` 或 `JOIN FETCH` 来达到同样效果。

**`relationship(back_populates="author")`** 声明双向关系。JPA 的 `@OneToMany(mappedBy = "author")` 做同样的事，`mappedBy` 对应 `back_populates`。

### Post 模型

```python
# app/models/post.py
from sqlalchemy import String, Text, Integer, ForeignKey, DateTime, Enum
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.models.base import Base, TimestampMixin


class Post(Base, TimestampMixin):
    __tablename__ = "posts"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    title: Mapped[str] = mapped_column(String(200))
    slug: Mapped[str] = mapped_column(String(200), unique=True, index=True)
    content: Mapped[str] = mapped_column(Text)
    status: Mapped[str] = mapped_column(
        String(20), default="draft"
    )
    author_id: Mapped[int] = mapped_column(ForeignKey("users.id"), index=True)
    view_count: Mapped[int] = mapped_column(Integer, default=0)
    published_at: Mapped[datetime | None] = mapped_column(
        DateTime(timezone=True), default=None
    )

    # 关系
    author: Mapped["User"] = relationship(back_populates="posts")
    comments: Mapped[list["Comment"]] = relationship(
        back_populates="post", lazy="selectin"
    )
    tags: Mapped[list["Tag"]] = relationship(
        secondary="post_tags",
        back_populates="posts",
        lazy="selectin",
    )

    def __repr__(self) -> str:
        return f"<Post(id={self.id}, title='{self.title}')>"
```

`ForeignKey("users.id")` 和 JPA 的 `@JoinColumn(name = "author_id")` 等价，都声明了外键约束。`secondary="post_tags"` 指向多对多的关联表，JPA 里用 `@JoinTable(name = "post_tags")` 做同样的事。

### Tag 模型与多对多关联

```python
# app/models/tag.py
from sqlalchemy import String, Table, Column, ForeignKey, Integer
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.models.base import Base


# 多对多关联表
post_tags = Table(
    "post_tags",
    Base.metadata,
    Column("post_id", ForeignKey("posts.id"), primary_key=True),
    Column("tag_id", ForeignKey("tags.id"), primary_key=True),
)


class Tag(Base):
    __tablename__ = "tags"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    name: Mapped[str] = mapped_column(String(50), unique=True)
    slug: Mapped[str] = mapped_column(String(50), unique=True)
    color: Mapped[str] = mapped_column(String(7), default="#6366f1")

    # 关系
    posts: Mapped[list["Post"]] = relationship(
        secondary="post_tags",
        back_populates="tags",
        lazy="selectin",
    )

    def __repr__(self) -> str:
        return f"<Tag(id={self.id}, name='{self.name}')>"
```

JPA 里多对多也用中间表，但 JPA 可以通过 `@ManyToMany` + `@JoinTable` 自动管理中间表的创建。SQLAlchemy 需要你显式定义 `Table` 对象——这多写了几行代码，但也意味着你对中间表有完全控制权（比如加额外字段）。

### Comment 模型

```python
# app/models/comment.py
from sqlalchemy import String, Text, Integer, ForeignKey
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.models.base import Base, TimestampMixin


class Comment(Base, TimestampMixin):
    __tablename__ = "comments"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    content: Mapped[str] = mapped_column(Text)
    post_id: Mapped[int] = mapped_column(ForeignKey("posts.id"), index=True)
    author_id: Mapped[int] = mapped_column(ForeignKey("users.id"), index=True)
    parent_id: Mapped[int | None] = mapped_column(
        ForeignKey("comments.id"), default=None
    )

    # 关系
    post: Mapped["Post"] = relationship(back_populates="comments")
    author: Mapped["User"] = relationship(back_populates="comments")

    def __repr__(self) -> str:
        return f"<Comment(id={self.id})>"
```

`parent_id` 的自引用外键实现了评论的嵌套回复。JPA 里用 `@ManyToOne` + `@JoinColumn(name = "parent_id")` + `@OneToMany(mappedBy = "parent")` 实现同样效果，代码量是 SQLAlchemy 的两三倍。

## 会话管理：Session vs EntityManager

JPA 的 `EntityManager` 和 SQLAlchemy 的 `Session` 做同样的事：管理数据库连接、追踪对象状态变化、提交事务。但管理方式完全不同。

```python
# app/database.py
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession
from app.config import settings

engine = create_async_engine(
    settings.database_url,
    echo=settings.debug,
    pool_size=10,
    max_overflow=20,
)

AsyncSessionLocal = async_sessionmaker(
    engine,
    class_=AsyncSession,
    expire_on_commit=False,
)


async def get_db():
    """获取数据库会话——供 FastAPI Depends 使用"""
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
```

JPA 的 `EntityManager` 通常由 Spring 容器管理，你通过 `@PersistenceContext` 注入，事务由 `@Transactional` 控制。SQLAlchemy 的 Session 需要你手动管理——`yield session` 把会话交给路由处理函数，处理完自动 `commit()` 或 `rollback()`。

两者的 trade-off：

| 方面         | JPA / Spring Boot                   | SQLAlchemy / FastAPI                  |
| ------------ | ----------------------------------- | ------------------------------------- |
| 会话生命周期 | 容器管理，`@Transactional` 自动提交 | 手动管理，`yield` + `commit/rollback` |
| 事务边界     | 注解驱动，AOP 拦截                  | 显式代码块，上下文管理器              |
| 线程安全     | 每个事务一个 EntityManager          | 每个请求一个 Session                  |
| 灵活性       | 受限于 JPA 规范                     | 可以直接执行原生 SQL                  |

JPA 的 `@Transactional` 写起来更简洁——一个注解就声明了事务边界。SQLAlchemy 的显式管理更透明——你能清楚地看到 `commit` 和 `rollback` 在哪里发生。对于需要精细控制事务的场景（比如一个请求里需要多次提交），SQLAlchemy 的方式更直接。

## Alembic 数据库迁移

```bash
# 初始化
alembic init alembic

# 生成迁移脚本（自动检测模型变更）
alembic revision --autogenerate -m "create cms tables"

# 执行迁移
alembic upgrade head

# 回滚
alembic downgrade -1
```

Alembic 的角色等同于 JPA 的 `hibernate.hbm2ddl.auto=update`，但 Alembic 生成的是版本化的 SQL 迁移脚本，你可以审查、修改、回滚。JPA 的自动更新在生产环境中不推荐使用，因为它无法处理复杂的列修改和索引变更。Spring Boot 项目通常搭配 Flyway 或 Liquibase 来做迁移管理。

在 `alembic/env.py` 中导入所有模型：

```python
# alembic/env.py（关键部分）
from app.models.base import Base
from app.models.user import User
from app.models.post import Post
from app.models.tag import Tag, post_tags
from app.models.comment import Comment

target_metadata = Base.metadata
```

## ORM 风格对比总结

| 方面     | Spring Data JPA                         | SQLAlchemy 2.0                 |
| -------- | --------------------------------------- | ------------------------------ |
| 模型定义 | `@Entity` + 字段注解                    | `class(Base)` + `Mapped[T]`    |
| 查询方式 | 方法名推导（`findByStatus`）或 `@Query` | `select()` 表达式语言          |
| SQL 控制 | 有限（需要 `nativeQuery`）              | 完全（ORM 和 Core 双模式）     |
| 关系加载 | `FetchType.EAGER/LAZY`                  | `lazy="selectin/joined/raise"` |
| 多对多   | `@JoinTable` 自动管理                   | 显式定义 `Table` 对象          |
| 自引用   | 多个 `@OneToMany/@ManyToOne`            | 单个 `ForeignKey("self.id")`   |
| 类型安全 | 编译时检查（Java 强类型）               | IDE 提示（Python 类型注解）    |

## 回顾

这篇用 SQLAlchemy 2.0 定义了 CMS 的完整数据模型：

- `Mapped[T]` + `mapped_column()` 是 2.0 推荐的类型注解风格，比旧版 `Column()` 更符合 Python 的类型系统
- 混入（Mixin）实现时间戳复用，比 JPA 的 `@MappedSuperclass` 更轻量
- `relationship()` + `back_populates` 声明双向关系，`lazy="selectin"` 避免 N+1 查询
- Session 的显式管理比 JPA 的 `@Transactional` 更透明，但代码量更多
- Alembic 提供版本化的数据库迁移，比 JPA 的自动更新更适合生产环境

模型定义好了，数据有了归宿。下一篇把 FastAPI 和 SQLAlchemy 串起来，用 Pydantic Schema 分离请求和响应，用 `Depends` 注入数据库会话，实现 CMS 文章的完整 CRUD API。
