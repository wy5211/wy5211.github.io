你在 Java 里写一个单元测试，先创建一个类继承 `Test`（JUnit 4）或者用 `@Test` 标记方法（JUnit 5），然后在 `@BeforeEach` 里初始化数据，最后用 `assertEquals`、`assertTrue` 做断言。哪怕只测一个简单的除法函数，你也得写一个完整的类结构。现在你打开一个 Python 项目的测试目录，发现测试文件里没有 class，没有 `setUp`，测试函数直接 `assert`——Python 的测试看起来像普通代码，不像测试代码。

这就是 pytest 和 JUnit 最直观的区别：JUnit 是面向测试框架的，你需要按框架的规则组织代码；pytest 是面向代码的，你按自然的思路写代码，pytest 来发现和运行它。这篇我们从 JUnit 5 的角度理解 pytest 的设计，看看 Python 测试为什么不需要 class，fixture 为什么比 `@BeforeEach` 更强大，以及 Python 的 mock 机制和 Java 的 Mockito 有什么不同。

## JUnit 5 vs pytest 对比

| 维度      | JUnit 5                                   | pytest                                     |
| --------- | ----------------------------------------- | ------------------------------------------ |
| 测试组织  | class + `@Test` 方法                      | 普通函数，`test_` 前缀                     |
| 断言方式  | `assertEquals(a, b)`                      | `assert a == b`（原生 assert）             |
| 前置/后置 | `@BeforeEach` / `@AfterEach`              | fixture（`@pytest.fixture`）               |
| 依赖注入  | `@Autowired`（Spring）                    | fixture 参数注入                           |
| 参数化    | `@ParameterizedTest` + `@CsvSource`       | `@pytest.mark.parametrize`                 |
| Mock      | Mockito + `@MockBean`（Spring）           | `unittest.mock` + `patch`                  |
| 测试套件  | `@Suite` / `@Nested`                      | 自动发现，按目录/标记分组                  |
| 生命周期  | class 实例（每个方法新建或不新建）        | scope：function / class / module / session |
| 异常测试  | `assertThrows(Exception.class, () -> {})` | `with pytest.raises(Exception):`           |
| 临时文件  | `@TempDir`                                | `tmp_path` fixture                         |

最关键的区别是前两行：JUnit 要求你写 class，pytest 不需要。JUnit 的 `assertEquals` 是框架提供的方法，pytest 直接用 Python 的 `assert` 关键字。这两个选择决定了两个框架的整体风格——JUnit 偏"正式"，pytest 偏"自然"。

## 为什么 pytest 不需要 class

JUnit 要求测试方法放在 class 里，因为 Java 本身就要求所有代码都在 class 里——这是 Java 的语言规则，不是测试框架的规则。JUnit 只是顺应了这个规则。Python 没有这个限制，函数可以独立存在，pytest 就让测试函数也独立存在。

```python
# tests/test_article_service.py

def test_create_article_with_valid_data():
    """pytest 测试 — 就是一个普通函数"""
    service = ArticleService()
    article = service.create(title="Python 测试", content="pytest 入门指南")
    assert article["id"] == 1
    assert article["title"] == "Python 测试"
    assert article["status"] == "draft"


def test_create_article_with_short_content_raises_error():
    """异常测试 — 用 pytest.raises 上下文管理器"""
    service = ArticleService()
    with pytest.raises(ValidationError, match="内容至少需要 10 个字符"):
        service.create(title="短文", content="太短")


def test_get_nonexistent_article_returns_none():
    """空值测试 — 直接 assert"""
    service = ArticleService()
    assert service.get_article(999) is None
```

等价的 JUnit 5 写法：

```java
// Java — JUnit 5
class ArticleServiceTest {

    private ArticleService service;

    @BeforeEach
    void setUp() {
        service = new ArticleService();
    }

    @Test
    void createArticleWithValidData() {
        var article = service.create("Python 测试", "pytest 入门指南");
        assertEquals(1, article.getId());
        assertEquals("Python 测试", article.getTitle());
        assertEquals("draft", article.getStatus());
    }

    @Test
    void createArticleWithShortContentRaisesError() {
        assertThrows(ValidationError.class,
            () -> service.create("短文", "太短"));
    }

    @Test
    void getNonexistentArticleReturnsNull() {
        assertNull(service.getArticle(999));
    }
}
```

JUnit 版本多了 `class` 声明、`@BeforeEach`、`@Test` 注解、`setUp` 方法——这些是框架的仪式感。pytest 版本没有这些，每个测试就是一个 `def`，直接写逻辑。

这不是说 pytest 不能用 class。当你需要共享 setup 逻辑时，pytest 也支持在 class 里组织测试：

```python
# pytest 也支持 class，但不是必须的
class TestArticleService:
    def test_create(self):
        service = ArticleService()
        article = service.create(title="Python 测试", content="内容足够长")
        assert article["status"] == "draft"
```

但 pytest 的 class 不需要继承任何基类，也不强制要求方法名以 `test` 开头（虽然这是惯例）。它只是组织代码的一种方式，不像 JUnit 那样是框架的要求。

### assert 的魔法：pytest 如何让原生 assert 有用

Java 的 `assertEquals` 能显示期望值和实际值，是因为它在方法内部做了比较和格式化。Python 的 `assert` 只是一个关键字，失败时只会抛 `AssertionError`，默认信息是 `assert a == b`，不会告诉你 `a` 和 `b` 分别是什么。

pytest 在运行时重写了 `assert` 语句——它通过 AST（抽象语法树）分析，在断言失败时自动提取变量的值并显示：

```
>       assert article["status"] == "published"
E       assert 'draft' == 'published'
E         - published
E         + draft
```

这和 JUnit 的 `assertEquals("published", article.getStatus())` 输出类似，但你不需要调用任何框架方法——就写 `assert`，pytest 帮你处理其余部分。

## fixture：比 @BeforeEach 更强大的依赖注入

JUnit 的 `@BeforeEach` 在每个测试方法前执行，用于初始化测试数据。但它有明显的局限：它只能初始化当前 class 内的成员变量，不能跨 class 共享；它不能参数化；它不能声明依赖关系。pytest 的 fixture 解决了所有这些问题。

### 基础 fixture

```python
# tests/conftest.py
import pytest


@pytest.fixture
def article_service():
    """创建 ArticleService 实例 — 相当于 JUnit 的 @BeforeEach"""
    service = ArticleService()
    yield service  # yield 之前是 setup，之后是 teardown
    service.cleanup()  # 测试结束后自动执行（即使测试失败）


@pytest.fixture
def sample_article():
    """返回测试数据"""
    return {
        "title": "pytest 入门",
        "content": "从 Java 开发者视角学习 Python 测试框架",
        "tags": ["python", "testing"],
    }


# tests/test_article_service.py
def test_create_article(article_service, sample_article):
    """fixture 通过函数参数注入 — 不需要手动创建"""
    article = article_service.create(**sample_article)
    assert article["id"] == 1
    assert article["title"] == "pytest 入门"
```

fixture 和 `@BeforeEach` 的关键区别在于**注入方式**：JUnit 的 `@BeforeEach` 通过 class 成员变量传递（`this.service`），pytest 的 fixture 通过函数参数注入。这意味着你不需要在每个测试方法里写 `service.xxx`——fixture 的值直接作为参数传入，更简洁也更明确。

### fixture 作用域 vs JUnit 生命周期

JUnit 的 `@BeforeEach` / `@BeforeAll` 只有两个粒度：每个方法前 / 每个 class 前。pytest 的 fixture 有四个作用域：

| 作用域   | JUnit 对应               | 适用场景                           |
| -------- | ------------------------ | ---------------------------------- |
| function | `@BeforeEach`            | 每个测试函数独立的数据             |
| class    | `@BeforeAll`（class 级） | 同一 class 内的测试共享            |
| module   | 无直接对应               | 同一文件内的测试共享               |
| session  | 无直接对应               | 整个测试会话共享（数据库、Docker） |

```python
@pytest.fixture(scope="session")
def db_engine():
    """整个测试会话只创建一次 — 相当于 Spring 的 @TestConfiguration singleton"""
    engine = create_engine("sqlite:///:memory:")
    Base.metadata.create_all(engine)
    yield engine
    engine.dispose()


@pytest.fixture(scope="function")
def db_session(db_engine):
    """每个测试函数一个独立事务 — 测试间完全隔离"""
    connection = db_engine.connect()
    transaction = connection.begin()
    session = Session(bind=connection)

    yield session

    session.close()
    transaction.rollback()  # 回滚，不影响其他测试
    connection.close()
```

JUnit 5 要实现同样的效果需要 `@TestInstance(Lifecycle.PER_CLASS)` 注解 + 手动管理状态。pytest 的 scope 是声明式的——你指定 `scope="session"`，pytest 自动管理创建和销毁的时机。

### fixture 依赖链

fixture 可以依赖其他 fixture，形成依赖链——JUnit 的 `@BeforeEach` 做不到这一点：

```python
@pytest.fixture
def db_engine():
    engine = create_engine("sqlite:///:memory:")
    Base.metadata.create_all(engine)
    yield engine
    engine.dispose()


@pytest.fixture
def db_session(db_engine):
    """依赖 db_engine — pytest 自动按依赖顺序执行"""
    connection = db_engine.connect()
    transaction = connection.begin()
    session = Session(bind=connection)
    yield session
    session.close()
    transaction.rollback()
    connection.close()


@pytest.fixture
def article_service(db_session):
    """依赖 db_session — 自动获得数据库会话"""
    return ArticleService(session=db_session)


def test_create_and_find_article(article_service):
    """只需要注入最顶层的 fixture，依赖链自动解析"""
    article = article_service.create(title="测试", content="内容")
    found = article_service.get_article(article["id"])
    assert found["title"] == "测试"
```

测试函数只需要声明它需要 `article_service`，pytest 自动解析整个依赖链：`db_engine` -> `db_session` -> `article_service`。这和 Spring 的依赖注入思路一样，但不需要 XML 配置或 `@Autowired` 注解——函数参数名就是声明。

## conftest.py：共享 fixture 的约定

JUnit 项目通常有一个 `AbstractBaseTest` 基类，把 `@BeforeEach` 和公共 fixture 放在里面，所有测试类继承它。这种方式的问题是继承耦合——你要修改 setup 逻辑就得改基类，影响所有子类。

pytest 用 `conftest.py` 文件替代继承。pytest 在收集测试时自动发现 `conftest.py`，里面的 fixture 对同级及下级目录的所有测试可见：

```
tests/
├── conftest.py              # 全局 fixture（所有测试可用）
├── test_models/
│   ├── conftest.py          # 模型测试专用 fixture
│   └── test_article.py
├── test_services/
│   ├── conftest.py          # 服务测试专用 fixture
│   └── test_article_service.py
└── test_api/
    └── test_article_api.py
```

`tests/conftest.py` 里的 fixture 对所有测试可见，`tests/test_models/conftest.py` 里的 fixture 只对模型测试可见。子目录的 `conftest.py` 可以覆盖父目录的同名 fixture。这种基于目录层级的作用域比 Java 的 class 继承更灵活——你不需要决定"所有测试类都继承哪个基类"，只需要把 fixture 放在合适的目录里。

## 参数化测试：一个函数测多组数据

JUnit 5 的 `@ParameterizedTest` 支持多种数据源（`@ValueSource`、`@CsvSource`、`@MethodSource`），功能强大但配置繁琐。pytest 的 `@pytest.mark.parametrize` 更简洁：

```python
import pytest


@pytest.mark.parametrize(
    "title, content, expected_valid",
    [
        ("正常标题", "这是一篇正常的文章内容", True),
        ("", "没有标题", False),
        ("标题", "短", False),
        ("a" * 201, "标题超过200字符", False),
    ],
)
def test_article_validation(title: str, content: str, expected_valid: bool):
    """一个函数覆盖 4 种场景 — JUnit 需要 4 个 @Test 方法"""
    validator = ArticleValidator()
    is_valid = validator.validate(title=title, content=content)
    assert is_valid == expected_valid
```

等价的 JUnit 5 写法：

```java
// Java — JUnit 5
@ParameterizedTest
@CsvSource({
    "正常标题, 这是一篇正常的文章内容, true",
    ", 没有标题, false",
    "标题, 短, false",
})
void articleValidation(String title, String content, boolean expectedValid) {
    var validator = new ArticleValidator();
    assertEquals(expectedValid, validator.validate(title, content));
}

// 超长标题的测试需要单独写，因为 @CsvSource 不方便处理 201 字符的字符串
@Test
void articleValidationWithTooLongTitle() {
    var validator = new ArticleValidator();
    assertFalse(validator.validate("a".repeat(201), "标题超过200字符"));
}
```

pytest 的参数化可以直接放任何 Python 对象——字符串、数字、列表、字典。JUnit 的 `@CsvSource` 只能传字符串，数字需要手动转换，复杂对象需要 `@MethodSource`。

### 多参数组合

pytest 支持笛卡尔积参数化——多个 `@pytest.mark.parametrize` 叠加时，会生成所有可能的组合：

```python
@pytest.mark.parametrize("role", ["admin", "editor", "viewer"])
@pytest.mark.parametrize("action", ["read", "write", "delete"])
def test_permission_check(role: str, action: str):
    """3 x 3 = 9 种组合 — pytest 自动生成"""
    checker = PermissionChecker()
    result = checker.check(role=role, action=action)
    # admin 可以做任何事，viewer 只能读...
```

JUnit 5 的 `@ParameterizedTest` 不支持多个数据源的笛卡尔积——你需要用 `@ArgumentsSources` 或者自己写循环。pytest 的装饰器叠加方式更直观。

## mock 和 patch：隔离外部依赖

Java 用 Mockito 做 mock，Python 用标准库的 `unittest.mock`。两者的核心思路一样：创建一个假对象替代真实依赖，让测试只关注被测代码的逻辑。

### MagicMock vs Mockito.mock()

```python
from unittest.mock import MagicMock, patch


def test_article_service_with_mock_repository():
    """mock 数据库访问 — 不需要真正的数据库"""
    mock_repo = MagicMock()
    mock_repo.find_by_id.return_value = {"id": 1, "title": "Mock 文章"}

    service = ArticleService(repository=mock_repo)
    article = service.get_article(1)

    assert article["title"] == "Mock 文章"
    # 验证 mock 被正确调用
    mock_repo.find_by_id.assert_called_once_with(1)
```

等价的 Mockito 写法：

```java
// Java — Mockito
@Test
void articleServiceWithMockRepository() {
    var mockRepo = mock(ArticleRepository.class);
    when(mockRepo.findById(1)).thenReturn(Optional.of(
        new Article(1, "Mock 文章")
    ));

    var service = new ArticleService(mockRepo);
    var article = service.getArticle(1);

    assertEquals("Mock 文章", article.getTitle());
    verify(mockRepo).findById(1);
}
```

语法非常相似：`return_value` 对应 `when().thenReturn()`，`assert_called_once_with` 对应 `verify()`。区别在于 Python 的 mock 对象更"宽容"——MagicMock 会自动为任何属性和方法创建 mock，不会因为访问了不存在的属性而报错。Mockito 默认对 null 返回更严格。

### patch：替换模块中的对象

`patch` 是 Python mock 机制中最独特的部分。它可以在运行时替换模块中的任何对象——函数、类、变量。这解决了"被测代码内部 import 了外部模块，你没法通过参数注入 mock"的问题。

```python
# src/techblog/services/article_service.py
import httpx


class ArticleService:
    def fetch_external_article(self, url: str) -> dict:
        """从外部 API 获取文章"""
        response = httpx.get(url)
        return response.json()


# tests/test_article_service.py
from unittest.mock import patch


@patch("techblog.services.article_service.httpx.get")
def test_fetch_external_article(mock_get):
    """patch 替换 httpx.get — 不需要真正的网络请求"""
    mock_get.return_value = MagicMock(
        status_code=200,
        json=lambda: {"id": 1, "title": "外部文章"},
    )

    service = ArticleService()
    article = service.fetch_external_article("https://api.example.com/articles/1")

    assert article["title"] == "外部文章"
    mock_get.assert_called_once_with("https://api.example.com/articles/1")
```

`patch` 的第一个参数是"对象的完整路径"——`techblog.services.article_service.httpx.get`。pytest 在运行时把 `httpx.get` 替换成 mock 对象，测试结束后自动恢复。Java 没有直接等价的机制——Spring 的 `@MockBean` 在测试上下文中替换 Spring Bean，但那是框架级别的，不是语言级别的。

`patch` 的一个常见陷阱是路径写错。你要 patch 的是"被测代码引用的对象"，不是"对象定义的位置"：

```python
# 正确 — patch 被测代码模块中的引用
@patch("techblog.services.article_service.httpx.get")

# 错误 — patch httpx 模块本身的 get（不起作用，因为被测代码已经 import 了引用）
@patch("httpx.get")
```

这和 Java 的 `@MockBean` 不一样——`@MockBean` 替换的是 Spring 容器中的 Bean，和 import 路径无关。Python 的 `patch` 操作的是模块的命名空间，你必须 patch "被测代码看到的名字"。

## Trade-off：自由与约定的平衡

pytest 的"自由"体现在你不需要按框架的规则组织代码——不需要 class，不需要继承，不需要特定的方法签名。一个 `test_` 前缀的函数就是一个测试，一个 `assert` 就是一个断言。这种自由让 Python 测试的编写速度很快，学习曲线很低。

但自由也有代价。JUnit 的 class 结构天然提供了组织约束——`ArticleServiceTest` 里只放 `ArticleService` 的测试，`ArticleControllerTest` 里只放 Controller 的测试。pytest 没有这种强制，如果你的测试文件里放了 50 个测试函数，组织性就会变差。你需要靠目录结构、`conftest.py` 和命名规范来维持秩序——这是 Python 项目的老问题，测试也不例外。

fixture 的依赖注入比 JUnit 的 `@BeforeEach` 更强大，但也更容易写出难以理解的 fixture 链。当你看到 `def test_something(service, session, client, admin_user)` 时，你需要追踪四个 fixture 的定义才能理解测试的前置条件。JUnit 的 `@BeforeEach` 虽然简单，但至少所有初始化逻辑都在同一个 class 里，容易找到。

另一个 trade-off 是 assert 的安全性。pytest 重写 assert 让输出更好看，但它本质上还是 Python 的 `assert`——如果你用 `python -O`（优化模式）运行代码，所有 assert 都会被跳过。JUnit 的 `assertEquals` 不受运行模式影响。所以在 CI 环境中，确保 pytest 不在 `-O` 模式下运行。
