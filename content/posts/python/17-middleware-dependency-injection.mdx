## 一个真实的问题

CMS 需要在每个请求中记录日志、计算耗时、注入数据库会话、检查用户认证。在 Spring Boot 中，这些分别对应 Interceptor、Filter、`@Autowired` 和 `@PreAuthorize`，各自有不同的注册方式和执行时机。FastAPI 把这些能力统一到了两个机制中：**中间件**（Middleware）和**依赖注入**（Depends）。

这两种机制的设计哲学和 Spring Boot 截然不同。理解它们的区别，是写出结构清晰的 FastAPI 应用的关键。

---

## 请求处理流程

先看一个请求在 FastAPI 中的完整生命周期：

```
客户端请求
  |
  v
ASGI Server (Uvicorn)
  |
  v
Middleware 1 (最外层)
  +-- Middleware 2
  |     +-- Middleware 3
  |     |     +-- 路由处理函数 (Depends 注入在这里执行)
  |     |     +-- 响应返回
  |     +-- 响应返回
  +-- 响应返回
  |
  v
客户端收到响应
```

中间件按注册顺序执行请求阶段，按逆序执行响应阶段（洋葱模型）。Depends 在路由函数执行前解析依赖，在路由函数执行后清理资源。

对应 Spring Boot 的请求链路：

```
Filter (Servlet 级别)
  -> Interceptor (Spring MVC 级别)
    -> AOP 切面 (@Before, @After)
      -> Controller 方法
        -> @Autowired 注入的 Service
```

FastAPI 的中间件对应 Filter + Interceptor，Depends 对应 `@Autowired` + AOP。只是 FastAPI 没有把这些概念拆成四层，而是合并成了两层。

## FastAPI 中间件

### 请求日志中间件

CMS 需要记录每个请求的方法、路径、状态码和耗时。在 Spring Boot 中通常用 `HandlerInterceptor` 或 `CommonsRequestLoggingFilter`，FastAPI 中继承 `BaseHTTPMiddleware`：

```python
import time, uuid, logging
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response

logger = logging.getLogger(__name__)

class RequestLoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next) -> Response:
        request_id = str(uuid.uuid4())[:8]
        request.state.request_id = request_id
        start = time.time()

        response = await call_next(request)

        process_time = time.time() - start
        response.headers["X-Request-ID"] = request_id
        response.headers["X-Process-Time"] = f"{process_time:.3f}"

        logger.info(f"[{request_id}] {request.method} {request.url.path} "
                     f"-> {response.status_code} ({process_time:.3f}s)")
        return response

# 注册
app.add_middleware(RequestLoggingMiddleware)
```

对应 Spring Boot 的写法：

```java
@Component
public class LoggingInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        MDC.put("requestId", UUID.randomUUID().toString().substring(0, 8));
        request.setAttribute("startTime", System.currentTimeMillis());
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) {
        long duration = System.currentTimeMillis() - (Long) request.getAttribute("startTime");
        log.info("[{}] {} {} -> {} ({}ms)",
            MDC.get("requestId"), request.getMethod(), request.getRequestURI(),
            response.getStatus(), duration);
        MDC.clear();
    }
}
// 还需要 WebMvcConfigurer 注册这个 Interceptor
```

Python 的实现是一个类一个方法，不需要注册配置。Java 需要定义 Interceptor、实现三个回调方法、写 `WebMvcConfigurer` 注册。对于简单的日志记录，FastAPI 的方式更直接。

### 内置中间件

FastAPI 内置了常用中间件，和 Spring Boot 的内置 Filter 对应：

```python
from fastapi.middleware.cors import CORSMiddleware
from fastapi.middleware.gzip import GZipMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
app.add_middleware(GZipMiddleware, minimum_size=1000)
```

对应 Spring Boot 的配置：

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
            .allowedOrigins("http://localhost:3000")
            .allowedMethods("*")
            .allowCredentials(true);
    }
}
// GZip 压缩在 application.yml 中配置：server.compression.enabled=true
```

两者功能等价。区别在于 Spring Boot 的 CORS 配置是声明式的（配置类或 yml），FastAPI 的中间件是命令式的（Python 代码）。

## Depends 依赖注入

### 核心概念

FastAPI 的 `Depends` 和 Spring 的 `@Autowired` 解决同一个问题：**把依赖的创建和使用解耦**。但注入模型完全不同：

Spring 的依赖注入是**容器驱动的**：Spring 容器启动时扫描 `@Component`、`@Service`、`@Repository`，创建 Bean 实例并缓存。使用时通过 `@Autowired` 或构造器注入从容器获取。

FastAPI 的依赖注入是**函数驱动的**：依赖就是一个 Python 函数，`Depends` 告诉 FastAPI 在路由执行前调用这个函数，把返回值注入到路由参数中。没有容器，没有 Bean 生命周期管理。

```python
# 数据库会话依赖
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionFactory() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise

# 认证依赖
async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> User:
    payload = decode_token(token)
    user = await UserService.get_by_id(db, int(payload["sub"]))
    if not user:
        raise HTTPException(status_code=401)
    return user

# 在路由中使用
@router.get("/posts")
async def list_posts(
    user: User = Depends(get_current_user),  # 自动注入当前用户
    db: AsyncSession = Depends(get_db),      # 自动注入数据库会话
):
    return await PostService.get_list(db)
```

对应 Spring Boot 的写法：

```java
@RestController
@RequestMapping("/posts")
public class PostController {
    private final PostService postService;
    private final CurrentUserService currentUserService;

    // 构造器注入
    public PostController(PostService postService, CurrentUserService currentUserService) {
        this.postService = postService;
        this.currentUserService = currentUserService;
    }

    @GetMapping
    @PreAuthorize("isAuthenticated()")
    public List<Post> listPosts(@AuthenticationPrincipal User user) {
        return postService.getByUser(user);
    }
}
```

两种方案的区别在于粒度：Spring 的 `@Autowired` 注入的是类级别的 Bean（Service、Repository），生命周期由容器管理。FastAPI 的 `Depends` 注入的是函数的返回值，每次请求调用一次函数。Spring 的方式适合有状态的 Bean（连接池、缓存），FastAPI 的方式适合无状态的依赖（数据库会话、当前用户）。

### yield 依赖：资源清理

`yield` 是 FastAPI 依赖注入中最重要的机制之一，它解决了资源清理的问题：

```python
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    session = AsyncSessionFactory()
    try:
        yield session          # yield 之前：路由执行前
    finally:
        await session.close()  # yield 之后：路由执行后（无论成功或异常）
```

这和 Java 中 `try-with-resources` 或 Spring 的 `@Bean(destroyMethod=...)` 效果相同，但写法更简洁。在 Spring Boot 中，数据库事务管理通常依赖 `@Transactional` 注解：

```java
@Service
public class PostService {
    @Transactional  // 方法结束后自动 commit/rollback
    public Post create(PostCreateDTO dto) {
        Post post = new Post(dto.getTitle());
        return postRepository.save(post);
    }
}
```

FastAPI 没有声明式事务注解，事务管理通过 `yield` 依赖的 `commit/rollback` 显式控制。少了注解的便利，但事务边界更清晰。

### 依赖链与子依赖

FastAPI 的依赖可以组合形成链式结构。CMS 的权限检查依赖就是一个典型例子 -- 它依赖认证依赖：

```python
# 依赖链：oauth2_scheme -> get_current_user -> require_role
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/login")

async def get_current_user(token: str = Depends(oauth2_scheme), ...) -> User:
    # 验证 Token，返回 User
    ...

def require_role(*roles: str):
    """权限检查依赖工厂"""
    async def checker(user: User = Depends(get_current_user)) -> User:
        if user.role not in roles:
            raise HTTPException(status_code=403, detail="Permission denied")
        return user
    return checker

# 使用
@router.delete("/{post_id}")
async def delete_post(
    post_id: int,
    user: User = Depends(require_role("admin")),  # 完整链路
):
    ...
```

`require_role("admin")` 返回一个函数，这个函数通过 `Depends(get_current_user)` 依赖认证逻辑。FastAPI 会自动解析整条依赖链：先提取 Token -> 验证 Token -> 查询用户 -> 检查角色。

Java 中的等价实现：

```java
// Spring Security 的 Filter Chain 天然形成依赖链
// JwtAuthenticationFilter -> SecurityContext -> @PreAuthorize
@DeleteMapping("/{id}")
@PreAuthorize("hasRole('ADMIN')")
public void deletePost(@PathVariable Long id) { ... }
```

Spring Security 的链路是隐式的（Filter Chain 按顺序执行），FastAPI 的链路是显式的（代码中可以看到依赖关系）。对于调试和维护，显式链路更有优势。

### Annotated 类型别名

当同一个依赖在多个路由中重复使用时，`Annotated` 可以简化代码：

```python
from typing import Annotated

# 定义类型别名
DbSession = Annotated[AsyncSession, Depends(get_db)]
CurrentUser = Annotated[User, Depends(get_current_user)]
AdminUser = Annotated[User, Depends(require_role("admin"))]

# 使用：签名更简洁
@router.get("/posts")
async def list_posts(db: DbSession, user: CurrentUser):
    ...

@router.delete("/{post_id}")
async def delete_post(post_id: int, db: DbSession, user: AdminUser):
    ...
```

这类似于 Java 中自定义注解 `@CurrentUser` 或 Spring Security 的 `@WithMockUser`，但 `Annotated` 是纯类型层面的语法糖，不需要定义新的注解类。

## 中间件 vs 依赖注入的选择

两者都能拦截请求，但适用场景不同：

| 场景          | 用中间件         | 用 Depends                   |
| ------------- | ---------------- | ---------------------------- |
| 请求日志      | 是               | 否（需要处理所有请求）       |
| CORS 配置     | 是               | 否                           |
| 认证检查      | 可以，但不够灵活 | 是（可以精确到路由级别）     |
| 数据库会话    | 否（需要返回值） | 是（yield 管理生命周期）     |
| 分页参数      | 否               | 是（返回分页参数对象）       |
| 限流          | 是               | 否                           |
| 修改请求/响应 | 是               | 否（Depends 不接触请求对象） |

简单规则：**需要修改请求/响应对象、或需要处理所有请求，用中间件；需要返回值供路由使用、或只需要在特定路由生效，用 Depends。**

对应 Spring Boot 的选择规则：全局 Filter 对应 FastAPI 中间件，`@Autowired` + `@PreAuthorize` 对应 FastAPI Depends。

## 应用生命周期管理

FastAPI 用 `lifespan` 上下文管理器替代了旧版的 `@app.on_event`（类似于 Spring Boot 的 `ApplicationRunner` / `@PostConstruct`）：

```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动阶段
    engine = create_async_engine(settings.database_url)
    print("CMS 启动完成")
    yield
    # 关闭阶段
    await engine.dispose()
    print("CMS 已关闭")

app = FastAPI(lifespan=lifespan)
```

对应 Spring Boot：

```java
@Component
public class AppStartupRunner implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) {
        System.out.println("CMS 启动完成");
    }
}

@PreDestroy  // 或 DisposableBean
public void cleanup() { ... }
```

功能一样，FastAPI 用一个函数的 yield 前后两段替代了 Spring 的两个注解/接口。

## 对比总结

| 维度         | Spring Boot                                       | FastAPI                  |
| ------------ | ------------------------------------------------- | ------------------------ |
| 中间件       | Filter + Interceptor（两层）                      | Middleware（一层）       |
| 依赖注入     | IoC 容器 + @Autowired                             | Depends 函数             |
| 依赖生命周期 | Bean（Singleton/Prototype/Request）               | 每次请求调用函数         |
| 资源清理     | @PreDestroy / try-with-resources / @Transactional | yield 依赖的 finally     |
| 权限控制     | @PreAuthorize + SecurityFilterChain               | RoleChecker + Depends    |
| 依赖组合     | Bean 之间的 @Autowired 注入                       | 子依赖链（Depends 嵌套） |
| 应用生命周期 | @PostConstruct / ApplicationRunner                | lifespan 上下文管理器    |
| 全局配置     | @Configuration + @Bean                            | 函数 + lru_cache         |

---

## Trade-off

FastAPI 的中间件和依赖注入的设计哲学是**显式优于隐式**。没有 IoC 容器意味着没有 Spring 那样的自动装配魔法，但每个依赖的来源和执行顺序都可以从代码中直接读出来。调试时，你不需要猜测 Spring 容器为什么没有注入某个 Bean -- 因为根本没有容器。

代价是缺少 Spring 的容器级能力：没有 AOP 代理（`@Transactional` 的底层）、没有 Bean 作用域（Singleton/Prototype）、没有条件化装配（`@ConditionalOnProperty`）。这些在 Python 中需要手动实现，或者在依赖函数内部用 `if/else` 处理。

对于 CMS 这种中等规模的应用，FastAPI 的方案在开发效率和可维护性上有明显优势。Spring 的 IoC 容器在大型企业应用中更有价值，但它的学习成本和调试复杂度也更高。
