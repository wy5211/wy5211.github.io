# 包管理与项目工程化

## 前言

在前五篇文章中，我们学习了 Python 的核心语言特性。但从"写代码"到"做工程"，中间还有很长的路。一个规范的 Python 项目应该有清晰的结构、合理的依赖管理、统一的代码风格和自动化的质量检查。

Python 生态在过去几年经历了巨大的变化：`pip` + `venv` 的传统组合正在被更现代的工具取代，`pyproject.toml` 成为统一的配置标准，`ruff` 一个工具就替代了 `flake8` + `isort` + `black` 三个工具。本文将带你搭建一套现代化的 Python 项目工程体系。

---

## 项目结构规范

### src layout vs flat layout

Python 项目有两种主流的目录布局：

**Flat Layout（扁平布局）**：

```
myproject/
├── pyproject.toml
├── mypackage/
│   ├── __init__.py
│   ├── module.py
└── tests/
    └── test_module.py
```

**Src Layout（源码布局）**：

```
myproject/
├── pyproject.toml
├── src/
│   └── mypackage/
│       ├── __init__.py
│       ├── module.py
└── tests/
    └── test_module.py
```

**推荐使用 src layout**，原因如下：

1. **避免导入歧义**：防止在项目根目录执行脚本时，意外导入本地未安装的代码，而不是 `site-packages` 中安装的版本
2. **强制正确安装**：必须通过 `pip install -e .` 安装后才能导入，确保测试的是真实安装后的行为
3. **社区推荐**：PyPA（Python Packaging Authority）官方推荐

### 标准项目结构

以下是一个生产级 Python 项目的完整结构：

```
myproject/
├── pyproject.toml          # 项目元数据与工具配置（统一入口）
├── README.md
├── .gitignore
├── .pre-commit-config.yaml # pre-commit 钩子配置
├── Makefile                # 任务自动化
├── src/
│   └── mypackage/
│       ├── __init__.py     # 包标识 + 公开 API
│       ├── __main__.py     # python -m mypackage 入口
│       ├── core/
│       │   ├── __init__.py
│       │   ├── config.py   # 配置管理
│       │   └── exceptions.py
│       ├── models/
│       │   ├── __init__.py
│       │   └── user.py
│       ├── services/
│       │   ├── __init__.py
│       │   └── user_service.py
│       └── api/
│           ├── __init__.py
│           └── routes.py
├── tests/
│   ├── __init__.py
│   ├── conftest.py         # pytest 共享 fixture
│   ├── test_models/
│   │   └── test_user.py
│   └── test_services/
│       └── test_user_service.py
└── scripts/
    └── seed_data.py
```

### `__init__.py` 的作用

`__init__.py` 是 Python 包的标识文件，它的职责包括：

```python
# src/mypackage/__init__.py

# 1. 标识这是一个 Python 包（可以是空文件）

# 2. 定义公开 API（控制 from mypackage import * 的行为）
__all__ = ["UserService", "create_app"]

# 3. 提供便捷导入
from mypackage.services.user_service import UserService
from mypackage.api.routes import create_app

# 4. 版本信息
__version__ = "1.0.0"
```

> **注意**：Python 3.3+ 支持"命名空间包"（Namespace Packages），即没有 `__init__.py` 的目录也能作为包被导入。但显式使用 `__init__.py` 仍然是推荐做法，因为它能明确表达意图并提供更好的兼容性。

## uv 包管理工具

[uv](https://github.com/astral-sh/uv) 是 Astral 团队（ruff 的开发者）推出的新一代 Python 包管理器，用 Rust 编写，速度极快。

### 安装 uv

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# 使用 Homebrew
brew install uv

# 使用 pip
pip install uv
```

### 初始化项目

```bash
# 创建新项目（自动生成 pyproject.toml 和虚拟环境）
uv init myproject
cd myproject

# 项目结构：
# myproject/
# ├── pyproject.toml
# ├── hello.py
# └── hello_test.py
```

### 常用命令

```bash
# 添加依赖（自动更新 pyproject.toml 和 uv.lock）
uv add fastapi uvicorn
uv add --dev pytest ruff mypy   # 开发依赖

# 安装所有依赖
uv sync

# 运行 Python
uv run python hello.py

# 运行脚本（自动处理虚拟环境）
uv run pytest
uv run ruff check .

# 删除虚拟环境并重新创建
uv venv --clear
```

### uv 的优势

| 特性            | uv           | pip + venv         |
| --------------- | ------------ | ------------------ |
| 速度            | 快 10-100 倍 | 基准               |
| 依赖解析        | 完整解析     | 不解析             |
| 锁文件          | `uv.lock`    | 无（需 pip-tools） |
| Python 版本管理 | 内置         | 需 pyenv           |
| 工具运行        | `uv run`     | 需激活 venv        |

## poetry 包管理工具

[Poetry](https://python-poetry.org/) 是另一个成熟的依赖管理工具，如果你已经熟悉它，完全可以继续使用。

### 安装与初始化

```bash
# 安装
curl -sSL https://install.python-poetry.org | python3 -

# 创建项目
poetry new myproject
cd myproject

# 在已有项目中初始化
poetry init
```

### 常用命令

```bash
# 添加依赖
poetry add fastapi uvicorn
poetry add --group dev pytest ruff mypy

# 安装依赖
poetry install

# 运行命令（自动在虚拟环境中执行）
poetry run pytest
poetry run python hello.py

# 进入虚拟环境 shell
poetry shell

# 构建与发布
poetry build
poetry publish
```

### uv vs poetry 如何选择

- **新项目**：推荐 uv，速度更快，更现代
- **已有 poetry 项目**：无需迁移，poetry 依然稳定可靠
- **需要 Python 版本管理**：uv 内置，poetry 需配合 pyenv

## pyproject.toml 配置详解

`pyproject.toml` 是 Python 项目的统一配置文件，取代了 `setup.py`、`setup.cfg`、`requirements.txt` 等多个配置文件。以下是使用 uv 时的完整配置：

```toml
# pyproject.toml

[project]
name = "myproject"
version = "0.1.0"
description = "A modern Python project"
readme = "README.md"
requires-python = ">=3.12"
license = { text = "MIT" }
authors = [
    { name = "Your Name", email = "you@example.com" }
]

# 生产依赖
dependencies = [
    "fastapi>=0.110.0",
    "uvicorn[standard]>=0.27.0",
    "pydantic>=2.0.0",
    "pydantic-settings>=2.0.0",
]

# 可选依赖组（替代 requirements-dev.txt）
[dependency-groups]
dev = [
    "pytest>=8.0.0",
    "pytest-asyncio>=0.23.0",
    "pytest-cov>=4.1.0",
    "ruff>=0.3.0",
    "mypy>=1.9.0",
    "pre-commit>=3.6.0",
]

# 项目入口点
[project.scripts]
myproject = "mypackage.__main__:main"

# 构建系统
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

# 工具配置 - ruff
[tool.ruff]
# 目标 Python 版本
target-version = "py312"
# 每行最大字符数
line-length = 88
# 排除目录
exclude = [".git", "__pycache__", ".venv", "dist"]

# ruff lint 规则配置
[tool.ruff.lint]
# 启用的规则集
select = [
    "E",    # pycodestyle errors
    "W",    # pycodestyle warnings
    "F",    # pyflakes
    "I",    # isort（导入排序）
    "N",    # pep8-naming
    "UP",   # pyupgrade（自动升级语法）
    "B",    # flake8-bugbear
    "SIM",  # flake8-simplify
    "RUF",  # ruff 特有规则
]
# 忽略的规则
ignore = ["E501"]  # 行长度由 formatter 处理

# isort 配置
[tool.ruff.lint.isort]
known-first-party = ["mypackage"]

# ruff formatter 配置
[tool.ruff.format]
quote-style = "double"
indent-style = "space"

# 工具配置 - mypy
[tool.mypy]
python_version = "3.12"
strict = true
# 第三方库的类型存根
[[tool.mypy.overrides]]
module = ["uvicorn.*"]
ignore_missing_imports = true

# 工具配置 - pytest
[tool.pytest.ini_options]
testpaths = ["tests"]
python_files = ["test_*.py"]
python_functions = ["test_*"]
addopts = "-v --tb=short"
# 异步测试配置
asyncio_mode = "auto"

# 工具配置 - coverage
[tool.coverage.run]
source = ["src"]
omit = ["*/tests/*", "*/__main__.py"]

[tool.coverage.report]
exclude_lines = [
    "pragma: no cover",
    "if TYPE_CHECKING:",
    "raise NotImplementedError",
]
```

## 开发工具配置

### ruff：一个工具替代三个

[ruff](https://github.com/astral-sh/ruff) 用 Rust 编写，将 linting（代码检查）和 formatting（代码格式化）合二为一：

```bash
# 安装
uv add --dev ruff

# 代码检查（替代 flake8）
uv run ruff check .

# 自动修复
uv run ruff check --fix .

# 代码格式化（替代 black）
uv run ruff format .

# 查看某个规则的具体说明
uv run ruff rule E501
```

**ruff 的规则映射**：

| 旧工具          | ruff 规则集   |
| --------------- | ------------- |
| flake8          | `E`, `W`, `F` |
| isort           | `I`           |
| black           | `ruff format` |
| pyupgrade       | `UP`          |
| pep8-naming     | `N`           |
| flake8-bugbear  | `B`           |
| flake8-simplify | `SIM`         |

### 任务自动化：Makefile

```makefile
# Makefile

.PHONY: install format lint test cov clean

# 安装依赖
install:
	uv sync

# 代码格式化
format:
	uv run ruff format .
	uv run ruff check --fix .

# 代码检查
lint:
	uv run ruff check .
	uv run mypy src/

# 运行测试
test:
	uv run pytest

# 运行测试（含覆盖率）
cov:
	uv run pytest --cov=src --cov-report=html

# 清理
clean:
	find . -type d -name __pycache__ -exec rm -rf {} +
	find . -type d -name .pytest_cache -exec rm -rf {} +
	find . -type d -name .ruff_cache -exec rm -rf {} +
	rm -rf htmlcov/ dist/ .coverage
```

使用方式：

```bash
make install    # 安装依赖
make format     # 格式化代码
make lint       # 代码检查
make test       # 运行测试
make cov        # 覆盖率报告
```

## pre-commit 钩子

[pre-commit](https://pre-commit.com/) 在每次 `git commit` 前自动运行代码检查，防止不规范的代码进入仓库。

### 安装与配置

```bash
# 安装 pre-commit
uv add --dev pre-commit

# 安装 git 钩子
uv run pre-commit install
```

### `.pre-commit-config.yaml`

```yaml
# .pre-commit-config.yaml
repos:
  # ruff - 代码检查与格式化
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.9.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

  # 通用检查
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v5.0.0
    hooks:
      - id: trailing-whitespace # 删除行尾空格
      - id: end-of-file-fixer # 文件末尾空行
      - id: check-yaml # YAML 语法检查
      - id: check-added-large-files # 防止提交大文件
        args: ["--maxkb=500"]
      - id: check-merge-conflict # 检查合并冲突标记

  # mypy 类型检查
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.14.0
    hooks:
      - id: mypy
        additional_dependencies:
          - pydantic
          - types-requests
        args: [--ignore-missing-imports]
```

### 运行方式

```bash
# 对所有文件运行（首次配置后建议运行一次）
uv run pre-commit run --all-files

# 日常开发中，commit 时自动触发
git commit -m "feat: add user service"
# pre-commit 会自动检查暂存的文件
```

## 环境变量管理

### python-dotenv

最基础的环境变量加载方式：

```bash
# .env 文件（不要提交到 git）
DATABASE_URL=postgresql://user:pass@localhost:5432/mydb
REDIS_URL=redis://localhost:6379
SECRET_KEY=your-secret-key
DEBUG=true
```

```python
# src/mypackage/core/config.py
from pathlib import Path
from dotenv import load_dotenv

# 加载 .env 文件
load_dotenv()

import os

class Config:
    """简单的环境变量配置"""
    DATABASE_URL: str = os.getenv("DATABASE_URL", "sqlite:///default.db")
    REDIS_URL: str = os.getenv("REDIS_URL", "redis://localhost:6379")
    SECRET_KEY: str = os.getenv("SECRET_KEY", "change-me")
    DEBUG: bool = os.getenv("DEBUG", "false").lower() == "true"
```

### pydantic-settings（推荐）

结合 Pydantic 的类型验证和环境变量管理，更加健壮：

```python
# src/mypackage/core/config.py
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import Field


class Settings(BaseSettings):
    """使用 pydantic-settings 管理环境变量配置"""

    model_config = SettingsConfigDict(
        env_file=".env",        # 从 .env 文件加载
        env_file_encoding="utf-8",
        case_sensitive=False,   # 环境变量不区分大小写
    )

    # 应用配置
    app_name: str = "MyApp"
    debug: bool = False
    secret_key: str = Field(..., min_length=16)  # 必填，最少 16 字符

    # 数据库配置
    database_url: str = "postgresql://localhost:5432/mydb"
    database_pool_size: int = Field(default=5, ge=1, le=20)

    # Redis 配置
    redis_url: str = "redis://localhost:6379"
    redis_ttl: int = Field(default=3600, ge=0)

    # 服务器配置
    host: str = "0.0.0.0"
    port: int = Field(default=8000, ge=1, le=65535)


# 全局单例
settings = Settings()

# 使用
if __name__ == "__main__":
    print(settings.app_name)
    print(settings.database_url)
    # 如果 SECRET_KEY 未设置或太短，启动时就会报错
```

**pydantic-settings 的优势**：

1. **类型自动转换**：`port: int` 会自动将 `"8000"` 转为整数
2. **自动验证**：启动时就能发现配置错误
3. **默认值**：每个字段都可以有默认值
4. **多环境支持**：通过 `.env.production`、`.env.development` 切换

## 最佳实践

1. **始终使用 `pyproject.toml`**：不要再创建 `setup.py` 或 `requirements.txt`（除非有特殊需求）
2. **使用 src layout**：避免导入歧义
3. **依赖分离**：生产依赖和开发依赖分开管理
4. **锁文件要提交**：`uv.lock` 或 `poetry.lock` 必须提交到 git，确保团队环境一致
5. **`.env` 不要提交**：在 `.gitignore` 中添加 `.env`，提供 `.env.example` 作为模板
6. **pre-commit 从第一天就配置**：比后期补加容易得多
7. **统一代码风格**：团队统一使用 ruff 的配置，避免"格式之争"

## 常见问题

**Q：uv 和 poetry 能否共存？**

A：可以，但不建议在同一个项目中混用。它们使用不同的锁文件格式（`uv.lock` vs `poetry.lock`）。选择一个作为主力工具即可。

**Q：什么时候需要 `__init__.py`？**

A：在 src layout 中，每个包目录都需要 `__init__.py`。它可以是空文件，但建议在其中定义 `__all__` 来明确公开 API。注意：`tests/` 目录下也需要一个 `__init__.py`，否则某些导入可能出问题。

**Q：`pyproject.toml` 中的 `[dependency-groups]` 和 `[project.optional-dependencies]` 有什么区别？**

A：`[dependency-groups]`（PEP 735）是 2024 年引入的新标准，专为开发依赖设计；`[project.optional-dependencies]`（PEP 621）是可选功能依赖，适合 `pip install myproject[extra]` 这种场景。开发工具推荐使用 `dependency-groups`。

**Q：如何管理多个 Python 版本？**

A：uv 内置了 Python 版本管理：`uv python install 3.12`、`uv python pin 3.12`。如果使用 poetry，可以配合 pyenv。

- **项目结构**：推荐 src layout，清晰分离源码和测试

- **代码质量**：ruff 替代 flake8 + isort + black，pre-commit 自动检查
