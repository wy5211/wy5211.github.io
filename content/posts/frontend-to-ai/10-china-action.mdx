前面九篇，我们讲了现实、五条活法、护城河、误区。这最后一篇，落地到**中国大陆读者**最关心的实操，再给你一份**按活法定制的行动清单**——不是泛泛的「努力学习」，而是「你这周该做什么」。

## 一、中国大陆的四个特殊现实

### 1. 出海几乎是最优解

如果你想做独立开发（第 5 篇），**出海几乎必选**。原因很现实：

- **国内白嫖多、付费意愿低**：同样一个工具，海外用户愿意每月付 $10，国内用户会找你要免费版或破解；
- **合规复杂**：国内做 AI 产品，备案、内容审核都是绕不开的成本；
- **idoubi、kulove 这些成功案例，都直接屏蔽了国内 IP**——这不是崇洋，是商业上更划算。

收款的标准做法是：注册一家美国公司（Stripe Atlas），用 Stripe 收款。这有成本（第 5 篇算过账），但它能让你触达付费意愿最强的市场。

### 2. 国内红利在「前端 × AI」复合岗

如果你不想独立开发，而是想在公司内转型，国内的真实红利在**复合岗**——北京、深圳、杭州都有「前端 × AI」岗位，开到 **30-55K**，要求是 React 精通 + workflow / 多 agent / AI coding 经验。

字节跳动 2025 年密集开放「AI 全栈开发」「AI Agent 前端」，社招 30-60K·15 薪，明确「熟悉大模型、AI Agent 者优先」。与此同时纯前端岗在缩编。**方向非常清楚：复合岗在扩，纯前端在缩。**

### 3. JD 关键词对齐：绕过 HR 的机器筛简历

这是个很实操、但很少有人明说的技巧。

国内大厂招聘，第一关往往是 HR 用机器筛简历——**系统按 JD 关键词匹配，你不写那些词，简历直接被刷**，技术再好也到不了面试官面前。

比如你学的是 LangGraph，但 JD 写的是「熟悉 CrewAI / AutoGen」——机器筛不上。怎么办？**在简历里把同义概念都列上**，写成「熟悉 LangGraph，对 CrewAI / AutoGen 等同概念框架同等了解」。这不是造假（你确实懂这个概念），而是让你的真实能力**通过机器这一关**。

这个技巧对所有走「就业」路线的人都适用——别让你的能力死在关键词匹配上。

### 4. 35 岁危机：是真的，但 AI 是扩大选项，不是根治

V2EX 上有很多讨论，有个共识很扎心：

> 「大部分公司二三十万的职位，宁愿招 30 来岁水平一般的，也不想接大厂七八十万的大龄程序员。」

35 岁危机是真的。但要把它的定位摆对：**AI 转型对冲不了宏观经济，但能扩大你在任何周期下的可选项**。

应对思路有三条：

- **趁年轻多攒钱**——runway 是你最大的自由（第 5 篇讲过）；
- **提早转 AI / 全栈 / 独立**——别等到 35 才开始想退路；
- **往「判断型」能力挪**——执行型岗位受年龄歧视最重，判断型、复合型岗位（第 8 篇的护城河）抗得多。

顺带一个反焦虑的点：**AI 带来了知识平权**。你现在能低成本获得翻译（做海外产品）、UI 设计辅助、后端学习资源、素材生成——这在三年前都是要花钱花人脉的。**这是你补全栈、做独立产品的窗口期**，别浪费。

## 二、行动清单：按你选的活法定制

这是这一篇的核心交付。**不要抄别人的清单**——选了不同活法，你该做的事完全不同。对号入座：

| 你选的活法   | 本周（1 件）                                               | 本月（1 件）                    | 3 个月里程碑                     |
| ------------ | ---------------------------------------------------------- | ------------------------------- | -------------------------------- |
| **03 Agent** | 注册大模型 API，用 LangChain + Vercel AI SDK 跑通流式 demo | 做出「3 轮记忆 + 流式」的 Agent | 一个可演示的 Agent + GitHub 仓库 |
| **04 全栈**  | Next.js + Supabase 跑通鉴权 + 数据库                       | 做出带 AI 功能的全栈小应用      | 线上可访问的端到端产品           |
| **05 独立**  | 选 niche + 抢域名 + **算清自己的 runway**                  | 上线能收 Stripe 测试支付的 MVP  | 验证首笔真实支付 / MRR 起步      |
| **06 增效**  | 写 cursor.rules + 记录基线工时                             | 把一个高频任务量化提效          | 一套可量化、可复用的 AI 工作流   |
| **07 产品**  | 选一个真实痛点，写一份产品 brief                           | 用 AI + 前端做出可交互原型      | 能讲清「为谁做、差异在哪」的原型 |

每一条都对应了那篇里的「毕业项目」——**这不是学习任务，是交付任务**。完成了，你就有了能写进简历、能演示、能量化的真实成果。

## 三、通用基础：不管走哪条，这两件事都要做

无论你选哪条活法，有两件事是**所有人都该持续做的**，因为它们是第 8 篇讲的护城河：

1. **强化 TypeScript 类型编程**——它是你验证 AI 代码的核心武器（拦掉 94% 的编译错误）；
2. **守住传统工程基本功**——性能、稳定性、体验，挑一个深挖到「面试能讲深」的程度。

这两件事不性感，但它们决定了你**在 AI 时代是「被放大」还是「被暴露」**。

## 四、收尾：你现在该做什么

读完整整十篇，如果你只记住三句话，记住这三句：

1. **前端没死，死的是「只会写页面」的旧身份**——别把岗位跟能力画等号（第 2 篇）。
2. **AI 是放大器，不是替代者**——基本功越扎实，你被放得越高；基本功弱，你的盲区被放得越大（第 8 篇）。
3. **这一轮没有捷径，只有「选对方向 + 持续动手 + 守住基本功」**——五条活法任选一条深做，比八条都浅尝强（第 3-9 篇）。

如果你读完还是不知道选哪条——**回到第 1 篇，做那个 4 题自测**：

1. 能接受 12 个月零收入吗？能 → 倾向独立开发（05）；
2. 想留在现有公司吗？想 → AI 增效（06）；
3. 想写代码还是定义产品？产品 → 产品工程师（07）；
4. TS 够扎实吗？够 → Agent（03），还在补 → 全栈（04）。

选定了，就回到上面的行动清单，**这周就开始第一件**。

转型这件事，最难的不是「知道该怎么做」，而是「真的开始做」。希望这十篇能帮你把焦虑转化成一张具体的待办清单——剩下的，就交给你了。
