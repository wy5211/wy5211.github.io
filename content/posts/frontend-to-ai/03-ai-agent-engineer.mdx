上一篇我们算清了账：前端没死，死的是「只会写页面」的旧身份，红利属于「前端 × AI」的复合能力。那这种能力具体怎么拿到手？从这一篇开始，我们把五条活法一条条拆开。

先说共识最高的一条：**AI Agent 工程师**。

你大概搜过「怎么学 AI Agent」，结果满屏都是 Python 教程、PyTorch、Transformer 原理图——作为前端，你看了一眼就想关页面。这篇就是写给那个想关页面的你的：**做 Agent 产品，根本不需要从底层原理学起，而且前端在这个方向上，起点比大多数人好。**

## 一、为什么前端做 Agent 天然占优

先说个可能颠覆你认知的事：做 AI Agent，TypeScript 不只是「能用」，在很多场景下是**首选**。

Mastra 创始人（Gatsby 联创）Sam Bhagwat 透露，YC 最新一批（X25）大约 **60-70% 的 AI Agent 创业公司在用 TypeScript 而非 Python 构建**。（⚠️ 说句实话：这个数字来自一位正在做 TS Agent 框架的创始人，没有公开统计口径，当作趋势信号就好，别当精确数据。但趋势是真的——LangChain、LlamaIndex 都有官方 TS 版，Vercel AI SDK 更是为前端量身定做。）

更关键的是，你已经会的那些前端手艺，恰好是做 Agent UI 最需要的：

- **组件化 → Agent 编排**：把一个复杂 Agent 拆成「感知 → 决策 → 行动」几个模块，和你拆组件是一回事。
- **Redux / 状态管理 → 多轮对话记忆**：Agent 要记住前面聊过什么，本质就是个带时间维度的状态机——你处理过比这更恶心的状态。
- **流式渲染 → LLM UI**：大模型一个字一个字往外吐，你要做的就是流式渲染。前端干这个本来就是基础活，但很多后端转过来的人反而要重新学。

这三条迁移，是后端转 AI 反而比前端吃力的地方。别小看它。

## 二、别走错路：从底层原理学起，是最快的劝退方式

知乎上有套流传很广的「科班路线」：Python → 高数 → 机器学习 → 深度学习 → 训练模型。16 年入行的资深前端「前端欧阳」直接说：**这是劝退路线。**

> 「不该在连 hello world 都没写出来的时候去学底层原理。」

他的修正路线是这样的（也是我推荐的）：

1. **先注册一个大模型 API**（国内通义 / 智谱 / DeepSeek 都有免费额度，海外用 OpenAI / Claude），拿到 token；
2. **用 LangChain 或 Vercel AI SDK 写个最简单的 demo**——哪怕就是「调 API + 流式输出」，先跑通；
3. **复刻一个开源项目**（比如字节的 deer-flow），在别人的代码上改，比从零写快十倍；
4. **真遇到不懂的底层概念，再去补**——看李宏毅的台大课程，那时候你带着问题学，效率完全不一样。

记住这个顺序：**先动手，再补原理。** 等你做出第一个能用的 Agent，再回头看 Transformer 论文，你会发现「原来 attention 就是这么回事」，而不是一开始就被公式劝退。

想要一张完整地图的话，业内比较认同的是五层结构，周期大概 9-12 个月：

| 层级         | 内容                 | 典型工具                             |
| ------------ | -------------------- | ------------------------------------ |
| 1. 基础      | AI 基本概念、Prompt  | 微软 AI for Beginners、Karpathy 系列 |
| 2. LLM / RAG | 大模型调用、检索增强 | LangChain、LlamaIndex、DSPy          |
| 3. Agent     | 多步编排、工具调用   | LangGraph、CrewAI、AutoGen           |
| 4. 生产化    | 部署、限流、护栏     | vLLM、Guardrails                     |
| 5. 向量库    | 语义检索存储         | Qdrant、pgvector                     |

不用一口气学完，**先扎到第 2-3 层做出东西，第 4-5 层等真要上线了再补**。

## 三、前端的杀手锏：Skill 协议里的 UI Renderer

如果说前面那些是「前端能跟上」，那这点就是「**只有前端能做好**」。

现在做 Agent，有个越来越重要的概念叫 **Skill 协议**（你可以理解成「给大模型定义一个可调用的能力」）。一个完整的 Skill 通常由三部分组成：

- **Manifest**：用 JSON Schema 告诉大模型「我有什么能力、参数是什么」；
- **Executor**：真正执行这个能力的代码；
- **UI Renderer**：把能力的结果渲染成用户能看懂的界面——卡片、图表、表单、可交互组件。

前两部分后端工程师也能写。但 **UI Renderer 这一层，是后端转 AI 普遍做不好的**。一个只返回 JSON 的 Agent，和一个返回「漂亮卡片 + 可点击操作」的 Agent，用户体验天差地别——而后者正是你的主场。

所以做 Agent 产品，别只盯着「怎么调模型」，**把精力投到「怎么把 Agent 的输出做成好的 UI」上**，这是你能拉开差距的地方。

## 四、你想训模型，还是想在模型之上造东西？

Datasette 作者 Simon Willison 在 Hacker News 上有段话，我觉得是整个方向的定海神针：

> 「你想训模型，还是想在模型之上造东西？……真正有用的，是 prompting、evals 和把模型做成生产级应用的那一堆东西。」

翻译一下：别纠结自己不懂训练模型（那是算法工程师的战场，也不是前端该卷的）。**你的战场是把现成的大模型，做成稳定、好用、能解决真实问题的产品。** 这话听着虚，但它精准定义了「AI Engineer」这个方向——海外已经把它当独立岗位了，而它要的能力，和优秀前端高度重合。

## 五、冷水：把 Agent 当「能力」，别只盯着「专职岗位」

说完优势，必须泼盆冷水，否则你会对就业市场抱有不切实际的期待。

V2EX 上有个帖子叫《AI Agent 的工作也不好找》，说得很现实：Agent 专职岗对**学历和大厂背景**要求很高，JD 关键词还很死板（你不写 Dify / CrewAI 这种特定词，简历会被 HR 的机器直接筛掉）。

所以更务实的看法是：**把 Agent 当成一种「能力」，而不是一个「岗位」。** 纯 Agent 专职岗国内不算多，但 Agent **能力**能落地的地方到处都是——数字人、工作流自动化、ASR（语音识别应用）、AI coding 工具。把这套能力带到现有业务里，比死磕「Agent 工程师」这个 title 容易得多。

（怎么把简历关键词对齐 HR 的机器筛，第 10 篇会专门讲。）

## 六、毕业项目 + 这条路适合谁

读完这篇别光收藏。给自己定一个最小的「毕业项目」：

> **做一个「调用单一 LLM + 3 轮上下文记忆 + 流式输出」的 Agent。** 验收标准：能贴出 GitHub 仓库 + 一段可运行录屏。

听起来简单，但真做起来你会踩 token 管理、上下文截断、流式中断这些坑——踩过去，你就入门了。

**这条路适合谁**：TypeScript 扎实、想做技术深度的中高级前端。如果你 TS 还在补，可能先看下一篇的全栈方向更顺。**时间预期**：认真做，9-12 个月能达到「可上岗 / 可独立做 Agent 产品」。

下一篇换一个方向——不往 AI 应用层深钻，而是**横向扩边界**，把前端变成能独立交付完整产品的全栈。
