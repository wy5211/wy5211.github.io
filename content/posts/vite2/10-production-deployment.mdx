但代码写完只是完成了一半。你的应用要真正服务用户，还需要可靠地部署到生产环境。手动 `npm run build` 然后 `scp` 到服务器？在 2026 年，这已经不够了。

你需要：自动化的构建和测试流水线、Docker 容器化的部署方式、PR 预览环境、灰度发布和回滚机制、线上监控和告警。

## Docker 多阶段构建

Vite 项目部署的第一步是 Docker 化。Vite 8 的依赖安装和构建都在 Node.js 环境中完成，最终产物是静态文件，由 Nginx 提供服务。

### 为什么要用多阶段构建

一个"朴素"的 Dockerfile 可能把 Node.js 和 Nginx 都装在同一个镜像里。但 Node.js 的运行时环境（大约 350MB）在部署时完全不需要——生产环境只需要 Nginx 和构建产物。多阶段构建让 Docker 镜像只包含必要的内容。

```dockerfile
# TaskFlow Dashboard 的 Dockerfile

# ---- 阶段 1：安装依赖 ----
FROM node:20-alpine AS deps
WORKDIR /app

# 先复制 package 文件，利用 Docker 缓存层
COPY package.json package-lock.json ./
RUN npm ci --frozen-lockfile

# ---- 阶段 2：构建 ----
FROM node:20-alpine AS builder
WORKDIR /app

# 从 deps 阶段复制 node_modules
COPY --from=deps /app/node_modules ./node_modules
COPY . .

# 安装 Vite 8 的二进制依赖（Rolldown、Oxc、lightningcss）
RUN npm install

# 构建生产版本
RUN npm run build:prod

# ---- 阶段 3：运行 ----
FROM nginx:alpine AS runner
WORKDIR /usr/share/nginx/html

# 从 builder 阶段复制构建产物
COPY --from=builder /app/dist ./

# 复制 Nginx 配置
COPY nginx.conf /etc/nginx/conf.d/default.conf

# 添加非 root 用户
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup
USER appuser

EXPOSE 8080

CMD ["nginx", "-g", "daemon off;"]
```

这个 Dockerfile 有几个关键设计：

**缓存层优化**。`package.json` 和 `package-lock.json` 单独复制，先执行 `npm ci`。只要这两个文件没变，Docker 就会用缓存，跳过依赖安装这一步。对于 TaskFlow Dashboard 来说，依赖不变的情况下，这一步能节省大约 30 秒。

**`npm ci --frozen-lockfile`**。`ci` 命令严格按照 `package-lock.json` 安装，不会修改 lockfile。`--frozen-lockfile` 在 lockfile 和 package.json 不一致时报错退出。在 CI/CD 环境中，这是必须的——你不想因为 lockfile 和 package.json 不同步导致构建出不可预期的结果。

**非 root 用户运行**。Nginx 默认用 root 用户运行，这在生产环境是一个安全风险。用 `adduser` 创建一个普通用户，`USER appuser` 切换到这个用户。

**Nginx 配置**。Nginx 配置单独用一个文件，不用默认配置。

### Nginx 配置

```nginx
# nginx.conf
server {
    listen 8080;
    server_name _;
    root /usr/share/nginx/html;
    index index.html;

    # Gzip 压缩
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml text/javascript image/svg+xml;

    # 静态资源：长期缓存（Vite 生成的文件名带 hash）
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # HTML 文件：不缓存
    location ~* \.html$ {
        add_header Cache-Control "no-cache, no-store, must-revalidate";
        add_header Pragma "no-cache";
        add_header Expires "0";
    }

    # SPA 路由：所有未匹配的请求回退到 index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 安全头
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
}
```

配置要点：`try_files` 支持 SPA 的客户端路由。`/assets/` 下的文件设置了 1 年缓存——因为 Vite 构建的文件名带 content hash，内容不变 hash 就不变。`index.html` 不缓存，确保每次部署后用户能拿到最新的文件引用。

最终的 Docker 镜像大小约 25MB（Nginx Alpine + 构建产物）。如果用 Node.js 运行 SSR 服务，镜像会大一些（约 180MB），但比包含完整 node_modules 的镜像（约 500MB+）小得多。

## CI/CD 流水线

有了 Docker 镜像，接下来用 GitHub Actions 搭建自动化流水线。

### 基本流水线结构

```yaml
# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, staging]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ---- Job 1：代码检查和测试 ----
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci --frozen-lockfile

      - name: Lint
        run: npm run lint

      - name: Type check
        run: npx tsc --noEmit

      - name: Unit tests
        run: npm run test -- --coverage

      - name: Build check
        run: npm run build:prod
        env:
          VITE_API_BASE_URL: https://api.taskflow.dev
          VITE_SENTRY_DSN: ${{ secrets.PROD_SENTRY_DSN }}

  # ---- Job 2：构建 Docker 镜像并推送 ----
  build-and-push:
    needs: lint-and-test
    runs-on: ubuntu-latest
    if: github.event_name == 'push'
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Determine image tag
        id: tag
        run: |
          if [[ "${{ github.ref }}" == "refs/heads/main" ]]; then
            echo "tag=latest" >> $GITHUB_OUTPUT
            echo "environment=production" >> $GITHUB_OUTPUT
          elif [[ "${{ github.ref }}" == "refs/heads/staging" ]]; then
            echo "tag=staging-$(date +%Y%m%d-%H%M%S)" >> $GITHUB_OUTPUT
            echo "environment=staging" >> $GITHUB_OUTPUT
          fi

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ steps.tag.outputs.tag }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ---- Job 3：部署 ----
  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: ${{ needs.build-and-push.outputs.environment }}
    steps:
      - name: Deploy to server
        run: |
          echo "Deploying image ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ steps.tag.outputs.tag }}"
          # 实际部署命令（SSH 到服务器、kubectl apply 等）
```

流水线分三个 Job，用 `needs` 建立依赖关系：先检查代码（lint、类型检查、单元测试、构建验证），通过后再构建 Docker 镜像，最后部署。

**`cache-from: type=gha` 和 `cache-to: type=gha`**。这是 Docker BuildKit 的 GitHub Actions 缓存支持。每次构建的 Docker 层缓存会存储在 GitHub Actions 的缓存中，下次构建时复用。对于 TaskFlow Dashboard 来说，缓存命中时构建时间从 2 分钟降到 40 秒。

**`if: github.event_name == 'push'`**。build-and-push 只在 push 事件时执行，PR 不构建镜像（PR 只做检查和测试）。这能节省 CI 资源——一个活跃的项目一天可能有 20+ 个 PR。

### 构建产物分析

在 lint-and-test 的构建步骤之后，加上产物分析：

```yaml
- name: Build for analysis
  run: npm run build:prod
  env:
    VITE_API_BASE_URL: https://api.taskflow.dev
    VITE_SENTRY_DSN: placeholder

- name: Check bundle size
  run: |
    TOTAL_SIZE=$(du -sb dist/assets/js | cut -f1)
    MAX_SIZE=524288  # 512KB
    if [ "$TOTAL_SIZE" -gt "$MAX_SIZE" ]; then
      echo "ERROR: Total JS size ${TOTAL_SIZE} bytes exceeds ${MAX_SIZE} bytes"
      exit 1
    fi
    echo "Total JS size: ${TOTAL_SIZE} bytes (within ${MAX_SIZE} limit)"
```

这个检查确保首屏 JS 总量不超过 512KB（未压缩）。如果某个 PR 引入了一个大型依赖导致 bundle 膨胀，CI 会直接报错，阻止合并。

## PR 预览环境

开发过程中，产品经理和设计师需要看实际效果。不能每次都等部署到 staging——staging 是共享的，别人也在用。

PR 预览环境解决这个问题：每个 PR 自动部署到一个独立的临时环境，PR 关闭后自动销毁。

用 Vercel 或者 Cloudflare Pages 可以零配置实现这个功能。如果用自己的基础设施，可以用 GitHub Actions + Docker：

```yaml
# .github/workflows/preview.yml
name: PR Preview

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  preview:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install and build
        run: |
          npm ci --frozen-lockfile
          npm run build
        env:
          VITE_API_BASE_URL: ${{ secrets.STAGING_API_URL }}
          VITE_SENTRY_DSN:

      - name: Deploy preview
        run: |
          # 部署到临时环境，URL 包含 PR 编号
          PREVIEW_URL="https://preview-${{ github.event.number }}.taskflow.dev"
          echo "Preview deployed to: ${PREVIEW_URL}"

      - name: Comment preview URL on PR
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `Preview deployed: https://preview-${{ github.event.number }}.taskflow.dev`
            })
```

这个流水线会在每个 PR 上自动构建并部署到一个临时 URL，然后在 PR 下评论预览链接。团队成员点开链接就能看到这个 PR 的实际效果。

## 灰度发布和回滚

直接把新版本全量推给所有用户是有风险的。万一新版本有严重 bug，影响的是 100% 的用户。灰度发布让新版本先覆盖一小部分用户，确认没问题后再扩大范围。

### 基于权重的灰度

如果你用 Kubernetes，可以用 Deployment 的滚动更新来实现简单的灰度：

```yaml
# kubernetes/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: taskflow-dashboard
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1 # 最多多创建 1 个 Pod
      maxUnavailable: 0 # 不允许有 Pod 不可用
  selector:
    matchLabels:
      app: taskflow-dashboard
  template:
    metadata:
      labels:
        app: taskflow-dashboard
    spec:
      containers:
        - name: taskflow
          image: ghcr.io/wy5211/taskflow-dashboard:latest
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet:
              path: /
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
```

Kubernetes 的滚动更新策略是：先创建新版本的 Pod，等新 Pod 通过 readinessProbe 之后，再删除旧版本的 Pod。`maxSurge: 1` + `maxUnavailable: 0` 意味着更新过程中始终有足够的 Pod 在服务。

更精细的灰度（比如只让 10% 的流量到新版本）需要 Service Mesh（Istio）或者 Ingress Controller（Nginx Ingress 的 Canary 注解）。

### 回滚

回滚的核心原则是：**回滚要足够快，且不需要重新构建**。

Docker 镜像的 tag 策略支持这一点。每次部署不只打 `latest` 标签，还打一个包含 git commit hash 的标签：

```bash
# 构建时打两个标签
docker tag taskflow-dashboard:latest taskflow-dashboard:abc1234
docker tag taskflow-dashboard:latest taskflow-dashboard:latest
```

回滚只需要把 Deployment 的镜像改回上一个版本：

```bash
kubectl set image deployment/taskflow-dashboard \
  taskflow=ghcr.io/wy5211/taskflow-dashboard:abc1234
```

Kubernetes 会用旧的镜像重新创建 Pod，通常 30 秒内完成回滚。

如果不使用 Kubernetes，回滚方式取决于你的部署方式。如果是直接 SSH 到服务器用 Docker 运行：

```bash
# 回滚到上一个版本
docker stop taskflow-current
docker rm taskflow-current
docker run -d --name taskflow-current ghcr.io/wy5211/taskflow-dashboard:abc1234
```

关键是要保留历史版本的镜像，不要每次部署都删除旧镜像。

## 监控和告警

部署上线只是开始。你需要持续监控应用的运行状态，在问题影响到大量用户之前发现并修复。

### 错误监控

Sentry 是目前最流行的前端错误监控服务。Vite 项目集成 Sentry 很简单：

```typescript
// src/lib/sentry.ts
import * as Sentry from "@sentry/react";

export function initSentry() {
  const dsn = import.meta.env.VITE_SENTRY_DSN;
  if (!dsn) return; // 开发环境不初始化

  Sentry.init({
    dsn,
    environment: import.meta.env.VITE_SENTRY_ENVIRONMENT || "production",
    release: __APP_VERSION__, // 构建时注入的版本号

    // 设置采样率（不要 100% 采样，太贵了）
    tracesSampleRate: 0.1, // 10% 的事务追踪
    replaysSessionSampleRate: 0, // 不自动录制回放
    replaysOnErrorSampleRate: 0.5, // 错误时 50% 概率录制回放

    // 过滤掉一些不重要的错误
    ignoreErrors: [
      "ResizeObserver loop limit exceeded",
      "Non-Error promise rejection captured",
      "NetworkError",
    ],

    // 在错误事件中附加上下文信息
    beforeSend(event) {
      if (event.user) {
        // 不要发送用户 IP
        delete event.user.ip_address;
      }
      return event;
    },
  });
}
```

在 `vite.config.ts` 中注入版本号：

```typescript
// vite.config.ts
import { version } from "./package.json";

export default defineConfig({
  define: {
    __APP_VERSION__: JSON.stringify(version),
  },
});
```

### 性能监控

上一篇提到的 Web Vitals 上报，配合 Sentry 的 Performance Monitoring，可以看到每个版本的 LCP、FID、CLS 的 P75 值。

在 Sentry 里设置告警规则：当 LCP 的 P75 超过 3 秒时发送 Slack 通知。这样当某个版本引入了性能退化，团队能第一时间收到告警。

### 健康检查

Nginx 配置中加一个健康检查端点：

```nginx
# nginx.conf
location /health {
    access_log off;
    return 200 'ok';
    add_header Content-Type text/plain;
}
```

部署平台（Kubernetes、Docker Compose、负载均衡器）可以定期请求这个端点来判断服务是否健康。

## 从 Webpack 迁移到 Vite 8：完整回顾

10 篇文章，从决策到上线。回顾一下 TaskFlow Dashboard 的迁移过程和实际收益。

### 迁移做了什么

1. **决策评估**（第 1 篇）：分析了 Webpack 的痛点，评估了 Vite 8 的能力和迁移成本，决定迁移。
2. **开发服务器**（第 2-3 篇）：理解了 Vite 的预构建和 HMR 机制，配置了开发环境。
3. **项目迁移**（第 4-5 篇）：处理了别名、代理、CSS、静态资源等配置的迁移。
4. **插件迁移**（第 6 篇）：替换了 Webpack 特有的 loader 和 plugin。
5. **SSR 支持**（第 7 篇）：用 Vite 的 SSR API 实现了服务端渲染。
6. **环境变量**（第 8 篇）：配置了 dev/staging/production 三个环境。
7. **性能优化**（第 9 篇）：Lighthouse 分数从 65 提到 96。
8. **生产部署**（本篇）：Docker 化、CI/CD 流水线、灰度发布。

### 实际收益

| 指标                   | Webpack 5 | Vite 8     |
| ---------------------- | --------- | ---------- |
| 开发服务器启动         | 18s       | 0.8s       |
| HMR 速度               | 3s        | 不足 100ms |
| 生产构建               | 4m12s     | 28s        |
| 配置文件数量           | 6 个      | 1 个       |
| CI/CD 总耗时           | 8min      | 3min       |
| Lighthouse Performance | 72        | 96         |
| Docker 镜像大小        | 180MB     | 25MB       |

### 付出的代价

- 迁移花了 2 周时间（1 周迁移 + 1 周测试验证）
- 团队学习 Vite 配置方式的时间（约 2-3 天）
- 安装体积增加约 15MB（Rolldown + Oxc + lightningcss 的二进制文件）
- 部分自定义 Webpack 插件需要重写为 Vite 插件
- SSR 的 Environment API 还是实验性的，未来可能有 breaking change

整体来看，这些代价是值得的。开发体验的改善直接提升了团队的开发效率——每天节省在等待构建上的时间大约 40 分钟（按 3 个前端开发者计算）。CI/CD 耗时从 8 分钟降到 3 分钟，一天的 20 次 commit 能节省将近 2 小时的等待时间。

## 权衡

部署方案的选择取决于团队规模和基础设施。

Docker 多阶段构建在所有场景下都是推荐的做法——镜像小、构建可复现、部署一致。唯一可能不用 Docker 的情况是纯静态站点（没有 SSR），可以直接用 CDN 托管（Vercel、Cloudflare Pages、Netlify），连 Docker 都不需要。

GitHub Actions 是最方便的 CI/CD 平台（对于 GitHub 上的项目），免费额度足够小团队使用。但如果你的代码在 GitLab 上，GitLab CI 也完全可以。CI/CD 工具的选择不是技术决策，而是团队习惯决策。

PR 预览环境对于前端团队来说价值很高——产品经理可以在开发过程中就看到效果，而不是等部署到 staging。但如果你的团队没有这个需求（比如后端为主的团队），PR 预览环境的维护成本可能不值得。

灰度发布的复杂度取决于你的部署架构。如果用的是 Vercel/Cloudflare Pages 这类平台，灰度发布是内置功能。如果自己用 Kubernetes 管理，灰度需要额外的 Service Mesh 或者 Ingress 配置。对于 TaskFlow Dashboard 的规模（日均 PV 约 5 万），滚动更新已经够用了——新版本上线有问题，30 秒内回滚。

这就是 Vite 实战指南系列的最后一篇。从为什么要迁移，到怎么迁移，再到怎么优化和部署——希望这 10 篇文章能帮你在自己的项目中做出更好的决策。
