# GitOps 实战：ArgoCD 实现声明式持续部署

在云原生时代，如何高效、可靠地管理 Kubernetes 应用的部署成为了运维团队的核心挑战。传统的 CI/CD 流程虽然能完成自动化部署，但往往缺乏对集群状态的持续监控和自愈能力。

**GitOps** 作为一种现代化的运维方法论，通过将 Git 作为单一事实来源（Single Source of Truth），实现了基础设施和应用的声明式管理。而 **ArgoCD** 作为 Kubernetes 原生的 GitOps 持续交付工具，完美践行了这一理念。

本文将带你深入理解 GitOps，并通过实战掌握 ArgoCD 的核心用法。

## 什么是 GitOps？

GitOps 是由 Weaveworks 在 2017 年提出的一种运维方法论，其核心理念是：

**Git 仓库中的声明式配置文件是系统的"期望状态"，集群会自动同步并维持这个状态。**

### GitOps 的四大原则

1. **声明式基础设施** - 所有配置（应用、基础设施、环境）都用声明式方式描述（如 YAML）
2. **版本化与不可变** - 所有配置存储在 Git 仓库中，具有完整的版本历史
3. **自动拉取** - 集群自动从 Git 拉取配置变更（而不是被动接收推送）
4. **持续调和** - 系统持续监控实际状态与期望状态的差异，并自动修复

### GitOps vs 传统 CI/CD

| 维度     | 传统 CI/CD         | GitOps                   |
| -------- | ------------------ | ------------------------ |
| 触发方式 | Push（CI 推送）    | Pull（集群拉取）         |
| 状态存储 | 分散在 CI 系统中   | Git 仓库统一管理         |
| 访问权限 | CI 需要 kubeconfig | 集群内运行，无需外部凭证 |
| 回滚能力 | 依赖 CI 历史       | Git revert 即可          |
| 审计能力 | CI 日志            | Git commit history       |
| 漂移检测 | 通常无             | 持续监控并自愈           |

## 为什么选择 ArgoCD？

ArgoCD 是 CNCF 毕业项目，专为 Kubernetes 设计的 GitOps 持续交付工具。它具有以下优势：

### 核心特性

- **声明式设置** - ArgoCD 本身也用 Kubernetes 资源配置
- **自动化部署** - 自动检测 Git 变更并同步到集群
- **多集群支持** - 一个 ArgoCD 可管理多个 Kubernetes 集群
- **多租户** - 支持 RBAC，适合团队协作
- **可视化界面** - 提供直观的应用拓扑和状态展示
- **健康检查** - 自动评估应用健康状态
- **回滚能力** - 一键回滚到任意历史版本
- **Sync Hooks** - 支持部署前后的自定义操作

## ArgoCD 安装与配置

### 前置要求

- Kubernetes 集群（v1.16+）
- kubectl 已配置好 kubeconfig
- Git 仓库（存放 Kubernetes 配置文件）

### 安装 ArgoCD

**1. 创建命名空间并安装**

```bash
# 创建命名空间
kubectl create namespace argocd

# 安装 ArgoCD（生产环境建议指定版本）
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

**2. 等待所有 Pod 就绪**

```bash
kubectl get pods -n argocd -w

# 期望输出
NAME                                                READY   STATUS    RESTARTS   AGE
argocd-application-controller-0                     1/1     Running   0          2m
argocd-applicationset-controller-xxx               1/1     Running   0          2m
argocd-dex-server-xxx                               1/1     Running   0          2m
argocd-redis-xxx                                    1/1     Running   0          2m
argocd-repo-server-xxx                              1/1     Running   0          2m
argocd-server-xxx                                   1/1     Running   0          2m
```

**3. 访问 ArgoCD UI**

```bash
# 方式一：Port-Forward（适合开发环境）
kubectl port-forward svc/argocd-server -n argocd 8080:443

# 访问 https://localhost:8080

# 方式二：通过 Ingress 暴露（生产环境推荐）
# 创建 Ingress 资源（见下文）
```

**4. 获取初始密码**

```bash
# 初始用户名为 admin
# 密码存储在 secret 中
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# 输出类似：Xy7kM9pL2qR4
```

**5. 安装 ArgoCD CLI（可选）**

```bash
# macOS
brew install argocd

# Linux
curl -sSL -o argocd-linux-amd64 https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
sudo install -m 555 argocd-linux-amd64 /usr/local/bin/argocd
rm argocd-linux-amd64

# 登录
argocd login localhost:8080
```

## 实战：部署第一个应用

### 准备 Git 仓库

创建一个 Git 仓库，存放 Kubernetes 配置文件。这里以一个简单的 Nginx 应用为例：

**目录结构**

```
gitops-demo/
├── apps/
│   └── nginx/
│       ├── deployment.yaml
│       └── service.yaml
└── README.md
```

**deployment.yaml**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-demo
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 200m
              memory: 256Mi
```

**service.yaml**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-demo-svc
spec:
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
  type: ClusterIP
```

推送到 GitHub：

```bash
git init
git add .
git commit -m "Add nginx demo app"
git branch -M main
git remote add origin https://github.com/YOUR_USERNAME/gitops-demo.git
git push -u origin main
```

### 创建 ArgoCD 应用

**方式一：通过 UI 创建**

1. 登录 ArgoCD UI（https://localhost:8080）
2. 点击 **New App**
3. 填写配置：
   - **Application Name**: nginx-demo
   - **Project**: default
   - **Sync Policy**: Manual（或 Automatic）
   - **Repository URL**: https://github.com/YOUR_USERNAME/gitops-demo.git
   - **Revision**: HEAD
   - **Path**: apps/nginx
   - **Cluster**: https://kubernetes.default.svc（当前集群）
   - **Namespace**: default

4. 点击 **Create**

**方式二：通过 CLI 创建**

```bash
argocd app create nginx-demo \
  --repo https://github.com/YOUR_USERNAME/gitops-demo.git \
  --path apps/nginx \
  --dest-server https://kubernetes.default.svc \
  --dest-namespace default
```

**方式三：通过 Kubernetes 资源创建（推荐）**

创建 `application.yaml`：

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: nginx-demo
  namespace: argocd
spec:
  project: default

  source:
    repoURL: https://github.com/YOUR_USERNAME/gitops-demo.git
    targetRevision: HEAD
    path: apps/nginx

  destination:
    server: https://kubernetes.default.svc
    namespace: default

  syncPolicy:
    automated:
      prune: true # 自动删除不在 Git 中的资源
      selfHeal: true # 自动修复手动修改
    syncOptions:
      - CreateNamespace=true
```

应用配置：

```bash
kubectl apply -f application.yaml
```

### 同步应用

应用创建后，状态为 **OutOfSync**（因为 Git 中的配置还未应用到集群）。

**手动同步**

```bash
# CLI 方式
argocd app sync nginx-demo

# 或在 UI 中点击 Sync 按钮
```

**自动同步**

如果配置了 `syncPolicy.automated`，ArgoCD 会自动检测 Git 变更并同步。

### 验证部署

```bash
# 查看应用状态
argocd app get nginx-demo

# 查看 Kubernetes 资源
kubectl get all -l app=nginx

# 期望输出

pod/nginx-demo-xxx-yyy            1/1     Running   0          1m

```

## 核心概念详解

### Application（应用）

Application 是 ArgoCD 的核心资源，定义了：

- **Source**：Git 仓库地址、分支、路径
- **Destination**：目标集群和命名空间
- **SyncPolicy**：同步策略（手动/自动）

**示例：带完整配置的 Application**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io # 删除应用时级联删除资源
spec:
  project: default

  source:
    repoURL: https://github.com/myorg/myapp.git
    targetRevision: main
    path: k8s/overlays/production

    # 使用 Kustomize
    kustomize:
      namePrefix: prod-

    # 或使用 Helm
    # helm:
    #   valueFiles:
    #   - values-prod.yaml
    #   parameters:
    #   - name: image.tag
    #     value: v1.2.3

  destination:
    server: https://kubernetes.default.svc
    namespace: production

  # 忽略某些字段的差异（如 replica count 由 HPA 管理）
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas

  # 同步策略
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
      allowEmpty: false

    syncOptions:
      - CreateNamespace=true
      - PrunePropagationPolicy=foreground
      - PruneLast=true

    # 重试策略
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m

  # 健康检查超时
  revisionHistoryLimit: 10
```

### Project（项目）

Project 用于多租户场景，限制应用可以访问的资源：

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: team-a
  namespace: argocd
spec:
  description: Team A's project

  # 允许的源仓库
  sourceRepos:
    - "https://github.com/myorg/team-a/*"

  # 允许的目标集群
  destinations:
    - namespace: "team-a-*"
      server: https://kubernetes.default.svc
    - namespace: "production"
      server: https://kubernetes.default.svc

  # 允许的资源类型
  clusterResourceWhitelist:
    - group: ""
      kind: Namespace
    - group: "rbac.authorization.k8s.io"
      kind: ClusterRole
    - group: "rbac.authorization.k8s.io"
      kind: ClusterRoleBinding

  # 禁止的资源类型
  namespaceResourceBlacklist:
    - group: ""
      kind: ResourceQuota
    - group: "networking.k8s.io"
      kind: NetworkPolicy

  # 同步窗口（限制何时可以同步）
  syncWindows:
    - kind: allow
      schedule: "10 1 * * *" # 每天 1:10 UTC 允许同步
      duration: 1h
      applications:
        - "*-prod"
```

### Sync Status（同步状态）

ArgoCD 提供丰富的状态信息：

| 状态          | 含义                       |
| ------------- | -------------------------- |
| **Synced**    | 集群状态与 Git 一致        |
| **OutOfSync** | Git 有更新未应用到集群     |
| **Unknown**   | 无法获取状态（网络问题等） |
| **Degraded**  | 应用不健康（Pod 失败等）   |

### Health Status（健康状态）

ArgoCD 自动评估应用健康：

| 状态            | 含义                     |
| --------------- | ------------------------ |
| **Healthy**     | 所有资源正常运行         |
| **Progressing** | 正在部署中               |
| **Degraded**    | 部分资源失败             |
| **Suspended**   | 资源被暂停（如 CronJob） |
| **Missing**     | 资源不存在               |

## 高级特性

### 1. 多集群管理

ArgoCD 可以管理多个 Kubernetes 集群。

**注册外部集群**

```bash
# 获取外部集群的 kubeconfig context 名称
kubectl config get-contexts -o name

# 注册到 ArgoCD
argocd cluster add external-cluster-context

# 验证
argocd cluster list
```

**在 Application 中指定集群**

```yaml
spec:
  destination:
    server: https://external-cluster-api-server:6443
    namespace: production
```

### 2. Helm 支持

ArgoCD 原生支持 Helm Chart。

**示例：从 Helm 仓库部署**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: redis
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://charts.bitnami.com/bitnami
    chart: redis
    targetRevision: 17.0.0
    helm:
      valuesObject:
        auth:
          enabled: false
        master:
          persistence:
            enabled: false
  destination:
    server: https://kubernetes.default.svc
    namespace: redis
```

**从 Git 仓库中的 Chart 部署**

```yaml
spec:
  source:
    repoURL: https://github.com/myorg/charts.git
    path: charts/mychart
    helm:
      valueFiles:
        - values-prod.yaml
      parameters:
        - name: image.tag
          value: v1.2.3
```

### 3. Kustomize 支持

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/myapp.git
    path: k8s/overlays/production
    kustomize:
      namePrefix: prod-
      nameSuffix: -v1
      images:
        - nginx=nginx:1.25
      commonLabels:
        env: production

    server: https://kubernetes.default.svc
    namespace: production
```

### 4. Sync Hooks（同步钩子）

在同步过程的不同阶段执行自定义操作：

```yaml
# job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  generateName: db-migrate-
  annotations:
    argocd.argoproj.io/hook: PreSync # 同步前执行
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  template:
    spec:
      containers:
        - name: migrate
          image: myapp:latest
          command: ["npm", "run", "migrate"]
      restartPolicy: Never
  backoffLimit: 2
```

**Hook 类型**

- `PreSync`：同步前执行（如数据库迁移）
- `Sync`：同步时执行（与主资源一起）
- `PostSync`：同步成功后执行（如通知）
- `SyncFail`：同步失败时执行（如告警）

**Hook 删除策略**

- `HookSucceeded`：成功后删除
- `HookFailed`：失败后删除
- `BeforeHookCreation`：新 Hook 创建前删除旧的

### 5. ApplicationSet（批量创建应用）

ApplicationSet 适合多环境、多集群场景，可以用模板批量创建 Application。

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: guestbook
  namespace: argocd
spec:
  generators:
    # List Generator：从列表生成
    - list:
        elements:
          - cluster: dev
            url: https://dev-cluster:6443
          - cluster: staging
            url: https://staging-cluster:6443
          - cluster: prod
            url: https://prod-cluster:6443

  template:
    metadata:
      name: "{{cluster}}-guestbook"
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/guestbook.git
        targetRevision: HEAD
        path: k8s/{{cluster}}
      destination:
        server: "{{url}}"
        namespace: guestbook
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
```

**其他 Generator**

- **Git Directory Generator**：根据 Git 目录结构生成
- **Cluster Generator**：根据注册的集群自动生成
- **Matrix Generator**：组合多个 Generator

### 6. Config Management Plugins（自定义配置管理）

如果 Helm、Kustomize 都不满足需求，可以自定义插件：

```yaml
# argocd-cm ConfigMap
apiVersion: v1

metadata:
  name: argocd-cm
  namespace: argocd

    - name: my-plugin
```

## GitOps 工作流最佳实践

### 1. 仓库结构设计

**单仓多目录（Monorepo）**

```
infrastructure/
├── apps/
│   ├── frontend/
│   ├── backend/
│   └── worker/
├── infrastructure/
│   ├── terraform/
│   └── k8s/
└── environments/
    ├── dev/
    ├── staging/
    └── production/
```

**多仓（Multi-repo）**

```
# 应用代码仓库
app-code/
├── src/
├── Dockerfile
└── .github/workflows/build.yaml

# 配置仓库
app-config/
├── dev/
├── staging/
└── production/
```

### 2. 分支策略

**推荐：基于环境的分支**

```
main (生产)
├── staging
└── dev
```

ArgoCD 配置：

```yaml
# Dev 环境
spec:
  source:
    targetRevision: dev

# Staging 环境
spec:
  source:
    targetRevision: staging

# Production 环境
spec:
  source:
    targetRevision: main
```

### 3. CI/CD 流程设计

**完整流程**

```
代码提交 → CI 构建 → 推送镜像 → 更新配置仓库 → ArgoCD 自动同步
```

**GitHub Actions 示例**

```yaml
# .github/workflows/ci.yaml
name: CI/CD

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Image
        run: docker build -t myapp:${{ github.sha }} .

      - name: Push to Registry
        run: |
          echo ${{ secrets.REGISTRY_PASSWORD }} | docker login -u ${{ secrets.REGISTRY_USER }} --password-stdin
          docker push myapp:${{ github.sha }}

      - name: Update GitOps Repo
        run: |
          git clone https://x-access-token:${{ secrets.GITOPS_TOKEN }}@github.com/myorg/app-config.git
          cd app-config
          # 更新镜像 tag
          kustomize edit set image myapp:latest=myapp:${{ github.sha }}
          git config user.name "CI Bot"
          git config user.email "ci@example.com"
          git add .
          git commit -m "Update image to ${{ github.sha }}"
          git push
```

### 4. RBAC 权限管理

**配置 RBAC**

```yaml
# argocd-rbac-cm
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: argocd
data:
  policy.csv: |
    # 给 team-a 组只读权限
    p, role:team-a-readonly, applications, get, */*, allow
    p, role:team-a-readonly, applications, sync, team-a/*, allow

    # 给 team-a 组写入权限
    p, role:team-a-admin, applications, *, team-a/*, allow
    p, role:team-a-admin, clusters, get, *, allow

    # 分配角色给组
    g, myorg:team-a, role:team-a-admin
    g, myorg:team-a-viewers, role:team-a-readonly

  policy.default: role:readonly
```

### 5. 通知集成

ArgoCD 支持在同步完成、失败等事件时发送通知。

**配置 Slack 通知**

```yaml
# argocd-notifications-cm
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-notifications-cm
  namespace: argocd
data:
  service.slack: |
    token: $slack-token

  template.app-sync-succeeded: |
    message: |
      Application {{.app.metadata.name}} sync succeeded!
      Commit: {{.app.status.sync.revision}}

  trigger.on-sync-succeeded: |
    - description: Application sync succeeded
      send:
      - app-sync-succeeded
      when: app.status.operationState.phase in ['Succeeded']
```

**在 Application 上启用通知**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  annotations:
    notifications.argoproj.io/subscribe.on-sync-succeeded.slack: my-channel
```

## 故障排查指南

### 常见问题

**1. 应用一直处于 Progressing 状态**

```bash
# 查看详细事件
kubectl describe application my-app -n argocd

# 检查 Pod 状态
kubectl get pods -n my-namespace -w

# 可能原因：
# - Readiness Probe 失败
# - ImagePullBackOff
# - 资源不足
```

**2. Sync 失败**

```bash
# 查看同步日志
argocd app logs my-app

# 常见错误：
# - RBAC 权限不足
# - 资源已存在（不在 Git 中）
# - 配置语法错误
```

**3. OutOfSync 无法自动修复**

```bash
# 检查是否启用了 selfHeal
argocd app get my-app --refresh

# 手动同步
argocd app sync my-app

# 强制替换资源
argocd app sync my-app --force
```

**4. Image Pull 错误**

```bash
# 检查镜像仓库凭证
kubectl get secrets -n my-namespace

# 创建 Docker Registry Secret
kubectl create secret docker-registry regcred \
  --docker-server=https://index.docker.io/v1/ \
  --docker-username=myuser \
  --docker-password=mypassword \
  -n my-namespace
```

### 调试技巧

```bash
# 查看 ArgoCD 日志
kubectl logs -n argocd deployment/argocd-application-controller

# 查看 Repo Server 日志
kubectl logs -n argocd deployment/argocd-repo-server

# 手动测试 Git 访问
argocd repo list
argocd repo add https://github.com/myorg/myrepo.git --username myuser --password mypass

# 测试清单生成
argocd app manifests my-app
```

## 生产环境建议

### 1. 高可用部署

```bash
# 使用 HA 模式安装
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/ha/install.yaml
```

**资源规划**

- **argocd-server**: 2+ 副本，每个 1 CPU / 256Mi
- **argocd-repo-server**: 2+ 副本，每个 1 CPU / 512Mi
- **argocd-application-controller**: 1 副本，2+ CPU / 2Gi（根据应用数量调整）
- **Redis**: HA 模式（3 副本）

### 2. 备份与恢复

```bash
# 备份所有 Application
kubectl get applications -n argocd -o yaml > argocd-apps-backup.yaml

# 备份 ArgoCD 配置
kubectl get cm,secret -n argocd -o yaml > argocd-config-backup.yaml

# 恢复
kubectl apply -f argocd-apps-backup.yaml
```

### 3. 监控 ArgoCD 本身

```yaml
# 启用 Metrics
# argocd-cm
data:
  server.metrics.enabled: "true"
  controller.metrics.enabled: "true"
  repo-server.metrics.enabled: "true"

# ServiceMonitor（如果使用 Prometheus Operator）
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: argocd
  namespace: argocd
spec:
  selector:
    matchLabels:
      app.kubernetes.io/name: argocd-metrics
  endpoints:
    - port: metrics
```

### 4. 安全加固

- **禁用 admin 用户**，使用 SSO（OIDC）
- **启用 RBAC**，按团队隔离权限
- **使用 SSH 密钥** 访问 Git 仓库
- **限制集群资源**，通过 AppProject 白名单
- **定期更新** ArgoCD 版本

```yaml
# argocd-cm
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: argocd
data:
  # 禁用 admin
  admin.enabled: "false"

  # OIDC 配置
  url: https://argocd.example.com
  oidc.config: |
    name: Okta
    issuer: https://dev-xxx.okta.com
    clientID: xxx
    clientSecret: $oidc.okta.clientSecret
```

## 总结

GitOps + ArgoCD 为 Kubernetes 应用交付带来了革命性的改变：

**核心优势**

1. **声明式管理** - Git 作为唯一事实来源，配置即代码
2. **自动化部署** - 无需手动触发，变更自动同步
3. **可追溯性** - 完整的 Git 历史，方便审计和回滚
4. **自愈能力** - 自动检测并修复漂移
5. **安全性** - 无需外部 CI 系统的集群凭证

**适用场景**

- 微服务架构的持续交付
- 多环境、多集群管理
- 需要严格审计的生产环境
- 追求高可用的关键业务

**学习路径**

1. 从单应用、单集群开始
2. 掌握 Helm / Kustomize 配置管理
3. 实践多环境管理（ApplicationSet）
4. 集成 CI 流程（镜像构建 + 配置更新）
5. 配置 RBAC 和通知
6. 生产环境高可用部署

GitOps 不仅仅是一套工具，更是一种运维理念的转变。ArgoCD 作为这一理念的优秀实践者，值得每个 Kubernetes 用户深入学习和应用。
