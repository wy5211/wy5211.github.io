## 为什么需要包管理？

在开发任务队列 API 时，你遇到了一个实际问题：如何组织代码？代码越来越多，都放在一个文件里显然不行。你还需要使用第三方库（如 Gin 框架），如何管理这些依赖？

Go 早期使用 GOPATH 模式，所有代码必须放在 `$GOPATH/src` 下，依赖管理很原始。Go 1.11 引入了 Modules，彻底改变了这个问题。现在你可以：

- 在任意目录创建项目
- 精确管理依赖版本
- 轻松发布和分享代码

## Go Modules 基础

### 初始化模块

创建一个新项目很简单：

```bash
# 创建项目目录
mkdir taskqueue-api
cd taskqueue-api

# 初始化 Go module
go mod init github.com/yourusername/taskqueue-api
```

这会创建 `go.mod` 文件：

```go
module github.com/yourusername/taskqueue-api

go 1.22
```

**为什么模块路径很重要？**

- 它是项目的唯一标识
- 通常使用 GitHub 仓库路径
- 其他项目通过这个路径导入你的代码

### 添加依赖

当你导入第三方包并运行 `go build` 或 `go mod tidy` 时，Go 会自动添加依赖：

```go
// main.go
package main

import "github.com/gin-gonic/gin"

func main() {
    r := gin.Default()
    r.GET("/", func(c *gin.Context) {
        c.String(200, "Hello, World!")
    })
    r.Run(":8080")
}
```

运行命令：

```bash
go mod tidy
```

`go.mod` 会自动更新：

```go
module github.com/yourusername/taskqueue-api

go 1.22

require github.com/gin-gonic/gin v1.9.1
```

同时会生成 `go.sum` 文件，记录依赖的校验和，确保依赖完整性。

### 依赖版本管理

Go 使用语义化版本（Semantic Versioning）：

```bash
# 获取最新版本
go get github.com/gin-gonic/gin@latest

# 获取特定版本
go get github.com/gin-gonic/gin@v1.9.0

# 获取某个分支
go get github.com/gin-gonic/gin@main

# 更新所有依赖到最新小版本
go get -u ./...
go mod tidy
```

**版本号含义**：

- `v1.9.1`：主版本.次版本.修订号
- 主版本变化表示不兼容的 API 变更
- 次版本变化表示向后兼容的功能新增
- 修订号变化表示向后兼容的问题修复

### 查看依赖关系

```bash
# 查看直接依赖
go list -m all

# 查看依赖图
go mod graph

# 查看为什么需要某个包
go mod why github.com/gin-gonic/gin
```

### 定义包

每个目录对应一个包，包名通常与目录名相同：

```go
// models/task.go
package models

import "time"

// Task 表示一个任务
// 注意：只有大写开头的标识符才能被其他包访问
type Task struct {
    ID          int       `json:"id"`
    Title       string    `json:"title"`
    Description string    `json:"description"`
    Status      string    `json:"status"`
    CreatedAt   time.Time `json:"created_at"`
    UpdatedAt   time.Time `json:"updated_at"`
}

// TaskStatus 定义任务状态常量
const (
    StatusPending   = "pending"
    StatusRunning   = "running"
    StatusCompleted = "completed"
    StatusFailed    = "failed"
)

// NewTask 创建新任务
func NewTask(title, description string) *Task {
    return &Task{
        Title:       title,
        Description: description,
        Status:      StatusPending,
        CreatedAt:   time.Now(),
        UpdatedAt:   time.Now(),
    }
}

// 可以添加更多方法
func (t *Task) Complete() {
    t.Status = StatusCompleted
    t.UpdatedAt = time.Now()
}

func (t *Task) Fail() {
    t.Status = StatusFailed
    t.UpdatedAt = time.Now()
}
```

```go
// models/worker.go
package models

import (
    "fmt"
    "time"
)

// Worker 表示工作线程
type Worker struct {
    ID        int
    TaskChan  chan *Task
    QuitChan  chan bool
}

// NewWorker 创建新 worker
func NewWorker(id int) *Worker {
    return &Worker{
        ID:       id,
        TaskChan: make(chan *Task),
        QuitChan: make(chan bool),
    }
}

// Start 启动 worker
func (w *Worker) Start() {
    go func() {
        for {
            select {
            case task := <-w.TaskChan:
                w.processTask(task)
            case <-w.QuitChan:
                fmt.Printf("Worker %d 退出\n", w.ID)
                return
            }
        }
    }()
}

// Stop 停止 worker
func (w *Worker) Stop() {
    close(w.QuitChan)
}

// processTask 处理任务（私有方法，小写开头）
func (w *Worker) processTask(task *Task) {
    fmt.Printf("Worker %d 开始处理任务 %d\n", w.ID, task.ID)
    task.Status = StatusRunning

    // 模拟处理
    time.Sleep(100 * time.Millisecond)

    task.Complete()
    fmt.Printf("Worker %d 完成任务 %d\n", w.ID, task.ID)
}
```

### 使用包

```go
// services/queue.go
package services

import (
    "sync"

    "github.com/yourusername/taskqueue-api/models"
)

// Queue 任务队列
type Queue struct {
    workers  []*models.Worker
    taskChan chan *models.Task
    wg       sync.WaitGroup
}

// NewQueue 创建新队列
func NewQueue(numWorkers int) *Queue {
    q := &Queue{
        workers:  make([]*models.Worker, numWorkers),
        taskChan: make(chan *models.Task, 100),
    }

    // 初始化 workers
    for i := 0; i < numWorkers; i++ {
        worker := models.NewWorker(i + 1)
        q.workers[i] = worker
        worker.Start()
    }

    return q
}

// Enqueue 添加任务到队列
func (q *Queue) Enqueue(task *models.Task) {
    q.taskChan <- task
}

// Start 启动队列处理
func (q *Queue) Start() {
    q.wg.Add(1)
    go func() {
        defer q.wg.Done()
        for task := range q.taskChan {
            // 分发任务给空闲的 worker（简化版，轮询）
            workerIdx := task.ID % len(q.workers)
            q.workers[workerIdx].TaskChan <- task
        }
    }()
}

// Stop 停止队列
func (q *Queue) Stop() {
    close(q.taskChan)
    q.wg.Wait()

    // 停止所有 workers
    for _, worker := range q.workers {
        worker.Stop()
    }
}
```

```go
// main.go
package main

import (
    "fmt"

    "github.com/yourusername/taskqueue-api/models"
```

## 包的可见性

Go 的可见性规则很简单：标识符首字母大写表示公开（public），小写表示私有（private）。

```go
package mypackage

// PublicVar 可以被其他包访问
var PublicVar int = 42

// privateVar 只能在当前包内访问
var privateVar int = 10

// PublicFunction 可以被其他包调用
func PublicFunction() {
    fmt.Println(privateVar)  // 可以访问私有变量
}

// privateFunction 只能在当前包内调用
func privateFunction() {
    fmt.Println("私有函数")
}

// PublicStruct 可以被其他包使用
type PublicStruct struct {
    PublicField  string  // 公开字段
    privateField string  // 私有字段
}

// 公开结构的私有字段仍然不能直接访问
```

使用示例：

```go
package main

import "github.com/yourusername/taskqueue-api/mypackage"

func main() {
    // 可以访问公开的标识符
    fmt.Println(mypackage.PublicVar)
    mypackage.PublicFunction()

    obj := mypackage.PublicStruct{
        PublicField: "公开",
        // privateField: "私有"  // 编译错误：不能访问私有字段
    }

    // 以下都会编译错误
    // fmt.Println(mypackage.privateVar)
    // mypackage.privateFunction()
}
```

## 导入包的高级用法

### 别名导入

当包名冲突或为了简洁时，可以使用别名：

```go
import (
    stdfmt "fmt"
    myfmt "myproject/utils/fmt"

    "github.com/gin-gonic/gin"
    ginhelper "myproject/gin"
)
```

### 点导入

点导入会将包的导出标识符引入当前作用域，通常用于测试：

```go
import . "fmt"

func main() {
    // 不需要包名前缀
    Println("Hello, World!")
}
```

**不推荐在生产代码使用点导入**，因为它会降低代码可读性，难以识别标识符来自哪个包。

### 匿名导入

匿名导入只执行包的 init 函数，不使用包的其他内容：

```go
import _ "github.com/lib/pq"  // 注册 PostgreSQL 驱动
import _ "image/jpeg"         // 注册 JPEG 解码器
```

这在数据库驱动和图像格式处理中很常见。

### 包的初始化

每个包可以有多个 `init()` 函数，它们在包被导入时自动执行：

```go
package models

import (
    "database/sql"
    "log"

    _ "github.com/lib/pq"
)

var db *sql.DB

func init() {
    // 包初始化逻辑
    var err error
    db, err = sql.Open("postgres", "conn-string")
    if err != nil {
        log.Fatal(err)
    }
}
```

`init()` 函数的特点：

- 在 `main()` 函数之前执行
- 每个包可以有多个 `init()` 函数
- 按声明顺序执行
- 不能手动调用
- 没有参数和返回值

## 内部包

Go 1.4 引入了"内部包"机制：以 `internal` 为名的目录及其子目录只能被其父目录导入。

```
project/
├── main.go
└── internal/
    ├── db/
    │   └── db.go
    └── auth/
        └── auth.go
```

`internal/db` 只能被 `project` 及其子目录导入，不能被外部项目导入。这用于实现真正私有的包。

```go
// project/main.go
import "project/internal/db"  // ✅ 可以导入

// external/main.go
import "project/internal/db"  // ❌ 编译错误
```

## 依赖管理最佳实践

### 1. 定期清理依赖

```bash
go mod tidy  # 移除未使用的依赖，添加缺失的依赖
```

### 2. 使用 vendor 目录（可选）

```bash
go mod vendor  # 将依赖复制到 vendor 目录
```

这可以将所有依赖包含在代码仓库中，确保构建的可重复性。但会增加仓库大小。

### 3. 指定 Go 版本

在 `go.mod` 中指定最低 Go 版本：

```go
module github.com/yourusername/taskqueue-api

go 1.22  // 使用 Go 1.22 的新特性

require ...
```

### 4. 间接依赖

某些依赖是间接的（被其他依赖引入），Go 会在 go.mod 中标记：

```go
require (
    github.com/gin-gonic/gin v1.9.1
    github.com/json-iterator/go v1.1.12 // indirect
)
```

`// indirect` 注释表示这个包不是直接依赖，而是被其他包依赖。

## 发布和使用私有模块

### 发布私有模块

1. 将代码推送到 Git 仓库（GitHub、GitLab 等）
2. 打标签作为版本号：

```bash
git tag v1.0.0
git push origin v1.0.0
```

3. 其他用户就可以导入：

```go
import "github.com/yourusername/taskqueue-api"
```

### 配置私有模块访问

对于私有仓库，需要配置 GOPRIVATE：

```bash
go env -w GOPRIVATE=github.com/yourusername/*
```

## Trade-off 分析

### Go Modules vs 其他依赖管理工具

**Go Modules 优势**：

- 官方支持，集成在 Go 工具链中
- 声明式管理，`go.mod` 文件清晰
- 支持语义化版本和版本选择
- 无需 GOPATH，可以在任意目录工作

**Go Modules 劣势**：

- 比起 npm 的 package.json，功能相对简单
- 依赖缓存可能导致磁盘占用较大
- 某些复杂的版本冲突场景处理不够灵活

### 包结构设计

**扁平结构**（小项目）：

```
project/
├── main.go
├── handler.go
└── model.go
```

**分层结构**（中大型项目）：

```
project/
├── api/
├── services/
├── models/
└── utils/
```

**按功能分包**（推荐）：

```
project/
├── task/
│   ├── model.go
│   ├── service.go
│   └── handler.go
├── user/
│   ├── model.go
│   ├── service.go
│   └── handler.go
└── common/
    └── util.go
```

按功能分包的优势：

- 高内聚低耦合
- 易于维护和测试
- 可以独立演化

## 总结

Go Modules 是现代 Go 项目的依赖管理系统，它让你能够：

- 在任意目录创建项目
- 精确管理依赖版本
- 轻松发布和分享代码

包组织是代码架构的基础，好的包结构让代码清晰、易维护。记住：

- 每个目录对应一个包
- 大写开头表示公开，小写表示私有
- 按功能分包，而不是按层次分包
- 使用 internal 目录实现真正的私有包

在任务队列 API 中，合理的包组织让代码职责清晰，便于多人协作和后续扩展。
