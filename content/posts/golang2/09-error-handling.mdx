## 为什么需要错误处理

在编写任务队列 API 时，很多事情可能出错：数据库连接失败、任务执行失败、网络超时等。如果这些错误没有被正确处理，程序可能会崩溃、数据可能损坏、用户可能看到不友好的错误信息。

许多语言使用异常机制来处理错误，但 Go 选择了不同的方式：显式的错误返回值。这种方式让错误处理更加明确和可控，不会因为被忽略的异常导致意外行为。

Go 的错误处理哲学是：错误是正常的，应该被显式检查和处理。这种哲学让代码更加可靠和可维护。

## 错误基础

### 创建和返回错误

Go 的 error 类型是一个接口，任何实现了 Error() 方法的类型都可以作为错误。

```go
package main

import (
    "errors"
    "fmt"
)

// 除法函数，返回错误
func divide(a, b float64) (float64, error) {
    if b == 0 {
        // 使用 errors.New 创建简单错误
        return 0, errors.New("除数不能为零")
    }
    return a / b, nil
}

// 验证函数，使用 fmt.Errorf 创建格式化错误
func validateAge(age int) error {
    if age < 0 {
        return fmt.Errorf("年龄 %d 无效：不能为负数", age)
    }
    if age > 150 {
        return fmt.Errorf("年龄 %d 无效：不能超过150", age)
    }
    return nil
}

func main() {
    // 处理除法错误
    result, err := divide(10, 0)
    if err != nil {
        fmt.Println("除法错误:", err)
    } else {
        fmt.Printf("结果: %.2f\n", result)
    }

    // 处理验证错误
    err = validateAge(-5)
    if err != nil {
        fmt.Println("验证错误:", err)
    }
}
```

### 错误处理模式

Go 代码中常见的错误处理模式。

```go
package main

import (
    "errors"
    "fmt"
)

func processData(data string) (string, error) {
    // 模式 1：尽早返回
    if data == "" {
        return "", errors.New("数据不能为空")
    }

    // 模式 2：错误检查后继续
    processed, err := transform(data)
    if err != nil {
        return "", fmt.Errorf("转换失败: %w", err)
    }

    // 模式 3：nil 表示成功
    return processed, nil
}

func transform(data string) (string, error) {
    if len(data) > 10 {
        return "", errors.New("数据过长")
    }
    return data + " (已处理)", nil
}

func main() {
    // 正常情况
    result, err := processData("Hello")
    if err != nil {
        fmt.Println("错误:", err)
    } else {
        fmt.Println("结果:", result)
    }

    // 错误情况
    _, err = processData("")
    if err != nil {
        fmt.Println("错误:", err)
    }
}
```

## 自定义错误类型

### 实现错误接口

自定义错误类型可以携带更多的上下文信息。

```go
package main

import "fmt"

// ValidationError 验证错误
type ValidationError struct {
    Field   string
    Message string
    Value   interface{}
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("字段 %s 验证失败: %s (值: %v)", e.Field, e.Message, e.Value)
}

// NotFoundError 未找到错误
type NotFoundError struct {
    Resource string
    ID       int
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("%s [ID:%d] 未找到", e.Resource, ID)
}

// 验证邮箱
func validateEmail(email string) error {
    if email == "" {
        return &ValidationError{
            Field:   "email",
            Message: "不能为空",
            Value:   email,
        }
    }
    return nil
}

// 查找用户
func findUser(id int) error {
    if id > 100 {
        return &NotFoundError{
            Resource: "用户",
            ID:       id,
        }
    }
    return nil
}

func main() {
    // 验证错误
    err := validateEmail("")
    if err != nil {
        fmt.Println("错误:", err)

        // 类型断言获取详细信息
        if ve, ok := err.(*ValidationError); ok {
            fmt.Printf("  字段: %s\n", ve.Field)
            fmt.Printf("  消息: %s\n", ve.Message)
            fmt.Printf("  值: %v\n", ve.Value)
        }
    }

    // 未找到错误
    err = findUser(101)
    if err != nil {
        fmt.Println("错误:", err)
    }
}
```

### 错误包装

Go 1.13+ 引入了错误包装机制，可以在不丢失原始错误的情况下添加上下文。

```go
package main

import (
    "errors"
    "fmt"
)

// 数据库操作
func queryDatabase(id int) (string, error) {
    if id < 0 {
        return "", errors.New("ID 不能为负数")
    }
    if id > 100 {
        return "", errors.New("记录不存在")
    }
    return fmt.Sprintf("数据-%d", id), nil
}

// 业务逻辑层
func getUserData(id int) (string, error) {
    data, err := queryDatabase(id)
    if err != nil {
        // 使用 %w 包装错误，保留原始错误
        return "", fmt.Errorf("获取用户数据失败 [ID:%d]: %w", id, err)
    }
    return data, nil
}

// 服务层
func processUserRequest(id int) error {
    data, err := getUserData(id)
    if err != nil {
        // 继续包装错误
        return fmt.Errorf("处理用户请求失败: %w", err)
    }
    fmt.Println("处理数据:", data)
    return nil
}

func main() {
    // 测试错误包装
    err := processUserRequest(-1)
    if err != nil {
        fmt.Println("错误:", err)

        // 检查特定错误
        if errors.Is(err, errors.New("ID 不能为负数")) {
            fmt.Println("  这是 ID 无效错误")
        }

        // 解包错误
        unwrapped := errors.Unwrap(err)
        if unwrapped != nil {
            fmt.Println("  解包后的错误:", unwrapped)
        }
    }
}
```

### errors.As 使用

errors.As 用于将错误转换为特定的自定义错误类型。

```go
package main

import (
    "errors"
    "fmt"
)

// 业务错误
type BusinessError struct {
    Code    int
    Message string
}

func (e *BusinessError) Error() string {
    return fmt.Sprintf("业务错误 [%d]: %s", e.Code, e.Message)
}

func doWork() error {
    return &BusinessError{
        Code:    1001,
        Message: "库存不足",
    }
}

func main() {
    err := doWork()
    if err != nil {
        // 使用 errors.As 检查特定错误类型
        var bizErr *BusinessError
        if errors.As(err, &bizErr) {
            fmt.Printf("捕获到业务错误:\n")
            fmt.Printf("  代码: %d\n", bizErr.Code)
            fmt.Printf("  消息: %s\n", bizErr.Message)
        }
    }
}
```

## panic 和 recover

Go 也提供了 panic 和 recover 机制，但它们应该只在特殊情况下使用。

### panic 使用场景

panic 用于处理不可恢复的错误，比如程序状态不一致。

```go
package main

import "fmt"

// 配置结构
type Config struct {
    Host string
    Port int
}

// 加载配置
func loadConfig() *Config {
    // 配置缺失导致程序无法运行，使用 panic
    return &Config{
        Host: "",
        Port: 0,
    }
}

func validateConfig(config *Config) {
    if config.Host == "" {
        panic("配置错误: Host 不能为空")
    }
    if config.Port == 0 {
        panic("配置错误: Port 必须设置")
    }
}

func main() {
    config := loadConfig()

    // 这会导致 panic
    validateConfig(config)
}
```

### recover 使用

recover 用于从 panic 中恢复，通常在 defer 中使用。

```go
package main

import (
    "fmt"
    "log"
)

// 安全执行函数
func safeExecute(fn func()) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panic 恢复: %v", r)
            log.Printf("发生 panic: %v", r)
        }
    }()

    fn()
    return nil
}

func riskyOperation() {
    panic("出错了!")
}

func main() {
    fmt.Println("开始执行")

    // 安全执行可能 panic 的函数
    err := safeExecute(riskyOperation)
    if err != nil {
        fmt.Println("捕获到错误:", err)
    }

    fmt.Println("程序继续执行")
}
```

## 实战示例：任务队列的错误处理

现在我们为任务队列添加完善的错误处理机制。

```go
package main

import (
    "errors"
    "fmt"
    "time"
)

// 任务错误类型
type TaskError struct {
    TaskID   int
    TaskName string
    Message  string
    Cause    error
}

func (e *TaskError) Error() string {
    if e.Cause != nil {
        return fmt.Sprintf("任务 %s [%d] 失败: %s (原因: %v)",
            e.TaskName, e.TaskID, e.Message, e.Cause)
    }
    return fmt.Sprintf("任务 %s [%d] 失败: %s",
        e.TaskName, e.TaskID, e.Message)
}

func (e *TaskError) Unwrap() error {
    return e.Cause
}

// 验证错误
type ValidationError struct {
    Field   string
    Message string
    Value   interface{}
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("验证失败 [%s]: %s (值: %v)", e.Field, e.Message, e.Value)
}

// 任务状态
type TaskStatus string

const (
    StatusPending    TaskStatus = "待处理"
    StatusProcessing TaskStatus = "处理中"
    StatusCompleted  TaskStatus = "已完成"
    StatusFailed     TaskStatus = "失败"
)

// Task 接口
type Task interface {
    GetID() int
    GetName() string
    GetStatus() TaskStatus
    Execute() error
}

// BaseTask 基础任务
type BaseTask struct {
    id     int
    name   string
    status TaskStatus
    error  error
}

func NewBaseTask(id int, name string) *BaseTask {
    return &BaseTask{
        id:     id,
        name:   name,
        status: StatusPending,
    }
}

func (bt *BaseTask) GetID() int          { return bt.id }
func (bt *BaseTask) GetName() string     { return bt.name }
func (bt *BaseTask) GetStatus() TaskStatus { return bt.status }
func (bt *BaseTask) SetStatus(status TaskStatus) { bt.status = status }
func (bt *BaseTask) SetError(err error)  { bt.error = err }
func (bt *BaseTask) GetError() error     { return bt.error }

// EmailTask 邮件任务
type EmailTask struct {
    *BaseTask
    To      string
    Subject string
    Body    string
}

func NewEmailTask(id int, to, subject, body string) *EmailTask {
    return &EmailTask{
        BaseTask: NewBaseTask(id, fmt.Sprintf("邮件-%s", to)),
        To:       to,
        Subject:  subject,
        Body:     body,
    }
}

func (et *EmailTask) Validate() error {
    if et.To == "" {
        return &ValidationError{Field: "To", Message: "收件人不能为空", Value: et.To}
    }
    if et.Subject == "" {
        return &ValidationError{Field: "Subject", Message: "主题不能为空", Value: et.Subject}
    }
    return nil
}

func (et *EmailTask) Execute() error {
    // 验证任务参数
    if err := et.Validate(); err != nil {
        return fmt.Errorf("邮件任务验证失败: %w", err)
    }

    et.SetStatus(StatusProcessing)
    fmt.Printf("发送邮件到: %s\n", et.To)

    // 模拟发送过程
    time.Sleep(time.Millisecond * 50)

    // 模拟失败情况
    if et.To == "invalid@example.com" {
        err := &TaskError{
            TaskID:   et.GetID(),
            TaskName: et.GetName(),
            Message:  "发送失败",
            Cause:    errors.New("邮箱地址无效"),
        }
        et.SetStatus(StatusFailed)
        et.SetError(err)
        return err
    }

    et.SetStatus(StatusCompleted)
    return nil
}

// SMSTask 短信任务
type SMSTask struct {
    *BaseTask
    Phone   string
    Message string
}

func NewSMSTask(id int, phone, message string) *SMSTask {
    return &SMSTask{
        BaseTask: NewBaseTask(id, fmt.Sprintf("短信-%s", phone)),
        Phone:    phone,
        Message:  message,
    }
}

func (st *SMSTask) Validate() error {
    if st.Phone == "" {
        return &ValidationError{Field: "Phone", Message: "手机号不能为空", Value: st.Phone}
    }
    if st.Message == "" {
        return &ValidationError{Field: "Message", Message: "消息内容不能为空", Value: st.Message}
    }
    return nil
}

func (st *SMSTask) Execute() error {
    if err := st.Validate(); err != nil {
        return fmt.Errorf("短信任务验证失败: %w", err)
    }

    st.SetStatus(StatusProcessing)
    fmt.Printf("发送短信到: %s\n", st.Phone)

    time.Sleep(time.Millisecond * 30)

    // 模拟失败情况
    if st.Phone == "error" {
        err := &TaskError{
            TaskID:   st.GetID(),
            TaskName: st.GetName(),
            Message:  "发送失败",
            Cause:    errors.New("网络超时"),
        }
        st.SetStatus(StatusFailed)
        st.SetError(err)
        return err
    }

    st.SetStatus(StatusCompleted)
    return nil
}

// TaskProcessor 任务处理器
type TaskProcessor struct {
    successCount int
    failureCount int
}

func NewTaskProcessor() *TaskProcessor {
    return &TaskProcessor{}
}

func (tp *TaskProcessor) ProcessTask(task Task) error {
    fmt.Printf("处理任务: %s\n", task.GetName())

    err := task.Execute()
    if err != nil {
        tp.failureCount++
        // 包装错误，添加任务信息
        return fmt.Errorf("处理器错误: 任务 %s 执行失败: %w",
            task.GetName(), err)
    }
    tp.successCount++
    return nil
}

func (tp *TaskProcessor) GetStats() (success, failure int) {
    return tp.successCount, tp.failureCount
}

// TaskQueue 任务队列
type TaskQueue struct {
    tasks     []Task
    processor *TaskProcessor
}

func NewTaskQueue() *TaskQueue {
    return &TaskQueue{
        tasks:     make([]Task, 0),
        processor: NewTaskProcessor(),
    }
}

func (tq *TaskQueue) AddTask(task Task) {
    tq.tasks = append(tq.tasks, task)
}

func (tq *TaskQueue) ProcessAll() {
    fmt.Println("=== 开始处理任务 ===")

    for _, task := range tq.tasks {
        if err := tq.processor.ProcessTask(task); err != nil {
            fmt.Printf("错误: %v\n", err)

            // 尝试提取特定错误类型
            var taskErr *TaskError
            if errors.As(err, &taskErr) {
                fmt.Printf("  任务错误详情: ID=%d, Name=%s\n",
                    taskErr.TaskID, taskErr.TaskName)
            }

            var validationErr *ValidationError
            if errors.As(err, &validationErr) {
                fmt.Printf("  验证错误: 字段=%s, 消息=%s\n",
                    validationErr.Field, validationErr.Message)
            }
        }
        fmt.Println()
    }

    success, failure := tq.processor.GetStats()
    fmt.Printf("=== 处理完成 ===\n")
    fmt.Printf("成功: %d, 失败: %d\n", success, failure)
}

func (tq *TaskQueue) PrintStatus() {
    fmt.Println("\n=== 任务状态 ===")
    for _, task := range tq.tasks {
        status := task.GetStatus()
        errMsg := ""
        if bt, ok := task.(*BaseTask); ok && bt.GetError() != nil {
            errMsg = fmt.Sprintf(" (错误: %v)", bt.GetError())
        }
        fmt.Printf("%s: %s%s\n", task.GetName(), status, errMsg)
    }
}

func main() {
    queue := NewTaskQueue()

    // 添加各种任务（包括会失败的任务）
    queue.AddTask(NewEmailTask(1, "user@example.com", "欢迎", "欢迎注册"))
    queue.AddTask(NewEmailTask(2, "invalid@example.com", "测试", "这会失败"))
    queue.AddTask(NewSMSTask(3, "13800138000", "验证码 123456"))
    queue.AddTask(NewSMSTask(4, "error", "这会失败"))
    queue.AddTask(NewEmailTask(5, "", "无主题", "")) // 验证失败

    // 处理所有任务
    queue.ProcessAll()

    // 打印最终状态
    queue.PrintStatus()
}
```

## 权衡与思考

### 显式错误处理 vs 异常机制

**Go 的显式错误处理优势**：

- 错误处理明确，不会被忽略
- 代码流程清晰
- 错误可以作为正常返回值处理
- 适合编写可靠的系统代码

**异常机制的优势**：

- 减少样板代码
- 集中处理错误
- 适合处理真正的异常情况

**何时使用 panic**：

- 程序状态不一致，无法继续执行
- 初始化失败，程序无法运行
- 开发阶段的断言

**何时不使用 panic**：

- 可预期的错误（如文件不存在）
- 输入验证失败
- 网络请求失败

### 错误包装的层次

**过度包装**：

```go
// 不好的做法
return fmt.Errorf("处理失败: %w",
    fmt.Errorf("转换失败: %w",
        fmt.Errorf("解析失败: %w", err)))
```

**适度包装**：

```go
// 好的做法
return fmt.Errorf("处理数据失败: %w", err)
```

**原则**：

- 只在需要添加上下文时包装
- 每层只包装一次
- 保持错误链清晰

### 错误处理的性能考虑

**错误处理的开销**：

- 创建错误对象有内存开销
- 错误检查增加了代码路径
- 过度的错误包装影响性能

**优化建议**：

- 热路径上避免频繁创建错误
- 考虑使用预定义的错误
- 错误信息不应该包含大量数据

## 总结

Go 的错误处理机制虽然不同于传统的异常机制，但它提供了一种更明确、更可控的错误处理方式：

- 显式的错误返回值让错误处理更加明确
- 自定义错误类型可以携带丰富的上下文信息
- 错误包装机制保留了错误的原始信息
- panic 和 recover 用于处理不可恢复的情况
- 良好的错误处理让程序更加可靠

在任务队列 API 中，完善的错误处理让我们能够：

- 验证任务参数
- 处理执行失败
- 提供详细的错误信息
- 区分不同类型的错误
