你刚吃完午饭回来，电脑风扇还在嗡嗡响，终端显示 "Webpack compiling..."，进度条卡在 85%。这已经是今天的第 5 次启动了，每次都要等半分钟才能开始工作。

同事的新项目倒是可以秒开——他换用了 Vite。

为什么 Webpack 需要打包整个应用才能启动，而 Vite 可以毫秒级启动？这背后的秘密是浏览器原生 ESM（ES Modules）。

## Webpack 的困境：必须打包

当你运行 `npm run dev` 时，Webpack 做了这些事：

```javascript
// 你的源码
import { Button } from "./components/Button";
import { debounce } from "lodash";

// Webpack 编译后（简化）
const modules = {
  "./src/components/Button.js": () => {
    /* ... */
  },
  "lodash/debounce": () => {
    /* ... */
  },
  // ... 打包所有依赖
};

// 启动前必须完成这个巨大的打包过程
```

Webpack 的困境是：**它假设浏览器不支持 ESM，所以必须提前把所有模块打包成一个或几个 Bundle**。

这个过程包括：

1. 解析所有 `import` 语句，构建依赖图
2. 把所有模块转换成浏览器能执行的代码
3. 合并、压缩、生成 Source Map

项目越大，这个步骤越慢。一个中型项目（1000+ 文件）启动要 15-30 秒。

## Vite 的魔法：利用浏览器 ESM

Vite 的核心思想：**现代浏览器已经原生支持 ESM 了，为什么还要打包？**

```html
<!-- 浏览器原生 ESM -->
<script type="module">
  import { Button } from "/src/components/Button.js";
  import { debounce } from "/node_modules/lodash/debounce.js";

  // 浏览器会自动解析并加载这些模块
</script>
```

Vite 开发服务器的工作流程：

```
浏览器请求 → Vite 服务器 → 按需编译 → 返回 ESM 代码
    ↓                        ↓
 http://localhost:3000/src/App.tsx
```

关键点：**Vite 不需要提前打包，而是按需编译**。你访问哪个文件，它就编译哪个文件。

## 但 node_modules 怎么办？

你可能会问：`import React from 'react'` 这样的代码，浏览器能直接处理吗？

不能。因为 `node_modules/react` 是 CommonJS 格式，浏览器不认识。

Vite 的解决方案是**依赖预构建**：

```javascript
// 第一次启动时，Vite 用 esbuild 把 node_modules 转换成 ESM
// node_modules/.vite/deps/react.js
import { jsx_runtime } from "react";
export { useState, useEffect, useMemo /* ... */ };
```

esbuild 是用 Go 写的，比 Webpack 快 10-100 倍。即使预构建所有依赖，也只需要几秒。

而且预构建只做一次——之后启动时直接用缓存。

## HMR：毫秒级热更新

当你修改代码时：

**Webpack HMR**：

```
修改文件 → 重新编译受影响的模块 → 生成新 Bundle → HMR 更新
            (1-3 秒)
```

**Vite HMR**：

```
修改文件 → 编译单个文件 → 通过 WebSocket 推送更新
            (< 100ms)
```

Vite 的 HMR 为什么这么快？

1. 只编译修改的那个文件
2. 不重新打包，直接替换模块
3. 基于 ESM，浏览器可以精确地重新加载特定模块

## 生产环境还是需要打包

Vite 只在开发环境不打包。生产构建时，它用 Rollup 打包，原因是：

1. **减少 HTTP 请求**：合并模块，减少网络开销
2. **代码压缩**：Terser 压缩，减小体积
3. **Tree Shaking**：移除未使用的代码
4. **兼容性**：转换 ES5+ 语法，支持旧浏览器
5. **更好的缓存**：hash 命名，内容变了才重新下载

## 性能对比

用实际数据说话：

| 项目规模           | Webpack 启动 | Vite 启动 | Webpack HMR | Vite HMR |
| ------------------ | ------------ | --------- | ----------- | -------- |
| 小型（100 文件）   | 3-5 秒       | < 1 秒    | 1-2 秒      | < 100ms  |
| 中型（1000 文件）  | 15-30 秒     | 1-3 秒    | 2-5 秒      | < 200ms  |
| 大型（10000 文件） | 60-120 秒    | 3-8 秒    | 5-10 秒     | < 500ms  |

## 什么时候不用 Vite

Vite 不是银弹。这些场景下 Webpack 更合适：

1. **需要极致的兼容性**：要支持 IE11，Vite 的 ESM 方案不work
2. **复杂的代码拆分**：需要非常细粒度的动态导入控制
3. **特定的 Loader 生态**：依赖某些 Webpack 独有的 Loader

但对于大多数现代项目（尤其是 React/Vue/Svelte），Vite 是更好的选择。
