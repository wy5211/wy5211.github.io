在上一篇文章中，我们学习了如何用关系装饰器把实体关联起来。现在你的数据模型已经建立好了，接下来要解决的问题就是：怎么对这些关联数据进行增删改查？

Repository 是 TypeORM 中最常用的数据访问接口，它提供的 find、save、update、remove 等方法可以覆盖大部分 CRUD 场景。但当你遇到"查询进行中的任务，按优先级排序，带标签信息，分页返回"这种需求时，就需要深入了解 FindOptions 和 FindOperator 了。

这篇文章会用我们正在构建的 SaaS 项目管理平台作为案例，实现 Task 的完整 CRUD 功能。你会学到如何灵活组合 find 选项、如何处理分页、以及 save/update/remove 之间的区别。

## 从关系模型到 CRUD 操作

在开始之前，让我们先回顾一下上一篇中定义的实体关系。我们的项目管理平台有这些核心实体：

- Workspace（工作空间）：包含多个 Project
- Project（项目）：包含多个 Task
- Task（任务）：指派给 User，关联多个 Tag
- User（用户）：可以是被指派人
- Tag（标签）：用于标记 Task

这些关系通过 @ManyToOne、@OneToMany、@ManyToMany 装饰器已经定义好了。现在我们需要通过 Repository 来操作这些数据。

## Repository 的获取方式

在实际项目中，有三种常见方式获取 Repository：

```typescript
import { DataSource, Repository } from "typeorm";
import { Task } from "./entity/Task";

// 方式 1：通过 DataSource 获取（独立应用）
const dataSource = new DataSource({
  /* ... */
});
const taskRepo = dataSource.getRepository(Task);

// 方式 2：在 NestJS 中通过依赖注入
@Injectable()
export class TaskService {
  constructor(
    @InjectRepository(Task)
    private taskRepo: Repository<Task>
  ) {}

  async findTasks() {
    return this.taskRepo.find();
  }
}

// 方式 3：自定义 Repository（扩展功能）
@EntityRepository(Task)
export class TaskRepository extends Repository<Task> {
  async findHighPriorityTasks() {
    return this.find({ where: { priority: "URGENT" } });
  }
}
```

在 NestJS 项目中，方式 2 是最常用的。如果你的业务逻辑比较复杂，可以自定义 Repository 来封装特定领域的查询方法。

## 创建任务：save vs insert

创建任务时，我们通常会同时创建任务与标签的关联关系。这里就涉及到 save 和 insert 的区别。

```typescript
import { Task, TaskStatus } from "./entity/Task";
import { Tag } from "./entity/Tag";

async function createTask(data: {
  title: string;
  description: string;
  projectId: number;
  assigneeId: number;
  tagIds: number[];
}) {
  // 方式 1：使用 save（推荐）
  const task = taskRepo.create({
    title: data.title,
    description: data.description,
    project: { id: data.projectId },
    assignee: { id: data.assigneeId },
    status: TaskStatus.TODO,
    tags: data.tagIds.map((id) => ({ id })), // 创建标签关联
  });

  const savedTask = await taskRepo.save(task);
  // save 返回完整的实体对象，包含自动生成的 id、createdAt 等字段
  // 并且会级联保存 tags 关联关系

  return savedTask;
}

// 方式 2：使用 insert（更快，但不支持级联）
const result = await taskRepo.insert({
  title: data.title,
  description: data.description,
  projectId: data.projectId,
  assigneeId: data.assigneeId,
});
// result.identifiers: [{ id: 1 }]
// insert 只返回插入结果，不返回完整实体
// 而且不会处理 tags 关联关系
```

save 和 insert 的核心区别在于：save 会先检查实体是否已存在（通过 id 判断），如果存在就更新，不存在就插入；而 insert 只能插入新记录。更重要的是，save 支持级联操作，可以自动处理关联实体的保存。

对于需要关联其他实体的场景（比如任务关联标签），应该使用 save。

## 查询任务：FindOptions 详解

Repository 的 find 方法接受一个 FindOptions 对象，这个对象比看起来要强大得多。

### 基础查询

```typescript
// 查询所有任务
const allTasks = await taskRepo.find();

// 条件查询
const todoTasks = await taskRepo.find({
  where: { status: TaskStatus.TODO },
});

// 查询单条
const task = await taskRepo.findOne({
  where: { id: 1 },
});

// 通过 ID 查询（更简洁）
const task = await taskRepo.findOneBy({ id: 1 });

// 查询或抛出异常
const task = await taskRepo.findOneOrFail({
  where: { id: 1 },
});
// 找不到时抛出 EntityNotFoundError
```

### 加载关联数据

```typescript
// 查询任务时加载指派人和标签
const taskWithRelations = await taskRepo.findOne({
  where: { id: 1 },
  relations: {
    assignee: true,
    tags: true,
    project: true,
  },
});

// 嵌套加载：查询任务时加载项目的所属工作空间
const taskWithNestedRelations = await taskRepo.findOne({
  where: { id: 1 },
  relations: {
    project: {
      workspace: true,
    },
    assignee: true,
  },
});
```

### 选择特定字段

```typescript
// 只查询需要的字段
const tasks = await taskRepo.find({
  select: {
    id: true,
    title: true,
    status: true,
    assignee: {
      id: true,
      name: true,
    },
    tags: {
      id: true,
      name: true,
    },
  },
  relations: {
    assignee: true,
    tags: true,
  },
});
```

### 排序和分页

```typescript
const [tasks, total] = await taskRepo.findAndCount({
  where: {
    status: TaskStatus.IN_PROGRESS,
  },
  relations: {
    assignee: true,
    tags: true,
  },
  order: {
    priority: "DESC", // 按优先级降序
    createdAt: "ASC", // 同级按创建时间升序
  },
  skip: 0, // 跳过多少条（offset）
  take: 10, // 取多少条（limit）
});

// 返回值
// {
//   tasks: [...], // 10 条任务
//   total: 45 // 总共 45 条记录
// }
```

findAndCount 是分页查询的最佳实践，它同时返回数据和总数，你只需要计算 totalPages = Math.ceil(total / take)。

## 复杂条件：FindOperator

简单查询用 where 对象就够了，但复杂条件需要 FindOperator。TypeORM 提供了丰富的操作符来构建查询条件。

### 常用操作符

```typescript
import {
  Like,
  ILike,
  Between,
  In,
  IsNull,
  Not,
  LessThan,
  MoreThan,
} from "typeorm";

// Like：模糊匹配（区分大小写）
const tasks = await taskRepo.find({
  where: {
    title: Like("%导航栏%"), // 标题包含"导航栏"
  },
});

// ILike：模糊匹配（不区分大小写）
const tasks = await taskRepo.find({
  where: {
    title: ILike("%BUG%"), // 标题包含"bug"（不区分大小写）
  },
});

// Between：范围查询
const tasks = await taskRepo.find({
  where: {
    createdAt: Between(new Date("2026-01-01"), new Date("2026-12-31")),
  },
});

// In：包含于列表
const tasks = await taskRepo.find({
  where: {
    status: In([TaskStatus.TODO, TaskStatus.IN_PROGRESS]),
  },
});

// IsNull：为空
const tasks = await taskRepo.find({
  where: {
    assignee: IsNull(), // 未分配的任务
  },
});

// Not：取反
const tasks = await taskRepo.find({
  where: {
    status: Not(TaskStatus.COMPLETED),
  },
});

// LessThan：小于
const tasks = await taskRepo.find({
  where: {
    priority: LessThan(3), // 优先级小于 3
  },
});
```

### 组合条件

```typescript
// AND：多个条件同时满足
const tasks = await taskRepo.find({
  where: {
    status: TaskStatus.IN_PROGRESS,
    assignee: { id: 1 },
    priority: MoreThan(2),
  },
});
// WHERE status = 'IN_PROGRESS'
//   AND assignee_id = 1
//   AND priority > 2

// OR：满足任一条件（使用数组）
const tasks = await taskRepo.find({
  where: [
    { status: TaskStatus.TODO },
    { priority: 3 }, // 优先级为 3
    { assignee: IsNull() }, // 未分配
  ],
});
// WHERE status = 'TODO'
//    OR priority = 3
//    OR assignee_id IS NULL

// 复杂组合：使用 Brackets
import { Brackets } from "typeorm";

const tasks = await taskRepo.find({
  where: {
    project: { id: 1 },
    ...new Brackets((qb) => {
      qb.where({ status: TaskStatus.TODO }).orWhere({ assignee: IsNull() });
    }),
  },
});
// WHERE project_id = 1
//   AND (status = 'TODO' OR assignee_id IS NULL)
```

### 在关联表上查询

```typescript
// 查询指派给特定用户的任务
const tasks = await taskRepo.find({
  where: {
    assignee: { id: 1 },
  },
  relations: {
    assignee: true,
  },
});

// 查询有特定标签的任务
const tasks = await taskRepo.find({
  where: {
    tags: {
      id: In([1, 2, 3]), // 有标签 1 或 2 或 3
    },
  },
  relations: {
    tags: true,
  },
});

// 查询特定项目下的任务
const tasks = await taskRepo.find({
  where: {
    project: {
      workspace: { id: 1 }, // 工作空间 1 下的任务
    },
  },
  relations: {
    project: true,
  },
});
```

## 更新任务：save vs update

更新操作也有两种方式：save 和 update。它们的区别和创建时类似。

```typescript
// 方式 1：查询后修改 + save（推荐用于复杂更新）
const task = await taskRepo.findOneBy({ id: 1 });
if (task) {
  task.status = TaskStatus.COMPLETED;
  task.completedAt = new Date();
  await taskRepo.save(task);
  // 会触发 @BeforeUpdate 和 @AfterUpdate 监听器
}

// 方式 2：直接 update（推荐用于简单更新）
await taskRepo.update(1, {
  status: TaskStatus.COMPLETED,
  completedAt: new Date(),
});
// 不触发监听器，但性能更好

// 方式 3：条件更新（批量）
await taskRepo.update(
  { status: TaskStatus.TODO }, // where 条件
  { priority: 1 } // 更新的值
);
// UPDATE tasks SET priority = 1 WHERE status = 'TODO'
```

什么时候用哪个？如果更新逻辑复杂（比如更新多个字段、需要触发监听器、有业务逻辑校验），用 save。如果只是简单更新几个字段，用 update 更快。

## 删除任务：remove vs delete vs softDelete

删除是三种操作，对应不同的场景。

```typescript
// 方式 1：remove（通过实体删除）
const task = await taskRepo.findOneBy({ id: 1 });
await taskRepo.remove(task);
// 会触发 @BeforeRemove 和 @AfterRemove 监听器

// 方式 2：delete（直接删除）
await taskRepo.delete(1);
await taskRepo.delete([1, 2, 3]); // 批量删除
await taskRepo.delete({ status: TaskStatus.CANCELLED }); // 条件删除
// 不触发监听器

// 方式 3：softDelete（软删除）
await taskRepo.softDelete(1);
// UPDATE tasks SET deleted_at = NOW() WHERE id = 1
// 数据不会真正删除，只是标记为已删除

// 恢复软删除
await taskRepo.restore(1);
// UPDATE tasks SET deleted_at = NULL WHERE id = 1
```

软删除需要在实体中定义 @DeleteDateColumn：

```typescript
@Entity("tasks")
export class Task {
  // ...

  @DeleteDateColumn({ name: "deleted_at" })
  deletedAt: Date;
}
```

使用软删除后，普通的 find 查询会自动过滤掉已删除的记录。如果要包含已删除的记录，使用 withDeleted: true。

```typescript
// 查询时包含已删除的记录
const tasks = await taskRepo.find({
  where: { id: 1 },
  withDeleted: true,
});
```

## 实战：任务管理 Service

让我们把上面的知识点整合起来，实现一个完整的任务管理 Service。

```typescript
import { Injectable } from "@nestjs/common";
import { InjectRepository } from "@nestjs/typeorm";
import { Repository, Like, In, Brackets } from "typeorm";
import { Task, TaskStatus, TaskPriority } from "./entity/Task";
import { User } from "./entity/User";
import { Tag } from "./entity/Tag";

interface CreateTaskDto {
  title: string;
  description: string;
  projectId: number;
  assigneeId?: number;
  tagIds?: number[];
  priority?: TaskPriority;
}

interface TaskQueryDto {
  page: number;
  limit: number;
  status?: TaskStatus;
  assigneeId?: number;
  projectId?: number;
  tagId?: number;
  keyword?: string;
}

@Injectable()
export class TaskService {
  constructor(
    @InjectRepository(Task)
    private taskRepo: Repository<Task>
  ) {}

  async createTask(dto: CreateTaskDto) {
    const task = this.taskRepo.create({
      title: dto.title,
      description: dto.description,
      project: { id: dto.projectId },
      assignee: dto.assigneeId ? { id: dto.assigneeId } : null,
      tags: dto.tagIds?.map((id) => ({ id })) || [],
      priority: dto.priority || TaskPriority.MEDIUM,
      status: TaskStatus.TODO,
    });

    return this.taskRepo.save(task);
  }

  async updateTask(id: number, dto: Partial<CreateTaskDto>) {
    const task = await this.taskRepo.findOne({
      where: { id },
      relations: {
        assignee: true,
        tags: true,
      },
    });

    if (!task) {
      throw new Error("任务不存在");
    }

    // 更新基本信息
    if (dto.title) task.title = dto.title;
    if (dto.description) task.description = dto.description;
    if (dto.priority) task.priority = dto.priority;

    // 更新指派人
    if (dto.assigneeId !== undefined) {
      task.assignee = dto.assigneeId ? ({ id: dto.assigneeId } as User) : null;
    }

    // 更新标签
    if (dto.tagIds) {
      task.tags = dto.tagIds.map((id) => ({ id })) as Tag[];
    }

    // 如果状态变为完成，设置完成时间
    if (dto.status === TaskStatus.COMPLETED && !task.completedAt) {
      task.status = TaskStatus.COMPLETED;
      task.completedAt = new Date();
    }

    return this.taskRepo.save(task);
  }

  async getTask(id: number) {
    return this.taskRepo.findOne({
      where: { id },
      relations: {
        project: true,
        assignee: true,
        tags: true,
      },
    });
  }

  async queryTasks(dto: TaskQueryDto) {
    const { page, limit, status, assigneeId, projectId, tagId, keyword } = dto;

    const queryBuilder = this.taskRepo
      .createQueryBuilder("task")
      .leftJoinAndSelect("task.assignee", "assignee")
      .leftJoinAndSelect("task.tags", "tag")
      .leftJoinAndSelect("task.project", "project");

    // 状态筛选
    if (status) {
      queryBuilder.andWhere("task.status = :status", { status });
    }

    // 指派人筛选
    if (assigneeId) {
      queryBuilder.andWhere("task.assigneeId = :assigneeId", { assigneeId });
    }

    // 项目筛选
    if (projectId) {
      queryBuilder.andWhere("task.projectId = :projectId", { projectId });
    }

    // 标签筛选
    if (tagId) {
      queryBuilder.andWhere("tag.id = :tagId", { tagId });
    }

    // 关键字搜索
    if (keyword) {
      queryBuilder.andWhere(
        new Brackets((qb) => {
          qb.where("task.title ILIKE :keyword", {
            keyword: `%${keyword}%`,
          }).orWhere("task.description ILIKE :keyword", {
            keyword: `%${keyword}%`,
          });
        })
      );
    }

    // 排序
    queryBuilder
      .orderBy("task.priority", "DESC")
      .addOrderBy("task.createdAt", "DESC");

    // 分页
    queryBuilder.skip((page - 1) * limit).take(limit);

    const [tasks, total] = await queryBuilder.getManyAndCount();

    return {
      data: tasks,
      total,
      page,
      limit,
      totalPages: Math.ceil(total / limit),
    };
  }

  async deleteTask(id: number) {
    return this.taskRepo.softDelete(id);
  }

  async assignTask(id: number, assigneeId: number) {
    return this.taskRepo.update(id, {
      assignee: { id: assigneeId } as User,
    });
  }

  async changeStatus(id: number, status: TaskStatus) {
    const updateData: any = { status };

    if (status === TaskStatus.COMPLETED) {
      updateData.completedAt = new Date();
    } else {
      updateData.completedAt = null;
    }

    return this.taskRepo.update(id, updateData);
  }

  async getTaskStats(projectId: number) {
    const stats = await this.taskRepo
      .createQueryBuilder("task")
      .select("task.status", "status")
      .addSelect("COUNT(*)", "count")
      .where("task.projectId = :projectId", { projectId })
      .groupBy("task.status")
      .getRawMany();

    return stats.reduce(
      (acc, item) => {
        acc[item.status] = parseInt(item.count);
        return acc;
      },
      {} as Record<string, number>
    );
  }
}
```

这个 Service 包含了任务管理的核心功能：创建、更新、查询、删除、分配、状态变更、统计。其中 queryTasks 方法展示了如何组合多个查询条件：状态、指派人、项目、标签、关键字搜索，再加上排序和分页。

## DTO 与 Entity 的边界

你可能注意到，在上面的 Service 中，我们没有直接把 Entity 返回给调用方。这是一个重要的最佳实践。

```typescript
// ❌ 不推荐：直接返回 Entity
async getTask(id: number) {
  return this.taskRepo.findOne({
    where: { id },
    relations: {
      assignee: true,
      tags: true,
    },
  });
  // 返回的 Task 实体包含所有字段，包括敏感信息
  // 而且 Entity 是数据库模型的直接映射，暴露给外部不合适
}

// ✅ 推荐：返回 DTO
async getTask(id: number) {
  const task = await this.taskRepo.findOne({
    where: { id },
    relations: {
      assignee: true,
      tags: true,
    },
    select: {
      id: true,
      title: true,
      description: true,
      status: true,
      priority: true,
      createdAt: true,
      assignee: {
        id: true,
        name: true,
      },
      tags: {
        id: true,
        name: true,
      },
    },
  });

  if (!task) {
    return null;
  }

  return {
    id: task.id,
    title: task.title,
    description: task.description,
    status: task.status,
    priority: task.priority,
    createdAt: task.createdAt,
    assignee: {
      id: task.assignee.id,
      name: task.assignee.name,
    },
    tags: task.tags.map(tag => ({
      id: tag.id,
      name: tag.name,
    })),
  };
}
```

为什么？因为 Entity 和 DTO 的职责不同：

- Entity 是数据库模型的映射，服务于数据持久化
- DTO 是 API 层的数据传输对象，服务于接口交互

把它们混在一起会导致：1）敏感信息泄露；2）API 结构与数据库结构耦合；3）难以控制返回字段。

正确的做法是：Service 层返回 DTO，Controller 层直接返回 DTO。可以用 class-transformer 的 instanceToPlain 方法简化转换。

## Trade-off 分析

Repository 和 QueryBuilder 如何选择？

Repository 的 find 方法简单直观，适合大部分查询场景。但它的表达能力有限，比如难以实现"查询每个项目下未完成任务数大于 10 的项目"这种统计查询。

QueryBuilder 提供了类似 SQL 的链式 API，可以构建任意复杂的查询，但代码会更冗长。对于简单查询，用 QueryBuilder 是过度设计。

一个实用的经验法则：能用 find 解决的就用 find，find 写不出来的再用 QueryBuilder。下一篇文章我们会专门讲 QueryBuilder 的高级用法。

save vs update 如何选择？

save 会先检查实体是否存在，然后决定插入还是更新，而且支持级联操作和触发监听器。update 直接执行 UPDATE 语句，性能更好但功能有限。

如果你需要更新关联实体、触发监听器、或者有复杂的业务逻辑，用 save。如果只是简单更新几个字段，用 update 更高效。

remove vs delete vs softDelete 如何选择？

remove 通过实体删除，会触发监听器，但需要先查询实体。delete 直接删除，不需要查询，但也不触发监听器。softDelete 是软删除，数据不会真正删除，只是标记为已删除。

对于需要追溯的业务数据（比如任务、订单），推荐用 softDelete。对于日志、临时数据，可以用 delete。remove 很少使用，除非你明确需要触发监听器。

## 总结

这篇文章我们深入了 Repository 的 CRUD 操作，从简单的 find 到复杂的 FindOperator，从 save/update 到 remove/delete/softDelete。我们还实现了一个完整的任务管理 Service，涵盖了创建、查询、更新、删除、统计等功能。

但 QueryBuilder 还没出场。当 find 方法无法满足你的查询需求时，比如需要 JOIN、GROUP BY、子查询，QueryBuilder 就会派上用场。下一篇文章我们会专门讲 QueryBuilder，它会让你感受到 TypeORM 查询的真正威力。
