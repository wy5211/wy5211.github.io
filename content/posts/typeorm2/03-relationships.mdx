你设计好了表结构，但表和表之间怎么关联？一对多、多对多、自引用……TypeORM 的关系装饰器一大堆，cascade、eager、orphanedRowAction 这些选项又是什么意思？

今天我们为一个社交媒体系统设计关系模型——User 发 Post、Post 有 Comment、Post 有 Tag、User 关注 User（自引用）。这个场景覆盖了所有关系类型，帮你建立对关系映射的直觉。

## 为什么需要关系映射

在关系型数据库中，表之间的联系通过外键实现。`posts` 表有 `author_id` 列指向 `users` 表的 `id` 列，这样就能知道每篇文章是谁写的。

但原始的外键操作很麻烦：

```typescript
// 查询用户的所有文章
const posts = await db.query("SELECT * FROM posts WHERE author_id = $1", [
  userId,
]);

// 查询文章及其作者信息
const postsWithAuthors = await db.query(`
  SELECT
    posts.*,
    users.name as author_name,
    users.email as author_email
  FROM posts
  JOIN users ON posts.author_id = users.id
`);
```

每次都要手写 JOIN，字段多了 SQL 会变得很长，而且类型没有保障。`postsWithAuthors` 的类型是 `any[]`，你不知道里面有哪些字段。

ORM 的关系映射让你通过对象属性来访问关联数据：

```typescript
const user = await userRepo.findOneBy({ id: userId });
const posts = await userRepo.find({
  relations: ["posts"], // 自动加载文章
});

// 通过对象访问
posts[0].author.name; // 用户名
posts[0].tags; // 文章的标签
```

TypeORM 自动生成 JOIN 语句，处理类型转换，你只需要声明"用户有多篇文章"，剩下的交给框架。

## 关系类型概览

TypeORM 支持四种关系类型，选择哪种取决于业务逻辑：

`@OneToOne` 是一对一关系，比如用户和用户资料。一个用户只有一个资料，一个资料也只属于一个用户。

`@ManyToOne` 是多对一关系，比如文章和用户。多篇文章属于一个用户。这是最常用的关系，因为任何"多"的一方最终都要指向"一"的一方。

`@OneToMany` 是一对多关系，是 `@ManyToOne` 的反向。比如用户有多篇文章，但实际存储的是文章表有外键指向用户表。

`@ManyToMany` 是多对多关系，比如文章和标签。一篇文章有多个标签，一个标签也可以属于多篇文章。这种关系需要一个中间表来连接。

## @ManyToOne：最常用的关系

让我们从文章模型开始。每篇文章有一个作者，这是典型的多对一关系：

```typescript
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  ManyToOne,
  JoinColumn,
  Index,
} from "typeorm";
import { User } from "./User";

@Entity("posts")
@Index(["authorId"])
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 255 })
  title: string;

  @Column({ type: "text" })
  content: string;

  // 外键列
  @Column({ name: "author_id" })
  authorId: number;

  // 关系定义
  @ManyToOne(() => User, (user) => user.posts)
  @JoinColumn({ name: "author_id" })
  author: User;
}
```

`@ManyToOne(() => User, (user) => user.posts)` 的第一个参数是关联的实体，第二个参数是反向关系的属性名。`user.posts` 是 User 实体上的属性，我们稍后定义。

`@JoinColumn({ name: "author_id" })` 告诉 TypeORM 这个关系的外键列名。`@ManyToOne` 必须加这个装饰器，因为它是拥有方（owning side）——包含外键的一方。

再看 User 实体：

```typescript
import { Entity, PrimaryGeneratedColumn, Column, OneToMany } from "typeorm";
import { Post } from "./Post";

@Entity("users")
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 100 })
  name: string;

  @Column({ unique: true, length: 255 })
  email: string;

  // 反向关系：一个用户有多篇文章
  @OneToMany(() => Post, (post) => post.author)
  posts: Post[];
}
```

`@OneToMany(() => Post, (post) => post.author)` 的第一个参数是关联的实体，第二个参数是 Post 实体上的外键属性。注意 `@OneToMany` 不需要 `@JoinColumn`，因为它不是拥有方。

## 拥有方和反向方

这是 TypeORM 关系映射的核心概念：每个关系都有一个拥有方（owning side）和一个反向方（inverse side）。

拥有方包含外键列，负责持久化关系。`@ManyToOne` 和 `@ManyToMany`（加了 `@JoinTable` 的那一边）是拥有方。

反向方不存储外键，只用于对象导航。`@OneToMany` 和 `@OneToOne`（没有 `@JoinColumn` 的那一边）是反向方。

为什么这样设计？因为数据库只能在一端存储外键。文章表有 `author_id` 列，用户表没有 `posts` 列。所以关系必须有一个明确的"拥有者"。

`mappedBy` 选项指定反向方的关系属性：

```typescript
@OneToMany(() => Post, (post) => post.author)
posts: Post[]
```

这里的 `post.author` 就是 `mappedBy`，告诉 TypeORM"这个关系通过 Post 实体的 author 属性来映射"。

## 查询关联数据

定义好关系后，怎么查询关联数据？

```typescript
// 查询文章时加载作者
const post = await postRepo.findOne({
  where: { id: 1 },
  relations: ["author"],
});

// 通过对象访问
console.log(post.author.name);

// 查询用户时加载文章
const user = await userRepo.findOne({
  where: { id: 1 },
  relations: ["posts"],
});

user.posts.forEach((post) => console.log(post.title));
```

`relations` 选项指定要加载的关联。TypeORM 会生成 JOIN 语句，一次查询把主表和关联表的数据都取回来。

如果需要加载多层关系：

```typescript
const user = await userRepo.findOne({
  where: { id: 1 },
  relations: {
    posts: {
      tags: true, // 加载文章的标签
    },
  },
});
```

这样会 JOIN 三张表：users -> posts -> post_tags -> tags。

## @ManyToMany：中间表

多对多关系需要一个中间表。文章和标签的关系就是这样：

```typescript
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  ManyToMany,
  JoinTable,
} from "typeorm";
import { Tag } from "./Tag";

@Entity("posts")
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 255 })
  title: string;

  @Column({ type: "text" })
  content: string;

  // 拥有方：文章有多个标签
  @ManyToMany(() => Tag, (tag) => tag.posts)
  @JoinTable({
    name: "post_tags",
    joinColumn: { name: "post_id", referencedColumnName: "id" },
    inverseJoinColumn: { name: "tag_id", referencedColumnName: "id" },
  })
  tags: Tag[];
}
```

`@JoinTable()` 在多对多关系的拥有方添加。它指定中间表的名称和列名。

```typescript
@Entity("tags")
export class Tag {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ unique: true, length: 50 })
  name: string;

  // 反向方：标签属于多篇文章
  @ManyToMany(() => Post, (post) => post.tags)
  posts: Post[];
}
```

生成的中间表结构：

```sql
CREATE TABLE post_tags (
  post_id INTEGER REFERENCES posts(id),
  tag_id INTEGER REFERENCES tags(id),
  PRIMARY KEY (post_id, tag_id)
)
```

操作多对多关系：

```typescript
// 创建文章并关联标签
const tag1 = await tagRepo.save({ name: "TypeScript" });
const tag2 = await tagRepo.save({ name: "TypeORM" });

const post = await postRepo.save({
  title: "TypeORM 教程",
  content: "内容...",
  tags: [tag1, tag2],
});

// 查询文章及其标签
const postWithTags = await postRepo.findOne({
  where: { id: post.id },
  relations: ["tags"],
});

console.log(postWithTags.tags); // [{ id: 1, name: 'TypeScript' }, ...]
```

## 级联操作

创建文章时，我们先创建标签，再创建文章并关联。能不能一步完成？这就需要级联操作：

```typescript
@ManyToMany(() => Tag, (tag) => tag.posts, {
  cascade: true // 保存文章时自动保存标签
})
@JoinTable()
tags: Tag[]
```

现在可以直接这样写：

```typescript
await postRepo.save({
  title: "TypeORM 教程",
  content: "内容...",
  tags: [
    { name: "TypeScript" }, // 标签不存在，会自动创建
    { name: "TypeORM" },
  ],
});
```

`cascade: true` 会级联插入、更新、软删除、恢复所有操作。如果只想级联部分操作：

```typescript
cascade: ["insert", "update"]; // 只级联插入和更新
```

级联删除要特别谨慎：

```typescript
// 危险：删除标签会删除所有关联的文章
@ManyToMany(() => Post, (post) => post.tags, {
  cascade: ["remove"]
})
tags: Tag[]
```

大多数场景下，应该用数据库级别的 `onDelete` 约束，而不是应用级联：

```typescript
@ManyToOne(() => User, (user) => user.posts, {
  onDelete: "CASCADE" // 删除用户时数据库自动删除文章
})
@JoinColumn({ name: "author_id" })
author: User
```

这样即使绕过应用直接操作数据库，约束也会生效。

## 自引用关系

有些关系是实体内部的，比如评论的嵌套回复、用户关注用户。这就是自引用关系。

评论的嵌套回复：

```typescript
@Entity("comments")
export class Comment {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ type: "text" })
  content: string;

  @ManyToOne(() => User)
  @JoinColumn({ name: "author_id" })
  author: User;

  @ManyToOne(() => Post)
  @JoinColumn({ name: "post_id" })
  post: Post;

  // 父评论：回复哪条评论
  @ManyToOne(() => Comment, (comment) => comment.replies, {
    nullable: true,
  })
  @JoinColumn({ name: "parent_id" })
  parent: Comment;

  // 子评论：这条评论的回复
  @OneToMany(() => Comment, (comment) => comment.parent)
  replies: Comment[];
}
```

一个实体内既有父级引用，又有子级引用。`nullable: true` 让顶级评论（没有父评论）的 `parent` 字段可以为 NULL。

查询评论树：

```typescript
const comments = await commentRepo
  .createQueryBuilder("comment")
  .leftJoinAndSelect("comment.author", "author")
  .leftJoinAndSelect("comment.replies", "replies")
  .leftJoinAndSelect("replies.author", "repliesAuthor")
  .where("comment.post = :postId", { postId })
  .andWhere("comment.parent IS NULL") // 只查顶级评论
  .getMany();
```

用户关注用户也是类似的：

```typescript
@Entity("users")
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 100 })
  name: string;

  // 关注的人
  @ManyToMany(() => User, (user) => user.followers)
  @JoinTable({
    name: "user_follows",
    joinColumn: { name: "follower_id", referencedColumnName: "id" },
    inverseJoinColumn: { name: "following_id", referencedColumnName: "id" },
  })
  following: User[];

  // 粉丝
  @ManyToMany(() => User, (user) => user.following)
  followers: User[];
}
```

## 完整的社交媒体模型

现在我们定义四个完整的实体：User、Post、Comment、Tag。它们的连接覆盖了所有关系类型：

```typescript
// src/entity/User.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  OneToMany,
  ManyToMany,
  JoinTable,
  BeforeInsert,
} from "typeorm";
import * as bcrypt from "bcrypt";

@Entity("users")
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 100 })
  name: string;

  @Column({ unique: true, length: 255 })
  email: string;

  @Column({ name: "password_hash", select: false })
  passwordHash: string;

  @Column({ length: 255, nullable: true })
  avatar: string;

  @Column({ type: "text", nullable: true })
  bio: string;

  @Column({ default: true })
  isActive: boolean;

  // 一个用户有多篇文章
  @OneToMany(() => import("./Post").Post, (post) => post.author)
  posts: any[];

  // 一个用户有多条评论
  @OneToMany(() => import("./Comment").Comment, (comment) => comment.author)
  comments: any[];

  // 关注的人
  @ManyToMany(() => User, (user) => user.followers)
  @JoinTable({
    name: "user_follows",
    joinColumn: { name: "follower_id" },
    inverseJoinColumn: { name: "following_id" },
  })
  following: User[];

  // 粉丝
  @ManyToMany(() => User, (user) => user.following)
  followers: User[];

  private _password?: string;

  set password(value: string) {
    this._password = value;
  }

  @BeforeInsert()
  async hashPassword() {
    if (this._password) {
      this.passwordHash = await bcrypt.hash(this._password, 10);
    }
  }
}
```

```typescript
// src/entity/Post.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  ManyToOne,
  ManyToMany,
  JoinTable,
  OneToMany,
  Index,
} from "typeorm";

export enum PostStatus {
  DRAFT = "draft",
  PUBLISHED = "published",
  ARCHIVED = "archived",
}

@Entity("posts")
@Index(["authorId", "status"])
@Index(["slug"], { unique: true })
export class Post {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 255 })
  title: string;

  @Column({ unique: true, length: 255 })
  slug: string;

  @Column({ type: "text" })
  content: string;

  @Column({ name: "author_id" })
  authorId: number;

  @ManyToOne(() => import("./User").User, (user) => user.posts, {
    onDelete: "CASCADE",
  })
  @JoinColumn({ name: "author_id" })
  author: any;

  @Column({
    type: "enum",
    enum: PostStatus,
    default: PostStatus.DRAFT,
  })
  status: PostStatus;

  @Column({ default: 0, name: "view_count" })
  viewCount: number;

  // 一篇文章有多条评论
  @OneToMany(() => import("./Comment").Comment, (comment) => comment.post, {
    cascade: ["insert"],
  })
  comments: any[];

  // 一篇文章有多个标签
  @ManyToMany(() => import("./Tag").Tag, (tag) => tag.posts, {
    cascade: ["insert", "update"],
  })
  @JoinTable({
    name: "post_tags",
    joinColumn: { name: "post_id" },
    inverseJoinColumn: { name: "tag_id" },
  })
  tags: any[];
}
```

## Trade-off

关系映射让代码更清晰，但也有一些代价。首先，N+1 查询问题很容易出现。如果你没有正确使用 `relations`，TypeORM 会在每次访问关联属性时单独查询一次数据库。10 篇文章就有 10 个作者，结果产生 11 次查询（1 次查文章，10 次查作者）。

其次是懒加载的陷阱。TypeORM 支持懒加载（通过 Promise 延迟加载），但需要额外的配置，而且在 Web 应用中基本不推荐。Web 请求是短命的，懒加载会导致连接池耗尽。最稳妥的方式是显式指定 `relations`，明确知道会执行几次查询。

级联操作能减少代码量，但可能带来意外的副作用。`cascade: ["remove"]` 删除一个标签，可能会删除所有关联的文章。这种危险操作很难在代码审查中发现。更安全的做法是使用数据库级别的 `onDelete` 约束，或者在应用层手动级联。

自引用关系的查询比较复杂。查询评论树需要多次 JOIN，或者用递归 CTE。TypeORM 的 QueryBuilder 能处理这些场景，但代码不如手写 SQL 直观。

下一篇文章，我们会深入 Repository 模式，学习如何用 `find` 方法的选项构建灵活的查询。我们也会介绍 `QueryBuilder`，用它来处理 N+1 查询和复杂的统计查询。这些是构建 CRUD API 的基础。
