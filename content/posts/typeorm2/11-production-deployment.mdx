在上一篇文章中，我们给项目加上了测试——Repository 集成测试验证 SQL 正确性，Service 单元测试验证业务逻辑，E2E 测试验证 API 的完整流程。有了这些测试，每次改动代码时都能快速发现是否引入了问题。

但测试只是在开发环境中运行，真正的挑战是生产环境。你肯定遇到过这种情况：本地跑得好好的，一部署就出问题——数据库连不上、迁移执行失败、端口被占用、环境变量没传过去……这些问题很常见，但也很烦人。

这篇文章的目标是让项目从"本地能跑"变成"生产能用"。我们会讨论：

1. **Docker 容器化**——把应用和数据库打包成容器，保证开发、测试、生产环境一致
2. **环境变量管理**——不同环境用不同的配置，敏感信息不能写死在代码里
3. **数据库迁移**——生产环境的表结构变更必须通过迁移，不能手改
4. **健康检查**——监控应用和数据库的状态，出问题能及时发现
5. **日志和监控**——生产环境的日志要能帮你快速定位问题

这些内容不是为了追求"完美的生产环境"，而是给你一个"能用的起点"。你可以在此基础上根据项目需求继续优化。

## Docker 容器化：让环境一致

Docker 的核心价值是"一次构建，到处运行"。你在本地打包的 Docker 镜像，在测试服务器、生产服务器上跑起来是完全一样的——不会出现"我本地是好的，怎么到你这就不行了"这种问题。

Docker 有两个层面：

1. **Dockerfile**——定义单个应用的镜像（比如 NestJS 应用）
2. **Docker Compose**——编排多个应用（比如 App + PostgreSQL）

先看 Dockerfile。一个生产级的 Dockerfile 通常用"多阶段构建"——构建阶段用完整的 Node 镜像（能编译 TypeScript），运行阶段用精简的 Alpine 镜像（体积小、攻击面小）：

```dockerfile
# Dockerfile

# ========== 阶段 1：构建 ==========
FROM node:22-alpine AS builder

# 设置工作目录
WORKDIR /app

# 先复制 package.json，利用 Docker 缓存
# 只要 package.json 不变，这一层就不会重新执行
COPY package*.json ./

# 安装依赖（包括 devDependencies，因为要编译 TypeScript）
RUN npm ci

# 复制源代码
COPY . .

# 编译 TypeScript
RUN npm run build

# ========== 阶段 2：生产运行 ==========
FROM node:22-alpine AS production

WORKDIR /app

# 设置环境变量
ENV NODE_ENV=production

# 只复制生产依赖（devDependencies 不会被打包进来）
COPY package*.json ./
RUN npm ci --only=production

# 从构建阶段复制编译后的代码
COPY --from=builder /app/dist ./dist

# 复制迁移文件（生产环境需要执行迁移）
COPY -from=builder /app/migrations ./migrations

# 暴露端口
EXPOSE 3000

# 启动应用
CMD ["node", "dist/main.js"]
```

这个 Dockerfile 的几个细节：

1. **用 Alpine 镜像**——`node:22-alpine` 比 `node:22` 小很多（100MB vs 900MB），下载和部署都更快
2. **npm ci 而不是 npm install**——`ci` 是"clean install"的缩写，它会严格按照 package-lock.json 安装，更可预测
3. **多阶段构建**——最终的镜像不包含 TypeScript 源码和 devDependencies，体积小、安全性高
4. **复制迁移文件**——生产环境需要 migrations 目录来执行数据库迁移

接下来是 `.dockerignore`，告诉 Docker 哪些文件不需要打包：

```dockerignore
# .dockerignore

# 依赖目录（重新安装）
node_modules

# 编译产物（重新构建）
dist

# Git 文件
.git
.gitignore

# 环境变量文件（包含敏感信息）
.env
.env.local
.env.*.local

# 测试和文档
test
coverage
*.md

# IDE 配置
.vscode
.idea
```

然后是 Docker Compose，用于编排多个容器。开发环境和生产环境的需求不同，通常写两个文件：

```yaml
# docker-compose.dev.yml（开发环境）
version: "3.8"

services:
  # PostgreSQL 数据库
  postgres:
    image: postgres:16-alpine
    container_name: pm_platform_db_dev
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: pm_platform_dev
    ports:
      - "5432:5432" # 暴露端口，方便本地连接工具（如 pgAdmin）
    volumes:
      - pg_dev_data:/var/lib/postgresql/data # 数据持久化
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  # NestJS 应用
  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: builder # 停留在构建阶段，支持热重载
    container_name: pm_platform_app_dev
    command: npm run start:dev # 开发模式，支持热重载
    ports:
      - "3000:3000"
    environment:
      DB_HOST: postgres
      DB_PORT: 5432
      DB_USERNAME: postgres
      DB_PASSWORD: postgres
      DB_DATABASE: pm_platform_dev
      NODE_ENV: development
    depends_on:
      postgres:
        condition: service_healthy # 等数据库健康检查通过后再启动
    volumes:
      - .:/app # 挂载源代码，实现热重载
      - /app/node_modules # 防止本地的 node_modules 覆盖容器内的

volumes:
  pg_dev_data:
```

```yaml
# docker-compose.prod.yml（生产环境）
version: "3.8"

services:
  postgres:
    image: postgres:16-alpine
    container_name: pm_platform_db_prod
    environment:
      POSTGRES_USER: ${DB_USERNAME}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_DATABASE}
    volumes:
      - pg_prod_data:/var/lib/postgresql/data
    restart: always # 自动重启
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USERNAME}"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - app-network

  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: pm_platform_app_prod
    ports:
      - "3000:3000"
    environment:
      DB_HOST: postgres
      DB_PORT: 5432
      DB_USERNAME: ${DB_USERNAME}
      DB_PASSWORD: ${DB_PASSWORD}
      DB_DATABASE: ${DB_DATABASE}
      NODE_ENV: production
      JWT_SECRET: ${JWT_SECRET}
    depends_on:
      postgres:
        condition: service_healthy
    restart: always
    networks:
      - app-network

  # Nginx 反向代理（可选）
  nginx:
    image: nginx:alpine
    container_name: pm_platform_nginx
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - app
    restart: always
    networks:
      - app-network

networks:
  app-network:
    driver: bridge

volumes:
  pg_prod_data:
```

开发环境和生产环境的主要区别：

1. **开发环境**：
   - 暴露数据库端口，方便本地连接工具调试
   - 挂载源代码，支持热重载
   - 不用 Nginx，直接访问应用端口
   - 用固定的开发密码

2. **生产环境**：
   - 不暴露数据库端口，只通过应用访问
   - 用环境变量传递敏感信息
   - 加 Nginx 反向代理，处理静态文件和负载均衡
   - 所有服务设置 `restart: always`，崩溃自动重启

Nginx 配置文件示例：

```nginx
# nginx.conf

events {
    worker_connections 1024;
}

http {
    upstream app {
        server app:3000;
    }

    server {
        listen 80;
        server_name localhost;

        # API 代理
        location /api/ {
            proxy_pass http://app;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # 健康检查（不用认证）
        location /health {
            proxy_pass http://app;
        }
    }
}
```

## 环境变量管理：敏感信息不能写死

生产环境有很多敏感信息——数据库密码、JWT 密钥、第三方 API 密钥。这些不能写死在代码里，否则一旦代码泄露，密钥就泄露了。

通常的做法是用 `.env` 文件（开发环境）和环境变量注入（生产环境）：

```env
# .env（开发环境，提交到 Git）

# 数据库配置
DB_HOST=localhost
DB_PORT=5432
DB_USERNAME=postgres
DB_PASSWORD=postgres
DB_DATABASE=pm_platform_dev

# 应用配置
PORT=3000
NODE_ENV=development

# JWT 配置
JWT_SECRET=dev-secret-key-do-not-use-in-production
JWT_EXPIRES_IN=7d
```

```env
# .env.example（提交到 Git，作为模板）

# 数据库配置
DB_HOST=
DB_PORT=5432
DB_USERNAME=
DB_PASSWORD=
DB_DATABASE=

# 应用配置
PORT=3000
NODE_ENV=

# JWT 配置
JWT_SECRET=
JWT_EXPIRES_IN=
```

```env
# .env.production（生产环境，不提交到 Git）

DB_HOST=postgres
DB_PORT=5432
DB_USERNAME=pm_user
DB_PASSWORD=strong_random_password_here
DB_DATABASE=pm_platform_prod

PORT=3000
NODE_ENV=production

JWT_SECRET=another_strong_random_secret
JWT_EXPIRES_IN=7d
```

在 NestJS 中，用 `@nestjs/config` 模块来管理环境变量：

```typescript
// src/config/app.config.ts
export default () => ({
  port: parseInt(process.env.PORT, 10) || 3000,
  nodeEnv: process.env.NODE_ENV || "development",

  database: {
    host: process.env.DB_HOST,
    port: parseInt(process.env.DB_PORT, 10) || 5432,
    username: process.env.DB_USERNAME,
    password: process.env.DB_PASSWORD,
    database: process.env.DB_DATABASE,
  },

  jwt: {
    secret: process.env.JWT_SECRET,
    expiresIn: process.env.JWT_EXPIRES_IN || "7d",
  },
});
```

```typescript
// src/app.module.ts
import { ConfigModule } from "@nestjs/config";
import appConfig from "./config/app.config";

@Module({
  imports: [
    ConfigModule.forRoot({
      isGlobal: true,
      load: [appConfig],
      envFilePath: [".env.local", ".env"], // 优先加载 .env.local
    }),
    // ...
  ],
})
export class AppModule {}
```

```typescript
// src/database/database.module.ts
import { Module } from "@nestjs/common";
import { TypeOrmModule } from "@nestjs/typeorm";
import { ConfigService } from "@nestjs/config";

@Module({
  imports: [
    TypeOrmModule.forRootAsync({
      imports: [ConfigModule],
      inject: [ConfigService],
      useFactory: (config: ConfigService) => ({
        type: "postgres",
        host: config.get("database.host"),
        port: config.get("database.port"),
        username: config.get("database.username"),
        password: config.get("database.password"),
        database: config.get("database.database"),
        entities: [__dirname + "/../**/*.entity{.ts,.js}"],
        synchronize: false, // 生产环境必须关闭
        migrations: [__dirname + "/../migrations/*{.ts,.js}"],
        logging: config.get("nodeEnv") === "development", // 只在开发环境打印 SQL
      }),
    }),
  ],
})
export class DatabaseModule {}
```

## 数据库迁移：生产环境必须规范

在开发环境中，我们用了 `synchronize: true` 让 TypeORM 自动同步表结构。但这在生产环境是绝对禁止的——自动同步可能会删除列、丢失数据，而且无法追溯变更历史。

生产环境的表结构变更必须通过迁移（Migration）：

```bash
# 1. 生成迁移文件（根据 Entity 变更）
npm run migration:generate -- src/migrations/AddTaskPriorityColumn

# 2. 检查生成的迁移文件，确认 SQL 正确
# src/migrations/timestamp-AddTaskPriorityColumn.ts

# 3. 在测试环境执行迁移，验证没问题
npm run migration:run

# 4. 如果有问题，可以回滚
npm run migration:revert

# 5. 确认无误后，在生产环境执行迁移
npm run migration:run
```

一个典型的迁移文件：

```typescript
// src/migrations/1704096000000-AddTaskPriorityColumn.ts
import { MigrationInterface, QueryRunner, Table, TableIndex } from "typeorm";

export class AddTaskPriorityColumn1704096000000 implements MigrationInterface {
  public async up(queryRunner: QueryRunner): Promise<void> {
    // 添加 priority 列
    await queryRunner.query(`
      ALTER TABLE "tasks"
      ADD COLUMN "priority" varchar(20)
    `);

    // 设置默认值
    await queryRunner.query(`
      UPDATE "tasks"
      SET "priority" = 'medium'
      WHERE "priority" IS NULL
    `);

    // 添加 NOT NULL 约束
    await queryRunner.query(`
      ALTER TABLE "tasks"
      ALTER COLUMN "priority" SET NOT NULL
    `);

    // 添加索引
    await queryRunner.createIndex(
      "tasks",
      new TableIndex({
        name: "IDX_TASK_PRIORITY",
        columnNames: ["priority"],
      })
    );
  }

  public async down(queryRunner: QueryRunner): Promise<void>> {
    // 删除索引
    await queryRunner.dropIndex("tasks", "IDX_TASK_PRIORITY");

    // 删除列
    await queryRunner.query(`
      ALTER TABLE "tasks"
      DROP COLUMN "priority"
    `);
  }
}
```

## 健康检查：及时发现问题

生产环境出问题时，越早发现越好。健康检查（Health Check）就是定期检查应用和依赖服务的状态，出问题时及时报警。

NestJS 推荐用 `@nestjs/terminus` 包来实现健康检查：

```bash
npm install @nestjs/terminus
```

```typescript
// src/health/health.module.ts
import { Module } from "@nestjs/common";
import { TerminusModule } from "@nestjs/terminus";
import { HealthController } from "./health.controller";

@Module({
  imports: [TerminusModule],
  controllers: [HealthController],
})
export class HealthModule {}
```

```typescript
// src/health/health.controller.ts
import { Controller, Get } from "@nestjs/common";
import {
  HealthCheck,
  HealthCheckService,
  HealthCheckResult,
  TypeOrmHealthIndicator,
  MemoryHealthIndicator,
} from "@nestjs/terminus";

@Controller("health")
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private db: TypeOrmHealthIndicator,
    private memory: MemoryHealthIndicator
  ) {}

  @Get()
  @HealthCheck()
  async check(): Promise<HealthCheckResult> {
    return this.health.check([
      // 数据库连接检查（3 秒超时）
      () => this.db.pingCheck("database", { timeout: 3000 }),

      // 堆内存检查（不超过 150MB）
      () => this.memory.checkHeap("memory_heap", 150 * 1024 * 1024),

      // RSS 内存检查（不超过 150MB）
      () => this.memory.checkRSS("memory_rss", 150 * 1024 * 1024),
    ]);
  }
}
```

访问 `GET /health` 会返回：

```json
{
  "status": "ok",
  "info": {
    "database": {
      "status": "up"
    },
    "memory_heap": {
      "status": "up"
    },
    "memory_rss": {
      "status": "up"
    }
  },
  "error": {},
  "details": {
    "database": {
      "status": "up"
    },
    "memory_heap": {
      "status": "up"
    },
    "memory_rss": {
      "status": "up"
    }
  }
}
```

如果数据库挂了，返回会是这样：

```json
{
  "status": "error",
  "info": {
    "memory_heap": {
      "status": "up"
    }
  },

    }
  }
}
```

你可以用监控工具（如 UptimeRobot、Datadog）定期访问 `/health` 端点，发现问题时发送报警。

## 日志管理：快速定位问题

生产环境的日志要能帮你快速定位问题，而不是输出一堆没用的信息。

TypeORM 的日志配置：

```typescript
// src/database/database.module.ts
TypeOrmModule.forRootAsync({
  useFactory: (config: ConfigService) => ({
    // ...
    logging: config.get('nodeEnv') === 'development'
      ? ['error', 'warn', 'query', 'slow']
      : ['error', 'warn'],  // 生产环境只记录错误和警告
    maxQueryExecutionTime: 1000,  // 超过 1 秒的查询会被记录为 slow
  }),
}),
```

生产环境的日志应该：

1. **记录错误和警告**——包括数据库连接失败、查询超时、死锁等
2. **记录慢查询**——超过阈值的查询要记录，方便优化
3. **结构化日志**——用 JSON 格式，方便日志工具（如 ELK、Splunk）解析
4. **日志级别可配置**——开发环境 DEBUG，生产环境 INFO 或 WARN
5. **避免敏感信息**——不要记录密码、token 等敏感数据

## 最佳实践总结

经过这 11 篇文章的实践，我总结了几个 TypeORM 的最佳实践：

**实体设计**：

- 用 Data Mapper 模式，不要把业务逻辑塞进 Entity
- 数据库列名用 snake_case，TypeScript 属性用 camelCase
- 敏感字段（如密码）设置 `select: false`
- 用枚举代替魔法字符串
- 用 `@BeforeInsert` 等监听器处理数据预处理

**查询优化**：

- 避免 N+1 查询，用 `relations` 或 `leftJoinAndSelect`
- 只查需要的字段，用 `select` 精确控制
- 合理使用索引，但要避免过度索引（影响写入性能）
- 分页查询用 `findAndCount`，一次返回数据和总数

**事务管理**：

- 事务保持简短，不要在事务中调外部 API
- 根据场景选择合适的隔离级别
- 高并发场景用乐观锁，低并发用悲观锁
- 嵌套事务要谨慎，避免死锁

**迁移策略**：

- 生产环境永远不要用 `synchronize: true`
- 每次表结构变更都要生成迁移文件
- 迁移前先备份数据库
- 大表迁移要分批执行，避免长时间锁表

**测试策略**：

- Repository 层测复杂查询，简单 CRUD 不测
- Service 层重点测业务逻辑，用 Mock 隔离数据库
- E2E 测试只测核心 API，不需要覆盖所有边界
- 测试覆盖率目标是 70%-80%，不是 100%

**部署运维**：

- 用 Docker 保证环境一致性
- 敏感信息用环境变量，不要写死在代码里
- 配置健康检查，及时发现问题
- 日志要结构化，方便检索和分析
