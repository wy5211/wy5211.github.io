你是一个有 Node.js 基础的后端开发者，团队要做一个新项目，技术选型到了数据库这一步。有人推荐 Prisma，有人说 TypeORM 更灵活，还有人提 Drizzle。你打开 TypeORM 文档，看到 Active Record、Data Mapper、Repository、QueryBuilder……一时不知道从哪下手。

这个系列就是为你准备的。我们不会把 TypeORM 的 API 翻译一遍，而是从"为什么需要它"出发，带你逐步构建一个 SaaS 项目管理平台的数据层。每解决一个真实的数据问题，你就自然掌握了一个 TypeORM 特性。

今天我们先从一个简单的个人任务追踪工具开始，用最少的代码体验 TypeORM 的核心流程。

## 为什么需要 ORM

在没有 ORM 的时代，我们需要手写 SQL：

```typescript
// 查询待办任务
const result = await db.query(
  "SELECT * FROM tasks WHERE status = $1 AND created_at > $2 ORDER BY created_at DESC",
  ["pending", "2026-01-01"]
);

// 插入任务
await db.query(
  "INSERT INTO tasks (title, status, priority, created_at) VALUES ($1, $2, $3, $4)",
  ["修复登录 bug", "pending", "high", new Date()]
);
```

这种方式有几个问题：

第一，SQL 字符串容易出错。字段名写错、参数顺序弄混，这些错误要到运行时才能发现。

第二，类型安全没有保障。`result` 是 `any[]`，你不知道里面有什么字段，IDE 也不会给你提示。

第三，数据库切换成本高。从 PostgreSQL 换到 MySQL，需要重写所有 SQL 语句。

ORM 解决了这些问题：你用 TypeScript 类定义数据模型，ORM 自动生成 SQL 并处理类型转换。

## TypeORM vs Prisma vs Drizzle

选择 ORM 不是"哪个更好"的问题，而是"哪个更适合你的场景"。

TypeORM 的特点是用装饰器定义模型，和 TypeScript 深度融合。你在类上写 `@Entity()`，在属性上写 `@Column()`，代码读起来就像在描述数据结构。如果你熟悉 Java 的 Hibernate 或 .NET 的 Entity Framework，TypeORM 的设计思想会让你觉得很亲切。

Prisma 走的是另一条路。你写一个 `schema.prisma` 文件，Prisma 会生成一个类型安全的客户端。它的类型生成非常强大，但需要额外的代码生成步骤，而且在处理复杂查询时，SQL 控制力不如 TypeORM。

Drizzle 是性能优先的选择。它的类型安全做得很好，生成的 SQL 也更高效，特别适合 Edge Computing 和 Serverless 场景。但它的学习曲线较陡，生态也不如 TypeORM 成熟。

如果你的项目基于 NestJS，或者团队有 Java/.NET 背景，TypeORM 是最自然的选择。它在灵活性和开发体验之间取得了很好的平衡。

## Active Record vs Data Mapper

TypeORM 支持两种设计模式，理解它们的区别是入门的关键。

Active Record 模式中，实体本身包含数据操作的方法。你调用 `user.save()` 把用户保存到数据库，调用 `user.remove()` 删除它。这种方式简单直观，适合小型项目，但实体和数据库操作耦合在一起，不利于单元测试。

Data Mapper 模式中，实体只定义数据结构，操作通过 Repository 进行。你调用 `userRepository.save(user)` 保存用户。这种方式职责分离，实体纯粹，易于测试，适合中大型项目。NestJS 官方推荐的就是 Data Mapper 模式，本系列也会全程使用它。

## 第一个项目：个人任务追踪工具

让我们搭建一个简单的任务追踪工具，包含创建任务、标记完成、查询待办列表这些基本功能。通过这个案例，你会掌握 TypeORM 的核心流程：配置数据源、定义实体、使用 Repository。

### 环境搭建

先创建项目并安装依赖：

```bash
# 创建项目目录
mkdir typeorm-tasks && cd typeorm-tasks

# 初始化项目
npm init -y

# 安装 TypeScript 和开发工具
npm install typescript tsx @types/node -D

# 安装 TypeORM 和 PostgreSQL 驱动
npm install typeorm reflect-metadata pg
```

TypeORM 需要两个关键的 TypeScript 配置。打开 `tsconfig.json`，确保启用装饰器支持：

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "commonjs",
    "esModuleInterop": true,
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}
```

`experimentalDecorators` 让你能在类上使用 `@Entity` 这种装饰器，`emitDecoratorMetadata` 让 TypeORM 能通过反射读取类型信息。

### 配置数据源

数据源是 TypeORM 管理数据库连接的方式。创建 `src/data-source.ts`：

```typescript
import "reflect-metadata";
import { DataSource } from "typeorm";

export const AppDataSource = new DataSource({
  type: "postgres",
  host: "localhost",
  port: 5432,
  username: "postgres",
  password: "postgres",
  database: "typeorm_tasks",
  synchronize: true,
  logging: true,
  entities: ["src/entity/**/*.ts"],
});
```

配置说明：

- `synchronize: true` 让 TypeORM 自动根据你的实体定义创建表。开发阶段很方便，但生产环境一定要关掉，否则可能导致数据丢失。
- `logging: true` 会打印执行的 SQL，便于学习调试。
- `entities` 指定实体文件的位置，我们用通配符让它自动加载。

如果本地没有 PostgreSQL，用 Docker 快速启动一个：

```bash
docker run -d \
  --name typeorm-postgres \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=typeorm_tasks \
  -p 5432:5432 \
  postgres:16
```

### 定义任务实体

在 TypeORM 中，数据库表对应一个 TypeScript 类，类属性对应表的列。创建 `src/entity/Task.ts`：

```typescript
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
} from "typeorm";

@Entity("tasks")
export class Task {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 255 })
  title: string;

  @Column({
    type: "enum",
    enum: ["pending", "in_progress", "completed"],
    default: "pending",
  })
  status: "pending" | "in_progress" | "completed";

  @Column({
    type: "enum",
    enum: ["low", "medium", "high"],
    default: "medium",
  })
  priority: "low" | "medium" | "high";

  @Column({ type: "text", nullable: true })
  description: string;

  @Column({ type: "date", nullable: true })
  dueDate: Date;

  @CreateDateColumn({ name: "created_at" })
  createdAt: Date;
}
```

`@Entity("tasks")` 告诉 TypeORM 这个类对应 `tasks` 表。`@PrimaryGeneratedColumn()` 定义自增主键。`@Column()` 定义普通列，可以通过选项指定类型、长度、默认值等。`@CreateDateColumn` 会自动记录创建时间。

### 实现任务管理

现在我们用 Repository 来操作任务数据。创建 `src/index.ts`：

```typescript
import "reflect-metadata";
import { AppDataSource } from "./data-source";
import { Task } from "./entity/Task";

async function main() {
  // 初始化数据源
  await AppDataSource.initialize();
  console.log("数据库已连接");

  const taskRepo = AppDataSource.getRepository(Task);

  // 创建任务
  console.log("\n=== 创建任务 ===");
  const task = taskRepo.create({
    title: "学习 TypeORM",
    description: "完成 TypeORM 系列教程的学习",
    status: "in_progress",
    priority: "high",
    dueDate: new Date("2026-04-30"),
  });
  await taskRepo.save(task);
  console.log("任务已创建：", task.title);

  // 查询所有任务
  console.log("\n=== 所有任务 ===");
  const allTasks = await taskRepo.find();
  allTasks.forEach((t) => console.log(`- ${t.title} [${t.status}]`));

  // 按状态查询
  console.log("\n=== 待处理任务 ===");
  const pendingTasks = await taskRepo.find({
    where: { status: "pending" },
    order: { priority: "DESC" },
  });
  pendingTasks.forEach((t) => console.log(`- ${t.title} [${t.priority}]`));

  // 更新任务状态
  console.log("\n=== 更新任务状态 ===");
  const taskToUpdate = await taskRepo.findOneBy({ id: task.id });
  if (taskToUpdate) {
    taskToUpdate.status = "completed";
    await taskRepo.save(taskToUpdate);
    console.log("任务已标记为完成");
  }

  // 关闭数据库连接
  await AppDataSource.destroy();
}

main().catch(console.error);
```

运行项目：

```bash
npx tsx src/index.ts
```

你会看到 TypeORM 自动创建了 `tasks` 表，并打印出执行的 SQL。这就是 `synchronize: true` 的作用——它检测到你的实体定义，自动生成对应的表结构。

### Repository 的核心方法

Repository 是 TypeORM 操作数据的主要接口。最常用的方法：

- `create()` 创建一个实体实例（不保存到数据库）
- `save()` 保存实体到数据库（插入或更新）
- `find()` 查询多条记录
- `findOneBy()` 根据条件查询单条记录
- `update()` 批量更新
- `delete()` 批量删除

`save()` 方法很智能：如果实体有 ID，它会执行 UPDATE；否则执行 INSERT。`create()` 不是必须的，你可以直接传对象给 `save()`，但使用 `create()` 能让代码更清晰，还能利用 TypeORM 的类型推断。

### 项目结构

一个规范的 TypeORM 项目结构：

```
src/
├── data-source.ts       # 数据源配置
├── entity/              # 实体定义
│   ├── Task.ts
│   └── index.ts         # 统一导出
├── repository/          # 自定义 Repository（可选）
└── index.ts             # 入口文件
```

实体文件单独放在 `entity/` 目录下，通过 `index.ts` 统一导出，这样维护起来更清晰。如果你的项目变大，可以添加 `repository/` 目录存放自定义的 Repository，`migration/` 目录存放数据库迁移文件。

## 什么时候该用 TypeORM

TypeORM 最适合这些场景：

第一，你的项目基于 NestJS。NestJS 官方推荐 TypeORM，两者集成非常自然。`TypeOrmModule.forFeature()` 可以自动注册 Entity 和 Repository，通过 `@InjectRepository()` 装饰器注入。

第二，你需要处理复杂的关系映射。TypeORM 的一对一、一对多、多对多关系定义非常清晰，级联操作和加载策略也很灵活。如果你的数据模型涉及大量关联表，TypeORM 会让你省不少心。

第三，你的团队有 Java 或 .NET 背景。TypeORM 的设计思想和 Hibernate、Entity Framework 很像，团队上手会很快。

第四，你需要精细控制 SQL。TypeORM 提供了 QueryBuilder，可以手写复杂的查询逻辑，同时保持类型安全。当 `find()` 方法不够用时，QueryBuilder 是你的武器。

但如果你的项目很简单，只是几个基础的 CRUD 操作，Prisma 的开发速度会更快。如果你追求极致性能，Drizzle 是更好的选择。

## Trade-off

TypeORM 的优势是灵活性和功能全面性。它的装饰器语法与 TypeScript 深度融合，关系映射和查询构建能力都很强大。但这也带来了学习曲线陡峭的问题。Active Record vs Data Mapper、owning side vs inverse side、cascade vs orphanedRowAction……这些概念需要时间消化。

另外，TypeORM 的性能不如 Drizzle，也不如手写 SQL。它的查询生成和类型转换都有开销，在高并发场景下可能成为瓶颈。但大多数应用达不到需要优化这个地步，过早优化是万恶之源。

`synchronize: true` 是开发阶段的便利工具，但不要依赖它。生产环境的数据库变更必须通过迁移来管理，否则你会遇到数据丢失的风险。迁移的话题我们在后续文章会详细讨论。
