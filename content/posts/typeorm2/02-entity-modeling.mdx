你用 Prisma 的 schema.prisma 文件定义过数据模型，但 TypeORM 用的是装饰器。装饰器能做的事情更多还是更少？字段类型、默认值、约束、索引……这些在 TypeORM 里怎么表达？

今天我们为一个电商订单系统设计数据模型——Order（订单）、OrderItem（订单明细）、Payment（支付记录）。这个场景涉及金额、状态枚举、时间戳、JSON 字段等多种数据类型，正好覆盖 TypeORM 装饰器的大部分用法。

## 为什么需要精确控制数据模型

上一篇文章我们定义了一个简单的 Task 实体，用的都是最基本的 `@Column()` 装饰器。但实际业务中，你需要更精细的控制：

金额字段应该用什么数据类型？PostgreSQL 的 `numeric` 类型能避免浮点数精度问题，但它在 TypeScript 中映射成 `string`，怎么处理？

订单状态是固定的几个值，用枚举还是字符串？如何保证数据库只存储这些值？

某些字段组合必须唯一，比如同一个订单不能有相同的商品，怎么定义复合唯一约束？

有些字段是可选的，有些是必填的，有些有默认值。这些约束应该在数据库层面强制，还是只在应用层校验？

TypeORM 的装饰器让你在 TypeScript 类中定义这些规则，然后自动转换成正确的 SQL。你写一遍代码，TypeORM 帮你生成对应的表结构、索引、约束，既保证类型安全，又不失去数据库的控制力。

## @Column 装饰器详解

`@Column()` 是最常用的装饰器，它的选项控制着列的所有属性。先看一个完整的订单实体：

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm";

@Entity("orders")
export class Order {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 50 })
  orderNumber: string;

  @Column({
    type: "enum",
    enum: ["pending", "processing", "shipped", "delivered", "cancelled"],
    default: "pending",
  })
  status: string;

  @Column({ type: "decimal", precision: 10, scale: 2 })
  totalAmount: string;

  @Column({ type: "jsonb", nullable: true })
  metadata: Record<string, any>;

  @Column({ default: true })
  isActive: boolean;

  @Column({ nullable: true })
  notes: string;
}
```

`type` 选项指定列的 SQL 类型。如果不提供，TypeORM 根据 TypeScript 类型自动推断：`string` 变成 `varchar(255)`，`number` 变成 `integer`，`Date` 变成 `timestamp`。

`length` 选项控制字符串列的长度。`varchar(255)` 是默认值，但对于订单编号这种固定格式的字段，50 个字符就够了。

`nullable` 控制字段是否允许为空。默认是 `false`，即必填字段。设置为 `true` 后，数据库列允许 NULL 值，TypeScript 类型也会变成 `string | undefined`。

`default` 设置数据库级别的默认值。当插入记录时没有提供这个字段，数据库会自动填充默认值。注意这和应用层设置默认值不同，数据库默认值即使绕过应用直接操作数据库也会生效。

## 数据类型映射

TypeScript 类型和 SQL 类型的映射关系需要特别注意，尤其是金额、日期、JSON 这些类型。

### 金额类型

金额应该用 `decimal` 类型，避免浮点数精度问题：

```typescript
@Column({ type: "decimal", precision: 10, scale: 2 })
totalAmount: string
```

`precision: 10, scale: 2` 表示总共 10 位数字，其中 2 位小数。能存储的最大值是 99999999.99。

你可能疑惑为什么 TypeScript 类型是 `string` 而不是 `number`。这是 JavaScript 的限制：Number 类型的最大安全整数是 2^53 - 1，而 `decimal(10,2)` 能表示的范围超出了这个限制。用字符串存储不会有精度损失，TypeORM 会自动处理转换。

计算金额时要小心：

```typescript
// 错误：直接相加会丢失精度
const total = order1.totalAmount + order2.totalAmount;

// 正确：转换为 number 后计算
const total = Number(order1.totalAmount) + Number(order2.totalAmount);
```

或者使用专门的货币计算库，比如 `dinero.js` 或 `currency.js`。

### 枚举类型

订单状态、支付方式这类字段，值是固定的几个选项，用枚举类型最合适：

```typescript
@Column({
  type: "enum",
  enum: ["pending", "processing", "shipped", "delivered", "cancelled"],
  default: "pending"
})
status: string
```

TypeORM 会创建一个 PostgreSQL 的 `enum` 类型，并在列上使用它。数据库层面会强制只能存储这些值，插入其他值会报错。

也可以用 TypeScript 的 `enum`：

```typescript
export enum OrderStatus {
  PENDING = "pending",
  PROCESSING = "processing",
  SHIPPED = "shipped",
  DELIVERED = "delivered",
  CANCELLED = "cancelled"
}

@Column({
  type: "enum",
  enum: OrderStatus,
  default: OrderStatus.PENDING
})
status: OrderStatus
```

这样类型安全更好，IDE 会自动补全枚举值，重构时也更安全。

### JSON 类型

PostgreSQL 的 `jsonb` 类型适合存储结构化数据：

```typescript
@Column({ type: "jsonb", nullable: true })
metadata: Record<string, any>
```

可以存储任意 JSON 对象：

```typescript
order.metadata = {
  customerNotes: "请周末配送",
  giftWrap: true,
  specialInstructions: ["轻放", "易碎"],
};
```

`jsonb` 比 `json` 类型查询更快，支持索引，但写入稍慢。大多数场景下用 `jsonb` 是正确的选择。

### 日期类型

TypeORM 提供了三个便捷的时间戳装饰器：

```typescript
import { CreateDateColumn, UpdateDateColumn, DeleteDateColumn } from "typeorm";

@Entity()
export class Order {
  @CreateDateColumn({ name: "created_at" })
  createdAt: Date;

  @UpdateDateColumn({ name: "updated_at" })
  updatedAt: Date;

  @DeleteDateColumn({ name: "deleted_at" })
  deletedAt: Date;
}
```

`@CreateDateColumn` 在插入时自动设置为当前时间，之后不会再改变。`@UpdateDateColumn` 在每次更新时自动刷新为当前时间。`@DeleteDateColumn` 用于软删除，设置为非 NULL 值表示记录已删除。

这三个装饰器生成的 SQL 列默认不允许 NULL，但 `@DeleteDateColumn` 除外，它必须是 nullable 的。

如果需要精确到日期（不含时间），用 `date` 类型：

```typescript
@Column({ type: "date" })
orderDate: Date
```

## 索引和约束

索引能大幅提升查询性能，但会降低写入速度并占用存储空间。要在查询频繁的字段上建立索引。

### 单列索引

```typescript
import { Index } from "typeorm";

@Entity("orders")
export class Order {
  @Column()
  @Index()
  orderNumber: string;
}
```

`@Index()` 装饰器在 `orderNumber` 上创建索引。如果这个字段的值必须唯一，可以创建唯一索引：

```typescript
@Column()
@Index({ unique: true })
orderNumber: string
```

唯一索引不仅提升查询性能，还保证数据完整性。尝试插入重复的订单号会报错。

### 复合索引

有些查询涉及多个字段，复合索引更有效：

```typescript
@Entity("orders")
@Index(["customerId", "status"])
@Index(["status", "createdAt"])
export class Order {
  @Column({ name: "customer_id" })
  customerId: number;

  @Column()
  status: string;

  @CreateDateColumn({ name: "created_at" })
  createdAt: Date;
}
```

第一个索引优化"查询某客户的所有订单"，第二个索引优化"查询某个状态范围的订单并按时间排序"。

复合索引的字段顺序很重要。如果把 `createdAt` 放在 `status` 前面，优化效果会大打折扣。

### 唯一约束

有些业务规则需要通过唯一约束保证：

```typescript
@Entity("order_items")
@Unique("UQ_order_product", ["orderId", "productId"])
export class OrderItem {
  @Column({ name: "order_id" })
  orderId: number;

  @Column({ name: "product_id" })
  productId: number;
}
```

这个约束确保同一个订单不能包含重复的商品。

## 实体监听器

有时候需要在数据操作前后执行额外逻辑，比如自动计算订单总金额、记录操作日志。TypeORM 提供了实体监听器来实现这类需求。

让我们为订单实体添加自动计算总金额的功能：

```typescript
import { Entity, BeforeInsert, BeforeUpdate } from "typeorm";
import { OrderItem } from "./OrderItem";

@Entity("orders")
export class Order {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ type: "decimal", precision: 10, scale: 2, default: "0.00" })
  totalAmount: string;

  // 关联的订单明细，稍后会在关系映射章节详细讲解
  items: OrderItem[];

  @BeforeInsert()
  @BeforeUpdate()
  calculateTotal() {
    // 在保存到数据库前自动计算总金额
    this.totalAmount = this.items
      .reduce((sum, item) => sum + Number(item.price) * item.quantity, 0)
      .toFixed(2);
  }
}
```

`@BeforeInsert()` 在插入前执行，`@BeforeUpdate()` 在更新前执行。这样无论通过什么方式创建或修改订单，总金额都会自动保持正确。

类似的监听器还有 `@AfterInsert`、`@AfterUpdate`、`@BeforeRemove`、`@AfterRemove`，分别在对应的数据库操作后执行。常见的用途是发送通知、清理缓存、记录日志。

## 软删除

有些数据不应该真正删除，比如订单记录。客户可能会要求查询历史订单，财务审计也需要保留所有记录。软删除通过标记记录为"已删除"来解决这个问题：

```typescript
import { DeleteDateColumn } from "typeorm";

@Entity("orders")
export class Order {
  @DeleteDateColumn({ name: "deleted_at" })
  deletedAt: Date;
}
```

调用 `softRemove()` 或 `softDelete()` 会软删除记录：

```typescript
await orderRepo.softDelete(orderId);
// 等价于 UPDATE orders SET deleted_at = NOW() WHERE id = ?

// 恢复软删除的记录
await orderRepo.restore(orderId);
// 等价于 UPDATE orders SET deleted_at = NULL WHERE id = ?
```

普通查询会自动过滤掉软删除的记录：

```typescript
const orders = await orderRepo.find();
// SELECT * FROM orders WHERE deleted_at IS NULL
```

如果需要包含已删除的记录：

```typescript
const orders = await orderRepo.find({ withDeleted: true });
// SELECT * FROM orders
```

软删除让数据可以恢复，查询行为也更符合业务逻辑。但它不是万能的，如果涉及隐私合规（如 GDPR 的"被遗忘权"），可能还是需要真正删除数据。

## 完整的订单系统模型

现在我们定义三个完整的实体：Order、OrderItem、Payment。关系映射会在下一篇文章详细讲解，这里先关注字段定义。

```typescript
// src/entity/Order.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
  DeleteDateColumn,
  Index,
} from "typeorm";

export enum OrderStatus {
  PENDING = "pending",
  PROCESSING = "processing",
  SHIPPED = "shipped",
  DELIVERED = "delivered",
  CANCELLED = "cancelled",
}

@Entity("orders")
@Index(["customerId", "status"])
@Index(["status", "createdAt"])
export class Order {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 50, unique: true })
  orderNumber: string;

  @Column({ name: "customer_id" })
  customerId: number;

  @Column({
    type: "enum",
    enum: OrderStatus,
    default: OrderStatus.PENDING,
  })
  status: OrderStatus;

  @Column({ type: "decimal", precision: 10, scale: 2, default: "0.00" })
  totalAmount: string;

  @Column({ type: "decimal", precision: 10, scale: 2, default: "0.00" })
  discountAmount: string;

  @Column({ type: "decimal", precision: 10, scale: 2, default: "0.00" })
  taxAmount: string;

  @Column({ type: "decimal", precision: 10, scale: 2, default: "0.00" })
  shippingAmount: string;

  @Column({ type: "decimal", precision: 10, scale: 2, nullable: true })
  paidAmount: string;

  @Column({ type: "jsonb", nullable: true })
  metadata: {
    customerNotes?: string;
    giftWrap?: boolean;
    specialInstructions?: string[];
  };

  @Column({ type: "text", nullable: true })
  notes: string;

  @CreateDateColumn({ name: "created_at" })
  createdAt: Date;

  @UpdateDateColumn({ name: "updated_at" })
  updatedAt: Date;

  @DeleteDateColumn({ name: "deleted_at" })
  deletedAt: Date;
}
```

```typescript
// src/entity/OrderItem.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  ManyToOne,
  JoinColumn,
  Unique,
} from "typeorm";
import { Order } from "./Order";

@Entity("order_items")
@Unique("UQ_order_product", ["orderId", "productId"])
export class OrderItem {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ name: "order_id" })
  orderId: number;

  @ManyToOne(() => Order)
  @JoinColumn({ name: "order_id" })
  order: Order;

  @Column({ name: "product_id" })
  productId: number;

  @Column({ length: 255 })
  productName: string;

  @Column({ type: "text", nullable: true })
  productDescription: string;

  @Column({ type: "decimal", precision: 10, scale: 2 })
  price: string;

  @Column({ type: "integer" })
  quantity: number;

  @Column({ type: "decimal", precision: 10, scale: 2 })
  discount: string;
}
```

```typescript
// src/entity/Payment.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  Index,
} from "typeorm";

export enum PaymentStatus {
  PENDING = "pending",
  COMPLETED = "completed",
  FAILED = "failed",
  REFUNDED = "refunded",
}

export enum PaymentMethod {
  CREDIT_CARD = "credit_card",
  DEBIT_CARD = "debit_card",
  PAYPAL = "paypal",
  BANK_TRANSFER = "bank_transfer",
}

@Entity("payments")
@Index(["orderId"])
@Index(["status", "createdAt"])
export class Payment {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ name: "order_id" })
  orderId: number;

  @Column({ length: 50, unique: true })
  transactionId: string;

  @Column({
    type: "enum",
    enum: PaymentStatus,
    default: PaymentStatus.PENDING,
  })
  status: PaymentStatus;

  @Column({
    type: "enum",
    enum: PaymentMethod,
  })
  method: PaymentMethod;

  @Column({ type: "decimal", precision: 10, scale: 2 })
  amount: string;

  @Column({ type: "jsonb", nullable: true })
  metadata: {
    cardLastFour?: string;
    cardBrand?: string;
    paypalEmail?: string;
    bankAccount?: string;
  };

  @Column({ type: "text", nullable: true })
  notes: string;

  @CreateDateColumn({ name: "created_at" })
  createdAt: Date;

  @Column({ type: "timestamp", nullable: true })
  completedAt: Date;
}
```

这三个实体涵盖了电商订单系统的核心数据模型。Order 存储订单基本信息，OrderItem 存储订单明细，Payment 记录支付信息。它们之间的关系（一个订单有多个明细，一个订单可能有多次支付尝试）会在下一篇文章详细讲解。

## Trade-off

TypeORM 装饰器的优势是类型安全和开发效率。你在 TypeScript 类中定义数据模型，IDE 能提供完整的类型提示，重构时也很安全。而且实体定义和表结构在同一个地方，不需要同步维护两个文件。

但装饰器也有局限性。某些高级数据库特性（如部分索引、表达式索引、触发器）无法通过装饰器定义，只能手写迁移。如果你的项目大量依赖这些特性，可能需要考虑其他方案。

另一个问题是学习曲线。`@Column` 的选项有几十个，每个数据库类型的支持程度也不同。新手容易在 `decimal` 类型的 TypeScript 映射、枚举类型的定义方式这些细节上卡住。

`synchronize: true` 在开发阶段很方便，但不要在生产环境使用。它通过比较实体定义和现有表结构来生成 ALTER 语句，但这种方式不够安全，可能删除重要数据。生产环境的数据库变更必须通过迁移来管理，这是下一篇文章的主题。

现在我们已经掌握了单表的定义方式，但实际的业务模型很少是单表的。下一篇文章会讲解关系映射：如何用 `@OneToOne`、`@ManyToOne`、`@OneToMany`、`@ManyToMany` 连接多个实体，以及如何控制级联操作和加载策略。我们会为一个社交媒体系统设计 User、Post、Comment、Tag 四个实体的关系。
