## 从本地到生产：那些让人崩溃的"最后一公里"

上一篇文章我们给任务管理平台搭了一套完整的测试策略——单元测试、集成测试、E2E 测试，三层防线把质量守住了。当时我说过一句话：测试的目的不是"没有 bug"，而是"上线前能发现 bug"。但现实是，哪怕你本地跑得再丝滑，部署到生产环境的时候，总有些东西会在你意想不到的地方出问题。

我见过太多这样的场景了：

- 本地用 `localhost:5432` 连数据库，部署上去直接连不上，因为生产数据库在内网
- `NEXTAUTH_SECRET` 忘了在生产环境配置，用户一登录就 500
- 图片上传功能本地好好的，部署后文件系统变成了只读的 Serverless 函数
- 首次部署成功，第二次部署构建失败，因为某个依赖在小版本更新后不兼容了

部署这件事，看起来就是"把代码放到服务器上"，但真正做起来，它涉及环境变量管理、数据库迁移、域名配置、HTTPS 证书、错误监控、性能追踪……每一个环节都是一个潜在的坑。

这篇文章，我们就围绕团队任务管理平台，一步步把它从本地开发环境推到生产环境。不是简单的"跑个命令"，而是一套完整的、可重复的、让人睡得着觉的部署流程。

## 先把环境变量这件事说清楚

环境变量是部署出问题最常见的原因，没有之一。我们先用 `@t3-oss/env-nextjs` 把所有环境变量做一个类型安全的校验，这样在构建阶段就能发现配置遗漏。

```typescript
// src/env.ts
import { createEnv } from "@t3-oss/env-nextjs";
import { z } from "zod";

export const env = createEnv({
  // 以 NEXT_PUBLIC_ 开头的变量会暴露给浏览器端
  client: {
    NEXT_PUBLIC_APP_URL: z.string().url(),
    NEXT_PUBLIC_SUPABASE_URL: z.string().url(),
  },
  // 仅服务端可用的变量，永远不会泄露到浏览器
  server: {
    DATABASE_URL: z.string().url(),
    NEXTAUTH_SECRET: z.string().min(32),
    NEXTAUTH_URL: z.string().url(),
    GITHUB_CLIENT_ID: z.string(),
    GITHUB_CLIENT_SECRET: z.string(),
    SENTRY_DSN: z.string().url().optional(),
    UPLOAD_BUCKET: z.string().optional(),
  },
  // 运行时变量（用于 Next.js Middleware 等需要在 edge 运行时访问的变量）
  runtimeEnv: {
    NEXT_PUBLIC_APP_URL: process.env.NEXT_PUBLIC_APP_URL,
    NEXT_PUBLIC_SUPABASE_URL: process.env.NEXT_PUBLIC_SUPABASE_URL,
    DATABASE_URL: process.env.DATABASE_URL,
    NEXTAUTH_SECRET: process.env.NEXTAUTH_SECRET,
    NEXTAUTH_URL: process.env.NEXTAUTH_URL,
    GITHUB_CLIENT_ID: process.env.GITHUB_CLIENT_ID,
    GITHUB_CLIENT_SECRET: process.env.GITHUB_CLIENT_SECRET,
    SENTRY_DSN: process.env.SENTRY_DSN,
    UPLOAD_BUCKET: process.env.UPLOAD_BUCKET,
  },
  // 如果有变量没配，构建时直接报错而不是在运行时悄悄出问题
  skipValidation: !!process.env.SKIP_ENV_VALIDATION,
});
```

有了这层校验，如果你的 `.env.production` 里漏了 `GITHUB_CLIENT_SECRET`，构建时就会直接报错告诉你缺少什么变量，而不是部署上去之后用户点登录才发现 500。

接下来是 `.env.example` 文件——把它提交到 Git 仓库，方便团队成员知道需要配置哪些变量：

```bash
# .env.example —— 提交到 Git，供团队参考
# 应用地址
NEXT_PUBLIC_APP_URL=http://localhost:3000
NEXTAUTH_URL=http://localhost:3000

# 数据库（本地开发用 Docker 运行的 PostgreSQL）
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/taskflow

# NextAuth 密钥（用 `openssl rand -base64 32` 生成）
NEXTAUTH_SECRET=your-secret-key-at-least-32-chars

# GitHub OAuth
GITHUB_CLIENT_ID=your-github-client-id
GITHUB_CLIENT_SECRET=your-github-client-secret

# Supabase（用于文件存储）
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co

# Sentry（错误监控，可选）
SENTRY_DSN=https://xxx@sentry.io/xxx
```

记住一个原则：`.env.local` 永远不要提交到 Git，`.env.example` 必须提交。这样团队里任何人拉代码都能立刻知道要配什么。

## 数据库部署：从本地 PostgreSQL 到 Supabase

本地开发我们用 Docker 跑的 PostgreSQL，生产环境需要换成托管数据库。Supabase 是一个不错的选择——它本质上是托管 PostgreSQL，额外提供了实时订阅、文件存储、认证等功能，而且免费额度对中小团队够用。

### 创建 Supabase 项目

在 supabase.com 上创建项目后，你会拿到一个数据库连接字符串：

```
postgresql://postgres.[project-ref]:[password]@aws-0-[region].pooler.supabase.com:6543/postgres
```

把这个连接字符串填到 Vercel 的环境变量里（后面会说怎么配）。注意 Supabase 的连接字符串有两种模式：

- **Direct connection**（端口 5432）：用于迁移操作，比如 `prisma migrate deploy`
- **Connection pooler**（端口 6543）：用于应用运行时查询，因为 Serverless 函数会频繁创建和销毁连接，池化可以避免连接数爆炸

### 执行数据库迁移

生产环境的数据库是空的，我们需要把 schema 迁移过去：

```bash
# 确保迁移文件是最新的
npx prisma migrate dev

# 设置生产数据库连接（临时指向 Supabase）
# 注意：迁移操作必须用 Direct connection（5432 端口），不能用 pooler
DATABASE_URL="postgresql://postgres.[ref]:[password]@aws-0-[region].pooler.supabase.com:5432/postgres" \
  npx prisma migrate deploy
```

`prisma migrate deploy` 和 `prisma migrate dev` 的区别很重要：前者只执行已有的迁移文件，不会创建新的；后者会检测 schema 变化并生成迁移。生产环境永远只用 `deploy`，绝对不要跑 `dev`。

### 用 Neon 作为替代方案

如果你不需要 Supabase 的额外功能（文件存储、实时订阅），Neon 是一个更轻量的选择——它同样提供 Serverless PostgreSQL，而且有一个很有用的功能：分支。就像 Git 分支一样，你可以为每个 PR 创建一个独立的数据库分支，测试完再合并。

```typescript
// 用 Neon 的 Serverless Driver 连接数据库，减少冷启动时间
// 相比传统的 pg 驱动，Neon 的 Serverless Driver 不需要建立 TCP 连接，
// 而是通过 WebSocket/HTTP 协议通信，更适合 Serverless 环境
import { neon } from "@neondatabase/serverless";

const sql = neon(process.env.DATABASE_URL!);

// 用法和 pg 的 tagged template 类似，但底层走的是 HTTP
const tasks =
  await sql`SELECT id, title, status FROM tasks WHERE assignee_id = ${userId}`;
```

选择 Supabase 还是 Neon？如果你已经在用 Supabase 做文件存储或认证，就直接用它的数据库，少一个服务就少一份运维心智负担。如果只需要一个干净的 PostgreSQL，Neon 的分支功能在 CI/CD 场景下非常有用。

## 部署到 Vercel：基本流程

Vercel 是 Next.js 的官方部署平台，部署体验最丝滑。基本流程其实就三步：连接仓库、配环境变量、点部署。但我们一步步来说，把每个环节都讲清楚。

### 连接 Git 仓库

登录 vercel.com，点击 "Add New Project"，选择你的 GitHub 仓库。Vercel 会自动检测到这是一个 Next.js 项目，预设好构建命令。你只需要确认以下配置：

- **Framework Preset**: Next.js（自动检测）
- **Root Directory**: `./`（如果你的 Next.js 项目在 monorepo 的某个子目录里，这里要改成对应路径）
- **Build Command**: `npm run build`（默认值，一般不需要改）
- **Install Command**: `npm ci`（用 ci 而不是 install，确保依赖版本和 lockfile 一致）

### 配置环境变量

在 Vercel Dashboard 的 Settings -> Environment Variables 里，把前面列出的所有变量都配上去。关键点：**环境变量可以按环境（Production / Preview / Development）分别设置**。

比如 `NEXTAUTH_URL`，生产环境是 `https://taskflow.yourdomain.com`，预览环境（每个 PR 自动生成一个域名）可能需要动态配置。Vercel 支持用 `VERCEL_URL` 这个内置变量：

```typescript
// 这样写就不用为每个预览环境手动配 NEXTAUTH_URL 了
const nextAuthUrl =
  process.env.NEXTAUTH_URL ?? `https://${process.env.VERCEL_URL}`;
```

### 自定义域名和 HTTPS

在 Settings -> Domains 里添加你的域名，比如 `taskflow.yourdomain.com`。然后去你的域名服务商那里添加一条 DNS 记录：

```
类型: CNAME
名称: taskflow（如果要用 taskflow.yourdomain.com）
值: cname.vercel-dns.com
```

DNS 生效后（通常几分钟到几小时），Vercel 会自动签发 SSL 证书并启用 HTTPS。你不需要手动配置证书，Vercel 全部处理好了。

如果你的域名在国内服务商（比如阿里云、腾讯云），CNAME 记录一般需要完成域名实名认证才能生效。提前把这个流程走完，别等部署当天才发现域名没备案。

## Docker 自托管：不想用 Vercel 怎么办

Vercel 虽然方便，但它毕竟是第三方平台——你的代码在他们的服务器上跑，数据经过他们的网络。有些团队因为合规要求、成本控制或者单纯的"想自己掌控一切"，会选择自托管。Docker 是最常见的方式。

### Dockerfile

Next.js 官方提供了 `standalone` 输出模式，专门为 Docker 部署优化过——它会自动把需要的依赖都收集到一个独立目录里，不需要在容器里跑 `npm install`。

```dockerfile
# 第一阶段：构建
FROM node:20-alpine AS builder
WORKDIR /app

# 先只复制 package 文件，利用 Docker 缓存层
# 只要 package.json 和 lockfile 没变，这一层就不会重新执行
COPY package.json package-lock.json ./
RUN npm ci

# 再复制源代码并构建
COPY . .
RUN npm run build

# 第二阶段：运行
FROM node:20-alpine AS runner
WORKDIR /app

# 安全最佳实践：用非 root 用户运行应用
ENV NODE_ENV=production
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs

# standalone 模式下，构建产物在 .next/standalone 目录
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

USER nextjs

# Next.js 默认监听 3000 端口
EXPOSE 3000
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"

CMD ["node", "server.js"]
```

要启用 standalone 模式，需要在 `next.config.ts` 里加一行配置：

```typescript
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  // standalone 模式会把所有依赖打包到一个独立目录，
  // 适合 Docker 部署，不需要在容器里装 node_modules
  output: "standalone",
};

export default nextConfig;
```

### Docker Compose

如果是小团队内部用，直接 `docker compose up` 就能把整个技术栈拉起来——应用、数据库、缓存，一键启动：

```yaml
# docker-compose.yml
version: "3.8"

services:
  # Next.js 应用
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgresql://postgres:postgres@db:5432/taskflow
      - NEXTAUTH_SECRET=${NEXTAUTH_SECRET}
      - NEXTAUTH_URL=http://localhost:3000
      - GITHUB_CLIENT_ID=${GITHUB_CLIENT_ID}
      - GITHUB_CLIENT_SECRET=${GITHUB_CLIENT_SECRET}
    depends_on:
      db:
        condition: service_healthy

  # PostgreSQL 数据库
  db:
    image: postgres:16-alpine
    environment:
      - POSTGRES_DB=taskflow
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=postgres
    volumes:
      # 把数据库数据持久化到宿主机，容器重建数据不丢失
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    # 健康检查：确保数据库完全启动后再让应用连接
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  pgdata:
```

```bash
# 启动所有服务（后台运行）
docker compose up -d

# 查看日志
docker compose logs -f app

# 停止并清理
docker compose down

# 停止并删除数据卷（慎用！会清空数据库）
docker compose down -v
```

自托管的好处是完全可控——你可以选择任何云服务器、随时调整配置、不用担心平台限制。代价是你得自己管 SSL 证书续期（用 Let's Encrypt + Certbot）、服务器安全加固、数据库备份、日志收集……这些在 Vercel 上都是自动的。

## CI/CD：让部署变成一个按钮

手动部署的问题在于——太依赖人了。谁记得上线前要跑数据库迁移？谁记得要把环境变量同步到 Vercel？CI/CD 就是把这些步骤自动化，让部署变成"推代码 -> 等绿灯 -> 合并"这样一个可重复的流程。

### GitHub Actions：代码质量门禁

我们在 `.github/workflows/ci.yml` 里配置一个 CI 流水线。每次推送代码或提 PR，它都会自动跑检查：

```yaml
# .github/workflows/ci.yml
name: CI

# 触发条件：推送到 main 分支，或对 main 分支提 PR
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    # 只在同一个 PR 中跑一次，不要重复跑
    concurrency:
      group: ci-${{ github.ref }}
      cancel-in-progress: true

    steps:
      # 检出代码
      - uses: actions/checkout@v4

      # 安装 Node.js 20，并启用 npm 缓存加速后续安装
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      # 用 ci 而不是 install，确保版本严格匹配 lockfile
      - name: Install dependencies
        run: npm ci

      # 代码格式和 lint 检查
      - name: Run linter
        run: npm run lint

      # TypeScript 类型检查（捕获类型错误，不要让它溜到运行时）
      - name: Type check
        run: npx tsc --noEmit

      # 跑测试套件
      - name: Run tests
        run: npm run test -- --coverage

      # 尝试构建（确保生产构建能成功）
      - name: Build
        run: npm run build
        env:
          # CI 环境中不需要真实的环境变量，但构建不能因此失败
          # 所以跳过环境变量校验
          SKIP_ENV_VALIDATION: "true"
```

这个流水线做了四件事：lint、类型检查、测试、构建。任何一步失败，PR 就不能合并（你在 GitHub 仓库设置里把分支保护规则打开就行）。

### Vercel 自动部署

如果你用的是 Vercel，部署其实已经自动了——连接 Git 仓库后，Vercel 会监听 `main` 分支的推送，自动构建和部署。推送到其他分支会生成预览环境（每个 PR 一个独立 URL）。

但你可能还想要一些额外的自动化。比如，在 PR 的 CI 评论里显示构建状态和预览链接：

```yaml
# .github/workflows/deploy-preview.yml
name: Deploy Preview

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  deploy-preview:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Vercel Preview
        # 用 Vercel 的官方 GitHub Action
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          # --preview 表示这是预览部署，不是生产部署
          vercel-args: "--preview"
```

这里需要在 GitHub 仓库的 Settings -> Secrets 里配置三个变量：`VERCEL_TOKEN`（在 Vercel 账户设置里生成）、`VERCEL_ORG_ID` 和 `VERCEL_PROJECT_ID`（在 Vercel 项目的 `.vercel/project.json` 里能找到）。

## 错误监控：上线后出事了怎么办

部署不是终点，而是运维的起点。代码上线后，你不可能盯着日志看有没有报错。你需要一套自动化的错误监控和告警系统，出了问题第一时间通知你。

### Sentry：捕获运行时错误

Sentry 是目前最流行的错误监控平台，对 Next.js 的支持也非常好。

```bash
# 安装 Sentry SDK
npm install @sentry/nextjs

# 运行 Sentry 向导，它会自动创建配置文件
npx @sentry/wizard@latest -i nextjs
```

Sentry 会在你的项目里生成三个配置文件。我们重点看一下客户端和服务端的配置：

```typescript
// sentry.client.config.ts —— 捕获浏览器端的错误
import * as Sentry from "@sentry/nextjs";

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,

  // 采样率：1.0 表示采集 100% 的性能数据
  // 生产环境建议降到 0.1~0.2，避免数据量太大
  tracesSampleRate: process.env.NODE_ENV === "production" ? 0.2 : 1.0,

  // Session Replay：录制用户操作回放，方便复现 bug
  // 正常用户中 0% 采样（只采样出错的 session）
  replaysSessionSampleRate: 0,
  // 出错的 session 100% 采样
  replaysOnErrorSampleRate: 1.0,

  // 过滤掉一些不需要上报的噪声
  ignoreErrors: [
    // 用户取消请求导致的错误（比如快速切换页面）
    "AbortError",
    // 网络中断
    "NetworkError",
  ],
});
```

```typescript
// sentry.server.config.ts —— 捕获服务端错误
import * as Sentry from "@sentry/nextjs";

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  tracesSampleRate: process.env.NODE_ENV === "production" ? 0.2 : 1.0,
});
```

然后在 Next.js 的错误边界里主动上报错误，这样用户看到的是友好的错误页面，开发者能在 Sentry 后台看到完整的错误信息和堆栈：

```typescript
// app/error.tsx —— 全局错误边界
"use client";

import { useEffect } from "react";
import * as Sentry from "@sentry/nextjs";

export default function GlobalError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  // 把错误上报到 Sentry
  // digest 字段是 Next.js 自动生成的错误指纹，可以用来关联日志
  useEffect(() => {
    Sentry.captureException(error);
  }, [error]);

  return (
    <div className="flex min-h-screen items-center justify-center">
      <div className="text-center">
        <h2 className="text-xl font-semibold mb-2">
          出了点问题
        </h2>
        <p className="text-gray-500 mb-4">
          我们已经记录了这个错误，会尽快修复。
        </p>
        <button
          onClick={reset}
          className="px-4 py-2 bg-blue-600 text-white rounded hover:bg-blue-700"
        >
          重试
        </button>
      </div>
    </div>
  );
}
```

在 Sentry 后台可以配置告警规则——比如"5 分钟内同一个错误出现超过 10 次，就发 Slack/邮件/钉钉通知"。这样你不需要盯着屏幕，出了问题它会主动找你。

### 健康检查端点

对于自托管场景，你需要一个健康检查端点让负载均衡器或监控服务知道你的应用是否还活着：

```typescript
// app/api/health/route.ts —— 健康检查端点
import { NextResponse } from "next/server";
import { sql } from "@/lib/db";

export async function GET() {
  try {
    // 不只是返回 200，还要检查关键依赖是否正常
    const start = Date.now();
    await sql`SELECT 1`; // 测试数据库连接
    const dbLatency = Date.now() - start;

    // 如果数据库响应超过 2 秒，返回 503（服务不可用）
    if (dbLatency > 2000) {
      return NextResponse.json(
        {
          status: "degraded",
          database: "slow",
          dbLatencyMs: dbLatency,
          timestamp: new Date().toISOString(),
        },
        { status: 503 }
      );
    }

    return NextResponse.json({
      status: "ok",
      database: "connected",
      dbLatencyMs: dbLatency,
      timestamp: new Date().toISOString(),
    });
  } catch (error) {
    // 数据库挂了，返回 503
    return NextResponse.json(
      {
        status: "unhealthy",
        database: "disconnected",
        error: error instanceof Error ? error.message : "unknown",
        timestamp: new Date().toISOString(),
      },
      { status: 503 }
    );
  }
}
```

这个端点不仅返回"活没活"，还返回数据库连接状态和延迟。配合 Uptime Robot（免费）之类的监控服务，每分钟 ping 一次，挂了就发邮件通知你。

### Vercel Analytics：性能数据

如果你用 Vercel 部署，可以直接用内置的 Speed Insights 和 Web Analytics，零配置就能收集 Core Web Vitals 数据：

```typescript

import { Analytics } from "@vercel/analytics/react";

}: {

}) {

      <body>
        {children}

        {/* Analytics 收集页面访问量、用户设备等基础数据 */}
        <Analytics />
        {/* Speed Insights 收集 LCP、FID、CLS 等 Core Web Vitals 指标 */}

```

Vercel 的监控数据会显示在项目的 Analytics 标签页里。虽然不如 Sentry 的功能丰富，但对于了解"用户从哪里来、页面快不快"已经够用了，而且完全免费。

## Trade-off 分析

到这一步，我们已经把部署的各个环节都过了一遍。但选择哪种方案，取决于你的具体情况。

**Vercel vs 自托管**：Vercel 的优势在于零运维——SSL、CDN、日志、监控全帮你处理了，适合不想管基础设施的团队。劣势是成本（Pro 版 $20/月/人）和平台锁定（你的代码深度依赖 Vercel 的边缘函数、Serverless 等特性）。自托管用一台 $5/月的云服务器就能跑起来，但你要自己处理证书续期、安全加固、数据库备份、日志收集……这些隐形成本不低。

**Supabase vs Neon**：Supabase 更像一个"全家桶"——数据库 + 文件存储 + 认证 + 实时订阅，适合快速搭建全栈应用。Neon 更专注——就是 Serverless PostgreSQL，但它的数据库分支功能在 CI/CD 场景下非常有用。如果你的项目已经用了 Supabase 的其他功能（比如我们系列里用 Supabase 存文件），就别再加一个数据库服务了。

**Sentry vs 自建日志**：Sentry 的错误聚合、堆栈解析、告警通知做得非常专业，免费额度每月 5000 个错误事件，对中小项目完全够用。自建方案（比如 ELK 栈）功能更灵活，但运维成本高很多。除非你有特殊需求（数据不能出内网），否则 Sentry 的性价比更高。

## 系列回顾：Next.js 全栈开发知识图谱

到这里，"Next.js 渐进式全栈开发"系列就完结了。我们从一个空的 Next.js 项目开始，一步步构建了一个功能完整的团队任务管理平台。回顾一下我们走过的路：

**基础篇（01-03）**：从环境搭建和 App Router 核心概念入手，理解了 Server Components 和 Client Components 的本质区别——前者在服务器上渲染、零 JS 送到浏览器；后者有状态和交互、需要在客户端运行。这个区分是理解整个 Next.js 架构的基石。

**数据篇（04-06）**：Server Actions 让我们不用写 API 路由就能处理表单提交和数据变更；数据获取和 API 路由让我们能灵活地对接外部服务；缓存策略（ISR、revalidate）让页面既新鲜又快。这三篇构成了 Next.js 全栈开发的数据层。

**工程篇（07-09）**：Zod 做数据校验，NextAuth.js 做认证授权，文件上传用 Supabase Storage。这三个看似独立的功能，其实都在解决同一个问题——让用户输入安全地流入系统，让系统资源安全地流出到用户。

**质量篇（10-12）**：性能优化不是玄学，是 Core Web Vitals 数据驱动的工程实践；测试策略不是写完代码再补的负担，而是和开发同步进行的质量保障；部署不是终点，而是运维的起点——监控、告警、日志，这些才是让你真正睡得着觉的东西。
