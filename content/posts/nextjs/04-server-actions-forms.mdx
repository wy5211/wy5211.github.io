上一篇我们把 Server Components 和 Client Components 的边界搞清楚了——什么该放服务端、什么必须放客户端，心里应该有数了。但有一个问题一直悬着：Server Components 能直接访问数据库读数据，那**写数据**呢？用户要提交一个表单、修改一条记录，总不能让 Server Component 直接执行 `db.insert()` 吧？

在传统的 Next.js（Pages Router 时代）或者普通的 React 项目里，处理一次表单提交大概是这样的：

1. 写一个 API Route（`/api/tasks`），处理请求体、验证数据、写数据库
2. 在前端写一个 `fetch` 调用，拼 URL、设 headers、处理 JSON
3. 自己管理 loading 状态（`setIsSubmitting(true)`）
4. 自己管理 error 状态（`try/catch` + `setError(msg)`）

改一个字段？API Route 和前端组件都要改。加一个验证规则？两个地方各加一次。这四件套写多了，你会想：难道没有更简单的方式？

Next.js 14 稳定的 Server Actions 就是干这个的。今天我们围绕一个真实的场景来学它——从这篇开始，整个系列的后续文章都会围绕同一个项目展开。

## 贯穿案例：团队任务管理平台

在进入具体技术之前，先介绍一下我们要构建的项目。整个系列从这一篇起，都会围绕一个**团队任务管理平台**（类似 Linear / Trello）来讲解。下面是贯穿全系列的数据示例，后面的每一篇文章都会用到它：

```
工作空间：字节前端团队 (ws_fe_8x9k2m)

项目：官网重构 v2.0
  └── 任务：修复导航栏在移动端的布局错位
        ├── 指派给：zhang_wei (usr_a1x9k2m)

        └── 活动日志：zhang_wei 将状态从 "待处理" 改为 "进行中"
```

数据看起来很简单，但要把它变成一个可用的 Web 应用，需要处理很多事情：创建任务、更新状态、添加评论、数据验证、权限控制……这些都需要"写数据"的操作。今天我们就从最基础的开始——用 Server Actions 实现任务的创建和状态更新。

## 传统表单处理的痛点

先看传统方式下，创建一个任务需要多少代码。

```typescript
// app/api/tasks/route.ts —— 第一步：写 API Route
import { NextRequest, NextResponse } from "next/server";
import { db } from "@/lib/db";

export async function POST(req: NextRequest) {
  try {
    const body = await req.json();

    // 手动验证
    if (!body.title || body.title.trim().length === 0) {
      return NextResponse.json({ error: "任务标题不能为空" }, { status: 400 });
    }

    if (!body.projectId) {
      return NextResponse.json({ error: "请选择所属项目" }, { status: 400 });
    }

    // 优先级校验：必须是预定义的枚举值
    const validPriorities = ["low", "medium", "high", "urgent"];
    if (!validPriorities.includes(body.priority)) {
      return NextResponse.json({ error: "无效的优先级" }, { status: 400 });
    }

    // 写入数据库
    const task = await db.task.create({
      data: {
        title: body.title.trim(),
        description: body.description || "",
        status: "todo",
        priority: body.priority || "medium",
        projectId: body.projectId,
        assigneeId: body.assigneeId || null,
        workspaceId: body.workspaceId,
      },
    });

    return NextResponse.json({ task }, { status: 201 });
  } catch (error) {
    console.error("创建任务失败:", error);
    return NextResponse.json({ error: "服务器内部错误" }, { status: 500 });
  }
}
```

```tsx
// app/workspace/[wsId]/projects/[projectId]/tasks/new/page.tsx
// 第二步：前端页面，手动管理 loading / error / 状态刷新
"use client";

import { useState } from "react";
import { useRouter } from "next/navigation";

export default function NewTaskPage({
  params,
}: {
  params: { wsId: string; projectId: string };
}) {
  const router = useRouter();
  // 手动管理 loading 和 error 状态
  const [isSubmitting, setIsSubmitting] = useState(false);
  const [error, setError] = useState<string | null>(null);

  async function handleSubmit(e: React.FormEvent<HTMLFormElement>) {
    e.preventDefault();
    setIsSubmitting(true);
    setError(null);

    try {
      const formData = new FormData(e.currentTarget);
      // 手动拼请求
      const res = await fetch("/api/tasks", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          title: formData.get("title"),
          description: formData.get("description"),
          priority: formData.get("priority"),
          projectId: params.projectId,
          workspaceId: params.wsId,
        }),
      });

      const data = await res.json();

      if (!res.ok) {
        // 手动处理各种 HTTP 状态码
        setError(data.error || "创建失败");
        return;
      }

      // 手动跳转
      router.push(`/workspace/${params.wsId}/projects/${params.projectId}`);
    } catch (err) {
      // 手动处理网络错误
      setError("网络错误，请稍后重试");
    } finally {
      setIsSubmitting(false);
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      {/* 表单字段... */}
      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? "创建中..." : "创建任务"}
      </button>
      {error && <p className="text-red-500">{error}</p>}
    </form>
  );
}
```

看到问题了吗？一个"创建任务"的功能，涉及两个文件、大约 80 行代码，而且逻辑高度重复——API Route 里验证一次，前端大概率也要验证一次（即时反馈）。loading 和 error 的管理更是样板代码。

Server Actions 要解决的就是这个问题：**让表单提交变成一次函数调用**。

## Server Actions 的本质

Server Actions 并不是什么黑魔法。它的本质是：**一段标记了 `"use server"` 的异步函数，运行在服务端，但可以被客户端组件直接调用**。

Next.js 在底层帮你做了这些事：

- 把函数引用序列化成一个 action ID
- 在表单提交时，自动发送 POST 请求到服务端
- 服务端根据 action ID 找到对应函数并执行
- 把执行结果返回给前端

你不需要写 API Route，不需要手动 `fetch`，不需要拼 URL。下面看具体怎么用。

## 基本用法：'use server' 与表单 action

Server Actions 有两种定义方式。第一种是内联定义，写在 Server Component 内部，只能在该组件中使用；第二种是独立文件定义，标记 `"use server"`，可以在任何组件中导入。

在实际项目中，推荐用独立文件的方式，这样 Action 可以在多个组件之间复用。我们用任务管理平台的"创建任务"来演示。

```typescript
// app/actions/task-actions.ts
// "use server" 告诉 Next.js：这个文件里的所有导出函数都是 Server Actions
"use server";

import { db } from "@/lib/db";
import { revalidatePath } from "next/cache";

/**
 * 创建任务
 *
 * 参数使用 FormData 类型——这是最简单的形式，
 * 对应 <form action={createTask}> 提交时浏览器自动收集的表单数据。
 * 你也可以让 Server Action 接收普通 JS 对象（用 bind 预填充参数），
 * 后面讲 useFormState 时会看到。
 */
export async function createTask(formData: FormData) {
  // 从 FormData 中提取字段值
  const title = formData.get("title") as string | null;
  const description = formData.get("description") as string | null;
  const priority = formData.get("priority") as string | null;
  const projectId = formData.get("projectId") as string | null;
  const workspaceId = formData.get("workspaceId") as string | null;

  // 服务端验证——永远不要信任客户端传来的数据
  if (!title || title.trim().length === 0) {
    // 返回错误对象而不是抛异常，让前端可以展示具体错误信息
    return { success: false, error: "任务标题不能为空" };
  }

  if (!projectId || !workspaceId) {
    return { success: false, error: "缺少必要参数" };
  }

  // 优先级枚举校验
  const validPriorities = ["low", "medium", "high", "urgent"] as const;
  const normalizedPriority = validPriorities.includes(
    priority as (typeof validPriorities)[number]
  )
    ? priority
    : "medium";

  try {
    // 写入数据库
    const task = await db.task.create({
      data: {
        title: title.trim(),
        description: description?.trim() || "",
        status: "todo", // 新任务默认状态为"待处理"
        priority: normalizedPriority,
        projectId,
        workspaceId,
        // assigneeId 可以稍后通过"指派"操作来设置
      },
    });

    // revalidatePath 告诉 Next.js：这个路径对应的缓存过期了，
    // 下次访问时需要重新渲染。这是 Server Actions 中触发 UI 更新的关键机制。
    revalidatePath(`/workspace/${workspaceId}/projects/${projectId}`);

    return { success: true, task };
  } catch (error) {
    // 生产环境应该用结构化日志，这里简化处理
    console.error("创建任务失败:", error);
    return { success: false, error: "服务器内部错误，请稍后重试" };
  }
}
```

但有一个问题：目前如果创建失败（比如标题为空），Server Action 返回了错误信息，但前端完全没有处理它。这就引出了下一个工具。

## useFormState：管理服务端返回的状态

`useFormState` 是 React 19 提供的 Hook（从 `react-dom` 导入），它把 Server Action 的返回值变成组件的状态。每次表单提交后，Action 的返回值会自动成为最新的 state。

```tsx
// app/workspace/[wsId]/projects/[projectId]/tasks/new/page.tsx
"use client";

import { useFormState } from "react-dom";
import { createTask } from "@/app/actions/task-actions";

// 定义状态的类型——这是 Server Action 返回值的联合类型
type TaskFormState =
  | { success: true; task: { id: string; title: string } }
  | { success: false; error: string }
  | null; // 初始状态为 null，表示还没有提交过

// 初始状态
const initialState: TaskFormState = null;

export default function NewTaskPage({
  params,
}: {
  params: { wsId: string; projectId: string };
}) {
  // useFormState 接收两个参数：
  // 1. 原始的 Server Action
  // 2. 初始状态
  // 返回值：[当前状态, 绑定了状态管理的 formAction]
  const [state, formAction] = useFormState(
    // createTask 本身接收 FormData
    // 我们需要把 projectId 和 workspaceId 预填充进去
    // 用 .bind(null, ...args) 来实现
    createTask.bind(null, {
      projectId: params.projectId,
      workspaceId: params.wsId,
    }),
    initialState
  );

  return (
    <div className="max-w-2xl mx-auto p-6">
      <h1 className="text-xl font-semibold mb-6">创建新任务</h1>

      {/* 用 formAction 替代原始的 createTask */}
      <form action={formAction}>
        <div className="mb-4">
          <label htmlFor="title" className="block text-sm font-medium mb-1">
            任务标题
          </label>
          <input
            id="title"
            name="title"
            type="text"
            placeholder="例：修复导航栏在移动端的布局错位"
            className="w-full border rounded-md px-3 py-2"
            // 根据服务端验证结果标记字段
            aria-invalid={state?.success === false}
          />
        </div>

        <div className="mb-4">
          <label
            htmlFor="description"
            className="block text-sm font-medium mb-1"
          >
            描述（可选）
          </label>
          <textarea
            id="description"
            name="description"
            rows={3}
            placeholder="补充任务的详细说明..."
            className="w-full border rounded-md px-3 py-2"
          />
        </div>

        <div className="mb-6">
          <label htmlFor="priority" className="block text-sm font-medium mb-1">
            优先级
          </label>
          <select
            id="priority"
            name="priority"
            defaultValue="medium"
            className="w-full border rounded-md px-3 py-2"
          >
            <option value="low">低</option>
            <option value="medium">中</option>
            <option value="high">高</option>
            <option value="urgent">紧急</option>
          </select>
        </div>

        <button
          type="submit"
          className="bg-blue-600 text-white px-4 py-2 rounded-md"
        >
          创建任务
        </button>

        {/* 根据 state 渲染不同的反馈信息 */}
        {state?.success === true && (
          <p className="mt-3 text-green-600">
            任务「{state.task.title}」创建成功！
          </p>
        )}

        {state?.success === false && (
          <p className="mt-3 text-red-600">{state.error}</p>
        )}
      </form>
    </div>
  );
}
```

配合 `useFormState`，Server Action 的签名需要做一点调整——使用 `bind` 预填充参数后，第一个参数会变成我们传入的对象，第二个才是 FormData：

```typescript
// app/actions/task-actions.ts（更新版）
"use server";

import { db } from "@/lib/db";
import { revalidatePath } from "next/cache";

// 定义预填充参数的类型
interface CreateTaskContext {
  projectId: string;
  workspaceId: string;
}

// 使用 bind 后，第一个参数是 CreateTaskContext，第二个才是 FormData
export async function createTask(
  context: CreateTaskContext,
  formData: FormData
) {
  const title = formData.get("title") as string | null;
  const description = formData.get("description") as string | null;
  const priority = formData.get("priority") as string | null;

  if (!title || title.trim().length === 0) {
    return { success: false, error: "任务标题不能为空" };
  }

  // context 中已经包含了 projectId 和 workspaceId，
  // 不再需要从 FormData 中提取
  const validPriorities = ["low", "medium", "high", "urgent"] as const;
  const normalizedPriority = validPriorities.includes(
    priority as (typeof validPriorities)[number]
  )
    ? priority
    : "medium";

  try {
    const task = await db.task.create({
      data: {
        title: title.trim(),
        description: description?.trim() || "",
        status: "todo",
        priority: normalizedPriority,
        projectId: context.projectId,
        workspaceId: context.workspaceId,
      },
    });

    revalidatePath(
      `/workspace/${context.workspaceId}/projects/${context.projectId}`
    );

    return { success: true, task };
  } catch (error) {
    console.error("创建任务失败:", error);
    return { success: false, error: "服务器内部错误，请稍后重试" };
  }
}
```

`useFormState` 解决了"怎么拿到服务端返回值"的问题。但还有一个体验问题：用户点了"创建任务"之后，按钮没有任何变化，他可能会以为没点上，于是连续点好几次。

## useFormStatus：显示提交中的 loading 状态

`useFormStatus` 也是一个来自 `react-dom` 的 Hook，它会告诉你当前所在的 `<form>` 是否正在提交。一个关键的细节：**它必须被放在 `<form>` 的子组件中**，不能和 `useFormState` 放在同一个组件里（因为 Hook 的上下文检测机制要求它必须是 form 的直接后代）。

```tsx
// components/submit-button.tsx
"use client";

import { useFormStatus } from "react-dom";

// 把按钮抽成独立组件，这样 useFormStatus 才能正确检测到所属的 form
export function SubmitButton({
  children,
  className = "",
}: {
  children: React.ReactNode;
  className?: string;
}) {
  // pending 为 true 时，表示 form 正在提交中（Server Action 还没返回）
  const { pending } = useFormStatus();

  return (
    <button type="submit" disabled={pending} className={className}>
      {pending ? (
        <span className="flex items-center gap-2">
          {/* 简单的 loading 指示器 */}
          <span className="inline-block w-4 h-4 border-2 border-white border-t-transparent rounded-full animate-spin" />
          处理中...
        </span>
      ) : (
        children
      )}
    </button>
  );
}
```

在表单中使用：

```tsx
import { SubmitButton } from "@/components/submit-button";

// 在 form 内部使用 SubmitButton
<form action={formAction}>
  {/* ...表单字段... */}

  <SubmitButton className="bg-blue-600 text-white px-4 py-2 rounded-md">
    创建任务
  </SubmitButton>
</form>;
```

`useFormStatus` 还有一个有用的字段——`data`，它返回当前提交的 `FormData`，可以用来做提交前的确认提示。不过最常见的用法就是 `pending`——禁用按钮、显示 loading，防止重复提交。

到目前为止，我们已经实现了"创建任务"的完整流程：Server Action 处理逻辑、useFormState 展示结果、useFormStatus 显示加载状态。接下来看一个进阶场景。

## 乐观更新：让操作"零延迟"

假设用户在任务详情页点击"将状态改为进行中"——这个操作很轻量，只是改一个字段，但每次都要等服务端响应才能更新 UI。网络慢的时候，用户点了一下，等了两秒才看到状态变化，体验不太好。

**乐观更新**的思路是：先在 UI 上立即反映变化，不等服务端响应。如果服务端返回失败，再回滚。

React 19 提供了 `useOptimistic` Hook 来做这件事。我们用"更新任务状态"来演示。

```tsx
// app/workspace/[wsId]/projects/[projectId]/tasks/[taskId]/page.tsx
"use client";

import { useOptimistic, useState } from "react";
import { updateTaskStatus } from "@/app/actions/task-actions";

// 任务状态的类型定义
type TaskStatus = "todo" | "in_progress" | "in_review" | "done";

// 状态选项的展示配置
const STATUS_OPTIONS: {
  value: TaskStatus;
  label: string;
  color: string;
}[] = [
  { value: "todo", label: "待处理", color: "bg-gray-100 text-gray-700" },
  { value: "in_progress", label: "进行中", color: "bg-blue-100 text-blue-700" },
  {
    value: "in_review",
    label: "审核中",
    color: "bg-yellow-100 text-yellow-700",
  },
  { value: "done", label: "已完成", color: "bg-green-100 text-green-700" },
];

export function TaskStatusUpdater({
  taskId,
  currentStatus,
  workspaceId,
  projectId,
}: {
  taskId: string;
  currentStatus: TaskStatus;
  workspaceId: string;
  projectId: string;
}) {
  // useOptimistic 接收两个参数：
  // 1. 服务端的真实状态（currentStatus）
  // 2. 一个 updater 函数——接收 (当前状态, 乐观值)，返回新的临时状态
  const [optimisticStatus, setOptimisticStatus] = useOptimistic(
    currentStatus,
    (state, newStatus: TaskStatus) => newStatus
  );

  async function handleStatusChange(newStatus: TaskStatus) {
    // 第一步：立即更新 UI（乐观更新）
    setOptimisticStatus(newStatus);

    // 第二步：发送到服务端
    // 如果失败，React 会自动回滚到 currentStatus
    const result = await updateTaskStatus({
      taskId,
      status: newStatus,
      workspaceId,
      projectId,
    });

    if (!result.success) {
      // 服务端返回了错误——React 会自动回滚乐观状态，
      // 但我们还可以给用户一个额外的提示
      alert(result.error);
    }
  }

  // 获取当前状态对应的展示配置
  const statusConfig = STATUS_OPTIONS.find((s) => s.value === optimisticStatus);

  return (
    <div className="flex items-center gap-3">
      <span className="text-sm text-gray-500">状态：</span>
      {/* 用下拉菜单展示状态选项 */}
      <select
        value={optimisticStatus}
        onChange={(e) => handleStatusChange(e.target.value as TaskStatus)}
        className={`px-3 py-1 rounded-full text-sm font-medium ${
          statusConfig?.color || ""
        }`}
      >
        {STATUS_OPTIONS.map((option) => (
          <option key={option.value} value={option.value}>
            {option.label}
          </option>
        ))}
      </select>
    </div>
  );
}
```

对应的 Server Action：

```typescript
// app/actions/task-actions.ts 中新增
"use server";

import { db } from "@/lib/db";
import { revalidatePath } from "next/cache";

type TaskStatus = "todo" | "in_progress" | "in_review" | "done";

interface UpdateStatusContext {
  taskId: string;
  status: TaskStatus;
  workspaceId: string;
  projectId: string;
}

export async function updateTaskStatus(context: UpdateStatusContext) {
  // 状态转换校验：不是所有状态之间都可以直接切换
  // 例如：不能从 "todo" 直接跳到 "done"（需要经过 "in_progress"）
  const allowedTransitions: Record<TaskStatus, TaskStatus[]> = {
    todo: ["in_progress"],
    in_progress: ["in_review", "todo"],
    in_review: ["done", "in_progress"],
    done: ["in_review", "todo"],
  };

  const currentTask = await db.task.findUnique({
    where: { id: context.taskId },
    select: { status: true },
  });

  if (!currentTask) {
    return { success: false, error: "任务不存在" };
  }

  const allowed = allowedTransitions[currentTask.status as TaskStatus];
  if (!allowed.includes(context.status)) {
    return {
      success: false,
      error: `不能从「${
        STATUS_LABELS[currentTask.status as TaskStatus]
      }」直接切换到「${STATUS_LABELS[context.status]}」`,
    };
  }

  try {
    await db.task.update({
      where: { id: context.taskId },
      data: { status: context.status },
    });

    // 记录活动日志——谁在什么时候改了什么
    await db.activityLog.create({
      data: {
        taskId: context.taskId,
        action: "status_changed",
        fromValue: currentTask.status,
        toValue: context.status,
        // userId 从 session 中获取，这里简化
      },
    });

    revalidatePath(
      `/workspace/${context.workspaceId}/projects/${context.projectId}/tasks/${context.taskId}`
    );

    return { success: true };
  } catch (error) {
    console.error("更新任务状态失败:", error);
    return { success: false, error: "更新失败，请稍后重试" };
  }
}
```

乐观更新的适用场景：操作很轻量、失败概率低、用户需要即时反馈。比如点赞、改状态、拖拽排序。如果你在做一个"删除项目"的操作，最好还是等服务端确认后再更新 UI——毕竟删错了想恢复就麻烦了。

## Server Actions 的安全性

Server Actions 在服务端执行，所以你天然拥有服务端的安全能力（访问数据库、读取环境变量等）。但它也面临一些特有的安全风险，需要了解。

### 输入验证：永远不要信任 FormData

前面我们已经做了基本的验证，但实际项目中建议用 Zod 做结构化验证（下一篇会专门讲数据验证，这里先给个示例）：

```typescript
// lib/validations/task.ts
import { z } from "zod";

// 用 zod 定义验证 schema，一份代码同时提供类型推导和运行时校验
export const createTaskSchema = z.object({
  title: z
    .string()
    .min(1, "任务标题不能为空")
    .max(200, "标题不能超过 200 个字符")
    .trim(),
  description: z
    .string()
    .max(5000, "描述不能超过 5000 个字符")
    .trim()
    .optional(),
  priority: z.enum(["low", "medium", "high", "urgent"]).default("medium"),
});

// z.infer 会从 schema 自动推导出 TypeScript 类型
// 这样前后端共享同一个 schema，类型永远一致
export type CreateTaskInput = z.infer<typeof createTaskSchema>;
```

在 Server Action 中使用：

```typescript
import { createTaskSchema } from "@/lib/validations/task";

export async function createTask(
  context: CreateTaskContext,
  formData: FormData
) {
  const rawData = {
    title: formData.get("title"),
    description: formData.get("description"),
    priority: formData.get("priority"),
  };

  // safeParse 不会抛异常，而是返回 { success, data, error }
  const result = createTaskSchema.safeParse(rawData);

  if (!result.success) {
    // flatten() 把嵌套的错误结构拍平，方便展示
    const errors = result.error.flatten();
    return {
      success: false,
      error: "请检查输入",
      fieldErrors: errors.fieldErrors, // { title?: string[], description?: string[], ... }
    };
  }

  // result.data 已经是经过验证和转换的类型安全数据
  const task = await db.task.create({
    data: {
      ...result.data,
      status: "todo",
      projectId: context.projectId,
      workspaceId: context.workspaceId,
    },
  });

  // ...
}
```

### CSRF 防护

好消息是，Next.js 自动为 Server Actions 提供了 CSRF 防护。每次页面加载时，Next.js 会在服务端生成一个 token，嵌入到表单中。表单提交时，服务端会验证这个 token 是否匹配。这不是你需要在代码里处理的事情，但了解一下原理有助于排查问题。

### 认证和授权

Server Actions 中可以做权限检查。当前用户的信息通常通过 session 获取（认证的详细实现会在第 8 篇讲）：

```typescript
import { auth } from "@/lib/auth";

export async function createTask(
  context: CreateTaskContext,
  formData: FormData
) {
  // 第一步：确认用户已登录
  const session = await auth();
  if (!session?.user) {
    return { success: false, error: "请先登录" };
  }

  // 第二步：确认用户有权在该工作空间创建任务
  const membership = await db.workspaceMember.findUnique({
    where: {
      workspaceId_userId: {
        workspaceId: context.workspaceId,
        userId: session.user.id,
      },
    },
  });

  if (!membership) {
    return { success: false, error: "你不是该工作空间的成员" };
  }

  // 第三步：验证输入 + 执行业务逻辑
  // ...
}
```

### 速率限制

防止恶意用户疯狂调接口。可以用 Upstash Redis 做简单的速率限制：

```typescript
import { Ratelimit } from "@upstash/ratelimit";
import { Redis } from "@upstash/redis";

// 每个 IP 每分钟最多创建 10 个任务
const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(10, "1 m"),
});

export async function createTask(
  context: CreateTaskContext,
  formData: FormData
) {
  const session = await auth();
  if (!session?.user) {
    return { success: false, error: "请先登录" };
  }

  // 用 userId 做速率限制
  const { success, remaining } = await ratelimit.limit(
    `create-task:${session.user.id}`
  );

  if (!success) {
    return {
      success: false,
      error: `操作太频繁了，请 ${Math.ceil(60)} 秒后再试`,
    };
  }

  // remaining 可以返回给前端，用来显示"你还能创建 X 个任务"
  // ...
}
```

## 什么时候该用 API Route 而不是 Server Action

Server Actions 解决了"前端到后端的函数调用"问题，但它不是万能的。有些场景必须用 API Route（也叫 Route Handler）：

**需要被外部系统调用的接口**。比如你需要给第三方系统提供一个 Webhook 接口来同步任务状态——GitHub 的 PR 关联了某个任务，PR 合并后自动把任务标记为"已完成"。这种场景下，调用方是 GitHub 的服务器，不是你的前端页面，Server Actions 用不了。

**需要支持非表单的 HTTP 方法**。Server Actions 本质上都是 POST 请求。如果你需要提供 GET 接口（供前端轮询数据）、DELETE 接口（RESTful API 风格）、或者需要自定义响应头（比如下载文件时的 `Content-Disposition`），API Route 更合适。

**需要被多个客户端共享的接口**。你的任务管理平台未来可能要出一个移动端 App，或者一个 CLI 工具。这些客户端不通过浏览器访问你的页面，没法用 Server Actions，必须通过 REST API 来交互。

一个简单的判断标准：**如果这个接口只有你的前端页面在用，用 Server Actions；如果外部系统或其他客户端也需要调，用 API Route。** 下一篇我们会详细讲 API Route 的设计和使用场景。

## Trade-off 分析

Server Actions 带来了明显的开发体验提升，但了解它的局限也很重要。

**收益方面**。代码量大幅减少，一个表单功能从"API Route + fetch + 状态管理"三件套简化成一个函数。类型安全贯穿前后端，不用手动定义 API 的 request/response 类型。表单提交天然支持渐进增强——即使 JavaScript 被禁用，表单仍然可以提交（虽然用户体验会差一些）。CSRF 防护开箱即用。

**代价方面**。Server Actions 是 Next.js 特有的概念，如果你以后要把应用迁移到其他框架，需要改写成 API Route 或其他形式。调试不如 API Route 直观——API Route 你可以用 curl / Postman 直接测，Server Actions 的调用链路更难追踪。Server Actions 没有标准的 OpenAPI 文档生成，团队协作时接口文档需要额外维护。对于复杂的 API（分页、过滤、排序），Server Actions 的参数组织不如 REST API 的 query string 直观。

总体来说，对于"自己的前端页面调自己的后端"这个场景，Server Actions 是更好的选择。但对于需要被外部系统调用的接口，API Route 不可替代。实际项目中两者通常是共存的。
