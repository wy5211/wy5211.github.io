## 一个让人后怕的场景

上篇文章我们优化了任务管理平台的性能——加了代码分割、图片懒加载、缓存策略。性能数据确实好看了，但改完之后你心里有没有一丝不安？这些优化有没有可能破坏已有的功能？

我遇到过一次线上事故：为了优化列表页的加载速度，我改了 Server Action 里查询任务的数据结构，把 `include` 的关联表从 eager loading 改成了按需加载。本地手动测了一遍，没问题，直接上线了。结果用户反馈"任务详情页的创建者信息全部丢失了"——因为关联数据没查出来，前端拿到的 `creator` 字段是 `null`。

那次事故让我彻底明白了一件事：**手动测试覆盖不了所有路径，自动化测试才是兜底的最后一道防线**。

问题来了，Next.js 应该怎么测？Server Actions 跑在服务端、Server Components 在服务端渲染、Client Components 在浏览器交互——这三层的东西混在一起，测试策略跟纯前端项目完全不一样。今天我们就围绕任务管理平台，把测试体系从里到外搭起来。

## 测试分层：先搞清楚该在哪层投入精力

在写任何一行测试代码之前，先明确一件事：不是所有代码都值得写测试。测试本身也是代码，也需要维护，如果投入产出比不对，就是在浪费时间。

我常用的分层策略是这样的：

**第一层：Server Actions 单元测试（投入最多）**

Server Actions 是业务逻辑的核心——创建任务、更新状态、分配成员、添加评论。这些逻辑一旦出错，直接影响数据正确性。而且 Server Actions 是纯函数，输入 FormData 或参数，返回结果，非常好测。我们把 60% 的测试精力放在这一层。

**第二层：组件集成测试（投入中等）**

验证组件之间的交互是否正确——比如提交任务表单后，列表是否更新了；点击筛选按钮后，是否只显示了对应状态的任务。这一层测的是"行为"，不是"渲染细节"。占 30% 的精力。

**第三层：E2E 测试（投入最少但覆盖面最广）**

模拟真实用户操作整个流程——登录、创建项目、添加任务、修改状态。这些测试跑得慢，但能捕获跨层集成的问题。占 10% 的精力，只覆盖最核心的用户路径。

## 测试环境搭建

先把工具链配好。我们的选择是：Vitest 做单元测试和组件测试，Playwright 做 E2E 测试。

```bash
# 单元测试和组件测试
npm install -D vitest @testing-library/react @testing-library/jest-dom
npm install -D @testing-library/user-event @vitejs/plugin-react jsdom

# E2E 测试
npm install -D @playwright/test
```

Vitest 的配置需要注意，Next.js 项目的路径别名（`@/`）需要在测试环境里也生效：

```typescript
// vitest.config.ts
import { defineConfig } from "vitest/config";
import react from "@vitejs/plugin-react";
import path from "path";

export default defineConfig({
  // 添加 React 插件，让 Vitest 能处理 JSX
  plugins: [react()],
  resolve: {
    alias: {
      // 让 @/ 别名在测试中也能正常工作
      "@": path.resolve(__dirname, "./src"),
    },
  },
  test: {
    // 使用 jsdom 模拟浏览器环境
    environment: "jsdom",
    // 全局导入测试 API，不用每个文件都 import { describe, it, expect }
    globals: true,
    // 测试文件匹配规则
    include: ["**/*.test.{ts,tsx}"],
    // 排除 e2e 目录，Playwright 有自己的运行器
    exclude: ["node_modules", "e2e"],
    // 启动前执行的设置文件
    setupFiles: ["./vitest.setup.ts"],
  },
});
```

设置文件负责引入断言扩展和清理副作用：

```typescript
// vitest.setup.ts
// 引入 jest-dom 的断言扩展，比如 toBeInTheDocument、toHaveTextContent 等
import "@testing-library/jest-dom";

// 每个测试用例跑完后自动清理 React 组件的渲染残留
// 如果不清理，上一次渲染的 DOM 可能影响下一次测试
import { cleanup } from "@testing-library/react";
import { afterEach } from "vitest";

afterEach(() => {
  cleanup();
});
```

Playwright 的配置也不复杂，关键是让它在跑测试前自动启动开发服务器：

```typescript
// playwright.config.ts
import { defineConfig, devices } from "@playwright/test";

export default defineConfig({
  // E2E 测试文件放在 e2e 目录下
  testDir: "./e2e",
  // 测试并行执行，加快速度
  fullyParallel: true,
  // CI 环境下禁止 only 跳过其他测试
  forbidOnly: !!process.env.CI,
  // CI 环境下失败自动重试 2 次（CI 环境偶发失败很常见）
  retries: process.env.CI ? 2 : 0,
  // CI 环境下只用 1 个 worker，避免资源竞争
  workers: process.env.CI ? 1 : undefined,
  reporter: "html",
  use: {
    baseURL: "http://localhost:3000",
    // 第一次重试时自动录制 trace，方便排查
    trace: "on-first-retry",
  },
  projects: [
    {
      name: "chromium",
      use: { ...devices["Desktop Chrome"] },
    },
    // 生产环境通常只测 Chrome，Firefox 和 Safari 可以在需要时开启
  ],
  // 自动启动开发服务器，测试跑完自动关闭
  webServer: {
    command: "npm run dev",
    url: "http://localhost:3000",
    reuseExistingServer: !process.env.CI,
  },
});
```

在 `package.json` 里加上测试脚本：

```json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run",
    "test:ui": "vitest --ui",
    "test:e2e": "playwright test",
    "test:e2e:ui": "playwright test --ui"
  }
}
```

## 测试数据管理：先搞定"造数据"这件事

写测试的第一步不是写断言，而是想清楚：测试数据从哪来？如果每个测试文件里都硬编码一堆 mock 数据，改一个字段就要改十几个文件。

我用工厂模式来管理测试数据。先定义基础数据，每个测试通过 `overrides` 覆盖需要的字段：

## Server Actions 测试：Mock 掉数据库，专注业务逻辑

Server Actions 是我们测试投入最大的地方。它们包含核心的业务规则和数据操作，而且测试起来相对简单——只需要 mock 掉数据库，验证输入输出和行为。

先看一个实际的 Server Action，这是创建任务的核心逻辑：

```typescript
// app/workspace/[workspaceId]/projects/[projectId]/actions.ts
"use server";

import { db } from "@/lib/db";
import { currentUser } from "@/lib/auth";
import { revalidatePath } from "next/cache";
import { z } from "zod";

// 验证规则：标题不能为空且不超过 200 字符，优先级只能是这四个值
const createTaskSchema = z.object({
  title: z.string().min(1).max(200),
  description: z.string().max(5000).optional(),
  priority: z.enum(["low", "medium", "high", "urgent"]),
  assigneeId: z.string().optional(),
  dueDate: z.string().optional(),
});

export async function createTask(
  projectId: string,
  prevState: { error?: string; success?: boolean },
  formData: FormData
) {
  // 1. 验证用户登录状态
  const user = await currentUser();
  if (!user) {
    return { error: "请先登录" };
  }

  // 2. 从 FormData 提取并验证数据
  const rawData = {
    title: formData.get("title") as string,
    description: formData.get("description") as string,
    priority: formData.get("priority") as string,
    assigneeId: formData.get("assigneeId") as string,
    dueDate: formData.get("dueDate") as string,
  };

  const result = createTaskSchema.safeParse(rawData);
  if (!result.success) {
    return { error: result.error.issues[0].message };
  }

  // 3. 验证用户是否有权限在当前项目创建任务
  const membership = await db.projectMember.findFirst({
    where: {
      projectId,
      userId: user.id,
    },
  });

  if (!membership) {
    return { error: "你没有权限在此项目中创建任务" };
  }

  // 4. 创建任务
  const task = await db.task.create({
    data: {
      ...result.data,
      projectId,
      creatorId: user.id,
      status: "todo",
    },
  });

  // 5. 刷新缓存，让项目页能看到新任务
  revalidatePath(`/workspace/${projectId}`);

  return { success: true, taskId: task.id };
}
```

这个 Action 有五个关键行为需要测试：未登录拦截、数据验证、权限检查、任务创建、缓存刷新。来看测试：

```typescript
// app/workspace/[workspaceId]/projects/[projectId]/actions.test.ts
import { describe, it, expect, vi, beforeEach } from "vitest";
import { createTask } from "./actions";
import { db } from "@/lib/db";
import { currentUser } from "@/lib/auth";

// mock 掉数据库模块，避免测试真的连数据库
vi.mock("@/lib/db", () => ({
  db: {
    projectMember: {
      findFirst: vi.fn(),
    },
    task: {
      create: vi.fn(),
    },
  },
}));

// mock 掉认证模块，控制用户登录状态
vi.mock("@/lib/auth", () => ({
  currentUser: vi.fn(),
}));

// mock 掉 next/cache，避免 revalidatePath 在测试环境报错
vi.mock("next/cache", () => ({
  revalidatePath: vi.fn(),
}));

describe("createTask Server Action", () => {
  beforeEach(() => {
    // 每个测试用例开始前清空所有 mock 的调用记录
    vi.clearAllMocks();
  });

  // 构造 FormData 的辅助方法，避免每个测试都重复写
  function buildFormData(overrides: Record<string, string> = {}) {
    const formData = new FormData();
    formData.set("title", "实现用户权限管理模块");
    formData.set("description", "支持管理员和普通成员两种角色");
    formData.set("priority", "high");
    formData.set("assigneeId", "user_002");
    formData.set("dueDate", "2026-04-15");
    // 允许调用方覆盖任意字段
    for (const [key, value] of Object.entries(overrides)) {
      formData.set(key, value);
    }
    return formData;
  }

  it("未登录时返回错误提示", async () => {
    // 模拟用户未登录
    vi.mocked(currentUser).mockResolvedValue(null);

    const result = await createTask("proj_001", {}, buildFormData());

    expect(result.error).toBe("请先登录");
    // 未登录时不应查询数据库
    expect(db.projectMember.findFirst).not.toHaveBeenCalled();
    expect(db.task.create).not.toHaveBeenCalled();
  });

  it("标题为空时返回验证错误", async () => {
    vi.mocked(currentUser).mockResolvedValue(createUser());

    const result = await createTask(
      "proj_001",
      {},
      buildFormData({ title: "" })
    );

    expect(result.error).toBeTruthy();
    // 验证失败就不该查数据库
    expect(db.projectMember.findFirst).not.toHaveBeenCalled();
  });

  it("优先级不合法时返回验证错误", async () => {
    vi.mocked(currentUser).mockResolvedValue(createUser());

    const result = await createTask(
      "proj_001",
      {},
      buildFormData({ priority: "super_urgent" })
    );

    expect(result.error).toBeTruthy();
    expect(db.task.create).not.toHaveBeenCalled();
  });

  it("非项目成员创建任务时返回权限错误", async () => {
    vi.mocked(currentUser).mockResolvedValue(createUser());
    // 模拟用户不是项目成员
    vi.mocked(db.projectMember.findFirst).mockResolvedValue(null);

    const result = await createTask("proj_001", {}, buildFormData());

    expect(result.error).toBe("你没有权限在此项目中创建任务");
    expect(db.task.create).not.toHaveBeenCalled();
  });

  it("项目成员创建任务成功", async () => {
    const user = createUser({ id: "user_001" });
    vi.mocked(currentUser).mockResolvedValue(user);
    // 模拟用户是项目成员
    vi.mocked(db.projectMember.findFirst).mockResolvedValue({
      id: "pm_001",
      projectId: "proj_001",
      userId: "user_001",
      role: "member",
    });
    // 模拟数据库创建任务成功
    vi.mocked(db.task.create).mockResolvedValue(
      createTask({ id: "task_new_001", creatorId: "user_001" })
    );

    const result = await createTask("proj_001", {}, buildFormData());

    expect(result.success).toBe(true);
    expect(result.taskId).toBe("task_new_001");

    // 验证数据库调用参数是否正确
    expect(db.task.create).toHaveBeenCalledWith({
      data: expect.objectContaining({
        title: "实现用户权限管理模块",
        priority: "high",
        projectId: "proj_001",
        creatorId: "user_001",
        status: "todo",
      }),
    });
  });

  it("创建成功后刷新项目页缓存", async () => {
    const user = createUser();
    vi.mocked(currentUser).mockResolvedValue(user);
    vi.mocked(db.projectMember.findFirst).mockResolvedValue({
      id: "pm_001",
      projectId: "proj_001",
      userId: user.id,
      role: "member",
    });
    vi.mocked(db.task.create).mockResolvedValue(createTask());

    await createTask("proj_001", {}, buildFormData());

    // 导入被 mock 的 revalidatePath，验证是否被正确调用
    const { revalidatePath } = await import("next/cache");
    expect(revalidatePath).toHaveBeenCalledWith("/workspace/proj_001");
  });
});
```

注意几个细节：每个测试只验证一个行为；mock 数据库调用时用的是 `expect.objectContaining`，只断言关心的字段，不关心 `createdAt` 这种由数据库自动生成的字段；`buildFormData` 辅助方法让测试数据构造变得简洁。

再测一个更复杂的场景——任务状态流转。任务管理平台里，状态的变更不是随意的，比如 `done` 不能回退到 `todo`，只有 `in_progress` 才能变成 `done`：

```typescript
// lib/task-state-machine.test.ts
import { describe, it, expect } from "vitest";
import { canTransitionTo } from "./task-state-machine";

// 任务状态机：定义哪些状态可以流转到哪些状态
// 这是纯函数测试，不需要 mock 任何东西，跑起来非常快
describe("canTransitionTo 状态流转校验", () => {
  const validTransitions = {
    todo: ["in_progress"],
    in_progress: ["done", "todo"],
    done: ["in_progress"],
  };

  it("新任务可以从 todo 切换到 in_progress", () => {
    expect(canTransitionTo("todo", "in_progress", validTransitions)).toBe(true);
  });

  it("新任务不能直接标记为 done", () => {
    expect(canTransitionTo("todo", "done", validTransitions)).toBe(false);
  });

  it("已完成任务可以重新打开为 in_progress", () => {
    expect(canTransitionTo("done", "in_progress", validTransitions)).toBe(true);
  });

  it("已完成任务不能直接回退到 todo", () => {
    expect(canTransitionTo("done", "todo", validTransitions)).toBe(false);
  });

  it("相同状态之间不触发流转", () => {
    expect(
      canTransitionTo("in_progress", "in_progress", validTransitions)
    ).toBe(false);
  });

  it("非法的当前状态返回 false", () => {
    expect(canTransitionTo("cancelled", "todo", validTransitions)).toBe(false);
  });
});
```

这种纯逻辑测试是投入产出比最高的——不需要 mock，跑得快，一旦出错就能精确定位问题。

## 组件测试：测行为，不测实现细节

组件测试的目标是验证"用户看到的和操作的结果是否正确"，而不是"组件内部用什么标签、什么 className"。Testing Library 的设计哲学就是：**用用户能感知的方式来查询 DOM**。

来看任务列表的核心组件：

```typescript
// components/task-list.tsx
"use client";

import { useState } from "react";
import { Task } from "@/lib/types";
import { updateTaskStatus } from "@/app/workspace/actions";

// 任务列表组件：展示任务、支持按状态筛选、支持拖拽修改状态
export function TaskList({ tasks }: { tasks: Task[] }) {
  const [filter, setFilter] = useState<string>("all");

  // 根据筛选条件过滤任务
  const filteredTasks =
    filter === "all"
      ? tasks
      : tasks.filter((task) => task.status === filter);

  // 按优先级排序：urgent > high > medium > low
  const priorityOrder = { urgent: 0, high: 1, medium: 2, low: 3 };
  const sortedTasks = [...filteredTasks].sort(
    (a, b) => priorityOrder[a.priority] - priorityOrder[b.priority]
  );

  return (
    <div>
      {/* 状态筛选按钮组 */}
      <div role="tablist" aria-label="按状态筛选任务">
        {[
          { key: "all", label: "全部" },
          { key: "todo", label: "待处理" },
          { key: "in_progress", label: "进行中" },
          { key: "done", label: "已完成" },
        ].map((tab) => (
          <button
            key={tab.key}
            role="tab"
            aria-selected={filter === tab.key}
            onClick={() => setFilter(tab.key)}
          >
            {tab.label}
          </button>
        ))}
      </div>

      {/* 任务列表 */}
      {sortedTasks.length === 0 ? (
        <p>没有匹配的任务</p>
      ) : (
        <ul>
          {sortedTasks.map((task) => (
            <li key={task.id} data-testid={`task-${task.id}`}>
              <span>{task.title}</span>
              <span>{task.priority}</span>
              <span>{task.status}</span>
              {task.dueDate && (
                <time dateTime={task.dueDate.toISOString()}>
                  截止: {task.dueDate.toLocaleDateString("zh-CN")}
                </time>
              )}
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

对应的测试：

```typescript
// components/task-list.test.tsx
import { describe, it, expect } from "vitest";
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { TaskList } from "./task-list";
import { createTask } from "@/test/factories";

describe("TaskList 组件", () => {
  // 构造测试用的任务列表，覆盖不同状态和优先级
  const tasks = [
    createTask({ id: "task_001", title: "修复登录 bug", status: "todo", priority: "urgent" }),
    createTask({ id: "task_002", title: "实现权限管理", status: "in_progress", priority: "high" }),
    createTask({ id: "task_003", title: "优化首页加载", status: "done", priority: "medium" }),

  it("默认显示全部任务", () => {

    // 用 getByText 查找——这是用户能看到的内容
    expect(screen.getByText("修复登录 bug")).toBeInTheDocument();

  it("紧急任务排在最前面", () => {
    render(<TaskList tasks={tasks} />);

    // 用 getAllByRole 获取所有任务项，检查顺序
    const items = screen.getAllByRole("listitem");
    // 第一条应该是 urgent 优先级的"修复登录 bug"
    expect(items[0]).toHaveTextContent("修复登录 bug");
    // 最后一条应该是 low 优先级的"补充单元测试"

    render(<TaskList tasks={tasks} />);
```

注意测试的写法：我们用的是 `getByText("修复登录 bug")` 而不是 `getByTestId("task-001")`——因为用户看到的是任务标题，不是 data-testid。如果将来组件重构，把 `<li>` 改成 `<div>`，用 `getByTestId` 的测试不会断，但用 `getByText` 的测试也不会断，因为用户关心的内容没变。

再测一个涉及表单提交的组件——创建任务的对话框：

```typescript
// components/create-task-dialog.test.tsx
import { describe, it, expect, vi } from "vitest";

import userEvent from "@testing-library/user-event";

    await user.click(screen.getByRole("button", { name: /确认/ }));

    await waitFor(() => {
      expect(screen.getByText(/标题不能为空/)).toBeInTheDocument();

  });

  it("填写完整信息后成功提交", async () => {

```

Server Components 怎么测？实际上 Server Components 在服务端渲染成 HTML，我们测的重点是它的渲染输出。但 Server Components 不能直接在 Vitest 的 jsdom 环境里跑（它们依赖 Next.js 的服务端运行时），所以一般通过 E2E 测试来覆盖，或者把数据获取逻辑抽成独立的函数来测。不要强行在 jsdom 里测 Server Components，投入产出比太低。

## E2E 测试：模拟真实用户的操作流程

E2E 测试是最接近真实使用场景的测试——启动完整的应用，打开浏览器，像用户一样操作。Playwright 是目前最好的 E2E 测试工具，速度快、API 设计好、调试体验优秀。

先看最核心的用户流程：登录后创建一个任务。

```typescript
// e2e/create-task.spec.ts
import { test, expect } from "@playwright/test";

// 登录状态复用：在所有测试前执行一次登录
// Playwright 会保存 cookies，后续测试自动保持登录状态
test.beforeEach(async ({ page }) => {
  // 直接访问登录页
  await page.goto("/login");

  // 填写登录表单
  await page.fill('input[name="email"]', "zhangming@example.com");
  await page.fill('input[name="password"]', "TestPass123!");

  // 提交登录
  await page.click('button[type="submit"]');

  // 等待跳转到 dashboard，确认登录成功
  await page.waitForURL("/dashboard");
});

test("创建任务的完整流程", async ({ page }) => {
  // 1. 进入项目页面
  await page.goto("/workspace/ws_001/projects/proj_001");

  // 验证页面标题包含了项目名
  await expect(page.locator("h1")).toContainText("Q2 产品迭代");

  // 2. 点击创建任务按钮
  await page.click('button:has-text("创建任务")');

  // 等待对话框出现
  await expect(page.locator('[role="dialog"]')).toBeVisible();

  // 3. 填写任务信息
  await page.fill('input[name="title"]', "实现用户权限管理模块");
  await page.fill(
    'textarea[name="description"]',
    "支持管理员和普通成员两种角色，管理员可以删除项目"
  );
  await page.selectOption('select[name="priority"]', "high");
  await page.fill('input[name="dueDate"]', "2026-04-15");

  // 4. 提交表单
  await page.click('button:has-text("确认创建")');

  // 5. 验证任务创建成功
  // 对话框应该关闭
  await expect(page.locator('[role="dialog"]')).not.toBeVisible();

  // 新任务应该出现在列表中
  await expect(page.locator("text=实现用户权限管理模块")).toBeVisible();

  // 任务状态应该默认为"待处理"
  const newTask = page.locator("text=实现用户权限管理模块").locator("..");
  await expect(newTask.locator("text=待处理")).toBeVisible();
});

test("按状态筛选任务", async ({ page }) => {
  await page.goto("/workspace/ws_001/projects/proj_001");

  // 先确认有多个任务
  const allTasks = page.locator("ul li");
  const initialCount = await allTasks.count();
  expect(initialCount).toBeGreaterThan(0);

  // 点击"进行中"筛选
  await page.click('button:has-text("进行中")');

  // 等待列表更新
  await page.waitForTimeout(500);

  // 验证所有可见任务的状态都是"进行中"
  const visibleTasks = page.locator("ul li");
  const count = await visibleTasks.count();
  for (let i = 0; i < count; i++) {
    await expect(visibleTasks.nth(i)).toContainText("进行中");
  }
});

test("未登录用户访问项目页面被重定向到登录页", async ({ browser }) => {
  // 创建一个全新的浏览器上下文，不使用 beforeEach 中保存的登录状态
  const context = await browser.newContext();
  const page = await context.newPage();

  await page.goto("/workspace/ws_001/projects/proj_001");

  // 应该被重定向到登录页
  await expect(page).toHaveURL(/\/login/);

  await context.close();
});
```

E2E 测试的几个实践要点：`beforeEach` 里复用登录状态，避免每个测试都重复登录；用 `waitForURL` 而不是 `setTimeout` 来等待页面跳转；创建新浏览器上下文来测试未登录场景。

## 什么值得测，什么不值得测

写了一段时间测试之后，你会发现一个规律：有些测试几乎从不失败，有些测试三天两头变红。关键在于搞清楚投入产出比。

**值得重点投入测试的：**

- Server Actions 中的业务规则和状态流转。这是最容易出 bug、出 bug 影响最大的地方。上文的状态机测试就是典型例子。
- 涉及金钱、权限、数据一致性的逻辑。比如"只有项目管理员才能删除项目"这个规则，如果不测，上线后出问题就是安全事故。
- 核心用户流程的 E2E 测试。不需要覆盖所有路径，只覆盖用户最常用的 3-5 个流程就够用了。

**不值得花太多精力的：**

- 纯展示型 Server Components 的渲染细节。标题用 h1 还是 h2、卡片有几个 div，这些是样式问题，不是功能问题。
- 简单的 getter/setter、类型转换函数。比如把日期格式化成字符串这种逻辑，出错了用户一眼就能看到，而且修复成本极低。
- 第三方库的行为。比如 `z.string().min(1)` 能不能正确验证空字符串——这是 Zod 的事，不是你的事。
- 样式和布局。视觉回归应该交给截图对比工具（比如 Percy），不要用断言去验证 `margin` 和 `padding`。

简单说，**测试的是"业务逻辑会出错的地方"，不是"代码可能写错的地方"**。

## CI 集成：让测试自动跑起来

测试写好了，但不能只在本地跑。每次提交代码、每次合并 PR，都应该自动跑一遍测试。GitHub Actions 是最常用的 CI 工具，配置起来也不复杂。

```yaml
# .github/workflows/test.yml
name: Test

# 在 push 到 main 分支和创建 PR 时触发
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

# 并行跑两个 job：单元测试和 E2E 测试
jobs:
  unit-test:
    name: 单元测试和组件测试
    runs-on: ubuntu-latest
    steps:
      # 1. 拉取代码
      - uses: actions/checkout@v4

      # 2. 安装 Node.js
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          # 开启缓存，加速依赖安装
          cache: "npm"

      # 3. 安装依赖
      - run: npm ci

      # 4. 运行单元测试和组件测试
      # --reporter=verbose 输出详细的测试结果
      # --coverage 生成覆盖率报告
      - run: npm run test:run -- --reporter=verbose

  e2e-test:
    name: E2E 测试
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci

      # 安装 Playwright 的浏览器
      - run: npx playwright install --with-deps chromium

      # 运行 E2E 测试
      - run: npm run test:e2e

      # 测试失败时上传 trace 文件，方便在本地调试
      - uses: actions/upload-artifact@v4
        if: ${{ failure() }}
        with:
          name: playwright-trace
          path: test-results/
          retention-days: 7
```

这个配置做了几件事：push 和 PR 自动触发测试；单元测试和 E2E 测试并行跑，节省时间；E2E 测试只跑 Chrome（生产环境主要用户用的是 Chrome）；测试失败时上传 Playwright 的 trace 文件，你可以在本地用 `npx playwright show-trace` 回放失败的步骤。

## 权衡与取舍

测试不是免费的。每一行测试代码都需要维护，mock 数据会过时，测试环境会跟生产环境产生偏差。在实际项目中，你需要做几个权衡：

**测试覆盖率的追求。** 我见过团队追求 100% 覆盖率，结果写了大量测试去验证 getter 和类型转换，这些测试几乎从不失败，但维护成本很高。我建议把精力放在业务逻辑密集的地方——Server Actions 的测试覆盖率做到 80% 以上，组件测试覆盖核心交互，E2E 测试覆盖 3-5 个主流程就够了。

**mock 的度。** mock 太多会让测试跟实际代码产生偏差——你测的是"mock 的世界"而不是"真实的世界"。比如你 mock 了数据库，但忘了 mock 数据库返回 `null` 的情况，上线后数据库异常就暴露了。我的做法是：Server Actions 测试可以 mock 数据库，但 E2E 测试一定要用真实的数据库（可以用 Docker 起一个测试实例）。

**测试速度。** 单元测试应该毫秒级完成，如果一个测试需要超过 1 秒，说明有外部依赖没有 mock 好。组件测试控制在几百毫秒。E2E 测试比较慢（一个流程可能需要几秒），所以只保留最核心的流程。整个测试套件的执行时间控制在 3 分钟以内是比较理想的，超过 5 分钟开发者就会倾向于跳过测试。

**测试的维护成本。** 当你重构代码时，测试也会跟着变。如果一个测试因为你改了组件内部实现而失败，但它验证的行为并没有变，说明这个测试耦合了实现细节——应该重写。好的测试应该在重构后依然能通过，只有当业务行为真的改变时才需要修改测试。

## 小结

这一篇我们从"改了 Server Action 怕出问题"出发，为任务管理平台搭建了三层测试体系：

- **Server Actions 单元测试**：mock 数据库和认证，验证业务规则、数据验证、权限控制、状态流转。这是投入产出比最高的一层。
- **组件集成测试**：用 Testing Library 测用户行为——筛选、表单提交、列表排序。测行为不测实现。
- **E2E 测试**：用 Playwright 模拟真实用户操作核心流程。数量少但覆盖面广。
- **CI 集成**：通过 GitHub Actions 自动跑测试，PR 合并前自动把关。

到目前为止，我们的任务管理平台功能完整、性能优化到位、测试兜底也做好了。但代码在本地跑得好好的，部署到线上又是一回事——环境变量怎么配、数据库怎么连、域名怎么绑、出了问题怎么监控。下一篇我们就来解决这个问题：把项目安全地推到生产环境。
