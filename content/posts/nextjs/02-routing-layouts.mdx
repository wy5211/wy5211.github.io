上一篇我们用 `create-next-app` 搭好了项目，跑通了第一个页面。你可能注意到 `app/` 目录下的文件结构直接对应了 URL 路径——`about/page.tsx` 就是 `/about`。这看起来很简单，但如果你之前用过 React Router，可能会好奇：嵌套路由怎么办？路由守卫呢？多个页面共享的导航栏和侧边栏呢？别急，Next.js 的路由系统比看起来强大得多。

## 先说为什么：React Router 的配置之痛

假设你在维护一个在线课程平台，用 React Router v6 管理路由。随着功能增长，你的路由配置文件变成了这样：

```tsx
// 传统 React Router 配置 —— 随着项目增长越来越难维护
const router = createBrowserRouter([
  {
    path: "/",
    element: <MarketingLayout />, // 营销页面布局
    children: [
      { index: true, element: <HomePage /> },
      { path: "courses", element: <CourseListPage /> },
      { path: "courses/:courseId", element: <CourseDetailPage /> },
      { path: "pricing", element: <PricingPage /> },
    ],
  },
  {
    path: "/dashboard",
    element: <DashboardLayout />, // 仪表板布局
    loader: requireAuthLoader, // 路由守卫
    children: [
      { index: true, element: <OverviewPage /> },
      { path: "courses", element: <MyCoursesPage /> },
      { path: "settings", element: <SettingsPage /> },
    ],
  },
  {
    path: "/courses/:courseId/lessons/:lessonId",
    element: <LessonPlayerLayout />, // 又一套布局
    children: [
      { index: true, element: <LessonContent /> },
      { path: "notes", element: <LessonNotes /> },
    ],
  },
]);
```

这个配置有几个痛点：第一，路由和 UI 组件分离在两个地方，加一个页面要改配置文件再建组件文件；第二，嵌套布局的层级关系不够直观，你得顺着 children 往下找才知道某个页面套了哪些布局；第三，loading 状态和错误处理要自己用 Suspense 和 ErrorBoundary 包，每个路由都要重复写。

Next.js 的文件系统路由要解决的就是这些问题——把路由结构从配置文件搬到目录结构里，布局、loading、error 全部通过约定式文件自动处理。下面我们用这个在线课程平台作为案例，逐步看看怎么实现。

## 文件系统路由：文件夹即路由

Next.js 的核心思路是：`app/` 目录下的每个文件夹对应 URL 的一段路径，`page.tsx` 文件就是该路径下的页面。没有 `page.tsx` 的文件夹不会生成路由——这个细节很重要，后面讲到路由组时会用到。

先看看课程平台的目录结构长什么样：

```
app/
├── layout.tsx                        # 根布局（全站共享）
├── page.tsx                          # / → 首页
├── courses/
│   ├── page.tsx                      # /courses → 课程列表
│   ├── [courseId]/                   # 动态路由段
│   │   ├── page.tsx                  # /courses/react-18 → 课程详情
│   │   └── lessons/
│   │       └── [lessonId]/
│   │           └── page.tsx          # /courses/react-18/lessons/hooks → 课时内容
│   └── layout.tsx                    # 课程模块的共享布局（侧边栏目录等）
├── dashboard/
│   ├── page.tsx                      # /dashboard → 学习仪表板
│   ├── settings/
│   │   └── page.tsx                  # /dashboard/settings → 账号设置
│   └── layout.tsx                    # 仪表板布局（侧边导航等）
├── pricing/
│   └── page.tsx                      # /pricing → 定价页
├── not-found.tsx                     # 全局 404
├── loading.tsx                       # 全局 loading
└── error.tsx                         # 全局错误边界
```

对比一下 React Router 的配置，你会发现文件结构本身就是路由配置——层级关系一目了然，加新页面只需要建文件夹和 `page.tsx`。除了 `page.tsx`，Next.js 还有一组特殊文件约定：

| 文件名          | 作用                                      |
| --------------- | ----------------------------------------- |
| `page.tsx`      | 该路由的页面 UI，必须有这个文件路由才生效 |
| `layout.tsx`    | 该路由段及其子路由共享的布局              |
| `loading.tsx`   | 该路由段的 Suspense fallback，自动包裹    |
| `error.tsx`     | 该路由段的错误边界，自动包裹              |
| `not-found.tsx` | 该路由段的 404 页面                       |
| `route.ts`      | API 端点（不返回 UI，返回 JSON 等数据）   |

注意路由的匹配优先级：预定义路由（`courses/new`）优先于动态路由（`courses/[courseId]`），动态路由又优先于捕获所有路由（`[...slug]`）。这意味着如果你有 `courses/new/page.tsx` 和 `courses/[courseId]/page.tsx`，访问 `/courses/new` 会命中前者。

## 动态路由：`[slug]` 和 `[...slug]`

课程详情页不可能为每门课单独建一个文件夹——课程可能有上千门。这时就需要动态路由，用方括号 `[]` 包裹的文件夹名会作为参数传入组件。

### 单段动态路由

最常见的场景，`[courseId]` 匹配 URL 中的这一段：

```tsx
// app/courses/[courseId]/page.tsx
import { notFound } from "next/navigation";

// 定义 props 类型，params 中的键名和文件夹名一致
type CourseDetailPageProps = {
  params: Promise<{ courseId: string }>; // Next.js 15 中 params 是 Promise
};

export default async function CourseDetailPage({
  params,
}: CourseDetailPageProps) {
  const { courseId } = await params; // 需要 await 解包

  // 从数据库或 API 获取课程数据
  // courseId 可能是 "react-18"、"typescript-5" 这样的 slug
  const course = await fetchCourseById(courseId);

  if (!course) {
    // 课程不存在时，触发最近的 not-found.tsx
    notFound();
  }

  return (
    <div>
      <h1>{course.title}</h1>
      <p>{course.description}</p>
      <p>讲师：{course.instructor.name}</p>
      <p>学员数：{course.enrollmentCount.toLocaleString()}</p>
    </div>
  );
}
```

上面代码里有两个值得注意的点：第一，Next.js 15 把 `params` 改成了 `Promise`，所以需要 `await`；第二，找不到数据时调用 `notFound()` 而不是自己返回 404 页面——这样会自动触发最近层级（往上找）的 `not-found.tsx`，保持错误处理的一致性。

### 多段动态路由

嵌套的动态路由也很常见，比如课程下的某个课时：

```tsx
// app/courses/[courseId]/lessons/[lessonId]/page.tsx
type LessonPageProps = {
  params: Promise<{ courseId: string; lessonId: string }>;
};

export default async function LessonPage({ params }: LessonPageProps) {
  const { courseId, lessonId } = await params;

  // 同时用两个动态参数获取课时数据
  const lesson = await fetchLesson(courseId, lessonId);

  return (
    <div>
      <nav>
        {/* 面包屑导航：课程 → 当前课时 */}
        <a href={`/courses/${courseId}`}>返回课程</a>
      </nav>
      <h1>{lesson.title}</h1>
      <video src={lesson.videoUrl} controls />
    </div>
  );
}
```

### 捕获所有路由 `[...slug]`

有时候你需要匹配多级路径，比如文档系统的任意嵌套路径。用 `[...slug]` 可以把剩余的路径段全部捕获为一个数组：

```tsx
// app/docs/[...slug]/page.tsx
type DocsPageProps = {
  params: Promise<{ slug: string[] }>;
};

export default async function DocsPage({ params }: DocsPageProps) {
  const { slug } = await params;
  // slug 是一个数组，把路径段拼起来用来查找文档
  // 例如 /docs/getting-started/installation → slug = ["getting-started", "installation"]
  const docPath = slug.join("/");
  const doc = await fetchDocByPath(docPath);

  return (
    <article>
      <h1>{doc.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: doc.content }} />
    </article>
  );
}
```

如果用 `[[...slug]]`（双括号），路由段就变成可选的了——`/docs` 也能匹配，此时 `slug` 为 `undefined`。这个语法在博客分类场景下特别好用：`/blog` 显示所有文章，`/blog/tech` 显示技术分类，`/blog/tech/react` 显示子分类。

### 搜索参数 `searchParams`

除了路径参数，你还可以通过 `searchParams` 接收查询字符串：

```tsx
// app/courses/page.tsx
type CourseListPageProps = {
  searchParams: Promise<{
    page?: string;
    category?: string;
    level?: string;
  }>;
};

export default async function CourseListPage({
  searchParams,
}: CourseListPageProps) {
  const { page = "1", category, level } = await searchParams;

  // 把查询参数传给后端接口做过滤和分页
  const courses = await fetchCourses({
    page: Number(page),
    category,
    level,
  });

  return (
    <div>
      <h1>全部课程</h1>
      {/* 分页链接：/courses?page=2&category=frontend */}
      <nav>{/* 分页控件 */}</nav>
      <ul>
        {courses.map((course) => (
          <li key={course.id}>
            <a href={`/courses/${course.slug}`}>{course.title}</a>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

注意 `searchParams` 的值永远是 `string | string[] | undefined`，要做类型转换。上面用了 `Number(page)` 把字符串页码转成数字。

## 布局系统：layout.tsx 为什么不会重新渲染

布局是 Next.js 路由系统里最实用的特性之一。先理解一个核心概念：当你从 `/courses` 导航到 `/courses/react-18` 时，`courses/layout.tsx` **不会** 重新渲染，只有 `page.tsx` 会变化。这是 Next.js 的一个关键优化——布局在路由切换时保持状态，就像页面间的"骨架"一直存在。

### 嵌套布局的层级关系

布局自动嵌套。子路由的布局会包在父路由的布局里面，形成类似洋葱的结构：

```
访问 /courses/react-18 时的渲染层级：

app/layout.tsx          ← 根布局（HTML、导航栏、页脚）—— 始终存在
  └── app/courses/layout.tsx   ← 课程布局（侧边栏、面包屑）—— 课程相关页面共享
        └── app/courses/[courseId]/page.tsx  ← 课程详情页 —— 切换课程时只替换这层
```

来看看根布局和课程布局的具体实现：

```tsx
// app/layout.tsx —— 根布局，全站共享
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: {
    default: "CodePath 在线课程平台",
    template: "%s | CodePath", // 子页面可以用 metadata.title 覆盖，自动套用模板
  },
  description: "学习前端、后端、DevOps 实战课程",
};

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="zh-CN">
      <body>
        {/* 全站导航栏 —— 在所有页面间保持状态，不会重新渲染 */}
        <SiteHeader />
        {/* children 就是当前路由的 page.tsx 渲染结果 */}
        <main>{children}</main>
        <SiteFooter />
      </body>
    </html>
  );
}
```

```tsx
// app/courses/layout.tsx —— 课程模块的共享布局
export default function CoursesLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex">
      {/* 课程分类侧边栏 —— 在课程相关页面间导航时不会重新渲染 */}
      <aside className="w-64 border-r">
        <nav>
          <a href="/courses?category=frontend">前端开发</a>
          <a href="/courses?category=backend">后端开发</a>
          <a href="/courses?category=devops">DevOps</a>
        </nav>
      </aside>
      {/* 右侧内容区 —— 只有这块会随路由切换而变化 */}
      <div className="flex-1">{children}</div>
    </div>
  );
}
```

这里有个重要的细节：根布局 `app/layout.tsx` 必须包含 `<html>` 和 `<body>` 标签，这是整个应用的 HTML 壳。子布局不需要，它们只是包裹 `children` 的 UI 容器。

### Layout vs Template：什么时候用哪个

`layout.tsx` 在路由切换时**不**会重新渲染——这就是为什么导航栏不会闪烁、侧边栏不会收起。但有些场景你需要在每次导航时都重新执行初始化逻辑，比如进入每个新课程时重新播放一个过渡动画。这时就要用 `template.tsx`：

```tsx
// app/courses/[courseId]/template.tsx
// 每次从一门课切到另一门课时，template 都会重新挂载
// 但 layout 不会
export default function CourseTemplate({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div>
      {/* 进入动画：每次导航到新课程时重新播放 */}
      <div className="animate-fade-in">{children}</div>
    </div>
  );
}
```

实际项目中 `template.tsx` 用得不多，绝大多数场景用 `layout.tsx` 就够了。只在需要入场动画、重置滚动位置等少数情况下才需要 template。

## 路由组：共享布局但不影响 URL

回到课程平台的场景。你希望 `/courses`、`/pricing`、`/about` 这些公开页面共享一套营销布局（顶部导航 + 宽松间距），而 `/dashboard`、`/dashboard/settings` 这些需要登录的页面共享另一套布局（侧边导航 + 紧凑布局）。在 React Router 里你要写两套 `children` 嵌套，而在 Next.js 里只需要用路由组。

路由组是用括号包裹的文件夹名，比如 `(marketing)` 和 `(dashboard)`。它们**不会**出现在 URL 里——纯纯的代码组织手段：

```
app/
├── layout.tsx                      # 根布局（HTML 壳）
├── (marketing)/                    # 路由组：公开页面，不影响 URL
│   ├── layout.tsx                  # 营销布局：顶部导航 + 宽松布局
│   ├── page.tsx                    # / → 首页
│   ├── courses/
│   │   ├── page.tsx                # /courses
│   │   └── [courseId]/
│   │       └── page.tsx            # /courses/react-18
│   └── pricing/
│       └── page.tsx                # /pricing
├── (dashboard)/                    # 路由组：需要登录的页面
│   ├── layout.tsx                  # 仪表板布局：侧边导航 + 紧凑布局
│   ├── dashboard/
│   │   ├── page.tsx                # /dashboard
│   │   └── settings/
│   │       └── page.tsx            # /dashboard/settings
│   └── learn/
│       └── [courseId]/
│           └── page.tsx            # /learn/react-18（学习页面，需要登录）
└── (auth)/                         # 路由组：认证页面
    ├── layout.tsx                  # 认证布局：极简布局，无导航
    ├── login/
    │   └── page.tsx                # /login
    └── register/
        └── page.tsx                # /register
```

两个关键点：第一，`(marketing)`、`(dashboard)`、`(auth)` 这些名字只存在于代码里，URL 中看不到；第二，每个路由组可以有自己的 `layout.tsx`，这意味着你的根布局下面可以"分支"出多个不同的布局树。

看看不同布局的具体实现：

```tsx
// app/(marketing)/layout.tsx —— 公开页面的营销布局
export default function MarketingLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="min-h-screen">
      {/* 顶部导航：Logo + 课程链接 + 登录按钮 */}
      <header className="sticky top-0 z-50 border-b bg-white">
        <nav className="mx-auto flex h-16 items-center justify-between px-6">
          <a href="/" className="text-xl font-bold">
            CodePath
          </a>
          <div className="flex gap-6">
            <a href="/courses">全部课程</a>
            <a href="/pricing">定价</a>
            <a href="/login">登录</a>
          </div>
        </nav>
      </header>
      {/* 内容区：宽松间距，适合营销内容 */}
      <div className="mx-auto max-w-6xl px-6 py-12">{children}</div>
      <SiteFooter />
    </div>
  );
}
```

```tsx
// app/(dashboard)/layout.tsx —— 登录后的仪表板布局
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex h-screen">
      {/* 侧边导航：固定的侧边栏 */}
      <aside className="w-60 border-r bg-gray-50">
        <nav className="flex flex-col gap-1 p-4">
          <a href="/dashboard" className="nav-item">
            学习概览
          </a>
          <a href="/dashboard/settings" className="nav-item">
            账号设置
          </a>
        </nav>
      </aside>
      {/* 内容区：紧凑布局，最大化利用屏幕空间 */}
      <div className="flex-1 overflow-y-auto">{children}</div>
    </div>
  );
}
```

```tsx
// app/(auth)/layout.tsx —— 登录/注册页的极简布局
// 认证页面不需要导航栏，用一个干净的全屏居中布局
export default function AuthLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex min-h-screen items-center justify-center bg-gray-50">
      <div className="w-full max-w-md">{children}</div>
    </div>
  );
}
```

路由组解决了 React Router 中一个很烦人的问题：你无法在不影响 URL 的前提下给一组路由加统一的布局。以前要么把布局组件在每个页面里手动引入，要么用嵌套的 `<Outlet>` 把配置搞得很复杂。路由组让这件事变得自然——你想把哪些页面组织在一起，就放进同一个文件夹。

## 并行路由：同一个 URL 渲染多个独立模块

有时候一个页面需要同时展示多个独立的内容区域，而且这些区域要独立加载、独立出错。比如课程平台的仪表板页面，左侧是"我的学习进度"，右侧是"推荐课程"，底部是"社区动态"——它们的数据来源不同、加载速度不同，不应该互相阻塞。

并行路由用 `@` 前缀的文件夹（叫"插槽" slot）来实现：

```
app/
└── (dashboard)/
    ├── layout.tsx                    # 布局接收多个插槽
    ├── dashboard/
    │   ├── layout.tsx                # 定义插槽的位置
    │   ├── @progress/
    │   │   ├── page.tsx              # 学习进度模块
    │   │   └── loading.tsx           # 进度模块自己的 loading
    │   ├── @recommendations/
    │   │   ├── page.tsx              # 推荐课程模块
    │   │   └── loading.tsx           # 推荐模块自己的 loading
    │   └── @activity/
    │       ├── page.tsx              # 社区动态模块
    │       └── error.tsx             # 动态模块自己的 error
```

布局通过 props 接收各个插槽，像拼积木一样把它们放在不同位置：

```tsx
// app/(dashboard)/dashboard/layout.tsx
export default function DashboardPageLayout({
  children, // 默认插槽（可以放主内容）
  progress, // @progress 插槽
  recommendations, // @recommendations 插槽
  activity, // @activity 插槽
}: {
  children: React.ReactNode;
  progress: React.ReactNode;
  recommendations: React.ReactNode;
  activity: React.ReactNode;
}) {
  return (
    <div className="grid grid-cols-3 gap-6 p-6">
      {/* 左侧：学习进度（占一列） */}
      <section className="col-span-1">{progress}</section>

      {/* 右侧上方：推荐课程 + 社区动态（占两列） */}
      <div className="col-span-2 flex flex-col gap-6">
        {recommendations}
        {activity}
      </div>
    </div>
  );
}
```

并行路由最大的优势是**独立处理**。`@progress` 模块可以有自己的 `loading.tsx` 和 `error.tsx`，如果推荐课程的接口挂了，只会影响 `@recommendations` 那块区域，整个页面不会白屏。每个插槽的 loading 状态也是独立的——进度模块可能 200ms 就加载完了，推荐课程还在 loading，社区动态已经渲染出来了。

```tsx
// app/(dashboard)/dashboard/@recommendations/loading.tsx
// 只有推荐课程模块在加载时显示这个骨架屏
export default function RecommendationsLoading() {
  return (
    <div className="animate-pulse space-y-4">
      <div className="h-4 w-48 rounded bg-gray-200" />
      <div className="h-32 rounded bg-gray-200" />
      <div className="h-32 rounded bg-gray-200" />
    </div>
  );
}
```

并行路由适合"仪表板"、"分析面板"这种一个页面展示多个独立数据模块的场景。如果你只是普通的页面跳转，不需要并行路由。

## 拦截路由：弹窗场景的优雅解法

课程平台有个常见的交互：在课程列表页点击某个课程，弹出一个详情弹窗（而不是跳转）。但如果用户直接在地址栏输入 `/courses/react-18`，应该显示完整的课程详情页。这种"同一个路由在不同上下文中有不同的展示形式"的需求，用拦截路由可以很优雅地实现。

拦截路由用 `(.)`、`(..)`、`(..)(..)` 语法来"拦截"其他位置的路由：

```
app/
├── courses/
│   ├── page.tsx                        # /courses → 课程列表
│   └── [courseId]/
│       ├── page.tsx                    # /courses/react-18 → 完整课程详情页
│       └── (.)course-preview/
│           └── page.tsx                # 拦截：在列表页点击时显示弹窗
└── dashboard/
    ├── learn/
    │   └── [courseId]/
    │       └── page.tsx                # /dashboard/learn/react-18
    │       └── (..)(..)course-preview/
    │           └── page.tsx            # 从仪表板也能拦截课程预览弹窗
```

`(.)` 表示同级目录，`(..)` 表示上一级，`(..)(..)` 表示上两级。上面 `(.)course-preview` 放在 `courses/[courseId]/` 下面，所以它拦截的是同级 `courses/[courseId]` 的路由。

列表页和弹窗页的实现：

```tsx
// app/courses/page.tsx —— 课程列表页
export default async function CourseListPage() {
  const courses = await fetchCourses();

  return (
    <div>
      <h1>全部课程</h1>
      <div className="grid grid-cols-3 gap-4">
        {courses.map((course) => (
          <a
            key={course.id}
            // 注意：用普通的 <a> 标签，拦截路由会自动处理
            href={`/courses/${course.slug}`}
            className="group relative overflow-hidden rounded-lg border"
          >
            <img src={course.coverUrl} alt={course.title} />
            <div className="p-4">
              <h3>{course.title}</h3>
              <p>{course.instructor.name}</p>
            </div>
          </a>
        ))}
      </div>
    </div>
  );
}
```

```tsx
// app/courses/[courseId]/(.)course-preview/page.tsx —— 拦截路由（弹窗）
import { useRouter } from "next/navigation";

type CoursePreviewProps = {
  params: Promise<{ courseId: string }>;
};

export default async function CoursePreview({ params }: CoursePreviewProps) {
  const { courseId } = await params;
  const course = await fetchCourseById(courseId);

  return (
    <dialog open className="fixed inset-0 z-50 bg-black/50">
      <div className="mx-auto mt-20 max-w-2xl rounded-xl bg-white p-8">
        <h2>{course.title}</h2>
        <p>{course.description}</p>
        <div className="flex gap-4">
          <a href={`/courses/${courseId}`} className="btn-primary">
            查看完整课程
          </a>
          <CoursePreviewCloseButton />
        </div>
      </div>
    </dialog>
  );
}

// 关闭弹窗需要客户端交互，所以抽成 Client Component
function CoursePreviewCloseButton() {
  const router = useRouter();

  return (
    <button
      onClick={() => router.back()} // 返回上一页，弹窗自动消失
      className="btn-secondary"
    >
      关闭
    </button>
  );
}
```

效果是这样的：用户在 `/courses` 页面点击课程卡片，URL 变成 `/courses/react-18`，但页面不会整页跳转——弹窗覆盖在列表上方。用户可以直接访问 `/courses/react-18`，看到的是完整的课程详情页（没有被拦截）。用户在弹窗里点"查看完整课程"，也会导航到完整页面。

拦截路由本质上是"在特定上下文中，用另一个组件替代目标路由的渲染"。这个模式特别适合：列表页的预览弹窗、图片画廊的灯箱、消息列表的详情侧滑面板。

## loading.tsx 和 error.tsx：内置的加载和错误处理

前面已经零散提到了 loading 和 error，这里系统说一下。Next.js 的 `loading.tsx` 和 `error.tsx` 是约定式文件——你只要在某个路由段下创建它们，Next.js 就会自动用 Suspense 和 ErrorBoundary 包裹该路由段的 `page.tsx`。

### loading.tsx：自动的 Suspense fallback

```tsx
// app/courses/[courseId]/loading.tsx
// 当课程详情页在获取数据时，这个组件会自动显示
// Next.js 内部用 <Suspense fallback={<Loading />}> 包裹了 page.tsx
export default function CourseLoading() {
  return (
    <div className="animate-pulse space-y-6 p-8">
      {/* 模拟课程标题 */}
      <div className="h-8 w-3/4 rounded bg-gray-200" />
      {/* 模拟讲师信息 */}
      <div className="h-4 w-1/4 rounded bg-gray-200" />
      {/* 模拟课程描述 */}
      <div className="space-y-2">
        <div className="h-4 w-full rounded bg-gray-200" />
        <div className="h-4 w-5/6 rounded bg-gray-200" />
        <div className="h-4 w-4/6 rounded bg-gray-200" />
      </div>
      {/* 模拟课程视频区域 */}
      <div className="aspect-video rounded bg-gray-200" />
    </div>
  );
}
```

`loading.tsx` 自动成为 Server Component，它会在数据加载时即时显示（不需要等任何 JS 加载），因为它是预渲染好的。这意味着用户在等待数据时看到的是静态骨架屏，而不是一个空白页。

### error.tsx：自动的错误边界

```tsx
// app/courses/[courseId]/error.tsx
"use client"; // error.tsx 必须是 Client Component，因为需要处理交互（重试按钮）

import { useEffect } from "react";

type ErrorPageProps = {
  error: Error & { digest?: string }; // digest 是服务端生成的错误标识，用于追踪
  reset: () => void; // 重试函数，重新渲染出错的页面
};

export default function CourseError({ error, reset }: ErrorPageProps) {
  useEffect(() => {
    // 把错误上报到监控服务（Sentry、Datadog 等）
    console.error("[CourseDetail] 渲染出错:", error);
  }, [error]);

  return (
    <div className="flex flex-col items-center justify-center gap-4 p-12">
      <h2 className="text-xl font-semibold">课程加载失败</h2>
      <p className="text-gray-500">{error.message || "请检查网络连接后重试"}</p>
      <button
        onClick={reset} // 调用 reset() 会重新渲染出错的 page.tsx
        className="rounded-lg bg-blue-600 px-6 py-2 text-white hover:bg-blue-700"
      >
        重试
      </button>
    </div>
  );
}
```

`error.tsx` 有几个特点：必须标记 `"use client"`（因为需要用 `useEffect` 上报错误、需要 `onClick` 重试）；接收 `error` 和 `reset` 两个 props；只捕获它自己层级的错误——子路由的 `error.tsx` 不会冒泡到父路由，除非子路由没有自己的 error.tsx。

### not-found.tsx：自定义 404


## 路由 vs React Router：什么场景该选哪个

到这里你已经看到 Next.js 文件系统路由的能力了。最后做个对比，帮你判断什么场景下该用 Next.js 内置路由，什么场景下可能还是需要 React Router。

**Next.js 文件系统路由的优势：**

- 路由即文件结构，新成员一眼就能看懂
- 布局自动嵌套，不会重新渲染，状态天然保持
- loading/error/not-found 全部约定式处理，零配置
- 并行路由和拦截路由解决了很多传统 SPA 中很麻烦的场景
- Server Components 可以在路由组件里直接 await 数据

**React Router 仍然适用的场景：**

- 纯客户端 SPA（不需要 SSR/SSG），React Router 更轻量
- 需要在运行时动态生成路由（Next.js 路由是构建时确定的）
- 需要非常细粒度的路由过渡动画控制
- 已有的大型 React SPA 迁移成本太高

对于大多数新项目，尤其是需要 SEO、首屏性能、或者有服务端数据获取需求的项目，Next.js 的文件系统路由是更好的选择。你在 React Router 中手写的那些布局嵌套、路由守卫、loading 状态，在 Next.js 里通过几个文件约定就解决了。

## Trade-off 分析

文件系统路由不是银弹，有一些需要注意的取舍：

**文件系统路由的代价：**

- 路由结构必须在构建时确定，无法根据数据库里的数据动态创建路由（只能用动态路由 `[slug]` 来匹配）
- 目录嵌套太深时，文件查找会变得困难——虽然有路由组可以缓解，但大型项目的 `app/` 目录仍然会很长
- 约定式文件有时候不够灵活，比如你想让某个路由的 error.tsx 同时处理两个不同路由段的错误，做不到——每个段独立处理
- 并行路由的 TypeScript 类型推导不太友好，slot props 的类型需要手动维护
- 拦截路由的 `(.)`、`(..)` 语法学习成本较高，第一次接触容易搞混层级关系

**选择建议：**

- 如果你的项目有超过 3 种不同的页面布局（营销页、仪表板、认证页、管理后台等），路由组能帮你大幅简化代码组织
- 如果你的仪表板有多个独立的数据面板，并行路由 + 独立的 loading/error 是最佳方案
- 如果你的列表页需要"点击弹出详情"的交互，拦截路由比手写状态管理要干净得多
- 如果你只需要一个简单的 SPA，不需要 SEO 也不需要服务端渲染，那 React Router + Vite 可能是更轻量的选择
