# GraphQL vs REST 深度对比与实践指南

API 是前后端交互的核心，选择合适的 API 设计方案对应用的可维护性和性能至关重要。REST API 已经是行业标准，而 GraphQL 作为新兴的查询语言，提供了一种全新的 API 设计思路。

本文将深入对比 GraphQL 和 REST，分析各自的优势和适用场景，并提供实践指南。

## REST API 基础

REST（Representational State Transfer）是一种软件架构风格，使用 HTTP 方法（GET、POST、PUT、DELETE）来操作资源。

### REST 核心原则

1. **资源导向**：一切皆资源，使用 URL 标识资源
2. **统一接口**：使用标准 HTTP 方法
3. **无状态**：服务器不保存客户端状态
4. **可缓存**：响应可被缓存
5. **分层系统**：可以分层部署

### REST API 示例

```typescript
// API 端点设计
GET    /api/users          # 获取用户列表
GET    /api/users/:id      # 获取单个用户
POST   /api/users          # 创建用户
PUT    /api/users/:id      # 更新用户
DELETE /api/users/:id      # 删除用户

GET    /api/users/:id/posts  # 获取用户的文章
GET    /api/posts/:id/comments # 获取文章的评论
```

```typescript
// Next.js API Routes 实现
// app/api/users/route.ts
import { NextRequest, NextResponse } from "next/server";
import { prisma } from "@/lib/db";

export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url);
  const page = parseInt(searchParams.get("page") || "1");
  const limit = parseInt(searchParams.get("limit") || "10");

  const users = await prisma.user.findMany({
    skip: (page - 1) * limit,
    take: limit,
  });

  return NextResponse.json({
    success: true,
    data: users,
  });
}

export async function POST(request: NextRequest) {
  const body = await request.json();

  const user = await prisma.user.create({ data: body });

  return NextResponse.json(
    {
      success: true,
      data: user,
    },
    { status: 201 }
  );
}

// app/api/users/[id]/route.ts
export async function GET(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const user = await prisma.user.findUnique({
    where: { id: parseInt(params.id) },
  });

  if (!user) {
    return NextResponse.json(
      { success: false, error: "User not found" },
      { status: 404 }
    );
  }

  return NextResponse.json({ success: true, data: user });
}
```

## GraphQL 基础

GraphQL 是一种用于 API 的查询语言，由 Facebook 于 2012 年创建，2015 年开源。它允许客户端精确指定需要的数据，避免过度获取。

### GraphQL 核心概念

1. **Schema**：定义 API 的类型系统
2. **Query**：读取数据
3. **Mutation**：修改数据
4. **Subscription**：实时数据推送
5. **Resolver**：解析查询和 mutation 的函数

### GraphQL Schema 示例

### GraphQL Server 实现

```typescript
import { ApolloServer } from "@apollo/server";
import { startStandaloneServer } from "@apollo/server/standalone";
import { PrismaClient } from "@prisma/client";
import { makeExecutableSchema } from "@graphql-tools/schema";
import { gql } from "graphql-tag";

const prisma = new PrismaClient();

const typeDefs = gql`
  type Query {
    users(page: Int, limit: Int): UserConnection!
    user(id: ID!): User
  }

  type Mutation {
    createUser(input: CreateUserInput!): User!
  }

  type User {
    id: ID!
    name: String!
    email: String!
    createdAt: DateTime!
  }

  type UserConnection {
    edges: [UserEdge!]!
    pageInfo: PageInfo!
  }

  type UserEdge {
    node: User!
    cursor: String!
  }

  type PageInfo {
    hasNextPage: Boolean!
    endCursor: String
  }

  input CreateUserInput {
    name: String!
    email: String!
  }

  scalar DateTime
`;

const resolvers = {
  Query: {
    users: async (_: any, { page = 1, limit = 10 }: any) => {
      const users = await prisma.user.findMany({
        skip: (page - 1) * limit,
        take: limit + 1, // 多取一个，判断是否有下一页
        orderBy: { createdAt: "desc" },
      });

      const edges = users.slice(0, limit).map((user) => ({
        node: user,
        cursor: Buffer.from(user.id.toString()).toString("base64"),
      }));

      return {
        edges,
        pageInfo: {
          hasNextPage: users.length > limit,
          endCursor: edges[edges.length - 1]?.cursor,
        },
      };
    },

    user: async (_: any, { id }: any) => {
      return prisma.user.findUnique({ where: { id: parseInt(id) } });
    },
  },

  Mutation: {
    createUser: async (_: any, { input }: any) => {
      return prisma.user.create({ data: input });
    },
  },

  User: {
    createdAt: (user: any) => user.createdAt.toISOString(),
    id: (user: any) => user.id.toString(),
  },
};

const server = new ApolloServer({
  typeDefs,
  resolvers,
});

const { url } = await startStandaloneServer(server, {
  listen: { port: 4000 },
});

console.log(`🚀 Server ready at ${url}`);
```

## GraphQL vs REST 核心差异

### 1. 数据获取方式

**REST**：

```typescript
// 获取用户及其文章
// 方案 1：多个请求
const user = await fetch("/api/users/1").then((r) => r.json());
const posts = await fetch(`/api/users/1/posts`).then((r) => r.json());

// 方案 2：自定义端点
const userWithPosts = await fetch("/api/users/1?include=posts").then((r) =>
  r.json()
);
// 返回所有字段，可能过度获取
```

**GraphQL**：

```graphql
# 单个请求，精确获取
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
    email
    posts {
      id
      title
      content
    }
  }
}
```

### 2. 版本控制

**REST**：

```
/api/v1/users
/api/v2/users
# 不同版本共存，维护成本高
```

**GraphQL**：

```graphql
# Schema 演进，无需版本控制
# 新增字段不影响现有客户端
type User {
  id: ID!
  name: String!
  email: String! # v1
  phone: String # v2 新增字段
  avatar: String # v3 新增字段
}
```

### 3. 错误处理

**REST**：

```typescript
// HTTP 状态码
GET /api/users/999
404 Not Found

{
  "success": false,
  "error": "User not found"
}
```

**GraphQL**：

```graphql
# 总是返回 200，错误在响应体中
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
  }
}

# 响应
{
  "data": {
    "user": null
  },
  "errors": [
    {
      "message": "User not found",
      "path": ["user"],
      "extensions": {
        "code": "NOT_FOUND"
      }
    }
  ]
}
```

### 4. 性能和缓存

**REST**：

```typescript
// HTTP 缓存
GET /api/users
Cache-Control: public, max-age=3600

// 浏览器和 CDN 可以缓存
```

**GraphQL**：

```graphql
# 默认 POST 请求，难以缓存
POST /graphql

# 可以使用 GET + 持久化查询
GET /graphql?query=hash_of_query
Cache-Control: public, max-age=3600
```

## GraphQL Schema 设计最佳实践

### 1. 命名规范

```graphql
# ✅ 好的命名
type Query {
  users(page: Int, limit: Int): UserConnection!
  user(id: ID!): User
}

type Mutation {
  createUser(input: CreateUserInput!): User!
  updateUser(id: ID!, input: UpdateUserInput!): User!
}

# ❌ 避免的命名
type Query {
  getUser(id: ID!): User # 应该是 user
  getAllUsers: [User!]! # 应该是 users，添加分页
  searchUsers(query: String!): [User!]! # 应该在 users 中添加 filter 参数
}
```

### 2. 输入验证

```graphql
# 使用输入类型进行验证
input CreateUserInput {
  name: String!
  email: String!
  password: String! @constraint(minLength: 8)
  role: UserRole @default(USER)
}

# 自定义指令
directive @constraint on INPUT_FIELD_DEFINITION | ARGUMENT_DEFINITION

# Resolver 中验证
const resolvers = {
  Mutation: {
    createUser: async (_: any, { input }: any) => {
      // 验证
      const { error } = validateUserInput(input);
      if (error) {
        throw new GraphQLError(error.message, {
          extensions: { code: 'VALIDATION_ERROR', details: error.details }
        });
      }

      return prisma.user.create({ data: input });
    }
  }
};
```

### 3. 分页设计

```graphql
# 游标分页（推荐）
type PostConnection {
  edges: [PostEdge!]!
  pageInfo: PageInfo!
}

type PostEdge {
  node: Post!
  cursor: String!
}

type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String
  endCursor: String
}

query GetPosts($first: Int, $after: String) {
  posts(first: $first, after: $after) {
    edges {
      node {
        id
        title
      }
      cursor
    }
    pageInfo {
      hasNextPage
      endCursor
    }
  }
}
```

### 4. 关系设计

```graphql
# 避免过度嵌套
# ❌ 不好的设计
type User {
  posts {
    comments {
      author {
        posts {
          comments {
            # 无限嵌套
          }
        }
      }
    }
  }
}

# ✅ 好的设计
type User {
  id: ID!
  name: String!
  posts(limit: Int): PostConnection!
}
```

## 数据加载优化

### N+1 查询问题

**REST**：

```typescript
// ❌ N+1 问题
const users = await prisma.user.findMany({ take: 10 });
for (const user of users) {
  const posts = await prisma.post.findMany({
    where: { authorId: user.id },
  }); // 每个用户一个查询
}

// ✅ 解决方案
const users = await prisma.user.findMany({
  include: {
    posts: true,
  },
});
```

**GraphQL**：

```graphql
# ❌ N+1 问题
query GetUsers {
  users {
    id
    name
    posts {
      id
      title
    }
  }
}

# 如果不使用 DataLoader，每个用户的 posts 都会触发一次查询

# ✅ 使用 DataLoader
import DataLoader from 'dataloader';

const userLoader = new DataLoader(async (ids: readonly number[]) => {
  const users = await prisma.user.findMany({
    where: { id: { in: ids } }
  });
  return ids.map(id => users.find(u => u.id === id));
});

const resolvers = {
  User: {
    posts: async (user: any) => {
      return postLoader.load(user.id);
    }
  }
};
```

### 批量加载优化

```typescript
import DataLoader from "dataloader";

// 创建 loader
const userLoader = new DataLoader(async (ids: readonly number[]) => {
  const users = await prisma.user.findMany({
    where: { id: { in: ids } },
  });

  // 按 ID 顺序返回
  const userMap = new Map(users.map((u) => [u.id, u]));
  return ids.map((id) => userMap.get(id));
});

// 使用 loader
const resolvers = {
  Query: {
    user: async (_: any, { id }: any) => {
      return userLoader.load(id);
    },
  },
};
```

### 查询复杂度限制

```typescript
import { createComplexityLimitRule } from "graphql-validation-complexity";

const server = new ApolloServer({
  typeDefs,
  resolvers,
  validationRules: [
    createComplexityLimitRule(1000, {
      onCost: (cost) => console.log(`Query cost: ${cost}`),
      createError: (cost, maxCost) =>
        new Error(
          `Query is too complex: ${cost}. Maximum allowed complexity: ${maxCost}`
        ),
    }),
  ],
});

// 字段添加复杂度权重
const typeDefs = gql`
  directive @cost(complexity: Int) on FIELD_DEFINITION

  type Query {
    users: [User!]! @cost(complexity: 10)
    posts: [Post!]! @cost(complexity: 20)
    user(id: ID!): User @cost(complexity: 5)
  }
`;
```

## GraphQL vs REST 选择指南

### 何时使用 REST

1. **简单 CRUD 应用**：增删改查为主的应用
2. **需要 HTTP 缓存**：浏览器和 CDN 缓存至关重要
3. **团队熟悉 REST**：降低学习成本
4. **第三方集成**：需要与其他 REST API 集成
5. **文件上传/下载**：处理二进制数据

### 何时使用 GraphQL

1. **复杂的数据关系**：需要深度嵌套查询
2. **多个客户端需求**：移动端、Web 端需要不同的数据
3. **减少请求次数**：客户端需要精确控制数据
4. **实时数据需求**：使用 Subscription
5. **快速迭代**：频繁的 API 变更

### 选择决策树

```
开始
  ↓
需要处理文件上传/下载？
  ├─ 是 → REST
  └─ 否 → 继续
  ↓
需要 HTTP 缓存？
  ├─ 是 → REST
  └─ 否 → 继续
  ↓
客户端需要不同数据？
  ├─ 是 → GraphQL
  └─ 否 → 继续
  ↓
数据关系复杂（深度嵌套）？
  ├─ 是 → GraphQL
  └─ 否 → REST
```

## 总结

GraphQL 和 REST 各有优势，选择取决于具体场景。REST 成熟稳定，适合简单应用；GraphQL 灵活强大，适合复杂需求。

### 核心要点

1. **REST**：成熟稳定，易于理解和实现
2. **GraphQL**：灵活强大，精确控制数据
3. **N+1 问题**：两种方式都需要注意
4. **缓存策略**：REST 更适合 HTTP 缓存
5. **混合架构**：可以结合两者优势

### 最佳实践

- 根据项目需求选择合适的方案
- GraphQL 使用 DataLoader 优化查询
- REST 合理设计 API 端点
- 两种方式都要考虑错误处理和性能
- BFF 模式可以聚合多个服务

没有银弹，理解各自的优势和局限，才能做出正确的技术选择。
