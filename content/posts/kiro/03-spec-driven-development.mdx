上篇我们用 Quick Plan 生成了 HabitFlow 的第一个注册 API，体验了 Kiro 的"快"——一句话需求，几秒钟代码。但 Quick Plan 适合简单任务，一旦功能复杂一点，AI 直接开写就容易跑偏。

这篇我们深入 Kiro 的核心功能：**Spec 驱动开发**。

## 为什么 AI 直接写代码总翻车

你让 AI "给 HabitFlow 加个习惯打卡功能"，它啪啪啪生成了一堆代码。你看了一眼：打卡记录存 localStorage、没有连续天数统计、日期用的 UTC 不是本地时间……方向全错了。

问题不是 AI 能力不够，而是**你没给它想清楚的机会**。人类写代码前也要先想需求、想设计、拆任务，凭什么指望 AI 直接从需求跳到代码？

Spec 就是解决这个问题的——让 AI 先写"文档"，确认对了再写代码。

## Spec 的三阶段工作流

一个完整的 Feature Spec 会生成三个文件：

1. **requirements.md** — 需求文档，定义"做什么"
2. **design.md** — 技术设计，定义"怎么做"
3. **tasks.md** — 任务拆解，定义"执行顺序"

你可以在每个阶段审查、修改，确认满意了再进入下一阶段。整个过程像带了一个很靠谱的实习生——他先给你看方案，你说 OK 他才动手。

## 实战：为 HabitFlow 创建习惯打卡 Spec

打开 Kiro，按 `Cmd+Shift+P` 输入 `Spec`，选择 **Create Feature Spec**。在弹出的输入框里描述你要什么：

> 为 HabitFlow 添加习惯打卡功能。用户可以创建习惯（名称、频率、目标天数），每天打卡一次，能看到连续打卡天数和历史记录。

Kiro 会分析你的项目结构，然后开始生成 requirements.md。

### 第一阶段：需求文档

生成的 requirements.md 大概长这样：

```markdown
# Habit Check-in Feature - Requirements

## User Stories

- As a user, I want to create a habit with name, frequency, and goal
- As a user, I want to check in once per day for each habit
- As a user, I want to see my current streak and total check-ins
- As a user, I want to view check-in history by date

## Acceptance Criteria

- Cannot check in twice for the same habit on the same day
- Streak resets to 0 if a day is missed
- Goal completion shows a celebration state
- All dates use local timezone
```

关键点：**仔细审查验收标准**。AI 可能漏掉你认为理所当然的规则，比如"同一天不能重复打卡"——如果你不在这里补上，后面代码就不会有这个校验。

我在这里加了"打卡可以补签最近 3 天内遗漏的"这个需求，然后点 **Approve**。

### 第二阶段：技术设计

Kiro 根据 requirements.md 生成 design.md：

```markdown
# Habit Check-in Feature - Design

## Data Model

- habits table: id, user_id, name, frequency, goal_days, created_at
- checkins table: id, habit_id, checked_date, created_at

## API Endpoints

- POST /habits — create habit
- POST /habits/:id/checkin — daily check-in
- GET /habits/:id/streak — get streak info
- GET /habits/:id/history — check-in history

## Key Logic: Streak Calculation

Query consecutive dates backward from today or latest check-in date.
Allow makeup check-ins for last 3 days.
```

这一步是整个 Spec 最有价值的环节。你需要关注：

- **数据模型**是否合理——checkins 表单独拆开是对的，不要把打卡记录塞进 habits 表
- **API 设计**是否符合你项目的 RESTful 风格
- **核心算法**的描述——连续天数怎么算、补签逻辑怎么处理

我发现 design.md 里没有提到"删除习惯时要不要级联删除打卡记录"，补上这个设计决策后点 Approve。

### 第三阶段：任务拆解

Kiro 会把设计拆成可执行的任务列表：

```markdown
# Tasks

## Wave 1 (Independent, can run in parallel)

- Task 1: Create database schema for habits and checkins tables
- Task 2: Add check-in validation utility (date check, streak calc)

## Wave 2 (Depends on Wave 1)

- Task 3: Implement POST /habits endpoint
- Task 4: Implement POST /habits/:id/checkin endpoint
- Task 5: Implement GET /habits/:id/streak endpoint

## Wave 3 (Depends on Wave 2)

- Task 6: Add history endpoint with date range query
- Task 7: Integration tests for full check-in flow
```

注意那个 **Wave**——Kiro 自动分析了任务之间的依赖关系，把没有依赖的任务分到同一波并行执行。Task 1 和 Task 2 互相不依赖，所以可以同时干。

确认任务没问题后，Kiro 开始按 Wave 顺序执行代码生成。

## 并行任务执行：Kiro 的隐藏杀器

传统 AI 工具是一个任务接一个任务串行执行。Kiro 不一样——它会在同一 Wave 里启动多个 **Subagent**，并行写代码。

比如 Wave 1 里，一个 Subagent 写数据库 schema，另一个写校验工具函数。它们改的文件不重叠，所以不会冲突。

这在大功能开发里效果很明显：一个 10 个任务的 Spec，如果 5 个可以并行，理论上速度快一倍。

实际体验中，并行执行的收益取决于任务拆分是否合理。如果你的 tasks.md 里都是互相依赖的大任务，并行就没意义了。所以第三阶段的审查很重要。

## Bugfix Spec：用 Spec 思路修 Bug

Spec 不只能做新功能，修 Bug 也行。选择 **Create Bugfix Spec**，描述你遇到的问题：

> 打卡后连续天数没有更新，怀疑是 streak 计算逻辑的时区问题。

Kiro 会生成 bugfix.md：

```markdown
# Bugfix: Streak Calculation Timezone Issue

## Root Cause Analysis

Streak calculation uses UTC dates via `new Date().toISOString()`,
but check-in dates are stored in local timezone. This causes
off-by-one errors near midnight.

## Fix Plan

- Standardize all date comparisons to local timezone
- Add timezone-aware date utility
- Add test case for midnight boundary
```

Bugfix Spec 的好处是强迫 AI 先分析根因，而不是上来就改代码。很多时候你以为是个时区问题，AI 分析完发现其实是异步竞态——这种 insight 比直接修 Bug 值钱得多。

## Spec 的最佳实践

用了几十个 Spec 之后，我总结了几个关键经验：

**需求阶段要抠细节。** 验收标准越精确，代码质量越高。"用户能看到连续天数"和"连续天数显示在习惯卡片右上角，灰色背景，数字加粗"——后者生成的代码差距巨大。

**设计阶段要参与决策。** 不要完全交给 AI。数据模型、API 风格、核心算法，这些架构决策应该由你来做，AI 帮你记录和补全。

**任务拆分要细到单文件。** "实现打卡功能"太大了，拆成"写 checkins 表的 migration"、"写 POST checkin 的路由"、"写 streak 计算函数"——颗粒度越小，并行效果越好。

**常见陷阱：** 不要跳过前两个阶段直接改 tasks.md。需求没想清楚就拆任务，等于在错误的方向上高效奔跑。也不要在 Spec 里塞太多功能——一个 Spec 一个核心功能，复杂需求拆成多个 Spec。

## Trade-off 总结

**适用场景：** 中大型功能开发（5 个文件以上的改动）、需要和 AI 反复对齐需求的任务、团队协作时需要留下设计文档的场景。

**不适用场景：** 改个 CSS 样式、加个 console.log、简单的 bug 修复——这些直接用 Chat 或 Quick Plan 更快。

**对比替代方案：** Cursor 的 Composer 模式也能生成多文件改动，但没有结构化的需求→设计→任务流程，更像是"高级聊天"。Windsurf 的 Cascade 有一定的规划能力，但没有 Spec 这种可审查的中间产物。Spec 的核心优势是**过程的可见性和可控性**。

**实践建议：** 从一个中等复杂度的功能开始练手，习惯三阶段的审查节奏。审查时间不要省——你在 Spec 上多花 5 分钟，能省下 30 分钟的返工。
