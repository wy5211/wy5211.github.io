上篇我们用 Steering 和 Hooks 让 Kiro 记住了项目规范——技术栈偏好、编码风格、提交前自动格式化。有了这些基础设施，终于可以放开手脚了。

这篇聊 Kiro 里最常用的功能：**Chat 面板和 Agent 工作流**。

你可能觉得 Chat 有什么好讲的——不就是对 AI 说话、它回你代码吗？Cursor 不就是这样？Kiro 确实也能这样用，但它的 Chat 背后有一套更完整的 Agent 系统，用对了效率翻倍，用错了就是高级版聊天框。

## 先搞清楚一个问题：什么时候用 Chat，什么时候用 Spec？

这是一个我在用了两个月之后才想明白的问题。

简单粗暴的判断标准：**需不需要改代码？**

- 不改代码的事——问问题、理解逻辑、探索方案 → **Chat**
- 要改代码的事——新功能、修 Bug、重构 → 先想想要不要走 **Spec**

但这里有个灰色地带：小改动。比如"给打卡记录加个排序字段"，改两行代码，走 Spec 流程太重，直接 Chat 说一句就完了。但如果改动涉及三个文件以上的联动，Spec 的结构化流程能帮你避免返工。

理解了这个大前提，我们来看 Chat 的两种模式。

## 两种会话模式：Vibe 和 Spec

打开 Chat 面板，顶部有一个切换按钮，可以在两种模式间选择：

**Vibe Session**——轻量、快速、随意。像跟同事在 Slack 里讨论技术问题。

**Spec Session**——结构化、有计划、可审查。像跟同事开了一个有议程的正式会议。

### 用 Vibe 模式探索"连续打卡统计"

我们的 HabitFlow 已经有了基本的打卡功能（第 3 篇做的），现在需要加"连续打卡天数统计"。在动手写代码之前，我想先搞清楚几个设计问题。

切到 Vibe 模式，输入：

```
HabitFlow 需要添加连续打卡天数统计功能。
我现在的 checkins 表存了 habit_id 和 checked_date。
帮我想想连续天数的计算逻辑——要考虑：
1. 用户漏打卡一天就断，还是有补签机制？
2. 时区怎么处理？跨天边界？
3. 性能：每次查询都算还是缓存？
```

Vibe 模式的特点是不触发任何代码修改，纯粹对话。Kiro 会分析你的项目结构（它能读到 checkins 表的 schema），给出分析。

它回答的要点大概是这样的：

- **断签策略**：建议区分"当前连续天数"和"最长连续天数"，断签只重置前者
- **时区**：用数据库的 `DATE(checked_date, 'localtime')` 统一为本地日期
- **性能**：用户量小的时候实时计算足够，加个索引就行

这个对话过程非常有价值——我在写一行代码之前就把设计想清楚了。以前用 Cursor，我往往直接说"帮我写连续打卡统计"，然后对着一堆不符合预期的代码反复修改。

### 用 Spec Session 执行"连续打卡统计"

设计想清楚了，该写代码了。切到 Spec Session，输入：

```
为 HabitFlow 添加连续打卡天数统计功能：
- checkins 表已有 habit_id, checked_date 字段
- 实现实时连续天数计算（从最近一次打卡往前数连续日期）
- 同时记录最长连续天数
- 新增 GET /habits/:id/streak 接口返回当前连续天数和最长记录
- 使用本地时区处理日期
```

Spec Session 会走结构化流程：先生成计划，列出要改的文件和步骤，你确认后才执行。这和 Vibe 的区别是——Vibe 只动嘴，Spec 会动手。

## Autopilot 和 Supervised：根据任务选择

上一篇文章提过这两个模式，这里从 Chat 视角再补充一下。

**Autopilot** 适合你对任务有信心的场景。比如刚才的连续打卡统计，需求明确、改动范围清晰，让 Kiro 自主完成就行。它会自动创建文件、修改代码、运行测试，一口气干完。

**Supervised** 适合以下情况：

- 你在学习 Kiro，想看清楚每一步
- 改动涉及核心逻辑，你需要逐个文件审查
- 任务描述不够精确，你想在中途调整方向

我的实际用法：**80% 的时间用 Autopilot，只在核心模块改动时切 Supervised**。全用 Supervised 太慢，全用 Autopilot 又容易失控。

## Subagents：并行执行的隐藏杀器

Kiro 的 Agent 不只是单线程执行任务。当 Spec Session 的任务拆解里有多个互不依赖的子任务时，Kiro 会启动 **Subagent** 并行处理。

拿连续打卡统计来说，Kiro 可能把任务拆成：

1. 写 streak 计算工具函数
2. 修改 habits 路由添加 streak 接口
3. 写 streak 相关的单元测试

任务 1 和任务 3 不依赖彼此（测试的是接口规范，不是具体实现），Kiro 会同时启动两个 Subagent 并行写代码。

你在 Chat 面板能看到每个 Subagent 的状态——哪个在跑、哪个完成了、哪个出错了。如果一个 Subagent 的改动和另一个冲突了，Kiro 会自动处理合并。

**实际体验**：并行执行在 5 个文件以上的改动中效果明显。但前提是任务拆分合理——如果你把所有逻辑塞进一个大任务，Subagent 就没机会并行了。

## Checkpoints：给自己留条后路

Agent 改了一堆代码，你发现方向不对——这时候怎么办？

Kiro 的 **Checkpoints** 就是干这个的。在 Agent 执行过程中，Kiro 会自动创建检查点（你也可以手动创建）。如果结果不满意，一键回退到之前的检查点，所有文件改动都会撤销。

使用场景举例：

- Spec Session 执行到一半，你看了前几个文件的改动，发现 AI 理解错了需求 → 回退到起点，重新描述需求
- Autopilot 跑完了，review 时发现某几个文件的改动不合理 → 回退到那个检查点，换 Supervised 模式重新来

在 Chat 面板顶部有个时间线图标，点开能看到所有检查点。选择任意一个，点 Restore 就行。

**我的习惯**：在 Spec Session 开始前手动创建一个检查点，标注为"改动前"。这样不管 AI 怎么折腾，我都能安全回到原点。

## Dev Servers：代码改了立刻能看到效果

这是一个很贴心的小功能。Kiro 可以连接你的开发服务器（比如 Vite、Next.js dev server），Agent 改完代码后自动刷新预览。

在 Chat 面板的设置里配置 Dev Server：

```bash
npm run dev
```

Kiro 会在后台启动这个命令，监控输出。当代码文件变化时，它能感知到开发服务器的热更新状态。

这有什么用？当你让 Agent 改前端代码时，你不用手动刷新浏览器去看效果。Kiro 会告诉你"改动已生效"或者"编译报错了，具体错误是……"。

对 HabitFlow 来说，当我们加完连续打卡统计的前端展示后，Kiro 直接告诉我"streak 数字已显示在习惯卡片上，当前值为 7"。我切到浏览器一看，果然更新了。

## Diagnostics：Agent 自己查问题

Agent 执行完任务后，Kiro 会自动运行 **Diagnostics**——检查代码质量、运行 lint、执行测试。如果发现问题，它会在 Chat 里直接报告，并且可以自动修复。

比如连续打卡统计功能加完后，Diagnostics 报告：

- TypeScript 类型检查：发现 streak 返回值缺少类型定义 → 自动补上
- 单元测试：一个边界 case 没覆盖（跨月连续天数） → 自动补充测试
- ESLint：一个未使用的 import → 自动移除

你不需要手动跑 `npm test` 或 `npx tsc --noEmit`，Kiro 在 Agent 完成任务后自动帮你检查一遍。配合 Steering 里配置的编码规范，Diagnostics 的检查会更有针对性。

## 一个完整的 Chat 工作流回顾

把刚才零散的操作串起来，完整的流程是这样的：

1. **Vibe Session** 问设计问题，搞清楚连续打卡的计算逻辑
2. 手动创建 **Checkpoint**
3. 切到 **Spec Session**，输入明确的需求描述
4. 审查 Kiro 生成的执行计划
5. 选择 **Autopilot** 执行，观察 **Subagents** 并行处理
6. **Diagnostics** 自动检查代码质量
7. review 结果，满意就保留，不满意就 **Restore Checkpoint** 重来

整个过程大概 10 分钟。如果用 Cursor，我通常会经历：描述需求 → AI 生成代码 → 发现问题 → 重新描述 → 再次生成 → 手动修补 → 跑测试发现 bug → 再修。同样的功能，往往要 30 分钟以上。

差距在哪？**Kiro 把"探索-规划-执行-验证"串成了一个结构化流程**，而不是靠来回聊天硬凑。

## Trade-off 总结

**适用场景：**

- Vibe Session：技术方案探索、代码理解、快速问答——不需要改代码的场合
- Spec Session：有明确改动目标的中大型任务——3 个文件以上的联动改动
- Subagents：任务间无依赖的并行场景，比如同时改前后端
- Checkpoints：任何你不 100% 确定结果的 Agent 操作

**不适用场景：**

- 简单的一句话修改（改个 CSS 值、加个字段）→ Vibe 模式直接说就行，不用 Spec
- 需要反复试错的探索性编码（比如调样式）→ Chat 来回对话比 Spec 流程高效
- 纯基础设施任务（配 CI/CD、写 Dockerfile）→ 不需要结构化规划

**对比替代方案：**

- 对比 Cursor Chat：Kiro 多了 Vibe/Spec 两种模式的明确区分，Cursor 是统一的 Chat + Agent 模式。Kiro 的结构化程度更高，但灵活度稍低
- 对比 Windsurf Cascade：Cascade 也有 Agent 能力，但没有 Checkpoints 和 Diagnostics 这样的安全保障
- 对比 GitHub Copilot Chat：Copilot 的 Chat 更轻量，适合问答和简单代码生成，不具备 Agent 的多步骤执行能力

**实践建议：**

- 先用 Vibe 模式想清楚，再用 Spec 模式动手——这个顺序能减少 50% 的返工
- 养成手动创建 Checkpoint 的习惯，尤其是大改动前
- Autopilot 不是偷懒——是你对需求描述有信心的体现。用 Supervised 只是因为你不信任自己的描述，那就回去改描述
- Diagnostics 的结果要看，但不要盲从——有时候 lint 警告是可以接受的
