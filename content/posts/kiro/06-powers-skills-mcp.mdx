上篇我们掌握了 Chat 和 Agent 两种工作流——简单任务用 Chat，复杂任务用 Agent。但到目前为止，Kiro 能操作的范围还局限在你的项目文件里。它碰不到数据库、调不了外部 API、更连不上 Figma 设计稿。

这篇要解决的核心问题就是：**怎么扩展 Kiro 的能力边界**。

Kiro 提供了三层扩展机制——Powers、Skills、MCP，各有各的定位。我们用一个真实场景串起来：**为 HabitFlow 接入 Supabase 数据库，再封装一个自定义的数据库迁移 Skill**。

## Powers 能力包：一键激活，按需加载

### Powers 是什么

Powers 是 Kiro 的"技能市场"。每个 Power 把一类工具、工作流和最佳实践打包在一起——Supabase Power 让 Kiro 直接读写数据库，AWS Power 让它管理云资源，Figma Power 让它读取设计稿生成代码。

最关键的一点：**Power 是动态加载的**。你提到"supabase"，Supabase Power 自动激活；任务结束后自动卸载，不占资源。不需要你手动开关，也不需要写 JSON 配置。

### 安装一个 Power

打开 Kiro 的 Power 面板（侧边栏的闪电图标），你会看到合作伙伴和社区提供的 Powers 列表。找到 Supabase，点击 Install，完事。

安装后 Kiro 会引导你做一次授权——填 Supabase 项目的 URL 和 API Key，保存在你的本地配置里。整个流程不到两分钟。

### 实战：用 Supabase Power 操作数据库

HabitFlow 之前用的 SQLite，现在要迁移到 Supabase。装好 Power 之后，直接在 Chat 面板里说：

> 把 HabitFlow 的数据库从 SQLite 迁移到 Supabase，表结构保持不变。

Kiro 检测到"Supabase"关键词，自动激活 Supabase Power。接下来它做的事情跟普通 Chat 不一样了——它能直接读取你 Supabase 项目的表结构、执行 SQL migration、验证数据迁移结果。

你会看到 Kiro 的工具栏里多了一组 Supabase 专属工具：`supabase_query`、`supabase_schema`、`supabase_migrate`。它用这些工具执行迁移，而不是靠猜你的数据库长什么样。

### 常用 Powers 一览

目前生态里比较实用的几个：

| Power    | 能力                         | 典型场景           |
| -------- | ---------------------------- | ------------------ |
| Supabase | 读写数据库、管理 Schema      | 数据库操作、迁移   |
| AWS      | 管理 Lambda、S3、DynamoDB 等 | 部署、云资源管理   |
| Postman  | 读取 API 集合、生成请求代码  | API 对接、接口测试 |
| Figma    | 读取设计稿、生成 UI 代码     | 设计还原、组件开发 |
| Stripe   | 管理 Products、Subscriptions | 支付集成           |
| Datadog  | 查询日志和监控数据           | 排查线上问题       |

安装方式都一样：Power 面板里点一下。社区贡献的 Power 也可以从 GitHub URL 安装。

## Agent Skills：把重复工作流封装起来

Power 解决的是"连外部工具"的问题。但还有一种痛点：**你总在重复做同一类事**。

比如 HabitFlow 项目里，每次加新表都要做一整套：写 migration SQL、创建 TypeScript 类型定义、写 CRUD 路由、加测试。每一步都不难，但每次都要从头描述一遍，烦。

Agent Skills 就是解决这个问题的——把重复工作流封装成一个可复用的"技能"。

### Skills 的组织方式

Skills 遵循开放的 agentskills.io 标准。每个 Skill 就是一个文件夹，里面包含一个 `SKILL.md` 文件来描述技能的用途和触发条件。

Skills 支持两种作用域：

- **工作区级**：放在项目的 `.kiro/skills/` 目录，只对当前项目生效
- **全局级**：放在 `~/.kiro/skills/` 目录，对所有项目生效

### 创建一个自定义 Skill

我们来创建一个"数据库迁移"Skill，让 Kiro 以后一键处理新建表的完整流程。

首先在项目里创建目录：

```
.kiro/skills/create-table/
```

然后写 `SKILL.md`：

```markdown
---
name: create-table
description: 为 HabitFlow 创建新的数据库表，包含完整流程
trigger: 创建新表、新建表、add table、create table
---

## 工作流

当用户要求创建新表时，按以下步骤执行：

1. 确认表名和字段定义，询问用户是否需要关联已有表
2. 在 supabase/migrations/ 目录生成迁移 SQL 文件
3. 在 src/types/database.ts 中添加对应的 TypeScript 类型
4. 在 src/routes/ 下创建 CRUD 路由文件
5. 生成基础的集成测试文件
6. 执行迁移并验证

## 规范

- 表名使用 snake_case 复数形式（如 habits, checkins）
- 所有表必须有 id, created_at, updated_at 字段
- 外键字段命名格式：关联表名的单数 + \_id
- 路由文件放在 src/routes/ 目录，命名与表名对应
```

注意那个 `trigger` 字段——Kiro 用它来判断什么时候应该自动激活这个 Skill。当你在 Chat 里说"帮我创建一个 notifications 表"，Kiro 匹配到关键词，自动加载这个 Skill 的工作流。

保存后 Skill 立刻生效，不需要重启。下次你说"创建新表"，Kiro 会按照 SKILL.md 里定义的六个步骤依次执行。

### Skills 的渐进式披露

Skills 的加载遵循三步机制：

1. **Discovery**：Kiro 扫描 `.kiro/skills/` 目录，建立索引
2. **Activation**：根据对话内容匹配 trigger 关键词，激活相关 Skill
3. **Execution**：按 SKILL.md 中定义的工作流执行

这个过程对用户是透明的——你不需要手动选择 Skill，Kiro 自己判断该用哪个。如果多个 Skill 同时匹配，它会告诉你它打算用哪个，你可以切换。

## MCP：连接一切外部工具

Powers 和 Skills 覆盖了大部分常见场景，但如果你要对接的工具不在官方列表里呢？比如公司内部的 API Gateway、自己搭的 Elasticsearch、或者一个冷门的 SaaS 服务。

这时候就该 **MCP（Model Context Protocol）** 出场了。

### MCP 是什么

MCP 是一个开放协议，让 AI 工具能够连接外部服务。你可以把它理解成一个"万能插座"——只要对方提供 MCP Server，Kiro 就能连上去用。

和 Power 的区别：Power 是 Kiro 官方打包好的能力，开箱即用；MCP 是你自己接线，灵活但需要配置。

### 配置一个 MCP Server

以连接 AWS 文档搜索为例。在 Kiro 的 MCP 管理界面（设置 → MCP Servers），添加一个新配置：

```json
{
  "mcpServers": {
    "aws-docs": {
      "command": "npx",
      "args": ["-y", "@aws/mcp-server-aws-docs"]
    }
  }
}
```

保存后 Kiro 会启动这个 MCP Server，管理界面会显示连接状态。连接成功后，你在对话中提到 AWS 相关问题，Kiro 就能调用这个 Server 搜索最新的 AWS 文档。

MCP 的配置虽然要手写 JSON，但好处是生态开放——社区有大量现成的 MCP Server 可以直接用，覆盖 GitHub、Slack、Notion、数据库、搜索引擎等几十种服务。

### MCP 管理界面

Kiro 提供了一个可视化的 MCP 管理面板，你可以看到：

- 已配置的所有 MCP Server 及其连接状态
- 每个 Server 提供的工具列表
- 实时日志，方便排查连接问题

如果某个 Server 挂了，面板上会直接标红，不用猜。

## 三层扩展机制对比

用一个例子说清三者的区别。假设你想让 Kiro 帮你管理数据库：

- **Powers**：官方打包好的 Supabase Power，一键安装，自动连接，最适合常用服务
- **Skills**：把你团队"创建新表"的标准流程封装起来，让 Kiro 每次都按规范走
- **MCP**：如果你用的是 PlanetScale 或 TiDB 这种没有官方 Power 的数据库，自己配一个 MCP Server 连上去

三层可以组合使用。Supabase Power 提供数据库连接能力，自定义 Skill 定义工作流规范，MCP 补充 Power 覆盖不到的外部工具。

## 和 Cursor 扩展机制的对比

Cursor 的扩展主要靠两件事：VS Code 插件生态 + `.cursorrules` 项目配置。好处是 VS Code 插件海量，坏处是这些插件不是为 AI 设计的——AI 用不了插件的功能。

Kiro 的思路不同：Powers、Skills、MCP 都是**给 AI 用的工具**，不是给人用的 UI 插件。每个扩展都会转化为 AI 可以调用的工具函数，AI 真的能用它们干活。

| 维度            | Cursor              | Kiro                          |
| --------------- | ------------------- | ----------------------------- |
| 扩展来源        | VS Code 插件市场    | Power 市场 + Skills + MCP     |
| AI 能否使用插件 | 大部分不能          | 全部可以                      |
| 自定义工作流    | .cursorrules 文件   | Skills（SKILL.md）            |
| 外部工具连接    | 需手动配置 MCP JSON | Power 一键安装 / MCP 手动配置 |
| 动态加载        | 无                  | Power 按需激活/卸载           |

简单说，Cursor 的扩展是给人用的，Kiro 的扩展是给 AI 用的。这是两种不同的设计哲学。

## Trade-off 总结

**适用场景**：项目依赖外部服务（数据库、云资源、设计工具）、团队有重复的工作流需要标准化、需要 AI 真正操作外部工具而不是只生成代码。

**不适用场景**：纯前端项目、不依赖外部服务的小项目、团队没有标准化的工作流。

**三层机制选择**：常用服务优先用 Powers（开箱即用，维护成本低）；团队重复流程用 Skills 封装（一次定义，反复使用）；Powers 覆盖不到的工具用 MCP（灵活但需要自己配置维护）。

**对比 Cursor**：Cursor 的 MCP 支持也在快速进步，但缺乏 Power 这种一键安装的体验和 Skill 这种工作流封装能力。Kiro 的扩展生态目前更面向 AI Agent 的实际操作需求。

**实践建议**：先从官方 Powers 开始，解决最频繁的外部服务对接需求。等你在项目里发现某个操作重复了三次以上，再考虑封装成 Skill。MCP 留给 Power 覆盖不到的长尾场景。不要一上来就把三个都用上——扩展越多，调试成本越高。
