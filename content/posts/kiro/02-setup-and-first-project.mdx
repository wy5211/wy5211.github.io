上篇聊了 Kiro 的定位——它不是又一个聊天写代码的工具，而是用 Spec 把 AI 编码拉到工程化轨道上的 IDE。道理听完了，这篇该动手了。

装软件这事听着无聊，但 Kiro 的安装和首次配置有几个容易踩的坑。我把自己从下载到跑通第一个 API 的完整过程记下来，你跟着走一遍，半小时以内能搞定。

## 下载安装：比你想象的快

去 [kiro.dev](https://kiro.dev) 点下载，三个平台都支持：

- **macOS**：Intel 和 Apple Silicon 都有，自动识别架构
- **Windows**：10 和 11，标准安装包
- **Linux**：需要 glibc 2.39+（Ubuntu 24.04+、Fedora 40+ 没问题）

我用的 macOS，下载 `.dmg` 拖进 Applications 就完事了。如果你更喜欢命令行，也可以用 CLI 安装：

```bash
curl -fsSL https://cli.kiro.dev/install | bash
```

安装完第一次打开，Kiro 会问你有没有在用 VS Code。这个问题很关键——

## VS Code 迁移：一键导入，但别全选

Kiro 基于 VS Code 内核，所以你的设置、快捷键、扩展大部分都能直接用。点「Import from VS Code」，它会扫描你现有的配置。

**但我要提醒一点**：别无脑全选。Kiro 有自己的一套 AI 相关面板和快捷键，如果你把 VS Code 的快捷键全部覆盖过来，可能会冲突。我的做法是：

1. **设置（Settings）**——导入，包括字体、主题、编辑器偏好
2. **扩展（Extensions）**——只导入你确定需要的，比如 GitLens、ESLint、Prettier
3. **快捷键（Keybindings）**——先不导入，用一段时间再按需调

迁移完成后，你会看到一个熟悉又陌生的界面——编辑器区域和 VS Code 几乎一样，但左侧多出了一些东西。

## 界面导览：三个关键面板

Kiro 在 VS Code 的基础上加了三个核心面板，搞清楚它们的位置和作用，后面就不会迷路：

**Spec 面板**（左侧栏）：管理所有 Spec 文件。Spec 是 Kiro 的灵魂——你可以理解为「让 AI 先写设计文档，再写代码」的工作流入口。后面第 3 篇会深入讲，现在只要知道它在哪就行。

**Chat 面板**（右侧栏）：和 AI 对话的地方。跟 Cursor 的 Chat 类似，但 Kiro 在 Chat 里也能触发 Spec 流程，不只是问答。底部可以切换模型。

**Steering 文件**（项目根目录 `.kiro/steering/`）：给 AI 的「项目说明书」。告诉 Kiro 你用什么技术栈、有什么编码规范。这个第 4 篇展开讲。

另外还有一个 **Powers 市场**，类似扩展商店，可以一键安装 Supabase、AWS、Postman 等能力包。按 `Cmd+Shift+P` 搜索「Powers」就能找到。

## 模型选择：不用纠结，先用 Auto

Chat 面板底部有个模型选择器，Kiro 目前支持这些模型：

| 模型              | 特点                 | 消耗倍率 |
| ----------------- | -------------------- | -------- |
| Auto              | 自动选择最合适的模型 | 变动     |
| Claude Opus 4.7   | 最强推理能力         | 2.2x     |
| Claude Sonnet 4.5 | 默认，均衡           | 1.3x     |
| Claude Haiku 4.5  | 快速响应             | 0.4x     |
| DeepSeek 3.2      | 便宜，中文不错       | 0.25x    |

我的建议：**日常用 Auto 就行**。Kiro 的 Auto 模式会根据任务复杂度自动匹配——简单的问答用 Haiku，复杂的设计用 Sonnet 或 Opus。你只有在对模型输出不满意、想手动控制的时候才需要切换。

想省钱？用 DeepSeek 做简单任务。想要最强推理？手动切 Opus。仅此而已。

## 创建 HabitFlow 项目

好，界面认识了，模型选了，现在来干正事——创建我们的主线项目 HabitFlow。

在终端里：

```bash
mkdir habitflow && cd habitflow
npm init -y
```

然后打开 Kiro，选择「Open Folder」，选中 `habitflow` 目录。Kiro 会识别这是一个新项目，可能会提示你创建 Steering 文件——先跳过，后面第 4 篇专门讲。

项目打开了，空的。接下来我们要用 Kiro 干第一件实事。

## 第一个 Spec：用 Quick Plan 生成注册 API

Kiro 有两种 Spec 模式：**Quick Plan**（快速计划）和 **Feature Spec**（完整规格）。Quick Plan 适合简单、明确的任务——比如「给我写个注册接口」。Feature Spec 适合复杂功能——需要需求分析、架构设计、任务拆分。

我们现在只需要一个注册 API，Quick Plan 正好。

在 Chat 面板输入：

```
用 Hono 框架创建一个用户注册 API：
- POST /api/register
- 接收 username 和 password
- 密码用 bcrypt 加密存储
- 用 SQLite 存数据
- 返回合适的错误信息（用户名重复等）
```

Kiro 会生成一个计划，列出它准备做的事——创建哪些文件、修改哪些文件、安装什么依赖。你确认后它就开始执行。

这里有个关键选择：

## Autopilot vs Supervised：什么时候用哪个

Kiro 执行计划时有两种模式：

**Autopilot（自动驾驶）**：AI 按计划自主完成所有步骤，中间不需要你确认。适合你信任计划内容、任务明确的时候。速度快，但你得事后 review 所有改动。

**Supervised（监督模式）**：每修改一个文件都会暂停，等你确认后才继续。适合你不放心 AI 输出、或者在学习阶段想看清楚它每一步在做什么。

**我的选择**：第一次用 Kiro，选 Supervised。你能看到它创建了什么文件、写了什么代码、装了什么依赖。等用熟了再切 Autopilot。

在 Supervised 模式下，Kiro 会逐步展示：

1. 安装 `hono`、`better-sqlite3`、`bcryptjs` 依赖
2. 创建 `src/db.ts` 初始化数据库和用户表
3. 创建 `src/routes/auth.ts` 实现注册接口
4. 创建 `src/index.ts` 作为入口文件

每一步你都能看到完整代码 diff，点 Approve 才继续。

全部完成后，跑一下：

```bash
npx tsx src/index.ts
```

用 curl 测试注册接口：

```bash
curl -X POST http://localhost:3000/api/register \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser","password":"mypassword"}'
```

看到 `{"success":true}` 的响应——第一个 API 就跑通了。

回顾一下刚才发生了什么：你没有逐行写代码，而是描述需求 → 确认计划 → 审核 AI 的实现。这就是 Spec-driven development 的简化版。当需求更复杂时，你会用 Feature Spec 走完整的三阶段流程（需求→设计→任务），那是第 3 篇的内容。

## Trade-off 总结

**适用场景**：

- 从 VS Code 迁移过来的开发者，几乎零学习成本
- 想用 AI 辅助写代码，但又不放心完全交给 AI 的人——Supervised 模式很适合
- 新项目从零开始，Kiro 的 Spec 流程最能发挥价值

**不适用场景**：

- 项目已经很大、代码库复杂——先从局部功能试用，别一上来就让 AI 改核心逻辑
- 团队里有不用 VS Code 系的人——Kiro 目前没有 Web 版，只有桌面客户端
- 纯后端/DevOps 场景——Kiro 对前端开发更友好

**对比替代方案**：

- 相比 Cursor：多了 Spec 流程和 Steering 系统，但扩展生态没 Cursor 丰富
- 相比 Windsurf：工作流更结构化，但自由度稍低
- 相比 GitHub Copilot：功能更全面，但需要切换编辑器

**实践建议**：

- 第一次装完选 Supervised 模式，认真看每一步 AI 做了什么
- 模型先用 Auto，别纠结选择
- VS Code 扩展别全导入，只带最必要的几个
- Steering 文件尽早配——下一篇就教你配
