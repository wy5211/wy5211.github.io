上篇用 Spec 流程完成了 HabitFlow 的习惯打卡功能，需求→设计→任务三阶段走下来，代码质量明显比直接让 AI 写好很多。但你有没有注意到一个问题：每次新开对话，AI 总要你重新告诉它"项目用 Hono 框架、TypeScript 严格模式、错误处理用 Result 类型"。

这不是 AI 笨，是它没有**项目记忆**。

这篇讲 Kiro 怎么解决这个问题——**Steering 引导系统**给 AI 加上持久记忆，**Hooks 自动化钩子**让代码规范自动执行。

## Steering 到底在解决什么问题

你跟一个新同事合作，第一天跟他说"我们项目用 TypeScript、API 路由放在 `src/routes/` 下、数据库用 Drizzle ORM"，他记住了。第二天他又问"ORM 用哪个来着？"第三天他提交的代码还是用了 Prisma。

AI 也一样。对话上下文有限，新对话就是一张白纸。你每次都要花 2 分钟重新描述项目背景，积累下来浪费的时间不少。

**Steering 就是一份给 AI 的入职文档**——项目用什么技术栈、遵循什么规范、文件怎么组织，全部写进 Markdown 文件，AI 每次对话都会自动读取。

## 三个基础 Steering 文件

在项目根目录运行 `kiro init`，Kiro 会在 `.kiro/steering/` 下生成三个文件：

### product.md — 告诉 AI 我们在做什么产品

```markdown
# HabitFlow

A habit tracking app that helps users build daily routines.

## Target Users

- People who want to build consistent daily habits
- Users who prefer simple, minimal UI over feature-heavy apps

## Core Features

- Create habits with custom frequency and goals
- Daily check-in with streak tracking
- History view with calendar heatmap
```

这个文件决定了 AI 对产品的理解深度。你写"习惯打卡应用"，AI 不会突然给你加社交功能；你写"极简 UI"，它生成代码时不会塞一堆 Material 组件。

### tech.md — 告诉 AI 用什么技术

```markdown
## Tech Stack

- Runtime: Bun
- Framework: Hono
- Language: TypeScript (strict mode)
- ORM: Drizzle
- Database: SQLite (dev) / PostgreSQL (prod)
- Testing: Vitest

## Conventions

- Use Result type for error handling, never throw
- All API responses follow `{ code, data, message }` format
- Use Zod for request validation
```

这是我改动最频繁的文件。每次发现 AI 又用了 `try/catch` 而不是 Result 类型，我就把这条规则加到 tech.md 里——下不为例。

### structure.md — 告诉 AI 项目怎么组织的

```markdown
src/
routes/ # API routes, one file per resource
services/ # Business logic
db/
schema.ts # Drizzle schema definitions
migrations/ # Generated migrations
utils/ # Shared utilities
types/ # TypeScript type definitions
```

有了这个，AI 不会把路由逻辑写到 service 里，也不会把数据库 schema 散落在各处。

## 自定义 Steering 文件：让规范更精准

三个基础文件覆盖了"做什么"和"用什么"，但还不够细。比如你的 API 设计有特殊约定、编码风格有团队规范——这些用自定义 Steering 文件解决。

在 `.kiro/steering/` 下新建 `api-design.md`：

```markdown
# API Design Guidelines

## Naming

- Use kebab-case for URLs: `/habit-checkins` not `/habitCheckins`
- Use camelCase for JSON fields: `{ "streakCount": 3 }`

## Response Format

All endpoints return:
`{ "code": number, "data": T, "message": string }`

## Error Codes

- 400: Validation error (include field name in message)
- 404: Resource not found
- 409: Conflict (e.g., duplicate check-in)
```

Kiro 会自动读取 `.kiro/steering/` 下的所有 Markdown 文件，不限文件名。你可以按主题拆分：`code-style.md`、`testing.md`、`git-conventions.md`——越细越好管理。

关键经验：**不要写空话**。"代码要写得好"没有信息量。"函数不超过 20 行、圈复杂度不超过 5"才是有效规则。

## 作用域：不是所有规范都该放在项目里

Steering 有三层作用域：

**工作区级别**（`.kiro/steering/`）：项目特定的规范，跟着代码仓库走，团队共享。比如"这个项目用 Hono"——放在这里。

**全局级别**（`~/.kiro/steering/`）：你个人的偏好，所有项目生效。比如"注释用英文"、"git commit message 用 conventional commits 格式"——这些是你的习惯，不是项目要求。

**团队级别**（通过 MDM 推送）：企业场景下，IT 管理员可以统一推送 Steering 文件到团队成员的机器上。比如"所有 API 必须通过安全审计检查"——这类强制规范。

实际操作中，我 80% 的规则放在工作区级别，20% 放全局。团队级别除非你在企业环境，否则用不到。

一个容易犯的错：把个人偏好放到工作区 Steering 里。比如你喜欢用 Tab 缩进，但团队统一 2 空格——这种就放全局，别污染项目的 Steering 文件。

## Hooks：让规范不只是"建议"

Steering 是告诉 AI 应该怎么做，但 AI 偶尔还是会"忘"。Hooks 是更强硬的手段——在特定事件发生时**自动执行**预设操作。

### Hook 怎么配

在 Kiro 的 Hooks 面板里，你可以用自然语言描述你想做什么：

> 每次保存 TypeScript 文件时，自动运行 Biome 检查，如果有格式问题就自动修复。

Kiro 会理解你的意图，生成对应的 Hook 配置。你也可以手动通过表单配置。

### 实战：给 HabitFlow 配一套 Hooks

我为 HabitFlow 配了两个 Hook，覆盖最常见的场景：

**Hook 1：保存时自动格式化**

触发事件：文件保存
执行操作：运行 `bunx biome check --write src/` 并报告结果

这确保每次保存后代码风格都是一致的。以前总有人提交的代码缩进不对、引号风格不统一，现在机器帮你管了。

**Hook 2：提交前跑测试**

触发事件：用户提示提交（commit）
执行操作：运行 `bun test`，如果有失败用例则阻止提交并报告失败的测试

这比 pre-commit hook 更灵活——它只在通过 Kiro 提交时触发，你手动 `git commit` 不受影响。但它会在 AI 生成提交信息之前先跑测试，避免 AI 提交了跑不过的代码。

Hooks 支持的触发事件很丰富：文件保存/创建/删除、工具调用前后、任务执行前后，甚至可以手动触发。你可以根据自己的工作流自由组合。

### Hooks 的边界

Hooks 不是万能的。它运行在 Kiro 内部，适合做轻量级的检查和自动化。如果你的项目需要重量级的 CI 流程（Docker 构建、端到端测试、部署验证），那还是交给 GitHub Actions 或类似工具。

我的原则是：**影响当前文件的检查用 Hooks，影响整个系统的检查用 CI**。

## 把 Steering 和 Hooks 结合起来

真正的威力在于两者配合。Steering 告诉 AI "我们用 Biome 做代码检查"，Hook 在保存时自动执行 Biome。一个是规范定义，一个是规范执行。

我给 HabitFlow 的完整配置：

- `product.md` — 产品定位和功能边界
- `tech.md` — 技术栈和编码约定
- `structure.md` — 目录结构和命名规范
- `api-design.md` — API 设计约定（自定义）
- Hook: 保存时 Biome 格式化
- Hook: 提交前跑测试

这套配置大概花了 30 分钟写完，但省下的时间远不止 30 分钟——之后每次开新对话，AI 都知道项目背景；每次保存文件，代码风格都自动统一。

## Trade-off 总结

**适用场景：** 团队项目（规范需要共识）、长期维护的项目（AI 需要持续记忆上下文）、有明确编码规范的团队。Hooks 特别适合格式化和 lint 这种高频重复操作。

**不适用场景：** 一次性脚本、快速原型验证（这时候加 Steering 反而增加认知负担）、个人小项目（如果规范只有你自己知道，直接在对话里说就行）。

**对比替代方案：** Cursor 有 `.cursorrules` 文件，功能类似 Steering，但没有结构化的三分文件划分，也没有 Hooks 自动执行能力。Windsurf 的 Rules 功能也类似，但作用域管理没有 Kiro 灵活。Steering + Hooks 的组合优势在于**规范的定义和执行是一体的**。

**实践建议：** 先写三个基础文件，用几天看看 AI 还在哪里犯重复错误，再有针对性地加自定义 Steering。Hooks 从一个最简单的开始（比如保存时 lint），确认没问题再加更多。别一次性配太多——规则越多，AI 处理上下文的开销越大，响应也会变慢。
