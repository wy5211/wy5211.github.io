上一篇文章我们设计了电商系统的用户表, 商品表和订单表，但这些都是孤立的表。实际业务中，这些表之间存在着紧密的关联：用户有多个订单，订单包含多个商品，商品属于某个分类。如果表之间没有建立关系，就会导致数据不一致, 查询困难, 维护成本高等问题。

关系型数据库的核心价值就在于"关系"二字。外键约束保证了数据的一致性，JOIN 查询让我们能够灵活地关联多张表的数据。但关系设计也是个技术活，外键设置错了可能导致数据级联删除，JOIN 写不好可能导致查询很慢。让我们通过电商订单系统这个真实案例，系统学习表关系设计的方方面面。

## 表关系类型

在数据库中，表之间的关系主要有三种：一对一, 一对多, 多对多。怎么判断两个表是什么关系呢？一个实用的方法是问自己：A 表的一条记录，对应 B 表的几条记录？

比如用户和订单的关系：一个用户可以有多个订单，但一个订单只属于一个用户。这就是典型的一对多关系。

订单和商品的关系就复杂了：一个订单包含多个商品，一个商品也可以出现在多个订单中。这就是多对多关系，需要通过中间表来实现。

用户和用户资料的关系：一个用户对应一个资料，一个资料也只属于一个用户。这是一对一关系。

## 一对一关系

什么时候需要一对一关系？最常见的情况是：某些字段不常用，或者某些敏感信息需要分开存储。电商系统中，用户的登录信息和详细资料就可以分开。

比如用户的登录凭证（邮箱, 密码）每次登录都要查询，而用户的详细资料（简介, 地址, 生日）很少用到。把它们放在一张表会导致查询时总是加载不需要的数据。

一对一关系有两种实现方式：共享主键和唯一外键。共享主键更简洁，唯一外键更灵活。

看一下共享主键的例子：

```sql
-- 用户表（存储核心登录信息）
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  full_name VARCHAR(100),
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 用户资料表（共享主键）
CREATE TABLE user_profiles (
  user_id UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  bio TEXT,
  website VARCHAR(255),
  location VARCHAR(100),
  birth_date DATE,
  gender VARCHAR(20),
  avatar_url TEXT,
  updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 插入用户
INSERT INTO users (email, password_hash, full_name)
VALUES ('zhang@example.com', 'hash_xyz', '张三');

-- 插入用户资料（使用相同的主键值）
INSERT INTO user_profiles (user_id, bio, location)
VALUES (
  (SELECT id FROM users WHERE email = 'zhang@example.com'),
  '全栈开发者，热爱技术',
  '深圳'
);

-- 查询用户及其资料（LEFT JOIN 确保没有资料的用户也能查出来）
SELECT
  u.id,
  u.email,
  u.full_name,
  p.bio,
  p.location
FROM users u
LEFT JOIN user_profiles p ON u.id = p.user_id
WHERE u.email = 'zhang@example.com';
```

这里有几个关键点：

第一，user_profiles 的主键就是 user_id，它同时也是外键指向 users 表。这样天然保证了一对一关系，因为主键本身就是唯一的。

第二，ON DELETE CASCADE 意味着删除用户时，对应的用户资料也会自动删除。这符合业务逻辑：用户没了，资料自然也没必要保留。

第三，查询时用 LEFT JOIN 而不是 INNER JOIN。因为有些用户可能还没有填写资料，用 INNER JOIN 就会漏掉这些用户。

再看唯一外键的实现方式：

```sql
-- 用户资料表（独立主键 + 唯一外键）
CREATE TABLE user_profiles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID UNIQUE NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  bio TEXT,
  location VARCHAR(100)
);
```

这里 user_profiles 有自己的主键 id，user_id 是一个外键，但加了 UNIQUE 约束。这个 UNIQUE 约束确保了一个用户只能对应一条资料记录。

这两种方式有什么区别呢？共享主键更紧凑，查询时少一次 JOIN，但不够灵活。唯一外键可以独立存在，未来如果需要扩展成一对多（比如一个用户有多个资料），只需要去掉 UNIQUE 约束即可。

## 一对多关系

一对多关系是最常见的，电商系统到处都是：用户和订单, 分类和商品, 订单和订单项。

设计一对多关系时，关键是在"多"的那一边添加外键指向"一"的那一边。订单表需要有 user_id 字段指向用户表，商品表需要有 category_id 字段指向分类表。

```sql
-- 用户表
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(255) UNIQUE NOT NULL,
  full_name VARCHAR(100)
);

-- 订单表（外键指向用户）
CREATE TABLE orders (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  order_number VARCHAR(50) UNIQUE NOT NULL,
  total NUMERIC(10, 2) NOT NULL,
  status VARCHAR(20) DEFAULT 'pending',
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 为外键创建索引（很重要！）
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 插入数据
INSERT INTO users (email, full_name)
VALUES ('zhang@example.com', '张三');

INSERT INTO orders (user_id, order_number, total)
VALUES
  (
    (SELECT id FROM users WHERE email = 'zhang@example.com'),
    'ord_a1b2c3d4',
    497.00
  ),
  (
    (SELECT id FROM users WHERE email = 'zhang@example.com'),
    'ord_e5f6g7h8',
    299.00
  );
```

这里有几个关键点：

第一，外键 user_id 在 orders 表中，不在 users 表中。这符合直觉：订单属于用户，所以订单表记录它属于哪个用户。

第二，ON DELETE CASCADE 表示删除用户时，相关订单也会被删除。这个设置要看业务场景：如果订单是重要历史数据，可能应该用 ON DELETE SET NULL 或者 ON DELETE RESTRICT。

第三，为外键创建索引 idx_orders_user_id。这个索引非常重要！如果没有这个索引，每次查询用户的订单时都要全表扫描 orders 表，性能会很差。

实际查询时，根据需求选择不同的 JOIN 类型：

```sql
-- INNER JOIN：只查询有订单的用户
SELECT
  u.id,
  u.email,
  u.full_name,
  o.order_number,
  o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id
ORDER BY o.created_at DESC;

-- LEFT JOIN：查询所有用户，包括没有订单的
SELECT
  u.id,
  u.email,
  u.full_name,
  COUNT(o.id) as order_count,
  COALESCE(SUM(o.total), 0) as total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id
ORDER BY total_spent DESC;
```

第一个查询用 INNER JOIN，因为它只关心有订单的用户。第二个查询用 LEFT JOIN，因为它要显示所有用户的订单统计，包括那些还没下单的用户。COALESCE 函数处理 NULL 值，确保没有订单的用户显示为 0 而不是 NULL。

另一个常见的一对多关系是自关联，比如商品分类的层级结构：

```sql
-- 商品分类表（支持多级分类）
CREATE TABLE categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(100) NOT NULL,
  parent_id UUID REFERENCES categories(id) ON DELETE SET NULL,
  level INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 插入分类数据
INSERT INTO categories (id, name, parent_id, level) VALUES
  ('cat_root', '电子产品', NULL, 0),
  ('cat_phone', '手机', NULL, 1),
  ('cat_laptop', '电脑', NULL, 1);

-- 更新父级关系
UPDATE categories SET parent_id = 'cat_root' WHERE id IN ('cat_phone', 'cat_laptop');

-- 查询某个分类的所有子分类
SELECT
  c.id,
  c.name,
  p.name as parent_name
FROM categories c
LEFT JOIN categories p ON c.parent_id = p.id
WHERE c.parent_id = 'cat_root';
```

这里 categories 表的 parent_id 字段指向同一张表的 id 字段，这就是自关联。ON DELETE SET NULL 表示删除父分类时，子分类的 parent_id 设为 NULL，子分类不会被删除。这样更安全，因为可能只是想调整分类结构，而不是删除整个分类树。

## 多对多关系

多对多关系在电商系统中也很常见：订单和商品, 用户和角色, 商品和标签。处理多对多关系需要一张中间表（也叫关联表或 junction table）。

电商系统中，一个订单包含多个商品，一个商品也可以出现在多个订单中。这需要一张 order_items 表作为中间表：

```sql
-- 订单表
CREATE TABLE orders (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  order_number VARCHAR(50) UNIQUE NOT NULL,
  total NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 商品表
CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  price NUMERIC(10, 2) NOT NULL,
  stock INTEGER DEFAULT 0
);

-- 订单项表（中间表）
CREATE TABLE order_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE RESTRICT,
  quantity INTEGER NOT NULL CHECK (quantity > 0),
  unit_price NUMERIC(10, 2) NOT NULL,
  subtotal NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,

  -- 防止同一订单重复添加同一商品
  UNIQUE (order_id, product_id)
);

-- 创建索引（提升查询性能）
CREATE INDEX idx_order_items_order ON order_items(order_id);
CREATE INDEX idx_order_items_product ON order_items(product_id);

-- 插入数据
INSERT INTO orders (user_id, order_number, total)
VALUES (
  (SELECT id FROM users WHERE email = 'zhang@example.com'),
  'ord_a1b2c3d4',
  497.00
);

INSERT INTO products (id, name, price, stock) VALUES
  ('prod_111', '无线蓝牙耳机', 99.00, 1250),
  ('prod_222', '智能手表', 299.00, 500);

-- 添加订单项
INSERT INTO order_items (order_id, product_id, quantity, unit_price, subtotal)
VALUES
  (
    (SELECT id FROM orders WHERE order_number = 'ord_a1b2c3d4'),
    'prod_111',
    2,
    99.00,
    198.00
  ),
  (
    (SELECT id FROM orders WHERE order_number = 'ord_a1b2c3d4'),
    'prod_222',
    1,
    299.00,
    299.00
  );
```

这个中间表设计有几个关键点：

第一，order_items 表有自己的主键 id，而不是用 (order_id, product_id) 作为组合主键。这样更灵活，未来如果需要记录同一商品多次添加（比如不同时段加购），可以去掉 UNIQUE 约束。

第二，ON DELETE CASCADE 和 ON DELETE RESTRICT 的选择很关键。删除订单时，订单项应该删除（CASCADE），但删除商品时，如果有订单包含该商品，应该禁止删除（RESTRICT）。这保护了历史订单数据的完整性。

第三，为外键创建索引。order_id 和 product_id 都会频繁用于 JOIN 查询，必须加索引。

查询多对多关系时，需要 JOIN 三张表：

```sql
-- 查询订单 ord_a1b2c3d4 的所有商品
SELECT
  o.order_number,
  p.name as product_name,
  oi.quantity,
  oi.unit_price,
  oi.subtotal
FROM orders o
INNER JOIN order_items oi ON o.id = oi.order_id
INNER JOIN products p ON oi.product_id = p.id
WHERE o.order_number = 'ord_a1b2c3d4';

-- 查询商品 prod_111 出现在哪些订单中
SELECT
  p.name as product_name,
  o.order_number,
  o.total as order_total,
  oi.quantity,
  o.created_at
FROM products p
INNER JOIN order_items oi ON p.id = oi.product_id
INNER JOIN orders o ON oi.order_id = o.id
WHERE p.id = 'prod_111'
ORDER BY o.created_at DESC;
```

## JOIN 类型详解

JOIN 查询是关系型数据库的核心操作，理解不同 JOIN 类型的区别非常重要。

INNER JOIN 只返回匹配的行。如果用户没有订单，就不会出现在结果中。LEFT JOIN 返回左表所有行，右表不匹配的填充 NULL。这是最常用的 JOIN 类型，因为它不会丢失数据。

```sql
-- INNER JOIN：只查询有订单的用户
SELECT u.email, o.order_number
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN：查询所有用户，包括没有订单的
SELECT u.email, o.order_number
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- 查询没有订单的用户（找出 LEFT JOIN 中右表为 NULL 的记录）
SELECT u.email
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;
```

第三个查询很实用，它能找出那些注册了但还没下单的用户，是精准营销的目标群体。

实际开发中，RIGHT JOIN 和 FULL JOIN 很少用。RIGHT JOIN 可以用 LEFT JOIN 替代（调换表顺序），FULL JOIN 的使用场景更少。大部分情况下，INNER JOIN 和 LEFT JOIN 就够了。

多表 JOIN 时，要注意顺序和性能：

```sql
-- 查询订单的完整信息（用户, 订单, 商品）
SELECT
  u.email as user_email,
  o.order_number,
  p.name as product_name,
  oi.quantity,
  oi.unit_price
FROM users u
INNER JOIN orders o ON u.id = o.user_id
INNER JOIN order_items oi ON o.id = oi.order_id
INNER JOIN products p ON oi.product_id = p.id
WHERE o.order_number = 'ord_a1b2c3d4';
```

这个查询 JOIN 了 4 张表，看起来很复杂，但 PostgreSQL 会自动优化执行顺序。我们只需要确保 JOIN 条件正确，有合适的索引。

## 外键约束的级联操作

外键约束不仅保证数据完整性，还提供了级联操作，自动处理相关记录的更新和删除。

有四种主要的级联操作：

CASCADE：删除/更新父记录时，自动删除/更新所有子记录。适用于强绑定关系，比如订单和订单项。

SET NULL：删除/更新父记录时，子记录的外键设为 NULL。适用于可空关系，比如商品和分类（分类删除后，商品的分类可以设为 NULL）。

RESTRICT：有子记录时，禁止删除/更新父记录（默认行为）。适用于需要保护的数据，比如商品和订单（有订单的商品不能删除）。

NO ACTION：类似 RESTRICT，但延迟到事务结束时检查。

```sql
-- 级联删除示例
CREATE TABLE orders (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  order_number VARCHAR(50) UNIQUE NOT NULL
);

-- 删除用户时，相关订单会自动删除
DELETE FROM users WHERE email = 'zhang@example.com';

-- 限制删除示例
CREATE TABLE order_items (
  id UUID PRIMARY KEY,
  order_id UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE RESTRICT,
  quantity INTEGER NOT NULL
);

-- 有订单项存在时，删除商品会报错
DELETE FROM products WHERE id = 'prod_111';
-- ERROR: update or delete on table "products" violates foreign key constraint
```

级联删除很方便，但也要小心。一个误操作可能导致大量数据被删除。很多系统会选择"软删除"：添加 is_deleted 字段，查询时过滤掉已删除的记录，而不是真的物理删除。

## 关系设计的权衡

关系设计需要在多个目标之间权衡：数据一致性, 查询性能, 维护复杂度。

外键约束保证了数据一致性，但会影响写入性能（每次插入/更新都要检查外键）和灵活性（不能随意删除数据）。有些互联网公司为了性能，会在应用层实现外键逻辑，数据库层面不建外键约束。但这要求数据库访问层必须非常可靠，否则很容易出现数据不一致。

索引能大幅提升查询性能，但会占用存储空间，降低写入性能。不是所有外键都需要索引，要看查询模式。如果一个外键很少用于 JOIN 或 WHERE 条件，可以考虑不加索引。

反范式设计（冗余字段）能减少 JOIN，提升查询性能，但增加了维护复杂度。比如在 orders 表中冗余 user_email，可以避免每次查询订单都要 JOIN users 表。但用户修改邮箱时，需要同步更新所有相关订单的 user_email 字段。

实际设计时，我的建议是：先保证正确性（外键, 约束, 范式化），再优化性能（加索引, 适度反范式）。先让系统跑起来，发现性能瓶颈后再针对性优化。过早优化是万恶之源，但不优化就是坐以待毙。

这篇文章我们学习了如何设计表之间的关系，包括一对一, 一对多, 多对多关系，以及各种 JOIN 查询和级联操作。但光有关系还不够，还需要评估这些关系设计是否合理。下一篇我们会学习范式理论，这套理论能帮助我们判断数据库设计是否足够规范，以及在什么情况下可以适度打破规范以换取性能。
