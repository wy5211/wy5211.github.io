上一篇文章我们学习了范式理论，理解了如何通过规范化设计来避免数据冗余。但规范化的设计有一个代价：查询时需要 JOIN 多张表，可能导致性能下降。比如用户查看订单列表，需要关联 orders、users、order_items 三张表，如果每张表都有上万条数据，查询就会很慢。

这时候索引就派上用场了。索引是数据库性能优化的核心工具，能让查询速度提升几十倍甚至上百倍。但索引不是万能的，它有适用场景，也有使用代价。今天我们就来深入理解索引原理，学会在合适的场景使用合适的索引。

## 什么是索引

想象一下，你要在一本 500 页的技术书里找到"索引原理"这一章。如果没有目录，你只能从第一页开始翻，平均需要翻 250 页才能找到。但如果书的前面有目录，你可以直接看到"索引原理"在第 187 页，几秒钟就找到了。

数据库索引就是书的目录。没有索引时，数据库要查询数据只能全表扫描（Seq Scan），就像从第一页开始翻书。有了索引后，数据库可以通过索引快速定位数据，就像通过目录直接跳到目标页。

但索引不是免费的午餐。建立索引需要占用额外的磁盘空间，每次插入、更新、删除数据时都要同步更新索引，这会影响写入性能。所以不能给所有字段都建索引，需要根据查询模式来选择。

PostgreSQL 默认使用 B-Tree 索引，这是一种平衡树结构，查找时间复杂度是 O(log n)。即使表中有 1000 万条数据，通过 B-Tree 索引查找也只需要几次磁盘 I/O，而全表扫描可能需要读取几万次。

## 主键索引：自动创建的性能优化

我们在设计表时通常会定义主键，PostgreSQL 会自动为主键创建唯一索引。这个索引不仅保证数据的唯一性，还能加速基于主键的查询。

电商系统的用户表中，user_id 是主键：

```sql
-- 创建用户表（主键自动创建索引）
CREATE TABLE users (
  id UUID PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  full_name VARCHAR(100),
  phone VARCHAR(20)
);

-- 插入测试数据
INSERT INTO users (id, email, full_name, phone)
VALUES
  ('usr_8x9k2m', 'zhang@example.com', '张三', '13812341234'),
  ('usr_3n7p4q', 'lisi@example.com', '李四', '13956785678'),
  ('usr_9w2r5t', 'wangwu@example.com', '王五', '13790909090');

-- 基于主键的查询（自动使用索引）
SELECT * FROM users WHERE id = 'usr_8x9k2m';

-- 查看执行计划（可以看到 Index Scan）
EXPLAIN ANALYZE SELECT * FROM users WHERE id = 'usr_8x9k2m';
```

主键索引是最高效的索引，因为主键值唯一且通常很短（UUID 或自增 ID）。查询时数据库可以通过主键直接定位到数据行，不需要扫描大量数据。

## 唯一索引：保证数据唯一性

有时候我们需要保证某个字段的值唯一，但又不想把它设为主键。比如用户的邮箱不能重复，但邮箱不是主键（主键是 UUID）。这时候就可以创建唯一索引。

```sql
-- 创建唯一索引
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- 尝试插入重复邮箱会报错
INSERT INTO users (id, email, full_name, phone)
VALUES ('usr_new', 'zhang@example.com', '赵六', '13612345678');
-- ERROR: duplicate key value violates unique constraint "idx_users_email"

-- 基于邮箱的查询会使用索引
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'zhang@example.com';
```

唯一索引有两个好处：一是保证数据完整性，防止重复数据；二是加速查询，基于唯一字段的查询性能接近主键查询。

电商系统中，订单号、用户邮箱、商品编码这些字段都应该创建唯一索引。它们是业务主键，虽然技术上不是表的主键，但业务上必须唯一。

## 普通索引：加速等值查询

如果经常基于某个字段查询，但这个字段的值不唯一，可以创建普通索引。比如订单表经常按 user_id 查询该用户的所有订单：

```sql
-- 创建订单表
CREATE TABLE orders (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  order_number VARCHAR(50) UNIQUE NOT NULL,
  status VARCHAR(20) DEFAULT 'pending',
  total NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 插入测试数据
INSERT INTO orders (id, user_id, order_number, status, total)
VALUES
  ('ord_a1b2c3d4', 'usr_8x9k2m', 'ORD20260326001', 'paid', 497.00),
  ('ord_b2c3d4e5', 'usr_8x9k2m', 'ORD20260326002', 'pending', 199.00),
  ('ord_c3d4e5f6', 'usr_3n7p4q', 'ORD20260326003', 'paid', 299.00);

-- 为 user_id 创建索引
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 基于 user_id 的查询会使用索引
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 'usr_8x9k2m';
```

普通索引可以大幅减少查询扫描的数据量。没有索引时，数据库需要扫描整个 orders 表；有索引后，只需要读取 user_id 等于 'usr_8x9k2m' 的那几行数据。

但要注意，普通索引适合等值查询（WHERE user_id = ?），对于范围查询（WHERE created_at > ?）和排序（ORDER BY created_at）效果有限，需要考虑其他类型的索引。

## 复合索引：优化多条件查询

实际业务中，查询往往包含多个条件。比如订单列表页可能需要查询某个用户在指定状态下的订单：

```sql
-- 复合条件的查询
SELECT * FROM orders
WHERE user_id = 'usr_8x9k2m'
  AND status = 'paid'
ORDER BY created_at DESC;
```

如果为 user_id 和 status 分别创建单列索引，数据库只能使用其中一个索引（通常是选择性更好的那个），然后对结果进行过滤。更好的方案是创建复合索引：

```sql
-- 创建复合索引（user_id, status）
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- 查看执行计划（可以看到使用了复合索引）
EXPLAIN ANALYZE SELECT * FROM orders
WHERE user_id = 'usr_8x9k2m' AND status = 'paid';
```

复合索引有多个列，查询时可以同时利用这些列快速定位数据。但复合索引有一个重要原则：最左前缀原则。

最左前缀原则意味着，只有查询条件包含索引的最左列，索引才会生效。比如索引是 (user_id, status, created_at)：

```sql
-- 创建三列复合索引
CREATE INDEX idx_orders_user_status_time ON orders(user_id, status, created_at);

-- 这些查询都能使用索引：
-- 使用完整索引
SELECT * FROM orders WHERE user_id = 'usr_8x9k2m' AND status = 'paid';
-- 只用最左列
SELECT * FROM orders WHERE user_id = 'usr_8x9k2m';
-- 用最左两列
SELECT * FROM orders WHERE user_id = 'usr_8x9k2m' AND created_at > '2026-01-01';

-- 但这些查询无法使用索引：
-- 跳过最左列
SELECT * FROM orders WHERE status = 'paid';
-- 只用右边的列
SELECT * FROM orders WHERE created_at > '2026-01-01';
```

设计复合索引时，要把选择性最好的列（区分度最高的列）放在最左边。比如 user_id 的选择性比 status 好（用户很多，状态只有几种），所以把 user_id 放在前面。

复合索引还有一个好处：可以覆盖查询。如果查询的所有字段都在索引中，数据库可以直接从索引返回数据，不需要访问表数据，这叫 Index Only Scan：

```sql
-- 创建覆盖索引（包含查询的所有字段）
CREATE INDEX idx_orders_covering ON orders(user_id, status, total);

-- 这个查询可以直接从索引获取所有数据（Index Only Scan）
EXPLAIN ANALYZE
SELECT status, total FROM orders
WHERE user_id = 'usr_8x9k2m';
```

## 部分索引：减少索引大小

有时候我们只需要索引表中的一部分数据。比如订单表有 1000 万条记录，但 99% 的订单状态都是 completed，我们只需要查询 pending 和 paid 的订单。这时候可以创建部分索引：

```sql
-- 只索引未完成的订单
CREATE INDEX idx_orders_active ON orders(user_id, created_at)
WHERE status IN ('pending', 'paid', 'shipping');

-- 查询未完成订单时使用部分索引
EXPLAIN ANALYZE SELECT * FROM orders
WHERE user_id = 'usr_8x9k2m'
  AND status = 'pending'
ORDER BY created_at DESC;

-- 查询已完成订单时不会使用部分索引（因为没有索引这些数据）
EXPLAIN ANALYZE SELECT * FROM orders
WHERE user_id = 'usr_8x9k2m'
  AND status = 'completed';
```

部分索引的好处是：索引更小，占用磁盘空间更少；查询时扫描的索引数据更少，性能更好；写入时需要维护的索引数据更少，写入性能更好。

电商系统中，部分索引很适合用来索引"活跃"数据。比如只索引最近 30 天的订单、只索引未支付的订单、只索引有库存的商品。这些查询都是业务中最频繁的，而历史数据、已完成数据的查询很少。

## 表达式索引：优化函数查询

有时候查询条件不是直接比较字段值，而是对字段进行函数运算。比如需要查询某个邮箱的用户（不区分大小写）：

```sql
-- 不使用索引的查询（LOWER 函数导致索引失效）
SELECT * FROM users WHERE LOWER(email) = 'ZHANG@EXAMPLE.COM';

-- 创建表达式索引
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- 现在查询可以使用索引
EXPLAIN ANALYZE SELECT * FROM users WHERE LOWER(email) = 'ZHANG@EXAMPLE.COM';
```

没有表达式索引时，数据库必须计算每行的 LOWER(email) 才能比较，无法使用普通索引。创建了表达式索引后，函数计算结果已经存储在索引中，查询可以直接使用。

表达式索引还常用在字符串匹配场景：

```sql
-- 查询姓名包含某个关键词的用户（不区分大小写）
SELECT * FROM users WHERE LOWER(full_name) LIKE '%张%';

-- 创建表达式索引优化 LIKE 查询
CREATE INDEX idx_users_name_trgm ON users USING gin(full_name gin_trgm_ops);

-- 需要启用 pg_trgm 扩展
CREATE EXTENSION pg_trgm;
```

pg_trgm 扩展提供了 trigram 索引，可以高效支持 LIKE 和 ILIKE 查询。电商系统中，商品搜索、用户搜索这些功能都可以用 trigram 索引优化。

但要注意，表达式索引只能用于完全匹配的函数表达式。如果创建了 LOWER(email) 索引，查询 WHERE email = 'xxx' 不会使用这个索引，必须写成 WHERE LOWER(email) = LOWER('xxx') 才能使用。

## 索引的代价：写入性能和磁盘空间

索引能显著提升查询性能，但也有代价。理解这些代价，才能做出合理的索引设计。

第一个代价是磁盘空间。每个索引都是额外的数据结构，占用磁盘空间。如果一张表有 5 个索引，索引的大小可能超过表数据本身。极端情况下，一张表的索引总大小是表数据的 2-3 倍。

```sql
-- 查看表和索引的大小
SELECT
  pg_size_pretty(pg_relation_size('orders')) as table_size,
  pg_size_pretty(pg_total_relation_size('orders')) as total_size,
  pg_size_pretty(pg_indexes_size('orders')) as indexes_size;
```

第二个代价是写入性能。每次插入、更新、删除数据时，不仅要修改表数据，还要同步修改所有相关索引。索引越多，写入越慢。如果一张表有 10 个索引，每次插入可能要更新 10 个索引结构，性能会很差。

```sql
-- 查看表的索引数量
SELECT
  schemaname,
  tablename,
  indexname
FROM pg_indexes
WHERE tablename = 'orders';
```

第三个代价是维护成本。索引多了之后，难以判断哪些索引有用、哪些没用。有些索引可能创建了但从来没用过，白白占用空间和影响性能。需要定期清理无用索引。

```sql
-- 查看索引使用情况（需要开启 pg_stat_statements）
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

SELECT
  schemaname,
  tablename,
  indexname,
  idx_scan as index_scans,
  idx_tup_read as tuples_read,
  idx_tup_fetch as tuples_fetched
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY idx_scan ASC;
```

如果 idx_scan 是 0 或很小的值，说明这个索引很少使用，可以考虑删除。但要注意，有些索引虽然查询频率低，但可能是关键业务的查询，删除前要确认。

## 什么时候该建索引

建索引的原则很简单：为频繁查询的字段建索引，不为其他字段建索引。但实际判断起来需要一些经验。

第一个判断标准是查询频率。如果某个字段经常出现在 WHERE 条件中，或者经常用于 JOIN，应该考虑建索引。电商系统中，orders.user_id（查询用户订单）、orders.status（按状态筛选订单）、products.category（按分类查询商品）这些字段都应该建索引。

第二个判断标准是数据选择性。选择性是指不同值的数量占总行数的比例。选择性越高，索引效果越好。比如 user_id 的选择性接近 100%（每个用户都不同），建索引很有价值；但 status 的选择性很低（只有几种状态），建索引效果有限。

```sql
-- 计算字段的选择性
SELECT
  COUNT(DISTINCT user_id)::FLOAT / COUNT(*) as user_id_selectivity,
  COUNT(DISTINCT status)::FLOAT / COUNT(*) as status_selectivity
FROM orders;
```

第三个判断标准是数据分布。如果某个字段的值分布很不均匀，索引效果可能不好。比如 99% 的订单都是 completed 状态，查询 WHERE status = 'completed' 时，索引几乎没用（还是要扫描几乎所有数据）。这种情况下，可以考虑部分索引，只索引那 1% 的非 completed 订单。

第四个判断标准是表的大小。小表（几千行）不需要索引，全表扫描也很快。大表（几十万行以上）才需要索引。如果表只有几百行数据，建索引的收益不大，反而浪费空间。

## 实战案例：优化订单查询

让我们用电商系统的订单查询，看看索引如何提升性能。

假设订单表有 1000 万条数据，需要查询某个用户的最近 20 条订单：

```sql
-- 没有索引时的查询（全表扫描）
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 'usr_8x9k2m'
ORDER BY created_at DESC
LIMIT 20;

-- 执行计划：Seq Scan（全表扫描），耗时可能几十秒
```

全表扫描要读取 1000 万条数据，即使有 LIMIT 20，数据库也要先扫描所有数据才能排序取前 20 条。这个查询可能要几十秒。

```sql
-- 为 user_id 创建索引
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 再次查询（使用了 user_id 索引）
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 'usr_8x9k2m'
ORDER BY created_at DESC
LIMIT 20;

-- 执行计划：Index Scan using idx_orders_user_id，耗时可能几秒
```

有了 user_id 索引后，数据库可以快速找到该用户的所有订单，但还需要对这些订单排序。如果用户有 1000 条订单，排序也要花时间。

```sql
-- 创建复合索引（包含排序字段）
CREATE INDEX idx_orders_user_created ON orders(user_id, created_at DESC);

-- 再次查询（使用了复合索引）
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 'usr_8x9k2m'
ORDER BY created_at DESC
LIMIT 20;

-- 执行计划：Index Scan using idx_orders_user_created，可能毫秒级返回
```

复合索引包含了 user_id 和 created_at，且 created_at 是降序排列。数据库可以直接从索引读取该用户的订单，并且已经是按时间降序排列的，直接取前 20 条即可，不需要额外排序。这个查询现在可能几毫秒就返回了。

## 总结：索引的权衡艺术

索引是数据库性能优化的核心工具，但不是万能的。理解索引原理和使用场景，才能做出合理的设计。

索引的核心价值是减少查询扫描的数据量。没有索引时，查询需要全表扫描；有索引后，可以快速定位目标数据。但要记住索引的代价：占用磁盘空间、影响写入性能、需要维护成本。

我的建议是：先建基础索引（主键、唯一键），再根据查询模式添加索引。每添加一个索引前，先问自己：这个查询真的需要优化吗？这个索引会影响写入性能吗？有没有更好的查询方式？
