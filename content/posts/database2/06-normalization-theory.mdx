上一篇文章我们设计了电商系统的表关系，通过外键把用户、订单、商品关联起来。但这样的设计是否合理呢？比如订单表里应该直接存用户的邮箱吗？商品表里应该存分类名称吗？如果把这些字段都放到其他表中，查询时要 JOIN 很多张表，会不会太慢？

这就是规范化与反范式的权衡问题。范式理论告诉我们如何设计"正确"的数据库结构，消除数据冗余，避免更新异常。但过度规范化又会导致查询时要 JOIN 太多表，性能很差。实际工作中，我们需要在数据一致性和查询性能之间找到平衡点。

## 什么是规范化

规范化是数据库设计的一种方法，通过分解表来消除数据冗余。想象一下，如果每个订单都存储用户的完整信息（邮箱、姓名、电话），当用户修改电话时，需要更新所有历史订单。这不仅效率低，还容易出现数据不一致（有些订单更新了，有些没更新）。

规范化解决的就是这类问题。它把数据分解到不同的表中，每个数据只存储一次。用户信息只在 users 表中存储一份，orders 表只存储 user_id，需要时通过 JOIN 关联查询。

规范化的好处显而易见：节省存储空间、避免数据不一致、更新时只需改一个地方。但代价也很明显：查询时需要 JOIN 多张表，性能可能下降。所以在实际设计中，需要根据业务场景权衡。

## 第一范式：字段不可再分

第一范式要求表中的每个字段都是不可再分的原子值。如果一个字段存储了多个值，就违反了第一范式。

电商系统中，一个常见的错误设计是在订单表中用数组存储商品信息：

```sql
-- 错误设计：违反第一范式
CREATE TABLE orders_bad (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  order_number VARCHAR(50),
  products TEXT[],              -- 数组存储多个商品名
  quantities INTEGER[],          -- 数组存储数量
  unit_prices NUMERIC(10, 2)[], -- 数组存储单价
  total NUMERIC(10, 2)
);

-- 插入数据
INSERT INTO orders_bad (id, user_id, order_number, products, quantities, unit_prices, total)
VALUES (
  'ord_a1b2c3d4',
  'usr_8x9k2m',
  'ord_a1b2c3d4',
  ARRAY['无线蓝牙耳机', '智能手表'],
  ARRAY[2, 1],
  ARRAY[99.00, 299.00],
  497.00
);
```

这个设计的问题很多：无法单独查询某个商品出现在哪些订单中，无法建立外键约束保证数据完整性，无法单独记录每个商品的下单时间。查询起来也很痛苦，需要处理数组的每个元素。

符合第一范式的设计应该是把商品信息分离到独立的表中：

```sql
-- 正确设计：符合第一范式
CREATE TABLE orders (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  order_number VARCHAR(50) UNIQUE NOT NULL,
  total NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE order_items (
  id UUID PRIMARY KEY,
  order_id UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_name VARCHAR(255) NOT NULL,
  quantity INTEGER NOT NULL,
  unit_price NUMERIC(10, 2) NOT NULL,
  subtotal NUMERIC(10, 2) NOT NULL
);

-- 插入数据
INSERT INTO orders (id, user_id, order_number, total)
VALUES ('ord_a1b2c3d4', 'usr_8x9k2m', 'ord_a1b2c3d4', 497.00);

INSERT INTO order_items (order_id, product_name, quantity, unit_price, subtotal)
VALUES
  ('ord_a1b2c3d4', '无线蓝牙耳机', 2, 99.00, 198.00),
  ('ord_a1b2c3d4', '智能手表', 1, 299.00, 299.00);
```

现在数据结构清晰了，每条记录都是一个完整的、不可再分的实体。查询某个订单的商品信息很简单，查询某个商品出现在哪些订单中也很方便。

## 第二范式：消除部分依赖

第二范式在第一范式的基础上，要求非主键字段必须完全依赖于主键，不能只依赖于主键的一部分。这个概念有点抽象，看个例子就明白了。

假设我们设计了一张订单项表，主键是 (order_id, product_id) 的组合：

```sql
-- 错误设计：违反第二范式
CREATE TABLE order_items_bad (
  order_id UUID NOT NULL,
  product_id UUID NOT NULL,
  product_name VARCHAR(255),      -- 依赖于 product_id
  product_price NUMERIC(10, 2),   -- 依赖于 product_id
  quantity INTEGER NOT NULL,
  subtotal NUMERIC(10, 2),
  PRIMARY KEY (order_id, product_id)
);
```

这个设计的问题在于：product_name 和 product_price 只依赖于 product_id，不依赖于 order_id。这意味着同一个商品的 name 和 price 会在多个订单项中重复存储。如果商品价格调整，需要更新所有包含该商品的订单项。

符合第二范式的设计应该把商品信息分离出去：

```sql
-- 正确设计：符合第二范式
CREATE TABLE products (
  id UUID PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  price NUMERIC(10, 2) NOT NULL,
  stock INTEGER DEFAULT 0
);

CREATE TABLE order_items (
  id UUID PRIMARY KEY,
  order_id UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE RESTRICT,
  quantity INTEGER NOT NULL,
  unit_price NUMERIC(10, 2) NOT NULL,  -- 快照：下单时的价格
  subtotal NUMERIC(10, 2) NOT NULL
);
```

现在商品信息只在 products 表中存储一份，order_items 表只记录订单和商品的关联关系，以及下单时的数量和价格（这里保留 unit_price 是合理的，因为商品价格可能变化，但订单记录的是历史价格）。

## 第三范式：消除传递依赖

第三范式在第二范式的基础上，要求非主键字段不能依赖于其他非主键字段，换句话说，消除传递依赖。

电商系统中，一个常见的错误设计是在订单表中存储用户的详细信息：

```sql
-- 错误设计：违反第三范式
CREATE TABLE orders_bad (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  user_email VARCHAR(255),        -- 依赖于 user_id
  user_full_name VARCHAR(100),    -- 依赖于 user_id
  user_phone VARCHAR(20),         -- 依赖于 user_id
  order_number VARCHAR(50),
  total NUMERIC(10, 2)
);
```

这里的问题很明显：user_email、user_full_name、user_phone 都依赖于 user_id，而 user_id 不是主键（主键是 id）。这就是传递依赖：user_email → user_id → id。

这意味着同一个用户的所有订单都会重复存储用户的邮箱、姓名、电话。用户修改邮箱时，需要更新该用户的所有订单。

符合第三范式的设计应该把用户信息分离到 users 表中：

```sql
-- 正确设计：符合第三范式
CREATE TABLE users (
  id UUID PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  full_name VARCHAR(100),
  phone VARCHAR(20)
);

CREATE TABLE orders (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  order_number VARCHAR(50) UNIQUE NOT NULL,
  total NUMERIC(10, 2) NOT NULL,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 查询订单和用户信息时，通过 JOIN 关联
SELECT
  o.id,
  o.order_number,
  o.total,
  u.email,
  u.full_name
FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE o.order_number = 'ord_a1b2c3d4';
```

现在用户信息只在 users 表中存储一份，orders 表只存储 user_id 外键。需要用户信息时通过 JOIN 获取，修改用户信息时也只需更新 users 表。

## 什么时候需要反范式

范式理论告诉我们要消除冗余，但实际工作中，有时候故意引入冗余能显著提升性能。这就是反范式设计。

反范式最常见的场景是：查询频繁、更新少。电商系统的订单列表页就是典型例子，需要显示订单号、金额、状态、用户邮箱、商品数量等信息。如果完全按照第三范式设计，每次查询都要 JOIN orders、users、order_items 三张表，性能可能不理想。

一种折中方案是在 orders 表中冗余一些字段：

```sql
-- 适度反范式的设计
CREATE TABLE orders (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  user_email VARCHAR(255),        -- 冗余：避免 JOIN users
  user_full_name VARCHAR(100),    -- 冗余：避免 JOIN users
  order_number VARCHAR(50) UNIQUE NOT NULL,
  status VARCHAR(20) DEFAULT 'pending',
  total NUMERIC(10, 2) NOT NULL,
  item_count INTEGER DEFAULT 0,   -- 冗余：商品数量，避免 JOIN order_items
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 查询订单列表（不需要 JOIN）
SELECT
  order_number,
  total,
  user_email,
  user_full_name,
  item_count,
  status
FROM orders
WHERE user_id = 'usr_8x9k2m'
ORDER BY created_at DESC
LIMIT 20;
```

这个设计的好处是查询订单列表时不需要 JOIN，性能更好。但代价是需要维护冗余字段的一致性：用户修改邮箱时，需要同步更新该用户的所有订单；订单项增减时，需要更新 item_count。

可以用触发器来自动维护冗余字段：

```sql
-- 订单插入时，自动填充用户信息
CREATE OR REPLACE FUNCTION fill_order_user_info()
RETURNS TRIGGER AS $$
BEGIN
  SELECT email, full_name INTO NEW.user_email, NEW.user_full_name
  FROM users WHERE id = NEW.user_id;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_fill_order_user_info
  BEFORE INSERT ON orders
  FOR EACH ROW
  EXECUTE FUNCTION fill_order_user_info();

-- 用户信息更新时，同步更新订单表
CREATE OR REPLACE FUNCTION sync_order_user_info()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE orders
  SET user_email = NEW.email,
      user_full_name = NEW.full_name
  WHERE user_id = NEW.id;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_sync_order_user_info
  AFTER UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION sync_order_user_info();
```

有了触发器，应用层就不需要关心冗余字段的同步问题，数据库会自动维护。但触发器也有代价：每次更新用户信息时，都要更新所有相关订单，可能影响性能。

## 物化视图：折中的选择

如果既想要查询性能，又不想要冗余字段带来的维护成本，物化视图是一个折中选择。

物化视图和普通视图的区别在于：普通视图只是保存了查询逻辑，每次查询都实时执行；物化视图则保存了查询结果，就像一张真实的表。

```sql
-- 创建物化视图：用户订单统计
CREATE MATERIALIZED VIEW user_order_stats AS
SELECT
  u.id as user_id,
  u.email,
  u.full_name,
  COUNT(o.id) as total_orders,
  COALESCE(SUM(o.total), 0) as total_spent,
  MAX(o.created_at) as last_order_at
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.email, u.full_name;

-- 查询物化视图（非常快，因为是预计算的结果）
SELECT * FROM user_order_stats
WHERE total_spent > 1000
ORDER BY total_spent DESC;

-- 刷新物化视图（更新数据）
REFRESH MATERIALIZED VIEW user_order_stats;

-- 并发刷新（不锁表，但需要唯一索引）
REFRESH MATERIALIZED VIEW CONCURRENTLY user_order_stats;
```

物化视图适合查询频繁、更新不频繁的场景。比如用户订单统计，可以每小时刷新一次，查询时直接从物化视图读取，性能很好。

但物化视图也有缺点：数据不是实时的，需要定期刷新；占用存储空间；刷新时可能有性能开销。所以要根据业务场景选择使用。

## 如何权衡规范化与反范式

规范化与反范式的权衡没有标准答案，需要根据业务场景判断。我的一般原则是：

默认情况下，遵循第三范式。保证数据一致性，避免冗余，这是数据库设计的基础。大部分系统（用户量 < 100 万）按第三范式设计性能都够用。

遇到性能瓶颈时，考虑适度反范式。通过监控找出慢查询，分析是否因为 JOIN 太多导致。如果是，且有明确的查询模式，可以冗余一些字段。冗余字段用触发器或应用层逻辑维护。

对于复杂的统计查询，使用物化视图。这比冗余字段更可控，不需要每次更新都维护，只需要定期刷新。

但要注意，反范式是性能优化的手段，不是偷懒的借口。有些系统为了省事，把所有字段都堆在一张表里，号称"反范式"，实际上就是懒。这种设计初期开发快，后期维护会很痛苦：数据不一致、难以扩展、代码复杂。

## 实战案例：订单列表查询

让我们用电商订单列表页这个真实场景，看看不同设计的性能差异。

完全规范化的设计（3NF）：

```sql
-- 查询用户的订单列表（包含用户信息和商品数量）
SELECT
  o.order_number,
  o.total,
  o.status,
  o.created_at,
  u.email,
  u.full_name,
  COUNT(oi.id) as item_count
FROM orders o
INNER JOIN users u ON o.user_id = u.id
LEFT JOIN order_items oi ON o.id = oi.order_id
WHERE o.user_id = 'usr_8x9k2m'
GROUP BY o.id, u.id
ORDER BY o.created_at DESC
LIMIT 20;
```

这个查询 JOIN 了三张表，还用了 GROUP BY，性能可能不理想。特别是在数据量大的情况下，每次查询都要扫描大量数据。

适度反范式的设计：

```sql
-- 订单表冗余用户信息和商品数量
CREATE TABLE orders (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  user_email VARCHAR(255),
  user_full_name VARCHAR(100),
  order_number VARCHAR(50) UNIQUE NOT NULL,
  status VARCHAR(20) DEFAULT 'pending',
  total NUMERIC(10, 2) NOT NULL,
  item_count INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 查询订单列表（不需要 JOIN）
SELECT
  order_number,
  total,
  status,
  user_email,
  user_full_name,
  item_count,
  created_at
FROM orders
WHERE user_id = 'usr_8x9k2m'
ORDER BY created_at DESC
LIMIT 20;
```

这个查询简单很多，不需要 JOIN，性能会好很多。代价是需要维护冗余字段的一致性，但用触发器可以自动处理。

如果这个查询是系统中最频繁的操作（比如订单列表页是访问量最大的页面），那么反范式是值得的。如果这个查询很少用，那就没必要为了它牺牲数据一致性。

## 数据库设计的艺术

数据库设计既是一门科学，也是一门艺术。范式理论提供了科学的原则，告诉我们什么是"正确"的设计。但实际工作中，没有绝对正确的设计，只有最适合的设计。

理解范式理论很重要，它帮助我们识别设计问题，避免低级错误。但更重要的是理解背后的权衡：数据一致性 vs 查询性能，存储空间 vs 计算开销，开发效率 vs 维护成本。

我的建议是：先保证正确性（规范化），再优化性能（反范式）。不要一开始就过度优化，也不要完全不考虑性能。随着业务增长，逐步调整设计，找到最合适的平衡点。

这篇文章我们学习了范式理论，包括第一、二、三范式的要求，以及什么时候需要反范式设计。但表设计只是基础，要让查询跑得快，还需要索引。下一篇我们会学习索引原理，理解为什么索引能让查询速度提升 100 倍，以及什么时候该加索引、什么情况下索引会失效。
