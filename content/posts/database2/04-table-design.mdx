上一篇文章我们搭建了 PostgreSQL 环境，学会了基本的数据库操作。但有个问题还没解决：怎么设计表结构？VARCHAR(255) 还是 TEXT？用 UUID 还是自增 ID？金额用什么类型存储？这些问题看似简单，但选择不当会埋下巨大的隐患。

数据库表设计是整个系统架构的基石。一个糟糕的表设计会导致性能问题, 数据不一致, 难以维护等各种隐患。让我们通过电商系统实战，系统讲解表设计原则, 数据类型选择和约束使用。

首先说说表设计原则。命名规范很重要，推荐使用蛇形命名法（snake*case），比如 user_profiles, order_items, created_at。避免使用驼峰命名，PostgreSQL 不推荐 userProfiles 这种写法。表名使用复数，这是国际惯例，比如 users, products, orders。列名使用单数，比如 user_name, product_name, order_status。布尔类型使用 is* 前缀，比如 is_active, is_verified, is_published。时间戳使用 \_at 后缀，比如 created_at, updated_at, deleted_at。

主键设计是个有争议的话题。自增整型（SERIAL/BIGSERIAL）性能好, 占用空间小，但可被枚举, 分布式场景不友好。UUID 全球唯一, 适合分布式, 安全性高，但占用空间大, 索引性能略低。组合主键适用于特定场景，比如订单项表用 order_id 和 product_id 作为组合主键。

选择建议是：单体应用用 SERIAL，分布式系统用 UUID，联合主键用组合主键。默认值与时间戳也很重要：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  is_verified BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = CURRENT_TIMESTAMP;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();
```

PostgreSQL 提供了丰富的数据类型，选择合适的类型对性能至关重要。数值类型包括整数类型和精确数值。SMALLINT 是 2 字节，范围从 -32768 到 +32767。INTEGER 是 4 字节，范围从 -2147483648 到 +2147483647。BIGINT 是 8 字节，用于超大整数。

根据业务选择很重要：

```sql
CREATE TABLE products (
  id SERIAL PRIMARY KEY,
  quantity SMALLINT,
  price INTEGER,
  views BIGINT
);
```

金额必须用精确数值 NUMERIC(10, 2)，10 位数字 2 位小数，最多 99999999.99。千万不要用 FLOAT/DOUBLE 存储金额，因为存在浮点数精度问题，可能出现 0.1 + 0.2 = 0.30000000000000004 这种情况。

```sql
CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  total NUMERIC(10, 2) NOT NULL
);

CREATE TABLE orders_bad (
  total DOUBLE PRECISION
);
```

序列类型用于自增，SERIAL 等同于 INTEGER 加自增，BIGSERIAL 等同于 BIGINT 加自增。

数值类型选择建议：金额货币用 NUMERIC/DECIMAL，数量计数用 INTEGER，ID 外键用 INTEGER/BIGINT，百分比用 NUMERIC(5, 2)。

字符类型包括固定长度 CHAR 和可变长度 VARCHAR, TEXT。CHAR(10) 固定 10 字符不足补空格，不推荐使用除非确定长度固定。VARCHAR(255) 最多 255 字符，TEXT 不限制长度但实际受 1GB 限制。性能方面 CHAR(100) 浪费存储查询时会补空格，VARCHAR(100) 灵活性能好，TEXT 适合长文本。

```sql
CREATE TABLE users (
  email VARCHAR(255) NOT NULL,
  phone VARCHAR(20),
  bio TEXT,
  avatar_url TEXT
);
```

VARCHAR 长度设置建议：邮箱用 VARCHAR(255)，URL 用 TEXT 或 VARCHAR(2048)，用户名用 VARCHAR(50)，标题用 VARCHAR(255)，内容用 TEXT。

日期时间类型包括 DATE（YYYY-MM-DD）, TIME（HH:MM:SS）, TIMESTAMP（时区无关）, TIMESTAMPTZ（带时区推荐）, INTERVAL（时间差）。

```sql
CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
  shipped_at TIMESTAMPTZ,
  estimated_delivery INTERVAL DEFAULT INTERVAL '3 days'
);

SELECT
  order_id,
  created_at,
  created_at + INTERVAL '7 days' AS delivery_due_date,
  AGE(CURRENT_TIMESTAMP, created_at) AS days_since_order
FROM orders;
```

时区处理建议：存储使用 TIMESTAMPTZ 带时区，显示根据用户时区转换，避免使用 TIMESTAMP 不带时区。

布尔类型很简单，BOOLEAN 表示 TRUE/FALSE/NULL：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  is_active BOOLEAN DEFAULT TRUE,
  is_verified BOOLEAN DEFAULT FALSE,
  is_deleted BOOLEAN DEFAULT FALSE
);

SELECT * FROM users WHERE is_verified = TRUE;
SELECT * FROM users WHERE is_active;
```

JSON 类型是 PostgreSQL 的特色。JSON 存储原始 JSON 查询时解析，JSONB 存储二进制格式查询性能更好推荐使用。

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  preferences JSONB DEFAULT '{}',
  profile JSONB
);

INSERT INTO users (email, preferences, profile)
VALUES (
  'user@example.com',
  '{"language": "zh-CN", "theme": "dark", "notifications": true}'::jsonb,
  '{"age": 25, "city": "北京", "hobbies": ["编程", "阅读"]}'::jsonb
);

SELECT
  email,
  preferences->>'language' AS language,
  preferences->>'theme' AS theme,
  profile->>'age' AS age,
  profile->>'city' AS city
FROM users;

SELECT * FROM users WHERE preferences->>'theme' = 'dark';

UPDATE users
SET preferences = preferences || '{"language": "en-US"}'
WHERE id = 1;

CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  order_number VARCHAR(50) UNIQUE NOT NULL,
  total NUMERIC(10, 2) NOT NULL,
  metadata JSONB DEFAULT '{}'
);

SELECT * FROM orders WHERE metadata->>'coupon_code' = 'SPRING2024';
```

JSON 和 JSONB 的对比：JSON 存储原始文本插入速度快查询时解析慢，JSONB 存储二进制插入时略慢查询快支持索引。JSON 保留空格和顺序，JSONB 不保留。需要保留原始格式用 JSON，大部分场景用 JSONB。

数组类型是 PostgreSQL 原生支持的：

```sql
CREATE TABLE posts (
  id SERIAL PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  tags INTEGER[]
);

CREATE TABLE products (
  id SERIAL PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  keywords TEXT[]
);

INSERT INTO products (name, keywords)
VALUES ('iPhone 15', ARRAY['手机', 'Apple', '5G']);

SELECT name, keywords FROM products WHERE 'Apple' = ANY(keywords);
SELECT * FROM products WHERE keywords @> ARRAY['Apple'];

UPDATE products
SET keywords = array_append(keywords, '新品')
WHERE id = 1;
```

特殊类型包括 UUID, ENUM, CIDR, BYTEA：

```sql

CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(255) NOT NULL
);

CREATE TYPE order_status AS ENUM (
  'pending',
  'paid',
  'shipped',
  'delivered',
  'cancelled'
);

CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  status order_status DEFAULT 'pending'
);

CREATE TABLE access_logs (
  id SERIAL PRIMARY KEY,
  ip_address CIDR,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE files (
  id SERIAL PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  data BYTEA

```

约束保证数据的完整性和一致性。NOT NULL 非空约束：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  full_name VARCHAR(100),
  phone VARCHAR(20) NOT NULL DEFAULT ''
);

ALTER TABLE users ALTER COLUMN phone SET NOT NULL;
ALTER TABLE users ALTER COLUMN phone DROP NOT NULL;
```

UNIQUE 唯一约束：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL
);

CREATE TABLE user_follows (
  follower_id INTEGER NOT NULL,
  followee_id INTEGER NOT NULL,
  UNIQUE (follower_id, followee_id)
);

ALTER TABLE users ADD CONSTRAINT unique_phone UNIQUE (phone);
```

CHECK 检查约束：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  age INTEGER CHECK (age >= 18),
  country VARCHAR(50) CHECK (country IN ('中国', '美国', '日本'))
);

CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  subtotal NUMERIC(10, 2) NOT NULL,
  discount NUMERIC(10, 2) DEFAULT 0 CHECK (discount >= 0 AND discount <= subtotal),
  total NUMERIC(10, 2) NOT NULL CHECK (total >= 0),
  CHECK (subtotal + tax - discount = total)
);

CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  CONSTRAINT email_format CHECK (email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$')
);
```

DEFAULT 默认值：

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  is_verified BOOLEAN DEFAULT FALSE,
  role VARCHAR(20) DEFAULT 'user',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  token VARCHAR(255) DEFAULT md5(random()::text)
);
```

EXCLUDE 排除约束是高级约束，用于排除冲突的数据：

```sql
CREATE TABLE reservations (
  id SERIAL PRIMARY KEY,
  room_id INTEGER NOT NULL,
  during TSRANGE NOT NULL,
  EXCLUDE USING GIST (room_id WITH =, during WITH &&)
);

INSERT INTO reservations (room_id, during)
VALUES (1, '[2024-01-01 10:00, 2024-01-01 12:00]');

INSERT INTO reservations (room_id, during)
VALUES (1, '[2024-01-01 11:00, 2024-01-01 13:00]');
```

订单模块：

```sql
CREATE TYPE order_status AS ENUM (
  'pending',
  'paid',
  'processing',
  'shipped',
  'delivered',
  'cancelled',
  'refunded'
);

CREATE TABLE orders (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  order_number VARCHAR(50) UNIQUE NOT NULL,
  status order_status DEFAULT 'pending',
  subtotal NUMERIC(10, 2) NOT NULL CHECK (subtotal >= 0),
  tax NUMERIC(10, 2) DEFAULT 0 CHECK (tax >= 0),
  shipping_fee NUMERIC(10, 2) DEFAULT 0 CHECK (shipping_fee >= 0),
  discount NUMERIC(10, 2) DEFAULT 0 CHECK (discount >= 0),
  total NUMERIC(10, 2) NOT NULL CHECK (total >= 0),
  recipient_name VARCHAR(100) NOT NULL,
  recipient_phone VARCHAR(20) NOT NULL,
  address TEXT NOT NULL,
  customer_notes TEXT,
  admin_notes TEXT,
  paid_at TIMESTAMPTZ,
  shipped_at TIMESTAMPTZ,
  delivered_at TIMESTAMPTZ,
  metadata JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT amount_calculation CHECK (
    subtotal + tax + shipping_fee - discount = total
  )
);

CREATE TABLE order_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE RESTRICT,
  sku_id UUID REFERENCES product_skus(id) ON DELETE SET NULL,
  product_name VARCHAR(255) NOT NULL,
  product_attributes JSONB,
  quantity INTEGER NOT NULL CHECK (quantity > 0),
  unit_price NUMERIC(10, 2) NOT NULL CHECK (unit_price >= 0),
  subtotal NUMERIC(10, 2) NOT NULL CHECK (subtotal >= 0),
  created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT subtotal_calculation CHECK (subtotal = quantity * unit_price)
);

CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_created ON orders(created_at DESC);
CREATE INDEX idx_order_items_order ON order_items(order_id);

```

设计原则方面，范式化与反范式化需要权衡。优先范式化（3NF），性能瓶颈时适度反范式，冗余字段保持同步用触发器。字段长度方面，VARCHAR 不要设置过长影响索引，URL 长文本使用 TEXT，金额使用 NUMERIC 不要用 FLOAT。默认值方面，布尔字段提供默认值，时间戳自动设置，数值字段默认 0。

索引设计很重要：

```sql
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_orders_user_status ON orders(user_id, status);
CREATE INDEX idx_users_active ON users(email) WHERE is_active = TRUE;
```

外键创建索引，频繁查询的字段创建索引，组合索引注意顺序，部分索引只索引活跃数据，避免过多索引影响写入性能。

数据一致性方面，使用外键约束, 使用 CHECK 约束, 使用触发器维护冗余字段, 使用事务保证数据完整性。

UUID 和自增 ID 的选择，UUID 优点是分布式友好, 安全性高, 可在客户端生成，自增 ID 优点是性能更好, 占用空间小, 有序利于范围查询。单体应用用自增 ID，分布式系统用 UUID，内部表小表用自增 ID。

金额必须使用 NUMERIC/DECIMAL，因为浮点数有精度问题。0.1 + 0.2 用 DOUBLE PRECISION 结果是 0.30000000000000004，用 NUMERIC 结果是 0.3。

VARCHAR 长度设置建议：邮箱用 VARCHAR(255) RFC 5321 标准，URL 用 TEXT 可能很长，用户名用 VARCHAR(50)，标题用 VARCHAR(255)，内容用 TEXT。PostgreSQL 的 VARCHAR 性能差异不大，主要是约束作用。

推荐使用 TIMESTAMPTZ，因为自动处理时区转换, 夏令时自动调整, 全球用户友好。

JSONB 和关联表的选择，使用 JSONB 的场景：数据结构灵活多变, 不需要查询排序关联, 配置元数据。使用关联表的场景：需要复杂查询, 需要外键约束, 需要保证数据完整性。
