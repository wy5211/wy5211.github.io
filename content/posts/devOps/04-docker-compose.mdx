上篇把 CloudPulse 的 Next.js 应用容器化了，镜像也从 1.2GB 优化到了 150MB。但 CloudPulse 不止一个服务——还有 PostgreSQL 数据库、Redis 缓存。

你试着手动启动这三个容器：

```bash
# 先启动 PostgreSQL
docker run -d --name db \
  -v pgdata:/var/lib/postgresql/data \
  -e POSTGRES_PASSWORD=secret \
  postgres:16

# 再启动 Redis
docker run -d --name redis redis:7

# 最后启动 Next.js（需要知道 db 和 redis 的 IP 地址）
docker run -d --name app \
  --link db:db \
  --link redis:redis \
  -p 3000:3000 \
  cloudpulse:latest
```

每次都要记住启动顺序、配置网络、挂载 Volume。一旦服务变多，这个脚本来得比应用本身还复杂。而且如果你换个服务器部署，还得重新敲一遍这些命令。

Docker Compose 解决的就是这个问题：**用 YAML 文件定义多服务应用，一条命令启动全部**。

## Docker Compose 是什么

Docker Compose 是 Docker 官方的编排工具，用于定义和运行多容器 Docker 应用。核心思想是**声明式配置**：你声明"我需要什么"，Composed 负责怎么实现。

它的价值：

- 一键启动：`docker compose up -d` 启动所有服务
- 自动处理依赖：先启动数据库，再启动应用
- 网络配置：容器间可以通过服务名互相访问
- 数据持久化：Volume 自动创建和管理
- 环境变量集中管理：不用每次 docker run 都写一堆 `-e`

## 编写 docker-compose.yml

在项目根目录创建 `docker-compose.yml`：

```yaml
# docker-compose.yml
services:
  # PostgreSQL 数据库
  db:
    image: postgres:16-alpine
    container_name: cloudpulse-db
    environment:
      POSTGRES_USER: cloudpulse
      POSTGRES_PASSWORD: ${DB_PASSWORD:-secret}
      POSTGRES_DB: cloudpulse
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U cloudpulse"]
      interval: 5s
      timeout: 5s
      retries: 5

  # Redis 缓存
  redis:
    image: redis:7-alpine
    container_name: cloudpulse-redis
    command: redis-server --appendonly yes
    volumes:
      - redisdata:/data
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5

  # Next.js 应用
  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: cloudpulse-app
    environment:
      DATABASE_URL: postgresql://cloudpulse:${DB_PASSWORD:-secret}@db:5432/cloudpulse
      REDIS_URL: redis://redis:6379
      NODE_ENV: production
    ports:
      - "3000:3000"
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy

volumes:
  pgdata:
  redisdata:
```

关键配置说明：

**services**：定义每个服务。这里我们定义了三个服务：db、redis、app。

**image vs build**：`image` 直接使用现有镜像（postgres、redis），`build` 从 Dockerfile 构建镜像（app）。

**environment**：环境变量。`${DB_PASSWORD:-secret}` 表示使用环境变量 `DB_PASSWORD`，如果没设置就用默认值 `secret`。这样可以避免把密码写死在配置文件里。

**volumes**：数据卷。`pgdata:/var/lib/postgresql/data` 把命名卷 `pgdata` 挂载到容器内的 PostgreSQL 数据目录。底部的 `volumes:` 声明会自动创建这些卷。

**ports**：端口映射。`"5432:5432"` 把宿主机的 5432 端口映射到容器的 5432 端口。

**depends_on + healthcheck**：`depends_on` 声明服务依赖，`condition: service_healthy` 表示等待依赖服务的健康检查通过后再启动。这解决了"应用启动时数据库还没就绪"的问题。

**服务名作为主机名**：注意 app 的 `DATABASE_URL` 用的是 `db:5432` 而不是 `localhost:5432`。Docker Compose 会自动创建一个网络，服务名就是主机名。

## 启动和管理

```bash
# 启动所有服务（后台运行）
docker compose up -d

# 查看运行状态
docker compose ps

# 查看日志（所有服务）
docker compose logs -f

# 查看特定服务的日志
docker compose logs -f app

# 停止所有服务
docker compose stop

# 停止并删除容器、网络（保留 Volume）
docker compose down

# 停止并删除一切（包括 Volume，慎用）
docker compose down -v

# 重启某个服务
docker compose restart app

# 进入容器调试
docker compose exec app sh
docker compose exec db psql -U cloudpulse
```

第一次运行 `docker compose up -d` 时，你会看到类似这样的输出：

```
[+] Running 4/4
 ✔ Network cloudpulse_default      Created
 ✔ Volume "cloudpulse_pgdata"      Created
 ✔ Volume "cloudpulse_redisdata"   Created
 ✔ Container cloudpulse-db         Started
 ✔ Container cloudpulse-redis      Started
 ✔ Container cloudpulse-app        Started
```

Compose 自动创建了网络、Volume，并按依赖顺序启动了三个容器。

## 开发环境配置

生产环境用上面的配置就够了。但开发时你可能需要：

- 挂载源码，代码改动自动热重载
- 启用调试工具
- 连接本地的数据库实例

Docker Compose 支持**覆盖配置**，用多个文件区分环境：

```yaml
# docker-compose.dev.yml（开发环境）
services:
  app:
    build:
      context: .
      target: development # 开发阶段的构建阶段
    volumes:
      - .:/app # 挂载源码
      - /app/node_modules # 避免覆盖 node_modules
      - /app/.next # 避免覆盖构建产物
    environment:
      NODE_ENV: development
    command: npm run dev # 开发模式启动
```

启动开发环境：

```bash
docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d
```

`-f` 指定配置文件，多个文件的配置会合并（后面的覆盖前面的）。这样你可以在 `docker-compose.yml` 放通用配置，`docker-compose.dev.yml` 放开发专属配置。

## 常见陷阱

### 健康检查的必要性

如果你不用 healthcheck，depends_on 只能保证启动顺序，不能保证依赖服务就绪。应用可能在数据库启动前就尝试连接，导致启动失败。

这就是为什么上面的配置给 PostgreSQL 和 Redis 都加了 healthcheck。`pg_isready -U cloudpulse` 会持续检查数据库是否可接受连接，`redis-cli ping` 检查 Redis 是否响应。

### 网络隔离

Docker Compose 创建的网络是隔离的。容器之间可以通过服务名访问，但从宿主机访问需要用 `localhost:映射端口`。

比如在 app 容器里访问数据库用 `db:5432`，但从你本机的 psql 客户端连接需要用 `localhost:5432`。

### Volume 数据不会自动删除

`docker compose down` 只删除容器和网络，不会删除 Volume。这意味着你下次启动时，之前的数据还在。

这是刻意设计的行为——防止误删数据。如果真的需要清空数据，用 `docker compose down -v`。

## 生产环境建议

上面的配置已经能用于生产，但还有几个改进点：

1. **环境变量用 .env 文件管理**：

```bash
# .env（不要提交到 git）
DB_PASSWORD=your-strong-password
REDIS_PASSWORD=your-redis-password
```

然后在 docker-compose.yml 里引用：

```yaml
environment:
  POSTGRES_PASSWORD: ${DB_PASSWORD}
```

2. **限制容器资源**：

```yaml
services:
  app:
    deploy:
      resources:
        limits:
          cpus: "1"
          memory: 1G
        reservations:
          cpus: "0.5"
          memory: 512M
```

3. **自动重启策略**：

```yaml
services:
  app:
    restart: unless-stopped # 除非手动停止，否则自动重启
```

到这里，CloudPulse 的三个服务已经能在本地或开发服务器上一键启动了。但如果你要把服务暴露到公网，直接用 `http://your-ip:3000` 访问既不专业也不安全。下一篇，我们用 Nginx 做反向代理，加上 HTTPS 让它像一个正式产品。
