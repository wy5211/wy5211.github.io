上篇用 Docker Compose 一键启动了 CloudPulse 的三个服务，访问方式是 `http://your-server-ip:3000`。

这对于开发环境没问题，但如果是给用户用的正式产品，有几个问题：

1. **端口暴露不专业**——用户需要记住端口号，而且浏览器默认访问 80/443 端口
2. **没有 HTTPS**——现代浏览器会标记为"不安全"，Chrome 甚至会显示警告
3. **没有域名**——IP 地址会变，而且用户记不住
4. **缺少缓存和压缩**——静态资源每次都从应用服务器取，浪费带宽

今天来解决这些问题，让 CloudPulse 看起来像一个正式产品。

## 为什么需要反向代理

反向代理是放在应用服务器前面的"中间层"。用户请求先到 Nginx，Nginx 再转发给后端应用。

它的价值：

**端口复用**：用户只需访问 `cloudpulse.com`（默认 80/443 端口），Nginx 转发到容器的 3000 端口。

**SSL 终止**：HTTPS 的加密解密在 Nginx 层完成，后端应用只用处理 HTTP，简化了应用配置。

**静态文件缓存**：Next.js 的静态文件（图片、CSS、JS）可以被 Nginx 缓存，减轻应用压力。

**负载均衡**：如果你后面有多个应用实例，Nginx 可以做负载均衡（虽然单机用不上）。

**Gzip 压缩**：Nginx 可以在响应前压缩文本内容，减少传输量。

## 安装和配置 Nginx

在服务器上安装 Nginx：

```bash
sudo apt install -y nginx
```

安装完成后，Nginx 会自动启动。你可以先访问 `http://your-server-ip` 看到默认页面。

然后创建 CloudPulse 的配置文件：

```bash
sudo vim /etc/nginx/sites-available/cloudpulse
```

配置内容：

```nginx
# HTTP 配置（用于重定向到 HTTPS）
server {
    listen 80;
    server_name cloudpulse.com www.cloudpulse.com;

    # Let's Encrypt 验证路径
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # 其他请求重定向到 HTTPS
    location / {
        return 301 https://$server_name$request_uri;
    }
}

# HTTPS 配置
server {
    listen 443 ssl http2;
    server_name cloudpulse.com www.cloudpulse.com;

    # SSL 证书（稍后配置）
    ssl_certificate /etc/letsencrypt/live/cloudpulse.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/cloudpulse.com/privkey.pem;

    # SSL 配置（使用 Mozilla 的推荐配置）
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;

    # 静态文件缓存
    location /_next/static {
        proxy_pass http://localhost:3000;
        proxy_cache_valid 200 7d;
        add_header Cache-Control "public, immutable";
    }

    location /static {
        proxy_pass http://localhost:3000;
        proxy_cache_valid 200 7d;
        add_header Cache-Control "public, immutable";
    }

    # 图片和字体缓存
    location ~* \.(jpg|jpeg|png|gif|ico|svg|woff|woff2|ttf|eot)$ {
        proxy_pass http://localhost:3000;
        proxy_cache_valid 200 30d;
        add_header Cache-Control "public, immutable";
    }

    # 其他请求转发到 Next.js
    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;

        # WebSocket 支持（Next.js HMR 需要）
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';

        # 传递真实客户端信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时设置
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    # Gzip 压缩
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css text/xml text/javascript application/x-javascript application/xml+rss application/json application/javascript;
}
```

配置说明：

**两段 server 块**：第一段处理 HTTP，第二段处理 HTTPS。HTTP 请求会被重定向到 HTTPS（除了 Let's Encrypt 的验证请求）。

**SSL 配置**：使用 TLS 1.2 和 1.3，禁用过时的协议。cipher 列表来自 Mozilla SSL Configuration Generator。

**静态文件缓存**：`/_next/static` 和 `/static` 是 Next.js 的静态资源路径，缓存 7 天。图片和字体缓存 30 天。`Cache-Control "public, immutable"` 告诉浏览器这些文件不会变，可以永久缓存。

**proxy_set_header**：把真实客户端 IP、协议等信息传递给后端应用。这对日志记录和安全检查很重要。

**WebSocket 支持**：`Upgrade` 和 `Connection` 头用于支持 WebSocket 连接。Next.js 开发模式的热重载需要 WebSocket。

创建符号链接启用配置：

```bash
sudo ln -s /etc/nginx/sites-available/cloudpulse /etc/nginx/sites-enabled/
sudo nginx -t  # 测试配置是否有语法错误
sudo systemctl reload nginx
```

## 配置 HTTPS 证书

现在 Nginx 配置好了，但证书文件还不存在。我们用 Let's Encrypt 的 Certbot 工具免费签发证书。

先安装 Certbot：

```bash
sudo apt install -y certbot python3-certbot-nginx
```

创建验证目录：

```bash
sudo mkdir -p /var/www/certbot
```

然后签发证书：

```bash
sudo certbot certonly --webroot \
  --webroot-path=/var/www/certbot \
  --email your@email.com \
  --agree-tos \
  --no-eff-email \
  -d cloudpulse.com \
  -d www.cloudpulse.com
```

参数说明：

- `certonly`：只签发证书，不修改 Nginx 配置（我们已经在配置文件里写好了）
- `--webroot`：使用 HTTP 验证方式（Certbot 会在 `/var/www/certbot` 创建验证文件）
- `--agree-tos`：同意服务条款
- `-d`：指定域名（可以有多个）

执行成功后，证书文件会被放到 `/etc/letsencrypt/live/cloudpulse.com/`。现在 Nginx 的 HTTPS 配置就能正常工作了。

访问 `https://cloudpulse.com`，你应该能看到浏览器地址栏的锁图标。

## 自动续期

Let's Encrypt 的证书有效期是 90 天，需要定期续期。Certbot 可以自动处理这个：

```bash
sudo certbot renew --dry-run  # 测试续期是否正常
```

Certbot 会在安装时自动创建一个 systemd timer，每天自动检查并续期。你可以用以下命令确认：

```bash
sudo systemctl status certbot.timer
```

输出应该显示 `active (waiting)`，表示定时任务在运行。

## 域名配置

上面的配置假设你已经有了域名 `cloudpulse.com`，并且把它的 A 记录指向你的服务器 IP。

在域名注册商（如 Cloudflare、Namecheap）的 DNS 设置里：

| 类型 | 名称 | 值             | TTL  |
| ---- | ---- | -------------- | ---- |
| A    | @    | your-server-ip | 3600 |
| A    | www  | your-server-ip | 3600 |

DNS 生效可能需要几分钟到几小时。你可以用 `dig cloudpulse.com` 确认是否生效。

## 常见问题排查

### 502 Bad Gateway

如果访问域名时看到 502，说明 Nginx 无法连接到后端应用。检查：

1. 应用是否在运行：`docker compose ps`
2. 应用是否监听正确的端口：`docker compose logs app`
3. Nginx 配置的 `proxy_pass` 是否正确（`http://localhost:3000`）

### 证书签发失败

如果 Certbot 报错 "Failed to connect to localhost"，检查：

1. 域名 DNS 是否正确解析
2. 防火墙是否开放 80 端口：`sudo ufw status`
3. Nginx 是否在运行：`sudo systemctl status nginx`

### 静态文件 404

如果 `/_next/static` 路径返回 404，检查：

1. 应用是否正常构建：`docker compose logs app | grep "Ready on"`
2. Nginx 的 `proxy_pass` 是否指向正确的端口

## 性能优化建议

配置已经能用了，但还有几个优化点：

1. **HTTP/2**：配置里的 `http2` 已启用，支持多路复用，提升加载速度
2. **访问日志格式**：可以自定义日志格式，记录响应时间等信息：

```nginx
log_format cloudpulse '$remote_addr - $remote_user [$time_local] '
                       '"$request" $status $body_bytes_sent '
                       '"$http_referer" "$http_user_agent" '
                       'rt=$request_time uct="$upstream_connect_time" '
                       'uht="$upstream_header_time" urt="$upstream_response_time"';

access_log /var/log/nginx/cloudpulse.access.log cloudpulse;
```

3. **限流**：防止 API 被滥用：

```nginx
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

location /api/ {
    limit_req zone=api burst=20 nodelay;
    proxy_pass http://localhost:3000;
}
```
