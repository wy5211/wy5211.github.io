## 一个真实的问题

用户 zhang_wei 在编辑器里敲完最后一个字，按下"发布"按钮。前端转了 6 秒圈，他才看到"发布成功"。

为什么会这么慢？打开后端日志看了一下：发布文章的接口里，代码串行执行了三件事——调用邮件服务给 28 个订阅者发送通知邮件（2.1 秒）、调用 AI 服务生成文章摘要（2.3 秒）、更新 Elasticsearch 全文搜索索引（1.8 秒）。三个操作加起来 6.2 秒。

用户根本不在乎邮件什么时候发出去，他只想知道文章已经存到数据库了。摘要可以稍后生成，搜索索引也不需要实时更新。但代码把它们串在一起，用户被迫等所有人干完活才能走。

如果并发量上来了就更糟。10 个用户同时发布文章，每个请求占一个线程 6 秒，很快连接池就满了。更致命的是，邮件服务和 AI 服务都有自己的并发限制，高并发下直接返回 429 Too Many Requests，导致发布接口也跟着报错。

上一篇我们用 Multer 处理文件上传，解决了"用户不该等文件处理完"的问题——图片上传后立即返回，压缩和缩略图生成在后台做。这篇解决的是同一类问题：**任何不需要用户等待结果的耗时操作，都应该从请求链路里剥离出去**。解决方案是消息队列。

## 同步 vs 异步：奶茶店的隐喻

想象你走进一家奶茶店，前面的人点了 10 杯不同口味的。没有排队系统的店，店员一杯一杯做，你站在柜台前干等 20 分钟。有了排队系统，你点完单拿到号码牌就能走，做好了发短信通知你。

代码层面，区别长这样：

```typescript
// 同步：用户等 6 秒
@Post('publish')
async publish(@Body() dto: PublishPostDto) {
  const post = await this.postsService.create(dto);
  await this.emailService.sendNotifications(post.id);   // 2.1s
  await this.aiService.generateSummary(post.id);         // 2.3s
  await this.searchService.updateIndex(post.id);         // 1.8s
  return post; // 6.2 秒后才到这里
}
```

```typescript
// 异步：用户瞬间看到结果
@Post('publish')
async publish(@Body() dto: PublishPostDto) {
  const post = await this.postsService.create(dto);
  // 只是把一条消息写入 Redis，大约 1 毫秒
  await this.postQueue.add('post-published', { postId: post.id });
  return post; // 50ms 内返回
}
```

`queue.add()` 不执行任何耗时操作，它只是往 Redis 里写了一条记录说"有活要干"。另一个进程（Worker）会监听这个队列，取出任务执行。你的 HTTP 请求和任务执行完全解耦了。

这个解耦带来了几个好处：

- **接口变快**：从 6.2 秒降到 50 毫秒，用户体验质的飞跃
- **服务隔离**：邮件服务挂了，发布接口不受影响。用户依然能发布文章，邮件稍后自动重试
- **削峰填谷**：高峰期涌入 100 个发布请求，任务先进队列排队，Worker 按自己的节奏处理，不会压垮下游服务
- **可观测性**：每个任务有独立的状态（等待、执行中、完成、失败），出问题时可以精确定位

## 为什么是 BullMQ 而不是 Bull

Bull 和 BullMQ 都是基于 Redis 的任务队列，但它们不是同一个东西，不能混用。

Bull 是 2015 年的老项目，基于 Node.js 的回调风格，内部使用 Redis 的 List 和 Sorted Set 数据结构。它功能完善、社区成熟，在 Express 生态里用得很多，但已经进入维护模式，不再开发新功能。

BullMQ 是 Taskforce.sh（Bull 的商业公司）在 2019 年推出的下一代重写。它全部基于 Promise 和 async/await，API 设计更现代。底层使用 Redis Streams（Redis 5.0+ 的新数据结构），支持消费组（Consumer Group）、消息确认（ACK）、消费者负载均衡。还支持 Redis 6+ 的 ACL 访问控制。

两者的队列名不能共用，Redis 里存的数据结构不同。如果你在老项目里用了 Bull，迁移到 BullMQ 需要等队列清空后再切换，否则数据会冲突。新项目直接上 BullMQ 就对了。

除了 BullMQ，还有没有其他选择？有。RabbitMQ 和 Kafka 是独立的消息中间件，功能更强大（支持消息路由、事务、持久化保证），但部署和运维成本也高得多。对于 NestJS 项目，BullMQ 依赖 Redis（你可能已经有了），上手成本低，功能足够覆盖 90% 的场景。除非你有非常特殊的消息可靠性要求，否则 BullMQ 是最务实的选择。

## 三个核心角色

BullMQ 的设计非常清晰，只有三个概念需要理解：

**Queue（队列）** 是生产者往里塞任务的容器。你可以把它想象成一张桌子，生产者把任务单放在桌上。它不执行任何任务，只负责存储和调度。你的 Controller 就是生产者。

**Worker（工人）** 是消费者，从队列里取任务执行。可以有多个 Worker 同时监听同一个队列，BullMQ 会自动分配任务——就像奶茶店可以雇佣多个员工同时做奶茶。你的 Processor 就是 Worker。

**Job（任务）** 是队列中的一条记录，包含数据（你传的参数）、状态（waiting/active/completed/failed）、重试次数、创建时间等信息。就像"一杯少糖抹茶拿铁，去冰"这张点单。

数据流是这样的：

```
Controller（生产者）          Redis             Worker（消费者）
     │                        │                     │
     │  1. add('post-published', {postId})            │
     ├───────────────────────►│                     │
     │  2. 返回 jobId（1ms）   │                     │
     │◄───────────────────────┤                     │
     │                        │  3. 取出任务          │
     │                        │◄────────────────────┤
     │                        │  4. 执行中...          │
     │                        │                     │
     │                        │  5. ACK 完成          │
     │                        │◄────────────────────┤
```

整个过程通过 Redis 协调，你的 HTTP 服务和 Worker 可以是不同的进程，甚至可以部署在不同的服务器上。

## 安装与配置

BullMQ 依赖 Redis，先在 docker-compose 里加一个 Redis 服务（如果你上一章还没加的话）：

```yaml
services:
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    command: redis-server --appendonly yes
    volumes:
      - redis_data:/data
volumes:
  redis_data:
```

`appendonly yes` 开启 AOF 持久化。Redis 默认是 RDB 持久化（定期快照），如果 Redis 在两次快照之间崩溃，可能丢失最近几分钟的任务。AOF 把每次写操作记录到日志文件里，最多丢失 1 秒的数据。对于任务队列来说，1 秒的数据丢失是可以接受的，但几分钟就不行了。

安装三个包：

```bash
pnpm add @nestjs/bullmq bullmq ioredis
```

`@nestjs/bullmq` 是 NestJS 的封装模块，把 BullMQ 包装成 NestJS 熟悉的 Module 形式。`bullmq` 是核心库，实际的任务队列逻辑。`ioredis` 是 Redis 客户端，BullMQ 底层通过它和 Redis 通信。

在 AppModule 里注册全局连接：

```typescript
// app.module.ts
import { BullMQModule } from "@nestjs/bullmq";

@Module({
  imports: [
    BullMQModule.forRoot({
      connection: {
        host: process.env.REDIS_HOST || "localhost",
        port: Number(process.env.REDIS_PORT) || 6379,
      },
      defaultJobOptions: {
        attempts: 3,
        backoff: { type: "exponential", delay: 1000 },
        removeOnComplete: { count: 1000 },
        removeOnFail: { count: 5000 },
      },
    }),
  ],
})
export class AppModule {}
```

这段配置值得逐条说清楚。

`attempts: 3` 表示任务失败最多重试 3 次。为什么是 3 而不是 5 或 10？因为大部分临时性故障（网络抖动、服务重启、限流）在 3 次内就能恢复。如果 3 次都失败，大概率是永久性错误（数据格式错误、地址不存在），重试更多次也没用。

`backoff: { type: 'exponential', delay: 1000 }` 是指数退避策略——第一次重试等 1 秒，第二次 2 秒，第三次 4 秒。给下游服务恢复的时间，而不是密集轰炸。

`removeOnComplete` 和 `removeOnFail` 控制历史任务在 Redis 里的保留数量。已完成任务保留 1000 条，失败任务保留 5000 条——失败任务多留一些是因为你需要排查失败原因。这些历史记录占用的 Redis 内存大约每条 1-2KB，1000 条也就 2MB，可以忽略。

## 文章发布：完整案例

回到 zhang_wei 的场景。他发布文章「从 Express 迁移到 NestJS 的踩坑记录」后，系统需要做三件事。我们创建一个专门的队列模块：

```typescript
// queue.module.ts
@Module({
  imports: [
    BullMQModule.registerQueue({
      name: "post-events",
    }),
  ],
  providers: [PostEventsProcessor],
  exports: [BullMQModule],
})
export class QueueModule {}
```

`registerQueue` 注册一个名为 `post-events` 的队列。这个名字是全局唯一的，Worker 和生产者通过这个名字来找到同一个队列。名字写错了不会报错，BullMQ 会默默创建一个新队列，任务就石沉大海了——所以队列名建议用常量管理。

Worker 是实际干活的地方。用 `@Processor` 装饰器声明监听哪个队列，用 `@Process` 声明处理哪种任务：

```typescript
// post-events.processor.ts
import {
  Processor,
  Process,
  OnQueueActive,
  OnQueueFailed,
} from "@nestjs/bullmq";
import { Job } from "bullmq";
import { Logger } from "@nestjs/common";

@Processor("post-events")
export class PostEventsProcessor {
  private readonly logger = new Logger(PostEventsProcessor.name);

  constructor(
    private readonly emailService: EmailService,
    private readonly aiService: AiService,
    private readonly searchService: SearchService
  ) {}

  @Process("post-published")
  async handlePostPublished(job: Job<{ postId: string }>) {
    const { postId } = job.data;
    this.logger.log(`开始处理文章发布后事件: ${postId}`);

    // 三件事并行执行，没有依赖关系
    const results = await Promise.allSettled([
      this.emailService.sendNotifications(postId),
      this.aiService.generateSummary(postId),
      this.searchService.updateIndex(postId),
    ]);

    // 检查是否有子任务失败
    const failures = results.filter((r) => r.status === "rejected");
    if (failures.length > 0) {
      failures.forEach((f, i) => {
        this.logger.warn(
          `子任务 ${i} 失败: ${(f as PromiseRejectedResult).reason.message}`
        );
      });
      // 部分失败不算整体失败，因为重试会重新执行所有子任务
      // 如果需要精确控制，可以把三个子任务拆成三个独立的 Job
    }

    this.logger.log(`文章发布后事件处理完成: ${postId}`);
  }

  @OnQueueActive()
  onActive(job: Job) {
    this.logger.log(`任务开始 [${job.id}]: ${job.name}`);
  }

  @OnQueueFailed()
  onFailed(job: Job, error: Error) {
    this.logger.error(
      `任务失败 [${job.id}]: ${job.name}, 第 ${job.attemptsMade} 次, ${error.message}`
    );
  }
}
```

这里用了 `Promise.allSettled` 而不是 `Promise.all`。三件事之间没有依赖关系——发邮件不需要等摘要生成完，搜索索引也不依赖邮件。`Promise.all` 的坑是：如果其中一个 Promise 抛异常，其他的也会被中断。`Promise.allSettled` 让每个操作独立处理自己的失败。

不过 `Promise.allSettled` 也有一个代价：BullMQ 的重试是整个 Job 级别的。如果邮件失败了，重试时三个子任务都会重新执行。摘要和索引已经成功过一次，再次执行可能是幂等的（重复执行没问题），也可能不是。更好的做法是把三个子任务拆成三个独立的 Job，各自重试：

```typescript
async publishPost(userId: string, postId: string) {
  const post = await this.prisma.post.update({ ... });

  // 三个独立任务，各自失败各自重试
  await Promise.all([
    this.postQueue.add('send-notifications', { postId: post.id }),
    this.postQueue.add('generate-summary', { postId: post.id }),
    this.postQueue.add('update-search-index', { postId: post.id }),
  ]);

  return post;
}
```

业务代码里添加任务只需要注入队列：

```typescript
// posts.service.ts
@Injectable()
export class PostsService {
  constructor(
    private prisma: PrismaService,
    @InjectQueue("post-events") private postQueue: Queue
  ) {}

  async publishPost(userId: string, postId: string) {
    const post = await this.prisma.post.update({
      where: { id: postId, authorId: userId },
      data: { status: "PUBLISHED", publishedAt: new Date() },
    });

    await this.postQueue.add("post-published", { postId: post.id });
    return post;
  }
}
```

`@InjectQueue('post-events')` 注入队列实例。`add` 的第一个参数是任务名（对应 `@Process('post-published')`），第二个参数是传递给 Worker 的数据。`add()` 返回后，任务已经安全地躺在 Redis 里了。

## 任务优先级：不是所有任务都平等

假设 zhang_wei 发布文章时触发了 28 封通知邮件入队。与此同时，用户 li_na 点击了"忘记密码"需要一封重置邮件。密码重置邮件的紧迫程度明显高于文章通知——li_na 可能在等着登录，每多等一秒体验就差一分。

BullMQ 支持优先级队列，数字越小优先级越高：

```typescript
// 密码重置：优先级 1，最紧急
await this.emailQueue.add("password-reset", data, { priority: 1 });

// 文章通知：优先级 5，普通
await this.emailQueue.add("post-notification", data, { priority: 5 });

// 欢迎邮件：优先级 10，最低，用户不急着看到
await this.emailQueue.add("welcome", data, { priority: 10 });
```

要启用优先级，注册队列时需要配置默认优先级：

```typescript
BullMQModule.registerQueue({
  name: 'email',
  defaultJobOptions: { priority: 5 },
}),
```

Worker 取任务时，总是先取优先级最高的。如果队列里有 28 封通知邮件（优先级 5）和 1 封密码重置邮件（优先级 1），Worker 会先处理密码重置。

不过这里有一个代价：启用优先级后，Worker 每次取任务都要对队列排序，性能会有轻微下降。如果你不需要优先级（所有任务同等重要），不要开启。

## 延迟执行：不是所有任务都急

有些任务不需要立刻执行。比如 zhang_wei 注册了账号但一直没填个人资料，你想在 1 小时后发一封"完善资料"的提醒邮件。BullMQ 支持延迟任务：

```typescript
await this.emailQueue.add(
  "profile-reminder",
  { userId: user.id },
  {
    delay: 60 * 60 * 1000, // 1 小时后执行（单位：毫秒）
  }
);
```

延迟任务在 Redis 里存储，到时间后自动变为 waiting 状态被 Worker 取走。即使服务重启，延迟任务也不会丢。

另一个常见场景是定时任务。比如每天凌晨统计全站的阅读数据并生成报告：

```typescript
// 用 BullMQ 的 repeat 选项实现定时任务
await this.reportQueue.add(
  "daily-report",
  {},
  {
    repeat: { every: 24 * 60 * 60 * 1000 }, // 每 24 小时执行一次
  }
);
```

比 `@nestjs/schedule` 的优势是：定时任务也走队列，可以享受重试、监控、并发控制等所有队列能力。`@nestjs/schedule` 是在进程内执行，服务挂了任务就丢了。

## 重试策略：指数退避的艺术

网络抖动、第三方服务限流、数据库连接超时——任务失败是常态。关键是怎么重试。

最天真的做法是立即重试三次。如果邮件服务正在重启，3 次请求间隔 0 秒，大概率还是失败。而且密集的重试请求可能让本就脆弱的服务雪上加霜。

指数退避的思路是：每次失败后等更长时间，给服务恢复的机会。

```typescript
await this.emailQueue.add("notification", data, {
  attempts: 5,
  backoff: {
    type: "exponential",
    delay: 2000,
  },
});
```

实际的时间线是：

```
第 1 次执行 → 失败 → 等 2 秒
第 2 次重试 → 失败 → 等 4 秒
第 3 次重试 → 失败 → 等 8 秒
第 4 次重试 → 失败 → 等 16 秒
第 5 次重试 → 失败 → 标记为 dead
```

如果邮件服务在 10 秒内恢复了，第三次重试就能成功，总等待时间只有 2 + 4 = 6 秒。5 次重试全部失败后，总耗时 2 + 4 + 8 + 16 = 30 秒。

BullMQ 支持三种退避策略：

| 策略          | 行为                              | 适用场景                         |
| ------------- | --------------------------------- | -------------------------------- |
| `exponential` | 每次等待时间翻倍（2s, 4s, 8s...） | 调用外部 API，网络可能临时故障   |
| `fixed`       | 每次等待固定时间（2s, 2s, 2s...） | 数据库连接超时等可预测的错误     |
| `custom`      | 自定义函数决定等待时间            | 需要根据 HTTP 状态码决定等待时间 |

`custom` 适合需要根据错误类型决定行为的场景。比如收到 429 Too Many Requests 就多等一会儿，收到 500 Internal Server Error 就少等一会儿：

```typescript
backoff: {
  type: 'custom',
  async delay(attemptsMade: number, err: Error) {
    if (err.message.includes('429')) {
      return 10000; // 限流了，等 10 秒
    }
    return 2000 * Math.pow(2, attemptsMade); // 其他错误，指数退避
  },
},
```

在 Processor 里可以通过 `job.attemptsMade` 知道当前是第几次尝试，据此调整行为：

```typescript
@Process('notification')
async handleNotification(job: Job) {
  if (job.attemptsMade > 1) {
    this.logger.warn(
      `通知邮件第 ${job.attemptsMade} 次重试，文章: ${job.data.postId}`,
    );
  }
  // 不需要 try-catch 手动重试
  // 只管抛错，BullMQ 自动捕获并根据退避策略调度
  await this.emailService.send(job.data);
}
```

## 死信队列：重试也救不了的任务

如果一封邮件重试 5 次仍然失败（比如收件地址格式永久错误），BullMQ 会把它移到 failed 队列。这就是"死信队列"——任务到了这里说明常规重试手段已经无效了。

死信任务不应该被遗忘。有两种常见的处理方式：

第一种是人工介入。通过 Bull Board（下一节介绍）查看失败任务列表，找到失败原因，修复后手动重试。适合需要人工判断的场景（比如数据错误）。

第二种是自动告警。在 `@OnQueueFailed` 钩子里，当重试次数耗尽时发送通知：

```typescript
@OnQueueFailed()
onFailed(job: Job, error: Error) {
  if (job.attemptsMade >= job.opts.attempts) {
    this.alertService.send(
      `任务 ${job.name}[${job.id}] 重试耗尽: ${error.message}\n` +
      `数据: ${JSON.stringify(job.data)}`,
    );
  }
}
```

实际生产中，两种方式要配合使用。告警让你第一时间知道有问题，Bull Board 让你方便地排查和处理。

## 并发控制：一个 Worker 能同时干几件事

默认情况下，一个 Worker 每次只处理一个 Job。`@Processor` 装饰器的第二个参数可以调并发数：

```typescript
@Processor("email", { concurrency: 10 })
export class EmailProcessor {
  @Process("send")
  async handleSend(job: Job) {
    await this.emailService.send(job.data);
  }
}
```

`concurrency: 10` 意味着这个 Worker 可以同时处理 10 封邮件。但这里有个常见的坑：你的上游服务能扛住 10 个并发请求吗？

如果你用 SMTP 发邮件，nodemailer 默认连接池大小是 5。并发设到 10，但连接池只有 5，多余的 5 个任务会在等连接，反而变慢。你需要先了解上游服务的并发承载能力，再决定 Worker 的并发数。

一般来说：

- I/O 密集型任务（发邮件、调 HTTP API）：并发可以设高一些，10-20
- CPU 密集型任务（图片压缩、PDF 生成）：设为 CPU 核心数，Node.js 单线程，设太高只会增加内存占用和上下文切换开销
- 有上游限流的任务：根据限流阈值来设，否则会被上游拒绝

不确定的时候怎么办？从 5 开始，通过 Bull Board 观察队列的 waiting 数量。如果 waiting 一直堆积说明 Worker 吃不消，加大并发；如果 Worker CPU 利用率很低但 waiting 也是 0，说明并发够用了。

## 多队列：不同类型的任务分开管理

随着业务增长，你可能会遇到不同类型的耗时任务。我们的博客系统目前有邮件发送和文章事件处理，它们的特点不同。如果未来加入"数据导出"这种耗时很长且占用内存的任务，和邮件放在同一个队列里，可能会互相影响。

BullMQ 支持注册多个队列，每个队列有独立的配置和 Worker：

```typescript
// queue.module.ts
@Module({
  imports: [
    // 邮件队列：重试次数多一些，因为网络问题常见
    BullMQModule.registerQueue({
      name: "email",
      defaultJobOptions: {
        attempts: 5,
        backoff: { type: "exponential", delay: 2000 },
      },
    }),
    // 文章事件队列：标准配置
    BullMQModule.registerQueue({
      name: "post-events",
      defaultJobOptions: {
        attempts: 3,
        backoff: { type: "exponential", delay: 1000 },
      },
    }),
    // 数据导出队列：任务耗时长，超时设大一些
    BullMQModule.registerQueue({
      name: "export",
      defaultJobOptions: {
        attempts: 2,
        timeout: 300_000, // 5 分钟超时
      },
    }),
  ],
  providers: [EmailProcessor, PostEventsProcessor, ExportProcessor],
  exports: [BullMQModule],
})
export class QueueModule {}
```

分开管理的好处是：Worker 可以独立部署和扩缩容。邮件发送量大的时候，单独给邮件队列加 Worker，不影响其他队列。数据库导出耗时很长，可以部署到另一台机器上，不占主服务的资源。每个队列的失败重试策略也可以不同。

## Bull Board：看见队列里在发生什么

在生产环境里，你不能靠猜来了解队列状态。Bull Board 是一个轻量的 Web 监控面板，能让你看到每个队列的运行全貌。

```bash
pnpm add @bull-board/api @bull-board/nestjs
```

在队列模块里注册需要监控的队列：

```typescript
import { BullBoardModule } from "@bull-board/nestjs";
import { BullMQAdapter } from "@bull-board/api/bullMQAdapter";

@Module({
  imports: [
    BullBoardModule.forRoot({
      route: "/admin/queues",
      adapter: BullMQAdapter,
    }),
    BullBoardModule.forFeature({
      name: "post-events",
      adapter: BullMQAdapter,
    }),
    BullBoardModule.forFeature({
      name: "email",
      adapter: BullMQAdapter,
    }),
  ],
})
export class QueueModule {}
```

启动后访问 `http://localhost:3000/admin/queues`，你能看到：

- 每个队列的等待、活跃、完成、失败任务数量
- 每个任务的详细数据、创建时间、处理耗时
- 可以手动重试失败的任务
- 可以往队列里添加测试任务

这对调试非常方便。zhang_wei 的通知邮件发送失败了？打开 Bull Board，找到 failed 列表，看错误信息，修复后点一下 Retry。

但有一个安全风险：这个面板没有任何鉴权，任何人访问这个 URL 就能看到你的任务数据。生产环境必须用前面学过的 Guard 来保护这个路由：

```typescript
@Controller("admin/queues")
@UseGuards(AdminGuard)
export class AdminQueueController {
  // Bull Board 会自动注册路由到这个路径
}
```

## 什么时候不该用队列

队列不是银弹。如果一个操作满足以下条件，就不要用队列：

- **用户需要看到结果才能继续操作**。比如创建文章后要拿到文章 ID 跳转到编辑页，这个必须同步。如果用队列，你拿不到创建结果。
- **操作本身很快**（< 100ms）。入队的开销（写 Redis 约 1ms + Worker 取任务 + 序列化/反序列化）可能比直接执行还大。
- **数据一致性要求极高**，不能接受"最终一致"。比如账户转账，扣款和加款必须是同一个事务。如果扣款成功但加款的任务失败了，钱就凭空消失了。

还有一个灰色地带是**数据导出**。如果导出数据很快（< 1 秒），直接同步返回就行。如果很慢，应该先返回一个"导出中"的状态，通过队列生成文件，完成后通知用户下载。这个状态可以用数据库的一个字段（`exportStatus: 'pending' | 'completed'`）来记录，前端轮询这个状态。

在我们的博客系统里，文章的增删改查走 HTTP 同步请求，邮件通知、摘要生成、索引更新走 BullMQ 异步。各司其职。

另一个容易忽略的点是：队列引入了对 Redis 的依赖。如果 Redis 挂了，不仅队列不能用，`queue.add()` 可能会阻塞 HTTP 请求（取决于配置）。生产环境需要 Redis 高可用（Sentinel 或 Cluster），并且给 `queue.add()` 设置合理的超时时间。

## 小结

回顾一下这篇解决的问题。zhang_wei 发布文章后，后端需要做三件耗时的事。通过 BullMQ，我们把 `POST /posts/:id/publish` 的响应时间从 6.2 秒降到了 50 毫秒以内。邮件、摘要、索引在后台异步完成，失败自动重试（指数退避），重试耗尽进死信队列，整个流程通过 Bull Board 可观测。

核心的 trade-off 是：你引入了对 Redis 的依赖，系统多了一个需要维护的组件。Worker 进程崩溃需要自动重启（用 PM2 或 Docker 的 restart policy），Redis 持久化需要正确配置，任务失败需要告警机制。如果你的项目总共就 3 个接口，根本没有耗时操作，那这些复杂性是得不偿失的。但当你的业务开始涉及邮件、推送、报表、文件处理这些场景，队列几乎是必选项。

BullMQ 本身依赖 Redis 来存储任务数据。下一篇我们会复用这个 Redis 实例，做另一件事——缓存。首页文章列表每次查数据库，热门文章浏览量更新频繁导致缓存失效，怎么设计一个又快又准的缓存方案，下一篇聊。
