上一篇我们给博客平台写了第一个控制器 `ArticlesController`，能返回文章列表、文章详情。但有个问题我们刻意回避了：那些文章数据是硬编码在控制器里的。真实项目里，数据从数据库来、要分页排序、要校验用户权限——这些逻辑如果全塞在控制器里，控制器很快就会变成一个几百行的"上帝类"。

更糟的是，你会发现同样的查询逻辑在三个地方被复制粘贴。改一个分页逻辑，要同时改三个控制器。这不是"维护"，这是"排雷"。

这一篇，我们把这些逻辑抽到 Service 层。

## 一个"胖控制器"的真实问题

先看看不拆分时的代码长什么样。假设 `ArticlesController` 需要支持分页、关键词搜索、按标签筛选，同时还要查每篇文章的作者信息：

```typescript
// articles.controller.ts — 胖控制器，所有逻辑混在一起
@Controller("articles")
export class ArticlesController {
  constructor(private prisma: PrismaService) {}

  @Get()
  async findAll(
    @Query("page") page: string = "1",
    @Query("limit") limit: string = "10",
    @Query("keyword") keyword?: string,
    @Query("tag") tag?: string
  ) {
    const pageNum = parseInt(page);
    const limitNum = parseInt(limit);
    const skip = (pageNum - 1) * limitNum;

    // 构建 where 条件
    const where: any = {};
    if (keyword) {
      where.OR = [
        { title: { contains: keyword, mode: "insensitive" } },
        { content: { contains: keyword, mode: "insensitive" } },
      ];
    }
    if (tag) {
      where.tags = { some: { slug: tag } };
    }

    const [articles, total] = await Promise.all([
      this.prisma.article.findMany({
        where,
        skip,
        take: limitNum,
        orderBy: { createdAt: "desc" },
        include: { author: true, tags: true },
      }),
      this.prisma.article.count({ where }),
    ]);

    return {
      data: articles,
      meta: {
        page: pageNum,
        limit: limitNum,
        total,
        totalPages: Math.ceil(total / limitNum),
      },
    };
  }
}
```

这还只是一个接口。如果把详情、创建、更新都写进去，控制器轻松超过 300 行。

问题不止于"文件太长"。如果你还有一个 `HomeController` 需要展示首页的推荐文章，或者一个 `AdminController` 需要统计文章数据——同样的查询逻辑、同样的分页计算、同样的 where 条件构建——你要么复制粘贴，要么把 PrismaService 和查询逻辑暴露给每个控制器。两条路都通向维护噩梦。

## Service 层：控制器的"后厨"

解决方案是把控制器比作餐厅的服务员——负责接待客人、传递菜单、上菜。至于菜怎么做、食材从哪来，那是后厨（Service）的事。

NestJS 里，Service 就是一个加了 `@Injectable()` 装饰器的普通类。它通过构造函数接收依赖（比如数据库连接、其他 Service），然后在模块的 `providers` 数组里注册。

```typescript
// articles.service.ts
import { Injectable } from "@nestjs/common";

@Injectable()
export class ArticlesService {
  // 业务逻辑放在这里
}
```

`@Injectable()` 告诉 NestJS 的 IoC 容器："这个类可以被注入到其他地方。"和控制器不同，Service 不关心 HTTP 请求长什么样，它只关心业务规则。

我们来看看把文章查询抽到 Service 后，控制器有多清爽：

```typescript
// articles.controller.ts — 瘦控制器
@Controller("articles")
export class ArticlesController {
  constructor(private articlesService: ArticlesService) {}

  @Get()
  findAll(@Query() query: ArticleQueryDto) {
    return this.articlesService.findPublished(query);
  }
}
```

控制器变成了一行代理。它不关心分页怎么算、关键词怎么搜——这些全在 Service 里。

## ArticleService：从假数据到真实查询

现在来写 `ArticleService`。目前数据库还没接（下一篇的事），我们先用内存数据模拟，重点是设计 Service 的接口和职责划分。

```typescript
// articles.service.ts
import { Injectable, NotFoundException } from "@nestjs/common";

export interface ArticleQuery {
  page?: number;
  limit?: number;
  keyword?: string;
  tag?: string;
}

@Injectable()
export class ArticlesService {
  // 临时数据，下一篇换成 TypeORM
  private articles = [
    {
      id: "post_001",
      title: "从 Express 迁移到 NestJS 的踩坑记录",
      excerpt: "记录了从 Express 迁移到 NestJS 过程中遇到的坑和解决方案",
      content: "（省略正文）...",
      views: 3842,
      authorId: "usr_a1x9k2m",
      publishedAt: "2025-12-15T08:30:00Z",
      createdAt: "2025-12-14T22:00:00Z",
    },
    {
      id: "post_002",
      title: "NestJS 依赖注入到底怎么工作",
      excerpt: "从源码角度理解 NestJS 的 IoC 容器",
      content: "（省略正文）...",
      views: 2156,
      authorId: "usr_a1x9k2m",
      publishedAt: "2026-01-08T10:00:00Z",
      createdAt: "2026-01-07T18:00:00Z",
    },
  ];

  async findPublished(query: ArticleQuery) {
    const { page = 1, limit = 10, keyword, tag } = query;
    const skip = (page - 1) * limit;

    let filtered = this.articles;

    // 关键词搜索：在标题和摘要中匹配
    if (keyword) {
      const lower = keyword.toLowerCase();
      filtered = filtered.filter(
        (a) =>
          a.title.toLowerCase().includes(lower) ||
          a.excerpt.toLowerCase().includes(lower)
      );
    }

    // 标签筛选（等接了数据库再做完整实现）
    // if (tag) { ... }

    const total = filtered.length;
    const items = filtered.slice(skip, skip + limit);

    return {
      data: items,
      meta: { page, limit, total, totalPages: Math.ceil(total / limit) },
    };
  }

  async findOne(id: string) {
    const article = this.articles.find((a) => a.id === id);
    if (!article) {
      throw new NotFoundException(`文章 ${id} 不存在`);
    }
    return article;
  }
}
```

注意几个设计决策：

**1. 方法名用业务语义，不用技术语义。** `findPublished` 而不是 `query` 或 `getList`——调用者一看就知道这是"查已发布的文章"，而不是泛泛的"获取数据"。

**2. Service 抛异常，控制器不处理。** `NotFoundException` 直接从 Service 抛出，NestJS 的异常过滤器会自动把它转成 404 响应。控制器不需要 try-catch。

**3. 返回值带 `meta`。** 分页信息是列表查询的标准组成部分，前端需要知道总数才能渲染分页器。把它作为约定的一部分，而不是让前端自己数。

**4. 临时数据结构不会影响接口设计。** 现在用的是内存数组，下一篇换成 TypeORM 的 Repository，但 `findPublished` 的入参和返回值结构不会变。这就是 Service 层抽象的价值——底层实现可以换，对外接口保持稳定。

## Service 之间的协作：ArticleService 调用 UserService

现实中的接口几乎不会只查一张表。文章详情页要显示作者头像和昵称，首页要显示"热门作者"——这意味着 `ArticleService` 需要调用 `UserService`。

先写一个简单的 `UserService`：

```typescript
// users.service.ts
import { Injectable, NotFoundException } from "@nestjs/common";

@Injectable()
export class UserService {
  private users = [
    {
      id: "usr_a1x9k2m",
      username: "zhang_wei",
      nickname: "张威",
      bio: "前端工程师，最近在学 NestJS",
      avatar: "https://cdn.example.com/avatars/zhang_wei.jpg",
      createdAt: "2025-03-20T10:00:00Z",
    },
  ];

  async findById(id: string) {
    const user = this.users.find((u) => u.id === id);
    if (!user) {
      throw new NotFoundException(`用户 ${id} 不存在`);
    }
    // 不返回敏感信息（虽然现在还没有密码字段）
    const { id: _id, ...profile } = user;
    return { id: _id, ...profile };
  }

  async findProfileBatch(ids: string[]) {
    return this.users
      .filter((u) => ids.includes(u.id))
      .map((u) => {
        const { id, username, nickname, avatar } = u;
        return { id, username, nickname, avatar };
      });
  }
}
```

然后在 `ArticleService` 里注入它：

```typescript
// articles.service.ts
@Injectable()
export class ArticlesService {
  constructor(private userService: UserService) {}

  async findPublished(query: ArticleQuery) {
    // ... 筛选逻辑同上 ...

    // 批量查作者信息，避免 N+1 查询
    const authorIds = [...new Set(items.map((a) => a.authorId))];
    const authors = await this.userService.findProfileBatch(authorIds);
    const authorMap = new Map(authors.map((a) => [a.id, a]));

    // 把作者信息附加到文章上
    const data = items.map((article) => ({
      ...article,
      author: authorMap.get(article.authorId) || null,
    }));

    return {
      data,
      meta: { page, limit, total, totalPages: Math.ceil(total / limit) },
    };
  }
}
```

关键点在于 `findProfileBatch` 这个方法。如果每次查一篇文章就调一次 `findById`，10 篇文章就是 10 次查询。批量查询一次拿回来，用 Map 做 O(1) 查找——这是 Service 层该做的优化。

等下一篇接了数据库，这个模式会变成 TypeORM 的 `IN` 查询，但 Service 层的接口设计不变。这就是抽离业务逻辑的好处：**换数据库不影响上层代码。**

## 模块注册：把 Service 装到 NestJS 的容器里

Service 写好了，但 NestJS 还不知道它的存在。需要在模块里注册：

```typescript
// articles.module.ts
import { Module } from "@nestjs/common";
import { ArticlesController } from "./articles.controller";
import { ArticlesService } from "./articles.service";
import { UserService } from "../users/users.service";

@Module({
  controllers: [ArticlesController],
  providers: [ArticlesService, UserService],
  exports: [ArticlesService], // 导出，供其他模块使用
})
export class ArticlesModule {}
```

`exports` 是关键。如果 `CommentsModule` 也需要查文章信息，只要在它的 `imports` 里加上 `ArticlesModule`，就能注入 `ArticlesService`。不导出的话，其他模块用不了——这是 NestJS 的封装机制。

```typescript
// comments.module.ts
import { Module } from "@nestjs/common";
import { CommentsService } from "./comments.service";
import { ArticlesModule } from "../articles/articles.module";

@Module({
  imports: [ArticlesModule], // 导入文章模块
  providers: [CommentsService],
})
export class CommentsModule {}

// comments.service.ts
@Injectable()
export class CommentsService {
  constructor(private articlesService: ArticlesService) {
    // NestJS 自动注入，因为 ArticlesModule 导出了 ArticlesService
  }
}
```

## 自定义 Provider：useFactory 做动态配置

有时候你要注入的东西不是一个简单的类，而是一个需要根据配置动态创建的对象。比如，你的日志服务在开发环境输出彩色日志，在生产环境写入文件。

NestJS 的 `useFactory` 就是干这个的：

```typescript
// logger.module.ts
import { Module } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";

// 日志服务接口
export abstract class LoggerService {
  abstract log(message: string): void;
  abstract error(message: string, trace?: string): void;
}

// 开发环境：彩色控制台输出
@Injectable()
export class DevLoggerService extends LoggerService {
  log(message: string) {
    console.log(`\x1b[32m[DEV]\x1b[0m ${message}`);
  }
  error(message: string, trace?: string) {
    console.error(`\x1b[31m[ERROR]\x1b[0m ${message}`);
    if (trace) console.error(trace);
  }
}

// 生产环境：写入文件
@Injectable()
export class ProdLoggerService extends LoggerService {
  log(message: string) {
    // 写入日志文件
  }
  error(message: string, trace?: string) {
    // 写入错误日志文件
  }
}

@Module({
  providers: [
    {
      provide: LoggerService,
      useFactory: (configService: ConfigService) => {
        const env = configService.get("NODE_ENV");
        return env === "production"
          ? new ProdLoggerService()
          : new DevLoggerService();
      },
      inject: [ConfigService], // 注入 ConfigService 到工厂函数
    },
  ],
  exports: [LoggerService],
})
export class LoggerModule {}
```

`useFactory` 的三个关键部分：

- `provide`：注入的 token，其他类用 `@Inject(LoggerService)` 来接收
- `useFactory`：工厂函数，返回实际的对象
- `inject`：工厂函数本身需要什么依赖，在这里声明

这比在 Service 里写 `if (process.env.NODE_ENV === 'production')` 优雅得多——日志服务本身不知道自己运行在什么环境，环境选择是模块配置的事。

`useValue` 适合提供固定值。比如博客平台有一些全局配置（站点名称、每页默认条数、最大上传大小），可以用 `useValue` 注入：

```typescript
// config.module.ts
@Module({
  providers: [
    {
      provide: "APP_CONFIG",
      useValue: {
        siteName: "技术博客",
        defaultPageSize: 10,
        maxUploadSize: 5 * 1024 * 1024, // 5MB
        allowedImageTypes: ["image/jpeg", "image/png", "image/webp"],
      },
    },
  ],
  exports: ["APP_CONFIG"],
})
export class AppConfigModule {}

// 在其他 Service 中使用
@Injectable()
export class UploadService {
  constructor(@Inject("APP_CONFIG") private config: typeof appConfig) {
    console.log(`最大上传大小: ${this.config.maxUploadSize}`);
  }
}
```

`useValue` 提供的是一个对象字面量，TypeScript 默认推断为 `{ siteName: string, ... }` 类型。接收方用 `@Inject('APP_CONFIG')` 获取，因为 `provide` 的 token 是字符串而不是类。如果觉得字符串 token 不够类型安全，可以定义一个 InjectionToken（TypeScript 3.2+ 的功能），但这个优化留到后续章节再展开。

还有 `useExisting`（给现有 Provider 取个别名），用法类似，在后续章节遇到具体场景时再说。

## 请求作用域 Provider：不是默认选择

默认情况下，NestJS 的 Provider 都是单例——整个应用生命周期只有一个实例，所有请求共享。这在绝大多数场景下是正确且高效的选择。

但偶尔你会遇到需要"每个请求独立一个实例"的情况，比如一个需要追踪当前请求上下文的服务：

```typescript
import { Injectable, Scope } from "@nestjs/common";

@Injectable({ scope: Scope.REQUEST })
export class RequestContextService {
  private requestId: string;
  private startTime: number;

  setContext(requestId: string) {
    this.requestId = requestId;
    this.startTime = Date.now();
  }

  getDuration() {
    return Date.now() - this.startTime;
  }
}
```

加了 `scope: Scope.REQUEST` 后，每个 HTTP 请求进来时 NestJS 都会创建一个新的 `RequestContextService` 实例。请求结束后，实例被回收。

但要注意代价：**请求作用域的 Provider 会显著增加内存和 GC 压力**，因为每个请求都要 new 一个对象。NestJS 官方也明确建议，除非有明确的理由，否则不要用请求作用域。大多数"需要请求上下文"的场景，用一个请求对象存储（比如 `AsyncLocalStorage`）就能解决，不需要创建整个 Service 的副本。

## Service 的可测试性：为什么要抽离

最后说说测试。这是 Service 层最有说服力的好处之一。

假设你要测试"文章不存在时抛出 404"这个逻辑。如果逻辑写在控制器里，你需要模拟 HTTP 请求、设置路由参数、启动整个 NestJS 应用。但如果逻辑在 Service 里，测试只需要几行：

```typescript
// articles.service.spec.ts
import { Test, TestingModule } from "@nestjs/testing";
import { ArticlesService } from "./articles.service";
import { NotFoundException } from "@nestjs/common";

describe("ArticlesService", () => {
  let service: ArticlesService;

  beforeEach(async () => {
    const module: TestingModule = await Test.createTestingModule({
      providers: [ArticlesService],
    }).compile();
    service = module.get<ArticlesService>(ArticlesService);
  });

  it("查询不存在的文章应抛出 NotFoundException", async () => {
    await expect(service.findOne("nonexistent_id")).rejects.toThrow(
      NotFoundException
    );
  });

  it("分页应正确计算 totalPages", async () => {
    const result = await service.findPublished({ page: 1, limit: 10 });
    // 2 篇文章，每页 10 篇，应该是 1 页
    expect(result.meta.totalPages).toBe(1);
  });
});
```

不需要数据库，不需要 HTTP 服务器，不需要任何外部依赖。`Test.createTestingModule` 是 NestJS 提供的测试工具，它能创建一个迷你的 DI 容器，只注册你需要的 Provider。这比启动整个应用快几百倍。

如果 Service 依赖其他 Service（比如 `ArticleService` 依赖 `UserService`），你可以用 mock 替换：

```typescript
beforeEach(async () => {
  const mockUserService = {
    findProfileBatch: jest
      .fn()
      .mockResolvedValue([
        { id: "usr_a1x9k2m", username: "zhang_wei", nickname: "张威" },
      ]),
  };

  const module: TestingModule = await Test.createTestingModule({
    providers: [
      ArticlesService,
      { provide: UserService, useValue: mockUserService },
    ],
  }).compile();
  service = module.get<ArticlesService>(ArticlesService);
});
```

`{ provide: UserService, useValue: mockUserService }` 就是用一个 mock 对象替换真实的 `UserService`。Service 层的依赖注入让这种替换变得轻而易举。

## Trade-off：什么时候该拆，什么时候不该

Service 层不是免费的。它增加了一层间接性，多了文件要维护，代码路径变长了。什么时候值得做？

**值得拆的情况：**

- 同一段查询逻辑出现在两个以上的地方
- 你需要单独测试业务规则（不依赖 HTTP 请求）
- Service 需要被多个控制器或模块复用
- 业务逻辑包含条件判断、数据转换、规则校验——这些不该放在控制器里

**暂时不拆的情况：**

- 一个 CRUD 接口，Service 只是 `return this.repository.find()`，没有任何业务逻辑——这种"透传 Service"不如直接在控制器里用 Repository
- 项目的早期原型阶段，业务逻辑还没稳定，过早拆分反而增加重构成本

一个实用的判断标准：如果你的 Service 方法里没有任何 if/else、没有任何数据转换、没有任何业务规则判断——它只是一个薄薄的 Repository 包装——那它可能还不值得存在。等业务逻辑复杂了再抽也不迟。

在实际项目中，我见过两种极端：

一种是"Service 万物论"——即使只有一个 `find()` 方法，也要创建一个 Service 文件。结果是项目里有一堆空壳 Service，每个只有两三行代码，维护它们的成本超过了收益。

另一种是"控制器通吃论"——所有逻辑都在控制器里，一个文件 800 行，方法之间靠复制粘贴共享逻辑。新人接手时根本不知道该从哪里看起。

好的做法是在中间找平衡。初期可以控制器直接用 Repository，等业务逻辑变复杂了（出现 if/else、出现数据转换、出现跨 Service 调用），再把逻辑抽到 Service 里。NestJS 的好处是，这个重构过程成本很低——把方法从控制器剪切到 Service，控制器改成调用 Service，完事。

## 循环依赖：Service 之间的"先有鸡还是先有蛋"

Service 之间互相调用是很常见的。`ArticleService` 需要 `UserService` 查作者信息，`UserService` 可能也需要 `ArticleService` 统计用户的文章数。一旦两个 Service 互相引用，NestJS 启动时会报错：`Circular dependency detected`。

原因很简单：NestJS 的 IoC 容器按顺序初始化 Provider。如果 A 依赖 B，B 又依赖 A，容器不知道该先创建谁。

NestJS 提供了一个应急方案 `forwardRef`：

```typescript
// articles.service.ts
import { forwardRef, Inject } from "@nestjs/common";

@Injectable()
export class ArticlesService {
  constructor(
    @Inject(forwardRef(() => UserService))
    private userService: UserService
  ) {}
}

// users.service.ts
import { forwardRef, Inject } from "@nestjs/common";

@Injectable()
export class UserService {
  constructor(
    @Inject(forwardRef(() => ArticlesService))
    private articlesService: ArticlesService
  ) {}
}
```

`forwardRef` 把引用延迟到运行时解析，绕过了初始化顺序的问题。能用，但这是一个"治标不治本"的方案。

更好的做法是重新审视设计：两个 Service 互相依赖，通常意味着它们的职责边界不清。比如"统计用户文章数"这件事，其实可以放在一个独立的 `UserStatsService` 里：

```typescript
// user-stats.service.ts — 打破循环依赖的中间层
@Injectable()
export class UserStatsService {
  constructor(private articleRepo: Repository<Article>) {}

  async getArticleCount(userId: string): Promise<number> {
    return this.articleRepo.count({ where: { authorId: userId } });
  }
}
```

`UserService` 依赖 `UserStatsService`，`ArticleService` 不需要知道 `UserService`。循环依赖消失了。

这个思路的本质是：**当你发现 A 和 B 互相依赖时，大概率是因为 A 和 B 之间有一个隐藏的 C——一个还没被识别出来的独立职责。** 找到 C，把它抽出来，依赖关系就变成了单向的。

## 小结

这一篇我们从"控制器太胖"这个真实问题出发，把业务逻辑抽到了 Service 层。核心收获：

- `@Injectable()` 让一个类可以被 NestJS 的 IoC 容器管理
- 通过构造函数注入依赖，Service 之间可以协作
- 模块的 `exports` 决定了哪些 Service 可以被外部使用
- `useFactory` 支持动态配置 Provider，`useValue` 提供固定值
- 抽离后的 Service 天然适合单元测试，用 `Test.createTestingModule` + mock 即可
- 请求作用域有性能代价，不是默认选择
- 循环依赖的根源通常是职责边界不清，`forwardRef` 只是应急方案

但有一个问题我们一直在回避：Service 里的数据是假的。数组里的 `post_001` 每次重启应用都会消失。下一篇，我们接入 PostgreSQL，用 TypeORM 让数据真正持久化。
