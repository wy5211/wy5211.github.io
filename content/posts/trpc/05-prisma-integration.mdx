TaskFlow 的认证和 API 都就绪了，但数据还住在内存里——上一章的 `db.project.findMany()` 实际上是从一个 JavaScript 数组里读数据。服务器一重启，所有项目都没了。

今天把数据迁移到 PostgreSQL。之所以选 Prisma，是因为它和 tRPC 在理念上高度一致：**都是"代码即定义"，不需要额外的 Schema 文件或代码生成步骤**。

## 为什么是 Prisma

tRPC 解决的是前后端之间的类型同步问题，Prisma 解决的是后端和数据库之间的类型同步问题。两者组合起来，就形成了一条完整的类型安全链路：

```
数据库 Schema → Prisma 自动生成类型 → tRPC Procedure → 前端自动推断
     ↓                  ↓                    ↓               ↓
  PostgreSQL         TypeScript           TypeScript      TypeScript
```

Prisma 的工作方式是：你在 `schema.prisma` 里用直观的语法定义数据模型，然后运行 `prisma generate`，Prisma 会自动生成对应的 TypeScript 类型。你在代码里写的 `db.project.findMany()` 返回的是 `Project[]` 类型，不需要手写任何 interface。

对比其他方案：TypeORM 需要用装饰器定义 Entity 和 Column（手写类型），Drizzle 需要用 TypeScript 定义 schema 对象（手写类型）。Prisma 的 schema 语法更接近自然语言，而且类型完全自动生成。

## TaskFlow 的数据模型

设计数据模型之前，先想清楚 TaskFlow 的核心业务关系：

- 一个用户可以属于多个项目
- 一个项目有多个成员
- 一个项目包含多个任务
- 一个任务可以有多条评论
- 一个任务可以有多个标签

对应的 Prisma Schema：

```prisma
model User {
  id           String    @id @default(cuid())
  name         String
  email        String    @unique
  passwordHash String
  avatar       String?
  role         String    @default("MEMBER")
  createdAt    DateTime  @default(now())
  updatedAt    DateTime  @updatedAt

  memberships ProjectMember[]
  tasks       Task[]
  comments    Comment[]

  @@index([email])
}

model Project {
  id          String    @id @default(cuid())
  name        String
  description String?
  color       String    @default("#3b82f6")
  ownerId     String
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  owner      User             @relation("ProjectOwner", fields: [ownerId], references: [id])
  members    ProjectMember[]
  tasks      Task[]

  @@index([ownerId])
}

model ProjectMember {
  userId    String
  projectId String
  role      String @default("MEMBER")
  joinedAt  DateTime @default(now())

  user    User    @relation(fields: [userId], references: [id], onDelete: Cascade)
  project Project @relation(fields: [projectId], references: [id], onDelete: Cascade)

  @@id([userId, projectId])
}

model Task {
  id          String    @id @default(cuid())
  title       String
  description String?
  status      String    @default("TODO")
  priority    String    @default("medium")
  dueDate     DateTime?
  projectId   String
  assigneeId  String?
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  project    Project     @relation(fields: [projectId], references: [id], onDelete: Cascade)
  assignee   User?       @relation(fields: [assigneeId], references: [id])
  comments   Comment[]
  tags       TaskTag[]

  @@index([projectId])
  @@index([assigneeId])
  @@index([status])
}

model Comment {
  id        String   @id @default(cuid())
  content   String
  taskId    String
  authorId  String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  task   Task @relation(fields: [taskId], references: [id], onDelete: Cascade)
  author User @relation(fields: [authorId], references: [id])

  @@index([taskId])
  @@index([authorId])
}

model Tag {
  id   String @id @default(cuid())
  name String @unique

  tasks TaskTag[]
}

model TaskTag {
  taskId String
  tagId  String

  task Task @relation(fields: [taskId], references: [id], onDelete: Cascade)
  tag  Tag  @relation(fields: [tagId], references: [id], onDelete: Cascade)

  @@id([taskId, tagId])
}
```

几个设计决策：

**ProjectMember 是独立模型**而不是 Project 和 User 的多对多字段，因为我们需要存储角色信息（MEMBER/ADMIN）。如果只需要简单的关联，可以用 Prisma 的隐式多对多。

**Tag 和 Task 通过 TaskTag 关联**，因为标签是全局的（多个任务可以共享同一个标签），而且我们可能需要按标签筛选任务。

**索引**：对经常查询的字段（外键、status、email）加索引。这些索引在生产环境很重要，不加的话数据量一大查询会变慢。

## Prisma Client 单例

在开发环境，Next.js 的热重载会多次执行模块代码。如果每次都创建新的 PrismaClient，会打开过多的数据库连接。标准做法是用一个全局变量缓存实例：

```typescript
// lib/prisma.ts
import { PrismaClient } from "@prisma/client";

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient };

export const prisma =
  globalForPrisma.prisma ||
  new PrismaClient({
    log: process.env.NODE_ENV === "development" ? ["query"] : [],
  });

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}
```

## 在 tRPC 中使用 Prisma

Prisma 和 tRPC 的集成非常直接——你只需要把之前用内存数据的地方换成 Prisma 查询。

TaskFlow 的项目列表，带成员计数和任务计数：

```typescript
list: protectedProcedure.query(async () => {
  return await prisma.project.findMany({
    where: {
      members: { some: { userId: ctx.user.id } },
    },
    select: {
      id: true,
      name: true,
      description: true,
      color: true,
      owner: { select: { id: true, name: true, avatar: true } },
      _count: { select: { tasks: true, members: true } },
      updatedAt: true,
    },
    orderBy: { updatedAt: "desc" },
  });
}),
```

这里用了 `members: { some: { userId: ctx.user.id } }` 做过滤——只返回当前用户所属的项目。`select` 控制返回字段，`_count` 提供关联计数。

创建项目并自动加入创建者为成员：

```typescript
create: protectedProcedure
  .input(z.object({
    name: z.string().min(1).max(50),
    description: z.string().max(500).optional(),
    color: z.string().regex(/^#[0-9a-f]{6}$/).default("#3b82f6"),
  }))
  .mutation(async ({ input, ctx }) => {
    // 用事务确保"创建项目"和"加入成员"是原子操作
    return await prisma.$transaction(async (tx) => {
      const project = await tx.project.create({
        data: {
          ...input,
          ownerId: ctx.user.id,
          members: {
            create: {
              userId: ctx.user.id,
              role: "ADMIN",
            },
          },
        },
        include: {
          owner: { select: { id: true, name: true, avatar: true } },
          _count: { select: { tasks: true, members: true } },
        },
      });

      return project;
    });
  }),
```

注意这里用了 `$transaction`。创建项目和加入成员是两个数据库操作，如果第一个成功但第二个失败，数据就不一致了。事务保证了它们要么都成功，要么都回滚。

## 关联查询的取舍

Prisma 的 `include` 和 `select` 都能加载关联数据，但行为不同：

**include**：加载关联对象的完整字段。简单但可能返回不必要的数据。

```typescript
// 返回 project 的所有字段，加上 owner 的所有字段
const project = await prisma.project.findUnique({
  where: { id },
  include: { owner: true },
});
```

**select**：精确控制返回哪些字段。更推荐的方式，因为它同时解决了"数据安全"（不返回密码等敏感字段）和"传输效率"（不返回不需要的字段）两个问题。

```typescript
// 只返回指定的字段
const project = await prisma.project.findUnique({
  where: { id },
  select: {
    id: true,
    name: true,
    owner: { select: { id: true, name: true, avatar: true } },
    _count: { select: { tasks: true } },
  },
});
```

在 TaskFlow 里，我推荐始终用 `select`。原因很简单：**你不需要的数据，就不应该从数据库取出来**。这不仅减少了网络传输量，也避免了不小心把敏感字段（比如 User 的 passwordHash）暴露给前端。

一个常见的纠结是：`select` 写起来比 `include` 繁琐。确实，但这是值得的。而且你不需要在每个 procedure 里重复写——可以提取成复用的 select 配置：

```typescript
const projectSelect = {
  id: true,
  name: true,
  description: true,
  color: true,
  owner: { select: { id: true, name: true, avatar: true } },
  _count: { select: { tasks: true, members: true } },
  updatedAt: true,
} satisfies Prisma.ProjectSelect;

// 在多个 procedure 中复用
list: protectedProcedure.query(async () => {
  return await prisma.project.findMany({ select: projectSelect });
}),
byId: protectedProcedure.input(z.object({ id: z.string() })).query(async ({ input }) => {
  return await prisma.project.findUnique({ where: { id: input.id }, select: projectSelect });
}),
```

## 游标分页

TaskFlow 的任务列表需要分页。之前我们用的是偏移分页（page 1, 2, 3...），但在有频繁插入删除的场景下，偏移分页会导致数据重复或遗漏。游标分页更稳定。

```typescript
list: protectedProcedure
  .input(z.object({
    projectId: z.string(),
    status: z.enum(["TODO", "IN_PROGRESS", "DONE"]).optional(),
    priority: z.enum(["low", "medium", "high", "urgent"]).optional(),
    limit: z.number().min(1).max(50).default(20),
    cursor: z.string().optional(),
  }))
  .query(async ({ input }) => {
    const items = await prisma.task.findMany({
      where: {
        projectId: input.projectId,
        ...(input.status && { status: input.status }),
        ...(input.priority && { priority: input.priority }),
      },
      take: input.limit + 1,
      cursor: input.cursor ? { id: input.cursor } : undefined,
      orderBy: [
        { status: "asc" },
        { priority: "desc" },
        { createdAt: "desc" },
      ],
      select: {
        id: true,
        title: true,
        status: true,
        priority: true,
        dueDate: true,
        assignee: { select: { id: true, name: true, avatar: true } },
        tags: { include: { tag: { select: { id: true, name: true } } } },
        createdAt: true,
      },
    });

    let nextCursor: string | undefined;
    if (items.length > input.limit) {
      items.pop();
      nextCursor = items[items.length - 1].id;
    }

    return { items, nextCursor };
  }),
```

游标分页的原理：多取一条数据来判断"还有没有下一页"。`take: limit + 1`，如果返回了 21 条（limit 是 20），说明还有更多数据，最后一条的 id 就是下一页的游标。

排序规则 `[{ status: "asc" }, { priority: "desc" }, { createdAt: "desc" }]` 反映了任务管理的业务逻辑：先按状态分组（待办 → 进行中 → 已完成），再按优先级排序，最后按创建时间倒序。

## TaskFlow 的完整 tasks Router

把上面的知识串起来：

```typescript
export const tasksRouter = t.router({
  list: protectedProcedure
    .input(listTasksSchema)
    .query(async ({ input, ctx }) => {
      // 验证用户是否有权限查看该项目的任务
      const member = await prisma.projectMember.findUnique({
        where: {
          userId_projectId: { userId: ctx.user.id, projectId: input.projectId },
        },
      });
      if (!member) {
        throw new TRPCError({
          code: "FORBIDDEN",
          message: "你不是该项目的成员",
        });
      }
      // ...游标分页查询（如上）
    }),

  create: protectedProcedure
    .input(
      z.object({
        projectId: z.string(),
        title: z.string().min(1).max(200),
        description: z.string().max(2000).optional(),
        priority: z.enum(["low", "medium", "high", "urgent"]).default("medium"),
        assigneeId: z.string().optional(),
        dueDate: z.string().datetime().optional(),
        tagIds: z.array(z.string()).default([]),
      })
    )
    .mutation(async ({ input, ctx }) => {
      return await prisma.$transaction(async (tx) => {
        const task = await tx.task.create({
          data: {
            projectId: input.projectId,
            title: input.title,
            description: input.description,
            priority: input.priority,
            assigneeId: input.assigneeId,
            dueDate: input.dueDate ? new Date(input.dueDate) : null,
            tags: {
              create: input.tagIds.map((tagId) => ({ tagId })),
            },
          },
          select: { id: true },
        });

        // 创建任务时给项目成员发通知（简化）
        await tx.notification.create({
          data: {
            projectId: input.projectId,
            type: "TASK_CREATED",
            message: `${ctx.user.name} 创建了任务：${input.title}`,
            taskId: task.id,
          },
        });

        return task;
      });
    }),

  update: protectedProcedure
    .input(
      z.object({
        id: z.string(),
        title: z.string().min(1).max(200).optional(),
        status: z.enum(["TODO", "IN_PROGRESS", "DONE"]).optional(),
        priority: z.enum(["low", "medium", "high", "urgent"]).optional(),
        assigneeId: z.string().nullable().optional(),
      })
    )
    .mutation(async ({ input, ctx }) => {
      const { id, ...data } = input;
      const task = await prisma.task.update({
        where: { id },
        data,
        select: {
          id: true,
          title: true,
          status: true,
          project: { select: { id: true } },
          assignee: { select: { id: true, name: true, email: true } },
        },
      });
      return task;
    }),
});
```

Prisma + tRPC 的组合在类型层面是零摩擦的：Prisma 生成的类型（`Project`、`Task`、`User`）通过 tRPC 的 procedure 自动传递给前端，前端不需要定义任何 interface。你在后端改了数据库 schema、重新生成 Prisma 类型后，前端如果有用到变化了的字段，TypeScript 会直接报错。

到现在，TaskFlow 的后端已经完整了：有认证、有权限控制、有数据持久化、有类型安全。下一步是给它一个 Web 界面。
