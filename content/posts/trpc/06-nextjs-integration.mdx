TaskFlow 的后端已经完整了——API、认证、数据库都到位。现在需要给它一个 Web 界面。

我们用 Next.js App Router 来构建前端。App Router 的服务端组件（RSC）和 tRPC 的组合很有意思：你可以在服务端组件里直接调用 tRPC procedure，就像调一个本地函数一样，没有 HTTP 开销，没有序列化。

## 项目结构

Next.js + tRPC 的推荐目录结构：

```
src/
├── app/
│   ├── layout.tsx              # 根布局，包裹 TRPCProvider
│   ├── page.tsx                # 首页
│   ├── (auth)/
│   │   ├── login/page.tsx      # 登录页
│   │   └── register/page.tsx   # 注册页
│   └── (dashboard)/
│       ├── layout.tsx          # 仪表板布局
│       ├── page.tsx            # 项目列表
│       └── projects/
│           └── [id]/
│               ├── page.tsx    # 项目详情
│               └── tasks/
│                   └── page.tsx  # 任务列表
├── lib/
│   ├── trpc/
│   │   ├── server.ts           # tRPC 服务端配置 + Context
│   │   ├── router.ts           # appRouter 定义
│   │   ├── client.ts           # tRPC React 客户端
│   │   └── server-caller.ts    # Server Caller
│   └── prisma.ts               # Prisma Client 单例
├── components/
│   └── ui/
└── server/
    └── trpc/
        └── routers/
            ├── projects.ts
            ├── tasks.ts
            ├── users.ts
            └── auth.ts
```

关键点：tRPC 的 router 定义放在 `server/trpc/routers/` 里（服务端代码），客户端代码放在 `lib/trpc/` 里。Next.js 的 App Router 会自动区分服务端模块和客户端模块。

## tRPC 服务端配置

```typescript
// lib/trpc/server.ts
import { initTRPC, TRPCError } from "@trpc/server";
import { ZodError } from "zod";
import { prisma } from "@/lib/prisma";

interface Context {
  user: {
    id: string;
    name: string;
    email: string;
    role: string;
    avatar: string | null;
  } | null;
}

export const createContext = async (): Promise<Context> => {
  // 在实际项目中从 JWT/NextAuth session 中获取用户
  // 这里简化处理
  return { user: null };
};

const t = initTRPC.context<Context>().create({
  errorFormatter({ shape, error }) {
    return {
      ...shape,
      data: {
        ...shape.data,
        zodError:
          error.code === "BAD_REQUEST" && error.cause instanceof ZodError
            ? error.cause.flatten()
            : null,
      },
    };
  },
});

export const createRouter = t.router;
export const createMiddleware = t.middleware;
export const publicProcedure = t.procedure;
export const protectedProcedure = t.procedure.use(async ({ ctx, next }) => {
  if (!ctx.user) {
    throw new TRPCError({ code: "UNAUTHORIZED" });
  }
  return next({
    ctx: { ...ctx, user: ctx.user },
  });
});
```

## API Route Handler

在 App Router 中，tRPC 通过 Route Handler 暴露 HTTP 接口：

```typescript
// app/api/trpc/[trpc]/route.ts
import { fetchRequestHandler } from "@trpc/server/adapters/fetch";
import { appRouter } from "@/lib/trpc/router";
import { createContext } from "@/lib/trpc/server";

const handler = (req: Request) =>
  fetchRequestHandler({
    endpoint: "/api/trpc",
    req,
    router: appRouter,
    createContext,
  });

export { handler as GET, handler as POST };
```

就这么简单。所有 tRPC 的请求都走这一个路由处理器，tRPC 内部会根据 URL 路径分发到对应的 procedure。

## Server Caller：服务端组件的利器

App Router 的服务端组件是 tRPC 的最佳搭档。在服务端组件里，你可以用 **Server Caller** 直接调用 procedure，就像调一个本地函数：

```typescript
// lib/trpc/server-caller.ts
import { createCallerFactory } from "@trpc/server";
import { appRouter } from "./router";
import { createContext } from "./server";

export const createCaller = async () => {
  const context = await createContext();
  return createCallerFactory()(appRouter, context);
};
```

使用：

```typescript
// app/(dashboard)/page.tsx
import { createCaller } from "@/lib/trpc/server-caller";

export default async function DashboardPage() {
  // 直接调用 procedure，没有 HTTP 请求，没有序列化
  const caller = await createCaller();
  const projects = await caller.projects.list();

  return (
    <div>
      <h1>我的项目</h1>
      {projects.length === 0 ? (
        <p>还没有项目，创建一个吧</p>
      ) : (
        <ul>
          {projects.map((project) => (
            <li key={project.id}>
              <span style={{ color: project.color }}>{project.name}</span>
              <span>{project._count.tasks} 个任务</span>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

这里发生的事情很有意思：`caller.projects.list()` 是一个**直接的函数调用**，不走 HTTP。在服务端组件里，tRPC 和 Next.js 运行在同一个进程里，完全不需要网络通信。这意味着：

- 没有 HTTP 开销（延迟更低）
- 不需要序列化/反序列化
- Date、BigInt 等类型不会丢失
- 但你仍然获得了完整的类型安全

### 什么时候用 Server Caller，什么时候用客户端 Hooks

一个简单的判断标准：

- **展示数据** → 服务端组件 + Server Caller。页面首次加载就拿到数据，SEO 友好，用户体验好。
- **需要交互** → 客户端组件 + React Hooks。表单提交、实时更新、乐观更新等场景需要客户端 Hooks 的能力（缓存、自动重试、loading 状态）。

一个页面通常是两者的混合：服务端组件负责初始数据加载和 SEO，客户端组件负责交互部分。

## 客户端组件：交互场景

需要用户交互的部分用客户端组件。TaskFlow 的创建项目表单：

```typescript
// components/create-project-button.tsx
"use client";

import { trpc } from "@/lib/trpc/client";
import { useState } from "react";
import { useRouter } from "next/navigation";

export function CreateProjectButton() {
  const [isOpen, setIsOpen] = useState(false);
  const router = useRouter();
  const utils = trpc.useContext();

  const createProject = trpc.projects.create.useMutation({
    onSuccess: (data) => {
      setIsOpen(false);
      utils.projects.list.invalidate();
      router.push(`/projects/${data.id}`);
    },
    onError: (error) => {
      if (error.code === "UNAUTHORIZED") {
        router.push("/login");
      }
    },
  });

  return (
    <>
      <button onClick={() => setIsOpen(true)}>创建项目</button>
      {isOpen && (
        <form
          onSubmit={(e) => {
            e.preventDefault();
            const formData = new FormData(e.currentTarget);
            createProject.mutate({
              name: formData.get("name") as string,
              description: formData.get("description") as string,
            });
          }}
        >
          <input name="name" placeholder="项目名称" required />
          <textarea name="description" placeholder="项目描述" />
          <button type="button" onClick={() => setIsOpen(false)}>取消</button>
          <button type="submit" disabled={createProject.isPending}>
            {createProject.isPending ? "创建中..." : "创建"}
          </button>
        </form>
      )}
    </>
  );
}
```

注意 `utils.projects.list.invalidate()`——创建项目成功后，让之前缓存的项目列表失效。React Query 会自动重新获取数据，列表会立即更新。

## 认证集成

TaskFlow 用 NextAuth.js 处理认证。NextAuth 的 session 信息需要传递给 tRPC 的 Context：

```typescript
// lib/trpc/server.ts
import { auth } from "@/auth"; // NextAuth 的 auth()

export const createContext = async (): Promise<Context> => {
  const session = await auth();

  return {
    user: session?.user
      ? {
          id: session.user.id,
          name: session.user.name,
          email: session.user.email,
          role: session.user.role,
          avatar: session.user.image,
        }
      : null,
  };
};
```

Server Caller 使用时，认证也是自动处理的：

```typescript
export default async function ProjectPage({ params }: { params: { id: string } }) {
  const caller = await createCaller();

  try {
    const project = await caller.projects.byId({ id: params.id });
    return <ProjectDetail project={project} />;
  } catch (error) {
    if (error instanceof TRPCError && error.code === "NOT_FOUND") {
      notFound();
    }
    throw error;
  }
}
```

在服务端组件里保护页面也很简单：

```typescript
// app/(dashboard)/layout.tsx
import { auth } from "@/auth";
import { redirect } from "next/navigation";

export default async function DashboardLayout({ children }) {
  const session = await auth();
  if (!session) {
    redirect("/login");
  }

  return <div>{children}</div>;
}
```

## 服务端组件 + 客户端组件的混合模式

一个真实页面通常是两者的混合。TaskFlow 的项目详情页：

```typescript
// app/(dashboard)/projects/[id]/page.tsx — 服务端组件
export default async function ProjectPage({ params }) {
  const caller = await createCaller();
  const project = await caller.projects.byId({ id: params.id });

  return (
    <div>
      <h1>{project.name}</h1>
      <p>{project.description}</p>

      {/* 任务列表需要交互（拖拽、筛选），用客户端组件 */}
      <TaskBoard projectId={project.id} initialTasks={project.tasks} />

      {/* 成员列表 */}
      <MemberList members={project.members} />
    </div>
  );
}
```

```typescript
// components/task-board.tsx — 客户端组件
"use client";

import { trpc } from "@/lib/trpc/client";

export function TaskBoard({ projectId }: { projectId: string }) {
  const [status, setStatus] = useState<string>("ALL");

  const { data, isLoading } = trpc.tasks.list.useQuery({
    projectId,
    status: status === "ALL" ? undefined : status,
    limit: 50,
  });

  return (
    <div>
      <div>
        {["ALL", "TODO", "IN_PROGRESS", "DONE"].map((s) => (
          <button key={s} onClick={() => setStatus(s)}>
            {s === "ALL" ? "全部" : s}
          </button>
        ))}
      </div>

      {isLoading ? (
        <div>加载中...</div>
      ) : (
        <ul>
          {data?.items.map((task) => (
            <li key={task.id}>
              <TaskCard task={task} />
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

这个模式的核心思路是：**服务端组件负责数据加载（首屏快、SEO 好），客户端组件负责交互（筛选、拖拽、乐观更新）。** 两者通过 props 传递数据，各自做各自擅长的事。

## 其他框架的简要对比

如果你用的是 Nuxt 而不是 Next.js，核心思路完全一样，只是 API 调用方式不同：

| 场景       | Next.js                  | Nuxt                                                    |
| ---------- | ------------------------ | ------------------------------------------------------- |
| 服务端调用 | Server Caller            | `$trpc.xxx.query()` (在 `useAsyncData` 中)              |
| 客户端查询 | `trpc.xxx.useQuery()`    | `useAsyncData(() => $trpc.xxx.query())`                 |
| 客户端修改 | `trpc.xxx.useMutation()` | `useMutation({ mutationFn: () => $trpc.xxx.mutate() })` |
| 缓存失效   | `utils.xxx.invalidate()` | `refresh()`                                             |

如果你用的是 Hono 做边缘部署，tRPC 也有官方适配器 `@hono/trpc-server`，集成方式和 Next.js 类似，区别在于部署目标（Cloudflare Workers 等）。

框架不同，但 tRPC 的核心——Router、Procedure、类型推断——在所有框架里都是一样的。学会了一个框架的集成方式，切换到其他框架成本很低。
