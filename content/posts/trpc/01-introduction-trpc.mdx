## 前言

你是否经历过这样的场景：

**早上 9:00**：后端小王在 TypeScript 中定义了一个用户接口：

```typescript
interface User {
  id: string;
  name: string;
  email: string;
}
```

**早上 9:05**：前端小李在前端代码中又定义了一遍：

```typescript
interface User {
  id: string;
  name: string;
  email: string;
}
```

**下午 2:00**：产品经理说需要添加 `avatar` 字段。

**下午 2:05**：小王更新了后端接口。

**下午 3:00**：小李更新了前端接口... 但忘了更新一个地方。

**晚上 8:00**：线上崩溃了，因为类型不匹配。

这不是虚构的故事，而是每天都在发生的真实场景。今天，我们要介绍一个工具，能彻底解决这个问题 —— **tRPC**。

## 传统 API 开发的痛点

### 1. 类型定义重复

在传统的 REST API 开发中，我们通常需要：

**后端定义**：

```typescript
// backend/src/types/user.ts
export interface User {
  id: string;
  name: string;
  email: string;
  createdAt: Date;
}

// backend/src/routes/users.ts
app.get("/users/:id", async (req, res) => {
  const user = await db.users.findUnique({
    where: { id: req.params.id },
  });
  res.json(user);
});
```

**前端定义**（再次定义！）：

```typescript
// frontend/src/types/user.ts
export interface User {
  id: string;
  name: string;
  email: string;
  createdAt: string; // 注意：Date 变成了 string
}

// frontend/src/api/users.ts
export async function getUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}
```

### 2. API 文档维护成本高

- Swagger/OpenAPI 文档需要手动编写和维护
- 文档与代码容易不同步
- 前端开发者需要等待文档更新才能开始开发

### 3. 运行时错误

```typescript
// 前端期望返回 User[]
const users = await fetch("/api/users").then((res) => res.json());

// 但后端实际返回了 { data: User[], total: number }
// TypeScript 无法在编译时发现这个问题！
```

### 4. 开发效率低

```typescript
// 每次调用 API 都要查看文档
const response = await fetch("/api/users?page=1&limit=10");

// 手动拼接 URL 参数
const url = new URL("/api/users", window.location.origin);
url.searchParams.set("page", "1");
url.searchParams.set("limit", "10");
const response = await fetch(url);

// 手动处理类型转换
const users = await response.json();
// users 的类型是 any，需要手动断言
```

## 什么是 tRPC？

**tRPC** (TypeScript Remote Procedure Call) 是一个让你在 TypeScript 全栈应用中实现**端到端类型安全**的工具，无需任何代码生成或 Schema 定义。

### 核心价值主张

> "Write your server with your favorite backend framework, then call your backend functions from the frontend in a type-safe manner without code generation or manual type declarations."

### 工作原理

```mermaid
graph LR
    A[后端定义 Router] -->|TypeScript 类型| B[tRPC Server]
    B -->|类型推断| C[tRPC Client]
    C -->|类型安全调用| D[前端代码]

    style A fill:#e1f5ff
    style B fill:#c8e6ff
    style C fill:#c8e6ff
    style D fill:#e1f5ff
```

**关键点**：

1. 后端定义一次类型
2. 前端自动获得类型推断
3. 无需 API 文档
4. 编译时类型检查

## tRPC vs 其他方案

### tRPC vs REST

| 特性       | REST                | tRPC              |
| ---------- | ------------------- | ----------------- |
| 类型安全   | ❌ 需要手动维护     | ✅ 自动推断       |
| API 文档   | ⚠️ Swagger/OpenAPI  | ✅ 代码即文档     |
| 开发体验   | ⚠️ 需要手动处理类型 | ✅ 完整的自动补全 |
| 传输协议   | HTTP                | HTTP/WebSocket    |
| 客户端限制 | 无                  | 仅 TypeScript     |

### tRPC vs GraphQL

| 特性     | GraphQL             | tRPC             |
| -------- | ------------------- | ---------------- |
| Schema   | ✅ 但需要定义和维护 | ✅ 代码即 Schema |
| 类型安全 | ✅ 但需要工具链     | ✅ 原生支持      |
| 学习曲线 | ⚠️ 较陡             | ✅ 平缓          |
| 过度获取 | ✅ 可控             | ⚠️ 返回整个对象  |
| 生态系统 | ✅ 成熟             | 🔄 快速发展中    |

### tRPC vs gRPC

| 特性       | gRPC             | tRPC                 |
| ---------- | ---------------- | -------------------- |
| 协议       | HTTP/2, Protobuf | HTTP/WebSocket, JSON |
| 类型安全   | ✅ 需要代码生成  | ✅ 无需代码生成      |
| 语言支持   | ✅ 多语言        | ⚠️ 仅 TypeScript     |
| 浏览器支持 | ⚠️ 需要工具      | ✅ 原生支持          |
| 性能       | ✅ 二进制协议    | ⚠️ JSON 协议         |

## 快速开始

### 第一步：初始化项目

```bash
# 创建项目目录
mkdir my-trpc-app
cd my-trpc-app

# 初始化 package.json
npm init -y

# 安装依赖
npm install @trpc/server@next zod
npm install -D typescript @types/node tsx
```

### 第二步：配置 TypeScript

创建 `tsconfig.json`：

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

### 第三步：创建服务端

创建 `server.ts`：

```typescript
import { initTRPC } from "@trpc/server";
import { z } from "zod";

// 1. 初始化 tRPC
const t = initTRPC.create();

// 2. 定义 Router
export const appRouter = t.router({
  // 简单的查询
  greeting: t.procedure
    .input(z.object({ name: z.string() }))
    .query(({ input }) => {
      return {
        message: `Hello, ${input.name}!`,
        timestamp: new Date().toISOString(),
      };
    }),

  // 获取用户列表
  getUsers: t.procedure.query(() => {
    return [
      { id: "1", name: "Alice", email: "alice@example.com" },
      { id: "2", name: "Bob", email: "bob@example.com" },
    ];
  }),

  // 根据ID获取用户
  getUserById: t.procedure
    .input(z.object({ id: z.string() }))
    .query(({ input }) => {
      const users = [
        { id: "1", name: "Alice", email: "alice@example.com" },
        { id: "2", name: "Bob", email: "bob@example.com" },
      ];
      return users.find((user) => user.id === input.id);
    }),
});

// 3. 导出类型
export type AppRouter = typeof appRouter;

// 4. 启动简单 HTTP 服务器
import { createHTTPServer } from "@trpc/server/adapters/standalone";

createHTTPServer({
  router: appRouter,
  createContext: () => ({}),
}).listen(3000, () => {
  console.log("🚀 Server running on http://localhost:3000");
});
```

### 第四步：创建客户端

创建 `client.ts`：

```typescript
import { createTRPCProxyClient, httpBatchLink } from "@trpc/client";
import type { AppRouter } from "./server";

// 1. 创建客户端
const client = createTRPCProxyClient<AppRouter>({
  links: [
    httpBatchLink({
      url: "http://localhost:3000",
    }),
  ],
});

// 2. 调用 API（完全类型安全！）
async function main() {
  // ✅ 完整的类型推断和自动补全
  const greeting = await client.greeting.query({ name: "World" });
  console.log(greeting); // { message: "Hello, World!", timestamp: "..." }

  // ✅ TypeScript 会自动推断返回类型
  const users = await client.getUsers.query();
  console.log(users); // Array<{ id: string, name: string, email: string }>

  // ✅ 输入参数也会被验证
  const user = await client.getUserById.query({ id: "1" });
  console.log(user); // { id: "1", name: "Alice", email: "alice@example.com" }

  // ❌ 编译时错误：Type 'number' is not assignable to type 'string'
  // const error = await client.getUserById.query({ id: 1 });

  // ❌ 编译时错误：Property 'doesNotExist' does not exist on type 'AppRouter'
  // const error = await client.doesNotExist.query();
}

main();
```

### 运行示例

```bash
# 终端 1：启动服务端
tsx server.ts

# 终端 2：运行客户端
tsx client.ts
```

## tRPC 的核心概念

### 1. Router（路由器）

Router 是 tRPC API 的组织单位，类似于 Express 的 Router 或 NestJS 的 Module。

```typescript
const appRouter = t.router({
  users: t.router({
    list: t.procedure.query(() => users),
    byId: t.procedure
      .input(z.object({ id: z.string() }))
      .query(({ input }) => getUser(input.id)),
  }),
  posts: t.router({
    list: t.procedure.query(() => posts),
    create: t.procedure
      .input(
        z.object({
          title: z.string(),
          content: z.string(),
        })
      )
      .mutation(({ input }) => createPost(input)),
  }),
});
```

### 2. Procedure（程序）

Procedure 是 tRPC API 的最小单位，类似于函数：

```typescript
// Query：查询数据（GET）
const getUsers = t.procedure.query(() => {
  return db.users.findMany();
});

// Mutation：修改数据（POST/PUT/DELETE）
const createUser = t.procedure
  .input(
    z.object({
      name: z.string(),
      email: z.string(),
    })
  )
  .mutation(({ input }) => {
    return db.users.create({ data: input });
  });
```

### 3. Context（上下文）

Context 用于在所有 procedure 之间共享数据：

```typescript
const createContext = async () => {
  return {
    user: await getCurrentUser(),
    db: await createDatabaseConnection(),
  };
};

const t = initTRPC
  .context<Awaited<ReturnType<typeof createContext>>>()
  .create();

// 在 procedure 中使用 context
const getUserProfile = t.procedure.query(({ ctx }) => {
  return ctx.user; // ✅ 类型安全
});
```

## 类型推断的魔法

让我们看看 tRPC 如何实现类型推断：

```typescript
// 1. 后端定义
export const appRouter = t.router({
  getUserById: t.procedure
    .input(z.object({ id: z.string() }))
    .query(({ input }) => {
      return {
        id: input.id,
        name: 'Alice',
        email: 'alice@example.com',
        createdAt: new Date()
      };
    })
});

export type AppRouter = typeof appRouter;

// 2. 前端使用（类型自动推断！）
const client = createTRPCProxyClient<AppRouter>({...});

// ✅ input 参数自动推断
await client.getUserById.query({ id: '123' });
// ❌ 编译错误: Property 'doesNotExist' does not exist
await client.getUserById.query({ doesNotExist: '123' });

// ✅ 返回值自动推断为：
// {
//   id: string;
//   name: string;
//   email: string;
//   createdAt: Date;
// }
const user = await client.getUserById.query({ id: '123' });
console.log(user.name); // ✅ TypeScript 知道 user 有 name 属性
console.log(user.age); // ❌ 编译错误: Property 'age' does not exist
```

## 最佳实践

### 1. 使用 Zod 进行输入验证

```typescript
import { z } from "zod";

const createUserSchema = z.object({
  name: z.string().min(2).max(50),
  email: z.string().email(),
  age: z.number().min(18).optional(),
});

const createUser = t.procedure.input(createUserSchema).mutation(({ input }) => {
  // ✅ input 已经被验证，类型安全
  return db.users.create({ data: input });
});
```

### 2. 合理组织 Router

```typescript
// app/router/index.ts
export const appRouter = t.router({
  users: usersRouter,
  posts: postsRouter,
  comments: commentsRouter
});

// app/router/users.ts
export const usersRouter = t.router({
  list: t.procedure.query(() => {...}),
  byId: t.procedure.query(() => {...}),
  create: t.procedure.mutation(() => {...})
});
```

### 3. 使用中间件处理横切关注点

```typescript
const isAuthed = t.middleware(({ ctx, next }) => {
  if (!ctx.user) {
    throw new TRPCError({ code: 'UNAUTHORIZED' });
  }
  return next();
});

const protectedProcedure = t.procedure.use(isAuthed);

// 使用
const deleteUser = protectedProcedure.mutation(() => {...});
```

## 什么时候使用 tRPC？

### ✅ 适合使用 tRPC 的场景

1. **TypeScript 全栈项目**：前后端都使用 TypeScript
2. **快速迭代**：需要频繁修改 API
3. **小型到中型团队**：团队规模 < 50人
4. **Web 应用**：单页应用、服务端渲染应用
5. **实时应用**：需要 WebSocket 支持

### ❌ 不适合使用 tRPC 的场景

1. **多语言后端**：后端不是 TypeScript
2. **公开 API**：需要提供给第三方使用
3. **微服务架构**：服务间使用不同语言
4. **超大规模团队**：需要严格的 API 版本管理

### ⚠️ 需要权衡的场景

1. **移动应用**：可以使用但需要考虑网络优化
2. **性能敏感**：JSON 序列化可能不如 Protobuf
3. **复杂查询**：GraphQL 在复杂查询场景更灵活

## 总结

tRPC 通过 TypeScript 的类型系统实现了前后端的**端到端类型安全**，让我们：

1. ✅ **告别类型定义重复**：定义一次，到处使用
2. ✅ **告别 API 文档**：代码即文档
3. ✅ **提升开发效率**：完整的自动补全和类型检查
4. ✅ **减少运行时错误**：编译时发现问题
5. ✅ **简化开发流程**：无需代码生成，开箱即用
