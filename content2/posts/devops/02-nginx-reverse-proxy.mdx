上一篇我们把 FlowAPI 容器跑起来了，`curl localhost:3000/health` 返回正常。现在有个尴尬的问题：用户在浏览器里输入 `example.com`，请求到达服务器，但没有任何东西在监听 80 端口，连接直接被拒绝。

你有几个选择：

- 让用户访问 `example.com:3000`——不现实，而且没有 HTTPS，密码和 token 全是明文传输
- 让 FlowAPI 直接监听 80 端口——那 HTTPS 怎么办？你不可能在每个应用里都实现 TLS
- 在前面放一个反向代理，统一处理 80/443 端口、HTTPS 证书、静态资源、安全头

第三种就是 Nginx 要干的事。

## 反向代理是什么

先区分两个概念。"正向代理"是代理客户端的，比如你用 VPN 访问 Google，VPN 服务器替你发请求。"反向代理"是代理服务端的，用户不直接访问后端应用，而是访问 Nginx，Nginx 再把请求转发给后端。

反向代理解决了几个问题：

**统一入口**。用户只需要访问 80 或 443 端口，不用关心后端应用跑在哪个端口。以后加新服务，只需要在 Nginx 里加一条路由规则。

**HTTPS 卸载**。TLS 握手和证书管理交给 Nginx，后端应用只处理 HTTP。不用在每个服务里单独配置证书。

**安全隔离**。后端应用不直接暴露在公网，Nginx 可以过滤恶意请求、设置安全头、限制请求频率。

**静态资源服务**。Nginx 处理静态文件（图片、CSS、JS）的效率远高于 Node.js 或 Java 应用。把静态资源交给 Nginx，应用只处理 API 逻辑。

**缓存和压缩**。Nginx 可以缓存后端响应、启用 Gzip 压缩，减少后端压力和传输体积。

## 安装 Nginx

```bash
# 安装 Nginx
sudo apt install nginx -y

# 启动并设置开机自启
sudo systemctl enable nginx
sudo systemctl start nginx

# 验证
sudo systemctl status nginx
```

安装完成后，访问 `http://你的服务器IP` 应该能看到 Nginx 的默认欢迎页面。能看到说明 Nginx 已经在 80 端口正常监听了。

看一下 Nginx 的目录结构，后面会频繁用到：

```
/etc/nginx/
├── nginx.conf          # 主配置文件
├── sites-available/    # 可用的站点配置（存放配置文件）
├── sites-enabled/      # 已启用的站点配置（软链接到 sites-available）
├── snippets/           # 可复用的配置片段
└── conf.d/             # 额外配置目录
```

`sites-available` 存放所有站点的配置文件，`sites-enabled` 存放软链接，只有链接到 `sites-enabled` 的配置才会生效。这是 Debian/Ubuntu 的约定，方便你临时禁用某个站点——删掉软链接就行，不用删配置文件。

## 基本反向代理配置

先写一个最简单的配置，把请求转发给 FlowAPI：

```nginx
# /etc/nginx/sites-available/flowapi
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;

        # 把原始请求的信息传递给后端
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

逐行解释一下关键部分：

`listen 80` 监听 HTTP 的标准端口。`server_name` 指定这个配置块响应哪个域名的请求。如果你的服务器只跑一个应用，也可以用 `_` 作为通配符，表示匹配所有域名。

`proxy_pass` 是核心指令，告诉 Nginx 把请求转发到 `http://127.0.0.1:3000`。注意这里用的是 `127.0.0.1` 而不是 `localhost`——`localhost` 在某些系统上可能解析到 IPv6 地址 `[::1]`，导致连接失败。

四个 `proxy_set_header` 是必须的：

- `Host`：把原始请求的域名传给后端。没有这个头，后端应用可能拿不到正确的域名，重定向或生成 URL 时会出问题。
- `X-Real-IP`：把客户端真实 IP 传给后端。没有这个头，后端日志里所有请求的 IP 都是 `127.0.0.1`。
- `X-Forwarded-For`：记录完整的代理链。如果有多层代理（CDN -> Nginx -> 后端），这个头会把所有 IP 串起来。
- `X-Forwarded-Proto`：告诉后端原始请求用的协议是 HTTP 还是 HTTPS。后端生成重定向 URL 时需要知道这个。

创建软链接并启用配置：

```bash
# 创建软链接
sudo ln -s /etc/nginx/sites-available/flowapi /etc/nginx/sites-enabled/

# 测试配置语法是否正确
sudo nginx -t

# 重新加载配置（不中断服务）
sudo systemctl reload nginx
```

`nginx -t` 会在不启动 Nginx 的情况下检查配置文件语法。这个命令值得养成习惯——每次改完配置都跑一次，确认没问题再 reload。`systemctl reload` 会平滑重载配置，不会断开现有连接。

现在访问 `http://api.example.com`，请求会被 Nginx 转发到 FlowAPI 的 3000 端口。但注意，这里用的是 HTTP，所有数据都是明文传输。

## HTTPS：Let's Encrypt + Certbot

HTTP 明文传输的问题很严重——用户的登录密码、API token、请求数据全部裸奔。浏览器现在会给 HTTP 页面标红提示"不安全"。HTTPS 是标配，不是可选项。

Let's Encrypt 是一个免费的证书颁发机构，Certbot 是官方推荐的证书管理工具，可以自动申请和续签证书。

```bash
# 安装 Certbot 和 Nginx 插件
sudo apt install certbot python3-certbot-nginx -y
```

申请证书之前，确保你的域名已经解析到服务器 IP。检查一下：

```bash
dig api.example.com +short
# 应该返回你的服务器公网 IP
```

如果 DNS 还没生效，Certbot 会申请失败。DNS 生效通常需要几分钟到几小时。

用 Certbot 自动申请证书并修改 Nginx 配置：

```bash
sudo certbot --nginx -d api.example.com
```

Certbot 会问你几个问题：

```
Enter email address (used for urgent renewal and security notices):
# 输入你的邮箱，用于证书到期提醒

Please read the Terms of Service at https://letsencrypt.org/documents/LE-SA-v1.4-April-2014.pdf.
You must agree in order to register with the ACME server.
# 输入 Y 同意服务条款

Would you be willing to share your email address with the EFF?
# 输入 N 或 Y，这是 EFF（电子前哨基金会）的推广邮件

Please choose whether or not to redirect HTTP to HTTPS.
# 选 2：Redirect - 所有 HTTP 请求自动跳转到 HTTPS
```

Certbot 会自动修改你的 Nginx 配置，加上证书路径和 HTTP 到 HTTPS 的重定向。改完之后你的配置大概会变成这样：

```nginx
# /etc/nginx/sites-available/flowapi（Certbot 修改后的版本）
server {
    listen 80;
    server_name api.example.com;

    # Certbot 自动添加：HTTP 跳转到 HTTPS
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name api.example.com;

    # 证书路径（Certbot 自动生成）
    ssl_certificate /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # SSL 优化配置
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Let's Encrypt 的证书有效期 90 天。Certbot 安装时会自动注册一个 systemd timer 或 cron job 来自动续签：

```bash
# 检查自动续签是否配置成功
sudo certbot renew --dry-run

# 查看 systemd timer 状态
sudo systemctl status certbot.timer
```

`--dry-run` 会模拟续签过程但不实际续签。如果这个命令成功了，说明自动续签配置没问题。

## 安全头

HTTPS 解决了传输加密的问题，但浏览器安全不止这些。Nginx 可以通过响应头告诉浏览器一些安全策略：

```nginx
# 在 server 块中添加安全头
server {
    listen 443 ssl;
    server_name api.example.com;

    # ... SSL 配置 ...

    # 防止 MIME 类型嗅探
    add_header X-Content-Type-Options "nosniff" always;

    # 防止点击劫持：禁止页面被嵌入 iframe
    add_header X-Frame-Options "DENY" always;

    # 启用浏览器 XSS 过滤
    add_header X-XSS-Protection "1; mode=block" always;

    # 控制 Referer 信息泄露
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # 内容安全策略（根据你的应用调整）
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'" always;

    # 严格传输安全：告诉浏览器接下来 1 年内只用 HTTPS 访问
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    location / {
        proxy_pass http://127.0.0.1:3000;
        # ... proxy_set_header ...
    }
}
```

每个头的作用：

**X-Content-Type-Options: nosniff**。浏览器默认会猜测资源的 MIME 类型。攻击者可能上传一个 `.jpg` 文件，实际上是 JavaScript 代码，浏览器如果猜测它是 JavaScript 就会执行。`nosniff` 告诉浏览器严格按服务器声明的 MIME 类型处理，不许猜。

**X-Frame-Options: DENY**。防止你的页面被其他网站嵌入 `<iframe>`。攻击者可以把你的登录页面嵌在自己的页面上，用户以为在输入你的密码，实际上在给攻击者发数据。`DENY` 完全禁止嵌入，如果你需要嵌入自己的页面，可以用 `SAMEORIGIN`。

**X-XSS-Protection: 1; mode=block**。启用浏览器内置的 XSS 过滤器。现代浏览器已经有更好的防护机制，但加上这个头作为兼容性保险没有坏处。

**Referrer-Policy: strict-origin-when-cross-origin**。控制从你的页面跳转到其他网站时，`Referer` 头携带多少信息。`strict-origin-when-cross-origin` 表示同源跳转带完整 URL，跨源跳转只带域名。避免敏感的 URL 路径泄露给第三方。

**Content-Security-Policy (CSP)**。这是最强大的安全头，用来控制页面能加载哪些资源。上面给的是最基础的配置——只允许从同源加载脚本和样式。如果你的前端用了 CDN、内联脚本、第三方字体，CSP 策略需要相应调整。CSP 策略太严格会搞坏页面，太宽松又没意义，需要根据实际情况调试。部署初期可以用 `Content-Security-Policy-Report-Only` 先观察报告，再切换到正式策略。

**Strict-Transport-Security (HSTS)**。告诉浏览器"这个域名只能用 HTTPS 访问，HTTP 直接拒绝"。`max-age=31536000` 是 1 年。`includeSubDomains` 表示所有子域名也适用。**注意：一旦设置了这个头，在 max-age 期间内你无法"取消"HTTPS**。如果证书出了问题，用户在 max-age 期间内完全无法访问你的网站。建议先设置一个较短的时间（比如 300 秒）测试，确认没问题后再改长。

所有 `add_header` 后面的 `always` 参数确保在所有响应（包括错误页面）中都添加这些头。

## Gzip 压缩

API 返回的 JSON 数据通常是可压缩的。开启 Gzip 可以把响应体积减少 60%-80%，对于带宽有限的服务器效果明显。

```nginx
# 在 http 块中添加（通常在 /etc/nginx/nginx.conf 中）
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 4;
gzip_min_length 1024;
gzip_types
    text/plain
    text/css
    text/xml
    text/javascript
    application/json
    application/javascript
    application/xml
    application/xml+rss
    image/svg+xml;
```

几个参数的含义：

- `gzip_comp_level 4`：压缩级别，1-9，越高压缩率越大但 CPU 消耗也越大。4 是个不错的平衡点。API 响应一般不大，没必要开到最高。
- `gzip_min_length 1024`：小于 1KB 的文件不压缩。压缩小文件反而可能让体积变大（gzip 头部开销），得不偿失。
- `gzip_types`：指定哪些 MIME 类型需要压缩。默认只压缩 text/html，其他类型要显式声明。`application/json` 对 API 来说是必须加的。

改完之后 reload Nginx，用 curl 验证压缩是否生效：

```bash
curl -H "Accept-Encoding: gzip" -I https://api.example.com/health
# 看响应头里有没有 Content-Encoding: gzip
```

## 完整配置文件

把前面的所有配置整合到一起。这是 FlowAPI 的完整 Nginx 配置：

```nginx
# /etc/nginx/sites-available/flowapi

# HTTP -> HTTPS 重定向
server {
    listen 80;
    listen [::]:80;
    server_name api.example.com;

    # Certbot 验证路径（用于证书续签）
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # 其他所有请求跳转到 HTTPS
    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS 主配置
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name api.example.com;

    # SSL 证书
    ssl_certificate /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # SSL 协议和加密套件
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;

    # SSL 会话缓存（减少重复握手开销）
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # 安全头
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # 日志
    access_log /var/log/nginx/flowapi_access.log;
    error_log /var/log/nginx/flowapi_error.log;

    # 反向代理
    location / {
        proxy_pass http://127.0.0.1:3000;

        # 传递客户端信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 代理超时设置
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;

        # WebSocket 支持（如果需要）
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # 健康检查不走日志（避免刷日志）
    location /health {
        proxy_pass http://127.0.0.1:3000;
        access_log off;
    }
}
```

Gzip 配置通常放在主配置文件 `/etc/nginx/nginx.conf` 的 `http` 块里，对所有站点生效。打开 `/etc/nginx/nginx.conf`，在 `http { }` 块中找到或添加：

```nginx
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 4;
gzip_min_length 1024;
gzip_types
    text/plain
    text/css
    text/xml
    text/javascript
    application/json
    application/javascript
    application/xml
    application/xml+rss
    image/svg+xml;
```

改完之后：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

## 多应用部署

你的服务器以后可能不止跑 FlowAPI 一个服务。Nginx 天然支持多站点配置——每个服务一个配置文件，通过 `server_name` 区分域名。

假设你要加一个前端应用 `app.example.com`：

```nginx
# /etc/nginx/sites-available/frontend
server {
    listen 80;
    listen [::]:80;
    server_name app.example.com;

    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name app.example.com;

    ssl_certificate /etc/letsencrypt/live/app.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/app.example.com/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;

    # 前端静态文件直接由 Nginx 服务，不经过后端
    root /var/www/frontend/dist;
    index index.html;

    # SPA 路由：所有未匹配的路径返回 index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 静态资源缓存（JS/CSS/图片）
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

前端应用和后端 API 用不同的 `server_name`，Nginx 根据请求的域名分发到对应的配置块。每个站点独立管理自己的证书、安全策略、缓存规则。

启用新站点并为它申请证书：

```bash
# 启用站点配置
sudo ln -s /etc/nginx/sites-available/frontend /etc/nginx/sites-enabled/

# 申请证书
sudo certbot --nginx -d app.example.com
```

如果你的服务器只有一个 IP 但有多个域名，这种配置方式完全没问题。Nginx 通过 `Host` 头区分不同域名，不需要额外绑定 IP。

如果多个服务用同一个域名的不同路径（比如 `example.com/api` 走 FlowAPI，`example.com/` 走前端），也可以用 `location` 块区分。但域名分离更干净，以后迁移或拆分更方便。

## 日志和基础调优

默认的 Nginx 日志格式够用，但做一些小调整能让排查问题更高效。

### 日志格式

默认的 access_log 格式比较简单，可以加一些有用的字段。在 `/etc/nginx/nginx.conf` 的 `http` 块中自定义日志格式：

```nginx
# 自定义日志格式（在 http 块中添加）
log_format detailed '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    'rt=$request_time';

# 使用自定义格式
# access_log /var/log/nginx/flowapi_access.log detailed;
```

加了 `rt=$request_time` 字段，可以记录每个请求的处理时间。排查慢请求时特别有用。

### worker 进程

```nginx
# 在 /etc/nginx/nginx.conf 的最外层（不在 http 块中）
worker_processes auto;
```

`worker_processes auto` 让 Nginx 根据服务器 CPU 核心数自动创建工作进程。默认值是 1，在多核服务器上会浪费性能。

### 连接数

```nginx
# 在 events 块中
events {
    worker_connections 1024;
}
```

每个 worker 进程能处理的最大并发连接数。对于 2 核服务器（2 个 worker），理论最大并发是 2048。这个值对大多数小应用来说绑绑有余。

### proxy buffer

默认的 proxy buffer 配置对 API 来说通常够用。如果你发现后端返回大响应时 Nginx 报错，可能需要调大：

```nginx
# 在 location 或 server 块中
proxy_buffer_size 16k;
proxy_buffers 4 16k;
```

## 容易踩的坑

Nginx 配置看起来简单，但有几个地方特别容易出错。

**proxy_pass 的斜杠问题**。`proxy_pass http://127.0.0.1:3000` 和 `proxy_pass http://127.0.0.1:3000/` 的行为不一样。有尾部斜杠时，Nginx 会把 `location` 匹配的路径替换掉；没有尾部斜杠时，完整路径会原样转发。比如配置了 `location /api/`，请求 `/api/users`，有尾部斜杠会转发 `/users`，没有会转发 `/api/users`。这个行为经常让人困惑，建议保持一致：要么 location 和 proxy_pass 都带斜杠，都不带。

**add_header 在 location 块中会覆盖父级**。如果在 `server` 块中设置了安全头，又在某个 `location` 块中设置了 `add_header`，那 `server` 级别的头在该 `location` 中会失效。这是 Nginx 的继承机制导致的。解决办法是把安全头放到一个单独文件中，用 `include` 在每个需要的块中引入：

```nginx
# /etc/nginx/snippets/security-headers.conf
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "DENY" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

```nginx
# 在 server 或 location 块中引入
include /etc/nginx/snippets/security-headers.conf;
```

**忘开防火墙端口**。如果配完 Nginx 外面访问不了，先检查防火墙有没有开 80 和 443 端口。上一篇已经配过了，但如果你换过服务器或重置过防火墙规则，可能需要重新开放。

**DNS 没解析**。Certbot 申请证书时需要验证域名所有权，会通过 HTTP 访问你的服务器。如果域名没解析到服务器 IP，验证会失败。

**HSTS 设置过猛**。前面提到过，HSTS 的 `max-age` 设太长、加上 `includeSubDomains`，一旦出问题很难回退。第一次设置建议 `max-age=300`（5 分钟），确认一切正常后再加大。

**配置改完不 reload**。Nginx 配置修改后不会自动生效，必须手动 `nginx -t && systemctl reload nginx`。如果直接 `systemctl restart` 会中断现有连接，而 `reload` 不会。

## 检查一下成果

```bash
# 验证 Nginx 配置
sudo nginx -t

# 测试 HTTP 是否跳转到 HTTPS
curl -I http://api.example.com
# 应该看到 301 跳转到 https://api.example.com

# 测试 HTTPS 和安全头
curl -I https://api.example.com/health
# 应该看到 200 OK，以及所有安全头

# 测试 API 功能
curl https://api.example.com/health
```

到这里，你的应用有了 HTTPS、安全头、Gzip 压缩、日志记录。用户访问 `https://api.example.com`，Nginx 接收请求、解密 TLS、转发给 FlowAPI、拿到响应后压缩、加安全头、返回给用户。整个过程对用户是透明的，他们只看到一个正常的 HTTPS 网站。

## Trade-off 分析

Nginx 作为反向代理增加了架构的一层，带来了额外的复杂度：多了一个需要维护的组件，配置文件需要理解，排查问题时要区分是 Nginx 的问题还是后端应用的问题。对于只有一个服务的简单场景，这个代价看起来有些多余。

但它解决的问题也是真实的：没有 Nginx，你就得在每个应用里自己处理 HTTPS、安全头、静态文件、Gzip，而且每个应用的实现方式可能不一样，维护成本更高。Nginx 把这些横切关注点统一抽出来，应用只需要专注业务逻辑。当你从 1 个服务扩展到 5 个、10 个服务时，Nginx 的价值就更明显了——加一个服务只是多一个配置文件，基础设施的改动可以控制在最小范围。

如果你的场景非常简单（比如一个内部工具，只有一个服务，不需要 HTTPS），直接暴露应用端口也不是不可以。但只要是面向外部用户的场景，Nginx 基本上是标配。多花半小时配置，换来的是标准化的安全防护和灵活的扩展能力。
