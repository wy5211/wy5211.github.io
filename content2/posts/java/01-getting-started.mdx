去年团队要做一个人事管理系统，后端技术选型投票时，我说选 Node.js。理由很充分：前后端统一语言、npm 生态丰富、Express/Koa 我闭着眼睛都能写。结果技术总监说了一句："这个系统要和公司现有的 SAP、OA、考勤机三个系统对接，数据一致性要求强，预算 50 万以内，6 个月交付。你用 Node.js 做，我不放心。"

最后选了 Java + Spring Boot。我不服气，觉得这是老程序员的路径依赖。直到项目做到第三个月，我发现：三个系统的对接协议全都是 SOAP WebService，Java 生态里有现成的 CXF 框架一键生成客户端；事务管理用 Spring 的 `@Transactional` 一个注解搞定；上线后系统稳定运行，没有出现过一次内存泄漏。

我开始重新审视 Java。

## 2026 年，Java 还值得学吗？

先看一组数据。根据 2026 年 Stack Overflow 开发者调查，Java 仍然是全球企业级后端使用率最高的语言（约 35%）。在国内的 Java 招聘市场上，中高级 Java 开发的岗位数量是 Node.js 的 4 倍以上，平均薪资也高出 20-30%。

但数据之外，真正重要的是 Java 在解决什么问题。Node.js 的强项是 I/O 密集型——高并发请求、实时通信、API 网关。Java 的强项是**复杂的业务逻辑 + 强一致性**：金融交易、ERP 系统、大型电商后端。这些场景的核心需求不是"每秒处理多少请求"，而是"业务逻辑不能出错，数据不能丢"。

还有一个现实原因：Java 不是只有 Spring Boot 这一条路。Kafka 的客户端是 Java 写的，Elasticsearch 的官方客户端是 Java，Hadoop 全家桶都是 Java。你要做大数据处理、消息中间件运维、搜索引擎二次开发，Java 是绕不过去的。

如果你是一个前端开发者，想在后端建立更宽的职业护城河，Java 是投入产出比最高的选择之一。

## 先搞清楚 JDK、JRE、JVM

学 Java 的第一道坎是三个缩写。很多教程给你一堆定义，看完还是记不住。换个方式来理解：

想象你开了一家餐厅。

**JVM（Java Virtual Machine）** 是厨房里的灶台。不管你用煤气灶还是电磁炉，菜最终都是在这上面做出来的。Java 程序编译后变成"字节码"，字节码不能直接被操作系统执行，必须由 JVM 来翻译运行。这就像 V8 引擎执行 JavaScript——浏览器不直接懂 JS，V8 帮你翻译成机器指令。

**JRE（Java Runtime Environment）** 是整个厨房。除了灶台（JVM），还包括碗碟、调料这些运行时需要的基础类库。你只需要吃菜（运行 Java 程序），不需要做菜（开发），装个 JRE 就够了。

**JDK（Java Development Kit）** 是厨房 + 菜谱 + 厨具。在 JRE 的基础上加了编译器（`javac`）、调试工具（`jdb`）、打包工具（`jar`）。你要开发 Java 程序，就得装 JDK。

用一句话：JDK 包含 JRE，JRE 包含 JVM。开发装 JDK，运行只装 JRE 就行。

## Java 程序是怎么跑起来的

写 JS 时你习惯这样：写完 `index.js`，直接 `node index.js` 就跑了。JavaScript 是解释执行的，V8 引擎边读边执行。

Java 多了一步编译。你写的 `.java` 文件先被 `javac` 编译成 `.class` 文件（字节码），然后 JVM 读取字节码并执行。

```
JavaScript 的路径：
  index.js → V8 引擎 → 机器码 → 运行

Java 的路径：
  Main.java → javac 编译 → Main.class（字节码） → JVM 执行 → 运行
```

这一步编译带来了两个好处。第一，JVM 可以在执行时做优化（JIT 编译），把反复执行的热点代码直接编译成机器码，运行速度接近 C++。第二，字节码是跨平台的——同一份 `.class` 文件可以在 Windows、macOS、Linux 上跑，只要目标机器装了对应版本的 JVM。

## 安装 JDK 17

2026 年，JDK 17 是企业级项目的首选版本。它是 LTS（长期支持）版本，主流框架都以此为基础。JDK 21 也是 LTS，但 17 的生态成熟度更高，线上稳定性经过更长时间验证。

macOS 上推荐用 Homebrew 安装：

```bash
# 安装 JDK 17
brew install openjdk@17

# 配置环境变量（加到 ~/.zshrc）
export JAVA_HOME=$(/usr/libexec/java_home -v 17)
export PATH=$JAVA_HOME/bin:$PATH

# 验证安装
java -version
# openjdk version "17.0.x" 202x-xx-xx
# OpenJDK Runtime Environment (build 17.0.x+x)
```

Windows 用户去 [Adoptium](https://adoptium.net/) 下载 `.msi` 安装包，安装时勾选"设置 JAVA_HOME"选项即可。

装好之后有一个关键细节：`java` 和 `javac` 都在 `$JAVA_HOME/bin` 目录下。如果终端提示找不到 `javac`，说明环境变量没配对，回去检查 `JAVA_HOME` 和 `PATH`。

## Maven vs npm：包管理的差异

习惯了 `npm install` 的人，第一次接触 Maven 会觉得别扭。两者的核心区别在于**依赖声明的位置**。

npm 把依赖写在项目根目录的 `package.json` 里，Maven 把依赖写在 `pom.xml` 里。看起来差不多，但 Maven 多了一层"仓库"的概念：

```
npm 的依赖查找顺序：
  本地 node_modules → npm 远程仓库

Maven 的依赖查找顺序：
  本地仓库（~/.m2/repository）→ 公司私服 → Maven 中央仓库
```

这个"本地仓库"是 Maven 和 npm 最大的区别。npm 的 `node_modules` 在每个项目里各存一份，Maven 的本地仓库在用户目录下全局共享。装过一次的 jar 包，所有项目都复用同一份。

Maven 的 `pom.xml` 长这样：

```xml
<project>
    <groupId>com.example</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0.0</version>

    <dependencies>
        <!-- 解析 JSON 用到的库，类似 npm 里的 npm install jackson-databind -->
        <dependency>
            <groupId>com.fasterxml.jackson.core</groupId>
            <artifactId>jackson-databind</artifactId>
            <version>2.17.0</version>
        </dependency>
    </dependencies>
</project>
```

三个坐标定位一个依赖：`groupId`（组织）、`artifactId`（项目名）、`version`（版本）。运行 `mvn install`，Maven 会自动下载依赖到本地仓库。这和 `npm install` 的体验基本一致，只是配置文件格式不同。

## IDE 选型

两个选择：IntelliJ IDEA 和 VSCode。

IntelliJ IDEA 是 Java 开发的标准答案。它的代码补全、重构支持、Spring 框架集成、调试器都是专门为 Java 优化的。社区版免费，对入门来说完全够用。缺点是吃内存，启动慢，习惯了 VSCode 的人会觉得"重"。

VSCode 配合 "Extension Pack for Java" 插件也能写 Java，轻量、启动快。但遇到复杂的项目结构（多模块 Maven 项目、Spring Boot 热重载），体验和 IDEA 差距明显。

建议是：前期学习用 VSCode 足够，等项目复杂度上来再切换到 IDEA。

## 第一个程序：解析 JSON 配置文件

用 Hello World 当第一个程序没有意义——你不会在工作中写一个只打印字符串的程序。我们来做一个实际场景：读取一个 JSON 配置文件，解析其中的字段，按条件过滤数据。

假设你有一个 `servers.json`，记录了团队管理的服务器列表：

```json
[
  { "name": "web-prod-01", "env": "production", "cpu": 8, "memory": 32 },
  { "name": "web-staging-01", "env": "staging", "cpu": 4, "memory": 16 },
  { "name": "db-prod-01", "env": "production", "cpu": 16, "memory": 64 },
  { "name": "cache-prod-01", "env": "production", "cpu": 4, "memory": 8 }
]
```

需求：筛选出所有生产环境（`env` 为 `production`）且内存大于等于 16GB 的服务器。用 JavaScript 你可能三行搞定，但 Java 的做法会让你理解这门语言的基本结构。

先创建 Maven 项目结构：

```
server-filter/
├── pom.xml
├── src/main/java/com/example/
│   └── ServerFilter.java
└── servers.json
```

核心代码如下。注意看 Java 的结构：一切代码必须放在类（`class`）里面，程序入口是 `main` 方法——这是 Java 和 JavaScript 最直观的区别。

```java
package com.example;

import com.fasterxml.jackson.databind.ObjectMapper;
import com.fasterxml.jackson.core.type.TypeReference;
import java.io.File;
import java.util.List;
import java.util.Map;
import java.util.stream.Collectors;

public class ServerFilter {
    public static void main(String[] args) throws Exception {
        // ObjectMapper 是 Jackson 库的核心，负责 JSON ↔ Java 对象的转换
        // 类似于 JavaScript 里的 JSON.parse()，但更严格——字段类型必须匹配
        ObjectMapper mapper = new ObjectMapper();

        // 读取 JSON 文件并解析为 List<Map>
        // TypeReference 解决了 Java 泛型擦除的问题，告诉 Jackson 目标类型是 List<Map>
        List<Map<String, Object>> servers = mapper.readValue(
            new File("servers.json"),
            new TypeReference<List<Map<String, Object>>>() {}
        );

        // Stream API 是 Java 8 引入的函数式编程风格
        // .stream() 把集合转为流，.filter() 做筛选，.collect() 把结果收集回 List
        List<Map<String, Object>> result = servers.stream()
            .filter(s -> "production".equals(s.get("env")))
            .filter(s -> (int) s.get("memory") >= 16)
            .collect(Collectors.toList());

        result.forEach(s -> System.out.println(s.get("name")));
    }
}
```

运行方式和 JS 类似，只是多了一步编译：

```bash
# 编译（Maven 会自动下载 Jackson 依赖）
mvn compile

# 运行
mvn exec:java -Dexec.mainClass="com.example.ServerFilter"

# 输出：
# web-prod-01
# db-prod-01
```

这段代码里有几个值得注意的点。第一，`public static void main(String[] args)` 是 Java 程序的固定入口签名——`public` 表示可被外部调用，`static` 表示不需要创建对象就能调用，`void` 表示没有返回值。第二，Java 的类型是静态的——`s.get("memory")` 返回的是 `Object`，你需要强转 `(int)` 才能比较大小。第三，Stream API 的链式调用和 JavaScript 的数组方法（`.filter().map()`）非常像，这是 Java 从函数式编程借鉴来的语法。

## JS vs Java 速查表

为了方便你在两个语言之间切换，这里列一张核心概念对照表：

| 概念       | JavaScript                       | Java                                         |
| ---------- | -------------------------------- | -------------------------------------------- |
| 执行方式   | 解释执行（V8）                   | 编译为字节码 → JVM 执行                      |
| 包管理     | npm/yarn/pnpm                    | Maven/Gradle                                 |
| 类型系统   | 动态类型                         | 静态类型                                     |
| 运行时     | Node.js / 浏览器                 | JVM                                          |
| 入口文件   | index.js                         | 包含 main 方法的类                           |
| 字符串拼接 | 模板字符串 `` `${a} and ${b}` `` | 字符串拼接 `a + " and " + b`                 |
| 数组操作   | `[1,2,3].filter(x => x > 1)`     | `List.of(1,2,3).stream().filter(x -> x > 1)` |
| 异常处理   | try/catch                        | try/catch（必须声明受检异常）                |
| 模块导入   | `import x from 'y'`              | `import com.example.Y;`                      |

## 什么时候选 Java，什么时候不选

说了这么多 Java 的好，但 Java 不是万能的。做一个技术选型决策时，核心是看你的项目特征。

选 Java 的场景：业务逻辑复杂（大量规则引擎、状态机、工作流）、对数据一致性要求高（金融、电商交易）、需要和大量 Java 生态中间件对接（Kafka、Elasticsearch、Zookeeper）、团队规模较大需要强类型约束降低协作成本。这些场景下 Java 的静态类型、成熟的生态、强大的事务管理能力是实打实的优势。

不选 Java 的场景：项目需要快速迭代验证想法（原型阶段 Java 的开发速度不如 Node.js）、I/O 密集型的高并发 API 网关（Node.js 的事件驱动模型更轻量）、团队全是前端背景短期内学不会 Java 的类型系统。

Java vs Node.js vs Go 的后端选型可以简化为：**复杂业务选 Java，快速开发选 Node.js，极致性能选 Go**。三者不是替代关系，而是各有擅长的领域。很多公司同时使用这三种语言——核心业务用 Java，内部工具和 BFF 层用 Node.js，网关和高性能服务用 Go。
