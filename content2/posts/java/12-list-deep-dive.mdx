你在做一个博客 CMS，首页需要展示文章列表——按发布时间倒序、支持分页，后台还要支持按浏览量排序。你习惯性地写了 `new ArrayList<>()`，然后开始想：这真的是最佳选择吗？LinkedList 会不会更好？

这个问题看似简单，但涉及到 List 最核心的取舍：数组连续内存 vs 链表分散节点。选错了不会立刻报错，但会在数据量增长后暴露出性能问题。

## ArrayList vs LinkedList：别被教科书误导

很多教程说"频繁插入删除用 LinkedList，频繁查询用 ArrayList"。这个结论对了一半，但遗漏了一个关键前提：**插入的位置**。

ArrayList 的底层是 `Object[]` 数组。插入和删除操作的时间复杂度取决于位置：

- **尾部插入**：均摊 O(1)，只在数组满时触发扩容
- **中间插入**：O(n)，需要把插入点之后的所有元素往后挪一位
- **按索引查询**：O(1)，直接通过 `base + index * elementSize` 定位

LinkedList 的底层是双向链表，每个节点持有前驱和后继的引用：

- **尾部插入**：O(1)，但每次 `new Node()` 的开销比数组赋值大
- **中间插入**：O(n)，因为要先遍历找到位置（虽然插入本身是 O(1)）
- **按索引查询**：O(n)，必须从头或尾遍历到目标位置

这就产生了一个反直觉的结论：**在实际场景中，ArrayList 几乎总是比 LinkedList 快**。原因有三：

1. CPU 缓存对连续内存有极高偏好——数组可以一次性加载到 L1/L2 缓存，链表的节点散落在堆各处，每次访问都可能缓存未命中
2. LinkedList 的每个节点需要额外的 prev/next 指针，内存开销是 ArrayList 的 2-3 倍
3. 现代 JVM 对数组访问有大量优化（边界检查消除、循环展开），但链表遍历很难优化

那 LinkedList 究竟什么时候该用？答案是**几乎不用**。除非你有明确的"频繁在头尾插入删除"需求（比如实现队列/栈），直接用 `ArrayDeque` 比 LinkedList 更好。JDK 的 `Stack` 类甚至已被官方标注为不推荐使用，建议用 `ArrayDeque` 代替。

```java
// CMS 文章列表——ArrayList 是正确选择
List<Post> posts = new ArrayList<>();

// 大批量添加，预分配容量避免多次扩容
List<Post> posts = new ArrayList<>(1000);
```

| 特性                 | ArrayList           | LinkedList                     |
| -------------------- | ------------------- | ------------------------------ |
| 底层结构             | 动态数组 `Object[]` | 双向链表 Node                  |
| 随机访问 `get(i)`    | O(1)                | O(n)                           |
| 尾部插入 `add(e)`    | 均摊 O(1)           | O(1)                           |
| 中间插入 `add(i, e)` | O(n)                | O(n)（查找位置）+ O(1)（插入） |
| 内存开销             | 小（连续内存）      | 大（每个节点 2 个引用）        |
| CPU 缓存友好         | 高                  | 低                             |
| 实际推荐使用频率     | 绝大多数场景        | 极少                           |

### 扩容机制

ArrayList 的默认初始容量是 10。当元素数量超过当前容量时，扩容为原来的 1.5 倍（`newCapacity = oldCapacity + (oldCapacity >> 1)`）。扩容操作会创建新数组并拷贝全部元素，代价是 O(n)。

如果你能预估数据量，在构造时指定容量可以避免扩容：

```java
// 从数据库查出来 5000 篇文章，直接预分配
List<Post> posts = new ArrayList<>(5000);
```

## 用 Comparator 给文章排序

CMS 后台需要按不同维度排序文章：按发布时间、按浏览量、按标题字母序。Java 的 `Comparator` 就是干这个的。

`Comparable` 和 `Comparator` 的区别经常混淆：

- `Comparable<T>`：在类内部定义自然排序规则（"这个类怎么比"）
- `Comparator<T>`：在外部定义排序规则（"按这个规则比"）

对于 CMS 的 Post 类，发布时间是天然的比较依据，适合用 `Comparable`：

```java
public class Post implements Comparable<Post> {
    private String title;
    private LocalDateTime createdAt;
    private int viewCount;

    @Override
    public int compareTo(Post other) {
        // 按发布时间倒序（新的在前）
        return other.createdAt.compareTo(this.createdAt);
    }

    // getter...
}
```

但浏览量排序、标题排序这些需求，不应该写在 Post 类内部。用 `Comparator` 在调用端定义：

```java
// 按浏览量降序
posts.sort(Comparator.comparingInt(Post::getViewCount).reversed());

// 按标题字母序
posts.sort(Comparator.comparing(Post::getTitle));

// 组合排序：先按浏览量降序，浏览量相同再按时间降序
posts.sort(Comparator.comparingInt(Post::getViewCount).reversed()
    .thenComparing(Post::getCreatedAt, Comparator.reverseOrder()));
```

`Comparator.comparingInt` 和 `Comparator.comparing` 的区别在于装箱：前者直接操作 `int`，避免 `Integer` 的装箱开销。当排序字段是基本类型时，优先用 `comparingInt` / `comparingLong` / `comparingDouble`。

### JavaScript 对比

JS 的 `Array.sort()` 接受一个比较函数，概念上和 Java 的 `Comparator` 类似，但有两个重要区别：

| 特性     | Java `Comparator`           | JavaScript `Array.sort()` |
| -------- | --------------------------- | ------------------------- |
| 类型安全 | 编译时检查                  | 无（运行时才知道类型）    |
| 返回值   | int（负/零/正）             | 同左，但 JS 引擎更宽容    |
| 默认排序 | 自然顺序（需 `Comparable`） | 按字符串 Unicode 排序     |
| 组合排序 | `thenComparing()` 链式      | 需要手动实现              |
| 稳定性   | Java 的 `TimSort` 稳定      | V8 的 `TimSort` 稳定      |

JS 的默认排序行为是按字符串转换，这是一个经典坑：

```js
// JavaScript
[10, 2, 1]
  .sort() // [1, 10, 2]——按字符串排的！
  [(10, 2, 1)].sort((a, b) => a - b); // [1, 2, 10]——正确
```

Java 没有这个问题。`List<Integer>` 的 `sort(null)` 会按数值排序，因为 `Integer` 实现了 `Comparable`。

## subList 与分页

CMS 首页需要分页展示文章。最直觉的做法是用 `subList(fromIndex, toIndex)`：

```java
public List<Post> getPostsByPage(List<Post> allPosts, int page, int size) {
    int from = (page - 1) * size;
    int to = Math.min(from + size, allPosts.size());

    if (from >= allPosts.size()) {
        return Collections.emptyList();
    }
    return allPosts.subList(from, to);
}
```

但 `subList` 有一个容易踩的陷阱：**它返回的是原列表的视图（view），不是副本**。对 subList 的修改会直接影响原列表，反之亦然：

```java
List<String> original = new ArrayList<>(List.of("A", "B", "C", "D", "E"));
List<String> sub = original.subList(1, 3); // [B, C]

sub.set(0, "X");        // original 变成 [A, X, C, D, E]
original.set(2, "Y");   // sub 变成 [X, Y]

// 更危险的情况：修改原列表的结构（增删）会导致 subList 抛出 ConcurrentModificationException
original.add("F");      // 此时 sub 已经失效
sub.get(0);             // ConcurrentModificationException
```

如果要把分页结果传给其他层（比如返回给前端），应该创建一个独立的副本：

```java
// 创建副本，断开与原列表的关联
return new ArrayList<>(allPosts.subList(from, to));

// Java 10+ 更简洁
return List.copyOf(allPosts.subList(from, to));
```

## 不可变 List

CMS 中有些列表一旦构建就不应该被修改，比如分类列表、配置项。Java 提供了几种创建不可变 List 的方式：

```java
// Java 9+ 的 List.of()——最简洁
List<String> categories = List.of("Java", "Node.js", "Rust");

// Java 10+ 的 List.copyOf()——从已有集合创建
List<String> copy = List.copyOf(existingList);

// Collections.unmodifiableList()——包装视图（原列表仍可修改）
List<String> unmodifiable = Collections.unmodifiableList(mutableList);
```

`List.of()` 和 `List.copyOf()` 创建的是真正的不可变列表，任何修改操作都会抛出 `UnsupportedOperationException`。但要注意 `Collections.unmodifiableList()` 只是包装——如果有人持有原列表的引用，仍然可以修改它。

```java
List<String> mutable = new ArrayList<>(List.of("A", "B"));
List<String> unmodifiable = Collections.unmodifiableList(mutable);

unmodifiable.add("C");  // UnsupportedOperationException
mutable.add("C");       // 成功——unmodifiable 的内容也变了
```

所以如果要做真正的防御性拷贝，用 `List.copyOf()` 或 `new ArrayList<>(list)`。

## JS vs Java 对比

| 概念     | JavaScript                             | Java                                           |
| -------- | -------------------------------------- | ---------------------------------------------- |
| 有序列表 | `Array`（唯一选择）                    | `ArrayList` / `LinkedList`                     |
| 排序     | `arr.sort(comparator)`                 | `list.sort(comparator)` / `Collections.sort()` |
| 分页     | `arr.slice(start, end)`                | `list.subList(start, end)`（视图）             |
| 不可变   | `Object.freeze(arr)`（浅冻结，不报错） | `List.of()`（深不可变，抛异常）                |
| 空列表   | `[]`                                   | `List.of()` / `Collections.emptyList()`        |

JS 没有链表这种内置数据结构。当你需要一个"有序集合"时，JS 只有一种选择——数组。Java 提供了多种实现，多出来的选择也意味着你需要做更多的判断。

## 回顾与预告

上篇我们了解了集合框架的全貌：List、Set、Map 各自的基本用法。本篇深入了 List 的两个主要实现和实际使用中的排序、分页、不可变集合。

核心取舍：ArrayList 在绝大多数场景下是更好的选择，LinkedList 的理论优势在实际中往往被 CPU 缓存特性抵消。Comparator 让排序逻辑从数据类中解耦出来，`subList` 是视图而非副本，使用时需要注意生命周期。
