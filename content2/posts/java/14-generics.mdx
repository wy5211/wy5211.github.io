你给 CMS 设计了一个文章仓库 `PostRepository`，里面有 `findById`、`findAll`、`save`、`delete` 这些方法。后来你又需要一个 `TagRepository`、`CommentRepository`——方法签名几乎一模一样，只是操作的实体类型不同。你不想把同样的代码复制三遍。

在 TypeScript 中你会写 `class Repository<T>`，然后 `new Repository<Post>()`。Java 也有泛型，语法看起来一样，但行为上有一个根本区别：**Java 的泛型信息在编译后会被擦除**。这意味着 `List<String>` 和 `List<Integer>` 在运行时是同一个类型。这就是"类型擦除"。

理解类型擦除，是理解 Java 泛型一切行为的基础。

## 泛型类：CMS 的通用仓库

先从 CMS 的 Repository 入手：

```java
public class Repository<T> {
    private final Map<Long, T> store = new HashMap<>();
    private long nextId = 1;

    public T save(T entity) {
        store.put(nextId, entity);
        nextId++;
        return entity;
    }

    public Optional<T> findById(long id) {
        return Optional.ofNullable(store.get(id));
    }

    public List<T> findAll() {
        return new ArrayList<>(store.values());
    }

    public boolean delete(long id) {
        return store.remove(id) != null;
    }
}
```

使用时指定具体类型：

```java
Repository<Post> postRepo = new Repository<>();
Repository<Tag> tagRepo = new Repository<>();

postRepo.save(new Post("Java Generics Guide", "java-generics"));
postRepo.save(new Post("Spring Boot Intro", "spring-boot"));

tagRepo.save(new Tag("Java"));
tagRepo.save(new Tag("Generics"));

// 类型安全：findById 返回 Optional<Post>，不需要强转
Optional<Post> post = postRepo.findById(1L);

// 编译错误：不能把 Tag 放进 PostRepository
// postRepo.save(new Tag("Wrong")); // 编译不通过
```

`<T>` 是类型参数（type parameter），使用时用具体类型替换它。编译器会在所有需要 `T` 的地方做类型检查，但运行时这些信息会消失——这就是类型擦除。

## 类型擦除：编译时存在，运行时消失

类型擦除是 Java 泛型的核心机制。编译器在编译时会：

1. 用类型的**上界**（默认是 `Object`）替换所有类型参数
2. 在需要的地方插入**强制类型转换**
3. 生成**桥方法**（bridge method）保证多态正确

看一个具体的例子：

```java
public class Repository<T> {
    private final Map<Long, T> store = new HashMap<>();

    public T findById(long id) {
        return store.get(id);
    }
}
```

编译后，`T` 被替换为 `Object`（因为没有指定上界）：

```java
// 编译后大致等价于：
public class Repository {
    private final Map store = new HashMap();

    public Object findById(long id) {
        return store.get(id);
    }
}
```

当你调用 `postRepo.findById(1L)` 时，编译器自动插入了强制转换：

```java
// 你写的代码
Post post = postRepo.findById(1L);

// 编译后实际执行的
Post post = (Post) postRepo.findById(1L);
```

这就是为什么这段代码能编译通过但运行时会报错：

```java
List<String> strings = new ArrayList<>();
List<Integer> numbers = new ArrayList<>();
System.out.println(strings.getClass() == numbers.getClass()); // true

// 运行时无法区分 List<String> 和 List<Integer>
// 下面的代码能编译（因为类型擦除后参数类型都是 List），
// 但运行时会抛出 ClassCastException
```

这个设计是 Java 5 引入泛型时做的妥协——为了让泛型代码和 Java 5 之前的非泛型代码兼容。TypeScript 没有这个问题，因为 TS 的类型系统是完全编译时行为，JS 引擎本身就没有类型信息。

### 类型擦除带来的限制

类型擦除直接导致了一些看起来不合理的限制：

```java
// 1. 不能创建泛型数组
// List<String>[] array = new List<String>[10]; // 编译错误
// 原因：擦除后变成 List[]，类型不安全

// 2. 不能用 instanceof 检查泛型类型
// if (obj instanceof List<String>) { } // 编译错误
// 只能用无界通配符
if (obj instanceof List<?>) { } // OK

// 3. 不能实例化类型参数
public class Factory<T> {
    // T instance = new T(); // 编译错误
    // 擦除后变成 Object instance = new Object(); 不是你想要的
}
```

第 3 个限制的解决方案是传入 `Class<T>` 对象，用反射创建实例：

```java
public class Factory<T> {
    private final Class<T> type;

    public Factory(Class<T> type) {
        this.type = type;
    }

    public T newInstance() throws Exception {
        return type.getDeclaredConstructor().newInstance();
    }
}

Factory<Post> factory = new Factory<>(Post.class);
Post post = factory.newInstance();
```

## 泛型方法

有时候你不需要整个类都是泛型的，只需要某个方法是泛型的：

```java
public class CollectionUtils {

    // 泛型方法：类型参数 <T> 写在返回值前面
    public static <T> T findFirst(List<T> list, Predicate<T> condition) {
        for (T item : list) {
            if (condition.test(item)) {
                return item;
            }
        }
        return null;
    }

    // 使用
    public static void main(String[] args) {
        List<Post> posts = List.of(
            new Post("Java Guide", "java"),
            new Post("Node.js Guide", "nodejs")
        );

        // 编译器自动推断 T 为 Post
        Post found = findFirst(posts, p -> p.getSlug().equals("nodejs"));
    }
}
```

泛型方法的好处是**不需要在类级别声明类型参数**。你可以在静态方法中使用泛型（类级别的类型参数不能用于静态方法，因为静态成员不属于任何实例）。

带边界（bound）的泛型方法限制了类型参数的范围：

```java
// T 必须实现 Comparable 接口
public static <T extends Comparable<T>> T max(List<T> list) {
    if (list.isEmpty()) {
        throw new IllegalArgumentException("列表不能为空");
    }
    T result = list.get(0);
    for (T item : list) {
        if (item.compareTo(result) > 0) {
            result = item;
        }
    }
    return result;
}
```

`<T extends Comparable<T>>` 告诉编译器：T 必须是可以和自身比较的类型。这样在方法体内就可以安全调用 `compareTo()`。

## 通配符：`? extends` 和 `? super`

假设你有一个打印列表的方法：

```java
// 这样写只能打印 List<Post>
public static void printPosts(List<Post> posts) { ... }

// 你可能想让它更通用：
public static void printList(List<Object> list) { ... }
// 但 List<Post> 不是 List<Object> 的子类型！
// printPosts(List.of(new Post(...))) // 编译错误
```

这是泛型的一个反直觉特性：**泛型类型是不变的（invariant）**。`List<Post>` 不是 `List<Object>` 的子类型，即使 `Post` 是 `Object` 的子类。原因在于类型安全——如果允许 `List<Post>` 赋值给 `List<Object>`，你就可以往里面塞任意 Object，破坏了类型安全。

通配符解决了这个问题：

### `? extends T`（上界通配符）

```java
// 可以接受 List<Post>、List<Tag>、List<Comment>，只要元素是 Content 的子类
public static void printContents(List<? extends Content> list) {
    for (Content item : list) {
        System.out.println(item.getTitle());
    }
}
```

`? extends Content` 的含义是"Content 的某个未知子类型"。你可以从列表中**读取** Content（因为不管实际类型是什么，它至少是 Content），但不能往里面**写入**（因为你不知道实际类型是什么，写错了就会破坏类型安全）。

这就是 **PECS 原则**（Producer Extends, Consumer Super）的前半部分：如果你只需要从集合中**读取**数据（集合是生产者），用 `? extends`。

### `? super T`（下界通配符）

```java
// 可以接受 List<Post>、List<Content>、List<Object>
public static void addDefaultPost(List<? super Post> list) {
    list.add(new Post("Default", "default"));
}
```

`? super Post` 的含义是"Post 的某个未知父类型"。你可以往列表中**写入** Post（因为不管实际类型是什么，Post 都是它的子类型），但读取时只能得到 `Object`（因为你不知道实际类型是什么）。

这就是 PECS 的后半部分：如果你只需要向集合中**写入**数据（集合是消费者），用 `? super`。

### PECS 总结

```java
// 从集合读取——用 extends
public static double sum(List<? extends Number> numbers) {
    double total = 0;
    for (Number n : numbers) {
        total += n.doubleValue(); // 安全：读出来的一定是 Number
    }
    return total;
    // numbers.add(1); // 编译错误：不知道实际类型，不能写
}

// 向集合写入——用 super
public static void addAll(List<? super Integer> list) {
    list.add(1);
    list.add(2);
    list.add(3);
    // Integer i = list.get(0); // 编译错误：读出来的是 Object
}
```

### 无界通配符 `?`

`List<?>` 等价于 `List<? extends Object>`，表示"某种未知类型的列表"。你只能从中读取 Object，不能写入任何非 null 的值。适用于类型无关的工具方法：

```java
public static int size(List<?> list) {
    return list.size();
}
```

## JS vs Java 对比

| 概念           | TypeScript                               | Java                                  |
| -------------- | ---------------------------------------- | ------------------------------------- |
| 泛型语法       | `interface Repository<T>`                | `class Repository<T>`                 |
| 类型擦除       | 编译后完全消失（和 Java 相同）           | 编译后擦除为上界类型                  |
| 运行时类型信息 | 无（和 Java 相同）                       | 无（但可以通过反射获取部分信息）      |
| 通配符         | 无（TS 用条件类型 `extends` 做类似的事） | `? extends T` / `? super T`           |
| 原始类型       | 无（TS 必须指定类型参数）                | 允许（`List` 不带类型参数，编译警告） |
| 基本类型       | `number` 是泛型参数                      | 不能用 `int`，必须用 `Integer`        |
| `new T()`      | 不支持                                   | 不支持（原因相同：擦除）              |

TS 和 Java 的泛型在"编译后消失"这一点上是相同的——两者都是给编译器看的。但 Java 有运行时的原始类型兼容性问题（需要和 Java 1.4 之前的代码兼容），这催生了通配符等 Java 特有的设计。

## 回顾与预告

上篇讲了 Set 和 Map——用 Set 给标签去重，用 HashMap 按 slug 缓存文章。本篇讲了泛型——从 CMS 的通用 Repository 出发，理解了泛型类、泛型方法、通配符和类型擦除。

核心取舍：泛型让代码更通用、更类型安全，但类型擦除带来了运行时的局限性——你不能 `new T()`、不能用 `instanceof List<String>`、不能创建泛型数组。PECS 原则是使用通配符的决策指南：只读用 `extends`，只写用 `super`。
