CMS 需要一个"导出文章"功能：把数据库里的文章内容写成 `.md` 文件保存到磁盘。还需要一个"导入文章"功能：读取用户上传的 Markdown 文件，解析后存入数据库。

在 Node.js 里，这两件事各一行代码：

```js
await fs.writeFile("posts/java-stream.md", content, "utf-8");
const content = await fs.readFile("posts/java-stream.md", "utf-8");
```

到 Java 里一看——`FileInputStream`、`FileOutputStream`、`FileReader`、`FileWriter`、`BufferedReader`、`BufferedWriter`、`InputStreamReader`、`OutputStreamWriter`、`Files`、`Path`……光 IO 相关的类就有几十个。

Java 的 IO 类确实多，但它们有清晰的分层逻辑：**字节流处理一切二进制数据，字符流处理文本**。理解了这个分界线，IO 体系就不再混乱。

## 字节流 vs 字符流：为什么 Java 要分开

计算机底层处理的是字节。文本文件说到底是字节序列，需要按编码规则（UTF-8、GBK 等）解码成字符。JS 的 `fs` 模块默认用 UTF-8 编码，对开发者屏蔽了"字节和字符"的区别。Java 没有帮你做这个决定——它提供了两套独立的体系：

**字节流**（InputStream / OutputStream）：逐字节读写，适用于任何文件类型（图片、视频、压缩包等二进制文件，也包括文本文件）。

**字符流**（Reader / Writer）：逐字符读写，自动处理编码转换，专门用于文本文件。

对于 CMS 的 Markdown 读写，字符流是更自然的选择——Markdown 是文本，你需要按字符处理（按行读取、拼接字符串），不需要关心底层是 UTF-8 还是 GBK 编码。

但有些场景只能用字节流。比如 CMS 需要存储用户上传的封面图片，图片是二进制数据，用字符流会导致数据损坏。

## 字节流基础：InputStream 和 OutputStream

最基础的文件读写，用 `FileInputStream` 和 `FileOutputStream`：

```java
// 写入字节
try (FileOutputStream fos = new FileOutputStream("output.txt")) {
    fos.write("Hello, CMS".getBytes(StandardCharsets.UTF_8));
}

// 逐字节读取（效率低，仅作演示）
try (FileInputStream fis = new FileInputStream("output.txt")) {
    int b;
    while ((b = fis.read()) != -1) {
        System.out.print((char) b);
    }
}
```

`read()` 每次只读一个字节，返回 `int`（-1 表示文件结束）。逐字节读的效率很低，实际开发中用缓冲：

```java
byte[] buffer = new byte[1024];
int bytesRead;
while ((bytesRead = fis.read(buffer)) != -1) {
    // 处理 buffer[0..bytesRead-1]
}
```

包装成 `BufferedInputStream` / `BufferedOutputStream` 后，底层会自动管理缓冲区，你不需要手动处理 `byte[]`。不过对于文本文件，有更好的选择。

## 字符流：Reader 和 Writer

`FileReader` 和 `FileWriter` 是字符流最简单的入口：

```java
// 写入文本
try (FileWriter writer = new FileWriter("post.md")) {
    writer.write("# Java Stream API\n\n");
    writer.write("Stream 是 Java 8 引入的函数式数据处理 API。\n");
}

// 逐字符读取（效率低）
try (FileReader reader = new FileReader("post.md")) {
    int ch;
    while ((ch = reader.read()) != -1) {
        System.out.print((char) ch);
    }
}
```

和字节流一样，逐字符读取效率低。实际用 `BufferedReader`，它提供了 `readLine()` 方法，一次读一行：

```java
try (BufferedReader reader = new BufferedReader(new FileReader("post.md"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
}
```

注意 `readLine()` 返回的字符串**不包含换行符**。如果需要保留换行符，可以用 `BufferedWriter.newLine()` 写入标准换行，读取时自己拼回去。

对应的写入端，`BufferedWriter` 提供了缓冲写入和换行方法：

```java
try (BufferedWriter writer = new BufferedWriter(new FileWriter("post.md"))) {
    writer.write("# Java Stream API");
    writer.newLine();
    writer.newLine();
    writer.write("Stream 是 Java 8 引入的函数式数据处理 API。");
    writer.newLine();
}
```

## 编码问题：InputStreamReader 和 OutputStreamWriter

`FileReader` 和 `FileWriter` 使用系统默认编码（取决于操作系统，可能是 UTF-8 也可能是 GBK）。CMS 需要保证编码一致——Markdown 文件用 UTF-8 存储，读取时也必须用 UTF-8 解码，否则中文会乱码。

指定编码需要用 `InputStreamReader` 和 `OutputStreamWriter`——它们是字节流和字符流之间的桥梁：

```java
// 用指定编码读取文件
try (BufferedReader reader = new BufferedReader(
        new InputStreamReader(
            new FileInputStream("post.md"),
            StandardCharsets.UTF_8
        )
    )) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
}
```

这里的装饰链是：`FileInputStream`（字节流） -> `InputStreamReader`（字节转字符，指定编码） -> `BufferedReader`（按行缓冲读取）。三层包装，每层负责一件事。这种"装饰器模式"是 Java IO 的核心设计。

JS 的 `fs.readFile(path, 'utf-8')` 一行搞定，Java 需要三层包装。这是 Java IO 被诟病"繁琐"的主要原因。但反过来理解：每一层都可以单独替换——你要换成 GBK 编码？只换 `InputStreamReader` 的参数。你要换成从网络读取？把 `FileInputStream` 换成 `Socket.getInputStream()`。其余代码不变。

## try-with-resources：不再忘记关闭

IO 流打开后必须关闭，否则会泄漏文件句柄（操作系统对同时打开的文件数量有限制）。Java 7 之前的标准写法：

```java
BufferedReader reader = null;
try {
    reader = new BufferedReader(new FileReader("post.md"));
    String line;
    while ((line = reader.readLine()) != null) {
        // 处理
    }
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (reader != null) {
        try {
            reader.close();  // close() 本身也可能抛 IOException
        } catch (IOException e) {
            // 处理 close 失败
        }
    }
}
```

finally 里嵌套 try-catch，冗长且容易写错。Java 7 的 try-with-resources 彻底解决了这个问题：

```java
try (BufferedReader reader = new BufferedReader(new FileReader("post.md"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        // 处理
    }
} // reader 自动关闭，无论是否抛异常
```

只要放在 `try()` 括号里的对象实现了 `AutoCloseable` 接口，退出 try 块时会自动调用 `close()`。多个资源用分号分隔：

```java
try (BufferedReader reader = new BufferedReader(new FileReader("input.md"));
     BufferedWriter writer = new BufferedWriter(new FileWriter("output.md"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        writer.write(line);
        writer.newLine();
    }
} // reader 和 writer 都会自动关闭
```

关闭顺序和声明顺序相反——先关 writer，再关 reader。这符合"先开后关"的原则。

JS/TS 没有对应的语法。Node.js 的 `fs.readFile` 是异步的，不存在"忘记关闭"的问题（文件在回调/resolve 之后自动关闭）。但如果用 `fs.open` 获取文件句柄，仍然需要手动 `await fileHandle.close()`，通常放在 finally 里。

## Files 工具类：Java 7+ 的简化方案

如果你觉得装饰器链太长，Java 7 引入的 `java.nio.file.Files` 提供了更简洁的静态方法：

```java
// 一次性读取全部内容
String content = Files.readString(Path.of("post.md"), StandardCharsets.UTF_8);

// 按行读取
List<String> lines = Files.readAllLines(Path.of("post.md"), StandardCharsets.UTF_8);

// 一次性写入
Files.writeString(Path.of("post.md"), content, StandardCharsets.UTF_8);

// 按行写入
Files.write(Path.of("post.md"), lines, StandardCharsets.UTF_8);
```

这些方法内部自动处理了流的打开和关闭，不需要 try-with-resources。对于 CMS 的 Markdown 读写，大多数场景用 `Files` 就够了。

但 `Files.readAllLines()` 会一次性把所有行加载到内存。如果文件有几百 MB，这会导致内存溢出。这时候还是得回到 `BufferedReader.readLine()` 逐行处理。

```java
// 大文件：逐行处理，不会一次加载到内存
try (BufferedReader reader = Files.newBufferedReader(Path.of("large-post.md"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        // 处理每一行
    }
}
```

`Files.newBufferedReader()` 是 `new BufferedReader(new FileReader(...))` 的简写，同时支持指定编码。

## CMS 场景：导出文章为 Markdown

把一篇 CMS 文章导出为标准的 Markdown 文件：

```java
public class PostExporter {

    public static void exportPost(Post post, String directory) throws IOException {
        Path dir = Path.of(directory);
        Files.createDirectories(dir);

        Path filePath = dir.resolve(post.getSlug() + ".md");

        StringBuilder md = new StringBuilder();
        md.append("# ").append(post.getTitle()).append("\n\n");
        md.append("> ").append(post.getSummary()).append("\n\n");

        if (!post.getTags().isEmpty()) {
            md.append("标签：");
            md.append(String.join(", ", post.getTags()));
            md.append("\n\n");
        }

        md.append(post.getContent()).append("\n");

        Files.writeString(filePath, md.toString(), StandardCharsets.UTF_8);
    }
}
```

`Files.createDirectories()` 会创建所有不存在的父目录（等价于 `mkdir -p`），不需要手动检查目录是否存在。

对应的 JS 写法：

```js
async function exportPost(post, directory) {
  await fs.mkdir(directory, { recursive: true });
  const md = [
    `# ${post.title}`,
    "",
    `> ${post.summary}`,
    "",
    `标签：${post.tags.join(", ")}`,
    "",
    post.content,
  ].join("\n");
  await fs.writeFile(`${directory}/${post.slug}.md`, md, "utf-8");
}
```

逻辑相同，Java 用 `StringBuilder` 拼接，JS 用模板字符串和 `join`。

## CMS 场景：批量导入 Markdown 文件

导入一个目录下所有 `.md` 文件：

```java
public class PostImporter {

    public static List<Post> importFromDirectory(String directory) throws IOException {
        Path dir = Path.of(directory);

        if (!Files.isDirectory(dir)) {
            throw new IllegalArgumentException("目录不存在: " + directory);
        }

        List<Post> posts = new ArrayList<>();

        try (DirectoryStream<Path> stream = Files.newDirectoryStream(dir, "*.md")) {
            for (Path file : stream) {
                String content = Files.readString(file, StandardCharsets.UTF_8);
                Post post = parseMarkdown(content, file.getFileName().toString());
                posts.add(post);
            }
        }

        return posts;
    }

    private static Post parseMarkdown(String content, String filename) {
        // 简化的解析逻辑：提取第一行作为标题
        String title = content.lines()
            .filter(line -> line.startsWith("# "))
            .findFirst()
            .map(line -> line.substring(2))
            .orElse(filename.replace(".md", ""));

        return new Post(title, filename.replace(".md", ""));
    }
}
```

`Files.newDirectoryStream(dir, "*.md")` 用 glob 模式过滤文件，只返回 `.md` 结尾的文件。`content.lines()` 返回一个 `Stream<String>`，可以用上一篇学的 Stream API 来处理——这里用它提取标题。

JS 的等价写法：

```js
async function importFromDirectory(directory) {
  const files = await fs.readdir(directory);
  const mdFiles = files.filter((f) => f.endsWith(".md"));

  return Promise.all(
    mdFiles.map(async (file) => {
      const content = await fs.readFile(`${directory}/${file}`, "utf-8");
      return parseMarkdown(content, file);
    })
  );
}
```

Java 用 `DirectoryStream` 遍历，JS 用 `fs.readdir` + `filter`。Java 的流需要 try-with-resources 关闭，JS 的 `readdir` 返回数组不需要关闭。

## JS vs Java IO 对比

| 特性       | Node.js fs                                            | Java IO                                                |
| ---------- | ----------------------------------------------------- | ------------------------------------------------------ |
| 读文本文件 | `fs.readFile(path, 'utf-8')`                          | `Files.readString(path)` 或 `BufferedReader`           |
| 写文本文件 | `fs.writeFile(path, content, 'utf-8')`                | `Files.writeString(path, content)` 或 `BufferedWriter` |
| 按行读取   | `readline.createInterface()` 或 `content.split('\n')` | `BufferedReader.readLine()`                            |
| 指定编码   | 传入 `'utf-8'` 参数                                   | `StandardCharsets.UTF_8` 或 `InputStreamReader`        |
| 自动关闭   | 异步 API 自动关闭                                     | try-with-resources（同步流需要显式关闭）               |
| 遍历目录   | `fs.readdir()` + `filter`                             | `Files.newDirectoryStream(dir, glob)`                  |
| 创建目录   | `fs.mkdir(dir, { recursive: true })`                  | `Files.createDirectories(dir)`                         |
| 二进制文件 | `fs.readFile(path)`（返回 Buffer）                    | `FileInputStream` + `byte[]`                           |

Node.js 的 `fs` 模块 API 更简洁，因为它默认是异步的（Promise 版本），不需要手动管理资源生命周期。Java IO 是同步的，流用完必须关闭，所以需要 try-with-resources。两者各有取舍——异步 IO 更适合高并发场景，同步 IO 更直观、更容易调试。

## Trade-off

Java IO 体系庞大，初看确实吓人。但分层设计有它的道理：字节流和字符流的分离，让你在处理不同类型文件时有明确的选择；装饰器模式让你可以自由组合缓冲、编码转换等功能。`Files` 工具类覆盖了最常见的场景，大多数 CMS 开发用它就够了，只有在处理大文件或需要细粒度控制时才需要回到底层流。

try-with-resources 是 Java IO 必须掌握的语法——忘记关闭流是 Java 程序最常见的资源泄漏来源之一。和 JS 的异步 API 自动关闭不同，Java 的同步流需要你显式管理生命周期，这是语言模型的差异，不是设计缺陷。

Java 8 之后有了 `java.nio.file` 包（NIO.2），API 比 `java.io` 现代得多。如果你的项目在 Java 7+ 上运行，优先用 `Files` 和 `Path`，而不是 `File`、`FileInputStream` 这些老类。`java.io.File` 在 Java 7 之后基本被 `java.nio.file.Path` 取代了。
