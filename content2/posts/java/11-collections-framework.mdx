上一篇我们给 CMS 建立了异常体系。PostNotFoundException 告诉调用方"这篇不存在"，InvalidStatusTransitionException 告诉调用方"状态不能这么变"。异常处理的本质是把错误信息从"发生了什么"变成"为什么发生"——这是 Java 强类型思维在错误处理上的体现。

现在有一个新问题：CMS 里的文章越来越多，怎么管理它们？

用数组？`Post[] posts = new Post[1000]`——你得提前知道最多有多少篇文章，少了不够用，多了浪费内存。插入一篇新文章到中间位置？后面所有元素都得往后挪。删除一篇？又得往前挪。这在 JS 里用 `Array.splice()` 三行搞定的事，在 Java 数组里是手动的 for 循环。

Java 的解决方案是**集合框架（Collections Framework）**。它提供了一套接口和实现类，帮你处理"一组对象的存储和操作"这个普遍问题。如果你有 JS 背景，这篇文章的核心任务就是搞清楚：Java 的 List/Map/Set 和 JS 的 Array/Object/Map 到底有什么不同，什么时候该用哪个。

## JS Array vs Java List：看起来一样，本质不同

先做一个对比。JavaScript 里你天天用 Array，Java 里对应的叫 List。两者都能存一组元素、按索引访问、遍历。但它们的内部实现和设计哲学差别很大。

```js
// JavaScript：Array 是动态的、万能的
const posts = [
  { title: "Java 入门", status: "DRAFT" },
  { title: "Spring Boot", status: "PUBLISHED" },
];

// 什么都能往里塞
posts.push(42); // 数字混进来了，不会有任何警告
posts.push(null); // null 也行
posts[100] = "oops"; // 跳跃赋值，中间全是 empty

// 筛选已发布的文章
const published = posts.filter((p) => p.status === "PUBLISHED");
```

JavaScript 的 Array 是一个"什么都接受"的容器。没有类型约束，你可以往里面塞任何东西——数字、字符串、对象、null、undefined，甚至混合类型。它既是列表、又是栈、又是队列，还是一个稀疏数组（可以跳过索引直接赋值）。

Java 的 List 完全不同：

```java
// Java：List 有严格的类型约束
List<Post> posts = new ArrayList<>();

// 只能放 Post 类型的对象，编译器强制检查
posts.add(new Post(1L, "Java 入门", "java-intro", "# Hello", PostStatus.DRAFT));
// posts.add(42);          // 编译错误：Int 无法转为 Post
// posts.add("hello");     // 编译错误：String 无法转为 Post

// 筛选已发布的文章
List<Post> published = posts.stream()
    .filter(p -> p.getStatus() == PostStatus.PUBLISHED)
    .collect(Collectors.toList());
```

`List<Post>` 里的 `<Post>` 是泛型参数，告诉编译器这个列表只能存 Post 对象。试图往里面塞其他类型，编译直接报错。这不是运行时的类型检查——是编译时的类型保证。你在写代码的那一刻就排除了"类型混进去"的可能性。

| 特性       | JavaScript Array               | Java List                            |
| ---------- | ------------------------------ | ------------------------------------ |
| 类型约束   | 无，可以混存任意类型           | 泛型约束，编译时检查                 |
| 大小       | 动态自动扩容                   | 动态自动扩容（ArrayList）            |
| 按索引访问 | `arr[0]`                       | `list.get(0)`                        |
| 尾部添加   | `arr.push(item)`               | `list.add(item)`                     |
| 按索引删除 | `arr.splice(index, 1)`         | `list.remove(index)`                 |
| 按值删除   | `arr.filter(x => x !== item)`  | `list.remove(item)`                  |
| 查找元素   | `arr.indexOf(item)`            | `list.indexOf(item)`                 |
| 排序       | `arr.sort(compareFn)`          | `Collections.sort(list, comparator)` |
| 遍历       | `for...of` / `forEach` / `map` | 增强 for / `forEach` / Stream        |

操作方式几乎一一对应，但有一个关键区别：Java List 是**接口**，ArrayList 是**实现类**。你声明变量时用接口类型，创建对象时选实现类。这种"面向接口编程"的思路在 JS 里不常见——JS 没有"接口"这个概念（TS 的 interface 是类型层面的，不是运行时的）。

```java
// 接口类型声明，ArrayList 实现
List<Post> posts = new ArrayList<>();

// 如果某天需要频繁在头部插入，换成 LinkedList 就行
// List<Post> posts = new LinkedList<>();
// 其他代码不用改，因为它们都依赖 List 接口
```

## 集合框架全景

Java 集合框架不是一个类，而是一套**接口 + 实现类**的体系结构。核心是三个接口：List、Set、Map。

```
Collection (接口)
├── List (接口)        → 有序、可重复、可索引
│   ├── ArrayList      → 数组实现，随机访问快
│   ├── LinkedList     → 链表实现，插入删除快
│   └── Vector         → 线程安全版 ArrayList（已过时）
├── Set (接口)         → 无重复元素
│   ├── HashSet        → 基于哈希表，O(1) 查找
│   ├── LinkedHashSet  → 保持插入顺序
│   └── TreeSet        → 自动排序
└── Queue (接口)       → 先进先出
    └── PriorityQueue  → 优先队列

Map (接口)             → 键值对
├── HashMap            → 基于哈希表，O(1) 查找
├── LinkedHashMap      → 保持插入顺序
└── TreeMap            → 按键排序
```

不用一次记住所有类。记住三个核心问题就够了：

- **需要有序且可重复？** 用 List
- **需要去重？** 用 Set
- **需要键值映射？** 用 Map

实际开发中，List 占了 80% 的使用量。Set 和 Map 在需要去重或快速查找的场景才会用到。

## CMS 实战：PostRepository

现在把集合框架用到 CMS 里。我们要创建一个 `PostRepository`，负责管理所有文章的增删改查。在真实项目中，Repository 的数据最终会存到数据库里。但现阶段我们还没有学数据库（那是后面的内容），所以先在内存中用集合来管理。

先看 JavaScript 版本的 Repository 长什么样：

```js
class PostRepository {
  constructor() {
    this.posts = [];
    this.slugIndex = new Map(); // slug → post，快速查找
  }

  save(post) {
    this.posts.push(post);
    this.slugIndex.set(post.slug, post);
  }

  findBySlug(slug) {
    return this.slugIndex.get(slug);
  }

  findByStatus(status) {
    return this.posts.filter((p) => p.status === status);
  }
}
```

JS 里用 Array 管理文章列表，用 Map 做 slug 索引。Java 版本几乎一模一样的思路，但多了类型约束：

```java
import java.util.*;
import java.util.stream.Collectors;

public class PostRepository {
    // 文章列表
    private final List<Post> posts = new ArrayList<>();
    // slug → Post 的快速索引缓存
    private final Map<String, Post> slugIndex = new HashMap<>();

    public void save(Post post) {
        posts.add(post);
        slugIndex.put(post.getSlug(), post);
    }

    // 通过 slug 快速查找文章，O(1) 时间复杂度
    public Optional<Post> findBySlug(String slug) {
        return Optional.ofNullable(slugIndex.get(slug));
    }

    // 按状态筛选
    public List<Post> findByStatus(PostStatus status) {
        return posts.stream()
            .filter(p -> p.getStatus() == status)
            .collect(Collectors.toList());
    }

    public List<Post> findAll() {
        return new ArrayList<>(posts); // 返回副本，防止外部修改
    }
}
```

几个值得注意的细节。

`Optional<Post>` 是 Java 8 引入的容器类，用来表示"值可能存在也可能不存在"。它替代了直接返回 null 的做法——调用方必须显式处理"找不到"的情况，不会被 NullPointerException 偷袭。这和 TS 里的 `post | undefined` 是一个思路。

`new ArrayList<>(posts)` 创建了一个防御性副本。如果你直接 `return posts`，调用方拿到的是同一个列表的引用，可以调用 `list.clear()` 把你的数据清空。返回副本确保了 PostRepository 内部数据的安全性。这在 JS 里做不到——JS 没有语言级别的"不可变"保证，你只能靠团队约定。

`slugIndex` 是一个 HashMap，用来做 slug 到 Post 的快速查找。如果不建索引，每次查找都要遍历整个 posts 列表，时间复杂度是 O(n)。有了 HashMap，查找变成 O(1)。这个思路和 JS 的 `Map` 一样，但 Java 的 HashMap 有一个额外的要求：作为 key 的对象必须正确实现 `hashCode()` 和 `equals()`。String 类已经帮你实现好了，所以这里可以直接用。如果你用自己的类做 key（比如用 Post 对象本身），就必须自己实现这两个方法——后面讲 Set 和 Map 实现原理时会详细展开。

## 按状态筛选、排序、分页

CMS 后台最常见的需求：管理员查看所有草稿文章，按创建时间倒序排列，每页显示 10 条。用 Stream API 来实现：

```java
public class PostRepository {
    private final List<Post> posts = new ArrayList<>();
    private final Map<String, Post> slugIndex = new HashMap<>();

    // ... save、findBySlug 等方法 ...

    /**
     * 按状态筛选 + 按创建时间倒序 + 分页
     *
     * @param status  要筛选的状态（null 表示不筛选）
     * @param page    页码，从 1 开始
     * @param size    每页条数
     * @return        当前页的文章列表
     */
    public List<Post> findPaged(PostStatus status, int page, int size) {
        List<Post> source = posts;

        // 如果指定了状态，先筛选
        if (status != null) {
            source = posts.stream()
                .filter(p -> p.getStatus() == status)
                .collect(Collectors.toList());
        }

        // 按创建时间倒序
        source.sort((a, b) -> b.getCreatedAt().compareTo(a.getCreatedAt()));

        // 分页计算
        int fromIndex = (page - 1) * size;
        int toIndex = Math.min(fromIndex + size, source.size());

        if (fromIndex >= source.size()) {
            return Collections.emptyList(); // 超出范围，返回空列表
        }

        return source.subList(fromIndex, toIndex);
    }

    // 统计各状态的文章数量
    public Map<PostStatus, Long> countByStatus() {
        return posts.stream()
            .collect(Collectors.groupingBy(
                Post::getStatus,
                Collectors.counting()
            ));
    }
}
```

Stream API 是 Java 8 引入的函数式编程接口。`.stream()` 把集合转换成一个"流"，然后你可以链式调用 `.filter()`、`.map()`、`.sorted()`、`.collect()` 等操作。如果你用过 JavaScript 的数组方法链 `[1,2,3].filter(...).map(...)`, Java Stream 的写法非常接近。

但有一个重要区别：Java Stream 是**惰性求值**的。`.filter()` 不会立即执行筛选，只是记录了一个操作。直到遇到 `.collect()` 这样的终结操作，整个链才会真正执行。这意味着你可以把一个很长的操作链组合起来，但数据只遍历一次。JavaScript 的数组方法链则是立即执行的——每个 `.filter()`、`.map()` 都会创建一个新数组。

`countByStatus()` 用了 `Collectors.groupingBy()`，按 PostStatus 分组并计数。结果是一个 `Map<PostStatus, Long>`，比如 `{DRAFT: 5, PUBLISHED: 12, ARCHIVED: 3}`。这在 JS 里是 `reduce` 的典型用法：

```js
// JS 版本的 countByStatus
const counts = posts.reduce((acc, post) => {
  acc[post.status] = (acc[post.status] || 0) + 1;
  return acc;
}, {});
// 结果：{ DRAFT: 5, PUBLISHED: 12, ARCHIVED: 3 }
```

Java 的 `groupingBy` 更简洁，而且返回的是一个真正的 Map 对象（有类型约束），不是一个随便什么属性都能往里塞的普通对象。

## Set：去重的利器

CMS 里有一个场景很适合 Set：文章标签不能重复。

```js
// JavaScript：要手动去重
const tags = ["java", "spring", "java", "backend"];
const unique = [...new Set(tags)]; // ["java", "spring", "backend"]
```

Java 里用 Set 直接存储，天然去重：

```java
Set<String> tagNames = new HashSet<>();
tagNames.add("java");
tagNames.add("spring");
tagNames.add("java");     // 重复，不会被添加
tagNames.add("backend");
// tagNames: {"java", "spring", "backend"}（顺序不确定）
```

HashSet 依靠 `hashCode()` 和 `equals()` 判断两个对象是否"相同"。对于 String、Integer 这些 Java 标准库的类，这两个方法已经实现好了。但对于自定义类（比如 Tag），你需要自己实现。在 CMS 里，Tag 的 name 应该是唯一的，所以：

```java
public class Tag {
    private String name;

    // 两个 Tag 如果 name 相同，就认为是同一个 Tag
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Tag)) return false;
        return name.equals(((Tag) o).name);
    }

    @Override
    public int hashCode() {
        return name.hashCode();
    }
}
```

如果两个对象的 `equals()` 返回 true，它们的 `hashCode()` 也必须相同。反过来不一定——hashCode 相同不代表对象相等（哈希冲突）。这个约定是 HashSet 和 HashMap 正确工作的前提。违反它，Set 就会"漏掉"重复元素。

JS 的 Set 用 `===` 做比较（原始类型按值，对象按引用），所以两个 `{ name: "java" }` 对象会被认为是不同的元素。Java 的 Set 更灵活——你可以通过重写 `equals()` 来定义"相等"的含义。

## Map vs Object：键值对的选择

JavaScript 里存键值对有两个选择：普通对象和 Map。Java 里的 Map 只有一个核心实现：HashMap（以及排序版的 TreeMap 和保持插入顺序的 LinkedHashMap）。

```js
// JavaScript：普通对象 vs Map
const obj = { java: 5, spring: 3 }; // 键只能是 string/Symbol
const map = new Map(); // 键可以是任意类型
map.set("java", 5);
map.set(42, "number key"); // 数字也可以做键
```

```java
// Java：Map 的键可以是任何对象（只要正确实现 hashCode/equals）
Map<String, Integer> tagCount = new HashMap<>();
tagCount.put("java", 5);
tagCount.put("spring", 3);

// 如果 key 不存在，用 getOrDefault 避免返回 null
int count = tagCount.getOrDefault("rust", 0);

// putIfAbsent：只在 key 不存在时才插入
tagCount.putIfAbsent("java", 10); // 不会覆盖已有的 5
```

`getOrDefault` 和 `putIfAbsent` 这两个方法在 JS 里没有直接对应——JS 里你得自己写 `map.get(key) ?? 0` 或 `if (!map.has(key)) map.set(key, value)`。Java 的 Map 在 API 层面就把这些常见操作内置了。

在 CMS 的 PostRepository 里，`slugIndex` 就是一个 Map 的典型应用。slug 是文章的 URL 友好标识符（比如 "java-collections-framework"），每个 slug 唯一对应一篇文章。用 Map 做 slug 索引，查找一篇文章只需要 O(1) 的时间。这在文章数量多的时候（几千、几万篇），性能优势非常明显。

## JS Array/Object/Map vs Java List/Set/Map

用一张表总结核心差异：

| 特性     | JS Array               | Java List         | JS Set     | Java Set                  | JS Object/Map                   | Java Map                  |
| -------- | ---------------------- | ----------------- | ---------- | ------------------------- | ------------------------------- | ------------------------- |
| 类型约束 | 无                     | 泛型约束          | 无         | 泛型约束                  | 无                              | 泛型约束                  |
| 去重     | 手动                   | 不去重            | 自动       | 自动                      | 键唯一                          | 键唯一                    |
| 有序     | 按索引                 | 按索引            | 插入顺序   | 取决于实现                | 插入顺序                        | 取决于实现                |
| null 值  | 允许多个               | 允许一个          | 允许一个   | 允许一个                  | 允许                            | 允许一个 null key         |
| 元素比较 | `===`                  | `equals()`        | `===`      | `equals()` + `hashCode()` | `===`                           | `equals()` + `hashCode()` |
| 遍历     | `for...of` / `forEach` | 增强 for / Stream | `for...of` | 增强 for / Stream         | `for...in` / `Object.entries()` | `entrySet()` / Stream     |

最核心的差异就两个字：**泛型**。Java 集合通过泛型在编译时保证类型安全，JS 集合没有类型约束，运行时才可能发现问题。这不是"好"或"坏"的区别——JS 的灵活性适合快速原型开发，Java 的严格性适合大型项目的长期维护。

## 接口编程：为什么声明 List 而不是 ArrayList

前面提到过，Java 里推荐用接口类型声明变量：

```java
// 推荐：面向接口编程
List<Post> posts = new ArrayList<>();
Map<String, Post> slugIndex = new HashMap<>();

// 不推荐：绑定具体实现
ArrayList<Post> posts = new ArrayList<>();
HashMap<String, Post> slugIndex = new HashMap<>();
```

为什么？因为接口隔离了"你需要什么"和"具体怎么实现"。

假设你的 PostRepository 最初用 ArrayList 存文章。后来发现经常在列表头部插入文章（置顶功能），ArrayList 在头部插入需要移动所有元素，性能差。你想换成 LinkedList。如果声明的是 `ArrayList<Post>`，你需要改所有用到这个变量的地方。如果声明的是 `List<Post>`，你只需要改一行：

```java
// 只改这里
private final List<Post> posts = new LinkedList<>();
// 其他所有用到 posts 的地方都不用改
```

这就是**面向接口编程**的实际价值。它让你的代码更容易替换底层实现而不影响上层逻辑。JS 没有这个概念——JS 的 Array 只有一种实现（V8 内部可能有优化，但对开发者来说是透明的）。

## 集合的不可变性和空安全

Java 集合有两个 JS 开发者容易忽略的问题：不可变性和空值处理。

**不可变性**。Java 提供了几种创建不可变集合的方式：

```java
// Java 9+: 不可变集合
List<String> fixed = List.of("A", "B", "C");
Set<String> fixedSet = Set.of("X", "Y", "Z");
Map<String, Integer> fixedMap = Map.of("a", 1, "b", 2);

// 任何修改操作都会抛出 UnsupportedOperationException
// fixed.add("D");  // 运行时异常
```

`List.of()` 创建的是真正不可变的列表——不能添加、删除、修改。这和 JS 里用 `Object.freeze()` 冻结数组是类似的效果，但 Java 的不可变性是集合框架层面的保证，API 上就明确不支持修改操作。

**空安全**。Java 的 HashMap 允许一个 null key 和多个 null value，但 Java 9+ 的 `Map.of()` 不允许任何 null。HashSet 允许一个 null 元素，`Set.of()` 不允许。这些规则没有 JS 那么统一——JS 里 null 和 undefined 可以随便往数组里塞。Java 的态度是：尽量在编译时就杜绝 null。

## 篇幅回顾

这篇我们从 CMS 的文章管理出发，理解了 Java 集合框架的核心设计：

- **List**（ArrayList）对应 JS 的 Array，但通过泛型提供编译时类型安全
- **Set**（HashSet）提供自动去重，依赖 `equals()` + `hashCode()` 判断元素相等
- **Map**（HashMap）提供键值映射和 O(1) 查找，slug 索引是 CMS 里的典型应用
- **面向接口编程**——声明 `List<Post>` 而不是 `ArrayList<Post>`，方便后续替换实现
- **Stream API** 的链式调用风格和 JS 数组方法链相似，但 Stream 是惰性求值的

有一个问题没有展开：ArrayList 和 LinkedList 的内部实现到底有什么不同？什么时候该用哪个？ArrayList 的扩容机制是怎么工作的？这些问题在下一篇 List 深入中会详细讲。理解了底层实现，你才能真正做出"该用哪个"的正确判断——而不是只记住"ArrayList 查询快、LinkedList 增删快"这种表面结论。
