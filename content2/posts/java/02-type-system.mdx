你在 JS 里写过 `1 + '1' === '11'` 吗？被 `[] == false` 为 true 惊讶过吗？`typeof null === 'object'` 这种设计到底是谁想出来的？JS 的隐式类型转换让无数开发者踩过坑，而 Java 从第一天起就选择了另一条路：**不让你猜，编译期就告诉你对不对**。

上一篇我们搭好了 Java 环境，写了第一个程序。你可能注意到了一个和 JS 完全不同的地方——每个变量都要声明类型。`int count = 0;` 而不是 `let count = 0;`。这不仅仅是一个语法差异，它反映了两种根本不同的类型哲学。

## 静态类型 vs 动态类型：两种不同的安全观

JS 里你可以这样写：

```js
let data = 42;
data = "hello"; // 没问题
data = { name: "x" }; // 还没问题
data.sort(); // 运行时才炸：data is not a function
```

JS 引擎在运行时才知道 `data` 到底是什么。`sort` 是数组方法，但 `data` 此时是个对象，引擎不会提前告诉你有问题，直到这行代码真正执行。

Java 不允许这种事：

```java
int data = 42;
// data = "hello";  // 编译错误：不兼容的类型
// data.sort();     // 编译错误：int 没有 sort 方法
```

编译器在代码还没跑起来之前就拦住了类型错误。这就是静态类型的核心：**变量的类型在编译期确定，运行时不能变**。

这不是说 Java 完全没有运行时类型问题——`ClassCastException` 依然存在，但绝大多数类型错误在编译阶段就被消灭了。你付出的是更多的声明代码，换来的是 IDE 的精确提示和重构时的安全感。

## 8 种基本类型：和 JS 的 Number 说再见

JS 只有一个数字类型 `Number`（64 位浮点），所有的数字都是它。Java 拆成了 8 种基本类型（primitive types），分成四组：

```
整型（整数）          浮点型（小数）       字符型      布尔型
┌──────────┐      ┌──────────┐      ┌──────┐    ┌────────┐
│ byte     │      │ float    │      │ char │    │boolean │
│ 1字节    │      │ 4字节    │      │ 2字节│    │ true/  │
│          │      │          │      │      │    │ false  │
│ short    │      │ double   │      └──────┘    └────────┘
│ 2字节    │      │ 8字节    │
│          │      │          │
│ int ★    │      │          │
│ 4字节    │      │          │
│          │      │          │
│ long     │      │          │
│ 8字节    │      │          │
└──────────┘      └──────────┘
```

带星号的是 `int`，最常用的整数类型。日常开发中，95% 的整数场景用 `int`，95% 的小数场景用 `double`。其他类型在特定场景才出场。

### int vs Number：为什么 Java 不用一个类型包打天下

JS 的 `Number` 用 64 位浮点数表示一切数字，这带来了一个经典问题：

```js
// JS
console.log(0.1 + 0.2); // 0.30000000000000004
console.log(Number.MAX_SAFE_INTEGER + 1 === Number.MAX_SAFE_INTEGER + 2); // true
console.log(9007199254740992 + 1); // 9007199254740992，加 1 等于没加
```

超过 `Number.MAX_SAFE_INTEGER`（2^53 - 1）的整数就不再精确了。这对于处理大 ID、时间戳、金融计算来说是致命的。

Java 把整数和浮点数分开：

```java
// Java：整数用 int/long，小数用 double
int count = 1_000_000;              // 32 位整数，精确到 ±21 亿
long timestamp = 1_713_000_000_000L; // 64 位整数，精确到 ±922 亿亿

// 浮点数同样有精度问题，但至少整数不会受影响
double price = 19.99;
// Java 也有浮点精度问题
System.out.println(0.1 + 0.2);      // 0.30000000000000004
```

注意 `long` 值末尾的 `L` 后缀。不加的话，Java 会把它当成 `int`，超过 21 亿就编译报错。这个后缀让编译器知道"我就是要一个 64 位整数"。

数字中的下划线 `_` 是 Java 7 引入的语法糖，纯粹为了可读性，编译时会忽略。`1_000_000` 和 `1000000` 完全等价。

### char vs string：Java 严格区分单个字符和字符串

这是另一个 JS 开发者容易困惑的地方。JS 没有字符类型，`'a'` 和 `"a"` 是同一个东西。Java 里 `char` 和 `String` 是完全不同的类型：

```java
char letter = 'A';          // 单个字符，用单引号
// char word = 'Hello';     // 编译错误：char 只能放一个字符

String greeting = "Hello";  // 字符串，用双引号
// String bad = 'Hello';    // 编译错误：类型不匹配
```

`char` 本质上是一个 16 位无符号整数（存储 Unicode 编码），所以它能参与数学运算：

```java
char letter = 'A';
System.out.println((int) letter);        // 65，A 的 Unicode 码点
System.out.println((char) (letter + 1)); // B，码点 +1 就是下一个字母
```

在 JS 里你不需要想这些，因为 `'a' + 1` 直接得到 `"a1"`。但 Java 的 `char` 是数值，不是字符串——`'A' + 1` 的结果是 `66`（整数），不是 `"A1"`。

## 变量声明：每一行都在做承诺

JS 用 `let`/`const` 声明变量，类型由值推断：

```js
let count = 0; // JS：count 是什么类型？看赋的值
const name = "hello"; // JS：name 是什么类型？看赋的值
```

Java 要求你显式声明类型——你在告诉编译器一个承诺：

```java
int count = 0;            // 我承诺 count 永远是 int
String name = "hello";    // 我承诺 name 永远是 String
double price = 9.99;      // 我承诺 price 永远是 double
```

分号不能省略。这是从 C/C++ 继承的传统，也是 Java 语法的基本要求。

Java 10 引入了 `var` 关键字，看起来像 JS 的 `let`，但本质完全不同：

```java
var count = 0;            // 编译器推断为 int
var name = "hello";       // 编译器推断为 String
var price = 9.99;         // 编译器推断为 double

// count = "hello";       // 编译错误：count 已经是 int，不能变
```

`var` 只是省了打字，类型在编译期就确定了，之后不能改变。这和 JS 的 `let` 有本质区别——JS 的 `let count = 0` 之后可以 `count = "hello"`，Java 的 `var count = 0` 不行。

什么时候该用 `var`，什么时候该写全类型？实际开发中的经验：当类型显而易见时用 `var`（比如 `var list = new ArrayList<String>()`，右边已经写全了），当类型是关键信息时写全（比如方法参数、公开 API 的返回值）。

## 类型转换：规则明确，不搞隐式

JS 的类型转换规则堪称"黑魔法"：

```js
1 + '1'           // '11'——数字变字符串
'1' - 1           // 0——字符串变数字
'1' == 1          // true——宽松比较
[] == false       // true——数组变布尔再变数字
null == undefined // true——两个 falsy 值相等
```

每一行都有隐式转换在背后发生，而且规则之间互相矛盾（加号把数字变字符串，减号把字符串变数字）。这就是为什么 TypeScript 要加 `strictNullChecks` 和 `noImplicitAny`。

Java 的类型转换规则简单得多，只有两种情况。

### 自动提升：小类型到大类型，不会丢数据

```
byte → short → int → long → float → double
              ↑
             char
```

箭头方向表示可以自动转换。这个链条的设计逻辑是：目标类型的表示范围大于源类型，所以不会丢失信息。

```java
int score = 95;
double result = score;  // 自动提升：int → double
System.out.println(result); // 95.0
```

一个需要注意的陷阱：`byte` 和 `short` 参与运算时，结果会被提升为 `int`：

```java
byte a = 10;
byte b = 20;
// byte c = a + b;  // 编译错误：a + b 的结果是 int，不能直接赋给 byte
int c = a + b;      // 正确
```

这不是 Java 在刁难你，而是因为两个 `byte` 相加可能超过 127（`byte` 的最大值），所以编译器要求你用更大的类型来接收结果。

### 强制转换：大类型到小类型，可能丢数据

反方向的转换必须显式声明——你要告诉编译器"我知道可能丢数据，我接受"：

```java
double pi = 3.14159;
int intPi = (int) pi;     // 强制转换，截断小数部分
System.out.println(intPi); // 3

int big = 300;
byte small = (byte) big;  // 强制转换，但 300 超过了 byte 范围（-128~127）
System.out.println(small); // 44——溢出，结果不正确
```

`(int) pi` 这种写法就是强制转换，括号里是目标类型。注意 `double` 转 `int` 是截断（直接丢弃小数），不是四舍五入。要四舍五入得用 `Math.round()`。

和 JS 的隐式转换相比，Java 的策略很明确：**安全转换自动做，危险转换要你签字画押**。这让你在每一处可能丢数据的转换点都有意识地去审视。

## String：不是基本类型，但比基本类型还常用

`String` 在 Java 里是一个类（引用类型），不是基本类型。但 Java 给了它特殊的语法地位——你可以用双引号创建字符串字面量，不需要 `new`：

```java
// String 不是基本类型，但有字面量语法
String name = "hello";           // 字面量
// String bad = 'hello';         // 编译错误：单引号只用于 char

// 也可以用 new，但不推荐
String also = new String("hello"); // 创建了两个对象，浪费内存
```

### 不可变性：String 一旦创建就不能改

JS 的字符串在技术层面也是不可变的，但你很少会意识到这一点，因为你可以随时重新赋值：

```js
// JS
let str = "hello";
str += " world"; // 看起来像是修改了，实际上是创建了新字符串
console.log(str); // "hello world"
```

Java 的 String 不可变性更"显眼"，因为你会在很多地方感受到它的存在。比如字符串拼接在循环中会创建大量临时对象：

```java
// 低效：每次循环都创建新的 String 对象
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i;  // 每次拼接都创建一个新 String，旧对象等待 GC
}

// 高效：用 StringBuilder，内部是可变的字符数组
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);  // 在同一个缓冲区上操作
}
String result = sb.toString();
```

在 JS 里你不太需要想这个，因为 V8 的引擎优化通常能处理简单的字符串拼接。但 Java 的 String 不可变性是语言设计层面的承诺——所有 String 方法（`substring`、`toUpperCase`、`replace`）都会返回新对象，原字符串永远不变。

## 引用类型 vs 基本类型：两套内存规则

JS 的类型分为原始类型（primitive）和对象（object）。Java 也有类似的分界：基本类型（primitive types）和引用类型（reference types）。

区别在于内存模型：

```
基本类型                              引用类型
变量直接存储值                        变量存储的是引用（地址）
┌─────┐                             ┌─────────┐
│  42 │ ← int x = 42;               │ 0x3a2f │ ← String s = "hello";
└─────┘                             └────┬────┘
      栈上直接存值                        │
                                         ↓
                                    ┌─────────┐
                                    │ "hello" │ ← 堆上的对象
                                    └─────────┘
```

基本类型的值直接存在栈上（简单高效），引用类型的变量存的是堆上对象的地址。这带来了一个 JS 开发者需要特别注意的行为差异：

```java
int a = 10;
int b = a;
b = 20;
System.out.println(a); // 10——基本类型是值拷贝，互不影响

int[] arr1 = {1, 2, 3};
int[] arr2 = arr1;     // arr2 和 arr1 指向同一个数组对象
arr2[0] = 99;
System.out.println(arr1[0]); // 99——引用类型共享同一个对象
```

JS 里同样有这个区分（原始类型值拷贝，对象引用拷贝），但 Java 把它做成了语言层面的硬规则——哪 8 种是基本类型是固定的，你不能再加。而 JS 里 `typeof` 能识别的原始类型只有 `number`、`string`、`boolean`、`undefined`、`symbol`、`bigint`，其他全是 object。

## null 和 undefined：Java 只有 null

JS 有两个"空值"：`null` 和 `undefined`。Java 只认 `null`，而且 `null` 只能赋给引用类型：

```java
String name = null;    // 合法：String 是引用类型
// int count = null;   // 编译错误：int 是基本类型，不能是 null
```

基本类型必须有值。如果你需要一个"可能有值也可能没有"的整数，Java 8 提供了包装类 `OptionalInt`，或者用 `Integer`（`int` 的引用类型包装）：

```java
Integer count = null;  // 合法：Integer 是引用类型
if (count != null) {
    System.out.println(count);
}
```

这和 JS 的 `number | null` 或者 TypeScript 的 `number | undefined` 是类似的思路。但 Java 的处理更"重"，你需要意识到基本类型和引用类型之间的转换（自动装箱/拆箱）是有性能开销的。

## 对比速查表

| 概念           | JavaScript                | Java                                    |
| -------------- | ------------------------- | --------------------------------------- |
| 数字类型       | Number（64 位浮点）       | int(32位), long(64位), double(64位浮点) |
| `1 + '1'`      | `'11'`（隐式转换）        | 编译错误（类型不匹配）                  |
| 变量声明       | `let x = 10;`             | `int x = 10;`                           |
| 字符串         | 可变感觉（实际不可变）    | 完全不可变，不可变是核心设计            |
| null/undefined | null + undefined          | 只有 null                               |
| 类型检查       | typeof（运行时）          | 编译期检查                              |
| 大整数         | Number 精度到 2^53        | long 精度到 2^63                        |
| 字符           | 没有字符类型，'a' === "a" | char 是独立类型，'a' !== "a"            |

## Trade-off：静态类型的优势与代价

选择静态类型不是没有代价的。

**你获得的**：编译期错误检测。IDE 可以在写代码的时候告诉你"这个方法不存在"、"类型不匹配"、"可能为 null"。重构时改一个方法签名，所有调用处都会报错提示你更新。这在大型项目中是巨大的效率提升——你不需要跑到线上才发现 `sort()` 被用在了错误的对象上。

**你付出的**：更多的声明代码和心智负担。你需要考虑"这个 ID 用 int 还是 long"、"这里用 float 还是 double"、"要不要用 var"。在写原型和快速实验时，这种显式声明确实让人感觉啰嗦。

**中间地带**：Java 10 的 `var` 减轻了声明负担，但类型安全没有妥协。现代 IDE 的类型推断和自动补全也让大部分声明成本消失了——你写 `List<String>` 后，IDE 自动帮你补全所有相关的方法调用。

静态类型不是银弹。它解决的是"运行时的类型错误"这一类问题，但解决不了逻辑错误、并发问题、架构问题。它是一个工程层面的安全网，让你在大型团队协作和长期维护中更有底气。
