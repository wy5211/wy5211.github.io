上一篇我们把 Next.js 里"数据怎么进、怎么出"这件事理清楚了 -- Server Components 里直接 async/await 查数据库，Server Actions 处理表单提交，API Routes 对接外部系统。你兴冲冲地把团队任务管理平台部署上去，跑了两天，发现数据库连接池快被撑爆了。

问题出在哪？打开监控面板一看：

```
项目列表页    -- 每次访问查一次数据库，一天 12000 次 SELECT
工作空间设置   -- 几乎不更新的配置数据，也在重复查询
用户头像       -- 同一个用户的头像 URL 被查了 8000 次
任务详情页     -- 高峰期 TPS 到了 50，数据库 CPU 飙到 85%
```

这些数据有一个共同特征：它们的变化频率远低于访问频率。项目列表一天可能改几次，但一天被访问几千次。工作空间设置可能一个月改一次。用户头像更是几乎不变。

这就是缓存要解决的问题：**把重复的数据获取请求省下来，让相同数据在短时间内只计算/查询一次**。

Next.js 提供的不是单一缓存，而是一个四层缓存体系。每一层解决不同的问题，用错了反而会引发更严重的 bug。我们一层一层来看，结合团队任务管理平台的真实场景。

## 第一层：Request Memoization -- 一次渲染内的去重

先从一个你可能已经踩过的坑说起。

假设你有一个项目详情页，需要同时展示项目信息、项目成员、项目下的任务统计。三个数据都需要查项目的 `workspaceId`：

```tsx
// app/(dashboard)/projects/[id]/page.tsx
import { prisma } from "@/lib/prisma";

export default async function ProjectDetailPage({
  params,
}: {
  params: { id: string };
}) {
  // 这三次查询实际上都在查同一个 workspace
  const project = await prisma.project.findUnique({
    where: { id: params.id },
  });

  const workspace = await prisma.workspace.findUnique({
    where: { id: project.workspaceId },
  });

  const workspaceMembers = await prisma.workspaceMember.findMany({
    where: { workspaceId: project.workspaceId },
  });

  // ...渲染
}
```

看起来没什么问题，对吧？但如果另一个组件也恰好查了同一个 workspace 呢？

```tsx
// components/project-sidebar.tsx
// 这个组件在同一个页面中被使用
import { prisma } from "@/lib/prisma";

export async function ProjectSidebar({ workspaceId }: { workspaceId: string }) {
  // 又查了一次 workspace！同一次渲染周期内重复查询
  const workspace = await prisma.workspace.findUnique({
    where: { id: workspaceId },
  });

  return <aside>{workspace.name} - 团队空间</aside>;
}
```

在同一个 React 服务端渲染周期内，`prisma.workspace.findUnique({ where: { id: "ws-001" } })` 被调用了两次。两次都打到数据库，两次返回相同的结果。这就是浪费。

React 的 `cache()` 函数就是解决这个问题的。它不是 Next.js 发明的，而是 React 本身提供的请求级记忆化能力：

```tsx
// lib/data.ts
import { cache } from "react";
import { prisma } from "@/lib/prisma";

/**
 * cache() 包装后的函数在单次渲染周期内，
 * 相同参数只执行一次，后续调用直接返回缓存结果。
 * 注意：这个缓存只存在于当前这一次服务端渲染请求中，
 * 下一个请求会重新执行。
 */
export const getWorkspace = cache(async (workspaceId: string) => {
  return prisma.workspace.findUnique({
    where: { id: workspaceId },
    include: {
      members: {
        include: {
          user: {
            select: { id: true, name: true, avatarUrl: true },
          },
        },
      },
    },
  });
});

export const getProject = cache(async (projectId: string) => {
  return prisma.project.findUnique({
    where: { id: projectId },
    include: {
      workspace: true,
      tasks: {
        select: {
          id: true,
          status: true,
          priority: true,
        },
      },
    },
  });
});
```

现在不管页面里有多少个组件调用了 `getWorkspace("ws-001")`，在一次渲染周期内，数据库只被查询一次。这就是 Request Memoization。

要注意的是，Next.js 对 `fetch()` 请求也自动做了 Request Memoization -- 同一个 URL + 相同 options 的 fetch 请求在单次渲染中只执行一次。所以如果你是用 fetch 调自己的 API 或者外部接口，不用额外处理。

但 Prisma 这种 ORM 查询不走 fetch，所以需要你手动用 `cache()` 包一层。

## 第二层：Data Cache -- 跨请求的数据缓存

Request Memoization 只能在单次请求内去重。但更多时候，你希望这个请求的结果能被后续的请求也用上。

举个例子，你的团队任务管理平台有一个"项目列表"页。这个页面是全团队都在用的，一个 50 人的团队一天可能打开几百次。如果每次都查数据库，那就太浪费了。

Data Cache 就是干这个的：**把 fetch 请求的结果缓存在服务端，跨请求共享**。

```tsx
// app/(dashboard)/projects/page.tsx
import { prisma } from "@/lib/prisma";
import { auth } from "@/lib/auth";
import { cache } from "react";
import { ProjectCard } from "@/components/projects/project-card";

/**
 * 按需缓存函数：使用 React cache + unstable_cache 组合
 * unstable_cache 是 Next.js 提供的跨请求数据缓存
 *
 * - 第一个参数：要缓存的异步函数
 * - 第二个参数：缓存 key（数组形式，用于区分不同参数的调用）
 * - 第三个参数：缓存配置
 *   - tags: 用于按标签批量失效缓存
 *   - revalidate: 缓存有效期（秒），过期后下次请求触发后台重新生成
 */
import { unstable_cache } from "next/cache";

const getCachedProjects = unstable_cache(
  async (userId: string) => {
    // 这个查询的结果会被缓存
    // 在 revalidate 时间窗口内，所有用户看到的都是同一份缓存数据
    return prisma.project.findMany({
      where: {
        // 只查该用户有权限的项目
        members: {
          some: { userId },
        },
      },
      include: {
        _count: {
          select: {
            tasks: true, // 任务总数
            members: true, // 成员总数
          },
        },
      },
      orderBy: { updatedAt: "desc" },
    });
  },
  ["projects-list"], // 缓存 key 前缀
  {
    tags: ["projects"], // 后续可以用 revalidateTag("projects") 来失效
    revalidate: 300, // 5 分钟后过期
  }
);

export default async function ProjectsPage() {
  const session = await auth();
  if (!session?.user) return null;

  // 命中缓存时直接返回，不查数据库
  const projects = await getCachedProjects(session.user.id);

  return (
    <div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6">
      {projects.map((project) => (
        <ProjectCard key={project.id} project={project} />
      ))}
    </div>
  );
}
```

不过这里有个关键问题：`unstable_cache` 在有用户特定参数时需要特别小心。上面的代码中 `userId` 是缓存函数的参数，Next.js 会自动把它加入缓存 key，所以不同用户会拿到各自的缓存。但如果你的团队不大，你也可以在更粗的粒度上缓存，然后在应用层做权限过滤。

Data Cache 的核心机制是 **Stale-While-Revalidate**：缓存过期后，用户不会等待重新生成，而是立即拿到旧数据（stale），同时 Next.js 在后台触发重新生成。下次请求时拿到新数据。用户体验上几乎没有感知。

## 第三层：Full Route Cache -- 整个页面的 HTML 和 RSC Payload

前两层缓存的是数据。Full Route Cache 更进一步，**把整个页面渲染的结果（HTML + RSC Payload）都缓存下来**。

在构建时（`next build`），Next.js 会渲染所有静态页面，把结果存到磁盘上。当用户访问时，直接返回缓存的 HTML，不需要走 React 的服务端渲染流程。

回到我们的任务管理平台，大部分页面其实都适合这个策略：

```tsx
// app/(dashboard)/projects/[id]/page.tsx
// 告诉 Next.js：这个页面的缓存 10 分钟后过期
// 在过期之前，所有用户看到的是同一份预渲染的 HTML
export const revalidate = 600; // 10 分钟

export default async function ProjectDetailPage({
  params,
}: {
  params: { id: string };
}) {
  const project = await getProject(params.id);
  if (!project) return notFound();

  return (
    <div>
      <h1>{project.name}</h1>
      <TaskBoard projectId={project.id} />
    </div>
  );
}
```

但有些页面你不能缓存整个 HTML -- 比如包含用户个人信息的头部导航。这时候就需要动态渲染：

```tsx
// components/user-nav.tsx
// 导出 dynamic = "force-dynamic" 让这个组件始终动态渲染
export const dynamic = "force-dynamic";

export async function UserNav() {
  // 这里每次请求都会重新执行，不受 Full Route Cache 影响
  const session = await auth();

  return (
    <nav>
      <span>{session?.user?.name}</span>
    </nav>
  );
}
```

Full Route Cache 有一个容易混淆的地方：**它和 Data Cache 是独立的**。即使你的数据被缓存了，页面的 HTML 仍然可能被重新渲染（如果 Full Route Cache 没命中）。反过来，即使 HTML 被缓存了，里面的数据获取逻辑在缓存失效时也会重新执行。

## 第四层：Router Cache -- 客户端的 prefetch 缓存

前三层都在服务端。Router Cache 是 Next.js 在客户端做的缓存，主要针对导航场景。

当用户在你的任务管理平台里点击侧边栏的"项目列表"时，如果之前已经访问过这个页面，Next.js 不会重新请求服务端，而是直接从客户端缓存中恢复页面。这就是 Router Cache。

```tsx
// components/sidebar.tsx
// NextLink 默认会 prefetch 链接指向的页面
// 用户鼠标悬停时就会在后台加载
import Link from "next/link";

export function Sidebar() {
  return (
    <nav>
      {/* 用户鼠标悬停时，Next.js 会 prefetch /projects 的 RSC Payload */}
      <Link href="/projects">项目列表</Link>
      <Link href="/settings">工作空间设置</Link>
      {/* 如果不希望 prefetch，可以显式关闭 */}
      <Link href="/notifications" prefetch={false}>
        通知中心
      </Link>
    </nav>
  );
}
```

Router Cache 有两个关键参数需要了解：

- **prefetch 行为**：默认情况下，`next/link` 会在用户悬停时 prefetch 页面的 RSC Payload（不包含数据，只包含组件结构和布局），并在用户点击时 prefetch 完整数据。如果你用的是动态渲染页面，可以设置 `prefetch={false}` 来避免不必要的请求。
- **缓存时长**：客户端的 Router Cache 在以下情况下会失效：用户刷新页面、使用浏览器前进/后退按钮（后退会显示 30 秒内的缓存）、或者调用 `router.refresh()`。

## 按需失效：revalidateTag 和 revalidatePath

缓存的真正难点不在于"怎么存"，而在于"什么时候失效"。

假设你的团队任务管理平台中，项目经理创建了一个新项目。如果项目列表页被缓存了 5 分钟，那在这 5 分钟内其他人都看不到新项目 -- 这在某些业务场景下是不可接受的。

Next.js 提供了两个函数来按需失效缓存：

```tsx
// app/(dashboard)/projects/actions.ts
"use server";

import { revalidateTag, revalidatePath } from "next/cache";
import { prisma } from "@/lib/prisma";
import { auth } from "@/lib/auth";

/**
 * 创建新项目
 * 关键步骤：数据变更后，立即让相关缓存失效
 */
export async function createProject(formData: FormData) {
  const session = await auth();
  if (!session?.user) {
    return { error: "未登录" };
  }

  const name = formData.get("name") as string;
  const description = formData.get("description") as string;

  // 1. 写入数据库
  const project = await prisma.project.create({
    data: {
      name,
      description,
      workspaceId: formData.get("workspaceId") as string,
      members: {
        create: {
          userId: session.user.id,
          role: "OWNER",
        },
      },
    },
  });

  // 2. 让带 "projects" 标签的所有缓存失效
  // 这会让 getCachedProjects()、项目列表页的 Full Route Cache 全部失效
  revalidateTag("projects");

  // 3. 也可以让特定路径的缓存失效
  // revalidatePath("/projects");
  // revalidatePath(`/projects/${project.id}`);

  return { success: true, projectId: project.id };
}
```

`revalidateTag` 和 `revalidatePath` 的区别：

- **revalidateTag**：按标签失效，影响所有带该标签的 Data Cache 和 Full Route Cache。适合"某个实体变了，所有相关的页面都要更新"的场景。比如任务状态变了，项目列表的任务计数、看板页的任务卡片、统计页的数据都要更新。
- **revalidatePath**：按路径失效，只影响特定 URL 的 Full Route Cache。适合"我知道哪个页面需要更新"的场景。比如更新了项目设置，只需要刷新 `/projects/[id]/settings` 这个页面。

在实际项目中，这两个函数通常组合使用。我的一般原则是：**用 tag 管理数据层面的缓存失效，用 path 管理页面层面的缓存失效**。

## 把四层缓存串起来：一个完整的请求链路

说了这么多层，我们用一个完整的例子把四层缓存串起来。假设用户打开团队任务管理平台的项目列表页，看看每一层缓存分别在什么时候发挥作用：

```
用户点击"项目列表"链接
  |
  v
[Router Cache] 客户端有没有缓存？
  |-- 有 --> 直接恢复页面，结束（30ms）
  |-- 没有 --> 继续往下
  |
  v
[Full Route Cache] 服务端有没有这个页面的 HTML + RSC Payload？
  |-- 有且未过期 --> 直接返回，结束（5ms）
  |-- 有但过期了 --> 返回旧内容 + 后台重新渲染
  |-- 没有 --> 继续往下
  |
  v
执行 React 服务端渲染
  |
  v
[Data Cache] getCachedProjects() 的缓存有没有命中？
  |-- 有 --> 直接用缓存数据（跳过数据库查询）
  |-- 没有 --> 查询数据库
  |
  v
[Request Memoization] 渲染过程中如果有重复的数据请求？
  |-- 有 --> 用同一请求周期内的缓存结果
  |-- 没有 --> 正常执行
  |
  v
返回完整的 HTML 给用户
```

这个链路解释了一个重要的事情：**每一层缓存都是上一层的补充，不是替代关系**。Router Cache 命中了就不需要走服务端，Full Route Cache 命中了就不需要重新渲染，Data Cache 命中了就不需要查数据库，Request Memoization 命中了就不需要重复请求。四层加在一起，才能把性能压到极致。

## 缓存策略选择：不同场景用不同策略

不是所有数据都适合缓存。我们团队任务管理平台里的各类数据，我会这样选择：

**适合长时间缓存的数据**（revalidate: 3600 或更长）：

- 工作空间设置 -- 几乎不变，改了用 revalidateTag 手动失效
- 项目模板列表 -- 低频变更
- 用户角色和权限配置 -- 变更频率极低

```tsx
// lib/data.ts
const getCachedWorkspaceSettings = unstable_cache(
  async (workspaceId: string) => {
    return prisma.workspaceSettings.findUnique({
      where: { workspaceId },
    });
  },
  ["workspace-settings"],
  {
    tags: ["workspace-settings"],
    revalidate: 3600, // 1 小时，变更时手动 revalidateTag
  }
);
```

**适合短时间缓存的数据**（revalidate: 60~300）：

- 项目列表 -- 偶尔有新项目创建或项目信息更新
- 任务看板 -- 任务状态在变化，但几秒的延迟可以接受
- 通知列表 -- 不断有新通知，但不需要秒级实时

```tsx
// lib/data.ts
const getCachedTasks = unstable_cache(
  async (projectId: string, status?: string) => {
    return prisma.task.findMany({
      where: {
        projectId,
        ...(status ? { status } : {}),
      },
      orderBy: { createdAt: "desc" },
    });
  },
  ["project-tasks"],
  {
    tags: ["tasks"], // 创建/更新任务时 revalidateTag("tasks")
    revalidate: 60, // 1 分钟，够用了
  }
);
```

**完全不应该缓存的数据**（cache: "no-store"）：

- 实时通知推送 -- 必须是最新的
- 在线状态 -- 用户登录登出要即时反映
- 表单提交结果 -- 提交后要看到最新状态

```tsx
// app/(dashboard)/notifications/page.tsx
import { prisma } from "@/lib/prisma";
import { auth } from "@/lib/auth";

// 告诉 Next.js：这个页面不要缓存，每次都动态渲染
export const dynamic = "force-dynamic";

export default async function NotificationsPage() {
  const session = await auth();
  if (!session?.user) return null;

  // 实时数据，不缓存，每次都查数据库
  const notifications = await prisma.notification.findMany({
    where: {
      userId: session.user.id,
      read: false,
    },
    orderBy: { createdAt: "desc" },
    take: 50,
  });

  return (
    <div>
      {notifications.map((n) => (
        <div key={n.id}>{n.content}</div>
      ))}
    </div>
  );
}
```

## 缓存的坑：雪崩和穿透

缓存用好了是利器，用不好会变成灾难。两个最常见的问题：

### 缓存雪崩

假设你的项目列表页设置了 `revalidate: 300`（5 分钟）。中午 12:00 高峰期，大量用户同时访问，恰好此时缓存过期。结果所有请求同时打到数据库，数据库瞬间被压垮 -- 这就是缓存雪崩。

解决思路是给缓存加随机偏移，避免大量缓存同时过期：

```tsx
// lib/data.ts
/**
 * 给 revalidate 时间加上随机偏移（正负 10%）
 * 避免大量缓存在同一时刻过期导致数据库被压垮
 */
function addJitter(baseSeconds: number): number {
  const jitter = baseSeconds * 0.1; // 10% 的偏移
  const randomOffset = Math.floor(Math.random() * 2 * jitter - jitter);
  return baseSeconds + randomOffset;
}

const getCachedProjectList = unstable_cache(
  async (workspaceId: string) => {
    return prisma.project.findMany({
      where: { workspaceId },
      orderBy: { updatedAt: "desc" },
    });
  },
  ["project-list"],
  {
    tags: ["projects"],
    // 基础 300 秒 + 随机偏移，实际在 270~330 秒之间
    revalidate: addJitter(300),
  }
);
```

### 缓存穿透

另一个问题是缓存穿透：用户请求一个不存在的项目 ID，缓存里没有，数据库里也没有。如果有恶意用户大量请求不存在的 ID，每个请求都会穿透到数据库层。

解决思路是对"不存在"的结果也做缓存（但缓存时间短一些），或者用参数校验把明显无效的请求拦在入口：

```tsx
// lib/data.ts
import { z } from "zod"; // 下一篇会详细讲 Zod

// 用 Zod 校验 ID 格式，明显不合法的直接拒绝
const projectIdSchema = z.string().cuid();

const getCachedProjectSafe = unstable_cache(
  async (projectId: string) => {
    // 先校验格式，不合法的请求直接返回 null，不打数据库
    if (!projectIdSchema.safeParse(projectId).success) {
      return null;
    }

    const project = await prisma.project.findUnique({
      where: { id: projectId },
    });

    // 如果数据库里也没有，返回 null
    // unstable_cache 会缓存这个 null 结果（直到 revalidate 过期）
    // 这样后续相同 ID 的请求不会穿透到数据库
    return project;
  },
  ["project-detail"],
  {
    tags: ["projects"],
    revalidate: 60, // 不存在的结果缓存 1 分钟就够了
  }
);
```

## Trade-off 分析

缓存永远是在"一致性"和"性能"之间做取舍。你需要根据业务场景来决定天平往哪边倾斜：

**用缓存的收益**：

- 数据库查询量大幅下降，我们的团队任务管理平台在引入缓存后，数据库 QPS 从峰值 200 降到了 30 左右
- 页面响应时间从平均 800ms 降到了 50ms（Full Route Cache 命中时）
- 服务器 CPU 使用率从 70% 降到了 20%，同样的机器能支撑更多用户

**用缓存的代价**：

- 数据一致性窗口期：缓存了 5 分钟的项目列表，意味着创建新项目后最多要等 5 分钟其他用户才能看到（除非你手动 revalidateTag）
- 调试复杂度增加：页面显示了旧数据，你很难快速判断是缓存问题还是数据问题。Next.js 的开发模式默认不缓存（`next dev` 下 revalidate 被忽略），所以有些缓存相关的 bug 只在生产环境才出现
- 缓存失效逻辑需要和维护：每次数据变更都要记得调用 revalidateTag/revalidatePath，漏掉一个就会导致用户看到过期数据。这在多人协作的项目中尤其容易出问题

我的建议是：**先用 Data Cache + revalidateTag 解决 80% 的场景，只有确认性能瓶颈后再考虑更激进的 Full Route Cache 策略**。宁可在缓存时间上保守一点（短一些），也不要为了性能牺牲数据一致性。

下一篇我们会聊一个和缓存紧密相关的话题：数据验证与类型安全。缓存能让你的数据获取变快，但如果缓存进来的数据本身就是脏的，那再快也没用 -- 更危险的是，你可能在不知不觉间把脏数据展示给了所有用户。下一篇我们用 Zod 来解决这个问题，让数据在进入系统之前就被校验干净。
