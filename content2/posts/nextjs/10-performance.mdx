## 从一个真实的性能事故说起

上周我们的团队任务管理平台上线了新版本，产品经理在群里甩了一张 Lighthouse 截图：性能评分 38 分。项目列表页首屏加载 3.2 秒，Dashboard 的图表区域白屏了将近 2 秒才出现，页面切换时还能明显感受到布局跳动。更致命的是，Google Search Console 发来了 Core Web Vitals 不达标的警告，直接影响搜索排名。

这些问题不是某一个人的锅——它们是功能迭代过程中一点一点积累出来的。图片越来越大、依赖越来越多、没人关心 Bundle 体积，直到用户开始抱怨"你们的页面怎么这么慢"。

上一篇我们聊了文件处理和媒体管理，里面提到了 `next/image` 做图片优化。那篇文章解决的是"怎么正确地处理文件"，而这一篇要解决一个更广泛的问题：**怎么让整个应用快起来**。性能优化不是锦上添花，它直接影响用户留存、SEO 排名和转化率。Google 的研究数据表明，页面加载时间每增加 1 秒，转化率就下降 7%。

接下来，我会用我们的团队任务管理平台作为贯穿案例，从诊断到优化，把 Next.js 性能优化的核心手段梳理一遍。

## 先诊断：Core Web Vitals 到底在说什么

在做任何优化之前，你得先知道问题出在哪里。Core Web Vitals 是 Google 定义的一组衡量用户体验的核心指标，也是 Lighthouse 评分的基础。对 Next.js 应用来说，你最需要关注三个指标：

**LCP（Largest Contentful Paint）**——最大内容绘制。用户打开页面后，视口内最大的那个元素（通常是图片或大标题）多久渲染出来。我们的项目列表页 LCP 是 3.1 秒，达标线是 2.5 秒。

**INP（Interaction to Next Paint）**——交互到下次绘制。用户点了按钮、输入了文字之后，页面多久给出视觉反馈。这个指标取代了之前的 FID，更全面地衡量交互响应性。我们的 Dashboard 页面因为图表组件太大，INP 高达 420ms，远超 200ms 的达标线。

**CLS（Cumulative Layout Shift）**——累积布局偏移。页面加载过程中，元素位置发生了多少意外移动。用户想点一个按钮，结果图片突然加载出来把按钮挤走了——这种体验非常糟糕。CLS 达标线是 0.1，我们的页面测出来是 0.18。

这三个指标不是孤立的，它们往往相互关联。比如图片没设尺寸导致 CLS 高，图片加载慢又拖慢了 LCP。所以优化的时候要系统性地看，不要只盯着一个指标。

## Bundle 分析：找到体积大户

代码优化第一步是搞清楚你的 JavaScript 到底有多大。Next.js 官方提供了 `@next/bundle-analyzer`，构建时能生成一个可视化的 Bundle 分析报告。

安装和配置很简单。先安装依赖：

```bash
npm install @next/bundle-analyzer
```

然后在 `next.config.ts` 中引入：

```typescript
// next.config.ts
import bundleAnalyzer from "@next/bundle-analyzer";

const nextConfig = {
  // ... 其他配置
};

// 用 withBundleAnalyzer 包裹配置
// 通过环境变量 ANALYZE 控制是否开启，日常开发不影响构建速度
const withBundleAnalyzer = bundleAnalyzer({
  enabled: process.env.ANALYZE === "true",
});

export default withBundleAnalyzer(nextConfig);
```

运行构建命令并开启分析：

```bash
ANALYZE=true npm run build
```

构建完成后会自动打开一个可视化页面，你能看到每个路由的 Bundle 体积，以及每个依赖占了多少空间。

跑了一次之后，我们的分析结果是这样的：

- `dashboard/page.js`：487KB（主要被 `recharts` 和 `date-fns` 撑大的）
- `projects/page.js`：312KB（`@tanstack/react-table` 贡献了 180KB）
- `layout.js`（全局共享）：156KB（`next-auth` + `tailwindcss`）
- 总体积超过 1MB，这还没算上字体的网络请求

看到这个结果，优化方向就很清楚了：Dashboard 的图表库和项目列表的表格库是最大的体积大户，需要做代码分割；日期库可以换成轻量替代品；一些低频功能（导出 Excel、批量操作）应该懒加载。

## 代码分割：不要让用户下载用不到的代码

代码分割的核心思想很简单：用户访问项目列表页，就不应该下载 Dashboard 图表的代码。Next.js 默认对每个路由做代码分割，但同一个页面里的组件默认还是打包在一起的，需要手动拆。

### 用 dynamic import 拆分重型组件

`next/dynamic` 是 Next.js 提供的动态导入工具，它基于 `React.lazy` 做了 SSR 兼容。来看看我们怎么用它优化 Dashboard 页面：

```typescript
// app/dashboard/page.tsx
import dynamic from "next/dynamic";
import { Suspense } from "react";

// 将图表组件动态导入，不参与首屏 Bundle
// ssr: false 表示这个组件只在客户端渲染，因为 recharts 依赖 DOM API
const TaskTrendChart = dynamic(
  () => import("@/components/dashboard/TaskTrendChart"),
  {
    ssr: false,
    // loading 组件在图表加载期间展示，避免白屏
    loading: () => (
      <div className="h-[300px] animate-pulse rounded-lg bg-gray-100" />
    ),
  }
);

// 成员工作负载图表也是同理，只在需要时才加载
const WorkloadChart = dynamic(
  () => import("@/components/dashboard/WorkloadChart"),
  {
    ssr: false,
    loading: () => (
      <div className="h-[250px] animate-pulse rounded-lg bg-gray-100" />
    ),
  }
);

// 导出 Excel 功能使用频率很低，更要懒加载
const ExportButton = dynamic(
  () => import("@/components/dashboard/ExportButton"),
  {
    ssr: false,
    // 这个组件比较小，用一个简单的 loading 就够了
    loading: () => <div className="h-10 w-24 animate-pulse rounded bg-gray-200" />,
  }
);

export default function DashboardPage() {
  return (
    <div className="space-y-6 p-6">
      <h1 className="text-2xl font-bold">Dashboard</h1>

      {/* 关键数据卡片直接渲染，不做懒加载 */}
      <Suspense fallback={<StatsSkeleton />}>
        <StatsCards />
      </Suspense>

      {/* 图表组件懒加载，首屏不会被拖慢 */}
      <div className="grid grid-cols-1 gap-6 lg:grid-cols-2">
        <TaskTrendChart />
        <WorkloadChart />
      </div>

      {/* 低频功能懒加载 */}
      <div className="flex justify-end">
        <ExportButton />
      </div>
    </div>
  );
}
```

做了这步之后，Dashboard 的首屏 Bundle 从 487KB 降到了 92KB，图表代码只在用户滚动到对应区域时才加载。

### 替换重型依赖

Bundle 分析还暴露了另一个问题：`date-fns` 全量引入占了 67KB。我们实际上只用了 `format`、`parseISO`、`differenceInDays` 这几个函数。

解决方案是改用 `date-fns` 的子路径导入，或者换成更轻量的 `dayjs`（仅 2KB）：

```typescript
// 之前：全量导入，整个 date-fns 都会被打包
import { format, parseISO, differenceInDays } from "date-fns";

// 方案一：子路径导入，只打包用到的函数（推荐）
import { format } from "date-fns/format";
import { parseISO } from "date-fns/parseISO";
import { differenceInDays } from "date-fns/differenceInDays";

// 方案二：换用 dayjs，体积更小但 API 略有不同
import dayjs from "dayjs";
import relativeTime from "dayjs/plugin/relativeTime";

dayjs.extend(relativeTime);
// dayjs("2026-03-23").fromNow() => "8 days ago"
```

类似的问题还有 lodash。如果你在用 lodash，务必用子路径导入：

```typescript
// 不好：整个 lodash 打包进去（约 72KB minified）
import _ from "lodash";
const debouncedSearch = _.debounce(searchHandler, 300);

// 正确：只导入 debounce（约 1.5KB）
import debounce from "lodash/debounce";
const debouncedSearch = debounce(searchHandler, 300);
```

### 条件加载：只给需要的人加载

有些组件只有特定角色的用户才看得到，那就没必要在首屏就加载。比如管理员面板：

```typescript
// components/AdminGuard.tsx
"use client";

import dynamic from "next/dynamic";
import { useSession } from "next-auth/react";

// 管理员面板代码量很大（约 200KB），但只有管理员才用得到
// 动态导入 + 条件渲染，普通用户完全不会下载这段代码
const AdminPanel = dynamic(
  () => import("@/components/admin/AdminPanel"),
  {
    loading: () => <div className="p-4 text-sm text-gray-500">加载管理面板...</div>,
  }
);

export function AdminGuard() {
  const { data: session, status } = useSession();

  // 身份还没确认之前不渲染任何东西
  if (status === "loading") return null;

  // 不是管理员就不加载 AdminPanel 组件
  if (session?.user?.role !== "ADMIN") return null;

  return <AdminPanel />;
}
```

注意这里的顺序：先判断权限，再决定是否渲染 `AdminPanel`。因为 `dynamic` 在组件被挂载时才开始加载代码，所以如果 `AdminGuard` 在非管理员场景下提前渲染了 `<AdminPanel />` 再 return null，那代码还是会被加载。

## 流式渲染：让页面逐步呈现

Next.js App Router 提供了 `Suspense` + `loading.tsx` 的流式渲染机制。它的核心思路是：页面不需要等所有数据都拿到才开始渲染，可以一部分一部分地展示给用户。

### 路由级 loading.tsx

每个路由目录下都可以放一个 `loading.tsx`，Next.js 会在该路由加载期间自动展示它：

```typescript
// app/projects/loading.tsx
// 这个组件作为项目列表页的即时 loading 状态
// Next.js 会在服务端流式传输 HTML 的过程中先把这个骨架屏推给浏览器
// 用户不需要等到数据全部返回就能看到页面结构
export default function ProjectsLoading() {
  return (
    <div className="space-y-6 p-6">
      {/* 模拟搜索栏 */}
      <div className="h-12 animate-pulse rounded-lg bg-gray-200" />

      {/* 模拟筛选条件 */}
      <div className="flex gap-3">
        {[1, 2, 3, 4].map((i) => (
          <div key={i} className="h-8 w-20 animate-pulse rounded-full bg-gray-200" />
        ))}
      </div>

      {/* 模拟项目卡片列表 */}
      <div className="grid grid-cols-1 gap-4 md:grid-cols-2 lg:grid-cols-3">
        {Array.from({ length: 6 }).map((_, i) => (
          <div key={i} className="space-y-3 rounded-lg border p-4">
            <div className="h-5 w-3/4 animate-pulse rounded bg-gray-200" />
            <div className="h-4 w-full animate-pulse rounded bg-gray-100" />
            <div className="h-4 w-2/3 animate-pulse rounded bg-gray-100" />
            <div className="flex gap-2 pt-2">
              <div className="h-6 w-16 animate-pulse rounded-full bg-gray-200" />
              <div className="h-6 w-16 animate-pulse rounded-full bg-gray-200" />
            </div>
          </div>
        ))}
      </div>
    </div>
  );
}
```

### 组件级 Suspense

`loading.tsx` 是路由级别的，但如果页面里有多个独立的数据区块，可以用 `Suspense` 做更细粒度的控制：

```typescript
// app/dashboard/page.tsx
import { Suspense } from "react";

// 这三个数据区块的数据获取互不依赖
// 用 Suspense 包裹后，它们会并行请求、独立渲染
// 最快返回的区块最先展示给用户，不用等其他区块

async function StatsCards() {
  // 这个接口通常很快（~200ms），因为有缓存
  const stats = await getDashboardStats();
  return (
    <div className="grid grid-cols-4 gap-4">
      <StatCard label="进行中" value={stats.active} />
      <StatCard label="本周完成" value={stats.completedThisWeek} />
      <StatCard label="逾期任务" value={stats.overdue} />
      <StatCard label="团队成员" value={stats.teamSize} />
    </div>
  );
}

async function RecentActivities() {
  // 活动列表数据量稍大，可能需要 ~500ms
  const activities = await getRecentActivities();
  return (
    <ul className="space-y-3">
      {activities.map((activity) => (
        <ActivityItem key={activity.id} activity={activity} />
      ))}
    </ul>
  );
}

async function MyTasks() {
  // 我负责的任务，涉及权限过滤，可能 ~800ms
  const tasks = await getMyTasks();
  return (
    <ul className="space-y-2">
      {tasks.map((task) => (
        <TaskItem key={task.id} task={task} />
      ))}
    </ul>
  );
}

export default function DashboardPage() {
  return (
    <div className="space-y-6 p-6">
      <h1 className="text-2xl font-bold">Dashboard</h1>

      {/* 三个区块并行加载，各自独立展示 loading 状态 */}
      {/* 统计卡片最快出来，用户立刻能看到关键数据 */}
      <Suspense fallback={<StatsSkeleton />}>
        <StatsCards />
      </Suspense>

      {/* 活动列表和任务列表谁先回来谁先显示 */}
      <div className="grid grid-cols-1 gap-6 lg:grid-cols-2">
        <Suspense fallback={<ActivitiesSkeleton />}>
          <RecentActivities />
        </Suspense>
        <Suspense fallback={<TasksSkeleton />}>
          <MyTasks />
        </Suspense>
      </div>
    </div>
  );
}
```

这种模式下，用户感知到的加载速度会快很多，因为页面在逐步"生长"出来，而不是等所有数据都拿到后才一次性渲染。

## 图片优化：LCP 和 CLS 的重灾区

我们在上一篇文件处理中简单用过 `next/image`，这里深入看看它的高级用法，尤其是和性能指标相关的部分。

### 项目封面图的优化

项目列表页每个卡片都有一张封面图，之前用的是原生 `<img>` 标签，没有指定尺寸，导致两个问题：图片加载慢（没有格式优化和响应式裁剪），图片加载后布局跳动（CLS 飙升）。

改造方案：

```typescript
// components/ProjectCard.tsx
import Image from "next/image";

interface ProjectCardProps {
  name: string;
  description: string;
  coverUrl: string;
  status: string;
  memberCount: number;
}

export function ProjectCard({
  name,
  description,
  coverUrl,
  status,
  memberCount,
}: ProjectCardProps) {
  return (
    <div className="group overflow-hidden rounded-xl border bg-white transition-shadow hover:shadow-lg">
      {/* 外层容器用 aspect-ratio 预留固定空间
          这样即使图片还没加载出来，布局也不会跳动
          这是解决 CLS 问题最直接有效的方法 */}
      <div className="relative aspect-[16/10] w-full overflow-hidden bg-gray-100">
        <Image
          src={coverUrl}
          alt={`${name} 的封面图`}
          fill
          // sizes 告诉浏览器在不同断点下图片应该多大
          // 浏览器据此选择最合适的图片尺寸，避免下载过大的图片
          sizes="(max-width: 768px) 100vw, (max-width: 1024px) 50vw, 33vw"
          // placeholder="blur" 需要配合 blurDataURL 使用
          // 在图片加载完成前展示模糊的缩略图，提升感知速度
          placeholder="blur"
          blurDataURL={coverUrl}
          className="object-cover transition-transform duration-300 group-hover:scale-105"
        />
      </div>

      <div className="p-4">
        <div className="flex items-center justify-between">
          <h3 className="font-semibold text-gray-900">{name}</h3>
          <span className="rounded-full bg-blue-50 px-2 py-1 text-xs text-blue-600">
            {status}
          </span>
        </div>
        <p className="mt-1 line-clamp-2 text-sm text-gray-500">{description}</p>
        <p className="mt-2 text-xs text-gray-400">{memberCount} 位成员</p>
      </div>
    </div>
  );
}
```

### 首屏大图：priority 属性

如果页面的 LCP 元素是一张大图（比如项目详情页的 Banner），需要用 `priority` 属性告诉浏览器这张图很重要：

```typescript
// app/projects/[id]/page.tsx
import Image from "next/image";

// 项目详情页的 Banner 图是页面上最大的元素，就是 LCP 元素
// 加 priority 后，Next.js 会：
// 1. 给这张图添加 fetchpriority="high"
// 2. 禁用懒加载，图片会预加载
// 3. 自动生成 preload link 标签
export default async function ProjectDetailPage({
  params,
}: {
  params: { id: string };
}) {
  const project = await getProject(params.id);

  return (
    <article>
      {/* LCP 元素，必须优先加载 */}
      <div className="relative aspect-[21/9] w-full">
        <Image
          src={project.bannerUrl}
          alt={`${project.name} 的 Banner`}
          fill
          priority // 这一行是关键，把 LCP 从 3.1s 降到 1.8s
          sizes="100vw"
          className="object-cover"
        />
      </div>

      <div className="mx-auto max-w-4xl p-6">
        <h1 className="text-3xl font-bold">{project.name}</h1>
        <p className="mt-2 text-gray-600">{project.description}</p>
      </div>
    </article>
  );
}
```

## 字体优化：别让字体阻塞渲染

自定义字体如果处理不好，会导致两种问题：FOIT（Flash of Invisible Text，字体加载前文字不可见）和 FOUT（Flash of Unstyled Text，字体加载前用系统字体显示，加载后突然切换）。

`next/font` 能彻底解决这些问题——它在构建时就把字体文件下载好，内联到 CSS 中，浏览器不需要额外发起网络请求。

```typescript
// app/layout.tsx
import { Inter, Noto_Sans_SC } from "next/font/google";

// Inter 用于英文和数字，Noto Sans SC 用于中文
// next/font 会在构建时下载这些字体，生成自托管版本
// 不会有额外的网络请求，也不会产生 CLS（因为字体尺寸在构建时就算好了）
const inter = Inter({
  subsets: ["latin"], // 只加载拉丁字符集，减少字体体积
  display: "swap", // 在自定义字体加载完成前先用系统字体显示，避免文字不可见
  variable: "--font-inter", // 注入为 CSS 变量，可以在 Tailwind 中使用
});

const notoSansSC = Noto_Sans_SC({
  subsets: ["latin"], // next/font 会自动处理中文字符子集化
  weight: ["400", "500", "700"], // 只加载用到的字重，中文字体每个字重都有几 MB
  display: "swap",
  variable: "--font-noto",
});

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html
      lang="zh-CN"
      className={`${inter.variable} ${notoSansSC.variable}`}
    >
      <body className="font-sans antialiased">
        {children}
      </body>
    </html>
  );
}
```

对应地，在 `tailwind.config.ts` 中配置字体变量：

```typescript
// tailwind.config.ts
import type { Config } from "tailwindcss";

const config: Config = {
  theme: {
    fontFamily: {
      // 引用 next/font 注入的 CSS 变量
      // 这样 Tailwind 的 font-sans 类就会使用我们配置的字体
      sans: [
        "var(--font-inter)",
        "var(--font-noto)",
        "system-ui",
        "sans-serif",
      ],
    },
  },
};

export default config;
```

关键点在于 `weight` 参数——中文字体非常大，每个字重大约 4-8MB。如果只用了 400、500、700 三个字重，就不要加载全部字重。另外 `next/font` 对 Google 字体做了自动子集化，只打包页面中实际用到的字符，所以中文字体的实际传输体积远小于完整字体文件。

## 渲染优化：减少不必要的重渲染

代码分割解决的是"下载多少代码"的问题，而渲染优化解决的是"代码下载后跑得多快"的问题。对于客户端组件，React 的重渲染机制如果不注意，很容易造成性能浪费。

### React.memo：避免组件无条件重渲染

假设我们的项目列表有一个筛选功能，用户选择状态筛选后，列表会重新渲染。如果每个 `ProjectCard` 都用了 `React.memo`，只有数据发生变化的卡片才会重渲染：

```typescript
// components/ProjectCard.tsx
```

### useMemo 和 useCallback：稳定引用

`React.memo` 有一个前提条件：传给子组件的 props 引用必须是稳定的。如果每次父组件渲染都创建新的对象或函数，`memo` 就会失效。这时候就需要 `useMemo` 和 `useCallback`：

```typescript
// components/ProjectFilters.tsx
"use client";

import { useState, useMemo, useCallback } from "react";
import { ProjectCard } from "./ProjectCard";

interface Project {
  id: string;
  name: string;
  description: string;
  coverUrl: string;
  status: string;
  memberCount: number;
}

interface ProjectFiltersProps {
  projects: Project[];
}

export function ProjectFilters({ projects }: ProjectFiltersProps) {
  const [search, setSearch] = useState("");
  const [statusFilter, setStatusFilter] = useState<string>("all");

  // useMemo 缓存筛选结果
  // 只有 projects、search、statusFilter 变化时才重新计算
  // 如果用户只是在搜索框里打字但还没触发筛选，这个计算就不会执行
  const filteredProjects = useMemo(() => {
    return projects.filter((project) => {
      const matchesSearch = project.name
        .toLowerCase()
        .includes(search.toLowerCase());
      const matchesStatus =
        statusFilter === "all" || project.status === statusFilter;
      return matchesSearch && matchesStatus;
    });
  }, [projects, search, statusFilter]);

  // useCallback 缓存事件处理函数
  // 如果不缓存，每次 ProjectFilters 重新渲染都会创建新的函数
  // 传给子组件的 onChange prop 就会变化，导致子组件也重新渲染
  const handleSearchChange = useCallback(
    (e: React.ChangeEvent<HTMLInputElement>) => {
      setSearch(e.target.value);
    },
    []
  );

  const handleStatusChange = useCallback(
    (e: React.ChangeEvent<HTMLSelectElement>) => {
      setStatusFilter(e.target.value);
    },
    []
  );

  return (
    <div>
      <div className="flex gap-4">
        <input
          type="text"
          placeholder="搜索项目..."
          value={search}
          onChange={handleSearchChange}
          className="rounded-lg border px-4 py-2"
        />
        <select
          value={statusFilter}
          onChange={handleStatusChange}
          className="rounded-lg border px-4 py-2"
        >
          <option value="all">全部状态</option>
          <option value="active">进行中</option>
          <option value="completed">已完成</option>
          <option value="archived">已归档</option>
        </select>
      </div>

      <div className="mt-6 grid grid-cols-1 gap-4 md:grid-cols-2 lg:grid-cols-3">
        {filteredProjects.map((project) => (
          <ProjectCard key={project.id} {...project} />
        ))}
      </div>
    </div>
  );
}
```

这里有一个容易踩的坑：`useMemo` 和 `useCallback` 本身也有开销（需要做依赖比较）。如果计算本身很轻量（比如简单的字符串拼接），或者子组件很简单，用 `useMemo` 反而会更慢。所以只在有明显性能问题时才使用，不要到处加。

### 自定义性能上报

如果不用 Vercel，或者需要更细粒度的监控，可以自己实现 Web Vitals 上报：

```typescript
// lib/performance-monitor.ts
// 自定义性能上报逻辑，将 Web Vitals 数据发送到自己的后端
// 这样可以把性能数据和业务数据关联起来，做更深入的分析

type MetricName = "LCP" | "INP" | "CLS" | "FCP" | "TTFB";

interface PerformanceMetric {
  name: MetricName;
  value: number;
  rating: "good" | "needs-improvement" | "poor";
  url: string;
  timestamp: number;
}

export function reportMetric(metric: PerformanceMetric) {
  // 使用 navigator.sendBeacon 而不是 fetch
  // sendBeacon 在页面关闭时也能可靠发送，不会阻塞页面卸载
  if (typeof navigator !== "undefined" && navigator.sendBeacon) {
    navigator.sendBeacon(
      "/api/metrics",
      JSON.stringify({
        ...metric,
        timestamp: Date.now(),
        url: window.location.href,
      })
    );
  }
}

// 使用 web-vitals 库采集指标
// npm install web-vitals
import { onLCP, onINP, onCLS, onFCP, onTTFB } from "web-vitals";

// 在客户端组件中初始化监控
export function initPerformanceMonitoring() {
  // onLCP 在 LCP 值确定时回调
  onLCP((metric) => {
    reportMetric({
      name: "LCP",
      value: Math.round(metric.value),
      rating: metric.rating,
      url: location.href,
      timestamp: Date.now(),
    });
  });

  onINP((metric) => {
    reportMetric({
      name: "INP",
      value: Math.round(metric.value),
      rating: metric.rating,
      url: location.href,
      timestamp: Date.now(),
    });
  });

  onCLS((metric) => {
    // CLS 的值是 0-1 之间的小数，乘以 1000 便于存储和展示
    reportMetric({
      name: "CLS",
      value: Math.round(metric.value * 1000),
      rating: metric.rating,
      url: location.href,
      timestamp: Date.now(),
    });
  });
}
```

## 优化的优先级：先做什么

上面讲了这么多优化手段，但你不可能一口气全做。实际工作中，建议按下面的优先级来：

**第一优先级：影响最大的问题先解决。** 跑一次 Lighthouse，看哪个指标得分最低。通常 LCP 是最容易出问题的——检查首屏大图是否用了 `priority`、图片格式是否优化、是否有阻塞渲染的大脚本。

**第二优先级：Bundle 体积。** 跑一次 `@next/bundle-analyzer`，看看有没有不该出现的依赖。替换重型库、做代码分割，这一步投入产出比最高。

**第三优先级：流式渲染。** 检查页面里是否有可以并行加载的数据区块，用 `Suspense` 包裹。`loading.tsx` 骨架屏几乎是零成本的优化，但用户感知的提升非常明显。

**第四优先级：渲染层面的优化。** `React.memo`、`useMemo`、`useCallback` 这些是锦上添花，只在确认了性能瓶颈之后再使用。过早优化反而会让代码更复杂。

**第五优先级：监控体系。** 性能优化是一个持续的过程，搭建好监控后才能发现问题、衡量效果。如果条件允许，尽早接入。

## Trade-off 分析

性能优化不是免费的，每个优化手段都有代价：

**代码分割增加了复杂度。** `dynamic import` 会引入异步加载状态，你需要处理 loading、error、超时等情况。如果分割粒度太细，HTTP 请求数会增多，反而可能拖慢首屏（虽然 HTTP/2 多路复用缓解了这个问题，但每个请求仍然有建立连接和服务器调度的开销）。

**`React.memo` 不是万能药。** 它只做浅比较，如果 props 是嵌套对象，浅比较会失效，需要配合 `useMemo`。滥用 `memo` 会增加内存开销和比较开销，简单的组件用了反而更慢。

**骨架屏需要维护。** 每个页面的 loading 状态都要和实际布局保持一致，页面改版时骨架屏也要跟着改。如果不一致，反而会产生 CLS。

**`next/font` 不支持所有字体。** 它主要支持 Google Fonts 和本地字体文件。如果用了特殊的字体服务（比如 Adobe Fonts），可能需要用传统的 `@font-face` 方案。

**过度追求性能分数会损害开发体验。** 为了把 Lighthouse 分数从 95 提到 100，可能要花大量时间做微优化，但这些时间本可以用来开发新功能。90 分以上的页面，用户感知的差异已经很小了。

## 小结

回顾一下我们做了什么：从 Lighthouse 38 分到 92 分，主要手段是代码分割（`dynamic import` 替换重型组件）、流式渲染（`Suspense` + `loading.tsx`）、图片优化（`next/image` + `priority` + `sizes`）、字体优化（`next/font` 自托管）和依赖瘦身（子路径导入）。

性能优化的核心思路是：先诊断，找瓶颈；再优化，看数据；最后建监控，防退化。不要凭感觉优化，要用数据说话。

到这里，我们的任务管理平台功能已经比较完整了：路由、组件、数据获取、缓存、验证、认证、文件处理、性能优化都覆盖了。但有一个问题一直没解决——我们怎么确保这些功能不出 bug？下一篇我们会聊测试策略，从单元测试到 E2E 测试，建立一套可靠的测试体系来守护代码质量。
