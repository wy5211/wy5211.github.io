上一篇文章里，我们用 Zod 给任务管理平台加了一层"安检门"——所有进入系统的数据都会被严格校验，格式不对、类型不对直接拦住。但有一个更根本的问题还没解决：谁来证明"我是我"？换句话说，目前任何人在浏览器里输入地址就能看到所有任务、创建新项目、修改别人的工作。这在真实生产环境中是完全不可接受的。

今天我们就来给平台加上身份验证和权限控制。具体来说，我们要做三件事：让用户可以通过 GitHub 一键登录（面向开发者团队很常见），也可以用邮箱密码注册登录（面向非技术成员），然后根据用户的角色决定他能看到什么、操作什么。我们用的核心工具是 Auth.js（前身是 NextAuth.js），它是 Next.js 生态中最成熟的认证方案。

## 为什么需要认证，而不是自己手写

你可能会想：认证不就是查一下数据库、比对一下密码、设个 cookie 吗？自己写不行吗？

当然可以，但你需要自己处理这些事情：

- 密码的安全哈希和比对（bcrypt 还是 argon2？rounds 设多少？）
- Session 的生成、存储、刷新、过期（JWT 还是 database session？过期时间多长？）
- CSRF 防护、Session Fixation 防护
- OAuth 握手流程（授权码、token 交换、刷新 token）
- 多种登录方式（GitHub、Google、邮箱密码）的统一管理
- 数据库适配（用户表、账户表、Session 表的结构设计）

Auth.js 把这些都封装好了。你只需要配置 provider、写几个 callback，就能得到一个生产级别的认证系统。省下来的时间可以用来做业务逻辑。

## Auth.js 核心概念

在写代码之前，先搞清楚几个关键概念。

Auth.js 的工作流程其实很简单：

```
用户点击"GitHub 登录"
       |
       v
浏览器跳转到 GitHub 授权页面
       |
       v
用户在 GitHub 同意授权，GitHub 回调你的应用并带上 code
       |
       v
Auth.js 用 code 换取 access_token，再拿到用户信息
       |
       v
Auth.js 创建 Session（JWT 或数据库记录），写入 cookie
       |
       v
后续请求自动携带 cookie，Auth.js 解析出当前用户
```

对于邮箱密码登录，流程更短：用户提交表单，你在 `authorize` 函数里查数据库比对密码，通过就创建 Session。

Auth.js 有两个核心配置项决定它如何存储用户和 Session：

- `adapter`：负责把用户数据存到数据库（Prisma、Drizzle、MongoDB 等），不配置 adapter 就只能用 OAuth，用户信息存在 JWT 里
- `session.strategy`：`"jwt"` 把 Session 信息编码进 cookie（无状态，不需要数据库查 session），`"database"` 在数据库里存 session 记录（可以主动让 session 失效）

对于我们的任务管理平台，选择 JWT 策略更合适——不需要频繁查询 session 表，而且平台规模不会大到需要强制踢人下线。

## 数据模型

先定义 Prisma Schema。Auth.js 的 Prisma Adapter 需要几个标准表，我们加上自己的角色字段和团队关联。

```prisma
// prisma/schema.prisma

model User {
  id            String    @id @default(uuid())
  name          String?
  email         String    @unique
  emailVerified DateTime?
  image         String?
  password      String?   // OAuth 登录的用户没有密码，所以是可选的
  role          Role      @default(MEMBER)

  // Auth.js 标准关联
  accounts      Account[]
  sessions      Session[]

  // 业务关联
  assignments   TaskAssignment[]

  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  @@map("users")
}

model Account {
  id                String  @id @default(uuid())
  userId            String
  type              String  // 比如 "oauth"
  provider          String  // 比如 "github"
  providerAccountId String  // GitHub 返回的用户 ID
  refresh_token     String?
  access_token      String?
  expires_at        Int?
  token_type        String?
  scope             String?
  id_token          String?

  user              User    @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([provider, providerAccountId])
  @@map("accounts")
}

model Session {
  id           String   @id @default(uuid())
  sessionToken String   @unique
  userId       String
  expires      DateTime

  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@map("sessions")
}

// 角色枚举：管理员、项目经理、普通成员
enum Role {
  ADMIN       // 管理员：管理团队成员、系统设置
  MANAGER     // 项目经理：创建项目、分配任务、查看所有任务
  MEMBER      // 普通成员：只看自己被分配的任务、更新任务状态
}
```

这里有几个值得注意的设计决策：

- `password` 字段是可选的，因为通过 GitHub 登录的用户不需要密码
- `Account` 表用来关联第三方登录，一个用户可以绑定多个 OAuth 账号
- 角色只设了三个级别，实际项目中你可能需要更细粒度的权限，但三级别对大多数中小团队已经足够

运行迁移后，数据库里就有了完整的用户体系基础。

## 配置 Auth.js

接下来是核心配置文件。Auth.js v5（也就是 Auth.js）的配置方式和 v4（NextAuth.js）有些不同，我们用的是最新版本。

```typescript
// auth.ts（项目根目录，不是 app/ 下）

import NextAuth from "next-auth";
import GitHub from "next-auth/providers/github";
import Credentials from "next-auth/providers/credentials";
import { PrismaAdapter } from "@auth/prisma-adapter";
import { prisma } from "@/lib/prisma";
import { hash, compare } from "bcryptjs";

// 扩展 Session 和 JWT 类型，把 role 和 id 加进去
declare module "next-auth" {
  interface Session {
    user: {
      id: string;
      email: string;
      name: string | null;
      image: string | null;
      role: "ADMIN" | "MANAGER" | "MEMBER";
    };
  }
  interface User {
    role: "ADMIN" | "MANAGER" | "MEMBER";
  }
}

declare module "next-auth/jwt" {
  interface JWT {
    id: string;
    role: "ADMIN" | "MANAGER" | "MEMBER";
  }
}

export const { handlers, auth, signIn, signOut } = NextAuth({
  // 使用 Prisma 适配器，Auth.js 会自动处理用户和账户的 CRUD
  adapter: PrismaAdapter(prisma),

  // 使用 JWT 策略：session 信息编码在 cookie 中，无状态，不需要查数据库
  session: {
    strategy: "jwt",
    maxAge: 30 * 24 * 60 * 60, // 30 天过期
  },

  // 自定义页面路径（不配置就用 Auth.js 内置的默认页面）
  pages: {
    signIn: "/login",
    error: "/login", // 登录失败也回到登录页，通过 URL 参数传递错误信息
  },

  providers: [
    // GitHub OAuth 登录
    GitHub({
      clientId: process.env.GITHUB_CLIENT_ID!,
      clientSecret: process.env.GITHUB_CLIENT_SECRET!,
      // 请求的权限范围：读取用户基本信息和邮箱
      authorization: {
        params: {
          scope: "read:user user:email",
        },
      },
    }),

    // 邮箱密码登录
    Credentials({
      name: "credentials",
      credentials: {
        email: { label: "邮箱", type: "email" },
        password: { label: "密码", type: "password" },
      },
      // authorize 函数是凭证登录的核心：验证用户身份，返回用户对象
      async authorize(credentials) {
        // 基本校验：确保邮箱和密码都不为空
        if (!credentials?.email || !credentials?.password) {
          return null;
        }

        // 从数据库查找用户，密码字段也一并查出（默认 select 不含密码）
        const user = await prisma.user.findUnique({
          where: { email: credentials.email as string },
          select: {
            id: true,
            email: true,
            name: true,
            image: true,
            role: true,
            password: true, // 需要 password 来比对
          },
        });

        // 用户不存在或没有设置密码（可能是纯 OAuth 用户）
        if (!user || !user.password) {
          return null;
        }

        // 用 bcrypt 比对密码（compare 会自动提取 salt 并计算哈希）
        const isValid = await compare(
          credentials.password as string,
          user.password
        );

        if (!isValid) {
          return null;
        }

        // 验证通过，返回用户对象（不要返回 password 字段）
        const { password: _, ...safeUser } = user;
        return safeUser;
      },
    }),
  ],

  callbacks: {
    // jwt callback：每次生成或刷新 JWT 时调用
    // 用户首次登录时，user 对象存在，我们把 role 和 id 写入 token
    async jwt({ token, user, trigger }) {
      if (user) {
        // 首次登录，把数据库中的 role 和 id 注入到 JWT
        token.id = user.id!;
        token.role = user.role;
      }

      // session 被刷新时（比如用户角色被管理员修改了）
      // 重新从数据库读取最新的 role，保证权限实时生效
      if (trigger === "update") {
        const freshUser = await prisma.user.findUnique({
          where: { id: token.id as string },
          select: { role: true },
        });
        if (freshUser) {
          token.role = freshUser.role;
        }
      }

      return token;
    },

    // session callback：每次调用 auth() 或 useSession() 时触发
    // 把 JWT 中的信息映射到 session 对象上
    async session({ session, token }) {
      if (session.user) {
        session.user.id = token.id as string;
        session.user.role = token.role as "ADMIN" | "MANAGER" | "MEMBER";
      }
      return session;
    },
  },
});
```

然后创建 API Route Handler，让 Auth.js 的 HTTP 端点生效。

## 邮箱密码认证：注册

凭证登录需要一个注册流程。注册接口需要做好输入校验（刚好用上上一篇学的 Zod）和密码哈希。

```typescript
// app/api/auth/register/route.ts

import { NextResponse } from "next/server";
import { z } from "zod";
import { prisma } from "@/lib/prisma";
import { hash } from "bcryptjs";

// 注册数据的校验规则
const registerSchema = z.object({
  name: z
    .string()
    .min(2, "姓名至少 2 个字符")
    .max(50, "姓名不能超过 50 个字符"),
  email: z.string().email("请输入有效的邮箱地址"),
  password: z
    .string()
    .min(8, "密码至少 8 个字符")
    .regex(/[A-Z]/, "密码需要包含至少一个大写字母")
    .regex(/[a-z]/, "密码需要包含至少一个小写字母")
    .regex(/[0-9]/, "密码需要包含至少一个数字"),
});

export async function POST(request: Request) {
  try {
    const body = await request.json();
    const result = registerSchema.safeParse(body);

    if (!result.success) {
      // 返回字段级别的错误信息，前端可以精确提示
      return NextResponse.json(
        {
          error: "输入信息不正确",
          details: result.error.flatten().fieldErrors,
        },
        { status: 400 }
      );
    }

    const { name, email, password } = result.data;

    // 检查邮箱是否已被注册
    const existingUser = await prisma.user.findUnique({
      where: { email },
    });

    if (existingUser) {
      return NextResponse.json({ error: "该邮箱已被注册" }, { status: 409 });
    }

    // 用 bcrypt 哈希密码，rounds 设为 12（安全性和性能的平衡点）
    const hashedPassword = await hash(password, 12);

    // 创建用户，默认角色是 MEMBER
    const user = await prisma.user.create({
      data: {
        name,
        email,
        password: hashedPassword,
        role: "MEMBER",
      },
      select: {
        id: true,
        name: true,
        email: true,
        role: true,
      },
    });

    return NextResponse.json({ message: "注册成功", user }, { status: 201 });
  } catch (error) {
    // 生产环境应该用日志系统记录，不要暴露堆栈信息
    console.error("注册失败:", error);
    return NextResponse.json({ error: "服务器内部错误" }, { status: 500 });
  }
}
```

注册接口里有两个关键操作：用 Zod 校验输入（和上一篇的知识衔接上了），用 bcrypt 哈希密码。`hash(password, 12)` 中的 12 是 cost factor，值越大越安全但也越慢。12 是目前业界推荐的默认值，大约需要 250ms 左右。

## 邮箱密码认证：登录

登录页面需要处理表单提交和错误展示。我们用 Server Action 配合 `useFormState` 来管理状态。

```tsx
// components/email-login-form.tsx

"use client";

import { useState } from "react";
import { useFormState } from "react-dom";
import { signIn } from "@/auth";

// Server Action：处理凭证登录
async function loginAction(
  _prevState: { error: string | null },
  formData: FormData
) {
  const email = formData.get("email") as string;
  const password = formData.get("password") as string;

  try {
    // signIn 会跳转到 Auth.js 的 authorize 函数
    // redirectTo 指定登录成功后跳转的页面
    await signIn("credentials", {
      email,
      password,
      redirectTo: "/dashboard",
    });
  } catch (error) {
    // signIn 在凭证无效时会抛出错误（而不是返回）
    // 我们需要捕获它并返回错误信息
    // 如果错误是 NEXT_REDIRECT，说明登录成功正在跳转，不要处理
    if (error instanceof Error && error.message === "NEXT_REDIRECT") {
      throw error;
    }
    return { error: "邮箱或密码不正确" };
  }

  // 这行理论上不会执行，因为 signIn 要么跳转要么抛错
  return { error: null };
}

export function EmailLoginForm({ callbackUrl }: { callbackUrl?: string }) {
  const [state, formAction] = useFormState(loginAction, { error: null });

  return (
    <form action={formAction} className="space-y-4">
      <div>
        <label
          htmlFor="email"
          className="block text-sm font-medium text-gray-700 mb-1"
        >
          邮箱
        </label>
        <input
          id="email"
          name="email"
          type="email"
          required
          autoComplete="email"
          placeholder="you@company.com"
          className="w-full px-3 py-2 border border-gray-300 rounded-lg focus:ring-2 focus:ring-blue-500 focus:border-blue-500"
        />
      </div>

      <div>
        <label
          htmlFor="password"
          className="block text-sm font-medium text-gray-700 mb-1"
        >
          密码
        </label>
        <input
          id="password"
          name="password"
          type="password"
          required
          autoComplete="current-password"
          placeholder="至少 8 个字符"
          className="w-full px-3 py-2 border border-gray-300 rounded-lg focus:ring-2 focus:ring-blue-500 focus:border-blue-500"
        />
      </div>

      {state.error && <p className="text-sm text-red-600">{state.error}</p>}

      <button
        type="submit"
        className="w-full py-2.5 px-4 bg-blue-600 text-white rounded-lg hover:bg-blue-700 transition-colors font-medium"
      >
        登录
      </button>
    </form>
  );
}
```

这里有一个容易踩的坑：`signIn` 在凭证无效时会抛出错误。Auth.js 的设计逻辑是——如果登录失败就抛异常，让调用方处理。但如果是 `NEXT_REDIRECT` 类型的错误（登录成功后的跳转），必须重新抛出，否则页面不会跳转。

## 在 Server Components 中获取 Session

认证做好之后，最重要的事情就是在页面里获取当前用户信息。在 Server Components 中，直接调用 `auth()` 就行。

```tsx
// app/dashboard/page.tsx

import { auth } from "@/auth";
import { redirect } from "next/navigation";
import { prisma } from "@/lib/prisma";

export default async function DashboardPage() {
  // auth() 从请求的 cookie 中解析 JWT，返回 session 对象
  // 如果没有 session，说明用户未登录
  const session = await auth();

  if (!session?.user) {
    // 未登录用户重定向到登录页
    // callbackUrl 记录原始页面，登录后可以跳回来
    redirect(`/login?callbackUrl=/dashboard`);
  }

  // 根据用户角色查询不同的数据
  // MANAGER 和 ADMIN 可以看到所有任务，MEMBER 只能看到自己的
  const tasks = await prisma.task.findMany({
    where:
      session.user.role === "MEMBER"
        ? { assignees: { some: { userId: session.user.id } } }
        : {}, // MANAGER 和 ADMIN 没有过滤条件，查全部
    include: {
      project: { select: { name: true } },
      assignees: {
        include: { user: { select: { name: true, image: true } } },
      },
    },
    orderBy: { updatedAt: "desc" },
    take: 20,
  });

  // 统计数据
  const stats = await prisma.task.groupBy({
    by: ["status"],
    where:
      session.user.role === "MEMBER"
        ? { assignees: { some: { userId: session.user.id } } }
        : {},
    _count: true,
  });

  return (
    <div className="p-6">
      <div className="mb-8">
        <h1 className="text-2xl font-bold">欢迎回来，{session.user.name}</h1>
        <p className="text-gray-500 mt-1">
          你当前的角色是
          <span className="font-medium text-gray-700 ml-1">
            {session.user.role === "ADMIN"
              ? "管理员"
              : session.user.role === "MANAGER"
                ? "项目经理"
                : "成员"}
          </span>
        </p>
      </div>

      {/* 任务统计卡片 */}
      <div className="grid grid-cols-2 md:grid-cols-4 gap-4 mb-8">
        {stats.map((stat) => (
          <div
            key={stat.status}
            className="p-4 bg-white rounded-lg shadow-sm border"
          >
            <p className="text-sm text-gray-500">{stat.status}</p>
            <p className="text-2xl font-bold">{stat._count}</p>
          </div>
        ))}
      </div>

      {/* 任务列表 */}
      <div className="bg-white rounded-lg shadow-sm border">
        <div className="p-4 border-b">
          <h2 className="font-semibold">最近任务</h2>
        </div>
        {tasks.map((task) => (
          <div key={task.id} className="p-4 border-b last:border-b-0">
            <div className="flex justify-between items-start">
              <div>
                <h3 className="font-medium">{task.title}</h3>
                <p className="text-sm text-gray-500 mt-1">
                  {task.project.name} / {task.status}
                </p>
              </div>
              <span
                className={`px-2 py-1 text-xs rounded-full ${
                  task.priority === "HIGH"
                    ? "bg-red-100 text-red-700"
                    : task.priority === "MEDIUM"
                      ? "bg-yellow-100 text-yellow-700"
                      : "bg-green-100 text-green-700"
                }`}
              >
                {task.priority}
              </span>
            </div>
          </div>
        ))}
      </div>
    </div>
  );
}
```

在 Server Component 里获取 Session 的好处是零客户端 JavaScript 开销——`auth()` 在服务端完成，最终发给浏览器的是已经渲染好的 HTML。而且你可以直接用 `session.user.id` 和 `session.user.role` 去查数据库，实现行级权限控制。

## 在 Client Components 中获取 Session

有些场景必须在客户端获取 Session，比如下拉菜单、实时状态显示、条件渲染的按钮等。

```tsx
// components/user-nav.tsx

"use client";

import { useSession, signOut } from "next-auth/react";

export function UserNav() {
  // useSession() 从 SessionProvider 的 context 中读取 session
  // status 有三个值：loading（正在获取）、authenticated（已登录）、unauthenticated（未登录）
  const { data: session, status } = useSession();

  // 正在加载 session 时显示骨架屏，避免布局抖动
  if (status === "loading") {
    return (
      <div className="flex items-center gap-3">
        <div className="w-8 h-8 rounded-full bg-gray-200 animate-pulse" />
        <div className="h-4 w-20 bg-gray-200 rounded animate-pulse" />
      </div>
    );
  }

  // 未登录时显示登录按钮
  if (!session) {
    return (
      <a
        href="/login"
        className="px-4 py-2 text-sm font-medium text-blue-600 hover:text-blue-700"
      >
        登录
      </a>
    );
  }

  // 已登录时显示用户头像和下拉菜单
  return (
    <div className="flex items-center gap-3">
      <span className="text-sm text-gray-600">{session.user.name}</span>
      {session.user.image && (
        <img
          src={session.user.image}
          alt={session.user.name || ""}
          className="w-8 h-8 rounded-full"
        />
      )}
      <button
        onClick={() => signOut({ redirectTo: "/login" })}
        className="px-3 py-1.5 text-sm text-gray-500 hover:text-gray-700 hover:bg-gray-100 rounded"
      >
        退出登录
      </button>
    </div>
  );
}
```

使用 `useSession` 有一个前提：你的应用必须被 `SessionProvider` 包裹。在 Next.js App Router 中，最干净的做法是在 Layout 层传入 session。

```tsx
// app/layout.tsx

import { SessionProvider } from "next-auth/react";
import { auth } from "@/auth";

export default async function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  // 在服务端获取 session，传给 SessionProvider
  // 这样客户端组件就能通过 useSession() 拿到初始 session
  const session = await auth();

  return (
    <html lang="zh-CN">
      <body>
        <SessionProvider session={session}>{children}</SessionProvider>
      </body>
    </html>
  );
}
```

这里有个细节：`SessionProvider` 的 `session` prop 是可选的。不传的话，`useSession` 会自己发请求去获取。但传了的话，客户端可以立即拿到 session 数据，不用等额外请求，首屏加载更快。

## middleware.ts：路由级别的权限控制

Server Component 里的 `if (!session) redirect(...)` 虽然能用，但有个问题：每个页面都要写一遍，容易遗漏。更好的做法是在 middleware 层统一拦截。

```typescript
// middleware.ts（项目根目录，和 app/ 同级）

import { auth } from "@/auth";
import { NextResponse } from "next/server";

// 定义路由权限规则
// routes 数组中的规则按顺序匹配，匹配到就按对应规则处理
const routeRules = [
  // 需要登录才能访问的路径
  {
    path: "/dashboard",
    authRequired: true,
  },
  {
    path: "/projects",
    authRequired: true,
  },
  {
    path: "/settings",
    authRequired: true,
  },
  // 需要管理员权限的路径
  {
    path: "/admin",
    roleRequired: "ADMIN" as const,
  },
  {
    path: "/api/tasks",
    authRequired: true,
  },
];

export default auth((req) => {
  const { pathname } = req.nextUrl;
  const isLoggedIn = !!req.auth;

  // 已登录用户访问登录/注册页，重定向到仪表盘
  if ((pathname === "/login" || pathname === "/register") && isLoggedIn) {
    return NextResponse.redirect(new URL("/dashboard", req.url));
  }

  // 检查每条路由规则
  for (const rule of routeRules) {
    if (!pathname.startsWith(rule.path)) continue;

    // 需要登录但未登录
    if (rule.authRequired && !isLoggedIn) {
      const loginUrl = new URL("/login", req.url);
      loginUrl.searchParams.set("callbackUrl", pathname);
      return NextResponse.redirect(loginUrl);
    }

    // 需要特定角色
    if (rule.roleRequired && req.auth?.user?.role !== rule.roleRequired) {
      // 角色不足，返回 403 或者重定向到无权限页面
      return NextResponse.redirect(new URL("/unauthorized", req.url));
    }
  }

  // 其他请求正常放行
  return NextResponse.next();
});

// matcher 定义 middleware 作用的路由范围
// 排除 _next/static、_next/image 等静态资源，以及 favicon
export const config = {
  matcher: ["/((?!_next/static|_next/image|favicon.ico|api/auth).*)"],
};
```

Auth.js v5 的 middleware 集成非常简洁——`auth()` 本身就是一个 middleware 工厂函数，它会自动解析请求中的 session 并挂载到 `req.auth` 上。你不需要手动解析 cookie 或调用 `getToken()`。

中间件里做的权限检查是第一道防线。但记住，middleware 只能做粗粒度的控制（比如"这个路径需要登录"）。更细粒度的权限（比如"这个任务只有创建者和项目经理能编辑"）还是要在业务逻辑里处理。

## 基于角色的访问控制：RBAC

到目前为止，我们只做了"能不能进来"的判断。但真实业务中更常见的需求是：同样登录了，不同角色能看到和操作的东西不一样。这就是 RBAC（Role-Based Access Control）。

我们写一组可复用的权限检查函数：

```typescript
// lib/permissions.ts

import { auth } from "@/auth";
import { prisma } from "@/lib/prisma";
import { redirect } from "next/navigation";

// 角色层级：ADMIN > MANAGER > MEMBER
const ROLE_HIERARCHY = {
  ADMIN: 3,
  MANAGER: 2,
  MEMBER: 1,
} as const;

type Role = keyof typeof ROLE_HIERARCHY;

/**
 * 要求用户已登录，否则重定向到登录页
 * 用在 Server Component 或 Server Action 中
 */
export async function requireAuth() {
  const session = await auth();

  if (!session?.user) {
    redirect("/login");
  }

  return session;
}

/**
 * 要求用户具有指定角色或更高角色
 * 比如 requireMinRole("MANAGER") 允许 MANAGER 和 ADMIN 通过
 */
export async function requireMinRole(minRole: Role) {
  const session = await requireAuth();

  const userRoleLevel = ROLE_HIERARCHY[session.user.role as Role];
  const requiredLevel = ROLE_HIERARCHY[minRole];

  if (userRoleLevel < requiredLevel) {
    redirect("/unauthorized");
  }

  return session;
}

/**
 * 检查当前用户是否有权限操作某个任务
 * 规则：
 * - ADMIN 可以操作所有任务
 * - MANAGER 可以操作自己管理的项目下的任务
 * - MEMBER 只能操作分配给自己的任务
 */
export async function checkTaskPermission(
  taskId: string,
  action: "view" | "edit" | "delete"
) {
  const session = await requireAuth();

  // 查询任务及其关联的项目
  const task = await prisma.task.findUnique({
    where: { id: taskId },
    include: {
      project: {
        select: { managerId: true },
      },
      assignees: {
        select: { userId: true },
      },
    },
  });

  if (!task) {
    return { allowed: false, reason: "任务不存在" };
  }

  // ADMIN 无限制
  if (session.user.role === "ADMIN") {
    return { allowed: true, task };
  }

  // 删除操作只有 ADMIN 能执行
  if (action === "delete" && session.user.role !== "ADMIN") {
    return { allowed: false, reason: "只有管理员可以删除任务" };
  }

  // MANAGER 可以查看和编辑自己管理项目下的任务
  if (session.user.role === "MANAGER") {
    const isProjectManager = task.project.managerId === session.user.id;
    if (!isProjectManager && action === "view") {
      // MANAGER 可以查看所有项目（因为需要跨项目协调）
      return { allowed: true, task };
    }
    if (!isProjectManager && action === "edit") {
      return { allowed: false, reason: "你只能编辑自己管理的项目下的任务" };
    }
    return { allowed: true, task };
  }

  // MEMBER 只能操作分配给自己的任务
  const isAssigned = task.assignees.some((a) => a.userId === session.user.id);
  if (!isAssigned) {
    return { allowed: false, reason: "你只能操作分配给你的任务" };
  }

  return { allowed: true, task };
}
```

然后在 Server Action 或页面里使用这些权限检查：

```typescript
// app/actions/tasks.ts

"use server";

import { checkTaskPermission, requireMinRole } from "@/lib/permissions";
import { prisma } from "@/lib/prisma";
import { revalidatePath } from "next/cache";

// 更新任务状态（任何有权限的用户都可以操作）
export async function updateTaskStatus(taskId: string, status: string) {
  // 先检查权限
  const { allowed, reason, task } = await checkTaskPermission(taskId, "edit");

  if (!allowed) {
    return { error: reason };
  }

  // 权限通过，执行更新
  await prisma.task.update({
    where: { id: taskId },
    data: {
      status,
      // 记录谁在什么时候修改了状态（审计日志）
      updatedBy: task?.updatedBy,
    },
  });

  revalidatePath("/dashboard");
  return { success: true };
}

// 删除任务（只有管理员）
export async function deleteTask(taskId: string) {
  // requireMinRole 会自动检查权限，不足时重定向
  await requireMinRole("ADMIN");

  await prisma.task.delete({
    where: { id: taskId },
  });

  revalidatePath("/dashboard");
  return { success: true };
}
```

RBAC 的核心思想就是把"谁能做什么"从业务代码里抽离出来，变成可配置、可复用的规则。上面的实现是硬编码在代码里的，如果你需要更灵活的权限系统（比如动态配置权限、权限继承、资源级别的细粒度控制），可以考虑引入 Casbin 或自己设计一套权限表结构。但对于大多数中小规模的应用，基于角色的权限检查已经足够了。

## Token 管理与安全

最后聊几个生产环境中容易被忽略的安全细节。

**JWT 的安全隐患**

我们选择了 JWT 策略，JWT 的问题是：一旦签发就无法撤销（除非你维护一个黑名单，那就失去了无状态的优势）。所以需要注意：

- JWT 的过期时间不要太长（我们设了 30 天，可以根据业务调整）
- 如果用户被禁用或角色被修改，旧 JWT 在过期前仍然有效（我们在 `jwt` callback 里通过 `trigger === "update"` 处理了角色更新，但需要客户端主动刷新 session）
- 敏感操作（删除数据、修改权限）应该重新验证用户身份，不要只依赖 JWT 中的 role

**密码安全**

```typescript
// lib/auth.ts

import { hash, compare } from "bcryptjs";

// 哈希密码：rounds=12 是安全性和性能的良好平衡
// 在普通服务器上大约需要 250ms，足以阻止暴力破解
// 但如果 rounds 太高（比如 15+），用户体验会变差
export async function hashPassword(password: string): Promise<string> {
  return hash(password, 12);
}

// 验证密码：compare 会自动从哈希值中提取 salt
// 所以不需要单独存储 salt
export async function verifyPassword(
  password: string,
  hashedPassword: string
): Promise<boolean> {
  return compare(password, hashedPassword);
}
```

**CSRF 防护**

Auth.js 默认启用了 CSRF 保护——它会检查 `Origin` 和 `Referer` 头，并要求 POST 请求携带 CSRF token。所以使用 `signIn` 和 `signOut` 时，不要用 `fetch` 手动调接口，而是用 Auth.js 提供的方法（Server Action 或 `next-auth/react` 的 hook），它们会自动处理 CSRF。

**登录频率限制**

在生产环境中，应该对登录接口做频率限制，防止暴力破解。可以用 `rate-limiter-flexible` 或者简单的内存计数器：

```typescript
// lib/rate-limit.ts

// 简单的内存限流器（单进程环境够用）
// 生产环境建议用 Redis 实现，支持多实例共享
const loginAttempts = new Map<string, { count: number; lastAttempt: number }>();

const MAX_ATTEMPTS = 5; // 最多尝试 5 次
const WINDOW_MS = 15 * 60 * 1000; // 15 分钟窗口

export function checkLoginRateLimit(email: string): boolean {
  const now = Date.now();
  const record = loginAttempts.get(email);

  if (!record || now - record.lastAttempt > WINDOW_MS) {
    // 没有记录或窗口已过期，重置计数
    loginAttempts.set(email, { count: 1, lastAttempt: now });
    return true; // 允许
  }

  if (record.count >= MAX_ATTEMPTS) {
    return false; // 超过限制，拒绝
  }

  record.count++;
  record.lastAttempt = now;
  return true; // 允许
}
```

然后在 `authorize` 函数里调用：

```typescript
// 在 authorize 函数开头加入频率检查
if (!checkLoginRateLimit(credentials.email as string)) {
  return null; // 不透露具体原因，统一返回 null
}
```

## Trade-off 分析

在 Auth.js 的方案选型中有几个值得权衡的点：

**JWT vs Database Session**

JWT 的优势是无状态、不需要查数据库、天然支持水平扩展。劣势是无法主动撤销（用户退出登录只是清除了客户端 cookie，服务端的 JWT 在过期前仍然有效）、payload 越大 cookie 越大（浏览器有 4KB 限制）。Database Session 的优势是可以随时让 session 失效（比如踢人下线、密码修改后强制重新登录），劣势是每次请求都要查数据库。我们的选择是 JWT，因为任务管理平台不需要强制下线功能，而且无状态的特性更适合 Vercel 等边缘部署平台。

**自建认证 vs Auth.js**

Auth.js 覆盖了 90% 的认证场景，开箱即用。但如果你有非常特殊的需求（比如多租户认证、自定义 SSO、WebAuthn 无密码登录），Auth.js 的扩展可能会比较痛苦，这时候自建或者用更底层的库（比如 jose 处理 JWT、passport.js 处理 OAuth）可能更合适。

**Credentials Provider vs Magic Link**

我们选择了传统的邮箱密码登录，但另一种越来越流行的方案是 Magic Link（邮箱验证码登录）。Magic Link 不需要用户记密码，用户体验更好，但实现上依赖邮件服务，而且每次登录都要收邮件。如果你的用户群主要是开发者（他们已经有很多密码要记），Magic Link 可能更合适。

**角色层级 vs 权限矩阵**

我们用了简单的三层角色模型（ADMIN > MANAGER > MEMBER），实现简单、容易理解。但如果你需要更细粒度的控制（比如"MEMBER 可以编辑任务但不能删除"、"MANAGER 可以创建项目但不能管理用户"），就需要引入权限矩阵（RBAC 的完整形式）或者 ABAC（基于属性的访问控制）。这会增加系统复杂度，但对于中大型团队是必要的。
