上一篇我们学习了各种查询运算符，能用 find() 构建复杂的查询条件。但实际开发中，数据很少是独立的，用户和文章、文章和评论、文章和标签之间都存在关联关系。

在关系型数据库中，我们会用外键来关联表，但 MongoDB 没有外键这个概念，这给了我们更大的自由度，也带来了更多选择。这篇文章我们就来聊聊在 MongoDB 中如何设计数据关系。

## 嵌入还是引用

MongoDB 提供两种主要的数据关联方式：嵌入式和引用式。

嵌入式就是把相关数据直接放在一个文档里，比如用户文档里直接包含他的资料信息。引用式则是像关系型数据库那样，只存储其他文档的 \_id，需要时再通过 populate 去获取完整数据。

这两种方式没有绝对的优劣，关键看你的使用场景。如果数据总是被一起查询，比如文章和它的评论，那嵌入通常更好，因为一次查询就能拿到所有数据。但如果数据需要被独立管理，比如用户可能有很多文章，而文章又需要单独查询和编辑，那引用式更合适。

数据量也是一个重要考虑因素。MongoDB 文档最大 16MB，虽然实际应用中很少会触及这个限制，但如果嵌入的数据太多，会影响查询性能。而且当多个文档都嵌入同一份数据时，数据更新会变得很麻烦。

## 一对一关系：用户资料

在 CMS 系统中，用户和他的个人资料是典型的一对一关系。这种关系通常适合用嵌入式设计，因为：

第一，资料总是和用户一起显示，我们查询用户时几乎总是需要他的资料
第二，资料数据量不大，不会导致文档过大
第三，资料更新不频繁，不会有并发冲突问题

```typescript
import mongoose, { Schema, Model, Document } from "mongoose";

interface IProfile {
  bio?: string;
  avatar?: string;
  website?: string;
  location?: string;
  social?: {
    twitter?: string;
    github?: string;
    linkedin?: string;
  };
  preferences?: {
    emailNotifications: boolean;
    theme: "light" | "dark";
    language: string;
  };
}

const profileSchema = new Schema<IProfile>(
  {
    bio: { type: String, maxlength: 500 },
    avatar: String,
    website: String,
    location: String,
    social: {
      twitter: String,
      github: String,
      linkedin: String,
    },
    preferences: {
      emailNotifications: { type: Boolean, default: true },
      theme: { type: String, enum: ["light", "dark"], default: "light" },
      language: { type: String, default: "zh-CN" },
    },
  },
  { _id: false }
);

interface IUser extends Document {
  name: string;
  email: string;
  password?: string;
  role: "user" | "author" | "admin";
  isActive: boolean;
  profile: IProfile;
}

const userSchema = new Schema<IUser>({
  name: {
    type: String,
    required: true,
    trim: true,
  },
  email: {
    type: String,
    required: true,
    unique: true,
    lowercase: true,
  },
  password: {
    type: String,
    select: false,
  },
  role: {
    type: String,
    enum: ["user", "author", "admin"],
    default: "user",
  },
  isActive: {
    type: Boolean,
    default: true,
  },
  profile: {
    type: profileSchema,
    default: () => ({}),
  },
});

const User = mongoose.model<IUser>("User", userSchema);

export default User;
```

这个设计把用户的个人资料作为子文档嵌入到用户文档中。注意我们在 profileSchema 定义时使用了 `_id: false` 选项，这样子文档就不会生成自己的 \_id，因为资料不会独立存在。

创建用户时可以直接包含资料信息：

```typescript
const user = await User.create({
  _id: new mongoose.Types.ObjectId("usr_8x9k2m"),
  name: "张三",
  email: "zhangsan@example.com",
  password: "hashedpassword",
  profile: {
    bio: "全栈开发者，热爱分享",
    avatar: "https://cdn.example.com/avatars/usr_8x9k2m.jpg",
    location: "北京",
    social: {
      github: "https://github.com/zhangsan",
      twitter: "https://twitter.com/zhangsan",
    },
    preferences: {
      theme: "dark",
      language: "zh-CN",
    },
  },
});

// 查询时自动包含资料
const foundUser = await User.findById("usr_8x9k2m");
console.log(foundUser.profile.bio); // "全栈开发者，热爱分享"
```

查询用户时，资料信息会自动包含在结果中，不需要额外的查询或 populate 操作。更新资料也很简单，使用点号路径即可：

```typescript
await User.findByIdAndUpdate("usr_8x9k2m", {
  "profile.bio": "全栈开发者，专注于 Node.js 和 React",
  "profile.preferences.theme": "light",
});
```

## 一对多关系：文章与评论

文章和评论是一对多关系，这个场景比较复杂。如果评论数量不多，我们可以把评论嵌入到文章文档中；但如果评论很多，或者评论需要独立管理，就应该用引用式。

对于博客系统，文章的评论通常不会特别多（一般几十到几百条），而且我们总是和文章一起显示评论，所以嵌入是合理的选择。同时，我们会保存评论作者的快照信息（名称、头像），避免每次都要 populate 用户表。

```typescript
import mongoose, { Schema, Model, Document } from "mongoose";

interface IComment {
  content: string;
  author: mongoose.Types.ObjectId;
  authorName?: string;
  authorAvatar?: string;
  createdAt: Date;
  updatedAt: Date;
  likes: number;
  replies?: IComment[];
}

const commentSchema = new Schema<IComment>(
  {
    content: {
      type: String,
      required: true,
      minlength: 1,
      maxlength: 1000,
    },
    author: {
      type: Schema.Types.ObjectId,
      ref: "User",
      required: true,
    },
    authorName: String,
    authorAvatar: String,
    createdAt: {
      type: Date,
      default: Date.now,
    },
    updatedAt: {
      type: Date,
      default: Date.now,
    },
    likes: {
      type: Number,
      default: 0,
      min: 0,
    },
    replies: [commentSchema],
  },
  { timestamps: true }
);

interface IPost extends Document {
  title: string;
  content: string;
  author: mongoose.Types.ObjectId;
  status: "draft" | "published" | "archived";
  comments: IComment[];
  tags: string[];
  views: number;
}

const postSchema = new Schema<IPost>({
  title: {
    type: String,
    required: true,
    trim: true,
  },
  content: {
    type: String,
    required: true,
  },
  author: {
    type: Schema.Types.ObjectId,
    ref: "User",
    required: true,
    index: true,
  },
  status: {
    type: String,
    enum: ["draft", "published", "archived"],
    default: "draft",
  },
  comments: [commentSchema],
  tags: [String],
  views: {
    type: Number,
    default: 0,
  },
});

const Post = mongoose.model<IPost>("Post", postSchema);

export default Post;
```

注意这里 comments 字段是一个数组，存储评论对象。同时 author 字段是引用式的，因为一个作者可能写很多文章，我们需要独立管理作者信息。

添加评论时使用 $push 操作符：

```typescript
const post = await Post.create({
  _id: new mongoose.Types.ObjectId("post_a1b2c3d4"),
  title: "Mongoose 入门教程",
  content: "从零开始学习 Mongoose...",
  author: new mongoose.Types.ObjectId("usr_8x9k2m"),
  tags: ["mongodb", "nodejs", "tutorial"],
});

// 添加评论
const updatedPost = await Post.findByIdAndUpdate(
  "post_a1b2c3d4",
  {
    $push: {
      comments: {
        _id: new mongoose.Types.ObjectId("cmt_x7y8z9"),
        content: "很棒的文章，对我帮助很大！",
        author: new mongoose.Types.ObjectId("usr_5f6g7h"),
        authorName: "李四",
        authorAvatar: "https://cdn.example.com/avatars/usr_5f6g7h.jpg",
        createdAt: new Date("2026-04-10T14:30:00Z"),
        likes: 5,
      },
    },
  },
  { new: true }
);

// 查询文章时自动包含评论
const foundPost = await Post.findById("post_a1b2c3d4")
  .populate("author", "name avatar email")
  .populate("comments.author", "name avatar");

console.log(foundPost.title); // "Mongoose 入门教程"
console.log(foundPost.author.name); // "张三"
console.log(foundPost.comments[0].content); // "很棒的文章..."
console.log(foundPost.comments[0].authorName); // "李四" (快照)
```

这里有个有趣的设计：我们既保存了 author 的 \_id（用于关联），又保存了 authorName 和 authorAvatar（快照）。这样即使原用户改名或换头像，评论中显示的信息也不会变，符合大多数系统的设计。

## 多对多关系：文章与标签

文章和标签是多对多关系：一篇文章可以有多个标签，一个标签也可以用于多篇文章。这种关系在 MongoDB 中通常用双向引用实现。

我们可以在文章中存储标签名数组，同时在标签中通过虚拟字段反向引用文章。这种设计的优点是标签名简单易读，不需要每次都查标签表。

```typescript
import mongoose, { Schema, Model, Document } from "mongoose";

interface ITag extends Document {
  name: string;
  slug: string;
  description?: string;
  postCount: number;
  color?: string;
}

const tagSchema = new Schema<ITag>({
  name: {
    type: String,
    required: true,
    unique: true,
    trim: true,
  },
  slug: {
    type: String,
    unique: true,
    lowercase: true,
    trim: true,
  },
  description: String,
  postCount: {
    type: Number,
    default: 0,
  },
  color: {
    type: String,
    default: "#3B82F6",
  },
});

// 自动生成 slug
tagSchema.pre("save", function (next) {
  if (!this.slug) {
    this.slug = this.name
      .toLowerCase()
      .replace(/[^a-z0-9]+/g, "-")
      .replace(/^-+|-+$/g, "");
  }
  next();
});

// 反向虚拟字段：标签 → 文章
tagSchema.virtual("posts", {
  ref: "Post",
  localField: "_id",
  foreignField: "tags",
});

const Tag = mongoose.model<ITag>("Tag", tagSchema);

export default Tag;
```

注意这里 foreignField 是 "tags"，但文章 Schema 中的 tags 字段是字符串数组，不是 ObjectId。这种设计在 Mongoose 中是可行的，它会尝试匹配标签的 name 字段。

在 Post Schema 中添加反向虚拟字段：

```typescript
postSchema.virtual("tagObjects", {
  ref: "Tag",
  localField: "tags",
  foreignField: "name",
});
```

使用方式：

```typescript
// 创建标签
const mongoTag = await Tag.create({
  _id: new mongoose.Types.ObjectId("tag_m1n2o3"),
  name: "MongoDB",
  slug: "mongodb",
  description: "MongoDB 相关文章",
  color: "#47A992",
});

const nodeTag = await Tag.create({
  _id: new mongoose.Types.ObjectId("tag_n4o5p6"),
  name: "Node.js",
  slug: "nodejs",
  description: "Node.js 相关文章",
  color: "#68A063",
});

// 创建文章（使用标签名）
const post = await Post.create({
  _id: new mongoose.Types.ObjectId("post_b2c3d4e5"),
  title: "Node.js 操作 MongoDB 实战",
  content: "详细介绍如何在 Node.js 中使用 MongoDB...",
  author: new mongoose.Types.ObjectId("usr_8x9k2m"),
  tags: ["MongoDB", "Node.js"],
  status: "published",
});

// 查询文章并 populate 标签
const foundPost = await Post.findById("post_b2c3d4e5").populate(
  "tagObjects",
  "name slug description color"
);

console.log(foundPost.title);
foundPost.tagObjects.forEach((tag: any) => {
  console.log(`标签: ${tag.name} (${tag.slug}) - ${tag.description}`);
});

// 反向查询：查标签及其文章
const tag = await Tag.findOne({ name: "MongoDB" }).populate(
  "posts",
  "title status views"
);

console.log(`\n标签: ${tag.name}`);
console.log(`文章数: ${tag.postCount}`);
tag.posts.forEach((post: any) => {
  console.log(`  - ${post.title} (${post.views} 浏览)`);
});
```

这个设计的好处是文章中的 tags 字段直接存储可读的标签名，即使标签表被删除，文章数据仍然有意义。同时通过虚拟字段，我们可以在需要时查询某个标签下的所有文章。

## 反向关系设计

有时我们需要从"多"方反向查询"一"方，比如查询某个用户的所有文章。可以通过虚拟字段实现：

```typescript
userSchema.virtual("posts", {
  ref: "Post",
  localField: "_id",
  foreignField: "author",
});

userSchema.virtual("publishedPosts", {
  ref: "Post",
  localField: "_id",
  foreignField: "author",
  match: { status: "published" }, // 只查询已发布的文章
});
```

使用时可以加上各种选项：

```typescript
const user = await User.findById("usr_8x9k2m")
  .populate("posts", "title status views createdAt")
  .populate("publishedPosts", "title views");

console.log(`用户: ${user.name}`);
console.log(`总文章数: ${user.posts.length}`);
user.posts.forEach((post: any) => {
  console.log(`  - ${post.title} (${post.status}, ${post.views} 浏览)`);
});

// 使用 populate 选项进行更精细的控制
const userWithLimit = await User.findById("usr_8x9k2m").populate({
  path: "posts",
  select: "title views createdAt tags",
  options: { limit: 10, sort: { createdAt: -1 } },
  match: { status: "published" },
});
```

## 关系设计的权衡

嵌入式和引用式各有优劣，选择时需要考虑多个维度。

嵌入式设计的优势在于查询性能，一次查询就能获取所有相关数据，不需要多次往返数据库。同时嵌入式数据的更新是原子的，不存在数据不一致的问题。但缺点也很明显：数据会重复，而且当嵌入数据量太大时会影响性能。

引用式设计则避免了数据重复，每个数据只存储一份，更新时只需改一处。而且引用式没有文档大小限制，适合关联大量数据。但代价是需要额外的查询或 join 操作，在应用层做关联会增加复杂度。

实际项目中，经常需要混合使用两种方式。比如文章作者用引用式（因为一个作者很多文章），文章评论用嵌入式（因为评论不多且总是一起显示），文章标签用双向引用（因为需要双向查询）。

关键是要理解你的数据访问模式：数据总是一起访问吗？数据量大吗？数据更新频繁吗？回答这些问题，就能做出合理的选择。
