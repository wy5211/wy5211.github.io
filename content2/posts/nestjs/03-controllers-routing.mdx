# 控制器与路由：设计你的第一个 API

上一篇我们把健康检查服务拆成了 ConfigModule、DatabaseModule、HealthModule 三个模块，代码各归其位。但健康检查只是个运维接口，用户真正会用的是业务 API。这篇文章我们来做第一个业务接口：文章摘要列表。

这个接口需要支持分页和标签筛选，还要考虑 API 版本管理——这些是真实项目中绕不开的问题。

## Express 的 app.use 和 NestJS 的 @Controller 有什么区别

用 Express 写路由，你大概是这样做的：

```javascript
const express = require("express");
const app = express();

// 手动挂载路由
const articlesRouter = require("./routes/articles");
app.use("/api/v1/articles", articlesRouter);

// 路由文件里继续手动匹配
articlesRouter.get("/snippets", async (req, res) => {
  const page = parseInt(req.query.page) || 1;
  // ...
});
```

每个路由文件需要手动 `require`、手动 `app.use` 挂载、手动从 `req.query` 里提取参数。参数全是 string 类型，`parseInt` 散落在各个路由里。路由前缀在 `app.use` 里定义，路径在路由文件里定义，要看完整的路由地址得两头跑。

NestJS 把这些事情收拢到了装饰器里：

```typescript
@Controller("api/v1/articles")
export class ArticlesController {
  @Get("snippets")
  getSnippets(@Query("page") page: string, @Query("tag") tag?: string) {
    // 参数已经通过装饰器提取好了
  }
}
```

路由前缀 `api/v1/articles` 和方法路径 `snippets` 放在一起，完整的路由地址一目了然。参数提取也通过装饰器完成，不需要手动 `req.query.page`。更重要的是，NestJS 知道每个路由接受什么参数，这为后续的参数验证管道（Pipe）打下了基础。

## 设计文章摘要 API

我们要实现的接口是 `GET /api/v1/articles/snippets`，返回最新的文章摘要列表。请求参数包括：

- `page`：页码，默认 1
- `pageSize`：每页条数，默认 10
- `tag`：按标签筛选，可选

响应数据大概长这样：

```json
{

    {
      "id": "art_20260328_001",
      "title": "从 Express 迁移到 NestJS 的踩坑记录",
      "summary": "团队花了三周时间把一个运行了两年的 Express 项目迁移到 NestJS，中间遇到了不少意想不到的问题...",
      "author": "zhang_wei",
      "publishedAt": "2026-03-28T14:30:00Z",
      "tags": ["NestJS", "Express", "迁移"],
      "readTime": 12
    }
  ],
  "pagination": {
    "page": 1,

```

先创建模块结构：

```bash

```

CLI 会在 `src/articles/` 下生成对应的文件，并且自动在模块里注册控制器和服务。

## 参数提取装饰器

NestJS 提供了一组装饰器来从请求中提取不同位置的数据。理解每个装饰器的适用场景很重要，因为用错了会在运行时拿到 `undefined`。

**@Query**：提取 URL 查询参数，也就是 `?` 后面的键值对。

```typescript
@Get("snippets")
getSnippets(
  @Query("page") page: string,       // /snippets?page=2
  @Query("pageSize") pageSize: string, // /snippets?pageSize=20
  @Query("tag") tag?: string           // /snippets?tag=NestJS
) {
  // 注意：page 和 pageSize 的类型是 string，不是 number
  // URL 里的所有参数都是字符串
}
```

这里有个常见的坑：很多人以为加了 `@Query("page") page: string` 之后，传 `?page=2` 拿到的就是数字 2。不是的。HTTP 协议里查询参数全是字符串，`page` 的值是 `"2"`（string），不是 `2`（number）。如果你直接拿它去做数据库查询的 offset 计算，结果会出乎意料。

**@Param**：提取 URL 路径参数。

```typescript
@Get(":id")
getArticle(@Param("id") id: string) {
  // GET /api/v1/articles/art_20260328_001
  // id = "art_20260328_001"
}
```

和 `@Query` 一样，`@Param` 拿到的也是 string。即使你的路径是 `/articles/42`，`id` 也是 `"42"` 而不是 `42`。

**@Body**：提取请求体数据，用于 POST、PUT、PATCH 等请求。

```typescript
@Post()
createArticle(@Body() createDto: CreateArticleDto) {
  // 请求体 JSON 会自动反序列化为 CreateArticleDto 实例
}
```

**@Headers**：提取请求头。

```typescript
@Get("snippets")
getSnippets(
  @Headers("x-request-id") requestId: string,
  @Headers("authorization") auth: string
) {
  // requestId = "req_abc123"
  // auth = "Bearer eyJhbGciOi..."
}
```

## 用 ParseIntPipe 解决类型问题

回到分页参数的问题。`@Query("page")` 拿到的是字符串，但分页逻辑需要数字。你当然可以手动 `parseInt(page)`，但这样每个接口都得写一遍，而且还得处理 `page=abc` 这种非法输入。

NestJS 的管道（Pipe）就是干这个的。`ParseIntPipe` 会自动把值转成整数，转换失败时抛出 400 错误。

```typescript
import {
  Controller,
  Get,
  Query,
  DefaultValuePipe,
  ParseIntPipe,
} from "@nestjs/common";

@Controller("api/v1/articles")
export class ArticlesController {
  constructor(private articlesService: ArticlesService) {}

  @Get("snippets")
  getSnippets(
    @Query("page", new DefaultValuePipe(1), ParseIntPipe) page: number,
    @Query("pageSize", new DefaultValuePipe(10), ParseIntPipe) pageSize: number,
    @Query("tag") tag?: string
  ) {
    return this.articlesService.getSnippets({ page, pageSize, tag });
  }
}
```

这段代码里有三个管道串在一起：先由 `DefaultValuePipe` 确保有默认值（如果请求里没传 page，就默认为 1），再由 `ParseIntPipe` 把值转成整数。转换之后，`page` 和 `pageSize` 的类型就真正变成了 `number`，不再是 string。

如果用户传了 `?page=abc`，`ParseIntPipe` 会直接返回 400 Bad Request，不需要你写任何错误处理代码。这就是管道的价值——把数据校验逻辑从业务代码中剥离出来。

## 实现服务层

控制器的代码应该尽量薄——只做参数提取和方法调用。业务逻辑放在 Service 里：

```typescript
// src/articles/articles.service.ts
import { Injectable } from "@nestjs/common";

@Injectable()
export class ArticlesService {
  // 模拟数据，真实项目中这里会是数据库查询
  private articles = [
    {
      id: "art_20260328_001",
      title: "从 Express 迁移到 NestJS 的踩坑记录",
      summary:
        "团队花了三周时间把一个运行了两年的 Express 项目迁移到 NestJS，中间遇到了不少意想不到的问题...",
      author: "zhang_wei",
      publishedAt: "2026-03-28T14:30:00Z",
      tags: ["NestJS", "Express", "迁移"],
      readTime: 12,
    },
    {
      id: "art_20260325_002",
      title: "PostgreSQL 索引优化实战：从查询 3s 到 30ms",
      summary:
        "线上慢查询报警，一个看似简单的 WHERE 条件背后，涉及联合索引的选择、覆盖索引的利用和统计信息的更新策略...",
      author: "li_ming",
      publishedAt: "2026-03-25T09:15:00Z",
      tags: ["PostgreSQL", "性能优化", "索引"],
      readTime: 15,
    },
    {
      id: "art_20260322_003",
      title: "用 Redis 实现分布式锁的正确姿势",
      summary:
        "网上的 Redis 分布式锁实现五花八门，很多都存在竞态条件。这篇文章从 SETNX 到 Redlock，分析每种方案的边界情况...",
      author: "wang_fang",
      publishedAt: "2026-03-22T16:45:00Z",
      tags: ["Redis", "分布式", "并发"],
      readTime: 10,
    },
  ];

  getSnippets(params: { page: number; pageSize: number; tag?: string }) {
    let filtered = this.articles;

    if (params.tag) {
      filtered = filtered.filter((a) => a.tags.includes(params.tag!));
    }

    const total = filtered.length;
    const totalPages = Math.ceil(total / params.pageSize);
    const start = (params.page - 1) * params.pageSize;
    const items = filtered.slice(start, start + params.pageSize);

    return {
      items,
      pagination: {
        page: params.page,
        pageSize: params.pageSize,
        total,
        totalPages,
      },
    };
  }
}
```

这个 Service 接收的 `page` 和 `pageSize` 已经是 number 类型了，因为在控制器层就被 `ParseIntPipe` 转换过了。这就是管道和控制器协作的效果——每一层只做自己该做的事。

## 响应控制：状态码和响应头

NestJS 默认的响应规则是：POST 请求返回 201，其他请求返回 200，返回值自动序列化为 JSON。大多数情况下这够用了，但有些场景需要手动控制。

比如登录接口，它用的是 POST 方法，但你不想返回 201（201 表示"资源已创建"，登录不是创建资源），而是想返回 200：

```typescript
import { HttpCode, HttpStatus } from "@nestjs/common";

@Post("login")
@HttpCode(HttpStatus.OK) // 覆盖 POST 默认的 201
login(@Body() loginDto: LoginDto) {
  return this.authService.login(loginDto);
}
```

设置响应头用 `@Header` 装饰器：

```typescript
@Get("snippets")
@Header("Cache-Control", "public, max-age=60")
getSnippets() {
  // 文章摘要列表可以缓存 60 秒
}
```

重定向用 `@Redirect`：

```typescript
@Get("legacy-docs")
@Redirect("/api/v1/articles/snippets", 301)
legacyDocs() {}
```

这些都是声明式的——你用装饰器声明意图，NestJS 帮你执行。和 Express 里 `res.status(200).set("Cache-Control", "...").json(data)` 这种命令式写法相比，代码更紧凑，意图也更清晰。

## API 版本控制的三种方式

API 版本管理是每个有生命周期的项目都要面对的问题。NestJS 支持三种方式。

**方式一：URL 前缀**。把版本号直接写在路由里。

```typescript
@Controller("api/v1/articles")
export class ArticlesV1Controller {}

@Controller("api/v2/articles")
export class ArticlesV2Controller {}
```

这是最常见、最直观的方式。缺点是路由前缀变长了，而且需要在每个控制器上重复写版本号。

**方式二：NestJS 内置的 Version 装饰器**。在 main.ts 里启用版本控制，然后在控制器或路由上用 `@Version` 声明版本：

```typescript
// main.ts
const app = await NestFactory.create(AppModule);
app.enableVersioning({
  type: RoutingVersion.MediaType,
  defaultVersion: "1",
});

// 控制器
@Controller("articles")
export class ArticlesController {
  @Get("snippets")
  @Version("1")
  getSnippetsV1() {}

  @Get("snippets")
  @Version("2")
  getSnippetsV2() {}
}
```

客户端通过 `Accept` 请求头指定版本：

```bash
curl -H "Accept: application/vnd.api.v1+json" http://localhost:3000/articles/snippets
```

这种方式 URL 更干净，但客户端需要知道在请求头里传版本号，对接成本更高。

**方式三：全局前缀**。在 main.ts 里统一设置，所有路由自动加上前缀：

```typescript
app.setGlobalPrefix("api/v1");
```

设置之后，`@Controller("articles")` 对应的路由就变成了 `/api/v1/articles`。要升级版本时，改一处全局前缀就行。缺点是没法在同一应用里同时支持多个版本。

实际项目中，方式一（URL 前缀）用得最多，因为它对 API 消费者最友好——看 URL 就知道是哪个版本，不需要额外的请求头约定。

## 把 ArticlesModule 注册到根模块

最后一步，把文章模块加到应用里：

```typescript
// src/app.module.ts
import { Module } from "@nestjs/common";
import { ConfigModule } from "./config/config.module";
import { DatabaseModule } from "./database/database.module";
import { HealthModule } from "./health/health.module";
import { ArticlesModule } from "./articles/articles.module";

@Module({
  imports: [ConfigModule, DatabaseModule, HealthModule, ArticlesModule],
})
export class AppModule {}
```

启动项目后，你可以测试一下接口：

```bash
# 获取第一页
curl http://localhost:3000/api/v1/articles/snippets

# 获取第二页，每页 5 条
curl "http://localhost:3000/api/v1/articles/snippets?page=2&pageSize=5"

# 按标签筛选
curl "http://localhost:3000/api/v1/articles/snippets?tag=NestJS"

# 传非法参数
curl "http://localhost:3000/api/v1/articles/snippets?page=abc"
# 返回 400 Bad Request: Validation failed (numeric string is expected)
```

## 什么时候该用哪个装饰器

简单总结一下选择逻辑：

- 参数在 URL 路径里（`/articles/:id`）→ 用 `@Param`
- 参数在查询字符串里（`?page=1`）→ 用 `@Query`
- 参数在请求体里（POST 的 JSON）→ 用 `@Body`
- 需要读取请求头 → 用 `@Headers`
- 需要客户端 IP → 用 `@Ip`

如果同时需要从多个位置提取参数，装饰器可以组合使用，NestJS 会从对应的位置分别取出值。

## 控制器的边界在哪

控制器应该只做三件事：提取参数、调用 Service、返回结果。如果你发现自己在控制器里写了数据库查询、数据格式化、条件判断等逻辑，大概率应该把它挪到 Service 里。

这样做的好处不仅是代码更干净，更重要的是**可测试性**。控制器的测试只需要 mock Service，验证参数提取和返回值是否正确。Service 的测试不需要关心 HTTP 层的事，专注业务逻辑。两者解耦后，测试起来互不干扰。

控制器写好了，但业务逻辑不应该一直放在 Service 里做数据模拟。下一篇来看看 Service 层怎么设计，以及如何接入真实的数据库。
