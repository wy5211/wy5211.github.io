## 从一个需求说起

zhang_wei（`usr_a1x9k2m`）注册了账号，发了几篇文章。现在他想做两件事：上传头像、给文章「从 Express 迁移到 NestJS 的踩坑记录」加一张封面图。

看起来简单，但仔细想想有几个问题：

- 用户传了一个叫 `avatar.jpg` 的文件，你直接存？100 个用户都传 `avatar.jpg`，覆盖了。得重命名。
- 文件后缀是 `.jpg` 就一定是图片？把一个 `.html` 文件改后缀成 `.jpg` 上传呢？不能只看后缀。
- 用户传了一张 4000x3000 的高清照片当头像，加载要好几秒。得压缩。
- 现在存本地，以后要迁移到云存储。得设计好接口，换实现不换调用方。

这一篇我们不只实现"能上传文件"，而是把整个链路做对。

## Multer 做了什么

NestJS 底层用 Express，Express 处理文件上传用 Multer。当前端用 `FormData` 发送文件时，请求的 `Content-Type` 是 `multipart/form-data`——这是一种把文件和表单字段混在一起编码的格式。

浏览器发的实际请求长这样：

```http
POST /upload/avatar HTTP/1.1
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW

------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="avatar"; filename="photo.jpg"
Content-Type: image/jpeg

[二进制数据...]
------WebKitFormBoundary7MA4YWxkTrZu0gW--
```

Multer 做的事情就是解析这种格式：把文件从请求体中提取出来，存到内存（Buffer）或磁盘，然后在 `req.file`（单文件）或 `req.files`（多文件）上挂载文件信息。

NestJS 通过 `@UseInterceptors(FileInterceptor('fieldName'))` 把 Multer 接入请求管道。`FileInterceptor` 的第一个参数是字段名，必须和前端 `formData.append('avatar', file)` 的字段名一致——如果不一致，Multer 找不到对应的字段，`req.file` 就是 undefined。

```typescript
// src/upload/upload.controller.ts
import {
  Controller,
  Post,
  UseInterceptors,
  UploadedFile,
} from "@nestjs/common";
import { FileInterceptor } from "@nestjs/platform-express";

@Controller("upload")
export class UploadController {
  @Post("avatar")
  @UseInterceptors(FileInterceptor("avatar"))
  uploadAvatar(@UploadedFile() file: Express.Multer.File) {
    return { url: `/uploads/avatars/${file.filename}` };
  }
}
```

`Express.Multer.File` 对象包含这些关键字段：

- `originalname`：用户上传时的原始文件名，比如 `我的头像.jpg`
- `filename`：存储后的文件名，可以是重命名后的
- `mimetype`：浏览器声明的 MIME 类型，比如 `image/jpeg`
- `size`：文件大小，单位字节
- `buffer`：文件内容（内存存储时可用）
- `path`：文件在磁盘上的路径（磁盘存储时可用）
- `fieldname`：FormData 中的字段名，即 `FileInterceptor('avatar')` 中的 `'avatar'`

## 不能信任前端传来的 MIME 类型

`file.mimetype` 是浏览器告诉你的，不是后端自己检测的。把一个 `.exe` 改名为 `.jpg`，浏览器照样说 `mimetype` 是 `image/jpeg`。

验证文件类型有三种策略，从简单到严格：

**策略 1：检查扩展名（最简单，最弱）**

```typescript
import { extname } from "path";

const ext = extname(file.originalname).toLowerCase();
if (![".jpg", ".jpeg", ".png", ".webp"].includes(ext)) {
  return cb(new BadRequestException("不支持的图片格式"), false);
}
```

用户改后缀就能绕过。但作为第一层过滤，把明显不对的请求挡掉，减少后续处理压力，还是有价值的。

**策略 2：使用 file-type 库检测文件头（推荐）**

```typescript
import { fileTypeFromBuffer } from "file-type";

const type = await fileTypeFromBuffer(file.buffer);
const allowedMimes = ["image/jpeg", "image/png", "image/webp"];
if (!type || !allowedMimes.includes(type.mime)) {
  throw new BadRequestException("文件类型不合法");
}
```

`file-type` 读取文件头部的魔数（magic number）来判断真实文件类型。一个 exe 文件不管怎么改名，它的文件头不会变成 JPEG 的 `FF D8 FF`。

**策略 3：白名单 MIME + 文件头检测 + 扩展名校验（最严格）**

三层都做。对于博客平台，策略 2 就够了。下面我们在实际代码中组合使用策略 1 和策略 2。

Multer 自带的 `fileFilter` 选项可以在文件写入前做拦截：

```typescript
@UseInterceptors(FileInterceptor('avatar', {
  fileFilter: (req, file, cb) => {
    const ext = extname(file.originalname).toLowerCase();
    if (!['.jpg', '.jpeg', '.png', '.webp'].includes(ext)) {
      cb(new BadRequestException('不支持的图片格式'), false);
      return;
    }
    cb(null, true);
  },
}))
```

注意 `fileFilter` 是同步的——它不能做 `fileTypeFromBuffer` 这种异步检测。所以扩展名检查放在 `fileFilter` 里做第一道拦截，`file-type` 检测放在 Service 层做第二道确认。

## 完整的头像上传实现

先看最终效果。zhang_wei 上传头像后，后端做这些事：

1. Multer 接收文件到内存（buffer）
2. Service 层检查文件大小
3. Service 层用 file-type 检测真实文件类型
4. 用 Sharp 压缩到 256x256 正方形
5. 重命名后存到磁盘
6. 返回可访问的 URL

Controller 只负责接收请求和转发：

```typescript
// src/upload/upload.controller.ts
import {
  Controller,
  Post,
  UseInterceptors,
  UploadedFile,
} from "@nestjs/common";
import { FileInterceptor } from "@nestjs/platform-express";
import { memoryStorage } from "multer";
import { UploadService } from "./upload.service";
import { CurrentUser } from "../common/decorators/current-user.decorator";
import { Roles } from "../common/decorators/roles.decorator";

@Controller("upload")
export class UploadController {
  constructor(private uploadService: UploadService) {}

  @Post("avatar")
  @UseInterceptors(
    FileInterceptor("avatar", {
      storage: memoryStorage(),
      limits: { fileSize: 5 * 1024 * 1024 }, // Multer 层的硬限制
    })
  )
  async uploadAvatar(
    @CurrentUser("userId") userId: string,
    @UploadedFile() file: Express.Multer.File
  ) {
    return this.uploadService.uploadAvatar(userId, file);
  }

  @Post("cover")
  @Roles("AUTHOR", "ADMIN")
  @UseInterceptors(
    FileInterceptor("cover", {
      storage: memoryStorage(),
      limits: { fileSize: 10 * 1024 * 1024 },
    })
  )
  async uploadCover(
    @CurrentUser("userId") userId: string,
    @UploadedFile() file: Express.Multer.File
  ) {
    return this.uploadService.uploadCover(userId, file);
  }
}
```

注意两处限制：

- 头像上传没有 `@Roles()`，意味着所有已登录用户都能上传自己的头像
- 封面图上传限制了 `@Roles('AUTHOR', 'ADMIN')`，reader 不能上传封面图

Multer 的 `limits.fileSize` 是第一道防线。如果文件超过这个大小，Multer 会直接报错，不会把整个文件读进内存。但这个限制基于 `Content-Length` header，不是实际文件大小——理论上可以被伪造。所以在 Service 层还要做一次检查。

Service 里的处理逻辑：

```typescript
// src/upload/upload.service.ts
import { Injectable, BadRequestException, Logger } from "@nestjs/common";
import * as sharp from "sharp";
import * as path from "path";
import * as fs from "fs";
import { fileTypeFromBuffer } from "file-type";
import { randomUUID } from "crypto";

const UPLOAD_DIR = path.resolve(process.cwd(), "uploads");
const ALLOWED_IMAGE_MIMES = ["image/jpeg", "image/png", "image/webp"];

@Injectable()
export class UploadService {
  private readonly logger = new Logger(UploadService.name);

  constructor() {
    this.ensureDirs();
  }

  private ensureDirs() {
    const dirs = [
      path.join(UPLOAD_DIR, "avatars"),
      path.join(UPLOAD_DIR, "covers"),
    ];
    dirs.forEach((dir) => {
      if (!fs.existsSync(dir)) {
        fs.mkdirSync(dir, { recursive: true });
      }
    });
  }

  async uploadAvatar(userId: string, file: Express.Multer.File) {
    // 1. 大小检查（Service 层二次确认）
    if (file.size > 2 * 1024 * 1024) {
      throw new BadRequestException("头像大小不能超过 2MB");
    }

    // 2. 文件类型检测（file-type 读文件头，不看扩展名）
    const type = await fileTypeFromBuffer(file.buffer);
    if (!type || !ALLOWED_IMAGE_MIMES.includes(type.mime)) {
      throw new BadRequestException("只支持 JPG、PNG、WebP 格式的图片");
    }

    // 3. 用 Sharp 压缩 + 裁剪为正方形 256x256
    const ext = type.ext === "jpg" ? "jpg" : type.ext;
    const filename = `${userId}-${randomUUID().slice(0, 8)}.${ext}`;
    const filepath = path.join(UPLOAD_DIR, "avatars", filename);

    await sharp(file.buffer)
      .resize(256, 256, { fit: "cover", position: "center" })
      .jpeg({ quality: 85 })
      .toFile(filepath);

    this.logger.log(`用户 ${userId} 上传头像: ${filename}`);

    return { url: `/uploads/avatars/${filename}` };
  }

  async uploadCover(userId: string, file: Express.Multer.File) {
    if (file.size > 5 * 1024 * 1024) {
      throw new BadRequestException("封面图大小不能超过 5MB");
    }

    const type = await fileTypeFromBuffer(file.buffer);
    if (!type || !ALLOWED_IMAGE_MIMES.includes(type.mime)) {
      throw new BadRequestException("只支持 JPG、PNG、WebP 格式的图片");
    }

    // 封面图保持比例，宽度最大 1200px，高度最大 630px（适合社交分享）
    const ext = type.ext === "jpg" ? "jpg" : type.ext;
    const filename = `${userId}-${randomUUID().slice(0, 8)}.${ext}`;
    const filepath = path.join(UPLOAD_DIR, "covers", filename);

    await sharp(file.buffer)
      .resize(1200, 630, { fit: "inside", withoutEnlargement: true })
      .jpeg({ quality: 85 })
      .toFile(filepath);

    return { url: `/uploads/covers/${filename}` };
  }
}
```

这里有几个设计决策值得说清楚。

### 为什么用内存存储而不是磁盘存储

`FileInterceptor` 配置 `storage: memoryStorage()` 后，文件内容在 `file.buffer` 中。对于头像和封面图（通常不超过 5MB），内存存储没问题。好处是可以直接把 buffer 传给 Sharp 处理，不用先写磁盘再读出来。

如果文件更大（比如几十 MB 的图片），内存存储会占用较多内存。这时候应该切换到磁盘存储：

```typescript
import { diskStorage } from 'multer';
import { extname } from 'path';

@UseInterceptors(FileInterceptor('file', {
  storage: diskStorage({
    destination: './uploads/temp',
    filename: (req, file, cb) => {
      cb(null, `${randomUUID()}${extname(file.originalname)}`);
    },
  }),
}))
```

磁盘存储时，`file.buffer` 为 undefined，文件内容在 `file.path` 指向的磁盘路径上。Sharp 可以直接读文件路径：

```typescript
await sharp(file.path).resize(256, 256).toFile(outputPath);
// 处理完后删除临时文件
fs.unlinkSync(file.path);
```

### 文件命名规则

`${userId}-${randomUUID前8位}.jpg`——用户 ID 保证同一用户的文件可追溯，UUID 防止覆盖。不使用原始文件名，因为：

- 中文文件名在 URL 中需要编码，`我的头像.jpg` 变成 `%E6%88%91%E7%9A%84%E5%A4%B4%E5%83%8F.jpg`
- 可能包含路径遍历字符（`../../etc/passwd`），虽然 Multer 会做一定程度的清理，但不能依赖它
- 不同用户可能上传同名文件

### Sharp 的 resize 参数

头像用 `fit: 'cover'`，意思是裁剪为正方形，居中取最大区域。一张 4000x3000 的横图会被裁成 256x256，左右各裁掉一部分。`position: 'center'` 保证取中间区域——大多数头像的人脸在中间。

`fit: 'inside'` 则保持比例，图片不会超出目标尺寸。封面图用 `inside`，因为宽图和窄图的比例不同，强行裁剪会丢内容。`withoutEnlargement: true` 确保小图不会被放大——一张 800x400 的图不会被拉伸到 1200x630。

## 抽象存储策略

现在文件存在本地磁盘。明天产品说"我们要迁移到阿里云 OSS"，后天又说要支持 AWS S3。如果存储逻辑散落在 Service 里，迁移就是灾难。

抽象一个 `StorageService` 接口：

```typescript
// src/upload/storage/storage.interface.ts
export interface StorageService {
  save(buffer: Buffer, filepath: string): Promise<string>;
  delete(filepath: string): Promise<void>;
  getUrl(filepath: string): string;
}
```

接口只有三个方法：保存、删除、获取 URL。所有上传逻辑都通过这个接口操作文件，不直接调用 `fs`。

本地存储的实现：

```typescript
// src/upload/storage/local.storage.ts
import { Injectable } from "@nestjs/common";
import * as fs from "fs";
import * as path from "path";
import { StorageService } from "./storage.interface";

const BASE_PATH = path.resolve(process.cwd(), "uploads");

@Injectable()
export class LocalStorageService implements StorageService {
  async save(buffer: Buffer, filepath: string): Promise<string> {
    const fullPath = path.join(BASE_PATH, filepath);
    const dir = path.dirname(fullPath);

    if (!fs.existsSync(dir)) {
      fs.mkdirSync(dir, { recursive: true });
    }

    fs.writeFileSync(fullPath, buffer);
    return filepath;
  }

  async delete(filepath: string): Promise<void> {
    const fullPath = path.join(BASE_PATH, filepath);
    if (fs.existsSync(fullPath)) {
      fs.unlinkSync(fullPath);
    }
  }

  getUrl(filepath: string): string {
    return `/uploads/${filepath}`;
  }
}
```

以后加一个 `OssStorageService` 实现同一个接口。Controller 和 Service 不需要改一行代码，只需要在 Module 里切换 provider。

用工厂模式根据环境变量决定用哪个实现：

```typescript
// src/upload/upload.module.ts
import { Module, DynamicModule, Provider } from "@nestjs/common";
import { ConfigService } from "@nestjs/config";
import { LocalStorageService } from "./storage/local.storage.service";
import { StorageService } from "./storage/storage.interface";
import { UploadService } from "./upload.service";
import { UploadController } from "./upload.controller";

@Module({})
export class UploadModule {
  static register(): DynamicModule {
    const storageProvider: Provider = {
      provide: "StorageService",
      useFactory: (configService: ConfigService) => {
        const storageType = configService.get<string>("STORAGE_TYPE", "local");
        if (storageType === "oss") {
          // return new OssStorageService(configService);
        }
        return new LocalStorageService();
      },
      inject: [ConfigService],
    };

    return {
      module: UploadModule,
      providers: [storageProvider, UploadService],
      controllers: [UploadController],
      exports: [UploadService, "StorageService"],
    };
  }
}
```

环境变量 `STORAGE_TYPE=local` 用本地，`STORAGE_TYPE=oss` 用云存储。Service 里注入 `@Inject('StorageService')` 就行。

`AppModule` 里注册：

```typescript
import { UploadModule } from "./upload/upload.module";

@Module({
  imports: [
    // ...
    UploadModule.register(),
  ],
})
export class AppModule {}
```

`DynamicModule` 让模块可以接受参数，根据配置动态决定提供哪些 provider。比在 Module 里硬编码灵活得多。

## Multer 配置：统一拦截器

现在 Multer 的配置还在 Controller 的装饰器里。如果多个上传接口都需要内存存储和大小限制，最好抽一个通用的拦截器工厂：

```typescript
// src/upload/interceptors/memory-upload.interceptor.ts
import { UseInterceptors, applyDecorators } from "@nestjs/common";
import { FileInterceptor, FilesInterceptor } from "@nestjs/platform-express";
import { memoryStorage } from "multer";

export function MemoryFileUpload(fieldName: string, maxSize: number) {
  return applyDecorators(
    UseInterceptors(
      FileInterceptor(fieldName, {
        storage: memoryStorage(),
        limits: { fileSize: maxSize },
      })
    )
  );
}

export function MemoryFilesUpload(
  fieldName: string,
  maxCount: number,
  maxSize: number
) {
  return applyDecorators(
    UseInterceptors(
      FilesInterceptor(fieldName, maxCount, {
        storage: memoryStorage(),
        limits: { fileSize: maxSize },
      })
    )
  );
}
```

使用时：

```typescript
@Post('avatar')
@MemoryFileUpload('avatar', 2 * 1024 * 1024)
async uploadAvatar(...) { ... }

@Post('covers')
@MemoryFilesUpload('covers', 5, 5 * 1024 * 1024)
async uploadCovers(...) { ... }
```

`applyDecorators` 把多个装饰器合并成一个，保持代码整洁。NestJS 会自动把它们展开为原始装饰器。

`FilesInterceptor` 和 `FileInterceptor` 的区别在于：`FilesInterceptor` 接收多个文件（同一字段名），用 `@UploadedFiles()` 获取文件数组。适用于批量上传场景，比如一次上传 5 张文章配图。

## 静态文件访问

文件存在 `uploads/` 目录下，但 HTTP 请求默认访问不到这个目录。需要在 `main.ts` 里注册静态文件中间件：

```typescript
// src/main.ts
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";
import { ValidationPipe } from "@nestjs/common";
import * as path from "path";

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  app.useGlobalPipes(new ValidationPipe());

  // 把 /uploads 路径映射到 uploads/ 目录
  app.useStaticAssets(path.resolve(process.cwd(), "uploads"), {
    prefix: "/uploads/",
  });

  await app.listen(3000);
}
bootstrap();
```

配置之后，`/uploads/avatars/usr_a1x9k2m-abc12345.jpg` 就能直接通过浏览器访问了。

注意 `prefix: '/uploads/'` 和文件路径的对应关系。`app.useStaticAssets(baseDir, { prefix })` 的意思是：URL 中以 `/uploads/` 开头的请求，会去 `baseDir` 目录下找对应文件。所以 `/uploads/avatars/xxx.jpg` 实际找的是 `<baseDir>/avatars/xxx.jpg`。

## 错误处理：Multer 的异常不友好

当文件超过大小限制时，Multer 抛出的错误长这样：

```json
{
  "statusCode": 500,
  "message": "File too large",
  "error": "Internal Server Error"
}
```

状态码是 500，不应该是 500——文件太大应该返回 413 Payload Too Large。而且 `File too large` 这个消息对中文用户不友好。

加一个全局异常过滤器来处理 Multer 错误：

```typescript
// src/common/filters/multer.filter.ts
import {
  ExceptionFilter,
  Catch,
  ArgumentsHost,
  HttpException,
  PayloadTooLargeException,
} from "@nestjs/common";
import { Response } from "express";

@Catch()
export class MulterExceptionFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();

    // Multer 错误的 name 是 'MulterError'
    if (exception instanceof Error && exception.name === "MulterError") {
      let status = 400;
      let message = "文件上传失败";

      if (exception.message === "File too large") {
        status = 413;
        message = "文件大小超过限制";
      } else if (exception.message === "Unexpected field") {
        message = "请求字段名不正确";
      } else if (exception.message === "Too many files") {
        message = "文件数量超过限制";
      }

      response.status(status).json({
        statusCode: status,
        message,
        error: "Bad Request",
      });
      return;
    }

    // 非 Multer 错误，继续抛出
    if (exception instanceof HttpException) {
      throw exception;
    }

    response.status(500).json({
      statusCode: 500,
      message: "服务器内部错误",
      error: "Internal Server Error",
    });
  }
}
```

在 `main.ts` 里注册：

```typescript
import { MulterExceptionFilter } from "./common/filters/multer.filter";

app.useGlobalFilters(new MulterExceptionFilter());
```

注意这个过滤器的位置：它应该放在其他异常过滤器之前，这样 Multer 错误会先被它捕获。如果你的项目里有全局的 `HttpExceptionFilter`，确保 `MulterExceptionFilter` 先注册。

## 文件路径设计

最后聊一个容易被忽略的问题：文件在服务器上的目录结构。

方案 A：按用户分目录

```
uploads/
├── avatars/
│   └── usr_a1x9k2m-abc12345.jpg
└── covers/
    ├── usr_a1x9k2m-def67890.jpg
    └── usr_b2c3d4e-ghi11223.jpg
```

方案 B：按日期分目录

```
uploads/
├── avatars/
│   └── 2026/03/30/
│       └── usr_a1x9k2m-abc12345.jpg
└── covers/
    └── 2026/03/30/
        └── usr_a1x9k2m-def67890.jpg
```

方案 C：按类型哈希打散

```
uploads/
├── avatars/
│   └── a1/
│       └── usr_a1x9k2m-abc12345.jpg
└── covers/
    └── b2/
        └── usr_b2c3d4e-def67890.jpg
```

方案 A 方便查找某个用户的所有文件，方案 B 方便按时间做归档和清理，方案 C 适合单目录文件过多的情况。博客平台推荐方案 A——用户量不大（通常几千到几万），按类型分目录就不会有单目录文件过多的问题。如果用户量到了百万级，再加一层哈希打散。

有一点要注意：文件名中包含用户 ID 时，要确保用户 ID 不包含特殊字符。UUID 格式（`usr_a1x9k2m`）是安全的。如果用户 ID 可能包含 `/`、`\`、`..` 等字符，需要做清理。

## 更新数据库中的文件路径

上传头像后，要把 URL 写回用户表，不然下次请求用户信息时拿不到头像地址。

```typescript
// src/users/users.service.ts
async updateAvatar(userId: string, avatarUrl: string) {
  const user = await this.prisma.user.findUnique({ where: { id: userId } });
  if (!user) {
    throw new NotFoundException('用户不存在');
  }

  // 如果用户已有旧头像，删除旧文件
  if (user.avatar) {
    const oldPath = user.avatar.replace('/uploads/', '');
    await this.storageService.delete(oldPath);
  }

  await this.prisma.user.update({
    where: { id: userId },
    data: { avatar: avatarUrl },
  });

  return avatarUrl;
}
```

Service 层注入 `StorageService`，删除旧头像时通过接口操作——不直接调用 `fs.unlinkSync`。这样切换到 OSS 后，删除逻辑不用改。

完整的头像上传流程变成：

```
用户选择图片
  → 前端 FormData 发送到 POST /upload/avatar
  → Multer 接收到内存 buffer
  → UploadService 校验类型、压缩图片、存储文件
  → 返回 { url: "/uploads/avatars/usr_a1x9k2m-abc12345.jpg" }
  → 前端拿到 url，调用 PUT /users/me/avatar { avatar: url }
  → UsersService 更新数据库
```

也可以在 UploadService 里直接更新数据库，省掉前端的那次额外请求。选择哪种取决于你的架构偏好——UploadService 只管文件，User 更新由 UserService 做（职责清晰），还是 UploadService 管到底（请求次数少）。

## 图片格式转换

用户可能上传 PNG（有透明通道）或 WebP 格式的图片。Sharp 统一输出为 JPEG 可以减小文件体积，但会丢失透明度。如果封面图需要透明背景（比如 logo），需要保留原始格式：

```typescript
async uploadCover(userId: string, file: Express.Multer.File) {
  const type = await fileTypeFromBuffer(file.buffer);

  // PNG 保留格式（可能有透明通道），其他转 JPEG
  const isPng = type?.mime === 'image/png';
  const ext = isPng ? 'png' : 'jpg';
  const filename = `${userId}-${randomUUID().slice(0, 8)}.${ext}`;
  const filepath = path.join(UPLOAD_DIR, 'covers', filename);

  const pipeline = sharp(file.buffer)
    .resize(1200, 630, { fit: 'inside', withoutEnlargement: true });

  if (isPng) {
    await pipeline.png({ quality: 85 }).toFile(filepath);
  } else {
    await pipeline.jpeg({ quality: 85 }).toFile(filepath);
  }

  return { url: `/uploads/covers/${filename}` };
}
```

WebP 格式更复杂——它通常比 JPEG 体积更小，但不是所有浏览器都支持。可以在响应头中根据 `Accept` header 决定返回 WebP 还是 JPEG。不过这属于 CDN 层面的优化，应用层不需要关心。

## 生成缩略图

封面图在文章列表页展示时，不需要 1200px 宽的大图。生成一张 400px 宽的缩略图用于列表展示，可以大幅减少页面加载时间。

Sharp 的 `clone()` 方法可以在同一条 pipeline 上创建分支，一次读取、两次处理：

```typescript
async uploadCoverWithThumbnail(userId: string, file: Express.Multer.File) {
  const type = await fileTypeFromBuffer(file.buffer);
  const ext = type?.ext === 'jpg' ? 'jpg' : type?.ext;
  const filename = `${userId}-${randomUUID().slice(0, 8)}.${ext}`;

  const originalPath = path.join(UPLOAD_DIR, 'covers', filename);
  const thumbPath = path.join(UPLOAD_DIR, 'covers', 'thumb', filename);

  // 确保缩略图目录存在
  const thumbDir = path.dirname(thumbPath);
  if (!fs.existsSync(thumbDir)) {
    fs.mkdirSync(thumbDir, { recursive: true });
  }

  // 原图：1200x630
  await sharp(file.buffer)
    .resize(1200, 630, { fit: 'inside', withoutEnlargement: true })
    .jpeg({ quality: 85 })
    .toFile(originalPath);

  // 缩略图：400x210
  await sharp(file.buffer)
    .resize(400, 210, { fit: 'inside', withoutEnlargement: true })
    .jpeg({ quality: 80 })
    .toFile(thumbPath);

  return {
    url: `/uploads/covers/${filename}`,
    thumbnailUrl: `/uploads/covers/thumb/${filename}`,
  };
}
```

两次调用 `sharp(file.buffer)`，每次都从同一个 buffer 创建新的 pipeline。这比读两次磁盘快得多——buffer 已经在内存里了。

## Trade-off 小结

**内存存储 vs 磁盘存储**：5MB 以下用内存存储，省一次磁盘 IO。超过 5MB 切磁盘存储，避免内存压力。选择边界取决于服务器的可用内存和并发上传量。

**file-type 检测 vs 扩展名检查**：file-type 更安全但需要额外的 npm 包和计算开销。两层都做，扩展名做快速过滤（Multer 的 `fileFilter` 里同步执行），file-type 做最终确认（Service 层异步执行）。

**存储策略抽象**：多写一个接口和几个实现类，换来的是随时切换存储后端的能力。对于确定只用本地存储的小项目来说可能过度设计，但对于博客平台这种大概率会迁移云存储的场景，值得投入。

**Sharp 处理 vs 前端处理**：前端可以做图片压缩（Canvas API），但后端处理更可靠——前端压缩的结果取决于用户浏览器，不同浏览器行为可能不一致。后端统一处理，输出质量和尺寸可控。

**Multer 大小限制 vs Service 层检查**：两层都做。Multer 的限制防止大文件进入内存（内存存储时），Service 层的检查是兜底。Multer 的限制基于 Content-Length header，理论上可以被绕过。
