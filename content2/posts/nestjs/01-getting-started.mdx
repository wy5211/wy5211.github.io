# 从零搭建：为什么选 NestJS

团队要启动一个新项目，技术选型会上吵了半个小时。后端用 Express 吧，自由度太高，半年后代码一定是一锅粥；用 Spring Boot 吧，Java 生态是强，但团队全是 TypeScript，转型成本太高；Fastify 性能好，可它本质上还是个路由库，项目大了照样乱。

这种选型困境你大概也遇到过。你需要的是一个东西：**既有一套强制性的架构规范，又不会因为规范本身把项目搞得太重**。

NestJS 试图回答的就是这个问题。

## NestJS 到底是什么

很多人把 NestJS 和 Express、Koa 放在一起比较，觉得它们是同一类东西。这个认知在起点就偏了。

Express 是一个 HTTP 路由库——它帮你监听端口、匹配路径、返回响应，剩下的全由你自己决定。Fastify 是 Express 的高性能替代品，做的事情一样，只是更快。而 NestJS 不和它们竞争，它**运行在 Express 或 Fastify 之上**，提供的是架构层面的东西。

打个比方：Express 是砖头和水泥，你可以盖任何风格的房子，也可能盖成危房。NestJS 是一个建筑框架，它规定了承重墙在哪、管线怎么走、房间怎么分隔，你在这个框架里盖房子，至少结构上是安全的。

这个建筑框架的设计灵感来自 Angular。NestJS 的创始人 Kamil Mysliwiec 之前做 Angular 开发，他把前端的依赖注入、模块化、装饰器那一套搬到了 Node.js 后端。所以如果你用过 Angular，NestJS 的代码组织方式会让你觉得非常熟悉。

## 用 NestJS 搭一个健康检查服务

光说概念没意思，我们直接动手搭一个东西。假设你刚接手一个线上服务，运维要求你提供一个健康检查接口，返回服务状态、数据库连接情况和内存使用量。这在真实的生产环境里几乎是标配——Kubernetes 的 liveness probe、负载均衡器的健康检测，都需要这样一个端点。

先安装 Nest CLI，然后创建项目：

```bash
npm install -g @nestjs/cli
nest new health-check-api --package-manager pnpm
```

CLI 会生成一套标准的目录结构。进去看看 src 下面有什么：

```
src/
├── main.ts              # 入口文件
├── app.module.ts        # 根模块
├── app.controller.ts    # 根控制器
└── app.service.ts       # 根服务
```

就四个文件。我们先把 `main.ts` 看清楚——这是整个应用的起点。

## main.ts 里发生了什么

打开 `src/main.ts`，你会看到一段非常短的代码：

```typescript
import { NestFactory } from "@nestjs/core";
import { AppModule } from "./app.module";

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  await app.listen(3000);
}
bootstrap();
```

别被这短短几行骗了。`NestFactory.create(AppModule)` 这一行背后，NestJS 做了很多事：它创建了一个 Express（或 Fastify）实例，实例化了 IoC 容器，扫描了 AppModule 里声明的所有模块、控制器和服务，建立了它们之间的依赖关系，然后才把 HTTP 服务器启动起来。

这就是 NestJS 和 Express 的本质区别。用 Express 写 `app.listen(3000)`，你只是启动了一个空壳服务器，路由和中间件全靠你手动挂载。而 NestJS 的 `app.listen` 启动的是一个已经装配好的应用——模块、依赖注入、生命周期钩子，全部就绪。

## 实现健康检查端点

我们把这个默认的 Hello World 改成真正的健康检查。先改 `app.service.ts`，让它返回有意义的信息：

```typescript
import { Injectable } from "@nestjs/common";
import * as os from "os";

@Injectable()
export class AppService {
  getHealth() {
    const memUsage = process.memoryUsage();
    const totalMem = os.totalmem();
    const freeMem = os.freemem();

    return {
      status: "ok",
      timestamp: new Date().toISOString(),
      uptime: process.uptime(),
      memory: {
        rss: `${(memUsage.rss / 1024 / 1024).toFixed(2)} MB`,
        heapUsed: `${(memUsage.heapUsed / 1024 / 1024).toFixed(2)} MB`,
        systemTotal: `${(totalMem / 1024 / 1024 / 1024).toFixed(2)} GB`,
        systemFree: `${(freeMem / 1024 / 1024 / 1024).toFixed(2)} GB`,
      },
    };
  }
}
```

这里用到了 Node.js 自带的 `process.memoryUsage()` 和 `os` 模块，不需要装任何额外依赖。`rss` 是常驻内存，`heapUsed` 是 V8 堆内存使用量——这些是排查内存泄漏时最常看的两个指标。

然后改控制器，把它暴露成 HTTP 端点：

```typescript
import { Controller, Get } from "@nestjs/common";
import { AppService } from "./app.service";

@Controller("health")
export class AppController {
  constructor(private readonly appService: AppService) {}

  @Get()
  getHealth() {
    return this.appService.getHealth();
  }
}
```

注意控制器里没有任何业务逻辑，它只做一件事：调用服务，返回结果。这种分离不是形式主义——当你的应用有几十个接口时，控制器只管 HTTP 层的事情（参数提取、状态码设置），业务逻辑全部在 Service 里，测试和复用都会轻松很多。

启动项目，访问 `http://localhost:3000/health`，你会看到类似这样的响应：

```json
{
  "status": "ok",
  "timestamp": "2026-03-30T10:30:00.000Z",
  "uptime": 123.456,
  "memory": {
    "rss": "52.14 MB",
    "heapUsed": "28.73 MB",
    "systemTotal": "16.00 GB",
    "systemFree": "4.21 GB"
  }
}
```

这就是一个能直接用在生产环境的健康检查接口了。

## 热重载是怎么回事

开发时运行 `pnpm run start:dev`，你会发现改完代码保存后，终端会自动重启应用。这不是 nodemon，是 NestJS 自己实现的。

NestJS 的 `start:dev` 命令底层用了两个进程：一个 watch 进程监听文件变化，一个编译进程把 TypeScript 编译成 JavaScript。文件变化时，watch 进程会重新触发编译，然后重启应用。和 nodemon 相比，NestJS 的方案更智能——它知道哪些文件改了需要重启，哪些只是类型变化不需要重启。

如果你的项目编译比较慢（几百个文件那种），可以在 `nest-cli.json` 里开启 webpack 模式来加速：

```json
{
  "compilerOptions": {
    "webpack": true,
    "deleteOutDir": true
  }
}
```

开启后，NestJS 会用 webpack 打包，利用它的增量编译能力，重启速度会快不少。不过对于大多数中小项目，默认的 tsc 编译就够了。

## 项目结构背后的设计意图

回头看 CLI 生成的那四个文件，每个文件都有明确的职责：

- `main.ts` 只负责启动应用，不应该放任何业务逻辑
- `app.module.ts` 是根模块，描述应用的组成部分
- `app.controller.ts` 处理 HTTP 请求
- `app.service.ts` 包含业务逻辑

这个结构不是 NestJS 强行规定的，而是**依赖注入 + 模块化**这套架构的自然结果。控制器需要服务，所以通过构造函数声明依赖；服务和控制器都属于某个模块，所以在模块里声明归属关系。每个组件都只做一件事，职责清晰。

你可能会觉得："就一个健康检查接口，分三个文件是不是过度设计了？"确实，小项目这样分显得啰嗦。但 NestJS 的目标用户不是写一个接口就收工的场景，而是那种要维护两三年、几十个开发者协作的中大型项目。在这类项目里，强制性的结构规范带来的收益远大于多写几个文件的成本。

## 什么时候该选 NestJS

说了这么多，给一个务实的判断标准：

**适合用 NestJS 的场景**：团队超过 3 个人、项目预期会持续迭代、后端接口超过 20 个、需要对接数据库/缓存/消息队列等中间件。这种情况下，NestJS 的模块化和依赖注入会显著降低维护成本。

**不适合的场景**：一个临时脚本、一个只有两三个接口的内部工具、对启动速度有极端要求的 Serverless 函数。这种场景下，Express 甚至一个简单的 `http.createServer` 就够了，引入 NestJS 的学习成本和运行时开销不划算。

**和同类方案的对比**：如果你已经在用 Spring Boot 并且团队以 Java 为主，没必要为了用 TypeScript 而迁移到 NestJS。如果你想要更轻量的 Node.js 方案同时保持一定的架构约束，可以看看 Fastify + @fastify/awilix 这个组合——但它需要你自己组装，不像 NestJS 开箱即用。
