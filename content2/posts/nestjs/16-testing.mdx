## 一个真实的问题

上周五下午，团队改了一段认证逻辑——把 JWT 过期时间从 30 分钟改成 15 分钟。改动只涉及一个常量，代码审查也通过了。部署之后，客服开始收到反馈：用户说登录后没过多久就被踢出来了，而且有人反馈文章编辑到一半保存失败。

排查了两个小时，发现原因：认证中间件在 token 过期后没有正确返回 401，而是抛了一个未捕获的异常，导致 NestJS 全局异常过滤器返回了 500。前端没有处理 500 的逻辑，直接白屏。

问题根源不是那行常量——而是没有人测过"token 过期后调用需要认证的接口"这条路径。手动测一次当然能发现，但你的博客平台有 20 多个接口，每次改代码都手动过一遍不现实。

这篇来解决这件事：怎么给博客平台写测试，让核心路径有保障，同时不把测试变成维护负担。

## 测试的三层金字塔

先搞清楚一个事情：不是所有测试都一样。测试按范围分为三层：

**单元测试**——测一个函数、一个方法。输入确定，输出确定。Mock 掉所有外部依赖（数据库、HTTP 请求），只验证逻辑本身。

**集成测试**——测一个模块的多个组件协同工作。比如测试 Controller + Service 的组合，Service 用真实依赖但数据库可以用测试库。

**E2E 测试**——启动整个应用，从 HTTP 请求进去，到 HTTP 响应出来。模拟真实用户操作。

用博客平台举例：

- 单元测试：`UsersService.findByEmail()` 传入 `zhang_wei@example.com`，应该返回对应用户数据，找不到时返回 `null`。
- 集成测试：`POST /auth/register` 传入合法数据，Controller 能正确接收、Service 能正确创建用户、响应格式正确。
- E2E 测试：注册 → 登录 → 拿到 token → 发文章 → 查文章，完整链路走通。

三者的 trade-off 很明确：单元测试快但覆盖面窄，E2E 测试覆盖面广但慢且脆弱。一个合理的比例大致是 70/20/10——单元测试最多，E2E 测试只覆盖最关键的用户路径。

## Jest + @nestjs/testing 的基本配置

NestJS 新项目自带 Jest 配置，`package.json` 里已经预设了几个命令：

```bash
pnpm run test           # 单元测试
pnpm run test:watch     # 监听模式
pnpm run test:cov       # 带覆盖率
pnpm run test:e2e       # E2E 测试
```

Jest 配置文件在 `package.json` 的 `jest` 字段或单独的 `jest.config.ts` 里，默认已经配好了 ts-jest 转换和 spec 文件匹配规则。大多数情况下不需要改。

E2E 测试需要 `supertest`，如果项目里没有，装一下：

```bash
pnpm add -D supertest @types/supertest
```

## UserService 单元测试：Mock 数据库

先给 `UsersService` 写单元测试。这个 Service 依赖 `PrismaService`，但单元测试不应该连接真实数据库——用 Mock 替代。

假设 `UsersService` 有这些方法（前面第 8 篇实现过的）：

```typescript
// src/users/users.service.ts
@Injectable()
export class UsersService {
  constructor(private prisma: PrismaService) {}

  async findByEmail(email: string) {
    return this.prisma.user.findUnique({ where: { email } });
  }

  async create(dto: RegisterDto) {
    const existing = await this.prisma.user.findUnique({
      where: { email: dto.email },
    });
    if (existing) throw new ConflictException("该邮箱已被注册");

    const hashedPassword = await bcrypt.hash(dto.password, 10);
    return this.prisma.user.create({
      data: { email: dto.email, name: dto.name, password: hashedPassword },
      select: {
        id: true,
        email: true,
        name: true,
        role: true,
        createdAt: true,
      },
    });
  }
}
```

测试文件：

```typescript
// src/users/users.service.spec.ts
import { Test, TestingModule } from "@nestjs/testing";
import { ConflictException } from "@nestjs/common";
import { UsersService } from "./users.service";
import { PrismaService } from "../database/prisma.service";
import * as bcrypt from "bcrypt";

// Mock bcrypt，避免单元测试里真的做哈希运算
jest.mock("bcrypt");

describe("UsersService", () => {
  let service: UsersService;
  let prisma: { user: Record<string, jest.Mock> };

  beforeEach(async () => {
    // 创建 Mock 的 PrismaService
    prisma = {
      user: {
        findUnique: jest.fn(),
        create: jest.fn(),
      },
    };

    const module: TestingModule = await Test.createTestingModule({
      providers: [UsersService, { provide: PrismaService, useValue: prisma }],
    }).compile();

    service = module.get<UsersService>(UsersService);
  });

  afterEach(() => jest.clearAllMocks());

  describe("findByEmail", () => {
    it("存在时返回用户对象", async () => {
      const mockUser = {
        id: "usr_a1x9k2m",
        email: "zhang_wei@example.com",
        name: "zhang_wei",
        password: "$2b$10$...",
        role: "USER",
        createdAt: new Date("2025-03-15"),
      };
      prisma.user.findUnique.mockResolvedValue(mockUser);

      const result = await service.findByEmail("zhang_wei@example.com");

      expect(result).toEqual(mockUser);
      expect(prisma.user.findUnique).toHaveBeenCalledWith({
        where: { email: "zhang_wei@example.com" },
      });
    });

    it("不存在时返回 null", async () => {
      prisma.user.findUnique.mockResolvedValue(null);

      const result = await service.findByEmail("notexist@example.com");

      expect(result).toBeNull();
    });
  });

  describe("create", () => {
    it("邮箱已存在时抛出 ConflictException", async () => {
      prisma.user.findUnique.mockResolvedValue({ id: "usr_a1x9k2m" });

      await expect(
        service.create({
          email: "zhang_wei@example.com",
          name: "zhang_wei",
          password: "Password123!",
        })
      ).rejects.toThrow(ConflictException);

      // 不应该调用 create
      expect(prisma.user.create).not.toHaveBeenCalled();
    });

    it("创建成功时返回不含密码的用户", async () => {
      prisma.user.findUnique.mockResolvedValue(null);
      (bcrypt.hash as jest.Mock).mockResolvedValue("$2b$10$mockhash");

      const mockCreated = {
        id: "usr_b3y7n4p",
        email: "li_na@example.com",
        name: "li_na",
        role: "USER",
        createdAt: new Date("2026-03-30"),
      };
      prisma.user.create.mockResolvedValue(mockCreated);

      const result = await service.create({
        email: "li_na@example.com",
        name: "li_na",
        password: "Password123!",
      });

      expect(bcrypt.hash).toHaveBeenCalledWith("Password123!", 10);
      expect(result).toEqual(mockCreated);
      // 确认 create 调用时没有包含原始密码
      expect(prisma.user.create).toHaveBeenCalledWith(
        expect.objectContaining({
          data: expect.objectContaining({
            password: "$2b$10$mockhash",
          }),
        })
      );
    });
  });
});
```

几个关键点解释一下：

`Test.createTestingModule()` 创建一个精简的 NestJS 模块容器，只注册你指定的 provider。用 `useValue` 把 `PrismaService` 替换成 Mock 对象，这样测试不连接任何数据库。

`jest.mock('bcrypt')` 在文件顶部把整个 bcrypt 模块替换成自动 Mock，避免每次测试真的做哈希运算（bcrypt 是 CPU 密集型的）。

每个 `it` 块只验证一件事。"邮箱已存在时抛异常"和"创建成功时返回数据"是两个独立的测试用例，不要合并成一个。

## AuthController 集成测试：supertest 发真实请求

单元测试验证了 Service 逻辑，但 Service 和 Controller 之间的协作也需要验证——参数传递对不对、HTTP 状态码对不对、响应格式对不对。

集成测试用 `supertest` 向 Controller 发真实 HTTP 请求，但 Service 层仍然用 Mock：

```typescript
// src/auth/auth.controller.spec.ts
import { Test, TestingModule } from "@nestjs/testing";
import { INestApplication, ValidationPipe } from "@nestjs/common";
import * as request from "supertest";
import { AuthController } from "./auth.controller";
import { AuthService } from "./auth.service";

describe("AuthController（集成测试）", () => {
  let app: INestApplication;

  const mockAuthService = {
    register: jest.fn(),
    login: jest.fn(),
    getCurrentUser: jest.fn(),
  };

  beforeAll(async () => {
    const module: TestingModule = await Test.createTestingModule({
      controllers: [AuthController],
      providers: [{ provide: AuthService, useValue: mockAuthService }],
    }).compile();

    app = module.createNestApplication();
    // 和真实应用一样启用 ValidationPipe
    app.useGlobalPipes(
      new ValidationPipe({
        whitelist: true,
        forbidNonWhitelisted: true,
        transform: true,
      })
    );
    await app.init();
  });

  afterAll(async () => await app.close());
  afterEach(() => jest.clearAllMocks());

  describe("POST /auth/register", () => {
    it("合法数据返回 201 和用户信息", async () => {
      mockAuthService.register.mockResolvedValue({
        user: { id: "usr_b3y7n4p", email: "li_na@example.com", name: "li_na" },
        accessToken: "mock-access-token",
        refreshToken: "mock-refresh-token",
      });

      const res = await request(app.getHttpServer())
        .post("/auth/register")
        .send({
          email: "li_na@example.com",
          name: "li_na",
          password: "Password123!",
        });

      expect(res.status).toBe(201);
      expect(res.body.user.email).toBe("li_na@example.com");
      expect(res.body.accessToken).toBeDefined();
      expect(mockAuthService.register).toHaveBeenCalledWith({
        email: "li_na@example.com",
        name: "li_na",
        password: "Password123!",
      });
    });

    it("密码太弱返回 400", async () => {
      const res = await request(app.getHttpServer())
        .post("/auth/register")
        .send({
          email: "li_na@example.com",
          name: "li_na",
          password: "123",
        });

      expect(res.status).toBe(400);
      expect(mockAuthService.register).not.toHaveBeenCalled();
    });

    it("缺少必填字段返回 400", async () => {
      const res = await request(app.getHttpServer())
        .post("/auth/register")
        .send({ name: "li_na" });

      expect(res.status).toBe(400);
    });
  });

  describe("POST /auth/login", () => {
    it("正确凭据返回 token", async () => {
      mockAuthService.login.mockResolvedValue({
        accessToken: "mock-access-token",
        refreshToken: "mock-refresh-token",
      });

      const res = await request(app.getHttpServer()).post("/auth/login").send({
        email: "zhang_wei@example.com",
        password: "Password123!",
      });

      expect(res.status).toBe(200);
      expect(res.body.accessToken).toBeDefined();
    });

    it("错误凭据返回 401", async () => {
      mockAuthService.login.mockRejectedValue(
        new (await import("@nestjs/common")).UnauthorizedException(
          "邮箱或密码错误"
        )
      );

      const res = await request(app.getHttpServer()).post("/auth/login").send({
        email: "zhang_wei@example.com",
        password: "wrongpassword",
      });

      expect(res.status).toBe(401);
    });
  });
});
```

注意这里的区别：集成测试启动了一个真正的 NestJS 应用实例（`app`），通过 `supertest` 发送 HTTP 请求。`ValidationPipe` 也启用了，所以密码太弱、缺少字段这些验证逻辑会被真实执行。

这就是集成测试的价值——它验证了"管道、控制器、服务之间的连接"是否正确，而单元测试只验证了"服务内部的逻辑"。

## 文章发布 E2E 测试：完整链路

E2E 测试是最接近用户操作的测试。启动完整应用，连接测试数据库，从注册开始走完整个流程。

先准备测试配置。E2E 测试用单独的数据库，避免污染开发数据：

```typescript
// test/auth.e2e-spec.ts
import { Test, TestingModule } from "@nestjs/testing";
import { INestApplication, ValidationPipe } from "@nestjs/common";
import * as request from "supertest";
import { AppModule } from "../src/app.module";
import { PrismaService } from "../src/database/prisma.service";

describe("文章发布流程（E2E）", () => {
  let app: INestApplication;
  let prisma: PrismaService;

  beforeAll(async () => {
    const moduleFixture: TestingModule = await Test.createTestingModule({
      imports: [AppModule],
    }).compile();

    app = moduleFixture.createNestApplication();
    app.useGlobalPipes(
      new ValidationPipe({ whitelist: true, transform: true })
    );
    await app.init();

    prisma = moduleFixture.get(PrismaService);
  });

  afterAll(async () => {
    // 清理测试数据
    await prisma.comment.deleteMany();
    await prisma.post.deleteMany();
    await prisma.user.deleteMany();
    await app.close();
  });

  let accessToken: string;
  let postId: string;

  it("完整流程：注册 → 登录 → 发文章 → 查文章", async () => {
    // 第一步：注册
    const registerRes = await request(app.getHttpServer())
      .post("/auth/register")
      .send({
        email: "e2e_test@example.com",
        name: "e2e_user",
        password: "Password123!",
      });

    expect(registerRes.status).toBe(201);
    expect(registerRes.body.accessToken).toBeDefined();
    accessToken = registerRes.body.accessToken;

    // 第二步：登录（验证已注册的用户能登录）
    const loginRes = await request(app.getHttpServer())
      .post("/auth/login")
      .send({
        email: "e2e_test@example.com",
        password: "Password123!",
      });

    expect(loginRes.status).toBe(200);
    accessToken = loginRes.body.accessToken;

    // 第三步：发文章
    const createPostRes = await request(app.getHttpServer())
      .post("/posts")
      .set("Authorization", `Bearer ${accessToken}`)
      .send({
        title: "E2E 测试文章：从 Express 迁移到 NestJS 的踩坑记录",
        content: "这篇文章记录了从 Express 迁移到 NestJS 的完整过程...",
        tags: ["Node.js", "NestJS"],
        status: "PUBLISHED",
      });

    expect(createPostRes.status).toBe(201);
    expect(createPostRes.body.id).toBeDefined();
    postId = createPostRes.body.id;

    // 第四步：查文章（不带认证也能查到已发布的文章）
    const getPostRes = await request(app.getHttpServer()).get(
      `/posts/${postId}`
    );

    expect(getPostRes.status).toBe(200);
    expect(getPostRes.body.title).toBe(
      "E2E 测试文章：从 Express 迁移到 NestJS 的踩坑记录"
    );

    // 第五步：查文章列表
    const listRes = await request(app.getHttpServer()).get(
      "/posts?page=1&pageSize=10"
    );

    expect(listRes.status).toBe(200);
    expect(listRes.body.data).toBeInstanceOf(Array);
    expect(listRes.body.data.length).toBeGreaterThan(0);
  });

  it("未认证时发文章返回 401", async () => {
    const res = await request(app.getHttpServer()).post("/posts").send({
      title: "未认证的文章",
      content: "不应该被创建",
    });

    expect(res.status).toBe(401);
  });

  it("编辑自己的文章返回 200", async () => {
    const res = await request(app.getHttpServer())
      .patch(`/posts/${postId}`)
      .set("Authorization", `Bearer ${accessToken}`)
      .send({ title: "修改后的标题" });

    expect(res.status).toBe(200);
    expect(res.body.title).toBe("修改后的标题");
  });
});
```

E2E 测试的关键设计：

`afterAll` 里做了数据清理——先删 comment 再删 post 再删 user，因为外键约束要求按依赖顺序删。这保证了多次运行测试不会因为残留数据而失败。

一个 `it` 块里走了注册→登录→发文章→查文章的完整链路。有人会说"测试应该互相独立"，这没错，但 E2E 测试本身就是验证流程连贯性的，步骤之间有依赖关系是合理的。

`POST /auth/login` 那一步看起来多余（注册已经拿到了 token），但它验证了"登录接口能正确验证密码并返回 token"，这是独立的价值。

## 测试覆盖率：关注路径而不是百分比

运行 `pnpm run test:cov` 会生成覆盖率报告。你会看到类似这样的输出：

```
-------------|---------|----------|---------|---------|
File         | % Stmts | % Branch | % Funcs | % Lines |
-------------|---------|----------|---------|---------|
users.service|    87.5  |     75   |     100 |    87.5 |
auth.service |    72.3  |     60   |     100 |    72.3 |
posts.service|    91.2  |     83.3 |     100 |    91.2 |
-------------|---------|----------|---------|---------|
```

一个常见误区是追求 100% 覆盖率。但 100% 覆盖率不等于没有 bug——它只意味着每一行代码都被执行到了，不代表所有边界情况都被测了。

对博客平台来说，哪些路径最值得测试：

- 认证链路（注册、登录、token 过期、错误凭据）——安全相关，出错影响大
- 文章创建和权限控制（只能编辑自己的文章）——核心业务逻辑
- 数据验证边界（密码太短、邮箱格式错误、标题为空）——防止脏数据入库
- 并发场景（同一邮箱重复注册）——线上真实会遇到的

哪些可以不测或轻量测：

- 简单的 CRUD 查询（如果只是 `prisma.user.findMany()` 没有任何额外逻辑）
- 纯粹的 getter 方法
- 框架自身的行为（NestJS 的装饰器、管道）

## 测试数据库：E2E 测试不应该污染开发数据

E2E 测试连接了真实的数据库。如果你直接用开发数据库跑 E2E 测试，每次运行都会创建用户、发文章，数据越积越多。你需要一个独立的测试数据库。

在 `test` 目录下创建测试专用的 Jest 配置：

```json
// test/jest-e2e.json
{
  "moduleFileExtensions": ["js", "json", "ts"],
  "rootDir": ".",
  "testEnvironment": "node",
  "testRegex": ".e2e-spec.ts$",
  "transform": {
    "^.+\\.(t|j)s$": "ts-jest"
  }
}
```

在 `package.json` 里配置 E2E 测试的环境变量：

```json
{
  "scripts": {
    "test:e2e": "jest --config ./test/jest-e2e.json --runInBand"
  }
}
```

`--runInBand` 让 E2E 测试串行执行。E2E 测试通常涉及共享状态（比如先注册用户再发文章），并行执行会导致竞态条件。

测试数据库的环境变量可以通过 `test/.env.test` 文件配置，在 E2E 测试的 `beforeAll` 里加载：

```typescript
// test/setup.ts
import { ConfigModule } from "@nestjs/config";

// 在测试模块里加载测试环境变量
ConfigModule.forRoot({
  envFilePath: ".env.test",
  isGlobal: true,
});
```

一个更简单的做法是让 Docker Compose 跑一个专门的测试数据库：

```bash
# 启动测试数据库
docker run -d --name blog-test-postgres \
  -e POSTGRES_USER=blog_test \
  -e POSTGRES_PASSWORD=test123 \
  -e POSTGRES_DB=blog_test \
  -p 5433:5432 \
  postgres:16-alpine

# 运行 E2E 测试
DATABASE_URL=postgresql://blog_test:test123@localhost:5433/blog_test pnpm run test:e2e

# 测试完成后清理
docker stop blog-test-postgres && docker rm blog-test-postgres
```

## 测试的投入产出比

写测试有成本。一个现实的问题是：你的项目工期紧，老板问你"写测试要花多少时间"。

一个务实的策略：只给会频繁变化的模块和出 bug 代价高的模块写测试。博客平台里，认证模块和权限控制属于"改了容易出事"的，文章 CRUD 属于"逻辑简单但核心"，WebSocket 和文件上传属于"边界复杂但不容易改"。

对于我们的博客平台，推荐的测试优先级：

1. `AuthService` 单元测试——认证逻辑出错直接导致安全问题
2. `UsersService` 单元测试——用户创建和查询是其他功能的基础
3. `AuthController` 集成测试——验证参数传递和响应格式
4. 文章发布 E2E 测试——验证核心业务链路
5. 其他模块视情况补充

不要试图一次性把所有测试都写完。先覆盖核心路径，后续迭代中逐步补充。一个有 50 个测试用例覆盖核心路径的项目，比一个有 200 个测试用例但大部分是废话的项目有价值得多。

---

## 小结

三种测试各司其职：单元测试保证 Service 逻辑正确，集成测试保证 Controller 和 Service 协作正常，E2E 测试保证完整链路走通。Mock 是单元测试的核心手段，`Test.createTestingModule()` + `useValue` 是 NestJS 提供的标准方式。supertest 让你能不发真实请求就测 Controller 的 HTTP 行为。覆盖率数字是参考，不是目标——关注关键路径是否被覆盖。
