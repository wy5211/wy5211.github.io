上一篇我们讲了 Hash 类型，适合存储字段固定的对象。但有些场景需要保持数据的顺序，比如"最新的 10 条弹幕"、"先到先得的任务队列"。这时候就需要 List 类型。

List 是一个链表结构，可以快速地在头部或尾部插入和删除元素。

## 为什么需要队列

想象一个直播场景：用户发送弹幕后，需要做很多处理 - 敏感词过滤、存入数据库、推送给其他观众。如果这些操作都在主流程中同步执行，用户会感觉弹幕发送很慢。

更好的做法是把弹幕先放入队列，然后异步处理。这样用户可以立即看到"发送成功"，后台慢慢处理实际的业务逻辑。

Redis List 非常适合实现这个队列。

## 基础队列实现

List 提供了两对操作：LPUSH/RPOP 和 RPUSH/LPOP。LPUSH 从左边（头部）插入，RPOP 从右边（尾部）弹出。

```typescript
import Redis from "ioredis";

const redis = new Redis();

class MessageQueue {
  constructor(
    private redis: Redis,
    private queueName: string
  ) {}

  // 发送消息（从右边插入）
  async send(message: any): Promise<number> {
    const messageStr = JSON.stringify(message);
    return await this.redis.rpush(this.queueName, messageStr);
  }

  // 接收消息（从左边弹出，实现 FIFO）
  async receive(): Promise<any | null> {
    const messageStr = await this.redis.lpop(this.queueName);
    if (!messageStr) {
      return null;
    }
    return JSON.parse(messageStr);
  }

  // 获取队列长度
  async size(): Promise<number> {
    return await this.redis.llen(this.queueName);
  }

  // 清空队列
  async clear(): Promise<number> {
    return await this.redis.del(this.queueName);
  }
}

// 使用示例：弹幕队列
const danmakuQueue = new MessageQueue(redis, "queue:danmaku");

// 发送弹幕
await danmakuQueue.send({
  roomId: "abc123",
  userId: "usr_a1b2c3",
  content: "主播好帅！",
  timestamp: Date.now(),
});

// 消费者处理弹幕
async function processDanmaku() {
  while (true) {
    const message = await danmakuQueue.receive();
    if (!message) {
      // 队列为空，等待一下
      await new Promise((resolve) => setTimeout(resolve, 100));
      continue;
    }

    // 处理弹幕：过滤、存储、推送等
    console.log("处理弹幕:", message);
    await handleDanmaku(message);
  }
}
```

这个实现有个问题：消费者需要不断轮询，队列空时也会占用 CPU。更好的方式是用阻塞操作。

## 阻塞队列

BLPOP 和 BRPOP 可以阻塞式地弹出元素。当队列为空时，它会一直等待，直到有新元素或超时。

```typescript
class BlockingQueue {
  constructor(
    private redis: Redis,
    private queueName: string
  ) {}

  // 阻塞式接收消息（timeout 为 0 表示永久阻塞）
  async receive(timeout: number = 30): Promise<any | null> {
    const result = await this.redis.blpop(this.queueName, timeout);
    if (!result) {
      return null;
    }

    // result 是一个数组：[queueName, message]
    const [, messageStr] = result;
    return JSON.parse(messageStr);
  }

  // 发送消息（非阻塞）
  async send(message: any): Promise<number> {
    return await this.redis.rpush(this.queueName, JSON.stringify(message));
  }
}

// 使用示例
const danmakuQueue = new BlockingQueue(redis, "queue:danmaku");

async function danmakuWorker() {
  while (true) {
    // 阻塞等待消息，最多等待 30 秒
    const message = await danmakuQueue.receive(30);
    if (!message) {
      console.log("超时，继续等待...");
      continue;
    }

    console.log("收到弹幕:", message);
    await handleDanmaku(message);
  }
}

danmakuWorker();
```

阻塞的好处是节省 CPU 资源。队列为空时，消费者线程会挂起，不会占用 CPU。

## 多消费者模式

实际生产环境中，通常会有多个消费者并行处理消息，提高吞吐量。

```typescript
// 启动多个消费者
async function startWorkers(queueName: string, workerCount: number) {
  const workers: Promise<void>[] = [];

  for (let i = 0; i < workerCount; i++) {
    const worker = async () => {
      const queue = new BlockingQueue(redis, queueName);

      while (true) {
        const message = await queue.receive(30);
        if (!message) continue;

        try {
          await handleMessage(message);
        } catch (error) {
          console.error(`Worker ${i} 处理消息失败:`, error);
          // 可以把失败的消息放入错误队列
          await moveToDeadLetterQueue(message, error);
        }
      }
    };

    workers.push(worker());
  }

  await Promise.all(workers);
}

// 使用示例：启动 5 个弹幕处理消费者
await startWorkers("queue:danmaku", 5);
```

多个消费者可以从同一个队列中取消息，Redis 会保证每条消息只被一个消费者获取（在 BLPOP/BRPOP 的竞争情况下）。

## 弹幕消息队列的完整实现

让我们把弹幕队列做得更完整一些：

```typescript
class DanmakuQueue {
  constructor(private redis: Redis) {}

  // 发送弹幕
  async sendDanmaku(danmaku: {
    roomId: string;
    userId: string;
    username: string;
    content: string;
    timestamp: number;
  }): Promise<boolean> {
    // 1. 敏感词过滤
    if (this.containsSensitiveWords(danmaku.content)) {
      return false;
    }

    // 2. 加入队列
    const queueKey = `queue:danmaku:${danmaku.roomId}`;
    await this.redis.rpush(queueKey, JSON.stringify(danmaku));

    return true;
  }

  // 消费弹幕（直播间维度）
  async consumeDanmaku(roomId: string, timeout: number = 10) {
    const queueKey = `queue:danmaku:${roomId}`;
    const result = await this.redis.blpop(queueKey, timeout);

    if (!result) {
      return null;
    }

    const [, messageStr] = result;
    return JSON.parse(messageStr);
  }

  // 获取队列积压情况
  async getQueueBacklog(roomId: string): Promise<number> {
    const queueKey = `queue:danmaku:${roomId}`;
    return await this.redis.llen(queueKey);
  }

  // 批量获取弹幕（用于页面初始化）
  async getRecentDanmaku(roomId: string, count: number = 50) {
    const queueKey = `queue:danmaku:${roomId}`;
    // LRANGE 从左到右获取，但不会删除元素
    const messages = await this.redis.lrange(queueKey, 0, count - 1);

    return messages.map((msg) => JSON.parse(msg));
  }

  private containsSensitiveWords(content: string): boolean {
    const sensitiveWords = ["脏话", "广告", "敏感词"];
    return sensitiveWords.some((word) => content.includes(word));
  }
}

// 使用示例
const danmakuQueue = new DanmakuQueue(redis);

// 发送弹幕
await danmakuQueue.sendDanmaku({
  roomId: "abc123",
  userId: "usr_a1b2c3",
  username: "wang_dev",
  content: "主播好帅！",
  timestamp: Date.now(),
});

// 消费者处理弹幕
async function danmakuWorker(roomId: string) {
  while (true) {
    const danmaku = await danmakuQueue.consumeDanmaku(roomId);
    if (!danmaku) continue;

    // 1. 存入数据库
    await saveToDatabase(danmaku);

    // 2. 推送给在线观众（用 Pub/Sub，后续会讲）
    await pushToViewers(roomId, danmaku);

    // 3. 更新直播间统计数据
    await updateRoomStats(roomId);
  }
}
```

这个实现按直播间分了队列，不同直播间的弹幕不会互相影响。一个直播间卡了，不会影响其他直播间。

## 栈实现

List 还可以实现栈（LIFO，后进先出）。只需要从同一端插入和弹出：

```typescript
class Stack {
  constructor(
    private redis: Redis,
    private stackName: string
  ) {}

  // 压栈（从左边插入）
  push(item: any): Promise<number> {
    return this.redis.lpush(this.stackName, JSON.stringify(item));
  }

  // 出栈（从左边弹出）
  async pop(): Promise<any | null> {
    const item = await this.redis.lpop(this.stackName);
    return item ? JSON.parse(item) : null;
  }

  // 查看栈顶元素（不删除）
  async peek(): Promise<any | null> {
    const item = await this.redis.lindex(this.stackName, 0);
    return item ? JSON.parse(item) : null;
  }

  // 获取栈大小
  size(): Promise<number> {
    return this.redis.llen(this.stackName);
  }
}

// 使用示例：操作撤销栈
const undoStack = new Stack(redis, "stack:undo:usr_a1b2c3");

await undoStack.push({ action: "delete", target: "comment_123" });
await undoStack.push({ action: "edit", target: "post_456" });

// 撤销时弹出
const lastAction = await undoStack.pop();
console.log("撤销操作:", lastAction);
```

栈的一个典型应用是"撤销"功能。每次用户操作时，把操作记录压入栈；撤销时从栈中弹出，执行反向操作。

## 有限队列

有时候需要限制队列的长度，避免无限增长。可以用 LTRIM 实现：

```typescript
class BoundedQueue {
  constructor(
    private redis: Redis,
    private queueName: string,
    private maxSize: number
  ) {}

  // 发送消息（超过最大长度时，删除最旧的消息）
  async send(message: any): Promise<number> {
    const messageStr = JSON.stringify(message);

    // 先插入
    await this.redis.rpush(this.queueName, messageStr);

    // 修剪队列，只保留最新的 maxSize 条
    await this.redis.ltrim(this.queueName, -this.maxSize, -1);

    return await this.redis.llen(this.queueName);
  }

  // 接收消息
  async receive(): Promise<any | null> {
    const messageStr = await this.redis.lpop(this.queueName);
    return messageStr ? JSON.parse(messageStr) : null;
  }
}

// 使用示例：最多保留 1000 条弹幕
const danmakuQueue = new BoundedQueue(redis, "queue:danmaku:abc123", 1000);
```

LTRIM 会保留指定范围内的元素，删除其他元素。`LTRIM key -1000 -1` 表示只保留最后 1000 条。

## List vs 专门的消息队列

Redis List 虽然可以实现消息队列，但它不是专门的消息队列。和 RabbitMQ、Kafka 相比，有什么区别？

**List 队列的优点：**

- 简单易用，Redis 本身就有
- 性能高，内存操作
- 可以实现基本的队列功能

**List 队列的缺点：**

- 消息一旦被消费就删除，没有持久化（除非开启 AOF）
- 不支持消息确认机制（ACK），消费失败无法重试
- 不支持消息路由、死信队列等高级功能
- Redis 重启且没持久化时，消息会丢失

**什么时候用 List 队列：**

- 消息量不大，丢失几条也能接受
- 需要简单的队列功能，不需要复杂的路由
- 已经在用 Redis，不想引入额外组件
- 对性能要求高，消息处理要快

**什么时候用专门的消息队列：**

- 消息不能丢，需要持久化和确认机制
- 需要消息路由、优先级、延迟等高级功能
- 消息量很大，需要水平扩展
- 需要消息追溯、审计等功能

对于直播平台的弹幕，用 List 队列就够了。弹幕丢失一两条影响不大，但需要高性能。但对于订单、支付这类关键业务，建议用专门的消息队列。

## 总结与 Trade-off

List 类型适合实现简单的消息队列和栈。

**List 的优点：**

- 有序的元素集合，可以保证顺序
- 两端插入删除都很快（O(1)）
- 支持阻塞操作，可以实现消费者模式
- 可以实现队列、栈、有限队列等多种结构

**List 的缺点：**

- 按索引访问是 O(N)，中间元素操作慢
- 元素唯一性需要自己保证
- 消息消费后即删除，没有持久化保证
- 不支持消息确认、重试等机制

**适用场景：**

- 简单的消息队列（弹幕、通知）
- 栈结构（撤销、回溯）
- 最新列表（最新 N 条记录）
- 有限队列（限制长度的队列）
