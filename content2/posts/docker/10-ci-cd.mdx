手动部署是什么感觉？SSH 到服务器、`git pull`、`docker compose build`、`docker compose up -d`，然后盯着日志看有没有报错。一个人维护还好，团队协作的时候，每次部署都是一场仪式——谁有服务器权限、谁负责部署、部署出了问题谁来回滚。

而且手动操作容易出错。有一次我把 `docker compose down -v` 敲成了 `docker compose down`（少了 `-v`），然后以为旧数据卷已经清理了，创建新的数据卷时名字冲突。排查了半小时才发现旧 Volume 还在。

CI/CD 把这些手动步骤变成自动化的流水线：代码推送 → 自动测试 → 自动构建镜像 → 自动部署。这篇用 GitHub Actions 来实现。

## 手动部署的完整流程

先回顾一下手动部署需要哪些步骤，后面每一步都会自动化：

```bash
# 1. SSH 到服务器
ssh deploy@your-server.com

# 2. 进入项目目录，拉取最新代码
cd /opt/order-api
git pull origin main

# 3. 构建镜像
docker compose build

# 4. 停止旧容器，启动新容器
docker compose up -d

# 5. 等待服务启动，检查健康状态
sleep 10
curl http://localhost:3000/health

# 6. 如果出问题，回滚
git checkout HEAD~1
docker compose build
docker compose up -d
```

六个步骤，每一步都可能出错。而且这个过程是不可重复的——不同的人执行可能有细微差异。

## GitHub Actions 基础

GitHub Actions 是 GitHub 内置的 CI/CD 工具。你在项目根目录创建 `.github/workflows/` 目录，在里面写 YAML 文件定义工作流。

一个最简单的 workflow：

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test
```

这个 workflow 做的事情：当代码推送到 `main` 分支或创建 PR 时，在 Ubuntu 虚拟机上拉取代码、安装 Node.js 20、安装依赖、运行测试。

## 自动构建并推送镜像

测试通过后，构建 Docker 镜像并推送到 Docker Hub（或其他镜像仓库）：

```yaml
# .github/workflows/deploy.yml
name: Build and Push

on:
  push:
    branches: [main]

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            yourname/order-api:latest
            yourname/order-api:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

几个关键点：

**`secrets.DOCKERHUB_USERNAME` 和 `secrets.DOCKERHUB_TOKEN`**：这两个值存在 GitHub 仓库的 Settings → Secrets and variables → Actions 里，不会出现在代码中。Docker Hub Token 可以在 Docker Hub 的 Account Settings → Security 里生成。

**镜像标签**：同时推了两个标签。`latest` 是方便拉取最新版，`${{ github.sha }}` 是 Git 提交的 SHA 值（如 `abc123def456`），用于精确回滚到某个版本。

**构建缓存**：`cache-from: type=gha` 和 `cache-to: type=gha,mode=max` 使用 GitHub Actions 的缓存机制。Docker 的层缓存会保存在 GitHub 上，下次构建时直接使用，大幅加速构建过程。第一次构建可能要 2-3 分钟，后续利用缓存只需 30-60 秒。

## 镜像标签策略

推了 `latest` 和 `git sha` 两个标签，什么时候用哪个？

- **`latest`**：开发环境可以用。`docker pull yourname/order-api:latest` 总是拉最新的。
- **`git sha`**：生产环境推荐。精确对应某次代码提交，可以完整追溯。回滚时指定 SHA 即可。
- **语义化版本**：如果你有版本发布流程，可以加一个 `v1.2.3` 的标签。在打 Git tag 时自动触发：

```yaml
on:
  push:
    tags:
      - "v*"
```

生产环境的 Compose 文件应该用精确标签：

```yaml
# docker-compose.prod.yml
services:
  api:
    image: yourname/order-api:abc123def456 # 精确到 git sha
```

## 自动部署到服务器

镜像推送到 Docker Hub 后，需要 SSH 到服务器拉取新镜像并重启服务。

```yaml
# 在 deploy.yml 的 build-and-push job 后面加一个 deploy job
deploy:
  needs: build-and-push
  runs-on: ubuntu-latest
  steps:
    - name: Deploy to server
      uses: appleboy/ssh-action@v1
      with:
        host: ${{ secrets.SERVER_HOST }}
        username: ${{ secrets.SERVER_USER }}
        key: ${{ secrets.SERVER_SSH_KEY }}
        script: |
          cd /opt/order-api
          docker compose pull
          docker compose up -d
          sleep 10
          curl -f http://localhost:3000/health || exit 1
```

`appleboy/ssh-action` 是一个社区维护的 GitHub Action，用于 SSH 到远程服务器执行命令。`SERVER_HOST`、`SERVER_USER`、`SERVER_SSH_KEY` 都存在 GitHub Secrets 里。

部署脚本做了四件事：`docker compose pull` 拉取最新镜像 → `docker compose up -d` 重启服务 → 等待 10 秒 → 检查健康接口。如果健康检查失败（`curl -f` 返回非 200），脚本退出码为 1，GitHub Actions 会标记这次部署为失败。

## 部署后验证

健康检查只是最基本的验证。更完善的部署流程还包括：

**等待就绪**：10 秒的 `sleep` 不太靠谱——有时候服务启动慢一点就误报失败。更好的做法是循环检查：

```bash
for i in $(seq 1 12); do
  if curl -sf http://localhost:3000/health > /dev/null; then
    echo "Service is healthy"
    exit 0
  fi
  echo "Waiting... ($i/12)"
  sleep 5
done
echo "Health check failed"
exit 1
```

最多等 60 秒（12 次 x 5 秒），服务健康了就继续，超时就失败。

**自动回滚**：如果部署失败了，自动回滚到上一个版本。在部署前先记录当前使用的镜像标签：

```bash
# 部署前记录当前版本
CURRENT_IMAGE=$(docker compose images -q api)
echo "Current image: $CURRENT_IMAGE"

# 拉取并启动新版本
docker compose pull api
docker compose up -d

# 健康检查
if ! curl -sf http://localhost:3000/health > /dev/null; then
  echo "Deployment failed, rolling back..."
  # 回滚：把镜像标签指回上一个版本
  docker compose up -d
fi
```

更完善的回滚方案需要在部署前先把上一个版本的镜像 tag 保存下来，这里不展开了。核心思路是：**部署前有快照，失败时有退路**。

## 漏洞扫描集成到 CI

上一篇讲的 `docker scout` 或 `trivy` 可以集成到 CI 流程里，每次构建后自动扫描：

```yaml
security-scan:
  needs: build-and-push
  runs-on: ubuntu-latest
  steps:
    - name: Run Trivy scanner
      uses: aquasecurity/trivy-action@master
      with:
        image-ref: "yourname/order-api:${{ github.sha }}"
        severity: "HIGH,CRITICAL"
        exit-code: "1"
```

`exit-code: '1'` 表示如果发现 HIGH 或 CRITICAL 级别的漏洞，CI 会失败，阻止部署。这样有安全问题的镜像就不会上线。

## 完整的 CI/CD 流程

把上面所有步骤串起来，完整的流程是：

```
代码推送
  ↓
自动测试（npm test）
  ↓
构建 Docker 镜像（利用缓存加速）
  ↓
推送镜像到 Docker Hub（latest + git sha）
  ↓
漏洞扫描（trivy，HIGH/CRITICAL 则阻止部署）
  ↓
SSH 到服务器
  ↓
docker compose pull 拉取新镜像
  ↓
docker compose up -d 重启服务
  ↓
健康检查（最多等 60 秒）
  ↓
成功 → 部署完成
失败 → 自动回滚
```

## CI/CD 不是银弹

自动化部署解决了手动操作的问题，但也引入了新的复杂性：

- **Secret 管理**：Docker Hub Token、SSH Key、数据库密码等敏感信息需要安全存储。GitHub Secrets 是最简单的方式，企业环境可能需要 Vault 或 AWS Secrets Manager。
- **构建时间**：如果镜像构建很慢，每次推送都要等几分钟。构建缓存（我们用了 GitHub Actions 的缓存）能大幅缩短时间。
- **部署速度**：`docker compose pull` 需要从 Docker Hub 下载镜像，网络不好的时候可能很慢。可以用国内镜像加速，或者私有仓库。
- **回滚复杂度**：自动化回滚需要提前设计好版本管理和回滚机制，不是简单的 `git checkout`。

如果你的项目只有一个服务、一个人维护，手动部署可能反而更简单。CI/CD 的价值在团队协作和多环境部署时才会充分体现。

## 回顾

这篇我们用 GitHub Actions 实现了完整的 CI/CD 流程：

- **自动测试**：代码推送后自动运行测试。
- **自动构建**：构建 Docker 镜像并利用 GitHub Actions 缓存加速。
- **镜像标签**：`latest` 用于开发，`git sha` 用于生产。
- **漏洞扫描**：Trivy 集成到 CI，HIGH/CRITICAL 漏洞阻止部署。
- **自动部署**：SSH 到服务器，拉取新镜像，重启服务，健康检查。
- **自动回滚**：部署失败时回退到上一个版本。
