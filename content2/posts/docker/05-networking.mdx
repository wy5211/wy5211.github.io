上一篇我们解决了数据持久化的问题。但有一个更基础的问题还没回答：订单 API 的代码里数据库连接地址写的是 `localhost:5432`，API 容器和 PostgreSQL 容器是两个完全独立的进程——它们各自的 `localhost` 指向完全不同的地方。

先试一下，看看会发生什么：

```bash
# 启动 PostgreSQL
docker run -d --name order-db \
  -e POSTGRES_PASSWORD=order_secret_2025 \
  -p 5432:5432 \
  postgres:16

# 启动订单 API
docker run -d --name order-api \
  -p 3000:3000 \
  -e DATABASE_URL=postgres://postgres:order_secret_2025@localhost:5432/postgres \
  order-api:latest

# 看日志
docker logs order-api
# Error: connect ECONNREFUSED 127.0.0.1:5432
```

连不上。因为 API 容器里的 `localhost` 指向的是 API 容器自己，不是 PostgreSQL 容器。这就像你在北京的公司里说"隔壁办公室"，和你在上海的公司里说"隔壁办公室"，指的是完全不同的房间。

容器之间需要一种机制来互相找到对方。这就是 Docker 网络要解决的问题。

## Docker 的默认网络

每个容器启动时，Docker 会自动把它接入一个默认网络。让我们看看：

```bash
# 查看所有网络
docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# a1b2c3d4e5f6   bridge    bridge    local
# f6e5d4c3b2a1   host      host      local
# 1a2b3c4d5e6f   none      none      local
```

默认网络叫 `bridge`。当你在 `docker run` 时不指定 `--network`，容器就会接入这个默认 bridge 网络。

问题是：**默认 bridge 网络不支持容器名解析。** 也就是说，在默认 bridge 网络里，容器之间只能通过 IP 地址互相访问，不能用容器名。

```bash
# 查看 API 容器的 IP
docker inspect order-api --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'
# 172.17.0.3

# 在 API 容器里 ping PostgreSQL 容器（用容器名——会失败）
docker exec order-api ping order-db
# ping: order-db: Name or service not known

# 用 IP 地址（可以通，但 IP 可能会变）
docker exec order-api ping 172.17.0.2
# PING 172.17.0.2 (172.17.0.2): 56 data bytes
# 64 bytes from 172.17.0.2: ...
```

IP 地址能用，但不可靠。容器每次重新创建，IP 可能会变。你总不能每次部署都手动查 IP 再改配置。

## 自定义 Bridge 网络

解决方案是创建一个自定义 bridge 网络。自定义网络自带 DNS 服务，容器之间可以通过容器名互相访问。

```bash
# 创建自定义网络
docker network create order-network

# 启动 PostgreSQL，接入自定义网络
docker run -d --name order-db \
  --network order-network \
  -e POSTGRES_PASSWORD=order_secret_2025 \
  postgres:16

# 启动 API，接入同一个网络，数据库地址用容器名
docker run -d --name order-api \
  --network order-network \
  -p 3000:3000 \
  -e DATABASE_URL=postgres://postgres:order_secret_2025@order-db:5432/postgres \
  order-api:latest
```

注意变化：`DATABASE_URL` 里的 `localhost` 换成了 `order-db`（PostgreSQL 的容器名）。

验证连通性：

```bash
# 在 API 容器里 ping PostgreSQL（用容器名——成功了）
docker exec order-api ping -c 3 order-db
# PING order-db (172.18.0.2): 56 data bytes
# 64 bytes from 172.18.0.2: seq=0 ttl=64 time=0.123 ms
# 64 bytes from 172.18.0.2: seq=1 ttl=64 time=0.045 ms
# 64 bytes from 172.18.0.2: seq=2 ttl=64 time=0.042 ms

# 测试 API 接口
curl http://localhost:3000/api/orders
# [{"id": 1, "userId": "usr_a1x9k2m", "totalAmount": "8999.00", ...}]
```

连上了。在自定义网络里，Docker 内置的 DNS 会把容器名 `order-db` 解析为对应的 IP 地址。你不需要关心 IP 是什么，只需要记住容器名。

## 端口映射的本质

前面我们一直在用 `-p 3000:3000` 做端口映射，但它的含义值得仔细理解。

```bash
docker run -d --name order-api \
  --network order-network \
  -p 3000:3000 \
  order-api:latest
```

`-p 3000:3000` 表示把宿主机的 3000 端口映射到容器的 3000 端口。外部请求到达宿主机的 3000 端口后，会被转发到容器内的 3000 端口。

但这里有个关键点：**端口映射只影响从宿主机外部访问容器的方式。** 在同一个 Docker 网络内部，容器之间通信不需要端口映射。

什么意思？假设你还有一个 Nginx 容器需要反向代理到 API 容器，它们在同一个网络里：

```bash
docker run -d --name nginx \
  --network order-network \
  -p 80:80 \
  nginx:alpine
```

Nginx 的配置里可以直接用 `http://order-api:3000` 来访问 API，不需要 API 容器做任何端口映射。`-p 3000:3000` 只是让你能从浏览器（宿主机外部）访问 API。

这意味着：在容器网络内部，每个容器的所有端口都是开放的（受容器内防火墙限制）。端口映射只是打开了从宿主机到容器的通道。

这也是为什么 `order-db` 容器没有做 `-p 5432:5432` 端口映射——API 容器可以通过网络内部直接访问 `order-db:5432`，不需要从宿主机暴露 PostgreSQL 的端口。事实上，数据库不对外暴露端口是一个安全最佳实践。

## 把已有的容器加入网络

有时候容器已经启动了，你想把它加入一个网络：

```bash
# 把已运行的 order-api 容器加入 order-network
docker network connect order-network order-api

# 从网络中断开
docker network disconnect order-network order-api
```

一个容器可以同时接入多个网络。这在某些场景下很有用——比如一个 API 容器既需要访问内部的数据库网络，又需要访问外部的缓存网络。

## Host 网络模式

除了 bridge 网络，Docker 还支持 host 网络模式：

```bash
docker run -d --name order-api \
  --network host \
  order-api:latest
```

host 模式下，容器直接使用宿主机的网络栈，不进行任何网络隔离。容器里监听 3000 端口，宿主机上也能直接通过 3000 端口访问，不需要 `-p` 映射。

这个模式调试时很方便——你能直接在宿主机上 `curl localhost:3000` 访问容器内的服务。但在生产环境中不推荐使用，因为它绕过了 Docker 的网络隔离，不同容器的端口可能冲突。

## 排查网络问题

当容器之间连不上时，按这个顺序排查：

```bash
# 1. 确认两个容器在同一个网络里
docker inspect order-api --format '{{json .NetworkSettings.Networks}}'
docker inspect order-db --format '{{json .NetworkSettings.Networks}}'

# 2. 在容器内部测试 DNS 解析
docker exec order-api ping -c 1 order-db

# 3. 如果 ping 不通（容器里可能没有 ping），用其他方式测试
docker exec order-api getent hosts order-db
# 172.18.0.2   order-db

# 4. 测试端口连通性
docker exec order-api nc -zv order-db 5432
# order-db (172.18.0.2:5432) open

# 5. 查看容器的端口监听情况
docker exec order-db ss -tlnp
# LISTEN  0  128  0.0.0.0:5432  *:*
```

常见的网络问题原因：

- **容器不在同一个网络**：最常见的原因。检查两个容器的网络配置。
- **目标服务还没启动完**：PostgreSQL 启动需要几秒钟，如果 API 比 PostgreSQL 先启动，连接会失败。解决方案是应用层面加重试逻辑，或者用 Compose 的 `depends_on`（下一篇讲）。
- **防火墙规则**：某些 Linux 发行版（如 CentOS）的防火墙可能阻止 Docker 的网络流量。

## 回顾

这篇我们解决了容器间通信的问题：

- **默认 bridge 网络不支持容器名解析**，必须用 IP 地址，不可靠。
- **自定义 bridge 网络自带 DNS**，容器之间可以通过容器名通信。始终创建自定义网络。
- **端口映射只影响外部访问**，网络内部通信不需要端口映射。数据库不应该暴露端口到宿主机。
- **排查网络问题**：先确认是否在同一个网络，再测试 DNS 解析和端口连通性。

现在我们知道了怎么创建网络、怎么让容器互相通信。但每次部署都要手动敲一长串命令——创建网络、启动 PostgreSQL、启动 API、挂载 Volume……下一篇的 Docker Compose 就是来终结这个痛苦的。
