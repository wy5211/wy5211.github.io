上一篇我们用 Agent System 快速摸清了 TaskFlow 的整体架构。接下来你开始日常开发，很快发现一个让人头疼的事：每次提交代码都要想 commit message，每次 code review 都要检查"类型安全、错误处理、N+1 查询"这几个点，每次部署前都要手动跑 lint、test、build。

这些事不难，但重复做很烦。而且人工做容易遗漏——哪天你忘了跑测试就提交了，CI 报错，又得回来修。

Claude Code 有两套机制来解决这个问题：**Skills** 和 **Hooks**。它们解决的是同一个问题——把重复性工作自动化——但方式不同。Skills 是你手动触发的指令模板，Hooks 是在特定事件自动执行的命令。

## Skills：一键触发的指令模板

什么是 Skill？简单说，它就是一个**预定义的 Prompt 模板**。你把一套复杂的指令写好，存成一个 Markdown 文件，以后只需要输入 `/skill-name` 就能触发。

举个例子。你的团队 code review 有一套固定标准：检查类型安全、错误处理、N+1 查询、安全漏洞。每次 review 你都要把这些标准重新说一遍。但如果你把它做成一个 Skill，以后只需要 `/code-review` 就能触发。

Skill 文件放在 `.claude/skills/` 目录下（项目级别）或 `~/.claude/skills/` 目录下（用户级别）。一个 Skill 就是一个 Markdown 文件：

```markdown
---
name: code-review
description: 审查代码变更，检查类型安全、错误处理和性能问题
---

审查当前工作区的代码变更，按以下维度检查：

1. 类型安全：是否有 `as any`、不安全的类型断言
2. 错误处理：async 方法是否有 try-catch，异常是否正确传播
3. N+1 查询：循环中是否有数据库查询
4. 安全：用户输入是否经过校验和清洗

输出格式：

- 按严重程度分组（必须修复 / 建议修复）
- 每个问题指出具体文件和行号
- 给出修复建议
```

保存为 `.claude/skills/code-review.md` 之后，你在对话中输入 `/code-review`，Claude Code 就会按照这个模板执行代码审查。

### Skill 的触发条件

Skill 可以配置触发条件，在特定场景下自动激活。比如：

```yaml
---
name: api-error-handler
description: 修复 API 错误处理问题
trigger: "代码中有未处理的异常"
---
```

当 Claude Code 检测到你的请求涉及"代码中有未处理的异常"时，会自动建议你使用这个 Skill。

### 什么时候该做成 Skill

不是所有重复性工作都适合做成 Skill。几个判断标准：

**值得做成 Skill 的**：

- 需要多步指令才能完成的任务（比如"分析代码 → 找问题 → 给修复建议"）
- 团队共享的标准流程（code review 标准、部署检查清单）
- 你经常用但每次都要重新描述的任务

**不值得做成 Skill 的**：

- 一次性任务（分析完就不需要了）
- 太简单的任务（"帮我跑一下测试"直接说就行了）
- 每次参数都完全不同的任务（Skill 模板无法适配）

一个经验法则：**如果你发现自己每周至少用 3 次同一套指令，就该做成 Skill**。

## Hooks：自动执行的命令

如果说 Skill 是"手动触发的自动化"，那 Hook 就是"完全自动化的自动化"。

Hook 是一段 shell 命令，在 Claude Code 的特定生命周期事件发生时自动执行。比如：每次 Claude Code 要执行一个 Bash 命令之前，先检查这个命令是否安全；每次对话开始时，自动加载一些环境变量；每次文件修改之后，自动跑 lint 检查。

Hooks 在 `settings.json` 中配置，支持 4 个层级：managed policy、user（`~/.claude/settings.json`）、project（`.claude/settings.json`）、local（`.claude/settings.local.json`）。

一个实用的 Hook 配置示例：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "if echo \"$TOOL_INPUT\" | grep -q 'rm -rf\\|DROP TABLE'; then exit 2; fi"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "npx prettier --write \"$FILE_PATH\" 2>/dev/null"
      }
    ]
  }
}
```

这个配置做了两件事：

- `PreToolUse`：在执行 Bash 命令前检查是否包含危险操作（`rm -rf`、`DROP TABLE`），如果有则阻止执行（exit code 2 表示阻止）
- `PostToolUse`：在每次编辑或写入文件后，自动用 Prettier 格式化

### Hook 的事件类型

Claude Code 支持 25+ 种 Hook 事件，覆盖了整个生命周期：

**会话相关**：SessionStart（会话开始）、SessionEnd（会话结束）

**工具执行**：PreToolUse（工具执行前）、PostToolUse（工具执行后）

**权限相关**：PermissionRequest（请求权限）、PermissionDenied（权限被拒绝）

**Agent 相关**：SubagentStart（子 Agent 启动）、SubagentStop（子 Agent 停止）

**任务相关**：TaskCreated（任务创建）、TaskCompleted（任务完成）

**文件变更**：FileChanged（文件被修改）

**上下文相关**：PreCompact（上下文压缩前）、PostCompact（上下文压缩后）

最常用的是 `PreToolUse` 和 `PostToolUse`——它们让你在任何工具执行前后插入自定义逻辑。

### Hook 的条件匹配

不是每个 Hook 都需要在所有场景下触发。你可以用 matcher 来过滤：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo '即将执行 Bash 命令'"
          }
        ]
      },
      {
        "matcher": "Edit",
        "if": "$FILE_PATH matches '*.spec.ts'",
        "hooks": [
          {
            "type": "command",
            "command": "echo '即将修改测试文件'"
          }
        ]
      }
    ]
  }
}
```

matcher 支持精确匹配和 glob 模式，`if` 条件支持更复杂的逻辑判断。这样你可以精确控制 Hook 在什么情况下触发。

### Hook 的返回值

Hook 命令的退出码决定了 Claude Code 的行为：

- `exit 0`：Hook 执行成功，继续执行原操作
- `exit 2`：阻止原操作执行（比如安全检查不通过）
- 其他退出码：Hook 执行失败，但原操作继续

这让 Hook 不仅能做"辅助性"的事情（比如自动格式化），还能做"拦截性"的事情（比如阻止危险操作）。

## 实战：为 TaskFlow 团队创建自动化工作流

现在让我们把 Skills 和 Hooks 组合起来，为 TaskFlow 团队搭建一套自动化工作流。

### 第一步：创建代码审查 Skill

```markdown
---
name: review
description: 按 TaskFlow 团队标准审查代码变更
---

审查当前工作区的代码变更，按 TaskFlow 团队的代码标准检查：

## 必须检查项

1. **类型安全**：不允许 `as any`，TypeORM 查询返回必须有类型
2. **错误处理**：Service 方法必须有 try-catch，抛 RpcException
3. **DTO 校验**：Controller 参数必须有 class-validator 装饰器
4. **N+1 查询**：循环中不能有数据库查询，用 JOIN 或批量查询
5. **敏感信息**：不能硬编码 API Key、密码、Token

## 输出格式

按严重程度分组：

- 🔴 必须修复（会导致 bug 或安全问题）
- 🟡 建议修复（代码质量或性能问题）
- 🟢 可选优化（风格建议）

每个问题包含：文件路径、行号、问题描述、修复建议。
```

保存为 `.claude/skills/review.md`。以后团队成员只需要输入 `/review`，就能执行标准化的代码审查。

### 第二步：配置自动化 Hooks

在项目的 `.claude/settings.json` 中添加：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "npx prettier --write \"$FILE_PATH\" 2>/dev/null || true"
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "if echo \"$TOOL_INPUT\" | grep -qE 'DROP\\|TRUNCATE\\|rm -rf /'; then exit 2; fi"
      }
    ]
  }
}
```

这样团队成员每次修改文件都会自动格式化，每次执行危险命令都会被拦截。

### 第三步：创建提交前检查 Hook

团队的 pre-commit 检查也可以通过 Hook 实现。但要注意，Claude Code 的 Hook 和 Git 的 pre-commit hook 是不同的东西——Claude Code 的 Hook 在 Claude Code 执行操作时触发，Git 的 Hook 在 git 命令执行时触发。

如果你想让 Claude Code 在每次准备提交代码前自动检查，可以这样做：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "if": "$TOOL_INPUT contains 'git commit'",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint && npm run test -- --passWithNoTests"
          }
        ]
      }
    ]
  }
}
```

这样每次 Claude Code 执行 `git commit` 之前，会先跑 lint 和测试。如果 lint 或测试失败，你可以选择阻止提交。

## 不要过度自动化

Skills 和 Hooks 很好用，但容易犯一个错误：把所有事情都自动化了。

有些团队定义了十几个 Skill，配置了二十几个 Hook，结果每次操作都要等半天（因为 Hook 一个接一个执行），团队成员甚至不知道某些 Hook 在后台做了什么。

几个原则：

**只自动化真正重复的事情**。如果你一周才做一次的事，不值得做成 Skill。

**Hook 要快**。PostToolUse 的 Hook 会在每次文件修改后执行，如果它要跑 30 秒的测试，你的编辑体验会很差。Hook 应该是轻量的——格式化、简单检查、日志记录。

**团队成员要知道有哪些 Hook 在运行**。把 Hook 配置提交到版本控制，让每个人都能看到。不要在个人配置里偷偷加 Hook，这会让团队其他人的开发体验不可预测。

**保持可关闭**。如果某个 Hook 或 Skill 妨碍了正常工作，团队成员应该能方便地关闭它。不要把自动化做成铁板一块。

Skills 和 Hooks 的最终目标是**减少重复劳动，而不是增加认知负担**。好的自动化是"不知不觉就发生了"，坏的自动化是"每次都要等半天然后莫名其妙失败了"。
