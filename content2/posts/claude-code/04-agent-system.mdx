从这篇开始，我们的场景变了。你刚加入一个 5 人团队，负责维护 **TaskFlow**——一个类似 Linear 的团队任务管理平台。项目运行了一年多，代码库有 127 个 TypeScript 文件，没有文档，测试覆盖率 22%，技术债不少。

你的第一件事是搞清楚这个项目的整体架构。127 个文件，手动一个个看？那得看到什么时候。Claude Code 的 Agent System 就是为这种场景设计的。

## Agent System 是什么

上一篇文章讲了 Plan Mode——它让 Claude Code 先探索代码再动手。但 Plan Mode 是你在主对话里操作，一次只能做一件事。

Agent System 把这个思路往前推了一步：**你可以派多个 Agent 同时干不同的事**。每个 Agent 有自己独立的上下文窗口、自己的工具集、自己的执行逻辑。它们互不干扰，可以并行运行。

打个比方：Plan Mode 是你让一个助手先调研再汇报。Agent System 是你可以同时派三个助手——一个去调研前端架构，一个去分析后端 API，一个去检查数据库设计——然后汇总他们的报告。

Claude Code 内置了三种 Agent 类型，每种有自己的专长。

## 三种内置 Agent

### Explore Agent：快速搜索和探索

Explore Agent 的设计目标只有一个：**快速理解代码库**。它用轻量级的模型（Claude Haiku），只能读代码和搜索，不能修改任何文件。

它的速度很快——因为你不需要它做复杂的推理，只需要它帮你找到文件、定位函数、梳理结构。轻量模型加上只读权限，让它的响应速度明显快于主对话。

什么时候用 Explore Agent：

- 你刚接手一个项目，想快速了解目录结构和模块划分
- 你想找一个函数的定义，但不知道它在哪个文件里
- 你想了解某个模块的依赖关系

```
用 Explore Agent 分析 TaskFlow 的项目结构，
告诉我有哪些核心模块、模块之间怎么依赖、
入口文件在哪、数据库实体有哪些。
```

Claude Code 会启动一个 Explore Agent，它会在后台扫描目录、读关键文件、搜索 import 语句。几十秒后给你一份结构报告。

### Plan Agent：只读规划

Plan Agent 和 Plan Mode 类似，也是只读的——只能读代码和搜索，不能修改文件。但它的定位不同：Plan Agent 是一个独立的 Agent，有自己的上下文窗口，不会占用主对话的空间。

主要用在需要深入研究但不想影响当前对话的场景。比如你在主对话里正在改一个 bug，突然需要研究另一个模块的架构——你可以派一个 Plan Agent 去研究，你继续改 bug，两件事同时进行。

### General Purpose Agent：通用执行

这是功能最全的 Agent——它有完整的工具访问权限，能读代码、改代码、跑命令、搜索文件。和主对话的区别在于：**它有自己独立的上下文窗口**。

这意味着什么？主对话的上下文窗口是共享的——你让它做了 20 轮对话，上下文快满了，它就开始"忘记"前面的内容。General Purpose Agent 有自己的窗口，不会影响主对话的上下文。

什么时候用 General Purpose Agent：

- 任务复杂，需要多轮对话才能完成，但你不想影响主对话的上下文
- 你想让 Claude Code 在后台执行一个耗时任务，你继续做别的事
- 你需要同时处理多个独立的任务

## 实战：用 Agent 分析 TaskFlow

让我们用一个真实的场景来体验 Agent System 的能力。

你刚加入 TaskFlow 团队，面对 127 个文件的项目，你想快速了解整体架构。你可以同时派多个 Agent 做不同的事：

```
我需要快速了解这个项目。同时帮我做三件事：

1. 用 Explore Agent 分析整体项目结构——
   有哪些模块、模块之间的依赖关系、入口文件在哪

2. 用 Explore Agent 检查数据库层——
   有哪些实体、实体之间的关系、用了什么 ORM

3. 用 Explore Agent 检查测试情况——
   哪些模块有测试、哪些没有、测试覆盖率大概什么水平
```

Claude Code 会同时启动三个 Explore Agent，并行执行。每个 Agent 独立工作，几十秒后陆续返回结果。你不需要等第一个做完再开始第二个——这就是并行的价值。

等三个 Agent 都返回了结果，你对项目就有了一个比较全面的了解：

- 项目有 6 个核心模块：task、user、project、notification、webhook、auth
- 模块之间有循环依赖：task 依赖 notification，notification 又依赖 task（这是一个问题）
- 数据库有 8 个实体，用 TypeORM 管理
- 127 个文件中只有 14 个有对应的测试文件，覆盖率确实很低

这些信息帮你快速建立了对项目的整体认知，不需要手动翻几十个文件。

## 后台执行与并行任务

Agent System 还有一个很实用的能力：后台执行。

有些任务很耗时——比如"检查所有 Controller 的错误处理是否规范"、"找出所有没有测试的 Service 方法"。这类任务你可能要等一两分钟。如果在前台等，这两分钟你什么都干不了。

用后台执行，你可以让 Agent 在后台跑，你继续在主对话里做别的事。Agent 完成后，结果会自动通知你。

```
在后台用 General Purpose Agent 做这件事：
检查所有 Controller 里是否有未处理的异常情况，
特别是没有 try-catch 包裹的 async 方法。
把结果整理成一份清单给我。
```

这个任务在后台跑了，你可以继续说：

```
帮我看看 task.service.ts 里的 create 方法，
参数校验是不是够严格。
```

两件事同时进行，不浪费时间。

## worktree 隔离：安全地试验

有时候你让 Agent 做一件有风险的事——比如"尝试重构 NotificationService，看看能不能拆成 5 个独立的 Service"。你不确定重构方案是否可行，不想污染你当前的工作分支。

Claude Code 的 worktree 隔离可以帮你。Agent 可以在一个独立的 git worktree 中工作——相当于一个临时分支，和你的主分支完全隔离。如果 Agent 的改动有问题，直接丢弃 worktree 就行，不影响你的代码。

这个能力在 General Purpose Agent 上特别有用。当你让 Agent 做实验性的修改时，开启 worktree 隔离是一个好习惯。

## 自定义 Agent

除了三种内置 Agent，你还可以创建自定义 Agent。自定义 Agent 是 Markdown 文件，放在 `.claude/agents/` 目录下（项目级别）或 `~/.claude/agents/` 目录下（用户级别）。

一个自定义 Agent 的定义文件长这样：

```markdown
---
name: code-reviewer
description: 审查代码变更，检查类型安全、错误处理和性能问题
model: sonnet
tools:
  - Read
  - Grep
  - Glob
permissionMode: plan
---

你是一个代码审查专家。审查代码时关注：

- 类型安全：是否有不安全的类型断言
- 错误处理：异常是否被正确处理
- 性能：是否有 N+1 查询、不必要的循环
- 安全：是否有注入风险、敏感信息泄露

输出格式：按严重程度排列问题清单。
```

几个值得注意的配置项：

- `model`：可以指定使用 Haiku（快但浅）、Sonnet（平衡）、Opus（深但慢）
- `tools`：限制 Agent 只能用指定的工具（比如只读工具）
- `permissionMode`：`plan` 表示只读，`default` 表示需要确认，`bypassPermissions` 表示完全自动

自定义 Agent 的价值在于：**把团队的最佳实践固化成可复用的 Agent**。比如团队的代码审查标准、部署检查流程、数据库迁移规范——都可以封装成 Agent，保证每次执行都遵循同样的标准。

## 什么时候不该用 Agent

Agent System 很强大，但不是万能的。几个不适合用 Agent 的场景：

**简单任务**。"帮我读一下这个文件"——直接在主对话里用 @ 引用就行了，启动 Agent 反而多了一层开销。

**需要频繁和你交互的任务**。如果你需要在执行过程中多次给出反馈、调整方向，Agent 的独立上下文反而成了障碍——它看不到主对话的上下文。

**已经进入上下文很深的对话**。如果你在主对话里已经讨论了 20 轮，这时候派 Agent 去做一个任务，Agent 不知道前面 20 轮的上下文。你需要把关键信息重新告诉 Agent。

一个实用的原则：**如果这个任务你自己描述需求不超过 3 句话就能完成，直接在主对话里做。超过 3 句话才能说清楚，或者需要多步探索，考虑用 Agent**。

Agent System 是一个提升效率的工具，不是让你把所有事情都派给 Agent 做。用它来处理那些"需要探索、需要并行、需要隔离"的任务，简单的事情还是在主对话里直接完成。
