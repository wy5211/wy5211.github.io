你写过代码，但有没有想过：数据存在哪里？程序关闭后数据还在吗？两个人同时下单同一个商品，库存会怎么处理？

这些问题的答案都在数据库里。数据库是现代软件系统的基石，但很多开发者对它的理解停留在"会写 CRUD"的程度。为什么有了文件系统还需要数据库？PostgreSQL, MongoDB, Redis 这些到底有什么区别？什么时候该用哪个？

让我们从最基本的问题开始：为什么不直接用文件存数据？

想象一下，你要开发一个电商系统，需要存储用户信息。最简单的想法是用 JSON 文件：

```json
// users.json
[
  { "id": 1, "email": "zhang@example.com", "name": "张三" },
  { "id": 2, "email": "li@example.com", "name": "李四" }
]
```

这看起来很简单，但很快就会遇到问题。当用户注册时，你需要读取整个文件, 解析 JSON, 添加新用户, 再写回文件。如果有两个人同时注册，可能会互相覆盖。当用户数量达到十万级别时，每次读取整个文件会很慢。如果你想查询"所有邮箱以 example.com 结尾的用户"，需要遍历整个数组。

数据库就是为了解决这些问题而诞生的。它提供了文件系统没有的核心能力：持久化存储, 高效查询, 并发控制, 数据完整性保证。

数据库的发展历程很有意思。1960 年代，数据分散在各个文件中，缺乏统一管理。1970 年代出现了层次型和网状型数据库，但结构复杂不易使用。1980 年代关系型数据库兴起，Oracle, MySQL, PostgreSQL 用表格和 SQL 语言改变了游戏规则。1990 年代有人尝试面向对象数据库，但没能成为主流。2000 年代 NoSQL 革命爆发，MongoDB, Redis, Cassandra 应对大数据量和高并发场景。2010 年代出现 NewSQL，试图兼顾 SQL 的易用性和 NoSQL 的扩展性。2020 年代进入云原生数据库时代，Aurora, Spanner, Cloud SQL 提供了 Serverless 和自动扩展能力。

数据库按数据模型可以分为几大类。关系型数据库以表格形式存储，表之间通过外键建立关系。PostgreSQL 是功能最强大的开源数据库，MySQL 是最流行的，SQLite 则是嵌入式零配置的选择。关系型数据库适合需要事务支持, 数据结构稳定, 需要复杂查询, 对数据一致性要求高的场景，比如电商系统, 金融系统, 企业管理系统。

文档型数据库存储 JSON/BSON 格式的文档，不需要预定义结构。MongoDB 是最流行的选择，适合数据结构灵活多变, 需要快速迭代, 内容管理, 日志存储, 嵌套文档结构的场景。

键值型数据库通过键快速访问值，性能极高。Redis 是功能丰富的内存数据库，适合缓存, 会话存储, 排行榜, 计数器, 消息队列等场景。

列族型数据库按列存储，适合大数据量, 写密集场景。Cassandra 高可用无单点故障，HBase 是 Hadoop 生态系统的一部分，适合海量数据存储, 高并发写入, 时序数据, 日志存储。

搜索引擎专门优化全文搜索和复杂查询。Elasticsearch 基于 Lucene，适合全文搜索, 日志分析, 数据分析可视化。

图数据库专门存储图结构，擅长处理复杂关系。Neo4j 是最流行的图数据库，适合社交网络, 知识图谱, 推荐引擎, 欺诈检测。

理解这些分类后，还需要理解关系型数据库的 ACID 特性。ACID 是保证数据可靠性的四大特性：原子性, 一致性, 隔离性, 持久性。

原子性意味着事务中的所有操作要么全部成功，要么全部失败。比如银行转账，从账户 A 扣除 100 元，向账户 B 增加 100 元，这两个操作必须同时成功或同时失败，不会出现钱扣了但没收到的情况。

一致性意味着事务执行前后，数据库都必须处于一致的状态。比如库存管理中可以约束库存不能为负数，如果扣减后库存小于 0，事务会失败。

隔离性意味着多个事务并发执行时，每个事务都感觉不到其他事务的存在。如果不隔离，会出现脏读（读到未提交的数据）, 不可重复读（同一查询两次结果不同）, 幻读（查询范围数据时，其他事务插入了新行）等问题。PostgreSQL 提供了读已提交, 可重复读, 可串行化三种隔离级别。

持久性意味着事务一旦提交，数据就会永久保存，即使系统崩溃也不会丢失。这通过 WAL（Write-Ahead Logging）实现，先写日志再写数据。

对于分布式数据库，ACID 可能无法完全满足，于是有了 CAP 定理。分布式系统只能同时满足一致性, 可用性, 分区容错性中的两个。传统单机数据库如 PostgreSQL, MySQL 选择 CA，强调可用性的 Cassandra, DynamoDB 选择 AP，强调一致性的 MongoDB, HBase 选择 CP。

BASE 理论是 ACID 的反向思维，放弃强一致性，追求最终一致性。Basically Available 表示基本可用允许部分失败，Soft state 表示状态可以随时间变化，Eventually consistent 表示最终一致短时间内不一致可接受。比如社交媒体的点赞数，用户点赞后数字可能不会立即更新，但几秒钟后所有用户看到的数字会一致。

那么如何为项目选择合适的数据库？如果需求不明确，PostgreSQL 是最安全的选择，它足够强大能应对大多数场景。对于简单项目或学习，SQLite 零配置单文件很方便。缓存层可以选择 Redis，性能极高且数据结构丰富。内容管理或日志可以选择 MongoDB，schema 灵活且支持水平扩展。全文搜索可以选择 Elasticsearch，强大的搜索能力且支持中文分词。

现代应用通常组合多种数据库各司其职。电商系统可能用 PostgreSQL 存储用户订单商品，Redis 做会话和热点数据缓存，Elasticsearch 做商品搜索，MongoDB 存储商品详情和用户评论，Redis 做消息队列处理异步任务。核心原则是事务性数据用 PostgreSQL，热点数据用 Redis，灵活文档用 MongoDB，全文搜索用 Elasticsearch。

在选择数据库时，优先选择关系型数据库，不要过早优化。小规模应用不需要复杂的分库分表，做好索引优化足够了。要理解 CAP 权衡，金融系统一致性优先，社交应用可用性优先，电商系统最终一致性可接受。缓存不能替代数据库，缓存是补充不是替代，数据库才是数据的唯一真实来源。选择社区活跃的数据库，检查最后更新时间, 查看 GitHub Stars, 阅读问题讨论。

了解了数据库的基本概念和分类，下一步我们将学习 SQL 语言基础，掌握数据库查询的核心技能，了解如何与数据库交互，如何创建表, 查询数据, 建立关系。
