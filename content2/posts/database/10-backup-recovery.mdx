上一篇文章我们学习了执行计划分析，可以诊断性能问题。但性能问题顶多是查询慢，如果数据丢了，那就真完了。硬件故障、人为误删、程序 bug、机房着火……数据丢失的风险无处不在。备份就是你的保险。

## 为什么需要备份

你可能觉得数据库有主从复制，应该安全了吧？但主从复制不是备份，它是同步数据。如果你在主库误删了一张表，从库也会跟着删。备份是某个时间点的数据快照，能让你恢复到那个时刻。

真实案例：某电商公司的运维误操作，DROP DATABASE，主库和从库都没了。幸好有每天凌晨的备份，只丢了当天的数据。从那以后他们改成每小时备份一次。

## 逻辑备份：pg_dump

pg_dump 是 PostgreSQL 最常用的备份工具，导出成 SQL 文件或自定义格式。

### 备份单个数据库

```bash
# 导出成 SQL 文本（可读，但恢复慢）
pg_dump -h localhost -U postgres -d myshop -f myshop_20260412.sql

# 导出成自定义格式（压缩，恢复快，支持选择性恢复）
pg_dump -h localhost -U postgres -d myshop -F c -f myshop_20260412.dump

# 并行备份（多核 CPU，更快）
pg_dump -h localhost -U postgres -d myshop -j 4 -F d -f myshop_backup_dir
```

-F 指定格式：

- p（plain）：纯文本 SQL，默认值
- c（custom）：自定义格式，压缩，支持并行恢复
- d（directory）：目录格式，每个表一个文件
- t（tar）：tar 格式

### 只备份部分数据

```bash
# 只备份 schema（不包含数据）
pg_dump -h localhost -U postgres -d myshop --schema-only -f schema.sql

# 只备份特定表
pg_dump -h localhost -U postgres -d myshop -t users -t orders -f tables.sql

# 只备份特定表的数据（INSERT 语句）
pg_dump -h localhost -U postgres -d myshop -t users --data-only -f users_data.sql
```

### 恢复逻辑备份

```bash
# 恢复 SQL 文本
psql -h localhost -U postgres -d myshop -f myshop_20260412.sql

# 恢复自定义格式（pg_restore）
pg_restore -h localhost -U postgres -d myshop -c myshop_20260412.dump

# -c 参数先清空数据库（DROP），慎用
# 只恢复特定表
pg_restore -h localhost -U postgres -d myshop -t users myshop_20260412.dump
```

### pg_dump 的优缺点

优点：

- 跨版本兼容（PG 12 的备份可以恢复到 PG 16）
- 可选择恢复部分表
- 文件小（压缩后）
- 可以在线备份（不锁表）

缺点：

- 恢复慢（要重新执行所有 SQL）
- 大数据量备份时间长

## 物理备份：pg_basebackup

pg_basebackup 直接复制数据库的数据文件，适合大数据量。

```bash
# 基础备份
pg_basebackup -h localhost -U postgres -D /backup/basebackup -Ft -z -P

# -D 指定备份目录
# -Ft 打包成 tar
# -z 压缩
# -P 显示进度
```

物理备份恢复时，需要停止数据库，复制文件回来，然后重启。适合灾难恢复，不适合日常备份。

## 时间点恢复（PITR）

逻辑备份和物理备份都只能恢复到备份的那个时间点。但如果你每天凌晨备份一次，晚上 8 点误删了数据，就丢了 20 小时的数据。PITR（Point-In-Time Recovery）可以让你恢复到任意时间点。

PITR 的原理是 WAL（Write-Ahead Log）。PostgreSQL 的每次修改都先写 WAL 日志，再改数据文件。WAL 日志就像数据库的 redo log，可以重放所有操作，恢复到任意时间点。

### 配置 WAL 归档

编辑 postgresql.conf：

```bash
# 启用 WAL 归档
wal_level = replica
archive_mode = on
archive_command = 'cp %p /backup/wal/%f'
```

archive_command 把 WAL 文件复制到备份目录。%p 是 WAL 文件路径，%f 是文件名。

### 备份流程

```bash
# 1. 基础备份（pg_basebackup）
pg_basebackup -h localhost -U postgres -D /backup/basebackup -Ft -z -P

# 2. WAL 归档（自动）
# PostgreSQL 会自动把 WAL 文件复制到 /backup/wal/
```

### 恢复到指定时间点

假设你要恢复到 2026-04-12 18:30:00：

```bash
# 1. 停止数据库
pg_ctl stop

# 2. 清空数据目录
rm -rf /var/lib/postgresql/data/*

# 3. 恢复基础备份
tar -xzf /backup/basebackup/base.tar.gz -C /var/lib/postgresql/data/

# 4. 创建 recovery.signal 文件（告诉 PostgreSQL 要恢复）
touch /var/lib/postgresql/data/recovery.signal

# 5. 配置恢复目标
echo "restore_command = 'cp /backup/wal/%f %p'" >> /var/lib/postgresql/data/postgresql.conf
echo "recovery_target_time = '2026-04-12 18:30:00'" >> /var/lib/postgresql/data/postgresql.conf

# 6. 启动数据库
pg_ctl start

# PostgreSQL 会重放 WAL 日志，直到 18:30:00
```

## 自动化备份策略

手动备份容易忘，自动化才是王道。

### 每日备份脚本

```bash
#!/bin/bash
# backup.sh

BACKUP_DIR="/backup/postgresql"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=7

# 创建备份目录
mkdir -p "$BACKUP_DIR/daily"
mkdir -p "$BACKUP_DIR/wal"

# 逻辑备份（保留 7 天）
pg_dump -h postgres -U postgres -d myshop -F c -f "$BACKUP_DIR/daily/myshop_$DATE.dump"

# 压缩
gzip "$BACKUP_DIR/daily/myshop_$DATE.dump"

# 清理旧备份
find "$BACKUP_DIR/daily" -type f -mtime +$RETENTION_DAYS -delete

# 验证备份（不解压，只检查文件完整性）
pg_restore -l "$BACKUP_DIR/daily/myshop_$DATE.dump.gz" > /dev/null
if [ $? -eq 0 ]; then
  echo "Backup successful: myshop_$DATE.dump.gz"
else
  echo "Backup failed: myshop_$DATE.dump.gz"
  exit 1
fi
```

### 定时任务

```bash
# 每天凌晨 2 点备份
0 2 * * * /scripts/backup.sh >> /var/log/postgresql_backup.log 2>&1

# 每小时增量备份（只备份 WAL）
0 * * * * rsync -av /var/lib/postgresql/pg_wal/ /backup/wal/
```

## 异地备份和云存储

本地备份不够，如果机房出问题（火灾、地震），备份也没了。要异地备份。

### 备份到云存储

```bash
# 备份到 AWS S3
aws s3 cp /backup/daily/myshop_20260412.dump.gz s3://myshop-backups/postgresql/

# 备份到阿里云 OSS
ossutil cp /backup/daily/myshop_20260412.dump.gz oss://myshop-backups/postgresql/

# 自动脚本
#!/bin/bash
DATE=$(date +%Y%m%d)
pg_dump -h postgres -U postgres -d myshop -F c - | gzip | \
  aws s3 cp - s3://myshop-backups/postgresql/myshop_$DATE.dump.gz
```

### 3-2-1 备份原则

行业标准的 3-2-1 原则：

- 3 份副本（1 份生产 + 2 份备份）
- 2 种介质（磁盘 + 云存储）
- 1 份异地（云存储就是异地）

## 恢复演练

备份不验证，等于没备份。定期演练恢复流程，确保备份真的可用。

```bash
#!/bin/bash
# restore_test.sh

BACKUP_FILE=$1
TEST_DB="myshop_test"

echo "Testing backup: $BACKUP_FILE"

# 创建测试数据库
psql -h localhost -U postgres -c "DROP DATABASE IF EXISTS $TEST_DB"
psql -h localhost -U postgres -c "CREATE DATABASE $TEST_DB"

# 恢复备份
pg_restore -h localhost -U postgres -d $TEST_DB "$BACKUP_FILE"

# 验证数据
RESULT=$(psql -h localhost -U postgres -d $TEST_DB -t -c "SELECT COUNT(*) FROM orders")
if [ "$RESULT" -gt 0 ]; then
  echo "Restore test passed: $RESULT orders found"
else
  echo "Restore test failed: no orders found"
  exit 1
fi
```

## 监控备份

确保备份真的在运行，没报错。

```sql
-- 检查最近的 WAL 归档
SELECT
  pg_ls_dir('/backup/wal/') as wal_files,
  pg_stat_file('/backup/wal/' || pg_ls_dir('/backup/wal/')).modification as mod_time
ORDER BY mod_time DESC
LIMIT 10;

-- 检查备份大小
SELECT
  pg_size_pretty(pg_total_relation_size('orders')) as table_size,
  pg_size_pretty(pg_database_size('myshop')) as db_size;
```

## 常见问题

### 备份文件太大

逻辑备份压缩后一般是原数据库的 10-20%。如果还是太大，考虑：

- 只备份重要表
- 增加备份频率，减少保留天数
- 用 pg_basebackup（物理备份）

### 恢复时主键冲突

如果你用 -c 参数恢复（先清空数据库），但序列没重置，会导致主键冲突。

```sql
-- 恢复后重置序列
SELECT setval('orders_id_seq', (SELECT MAX(id) FROM orders));
```

### 恢复很慢

逻辑备份恢复要重新执行所有 SQL，慢是正常的。加速方法：

- 用自定义格式（-F c）而不是纯文本
- 用 pg_restore -j 4 并行恢复
- 用物理备份（pg_basebackup）

## 小结

备份和恢复是数据安全的最后一道防线，但经常被忽视。等你需要的时候已经晚了。

核心要点：

- 逻辑备份（pg_dump）适合小数据库，灵活但恢复慢
- 物理备份（pg_basebackup）适合大数据库，快但不灵活
- PITR 可以恢复到任意时间点，减少数据丢失
- 3-2-1 原则：3 份副本、2 种介质、1 份异地
- 自动化备份，定期演练恢复

Trade-off：备份占用存储空间，恢复有停机时间。但比起数据丢失，这些代价都是值得的。另外，备份不是一次性的，要持续监控和验证。
