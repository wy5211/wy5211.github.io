上一篇文章我们学习了索引原理，知道合适的索引能让查询速度提升 100 倍。但实际工作中，经常遇到这种情况：明明建了索引，查询还是很慢。这通常是因为查询写法有问题，导致索引无法发挥作用。

比如用函数包裹索引列、SELECT \* 查询所有字段、深分页查询、N+1 查询问题，这些都会让索引失效或效率低下。查询优化不需要修改数据库结构，只需要改写 SQL，就能让速度提升 10 倍甚至更多。

## 找出慢查询

优化的第一步是找出哪些查询需要优化。PostgreSQL 提供了几种方式：

第一种是开启慢查询日志，记录执行时间超过阈值的查询：

```sql
-- 记录执行时间超过 1 秒的查询
ALTER DATABASE your_database SET log_min_duration_statement = 1000;

-- 记录所有查询（开发环境可以用，生产环境要慎重）
ALTER DATABASE your_database SET log_min_duration_statement = 0;

-- 查看日志文件位置
SHOW log_directory;
SHOW log_filename;
```

第二种是使用 pg_stat_statements 扩展，统计所有查询的执行情况：

```sql
-- 启用 pg_stat_statements 扩展
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 查看最慢的查询
SELECT
  calls,
  total_exec_time / 1000 as total_time_seconds,
  mean_exec_time / 1000 as avg_time_seconds,
  query
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

-- 查看执行次数最多的查询
SELECT
  calls,
  total_exec_time / 1000 as total_time_seconds,
  query
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 10;
```

第三种是用 EXPLAIN ANALYZE 分析单个查询：

```sql
-- 分析查询执行计划
EXPLAIN ANALYZE
SELECT o.order_number, o.total, u.email
FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE u.email = 'zhang@example.com';

-- 输出示例：
-- Hash Join  (cost=8.44..21.33 rows=1 width=200) (actual time=0.123..1.456 rows=100 loops=1)
--   Hash Cond: (o.user_id = u.id)
--   ->  Seq Scan on orders o  (cost=0.00..15.00 rows=1000 width=200) (actual time=0.012..0.234 rows=1000 loops=1)
--   ->  Hash  (cost=8.43..8.43 rows=1 width=100) (actual time=0.045..0.045 rows=1 loops=1)
--         ->  Index Scan using idx_users_email on users u  (cost=0.42..8.43 rows=1 width=100) (actual time=0.032..0.041 rows=1 loops=1)
```

EXPLAIN ANALYZE 的输出很丰富，关键信息包括：actual time（实际执行时间）、rows（实际返回行数）、Index Scan（使用索引）、Seq Scan（全表扫描）。如果看到 Seq Scan 在大表上，说明需要优化或加索引。

## 避免 SELECT \*

SELECT \* 是很多开发者的习惯，写起来方便，但性能很差。它的缺点包括：增加网络传输、增加内存占用、可能无法使用覆盖索引、增加解析成本。

```sql
-- 查询订单列表（不推荐）
SELECT * FROM orders WHERE user_id = 'usr_8x9k2m' LIMIT 20;

-- 只查询需要的字段（推荐）
SELECT
  order_number,
  total,
  status,
  created_at
FROM orders
WHERE user_id = 'usr_8x9m2k'
LIMIT 20;
```

如果 orders 表有 20 个字段，SELECT \* 会返回所有字段，但列表页只需要显示 4-5 个字段。这不仅浪费带宽，还可能加载了很大的字段（比如 JSONB、TEXT），影响性能。

更好的做法是明确指定需要的字段：

```sql
-- 订单列表页只需要这几个字段
SELECT
  id,
  order_number,
  total,
  status,
  created_at
FROM orders
WHERE user_id = 'usr_8x9k2m'
ORDER BY created_at DESC
LIMIT 20;
```

## 避免在 WHERE 中使用函数

如果在 WHERE 条件中对索引列使用函数，索引就会失效。这是最常见的性能问题之一。

```sql
-- 假设有索引：CREATE INDEX idx_users_email ON users(email);

-- 索引失效（函数包裹）
SELECT * FROM users WHERE LOWER(email) = 'ZHANG@EXAMPLE.COM';

-- 索引生效（直接比较）
SELECT * FROM users WHERE email = 'zhang@example.com';

-- 或者在应用层统一转小写
const email = 'ZHANG@EXAMPLE.COM'.toLowerCase();
// SELECT * FROM users WHERE email = ${email}
```

另一个常见问题是日期函数：

```sql
-- 索引失效（函数包裹）
SELECT * FROM orders WHERE DATE(created_at) = '2026-04-12';

-- 索引生效（范围查询）
SELECT * FROM orders
WHERE created_at >= '2026-04-12 00:00:00'
  AND created_at < '2026-04-13 00:00:00';
```

如果必须用函数，考虑创建表达式索引：

```sql
-- 创建表达式索引
CREATE INDEX idx_users_lower_email ON users(LOWER(email));

-- 现在这个查询可以使用索引了
SELECT * FROM users WHERE LOWER(email) = 'ZHANG@EXAMPLE.COM';
```

## 使用 EXISTS 替代 IN

IN 和 EXISTS 都可以用于子查询，但性能差异很大。EXISTS 只要找到一条匹配记录就停止，IN 会扫描所有结果。

```sql
-- 使用 IN（可能较慢）
SELECT * FROM orders o
WHERE o.user_id IN (
  SELECT u.id FROM users u WHERE u.is_verified = TRUE
);

-- 使用 EXISTS（通常更快）
SELECT * FROM orders o
WHERE EXISTS (
  SELECT 1 FROM users u
  WHERE u.id = o.user_id AND u.is_verified = TRUE
);
```

这个差异在大表上特别明显。如果子查询返回很多记录，IN 需要构建一个很大的临时表，而 EXISTS 只要找到一条就返回。

另一个优化是用 JOIN 替代 IN：

```sql
-- 使用 IN
SELECT * FROM orders
WHERE user_id IN (SELECT id FROM users WHERE is_verified = TRUE);

-- 使用 JOIN（可能更快）
SELECT DISTINCT o.*
FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE u.is_verified = TRUE;
```

但要注意，如果子查询只返回少量记录，IN 可能更快。所以最好用 EXPLAIN ANALYZE 对比两种方案。

## 使用 UNION ALL 替代 UNION

UNION 会合并结果并去重排序，UNION ALL 只是合并结果不去重。如果确定结果没有重复，用 UNION ALL 会快很多。

```sql
-- 使用 UNION（去重排序，较慢）
SELECT email FROM users_a
UNION
SELECT email FROM users_b;

-- 使用 UNION ALL（不去重，很快）
SELECT email FROM users_a
UNION ALL
SELECT email FROM users_b;
```

UNION 的性能问题在大结果集上特别明显。如果两个表各 100 万行，UNION 需要对 200 万行去重排序，而 UNION ALL 直接返回 200 万行。

如果需要去重，但确定某个字段是唯一的，可以这样优化：

```sql
-- 优化 UNION
SELECT email FROM users_a
UNION
SELECT email FROM users_b;

-- 改写为（利用唯一性）
SELECT email FROM (
  SELECT email, 'a' as source FROM users_a
  UNION ALL
  SELECT email, 'b' as source FROM users_b
) t
GROUP BY email;
```

## 优化分页查询

分页查询是最常见的操作之一，但深分页（OFFSET 很大）性能很差。因为数据库要扫描并丢弃前面的所有记录。

```sql
-- 深分页（很慢）
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20 OFFSET 10000;

-- 问题：数据库要读取 10020 条记录，丢弃前 10000 条，只返回后 20 条
```

优化方案之一是使用游标分页（基于上一页的最后一条记录）：

```sql
-- 游标分页（很快）
SELECT * FROM orders
WHERE created_at < '2026-04-11 10:30:00'  -- 上一页最后一条记录的时间
ORDER BY created_at DESC
LIMIT 20;
```

这种方案不需要 OFFSET，直接定位到上一页结束的位置，性能很好。但要求 created_at 是唯一的（或者用组合唯一条件）。

另一个方案是延迟关联，先通过索引找到需要的 ID，再 JOIN 获取完整数据：

```sql
-- 延迟关联（优化深分页）
SELECT o.*
FROM orders o
INNER JOIN (
  SELECT id FROM orders
  ORDER BY created_at DESC
  LIMIT 20 OFFSET 10000
) t ON o.id = t.id;
```

这个查询先通过索引快速找到需要的 20 个 ID，再 JOIN 获取完整数据。因为索引扫描比全表扫描快很多，所以性能更好。

## 批量操作

批量插入、更新比逐条操作快很多，因为减少了网络往返和事务开销。

```sql
-- 逐条插入（很慢）
INSERT INTO order_items (order_id, product_id, quantity, unit_price, subtotal)
VALUES ('ord_a1b2c3d4', 'prod_111', 2, 99.00, 198.00);

INSERT INTO order_items (order_id, product_id, quantity, unit_price, subtotal)
VALUES ('ord_a1b2c3d4', 'prod_222', 1, 299.00, 299.00);

-- 批量插入（很快）
INSERT INTO order_items (order_id, product_id, quantity, unit_price, subtotal)
VALUES
  ('ord_a1b2c3d4', 'prod_111', 2, 99.00, 198.00),
  ('ord_a1b2c3d4', 'prod_222', 1, 299.00, 299.00);
```

对于大批量数据，使用 COPY 命令比 INSERT 快 10-50 倍：

```sql
-- 使用 COPY 批量导入（非常快）
COPY orders(user_id, order_number, total, status)
FROM '/path/to/orders.csv'
DELIMITER ','
CSV HEADER;
```

## 优化 JOIN

JOIN 优化有几个要点：小表驱动大表、只查询需要的字段、为外键创建索引、避免过深的 JOIN。

```sql
-- 小表驱动大表（推荐）
SELECT u.email, o.order_number
FROM users u              -- 小表
INNER JOIN orders o ON u.id = o.user_id;

-- 大表驱动小表（不推荐）
SELECT u.email, o.order_number
FROM orders o             -- 大表
INNER JOIN users u ON o.user_id = u.id;
```

这个差异在数据量大时很明显。小表驱动大表意味着：先扫描小表，再根据小表的记录去大表中查找。大表驱动小表意味着：先扫描大表，再根据大表的记录去小表中查找。显然前者效率更高。

JOIN 的深度也要控制。超过 5 个表的 JOIN 通常要重新设计数据模型，或者考虑冗余字段。

```sql
-- 过深的 JOIN（不推荐）
SELECT * FROM table1
INNER JOIN table2 ON table1.id = table2.t1_id
INNER JOIN table3 ON table2.id = table3.t2_id
INNER JOIN table4 ON table3.id = table4.t3_id
INNER JOIN table5 ON table4.id = table5.t4_id
INNER JOIN table6 ON table5.id = table6.t5_id;

-- 分步查询（推荐）
WITH step1 AS (
  SELECT * FROM table1 INNER JOIN table2 ON table1.id = table2.t1_id
),
step2 AS (
  SELECT * FROM step1 INNER JOIN table3 ON step1.t2_id = table3.id
)
SELECT * FROM step2;
```

## 使用 CTE 优化复杂查询

CTE（Common Table Expression，公共表表达式）能让复杂查询更清晰，有时也能提升性能。

```sql
-- 不使用 CTE（重复子查询）
SELECT
  u.id,
  u.email,
  (SELECT COUNT(*) FROM orders WHERE user_id = u.id) as order_count,
  (SELECT SUM(total) FROM orders WHERE user_id = u.id) as total_spent
FROM users u;

-- 使用 CTE（只计算一次）
WITH user_stats AS (
  SELECT
    user_id,
    COUNT(*) as order_count,
    SUM(total) as total_spent
  FROM orders
  GROUP BY user_id
)
SELECT
  u.id,
  u.email,
  COALESCE(us.order_count, 0) as order_count,
  COALESCE(us.total_spent, 0) as total_spent
FROM users u
LEFT JOIN user_stats us ON u.id = us.user_id;
```

这个优化不仅提升了性能（避免重复计算），还让查询更清晰。

## 避免 N+1 查询

N+1 查询是指：先查询 1 次获取 N 条记录，再对每条记录查询 1 次，总共执行 N+1 次查询。这是 ORM 框架常见的性能问题。

```sql
-- 假设用 ORM 查询用户的订单（N+1 问题）
const users = await User.findAll();  // 查询 1 次
for (const user of users) {
  const orders = await Order.findAll({ where: { userId: user.id } });  // 查询 N 次
  user.orders = orders;
}

// 如果有 100 个用户，就会执行 101 次查询
```

优化方案是使用 JOIN 或批量查询：

```sql
-- 使用 JOIN（推荐）
SELECT
  u.id,
  u.email,
  o.order_number,
  o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- 或使用批量查询
const userIds = users.map(u => u.id);
const orders = await Order.findAll({ where: { userId: { [Op.in]: userIds } } });  // 只查询 1 次
```

## 定期维护数据库

数据库使用一段时间后，性能可能下降，需要定期维护：

```sql
-- 更新统计信息（帮助查询优化器做正确决策）
ANALYZE users;
ANALYZE orders;

-- 清理死元组（回收空间）
VACUUM users;
VACUUM orders;

-- 清理并分析（组合操作）
VACUUM ANALYZE users;
VACUUM ANALYZE orders;

-- 自动清理（PostgreSQL 的 autovacuum 会自动运行）
-- 但可以调整参数
ALTER TABLE orders SET (autovacuum_vacuum_scale_factor = 0.1);
ALTER TABLE orders SET (autovacuum_analyze_scale_factor = 0.05);
```

死元组是指被标记为删除但未真正清理的记录。PostgreSQL 的 MVCC 机制导致每次 UPDATE 或 DELETE 都会产生死元组，占用空间并影响性能。VACUUM 操作会清理这些死元组。

## 优化检查清单

遇到慢查询时，可以按这个清单排查：

1. 用 EXPLAIN ANALYZE 分析执行计划，看看是否用了索引，哪一步最慢
2. 检查是否有 SELECT \*，改为只查询需要的字段
3. 检查 WHERE 条件是否有函数包裹，改写查询或创建表达式索引
4. 检查是否有 IN (SELECT ...)，改写为 EXISTS 或 JOIN
5. 检查是否有 UNION，改为 UNION ALL
6. 检查是否有深分页，改用游标分页或延迟关联
7. 检查是否逐条插入/更新，改为批量操作
8. 检查是否有 N+1 查询，改为 JOIN 或批量查询
9. 检查是否有不必要的 JOIN，考虑冗余字段
10. 更新表的统计信息（ANALYZE），清理死元组（VACUUM）

## 优化的边界

查询优化不是万能的，有些问题不是优化查询能解决的：

数据量太大：单表超过 1 亿行，应该考虑分库分表，而不是优化查询

硬件瓶颈：CPU、内存、磁盘 I/O 到了极限，应该升级硬件，而不是优化查询

网络瓶颈：大量数据传输，应该在应用层做缓存，而不是优化查询

业务逻辑复杂：某些查询本身就很复杂，应该在应用层做计算，而不是全推给数据库

优化应该有针对性：先找出真正的瓶颈，再对症下药。过早优化是万恶之源，但不优化就是坐以待毙。监控、分析、优化、验证，这是一个持续的过程。

这篇文章我们学习了 SQL 查询优化的各种技巧，包括避免 SELECT \*、避免函数包裹、使用 EXISTS 替代 IN、优化分页、批量操作等。但查询优化只是"术"，要真正理解为什么这样优化有效，还需要深入执行计划。下一篇我们会学习执行计划分析，理解 PostgreSQL 如何执行查询，以及如何根据执行计划做更精准的优化。
