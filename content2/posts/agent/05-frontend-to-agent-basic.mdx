# 前端工程师的 Agent 进阶之路（上）：入门阶段

在上一篇文章中，我们了解了 Agent 开发的技术栈：LLM、工具调用、向量数据库和框架。现在让我们进入实战阶段，看看前端开发者如何利用已有的 JavaScript/TypeScript 技能，快速上手 Agent 开发。

你会发现，前端开发者转型做 Agent 其实非常自然，因为很多能力是可以直接迁移的。

## 从前端到 Agent：你的技能可以复用

先说说为什么前端工程师适合做 Agent 开发。这不仅仅是换个方向，而是很多核心能力是通用的。

你之前调用 REST API 获取数据，现在调用 LLM API 获取智能回复。你之前处理异步操作用 Promise/async-await，Agent 开发同样需要处理异步的 LLM 调用。你之前用状态管理管理组件状态，现在用类似的方式管理 Agent 的记忆。你之前的组件化思维，可以直接应用到 Agent 的模块化设计中。

换个角度看，你之前的用户体验设计经验，在 Agent 交互设计中同样重要。一个好的 Agent 不仅需要智能，更需要让用户用起来舒服、自然。

## 第一步：理解 LLM API 调用

让我们从最基础的开始：调用 LLM API。这和调用任何其他 API 没有本质区别，但有一些特定的模式需要掌握。

先看一个最简单的例子：

```typescript
// 安装 OpenAI SDK
// npm install openai

import OpenAI from "openai";

const openai = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
});

async function chat(message: string) {
  const response = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [{ role: "user", content: message }],
  });

  return response.choices[0].message.content;
}

// 使用
const answer = await chat("什么是 Agent？");
console.log(answer);
```

这段代码应该很熟悉。创建一个客户端，调用接口，获取响应。但有几个关键点需要理解：

首先是 `model` 参数。不同的模型有不同的能力和成本，`gpt-4o-mini` 是一个性价比很高的选择，适合开发测试。生产环境可以根据需求选择更强的模型。

其次是 `messages` 数组。这不是简单的 prompt，而是对话历史数组。每个消息有 `role`（可以是 `system`、`user`、`assistant`）和 `content`。这种设计让多轮对话变得简单。

## 让 AI 扮演特定角色

在 Agent 开发中，我们经常需要 AI 扮演特定角色。这通过 `system` 消息实现：

```typescript
async function createExpertAI(role: string, question: string) {
  const response = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: [
      {
        role: "system",
        content: `你是一个${role}。请用专业但易懂的方式回答问题。`,
      },
      {
        role: "user",
        content: question,
      },
    ],
  });

  return response.choices[0].message.content;
}

// 使用
const answer = await createExpertAI(
  "资深前端工程师",
  "React Server Components 有什么优势？"
);
```

这里的 `system` 消息定义了 AI 的角色和行为准则。它就像是给 AI 一个"人设"，让后续的回答更符合预期。

在 Agent 开发中，系统提示词的设计至关重要。好的系统提示词能让 Agent 更稳定、更可控。

## 实现多轮对话

现实中的 Agent 需要记住对话上下文。这需要我们维护对话历史：

```typescript
class ChatSession {
  private messages: Array<{ role: string; content: string }> = [];

  constructor(private systemPrompt: string) {
    // 初始化时添加系统提示
    this.messages.push({
      role: "system",
      content: systemPrompt,
    });
  }

  async chat(userMessage: string): Promise<string> {
    // 添加用户消息到历史
    this.messages.push({
      role: "user",
      content: userMessage,
    });

    // 调用 API，带上完整历史
    const response = await openai.chat.completions.create({
      model: "gpt-4o-mini",
      messages: this.messages,
    });

    const assistantMessage = response.choices[0].message.content;

    // 添加助手回复到历史
    this.messages.push({
      role: "assistant",
      content: assistantMessage,
    });

    return assistantMessage;
  }

  getHistory() {
    return this.messages;
  }
}

// 使用
const session = new ChatSession("你是一个代码助手");

const q1 = await session.chat("什么是 TypeScript？");
console.log(q1);

const q2 = await session.chat("它和 JavaScript 有什么区别？");
console.log(q2);
// 这里的回答会基于第一轮的上下文
```

这个简单的类实现了对话记忆。每次对话都把历史传给 API，这样 AI 就能理解上下文关联。

但要注意，对话越长，token 消耗越大。实际项目中需要做历史截断或摘要，这个后面会讲。

## 使用 Vercel AI SDK：前端开发者的福音

虽然可以直接用 OpenAI SDK，但 Vercel AI SDK 为前端开发者做了很多优化，特别是与 React/Next.js 的集成。

先看基础用法：

```typescript
// 安装
// npm install ai @ai-sdk/openai

import { openai } from "@ai-sdk/openai";
import { generateText } from "ai";

async function generateResponse(prompt: string) {
  const { text } = await generateText({
    model: openai("gpt-4o-mini"),
    prompt: prompt,
  });

  return text;
}

// 使用
const response = await generateResponse("解释什么是闭包");
```

Vercel AI SDK 的优势在于统一的接口。你可以轻松切换不同的模型提供商，而不需要改变太多代码。

更重要的是，它在 React 中的使用体验非常好。看一个完整的前端集成例子：

```typescript
// app/api/chat/route.ts
import { openai } from "@ai-sdk/openai";
import { streamText } from "ai";

export const runtime = "edge";

export async function POST(req: Request) {
  const { messages } = await req.json();

  const result = streamText({
    model: openai("gpt-4o-mini"),
    messages,
  });

  return result.toDataStreamResponse();
}
```

```tsx
// app/chat/page.tsx
"use client";

import { useChat } from "ai/react";

export default function ChatPage() {
  const { messages, input, handleInputChange, handleSubmit } = useChat({
    api: "/api/chat",
  });

  return (
    <div className="flex flex-col h-screen">
      <div className="flex-1 overflow-y-auto p-4">
        {messages.map((message) => (
          <div
            key={message.id}
            className={`p-3 rounded-lg mb-2 ${
              message.role === "user"
                ? "bg-blue-500 text-white ml-auto max-w-[70%]"
                : "bg-gray-100 mr-auto max-w-[70%]"
            }`}
          >
            {message.content}
          </div>
        ))}
      </div>

      <form onSubmit={handleSubmit} className="p-4 border-t">
        <input
          value={input}
          onChange={handleInputChange}
          placeholder="输入消息..."
          className="w-full p-2 border rounded-lg"
        />
      </form>
    </div>
  );
}
```

这个 `useChat` hook 封装了所有复杂的状态管理：消息列表、加载状态、流式响应处理等。你只需要关注业务逻辑。

流式响应是 Vercel AI SDK 的另一个优势。用户不需要等全部生成完才能看到内容，而是像真人打字一样逐字显示，体验更好。

## 工具调用：让 Agent 能做事情

前面说的都是对话，但 Agent 的核心能力是能"做事情"。这通过工具调用实现。

工具调用本质上是让 LLM 能够在需要时调用外部函数。看个例子：

```typescript
import { openai } from "@ai-sdk/openai";
import { generateText, tool } from "ai";
import { z } from "zod";

async function agentWithTools(userMessage: string) {
  const { text } = await generateText({
    model: openai("gpt-4o-mini"),
    tools: {
      // 定义一个工具：获取 GitHub 仓库信息
      getGitHubRepo: tool({
        description: "获取 GitHub 仓库的基本信息",
        parameters: z.object({
          owner: z.string().describe("仓库所有者"),
          repo: z.string().describe("仓库名称"),
        }),
        execute: async ({ owner, repo }) => {
          // 调用 GitHub API
          const response = await fetch(
            `https://api.github.com/repos/${owner}/${repo}`
          );
          const data = await response.json();

          return {
            name: data.full_name,
            stars: data.stargazers_count,
            description: data.description,
            language: data.language,
          };
        },
      }),
    },
    toolChoice: "auto",
    prompt: userMessage,
  });

  return text;
}

// 使用
const result = await agentWithTools("帮我查看 facebook/react 这个仓库的信息");
console.log(result);
```

这段代码定义了一个工具，让 Agent 能够调用 GitHub API。当用户问关于仓库的问题时，Agent 会自动判断需要调用这个工具，并提取正确的参数。

工具的关键在于 `description`。LLM 通过描述来理解工具的作用，决定何时调用。所以描述要清晰准确，说明工具做什么、什么情况下使用。

## 实战案例：代码仓库分析 Agent（入门版）

让我们用学到的知识，构建一个简单的代码仓库分析 Agent。这是我们的主线案例，会逐步完善。

先从最简单的版本开始：能够分析一个 GitHub 仓库的基本信息。

```typescript
// app/api/analyze/route.ts
import { openai } from "@ai-sdk/openai";
import { generateText, tool } from "ai";
import { z } from "zod";

export async function POST(req: Request) {
  const { repoUrl } = await req.json();

  // 解析仓库 URL
  const match = repoUrl.match(/github\.com\/([^/]+)\/([^/]+)/);
  if (!match) {
    return Response.json({ error: "无效的 GitHub URL" }, { status: 400 });
  }

  const [, owner, repo] = match;

  const { text } = await generateText({
    model: openai("gpt-4o-mini"),
    tools: {
      getRepoInfo: tool({
        description: "获取 GitHub 仓库的详细信息",
        parameters: z.object({
          owner: z.string(),
          repo: z.string(),
        }),
        execute: async ({ owner, repo }) => {
          // 获取仓库基本信息
          const repoRes = await fetch(
            `https://api.github.com/repos/${owner}/${repo}`
          );
          const repoData = await repoRes.json();

          // 获取仓库语言分布
          const langRes = await fetch(
            `https://api.github.com/repos/${owner}/${repo}/languages`
          );
          const langData = await langRes.json();

          return {
            name: repoData.full_name,
            description: repoData.description,
            stars: repoData.stargazers_count,
            forks: repoData.forks_count,
            openIssues: repoData.open_issues_count,
            createdAt: repoData.created_at,
            updatedAt: repoData.updated_at,
            languages: Object.keys(langData),
          };
        },
      }),
    },
    prompt: `请分析这个 GitHub 仓库：${repoUrl}

提供一个简洁的分析，包括：
1. 仓库的用途（基于描述）
2. 流行程度（stars、forks）
3. 主要编程语言
4. 活跃程度（基于更新时间）`,
  });

  return Response.json({ analysis: text });
}
```

```tsx
// app/analyze/page.tsx
"use client";

import { useState } from "react";

export default function AnalyzePage() {
  const [repoUrl, setRepoUrl] = useState("");
  const [analysis, setAnalysis] = useState("");
  const [isLoading, setIsLoading] = useState(false);

  const analyzeRepo = async () => {
    if (!repoUrl.trim()) return;

    setIsLoading(true);
    try {
      const response = await fetch("/api/analyze", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ repoUrl }),
      });

      const data = await response.json();
      setAnalysis(data.analysis);
    } catch (error) {
      console.error("分析失败:", error);
    } finally {
      setIsLoading(false);
    }
  };

  return (
    <div className="max-w-2xl mx-auto p-6">
      <h1 className="text-2xl font-bold mb-6">代码仓库分析 Agent</h1>

      <div className="mb-4">
        <input
          type="text"
          value={repoUrl}
          onChange={(e) => setRepoUrl(e.target.value)}
          placeholder="输入 GitHub 仓库 URL，如：https://github.com/facebook/react"
          className="w-full p-3 border rounded-lg"
        />
      </div>

      <button
        onClick={analyzeRepo}
        disabled={isLoading || !repoUrl.trim()}
        className="w-full p-3 bg-blue-500 text-white rounded-lg disabled:opacity-50"
      >
        {isLoading ? "分析中..." : "分析仓库"}
      </button>

      {analysis && (
        <div className="mt-6 p-4 bg-gray-50 rounded-lg">
          <h2 className="font-semibold mb-2">分析结果</h2>
          <div className="whitespace-pre-wrap">{analysis}</div>
        </div>
      )}
    </div>
  );
}
```

这是一个入门版的代码仓库分析 Agent。它能够：

- 解析 GitHub URL
- 调用 GitHub API 获取仓库信息
- 让 LLM 分析这些信息
- 返回简洁的分析报告

虽然简单，但它展示了 Agent 的核心模式：接收用户请求 → 调用工具获取信息 → 让 LLM 处理并生成回复。

## 入门阶段的权衡

在入门阶段，我们用了相对简单的方式实现功能。这种做法有优势也有局限。

优势是快速上手、易于理解。直接调用 API，不用学习复杂的框架，适合快速验证想法。代码量少，调试方便。

局限是扩展性不足。当需要添加更多工具、更复杂的逻辑时，代码会变得难以维护。缺少记忆管理、错误重试等生产级特性。

这正是为什么需要进阶阶段的学习。我们会引入框架（如 LangChain），学习如何构建更强大、更可维护的 Agent。

但记住，理解基础是掌握高级技巧的前提。你现在应该能理解 Agent 的核心工作了：接收请求、调用工具、处理结果、生成回复。
