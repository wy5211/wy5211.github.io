前 6 篇我们用原生 MyBatis 完成了 BookHive 的核心功能。每次新增一个实体，要写三样东西：实体类、Mapper 接口、Mapper XML。Mapper XML 里大部分是样板代码——insert、update、delete、selectById、selectList……这些操作的模式几乎一模一样，只是表名和字段不同。

你可能在想：这些重复的 CRUD 能不能自动生成？

MyBatis-Plus 就是干这个的。它在不改变 MyBatis 原有功能的前提下，提供了通用 CRUD、条件构造器、分页插件、代码生成器等增强功能。2026 年，MyBatis-Plus 已经是国内 Java 项目的标配——用 MyBatis 基本就意味着用 MyBatis-Plus。

## BaseMapper：零 XML 的 CRUD

MyBatis-Plus 的核心是 `BaseMapper<T>` 接口。让你的 Mapper 接口继承它，立刻获得一套通用的 CRUD 方法。

```java
package com.bookhive.mapper;

import com.baomidou.mybatisplus.core.mapper.BaseMapper;
import com.bookhive.entity.Book;

public interface BookMapper extends BaseMapper<Book> {
    // 不需要写任何方法，BaseMapper 已经提供了 17+ 个通用方法
    // 只需要在这里添加 BaseMapper 没有的自定义查询
}
```

不需要写 Mapper XML。`BaseMapper` 提供的方法包括：

```java
// 插入
int insert(T entity);

// 删除
int deleteById(Serializable id);
int deleteByMap(Map<String, Object> columnMap);
int delete(Wrapper<T> queryWrapper);
int deleteBatchIds(Collection<? extends Serializable> idList);

// 更新
int updateById(T entity);
int update(T entity, Wrapper<T> updateWrapper);

// 查询
T selectById(Serializable id);
List<T> selectBatchIds(Collection<? extends Serializable> idList);
List<T> selectByMap(Map<String, Object> columnMap);
T selectOne(Wrapper<T> queryWrapper);
Long selectCount(Wrapper<T> queryWrapper);
List<T> selectList(Wrapper<T> queryWrapper);
Page<T> selectPage(Page<T> page, Wrapper<T> queryWrapper);
```

用法：

```java
@Service
public class BookService {

    private final BookMapper bookMapper;

    public BookService(BookMapper bookMapper) {
        this.bookMapper = bookMapper;
    }

    // 插入
    public void addBook(Book book) {
        bookMapper.insert(book);
        // 主键自动回填到 book.getId()
    }

    // 根据 ID 查询
    public Book getById(Long id) {
        return bookMapper.selectById(id);
    }

    // 根据 ID 列表批量查询
    public List<Book> getByIds(List<Long> ids) {
        return bookMapper.selectBatchIds(ids);
    }

    // 查询所有
    public List<Book> getAll() {
        return bookMapper.selectList(null); // null 表示无条件
    }

    // 更新
    public void updatePrice(Long bookId, BigDecimal newPrice) {
        Book book = new Book();
        book.setId(bookId);
        book.setPrice(newPrice);
        bookMapper.updateById(book);
        // 只更新非 null 字段（选择性更新）
    }

    // 删除
    public void removeBook(Long id) {
        bookMapper.deleteById(id);
    }
}
```

注意 `updateById` 的行为：**只更新非 null 字段**。传一个只设置了 `id` 和 `price` 的 `Book` 对象，只有 `price` 字段会被更新，其他字段不受影响。这和原生 MyBatis 的 `update` 标签不同——原生 MyBatis 的 `update` 需要你手动写 `<set>` + `<if>` 来实现选择性更新。

## 条件构造器：替代动态 SQL 的简单查询

复杂的多条件搜索，原生 MyBatis 要写动态 SQL 的 XML。MyBatis-Plus 提供了条件构造器，在 Java 代码中构建查询条件。

### QueryWrapper

```java
// 查找价格在 20-50 之间、标题包含"Java"的在售书籍
QueryWrapper<Book> wrapper = new QueryWrapper<>();
wrapper.eq("status", 1)                        // status = 1
       .like("title", "Java")                   // title LIKE '%Java%'
       .between("price", 20, 50)                // price BETWEEN 20 AND 50
       .orderByDesc("created_at")               // ORDER BY created_at DESC
       .last("LIMIT 10");                        // 追加 SQL 片段

List<Book> books = bookMapper.selectList(wrapper);
```

条件构造器的方法名和 SQL 关键字一一对应：`eq`、`ne`、`gt`、`ge`、`lt`、`le`、`like`、`notLike`、`in`、`notIn`、`between`、`isNull`、`isNotNull`、`orderByAsc`、`orderByDesc`、`groupBy`、`having`……覆盖了绝大部分 SQL 条件。

### LambdaQueryWrapper

`QueryWrapper` 用字符串指定列名，拼错列名编译不会报错。`LambdaQueryWrapper` 用方法引用指定列名，编译期就能检查：

```java
LambdaQueryWrapper<Book> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(Book::getStatus, 1)
       .like(Book::getTitle, "Java")
       .between(Book::getPrice, new BigDecimal("20"), new BigDecimal("50"))
       .orderByDesc(Book::getCreatedAt)
       .last("LIMIT 10");

List<Book> books = bookMapper.selectList(wrapper);
```

`Book::getTitle` 是 Java 8 的方法引用。MyBatis-Plus 通过它推导出列名是 `title`。如果 `Book` 类没有 `getTitle()` 方法，编译就报错。

实际项目中，**推荐使用 LambdaQueryWrapper**。虽然多打几个字，但列名拼错的 bug 在编译期就能发现，而不是运行时看到一个莫名其妙的 SQL 错误。

### 动态条件

条件构造器支持条件性地添加查询条件，替代了 XML 中的 `<if>` 标签：

```java
public List<Book> searchBooks(BookQueryDTO query) {
    LambdaQueryWrapper<Book> wrapper = new LambdaQueryWrapper<>();

    // 条件性地添加查询条件
    wrapper.like(query.getTitle() != null, Book::getTitle, query.getTitle())
           .ge(query.getMinPrice() != null, Book::getPrice, query.getMinPrice())
           .le(query.getMaxPrice() != null, Book::getPrice, query.getMaxPrice())
           .eq(query.getPublisherId() != null, Book::getPublisherId, query.getPublisherId())
           .orderByDesc(Book::getCreatedAt);

    return bookMapper.selectList(wrapper);
}
```

第一个参数是 boolean——只有为 true 时才会添加这个条件。这和 XML 中的 `<if test="title != null">` 效果一样，但写在 Java 代码里更直观。

### 条件构造器的局限

条件构造器适合中等复杂度的查询。但如果查询涉及多表 JOIN、子查询、复杂的聚合函数，条件构造器就力不从心了——代码会变得比 XML 动态 SQL 还难读。

这种时候，用原生 MyBatis 的 XML 方式写查询，和 MyBatis-Plus 的 BaseMapper 并存：

```java
public interface BookMapper extends BaseMapper<Book> {
    // BaseMapper 提供了通用 CRUD

    // 自定义的复杂查询用 XML 实现
    List<BookSalesDTO> selectSalesStatistics(
        @Param("startDate") LocalDate startDate,
        @Param("endDate") LocalDate endDate
    );
}
```

MyBatis-Plus 不会覆盖你在 XML 中自定义的方法。两者和平共处。

## 自动填充：createdAt 和 updatedAt

每个表都有 `created_at` 和 `updated_at` 字段。每次 insert 时设置 `createdAt`，每次 update 时更新 `updatedAt`——这是样板代码中的样板代码。

MyBatis-Plus 的自动填充功能让这个工作自动完成。

### 实体类标注

```java
public class Book {
    private Long id;
    private String title;
    private String isbn;
    private BigDecimal price;
    private Integer stock;

    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime createdAt;

    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updatedAt;
    // ...
}
```

`FieldFill.INSERT`：插入时自动填充。`FieldFill.INSERT_UPDATE`：插入和更新时都自动填充。

### MetaObjectHandler

```java
@Component
public class AutoFillHandler implements MetaObjectHandler {

    @Override
    public void insertFill(MetaObject metaObject) {
        this.strictInsertFill(metaObject, "createdAt", LocalDateTime.class, LocalDateTime.now());
        this.strictInsertFill(metaObject, "updatedAt", LocalDateTime.class, LocalDateTime.now());
    }

    @Override
    public void updateFill(MetaObject metaObject) {
        this.strictUpdateFill(metaObject, "updatedAt", LocalDateTime.class, LocalDateTime.now());
    }
}
```

`strictInsertFill` 和 `strictUpdateFill` 只在字段为 null 时才填充——如果你手动设置了值，不会被覆盖。

配置完成后，insert 和 update 时不需要手动设置这两个字段了：

```java
Book book = new Book();
book.setTitle("深入理解 Java 虚拟机");
book.setPrice(new BigDecimal("99.00"));
bookMapper.insert(book);
// book.getCreatedAt() 和 book.getUpdatedAt() 自动填充
```

## 逻辑删除

实际项目中，大部分数据不应该被物理删除——删除了就找不回来了。更常见的做法是逻辑删除：加一个 `deleted` 字段（0 表示正常，1 表示已删除），删除操作实际上是更新这个字段。

MyBatis-Plus 内置了逻辑删除支持。

### 配置

```yaml
# application.yml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: deleted # 全局逻辑删除字段名
      logic-delete-value: 1 # 删除后的值
      logic-not-delete-value: 0 # 未删除的值
```

实体类：

```java
public class Book {
    private Long id;
    private String title;
    // ...
    @TableLogic
    private Integer deleted;
}
```

### 效果

配置逻辑删除后，MyBatis-Plus 会自动修改你的 SQL：

```java
// 你写的代码
bookMapper.deleteById(1L);

// MyBatis-Plus 实际执行的 SQL
UPDATE books SET deleted = 1 WHERE id = 1 AND deleted = 0

// 你写的代码
bookMapper.selectById(1L);

// MyBatis-Plus 实际执行的 SQL
SELECT * FROM books WHERE id = 1 AND deleted = 0
```

所有的查询操作自动加上 `deleted = 0` 条件，删除操作变成更新操作。对业务代码完全透明——你不需要在每个查询里手动加 `WHERE deleted = 0`。

### 逻辑删除的注意事项

逻辑删除方便，但有几个需要注意的地方：

- **唯一约束**。逻辑删除的记录还留在数据库里。如果 `username` 字段有 UNIQUE 约束，用户删除后重新注册同名用户会报唯一约束冲突。解决方案：唯一约束加上 `deleted` 字段（`UNIQUE(username, deleted)`），或者用软删除时间戳代替布尔标志。

- **数据量增长**。逻辑删除的记录永远在表里，数据量会持续增长。需要定期归档或清理。

- **关联查询**。逻辑删除的记录在 JOIN 查询中可能出现。比如订单关联了一个被逻辑删除的书籍，查询结果中会包含这本书的信息。如果不想这样，需要手动加过滤条件。

## IService：通用 Service 层

`BaseMapper` 在 Mapper 层提供了通用 CRUD。MyBatis-Plus 还提供了 `IService<T>` 接口和 `ServiceImpl<M, T>` 实现类，在 Service 层提供同样的能力。

```java
// Service 接口
public interface BookService extends IService<Book> {
    // IService 提供了通用的 CRUD 方法
    // 在这里添加自定义的业务方法
}

// Service 实现
@Service
public class BookServiceImpl extends ServiceImpl<BookMapper, Book>
        implements BookService {
    // BookMapper 是 Mapper 接口，Book 是实体类
    // ServiceImpl 自动注入了 BookMapper
}
```

`IService` 提供的方法比 `BaseMapper` 更多，包括批量操作、保存或更新、链式调用等：

```java
// 批量保存
bookService.saveBatch(bookList);

// 保存或更新（有 id 就更新，没有 id 就插入）
bookService.saveOrUpdate(book);

// 链式查询
List<Book> books = bookService.lambdaQuery()
        .eq(Book::getStatus, 1)
        .like(Book::getTitle, "Java")
        .orderByDesc(Book::getCreatedAt)
        .list();

// 链式更新
bookService.lambdaUpdate()
        .eq(Book::getId, bookId)
        .set(Book::getPrice, newPrice)
        .set(Book::getUpdatedAt, LocalDateTime.now())
        .update();
```

链式调用是 MyBatis-Plus 3.x 新增的特性，写起来比单独创建 `LambdaQueryWrapper` 更简洁。

### 要不要用 IService

这取决于你的 Service 层有多复杂。

如果你的 Service 只是简单封装 Mapper 的 CRUD（没有复杂的业务逻辑），用 `IService` 可以省不少代码。

但如果你的 Service 有复杂的业务逻辑（事务管理、跨 Mapper 调用、业务规则校验），继承 `ServiceImpl` 可能会让代码结构不清晰——业务方法和通用 CRUD 方法混在一起，分不清哪些是通用能力、哪些是业务逻辑。

建议：简单的模块（比如字典管理、配置管理）用 `IService`。复杂的模块（比如订单、支付）还是手动写 Service，保持业务逻辑的清晰。

## MyBatis-Plus 的局限性

MyBatis-Plus 不是万能的。了解它的局限，才能在合适的场景使用。

**复杂查询还是要写 XML**。条件构造器处理不了多表 JOIN、子查询、复杂的聚合函数。这种查询还是需要 XML 方式。

**代码生成器的模板可能不符合你的规范**。MyBatis-Plus 的代码生成器可以生成实体类、Mapper、Service、Controller，但生成的代码可能不符合你团队的编码规范。需要自定义模板。

**和原生 MyBatis 的版本兼容**。MyBatis-Plus 依赖特定版本的 MyBatis。升级 MyBatis 时要注意兼容性。

**自动填充和逻辑删除是全局行为**。一旦配置了全局的逻辑删除策略，所有的查询都会带上 `deleted = 0` 条件。如果有特殊的查询需要包含已删除的数据，需要单独处理。

**学习成本**。MyBatis-Plus 有自己的 API（Wrapper、IService、链式调用等），团队需要学习。但这个学习成本不高——API 设计很直观，大部分方法名一看就懂。

对于 BookHive 项目，MyBatis-Plus 的 BaseMapper 和 LambdaQueryWrapper 覆盖了 80% 的数据访问需求。剩下的 20%（多表关联查询、统计报表）用 XML 实现。下一篇讲分页查询和性能优化，在 MyBatis-Plus 基础上进一步提升查询效率。
