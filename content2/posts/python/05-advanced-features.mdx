# 高级特性：装饰器、生成器、上下文管理器

上一篇我们在 `Article` 类上写了 `@property`、`@staticmethod`、`@classmethod`，还在 `@dataclass` 上看到了同样的 `@` 符号。如果你有 Java 背景，第一个反应大概是："这不就是注解（Annotation）吗？" 看起来确实像，但机制完全不同。Java 的 `@Override` 和 `@Autowired` 是标记，本身不执行任何代码，需要 Spring 之类的框架在运行时通过反射读取并处理。Python 的 `@property` 是**立刻执行**的——它在类定义时就把一个方法变成了属性描述符。

这个区别是理解 Python 高级特性的钥匙。装饰器、生成器、上下文管理器是 Python 的三大"元编程"工具，它们让你在不修改原有代码的前提下注入行为、惰性处理数据、安全管理资源。Flask 的 `@app.route`、FastAPI 的 `@router.get`、Django 的 `@login_required`，都是装饰器的实际应用。

---

## 装饰器：高阶函数，不是注解

先回到上一篇留下的谜题：`@property` 上的 `@` 到底是什么？答案是：**语法糖**，等价于一次函数调用。

```python
class Article:
    def __init__(self, title: str, status: str = "draft"):
        self.title = title
        self._status = status

    @property                          # 语法糖
    def status(self) -> str:
        return self._status

    # 等价于：status = property(status)  — 把 status 方法传给 property，返回一个描述符对象
```

`@property` 实际上是 `property()` 这个内置函数的调用。Python 在定义类体时，把 `status` 方法传给 `property()`，返回的结果赋值回 `status` 这个名字。整个过程中没有任何"标记"，没有反射，没有框架介入——就是一次纯粹的函数调用。

理解了这个，装饰器就没什么神秘的了。

### 函数装饰器的本质

装饰器就是一个**高阶函数**：接收一个函数作为参数，返回一个新函数。最常见的场景是给函数添加计时、日志、权限检查等横切关注点——Java 里你用 AOP + 自定义注解做的事，Python 里用装饰器一行搞定。

```python
import time
import functools


def timer(func):
    """计时装饰器：测量函数执行时间"""
    @functools.wraps(func)  # 保留原函数的 __name__、__doc__ 等元信息
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"[{func.__name__}] 耗时 {elapsed:.4f}s")
        return result
    return wrapper


@timer  # 等价于：fetch_articles = timer(fetch_articles)
def fetch_articles():
    """获取文章列表"""
    time.sleep(0.3)
    return [{"title": "Python 入门", "views": 1200}]


print(fetch_articles().__name__)  # fetch_articles — 因为 @functools.wraps
# 输出：[fetch_articles] 耗时 0.3001s
```

`@functools.wraps(func)` 不是可有可无的装饰。不加它，`fetch_articles.__name__` 会变成 `wrapper`，调试时你看到一堆 `wrapper` 函数，根本不知道原来的函数叫什么。这是装饰器的第一个坑：**必须用 wraps 保留元信息**。

### 带参数的装饰器：三层嵌套

上面的 `timer` 不接受配置参数。如果我想指定日志级别或者只记录超过阈值的调用呢？这就需要带参数的装饰器——三层嵌套函数。

```python
def retry(max_retries: int = 3, delay: float = 1.0,
          exceptions: tuple = (Exception,)):
    """重试装饰器：函数失败时自动重试

    三层结构：
    - retry() 接收配置参数，返回 decorator
    - decorator() 接收被装饰函数，返回 wrapper
    - wrapper() 执行实际的装饰逻辑
    """
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(1, max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except exceptions as e:
                    if attempt == max_retries:
                        print(f"[{func.__name__}] 重试 {max_retries} 次后失败: {e}")
                        raise
                    print(f"[{func.__name__}] 第 {attempt} 次失败，{delay}s 后重试")
                    time.sleep(delay)
        return wrapper
    return decorator


@retry(max_retries=3, delay=0.5, exceptions=(ConnectionError, TimeoutError))
def fetch_remote_data(url: str) -> str:
    """模拟可能失败的网络请求"""
    import random
    if random.random() < 0.7:
        raise ConnectionError("Connection refused")
    return f"Data from {url}"
```

调用链是 `retry(3, 0.5, ...)(fetch_remote_data)` —— 先用配置参数调用 `retry`，得到 `decorator`，再用 `decorator` 包装 `fetch_remote_data`。三层嵌套是 Python 装饰器的常见模式，虽然看起来有些绕，但每层职责清晰：外层管配置、中层管包装、内层管执行。

### 类装饰器

装饰器不一定非要是函数，任何实现了 `__call__` 的类都可以充当装饰器。类装饰器的好处是可以持有状态。

```python
class CountCalls:
    """统计函数调用次数的类装饰器"""

    def __init__(self, func):
        self.func = func
        self.count = 0
        functools.update_wrapper(self, func)

    def __call__(self, *args, **kwargs):
        self.count += 1
        print(f"[Call #{self.count}] {self.func.__name__}")
        return self.func(*args, **kwargs)


@CountCalls
def save_article(title: str):
    print(f"保存文章: {title}")


save_article("Python 入门")   # [Call #1] save_article
save_article("Docker 部署")   # [Call #2] save_article
print(f"总调用次数: {save_article.count}")  # 2
```

`CountCalls` 在装饰时被实例化，`func` 被存储为实例属性。之后每次调用 `save_article`，实际上是在调用 `CountCalls` 实例的 `__call__` 方法。调用次数作为实例状态被持久化了——这是函数装饰器做不到的（函数装饰器的状态只能通过闭包变量保存，不如实例属性直观）。

### 装饰器的叠加顺序

多个装饰器叠加时，执行顺序是从下往上包装、从上往下调用：

```python
@timer
@retry(max_retries=2, delay=0.1)
def publish_article(title: str):
    """先重试，再计时"""
    time.sleep(0.1)
    return f"{title} 已发布"

# 等价于：publish_article = timer(retry(max_retries=2, delay=0.1)(publish_article))
# 调用时：timer.wrapper -> retry.wrapper -> publish_article
```

`@timer` 在最外层，所以计时包含了重试的时间。如果你想分别计时每次重试和总时间，需要调整装饰器的顺序。这个叠加顺序和 Java Spring AOP 的 `@Order` 概念类似，但 Python 靠物理位置决定，更直接。

### 和 Java 注解的本质区别

说了这么多，回到最初的问题：装饰器和 Java 注解到底是不是一回事？

```java
// Java — 注解只是标记，需要框架处理
@Transactional
@Retry(maxAttempts = 3)
public void publishArticle(String title) { ... }
// Spring 在启动时扫描这些注解，通过动态代理生成代理对象
// 注解本身不执行任何代码
```

```python
# Python — 装饰器是立即执行的高阶函数
@retry(max_retries=3)
@timer
def publish_article(title: str): ...
# 在模块加载时，publish_article 就已经被 retry(timer(publish_article)) 替换了
# 不需要任何框架介入
```

| 维度     | Java 注解                      | Python 装饰器                  |
| -------- | ------------------------------ | ------------------------------ |
| 本质     | 标记（metadata）               | 高阶函数调用                   |
| 执行时机 | 框架运行时读取处理             | 模块加载时立即执行             |
| 处理方式 | 反射 + 动态代理（Spring AOP）  | 直接替换被装饰对象             |
| 依赖     | 需要框架支持（Spring、JPA 等） | 无需框架，语言原生支持         |
| 灵活性   | 注解参数有限，逻辑由框架决定   | 可以做任何 Python 函数能做的事 |

Python 装饰器更接近 Java 的**代理模式（Proxy Pattern）**加 **AOP 切面**的结合体，但它是语言级的，不需要 Spring 介入。这是 FastAPI 的路由注册比 Spring Boot 的 Controller 扫描更简洁的原因——`@app.get("/articles")` 直接就是一个函数调用，不需要启动 IoC 容器来处理它。

## 生成器：用 yield 实现惰性求值

如果你用 Java Stream 处理过大数据集，你应该熟悉"惰性求值"的概念——中间操作（`filter`、`map`）不会立即执行，只有遇到终端操作（`collect`、`forEach`）时才会触发计算。Python 的生成器是更原生的惰性求值机制：用 `yield` 关键字把一个函数变成"可暂停的迭代器"。

### yield：函数的暂停键

普通函数调用 `return` 后就结束了，调用栈被销毁。生成器函数调用 `yield` 后暂停执行，保留完整的调用栈状态（局部变量、执行位置），下次调用时从暂停点继续。

```python
def article_feed(total: int, batch_size: int = 3):
    """分批生成文章数据 — 模拟分页查询"""
    offset = 0
    while offset < total:
        # 模拟数据库查询：每次取 batch_size 条
        batch = [
            {"id": i, "title": f"文章 {i}"}
            for i in range(offset, min(offset + batch_size, total))
        ]
        yield batch  # 暂停，返回当前批次，下次从这里继续
        offset += batch_size
        print(f"  已返回 {offset}/{total} 条")


# 第一次调用
feed = article_feed(10, batch_size=3)
print(next(feed))  # [{"id": 0, ...}, {"id": 1, ...}, {"id": 2, ...}]
print(next(feed))  # [{"id": 3, ...}, {"id": 4, ...}, {"id": 5, ...}]
print(next(feed))  # [{"id": 6, ...}, {"id": 7, ...}, {"id": 8, ...}]
print(next(feed))  # [{"id": 9, ...}]
# next(feed)  # StopIteration — 没有更多数据
```

每次 `next(feed)` 调用时，函数从上一次 `yield` 的位置继续执行，直到遇到下一个 `yield` 或函数结束。`offset`、`batch_size`、`total` 这些局部变量的值被完整保留——这就是生成器和普通函数的核心区别。

### 惰性求值 vs 一次性生成

来看一个直观的内存对比：

```python
# 列表推导式 — 一次性生成 100 万个结果，全部存入内存
squares_list = [x ** 2 for x in range(1_000_000)]
# 占用约 8MB 内存

# 生成器表达式 — 惰性求值，只在需要时计算下一个值
squares_gen = (x ** 2 for x in range(1_000_000))
# 占用约 200 字节 — 只有迭代器对象本身

# 生成器表达式常用于函数参数
total = sum(x ** 2 for x in range(1_000_000))  # 不创建中间列表
max_val = max(x ** 2 for x in range(1_000_000))
```

语法上唯一的区别是括号：列表推导式用 `[]`，生成器表达式用 `()`。但行为完全不同——列表推导式立即创建完整列表，生成器表达式只在迭代时逐个计算。

### 处理大文件：生成器的典型场景

处理 GB 级日志文件时，你不能 `readlines()` 把所有行加载到内存。生成器让你逐行处理，内存占用恒定：

```python
def analyze_log(file_path: str, min_level: str = "ERROR"):
    """逐行读取日志，只返回指定级别以上的条目"""
    with open(file_path, "r", encoding="utf-8") as f:
        for line in f:
            line = line.strip()
            if not line:
                continue
            # 简单解析：假设格式为 "2026-03-30 ERROR message..."
            parts = line.split(" ", 2)
            if len(parts) < 2:
                continue
            level = parts[1]
            if level >= min_level:  # 字符串比较，ERROR > WARNING > INFO
                yield {"timestamp": parts[0], "level": level, "message": parts[2]}


# 处理 10GB 日志文件，内存占用不超过几 MB
error_count = 0
for entry in analyze_log("/var/log/app.log", min_level="ERROR"):
    error_count += 1
    if error_count <= 3:
        print(f"[{entry['timestamp']}] {entry['message']}")
print(f"共 {error_count} 条错误日志")
```

`for line in f` 本身就是一个惰性迭代器——Python 的文件对象逐行读取，不会一次性加载整个文件。生成器函数在此基础上加了一层过滤和解析，但内存模型不变。

### yield from：委托给子生成器

当一个生成器需要消费另一个生成器的所有值时，用 `yield from` 比手写 `for ... yield` 更简洁，也更高效（它直接在 C 层传递值，不经过 Python 循环）：

```python
def parse_tags(tag_line: str):
    """解析标签字符串"""
    for tag in tag_line.split(","):
        tag = tag.strip()
        if tag:
            yield tag


def article_entries(file_path: str):
    """逐行解析文章数据，委托给子生成器"""
    with open(file_path, "r", encoding="utf-8") as f:
        for line in f:
            parts = line.strip().split("|")
            if len(parts) >= 3:
                yield {
                    "title": parts[0],
                    "content": parts[1],
                    "tags": list(parse_tags(parts[2]))  # 子生成器
                }


# yield from 等价写法对比
def flatten_manual(nested):
    for item in nested:
        if isinstance(item, (list, tuple)):
            for sub in item:  # 手动嵌套循环
                yield sub
        else:
            yield item

def flatten_with_yield_from(nested):
    for item in nested:
        if isinstance(item, (list, tuple)):
            yield from flatten_with_yield_from(item)  # 委托
        else:
            yield item
```

### 和 Java Stream 的对比

| 维度     | Java Stream                           | Python 生成器                                  |
| -------- | ------------------------------------- | ---------------------------------------------- |
| 惰性求值 | 中间操作惰性，终端操作触发            | 天然惰性，每次 `next()` 计算一个值             |
| 可重用性 | Stream 一次性的，终端操作后不能再消费 | 生成器也是一次性的，但重新调用函数就能创建新的 |
| 创建方式 | `Collection.stream()`                 | `yield` 函数 或 `()` 生成器表达式              |
| 并行     | `.parallelStream()` 原生支持          | 无内置并行，需用 `multiprocessing`             |
| 内存模型 | 内部用 Spliterator 分片               | 单线程，保持调用栈状态                         |
| 典型用途 | 集合的链式处理                        | 大文件处理、分页、管道式数据流                 |

Java Stream 更像是一个"数据处理管道"——你用 `filter`、`map`、`collect` 串起来，最后一次性执行。Python 生成器更像是一个"数据生产者"——它一个一个地产出数据，消费者按需拉取。在处理 GB 级文件或数据库分页时，生成器模型更自然。

## 上下文管理器：Python 的 try-with-resources

Java 7 引入的 `try-with-resources` 是资源管理的标准写法：任何实现了 `AutoCloseable` 的对象都可以放在 `try` 后面的括号里，无论是否发生异常，`close()` 都会被调用。Python 用 `with` 语句做同样的事，但实现方式不同。

### `__enter__` 和 `__exit__`

任何实现了 `__enter__` 和 `__exit__` 方法的对象都是上下文管理器：

```python
class DatabaseConnection:
    """数据库连接的上下文管理器"""

    def __init__(self, host: str, port: int, database: str):
        self.host = host
        self.port = port
        self.database = database
        self.conn = None

    def __enter__(self):
        """进入 with 块时调用 — 获取资源"""
        print(f"连接 {self.host}:{self.port}/{self.database}")
        self.conn = f"Connection({self.host}:{self.port})"
        return self.conn  # 返回值赋给 as 后的变量

    def __exit__(self, exc_type, exc_val, exc_tb):
        """退出 with 块时调用 — 释放资源"""
        print("关闭数据库连接")
        self.conn = None
        # 返回 False 表示不抑制异常（异常会继续向上传播）
        # 返回 True 表示异常已被处理，不再向上传播
        return False


# 使用
with DatabaseConnection("localhost", 5432, "techblog") as conn:
    print(f"使用连接: {conn}")
    # 即使这里抛出异常，__exit__ 也会被调用
# 离开 with 块时自动调用 __exit__
```

对比 Java 的等价写法：

```java
// Java — 实现 AutoCloseable 接口
public class DatabaseConnection implements AutoCloseable {
    private Connection conn;

    public DatabaseConnection(String host, int port, String db) {
        this.conn = DriverManager.getConnection(...);
    }

    @Override
    public void close() {
        conn.close();
    }
}

// 使用 — try-with-resources
try (var db = new DatabaseConnection("localhost", 5432, "techblog")) {
    db.execute("SELECT ...");
}  // 自动调用 close()
```

Python 的 `__enter__` 负责获取资源并返回使用对象，`__exit__` 负责释放资源。Java 的构造器负责获取资源，`close()` 负责释放。Python 把"进入"和"退出"拆成两个方法的好处是：`__exit__` 能接收异常信息（`exc_type`、`exc_val`、`exc_tb`），你可以在资源清理时根据异常类型做不同处理。

### 用 contextmanager 装饰器简化

不是每个上下文管理器都需要一个完整的类。Python 标准库的 `contextlib.contextmanager` 装饰器让你用生成器函数写上下文管理器——`yield` 之前的代码是 `__enter__`，`yield` 之后的代码是 `__exit__`：

```python
from contextlib import contextmanager
import time


@contextmanager
def measure_time(label: str):
    """计时上下文管理器 — 用生成器替代类"""
    start = time.perf_counter()
    print(f"[{label}] 开始")
    try:
        yield  # yield 之前的代码 = __enter__，之后的代码 = __exit__
    finally:
        elapsed = time.perf_counter() - start
        print(f"[{label}] 结束，耗时 {elapsed:.4f}s")


@contextmanager
def transaction(connection):
    """数据库事务上下文管理器"""
    print("BEGIN TRANSACTION")
    try:
        yield connection
        print("COMMIT")
    except Exception:
        print("ROLLBACK")
        raise


# 使用
with measure_time("数据导入"):
    time.sleep(0.5)

with transaction(conn):
    conn.execute("INSERT INTO articles ...")
    conn.execute("UPDATE counters ...")
    # 如果任一语句失败，自动 ROLLBACK
```

`@contextmanager` 是装饰器和生成器结合的经典案例——生成器负责管理"进入"和"退出"的状态，装饰器负责把生成器变成上下文管理器。这种组合在 Python 标准库里大量出现。

### 多资源管理

同时管理多个资源时，Python 可以嵌套 `with` 语句，也可以用逗号分隔：

```python
# 嵌套写法
with open("input.csv", "r") as infile:
    with open("output.csv", "w") as outfile:
        for line in infile:
            outfile.write(line.upper())

# Python 3.1+ 支持逗号分隔
with open("input.csv", "r") as infile, open("output.csv", "w") as outfile:
    for line in infile:
        outfile.write(line.upper())
```

Java 的 try-with-resources 也支持多资源，用分号分隔：

```java
try (var in = new FileReader("input.csv");
     var out = new FileWriter("output.csv")) {
    // ...
}
```

两者功能等价。Python 的优势在于 `@contextmanager` 让你不需要为了一个简单的资源管理逻辑去定义一个完整的类。

## 三者的协同：一个完整的实战场景

装饰器、生成器、上下文管理器经常一起出现。来看一个文章数据导出的场景，三个特性各司其职：

```python
import functools
import csv
from contextlib import contextmanager


# 1. 装饰器 — 记录导出操作日志
def log_export(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print(f"[EXPORT] 开始导出: {func.__name__}")
        result = func(*args, **kwargs)
        print(f"[EXPORT] 导出完成，共 {result} 条")
        return result
    return wrapper


# 2. 生成器 — 惰性查询文章数据（不一次性加载全部）
def query_articles(status: str = "published"):
    """逐批从数据库查询文章"""
    # 模拟分批查询
    for batch_start in range(0, 100, 20):
        batch = [
            {"id": i, "title": f"文章 {i}", "status": status}
            for i in range(batch_start, batch_start + 20)
        ]
        yield from batch


# 3. 上下文管理器 — 管理 CSV 文件的打开和关闭
@contextmanager
def csv_writer(file_path: str):
    """CSV 写入上下文管理器"""
    f = open(file_path, "w", newline="", encoding="utf-8")
    writer = csv.DictWriter(f, fieldnames=["id", "title", "status"])
    writer.writeheader()
    try:
        yield writer  # 把 writer 传给调用者
    finally:
        f.close()


# 组合使用
@log_export
def export_articles_to_csv(output_path: str):
    count = 0
    with csv_writer(output_path) as writer:
        for article in query_articles():  # 生成器惰性产出
            writer.writerow(article)
            count += 1
    return count


export_articles_to_csv("articles.csv")
```

装饰器管"横切关注点"（日志），生成器管"数据生产"（惰性查询），上下文管理器管"资源生命周期"（文件打开/关闭）。三者各司其职，组合起来就是一个干净的数据导出流程。Java 里实现同样的功能，你需要 Spring AOP 做日志、JPA 的 `Stream` 或 `Slice` 做分页、try-with-resources 管文件——能做，但分散在三个不同的机制里。

## 对比总览

| 特性         | Java                                              | Python                                     |
| ------------ | ------------------------------------------------- | ------------------------------------------ |
| 装饰器/注解  | 注解是 metadata，由框架（Spring AOP）通过反射处理 | 装饰器是高阶函数，模块加载时直接执行       |
| 行为注入方式 | 动态代理 / 字节码增强                             | 直接替换被装饰对象                         |
| 惰性求值     | Stream API（中间操作惰性，终端操作触发）          | 生成器（`yield`），逐个产出                |
| 大文件处理   | `BufferedReader.readLine()` 或 Stream             | 生成器 + `for line in f`                   |
| 资源管理     | `try-with-resources` + `AutoCloseable`            | `with` 语句 + `__enter__`/`__exit__`       |
| 简化资源管理 | 无对应（必须实现接口）                            | `@contextmanager` 装饰器                   |
| 代码复用     | AOP + 自定义注解                                  | 装饰器组合                                 |
| 调试         | 注解不影响调用栈                                  | 装饰器增加调用层级，需用 `@wraps` 保留信息 |

---

## Trade-off：Python 高级特性的设计取舍

装饰器的"立即执行"是它强大之处，也是隐患。在 Java 里，`@Transactional` 的行为由 Spring 事务管理器统一控制，你可以集中配置传播级别、隔离级别、回滚策略。Python 的装饰器是分散的——每个装饰器独立工作，叠加顺序依赖物理位置，没有统一的"装饰器容器"。当你看到 `@timer @retry @cache @log` 四层装饰器堆叠时，调试会变得困难，因为异常堆栈被包装了好几层。实际项目中，装饰器叠加不建议超过两层，超过时考虑用中间件或类来组织逻辑。

生成器是 Python 处理大数据集的标准方式，但它不支持随机访问——你不能 `gen[100]`，只能从头遍历到第 100 个元素。Java 的 `ArrayList` 支持随机访问，Stream 不支持但可以 `collect()` 到列表后再访问。Python 的应对方式是：如果需要多次遍历或随机访问，就用 `list(generator)` 把生成器物化成列表，代价是内存占用。选择生成器还是列表，本质上是在**时间**（物化的开销）和**空间**（内存占用）之间做权衡。

上下文管理器的 `@contextmanager` 虽然比定义类更简洁，但它有一个 Java `AutoCloseable` 没有的陷阱：如果在 `yield` 之前发生异常（资源还没获取成功），`finally` 块里可能没有资源可释放，但代码仍然会执行。你需要确保 `finally` 里的清理逻辑能处理"资源从未获取"的情况。Java 的 `AutoCloseable.close()` 规范也要求 `close()` 方法是幂等的（多次调用不报错），但 Python 的 `@contextmanager` 不会自动保证这一点。
