你在 Java 里调用 `FileReader("config.json")`，编译器直接报红——你没有处理 `FileNotFoundException`。要么 `try-catch`，要么在方法签名上 `throws`。不管你愿不愿意，编译器逼着你面对每个可能出错的地方。你转到 Python，写了同样的 `open("config.json")`，什么都没发生——没有编译器提醒你，运行时才会抛 `FileNotFoundError`，如果你没写 `try-except`，程序直接崩溃。

这是 Java 和 Python 错误处理最根本的区别：Java 有 checked exception，编译器强制你处理；Python 没有，一切异常都是 unchecked，处理不处理由你自己决定。Java 的设计哲学是"能出错的地方必须显式处理"，Python 的设计哲学是"约定优于强制，简洁优于安全"。这篇我们从 checked exception 的角度，理解 Python 的异常处理设计，以及在没有编译器强制的情况下，如何构建一套健壮的错误处理体系。

## Java vs Python 异常处理对比

| 维度       | Java                                   | Python                                        |
| ---------- | -------------------------------------- | --------------------------------------------- |
| 异常分类   | checked + unchecked（Error）           | 统一为 unchecked                              |
| 编译器强制 | checked exception 必须处理或 throws    | 不强制，所有异常都可以不处理                  |
| 异常声明   | `throws IOException` 写在方法签名上    | 不需要声明，`raise` 即可                      |
| 捕获语法   | `try-catch-finally`                    | `try-except-else-finally`                     |
| 异常链     | `new Exception(msg, cause)`            | `raise NewException from cause`               |
| 异常层次   | `Throwable` -> `Exception` / `Error`   | `BaseException` -> `Exception` / `SystemExit` |
| 自定义异常 | `class MyException extends Exception`  | `class MyError(Exception)`                    |
| 资源管理   | `try-with-resources` + `AutoCloseable` | `with` 语句 + 上下文管理器                    |
| 常见异常   | `NullPointerException`、`IOException`  | `TypeError`、`ValueError`、`KeyError`         |

最关键的行是第二行和第三行：Java 把异常分成 checked 和 unchecked，checked 的必须在编译时处理；Python 不区分，所有异常都是 unchecked。这个设计选择影响了两个语言整个错误处理体系的风格。

## checked exception 的争议：为什么 Python 不需要它

Java 的 checked exception 机制来自一个合理的想法：如果 `FileReader` 构造函数可能抛 `FileNotFoundException`，调用者应该知道并处理它。编译器帮你确保不会遗漏。

但实际使用中，checked exception 带来了一系列问题：

```java
// Java — checked exception 的连锁反应
public List<Article> loadArticles(String path) throws IOException {
    // FileReader 抛 IOException，你必须 throws
    var reader = new FileReader(path);
    // ...
}

public List<Article> getPublishedArticles(String path) throws IOException {
    // 调用 loadArticles，也必须 throws
    return loadArticles(path).stream()
        .filter(a -> "published".equals(a.getStatus()))
        .toList();
}

// 逐层 throws 直到 Controller，或者到处 try-catch
```

异常一层层往上传播，每个中间方法都要 `throws IOException`，或者用 `try-catch` 转换成 `RuntimeException`。最终很多 Java 代码变成了这样：

```java
// Java — checked exception 的常见"反模式"
try {
    service.doSomething();
} catch (IOException e) {
    throw new RuntimeException(e);  // 包装成 unchecked，等于白检查
}
```

Python 的设计者（Guido van Rossum）在 1999 年就写过一篇文章解释为什么 Python 不需要 checked exception：**大多数异常的处理方式是一样的——记录日志、包装后重新抛出、或者让程序崩溃**。编译器强制你处理每个 checked exception，并不能帮你写出更好的错误处理代码，反而增加了样板代码。

C# 的设计者（Anders Hejlsberg）也持同样观点：checked exception 的理念是好的，但实际效果是在大规模项目中增加了代码噪声，而不是提高了错误处理质量。C# 从第一个版本就没有 checked exception。

## Python 的异常层次结构

Python 的异常类构成一棵继承树，和 Java 的 `Throwable` -> `Exception` -> `RuntimeException` 类似，但更简洁：

```
BaseException
├── SystemExit            # sys.exit()
├── KeyboardInterrupt     # Ctrl+C
├── GeneratorExit         # 生成器关闭
└── Exception             # 所有常规异常的基类
    ├── StopIteration
    ├── ArithmeticError
    │   ├── ZeroDivisionError
    │   └── OverflowError
    ├── LookupError
    │   ├── IndexError     # list[10] 越界
    │   └── KeyError       # dict["missing"] 键不存在
    ├── TypeError          # 类型错误（传了 str 给需要 int 的参数）
    ├── ValueError         # 值错误（int("abc")）
    ├── AttributeError     # obj.not_exist 属性不存在
    ├── RuntimeError
    │   ├── NotImplementedError
    │   └── RecursionError
    ├── OSError
    │   ├── FileNotFoundError
    │   ├── PermissionError
    │   └── ConnectionError
    │       ├── ConnectionRefusedError
    │       └── TimeoutError
    └── ImportError
        └── ModuleNotFoundError
```

对比 Java：

```
Throwable
├── Error                  # 不应该被 catch 的严重错误
│   ├── OutOfMemoryError
│   └── StackOverflowError
└── Exception              # 应该被 catch 的异常
    ├── RuntimeException   # unchecked（不强制处理）
    │   ├── NullPointerException
    │   ├── IllegalArgumentException
    │   ├── IndexOutOfBoundsException
    │   └── ClassCastException
    └── IOException         # checked（强制处理）
        ├── FileNotFoundException
        └── SocketException
```

Java 把 `Exception` 下的异常分成 checked（如 `IOException`）和 unchecked（`RuntimeException` 及其子类）。Python 没有这种区分——`Exception` 下全是 unchecked。Python 的 `TypeError` 约等于 Java 的 `ClassCastException`，`ValueError` 约等于 `IllegalArgumentException`，`KeyError` 约等于 `NullPointerException`（在字典场景下）。

## try-except-else-finally：比 Java 多一个 else

Python 的异常处理语法比 Java 多一个 `else` 子句——只有 `try` 块没有抛异常时才执行：

```python
import json

def load_config(file_path: str) -> dict:
    """完整的异常处理结构"""
    try:
        with open(file_path, "r") as f:
            config = json.load(f)
    except FileNotFoundError:
        # 文件不存在 — 使用默认配置
        print(f"配置文件 {file_path} 不存在，使用默认配置")
        return {"debug": False, "port": 8000}
    except json.JSONDecodeError as e:
        # JSON 格式错误 — 包装后抛出
        raise ValueError(f"配置文件格式错误: {file_path}") from e
    else:
        # try 块没有异常时执行 — config 是合法的 dict
        print(f"成功读取配置: {file_path}")
        return config
    finally:
        # 无论是否异常都执行 — 这里不需要，因为 with 已经关闭了文件
        pass
```

等价的 Java 写法：

```java
// Java — 没有 else，逻辑需要放在 try 块末尾
public Map<String, Object> loadConfig(String path) throws IOException {
    try {
        String content = Files.readString(Path.of(path));
        Map<String, Object> config = new ObjectMapper().readValue(content, Map.class);
        // "成功读取" 的逻辑只能放在这里（try 块末尾）
        System.out.println("成功读取配置: " + path);
        return config;
    } catch (FileNotFoundException e) {
        System.out.println("配置文件不存在，使用默认配置");
        return Map.of("debug", false, "port", 8000);
    } catch (JsonProcessingException e) {
        throw new IllegalArgumentException("配置文件格式错误: " + path, e);
    }
}
```

`else` 的好处是把"可能出错"和"出错后"的逻辑分开。`try` 块里只放可能抛异常的代码，`else` 里放成功后的处理。这样你不会在 `try` 块里不小心 catch 到自己代码抛出的异常。Java 没有这个语法，"成功后的处理"只能放在 `try` 块末尾，如果那段代码本身也抛异常，就会和原始异常混在一起。

## 精确捕获：不要用裸 except

Python 有一个 Java 没有的陷阱——裸 `except:`（不指定异常类型）会捕获**所有异常**，包括 `KeyboardInterrupt` 和 `SystemExit`：

```python
# 错误 — 裸 except 会捕获 KeyboardInterrupt，Ctrl+C 都不管用了
try:
    process_data()
except:
    pass

# 正确 — 只捕获 Exception（不包括 KeyboardInterrupt 和 SystemExit）
try:
    process_data()
except Exception:
    logger.error("处理数据失败")
    raise

# 更好 — 精确捕获特定异常
try:
    process_data()
except (ValueError, KeyError) as e:
    logger.error("数据格式错误: %s", e)
    raise
```

Java 的 `catch (Exception e)` 不会捕获 `Error`（如 `OutOfMemoryError`），但会捕获所有 checked 和 unchecked exception。Python 的 `except Exception` 不会捕获 `KeyboardInterrupt` 和 `SystemExit`——这是 `BaseException` 的子类，不在 `Exception` 的继承树下。

| 捕获方式                | Java 捕获范围                     | Python 捕获范围                         |
| ----------------------- | --------------------------------- | --------------------------------------- |
| 最精确                  | `catch (FileNotFoundException e)` | `except FileNotFoundError as e:`        |
| 捕获多个                | `catch (IOException \| ... e)`    | `except (ValueError, KeyError) as e:`   |
| 宽泛                    | `catch (Exception e)`             | `except Exception:`                     |
| 捕获一切（含 Error）    | `catch (Throwable t)`             | `except BaseException:`（几乎不应该用） |
| 裸捕获（含 SystemExit） | 无                                | `except:`（危险，不要用）               |

## 自定义异常：构建业务错误体系

Java 项目通常定义一套异常层次：`BusinessException` -> `ValidationException`、`NotFoundException`、`ConflictException` 等。Python 也需要这样做——虽然编译器不强制，但统一的异常体系让错误处理更一致。

```python
# src/techblog/core/exceptions.py
from typing import Any


class AppError(Exception):
    """业务异常基类 — 所有自定义异常都继承它"""

    def __init__(
        self,
        message: str = "An application error occurred",
        code: str = "INTERNAL_ERROR",
        status_code: int = 500,
        details: dict[str, Any] | None = None,
    ) -> None:
        self.message = message
        self.code = code
        self.status_code = status_code
        self.details = details or {}
        super().__init__(self.message)

    def to_dict(self) -> dict[str, Any]:
        """序列化为字典 — 用于 API 错误响应"""
        return {
            "error": {
                "code": self.code,
                "message": self.message,
                "details": self.details,
            }
        }


class NotFoundError(AppError):
    """资源不存在 — 404"""

    def __init__(self, resource: str, resource_id: int | str) -> None:
        super().__init__(
            message=f"{resource} with id '{resource_id}' not found",
            code="NOT_FOUND",
            status_code=404,
            details={"resource": resource, "id": resource_id},
        )


class ValidationError(AppError):
    """数据验证失败 — 400"""

    def __init__(self, message: str, fields: dict[str, str] | None = None) -> None:
        super().__init__(
            message=message,
            code="VALIDATION_ERROR",
            status_code=400,
            details={"fields": fields or {}},
        )


class ConflictError(AppError):
    """资源冲突 — 409"""

    def __init__(self, message: str, details: dict[str, Any] | None = None) -> None:
        super().__init__(
            message=message,
            code="CONFLICT",
            status_code=409,
            details=details,
        )


class AuthenticationError(AppError):
    """认证失败 — 401"""

    def __init__(self, message: str = "Authentication failed") -> None:
        super().__init__(
            message=message,
            code="AUTHENTICATION_ERROR",
            status_code=401,
        )
```

对比 Java 的等价写法：

```java
// Java — 自定义异常需要 extends Exception（checked）或 RuntimeException（unchecked）
public class AppError extends RuntimeException {
    private final String code;
    private final int statusCode;
    private final Map<String, Object> details;

    public AppError(String message, String code, int statusCode) {
        super(message);
        this.code = code;
        this.statusCode = statusCode;
        this.details = new HashMap<>();
    }

    // getter 方法（Python 不需要，直接访问属性）
    // toDict() 方法...
}

// 每个子类都要写完整的构造函数
public class NotFoundError extends AppError {
    public NotFoundError(String resource, String resourceId) {
        super(
            resource + " with id '" + resourceId + "' not found",
            "NOT_FOUND",
            404
        );
    }
}
```

Python 的自定义异常比 Java 简洁得多——不需要 getter 方法，不需要 `private final` 字段，不需要一堆构造函数参数的 `this.xxx = xxx` 赋值。`super().__init__()` 一行就完成了基类初始化。Python 的异常可以直接添加任意属性（`self.message`、`self.code`），Java 需要预先声明字段。

在业务代码中使用：

```python
from techblog.core.exceptions import NotFoundError, ConflictError, ValidationError


class ArticleService:
    def __init__(self) -> None:
        self._articles: dict[int, dict] = {}

    def get_article(self, article_id: int) -> dict:
        article = self._articles.get(article_id)
        if not article:
            raise NotFoundError("Article", article_id)
        return article

    def create_article(self, title: str, content: str) -> dict:
        # 检查标题是否已存在
        for article in self._articles.values():
            if article["title"] == title:
                raise ConflictError(
                    message=f"标题 '{title}' 已存在",
                    details={"field": "title"},
                )
        # 验证内容长度
        if len(content) < 10:
            raise ValidationError(
                message="文章内容至少需要 10 个字符",
                fields={"content": "min_length: 10"},
            )

        article_id = len(self._articles) + 1
        article = {"id": article_id, "title": title, "content": content}
        self._articles[article_id] = article
        return article
```

在 FastAPI 的异常处理器中统一拦截：

```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from techblog.core.exceptions import AppError

app = FastAPI()


@app.exception_handler(AppError)
async def app_error_handler(request: Request, exc: AppError):
    """统一拦截业务异常 — 相当于 Spring 的 @ControllerAdvice"""
    return JSONResponse(
        status_code=exc.status_code,
        content=exc.to_dict(),
    )
```

这和 Spring Boot 的 `@ControllerAdvice` + `@ExceptionHandler` 是同一个思路：在框架层面统一拦截业务异常，返回标准化的错误响应。区别在于 Spring 的 `@ExceptionHandler` 用注解标记方法，FastAPI 的用装饰器——回到第 5 篇讲过的区别：注解是标记，装饰器是函数调用。

## 异常链：raise from 保留上下文

异常链是错误处理中最重要的实践之一。当底层异常被包装成业务异常时，你需要保留原始异常的信息，否则排查问题时你只知道"文章创建失败"，不知道是数据库连接超时还是磁盘空间不足。

Java 用异常的 `cause` 参数保留异常链：

```java
// Java — 异常链
try {
    repository.save(article);
} catch (SQLException e) {
    throw new ServiceException("保存文章失败", e);  // e 成为 cause
}
```

Python 用 `raise from` 语法：

```python
import json

class ArticleRepository:
    def load_articles(self, file_path: str) -> list[dict]:
        try:
            with open(file_path, "r") as f:
                return json.load(f)
        except FileNotFoundError as e:
            # raise from e — 保留原始异常作为 __cause__
            raise NotFoundError("ArticleData", file_path) from e
        except json.JSONDecodeError as e:
            raise ValidationError(
                message=f"文章数据文件格式错误: {file_path}",
            ) from e

    def save_articles(self, file_path: str, articles: list[dict]) -> None:
        try:
            with open(file_path, "w") as f:
                json.dump(articles, f, indent=2)
        except OSError as e:
            raise AppError(
                message=f"保存文章数据失败: {file_path}",
                code="IO_ERROR",
                status_code=500,
            ) from e
```

`raise X from e` 把 `e` 设为 `X` 的 `__cause__`。Python 的 traceback 输出会自动显示完整的异常链：

```
Traceback (most recent call last):
  File "example.py", line 3, in load_articles
    return json.load(f)
           ^^^^^^^^^^^
json.decoder.JSONDecodeError: Expecting value: line 1 column 1 (char 0)

The above exception was the direct cause of the following exception:

Traceback (most recent call last):
  File "example.py", line 10, in <module>
    articles = repo.load_articles("broken.json")
                ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
techblog.core.exceptions.ValidationError: 文章数据文件格式错误: broken.json
```

"direct cause of the following exception"——Python 的 traceback 自动帮你展示了异常链。Java 的异常链也需要在日志中手动打印 cause，但 IDE 的异常断点通常能直接跳到 cause。

### raise from None：显式断开异常链

有时候你不希望把内部异常暴露给调用者。比如数据库连接失败时，你不想让 API 响应里出现数据库地址和凭据：

```python
def get_api_data() -> dict:
    try:
        result = internal_http_call()
    except ConnectionError as e:
        # from None — 不保留原始异常
        raise AppError(
            message="服务暂时不可用，请稍后重试",
            code="SERVICE_UNAVAILABLE",
            status_code=503,
        ) from None
```

`raise from None` 设置 `__cause__` 为 `None`，traceback 中不会显示原始异常。Java 没有直接等价的语法——你要么传 cause（`new Exception(msg, cause)`），要么不传（`new Exception(msg)`），但即使不传 cause，Java 的堆栈跟踪也会显示 `Caused by: ...` 如果异常是在 catch 块中抛出的。

## 不要吞掉异常

Python 比 Java 更容易"吞掉"异常——因为没有任何机制强制你处理。Java 的 checked exception 至少逼你写一个 try-catch，即使你只是 `throw new RuntimeException(e)`，也比什么都不做强。

Python 中吞掉异常的常见方式：

```python
# 最常见 — except 里什么都不做
try:
    risky_operation()
except Exception:
    pass  # 异常被吞掉了，出了问题你完全不知道

# 稍好一点 — 打印了但没处理
try:
    risky_operation()
except Exception as e:
    print(f"出错了: {e}")  # 用 print 而不是 logger

# 正确做法 — 记录日志并重新抛出
try:
    risky_operation()
except Exception as e:
    logger.error("操作失败: %s", e, exc_info=True)
    raise  # 重新抛出，让上层处理
```

Java 项目里最常见的问题不是吞掉异常，而是过度包装——每个 catch 块都 `throw new BusinessException("xxx", e)`，最终你看到的是 5 层嵌套的异常链，每层都只是换了个消息。Python 项目更倾向于直接让异常传播，在顶层统一处理。这两种方式各有优劣：Java 的方式让每一层的错误语义更清晰，但嵌套太深；Python 的方式更简洁，但中间层丢失了错误上下文。

最佳实践是在**业务边界**处理异常——Service 层捕获底层异常（数据库、网络），包装成业务异常抛出；Controller/Handler 层统一捕获业务异常，返回标准化错误响应。中间的工具函数、数据转换函数不应该捕获异常，让它们自然传播。

## Trade-off：没有 checked exception 的自由与责任

Python 的 unchecked exception 设计赋予了开发者更大的自由度——你不需要为了调用一个可能抛 `IOException` 的函数而写 try-catch，代码更简洁。在写脚本、数据处理、快速原型时，这种自由度非常宝贵：`open()` 抛 `FileNotFoundError`？让它崩，反正是一次性脚本。

但在大型项目中，没有 checked exception 意味着你必须靠**团队纪律**来保证异常处理的质量。Java 的 checked exception 是"编译器帮你兜底"，Python 没有这个兜底——如果你忘了处理异常，运行时才会发现。这是 Python 和 Java 在错误处理上最根本的 trade-off：**自由 vs 安全**。

Python 应对这个问题的工具是：

- **自定义异常体系**：统一的 `AppError` 层次，让异常有结构化的错误码和状态码
- **框架级拦截**：FastAPI 的 `@app.exception_handler` 统一处理异常，确保错误响应格式一致
- **mypy 的 `--strict` 模式**：不能直接帮你检查异常，但能确保类型正确，减少运行时 `TypeError`
- **测试覆盖**：用 pytest 的 `pytest.raises` 测试异常场景，确保关键路径的错误处理被测试覆盖

这些工具加在一起，虽然没有 Java 编译器那样的强制力，但能在 CI 层面保证异常处理的基本质量。关键是：你的团队需要有一个共识——哪些异常必须处理，哪些可以让它传播，哪些需要包装。
