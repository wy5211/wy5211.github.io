CMS 的文章 API 已经有了 CRUD、校验、异常处理、JWT 认证、Redis 缓存——功能越堆越多。每次改代码都在祈祷"别把别的地方搞坏"。这个祈祷可以自动化，方式就是测试。

如果你写过 Node.js 项目，大概率用过 Jest。`describe`、`it`、`expect`、`jest.mock()`，一套组合拳打完，测试就写好了。Jest 的哲学是"零配置"——安装完就能用，mock 内置，断言库内置，覆盖率报告内置。Spring Boot Test 的哲学不同。它不做"零配置"，而是给你一个分层体系：从单元测试到切片测试到集成测试，每一层有对应的注解和工具。理解这个分层，是从 Jest 转到 Spring Boot Test 的关键。

## Jest 和 Spring Boot Test 的核心差异

先看一张对比表，建立整体印象：

| 维度       | Jest                | Spring Boot Test                             |
| ---------- | ------------------- | -------------------------------------------- |
| 运行器     | Jest（内置）        | JUnit 5（需引入）                            |
| 断言库     | `expect()` 内置     | AssertJ / Mockito                            |
| Mock       | `jest.mock()` 内置  | `@MockBean` / Mockito                        |
| HTTP 测试  | supertest           | `@WebMvcTest` + `MockMvc`                    |
| 集成测试   | 自己组装 Express    | `@SpringBootTest` 启动完整容器               |
| 数据库测试 | 内存 SQLite 或 mock | H2 内存数据库 + `@DataJpaTest`               |
| 测试分层   | 没有明确分层        | 单元 / 切片 / 集成 三层体系                  |
| 配置方式   | jest.config.js      | `@TestPropertySource` / application-test.yml |

最大的区别在于**分层**。Jest 不区分测试类型——所有测试都是 `describe` + `it`，全凭开发者自觉。Spring Boot Test 通过注解强制分层，每一层只加载需要的组件，启动速度和测试隔离性都更有保障。

## 依赖引入

Spring Boot Test 不需要单独引入 starter，`spring-boot-starter-test` 包含了所有测试工具：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>
```

这个依赖拉进来的是：JUnit 5、AssertJ、Mockito、MockMvc、JsonPath、TestRestTemplate、Hamcrest。等价的 Node.js 项目里，你需要单独装 `jest`、`supertest`、`faker`，有时还要配 ts-jest。

## 单元测试：Mockito

单元测试只测一个类，不启动 Spring 容器。在 Node.js 里，单元测试就是 `jest.mock()` 模块，然后测试函数逻辑。Spring Boot Test 里用 Mockito：

```java
class PostServiceTest {

    private PostRepository postRepository = mock(PostRepository.class);
    private PostService postService = new PostService(postRepository);

    @Test
    void shouldCreatePost() {
        // given
        PostCreateRequest request = new PostCreateRequest("测试标题", "测试内容");
        Post saved = new Post(1L, "测试标题", "测试内容", PostStatus.DRAFT);
        when(postRepository.save(any(Post.class))).thenReturn(saved);

        // when
        PostResponse response = postService.createPost(request, 1L);

        // then
        assertThat(response.getTitle()).isEqualTo("测试标题");
        verify(postRepository).save(any(Post.class));
    }

    @Test
    void shouldThrowWhenTitleEmpty() {
        PostCreateRequest request = new PostCreateRequest("", "内容");
        assertThatThrownBy(() -> postService.createPost(request, 1L))
            .isInstanceOf(IllegalArgumentException.class)
            .hasMessage("标题不能为空");
    }
}
```

等价的 Jest 写法：

```js
describe("PostService", () => {
  const postRepository = { create: jest.fn() };
  const postService = new PostService(postRepository);

  it("should create post", () => {
    postRepository.create.mockResolvedValue({ id: 1, title: "测试标题" });
    const result = await postService.createPost({ title: "测试标题", content: "内容" }, 1);
    expect(result.title).toBe("测试标题");
    expect(postRepository.create).toHaveBeenCalled();
  });
});
```

核心区别：Jest 的 `jest.fn()` 是全局 API，Mockito 的 `mock()` 是静态导入。功能上等价，但 Mockito 的 `verify()` 比 Jest 的 `toHaveBeenCalled()` 表达力更强——`verify(repository, times(2)).save(any())` 可以精确断言调用次数和参数。

## 切片测试：@WebMvcTest

切片测试是 Spring Boot Test 的特色概念。它只启动应用的一部分——比如只启动 MVC 层（Controller + Filter + 拦截器），不启动数据库连接、不启动 Redis、不启动 Service 层的真实实现。Service 层用 `@MockBean` 替换。

这和 Jest + supertest 的组合类似：supertest 只启动 Express 应用，数据库用 mock。区别在于 Spring Boot Test 的切片是框架自动控制的，你只需要加注解。

```java
@WebMvcTest(PostController.class)
class PostControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private PostService postService;

    @MockBean
    private JwtUtil jwtUtil;

    @Test
    void shouldReturnPostList() throws Exception {
        // mock Service 返回值
        when(postService.listPosts(any(), any())).thenReturn(
            new PageImpl<>(List.of(new PostResponse(1L, "标题", "摘要", PostStatus.PUBLISHED)))
        );

        // 模拟 JWT 认证通过
        when(jwtUtil.validateToken(any())).thenReturn(true);
        when(jwtUtil.getUsername(any())).thenReturn("admin");

        mockMvc.perform(get("/api/posts")
                .header("Authorization", "Bearer fake-token")
                .param("page", "0")
                .param("size", "10"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.content[0].title").value("标题"));
    }

    @Test
    void shouldReturn401WithoutToken() throws Exception {
        mockMvc.perform(get("/api/posts"))
            .andExpect(status().isUnauthorized());
    }

    @Test
    void shouldReturn400WhenTitleEmpty() throws Exception {
        String json = """
            {"title": "", "content": "内容"}
            """;

        mockMvc.perform(post("/api/posts")
                .contentType(MediaType.APPLICATION_JSON)
                .content(json)
                .header("Authorization", "Bearer fake-token"))
            .andExpect(status().isBadRequest());
    }
}
```

`@WebMvcTest(PostController.class)` 做了三件事：只实例化 `PostController` 及其依赖的 MVC 组件（拦截器、参数解析器等），用 `@MockBean` 替换 Controller 依赖的 Service，自动配置 `MockMvc`。等价的 Jest + supertest 写法：

```js
describe("POST /api/posts", () => {
  it("should return 400 when title empty", async () => {
    const res = await request(app)
      .post("/api/posts")
      .set("Authorization", "Bearer " + token)
      .send({ title: "", content: "内容" });
    expect(res.status).toBe(400);
  });
});
```

两者在测试表达上非常接近。`mockMvc.perform(...).andExpect(...)` 的链式调用和 supertest 的 `.expect(...)` 风格类似。但 `@WebMvcTest` 的优势在于**框架帮你隔离了数据库和其他基础设施**——你不需要手动 mock Sequelize/TypeORM 的连接，框架直接不启动它们。

## 集成测试：@SpringBootTest

集成测试启动完整的 Spring 容器，包括数据库、Redis、Security 等所有组件。这是最接近真实运行环境的测试。测试数据库用 H2 内存数据库替代 MySQL，避免依赖外部环境。

```java
@SpringBootTest
@AutoConfigureMockMvc
@TestPropertySource(properties = {
    "spring.datasource.url=jdbc:h2:mem:testdb",
    "spring.datasource.driver-class-name=org.h2.Driver",
    "spring.jpa.hibernate.ddl-auto=create-drop",
    "spring.data.redis.host=localhost"
})
class PostApiIntegrationTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private PostRepository postRepository;

    @Autowired
    private UserRepository userRepository;

    @Autowired
    private PasswordEncoder passwordEncoder;

    @Autowired
    private JwtUtil jwtUtil;

    private String adminToken;

    @BeforeEach
    void setUp() {
        postRepository.deleteAll();
        userRepository.deleteAll();

        // 创建测试用户
        User admin = new User();
        admin.setUsername("admin");
        admin.setPasswordHash(passwordEncoder.encode("password"));
        admin.setRole(UserRole.ADMIN);
        userRepository.save(admin);

        adminToken = jwtUtil.generateToken("admin");
    }

    @Test
    void fullCrudFlow() throws Exception {
        // 创建文章
        String createJson = """
            {"title": "集成测试文章", "content": "这是内容", "status": "DRAFT"}
            """;
        mockMvc.perform(post("/api/posts")
                .header("Authorization", "Bearer " + adminToken)
                .contentType(MediaType.APPLICATION_JSON)
                .content(createJson))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.title").value("集成测试文章"));

        // 查询文章列表
        mockMvc.perform(get("/api/posts")
                .header("Authorization", "Bearer " + adminToken))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.content").isArray())
            .andExpect(jsonPath("$.content.length()").value(1));

        // 查询单篇文章
        mockMvc.perform(get("/api/posts/1")
                .header("Authorization", "Bearer " + adminToken))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.title").value("集成测试文章"));

        // 删除文章
        mockMvc.perform(delete("/api/posts/1")
                .header("Authorization", "Bearer " + adminToken))
            .andExpect(status().isNoContent());
    }
}
```

`@TestPropertySource` 覆盖了 application.yml 的配置，把数据库换成 H2。这是 Spring Boot Test 的标准做法——测试环境和生产环境用相同的代码、不同的配置。在 Node.js 里你通常用 `dotenv.config({ path: '.env.test' })` 或者在 Jest 的 setup 文件里设置环境变量。

## TestRestTemplate：另一种 HTTP 测试方式

`MockMvc` 在测试容器内发送请求，不经过真实的 HTTP 协议栈。如果你需要测试真实的 HTTP 通信（包括序列化、过滤器链、端口绑定），可以用 `TestRestTemplate`：

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class PostApiRestTemplateTest {

    @Autowired
    private TestRestTemplate restTemplate;

    @Autowired
    private JwtUtil jwtUtil;

    @Test
    void shouldReturnPosts() {
        HttpHeaders headers = new HttpHeaders();
        headers.setBearerAuth(jwtUtil.generateToken("admin"));

        ResponseEntity<PostPageResponse> response = restTemplate.exchange(
            "/api/posts?page=0&size=10",
            HttpMethod.GET,
            new HttpEntity<>(headers),
            PostPageResponse.class
        );

        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.OK);
        assertThat(response.getBody()).isNotNull();
    }
}
```

`TestRestTemplate` 和 Node.js 里用 `fetch()` 或 `axios` 调用自己的 API 做端到端测试是同一回事。`MockMvc` 更快但跳过了 HTTP 协议层，`TestRestTemplate` 更慢但更接近真实。大部分场景用 `MockMvc` 就够了。

## 测试分层策略

Spring Boot Test 的三层体系不是教条，而是工程实践中的取舍：

| 测试类型 | 注解              | 启动范围   | 适用场景              | 速度     |
| -------- | ----------------- | ---------- | --------------------- | -------- |
| 单元测试 | 无（纯 Mockito）  | 不启动容器 | Service 逻辑、工具类  | 毫秒级   |
| 切片测试 | `@WebMvcTest`     | MVC 层     | Controller 路由和校验 | 百毫秒级 |
| 切片测试 | `@DataJpaTest`    | JPA 层     | Repository 查询逻辑   | 百毫秒级 |
| 集成测试 | `@SpringBootTest` | 完整容器   | API 端到端流程        | 秒级     |

在 Node.js 项目里，你通常不会做这样的区分——大部分测试要么是 mock 所有依赖的单元测试，要么是 supertest + 内存数据库的集成测试。Spring Boot Test 的切片测试填补了中间地带：它比集成测试快（不启动数据库），但比纯 mock 测试覆盖面广（测试了 Controller 的路由绑定、参数校验、序列化）。

**实际建议：**

- Service 层的纯逻辑（不依赖数据库的）写单元测试，用 Mockito mock Repository
- Controller 的路由、参数校验、权限注解写 `@WebMvcTest` 切片测试
- 关键 API 的完整流程（从请求到数据库）写 `@SpringBootTest` 集成测试
- 不需要追求覆盖率——核心业务路径的集成测试比 90% 覆盖率的单元测试更有价值

## 完整对比

| 维度       | Jest + supertest             | Spring Boot Test                                  |
| ---------- | ---------------------------- | ------------------------------------------------- |
| 上手难度   | 低，零配置                   | 中，需要理解注解体系                              |
| Mock 机制  | `jest.mock()` 自动 mock 模块 | `@MockBean` 替换 Spring 容器中的 Bean             |
| 测试隔离   | 靠 `beforeEach` 清理状态     | 框架自动管理（`@DirtiesContext`、事务回滚）       |
| 数据库测试 | 手动管理内存数据库连接       | `@DataJpaTest` 自动配置 H2 + 事务回滚             |
| HTTP 测试  | supertest（真实 HTTP）       | MockMvc（容器内）或 TestRestTemplate（真实 HTTP） |
| 测试速度   | 快，不需要启动容器           | 切片测试快，集成测试慢                            |
| 测试分层   | 无强制分层                   | 注解强制分层，框架保障隔离                        |

## Trade-off 分析

**Spring Boot Test 的收益：**

- 测试分层有框架保障。`@WebMvcTest` 自动只加载 MVC 层，你不需要手动控制哪些模块启动、哪些 mock。在 Node.js 项目里，测试之间的隔离全靠开发者自觉
- `@MockBean` 和 Spring 容器的深度集成。它能替换容器中的任意 Bean，而不只是 mock 模块导出。这意味着你可以精确控制依赖注入的每个环节
- 事务回滚机制。`@DataJpaTest` 默认每个测试方法在事务中运行，测试结束后自动回滚，数据库状态完全干净。Jest 没有等价机制，你需要手动 `beforeEach` 清表

**代价：**

- 注解体系复杂。`@SpringBootTest`、`@WebMvcTest`、`@DataJpaTest`、`@MockBean`、`@TestPropertySource`、`@DirtiesContext`、`@AutoConfigureMockMvc`——光记这些注解就需要时间
- 启动慢。`@SpringBootTest` 启动完整容器需要几秒到十几秒，测试越多越慢。Jest 的测试启动几乎是瞬时的
- Mock 和 Spring 容器的耦合。`@MockBean` 会修改 Spring 容器的 Bean 定义，如果被多个测试类使用，可能导致上下文缓存失效，进一步拖慢速度

## 回顾与预告

到这篇为止，CMS 的质量保障体系已经搭好：

- 理解了 Spring Boot Test 的三层测试体系——单元测试用 Mockito，切片测试用 `@WebMvcTest`，集成测试用 `@SpringBootTest`
- `@MockBean` 替换容器中的 Bean 实现依赖隔离，等价于 Jest 的 `jest.mock()` 但更精确
- `MockMvc` 在容器内测试 HTTP 层，`TestRestTemplate` 走真实 HTTP 协议栈，各有适用场景
- `@TestPropertySource` 让测试环境用 H2 替代 MySQL，不需要外部依赖

CMS 现在有了完整的功能和测试。下一步是让它跑在容器里，而不是开发者的笔记本上。
