目前创建文章时没有输入校验，title 可以是空字符串、content 可以是 null。用户发一个 `{"title": "", "content": null}` 过来，JPA 照存不误，数据库里就多了一条垃圾数据。更糟的是，如果 slug 重复了，数据库会抛唯一约束异常，但你拿到的只有一堆 Hibernate SQL 错误日志，根本看不出是哪个字段出了问题。

在 Node.js 里你可能习惯用 zod 或 Joi 做校验，在路由处理函数的最开头调一次 `schema.parse(request.body)`，不合法直接抛错。Spring Boot 里做同样的事情，但思路完全不同——它用的是注解（Annotation）驱动的声明式校验，校验规则写在 DTO 类的字段上，由框架在请求进入 Controller 时自动执行。

## 从手写 if-else 到声明式校验

先看没有校验时，你可能在 Service 里写的东西：

```java
public Post create(PostCreateRequest request) {
    if (request.getTitle() == null || request.getTitle().isBlank()) {
        throw new IllegalArgumentException("title 不能为空");
    }
    if (request.getTitle().length() > 200) {
        throw new IllegalArgumentException("title 不能超过 200 个字符");
    }
    if (request.getContent() == null) {
        throw new IllegalArgumentException("content 不能为 null");
    }
    // ... 每加一个字段就多几行 if
    return postRepository.save(/* ... */);
}
```

每加一个校验规则就多一个 if 分支，create 和 update 方法里还要重复写一遍。这在 Node.js + Express 里也一样——如果你不用任何校验库，就是一堆 `if (!req.body.title)` 的判断。

zod 的做法是把校验规则抽成一个独立的 schema 对象：

```typescript
const postSchema = z.object({
  title: z.string().min(1).max(200),
  slug: z.string().regex(/^[a-z0-9-]+$/),
  content: z.string().min(1),
});
```

Spring Boot 的做法是把校验规则写在 DTO 字段上：

```java
public class PostCreateRequest {

    @NotBlank(message = "title 不能为空")
    @Size(max = 200, message = "title 不能超过 200 个字符")
    private String title;

    @Pattern(regexp = "^[a-z0-9-]+$", message = "slug 只允许小写字母、数字和连字符")
    private String slug;

    @NotBlank(message = "content 不能为空")
    private String content;
}
```

两种方式表达的信息完全一样，只是"规则放哪里"这个问题上的选择不同。zod 把规则集中在一个 schema 对象里，Spring Validation 把规则分散在各个字段上。

## @Valid 触发校验

光在 DTO 上加注解不会自动生效。你需要在 Controller 方法参数前加 `@Valid`，告诉 Spring："在绑定请求体到这个对象之前，先跑一遍校验"。

```java
@PostMapping
public ResponseEntity<Post> create(@Valid @RequestBody PostCreateRequest request) {
    Post post = postService.create(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(post);
}
```

校验失败时，Spring 会抛出 `MethodArgumentNotValidException`。如果不处理这个异常，Spring Boot 默认返回 400 状态码和一段不太友好的错误信息。如何把它变成结构化的 JSON 响应，是下一篇文章全局异常处理的内容——这里先了解一个简单的处理方式。

校验流程是这样的：

```
HTTP 请求
  │
  ▼
DispatcherServlet
  │
  ▼
@Valid 触发 Hibernate Validator
  │
  ├─ 校验通过 → 进入 Controller 方法体
  │
  └─ 校验失败 → 抛出 MethodArgumentNotValidException
                    │
                    ▼
               @ExceptionHandler 处理（第 07 篇讲）
```

## 依赖配置

Spring Boot 3.x 已经把 `spring-boot-starter-validation` 作为 `spring-boot-starter-web` 的传递依赖包含了。如果你的项目里 `@Valid` 找不到，手动加一下：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

底层实现是 Hibernate Validator，它是 Jakarta Bean Validation（原名 JSR-380）规范的参考实现。`@NotBlank`、`@Size` 这些注解都来自 `jakarta.validation.constraints` 包，不是 Spring 自己定义的。

## 常用校验注解速查

不要背，用的时候查就行。这里按用途分组，带 Node.js 对比：

### 空值判断

| 注解        | 适用类型                    | 行为                            | zod 等价写法                             |
| ----------- | --------------------------- | ------------------------------- | ---------------------------------------- |
| `@NotNull`  | 任意类型                    | 不能为 null，空字符串 OK        | `z.string().nullable()` 取反             |
| `@NotBlank` | String                      | 不能为 null，且 trim 后长度 > 0 | `z.string().min(1)`                      |
| `@NotEmpty` | String/Collection/Map/Array | 不能为 null，且 size > 0        | `z.string().min(1)` / `z.array().min(1)` |

`@NotBlank` 是字符串场景最常用的。很多人分不清 `@NotNull` 和 `@NotBlank` 的区别——`@NotNull` 允许空字符串 `""`，`@NotBlank` 不允许。

### 字符串

| 注解               | 行为     | zod 等价                   |
| ------------------ | -------- | -------------------------- |
| `@Size(min, max)`  | 长度范围 | `z.string().min(n).max(m)` |
| `@Email`           | 邮箱格式 | `z.string().email()`       |
| `@Pattern(regexp)` | 正则匹配 | `z.string().regex(/.../)`  |
| `@URL`             | URL 格式 | `z.string().url()`         |

### 数值

| 注解              | 行为           | zod 等价                |
| ----------------- | -------------- | ----------------------- |
| `@Min(value)`     | 最小值         | `z.number().min(n)`     |
| `@Max(value)`     | 最大值         | `z.number().max(n)`     |
| `@Positive`       | 必须为正数     | `z.number().positive()` |
| `@DecimalMin/Max` | 支持小数的范围 | `z.number().min(n)`     |

## 给 CMS 加上校验

回到 CMS 项目。`PostCreateRequest` 加上校验注解：

```java
public class PostCreateRequest {

    @NotBlank(message = "标题不能为空")
    @Size(max = 200, message = "标题不能超过 200 个字符")
    private String title;

    @Pattern(regexp = "^[a-z0-9][a-z0-9-]*[a-z0-9]$",
             message = "slug 格式不合法，只允许小写字母、数字和连字符")
    private String slug;

    @NotBlank(message = "内容不能为空")
    private String content;

    @Size(max = 10, message = "标签数量不能超过 10 个")
    private List<@NotBlank(message = "标签名不能为空") String> tags;
}
```

`List<@NotBlank String>` 这种写法是嵌套校验——泛型参数上的注解默认不会生效，需要在列表字段上配合 `@Valid` 才行。但在 `@RequestBody` 场景下，Spring Boot 对集合元素上的约束注解也能处理，所以这里可以直接写。如果遇到不生效的情况，加一个 `@Valid`：

```java
@Valid
@Size(max = 10, message = "标签数量不能超过 10 个")
private List<@NotBlank(message = "标签名不能为空") String> tags;
```

Controller 不需要改，`@Valid` 之前就加上了。现在发一个空请求试试：

```bash
curl -X POST http://localhost:8080/api/posts \
  -H "Content-Type: application/json" \
  -d '{"title": "", "content": null}'
```

校验失败，抛出 `MethodArgumentNotValidException`。Spring Boot 默认的响应体大致是：

```json
{
  "status": 400,
  "error": "Bad Request",
  "errors": [
    {
      "field": "title",
      "message": "标题不能为空"
    },
    {
      "field": "content",
      "message": "内容不能为空"
    }
  ]
}
```

格式不够统一，下一篇全局异常处理会把它变成和 CMS 其他接口一致的 `{ code, message, data }` 结构。

## 自定义校验：@UniqueSlug

内置注解能处理格式和长度，但有一个业务规则它处理不了：slug 必须全局唯一。这需要查数据库，内置注解做不到。

在 Node.js + zod 里，你会用 `.refine()` 加一个异步函数：

```typescript
const postSchema = z.object({
  slug: z.string().regex(/^[a-z0-9-]+$/).refine(
    async (slug) => !(await postRepo.existsBySlug(slug)),
    { message: "slug 已被使用" }
  );
```

Spring Boot 里对应的概念是自定义校验注解 + 校验器。需要写两个东西：一个注解定义，一个 `ConstraintValidator` 实现。

**第一步，定义注解：**

```java
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = UniqueSlugValidator.class)
public @interface UniqueSlug {
    String message() default "slug 已被使用";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}
```

`groups` 和 `payload` 是 Bean Validation 规范要求的固定属性，所有自定义注解都必须有。`@Constraint(validatedBy = ...)` 指定了校验逻辑的实现类。

**第二步，实现校验器：**

```java
@Component
public class UniqueSlugValidator implements ConstraintValidator<UniqueSlug, String> {

    @Autowired
    private PostRepository postRepository;

    @Override
    public boolean isValid(String slug, ConstraintValidatorContext context) {
        if (slug == null || slug.isBlank()) {
            return true; // 空值交给 @NotBlank 处理，这里放行
        }
        return !postRepository.existsBySlug(slug);
    }
}
```

`ConstraintValidator<UniqueSlug, String>` 的两个泛型分别是注解类型和被校验字段的类型。因为加了 `@Component`，Spring 会管理这个校验器的生命周期，`postRepository` 可以正常注入。

**第三步，在 DTO 上使用：**

```java
@UniqueSlug
@Pattern(regexp = "^[a-z0-9][a-z0-9-]*[a-z0-9]$")
private String slug;
```

校验顺序：先跑 `@NotBlank`（如果有的话），再跑 `@Pattern`，最后跑 `@UniqueSlug`。格式校验在前，数据库查询在后，避免对格式都不合法的值去查数据库。

有一个细节值得注意：`ConstraintValidator` 默认不是 Spring Bean。上面用了 `@Component` 让它成为 Bean，这样可以通过 `@Autowired` 注入 `PostRepository`。如果你不加 `@Component`，就需要通过 `ConstraintValidatorContext` 或者 Spring 的 `LocalValidatorFactoryBean` 来间接获取 Bean——麻烦得多。Spring Boot 的自动配置已经处理好了这个集成，直接加 `@Component` 就行。

## 分组校验：创建和更新的规则不同

一个常见的需求：创建文章时 title 必填，更新文章时 title 可选（部分更新）。zod 用 `.partial()` 解决这个问题，Spring Validation 用分组（Group）。

定义两个空接口作为分组标记：

```java
public interface Create {}
public interface Update {}
```

在 DTO 上用 `groups` 指定规则属于哪个分组：

```java
public class PostRequest {

    @Null(groups = Create.class, message = "创建时不能指定 ID")
    @NotNull(groups = Update.class, message = "更新时必须指定 ID")
    private Long id;

    @NotBlank(groups = Create.class, message = "创建时标题不能为空")
    @Size(max = 200)
    private String title;

    @NotBlank(groups = Create.class, message = "创建时内容不能为空")
    private String content;
}
```

Controller 里用 `@Validated(Create.class)` 指定激活哪个分组：

```java
@PostMapping
public ResponseEntity<Post> create(
        @Validated(Create.class) @RequestBody PostRequest request) {
    // ...
}

@PutMapping("/{id}")
public ResponseEntity<Post> update(
        @Validated(Update.class) @RequestBody PostRequest request) {
    // ...
}
```

注意这里用的是 `@Validated` 而不是 `@Valid`。`@Valid` 是 Jakarta 标准注解，不支持分组；`@Validated` 是 Spring 提供的增强版，支持分组校验。这是很多人踩过的坑——加了 groups 但校验不生效，就是因为用了 `@Valid`。

## 响应格式化：处理校验错误

前面说过，校验失败抛出的 `MethodArgumentNotValidException` 需要处理。写一个简单的 `@ExceptionHandler` 让错误信息更友好（完整方案在下一篇）：

```java
@RestControllerAdvice
public class ValidationExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new LinkedHashMap<>();
        ex.getBindingResult().getFieldErrors().forEach(error ->
            errors.put(error.getField(), error.getDefaultMessage())
        );
        Map<String, Object> body = new LinkedHashMap<>();
        body.put("code", 400);
        body.put("message", "参数校验失败");
        body.put("errors", errors);
        return ResponseEntity.badRequest().body(body);
    }
}
```

错误响应变成这样：

```json
{
  "code": 400,
  "message": "参数校验失败",
  "errors": {
    "title": "标题不能为空",
    "slug": "slug 格式不合法，只允许小写字母、数字和连字符"
  }
}
```

`errors` 是一个 field -> message 的映射，前端可以直接拿字段名定位到表单的对应输入框，比一段纯文本错误信息实用得多。

## zod vs @Valid + Hibernate Validator

把两种方案的完整对比拉出来：

| 维度             | zod (Node.js)                             | @Valid + Hibernate Validator (Spring Boot) |
| ---------------- | ----------------------------------------- | ------------------------------------------ |
| **校验规则定义** | 独立 schema 对象                          | 注解写在 DTO 字段上                        |
| **类型推断**     | `z.infer<typeof schema>` 自动推导 TS 类型 | 无，Java 字段本身就有类型                  |
| **校验触发**     | 手动调用 `schema.parse()` 或中间件        | `@Valid` / `@Validated` 自动触发           |
| **自定义校验**   | `.refine()` / `.superRefine()`            | `ConstraintValidator` 接口                 |
| **数据库校验**   | `.refine(async fn)`                       | `ConstraintValidator` + 注入 Repository    |
| **分组校验**     | `.pick()` / `.omit()` / `.partial()`      | `groups` + `@Validated`                    |
| **错误格式**     | `issue.path` + `issue.message`            | `FieldError.getField()` + `getMessage()`   |
| **嵌套对象**     | `z.object({ address: addressSchema })`    | `@Valid` 注解在嵌套字段上                  |
| **异步校验**     | 原生支持 `async`                          | 需要额外配置，不常用                       |
| **运行时开销**   | 每次调用 `parse()` 都执行校验逻辑         | 注解在类加载时解析，校验时反射调用         |
| **可测试性**     | schema 是纯函数，直接单元测试             | 需要 `Validator` 实例或 Mock               |
| **生态**         | zod-to-openapi、zod-i18n 等               | 与 JPA、Spring Security 深度集成           |

两种方案没有绝对的好坏。zod 的优势在于 schema 即类型、函数式风格、异步支持好。Spring Validation 的优势在于和 Spring 生态无缝集成——`@Valid` 放在 Controller 参数上就完了，不用额外写中间件；自定义校验器能直接注入 Spring Bean 访问数据库。

## Trade-off 分析

**Spring Validation 的收益：**

- 校验逻辑和 DTO 绑定，不会出现"校验规则写在 A 文件，DTO 在 B 文件"的分离问题
- 与 Spring MVC 集成紧密，`@Valid` 一加就生效
- 自定义校验器可以注入任何 Spring Bean，做数据库查询、缓存检查都很方便
- 规范是 Jakarta 标准，换实现（比如从 Hibernate Validator 换成 Apache BVal）不改代码

**代价：**

- 注解太多时 DTO 类变得密集，阅读体验不如 zod 的 schema 那样干净
- 自定义校验需要写注解 + 校验器两个类，比 zod 的 `.refine()` 麻烦
- 反射调用有性能开销（生产环境中通常可以忽略，但极端高并发场景需要注意）
- `@NotNull` 和 `@NotBlank` 这种细微差别容易搞混，新人上手成本高
- 分组校验的 `groups` 机制不如 zod 的 `.partial()` 直观

**实际选择建议：** 如果项目已经在用 Spring Boot，没有理由不用 `@Valid`——它是免费的，和框架集成度最高。如果你是写一个独立的校验库或者非 Spring 项目的校验层，zod 的函数式风格可能更灵活。

## 回顾与预告

到这篇为止，CMS 已经有了：

- Post/Tag/User 实体 + Repository（JPA 数据访问）
- RESTful API（CRUD 完整）
- 参数校验（`@Valid` + Hibernate Validator + 自定义 `@UniqueSlug`）

但有一个问题一直没解决：校验失败的错误响应格式是临时写的，其他类型的异常（资源不存在、数据库错误）还没有统一处理。目前如果用户访问一个不存在的文章 ID，你需要在每个 Controller 方法里手动 `findById().orElseThrow()`，而且抛出来的异常格式和校验错误格式不一致。
