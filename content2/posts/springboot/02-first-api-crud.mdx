如果你用 Express 写过一个 REST API，大概是这样的：`router.get('/posts/:id', handler)`，然后在 handler 里用 `req.params.id` 拿参数、`req.body` 拿请求体、`res.json()` 返回响应。整个路由定义是一个个函数调用，路径和处理器之间的绑定关系散落在多个文件里。

Spring Boot 的做法完全不同。你不需要调用任何函数来"注册"路由——只需要在类和方法上放注解（Annotation），框架在启动时自动扫描并完成绑定。这种"声明式"风格是 Spring MVC 最核心的设计选择，也是从 Express 转过来最先需要适应的地方。

上一篇我们搭好了 Spring Boot 项目，理解了 IoC 和自动配置。今天来写真正的 API——用 Controller 实现 CMS 的文章 CRUD。

## 从 Express Router 到 @RestController

先看一个最直接的路由对比：

```js
// Express
const router = express.Router();
router.get("/api/posts/:id", (req, res) => {
  const id = req.params.id;
  res.json({ id });
});
```

```java
// Spring Boot
@RestController
@RequestMapping("/api/posts")
public class PostController {

    @GetMapping("/{id}")
    public Post getPost(@PathVariable Long id) {
        return postService.findById(id);
    }
}
```

几个关键区别：

- **`@RestController`** 等价于给每个方法都加了 `@ResponseBody`，告诉框架"返回值直接序列化成 JSON 写入响应体"。Express 没有这个概念，因为 Express 的 `res.json()` 已经显式表达了同样的意图。
- **`@RequestMapping("/api/posts")`** 是类级别的路径前缀。Express 也可以通过 `app.use('/api/posts', router)` 做类似的事，但注解的方式把路径和处理器绑定在同一个类里，更容易定位。
- **`@GetMapping("/{id}")`** 是 `@RequestMapping(method = RequestMethod.GET, value = "/{id}")` 的简写。Spring 提供了 `@GetMapping`、`@PostMapping`、`@PutMapping`、`@DeleteMapping` 四个组合注解，对应 Express 的 `router.get/post/put/delete`。

完整的对照关系：

| 概念      | Express                             | Spring Boot                                           |
| --------- | ----------------------------------- | ----------------------------------------------------- |
| 定义路由  | `router.get('/posts/:id', handler)` | `@GetMapping("/posts/{id}")`                          |
| 路径参数  | `req.params.id`                     | `@PathVariable Long id`                               |
| 查询参数  | `req.query.page`                    | `@RequestParam(defaultValue="1") int page`            |
| 请求体    | `req.body`                          | `@RequestBody CreatePostRequest request`              |
| 响应      | `res.status(201).json(data)`        | `ResponseEntity.status(201).body(data)`               |
| HTTP 方法 | `app.get/post/put/delete`           | `@GetMapping/@PostMapping/@PutMapping/@DeleteMapping` |

注意路径参数的占位符：Express 用 `:id`，Spring Boot 用 `{id}`。

## 路径参数与查询参数

路径参数 `@PathVariable` 和查询参数 `@RequestParam` 对应 Express 的 `req.params` 和 `req.query`，但 Spring Boot 多了一层**类型转换**。

在 Express 中，`req.params.id` 永远是字符串，你需要手动转换和校验：

```js
// Express 中你需要自己处理类型转换
const id = parseInt(req.params.id, 10);
if (isNaN(id)) {
  return res.status(400).json({ error: "Invalid id" });
}
```

Spring Boot 的 `@PathVariable Long id` 直接声明参数类型为 `Long`。如果请求传入的不是数字，框架在到达你的方法之前就返回 400 错误。这种"框架帮你做"的校验在早期开发阶段很方便，但要注意：默认的错误响应格式不一定符合你的 API 规范，后续需要通过全局异常处理来统一（第 7 篇会讲）。

查询参数的 `@RequestParam` 支持 `required` 和 `defaultValue`：

```java
// 如果不传 page，默认是 1；如果不传 size，默认是 20
@GetMapping
public List<Post> getPosts(
    @RequestParam(defaultValue = "1") int page,
    @RequestParam(defaultValue = "20") int size
) {
    return postService.findAll(page, size);
}
```

Express 里要实现同样的逻辑，需要写更多代码：

```js
const page = parseInt(req.query.page, 10) || 1;
const size = parseInt(req.query.size, 10) || 20;
```

`@RequestParam` 的 `defaultValue` 还有一个隐含效果：设置了 `defaultValue` 之后，`required` 自动变成 `false`。也就是说，参数可以不传。

## DTO：为什么要多一层 Request 对象

在 Express 中，我们通常直接用 `req.body`：

```js
router.post("/posts", (req, res) => {
  const { title, content, slug } = req.body;
  // 直接用 req.body 里的字段
});
```

Spring Boot 推荐用专门的 DTO（Data Transfer Object）来接收请求体：

```java
public class CreatePostRequest {
    private String title;
    private String content;
    private String slug;

    // getter 和 setter 省略
}
```

为什么不能像 Express 那样直接用？原因有三：

**第一，类型安全。** `req.body` 在 Express 里是 `any`，你拿到的任何字段都是隐式的。而 DTO 明确定义了字段名和类型，Jackson 在反序列化时会自动校验——传了一个不存在的字段会被忽略，类型不匹配会报错。

**第二，读写分离。** 创建文章时需要的字段和返回的字段往往不同。创建时客户端提交 `title`、`content`、`slug`；返回时需要 `id`、`createdAt`、`viewCount`。用不同的 DTO 分别处理入参和出参，能避免不小心把不该暴露的字段（比如内部 ID 生成策略）泄露给客户端。

**第三，校验前置。** 在 DTO 上加 `@NotBlank`、`@Size` 之类的校验注解，框架会在进入业务逻辑之前就拦截非法请求。这在 Express 中通常需要用 Joi 或 Zod 中间件来实现。

对于我们的 CMS 项目，先定义两个 DTO：

```java
// 创建文章的请求
public class CreatePostRequest {
    @NotBlank(message = "标题不能为空")
    @Size(max = 100, message = "标题不能超过 100 个字符")
    private String title;

    @NotBlank(message = "slug 不能为空")
    @Pattern(regexp = "^[a-z0-9-]+$", message = "slug 只能包含小写字母、数字和连字符")
    private String slug;

    private String content;
}

// 更新文章的请求
public class UpdatePostRequest {
    private String title;
    private String slug;
    private String content;
    private PostStatus status;
}
```

注意 `UpdatePostRequest` 的所有字段都是可选的——客户端只传需要修改的字段，这在 RESTful PATCH 语义下是合理的。

## ResponseEntity：比 res.json 更精细的响应控制

Express 的 `res.status(201).json(data)` 用链式调用设置状态码和响应体。Spring Boot 中等价的写法是 `ResponseEntity`：

```java
@PostMapping
public ResponseEntity<Post> createPost(@RequestBody @Valid CreatePostRequest request) {
    Post post = postService.create(request);
    return ResponseEntity
        .status(HttpStatus.CREATED)
        .body(post);
}
```

如果只是返回 200 和一个对象，可以直接返回对象，框架自动包装成 200 响应。但当你需要精确控制状态码和响应头时，`ResponseEntity` 是必须的。

几个常用的响应模式：

```java
// 201 Created —— 资源创建成功
return ResponseEntity.status(HttpStatus.CREATED).body(post);

// 204 No Content —— 删除成功，无响应体
return ResponseEntity.noContent().build();

// 200 with Location header —— 创建成功并返回资源 URI
return ResponseEntity
    .created(URI.create("/api/posts/" + post.getId()))
    .body(post);

// 404 Not Found —— 资源不存在
return ResponseEntity.notFound().build();
```

相比 Express 的 `res.status().json()`，`ResponseEntity` 的优势在于它是方法的返回值而不是对 `res` 对象的副作用调用。这意味着你可以在业务逻辑中构造响应，最后统一返回，而不是在 handler 的不同分支里调用 `res` 方法。

## 完整的 CRUD 实现

现在把所有概念组合起来，实现文章的完整 CRUD。为了专注 Controller 层本身，暂时用内存中的 `List` 模拟数据存储，数据库接入在第 5 篇。

先定义实体和枚举：

```java
public enum PostStatus {
    DRAFT, PUBLISHED, ARCHIVED
}

public class Post {
    private Long id;
    private String title;
    private String slug;
    private String content;
    private PostStatus status;
    private Long authorId;
    private int viewCount;
    private LocalDateTime createdAt;

    // 构造函数、getter、setter 省略
}
```

初始数据：

```java
Post(id=1, title="深入理解 Java 内存模型", slug="java-memory-model",
     status=PUBLISHED, authorId=1, viewCount=1520, createdAt=2026-03-15)
Post(id=2, title="Spring Boot 3 新特性详解", slug="spring-boot-3-features",
     status=DRAFT, authorId=2, viewCount=0, createdAt=2026-03-20)
```

然后是 Controller：

```java
@RestController
@RequestMapping("/api/posts")
public class PostController {

    private final List<Post> posts = new ArrayList<>();
    private long nextId = 3;

    public PostController() {
        posts.add(new Post(1L, "深入理解 Java 内存模型", "java-memory-model",
            "JMM 定义了线程如何与内存交互...", PostStatus.PUBLISHED,
            1L, 1520, LocalDateTime.of(2026, 3, 15, 10, 0)));
        posts.add(new Post(2L, "Spring Boot 3 新特性详解", "spring-boot-3-features",
            "Spring Boot 3 带来了诸多改进...", PostStatus.DRAFT,
            2L, 0, LocalDateTime.of(2026, 3, 20, 14, 30)));
    }

    @GetMapping
    public List<Post> getPosts(
            @RequestParam(defaultValue = "1") int page,
            @RequestParam(defaultValue = "20") int size) {
        int from = Math.max(0, (page - 1) * size);
        int to = Math.min(posts.size(), from + size);
        return posts.subList(from, to);
    }

    @GetMapping("/{id}")
    public ResponseEntity<Post> getPost(@PathVariable Long id) {
        return posts.stream()
            .filter(p -> p.getId().equals(id))
            .findFirst()
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }

    @PostMapping
    public ResponseEntity<Post> createPost(
            @RequestBody @Valid CreatePostRequest request) {
        Post post = new Post(
            nextId++, request.getTitle(), request.getSlug(),
            request.getContent(), PostStatus.DRAFT,
            1L, 0, LocalDateTime.now()
        );
        posts.add(post);
        return ResponseEntity
            .status(HttpStatus.CREATED)
            .body(post);
    }

    @PutMapping("/{id}")
    public ResponseEntity<Post> updatePost(
            @PathVariable Long id,
            @RequestBody @Valid UpdatePostRequest request) {
        return posts.stream()
            .filter(p -> p.getId().equals(id))
            .findFirst()
            .map(existing -> {
                if (request.getTitle() != null) {
                    existing.setTitle(request.getTitle());
                }
                if (request.getContent() != null) {
                    existing.setContent(request.getContent());
                }
                if (request.getStatus() != null) {
                    existing.setStatus(request.getStatus());
                }
                return ResponseEntity.ok(existing);
            })
            .orElse(ResponseEntity.notFound().build());
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deletePost(@PathVariable Long id) {
        boolean removed = posts.removeIf(p -> p.getId().equals(id));
        return removed
            ? ResponseEntity.noContent().build()
            : ResponseEntity.notFound().build();
    }
}
```

逐个看每个端点的设计意图：

**GET /api/posts** — 列表查询，支持分页。`subList` 做了边界保护，`page` 传负数不会越界。生产环境中分页逻辑应该下沉到 Service 层甚至数据库查询层面，这里只是为了演示 Controller 的参数接收。

**GET /api/posts/{id}** — 用 Stream 的 `findFirst` + `map` 替代了 if-else。找不到返回 404，找到返回 200。这种函数式的写法比显式的 `if (post == null) return notFound()` 更简洁，但也更难加日志——实际项目中如果需要在 404 时记录日志，显式 if-else 可能更合适。

**POST /api/posts** — 创建文章时 `status` 默认设为 `DRAFT`，`viewCount` 设为 0。客户端不应该能指定这些字段，所以 `CreatePostRequest` 里没有它们。这是 DTO 做读写分离的实际体现。

**PUT /api/posts/{id}** — 部分更新逻辑通过 `null` 检查实现：客户端只传要改的字段。这里用的是 PUT 语义做部分更新，严格来说应该用 PATCH。但在实际项目中，很多团队并不严格区分 PUT 和 PATCH，因为部分更新远比全量替换常用。

**DELETE /api/posts/{id}** — 返回 204 No Content，没有响应体。`removeIf` 的返回值告诉我们是否真的删除了一条记录，据此返回 204 或 404。

## 一个容易忽略的细节：JSON 序列化

Express 默认用 `JSON.stringify` 序列化响应，而 Spring Boot 默认用 Jackson。两者的行为差异在处理日期时尤其明显。

Jackson 默认把 `LocalDateTime` 序列化成数组：`[2026, 3, 15, 10, 0, 0]`。这通常不是你想要的。在 `application.yml` 里加一行配置就能统一成 ISO 格式：

```yaml
spring:
  jackson:
    date-format: yyyy-MM-dd HH:mm:ss
    time-zone: Asia/Shanghai
```

或者更推荐的方式，在字段上用 `@JsonFormat` 精确控制：

```java
@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
private LocalDateTime createdAt;
```

## Trade-off：注解驱动的代价

Spring Boot 的注解驱动路由在代码组织上确实比 Express 的函数调用更结构化，但这个设计不是没有代价。

**编译期不可见的错误。** Express 中拼错了一个路由路径，至少代码还能跑，只是那个路由永远匹配不上。Spring Boot 中拼错注解值，应用照样启动，但你可能要到请求时才发现 404。更糟的是，`@PathVariable` 的变量名和路径中 `{id}` 不匹配时，在某些 Java 编译配置下不会报错，运行时直接 500。

**隐式魔法太多。** 一个 `@RestController` 类被 Spring 扫描、实例化、注册路由——这些步骤都是框架自动完成的。对于习惯了"看到什么就是什么"的 Express 开发者来说，需要一段时间适应这种"在别的地方帮你做了"的感觉。调试时，你经常需要理解框架的内部流程才能定位问题。

**DTO 层的代码膨胀。** Express 中一个 `req.body` 就能拿到所有数据，Spring Boot 中你需要为每个请求定义一个 DTO 类，加上 getter、setter（或者用 Lombok 的 `@Data`）。对于一个简单的 CRUD，DTO 类的数量可能比业务逻辑还多。但这不是冗余——在项目变大之后，DTO 层提供的类型安全和校验能力会帮你省下大量时间。
