日志服务跑了三天，凌晨两点突然挂了。你爬起来看日志，只看到一行 `Error: ECONNREFUSED`，完全不知道是哪个请求触发的、当时的上下文是什么、还有多少请求在排队。

你之前写 Java 的时候，`try-catch` 包一切，很安心。到了 Node.js，`try-catch` 对 `Promise` 里的错误完全无效，`async` 函数里漏一个 `await` 就可能产生一个 `unhandledRejection`，进程直接退出。异步世界的错误处理为什么这么让人头疼？

根本原因在于：`try-catch` 只能捕获**当前调用栈**中的同步异常。而异步操作（`fs.readFile`、`http.request`、`setTimeout`）的回调会在完全不同的调用栈中执行，`try-catch` 早就不在了。

## 同步错误 vs 异步错误：为什么 try-catch 失效

先看一个最直观的例子：

```js
// 同步错误 —— try-catch 可以捕获
try {
  JSON.parse("{invalid}");
} catch (err) {
  console.log("捕获到了:", err.message); // ✓
}

// 异步错误 —— try-catch 无能为力
try {
  setTimeout(() => {
    throw new Error("boom"); // ✗ uncaughtException，进程崩溃
  }, 0);
} catch (err) {
  console.log("永远不会执行");
}
```

`setTimeout` 的回调在事件循环的下一个 tick 执行，那时候 `try-catch` 的调用栈已经不存在了。[第 3 篇](/posts/nodejs-core/03-event-loop) 讲过事件循环的阶段划分，异步回调会在不同的阶段被取出执行，和同步代码的调用栈完全隔离。

`async/await` 看起来像同步代码，但它本质上是 Promise 的语法糖。`await` 只能捕获**你真正 await 了**的 Promise：

```js
// 这个可以捕获
async function readFile() {
  try {
    const data = await fs.promises.readFile("logs/app.log");
    return data;
  } catch (err) {
    console.log("捕获到了:", err.message); // ✓
  }
}

// 这个捕获不到 —— 忘了 await
async function broken() {
  try {
    fs.promises.readFile("nonexistent.txt"); // 忘了 await
    // Promise 被创建但没被 await，错误变成 unhandledRejection
  } catch (err) {
    console.log("捕获不到"); // ✗
  }
}
```

这是 Node.js 开发中最常见的坑之一。`eslint` 的 `require-await` 规则和 `@typescript-eslint/no-floating-promises` 规则就是专门检测这种问题的。

## 自定义错误类：让错误携带业务上下文

只记录 `error.message` 在生产环境几乎没用。你需要知道：这是什么类型的错误？发生在哪个模块？影响范围多大？需要人工介入吗？

自定义错误类就是解决这个问题的——让错误自己携带上下文：

```js
class AppError extends Error {
  constructor(
    message,
    { code, statusCode = 500, isOperational = true, cause } = {}
  ) {
    super(message);
    this.name = this.constructor.name;
    this.code = code;
    this.statusCode = statusCode;
    this.isOperational = isOperational; // 区分业务错误和编程错误
    this.timestamp = new Date().toISOString();
    if (cause) this.cause = cause;
    Error.captureStackTrace(this, this.constructor);
  }
}

class ValidationError extends AppError {
  constructor(message, fields) {
    super(message, { code: "VALIDATION_ERROR", statusCode: 400 });
    this.fields = fields; // 哪些字段校验失败了
  }
}

class StorageError extends AppError {
  constructor(message, operation, path) {
    super(message, { code: "STORAGE_ERROR", statusCode: 500 });
    this.operation = operation; // 'read' | 'write' | 'delete'
    this.path = path; // 操作的文件路径
  }
}
```

为什么要区分 `isOperational`？因为不是所有错误都一样处理：

- **业务错误**（`isOperational: true`）：参数校验失败、文件不存在、网络超时——这些是预期内会发生的，应该优雅处理并返回有意义的错误信息。
- **编程错误**（`isOperational: false`）：`TypeError`、`ReferenceError`、`null` 上调方法——这些说明代码有 bug，不应该恢复，应该让进程崩溃然后自动重启。

日志服务中的实际使用：

```js
// 查询日志时的参数校验
function queryLogs(params) {
  if (!params.from || !params.to) {
    throw new ValidationError("Missing required fields: from, to", {
      missing: ["from", "to"],
    });
  }
  if (new Date(params.from) > new Date(params.to)) {
    throw new ValidationError("from must be before to", {
      from: params.from,
      to: params.to,
    });
  }
  // ...
}

// 日志文件操作失败
async function appendLog(entry) {
  try {
    await fs.promises.appendFile(
      `logs/${entry.date}.log`,
      JSON.stringify(entry) + "\n"
    );
  } catch (err) {
    throw new StorageError(
      `Failed to append log: ${err.message}`,
      "write",
      `logs/${entry.date}.log`
    );
  }
}
```

## 全局兜底：最后一道防线

即使你在每个函数里都写了错误处理，也难免有漏网之鱼。Node.js 提供了两个全局事件作为最后防线：

```js
// 兜住同步代码中未被 try-catch 捕获的异常
process.on("uncaughtException", (err) => {
  // 记录完整的错误信息
  errorLogger.fatal("Uncaught exception", {
    message: err.message,
    stack: err.stack,
    name: err.name,
  });
  // 编程错误不应该恢复，让进程退出让 PM2/systemd 重启
  process.exit(1);
});

// 兜住未被 .catch() 或 try-catch 处理的 Promise 拒绝
process.on("unhandledRejection", (reason, promise) => {
  errorLogger.error("Unhandled rejection", {
    reason: String(reason),
    stack: reason?.stack,
  });
  // Node.js 15+ 中，unhandledRejection 默认会导致进程退出
  // 这里只是做记录，进程退出行为由 Node.js 控制
});
```

`uncaughtException` 的处理有一条铁律：**记录日志后必须退出进程**。Node.js 官方文档明确警告，在 `uncaughtException` 中继续运行是不安全的——进程的内部状态可能已经不一致了，继续运行可能产生更严重的问题（比如把错误数据写入数据库）。

`unhandledRejection` 的处理更灵活一些。如果一个 Promise rejection 不影响核心功能（比如非关键的日志上报失败），你可以只记录不退出。但如果是核心流程的 Promise 被拒绝，也应该让进程退出。

## HTTP 接口的错误处理：统一错误响应

[第 6 篇](/posts/nodejs-core/06-http) 我们用原生 `http` 模块搭建了查询 API。现在给它加上统一的错误处理：

```js
function sendError(res, err) {
  const statusCode = err.statusCode || 500;
  const body = {
    error: {
      code: err.code || "INTERNAL_ERROR",
      message:
        statusCode >= 500
          ? "Internal server error" // 不向客户端暴露内部错误详情
          : err.message,
      timestamp: err.timestamp,
    },
  };

  res.writeHead(statusCode, { "Content-Type": "application/json" });
  res.end(JSON.stringify(body));
}

// 在路由处理中统一使用
async function handleQueryLogs(req, res) {
  try {
    const params = parseQueryParams(req.url);
    const logs = await logStore.query(params);
    res.writeHead(200, { "Content-Type": "application/json" });
    res.end(JSON.stringify({ data: logs, total: logs.length }));
  } catch (err) {
    // ValidationError → 400
    // StorageError → 500
    // 其他未知错误 → 500
    sendError(res, err);
    if (!err.isOperational) {
      errorLogger.error("Unexpected error in query handler", { err });
    }
  }
}
```

注意一个安全细节：5xx 错误不向客户端暴露 `error.message`。内部错误信息（文件路径、SQL 语句、堆栈）可能包含敏感信息，不应该返回给前端。只返回一个通用的 "Internal server error"，详细的服务端日志自己留着排查。

## 结构化错误日志：只记有用的信息

日志服务的错误日志要和业务日志分开存储、分开分析。一条好的错误日志应该包含什么？

```js
function formatErrorLog(err, context = {}) {
  return {
    traceId: context.traceId || "unknown",
    timestamp: err.timestamp || new Date().toISOString(),
    level: err.isOperational ? "ERROR" : "FATAL",
    error: {
      type: err.name, // 'ValidationError' | 'StorageError'
      code: err.code, // 'VALIDATION_ERROR' | 'STORAGE_ERROR'
      message: err.message,
      stack: err.stack,
    },
    context: {
      path: err.path, // 如果是文件操作错误
      operation: err.operation, // 如果是文件操作错误
      fields: err.fields, // 如果是校验错误
      ...context, // 请求路径、用户 ID 等额外上下文
    },
  };
}

// 使用
try {
  await appendLog(entry);
} catch (err) {
  const log = formatErrorLog(err, {
    traceId: entry.traceId,
    endpoint: "/api/logs/ingest",
  });
  await fs.promises.appendFile("logs/error.log", JSON.stringify(log) + "\n");
  // 如果是操作错误，返回给客户端；如果是编程错误，让它继续抛出
  if (!err.isOperational) throw err;
}
```

关键点：`traceId`。在分布式系统中，一个请求可能经过多个服务。把请求的 `traceId` 记录到错误日志里，你才能把同一次请求产生的多条日志关联起来，快速定位问题链路。这个 traceId 正是我们在[第 1 篇](/posts/nodejs-core/01-buffer)设计的二进制协议中的第一个字段。

## 什么时候崩溃，什么时候恢复

这是错误处理中最难做决策的地方。一个简单的判断标准：

| 错误类型 | 例子                                 | 处理方式                 |
| -------- | ------------------------------------ | ------------------------ |
| 业务错误 | 参数校验失败、文件不存在、重复操作   | 捕获，返回友好提示       |
| 瞬时故障 | 网络超时、数据库连接断开             | 重试（带退避策略）       |
| 资源耗尽 | 内存不足、文件描述符用完             | 拒绝新请求，尝试释放资源 |
| 编程错误 | `TypeError`、读取 `undefined` 的属性 | 崩溃重启，不要尝试恢复   |

瞬时故障的重试要小心。网络超时后立刻重试大概率还是超时——服务端可能还在过载。应该用**指数退避**（exponential backoff）：

```js
async function withRetry(fn, maxRetries = 3) {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt === maxRetries) throw err;
      const delay = Math.min(1000 * 2 ** attempt, 10000); // 最多等 10 秒
      await new Promise((resolve) => setTimeout(resolve, delay));
    }
  }
}

// 使用：查询远程日志存储时自动重试
const logs = await withRetry(
  () => logStore.query(params),
  3 // 最多重试 3 次
);
```

## 什么时候用什么

错误处理的策略取决于错误的性质：

- **预期内的错误**（校验失败、资源不存在）：用自定义错误类，返回有意义的错误码和消息，让调用方知道该怎么处理。
- **瞬时故障**（网络超时、连接断开）：重试 + 退避，不要立刻放弃也不要无限重试。
- **未预期的编程错误**：不要尝试恢复。记录完整的上下文信息后让进程退出，交给进程管理器（PM2、systemd）重启。一个状态不一致的进程继续运行比崩溃更危险。

错误处理不是到处加 `try-catch`。过度捕获错误反而会让问题更难排查——错误被吞掉了，你甚至不知道它发生过。好的错误处理策略是：在最接近错误发生的地方捕获**业务错误**，让**编程错误**一路冒泡到全局处理器，由它决定进程的命运。
