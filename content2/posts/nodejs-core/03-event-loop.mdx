你写了一个 Node.js 脚本：先读配置文件，再初始化数据库连接，最后启动 HTTP 服务。运行发现，"启动 HTTP 服务"的代码在数据库还没连上时就执行了。你知道这是"异步"，但具体是"怎么异步"的？

更让你困惑的是这些面试题：`setTimeout(fn, 0)` 和 `Promise.resolve().then(fn)` 谁先执行？`process.nextTick` 和 `setImmediate` 呢？为什么同样的代码，放在 `fs.readFile` 回调里执行顺序会变？

这篇彻底搞清楚这些问题。

## 为什么 Node.js 用单线程 + 事件循环

Java 的 Servlet 模型是"一个请求一个线程"。来 1000 个请求就开 1000 个线程，每个线程占 1MB 栈空间，光线程就吃掉 1GB 内存。再加上线程上下文切换的开销，并发一高服务器就扛不住了。

Node.js 走了另一条路：单线程 + 事件循环。主线程只负责接收请求和调度回调，真正耗时的 I/O 操作（文件读写、网络请求）交给操作系统的线程池或 libuv 的线程池去做，完成后通过事件通知主线程来处理结果。

```
Java 模型（每请求一线程）：
请求1 → [线程1] ── 阻塞等待 I/O ──→ 返回
请求2 → [线程2] ── 阻塞等待 I/O ──→ 返回
请求3 → [线程3] ── 阻塞等待 I/O ──→ 返回

Node.js 模型（单线程 + 事件循环）：
请求1 → [注册回调] → 事件循环 ← I/O 完成，执行回调1
请求2 → [注册回调] → 事件循环 ← I/O 完成，执行回调2
请求3 → [注册回调] → 事件循环 ← I/O 完成，执行回调3
```

这个模型的天然优势是：处理 I/O 密集型任务时，不需要为每个请求创建线程，内存开销极小。但代价是你不能在主线程做 CPU 密集型操作——否则事件循环被阻塞，所有请求都会卡住。

## libuv 的六个阶段

事件循环由 libuv 实现，每次循环经过六个阶段：

```
   ┌──────────────────────────┐
┌─>│        timers             │  setTimeout、setInterval
│  └────────┬─────────────────┘
│  ┌────────▼─────────────────┐
│  │   pending callbacks       │  上轮未执行的 I/O 回调
│  └────────┬─────────────────┘
│  ┌────────▼─────────────────┐
│  │   idle, prepare           │  内部使用（可忽略）
│  └────────┬─────────────────┘
│  ┌────────▼─────────────────┐
│  │        poll               │  获取新的 I/O 事件，执行 I/O 回调
│  └────────┬─────────────────┘
│  ┌────────▼─────────────────┐
│  │        check              │  setImmediate 回调
│  └────────┬─────────────────┘
│  ┌────────▼─────────────────┐
│  │   close callbacks         │  socket.close 等关闭回调
│  └────────┬─────────────────┘
└────┴────────────────────────┘
```

每个阶段做什么：

- **timers**：执行到期的 `setTimeout` 和 `setInterval` 回调。注意"到期"是指时间到了，但不保证立即执行——如果 poll 阶段正在处理 I/O，timer 回调要等 poll 完成才执行。
- **pending callbacks**：执行上一轮循环中未完成的 I/O 回调（比如某些系统操作的延迟回调）。
- **idle / prepare**：libuv 内部使用，Node.js 应用代码不会直接接触。
- **poll**：核心阶段。两件事：计算应该阻塞多久（距离下一个 timer 到期还有多久），然后阻塞等待 I/O 事件。如果有到期的 timer，poll 不会一直等，会立即进入 check 阶段。
- **check**：执行 `setImmediate` 的回调。
- **close callbacks**：执行 `socket.on('close', ...)` 之类的关闭回调。

理解这个模型后，很多看似奇怪的行为就解释得通了。

## 微任务 vs 宏任务

如果你了解浏览器的事件循环，可能知道微任务（microtask）和宏任务（macrotask）的概念。Node.js 也有类似机制，但细节不同。

Node.js 中的微任务：

- `Promise.resolve().then()`
- `queueMicrotask()`
- `process.nextTick()`

Node.js 中的宏任务：

- `setTimeout` / `setInterval`
- `setImmediate`
- I/O 回调（`fs.readFile` 等）

关键区别：**微任务在每个宏任务之间、以及每个阶段结束后都会被清空**。也就是说，每执行完一个回调函数，Node.js 都会检查微任务队列并执行所有微任务，然后再继续下一个回调。

执行顺序可以简化为：

```
同步代码 → process.nextTick → Promise.then → 宏任务回调
```

来看一段经典的代码：

```js
console.log("1");

setTimeout(() => console.log("2"), 0);

Promise.resolve().then(() => console.log("3"));

process.nextTick(() => console.log("4"));

console.log("5");
```

输出：`1 5 4 3 2`

执行过程：

1. 同步执行 `console.log('1')`
2. 注册 `setTimeout` 回调到 timers 队列
3. 注册 `Promise.then` 到微任务队列
4. 注册 `nextTick` 到 nextTick 队列
5. 同步执行 `console.log('5')`
6. 清空 nextTick 队列：`console.log('4')`
7. 清空微任务队列：`console.log('3')`
8. 进入事件循环，timers 阶段：`console.log('2')`

注意 `nextTick` 比 `Promise.then` 先执行。`process.nextTick` 有自己的独立队列，优先级比 Promise 微任务更高。这个设计是有历史原因的——`nextTick` 在 Promise 被引入之前就存在了，Node.js 团队选择保持 `nextTick` 的更高优先级以向后兼容。

## setTimeout vs setImmediate

`setTimeout(fn, 0)` 和 `setImmediate(fn)` 都是想"尽快执行"，但行为不同：

```js
// 在主模块中直接运行——顺序不确定
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
```

你可能以为 `setTimeout(fn, 0)` 更快，但实际输出可能是 `immediate timeout`，也可能是 `timeout immediate`。

原因：在主模块中，事件循环还没开始。`setTimeout` 需要进入 timers 阶段才能执行，而 `setImmediate` 在 check 阶段执行。由于主模块代码执行完毕后，事件循环先进入哪个阶段取决于进程的性能（如果 I/O 准备得快，可能先 poll 再 timers）。

但在 I/O 回调中，顺序是确定的：

```js
import { readFile } from "node:fs";

readFile("some-file.txt", () => {
  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));
  // 输出一定是：immediate timeout
});
```

为什么？因为 I/O 回调在 poll 阶段执行。poll 阶段结束后，下一个阶段是 check（`setImmediate`），然后下一轮循环才到 timers（`setTimeout`）。

## 实战：事件循环阻塞检测

你的 Node.js 脚本同时做三件事：读取配置文件（I/O）、定时发送心跳（Timer）、处理命令行输入（stdin）。如果某个操作阻塞了事件循环，心跳就会停止发送。

怎么检测事件循环是否被阻塞？Node.js 提供了 `monitorEventLoopDelay`：

```js
import { monitorEventLoopDelay } from "node:perf_hooks";

const histogram = monitorEventLoopDelay({ resolution: 20 });

histogram.enable();

// 定期检查事件循环延迟
setInterval(() => {
  const delay = histogram.mean / 1e6; // 纳秒转毫秒
  if (delay > 50) {
    console.warn(`事件循环延迟 ${delay.toFixed(1)}ms，可能被阻塞`);
  }
}, 5000);
```

`monitorEventLoopDelay` 的工作原理很简单：它每隔 `resolution` 毫秒安排一个定时器，如果定时器被延迟执行了，说明事件循环在那个时间段内被阻塞了。

### CPU 密集任务拆分

如果确实有 CPU 密集的任务（比如解析一个大 JSON 文件），怎么避免阻塞事件循环？

一种方式是用 `setImmediate` 分片：把大任务拆成小块，每块之间让出事件循环：

```js
function processLargeArray(items, onProgress, onComplete) {
  let index = 0;
  const batchSize = 1000;

  function processBatch() {
    const end = Math.min(index + batchSize, items.length);

    for (; index < end; index++) {
      // 处理 items[index]
    }

    if (index < items.length) {
      // 让出事件循环，下一轮继续
      setImmediate(processBatch);
      onProgress(index / items.length);
    } else {
      onComplete();
    }
  }

  processBatch();
}
```

这种方式把一个同步的 CPU 密集任务变成了异步的，每处理 1000 条数据就让出一次事件循环，让 I/O 回调和定时器有机会执行。代价是总耗时变长了（因为每次让出都有额外开销），但事件循环不会被长时间阻塞。

更好的方式是用 Worker Threads，把 CPU 密集任务完全放到另一个线程。

## 和浏览器事件循环的区别

如果你是从前端转过来的，可能会把浏览器的事件循环和 Node.js 的搞混。核心区别：

| 特性               | 浏览器               | Node.js                          |
| ------------------ | -------------------- | -------------------------------- |
| 宏任务队列         | 多个（按优先级排序） | 无明确队列概念，按阶段执行       |
| 微任务             | 在每个宏任务之后清空 | 在每个阶段结束和每个回调之后清空 |
| `process.nextTick` | 不存在               | 独立队列，优先级高于 Promise     |
| 渲染               | 在微任务之后         | 不存在                           |
| I/O                | 不直接处理           | poll 阶段专门处理                |

一个重要的实践差异：在浏览器中，`Promise.then` 的微任务是在当前宏任务结束后立即执行的。在 Node.js 中，微任务是在**每个回调函数之后**执行的——这意味着一个阶段中的多个回调之间，微任务也会被执行。

理解事件循环不是为了应付面试题，而是为了写出真正高效的异步代码。当你的服务出现"请求变慢"、"定时器不准"、"回调不执行"这类问题时，事件循环的知识能帮你快速定位原因。下一篇我们来看模块系统——Node.js 怎么组织代码，以及 CommonJS 和 ESM 之间那些让你头疼的兼容性问题。
