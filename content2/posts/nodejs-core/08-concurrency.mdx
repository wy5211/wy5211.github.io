日志服务需要调用外部命令——用 `gzip` 压缩日志、用 `grep` 快速搜索历史日志。你用 `exec('gzip ' + filename)` 压缩文件，同事 Code Review 时脸色大变：你知道这行代码有 Shell 注入风险吗？

另一个问题：日志服务要对过去 24 小时的日志做统计分析——按小时聚合错误数量、计算 P99 响应时间。单线程处理 500MB 数据要跑 30 秒，事件循环被完全阻塞。

Node.js 是单线程的，但有两种方式突破这个限制：**子进程**（适合调用外部命令）和 **Worker Threads**（适合纯 JavaScript 计算）。选错 API 不仅性能差，还可能引入安全风险。

## 什么时候用子进程，什么时候用 Worker Threads

先给一个判断标准：

| 特性     | 子进程 (fork) | Worker Threads                 |
| -------- | ------------- | ------------------------------ |
| 创建开销 | 大（几十 MB） | 小（几 MB）                    |
| 启动速度 | 慢（~100ms）  | 快（~10ms）                    |
| 通信方式 | IPC（序列化） | 结构化克隆 / SharedArrayBuffer |
| 内存共享 | 不共享        | 可共享                         |
| 适用场景 | 调用外部命令  | 纯 JavaScript 计算             |

如果你的任务是纯 JavaScript 计算（不需要调用外部命令、不需要独立的环境隔离），Worker Threads 是更好的选择。如果需要调用 `grep`、`gzip` 等系统命令，用子进程。

## 子进程：四种创建方式

```js
import { spawn, exec, execFile, fork } from "node:child_process";
```

这四个函数看起来都能"执行命令"，但行为完全不同：

| 函数       | 创建方式              | 输出处理                         | 适用场景                                |
| ---------- | --------------------- | -------------------------------- | --------------------------------------- |
| `spawn`    | 直接创建进程          | 流式（stdin/stdout/stdout 管道） | 长时间运行、大量输出                    |
| `exec`     | 创建 Shell 再执行命令 | 缓冲全部输出后回调               | 短命令、需要 Shell 特性（管道、通配符） |
| `execFile` | 直接创建进程          | 缓冲全部输出后回调               | 短命令、不需要 Shell                    |
| `fork`     | 创建 Node.js 子进程   | IPC 通道通信                     | Node.js 之间的通信                      |

### spawn：流式处理大量输出

`spawn` 是最灵活的方式。它不会创建 Shell，直接执行命令，输出通过管道以流的方式传输：

```js
import { spawn } from "node:child_process";

// 用 gzip 压缩日志文件
const gzip = spawn("gzip", ["-c", "logs/2026-03-15.log"]);

// stdin/stdout 都是 Stream，可以用 pipe 连接
import { createReadStream, createWriteStream } from "node:fs";
import { pipeline } from "node:stream/promises";

await pipeline(
  createReadStream("logs/2026-03-15.log"),
  gzip.stdin,
  createWriteStream("logs/2026-03-15.log.gz")
);
```

注意 `spawn` 的第一个参数是命令本身，第二个参数是参数数组——它不会经过 Shell 解析，所以不用担心 Shell 注入。

### exec：方便但有安全风险

`exec` 会先创建一个 Shell（`/bin/sh`），然后在 Shell 中执行命令字符串。这让它支持 Shell 的管道、通配符等特性，但也带来了注入风险：

```js
import { exec } from "node:child_process";

// 危险！如果 filename 来自用户输入，比如 "; rm -rf /"
exec(`gzip ${filename}`, (err, stdout, stderr) => {
  /* ... */
});

// 安全：用 execFile 代替
import { execFile } from "node:child_process";
execFile("gzip", [filename], (err, stdout, stderr) => {
  /* ... */
});
```

`exec` 的另一个问题是它会把 stdout 和 stderr 全部缓冲到内存中。如果命令输出很大（比如 `grep` 一个大文件），内存可能爆掉。对于大量输出，用 `spawn`。

### fork：Node.js 进程间通信

`fork` 专门用于创建 Node.js 子进程，自带 IPC 通道：

```js
// parent.js
import { fork } from "node:child_process";
const child = fork("./worker.js");

child.send({ task: "analyze", filePath: "logs/2026-03-15.log" });
child.on("message", (result) => {
  console.log("分析结果:", result);
});

// worker.js
process.on("message", (msg) => {
  if (msg.task === "analyze") {
    // 执行分析任务...
    process.send({ totalLines: 15000, errorCount: 42 });
  }
});
```

`fork` 底层就是 `spawn`，但自动设置了 `ipc` 通道。它只能用于 Node.js 文件，不能执行系统命令。

### 实战：grep 进程池

日志服务经常需要在历史日志中搜索关键词。每次搜索都 `spawn` 一个新的 `grep` 进程太浪费——启动进程的开销远大于搜索本身。用进程池复用 grep 进程：

```js
import { spawn } from "node:child_process";

class GrepPool {
  constructor(size = 4) {
    this.queue = [];
    this.workers = [];

    for (let i = 0; i < size; i++) {
      const worker = spawn("grep", ["--line-buffered", "-F", ""], {
        stdio: ["pipe", "pipe", "pipe"],
      });

      worker.stdin.setDefaultEncoding("utf-8");
      worker.stdout.setEncoding("utf-8");
      this.workers.push({ process: worker, busy: false });

      worker.on("exit", () => {
        this._restartWorker(i);
      });
    }
  }

  async search(pattern, logPath) {
    return new Promise((resolve, reject) => {
      const task = { pattern, logPath, resolve, reject };
      const worker = this.workers.find((w) => !w.busy);

      if (worker) {
        this._execute(worker, task);
      } else {
        this.queue.push(task);
      }
    });
  }

  _execute(worker, task) {
    worker.busy = true;
    const { process: grep } = worker;
    const results = [];

    // 重新启动 grep 进程（grep 不支持动态切换搜索模式）
    grep.kill();
    const newGrep = spawn("grep", [
      "--line-buffered",
      "-F",
      task.pattern,
      task.logPath,
    ]);
    worker.process = newGrep;

    newGrep.stdout.on("data", (data) => {
      results.push(data.trim());
    });

    newGrep.on("close", (code) => {
      worker.busy = false;
      if (code === 0) {
        task.resolve(results);
      } else if (code === 1) {
        task.resolve([]); // 没有匹配，不是错误
      } else {
        task.reject(new Error(`grep 退出码: ${code}`));
      }

      // 处理队列中的下一个任务
      if (this.queue.length > 0) {
        this._execute(worker, this.queue.shift());
      }
    });
  }
}
```

## Worker Threads：多线程 JavaScript 计算

子进程能调用外部命令，但如果你需要的是纯 JavaScript 计算（数据排序、统计分析、加密），创建进程的开销太大了。Worker Threads 是更轻量的选择。

### 基本用法

```js
// main.js
import { Worker } from "node:worker_threads";

const worker = new Worker("./analyze.js", {
  workerData: { filePath: "logs/2026-03-15.log", startLine: 0, endLine: 10000 },
});

worker.on("message", (result) => {
  console.log("分析结果:", result);
});

worker.on("error", (err) => {
  console.error("Worker 出错:", err);
});

worker.on("exit", (code) => {
  if (code !== 0) console.error(`Worker 退出码: ${code}`);
});
```

```js
// analyze.js
import { workerData, parentPort } from "node:worker_threads";
import { readFileSync } from "node:fs";

const { filePath, startLine, endLine } = workerData;

const content = readFileSync(filePath, "utf-8");
const lines = content.split("\n").slice(startLine, endLine);

const errorCount = lines.filter((l) => l.includes('"level":"ERROR"')).length;
const warnCount = lines.filter((l) => l.includes('"level":"WARN"')).length;

parentPort.postMessage({ errorCount, warnCount, totalLines: lines.length });
```

`workerData` 是初始化时传给 Worker 的数据（只能传一次）。`parentPort` 是 Worker 和主线程通信的通道——本质上也是结构化克隆（类似 JSON 序列化），不能共享引用。

### SharedArrayBuffer：共享内存

`parentPort.postMessage` 每次都会复制数据。如果需要频繁地在主线程和 Worker 之间共享数据（比如一个不断更新的统计计数器），用 `SharedArrayBuffer`：

```js
// main.js
import { Worker } from "node:worker_threads";

// 创建共享内存：4 个 32 位整数
const sharedBuffer = new SharedArrayBuffer(16);
const counters = new Int32Array(sharedBuffer);

const worker = new Worker("./counter.js", { workerData: { sharedBuffer } });

worker.on("message", () => {
  console.log("错误数:", counters[0]);
  console.log("警告数:", counters[1]);
});
```

```js
// counter.js
import { workerData, parentPort } from "node:worker_threads";

const { sharedBuffer } = workerData;
const counters = new Int32Array(sharedBuffer);

// 直接在共享内存上修改
counters[0] += 15; // errorCount
counters[1] += 42; // warnCount

parentPort.postMessage("done");
```

`SharedArrayBuffer` 的数据在所有 Worker 和主线程之间共享，修改是立即可见的。但这也意味着你需要处理并发问题——多个 Worker 同时写同一个位置会产生竞态条件。用 `Atomics` 提供原子操作：

```js
import { Atomics } from "node:atomics";

// 原子递增（返回旧值）
const oldCount = Atomics.add(counters, 0, 1); // counters[0]++

// 原子加载
const current = Atomics.load(counters, 0);
```

### 实战：多线程日志分析引擎

把 500MB 的日志文件分片，分配给多个 Worker 并行处理：

```js
import { Worker, availableParallelism } from "node:worker_threads";
import { readFileSync } from "node:fs";

async function analyzeLog(filePath) {
  const content = readFileSync(filePath, "utf-8");
  const lines = content.split("\n");
  const totalLines = lines.length;

  const numWorkers = Math.min(availableParallelism(), 4);
  const linesPerWorker = Math.ceil(totalLines / numWorkers);

  const results = await Promise.all(
    Array.from({ length: numWorkers }, (_, i) => {
      const start = i * linesPerWorker;
      const end = Math.min(start + linesPerWorker, totalLines);

      return new Promise((resolve, reject) => {
        const worker = new Worker("./analyze-worker.js", {
          workerData: { lines: lines.slice(start, end) },
        });

        worker.on("message", resolve);
        worker.on("error", reject);
        worker.on("exit", (code) => {
          if (code !== 0) reject(new Error(`Worker ${i} 退出码: ${code}`));
        });
      });
    })
  );

  // 汇总所有 Worker 的结果
  return results.reduce(
    (acc, r) => ({
      errorCount: acc.errorCount + r.errorCount,
      warnCount: acc.warnCount + r.warnCount,
      totalLines: acc.totalLines + r.totalLines,
    }),
    { errorCount: 0, warnCount: 0, totalLines: 0 }
  );
}
```

`availableParallelism()` 返回 CPU 逻辑核心数。在 8 核机器上，4 个 Worker 并行处理，理论上是单线程的 4 倍速度（实际可能 3 倍左右，因为有线程创建和数据传递的开销）。

### Transferable 对象：零拷贝传输

上面的例子中，`workerData: { lines }` 会把 lines 数组复制一份传给 Worker。对于大数据集，这个复制开销不小。用 Transferable 可以实现零拷贝：

```js
// 把 ArrayBuffer 转移给 Worker（转移后主线程就不能访问了）
const buffer = new TextEncoder().encode(content).buffer;
const worker = new Worker("./analyze-worker.js", {
  workerData: { buffer },
  transferList: [buffer],
});
```

`transferList` 中的对象会被"转移"而不是"复制"——所有权的转移是瞬间的，不需要数据拷贝。代价是转移后原线程不能再访问这个对象。

## 信号处理与优雅退出

子进程管理中最容易忽略的是信号处理。当你的服务收到 `SIGTERM`（比如 Kubernetes 要终止 Pod），你需要：

1. 停止接受新请求
2. 等待正在处理的任务完成
3. 关闭所有子进程和 Worker
4. 退出

```js
import { spawn } from "node:child_process";

const workers = [];

function cleanup() {
  console.log("收到退出信号，正在清理...");

  for (const worker of workers) {
    worker.kill("SIGTERM");
  }

  // 5 秒后还没退出就强制杀掉
  setTimeout(() => {
    for (const worker of workers) {
      if (!worker.killed) worker.kill("SIGKILL");
    }
    process.exit(0);
  }, 5000);
}

process.on("SIGTERM", cleanup);
process.on("SIGINT", cleanup);
```

`SIGTERM` 让进程有机会清理资源，`SIGKILL` 是强制终止，进程无法捕获。先用 `SIGTERM`，超时再用 `SIGKILL`，这是标准的优雅退出模式。

`child.on('error')` 和 `child.on('exit')` 也是必须监听的——不监听 `error` 会导致未捕获异常让主进程崩溃。

## Worker Threads 的限制

Worker Threads 不是万能的：

- 每个 Worker 有独立的 V8 实例和事件循环，创建开销比协程大得多
- 不能共享 DOM、原生模块（C++ addon）的部分功能
- `SharedArrayBuffer` 在浏览器中需要特定的安全头，Node.js 中不受此限制
- 调试 Worker 比调试主线程更复杂

## 什么时候用什么

一个简单的判断标准：如果任务需要调用外部命令（`grep`、`gzip`），用子进程。如果任务是纯 JavaScript 计算（数据排序、加密、统计分析），用 Worker Threads。

日志服务的并发能力到此就完整了——子进程调用外部工具，Worker Threads 处理 CPU 密集计算。下一篇我们来看安全——日志中包含敏感信息，API 接口需要认证，怎么用 Node.js 的 `crypto` 模块保护数据。
