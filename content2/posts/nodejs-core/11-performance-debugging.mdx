日志服务上线一周，一切正常。第二周开始，客服反馈查询接口响应变慢了——从 50ms 涨到了 2s。你打开服务器一看，CPU 没跑满，内存也没超限，但就是慢。

性能问题最难的不是修复，而是定位。你不知道瓶颈在 CPU、内存、磁盘 I/O 还是网络。盲目优化（"加点缓存试试"）往往是在浪费时间和掩盖问题。你需要一套系统的方法来诊断。

## 从哪里开始：建立性能基线

在优化之前，先搞清楚"正常"是什么样的。没有基线，你就不知道当前状态是快还是慢、优化有没有效果。

Node.js 内置的 `perf_hooks` 模块提供了高精度计时器：

```js
import { performance, PerformanceObserver } from "node:perf_hooks";

const obs = new PerformanceObserver((list) => {
  const entries = list.getEntries();
  for (const entry of entries) {
    metrics.record(entry.name, entry.duration);
  }
});
obs.observe({ entryTypes: ["measure"] });

// 在关键路径上打点
performance.mark("query-start");
const logs = await logStore.query(params);
performance.mark("query-end");
performance.measure("log-query", "query-start", "query-end");
```

但这只是在一次请求中测量。生产环境需要的是**聚合指标**——P50、P95、P99 的响应时间：

```js
class Metrics {
  #histograms = new Map();

  record(name, value) {
    if (!this.#histograms.has(name)) {
      this.#histograms.set(name, []);
    }
    const samples = this.#histograms.get(name);
    samples.push(value);
    if (samples.length > 1000) samples.shift();
  }

  percentile(name, p) {
    const samples = this.#histograms.get(name);
    if (!samples?.length) return 0;
    const sorted = [...samples].sort((a, b) => a - b);
    const idx = Math.ceil((sorted.length * p) / 100) - 1;
    return sorted[idx];
  }
}
```

为什么看 P99 而不是平均值？平均值会被极端值拉偏——100 个请求都是 50ms，1 个请求是 5000ms，平均值变成 99.5ms，看起来还好，但那一个 5 秒的请求可能已经让用户等得不耐烦了。P99 告诉你：99% 的请求在多少毫秒内完成。

## 事件循环延迟：最容易被忽视的瓶颈

[第 3 篇](/posts/nodejs-core/03-event-loop) 讲过事件循环的执行模型。如果事件循环被阻塞，所有 I/O 回调、定时器、微任务都会被延迟。这是 Node.js 性能问题的头号嫌疑犯。

怎么知道事件循环有没有被阻塞？Node.js 22 提供了 `monitorEventLoopDelay`：

```js
import { monitorEventLoopDelay } from "node:perf_hooks";

const histogram = monitorEventLoopDelay({ resolution: 20 });
histogram.enable();

setInterval(() => {
  const p99 = histogram.percentile(99);
  console.log(`Event loop delay p99: ${(p99 / 1e6).toFixed(2)}ms`);
  histogram.reset();
}, 10_000);
```

事件循环延迟的正常值应该在几毫秒以内。如果 P99 超过 50ms，说明有东西在阻塞事件循环——通常是 [第 8 篇](/posts/nodejs-core/08-concurrency) 提到的 CPU 密集任务（大 JSON 解析、正则匹配、加密计算）。

一个实用的经验法则：事件循环延迟超过 100ms，用户体验就会明显变差。超过 1 秒，基本可以认为服务不可用了。

## 内存：不只是"够不够用"

`process.memoryUsage()` 返回的四个指标各有含义：

```js
const mem = process.memoryUsage();
console.log({
  rss: `${(mem.rss / 1024 / 1024).toFixed(1)} MB`, // 进程占用的总物理内存
  heapTotal: `${(mem.heapTotal / 1024 / 1024).toFixed(1)} MB`, // V8 堆的总大小
  heapUsed: `${(mem.heapUsed / 1024 / 1024).toFixed(1)} MB`, // V8 堆中已使用的部分
  external: `${(mem.external / 1024 / 1024).toFixed(1)} MB`, // Buffer 等堆外内存
});
```

- **RSS**（Resident Set Size）：进程占用的总物理内存。RSS 持续增长通常意味着内存泄漏。
- **heapTotal / heapUsed**：V8 堆是 JavaScript 对象分配内存的地方。`heapUsed` 接近 `heapTotal` 时，V8 会频繁触发 GC，导致延迟抖动。
- **external**：Buffer、TypedArray 等堆外内存。

V8 的堆大小有默认上限（64 位系统约 1.5GB）。超过这个限制，进程会直接 OOM 崩溃。可以通过 `--max-old-space-size` 调整：

```bash
node --max-old-space-size=4096 server.js  # 最大 4GB
```

但调大堆上限不是解决内存泄漏的方法，只是推迟崩溃的时间。

## 内存泄漏：最常见的三种模式

内存泄漏在 Node.js 中特别隐蔽——进程不会像浏览器那样刷新页面就释放，一个微小的泄漏持续几天就能吃光内存。

**模式一：闭包引用**

```js
// 每次请求都创建一个新的闭包，但 cache 永远不清理
const cache = new Map();

function handleRequest(req, res) {
  const key = req.headers["x-trace-id"];
  cache.set(key, {
    data: processBody(req),
    req, // ← 这个引用导致 req 和 res 无法被回收
    timestamp: Date.now(),
  });
}
```

**模式二：事件监听器泄漏**

```js
// 每次调用都添加新的监听器，从不移除
function subscribe(emitter, handler) {
  emitter.on("log", handler); // ← 每次都加一个
}

// 修复：用 once，或者手动 removeListener
emitter.once("log", handler);
```

Node.js 有一个保护机制——如果同一个事件的监听器超过 10 个，会发出警告。通常不应该调高这个阈值，而是应该排查为什么会注册这么多监听器。

**模式三：全局缓存无限增长**

```js
// 经典错误：缓存没有淘汰策略
const queryCache = new Map();

async function queryLogs(params) {
  const key = JSON.stringify(params);
  if (queryCache.has(key)) return queryCache.get(key);

  const result = await logStore.query(params);
  queryCache.set(key, result); // ← 永远不清理
  return result;
}

// 修复：限制缓存大小
const MAX_CACHE_SIZE = 500;
if (queryCache.size >= MAX_CACHE_SIZE) {
  const oldest = queryCache.keys().next().value;
  queryCache.delete(oldest);
}
```

## CPU Profiling：找到那个吃 CPU 的函数

当你确认事件循环被阻塞了，下一步就是找出是哪个函数在吃 CPU。Node.js 内置了 V8 Profiler：

```bash
# 采集 60 秒的 CPU profile
node --prof server.js
# Ctrl+C 停止后，当前目录会生成 isolate-*.log

# 解析 profile 文件
node --prof-process isolate-*.log > profile.txt
```

`profile.txt` 的关键部分是 Bottom up（自底向上）视图。关注两个数字：

- **self**：函数自身执行的时间（不含子函数调用）。这个数字大说明瓶颈在这个函数内部。
- **total**：函数及其所有子函数的总执行时间。

日志服务中常见的 CPU 热点：

| 热点                            | 原因                | 优化方向                                                       |
| ------------------------------- | ------------------- | -------------------------------------------------------------- |
| `JSON.parse` / `JSON.stringify` | 解析大量日志 JSON   | 用 Stream 逐行解析，或用二进制协议                             |
| 正则表达式                      | 复杂的正则匹配      | 预编译正则，避免回溯爆炸                                       |
| 加密操作                        | HMAC 签名、AES 加密 | [第 8 篇](/posts/nodejs-core/08-concurrency) 的 Worker Threads |
| 字符串拼接                      | 大量 `+=` 操作      | 用数组 `push` + `join`，或 Buffer                              |

## 调试：Node.js Inspector

找到瓶颈后，如果是代码 bug 导致的，光看 Profiler 还不够——你需要用调试器深入运行中的程序。

### --inspect 与 Chrome DevTools

Node.js 内置了基于 Chrome DevTools Protocol 的调试器：

```bash
node --inspect server.js
# 输出：Debugger listening on ws://127.0.0.1:9229/...
```

在 Chrome 浏览器打开 `chrome://inspect`，点击 Node.js 进程下面的 "inspect"，就能看到和调试前端代码一样的 DevTools 界面。

两个常用标志：

```bash
# 等调试器连接后才开始执行
node --inspect-brk server.js

# 正常启动，调试器随时可以连
node --inspect server.js
```

### VS Code 调试配置

在项目根目录创建 `.vscode/launch.json`：

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "node",
      "request": "launch",
      "name": "启动日志服务",
      "program": "${workspaceFolder}/src/server.ts",
      "runtimeArgs": ["--enable-source-maps"],
      "console": "integratedTerminal"
    },
    {
      "type": "node",
      "request": "attach",
      "name": "连接运行中的进程",
      "port": 9229,
      "restart": true,
      "localRoot": "${workspaceFolder}",
      "remoteRoot": "/app"
    }
  ]
}
```

几个实用的断点技巧：

- **条件断点**：右键断点设置条件，比如 `traceId === 'trace_a1x9k2m'`，只有处理这个特定请求时才停下来
- **日志断点**：不断下程序，只是在命中断点时输出一条日志——相当于不修改代码的 `console.log`
- **Watch 表达式**：添加表达式，每次断点命中时自动计算并显示值

### Source Map

如果你用 TypeScript 开发，编译后的代码和源码的行号对不上。Source Map 就是那个映射表：

```json
// tsconfig.json
{
  "compilerOptions": {
    "sourceMap": true,
    "inlineSources": true // 把源码内容嵌入 map 文件
  }
}
```

Node.js 需要加 `--enable-source-maps` 才会使用它：

```bash
node --enable-source-maps dist/server.js
```

一个容易踩的坑：如果 `dist/` 目录只有 `.js` 文件，没有 `.ts` 源文件，即使加载了 Source Map，调试器也找不到源码。用 `inlineSources: true` 把源码内容嵌入 `.map` 文件可以解决。

### 远程调试

生产环境的代码跑在 Docker 容器或远程服务器上，需要把调试端口转发到本地：

```bash
# SSH 隧道
ssh -L 9229:localhost:9229 deploy@prod-server-01

# Docker 端口映射
docker run -p 9229:9229 log-service:latest
```

**安全提醒**：`--inspect` 默认绑定 `127.0.0.1`。如果绑定 `0.0.0.0`，网络上的任何人都能连接你的调试器。**永远不要在生产环境对公网暴露 9229 端口**。更安全的做法是用 `kill -USR1 <pid>` 按需激活调试器。

## 诊断顺序

性能诊断不是上来就开 Profiler。按这个顺序排查：

1. **先看指标**：响应时间 P99 有没有恶化？事件循环延迟有没有升高？内存 RSS 有没有持续增长？指标告诉你"有没有问题"。
2. **事件循环延迟高** → CPU 密集任务阻塞。用 `--prof` 找到热点函数，考虑用 Worker Threads 卸载。
3. **内存持续增长** → 内存泄漏。用 `v8.writeHeapSnapshot()` 生成堆快照，对比两个时间点的快照找出泄漏对象。
4. **磁盘 I/O 高** → 日志写入太频繁。用 [第 2 篇](/posts/nodejs-core/02-stream) 的 `highWaterMark` 调整缓冲区，或者批量写入减少 fsync 次数。
5. **都不高但就是慢** → 可能是外部依赖（数据库、第三方 API）慢。给外部调用加上超时和断路器。

性能优化的黄金法则是：**先测量，再优化，然后再次测量验证效果**。没有测量数据的优化等于盲人摸象。
