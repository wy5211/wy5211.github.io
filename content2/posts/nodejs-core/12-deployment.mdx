日志服务在本地跑得好好的。你满怀信心地部署到服务器上，然后：

- Node 版本不对，`globalThis.navigator` 报错
- 环境变量缺失，数据库连接字符串是 `undefined`
- 进程跑了半小时就挂了，没人知道
- 部署新版本时，正在处理的请求被直接中断

开发和生产环境之间的鸿沟比你想象的大。这篇文章把日志服务从"能在本地跑"变成"能在生产环境稳定运行"。

## Docker：让"在我机器上能跑"变成真正的可复现

部署的第一步是解决环境一致性问题。Docker 通过把应用和它的运行环境打包成一个镜像，确保开发和生产环境完全一致。

日志服务的 Dockerfile：

```dockerfile
# 构建阶段
FROM node:22-alpine AS builder
WORKDIR /app

# 先拷贝依赖文件，利用 Docker 缓存层
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile

# 再拷贝源码并构建
COPY . .
RUN pnpm run build

# 运行阶段——更小的镜像
FROM node:22-alpine
WORKDIR /app

# 不需要 devDependencies
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package.json ./

ENV NODE_ENV=production
EXPOSE 3000

USER node
CMD ["node", "dist/server.js"]
```

两个关键细节：

**多阶段构建**：构建阶段用完整的 Node.js 镜像安装依赖和编译 TypeScript，运行阶段只拷贝产物。最终镜像里没有 TypeScript 编译器、没有测试文件、没有源码，体积从 1GB+ 缩小到 200MB 左右。

**层缓存优化**：`package.json` 和 `pnpm-lock.yaml` 放在源码之前拷贝。因为依赖文件变化频率远低于源码，这样大部分构建都能命中缓存，不用每次都重新安装依赖。

## 环境变量：配置和代码分离

日志服务需要知道数据库连接串、加密密钥、监听端口。这些值不应该硬编码在代码里（代码进了 Git 就泄露了），也不应该写死在 Docker 镜像里（同一个镜像要部署到不同环境）。

环境变量是 Node.js 事实上的配置标准：

```js
const config = {
  port: parseInt(process.env.PORT || "3000"),
  logDir: process.env.LOG_DIR || "./logs",
  maxLogSize: parseInt(process.env.MAX_LOG_SIZE || "104857600"), // 100MB
  encryptionKey: process.env.ENCRYPTION_KEY, // 必须提供
};

// 启动时校验必需的环境变量
function validateConfig(config) {
  const required = ["ENCRYPTION_KEY"];
  const missing = required.filter((key) => !process.env[key]);
  if (missing.length > 0) {
    console.error(
      `Missing required environment variables: ${missing.join(", ")}`
    );
    process.exit(1); // 缺少必需配置，直接退出
  }
}
```

在 Docker 中传入环境变量：

```bash
# 命令行传入
docker run -e ENCRYPTION_KEY=abc123 -e PORT=3000 log-service

# 或者用 env 文件
docker run --env-file .env.production log-service
```

一条安全原则：**密钥、密码、Token 这类敏感信息不要放在环境变量里**。`docker inspect`、`/proc/<pid>/environ` 都能看到容器的环境变量。生产环境应该用专门的密钥管理服务（AWS Secrets Manager、HashiCorp Vault），或者在启动时从密钥文件读取。

## 优雅退出：不要暴力杀进程

[第 10 篇](/posts/nodejs-core/10-error-handling) 我们让进程在编程错误时崩溃重启。但重启之前，你需要给进程一个机会完成手头的工作——正在写入的日志文件要关掉、正在处理的 HTTP 请求要返回响应、数据库连接要释放。

Docker 停止容器时，先发 SIGTERM，等 10 秒后如果进程还在就发 SIGKILL 强制杀掉。你的代码需要在收到 SIGTERM 时完成清理：

```js
import { createServer } from "node:http";

const server = createServer(handleRequest);
const connections = new Set();

// 跟踪所有活跃连接
server.on("connection", (socket) => {
  connections.add(socket);
  socket.on("close", () => connections.delete(socket));
});

// 收到 SIGTERM 时优雅退出
function shutdown(signal) {
  console.log(`${signal} received, shutting down gracefully...`);

  // 停止接受新连接
  server.close(() => {
    console.log("Server closed");
    process.exit(0);
  });

  // 给正在处理的请求一个超时时间
  setTimeout(() => {
    console.log("Force closing remaining connections");
    for (const socket of connections) {
      socket.destroy();
    }
    process.exit(1);
  }, 10_000).unref(); // 不让这个 timer 阻止退出
}

process.on("SIGTERM", () => shutdown("SIGTERM"));
process.on("SIGINT", () => shutdown("SIGINT"));
```

`server.close()` 不会立即关闭——它只是停止接受新连接，已经建立的连接会继续处理直到完成。配合 `setTimeout` 设置一个强制退出的兜底，防止某些连接永远不关闭导致进程卡住。

`.unref()` 是一个容易被忽视的 API——它告诉 Node.js"如果这个 timer 是事件循环中唯一剩下的事件，不要等它，直接退出"。没有它，即使所有连接都关闭了，10 秒的 timer 也会阻止进程退出。

## 健康检查：告诉调度器"我还活着"

在容器编排（Docker Compose、Kubernetes）中，健康检查让调度器知道容器是否正常工作。不健康的容器会被自动重启或从负载均衡中移除。

实现一个真正有用的健康检查端点：

```js
async function handleHealthCheck(req, res) {
  const checks = {
    server: "ok",
    disk: await checkDiskSpace(),
    eventLoop: checkEventLoopDelay(),
  };

  const isHealthy = Object.values(checks).every((v) => v === "ok");

  res.writeHead(isHealthy ? 200 : 503, { "Content-Type": "application/json" });
  res.end(
    JSON.stringify({ status: isHealthy ? "healthy" : "degraded", checks })
  );
}

async function checkDiskSpace() {
  try {
    const stats = await fs.promises.statfs("/app/logs");
    const freePercent = stats.bavail / stats.blocks;
    return freePercent > 0.1 ? "ok" : "low"; // 剩余空间不足 10%
  } catch {
    return "unknown";
  }
}
```

为什么不只是返回 `{ status: 'ok' }`？因为进程活着不代表服务正常。磁盘满了、事件循环卡死了、数据库连不上了——这些情况下进程还在，但已经无法正常工作。健康检查应该验证**核心功能是否可用**。

Docker 中的健康检查配置：

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1
```

`retries=3` 意味着连续 3 次检查失败才标记为 unhealthy。避免因为一次网络抖动就重启容器。

## PM2：不需要 K8s 时的进程管理

如果你的部署规模不大（几台服务器、几十个进程），Kubernetes 可能过于复杂。PM2 是一个轻量级的 Node.js 进程管理器，开箱即用。

```bash
# 启动并监控
pm2 start dist/server.js --name log-service -i max

# cluster 模式，自动利用所有 CPU 核心
# -i max = 启动与 CPU 核心数相同的进程数
```

PM2 的 cluster 模式通过 `node:cluster` 模块（Node.js 内置）创建多个进程，共享同一个端口。和 [第 8 篇](/posts/nodejs-core/08-concurrency) 的 Worker Threads 不同，cluster 模式的每个进程是完全独立的，有自己的事件循环和内存，适合 I/O 密集型服务。

PM2 配置文件：

```js
// ecosystem.config.cjs
module.exports = {
  apps: [
    {
      name: "log-service",
      script: "dist/server.js",
      instances: "max",
      exec_mode: "cluster",
      max_memory_restart: "512M", // 内存超过 512MB 自动重启
      env: {
        NODE_ENV: "production",
        PORT: 3000,
      },
      log_date_format: "YYYY-MM-DD HH:mm:ss Z",
      error_file: "/var/log/log-service/error.log",
      out_file: "/var/log/log-service/out.log",
      merge_logs: true,
      autorestart: true,
      max_restarts: 10,
      restart_delay: 5000,
    },
  ],
};
```

`max_memory_restart` 是一个实用的防护——如果存在内存泄漏，进程会在内存超过阈值时自动重启，不会导致服务器 OOM。

## 日志收集：为什么 console.log 在生产不够用

`console.log` 在开发时够用，但在生产环境有几个问题：没有时间戳、没有日志级别、写入文件需要自己实现、不能集中管理多个实例的日志。

生产环境的日志应该写入文件，由专门的日志收集系统（Loki、ELK、Fluentd）统一收集和分析。一个简单但够用的方案：

```js
function writeLog(level, message, meta = {}) {
  const entry = {
    timestamp: new Date().toISOString(),
    level,
    service: "log-service",
    host: process.env.HOSTNAME || os.hostname(),
    pid: process.pid,
    message,
    ...meta,
  };
  // 写入 stdout，让 Docker 的日志驱动收集
  process.stdout.write(JSON.stringify(entry) + "\n");
}
```

把日志写到 stdout 而不是文件。Docker 和 Kubernetes 的日志驱动会自动收集容器的 stdout，统一发送到日志系统。这样你不需要在容器里管理日志文件轮转和清理。

## 部署前检查清单

上线之前，逐项确认：

- [ ] 环境变量已配置（通过 `.env` 文件或密钥管理服务）
- [ ] `NODE_ENV=production` 已设置（V8 会做更多优化，Express 不会输出详细错误堆栈）
- [ ] 优雅退出逻辑已测试（`kill -SIGTERM <pid>` 后进程能在超时前正常退出）
- [ ] 健康检查端点可用且检查了核心依赖
- [ ] 日志输出到 stdout，格式为 JSON（方便日志系统解析）
- [ ] Docker 镜像使用非 root 用户运行（`USER node`）
- [ ] 锁文件（`pnpm-lock.yaml`）已提交，构建时用 `--frozen-lockfile`
- [ ] 错误监控已接入（Sentry、自建错误收集服务）
- [ ] 进程管理器已配置自动重启和内存限制
- [ ] `console.log` 中没有输出敏感信息（密钥、Token、用户数据）
