## 为什么需要智能指针？

在构建分布式任务处理系统时，我们遇到了普通引用无法解决的场景：

1. 任务配置数据需要在多个 Worker 之间共享，但编译时不知道有多少个 Worker
2. 任务树结构是递归的，编译期无法确定其大小
3. 我们需要在保持引用不可变的同时，修改内部的计数器

这些场景下，Rust 的默认所有权规则过于严格。智能指针就是为此设计的——它们在拥有数据所有权的同时，提供了额外的能力：引用计数、运行时借用检查、内部可变性。

智能指针本质上是一个结构体，实现了 `Deref` trait（像引用一样使用）和 `Drop` trait（自动清理资源）。

## Box：堆分配与递归类型

`Box<T>` 是最简单的智能指针，它在堆上分配数据，栈上只保留一个指针。在任务系统中，Box 主要用于两个场景：处理递归数据结构和 Trait 对象。

```rust
// 任务树是一个递归结构，需要 Box 来打破无限大小
#[derive(Debug)]
enum TaskNode {
    Leaf {
        id: u32,
        name: String,
    },
    Branch {
        name: String,
        // 不用 Box 会产生无限递归，编译器无法确定大小
        left: Box<TaskNode>,
        right: Box<TaskNode>,
    }
}

fn main() {
    // 创建一个简单的任务树
    let task_tree = TaskNode::Branch {
        name: String::from("主任务"),
        left: Box::new(TaskNode::Leaf {
            id: 1,
            name: String::from("子任务 A"),
        }),
        right: Box::new(TaskNode::Leaf {
            id: 2,
            name: String::from("子任务 B"),
        }),
    };

    println!("任务树创建成功: {:?}", task_tree);
}
```

Box 的另一个用途是创建 Trait 对象，当我们要存储不同类型但实现了相同 trait 的值时：

```rust
trait TaskExecutor {
    fn execute(&self);
    fn name(&self) -> &str;
}

struct DataProcessingTask {
    data_size: usize,
}

struct NetworkTask {
    endpoint: String,
}

impl TaskExecutor for DataProcessingTask {
    fn execute(&self) {
        println!("处理 {} 字节数据", self.data_size);
    }

    fn name(&self) -> &str {
        "数据处理任务"
    }
}

impl TaskExecutor for NetworkTask {
    fn execute(&self) {
        println!("连接到 {}", self.endpoint);
    }

    fn name(&self) -> &str {
        "网络任务"
    }
}

fn main() {
    // Box<dyn TaskExecutor> 可以存储任何实现了 TaskExecutor 的类型
    let tasks: Vec<Box<dyn TaskExecutor>> = vec![
        Box::new(DataProcessingTask { data_size: 1024 }),
        Box::new(NetworkTask { endpoint: String::from("https://api.example.com") }),
    ];

    for task in &tasks {
        println!("执行: {}", task.name());
        task.execute();
    }
}
```

Box 是零开销抽象——解引用 `*boxed_value` 和直接使用值的性能完全相同，编译器会将其优化为直接访问。

## Rc：引用计数共享

当多个地方需要共享同一个数据，但都是只读时，可以使用 `Rc<T>`（Reference Counted）。在任务系统中，这很常见：多个任务引用同一个配置对象。

```rust
use std::rc::Rc;

#[derive(Debug)]
struct TaskConfig {
    max_retries: u32,
    timeout_seconds: u64,
    worker_count: usize,
}

#[derive(Debug)]
struct Task {
    id: u32,
    name: String,
    // 多个任务可以共享同一个配置
    config: Rc<TaskConfig>,
}

fn main() {
    // 创建共享配置
    let config = Rc::new(TaskConfig {
        max_retries: 3,
        timeout_seconds: 30,
        worker_count: 4,
    });

    // 创建多个任务，都引用同一个配置
    let task1 = Task {
        id: 1,
        name: String::from("数据导入"),
        config: Rc::clone(&config),  // 增加引用计数，不拷贝数据
    };

    let task2 = Task {
        id: 2,
        name: String::from("生成报表"),
        config: Rc::clone(&config),  // 引用计数继续增加
    };

    let task3 = Task {
        id: 3,
        name: String::from("发送通知"),
        config: Rc::clone(&config),
    };

    // 查看引用计数
    println!("配置被引用次数: {}", Rc::strong_count(&config));  // 4

    println!("\n任务 1: {:?}", task1);
    println!("任务 2: {:?}", task2);
    println!("任务 3: {:?}", task3);

    // config 仍然有效，因为还有任务引用它
    println!("\n原始配置仍然有效: {:?}", config);
}
```

Rc 使用引用计数来跟踪有多少个所有者。当最后一个 Rc 被销毁时，数据会被自动清理。Rc 只能在单线程中使用，因为它的引用计数操作不是原子操作，多线程使用会导致数据竞争。

## RefCell：内部可变性

Rust 默认的借用规则在编译期检查，但有时我们需要"在不可变引用下修改内部数据"。`RefCell<T>` 将借用检查推迟到运行时，允许这种操作。

在任务系统中，这很有用：我们可能想在保持任务队列不可变的同时，修改内部的统计信息。

```rust
use std::cell::RefCell;

#[derive(Debug)]
struct TaskQueue {
    tasks: Vec<String>,
    // RefCell 允许我们在不可变引用下修改内部值
    stats: RefCell<QueueStats>,
}

#[derive(Debug)]
struct QueueStats {
    total_added: u32,
    total_processed: u32,
}

impl TaskQueue {
    fn new() -> Self {
        Self {
            tasks: Vec::new(),
            stats: RefCell::new(QueueStats {
                total_added: 0,
                total_processed: 0,
            }),
        }
    }

    fn add_task(&mut self, task: String) {
        self.tasks.push(task);
        // 通过 borrow_mut() 获取可变引用
        self.stats.borrow_mut().total_added += 1;
    }

    fn process_next(&mut self) -> Option<String> {
        if let Some(task) = self.tasks.pop() {
            self.stats.borrow_mut().total_processed += 1;
            Some(task)
        } else {
            None
        }
    }

    // 即使 self 是不可变引用，也能读取统计信息
    fn get_stats(&self) -> (u32, u32) {
        let stats = self.stats.borrow();  // borrow() 获取不可变引用
        (stats.total_added, stats.total_processed)
    }
}

fn main() {
    let mut queue = TaskQueue::new();

    queue.add_task(String::from("任务 A"));
    queue.add_task(String::from("任务 B"));
    queue.add_task(String::from("任务 C"));

    let (added, processed) = queue.get_stats();
    println!("已添加: {}, 已处理: {}", added, processed);

    if let Some(task) = queue.process_next() {
        println!("处理: {}", task);
    }

    let (added, processed) = queue.get_stats();
    println!("已添加: {}, 已处理: {}", added, processed);
}
```

RefCell 的运行时检查意味着如果你违反借用规则（比如同时存在不可变和可变借用），程序会 panic 而不是编译失败。这是其代价——换取了灵活性，但失去了编译期保证。

## Rc + RefCell 组合

这是非常强大的组合——多个所有者共享可变数据。在任务系统中，这允许不同的组件共享和修改同一个数据结构。

```rust
use std::cell::RefCell;
use std::rc::Rc;

#[derive(Debug)]
struct SharedProgress {
    total: u32,
    completed: u32,
}

#[derive(Debug)]
struct Worker {
    id: u32,
    // 多个 Worker 可以共享并修改同一个进度对象
    progress: Rc<RefCell<SharedProgress>>,
}

impl Worker {
    fn new(id: u32, progress: Rc<RefCell<SharedProgress>>) -> Self {
        Self { id, progress }
    }

    fn complete_task(&self) {
        let mut progress = self.progress.borrow_mut();
        progress.completed += 1;
        println!("Worker {} 完成任务，进度: {}/{}",
            self.id, progress.completed, progress.total);
    }
}

fn main() {
    let progress = Rc::new(RefCell::new(SharedProgress {
        total: 10,
        completed: 0,
    }));

    // 创建多个 Worker，共享同一个进度对象
    let workers: Vec<Worker> = (1..=3)
        .map(|id| Worker::new(id, Rc::clone(&progress)))
        .collect();

    // 模拟任务完成
    workers[0].complete_task();  // Worker 1 完成任务，进度: 1/10
    workers[1].complete_task();  // Worker 2 完成任务，进度: 2/10
    workers[2].complete_task();  // Worker 3 完成任务，进度: 3/10
    workers[0].complete_task();  // Worker 1 完成任务，进度: 4/10

    // 所有 Worker 都能看到最新的进度
    let progress = progress.borrow();
    println!("\n最终进度: {}/{}", progress.completed, progress.total);
}
```

## Arc：多线程共享

`Arc<T>`（Atomic Reference Counted）是 Rc 的线程安全版本。它使用原子操作来维护引用计数，因此可以安全地在线程间传递。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

#[derive(Debug)]
struct Task {
    id: u32,
    data: String,
}

fn main() {
    // Arc + Mutex 是多线程共享可变数据的标准组合
    let tasks = Arc::new(Mutex::new(Vec::new()));

    let mut handles = vec![];

    // 创建多个线程，每个都向共享的任务列表添加任务
    for i in 1..=3 {
        let tasks = Arc::clone(&tasks);
        let handle = thread::spawn(move || {
            let task = Task {
                id: i,
                data: format!("来自线程 {} 的任务", i),
            };

            // 获取锁并添加任务
            let mut tasks = tasks.lock().unwrap();
            tasks.push(task);
            println!("线程 {} 添加了任务", i);
        });
        handles.push(handle);
    }

    // 等待所有线程完成
    for handle in handles {
        handle.join().unwrap();
    }

    // 查看最终的任务列表
    let tasks = tasks.lock().unwrap();
    println!("\n所有任务: {:?}", tasks);
}
```

Arc 和 Rc 的主要区别：

- Arc 使用原子操作，性能略低
- Arc 是线程安全的（实现了 Send 和 Sync）
- Rc 只能在单线程使用

## 循环引用问题

Rc 和 Arc 的引用计数无法检测循环引用，会导致内存泄漏。Rust 提供了 `Weak<T>` 弱引用来打破循环。

```rust
use std::cell::RefCell;
use std::rc::{Rc, Weak};

#[derive(Debug)]
struct Task {
    id: u32,
    name: String,
    // 使用弱引用指向父任务，避免循环引用
    parent: RefCell<Weak<Task>>,
    children: RefCell<Vec<Rc<Task>>>,
}

impl Task {
    fn new(id: u32, name: String) -> Rc<Self> {
        Rc::new(Self {
            id,
            name,
            parent: RefCell::new(Weak::new()),
            children: RefCell::new(Vec::new()),
        })
    }

    fn add_child(parent: &Rc<Task>, child: Rc<Task>) {
        // 子任务持有父任务的弱引用
        *child.parent.borrow_mut() = Rc::downgrade(parent);
        // 父任务持有子任务的强引用
        parent.children.borrow_mut().push(child);
    }
}

fn main() {
    let parent = Task::new(1, String::from("主任务"));
    let child1 = Task::new(2, String::from("子任务 1"));
    let child2 = Task::new(3, String::from("子任务 2"));

    Task::add_child(&parent, child1);
    Task::add_child(&parent, child2);

    println!("任务树创建成功");
    println!("父任务: {}", parent.name);
    println!("子任务数: {}", parent.children.borrow().len());

    // 升级弱引用为强引用（如果父任务仍然存活）
    let child1_ref = parent.children.borrow()[0].clone();
    if let Some(parent_ref) = child1_ref.parent.borrow().upgrade() {
        println!("{} 的父任务: {}", child1_ref.name, parent_ref.name);
    }
}
```

## 权衡与考虑

每种智能指针都有自己的适用场景和代价：

**Box：**

- 最轻量，只需一次堆分配
- 适合递归类型、Trait 对象、大数据传递
- 不支持共享所有权

**Rc：**

- 允许多个所有者，但只能在单线程使用
- 每次克隆都有引用计数的原子操作开销
- 配合 RefCell 实现可变共享

**RefCell：**

- 提供内部可变性
- 运行时借用检查，panic 而非编译错误
- 谨慎使用，优先考虑编译时检查

**Arc：**

- 线程安全的共享所有权
- 原子操作有性能开销
- 配合 Mutex 实现线程安全的可变共享

**选择建议：**

- 优先使用普通引用，零成本
- 需要堆分配时用 Box
- 单线程共享时用 Rc
- 多线程共享时用 Arc
- 需要内部可变性时用 RefCell
